1.1 系统全景骨架

一句话定位：豆包上车是搭载在上汽荣威D6X上的车载AI助手。Director（又名Planner）是整个系统的"大脑"——它不直接控制车辆，而是理解用户意图、制定计划、调度26个下游工具/Agent执行任务、跟踪进度并反馈用户。底层模型为字节豆包Seed 1.8，部署在云端。整套系统的设计哲学可以浓缩为三个原则：层层过滤、分阶段注入、数据回流。
三个时间面：
(1) 已上线 (S7)：单步复杂任务（一句话拆多个并行动作）、FC快车道（简单车控端侧毫秒级执行）。这是系统的"底座"，保证日常车控指令"打开车窗""空调调低"不经过大脑、不走云端推理。
(2) 落地中 (S8)：多步任务（串行依赖，如"去昨天游泳馆"需先查记忆再拿地址再导航）、条件任务（"十分钟后关座椅加热"）、触发器引擎上线。这是当前的开发主线。
(3) 实验 (S9-S10)：持续任务（"帮我介绍沿途风景"）、多Agent架构（4个静态Advisor+N个动态Advisor并行深度思考），向1.0长期架构靠拢。
设计三原则详解：
原则1 — 层层过滤：能用简单方法搞定的绝不漏到后面。用户说"打开车窗"，端侧白名单150ms就解决了，凭什么还要走云端大模型？每一层都是一道筛子：端侧白名单筛掉最确定的指令 → 云侧前置拒识筛掉不该接的 → 仲裁模型分流简单/复杂 → Director内部plan_describe前还有一道拒识。四道关卡把进入深度推理的请求量压缩到只剩真正"需要动脑子"的那部分。
原则2 — 分阶段注入知识：不是一开始就把所有信息塞给模型。端侧只需要query本身；到云侧前置时注入情景RAG（几条few-shot示例 + 知识片段）；真正进入Director时才注入完整的SP（角色定义、26个工具描述、69个参考示例、车辆规格、用户记忆、目标队列等~30K tokens）。这样做的核心原因是token成本和推理延迟——在前面几层处理掉70%以上的请求后，后面的"重炮"才用得值。
原则3 — 数据回流，后面推前面：Director/Planner是整个系统的"云端教师"。它处理过的复杂case会被标注、蒸馏回FC.SFT模型（让简单链路能处理的范围越来越广）；badcase分析结果会回流到拒识模型、仲裁模型的迭代训练中；白名单也会根据高频命中的新指令不断扩充。系统是一个持续学习的闭环。
真实走通案例："我好热啊"
用户说"我好热啊" → ASR转文字 → 端侧白名单没有精确命中（这不是"打开空调"这种标准指令） → 上云 → 云侧前置：不拒识（是合理需求） → 仲裁模型判断：这是模糊表达、需要推理（要查车内温度、判断开空调还是开车窗、还是调低温度）→ 分流到Director → Director调用search_vehicle_status_info查温度 → 发现30度 → 规划action_list: [开空调26度, 调风速3档, 关车窗] → 下发执行 → tool_feedback回传 → Director组织话术"帮你把空调打开到26度啦，车窗也关上了"。整个链路约1.5秒。
为什么不直接让大模型处理所有请求？
三个硬约束：(1) 延迟——云端LLM一次推理TTFT P50约500ms，加上回答生成又500ms，总共1秒起步。而端侧车控150ms就完成了，用户感知差距巨大。(2) 成本——每次Director调用消耗~40K tokens输入，按云端推理定价算，如果每句话都走Director，日均成本会是现在的5-8倍。(3) 稳定性——弱网/隧道/地库场景下云端不可用，端侧必须有兜底能力。所以"能不走大脑就不走大脑"不是可选优化，而是产品底线。
全景层面的常见坑：
(1) 链路断裂：ASR错误（如把"虹梅路"识别成"虹煤路"）会导致后续所有模块判断错误。好消息是Director有上下文纠错能力（用户第二句说"是梅花的梅"），但前提是对话要连续，一旦session断开纠错就失效。
(2) KV Cache断裂：Director的SP约30K tokens，如果动态变量（{{status}}/{{memory}}/{{goal_list}}）频繁变化，会导致每次请求都无法命中KV Cache前缀，推理延迟翻倍。这是长对话场景下性能退化的主因。
(3) 端云同步延迟：端侧信号库→云端信号库的同步链路如果出现延迟（如网络抖动），触发器引擎扫描到的信号可能是过时的。比如用户已经下车了，但云端还以为人在车上，导致误触发。

1.2 多级拒识和召回

三道安检类比：把用户的每句话想象成进机场——
第一道安检（端侧白名单 + 端侧LLM）：就像机场入口的身份证扫描。白名单里存了约500条最高频的确定性指令（"打开车窗""空调调到25度""下一首歌"），精确匹配。如果命中，直接放行到端侧执行，150ms完成，根本不上云。如果是明确的拒识（用户自言自语、乘客间聊天），端侧LLM也能直接判断不应答。这一道处理掉大约40-50%的请求。
第二道安检（云侧前置拒识 + 仲裁模型）：就像安检X光机。请求上云后，拒识仲裁意图模型（⑨）做四路分流：判定是车控的甩回端侧FC.SFT执行、判定是条件话的送去意图模型解析、判定是复杂需求的才放进Director、判定不该接的拒识并记录。这一道又过滤掉约30%。
第三道安检（Director内部拒识）：就像登机口的最后核查。即使请求进了Director，在plan_describe阶段之前还有一道拒识思考：在当前语境下（有完整的对话历史、用户记忆、车辆状态），是否应当拒识？如果是，content直接输出reject，不进后续plan流程。这是最后的兜底防线。
"空调调低一点"实走详解：
Step 1：用户说"空调调低一点"，ASR转成文字，带上speaker_name和position。
Step 2：进入第一级——端侧白名单查找。"空调调低"这种短指令在白名单里有精确匹配（属于vehicle_basic_control的高频子集）。
Step 3：命中！输出格式为 {"type":"directExecute","reject_ornot":"react","task_list":[{vehicle_basic_control: "空调温度降低2度"}]}。
Step 4：直达端侧车控执行，150ms内空调温度调低2度，弹卡片通知用户。
全程不经过云端、不经过Director。
对比："车里好吵调安静点"实走详解：
Step 1：ASR转文字。注意这句话是模糊复杂表达——"好吵"可能指音乐太大声、可能指车窗开着路噪大、可能指空调风量太大。
Step 2：第一级白名单——未命中。"车里好吵调安静点"不在白名单中（白名单只有确定性指令，不处理模糊表达）。
Step 3：上云，进入第二级。仲裁模型判断：这不是纯句法能解的车控指令、需要结合情景推理（当前音乐音量、车窗状态、空调风量）→ 判定为"复杂" → 分流到Director。
Step 4：Director接收query，注入完整context（当前车内正在播音乐音量80%、前排车窗开着、空调风速4档）→ plan_describe推理 → 输出action_list: [music音量调到40%, 关闭前排车窗, 空调风速调到2档] → talk_content: "帮你把音乐调小了、车窗关上了、空调风也调低了，这样安静多了"。
全程约1500ms。这就是为什么约30%的车控指令是模糊复杂表达，必须走Director。

为什么是三级而不是一级或两级？
如果只有一级（端侧白名单），那所有模糊表达都处理不了——"我好热"不在白名单里，系统就卡死了。如果只有两级（端侧+Director），所有未命中白名单的请求都直接打到Director，Director就成了瓶颈——它要同时处理"打开空调"（本来FC就能搞定）和"帮我规划一条去昨天那个游泳馆的路线"（真正需要推理的），既浪费推理资源又拉高延迟。第二级（云侧仲裁）的存在，就是在端侧和Director之间加了一个"快速分诊台"：能走简单FC的走FC（300ms），需要条件解析的走解析，只有真正复杂的才进Director。三级架构把Director的负载压缩了60-70%。
拒识环节的典型坑点：
(1) 过度拒识（误拒）：用户语气比较随意地说"这个温度还行吧"，端侧LLM可能误判为"用户在自言自语"而拒识。但实际上用户是在确认空调设置、期待一个回应。解法：在对话上下文中如果上一轮是系统在调温度，当前轮的模糊表达大概率是对系统行为的反馈，不应拒识。
(2) 过度召回（漏拒）：副驾对主驾说"你帮我把空调调低"，ASR识别后系统以为是对它说的，直接执行了。解法：结合speaker_position和对话方向判断——如果副驾面朝主驾说话（视觉VLM可以判断面部朝向），大概率不是对系统说的。
(3) 白名单维护成本：白名单需要持续人工维护，新功能上线后对应的指令变体要及时添加。如果一个新工具上线了但白名单没更新，用户的简单指令就会被推到Director，白白增加延迟。这就是"数据回流"原则的落地点——Director处理过的高频简单case要定期蒸馏回白名单和FC.SFT。

1.3 仲裁模型

仲裁模型的位置和选型：
仲裁模型位于"三级拒识漏斗"的第二级，是云侧的快速分诊台。选型为 Seed 1.6 flash SFT——为什么用flash？因为它要快。仲裁模型的职责不是给答案，而是快速做出路由决策：这句话该走哪条路。所以用轻量级flash模型 + SFT微调足矣，不需要动用Seed 1.8的完整推理能力。仲裁模型本身的推理时间控制在50-100ms，不能成为延迟瓶颈。
情景RAG的关键设计：注入知识但不输出结果：
这是仲裁模型最容易被误解的设计点。情景RAG会在仲裁阶段向模型注入用户记忆片段、当前车况、情景信息、以及少量few-shot示例，但仲裁模型的输出格式是 {"type":"modelExecute","knowledge":"...","agentLimit":[...]}——注意它只输出路由决策和关联知识摘要，不输出最终答案。为什么？因为仲裁模型是flash级别的轻量模型，让它直接给答案质量不够；但让它带着情景信息做分流判断，准确率会高得多。真正的答案留给Director这个"重炮"去生成。
四条分流路径的判定规则：
路径1 - FC快车道（~300ms）：条件是"未拒识 + 车控意图 + 命中6大标准意图"。6大意图包括空调控制、座椅控制、车窗车门、灯光控制、导航基础控制、音乐播控。满足这三个条件才走FC.SFT，保证快速且准确。
路径2 - 条件话 → 意图模型解析：识别出用户表达中包含时间条件或触发条件（"十分钟后""到家的时候""超过25度就"），转给意图模型做条件提槽，生成触发器协议。
路径3 - Director（~500ms+500ms）：兜底路径。所有不满足路径1和2的、以及超时的请求都进Director。包括：模糊表达、多步任务、需要记忆推理、需要联网搜索、闲聊等。
路径4 - 拒识：判定为不应接的（自言自语、乘客间对话、无意义噪声），静默记录不应答。
超时兜底逻辑：
如果云端仲裁超时（弱网、隧道场景），系统不会让用户空等。回退逻辑是：端侧有结果就用端侧的（白名单命中），端侧也没有就走Director兜底（Director是最终兜底层）。如果连云端Director都不可达，则端侧LLM给出最基本的应答（"现在网络不太好，基础车控我可以帮你，复杂的等一下"）。永远不能让用户觉得系统"死了"。
简单路径 vs 复杂路径的完整对比：
简单路径实例："打开座椅加热"
→ 仲裁判定：未拒识 + 车控意图 + 命中"座椅控制"标准意图 → FC.SFT快车道
→ FC模型直接输出function call: vehicle_basic_control("打开主驾座椅加热") → 端侧执行
→ 总耗时：ASR 200ms + 仲裁 80ms + FC 100ms = ~380ms

复杂路径实例："去昨天那个游泳馆，顺便帮我查一下今天水温"
→ 仲裁判定：需要记忆推理（"昨天那个"= 要查记忆）+ 多步串行 + 联网搜索 → Director
→ Director: step1 调 search_user_memory("昨天去的游泳馆") → 得到"星辉游泳馆 浦东新区XX路" → step2 调 route_planning_qa("导航到星辉游泳馆") → step3 调 web_search("星辉游泳馆今天水温") → 综合输出
→ 总耗时：ASR 200ms + 仲裁 80ms + Director 3轮推理(500ms*3) + 工具调用耗时 = ~2500ms
注意Director是流式输出的，talk_content "好的，我帮你查一下昨天那个游泳馆..." 在第一轮推理结束后就开始吐给用户了，用户不会感觉等了2.5秒。
仲裁误判的三大根因深度分析：

根因1：意图分类错误叠加简单过召（双重路由错误）
这是当前最严重的问题，属于bug级别。具体表现：语气演绎类表达（用户模仿别人说话）被误判为车控意图、Face ID相关query被归入车控、人名座位映射被简单过召。测试报告中8类问题中多数都是"意图错误分类 + 仲裁简单过召"的组合——意图模型先把query归错了类，然后仲裁模型又判定为"简单"，结果送到FC.SFT链路后完全处理不了。本质原因：仲裁模型训练时的正样本覆盖不全，大量边界case没有标注。

根因2：判断标准单一——只看句式复杂度
当前仲裁模型主要依据句式复杂度来判断简单/复杂。"导航去浦东机场"句式简单但确实简单；"帮我查一下上次那个什么什么餐厅"句式也不算复杂，但需要记忆推理。"十分钟后关空调"句式极其简单，但它是条件任务，FC根本处理不了。核心建议：重训仲裁模型，从"纯句式复杂度"改为"句式 + 意图（SFT能处理的走，其他都进Planner）"综合判定。具体做法是把SFT链路能处理的case做成正样本，其他全部做成负样本（应该进Director），以此为训练目标。

根因3：情景信息缺失导致误判
"车里好吵调安静点"——这句话如果只看文字，句式像一个简单车控指令，仲裁模型可能把它分到FC快车道。但实际上"好吵"可能指音乐、车窗噪音、空调风声中的任意一种，需要查车况才能决定怎么做。如果情景RAG恰好没有召回当前音乐播放状态或车窗状态，仲裁模型就缺少了关键判断依据。上游链路问题：记忆模块滞后、向量检索召回不全、context可用性差——这些在Director优化讨论会上被明确列为四大问题之一。
仲裁相关的排查路径（Badcase排查SOP）：
当遇到用户反馈"系统答错了/没答/执行错了"时，按以下顺序排查：
Step 1 — 链路问题（走错模块了吗？）：query到底走了哪条路？是白名单命中？还是FC快车道？还是Director？用链路日志确认。如果走错了模块，问题出在仲裁层。
Step 2 — 工具描述问题（SP里工具定义清楚吗？）：如果确实走到了Director但执行错误，检查是不是Director的SP中工具描述不够清晰，导致模型调错了工具或传错了参数。
Step 3 — 上下文问题（缺记忆/车况/情景吗？）：Director拿到的context够不够？用户记忆有没有召回？车况信号是不是最新的？情景RAG有没有注入关键信息？
Step 4 — 模型能力问题（信息全了但推理还是错了？）：如果前三步都没问题，query走对了模块、工具描述清楚、context完整，但输出还是错——那就是模型本身的推理能力不够。这种case要攒起来做SFT训练数据。

当前数据：177条badcase中，任务规划与拆解问题占14%（26条）、稳定性/交互问题占7.3%（13条）、下游工具问题占6.2%（11条）。仲裁误判导致的问题没有单独统计，但根据测试报告分析，约30-40%的badcase根因可追溯到仲裁分流阶段。
