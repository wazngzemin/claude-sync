§1 产品定位：端侧触发器是什么

端侧触发器 = AI Box 上的手脚 + 眼睛 + 闹钟

它跑在车机（AI Box）上，做三件事：掐表（本地定时器，断网也准）、开关 VLM（管 AlwaysOn 任务的启停）、硬条件判定（挡位变P、温度超30度，认死规矩不判语义）。它不联网推理，不做语义理解——贵的活全交给云端触发器。

不是什么：端侧触发器不是"又一个触发器"。6.22 西钺警告"不要开发两个触发器"——逻辑上它和云端触发器是一个触发器的两半，一个条件只判一次。端侧做的事=物理上只能在车端做的事。

1.1 为什么必须有端侧这一半

物理约束只能在端侧做的事如果放云端会怎样
掐表"10分钟后关加热"→本地倒计时车在隧道断网→云端够不着→闹钟失灵
离摄像头近舱内画面合规上不了云→VLM 眼睛只能长在车上隐私违规
开关在手边上电/下电/按钮→端侧直接感知→开关 AlwaysOn多一跳网络延迟→上电后几秒才开始观测
信号就在本地挡位、速度、导航状态→端状态直接读先传云端再判→延迟+带宽浪费

1.2 端侧触发器的三个不做

不做为什么谁做
不做语义判断"宝宝哭没哭"要理解自然语言描述→需要大模型云端触发器事件判断引擎（L1/L2/L3）
不做 SP 拼接7.6 丁彬："SP 拼接先不做，由上游业务方推动下游沟通"一期不做；二期归云端触发器
不做冲突仲裁"开窗"撞"关窗"→要全局视角→端上各管各的看不到别人云端触发器⑮注册+仲裁

§2 系统架构（含 7.6 变更后链路）

2.1 变更前 vs 变更后（7.6 定版）

核心变更：VLM 结果流向从 "VLM → AI Service 直接" 改为 "VLM → 端侧触发器 → AI Service（数据中心）"
原因：AI Box 架构统一原则——"谁调用我，结果返回给谁"（丁彬 7.6 ~10:21）
结论：端侧触发器调用 VLM → VLM 把结果返回给端侧触发器 → 端侧触发器整包透传给 AI Service（数据中心）
对比项变更前变更后（7.6 定版）
VLM结果流向 端侧触发器 → 开关VLM → VLM结果 → 直接给 AI Service 端侧触发器 → 开关VLM → VLM结果 → 返回端侧触发器 → 端侧触发器 → AI Service
端侧触发器角色 只管开关，不碰结果 开关 + 结果透传中继
是否解析 AI Service 侧解析 端侧触发器不解析，整包透传原始字符串；数据中心自己解析
设计原则 触发者和结果接收者分离 谁调用，结果返回给谁——与赛力斯 AI Box 架构一致
丁彬原话（7.6 ~11:18）："你都不用管里头是啥，你就整包给就行了。不解析了，整包透传，所以就更简单一些了。模型给你啥你就给啥，它是一串字符串，你不需要解析里面的东西了。"

㉔-8 AlwaysOn 点火 ✅ 一期做

职责：管理 3 个 VLM 常驻任务（舱外 AlwaysOn / 舱内动态低频 / 舱内静态高频）的开和关。

AlwaysOn 任务观测周期开条件关条件
舱外 AlwaysOn
天气/路况/路面		 3 秒 系统上电（单条件） 系统下电（单条件）
舱内动态 AlwaysOn
动作/神态/情绪/手势		 5 秒 系统上电 AND 设置中 Awake 按钮开
（并——两个同时满足才开）		 系统下电 OR 按钮关
（或——任一满足就关）
舱内静态 AlwaysOn
座位/性别/年龄/物品		 5 分钟 同上（AND） 同上（OR）
7.6 定版原话：丁彬 ~09:34："上面那个开它是一个并的关系，底下那个关它是个或的关系。开就是这两个条件必须同时满足，才能开。"
王泽民 ~09:45 确认："上边那个开，不是你就一个条件，不是就系统上电嘛。上电也是并的。下边是关是或，然后开是并嘛。"

点火流程

Q: 上电后 Button 还没开，舱外先开了，舱内呢？
舱外立即开（只看上电），舱内等 Button 打开后才开。两个独立判断，互不阻塞。
Q: 行驶中用户关了 Button，舱外受影响吗？
不影响。舱外只看上电/下电，和 Button 无关。
Q: 开关信号从哪来？
上电/下电 = 系统广播（AI Box 系统事件）；Button = 设置页状态（赛力斯 UI 控件状态）。

㉔-9 VLM 调度 + 整包透传 ✅ 一期做（重定义）

职责变更：原名"VLM 五态归一"——原先设计是端侧触发器要解析 VLM 输出的五种状态进行归一化。7.6 会后取消解析，改为整包透传。

子能力说明
VLM 调度调摄像头（前/后/左/右）、决定跑哪个任务、管优先级、超时释放、oneshot 插队
Topic 订阅AlwaysOn 开启后，订阅对应 topic → VLM 按周期返回结果 → 端侧触发器接收
整包透传VLM 返回的原始 JSON 字符串，不解析、不拆字段，原样转发给 AI Service 数据中心
oneshot 执行收到云端下发的单次视觉查询请求 → 插队执行 → 结果返回云端
不解析意味着什么：端侧触发器不关心 VLM 输出内容是"座位位置=主驾"还是"情绪=哭泣"——拿到的是一个 string，直接推给 AI Service。数据中心自己做拆字段、分发、入库。
丁彬 7.6 ~12:38："模型给你啥你就给啥，它是一串字符串，你不需要解析里面的东西了。"

㉔-7 本地定时器 ✅ 一期做

职责：管理用户注册的定时类任务——本地掐表，断网不停。

能力说明
本地倒计时"10分钟后关加热"→本地倒计时10分钟，到点直接执行
循环定时"每2分钟告诉我比分"→循环触发，最多循环30次
断网兜底车进隧道断网→定时器不受影响→到点该干啥干啥
持久化任务存本地→掉电/重启后恢复未完成的定时任务
场景：用户进隧道前说"10分钟后关座椅加热"。Director 拆解→向云端触发器注册定时任务→云端校验通过→下发端侧→端侧本地倒计时10分钟。3分钟后车进隧道断网——没关系，端侧闹钟在本地，7分钟后到点直接关加热。

㉔-J 硬条件判定 ✅ 一期做

职责：端侧信号的运算符比对——只认死规矩，不判语义。

判定类型举例信号来源
等于/变更为挡位 states 变更为 P（R→P）端状态
大于/小于距离 distance ≤ 300m导航信号
包含天气 weather_type 包含 [雾,大雾]外部天气数据
变更为不包含天气恢复 weather_type 变更为 不包含[雾]外部天气数据

端侧自闭环场景（不需要云端参与）：

场景条件动作
雾天→开后雾灯weather_type 包含 [雾,大雾]开后雾灯
雨天→湿滑模式weather_type 包含 [小雨,中雨,大雨,暴雨]开湿滑模式
雪天→雪地模式weather_type 包含 [小雪,中雪,大雪]切雪地模式
关键约束：硬条件判定只用运算符——等于、大于、小于、包含、变更为。不用模型、不判语义。"宝宝哭没哭"这种需要理解自然语言的，不在这做，归云端 L1/L2/L3 事件判断引擎。

㉔-U 透传上报 ✅ 一期做（7.6 新增）

职责：7.6 会后新增的职责——将 VLM 结果按"谁调返给谁"原则整包上报给 AI Service。

上报内容格式频率
舱外 AlwaysOn 结果VLM 原始 JSON 字符串每 3 秒
舱内动态 AlwaysOn 结果VLM 原始 JSON 字符串每 5 秒
舱内静态 AlwaysOn 结果VLM 原始 JSON 字符串每 5 分钟
上报链路：VLM topic → 端侧触发器订阅接收 → 原始字符串不做任何解析 → 推送给 AI Service 数据中心 → 数据中心透传 → 云端各消费方（云端触发器/Context/Advisor）各取所需

丁彬 7.6 ~10:57："你订阅，它这块有一个 topic，你一订阅，它就周期性地把结果给你，你拿到结果，你就都推给 AI Service 就行了。"

上报协议（一期）

{
  "task_type": "cabin_outside_alwayson",   // 任务类型标识
  "raw_output": "...(VLM原始输出字符串)...",  // 不解析·整包
  "timestamp": 1720252800000               // 端侧时间戳
}

字段名和具体协议待与丁彬（VLM）+ 仁杰（数据中心）下一步对齐。

㉔-SP SP 注入 ❌ 一期不做

原设计：收到云端下发的增量 SP → 注入 AlwaysOn 槽位 → VLM 按新 SP 执行。

不做原因：
① 7.6 丁彬 ~06:02："倒也没说不上，就这东西大家觉得没聊清，然后拆开了，就不要一块阻塞的。"
② 7.6 丁彬 ~28:20："像这种 SP 这种的，是由上游业务方来推动下游来去沟通这件事情，讲清楚逻辑，然后研发判断要走哪条链路。"
③ 7.6 张荣 ~30:03："第三块就是拼接 SP 这个东西，端侧的 VLM 和端云两边的触发器都存疑，需要找到源头。"

一期 scope：先保三个 AlwaysOn 能上，用固定 SP（系统预设），不接受动态 SP 注入。

二期规划：动态 Advisor 创建观测任务（"帮我看着宝宝毯子"）→ 云端触发器拼 SP → 下发端侧 → 端侧触发器替换当前 SP → VLM 用新 SP 执行。

4.2 开关逻辑真值表

任务开 = 条件关 = 条件开逻辑关逻辑
舱外 系统上电 系统下电 单条件 单条件

舱内静态 系统上电 + Button 开 系统下电 / Button 关 AND（并） OR（或）

步模块做什么

2㉔-9 VLM 调度发起 oneshot VLM 请求：用后置摄像头拍一帧 → SP="当前停车位是否有充电桩"

条件拆解：

P0 · 端侧自闭环

链路类型：端侧自闭环（信号→硬条件→直接执行，不上云）  端侧角色：信号监听 + 硬条件判定 + 直接下发执行

步模块做什么
1㉔-J 硬条件判定监听天气信号 weather_type，检测到包含 [雾/大雾] 或 [雨类] 或 [雪类]
2㉔-J 穿越检测F→T 变更检测：之前不包含雾→现在包含雾 = 穿越 → 触发
3㉔ 直接执行下发车控指令：开后雾灯 / 开湿滑模式 / 切雪地模式
4㉔-J 恢复检测天气恢复（变更为不包含雾/雨/雪）→ 关闭对应模式
为什么是端侧自闭环：
① 天气信号已经在端侧（外部天气数据/VLM 舱外 AlwaysOn 也能看到）
② 判断逻辑是死规矩（包含雾=开雾灯，不需要语义理解）
③ 动作是车控指令（直接下发赛力斯 CDC）
④ 断网也能干（隧道里下雪照样切雪地模式）

频控：可配置——周期内几小时/几天/自然日一次（7.2 泽民确认）
防抖：持续收到雾天状态 → 只在 F→T 边沿触发一次，不反复弹窗

§8 异常处理策略（7.4 评审通过——唯一通过的议题）

异常类型处理策略设计原则补充说明
网络波动积压 周期类：只传最新帧，压包丢弃中间帧
oneshot：暂存本地，网络恢复后续发一次		 周期=时效优先（过期没用）
oneshot=可靠优先（丢了无法重现）		 AlwaysOn 断网：本地缓存最新一帧，恢复后只传最新
信号 TTL 过期 信号作废→依赖该信号的场景 frozen 防过期数据误触发 天气信号 TTL = 30min；挡位信号 TTL = 实时
端侧不留存 端侧不做数据持久化，只管透传。断网积压的续发完即清 端侧算力/存储有限 唯一持久化的是：定时器状态 + 任务列表
时序脏数据 人换座位→先上报一帧空数据清掉前值→再上报新值 防旧 face_id 残留导致误判 超然（7.4）提出的方案：一空一实保证数据干净
VLM 超时 单次 VLM 执行超过超时阈值→释放资源→跳过本帧 不阻塞后续周期 超时阈值待与丁彬对齐（3s? 5s?）
