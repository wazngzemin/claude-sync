一、触发器到底是什么？为什么需要它？

你在做的产品叫"豆包上车"（Driver Agent 2.0 驾驶代理2.0版本），是装在上汽荣威D6X车上的AI助手。

这个AI助手有一个"大脑"叫 Director 导演/决策者——用户说话的时候，Director负责理解用户的意思、做决策、调工具执行。

但问题是：用户不说话的时候呢？

比如副驾驶睡着了，AI能不能自动把音量调小？比如车速超了130，能不能自动提醒？比如用户说了"10分钟后关座椅加热"，10分钟到了谁来执行？

这些事情，Director做不了——因为Director只有用户说话的时候才工作。

所以需要一个"7×24小时值班的哨兵"，一直盯着车上的所有传感器，一旦发现"某个条件满足了"，就主动通知该通知的人去做事。

这个哨兵，就是触发器（Trigger 触发器）。

触发器干3件事：
①盯信号：车速多少？温度多少？有人睡着了吗？——持续监控
②判条件：温度超过28度了吗？副驾安全带系上了吗？——条件匹配
③通知人：满足了→告诉Advisor（Advisor 顾问）或Director去做事
触发器在整个系统里的位置

用户说话时走的是一个"三层漏斗"：
第1层：RAG 检索增强生成，规则匹配——处理"打开车窗"这种明确指令
第2层：情景RAG + 仲裁——注入记忆/车况，判断简单还是复杂
第3层：Planner/Director 规划器/导演——云端大模型深度推理

触发器不在这个漏斗里。它是漏斗之外的一个独立系统，有自己的输入（信号）和输出（事件通知）。
唯一和漏斗交叉的地方：用户说了条件话（"10分钟后关座椅加热"）→ 漏斗处理 → Director把条件注册到触发器 → 触发器到时间了再通知Director去执行。

三、信号：触发器的"眼睛和耳朵"

触发器要盯着车上的情况，那它怎么知道车上发生了什么？——靠信号（Signal 信号）。

信号就是"车上发生的事"的数据化表达。比如"车门开了"是一个信号，"温度28度"是一个信号，"VLM看到有人睡觉"也是一个信号。

车上能产生信号的地方一共有6类。但是！触发器不直接去读这些原始信号——它只读"信号库"（Signal Library 信号库，类似数据仓库）里已经标准化好的数据。这是铁律第3条（后面会讲）。

四、云端触发器引擎⑮：18个子模块

⑮是整个触发器系统的心脏。你作为触发器PM，这就是你负责的核心。

它有18个子模块（sub-module 子模块），按功能分成6组。我一组一组讲，每组讲完告诉你和你有什么关系。

你可以把⑮想象成一条流水线：
规则进来（注册组）→ 检查信号（判定组）→ 防止误触（防抖/冷却）→ 排优先级（仲裁组）→ 打包发出去（输出组）→ 处理视觉需求（6.30新增组）

🔵 第1组：注册组（15-1 ~ 15-3）—— 接收规则，开始盯信号

编号名称做什么（人话）
15-1 注册校验
iCanDo "我能做"，校验是否能处理		 一条新的触发规则要注册进来。15-1先做"入学审查"：

① 这条规则用到的信号（signal_key 信号字段名）在信号库里存在吗？——比如你写了一个条件"当signal_key=abc时触发"，但信号库里根本没有abc这个信号，那就直接拒绝

② 要执行的动作（action 动作）允许吗？——比如你要求"触发后自动开车门"，但开车门不在允许列表里（太危险），拒绝

（版本 + 灰度grayscale 灰度，指只对部分用户/车辆生效）		 校验通过后存入数据库。每个规则叫一个"场景"（scene 场景），有唯一编号 scene_id。

15-3 信号扫描
（每5秒一次心跳）		 每5秒读一次⑫云端信号库的快照（snapshot 快照，某一时刻的数据截面）。看看哪些信号的值变了。

用的是incremental subscription 增量订阅，只看变化的部分，不是每次全量扫描。

🟡 第2组：判定组（15-4 ~ 15-7）—— 条件满足了吗？别误触发

编号名称做什么（人话）
15-4 条件判定
（核心运算）		 核心模块！拿到变化的信号，去匹配所有已注册的场景规则。

支持的运算符（operator 运算符）：
= 等于 / > 大于 / < 小于 / >= 大于等于 / <= 小于等于
range 区间，比如"温度在25到30之间"
change_to 变更为，比如"档位从D变成P"——只在变化的那一刻触发

多条件组合：AND 并且，所有条件都要满足 / OR 或者，满足任一条件就行

举例：场景"副驾上车后唱小星星"的条件 = {副驾占座==有人 AND 安全带==系上}
信号变化：副驾安全带从"未系"→"系上"
15-4检查：副驾占座==有人？✅ 安全带==系上？✅ → 条件满足！
15-5 VLM任务编译
（7个字段）		 如果触发条件需要VLM去看（比如"看看后排有没有小狗"），这个模块负责把需求"翻译"成VLM能理解的标准指令。

翻译出来的指令有7个字段：
① task_id 任务编号（如ADV-C-007）
② task_goal/query 任务目标/观察什么（如"观察后排有无犬类"）
③ priority 优先级（P0~P4，P0最高）
④ timeout 超时时间（比如30秒没出结果就算超时）
⑤ mutex_group 互斥组（同一组的任务不能同时执行，比如VLM_CABIN互斥组=舱内摄像头同时只能做一个VLM任务）
⑥ trigger_source 谁发起的（如advisor_comfort=舒适顾问发起）
⑦ expire_at 自动过期时间
15-6 抖动过滤
（Jitter Filter 抖动过滤）		 防止信号在阈值附近来回跳，导致反复触发。

什么是抖动：比如你设了"车速>130就提醒"，车速在129和131之间来回波动。如果不防抖，就会"提醒-取消-提醒-取消"来回跳。

怎么防：设一个"死区"（±区间）。比如设125~135的死区——只有车速从125以下涨到135以上，才算真正超速。在死区里来回跳的不算。

VLM也需要防抖！VLM可能前一帧说"有小狗"，下一帧说"没有"。张兰若原话："但凡出现一次错误，触发器就会有问题。"所以VLM信号要"连续N帧一致"才算有效。
15-7 冷却频控
（Cooldown 冷却）		 控制一个场景的触发频率，防止反复骚扰用户。

4个维度的控制：
· total_count 总次数上限：比如总共最多100次
· daily_count 每天上限：每天最多100次
· power_count 每个上电周期上限：每次启动车最多10次
· execute_interval 两次触发的最小间隔：至少隔60秒

举例："有人打哈欠"如果每5秒VLM都检测到，不做频控就5秒通知一次——用户会疯。设了"间隔10分钟"就变成最多10分钟提醒一次。

编号名称做什么（人话）
15-8 优先级仲裁
Priority Arbitration		 多个场景同时触发时，按优先级排队。高的先执行。

5级优先级：
P0 安全（最高）：疲劳驾驶/碰撞预警——这些抢占一切
P1 用户主动：用户自己设的条件（"超25度开空调"）
P2 顾问推荐：Advisor的主动服务建议
P3 系统预设：系统默认的舒适调节（环境光/温度自动调节）
P4 内容推荐（最低）：娱乐/信息建议

互斥组（mutex group 互斥组，同组的不能同时执行）：高优抢占低优。被抢占的暂停，等高优做完再恢复。

15-10 阈值穿越检测
（Threshold Crossing 阈值穿越）		 6.22新增的最重要的模块！

什么叫"穿越"：一直盯着一个信号值。当这个值从"不满足条件"变成"满足条件"的那一瞬间 = 穿越 = 触发事件。

举例：条件是"有人睡着了"
· 上一帧：没人睡 → 当前帧：有人睡 → 穿越！触发事件"有人睡着了"
· 上一帧：有人睡 → 当前帧：有人睡 → 不是穿越。已经在睡了，不重复触发

英文缩写：F→T = False→True 从不满足到满足

为什么需要穿越：如果不做穿越，只做"条件满足就触发"，那"有人在睡觉"这个条件每5秒VLM输出一次都满足，就会每5秒触发一次——用户会被反复打扰。穿越确保只在状态变化的那一刻触发一次。

施佳杰的定义："事件是唯一时刻的触发；condition（条件）是Context（上下文）中的持续状态。"

铁律二：端侧结果不直推Advisor，必须走信号库

什么意思：端侧（VLM/定时器等）的结果路径只能是：㉔ → ④端侧信号库 → ⑫云端信号库 → ㉑Context（上下文）

不允许：㉔直接推给Advisor / ㉔直接推给Director

为什么：
①信号库是唯一的"事实源头"（Single Source of Truth 单一事实来源）。绕过就会出现矛盾——信号库说"没有小狗"，但Advisor说"有小狗"
②断网时端侧结果先缓存在本地队列，等联网后同步——如果直推就没法做断网缓存
铁律三：触发器只读信号库，不直接监听原始信号源

什么意思：⑮读的是⑫云端信号库里标准化后的数据，不是直接去读CAN总线或直接调VLM。

不允许：⑮直接读CAN总线 / ⑮直接调VLM

为什么：解耦（Decoupling 解耦，让两个模块互不依赖）。信号源的格式千差万别——CAN是二进制、VLM是JSON、App是埋点。信号库做标准化转换，触发器只需要读标准格式，不需要关心"这个数据原来长什么样"。
铁律四（最核心）：condition→Context / event→⑮主动push

什么意思：
· condition（条件/状态）= "车里现在有小朋友" → 写入Context → Advisor推理时自动带上
· event（事件）= "小朋友刚上车" → ⑮组装后主动push给Advisor

不允许：Advisor自己去轮询（polling 轮询，反复问"有变化吗？有变化吗？"）Context来发现事件

为什么：轮询太慢太浪费。事件是"某个时刻发生的事"，需要触发器在发生的那一瞬间主动push——这就是触发器存在的核心价值。施佳杰原话："事件应该是更有价值的信息，不等同于状态。"

十、四条数据链路

整个系统的数据从进来到出去，有4条主要路径。每条标一个颜色。你可以把每条路径想象成一条"高速公路"——从起点到终点，中间经过哪些"收费站"（模块）。
🔵 蓝色链路：车控快速通道
场景：用户说"打开车窗"——简单明确的车控指令
走法：① 用户说话 → ② ASR转文字 → ⑨ 仲裁判"蓝路" → ⑤ 离线FC白名单匹配 → ⑥ 命中 → ⑦ 直接执行 → ⑧ 反馈卡片
特点：全程不上云，毫秒级响应。不经过触发器。
🟢 绿色链路：对话/查询通道
场景：用户说"我好无聊" / "帮我搜好吃的"——需要AI推理
走法：① → ② → ⑨ 仲裁判"绿路" → ⑲ Director汇入 → ⑳ 推理 → 调工具/TTS播报/车控执行
特点：工具反馈会循环回⑲（最多4轮），直到Director判断可以回复。不经过触发器。
🟠 橙色链路：条件/定时任务
场景：用户说"超25度开空调" / "10分钟后关座椅加热"——设条件
注册阶段：⑨仲裁→⑩意图解析→⑬配置平台注册→⑮引擎入库
或：⑳Director推理后→L33注册到⑮
触发阶段：⑮扫描信号→条件判定→防抖→频控→仲裁→分类→push Advisor/Director 或 下发端侧㉔执行
特点：经过触发器——这是触发器参与的第一种链路。
🟣 紫色链路：主动服务（用户没说话！）
场景：用户没说话，但车上信号变了（有人睡着了/温度升高了/检测到充电桩了）
走法：③ 信号源产生数据 → ④ 端信号库 → L9同步云端 → ⑫ 云信号库 → ⑮ 扫描→穿越检测（F→T!）→ 组装事件push → ⑯/⑰ Advisor → ⑲ Director推理 → 回复用户
特点：这就是触发器存在的核心价值——没有用户发起，完全由信号驱动。让系统从"被动应答"变成"主动服务"。

十二、6个真实Case

Case 1：定时器 —— "10分钟后关座椅加热"
走法：用户说话→⑨仲裁判"橙路"→⑩解析{条件:10min后, 动作:关座椅加热}→⑬注册→⑮入库→15-9分类=端侧（定时器必须端侧跑）→L14下发→㉔接收→24-5启动倒计时→10分钟后到期→24-12'白名单直执关座椅加热→24-11结果回写④→同步⑫→⑮确认→Director回复"已关闭座椅加热"

关键：定时器在端侧。进隧道断网了，10分钟到了照样执行。这就是24-5不可替代的原因。
Case 2：双条件AND —— "倒档+检测到充电桩→开充电口盖"
走法：产品预设场景{条件:档位==R AND 充电桩==有}→用户挂R档→⑮检测到档位==R→先查便宜条件（读信号库）→满足→再启动VLM去看充电桩（贵操作）→VLM看到充电桩→回写信号库→⑮判定两个条件都满足→触发→Director开充电口盖

关键：先查便宜的再查贵的。不满足便宜条件就不浪费VLM算力。
Case 3：AlwaysOn + 静态Advisor —— "有人打电话→调小音量"
走法：VLM每5秒输出一次→动态str含"握持手机"→写入④→同步⑫→⑮穿越检测（上一帧不含"手机"→当前帧含→F→T!）→15-6防抖（连续3帧15秒确认）→15-11 push给⑯舒适Advisor→Advisor建议调小音量→Director执行

关键："看手机"目前不是enum标签，是str。穿越靠关键词匹配，SFT后才能用==号判断。
Case 4：导航POI —— "快到外婆家时提醒我买水果"
走法：用户说话→Director解析→出卡片让用户选"外婆家"具体哪个→拿到坐标→L33注册到⑮→⑮注册围栏（500m半径）→车进入围栏→信号变化approaching→⑮穿越→Director说"记得买水果哦~"

关键：6.25决定优先端侧围栏（合规+实时性）。用围栏状态（in/out），不用距离计算。
Case 5：动态SP —— "宝宝毯子掉了告诉我"
参见第十一章《SP生命周期8步》——完整走了一遍SP拼接→VLM推理→穿越→push→清理→销毁的全流程。
Case 6：即时VQA —— "前面有没有加油站"
走法：Director直接调VQA工具→VQA内部路由：加油站=LBS（地图数据）→查地图→返回"前方500米有中石化"→Director回复用户

关键：这个case不经过触发器！即时查询走Director绿色链路，不是条件触发。触发器只处理"条件触发"（持续监听+穿越），不处理"即时查询"。
