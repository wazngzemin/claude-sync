Advisor 体系深度讲义：4个静态 + 动态advisor + 感知订阅，掰开揉碎讲透

这是之前讲义里最薄弱的部分——上篇只用了"四个部门经理"的类比一笔带过。实际上4个静态advisor各有独立SP、完全不同的推理逻辑和输出格式；动态advisor有9套prompt模板和完整的生命周期状态机；感知订阅有VLM阈值穿越检测和触发优先级。下面每个都掰开讲。

AAdvisor 体系总览：它们在系统里是什么角色

Director 是 CEO（快决策、对外说话），advisor 是部门经理（各管一条线、深度思考、盯进度）。但这不是4个"一样的"经理——它们各有完全不同的推理逻辑、触发方式和输出格式。

B舒适 Advisor——车内的"体感管家"

关注6+1个舒适维度：体感温度、空气质量、光线、声音、乘坐、空间、其他。它不直接控车，只给 Director 输出"分析+建议"。

它怎么"想"：感知异常 → 诊断维度 → 生成方案

舒适advisor的SP内置了5条knowhow推理规则（决策树式的if-then）：

温度舒适："当车内温度超出人体适宜温度，思考如何快速使车内温度达到适宜；当车内人员表现出温度不适感（打颤、反复擦汗），思考如何调整局部车辆控制使对应人员舒适。"
空气舒适：空气质量超标→快速恢复。
光线舒适：有人阅读/找东西/化妆/睡觉→判断光线是否够。
声音舒适：有人睡觉/听歌→判断音量是否合适。
空间舒适：车内人员变动→空间是否满足所有人。

触发有两种：① 事件触发（有人上车、温度变化、VLM看到有人在阅读）；② 1分钟内无新事件+有未完成的舒适子目标→自动触发一次定时review。

输入是5大类模板变量：{{attention}}事件、{{watcher}}VLM描述、{{car_condition}}端信号、{{web_information}}世界信息、{{memory}}记忆。

输出是两字段JSON：

{"当前的舒适性问题":"舱内温度已达24度，但空调还在16度且风速较大",
 "应对建议":"建议把空调温度调到24度，风速调到2档"}
真实case——夏季38度上车(P0)：VLM描述"乘客额头有汗、在擦脸"→端信号"车内温度38度、空调关闭"→舒适advisor判定"温度不舒适"→输出建议"开空调制冷20度+座椅通风3档+车窗先通风2分钟再关"→Director 接收并执行。然后进入持续追踪：1分钟后定时review→"车内降到26度了但风速还在7档，建议降到3档"→Director 再调一次。这就是"持续照看"。
真实case——观察到后排擤鼻涕(TBD)：VLM描述"后排乘客在擤鼻涕"→舒适advisor判定"可能鼻炎/感冒，大风吹脸不舒服"→建议"后排风量调小、出风口调离人吹"。注意这个case标TBD，因为需要VLM能力够细才行。
注意：舒适advisor不直接控车！它只是给Director提建议，最终执行不执行是Director的事。如果Director此时正在处理更高优先级的用户query，可能会延迟执行甚至丢弃这条建议（走交互管理的排队/丢弃策略）。

C出行 Advisor——不只是导航助手，而是"理解你去哪、为什么去"

覆盖全出行生命周期：行前(目的预测/路线推荐/缴停车费) → 行中(堵车绕行/POI推荐/充电/订座) → 行后(最后一公里指引)。

核心推理链：出行目的预测（这是出行advisor最独特的能力）

用户上了车，还没说去哪。出行advisor就要猜出来：

PRD给的推理示例（原文）：
"根据当前信息（12月3日周二早上8:31），益红和依依已上车，车辆位于长江小区。结合两人的日常安排：
- 依依应在8:30-8:35出发去光启小学上学，当前时间已接近最晚出发时间，且她带了书包，说明今天需要上学。
- 益红通常8:45-9:00出发去平金中心上班，但两人已上车，很可能计划先送依依上学，再去公司。
因此，本次出行的目的是：1.送依依上学 2.益红上班"

它怎么猜的？靠10+个数据源：历史出行偏好（什么时间去什么地方）、车内人员状态（谁坐哪干什么）、上车携带物品（带了书包→上学）、时间上下文（周二早上→通勤）、当前位置、常去地点、导航状态、落座后停留时长、日程/日历、交通通知、高德出行预测……

推理模式是：多因素融合 → 概率排序 → 输出预测+置信度 → 低置信时主动确认。置信度低就问用户"你是要去公司还是去学校？"，高就直接推荐路线。

停车缴费case：事件"车在停车场&起步/驶离"→出行advisor推理"已停3小时，有缴费需求"→输出建议"主动询问是否帮交停车费，同意后通过GUIAgent缴费"→Director接收并执行→GUIAgent操作捷停车小程序。
常走路堵车case：事件"堵车"→出行advisor推理"平时走中环内圈，今天堵，外环高速顺但收费10元"→输出建议"推荐走外环高速，路顺但付10元"→Director告知用户选择。
出行advisor PRD是v0.4，相比舒适v1.0和情感v1.0，成熟度偏低。很多功能（生态服务-订座/取号、个性化POI推荐、特殊充电情况）标P2未实现。输出格式和与Planner的对接"需要和planner详细沟通"，还未完全定义。

D内容 Advisor——"首席内容建议官"，克制比推荐更重要

核心设计哲学不是"推更多内容"，而是"判断该不该推"。默认不工作，没有Trigger就没有内容。成功指标是建议采纳率，不是播放量。

最核心的机制：L0-L3 介入等级（这是一个状态机）

内容advisor被触发后，第一件事不是"推什么"，而是"该不该推"。它会根据当前场景输出唯一的介入等级：

等级定义认知占用什么时候
L0 隐形不推荐不打扰0%愤怒/烦躁/高压力；暴雨冰雪；社交冲突；明确拒绝
L1 氛围白噪声/轻音乐<5%疲惫/低落/发呆；单调安全环境
L2 碎片笑话/脱口秀/新闻30%通勤/短途(导航<30min)；正常情绪
L3 沉浸专家解读/历史讲坛>80%长途巡航/驻车休息(NOA/充电)；好奇/高接收度

PRD原文的设计哲学："要像人一样察言观色——车上夫妻正在吵架，即使满足了Trigger条件，也不要去推送。"

8步推理Flow：静默态 → Trigger唤醒 → Context汇聚 → 介入判断(决策核心) → 策略参数生成 → 分发给Planner → 反馈感知 → Review回收。其中第4步"介入判断"是核心——只有判定L1以上才往后走；判定L0就直接回到静默，一个字都不说。
下班雨天case：触发→Context汇聚(下雨、下班、一人开车、情绪低落)→介入判断=L1氛围(不适合推深度内容)→输出"推一首钢琴版雨天轻音乐"→Director播放。注意：不会推一首欢快的歌来"对抗"低落情绪——PRD说的是"顺势而为的治愈内容"，不是"强行振奋"。
内容advisor有一个独特的触发源——"协同信号"：情感advisor判断"用户现在适合陪伴"时，会发一个信号给内容advisor，触发它推荐。这是4个advisor之间唯一明确的"跨advisor协同"接口。
反馈闭环很精细：连续跳过3次→归因分析(是不喜欢这个歌手还是不喜欢曲风？)→策略修正。路况从畅通变拥堵且持续>5min→自动降低认知门槛(从L3降到L2)。每20分钟后台扫描一次。

E情感 Advisor——4个advisor里推理最精细、红线最严格的

纯粹做情感层面的察言观色。不控车、不推内容、不提醒物品——只管"用什么语气说话、要不要主动搭话、该不该关心一下"。有两套完整SP（Event prompt + Review prompt），含大量正/反示例。

核心机制1：Event Prompt 的4步推理链

第一步：事件解析+情感影响评估——收到事件（如"副驾上车了"），判断有哪些情感影响。多情感优先级规则（这是决策树）：启动车辆方式 > 安全相关 > 节日/特殊日期 > 人物状态 > 情绪 > 人物 > 车内场景 > 车外场景 > 人物+时间 > 人物着装。安全相关强制执行安全驾驶知识中的建议。
第二步：即时建议生成——格式是"语气+表情+对象+情感方向+禁止"。6条红线：①只输出情感建议，不输出具体内容 ②不控车 ③不对未发生的事猜测 ④不对曾发生的事预测 ⑤不提醒物品 ⑥禁止"维持现状"类建议。
第三步：子目标生命周期管理（完整状态机）——两种类型：A目标达成型（"安抚主驾情绪直到不再愤怒"）和B条件响应型（"轻柔语气直到右后排睡醒"）。操作create/update/delete。Create需同时满足5个条件（有服务对象+情感可建议+非即时闭环+可判定+语义不重复）。子目标上限10个。
第四步：输出整合+冲突检查——即时建议 vs 子目标冲突→以即时建议为准；新增子目标 vs 已有重复→合并。

核心机制2：Review Prompt 的3步推理链

触发条件："10分钟没有关注用户了+有未完成子目标"。

三种情况：A进行中需调整（重度→轻度疲惫）→更新goal；B触发条件消失→恢复建议+删除goal；C目标已完成→仅删除，不输出任何建议（克制！完成了就安静离场）。
子目标冲突case——后排儿童睡着+主驾分心：按优先级规则"安全相关>人物状态"→即使后排有"轻柔语气"子目标，安全提醒优先→Director 用严肃语气提醒主驾，后排的轻柔子目标暂被安全覆盖。
被超车case：用户"卧槽"→事件触发→情感advisor判定"愤怒/受挫"→输出：语气=共情吐槽，方向=先认同再安抚→Director："那个开车也太横了吧。不过你没事就好，别因为他影响心情。"
情感advisor的红线是4个advisor里最严格的。它有大量"反面示例"写在SP里：
✘ "建议帮用户播放一首轻音乐缓解心情"（越界！不该推荐具体内容）
✘ "建议把空调调低一些，他可能出汗了"（越界！不该控车）
✘ "建议提醒用户别忘了带雨伞"（越界！不该提醒物品）
✘ "建议维持当前状态，不需要做任何事"（禁止！"维持现状"不算建议）
每条反例都标了"为什么错"和"正确做法"。
为什么情感advisor有两套SP（Event + Review），而其他advisor只有一套？因为情感管理的本质是"事件驱动的即时响应 + 时间驱动的持续关注"——两种模式的推理链完全不同。Event是"发生了什么→该怎么反应"，Review是"一直在盯的目标→进展如何→该不该调整"。分成两套SP让各自的推理链更清晰、更不容易混淆。

F动态 Advisor——9套Prompt + 4类任务 + 完整生命周期状态机

之前讲义里"一目标一advisor，基于prompt模板工程自动注册"只是概述。实际上它有9套不同的prompt、4类不同的创建流程、复杂的exec_state状态机，还有二期的Skill系统和Workflow编排。

创建流程：query 进来后经历了什么

Step 1: Director → CreateGoal——Director判断需要创建持续任务，把GoalText发给动态advisor的CreateGoal接口。
Step 2: 分类模块（LLM推理）——用专门的"任务分类prompt"把query分成4类：
优先级判定(原文)：① 是否调用点单/缴费/预订 → type=2(持续/长程) ② 包含明确时间/周期 → type=4(定时) ③ 包含条件表达 → type=3(条件) ④ 其他 → type=1(长时)
输出：{"type":"1~4", "vlm_use":true/false, "cycle":true/false, "max_execution":-1/1/N}
Step 3: 按类型走不同流水线——
· 长时(type=1)→直接Y/N判定→创建成功→加载Skill(二期)
· 持续/GUI(type=2)→绑定GUI Agent目标
· 条件(type=3)→解析条件→注册触发器(5-8s)→成功or失败
· 定时(type=4)→生成APScheduler参数→注册定时器→是否需VLM→是→生成VLM提示词
Step 4: Goal生命周期开始——创建成功后任务中心显示Widget卡片。条件任务创建需5-8s，期间显示"任务生成中..."。

输出JSON结构（长时任务，最完整的一种）

{
  "goal": "照顾二排宝宝",
  "goal_status": "active | achieved | terminated_by_user | blocked_by_system",
  "goal_progress": "宝宝安睡中，持续监测",
  "need_advice": true,
  "advice_timing_judgement": "距上次建议已120秒且状态变化",
  "advice_timing_confidence": 0.85,
  "goal_achieve_advise": "宝宝醒了在揉眼睛，建议放一首安抚音乐",
  "completion_rate": 0.3,
  "exec_state": "normal | suppressed | recoverable | short_circuit"
}

exec_state状态机（之前完全没讲过）：
· normal：正常，可输出建议
· suppressed：推进抑制，强制need_advice=false（如用户刚被打断不适合再推）
· suppressed_attention_shift：用户注意力偏移导致抑制
· recoverable：可解除抑制，等恢复条件
· short_circuit：上轮建议已执行且有效，跳过本轮

销毁规则

· 条件/定时任务：max_execution达上限→注销触发器/定时器→CloseGoal
· 用户主动删除（任务中心点"取消"）
· goal_status进入terminated_by_user或blocked_by_system
· 触发器注册失败时：CloseGoal, reason=创建失败

二期(DNA-616)关键变化

Skill系统+Workflow编排：长时任务从"特调"变"通用"。创建时load_skill()匹配Skill(如"比分播报Skill""兜风Skill")→加载Knowhow→组成实例。支持顺序执行/条件分支/循环。
端侧定时器：弱网/断网时也能触发(AI PUSH不依赖网络)；端侧倒计时展示(不再显示"30分钟后"不变)。
策略预生成：条件满足时不再现场推理，提前把执行策略算好放云端(后续考虑放端)。"可以在事件发生时减少一次推理所需的时间"。
导航场景感知：新增"距离某POI<指定距离"触发事件（如"快到外婆家时打电话"）。
触发优先级（之前没讲过）：订阅事件 > qa结束 > 定时。高优先级的触发会打断低优的，反之不成立。比如条件触发器callback进来时，如果定时触发正在处理，定时会被打断。

G感知订阅技术细节——VLM 阈值穿越检测到底怎么工作

之前讲的是"拆成三维度配置下游"。实际技术实现比这复杂——VLM有阈值穿越检测(last==False and this==True才触发)、有event vs state两种判定模式、有定时器4种类型。

VLM视觉感知：阈值穿越检测（这是触发的真实机制）

VLM不是"看到宝宝就报告"，而是"看到状态变化才报告"。具体流程：

1. 定时器触发（演示阶段5s/次，量产30-60s）
2. 调图片接口拍照→获取图片URL
3. 请求VLM→把图片+query发给云VLM，VLM只输出一行：true|reason 或 false|reason
4. 阈值穿越检测：对比上一次VLM输出(last)和这一次(this)。只有 last==False and this==True 才算事件触发！也就是说，状态从"未发生"变成"发生"的那个瞬间才触发，持续为True不会反复触发。
5. 触发后→请求advisor模型给出advice
"帮我照顾二排宝宝"的VLM query生成：{"type":"state","query":"二排存在宝宝（画面中出现婴儿或儿童）"}
· 第1轮VLM返回：false|后排宝宝没有睡着 → 不触发
· 第2轮VLM返回：true|后排宝宝睡着了 → last==False, this==True → 触发！
· 第3轮VLM返回：true|后排宝宝还在睡 → last==True, this==True → 不触发（状态没变）
· 第4轮VLM返回：false|宝宝醒了在哭 → last==True, this==False → 不触发（穿越方向反了）
但是下一轮如果又变成True就会再次触发——只检测从False到True的上升沿。

event vs state 两种VLM判定模式

类型定义举例
event(可判定事件)有明确触发动作，瞬时视觉特征，一帧可判"用户打哈欠了"→检测单帧哈欠动作
state(持续状态监控)无明确触发瞬间，长期持续关注"二排存在宝宝"→持续监控婴儿是否在座

视觉感知advisor的3类策略（输出差异大）

类型策略举例
风险监控有异常才说，只说一次，同一问题不重复提醒照顾宝宝（宝宝哭了才报，哭一次报一次）
信息讲解必须有新内容，间隔≥2-3轮(60-90s)介绍沿途风景（每次讲新的景点）
状态陪伴必须"明显状态信号"才触发，单次触发后长冷却≥5min缓解疲劳（打哈欠了才提醒，提醒后5分钟内不再说）
