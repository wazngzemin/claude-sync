第 0 拍 · 开场：给全场一个坐标系⏱ 2 分钟

各位，今天评一件事：VLM 条件任务的"判断"环节。整场我就讲这一张图，大家眼睛跟着我手走。

先建坐标系。这张图左边端侧、右边云端，中间这条虚线是端云分界。图上主角就五个，我先点一遍：

右上这个，Director（对话推理引擎）——管跟用户说话的；
右边中间，Task Service（任务服务）——管任务的生老病死，图上写着它创建四类任务：长时、持续、条件、定时；
它旁边红字这个，云端触发器——管判断的，今天的主角，先说清楚一句：它是现有规则服务的能力扩展，不是新建模块；
左边这个大框 AIBox 里的端侧感知模块 VLM——管看的；
中间这两个圆柱，AI Service 的消息处理和数据中心——管传的，消息处理管下行，数据中心管上行。

一句话记住这张图：Task Service 管任务，VLM 管看，触发器管判，数据中心管传，Director 管说。

图上还有三块是背景，今天不评，我先圈掉：左边虚线框 SLS 闭环业务，赛力斯自己的；中间黄色信号源面板，车速档位围栏这些，触发器既有输入；右边静态 Advisor（静态顾问），归弋途。一会儿走线经过它们时我各用一句话带过，不展开。

好，坐标系有了。现在拿用户真实的一句话，把这张图从左走到右。
说一个主角点一个：Director（右上）→ Task Service（黄框）→ 云端触发器（红字框）→ AIBox·端侧感知模块VLM（左）→ 两个圆柱。说"背景"时在 SLS 虚线框、信号源黄框、静态Advisor 框上各画一个圈。

第 1 拍 · 左上角起点：一句话进来⏱ 1 分钟

看图最左上角这个框——用户说："后排宝宝哭了告诉我。"

这句话沿着图顶上这条长线，直接进到右上角的 Director。Director 做它一直做的事：前置筛选、Context（上下文）注入，然后识别出——这不是"打开车窗"那种一句话就办完的事，这是一个要持续盯着的任务。于是往下，交给 Task Service。
从左上角 query 框，沿顶上那条横线一路划到右上 Director 框，再从 Director 往下划到 Task Service。

第 2 拍 · Task Service：把一句话劈成两半⏱ 2 分钟

Task Service 接到这句话，干的第一件事：劈开。大家看图上这两行红字——

user_goal（用户目标）："后排宝宝哭了告诉我"——原话，存档；
execute_text（执行文本）："告知用户：后排宝宝哭了"——这是后半句"告诉我"变的，动作，触发之后才用它。

那前半句"宝宝哭了"呢？——它是条件，被转写成一个疑问句："二排的宝宝哭了吗？"我们管它叫感知目标。

为什么非要劈？因为用户的话是"条件+动作"混在一起的。不劈开，碰上"宝宝哭了先别吵她、睡熟了再告诉我"这种话，判断环节直接就乱了。劈开之后：条件去判断，动作等触发，各走各的。这个转写有一套规范——劈条件动作、疑问句化、位置标准化、时长剥离——由 Task 侧的 Spec Builder（规格构建器）统一做，是今天待确认第九条。
点图上 "注册动态任务" 附近那两行红字 user_goal / execute_text，念的就是图上的原文。

第 3 拍 · 兵分两路：注册规则 + 下发观察⏱ 2 分钟

劈完，Task Service 同时发出去两个东西，靠任务编号串着。

第一路，往云端触发器，注册一条规则——看图上红字"动态任务注册"。注册单就是咱们配置平台"新建场景"那三件套：conditions（触发条件）放感知目标"二排的宝宝哭了吗"；actions（动作）是回调；execute_policy（执行策略）是命中一次即停、下电注销。等于程序替用户自动填了一张咱们天天手工配的场景单。触发器这边还有图上写的"全量/增量规则下发"能力，管规则的同步。

第二路，跨过中线往端侧，下发观察任务——就是图上这条红色长线，"动态感知模块生成高频任务UP并下发"。下发的内容看这个代码块：dynamic_watch_tasks，任务编号 8，感知目标"二排的宝宝哭了吗？"。

记住这个"8"，一会儿它会带着答案回来。
先点云端触发器框和"动态任务注册"红字；再沿那条红色长线从右往左划到端侧"消息处理"圆柱；最后点 dynamic_watch_tasks 代码块，念出"任务编号 8"。

第 4 拍 · 端侧：AIBox 排队、VLM 开工⏱ 2 分钟

到端侧。消息处理模块把观察任务透传给 AIBox——透传的意思是不解析不改写，原样递进去。

AIBox 里先过任务优先级管理——图上写着：OneShot（单次任务）P0 抢占，AlwaysOn（常开任务）P3，高优完成后自动恢复低优。为什么要排队？因为左边虚线框里，赛力斯的 SLS 应用也会触发任务——oneshot 或者周期推理——这是 SLS 自己的闭环，结果回 SLS，跟我们互不干扰，唯一的交点就是在这儿排个队。这块今天不评。

排完队，端侧感知模块 VLM 挂上任务编号 8——图上写着它的职责：封装 VLM 引擎、管理推理执行、数据发布到 topic（订阅主题）。左下角视频流处理把画面送进来，VLM 开始每 5 秒看一次二排。

补一句前提：图上左下这个"SLS 舱内 AlwaysOn 开关"，用户在赛力斯设置页的总开关——开着，舱内观察才跑。这是 7.8 评审定过的，不重复评。
依次点：消息处理圆柱 → AIBox 框里"任务优先级管理"那三行字（逐行念）→ SLS 虚线框（一句话带过）→ "端侧感知模块VLM"框 → 视频流处理框 → SLS舱内AlwaysOn开关框。

第 5 拍 · 上报：一包数据、两个框、两条路⏱ 3 分钟

VLM 每 5 秒吐一包数据。大家看中间这张 JSON（数据结构）截图——一包数据里有两个框，走两条完全不同的路。这是这张图上最容易看混的地方，我拆开讲。

绿框，乘客列表——谁坐哪、什么状态。图上的真实样例："位置：副驾后方，动态：头依靠在中间乘客肩膀上睡觉"。这一框走 Context 路线：数据中心整包缓存、按端状态协议上传，到云端 Context，二次组织之后给 Director 当对话背景。这条是 7.8 评过的老链路，今天不动它。

红框，高频任务UP结果——看样例：任务编号 8，识别结果"后排儿童睁眼，脸上有泪珠"。刚才下发的那个 8，带着答案回来了。这一框走动态任务路线：图上这条紫色长线，"透传高频任务UP结果"，不走二次组织，直达判断环节。

注意红框里 VLM 说的是什么——"睁眼，脸上有泪珠"。它没说"哭了"。VLM 是通用感知，它不知道今天用户问的是哭没哭、明天问的是睡没睡，它没法预置结论，只能如实描述。把"有泪珠"判成"哭了"，是下一拍的事——也就是今天评审的核心。
先点 JSON 截图的绿框部分念"头依靠在肩膀上睡觉"，手沿绿线划到云端 Context 圆柱；再点红框念"任务编号8、脸上有泪珠"，手沿紫色长线从数据中心划到判断区。"它没说哭了"这句，指着红框停两秒。
有人问："为什么不让 VLM 直接输出'哭了=是/否'？"
VLM 是通用感知，不预知每个用户会提什么任务——今天问哭没哭，明天问毯子掉没掉，它没法为每个问题预置结论。所以定的分工是：VLM 出证据，判断环节出结论——判断必须发生在知道任务目标的地方，目标注册在触发器，所以判断在触发器。我马上讲这块。

第 6 拍 · 判断区放大讲：两层判断 + A/B 两种判法⏱ 6 分钟 · 全场核心，讲慢

现在到今天的核心，图上这块紫色标注区，我放大讲。

先看判断的输入，图上写着：识别结果"后排儿童睁眼，脸上有泪珠"，加上感知目标"二排的宝宝哭了吗？"。一个是证据——每 5 秒流进来一条；一个是题目——注册那天就存在触发器规则库里。判断就是拿流动的证据，对静止的题目。

这里要先把一个词说清楚——"判断"其实有两层，图上我拆成了两个框：

第一层，条件命中：这条"有泪珠"符不符合"哭了"？每 5 秒判一次。这层在云端触发器。依据是 Task 技术方案的原文——规则服务管"延时、条件、周期和 VLM 语义匹配"；"Trigger 完成语义匹配并判定命中后回调到 Task，Task 不负责二次条件匹配"。

第二层，任务达成：命中之后，这个任务是"报一次就关"还是"持续盯着"？这层在 Task Service，按 EndPolicy（结束策略）走——这就是 Task Service 职责表里那条"判断是否达成用户任务目标"的真正含义，它管任务生死，不管条件真假。两层各归各，互不越界。

那第一层具体怎么判？两种判法，看图上这行字：标签=比对枚举值，语义=轻量模型判断。我拿同一句"宝宝哭了"给大家看两种数据——

A，标签比对。前提是"哭闹"进了预置标签，VLM 直接给选项：第一条"哭闹状态：无"，第二条"无"，第三十七条"哭闹状态：哭闹"。触发器干的事就一个字符串比对：值等于"哭闹"吗？——跟咱们配置平台"档位 等于 P"一模一样，零成本零误差。结论 VLM 已经下了，触发器对答案。

B，语义匹配。用户说"毯子滑落了告诉我"——这话没有预置标签，也永远预置不完。VLM 只能报文字："主驾后排宝宝的毯子已滑落"。这段文字跟目标字面不相等，现有十个运算符一个都判不了——所以加一个轻量模型，每条读一遍："这段描述符合'毯子滑落'吗？"答"是"就命中。VLM 交作文，触发器阅卷。

为什么两条都要？全走 A，长尾接不住——用户想说什么预置不到；全走 B，高频血亏——明明能枚举比对，每 5 秒烧模型钱还带误判率。所以分层：高频走 A 又快又准又免费，长尾走 B 什么都接得住。哪个场景走哪条，三问定：能不能收敛成有限选项、高不高频、VLM 描述撑不撑得住——附件表格 31 个场景每条的路线都按这三问标好了。

最后看图上这行："没达成，等下一条 5 秒上报，再判；达成，通知 Task Service。"补一个工程细节：连续十帧都是"哭了"，不会回调十次——判断结果当成布尔状态，引擎抓"没哭→哭"这个变化边沿才触发，跟现有"变更为"运算符同一个机制，配 one_shot（单次）策略。去重、边沿、执行策略全是引擎现成能力——这也是判断放触发器的第三个理由：放别处，这套全要重造。
这一拍全程停在紫色标注区：先逐字念"输入=识别结果+感知目标"那行；讲两层时点拆开的两个判断框（改图后）；讲 A/B 时如果图上有两组数据样例就分别点，没有就口头说；最后念图上"没达成等下一条5秒上报"原文。语速放到全场最慢，这 6 分钟值一半的会。
庆磊："触发器是规则引擎，挂个模型，架构上怎么挂？"
做成引擎的第三种运算类型。现有引擎两种运算：条件八个运算符比大小，事件两个运算符比变更——现在加一种"比含义"。对引擎来说就是个新运算符，注册协议不动、复用现有三件套。模型本身谁提供，待确认第六条，一会儿跟晓朝定。
刘杨："5 秒一条、多任务并发，引擎扛得住吗？"
量级可控：A 路线是字符串比对，量再大也没压力；真正过模型的只有 B 路线激活中的任务，单车同时挂的 B 任务是个位数。具体压测口径，你我下来对，今天先定机制。

第 7 拍 · 右侧收尾：回调 → 播报 → 拆台⏱ 2 分钟

判断命中，往右走完最后一段。

触发器回调 Task Service——带着 task_id（任务ID）、callback_id（回调ID）、命中时的原始识别结果、触发时刻。Task Service 收到，不再判一遍条件，它做第二层判断：按结束策略，这个任务是一次性的——报完就关。

还有个细节：用户如果说的是"每 60 秒告诉我一次"，那 VLM 该 5 秒还 5 秒地报、判断该 5 秒还 5 秒地判，但反馈按 60 秒来——判断的频率和打扰用户的频率是两个周期，解耦的。

然后走 Director：图上写的，Director 推理，执行动作拆解——talk_or_not（说不说）、talk_content（说什么，带情绪和 emoji 表情）、action_list（动作列表）——播报："后排宝宝哭了。"

最后拆台：任务关闭，触发器的规则注销，VLM 那边收到"结束任务编号 8"，停止观察。一句话从进到出，全链路闭环。整张图走完了。
从判断区沿线划到 Task Service，再划到 Director 框（点 talk_or_not/talk_content/action_list 那几行字），最后手从右往左大幅度划回 VLM——"拆台"这个动作配合手势，全场跟着你把图收拢。

第 8 拍 · 图上剩下的三块背景，一分钟带过⏱ 1 分钟

图上还剩三块没讲，各一句话。

中间黄色信号源面板——车辆端状态、手机外设 IOT（物联设备）、电子围栏、用户 query，这是触发器既有的非视觉信号输入，今天这条链路不动它，画出来是让大家看到触发器的完整输入版图。

静态 Advisor——舒适、出行、情感、内容四个，归弋途在意图侧做，图上画的是云端触发器跟它的触发关系，本次不评。

舱内外 VQA（视觉问答）——同步问答链路，跟今天的持续观察是两回事，另行评审。

到这，图讲完了。进入收结论。
三块各点一下，节奏快，每块一句话——这一拍的作用就是让全场知道"你看到了、你没漏、但今天不聊"，堵住跑题的口子。

第 9 拍 · 成本：主动报（韩杰段）⏱ 2 分钟

收结论之前，成本我主动交代，先把最吓人的数自己说了：单任务 5 秒一判，一天理论上限一万七千次。但真实量级三个折减——

一，分层就是成本方案：高频走 A 标签，字符串比对，零模型调用；过模型的只有 B 路线长尾。
二，只在激活期跑："宝宝哭了"这种任务平均挂几十分钟，不是全天；命中即停。
三，模型很轻：单次输入两句话、百级 token（词元），跟 Planner（规划器）单次三万 token 差两个数量级，非常驻。

真实测算——激活任务数乘激活时长乘 B 路线占比——我这周书面给韩杰。今天先认口径。
韩杰："一万辆车呢？"
口径一样：有激活任务的车 × 平均激活时长 × B 路线占比——不是一万辆车全天都在跑判断。激活率是关键系数，我拿真实任务分布算完书面给你，这周内。

第 11 拍 · 收尾复述⏱ 1.5 分钟

我复述今天的结论。

一，这张图上，Task Service 管任务、VLM 管看、触发器管判、Director 管说——判断两层，条件命中在触发器、任务达成在 Task Service。
二，判法分层：高频标签比对、长尾语义匹配。
三，口径：三态输出、宁漏判不误判、边沿去重。
四，作业：丁彬黄色列两张表 X 日前；种子数据口径我 X 日出；刘杨引擎改动量 X 日对；成本测算我 X 日书面给韩杰。
