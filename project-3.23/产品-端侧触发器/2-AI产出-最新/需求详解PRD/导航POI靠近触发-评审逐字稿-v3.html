② 背景：这事到底要干嘛

一句话：让触发器能做"快到某个地方就提醒你"这件事。

触发器目前不认"目的地""剩余距离"这些概念。用户说"快到外婆家提醒我打电话"，系统注册不了、监测不了、触发不了——一个字都做不了。

为什么做不了？因为触发器信号表里，导航状态、目的地位置、剩余距离这些信号之前被删了/没开放。信号编号 119 到 133 在老表里有定义，但端上没给数据、引擎没接、配置平台也配不了。

本期就是把它补回来。不是从零造，是把缺的信号重新开放、把缺的判定逻辑补上。

目标：补充触发器导航状态、目的地/途经点位置（地址 + 剩余距离）、收藏地址等信号。

这里面最核心的就是"接近目的地"——它不是简单的"距离小于 500 就触发"，需要 one_shot 过程判定。等一下我会展开讲为什么。

④ 典型场景 + 两类 Case 的区别

先给大家三个真实场景，有个画面。

场景一：早上导航去公司，距公司 500 米时车主动说"快到啦，要不要提前点杯咖啡"。
场景二：导航回外婆家，快到时主动帮你搜附近充电桩。
场景三：路过徐浦大桥，提醒你"准备右转下匝道"。

这三个场景其实分两类，性质不一样，链路走法也不一样。

A 类：目的地/途经点距离

POI 就是当前导航的目的地或途经点。
导航已经在实时算到它的剩余距离。
端状态信号已支持，只缺 one_shot 状态机。

例："离目的地还有 1km 提醒我拿咖啡"
例："快到途经点帮我找停车位"
B 类：靠近任意 POI

POI 不是当前导航目标（"外婆家"可能不在路线上）。
导航没有在算到它的距离。
需要先把它动态添加为途经点，地图才能给距离。

例："快到外婆家提醒我打电话"
例："路过徐浦大桥提醒我右转"

A 类相对简单——导航已经在算距离，我们加个状态机就行。

B 类难很多——"外婆家"不在导航路线上，系统不知道到它多远。要先把"外婆家"从用户记忆/收藏地址里解析出一个真实地址，再动态插入导航的途经点，地图才开始算距离。地图能不能动态插途经点，是 B 类做不做得了的生死问题。

这个区别很重要，等一下走流程图的时候大家会再看到。先有这个数。

⑤ 核心难点：为什么"距离 < 500m"行不通

有人可能想——配一条规则"剩余距离小于 500 就触发"不就行了？不行。我用数字走一遍。

周一早 8 点，用户导航去公司，出发时 15 公里。产品想做"距公司 500 米提醒拿咖啡"。
时刻剩余距离如果配"<500m 就触发"
8:00 - 8:2515km → 501m一路递减，都 >500m，不触发。没问题。
8:25:30500m第一次 ≤500m，触发！提醒拿咖啡。到这里没问题。
8:25:31499m还满足 <500m → 又触发
8:25:32498m又满足 → 又触发
8:25:33 …497m … 50m每帧都满足 → 一路触发几十上百次，"该拿咖啡了"刷屏

问题就出在这里：从 500 米一路开到停车场，距离 499、498、一直到 50，每次信号刷新都满足"小于 500"。触发器没有记忆，不知道刚才已经提醒过了，就一直响。

我要的是：第一次跨过 500 米提醒一次，然后闭嘴。像门铃，不是警报器。这就是 one_shot。

现有协议做不了这事，它缺三样东西：

  1. 记不住上一次的状态——不知道已经提醒过了。
  2. 判不了方向——不知道你是在靠近还是远离。
  3. 标不了"已触发"——触发了不能自动闭嘴。

所以必须加一套 POI 状态机。状态机怎么工作，我在流程图里展开讲。

④ 端侧信号库 → 端侧 POI 状态机（B 方案核心）

这是整条链路最关键的地方，展开讲。

刚才说了为什么"距离小于 500"行不通——会刷屏。解法就是这个状态机。给每个被监控的 POI 记一个状态，四种：远 → 已接近 → 已到达 → 已离开，单向翻，翻过去就不回来。

用数字走一遍：

距离刷新状态机里发生了什么
d = 15000m, 10000m, …, 501mstate=远，d 都大于 480m，条件不满足，不动。
d 从 501 变到 479mstate=远 且 d≤480 → 条件满足！触发 approach 事件。
state 立刻翻成 已接近。
d = 478m, 477m, …, 50mstate=已接近（不是"远"了），接近判定进不去，不会再触发。这就是 one_shot。
d ≤ 50m 且 档位 → P触发 arrive 事件，state 翻成 已到达。
d > 520m（开走了）触发 leave 事件，state 翻成 已离开。
为什么判定线是 480 不是 500？——防抖。

用户在距外婆家 500 米路口等红灯，GPS 在 498 和 502 之间来回抖。阈值卡在 500，每抖进来就触发一次。

所以设了死区：小于 480 才算接近，大于 520 才算离开，480~520 之间不动作。产品配 500，实际判定用死区值。
为什么用沿路线距离不用直线距离？

你家在高架旁，直线 300 米，但导航要绕到下一个出口再绕回来，沿路线 2 公里。直线距离算的话 300 米就触发"接近"了——可你实际还要开 2 公里。误触发。一律用导航给的沿路线剩余距离。

@金龙 @汪帅 ——这套状态机逻辑跑在端侧，由你们实现。四个状态、三条判定规则、死区防抖、状态重置（导航取消/离车时清空），这就是 B 方案你们要做的核心。

云端触发器拿到 approach 事件，匹配之前注册的条件——event_type eq "approach" AND poi_id eq "外婆家"。命中了。

最终组装出 advice（建议），推给 Director。

⑧ B 方案的难点和我的顾虑

链路走完了。B 方案的核心就是状态机放在端侧跑。630 来得及、引擎改动小，但有几个问题我得摆出来。

难点一：多端各写一遍

上汽一套端、赛力斯一套端。状态机跑在端侧，每个端都得实现一遍。逻辑一样，代码各写各的。后面产品要改死区从 480/520 到 450/550，我得找每个端的人说一遍"你改一下"，漏了就出 bug。

我的态度：本期先这么做，能跑起来。长期端多了维护成本上来了，再往引擎迁。接口不变——端上报事件、引擎匹配——改的只是"事件谁来算"。

⑨ 和传统地理围栏的区别

可能有人问——这不就是地理围栏吗？手机上早就有，进入一个区域触发通知。不一样，关键区别在这里。

维度我们的方案（沿路线距离 + 状态机）传统地理围栏
距离怎么算 导航给的沿路线剩余距离。高架上绕出口再回来，导航告诉你"还有 2 公里"。 画一个圆，GPS 直线距离判断在不在圈里。
高架旁 沿路线 2km → 不触发。准。 直线 300m → 进圈了，触发了。但你还得绕 2 公里。误触发。
前提条件 必须在导航状态下。 不需要导航，GPS 在就行。
高德支持 导航剩余距离——支持。 围栏 API——宫丽萍已确认：高德不支持。

选沿路线距离两个原因：准（不会误触发）、高德有数据（围栏得自己搭）。代价是必须在导航状态下才能用。

⑩ 如果 B 方案做不了，走围栏需要什么

退路先说清楚。端侧状态机做不了或 630 来不及，唯一退路是自己实现地理围栏。工作量大家感受一下：

要做的事谁做有多重

围栏不是更简单，是更重。状态机省不掉、高架问题多一个、围栏引擎还得自己写。

所以我的建议：B 方案能做就做 B。围栏是最后的退路。

⑫ 应答策略

金龙/汪帅："端侧状态机工期太紧了，630 来不及"

"那我们拆——A 类 Case 的状态机不复杂，四个状态三条规则。能不能先做 A 类，B 类排后面？A 类做出来，'快到公司提醒拿咖啡'主场景就跑了。"

有人："为什么不做地理围栏？"

"两个原因。高德不支持围栏 API，得自己从头写引擎，工期更大。而且围栏用直线距离，高架旁 300 米就误触发。沿路线距离不会。"

有人："直接配'距离小于 500'不就行了？"

"不行。499、498 一直到 50，每帧都满足，触发器会连着响几十次。必须只响一次就闭嘴，这就是状态机要解决的。"

刘杨："引擎不改可以，但 event_type 匹配你给我定义清楚"

"PRD 事件定义表——event_type 值是 approach/arrive/leave/pass_by，字符串，eq 匹配。你看文档，有问题随时对。"

导航产品："动态插途经点做不了"

"那 B 类 Case 本期砍。A 类不受影响。B 类记技术债，看地图能力什么时候到位。"

施佳杰："Advisor 怎么区分接近公司和接近外婆家？"

"事件 payload 里带 poi_id 和 address。按 poi_id 分，不同 POI 触发不同 advice。"

有人："以后能迁到引擎统一做吗？"

"能。接口不变——端上报事件、引擎匹配。改的只是'事件谁来算'。下游不用动。"

有人："没在导航怎么办？"

"做不了。没导航就没有沿路线距离。这是能力边界，不是缺陷。无导航的触发得靠围栏或 VLM，另一个需求。"

⑬ 收口：散会要拿到的 5 件事

  2. 地图能不能动态插途经点 导航产品/宫丽萍 ——B 类做不做的生死线。
  3. 距离格式 + 收藏地址匹配方式 导航产品/金龙 ——整数米 or 字符串？经纬度 or 字符串比对？
  4. nav_event 信号编号注册 韩天亮 ——前置依赖。
  5. eq 运算符支持 event_type 任军 ——确认一下。
