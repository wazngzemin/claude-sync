## 摘要

Dan Shipper 是 Every 的创始人兼 CEO，团队从 15 人增长到近 30 人，全员深度使用 AI。本期访谈提出一个核心判断：AI 时代 PM 和全栈设计师会变强，但前提是"把模型带进真实工作流"而非"写更多文档"。围绕这个判断，Dan 给出了五个操作级洞察：用"reach test"判断工具价值、公司应先养一个"超级 Agent"再下沉到个人、Codex 正在变成知识工作的桌面、SaaS 不会死反而用户会增加、"自动化是一种谎言"每个 Agent 都需要一个在乎它的人。

## 关键要点

- **reach test**：早上醒来会不会自然伸手去用它——跨过这条线的工具才有真正价值
- **公司级超级 Agent 优先**：个人 Agent 维护成本太高，先让一个小团队把 Agent 养好，再服务全组织
- **Codex 成为知识工作桌面**：AI 从"SaaS 里的按钮"变成"进入 SaaS 的外壳"
- **SaaS 不死论**：Agent 带来更多"用户"，SaaS 需要同时服务人和 Agent
- **自动化是谎言**：每自动化一个流程就需要一个人盯着，Every 人数反而翻倍
- **PM 能力迁移**：从"写 PRD + 协调资源"→"判断该做什么 + 直接构建"
- **全栈设计师崛起**：设计师能直接做出可运行版本并提交 PR
- **ride the models**：新模型出来就拿自己的工作流试，AI 的边缘在"AI遇到正在做事的人"那里

## 逐段拆解

### 第一段：开场定位（"把模型带进真实工作流"）

**功能**：核心论点预告 + 身份背书

**深度分析**：Dan 的论点开门见山：PM 和设计师会赢，但不是因为传统技能，而是因为 AI 放大了他们原本就有但被低估的能力——产品判断力和审美直觉。"前提不在于写更多文档"是一个精准的否定——它暗示大量 PM 正在用 AI 做错误的事（生成更多文档、更长的 PRD），而不是用 AI 做正确的事（验证判断、快速构建）。这个开场隐含了一个分流：你是用 AI 做更多旧工作的 PM，还是用 AI 做新类型工作的 PM？Dan 自己的身份背书也很重要——他不是技术创始人，而是写作者出身的 CEO，这让他对"非工程人用 AI"有直接体感。

**隐含假设**：PM 和设计师天然具备可被 AI 放大的"核心判断力"。但实际上很多 PM 的核心工作就是写文档和开会——如果核心能力就是执行和协调，AI 放大的可能是一片空白。

**精彩**：把"PM 会赢"作为标题级结论极有传播力。**破绽**：条件"亲自上手 AI"暗示大多数 PM 其实不会赢——这个"前提"本身就是一个硬筛选器。

### 第二段：reach test + 住进未来（"你不该做预言家"）

**功能**：方法论展开——如何判断工具价值 + 如何做组织探索

**深度分析**：Dan 提出了两个互相嵌套的方法。**第一个是"reach test"**——判断单个工具是否有价值的微观标准：早上醒来会不会自然伸手去用？这比任何 benchmark 都更接近真实价值，因为它衡量的是"行为改变"而非"能力评分"。一个工具可以在评测中跑分第一，但如果没有改变你的行为习惯，它对你就没有真正价值。**第二个是"住进未来"**——组织层面的探索方法：不做会议室预测，而是让团队在新工具中生活。Every 把试用变成组织习惯，工程师、设计师、作者互相观察谁在用什么。Dan 特别强调了写作的角色：试用→写下观察→命名变化。没有被命名的变化不会被组织学习——Claude Code 从"小技巧"变成"非工程人的工作伙伴"，就是被反复使用+写作命名后才清晰的。

**隐含假设**：团队成员愿意且有时间持续尝试新工具。在 30 人小公司这可以做到；在字节这种大公司，合规审批、安全审查、工具管控让"自由探索"变得困难。

**精彩**：reach test 简单到任何人都能用，且几乎无法自欺欺人——你骗不了自己的肌肉记忆。

### 第三段：公司级超级 Agent（"它真的需要一个在乎它的人"）

**功能**：实践验证 + 预测修正——Agent 部署的正确顺序

**深度分析**：这是全文最有实操价值的段落之一。Dan 原本相信个人 Agent（每人一个 AI 助手），但 Every 用 OpenClaw 后发现**个人 Agent 维护成本太高**——普通人不想 SSH 到服务器排查问题。这迫使他修正预测：**先公司级，后个人**。"Agent 需要一个在乎它的人"是一个极其深刻的判断——它定义了一个新角色：不是用户、不是开发者，而是"Agent 饲养员"。这个人需要：懂业务（知道 Agent 应该做什么）、会维护上下文（持续补信息）、愿意看日志（发现坏掉的流程）。Dan 建议 Agent 主要留在 Slack，个人 Agent 处理私事，两者分开让人更安心。部署路径：公司顶部通用能力→团队→个人。

**隐含假设**：公司级 Agent 的 ROI 足够高，能支撑一个全职维护者。小公司 30 人可以算过来，但很多场景下"养一个人盯 Agent"的成本可能高于 Agent 带来的效率提升。

**与你工作的直接关联**：你就是 Planner 的"饲养员"。Badcase 排查、SP 优化、示例更新——这些不是附属工作，这是 Agent 产品经理的核心工作。Dan 的经验验证了一个你可能已经感知到的事实：Agent 不是"做完上线就行"的产品，它需要持续的人工养护。

### 第四段：Codex 成为知识工作桌面（"进入 SaaS 的外壳"）

**功能**：趋势判断——工作入口迁移

**深度分析**：Dan 描述了一个正在发生的入口迁移：AI 从"SaaS 里的按钮"变成"进入 SaaS 的外壳"。用户不再逐个打开 SaaS 操作，而是在 Codex/Claude Code 环境中统一处理所有工具。他的 inbox zero 例子很具体——Codex 配合邮件 Agent Cora，把邮件变成页面，对每封口述处理意见（查合同、起草回复）。变化的本质是**上下文统一**：Codex 能同时看到项目文件、浏览器页面、文档，把跨工具的信息连起来。以前一封律师邮件需要你手动打开合同系统查四年前的文件；现在 Codex 在同一个线程里就能做到。Dan 说自己仍会在 Claude 和 Codex 间切换，但日常主力已是 Codex。

**隐含假设**：Codex 的上下文窗口和权限模型能支撑"统一入口"的需求——当前还有很多限制（token 上限、权限边界、隐私问题）。

**与你工作的直接关联**：Planner 对车载场景的意义，和 Codex 对知识工作的意义完全同构——都是"上下文统一的操作入口"。用户不需要分别操作空调、导航、音乐，通过一句话让 Planner 调度 23 个工具。Planner 的价值不在于每个工具调用的质量，而在于**把跨工具的上下文连起来的能力**。

### 第五段：SaaS 不死论（"Agent 会增加用户数量"）

**功能**：反直觉判断——反驳"SaaS 末日论"

**深度分析**：Lenny 追问"所有工作在 Codex 里做，SaaS 会不会被吃掉"，Dan 的回答很反直觉：Agent 会带来更多"用户"而非消灭 SaaS。论证逻辑是：当用户带着 AI 进入 SaaS（用自己的 token 和权限），SaaS 的"用户"从人扩展到人+Agent。SaaS 不需要替用户付推理成本，但需要让人和 Agent 同时能操作产品。这意味着 SaaS 的竞争维度发生变化——**从"UI 好不好看"→"页面结构是否 Agent-friendly"**：结构可读、状态清晰、操作可复现、日志能记录"人让 Agent 做了什么"。产品设计不再只服务人类用户，还要服务人类用户的 AI 代理。

**隐含假设**：SaaS 公司有动力和能力快速适配 Agent。实际上很多 SaaS 的前端结构对 Agent 极不友好（动态渲染、验证码、复杂权限模型）。

**与你工作的直接关联**：车载的 23 个下游工具就是 Planner 的"SaaS"。Dan 的判断直接指向一个优化方向：**工具定义是否对 Planner 足够友好？** 参数描述是否清晰？返回格式是否结构化？错误信息是否可解析？如果工具定义含糊，Planner 的决策质量就会被拖累——这不是模型的锅，是"SaaS"（工具）的锅。

### 第六段：自动化是谎言（"每个 Agent 都需要一个人"）

**功能**：核心洞察——重新定义人+AI 关系

**深度分析**：这是全文最有分量的一段。Every 作为 AI 使用极其激进的公司，人数反而从 15 翻到 30——Dan 用一句话解释：**"自动化是一种谎言。"** 每当你自动化一个流程，就需要一个人在上面盯着确认它真的服务业务。他给了一个惨烈的反面案例：vibe coded 产品发布后每 10 分钟崩一次，让 Codex 修→每修一处带出四个新错误→最后找两位高级工程师从头重写。这揭示了一个不对称性：**AI 的生成速度 >> 人的审核速度**。当生成能力被无限放大，判断力、系统感和责任感成为瓶颈。很多团队接下来要补的岗位，不是"会用 AI 的人"，而是"懂业务+懂工具+愿意照看 Agent 的人"。数据科学团队的变化也类似：以前是数据科学家回答问题，现在每个人都能让 AI 做分析，团队反而要检查结论、纠正方法、建设能处理基础请求的 Agent。

**隐含假设**："自动化需要人盯"是结构性特征而非暂时限制。Dan 似乎认为这是永久的；但如果 Hassabis 说的"持续学习"问题解决了，Agent 真的能从错误中自我改进，这个判断就需要更新。

**精彩**："自动化是谎言"这个表述极其有力，因为它直接对抗了硅谷的默认叙事（AI 会替代一切）。**破绽**：如果这是真的，那大规模降本增效的 narrative 就站不住——但 Every 自己的例子（30 人做了以前需要 100 人的事）说明降本增效确实存在，只是不是通过"替代人"而是通过"人+Agent"来实现。

**与你工作的直接关联**：这完美解释了为什么 Planner 需要你持续维护。SP 不是写完就行的，它需要一个"在乎它的人"持续看日志、处理 badcase、调优化。这不是因为 Planner 不够好——这是 Agent 产品的结构性特征。你的角色就是 Dan 说的"在乎 Agent 的那个人"。

### 第七段：PM 能力迁移（"他发货速度比任何人都快"）

**功能**：具体示例——PM 如何在 AI 时代变强

**深度分析**：Marcus 的例子是全文最有说服力的证据。他的背景：做过 Axios 写作产品、带过大团队、做到数千万 ARR——但不是传统工程师，只是"轻技术"（知道数据库迁移是什么、能看懂一些代码）。以前 Every 不会让这样的人独立负责产品实现，但 Cursor/Claude Code 把他的产品感、写作判断和用户理解**放大了**——他的发货速度"几乎比团队里任何人都快"。这里的关键不是 Marcus 学会了写代码，而是 AI 把"造出来"的门槛压低到了 PM 的判断力能直接发挥作用的程度。PM 的核心能力重新定义：从"敲 PRD"→"判断该做什么、好不好、解决什么问题"。Dan 对 AI 生成文档的态度也很精确：可以用 GPT 写策略文档（甚至常常比人硬敲更好），**红线是"发送者是否站得住每一行"**。如果开会发现你不知道文档里写了什么，不可接受。

**隐含假设**：PM 能快速学会 Cursor/Claude Code。实际上很多 PM 对命令行有恐惧，对代码逻辑缺乏基本直觉。Marcus 之所以能成功，可能是因为他本身就是"轻技术"——这个前提并不普遍。

**与你工作的直接关联**：你作为 AI PM，不应该只停留在"写 SP 和调参数"。Marcus 的例子暗示你可以更进一步——用 Claude Code 写 badcase 自动分类脚本、用 Cursor 做 Planner 的原型实验（比如测试一个新的工具组合策略）、直接分析用户交互 log 找模式。你的产品感+AI 工具的编码能力 = 比纯工程师更快的洞察-行动循环。

### 第八段：全栈设计师（"直接把想法造出来"）

**功能**：并行例证——设计师的能力升级

**深度分析**：设计师的痛点被精准描述：做了漂亮交互→工程实现被砍→最终效果打折。AI 工具让设计师跳过等待，直接做出可运行版本甚至提交 PR。这解决了"设计-工程交接"这个经典瓶颈。但更深层的意义在于：当更多人能快速生成同质化 UI，**创造力和审美品味成为真正的差异化因素**。Lenny 也提到"很多 AI 设计工具一眼就能看出模板味"——这恰好说明默认输出在快速同质化，只有真正有品味的设计师才能从一堆相似界面里跳出来。

**隐含假设**：设计师愿意且能够学习代码工具。实际上这个群体和 PM 一样，很多人对技术工具有抵触。

### 第九段：骑在模型上（"AI 的边缘不在旧金山"）

**功能**：行动号召——给普通从业者的建议

**深度分析**：Dan 拒绝"AI 末日"叙事，给出一个更精细的判断：模型每次进步→把昨天的人类能力变便宜→默认输出同质化→人的任务是用便宜能力做新东西。"ride the models"是一个动态策略：新模型出来→拿自己工作流试→以前做不了的事过几个月再试。最精辟的一句话："AI 的边缘不在旧金山，而在 AI 遇到一个正在做事的人那里。" 这句话的含义极深——AI 的价值不在技术本身（旧金山），而在技术与具体场景的交叉点（正在做事的人）。这意味着：越深入具体领域的人，在 AI 时代反而越有价值，因为你提供了 AI 缺少的"场景上下文"。

**隐含假设**：个人有时间和精力持续追踪和试用新模型。实际上大多数人的工作节奏不允许频繁探索。

**与你工作的直接关联**：你就在"AI 遇到正在做事的人"的位置——车载智能座舱 × AI 的交叉点。你积累的场景 know-how（什么时候用户想开空调、什么情况需要多步规划、什么话术让人舒服）就是 Dan 说的"边缘"。这种领域知识不会被模型更新取代。

## 论证链

"住进未来而非预言"（方法论）→ "reach test 判断工具价值"（微观标准）→ "公司先养超级 Agent"（部署路径）→ "Codex 成为工作桌面"（入口迁移）→ "SaaS 不死，Agent 增加用户"（生态变化）→ "自动化是谎言，Agent 需要人"（人+AI 关系）→ "PM 从 PRD 迁移到定义问题"（能力重定义）→ "设计师直接造出来"（并行验证）→ "骑在模型上，边缘在做事的人"（行动原则）

## 隐含假设

1. **PM/设计师天然具备可被 AI 放大的"判断力"**——但很多 PM 的核心工作就是写文档和协调，AI 放大的可能是空白
2. **非技术人员能快速上手 CLI 类 AI 工具**——Marcus 是"轻技术"背景，这不是普遍前提
3. **公司级 Agent 的 ROI 足够高**——能支撑一个全职维护者，但小团队可能算不过来
4. **"自动化需要人"是结构性特征而非暂时限制**——如果持续学习问题解决，这个判断需要更新
5. **AI 的价值在"场景交叉点"而非"技术前沿"**——这暗示非旧金山的从业者反而有优势，但前提是他们能接触到最新工具

## 观点边界

1. **公司规模限制**：Dan 的经验来自 30 人小公司，大公司的合规/安全/审批流程会显著限制 AI 工具的自由探索
2. **行业适用性**：所有例子都是"知识工作"（写作、产品、营销），不适用于制造业、医疗等受监管行业
3. **与 Hassabis 的张力**：Dan 说"自动化是谎言，需要人盯"，Hassabis 说"根因是缺持续学习"——如果持续学习问题解决了，Dan 的判断是否还成立？
