# 场景知识 Selector 评测规则文档

> 版本：v1.0 | 更新日期：2026-05-20
> 适用组件：司机 Agent 2.0 — Knowledge Selector（第一层场景分类器）
> 模型：doubao-seed-1.6-flash-250828

---

## 一、Selector 是什么

### 1.1 在系统中的位置

```
用户语音 → ASR 转文字
                ↓
        ┌──────────────┐
        │  Selector    │ ← 本文档评测的对象
        │ （场景分类器）│
        └──────┬───────┘
               │ 输出场景 ID
               ↓
        ┌──────────────┐
        │  知识注入     │ 根据 ID 把对应的"场景知识"注入下一层
        └──────┬───────┘
               ↓
        ┌──────────────┐
        │  Planner     │ 拿到场景知识后，生成回复 + 调用车辆工具
        │ （对话+执行） │
        └──────────────┘
```

### 1.2 Selector 的唯一职责

**根据用户 query + 上下文，输出正确的场景 ID。**

- 命中场景 → 输出 `ID_001` ~ `ID_013` 中的一个或多个
- 未命中任何场景 → 输出 `ID_000`
- 多场景同时命中 → 用英文逗号分隔，如 `ID_004,ID_012`

### 1.3 为什么 Selector 很重要

| Selector 表现 | 对用户的影响 |
|:---|:---|
| 分类正确 | Planner 拿到正确知识 → 精准回复 + 正确工具调用 |
| 分类错误 | Planner 拿到错误知识 → 执行错误操作（如用户说冷却触发降温） |
| 输出 ID_000（漏判） | Planner 没有场景知识 → 只能通用闲聊，无法执行场景方案，RT 变长 |
| 输出文字（格式错误） | 系统无法解析 → 整个流程中断，用户无响应 |

## 二、核心概念：什么走场景（001-013），什么走兜底（000）

### 2.1 判断原则

> **一句话：用户这句话背后，车是否需要"按一套方案协调多个系统"来响应。**

- **需要方案** → 走场景知识（ID_001 ~ ID_013）
- **不需要方案** → 走兜底（ID_000）

### 2.2 需要场景知识的情况（ID_001-013）

用户表达了一个意图，这个意图不是"执行某一个动作"就能满足的，而是需要多个车辆子系统协同配合。

**示例：**

| 用户说 | 为什么需要场景知识 | 场景 ID |
|:---|:---|:---|
| "好热啊" | 需要先开窗散热→再开空调最低温→2分钟后关窗回正常，多步骤编排 | ID_004 |
| "冻死了" | 需要空调+座椅加热+方向盘加热+除雾同时启动 | ID_005 |
| "帮我哄哄孩子" | 需要推荐适龄内容+调音量+开儿童锁+安全设置 | ID_001 |
| "我要眯一会" | 需要座椅放倒+遮阳帘+温度维持+静音+定时叫醒 | ID_009 |
| "后排放个电影" | 需要后排屏开启+音源路由+前排隔离+内容推荐 | ID_002 |
| "玻璃起雾看不清" | 需要除雾+雨刮+后视镜加热+雾灯联动 | ID_010 |

### 2.3 不需要场景知识的情况（ID_000）

分三种类型：

**类型 A：单一控制指令 — 用户已明确告知具体动作**

| 用户说 | 为什么不需要 |
|:---|:---|
| "空调调到 25 度" | 一个精确指令，直接执行，不需要方案 |
| "打开方向盘加热" | 一个开关操作 |
| "把车窗关上" | 一个动作 |
| "音量调大一点" | 一个调节 |

**类型 B：信息咨询/闲聊 — 不涉及车辆操控**

| 用户说 | 为什么不需要 |
|:---|:---|
| "今天天气怎么样" | 查信息，不操控车 |
| "有没有好看的动作片推荐" | 只是问推荐，没有要看 |
| "明天几点日出" | 纯闲聊 |

**类型 C：其他功能模块 — 有独立处理通道**

| 用户说 | 为什么不需要 |
|:---|:---|
| "帮我导航去公司" | 走导航模块 |
| "放首周杰伦的歌" | 走音乐模块 |
| "给老婆打个电话" | 走通讯模块 |

### 2.4 最易判错的边界（重点理解）

| 用户说 | 表面上像 | 实际应该判 | 原因 |
|:---|:---|:---|:---|
| "太冷了，调高温度" | 单一指令（调温度）→ 000 | **ID_005** | "太冷了"是体感不适，需要整套升温方案 |
| "方向盘太冷了" | 开方向盘加热 → 000 | **ID_005** | 是整体温度不适的表达，不是在下单一指令 |
| "方向盘冻得握不住，开加热" | 体感冷 → 005 | **ID_000** | "开加热"是明确的单一控制指令 |
| "帮我找古装剧全集" | 搜索信息 → 000 | **ID_003** | "找全集"隐含观看意图，需要开屏播放 |
| "有没有好看的动作片" | 想看电影 → 003 | **ID_000** | 只是问推荐，没有观看动作词 |
| "腰不舒服"（副驾说） | 闲聊抱怨 → 000 | **ID_006** | 副驾表达身体不适 → 座椅调节需求 |
| "腰不舒服"（主驾说） | 座椅问题 → 006 | **ID_000** | 006 仅限副驾座椅，主驾座椅无专属方案 |

## 四、评测维度与优先级

### 4.1 评测维度总览

| 优先级 | 评测维度 | 说明 | 占比建议 |
|:---|:---|:---|:---|
| **P0** | 输出格式合规 | 输出必须是纯 ID，不能有任何文字 | 15% |
| **P0** | 场景分类准确 | 命中正确的场景 ID | 50% |
| **P1** | 边界区分能力 | 易混淆场景对的判断准确率 | 25% |
| **P2** | 上下文利用 | 结合 speaker_position、env_info 等信息辅助判断 | 10% |

### 4.2 P0：输出格式合规

**规则：**
- Selector 的输出只能是场景 ID，不允许有任何其他内容
- 单场景：`ID_004`
- 多场景：`ID_004,ID_012`（英文逗号分隔）
- 未命中：`ID_000`

**通过标准：**
| 输出 | 判定 |
|:---|:---|
| `ID_004` | PASS |
| `ID_004,ID_012` | PASS |
| `ID_000` | PASS |
| `ID_004，因为用户说好热` | **FAIL**（包含解释文字） |
| `好的，我帮你开空调降温` | **FAIL**（对话回复） |
| `004` | **FAIL**（缺少 ID_ 前缀） |
| 空值 / 无返回 | **FAIL**（管道异常，需排查工程链路） |

### 4.3 P0：场景分类准确

**规则：**
- 输出的 ID 必须与预期 ID 完全一致
- 多场景输出时，ID 集合必须完全匹配（顺序不限）

**通过标准：**
| 预期 | 实际输出 | 判定 |
|:---|:---|:---|
| ID_004 | ID_004 | PASS |
| ID_004,ID_012 | ID_012,ID_004 | PASS（顺序不限） |
| ID_004,ID_012 | ID_004 | **FAIL**（漏召回 012） |
| ID_004 | ID_005 | **FAIL**（错判） |
| ID_004 | ID_000 | **FAIL**（漏判） |
| ID_000 | ID_004 | **FAIL**（误判） |

### 4.4 P1：边界区分能力

以下 8 组易混淆场景对，是测试重点。每组至少覆盖 2 条正例 + 2 条反例。

#### 组 1：儿童陪伴(001) vs 后排勿扰(013) vs 车内小憩(009)

| 用户说 | 正确 ID | 判断依据 |

| "帮我哄哄孩子" | 001 | 主动陪伴孩子 |
| "给孩子放个动画" | 001 | 给孩子娱乐 |
| "哄哄他他困了" | 001 | 给孩子放安抚助眠内容，仍是主动陪伴行为 |
| "孩子睡了别吵他" | 013 | 孩子已经睡了，要求安静勿扰 |
| "后面老人要睡了" | 013 | 后排有人要休息，勿扰 |
| "孩子困了帮我营造个睡眠环境" | 009 | 要营造睡眠环境（座椅+温度+遮光） |

#### 组 2：后排观影(002) vs 前排观影(003) vs 5D沉浸(007) vs 兜底(000)

| 用户说 | 正确 ID | 判断依据 |
|:---|:---|:---|
| "后排放个电影" | 002 | 后排 + 观看 |
| "给后排的朋友放点综艺" | 002 | 明确说给后排 |
| "我想看个电影"（前排说） | 003 | 前排 + 观看 |
| "帮我找古装剧全集" | 003 | "找全集"含观看意图 |
| "开个5D影院" | 007 | 明确要 5D |
| "有没有好看的动作片推荐" | 000 | 只是问推荐，无观看动作 |
| "最近有什么好剧吗" | 000 | 只是咨询，无观看动作 |

**观看意图判定标准：** 满足以下任一 = 有观看意图
- 含动作词：看/播/放/追剧/来一集
- 含搜索+内容词：找/搜 + 电视剧/电影/综艺/全集
- 含指向词：来一部/这部/最新一期/第X季
- 含具体片名 + 动作词

不满足以上任一（仅推荐/介绍/询问/讨论）= 无观看意图 → ID_000

## 五、测试用例编写规范

### 5.1 每条测试用例必须包含的字段

| 字段 | 说明 | 示例 |
|:---|:---|:---|
| case_id | 用例编号 | TC_001 |
| query | 用户输入文本 | "好热啊，快给我降降温" |
| speaker_position | 发言人座位（主驾/副驾/后排/未知） | 主驾 |
| env_info | 环境信息（可选） | 室外温度38度 |
| conversation_history | 对话历史（可选） | 上一轮提到孩子 |
| expected_id | 预期输出的场景 ID | ID_004 |
| priority | 用例优先级 | P0 / P1 / P2 |
| category | 所属评测维度 | 边界区分 / 格式合规 / 上下文利用 |
| notes | 判断依据说明 | 用户表达热体感，走急速降温 |

### 5.2 预期 ID 填写准则

在填写预期 ID 时，按以下顺序判断：

```
第一步：用户这句话是否跟车辆场景相关？
  → 不相关（闲聊/导航/音乐/电话）→ ID_000

第二步：相关的话，用户需要的是"一套方案"还是"单个动作"？
  → 单个动作（调到25度/开XX加热/关车窗）→ ID_000
  → 一套方案 → 进入第三步

第三步：匹配具体是哪个场景
  → 对照 13 个场景的定义和触发关键词
  → 如果有两个场景都像，查互斥规则确认
```

### 5.3 测试用例覆盖要求

| 分类 | 最低用例数 | 说明 |
|:---|:---|:---|
| 每个场景（001-013）正例 | 每场景 ≥ 3 条 | 覆盖不同表达方式 |
| ID_000 正例 | ≥ 10 条 | 覆盖三种类型（单一指令/闲聊/其他模块） |
| 8 组边界对 | 每组 ≥ 4 条 | 每组至少 2 正 2 反 |
| 上下文依赖 | ≥ 5 条 | 同一 query 不同 context 的对照 |
| 多场景召回 | ≥ 3 条 | 如 "好热好臭" → 004+012 |
| 格式验证 | ≥ 5 条 | 验证不输出文字 |

## 六、评测不覆盖的范围

以下内容**不属于** Selector 评测范围，测试时请排除：

| 不测什么 | 原因 | 谁负责 |
|:---|:---|:---|
| Planner 回复质量 | Selector 只管分类，不管分类后系统怎么回复 | Planner 评测 |
| 场景知识内容合理性 | "急速降温应该先开窗还是先开空调"是知识编写问题 | 产品/场景知识编辑 |
| 工具调用是否正确 | 调空调、开车窗等具体执行是 Planner+工具层的事 | 工具层评测 |
| 系统自动触发场景 | advisor 检测到孩子哭闹自动弹提示，不走 Selector | 系统侧评测 |
| ASR 识别错误 | 语音转文字出错不是 Selector 的问题 | ASR 评测 |
| 管道/平台问题 | 返回 nan/空值是工程链路问题 | 工程排查 |

## 七、通过标准与指标

### 7.1 准入指标

| 指标 | 计算方式 | 目标值 |
|:---|:---|:---|
| 整体准确率 | PASS 数 / 有效用例总数 × 100% | ≥ 95% |
| 格式合规率 | 格式正确数 / 总用例数 × 100% | 100%（0 容忍） |
| 边界准确率 | 边界 case PASS 数 / 边界 case 总数 × 100% | ≥ 90% |
| P50 延迟 | Selector 单层调用耗时中位数 | ≤ 800ms |
| P99 延迟 | Selector 单层调用耗时 99 分位 | ≤ 1500ms |

### 7.2 有效用例定义

以下情况不计入准确率分母：
- 返回 nan / 空值（管道问题，单独统计并提交工程排查）
- 预期 ID 为空（用例不完整）

### 7.3 失败分类

每条 FAIL 用例需标注失败类型：

| 失败类型 | 定义 | 严重度 |
|:---|:---|:---|
| 格式错误 | 输出包含非 ID 内容（文字回复/解释） | P0-Critical |
| 错判 | 输出了错误的场景 ID | P0-Major |
| 漏判 | 应命中场景但输出 ID_000 | P0-Major |
| 误判 | 不应命中场景但输出了非 000 的 ID | P1-Major |
| 漏召回 | 多场景命中时少输出了一个 ID | P1-Minor |
| 管道异常 | 返回 nan / 空值 / 超时 | P0-Blocker（工程） |
