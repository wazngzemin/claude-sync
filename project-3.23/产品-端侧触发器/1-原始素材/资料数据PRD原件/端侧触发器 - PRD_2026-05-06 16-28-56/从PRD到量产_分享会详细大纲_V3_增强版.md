### 一、开场：为什么一个AI PM跑去做汽车了

**时长：** 约5分钟

**目的：** 用一个问题引发好奇心——"AI产品从数字世界走到物理世界，到底有什么不一样？"

#### 1.1 破冰：一个反直觉的问题

- 开场提问："在座各位做AI产品的，有没有想过一个问题——你做的AI，犯一次错，代价是什么？"
- 引导思考：
  - 做推荐算法，推错了一条内容，用户划走就行，代价几乎为零
  - 做对话机器人，回错了一句话，用户重新问一遍，代价是几秒钟的耐心
  - **但如果你做的AI是在开车呢？犯一次错，代价可能是一条命。**
- 引出主题："这就是我现在每天面对的事情——让大模型在物理世界做决策。"

#### 1.2 我是谁

- 快速经历线：百度（Agent平台）→ 理想（车载AI产品）→ 字节（大模型上车）
- "一直在AI产品一线，从做平台到做端侧，从数字世界做到物理世界。"

#### 1.3 今天你们能带走什么

- "我会用一个车载场景作为案例，讲四件事：**怎么从用研推出产品定义、怎么设计一个Agent的系统架构、怎么给大模型产品建评测体系、以及在这个过程中踩过的真实的坑。**"

### 二、【方法论一】怎么从用研推导出AI产品的定义

**时长：** 约8分钟

**目的：** 展示"从用户洞察到产品定义"的思维链路。

#### 2.1 一个所有AI PM都会遇到的问题：用户到底要什么？

- 常见错误路径：
  - ❌ 从技术能力出发
  - ❌ 从竞品功能出发
  - ❌ 从老板需求出发
- 正确路径：**回到用户的真实场景，找到"期待"和"现实"之间的断层——断层就是产品机会**

#### 2.2 方法论：怎么做AI产品的用户调研

**第一步：找到对标物——用户心里的"满分体验"是什么？**

- 来自知识库的实际做法：
  - 与赛力斯联合开展多轮共创调研（Round 2、Round 3）
  - 调研框架：驾驶习惯观察、情绪情感需求访谈、语音助手偏好对比、智能化体验评价
  - Round 2 实际执行：深度访谈理想汽车用户，了解其使用体验、功能需求、品牌理念
  - 2.0 PRD 中的用户研究：与 11 位网约车/专车司乘用户进行 workshop，与 1 位私家司机、2 位 Robotaxi 深度用户进行 1V1 深度访谈
  - 核心发现：快车、专车、私人司机、Robotaxi 四类出行方式存在巨大体验断层——这就是司机Agent的差异化机会点

- **通用方法论：做AI产品调研时，不要问"你想要什么功能"，要问"你体验过的最好的服务是什么"。用户不会告诉你他要一个Agent，但他会告诉你他被一个专车司机感动过。**

**第二步：建体验阶梯——找到断层在哪里**

- 出行体验阶梯：

  | 等级 | 对标 | 核心体验 | 用户评价 |
  |------|------|---------|---------|
  | L1 基础工具 | 快车/基础导航 | 能到就行 | "凑合" |
  | L2 贴心服务 | 专车司机 | 经验驱动、主动适配 | "贴心好用" |
  | L3 专属无摩擦 | 私人司机 | 记忆驱动、预判需求 | "专属无摩擦" |
  | 当前AI出行 | 基础智驾+座舱 | 算法驱动、机械执行 | "安全但呆板" |

- **来自知识库的用户真实痛点**（Planner PRD 中记录）：
  - "我讲一句，它应一句，一锤子买卖，需求不闭环"
  - "想让座舱更舒适，但具体现在应该怎么操作呢？"
  - "搜一个我想看的电影很费劲，能不能结合我的偏好和当前指令把这件事搞定？"
  - "当前的闲聊聊不下去，既割裂又没内涵——不知道舱内外发生了什么、不知道车辆处于什么状态、不知道正在做的事情的进展"

- **通用方法论：不要问"产品还缺什么功能"，要问"用户期待的体验等级是什么，我们在哪个等级"。差距就是产品定义。**

**第三步：把断层翻译成产品语言**

- 断层洞察："用户想要的不是一个更好的语音助手，而是一个像人一样思考和行动的AI司机"
- 产品定义推导：
  - 旧定义：车载AI = 语音助手 + 智能驾驶（两个独立的东西）
  - 新定义：车载AI = 一个统一的Agent，既能理解意图、管理任务，又能操控车辆在物理世界执行
- 检验标准："物理世界图灵测试"——用户闭上眼，分不清是人还是AI在开车和服务

### 三、一个场景故事：周五傍晚的一句话

**时长：** 约5分钟

**目的：** 用故事把听众拉进来，展示"从场景故事推导产品功能"的思维方式。

#### 3.1 场景带入

> 周五傍晚6点，小桥下班了。他提前订好了电影票，准备接上还在加班的朋友小明一起看电影。
>
> 上车后，他对车里的AI说了一句话：
>
> **"激活智驾出发，路上帮我在炸鸡店买份双人套餐，然后去天河广场地库B2层接上小明，一起去星光影城看电影，到了帮我找个离电梯近的停车位。"**
>
> 一句话。四个任务。接下来50分钟的行程，全部交给AI。

#### 3.2 一句话拆出来的四段旅程

| 时间段 | 旅程阶段 | AI需要做什么 |
|--------|---------|-----------|
| 18:00→18:20 | 路上买炸鸡 | 理解"顺路" → 沿途找店 → 提前下单 → 临停取餐 → 继续行驶 |
| 18:20→18:35 | 地库接小明 | 进入地下停车场 → 找到B2层 → 临停等人 → 通知小明 |
| 18:35→18:50 | 去看电影 | 场景变了（多了一个人）→ 驾驶风格自动调整 → 影院信息查询 |
| 18:50 | 到达停车 | 找离电梯近的车位 → 泊车 → 推送车位信息 |

#### 3.3 为什么用场景故事来做产品设计

- "我做AI产品的第一步，永远不是画架构图、不是列功能清单，而是**先写一个用户故事，然后在脑子里把这个故事完整地跑一遍**。"
- "每跑到一个环节，我问自己三个问题：**① 用户此刻的真实诉求是什么？② AI需要具备什么能力？③ 如果AI做错了，最坏的后果是什么？**"
- **来自知识库的实际验证**——临停功能PRD中，产品团队正是通过枚举51条评测Query（即时17条+条件17条+目标17条）来反向验证场景覆盖。每一条Query都对应一个真实的用户故事和验收条件。

### 四、【方法论二】怎么设计一个AI Agent的系统架构

**时长：** 约14分钟（核心重头戏）

**目的：** 全场信息密度最高的部分。用故事中的旅程作为案例，讲清楚AI Agent系统设计的核心模块。

#### 4.1 先建认知：传统方案为什么不行，Agent方案解决了什么

**来自知识库的真实架构对比**（座舱Planner OnePage）：

- **传统座舱架构的三大缺陷：**
  1. 模块协同性不足——各模块独立闭环，缺乏全局信息共享，跨域任务衔接差
  2. 场景感知能力薄弱——未集成用户记忆与实时环境，无法自适应智能交互
  3. 成本与泛化瓶颈——长尾场景需单独定制逻辑，研发成本高、周期长

- **传统方案 vs AI Agent方案：**

  | 对比维度 | 传统方案 | AI Agent方案 |
  |---------|---------|------------|
  | 任务覆盖 | 部分领域的简单任务 | 全场景覆盖，不分领域 |
  | 扩展性 | 每增加一个场景要写新规则 | 新场景通过工具组合自动适配 |
  | 复杂任务 | 几乎无法处理 | 自动拆解，Agent协作完成 |
  | 维护成本 | 规则越来越多，持续上升 | 能力层稳定，可控 |

- **Agent架构类比（私人管家）：**
  - **Director = 管家的大脑：** 听懂主人说的话，判断要做什么，决定谁去做
  - **Planner = 管家的执行力：** 把任务变成一步步的计划，协调各方执行
  - **Context = 管家的记忆和五感：** 记得主人的偏好，知道现在几点、在哪里、外面什么天气
  - **Skill = 管家的工具箱：** 每个工具有明确的管辖范围和失败兜底

#### 4.2 架构核心原则：漏斗原则（Funnel Principle）

**来自知识库的核心设计思想**（座舱Planner OnePage）：

- "基于语音和多模的交互，流畅性和稳定性是第一要素。当前大模型技术水平达不到兼顾速度和对复杂任务的处理，因此这是一个过渡方案，后续模型能力提升后可逐步推进。"

| 层级 | 处理方式 | 响应时间 | 适用场景 |
|------|----------|----------|----------|
| **L1：句法RAG** | 规则匹配，直接执行 | <500ms | 明确单轮指令（车书、音乐、简单车控） |
| **L2：情景RAG** | 知识注入，辅助模型 | <1s | 模糊单轮指令（"我好热"、复合意图） |
| **L3：复杂Planner** | 模型推理，工具调用 | 1-3s | 复杂任务、多轮对话、多步推理 |

- **设计目标：** 80%以上的简单请求在第一层快速闭环；句法RAG可开放给车厂自行配置；复杂case的处理数据回流，逐步增强前置模块能力

- **实际应用案例**（临停功能PRD）：
  - "靠边停车"→ 句法RAG → directExecute → pullover_execute（<500ms，不过Planner）
  - "到了下个服务区停"→ 情景RAG → Planner → condition_monitor注册条件（<1s）
  - "停在拉行李箱的人旁边"→ 云侧Planner → VLM识别 → pullover_execute（1-3s）

#### 4.3 Director设计：怎么让AI听懂"人话"

**核心命题：用户说的是自然语言，系统需要的是结构化指令。**

- **回到故事：** 用户说了一句话包含四个任务，Director第一步做什么？
  - ① 意图识别：这句话表达了几个意图？每个是什么类型？
  - ② 任务拆解：哪些立即执行？哪些条件触发？
  - ③ 依赖排序：任务之间的先后关系和并行可能
  - ④ 工具匹配：每个任务需要调用哪些Skill？

- **来自知识库的实际踩坑：**
  - Badcase："前面星巴克靠边停"被理解成了"导航去星巴克"
  - 根因：prompt里没有清晰定义"沿途临停"和"目的地变更"的区分规则
  - 修复：在prompt中增加特征判断规则——当query包含方位词（前面/旁边/路边）+ 临停类动词（停一下/靠边），判定为沿途临停

- **来自2.0 PRD的临停参照物类型体系**（Director必须先判断类型，再决定走哪条链路）：

  | 参照物类型 | ParkRefType | 典型Query | 处理链路 |
  |-----------|-------------|-----------|---------|
  | 无参照临停 | NONE | "前面靠边""找个地方靠边" | 智驾自主选点，就近安全靠边 |
  | 路口参照 | INTERSECTION | "过了这个红绿灯靠边停" | TBT序列对齐，判断当前/下一路口 |
  | POI参照 | POI | "前面全家停一停" | 沿途POI搜索，150m→200m→对向道路 |
  | 视觉参照 | VISUAL | "停在拉行李箱的人旁边" | VLM实时识别，ODD ≤35km/h |

- **关键设计原则：**
  - **"能推断就不问"** ——降低用户的交互成本。反面教训：早期Director遇到不确定性都反问，用户说"你怎么比我妈还啰嗦"
  - **"所有澄清必须有兜底"** ——如果用户不回答，系统不能卡死。沉默=失控

#### 4.4 Context设计：AI怎么知道"现在是什么状况"

**核心命题：AI Agent不是生活在真空里，它需要感知环境。**

- **来自知识库的Context体系**（公共Context管理PRD）——13类Context：

  | Context类型 | 说明 | 来源 |
  |-------------|------|------|
  | 情景描述 | 舱内外环境、成员状态 | 端侧感知模型 |
  | 对话历史 | 用户与AI的多轮对话记录 | 对话系统 |
  | 事件日志 | 人物、车辆、AI的行为记录 | 各模块上报 |
  | 车辆状态 | 空调、车窗、座椅、能源等 | 车辆CAN信号 |
  | 屏幕UI | 当前屏幕显示内容 | HMI系统 |
  | 用户记忆 | 用户属性、偏好、行为 | 记忆模块 |
  | 任务状态 | 当前任务列表及状态 | Planner |
  | 基础联网信息 | 天气、节日等 | 云端服务 |
  | 豆包探索 | 主动探索和推荐 | 探索模块 |
  | 通用任务经验 | 常见任务处理经验 | 知识库 |
  | 工具知识 | 各工具的能力描述 | 工具注册 |
  | 专有知识 | 车书、产品手册等 | 知识库 |
  | 豆包人设 | AI性格特征 | 配置 |

- **"我好热"案例**——同一个query，不同Context → 完全不同的任务拆解：
  - 情景Context：外面正在下大雨 → 不拆解"开窗散热"
  - 端状态Context：空调温度已经调到最低了 → 不拆解"再调低空调"
  - 记忆Context：用户之前说过"不喜欢吹风" → 不拆解"大风量"
  - 最终拆解：打开空调 → 21度 → 3档风量 → "车里是有点热呢，我来帮你打开空调降降温～都搞定啦"

- **最深刻的踩坑教训：**
  - 早期把57条fewshot示例全部塞进prompt（SP总token膨胀到~34,000）
  - 结果：模型开始忽略真正重要的runtime信息——"当前车速35km/h"这个关键约束被淹没在一堆fewshot里
  - 修复：Context信息优先级排序和动态裁剪——核心层信息永远在最前面，条件层按需插入，背景层fewshot从57条砍到12条最关键的
  - 来自知识库的SP压缩目标：从~34,000 tokens → 15,000 tokens以下（减少>50%）

- **通用方法论：Context设计第一原则是"精准"，不是"全面"。Prompt不是写作文，是做信息架构。**

#### 4.5 Skill设计：AI手里应该有几把"刷子"

**核心命题：Agent的能力边界，取决于工具的数量和可靠性。**

- **来自知识库的实际工具体系：**
  - Planner PRD定义的工具库：车控工具、本地生活、音乐、记忆、条件任务、联网问答/搜索、语音扮演、Face ID、VQA、状态查询 ——共10大类
  - Planner SP中实际注册了18个Tool：vehicle_basic_control、search_vehicle_status_info、search_weather、search_poi_qa、route_planning_qa、navi_basic_control、vehicle_manual_qa、search_and_control_short_video、search_and_control_music、search_user_memory、face_id_register、operate_user_memory、web_search、goal_list_update、search_and_control_ai_broadcast、gui_agent_operation、image_generate、search_visual_info
  - 2.0 临停功能新增5个Tool：poi_search、visual_reference_detect、gesture_direction_resolve、condition_monitor、pullover_execute

- **关键设计原则：**

  - **原则一：Skill不是越多越好。** 每多一个Skill，Planner的决策空间就翻一倍。18个Skill意味着极其龐大的调用组合空间。来自知识库的经验：GUI Agent 当前仅用于4个用途（点外卖、订座、订咖啡、交停车费），克制扩展。

  - **原则二：每个Skill必须有明确的失败兜底。** 来自POI临停的实际处理链路：搜索前方150m → 扩大至200m → 扩大至对向道路 → 澄清"是否就近靠边停车？"。永远不能出现"系统沉默了"。

  - **原则三：Skill之间的边界必须清晰。** 真实badcase：导航POI搜索和联网搜索两个Skill输入条件重叠，Planner两个都调了，结果冲突。修复：每个Skill定义明确的"管辖范围"。

  - **原则四：条件状态不依赖LLM多轮记忆。** 来自临停PRD的架构决策——条件临停状态由工程侧显式状态机维护，Planner仅在设定条件和取消条件时参与，持续监控由condition_monitor工程模块独立完成。这是对LLM可靠性不足的务实应对。

#### 4.6 Planner设计：怎么把任务变成一步步的执行动作

**核心命题：Director决定"做什么"，Planner决定"怎么做、按什么顺序做"。**

- **回到故事快速说明——六个执行步骤：**
  - Step 1：激活智驾 + 设置导航路线（含炸鸡店途经点）
  - Step 2：临近炸鸡店时触发 → 临停 + 下单取餐
  - Step 3：取餐完成后接续智驾 → 前往天河广场
  - Step 4：到达天河广场地库 → 泊车 + 通知小明
  - Step 5：接人后继续智驾 → 前往星光影城 + 调整驾驶风格
  - Step 6：到达影城 → 泊车 + 推送车位信息

- **来自知识库的五种任务类型**（Planner PRD v2.4）：

  | 类型 | 典型Query | 拆解逻辑 | 评测标准 |
  |------|----------|---------|---------|
  | **单步单任务** | "打开空调" | 直接调用执行层 | 首字响应P50≤500ms |
  | **单步多任务** | "打开空调并播放音乐" | 并行拆解多个操作 | 推理ACC≥90% |
  | **条件任务** | "5分钟后关掉座椅加热" | 设置触发条件→条件满足时自动执行 | 条件触发准确率 |
  | **多步任务** | "去个好吃的火锅店" | 时序依赖+条件分支，推荐→导航→到达提醒 | 步骤完整率+过渡话术自然度 |
  | **持续任务** | "帮我介绍沿途风景" | 持续性交互，不阻塞其他任务 | 持续时长+内容多样性 |

- **三个设计原则：**
  - ① "最小可执行单元"——每一步只做一件事，做完检查状态再走下一步
  - ② "中途变更是常态"——用户随时可能改变主意，Planner必须支持动态调整（Replan机制、打断&恢复机制）
  - ③ "任务可追踪"——来自知识库的全局State管理：每个query对应唯一plan_id，Planner通过plan_update function call输出"失败/已完成/待完成"状态，工程端解析存入全局State

- **来自知识库的Teacher-Student数据飞轮：**
  - 前期：Planner作为Teacher，处理复杂任务的同时，把决策过程数据回流给句法RAG引擎
  - 中期：句法RAG学会了越来越多的场景，简单指令直接RAG秒回，不再过Planner
  - 后期：积累的数据反哺训练Planner模型本身，形成闭环
  - 训练数据管线：Kimi K2蒸馏 → GPT-4o打分 → Doubao-Seed-1.6训练 → Kimi K2评测

### 五、【方法论三】怎么给大模型产品建评测体系

**时长：** 约8分钟

**目的：** 输出一套可直接复用的评测方法论。

#### 5.1 一个灵魂拷问：你怎么知道你的AI做对了？

- "做大模型产品，你面对的是一个**概率系统**——同一个输入，今天输出对的，明天可能输出错的。"
- "那你怎么定义'做对了'？怎么保证'持续做对'？"

**车载AI评测的特殊性：**

| 维度 | 互联网AI | 车载AI |
|------|---------|-------|
| 验证方式 | A/B测试、点击率 | **不能A/B**（驾驶中不能随机分配体验） |
| 容错空间 | 错了用户重来就行 | **安全第一**，错了不可逆 |
| 评测粒度 | 测单点功能 | **测整条链路**（意图→拆解→执行→反馈） |

#### 5.2 评测集怎么建：生长出来的，不是一次建完的

**来自知识库的真实实践：**

- **第一阶段：PM手写种子数据（冷启动）**
  - 晓伟整理了约150条query，按模块分类覆盖全场景（像人一样对话、明确车控、模糊车控、人脸注册/记忆、POI/导航、内容媒体、信息查询、主动服务、跟进任务、高阶体验——共10大模块）
  - 7人分工，每人负责一个大模块构建Context字段
  - 目标：4月底2000条高质量数据，5月启动训练
  - 比例：20%评测 + 训练同步（1000条训练配200条评测）

- **第二阶段：Badcase反哺（真实迭代）**
  - 积累了近300条badcase，每一条都反向生成评测case
  - GUI Agent专项：泽民分析了51个GUI相关bug，按工具分类逐个过

- **第三阶段：规则化自动生成（规模扩展）**
  - 模板化和参数化：把"前面那个炸鸡店停一下"泛化成"前面那个{品类}停一下"

**来自知识库的实际评测体系：**

- 司机Agent统一评测集：113条case（T001-T113），三级场景分类，每个case包含：评测Query、期望Intent、parkRefType、期望目标语义槽位、期望输出Action、期望HMI话术、异常分支、优先级
- 舱驾融合评测Query集：101条case，A/B/C三大类（条件任务/目的地任务/异常边界）
- Planner评分规则：必要步骤缺失-30分/个，执行错误-20分/个，冗余操作-5分/个
- 评分体系：结果评分占70%（动作正确性+位置准确性+操作完整性+安全性），过程评分占30%（理解准确性+澄清合理性+反馈及时性+执行平顺性），稳定性指标（pass@3、pass^3）

#### 5.3 Badcase分析方法论：四步闭环

- **第一步：归类**——问题出在哪个模块？Director意图理解？Planner调用错误？Skill执行bug？下游能力不支持？
- **第二步：归因**——追到根因。"模型输出错了"不是根因，"prompt里没有定义这类case的处理规则"才是
- **第三步：排优先级**——频率 × 严重度。高频低危的排前面批量修，低频高危的单独重点修
- **第四步：闭环验证**——修复后必须全量回归。真实教训：修了一个case的prompt规则，结果影响了另外5个case

**来自知识库的实际badcase案例：**
- GUI Agent 三大类问题：指令理解错误（把澄清当闲聊）、参数混淆（新建任务 vs 补充信息）、任务难度高
- Planner 8类问题：GUI澄清被当闲聊、澄清内容与页面不符、受对话历史影响、type传错+播报过度承诺、advisor上报问题、Goal创建/拆分问题、多任务互相影响、POI定位问题

### 六、回到全局：从1.0到2.0的能力跃迁

**时长：** 约3分钟

**目的：** 把前面的细节收拢，建立完整认知。

#### 6.1 1.0 vs 2.0

- **1.0 = 数字世界的Agent上车**
  - 能力：语音交互、车控、导航、娱乐、生活服务
  - 本质：让座舱更好用
  - 核心架构：句法RAG + 情景RAG + Planner + 18个Tool

- **2.0 = 物理世界的具身智能体**
  - 产品定义（来自2.0 PRD原文）："让汽车成为像人一样思考&行动的物理世界的具身智能体"
  - 新增四大能力方向：
    1. **行车场景**：无参照/路口参照/POI参照/视觉参照四种临停模式 + 多任务编排
    2. **泊车场景**：静态参照物泊入、豆包猎手（车位空闲提醒+抢车位）
    3. **驾驶风格控制**：语音控制跟车距离、变道、速度
    4. **跨端协同**：手机App联动、记忆跨端同步

#### 6.2 架构演进

- 两个方案并行：上汽&MMT&字节方案 / 赛力斯&元戎&字节方案
- 关键挑战：有图无图策略不同、VLM能力边界、多方协作接口协议

### 七、收尾：三点个人思考

**时长：** 约3分钟

#### 7.1 三个判断

- **判断一："做AI产品最稀缺的能力不是懂模型，是懂场景"**
  - 大模型的能力在快速进化，今天做不到的事情明天可能就能做
  - 但"理解用户真实场景 → 推导产品定义 → 拆解成系统设计"这套能力，是需要在项目里反复磨练的

- **判断二："AI产品80%的工作量在处理20%的异常"**
  - 传统产品经理花80%精力设计正常流程
  - AI产品经理反过来：正常流程往往很简单，真正的工作量在处理各种异常、边界、兜底
  - "如果你的PRD里异常处理的篇幅不超过正常流程的3倍，那你的PRD一定不够完整"
  - 实际数据：临停PRD中，5个场景的异常降级路径覆盖30种（5场景×6种异常），评测集中C类（异常与边界）占14条

- **判断三："AI PM的核心能力正在从'写文档'进化为'做系统设计'"**
  - 不需要会写代码，但需要具备**跟技术团队进行对等对话**的系统设计思维
  - 来自知识库的真实写照：Planner SP的7个模块拆解、token膨胀根因分析、SP压缩策略设计——这些都需要PM深度参与

#### 7.2 结束语

- "今天用一个场景故事，走完了大模型产品从用研到量产的全链路。"
- "核心想传递的不是车载AI怎么做，而是**一套做AI产品的思考方式**——从场景出发、用故事驱动、对系统设计有全局认知、对异常处理保持敬畏。"
- "我是王泽民，一个实战派的AI产品经理。谢谢大家，欢迎交流。"

### 八、Q&A互动

**时长：** 5-10分钟

| 可能的提问 | 回答方向 |
|----------|---------|
| "大模型上车最大的挑战是什么？" | 不是模型能力，而是物理世界的不可逆性——在手机上犯错用户可以重来，在车上犯错没有撤回键。 |
| "AI PM需要懂技术到什么程度？" | 不需要会写代码，但需要理解系统架构、能跟工程师就技术方案做有效决策对话。核心是"知道什么能做、什么不能做、代价是什么"。 |
| "prompt优化有什么经验？" | 三个原则：规则化（每条规则对应一个badcase）、分层化（信息按优先级排列）、可测试化（每次改prompt必须跑回归评测）。SP从34K tokens压缩到15K以下是真实在做的事。 |
| "怎么处理多方协作中的需求冲突？" | 关键是在前期定义清楚接口协议和badcase归属规则。2.0项目涉及字节、OEM、智驾供应商三方，事前不定清楚，事后一定扯皮。 |
| "端云协同怎么设计？" | 核心原则：简单确定性任务走端侧（句法RAG直出，毫秒级响应），复杂开放性任务走云端（Planner+COT推理）。边界不是固定的，随数据积累持续调整。 |
| "车载AI和手机AI产品思维的本质差异？" | 三个核心差异：① 环境驱动>用户驱动（AI需要主动感知环境变化）② 安全第一>体验优先（不能出错的设计思维）③ 不可回退>可重试（每个动作都可能是终态） |
