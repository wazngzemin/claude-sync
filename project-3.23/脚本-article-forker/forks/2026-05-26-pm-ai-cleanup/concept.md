## 详细说明

### 一、核心诊断：你在哪一层？

文章区分了三种 PM 状态，这不是好坏判断，而是帮你定位自己在哪里、下一步往哪走。

| 层级 | 特征 | 典型表现 | 风险等级 |
|------|------|---------|---------|
| **L0: 不用 AI** | 拒绝或忽视 AI 工具 | 用旧模板改 PRD、手画原型、"AI 不靠谱" | 极高 — 文章说"已经被干掉了" |
| **L1: 假用 AI** | 只和 AI 聊天，不给上下文 | 丢一句话让 AI 写 PRD、只用来润色文字 | 高 — 得出"AI也就这样"的错误结论 |
| **L2: 真用 AI** | 喂上下文，让 AI 在知识框架里工作 | 导入历史文档、架构图、业务规则后让 AI 拆需求 | 低 — 正在拉开差距 |
| **L3: AI原生 PM** | 用 AI 构建、实验、验证判断 | 自己写脚本分析数据、做多套方案回测、提前验证方向 | 极低 — Dan说的Marcus |

**对你的定位**：你因为维护 Planner SP，本质上已经在 L2-L3 之间——你在"给 AI 上下文"（SP维护）和"用 AI 做实验"（badcase分析）之间。下一步是更多地做 L3 的事。

### 二、从 L1 到 L2 的关键跳跃："上下文"到底是什么

文章的核心诊断是："AI 不强，很多时候不是模型不行，而是你给的上下文太少。"

但"上下文"这个词太抽象了。具体到 PM 工作，上下文分为四类：

| 上下文类型 | 具体内容 | 在 Planner 中的对应 | 怎么给 AI |
|-----------|---------|-------------------|----------|
| **业务上下文** | 业务场景、用户画像、核心流程 | SP 中的角色定义 + 聊天风格 + 车辆设备知识 | 写进 System Prompt 的固定部分 |
| **架构上下文** | 产品架构、技术约束、接口规范 | SP 中的 23 个工具定义 + 输出格式 | 工具描述越精确，AI 决策越准 |
| **历史上下文** | 过去的 PRD、决策记录、badcase | SP 中的 69 个示例 + 注意事项 | 示例 = 历史经验的结晶 |
| **实时上下文** | 当前状态、用户偏好、环境信息 | UP 中的对话历史 + 动态变量（车况/记忆/目标） | 每次调用时注入 |

**关键洞察**：你每天维护 SP 的工作，本质上就是在"给 Planner 上下文"。SP 里的每一条规则、每一个示例、每一段工具描述，都是在教 Planner "带着上下文干活"。文章说的"不给上下文只丢一句话"，对应到你的工作就是"工具描述模糊、示例不够、规则不清晰"的 SP。

### 三、三件套框架的深度展开

#### 第一件：工具 = 协作环境（不是聊天窗口）

文章强调"工具不是让你和 AI 聊天，而是搭建协作环境"。

**对你的映射**：

| 文章说的"工具" | 你的"工具" | 本质 |
|---------------|----------|------|
| Cursor / VS Code | Claude Code / Cursor | 让 AI 能访问你的文件和代码 |
| 管理上下文和 Skill | SP 管理 + 示例库 | 让 AI 在你的知识框架里工作 |
| 方便 AI 随时调用 | Planner 的工具调用机制 | 让 AI 能执行操作不只是说话 |

**你可以做的进阶**：
1. 用 Cursor 打开 SP 文件，让 AI 帮你审查 SP 中的逻辑冲突
2. 用 Claude Code 把 badcase 日志和 SP 放在同一个工作空间，让 AI 看着日志建议 SP 修改
3. 这就是"搭建协作环境"——不是在聊天框里问"怎么改 SP"，而是让 AI 看到你的真实工作文件

#### 第二件：上下文 = 你积累的知识资产

文章说"慢慢导入就好"，但没说怎么导入、导入什么。具体化：

**你的上下文资产清单**（按优先级排序）：

| 优先级 | 资产 | 当前状态 | 怎么导入 AI 工作流 |
|--------|------|---------|----------------|
| P0 | Planner SP（当前版本） | 已有 | 已经在用，持续优化 |
| P0 | Badcase 日志 | 有但未结构化 | 用脚本自动分类（上篇 fork 的 Prompt 模板） |
| P1 | 历史 SP 版本变更记录 | 可能散落 | 建一个 SP changelog，每次改动记原因 |
| P1 | 用户交互 log | 有 | 定期分析唤醒率/功能使用分布 |
| P2 | 竞品分析（小鹏/蔚来/理想的车载 AI） | 可能没有 | 用 Claude Code 爬取公开信息整理对比表 |
| P2 | 团队决策记录（为什么选方案 A 不选 B） | 可能在脑子里 | 每次决策后写一行到决策日志 |

#### 第三件：Skill = 可复用的工作模式

文章说 Skill "可以让 AI 帮你写"。更精确地说，Skill = **你把自己的工作方法形式化，让 AI 可以重复执行**。

**你已经有的 Skill（但可能没意识到）**：

| 你的工作 | 对应的 Skill | 怎么形式化 |
|---------|------------|----------|
| Badcase 排查 | "链路→工具描述→上下文→模型"排查路径 | 写成一个 prompt，让 AI 按路径逐步排查 |
| SP 修改 | 看 badcase → 判断根因 → 改对应部分 | 写成一个 checklist，附修改模板 |
| 新功能评估 | 评估新工具是否需要加入 Planner | 写成一个评估框架（AlphaFold 三条件可以用上） |
| 性能监控 | 关注延迟/token 消耗/缓存命中率 | 写成一个自动化监控脚本 |

**Prompt 模板——让 AI 帮你写 Skill：**

```
我是车载AI助手（豆包上车）的产品经理，负责Planner模块。
我有一个反复做的工作：[描述你的工作流]
每次的步骤大概是：
1. [步骤1]
2. [步骤2]
3. [步骤3]

请帮我把这个工作流写成一个可复用的Skill（标准化操作文档），包括：
- 触发条件：什么情况下执行这个Skill
- 输入：需要哪些信息
- 步骤：每步具体做什么，用什么工具
- 输出：产出什么
- 检查点：怎么判断做对了
```

### 四、"10 倍效率"的真正落地：从"改一版等反馈"到"多版快速回测"

文章说 10 倍效率不是速度快 10 倍，是试错成本降 10 倍。

**旧循环** vs **新循环**：

| | 旧循环 | 新循环 |
|---|-------|-------|
| Step 1 | 憋一版 SP 修改 | 快速生成 3 套 SP 修改方案 |
| Step 2 | 上线等反馈 | 用历史 badcase 回测 3 套方案 |
| Step 3 | 发现不对再改 | 上线前就知道哪套最好 |
| Step 4 | 改了再等 | 只上线最佳方案 |
| 风险 | 高（生产环境试错） | 低（离线回测试错） |
| 周期 | 1-2 周/轮 | 1-2 天/轮 |

**具体操作（结合上篇 fork 的 Prompt 模板）**：
1. 确定要修改的 SP 部分（比如某个工具的描述）
2. 让 AI 生成 3 种不同方案
3. 准备 10 个历史 badcase 的 user_query
4. 分别用 3 套方案构造 prompt，看哪套解决了最多 badcase
5. 选最优方案上线

### 五、判断标准

**升级成功的信号：**
- 你的 AI 使用不再只是"润色文字"或"写周报"
- 你有一个持续积累的上下文资产库（SP changelog、badcase 分类、决策日志）
- 你有至少 3 个可复用的 Skill（Badcase 排查 Skill、SP 修改 Skill、性能监控 Skill）
- 你能在 1-2 天内回测一个 SP 修改方案，而不是等 1-2 周的线上反馈

**还在原地的信号：**
- AI 只用来写文档和润色
- 没有结构化的上下文资产（badcase 散落在聊天记录里）
- 每次改 SP 都是"凭感觉改一条试试"
- 你说"AI 也就这样"——但从没给过它足够的上下文
