一、Advisor 体系总览：为什么需要 advisor、它和 Director 是什么关系

1.1 旧架构的问题

旧架构里 Director（Planner）一人包揽所有事——理解意图、拆解任务、执行、跟进进度、主动关怀，全压在一个模型调用里。这带来四个病：

  1. 过载——要同时兼顾"快速响应"和"深度思考"，性能和质量没法两全
  2. 单线程——系统不具备多任务并行的能力
  3. 需求不闭环——"我讲一句它应一句，一锤子买卖"
  4. 涌现受限——为了稳定加了很多规则，限制了模型能力的发挥

1.2 新架构的分工

新架构把 Director 的职责拆开：

  • Director——重时延、快交互、串联中枢。负责快速响应用户、调工具执行、建/删目标。不盯进度、不更新目标进展。
  • 4 个静态 advisor——重思考、轻时延、守目标。各管一条线（舒适/出行/情感/内容），永久在岗，负责盯目标是否达成、更新进展、给 Director 主动服务建议。
  • 动态 advisor——临时监工。当静态 advisor 管不了的临时持续任务出现（如"帮我照顾宝宝"），Director 创建一个动态 advisor 实例来专门盯。任务完成后销毁。
  • goal list——共享看板。Director 和 advisor 共同读写，所有目标的进展都在这里。
核心逻辑用一句话概括：能当场做完的，Director 闭环；做不完的、要持续盯的，升级成 goal + advisor，Director 建完就撒手，advisor 接管盯进度。恢复被打断的任务也不靠 Director 记忆——靠 advisor 读 goal list 上的"细节和进展"重新触发。

二、舒适 Advisor——车内的"体感管家"

2.1 为什么做

车内环境（温度/空气/光线/声音/座椅/空间）时刻在变，影响乘客身体感受。靠用户主动说"我好热"才调空调太被动——好的体验是系统自己发现不舒适，主动建议调整。

2.2 怎么做的

触发方式：两种——① 事件触发（有人上车、温度变化、VLM 看到有人在阅读→光线可能不够）；② 1 分钟内无新事件 + 有未完成的舒适子目标 → 自动触发一次定时 review。

输入（5 类模板变量注入 SP）：

  • {{attention}}——触发本次推理的事件（如"后排有人上车"）
  • {{watcher}}——VLM 的视觉描述（如"后排有人在看书"）
  • {{car_condition}}——端信号（空调状态、车窗状态、温度等）
  • {{web_information}}——世界信息（时间、天气）
  • {{memory}}——记忆（如"这个用户有鼻炎，怕风直吹"）

SP 内置 5 条 knowhow 推理规则（决策树式 if-then）：温度不适 → 快速调温；空气超标 → 恢复空气质量；人员变动 → 检查空间是否够；有人阅读/找东西/睡觉 → 检查光线是否合适；有人睡觉/听歌 → 检查音量是否合适。

输出：JSON 给 Director，两个字段——"当前的舒适性问题"（分析）+ "应对建议"（方案）。舒适 advisor 不直接控车，只输出建议，Director 决定执行不执行。

夏季 38 度上车：VLM 描述"乘客额头有汗" + 端信号"车内 38 度、空调关闭" → 舒适 advisor 判定"温度不舒适" → 输出建议"开空调制冷 20 度 + 座椅通风 3 档 + 车窗先通风 2 分钟再关" → Director 接收执行。1 分钟后定时 review："车内降到 26 度了但风速还在 7 档" → 建议"降到 3 档" → Director 再调一次。这就是持续追踪。

2.3 做到什么程度

PRD v1.0，核心场景（上车高温/低温、多人分区、睡眠中调节）为 P0 已定义。但结合视觉的高感场景（如"观察到后排擤鼻涕→调小风量"）标 TBD，依赖 VLM 能力。反思机制（离线复盘推理质量）标 P1 待讨论。

三、出行 Advisor——"理解你去哪、为什么去"

3.1 为什么做

用户上了车，还没说去哪。如果系统能主动猜出目的地、推荐路线、顺手帮缴停车费、提前订座，就从"导航工具"升级成"出行管家"。

3.2 怎么做的

覆盖全出行生命周期：行前（目的预测/路线推荐/缴停车费）→ 行中（堵车绕行/POI 推荐/充电/订座/目的地服务）→ 行后（最后一公里指引/停车记位）。

核心推理 = 出行目的预测。靠 10+ 数据源：历史出行偏好（什么时间去什么地方）、车内人员状态（谁坐哪干什么）、上车携带物品（带了书包→上学）、时间上下文（周二早上→通勤）、当前位置、常去地点、导航状态、日程……

PRD 给的推理示例："12月3日周二早上8:31，益红和依依已上车，车在长江小区。依依应在8:30出发去光启小学上学，她带了书包；益红通常8:45出发去平金中心上班。因此本次目的是：1.送依依上学 2.益红上班。" → 推理模式：多因素融合 → 概率排序 → 输出预测+置信度 → 低置信时主动确认。

输出给 Director：出行目的预测 + 当前出行目的 + 需求分析 + 建议。

3.3 做到什么程度

PRD v0.4，成熟度偏低。行前目的预测和缴停车费为核心场景。但生态服务（美团订座/取号）、个性化 POI 推荐、部分充电场景标 P2 未实现。输出格式和与 Director 的对接"需要详细沟通"，还未完全定义。

四、内容 Advisor——"首席内容建议官"，克制比推荐更重要

4.1 为什么做

车里大量时间是沉默的——通勤/巡航/等充电。如果系统能在合适的时机推荐合适的内容（不只是随机放歌，而是根据场景/情绪/偏好精准匹配），就能让车内时光更有价值。但推错了比不推更糟——夫妻吵架时推欢快歌曲、暴雨急弯时推深度播客，都是灾难。所以核心设计哲学是"先判断该不该推，再判断推什么"。

4.2 怎么做的

默认不工作——没有 Trigger 就没有内容。保持用户注意力和系统资源的双重零占用。

触发后进入8步推理 Flow：静默态 → Trigger唤醒 → Context汇聚 → 介入判断（决策核心） → 策略参数生成 → 分发给 Planner → 反馈感知 → Review 回收。

介入判断输出唯一的介入等级（L0-L3），这是核心状态机：

等级定义认知占用什么时候
L0 隐形不推荐不打扰0%愤怒/烦躁；暴雨冰雪驾驶；社交冲突；明确拒绝
L1 氛围白噪声/轻音乐<5%疲惫/低落/发呆；单调安全环境
L2 碎片笑话/脱口秀/新闻30%通勤/短途(导航<30min)；正常情绪
L3 沉浸专家解读/历史讲坛>80%长途巡航/驻车休息(NOA/充电)；好奇/高接收度
原文设计哲学："要像人一样察言观色——车上夫妻正在吵架，即使满足了 Trigger 条件，也不要推送。"

反馈闭环：连续跳过 3 次 → 归因分析（不喜欢歌手还是曲风？）→ 策略修正。路况从畅通变拥堵持续>5min → 自动降认知门槛（L3→L2）。

独特的协同信号：情感 advisor 判断"用户适合陪伴"时，会发信号给内容 advisor 触发推荐。这是 4 个 advisor 之间唯一明确的跨 advisor 协同接口。

4.3 做到什么程度

PRD V0.4。L0-L3 判定逻辑和 8 步 Flow 已完整定义。具体内容源对接（QQ音乐/网易云/喜马拉雅/小宇宙 API）是工程层面的依赖，集成复杂度高。Benchmark 方案"暂不支持"。

五、情感 Advisor——4 个里推理最精细、红线最严格的

5.1 为什么做

车是一个封闭的私密空间，人在车里的情绪波动比其他场景更明显（堵车焦虑、深夜疲惫、被超车愤怒、久别重逢欢喜）。如果系统能察言观色、用合适的语气陪伴，就不只是工具，而是有温度的伙伴。但情感领域"做多了"比"做少了"更危险——过度关心、猜测隐私、擅自推荐、控制设备，都会让用户反感。所以情感 advisor 有最严格的红线。

5.2 怎么做的

情感 advisor 有两套完整 SP——Event Prompt（事件触发用）和 Review Prompt（定时复查用），各自有独立的推理链。

Event Prompt 的 4 步推理链

第一步：事件解析 + 情感影响评估——收到事件（如"副驾上车了"），判断有哪些情感影响。核心是多情感优先级规则：启动车辆方式 > 安全相关 > 节日/特殊日期 > 人物状态 > 情绪 > 人物 > 车内场景 > 车外场景 > 人物+时间 > 人物着装。安全类强制执行，不管其他情感目标怎么说。

第二步：即时建议生成——格式"语气+表情+对象+情感方向+禁止"。6 条红线（绝对禁止）：

  1. 只输出情感建议，不输出具体内容（如"播放一首轻音乐"→ 越界，该内容 advisor 管）
  2. 不控车（如"把空调调低"→ 越界，该舒适 advisor 管）
  3. 不对未发生的事猜测
  4. 不对曾发生的事预测
  5. 不提醒物品（如"别忘了带伞"→ 越界）
  6. 禁止"维持现状"类建议（"建议不做任何事"不算建议）

第三步：子目标生命周期管理——这是一个完整的状态机。两种子目标类型：A 目标达成型（"安抚主驾情绪直到不再愤怒"）和 B 条件响应型（"轻柔语气直到右后排睡醒"）。Create/Update/Delete 操作。子目标上限 10 个。

第四步：输出整合 + 冲突检查——即时建议 vs 子目标冲突 → 以即时建议为准；新增子目标 vs 已有重复 → 合并。

Review Prompt 的 3 步推理链

触发条件："10 分钟没关注用户了 + 有未完成子目标"。三种情况：A 进行中需调整（重度→轻度疲惫）→ 更新 goal；B 触发条件消失 → 恢复建议 + 删除 goal；C 目标已完成 → 仅删除，不输出任何建议（克制！完成了就安静离场）。

输出

{
  "reason": "所有可进行情感建议事件的描述",
  "advice": "用温柔关切的语气，询问主驾是否需要休息",
  "goal_operations": [
    {"action":"create", "fields":{"goal_id":1, "description":"持续关注主驾疲劳状态", "reason":"检测到打哈欠"}}
  ]
}

动作空间：DO_NOTHING / CONVO_STRATEGY（沟通策略给 Planner）/ CONVO_INITIATE（主动搭话）/ EMOJI_SUGGESTION（表情建议）/ TOOL_CALL（self ReACT，如调 car_info 查车况）。

子目标冲突 case——后排儿童睡着（子目标 B"轻柔语气直到睡醒"）+ 主驾分心驾驶（安全事件）→ 按优先级"安全相关 > 人物状态"→ 安全提醒优先，用严肃语气提醒主驾"注意前方"，后排的轻柔子目标暂被安全覆盖。
被超车 case——用户"卧槽" → 事件触发 → 情感 advisor 判定"愤怒/受挫" → 输出：语气=共情吐槽，方向=先认同再安抚 → Director："那个开车也太横了吧。不过你没事就好。"

5.3 做到什么程度

PRD v1.0，两套完整 SP 全文已写好（含大量正/反示例）。事件触发和子目标管理为 P0。但打扰预算自适应（P2）、用户反馈分析（P2）、实时表情陪伴（P2）未实现。主动搭话默认 3 次/天，自适应调整依赖正负反馈积累。

六、动态 Advisor——9 套 Prompt、4 类任务、完整生命周期

6.1 为什么做

4 个静态 advisor 管的是固定四维度（舒适/出行/情感/内容），但"帮我照顾宝宝""一路介绍风景""帮我点杯咖啡送公司"这种临时冒出来的持续任务，静态 advisor 管不了——它需要一个专属的、临时的、做完就销毁的监工。

6.2 怎么做的

创建流程

Step 1：Director 判断需要创建持续任务，调 goal_list_update 把 GoalText 发给 CreateGoal。

Step 2：任务分类模块（LLM 推理）用专门的"任务分类 prompt"把 query 分成 4 类：

优先级判定type例子
1是否调用点单/缴费/预订2（持续/长程）点咖啡送公司
2包含明确时间/周期4（定时）每5分钟播报NBA
3包含条件表达3（条件）副驾上车时开玩笑
4其他1（长时）帮我介绍这台车

输出分类参数：{"type":"1~4", "vlm_use":true/false, "cycle":true/false, "max_execution":-1/1/N}

Step 3：按类型走不同流水线——长时 → 加载 Skill（二期）；持续/GUI → 绑定 GUI Agent；条件 → 解析+注册触发器（5-8s）；定时 → 生成 APScheduler 参数+注册定时器 → 是否需 VLM → 是 → 生成 VLM 提示词。

输出 JSON（长时任务版本，最完整）

{
  "goal": "照顾二排宝宝",
  "goal_status": "active | achieved | terminated_by_user | blocked_by_system",
  "goal_progress": "宝宝安睡中",
  "need_advice": true,
  "advice_timing_confidence": 0.85,
  "goal_achieve_advise": "宝宝醒了在揉眼睛，建议放安抚音乐",

exec_state 状态机（之前没讲过）：normal（正常可输出）→ suppressed（抑制，强制 need_advice=false）→ suppressed_attention_shift（用户注意力偏移导致抑制）→ recoverable（可解除抑制）→ short_circuit（上轮建议有效，跳过本轮）。

销毁规则

max_execution 达上限 → 注销触发器/定时器 → CloseGoal；用户主动删除（任务中心点"取消"）；goal_status 进入 terminated_by_user 或 blocked_by_system；触发器注册失败 → CloseGoal。

二期（DNA-616）关键变化

  • Skill 系统 + Workflow 编排——长时任务从"特调"变"通用"。创建时 load_skill() 匹配 Skill（如"兜风 Skill"）→ 加载 Knowhow → 组成实例。
  • 端侧定时器——弱网/断网也能触发（AI PUSH 不依赖网络），端侧倒计时展示。
  • 策略预生成——条件满足时不再现场推理，提前算好执行策略放云端。"减少一次推理时间"。
  • 导航场景感知——新增"距离某 POI < 指定距离"触发事件。

七、感知订阅——把"帮我盯着"翻译成可执行的触发器

7.1 为什么做

动态 advisor 说的是人话（"每 10s 告诉我宝宝状态"），底层的 VLM/触发器/定时器只认配置参数。感知订阅模块就是把人话翻译成机器配置的翻译官。

7.2 三个子模块

① 动态感知需求推理——"看什么"。如果要看，就调 VLM 的 sp/attention（让视觉模型重点关注特定对象）。不是所有订阅都需视觉——"温度超 20 度"只要条件信号。

② 条件需求推理——"什么条件"。温度>xx / face id "xx 上车" / 车速>xx → 注册触发器持续监控。

③ 定时需求推理——"多久一次"。每 10s / 每 5min / 20min 后（一次性）→ 注册 APScheduler 定时器。

④ 信号组装 + 触发——把三个子模块的输出拼在一起落地，条件/定时满足时唤醒 advisor 并附带最新感知。

7.3 VLM 阈值穿越检测（触发的真实机制）

VLM 不是"看到宝宝就报告"，而是"看到状态变化才报告"。具体流程：

  1. 定时器触发（演示 5s/次，量产 30-60s）
  2. 调图片接口拍照 → 获取图片 URL
  3. 请求 VLM → 只输出一行：true|reason 或 false|reason
  4. 阈值穿越检测：对比上一次(last)和这一次(this)。只有 last==False and this==True 才触发——状态从"未发生"变"发生"的那个瞬间。持续为 True 不重复触发。
  5. 触发后 → 请求 advisor 模型给 advice
"照顾二排宝宝" → VLM query: {"type":"state","query":"二排存在宝宝"}
第 1 轮：false|宝宝没有睡着 → 不触发
第 2 轮：true|宝宝睡着了 → last==F, this==T → 触发！
第 3 轮：true|宝宝还在睡 → last==T, this==T → 不触发（没变化）
只检测从 False 到 True 的上升沿。

7.4 VLM 视觉感知的 3 类策略

类型策略例子
风险监控有异常才说，只说一次，同一问题不重复照顾宝宝（哭了才报）
信息讲解必须有新内容，间隔≥60-90s介绍沿途风景（每次讲新景点）
状态陪伴明显状态信号才触发，冷却≥5min缓解疲劳（打哈欠才提醒）

7.5 触发优先级

订阅事件（最高）> qa 结束 > 定时（最低）。高优先级打断低优的，反之不成立。

八、VLM / Watcher——让文本大脑"看见"车内发生了什么

8.1 为什么做

Director 是文本模型，它看不见图像。VLM 就是它的"眼睛"——把摄像头画面转成自然语言描述，写进 context 让 Director 读。

8.2 怎么做的

承担实体叫 Watcher，端侧部署、自带算力、Always on。三大功能：① 细致理解描述场景 ② 全时多模聆听 ③ 发现值得关注的事件（小孩哭闹/咳嗽/前方大货车/OK 手势）。

VLM 原理：大小 LLM 融合。小模型把图像编码成 token（兜底识别高优标签）；大模型按一段 SP（识别目标）看画面输出自然语言。输入：OMS 摄像头 video 10-15s。输出：结构化流式的"情景描述"文字写回 context。

"影响感知器 sp/attention"就是改 VLM 的 SP——同一个 VLM，拼不同 SP 切换关注对象。比如无订阅时用"既定 SP"通用观察；收到"关注二排宝宝"后拼上新 SP 重点关注后排婴儿。不需要为每个需求各起一个模型。

8.3 做到什么程度

端侧 Watcher Always on 已设计。VLM 视觉持续任务同时仅 1 个。事件发现一期为纯工程触发（字符串匹配），二期升级为多模态模型+工程混合。量产阶段 VLM 定时频率为 30-60s（演示阶段 5s）。

九、四类任务 → 架构承接映射（全篇收口）

任务类型卡片动态 advisor架构怎么承接
简单即时 / 单步复杂无❌Director 当场闭环，不留痕
多步无❌Director 在 ReAct 里串联 tool_feedback
