二、刘杨（触发器研发）
刘
刘杨
云端触发器研发 · 定时引擎 owner
核心关注点：这个功能到底要我做什么？接口边界在哪里？人日怎么给？多机部署和下电保持能不能做？
背景与目标
这个定时任务是不是要新建一个服务？和现有触发器什么关系？
不新建独立服务，是在现有云端触发器里新增一个定时引擎模块。它和条件触发、语义触发共享触发器的调度底座，只是激活条件变成时间到达。这样复用现有基础设施，也避免边界混乱。
任务系统把时间解析做了，触发器只剩一个到点 callback，凭什么占我这么多人日？
触发器侧工作量不只是 callback。具体包括：① 闹钟队列维护（优先队列/时间轮选型）；② 到点精度保证（大量任务下 3 秒内触发）；③ 多机部署一致性（同一个闹钟只响一次）；④ 循环续钟（固定间隔/日历周期）；⑤ 服务重启恢复；⑥ 下电保持。这些才是人日大头。
图 1 定时器任务范围
触发器对外暴露几个接口？传什么字段？
四个核心接口：创建闹钟（任务 ID、下次触发时间、循环规则、过期时间）、修改闹钟（更新下次触发时间或规则）、取消闹钟（按任务 ID 摘钟）、到点回调（触发器回传任务 ID 和触发时间）。回调不带执行动作。
循环任务续钟是触发器自己算，还是任务系统告诉触发器下一次几点？
建议触发器自己算。任务系统创建时把「间隔秒数」或「日历规则」传过来，触发器每次响铃后按规则算出下一次时间并重新排队。这样任务系统不用每次参与续钟，接口更干净。如果你们觉得触发器算日历周期太重，也可以任务系统算好下一次再通知触发器更新，这个可以会上定。
四种时间形态
相对延时和绝对时间对触发器来说有区别吗？
对触发器没有区别，都是「一个具体时间点」。区别在任务系统解析阶段：相对延时 = 当前时间 + 偏移量；绝对时间 = 直接给目标时间。触发器只收最终时间点。
日历周期首期做吗？规则复杂吗？
日历周期放二期。首期只做相对延时、绝对时间、固定间隔。触发器侧先支持按秒级间隔续钟，日历规则二期再扩展。
图 0 业务流程图
图里没画 Guard，到点后到底执不执行？
Guard 在任务系统侧做，不属于触发器职责。触发器到点 callback 后，任务系统先做 Guard，通过才下发执行。 Guard 不通过就跳过，跳过不算失败。这个在业务规则第 3、4 条写了。
第四张图「定时 + 看画面」，触发器要订阅画面流吗？
不需要。触发器只是每 60 秒响一次铃。画面流由任务系统直接订阅 VLM，和触发器无关。触发器只提供采样时钟。
图 2 状态机
状态机里「已关闭」和「已取消」对触发器侧有区别吗？
触发器侧动作一样，都是摘闹钟。区别在业务语义：已取消是用户主动或任务系统主动终止；已关闭是因为车辆下电且任务非跨天。任务系统调用触发器取消接口时可以把原因带过来，便于日志和排查。
循环任务续钟失败怎么办？比如算不出下一次时间。
如果循环规则有问题或达到最大次数，就不续钟，任务结束。如果是临时计算异常，可以按策略重试或回调任务系统处理，这个在待确认 Q5 里要细化重试策略。

三、庆磊（任务系统研发）
庆
庆磊
任务系统研发 · Task Service owner
核心关注点：协议里怎么表达时间规则？生命周期和 Guard 怎么做？上下电策略谁决定？和触发器的接口边界在哪里？
定义与范围
自然语言解析到底谁做？Planner 还是任务系统？
任务系统做。Planner 只识别「这是一个定时任务请求」并带出原始时间表达。任务系统把「十分钟后」「明天 9 点」翻译成结构化规则。
「模糊时间」比如「过一会儿」「待会儿」怎么处理？
任务系统定义兜底策略：无法解析时反问用户，或按默认值处理。这些细节不在触发器侧，不影响 PRD 主逻辑。
四种时间形态
P0 现有协议没有「相对延时 / 固定间隔」字段，你让我怎么序列化？
产品主张：把「时间规则」作为执行策略的扩展字段表达。字段可以包含：type（delay/absolute/interval/cron）、value（秒数或目标时间）、timezone、max_count。具体格式我们三人（王泽民、庆磊、刘杨）会前小会对齐，下周二前出结论。
绝对时间涉及时区吗？车辆跨时区怎么办？
本期先按车辆当前所在时区处理，不单独做跨时区。如果用户说「明天 9 点」，默认按车辆当前时区解析。跨时区作为后续优化项。
图 0 业务流程图
Guard 检查具体检查哪些内容？谁来定义规则？
Guard 规则由产品定义，任务系统实现。例如：关座椅加热前要检查「车辆在线、座椅加热开着、座椅有人」；提醒出发只要检查「车辆在线」。不同任务类型 Guard 条件不同，任务系统按任务类型配置。
跳过不算失败，那任务中心怎么展示？
跳过不上报失败，任务状态保持「等待中」或「已跳过」。任务中心可以展示「本轮条件不满足，已跳过」，但不显示红色失败。
Case 讲解
循环任务的「最大次数」存在哪里？
存在任务系统。触发器只关心「当前响铃后要不要续下一个」。任务系统可以在创建时告诉触发器最大次数，也可以自己计数，达到后通知触发器摘钟。
用户取消后，任务系统要先通知触发器摘钟，再更新卡片。如果摘钟失败怎么办？
摘钟失败要重试，并给用户反馈「取消失败，请重试」。卡片不允许在摘钟成功前消失。这是业务规则第 9 条。
业务规则
P0 第 7 条「满了报错不挤旧」，上限是全局还是按类型？
这是待确认 Q3。产品倾向先按全局上限做，简单可测。如果庆磊觉得应该按类型分，我们可以讨论，但今天最好拍一个。
四、晓伟（Planner + 任务系统产品）
晓
晓伟
Planner + 任务系统产品
核心关注点：Planner 要改什么？任务中心卡片怎么展示？取消链路怎么设计？上线时间能不能匹配？
定义与范围
Planner 要识别哪些表达为定时任务？
只要用户表达里包含「未来执行」或「周期执行」意图，Planner 就识别为定时任务。典型表达：「十分钟后…」「明天 9 点…」「每小时…」「每隔…」。Planner 不需要解析具体时间，只打标签并交给任务系统。
Planner 解析不了的时间，要不要反问？
可以反问，但建议先交给任务系统兜底。任务系统解析失败后再反问，避免 Planner 和任务系统重复判断。
图 1 定时器任务范围
任务中心卡片要展示哪些信息？
至少展示：任务名称、下次触发时间、循环规则、开关状态、取消入口。对于跨天任务，还要展示「跨天保留」提示。
图 0 业务流程图
流程图里用户说完后，卡片什么时候出现？
任务系统创建任务并通知触发器挂钟成功后，任务中心展示卡片。如果挂钟失败，要给用户错误提示，不展示卡片。
Case 5：用户取消
取消链路为什么一定要「后台真停成功后再消卡」？用户体验会不会卡？
这是为了避免「用户以为取消了，后台还在响铃」的更差体验。卡片可以先进入「取消中」状态，给用户反馈；确认成功后再消失。如果失败，卡片保留并提示重试。
语音取消和卡片取消是同一个接口吗？
是的。两个入口最终都转成「取消任务」意图，走同一条链路。区别只在入口形态。

韩杰
端状态
核心关注点：下电事件怎么上报？车辆在线状态怎么定义？本趟内/跨天的默认值谁定？
背景与目标
这个需求要我端状态这边做什么？
纯定时任务不需要端状态参与。只有两类情况需要你：① 下电事件上报，任务系统据此决定任务是否结束；② Guard 里用到车辆在线/座椅占用等车态信号。这些不是新信号，复用现有 Context 机制。
Case 6：上下电
下电事件从哪里上报？信号中枢还是车端直接报？
按现有链路走，车端通过 Context 或信号中枢上报下电事件给云端。任务系统订阅这个事件，决定任务下电策略。不需要为定时任务单独建上报链路。
「车辆下电」和「车辆不在线」是同一个信号吗？
不一定。下电是一个明确事件；不在线可能是网络问题。任务系统主要关心「下电事件」来处理本趟内任务结束。到点时的 Guard 检查车辆是否在线，用于决定是否跳过本轮。
本趟内和跨天的默认值谁定？
产品定义默认值，任务系统写入任务属性。比如车控类（关加热、开按摩）默认本趟内；提醒类（明天 9 点、每工作日开会）默认跨天。会后我会出一张默认值表。

丁彬
VLM 产品
核心关注点：视觉链路在定时 + 看画面里承担什么？为什么不是画面变了直接触发？状态跳变怎么定义？
图 0 第四张图
「定时 + 看画面」这个场景，VLM 要做什么？
VLM 只负责持续输出画面描述，比如「后排宝宝哭了」「后排宝宝没哭」「看不清」。它不参与定时、不参与决策、不参与提醒触发。
画面流已经在持续上报，为什么触发器还要 60 秒采样一次？
触发器不是「还要」采样，它本身就是提供采样时钟。业务上需要按固定节奏判断，避免 VLM 抖动导致频繁触发。同时状态跳变策略要配合这个节奏，防止持续状态重复打扰。
Case 4：看宝宝
状态跳变是谁来判断？
任务系统判断。任务系统维护一个「上次状态」记忆，每次采样时对比最新画面描述：只有「上次没哭 → 这次哭了」才触发提醒。
「看不清」怎么定义？是 VLM 返回空还是返回特定语义？
建议 VLM 返回特定语义，比如「画面不清晰，无法判断」。任务系统收到这个语义后，本轮不更新记忆。具体语义值我们二期联调时对。
这个场景一期做吗？
不做，二期。本期只做纯定时任务，不依赖视觉链路。二期和日历周期一起做。

八、绿坡（测试）
绿
绿坡
测试
核心关注点：怎么测？验收标准是什么？异常场景有哪些？能不能自动化？
验收标准有哪些？能不能量化？
PRD 里列了 V1–V10 验收用例，包括：创建一次性任务 600±3 秒内触发、循环任务做 3 次间隔 60±3 秒、超上限报错、重复创建只更新、下电后本趟内结束/跨天继续等。这些都可以自动化测。
到点精度怎么测？允许误差多少？
PRD 里建议 3 秒内。测试可以构造 1000 个并发任务，统计到点延迟分布。具体精度要求可以根据触发器选型再调，但 3 秒是产品底线。
多机部署场景怎么测「同一个闹钟只响一次」？
需要触发器侧实现分布式锁或分片机制。测试可以模拟多机同时部署，构造到点任务，验证 callback 只发生一次、任务只执行一次。
下电场景怎么测？
构造两类任务：本趟内任务和跨天任务。模拟车辆下电 → 验证本趟内任务结束并摘钟、跨天任务闹钟保留 → 模拟上电 → 验证跨天任务正常触发。
Guard 跳过的场景算通过还是不通过？
算预期行为，不是 BUG。测试用例要单独写「跳过场景」，验证任务状态不变、下次周期继续、不上报失败。
九、讲四个流程图时，各角色会怎么问

讲第一张图（一次性相对延时）时

刘杨： 触发器只 callback 一次，如果 callback 失败怎么办？
触发器侧要有失败重试机制。如果连续重试失败，可以标记任务为异常，由任务系统决定是跳过还是人工处理。重试策略待确认 Q5 细化。
庆磊： Guard 放在任务系统，具体规则谁提供？
产品提供 Guard 规则表，任务系统实现。例如关加热要检查车辆在线、加热开着、座椅有人。
绿坡： 到点时用户刚好下车，这个 case 怎么测？
构造「到点前车已下电」场景，验证触发器 still callback，任务系统 Guard 不通过，任务跳过并结束。

讲第二张图（一次性绝对时间）时

庆磊： 跨天时间解析涉及时区吗？
本期按车辆当前时区处理，不单独做跨时区。
韩杰： 下电期间到点，上电后要补发吗？
不补发。错过就错过，一次性任务直接结束。这是业务规则明确的。
晓伟： 用户说明天 9 点是早上还是晚上？
默认早上 9 点。晚上要说「晚上 9 点」才解析为 21:00。任务系统定义话术兜底。

讲第三张图（循环固定间隔）时

刘杨： 循环续钟是触发器算还是任务系统算？
建议触发器算。任务系统创建时把间隔规则传给触发器，触发器每次响铃后自己续下一个。如果日历规则复杂，可以任务系统算。
庆磊： 最大次数存在哪里？
任务系统维护。触发器只负责续钟，达到次数后任务系统通知触发器摘钟。
绿坡： 循环任务怎么验证它确实停止了？
