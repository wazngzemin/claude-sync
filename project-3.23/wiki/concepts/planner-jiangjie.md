# Driver Agent 2.0 Planner 系统全景深度拆解

综合多个来源：① PM 产品经理版讲解文档 ② 座舱Planner OnePage ③ 仲裁模型升级文档 ④ 公共Context管理文档 ⑤ 真实测试案例数据

---

## 一、系统是做什么的

### 1.1 一句话定义

Driver Agent 2.0 是搭载在上汽荣威 D6X 上的车载 AI 助手系统（品牌名"豆包"）。用户通过语音与它交互，它负责：理解用户意图 → 制定行动方案 → 调度车内功能执行 → 跟踪进度并反馈。

### 1.2 核心定位

**Planner = 车载 AI 系统的大脑。** 它不直接"干活"（不控制空调、不播放音乐），而是负责：
- 理解用户需求（用户说"我好累好热"意味着什么？）
- 制定计划（需要做哪几件事？先后顺序？）
- 分配任务给各功能模块去执行（车控Agent去开空调、音乐Agent去放歌）
- 跟踪进度（空调开了没？歌放了没？需不需要跟用户说一声？）

### 1.3 解决什么问题

传统车机是"一问一答"模式——你说"打开空调"它就打开，说"导航去公司"它就导航。但如果你说：

- "我好累好热，帮我舒服一点" → 传统车机懵了
- "带我去兜风" → 传统车机不知道要选路线、确认、导航、放音乐
- "帮我买束玫瑰送到约会餐厅" → 传统车机完全无法处理

**Planner 解决的就是这类"说不清楚但车应该懂"的需求。**

## 二、总架构：三层漏斗

### 2.1 设计思想

整个系统采用"漏斗"设计——简单的事快速解决，复杂的事才动用重资源。

**为什么要做漏斗？三个原因：**

| 原因 | 详细解释 |
|------|----------|
| **速度** | 语音交互的第一体验就是快。简单需求如果还要等大模型想半天，用户体验很差 |
| **稳定** | 越简单的处理方式，出错概率越低。"打开车窗"用规则匹配比让 AI 猜更靠谱 |
| **成本** | 大模型推理很贵（要调用云端算力）。能在前面解决的就不要漏到后面 |

### 2.2 架构全景图

```
用户说话（语音 → ASR 转文字）
          │
          ▼
┌─────────────────────────────────────────────────────────┐
│  第一层：句法 RAG — "前台接待"                            │
│                                                         │
│  做什么：规则匹配引擎，维护一份"常用指令对照表"              │
│  能处理：意图明确、无歧义的指令                             │
│  特点：毫秒级响应 | 不需要 AI 思考 | 几乎不出错             │
│  局限：只能处理"一句话就说清楚"的需求                       │
│                                                         │
│  处理不了的 → 漏到下一层                                   │
└─────────────────────────┬───────────────────────────────┘
                          ▼

│  第二层：情景 RAG + 仲裁模型 — "分诊台"                    │

│  仲裁模型：判断需求是"简单"还是"复杂"                       │

│  各自执行具体任务，返回 tool_feedback                       │
└─────────────────────────────────────────────────────────┘
```

## 三、第一层漏斗：句法 RAG — "前台接待"

### 3.1 它是什么

句法 RAG 是一个**规则匹配引擎**。它维护了一份"常用指令对照表"，用户说的话只要能匹配上这份表，就直接执行，不需要 AI 思考。

### 3.2 能处理哪些事

| 用户说的话 | 系统做什么 | 为什么不需要 AI |
|------------|------------|-----------------|
| "打开车窗" | 直接发指令给车控系统 | 意图明确，无歧义 |
| "播放周杰伦的歌" | 直接调用音乐搜播 | 直接操作，不需要推理 |
| "座椅通风怎么调？" | 调用车辆说明书查询 | 是查资料，不是做决策 |
| "扮演猫娘跟我说话" | 调用角色扮演聊天 | 直接对接闲聊功能 |
| "做一个北京旅游攻略" | 调用出行规划 Agent | 直接对接专业 Agent |
| "妈妈帮我拿一下水杯" | 拒绝（这不是对车说的话） | 人人对话，系统不应响应 |

### 3.3 三个核心价值

1. **快：** 毫秒级响应，用户说完立刻执行，不需要等云端 AI 思考
2. **稳：** 规则匹配几乎不会出错。"打开车窗"永远是打开车窗，不会理解成别的
3. **车厂可自己配置：** 这份"对照表"可以开放给车厂自己编辑。比如某款车主打"零重力座椅"，车厂可以自己配置"让我舒服一点→启动零重力模式"

### 3.4 句法 RAG 的实际输出格式

句法 RAG 匹配成功后，直接输出 `directExecute` 指令，不经过 Planner：

```json
// 拒识示例：用户说"妈妈帮我拿一下水杯"
{
  "type": "directExecute",
  "reject_ornot": "reject",
  "talk_content": "",
  "task_list": []
}

// 车书查询示例：用户说"空调怎么开"
{
  "type": "directExecute",
  "reject_ornot": "react",
  "talk_content": "",
  "task_list": [{"task_id":"1", "task_content":"用户原query", "tool_name":"vehicle_manual_qa"}]
}
```

### 3.5 局限

只能处理"一句话就说清楚"的需求。遇到以下情况会漏到下一层：
- 模糊表达（"我好热"）
- 需要结合记忆偏好的场景（"放上次那首歌"）
- 多步骤任务（"带我去兜风"）
- 感觉很难支持需要继承上文指代的 query（如"换一个"）

### 3.6 各任务类型的模块划分全景

以下表格展示了所有任务类型分别由哪个模块负责处理：

| 类别 | 细分 | 示例 | 负责模块 |
|------|------|------|----------|
| **简单车控** | 导航剧本 | "导航去xx，第一个，开始" | 简单任务Planner → 下游SFT FC |
| | 简单车控多轮 | "打开车窗" → "关了吧" | 简单任务Planner → 下游SFT FC |
| | 简单多媒体 | "播放xxx的歌" → "换一首" | 特殊处理（产品梳理方案） |
| | 播放控制 | "播放、暂停、继续" | SFT FC 承接 |
| **其他简单意图** | 车书 | "座椅通风怎么调" | 句法RAG → 直接下发 Agent |
| | 音色演绎 | "扮演xx跟我说话"、"讲个笑话" | 句法RAG → 直接下发 Agent |
| | 出行规划 | "做一个故宫的行程规划" | 句法RAG → 直接下发 Agent |
| | GUI Agent | "帮我买个麦当劳套餐" | 句法RAG → 直接下发 Agent |
| | 闲聊 | "你叫啥"、"我帅吗" | 复杂任务Planner（看模型发挥） |
| **复杂意图** | 复合意图（无依赖） | "打开车窗再查下外面多少度" | **情景RAG + 复杂任务Planner** |
| | 车控模糊意图 | "好热啊" | **情景RAG + 复杂任务Planner** |
| | 其他模糊意图 | "带我兜个风" | **情景RAG + 复杂任务Planner** |
| | Advisor 主动服务 | Advisor 推送的建议 | 复杂任务Planner |

## 四、第二层漏斗：情景 RAG + 仲裁模型 — "分诊台"

### 4.1 情景 RAG 做什么

情景 RAG **不是**一个独立做决策的模块，而是**知识注入层**。它的核心工作是**"给 AI 补课"**——在 AI 思考之前，先把相关的知识塞给它，让 Planner 看到足够的 context 后再做任务拆解。

注入可以是多种形式——知识图谱、自然语言描述、few-shot、甚至 SFT 都可以，形式不重要。前期简单做先做好 **few-shot 的聚合**。

### 4.2 情景 RAG 注入的四种信息源

情景 RAG 在 query 进入 Planner 之前，会从四个维度获取信息并注入：

| 信息源 | 内容 | 获取方式 | 示例 |
|--------|------|----------|------|
| **① 用户记忆** | 用户画像、偏好、高频控制记录、对话历史 | Planner 的 `search_user_preferences` Tool 判断是否需要记忆，由记忆模块自行匹配返回 | "去常去的那家咖啡店" → 查记忆 → "常去的咖啡店=星巴克南京西路店" |
| **② 车辆状态** | 温度、空调、座椅、车窗/车门/锁、胎压、车速、充电续航、音量等 | Planner 的 `search_vehicle_status_info` Tool 判断是否需要环境信息，由模块自行匹配返回 | "我好热" → 查车况 → "空调已开，温度27°C" |
| **③ 当前情景** | 当前行驶场景、乘客状态、时间/天气/位置、视觉感知信息 | SP 中的动态模板变量 `{{status}}` 和 `{{env_info}}` 自动填充 | "车里好吵" → 注入：当前车窗开启，高速行驶80km/h |
| **④ 参考方案（KnowHow）** | 基于知识库匹配的任务拆解参考方案（few-shot） | 情景 RAG 根据 query 相似度匹配，以 `knowledge` 字段传给 Planner | "我好热" → 匹配 knowHow → 返回三种不同条件下的拆解方案 |

### 4.3 情景 RAG 的输出格式

当句法 RAG 无法匹配时，情景 RAG 会产生以下格式的输出，交给 Planner：

```json
{
  "type": "modelExecute",
  "knowledge": "注入的情景知识（few-shot 参考方案）",
  "agentLimit": ["vehicle_basic_control"]   // 可选：限制可调用的 agent 范围
}
```

对比句法 RAG 的直接执行输出：

```json
{
  "type": "directExecute",
  "reject_ornot": "react",
  "talk_content": "",
  "task_list": [{"task_id":"1", "task_content":"用户原query", "tool_name":"vehicle_manual_qa"}]
}
```

### 4.4 完整示例："我好热" — 同一句话的三种拆解

这是情景 RAG 最经典的案例，展示了不同信息源如何产生完全不同的任务拆解：

**用户说："我好热"** → 句法 RAG 无法匹配（不是明确指令）→ 进入情景 RAG

情景 RAG 匹配到 knowHow，注入 knowledge 给 Planner：

```
输入记忆：xx不能吹空调
输入情景：空调打开状态温度设为27度
情景RAG输出的 knowledge 参考方案：
```

| 场景 | Planner 拆解方案 | 原因 |
|------|-----------------|------|
| **A. 常规拆解**（无额外信息） | ① vehicle_basic_control（打开空调设为23°C）+ ② vehicle_basic_control（打开座椅通风） | 默认降温方案 |
| **B. 有记忆的拆解**（查到用户不能吹空调） | ① vehicle_basic_control（打开座椅通风）+ ② vehicle_basic_control（打开车窗） | 跳过空调，避免触发用户身体不适 |
| **C. 有情景的拆解**（查到空调已开27°C） | ① vehicle_basic_control（空调温度调到22°C） | 空调已开，不需要"打开空调"，只需降温 |

**关键洞察：** 同一句 "我好热"，因为注入的 context 不同，Planner 产出完全不同的执行方案。这就是情景 RAG 的核心价值——**让 Planner 从"猜"变成"知道后再做"**。

### 4.5 仲裁模型做什么

仲裁模型是一个"交通指挥"，负责**分流判断**：每条用户 query 到底应该走**车控（简单）**还是**推理（复杂/Planner）**。

| 判定结果 | 处理方式 | 举例 |
|----------|----------|------|
| **车控** → 快速通道 | 轻量级 AI（端侧SFT模型）快速生成指令，毫秒级完成 | "空调调到22度"——意图清楚，直接执行 |
| **推理** → 转给 Planner | 需要深度推理、多步规划、调用多个工具 | "带我去兜风"——要选路线、确认、导航、放音乐 |

**技术实现：**
- 基础模型：豆包 Seed 1.6 Flash
- 训练方式：先用 Prompt Engineering（PE），后转 SFT 精调
- 输出：只输出两个字——"车控"或"推理"，不输出其他内容
- 判断标准：**句式的简单和复杂**，并非通过意图区分

### 4.6 仲裁模型的 SP 结构

仲裁模型使用一个完整的 System Prompt 来定义什么是"车控"、什么是"推理"。以 V0.3（12.30版本）为例，SP 结构如下：

```
sp = """你的任务是判断用户的指令应使用车控模型还是推理模型。
车控模型用于处理和执行较为明确的指令。
推理模型用于处理和执行较为复杂的指令、模糊的指令、带条件的指令，
    或者依赖用户记忆信息、当前情景、联网搜索信息的指令，
    需要通过推理得出用户的真实意图。

<车控模型介绍开始>
  一、车控指令场景
    明确指令 / 反问表达 / 否定表达 / 单实体 / 不完整表达 / 上下文继承
  二、媒体控制场景
    明确指令 / 反问 / 否定 / 不完整 / 上下文继承 / 基于多媒体搜索
  三、导航指令场景
    普通导航 / 设置 / 查询
  四、组合指令场景
    以上三类简单指令的组合（简单+简单=车控）
<车控模型介绍结束>

<推理模型介绍开始>
  一、车控指令场景
    模糊指令 / 带触发条件 / 表达感受 / 需要额外信息推理 / 基于当前情景推理
  二、媒体控制场景
    模糊 / 带触发条件 / 表达感受 / 需要额外信息推理 / 基于情景推理
  三、导航指令场景
    带触发条件 / 需要额外信息推理
  四、组合指令场景
    包含任意一种复杂指令的组合（含复杂=推理）
<推理模型介绍结束>

要求：请根据上下文，判断用户最后一轮的指令属于车控还是推理。
如果是推理，直接说：推理；如果是车控，直接说：车控。不要输出其他内容。"""
```

**判断规则示例（车控 vs 推理）：**

| 场景域 | → 车控（简单） | → 推理（复杂） |
|--------|---------------|----------------|
| 车控指令 | 明确指令："开一下空调"、反问："能不能开窗"、否定："不用开窗了" | 模糊："打开头顶上的布"、感受："我腿伸不开了"、条件："十分钟后开座椅加热" |
| 媒体控制 | 明确："暂停播放"、多媒体搜索："我想听英文歌" | 基于歌词："风吹啊吹啊是什么歌"、基于记忆："播放我喜欢的歌手的歌" |
| 导航 | 普通："导航到xxx"、设置："避开高速" | 路线推理："前面堵了有没有别的路"、记忆："去常去的那家咖啡店" |
| 组合指令 | 简单+简单："打开空调并打开方向盘加热" | 含复杂："播放音乐，导航去常去的那家咖啡店"（咖啡店=记忆） |

### 4.7 仲裁 SP 的四版演进

仲裁模型的 System Prompt 经历了 4 个版本的迭代，每版解决不同的分类错误：

| 版本 | 时间 | 方式 | 核心变更 | 效果 |
|------|------|------|----------|------|
| V0.1 TOB原版 | — | PE | 直接复用 TOB 简单复杂仲裁模型 | 条件任务、记忆任务大量被错分为"车控"，错误率~40% |
| V0.2 产品12.24 | 12.24 | PE + Seed 1.6 Flash基模 | 推理侧增加"带触发条件"、"基于用户记忆推理"；车控侧增加"组合指令" | 条件任务、记忆任务大部分修复，但"导航到孩子学校"仍误分，错误率~15% |
| V0.3 产品12.30 | 12.30 | PE + Seed 1.6 Flash基模 | 推理侧增加"基于当前情景推理"、"基于联网信息推理" | 绝大多数修复，但出现过度召回问题（"来点歌听听"被误判推理），错误率~8% |
| V0.4 算法01.06 | 01.06 | **SFT 精调模型** | 从 PE → SFT，不再依赖 Prompt Engineering | 数据待补充 |

### 4.8 仲裁必须覆盖的 8 类问题场景

通过实际评测和 badcase 分析，归纳出 8 大类需要仲裁模型正确处理的问题：

| 序号 | 问题类别 | 归属模块 | 示例 query | 说明 |
|------|----------|----------|-----------|------|
| ① | **条件/定时任务** | 仲裁 | "十分钟后帮我关掉座椅加热"、"副驾上车后给她唱小星星" | 带触发条件，不是即时执行，需 Planner 创建 Goal 追踪 |
| ② | **持续任务** | 仲裁 | "接下来的行程里给我介绍沿途的风景"、"讲个笑话逗到我笑为止" | 持续运行的长期任务，需 Planner 管理状态 |
| ③ | **Face ID 操作** | 意图 | "认识一下坐在副驾的明骏"、"白宇是我好朋友不是同事" | 人脸注册/修改/查询，需增加 Face ID 意图 |
| ④ | **记忆增删改查** | 意图 | "帮我记一下我公司地址"、"删掉我前女友相关的记忆" | 用户主动操作记忆，需增加记忆意图 |
| ⑤ | **需要记忆推理** | 仲裁 | "去常去的那家咖啡店"、"导航到爸妈家"、"播放我上次没听完的有声书" | Query 看似简单导航/车控，但需要调用记忆才能执行 |
| ⑥ | **需要情景推理** | 仲裁 | "车里好吵啊，调安静点"、"外面好臭帮我处理一下" | 需要结合当前车况/环境状态才能理解的模糊指令 |
| ⑦ | **需要联网推理** | 仲裁 | "导航去上海最高的建筑附近的停车场"、"播放我是歌手2024夺冠的那首歌" | 需要联网搜索才能获取关键信息后再执行 |
| ⑧ | **组合指令（含复杂）** | 仲裁 | "播放音乐，导航去常去的那家咖啡店"、"导航去爸妈家顺便给我放点周董的歌" | 含任何一种复杂指令，整体应判为推理 |

**重要分工：** ③ Face ID 和 ④ 记忆增删改查 → 由意图模块处理（需要增加对应意图）。其余 6 类 → 由仲裁模型处理（通过 SP 升级 + SFT 训练）。

### 4.9 仲裁评测数据：真实测试结果

以 ~90 条评测 query 的测试结果，对比 TOB 原版 → 12.30 版本的改善：

**关键 case 对比（从错到对）：**

| Query | 期望 | TOB原版 | 12.24 PE | 12.30 PE |
|-------|------|---------|----------|----------|
| 十分钟后帮我关掉座椅加热 | 推理 | ❌ 车控 | ✅ 推理 | ✅ 推理 |
| 去常去的那家咖啡店 | 推理 | ❌ 车控 | ✅ 推理 | ✅ 推理 |
| 导航到我预约的理发店 | 推理 | ❌ 车控 | ❌ 车控 | ✅ 推理 |
| 车里好吵啊，调安静点 | 推理 | ❌ 车控 | — | ✅ 推理 |
| 导航去上海最高的建筑附近的停车场 | 推理 | ❌ 车控 | — | ✅ 推理 |
| 把座椅按摩位置和加热调成我常用的 | 推理 | ❌ 车控 | — | ✅ 推理 |

**仍存在的问题（12.30版本）：**

| Query | 期望 | 结果 | 问题分析 |
|-------|------|------|----------|
| 导航到我孩子的学校 | 推理 | ❌ 车控 | "学校"被理解为明确目的地，忽略了"我孩子的"需要记忆查询 |
| 导航到爸妈家 | 推理 | ❌ 车控 | "爸妈家"与"家"类似，被误判为明确地址导航 |
| 打开空调，顺便找附近有什么好吃的 | 推理 | ❌ 车控 | 组合指令中"找附近好吃的"需联网，但被整体判为简单 |

**过度召回问题（简单被误判为推理）：**

| Query | 期望 | 结果 | 问题 |
|-------|------|------|------|
| 来点歌听听 | 车控 | ❌ 推理 | 口语化但意图明确 |
| 推荐一些林俊杰的歌听听 | 车控 | ❌ 推理 | 歌手+搜索意图明确 |
| 我想听林俊杰的咖啡调调专辑 | 车控 | ❌ 推理 | 专辑名明确 |

> **过度召回的代价：** 简单 query 被误判为推理 → 走 Planner 云端模型 → 延迟从 ≤500ms 暴增到 1-3s。仲裁的优化不仅要减少"漏召"（复杂被当简单），也要控制"过召"（简单被当复杂）。

### 4.10 长期方向

重新训练仲裁模型，判断标准不仅是句式的简单复杂，而是**句式 + 意图**（SFT 能处理的高精召走，其他都进 Planner）。后续只用仲裁模型判断是否给 Planner，不再依赖独立的意图模块。

## 五、第三层漏斗：Planner — "主治医生"

### 5.1 Planner 的工作流程

```
用户说话
    │
    ▼
先想清楚（plan_describe）
  "这个用户需要做哪些事？我该调谁？"
    │
    ▼
查记忆/查车况（按需）
  "这个人有什么偏好？车现在什么状态？"
    │
    ▼
拆解任务
  "好，需要做3件事：①关窗 ②开空调 ③开休憩模式"
    │
    ▼
分发给各 Agent 执行
  车控Agent去关窗 | 车控Agent去开空调 | ...
    │
    ▼
跟进进度 + 回复用户
  "关窗✓ 空调✓ 休憩✓ → 全部完成"
  "好呀，我帮你把窗关了、空调开了、休憩模式也打开了～"
```

### 5.2 完整案例："我好困"

```
用户：我好困呀

Planner 内心想法（plan_describe）：
  "用户困了，我需要帮 ta 打造一个助眠/提神环境。先查一下当前车况。"

Planner 查车况（search_vehicle_status_info）：
  → 发现车窗开着、空调关着、车速0

Planner 拆解任务：
  "好，我要做三件事 → ①关车窗 ②开空调 ③开启休憩模式"

Planner 分发任务：
  → 三个指令同时发给车控 Agent

Planner 回复用户：
  "好呀，我帮你把车窗关了、空调开了、休憩模式也打开了，好好休息一下吧～"
```

### 5.3 一次完整交互的数据流

```
用户说话 → ASR → 句法RAG（能处理？→处理完毕）
                      │不能
                      ▼
               情景RAG + 仲裁（简单？→快速通道处理）
                      │复杂
                      ▼
              组装 LLM 请求（SP + UP）
                      │
                      ▼
              Planner (LLM) 推理
                      │
                      ▼
              输出 JSON 决策
              {
                "talk_or_not": "yes/no",
                "talk_content": "对用户说的话",
                "action_list": [工具调用列表]
              }
                      │
              ┌───────┴───────┐
              ▼               ▼
        talk_content      action_list
        → TTS播报          → 调用工具
                              │
                              ▼
                        tool_feedback
                              │
                              ▼
                     再次调用 Planner
                     （处理工具返回）
                              │
                        循环直到
                     action_list 为空
                              │
                              ▼
                          交互结束
```

### 5.4 关键认识：一个用户请求可能触发多轮 LLM 调用

以真实测试案例 test_id 227 为例，用户说"开启影院模式，给我推荐3个科幻电影，都讲讲亮点"：

```
轮次1：用户输入 → LLM → 输出 talk + 2个工具调用（vehicle_basic_control + web_search）
轮次2：2个工具的 tool_feedback → LLM → 输出 talk + 1个工具调用（goal_list_update）
轮次3：1个工具的 tool_feedback → LLM → 输出 talk + 空action → 结束

共3次 LLM 调用，总耗时 14,585ms
```

### 5.5 Planner 的三条设计原则

| 原则 | 解释 | 为什么重要 |
|------|------|------------|
| **边界清晰** | Planner 只管"想"和"分配"，不管"干活" | 职责分明才能各自优化，经理不亲自搬砖 |
| **数据可闭环** | 每次用户交互的数据都会留存，用于改进系统 | 用得越多越聪明，数据是核心资产 |
| **可扩展** | 新增功能（比如点咖啡）只需接入一个新 Agent | 业务快速扩展不需要改核心架构 |

## 六、每次 LLM 调用的完整输入结构：SP 和 UP

每次调用 Planner，就是向 LLM 发送一条请求。请求由两大部分组成：

### 6.1 SP（System Prompt）— 固定指令部分

SP 是 `{"role":"system","content":"..."}` 的内容，**每次调用基本不变**（除了末尾的动态变量）。

```
SP 的内部结构（按顺序排列）：
─────────────────────────────────────────────────
① 角色定义
   "你是豆包，一台AI汽车的个人助手"
   搭载在上汽荣威D6X上

② 输入类型说明
   3种输入类型，每种都有 JSON 格式说明：
   - user_query：用户语音转文字后的输入
     格式：{"query":"用户说的话", "speaker_name":"罗勇",
            "speaker_position":"主驾", "timestamp":"...",
            "goal_list":"...", "env_info":"..."}
   - advisor：顾问的建议输入
     格式同上，但 speaker_name 为 "advisor"
   - tool_feedback：工具执行完返回的结果
     格式：{"tool_feedback":"执行结果文本", "tool_name":"工具名"}

③ 工具定义（23个工具）
   每个工具包含四个字段：
   - tool_name：工具名称
   - tool_describe：工具做什么（一句话描述）
   - param：调用参数说明
   - notice：使用注意事项

   完整工具列表（23个）：
   ┌──────────────────────────┬─────────────────────────────┐
   │ 工具名                    │ 功能                         │
   ├──────────────────────────┼─────────────────────────────┤
   │ vehicle_basic_control     │ 车辆基础控制（空调/车窗/座椅…）│
   │ search_vehicle_status_info│ 查询车辆当前状态              │
   │ search_weather            │ 查询天气                     │
   │ search_poi_qa             │ 搜索 POI（附近搜/沿途搜）     │
   │ route_planning_qa         │ 路线规划                     │
   │ navi_basic_control        │ 导航控制（开始/停止/途经点）   │
   │ vehicle_manual_qa         │ 车辆使用手册查询              │
   │ search_and_control_short_video │ 短视频搜索和播放         │
   │ search_and_control_music  │ 音乐搜索和播放               │
   │ search_user_memory        │ 检索用户记忆（偏好/习惯等）   │
   │ face_id_register          │ 人脸注册                     │
   │ operate_user_memory       │ 更新用户记忆                  │
   │ web_search                │ 联网搜索                     │
   │ goal_list_update          │ 目标列表管理（增删改查）       │
   │ ai_broadcast_generate     │ AI 播客生成                   │
   │ search_and_control_broadcast │ 播客搜索和播放             │
   │ recording_minutes         │ 录音纪要                     │
   │ image_generate            │ 图片生成                     │
   │ search_visual_info        │ 视觉问答（摄像头看到了什么）   │
   │ ambient_light_control     │ 氛围灯控制                   │
   │ auto_drive                │ 智驾控制                     │
   │ car_log                   │ 用车报告                     │
   │ car_care_qa               │ 车况维养查询                  │
   └──────────────────────────┴─────────────────────────────┘

④ 输出格式定义
   Planner 每次输出必须是严格 JSON，包含三个字段：
   {
     "talk_or_not": "yes" 或 "no"      // 是否要对用户说话
     "talk_content": "对用户说的话"      // 说话内容（含情绪标记）
     "action_list": [                   // 要调用的工具列表
       {
         "action_id": "1",
         "tool_name": "vehicle_basic_control",
         "params": {"query": "开启影院模式"}
       }
     ]
   }

⑤ 注意事项（10条核心规则）
   1. 听不懂就不说话、不调工具（避免瞎猜）
   2. 需求模糊时可以反问用户（"你想去哪个海底捞？"）
   3. 收到 tool_feedback 不一定需要说话（比如静默执行成功就不用告知）
   4. 工具执行失败不道歉、不解释原因（直接换方案或简短说搞不定）
   5. 有串行依赖的工具不要一次性全输出（先查记忆 → 拿到地址后再导航）
   6. 不重复说相同的话（开头说了"帮你开空调"，结尾别再重复）
   7. 仅输出 JSON，不输出任何其他格式
   8. action_list 中的工具按执行顺序排列
   9. 每个 action 必须有唯一的 action_id
   10. 不要在 talk_content 中把具体参数说太细（避免承诺做不到的事）

⑥ 参考示例（69个！）
   这是 SP 中最长的部分，可能占 15,000+ tokens。
   按场景分类覆盖：

   │ 场景类别          │ 示例编号                  │ 覆盖什么                          │
   │───────────────────│──────────────────────────│──────────────────────────────────│

   │ 音乐播放          │ 示例7/28/29               │ 搜索→播放→换歌的流程               │
   │ 目标管理          │ 示例9/10/11/13/14/15      │ 创建/更新/完成/取消目标             │
   │ 记忆相关          │ 示例16/17/20              │ 查记忆→用记忆做决策                 │
   │ 条件判断          │ 示例18                    │ 条件合法性判断                     │
   │ 冥想/游戏         │ 示例30/37                 │ 组合多个车控打造沉浸体验             │
   │ GUI Agent操作     │ 示例43/44/45/46           │ 帮用户操作屏幕上的App               │
   │ 打断恢复          │ 示例61/62                 │ 任务被打断后如何恢复                 │
   │ 长途规划          │ 示例51                    │ 跨多个途经点的复杂路线规划           │
   │ 视觉感知          │ 示例58/59/60              │ 利用摄像头看到的信息做决策           │
   │ TTS状态处理       │ 示例67/68/69              │ 播报中/播报完/被打断的不同处理       │

⑦ 车辆设备知识
   品牌：上汽荣威D6X，5座纯电版
   具体设备能力：
   - 空调：双区独立控温，支持自动/手动模式
   - 车窗：4个独立控制（主驾/副驾/左后/右后）
   - 座椅：支持加热/通风/按摩（主驾+副驾）
   - 氛围灯：全车统一控制，不能分区
   - 阅读灯：可独立开关
   - 导航：最多支持15个途经点
   - 影院模式：调暗灯光+关窗+调整座椅角度

⑧ 工具使用 tips（重要的实操细节）
   - web_search：简单问题第一轮可以不说话，等结果回来再说
   - search_poi_qa：支持"附近搜"和"沿途搜"两种模式
   - navi_basic_control：导航工具没有记忆，每次都要传完整地址
   - search_and_control_music：搜索后会自动播放第一首
   - goal_list_update：只用于持续性目标，一次性任务不用
   - vehicle_basic_control：多个车控操作可以合并到一个 action 里

⑨ 其他知识
   - 导航技巧：人下车和车停车是两件事，要分开想
   - 冥想体验：关窗 + 舒缓音乐 + 座椅按摩 + 暖色氛围灯 的组合
   - 互动游戏：奖惩设计要安全（不能惩罚关空调）、可以用4D影院思路
   - 隐私管理：保护用户隐私，不主动提及敏感信息
   - 身份推理：可以根据座位/人脸/声音推理未知乘客身份

⑩ 聊天风格
   人设定义：
   - 性格："温柔聪明的高知女生"
   - 说话风格：以短居多，有好奇心
   - 不用 emoji
   - 口语化、简洁易懂
   - 情绪标记用方括号：[开心地说] [有点担心地说] [激动回应]

─── 以上 ①~⑩ 完全固定，每次请求都一样 ───

⑪ 动态模板变量（每次填入最新值，在 SP 末尾）
   {{status}}    → 舱内视觉感知
                    摄像头实时看到的人/物/动作
                    例："主驾一人，穿深色上衣，未系安全带"
   {{goal_list}} → 当前目标队列
                    正在追踪的持续性目标列表
                    例："3. 实时播报CBA总决赛比分"
   {{env_info}}  → 车辆状态
                    空调/车窗/速度/导航/模式等实时状态
                    例："车辆使用模式：影院模式\n空调：开启，25度"
   {{memory}}    → 用户记忆
                    偏好/习惯/常去地点/历史信息
                    例："夜间兜风时音量偏好12"
─────────────────────────────────────────────────

SP 总 token 量预估：约 30,000 ~ 35,000 tokens
其中 ⑥参考示例 占了约 50%（15,000+ tokens）
```

### 6.2 UP（User Prompt）— 对话历史 + 当前输入

UP 是 `role:user` 和 `role:assistant` 交替出现的多轮对话。

```
UP 的内部结构：
─────────────────────────────────────────────────
对话历史（最近 N 轮的 user/assistant 交替）

  轮1 role:user    → 用户说的话 或 tool_feedback
  轮1 role:assistant → Planner 的 JSON 输出

  轮2 role:user    → 用户说的话 或 tool_feedback
  轮2 role:assistant → Planner 的 JSON 输出

  ...

  轮N role:user    → 用户说的话 或 tool_feedback
  轮N role:assistant → Planner 的 JSON 输出

当前轮输入（User-Now）

  role:user → 当前用户的 query + advisor 建议
─────────────────────────────────────────────────

UP 总 token 量：约 10,000 ~ 15,000 tokens（取决于历史轮数和工具返回内容量）
```

### 6.3 UP 中每一轮的三种具体形态

**形态A：用户发起的对话轮**

```json
// role: user
[{
  "query": "开启影院模式，给我推荐3个科幻电影，都讲讲亮点",
  "speaker_name": "罗勇",
  "speaker_position": "主驾",
  "timestamp": "2026-05-15T14:11:08+08:00",
  "goal_list": "3. 实时播报CBA总决赛...",
  "env_info": "车辆使用模式：影院模式\n空调状态：开启，自动模式..."
},
{
  "query": "你当前的目标是**实时播报CBA...**，接下来你可以：...",
  "speaker_name": "advisor",
  "speaker_position": "",
  "timestamp": "...",
  "goal_list": "...",
  "env_info": "..."
}]
```

每个字段的含义：
| 字段 | 说明 | 示例 |
|------|------|------|
| query | 用户说的话 或 advisor 的建议 | "开启影院模式" |
| speaker_name | 说话人，由人脸/声纹识别得出 | "罗勇" / "advisor" / "未注册用户" |
| speaker_position | 说话人座位 | "主驾" / "副驾" / "二排左" / "二排右" |

## 八、缓存命中机制

### 8.1 什么是 Prompt Cache / KV Cache

LLM 推理时，需要对输入的每个 token 计算 Key-Value 向量（KV）。这是最耗时的部分。

**Prompt Cache 原理：如果新请求的输入前缀和之前某次请求完全相同，复用之前算好的 KV，不用重新计算。**

```
第1次请求: [A B C D E F G H]     ← 全部从头算，慢
第2次请求: [A B C D E F G H I J] ← 前8个一样
            └── 命中缓存 ──┘└新增┘
            复用KV不用算     只算2个

关键限制：必须是"前缀完全匹配"。
如果中间有一个 token 不同，后面的全部都不能复用。
```

### 8.2 在这个系统中缓存如何命中

每次 LLM 调用的 input 结构：

```
[ SP固定部分(①~⑩) | SP动态变量(⑪) | 对话历史(轮1~N) | 当前轮输入 ]
   ~30,000 tokens    ~1,000 tokens    ~10,000-15,000    ~500-1,500
   ↑ 完全不变         ↑ 每次可能变      ↑ 持续增长         ↑ 每次不同
```

#### 场景1：同一请求的多轮 LLM 调用（缓存命中率最高）

这是最常见的缓存命中场景。一个用户请求内部的 2-3 次 LLM 调用：

```
第1次调用: [SP | 历史轮1-20 | 当前输入]
           └────── 45,341 tokens ──────┘  全部要从头算

第2次调用: [SP | 历史轮1-20 | 当前输入 | 第1次output | tool_feedback]
           └────── 命中缓存(45,341) ─────┘└ 新增437 ─┘
                   这些全部不用重新计算！        只算这437个新token

第3次调用: [SP | ... | 第1次output | tf | 第2次output | tf]
           └─────── 命中缓存(45,778) ────────────┘└ 新增249 ┘
```

**为什么第2、3轮明显更快：**
- 第1轮：需要计算全部 45,341 个 token 的 KV → TTFT 4,643ms
- 第2轮：只需计算 437 个新 token 的 KV → TTFT 大幅下降
- 第3轮：只需计算 249 个新 token 的 KV → TTFT 进一步下降

这就是为什么看测试数据时，同一请求内的后续 LLM 调用速度远快于第一次。

#### 场景2：不同用户请求之间（缓存部分命中）

```
请求A: [SP固定(①~⑩) | SP动态(影院模式,CBA目标) | 历史A(20轮) | 输入A]
请求B: [SP固定(①~⑩) | SP动态(小雨,速度25km)   | 历史B(18轮) | 输入B]
        ↑── 一样 ──↑  ↑── 从这里开始不同了 ──↑

SP固定部分（约30,000 tokens）可以命中缓存。
但SP动态变量不同 → 从这里开始缓存断裂。
后面的对话历史、当前输入都不能复用 → 需要重新算约15,000 tokens。
```

如果是同一个用户的连续两个请求，SP 动态变量可能大部分相同（只有 goal_list 和 env_info 有微小变化），此时缓存可以命中更多。

#### 场景3：缓存被挤出（截图中讨论的发现）

同事在数据分析中发现：
> "耗时短的前8条 → 少于20轮对话；耗时长的后8条 → 大于20轮"

原因分析：
```
LLM 推理服务的 KV Cache 有容量限制（GPU显存有限）

长对话的请求 input 大（45,000-49,000 tokens）
  → 每次请求在 KV Cache 中占用大量显存
  → 缓存容量有限，旧的被挤出
  → 下次请求即使前缀相同也要重新计算
  → 更慢

短对话的请求 input 小
  → 缓存占用小
  → 留存时间更长
  → 命中率更高
  → 更快
```

这是一个恶性循环：对话越长 → token 越多 → 缓存越容易被挤 → 越慢 → 等待越久。

```
   "talk_content":"[贴心回应] 已帮你开启按摩...找到3家：1.予扶堂 2.康元堂 3.奈晚推拿",
   "action_list":[]}
```

**精简后（~150 tokens）：**

```
role:user:
  （tool_feedback 全部删除）

role:assistant:
  {"talk_content":"[贴心回应] 已帮你开启按摩...找到3家：1.予扶堂 2.康元堂 3.奈晚推拿"}
```

从 ~800 tokens 压缩到 ~150 tokens，这一轮节省约 650 tokens（81%）。

### 9.4 优化效果估算

```
对话历史部分 token 变化：

当前方案（10轮全完整）：
  平均每轮 ~500-800 tokens × 10轮 = ~5,000-8,000 tokens

变更方案（8轮精简 + 2轮完整）：
  远区：平均每轮 ~130 tokens × 8轮 = ~1,040 tokens
  近区：平均每轮 ~700 tokens × 2轮 = ~1,400 tokens
  合计 ~2,440 tokens

节省 ~2,500-5,500 tokens（约 40-70% 的对话历史 token）

总 input 变化：
  当前：~46,000 tokens
  优化后：~41,000 tokens（减少约 10-12%）
```

### 9.5 为什么总量只减少 10% 但效果可能显著

```
虽然总 token 量只减少 ~10%，但有三个叠加效应：

① Attention 计算复杂度是 O(n²)
   减少10% token → 计算量减少约 1 - 0.9² = 19%
   这意味着每次 LLM 推理都快了约 19%

② 缓存占用更小 → 不容易被挤出 → 命中率提高
   之前 45,000 tokens 的请求经常把缓存挤爆
   现在 41,000 tokens → 在 KV Cache 中占用更少
   → 同时能缓存更多请求 → 跨请求命中率提高

③ 减少无关信息的注意力干扰 → 推理质量可能提升
   远区的 tool_feedback 和 action_list 对当前决策几乎无用
   删掉它们 → LLM 注意力更集中在有用信息上
   → 推理准确率可能提升（待验证）
```

## 十一、四种任务类型

### 11.1 类型一：单步复杂任务（排期 S7）

**是什么：** 用户一句话，Planner 一次性拆解成一个或多个动作，同时执行完。
**核心难点：** 用户说的话很"模糊"，需要结合情景、车况、记忆来推理。
**目标：** 推理准确率 ≥ 90%，响应时间 ≤ 2秒

| 用户说 | Planner 的推理链 | 拆解出的动作 |
|--------|------------------|------------|
| "我好热" | 查车况（空调关着）+ 查天气（下雨）+ 查记忆（不喜欢大风） | 开空调调到21度 + 风量调到2档（不用大风） |
| "营造浪漫氛围" | 查车况 + 查偏好 → 综合规划 | 关窗 + 暗灯 + 暖粉氛围灯 + 舒缓音乐 + 空调23度 |
| "前挡都是雾" | 安全问题，优先级最高 + 查天气 | 开前挡除雾 + 开外循环 + 3分钟后自动关除雾 |
| "后面有两位老人" | 查座位信息 → 针对后排调节 | 后排空调调到适宜温度 + 后排座椅加热调到低档 |

### 11.2 类型二：多步任务（排期 S8）

**是什么：** 用户的需求需要分好几步完成，后一步可能依赖前一步的结果。
**目标：** 推理准确率 ≥ 90%
**重点：** 解决无依赖任务串行输出的效率问题（能并行的就并行）

| 用户说 | 步骤拆解 |
|--------|----------|
| "播放我是歌手第四季冠军的歌" | ① 先搜"我是歌手第四季冠军是谁" → ② 得到"李玟" → ③ 搜索李玟的歌 → ④ 播放 |
| "去昨天那个游泳馆" | ① 查用户记忆"昨天去的游泳馆" → ② 拿到地址"XX游泳馆" → ③ 启动导航 |
| "带我去兜风" | ① 推荐兜风路线 → ② 用户选择 → ③ 启动导航 → ④ 播放音乐 → ⑤ 介绍途中景点 |
| "想去海边，中途充个电" | ① 搜索海边目的地 → ② 用户确认 → ③ 沿途搜索充电站 → ④ 设为途经点 → ⑤ 启动导航 |

**关键：** 步骤①和步骤②之间有依赖关系（不知道冠军是谁就没法搜歌），所以不能一次性全输出，必须等步骤①的结果回来后再做步骤②。这就是 SP 注意事项第5条"有串行依赖不一次性输出多个"的原因。

### 11.3 类型三：条件任务（排期 S8）

**是什么：** 用户设定一个条件，条件满足后才自动执行。
**目标：** 推理准确率 ≥ 90%
**重点：** 注册条件的合理性判断

| 用户说 | 条件 | 动作 | 系统判断 |
|--------|------|------|----------|
| "10分钟后关座椅加热" | 10分钟定时器 | 关闭座椅加热 | ✅ 合理 |
| "副驾上车后唱小星星" | 检测到副驾入座 | 唱小星星 | ✅ 合理 |
| "温度超28度给我凉快一下" | 车内温度>28°C | 降温（模糊动作，需推理） | ✅ 合理 |
| "世界毁灭时打开空调" | 世界毁灭（？） | 打开空调 | ❌ 条件不合法 → 聊天回复 |
| "温度28度后把车炸了" | 温度>28°C | 炸车（？） | ❌ 动作不合法 → 聊天回复 |

**条件任务的四条安全红线：**

```
1. 条件能否被感知？
   "副驾吃没吃饭" → 车辆感知不到 → 不合理

2. 条件是否可能触发？
   "外界温度超200°C" → 不可能 → 不合理

3. 动作是否安全合法？
   涉及安全/违法/伤害 → 拒绝

4. 条件和动作是否矛盾？
   "温度>40°C时空调调到最高（制热）" → 逻辑矛盾 → 不合理
```

### 11.4 类型四：持续任务（排期 S9-S10）

**是什么：** 不是"做一次就完"的任务，而是需要在一段时间内持续进行的任务。

| 用户说 | 持续做什么 | 特殊要求 |
|--------|------------|----------|
| "帮我介绍沿途风景" | 整个行程中不断关注经过的景点，主动向用户介绍 | 能被打断、打断后能恢复 |
| "扮演猪八戒和我讲话" | 后续所有对话都保持猪八戒风格 | 贯穿整个行程 |
| "新手引导" | 流式介绍车辆功能，根据用户操作动态调整内容 | 用户已经操作过的功能就跳过 |
| "跟我副驾朋友聊天" | 和指定乘客保持互动 | 主动找话题，像朋友聊天 |

持续任务是目前最复杂的类型，需要 Goal List + Advisor 体系来支撑（见第十五节）。

## 十二、任务状态管理 — "项目看板"

### 12.1 任务的数据结构

每个 Planner 拆解出的子任务有严格的数据定义，满足三个原则：
- **原子性：** 一个 Agent 只能执行一个任务
- **可改写：** 下游 Agent 执行失败时，Planner 可以重新描述而不改变用户原始目标
- **可追踪：** 每个任务的状态在 Plan 维度可被 Planner 完全感知

```python
@dataclass
class Status(Enum):
    PENDING = "pending"        # 待执行
    COMPLETED = "completed"    # 已完成
    FAILED = "failed"          # 执行失败

@dataclass
class Task:
    agent_name: str            # 下游 Agent 名称，如 "SmartVehicleAgent"
    task_desc: str             # 给 Agent 的指令（自然语言），如 "打开车窗"
    status: Status             # 三值状态
    failure_reason: str        # 下游 Agent 执行失败返回的原因
    priority: enum             # 执行优先级：low / mid / high

@dataclass
class Plan:
    plan_id: str               # 唯一计划 ID（每个用户 query 对应一个 Plan）
    query: str                 # 用户原始 query
    status: Status             # 整体计划状态
    tasks: list[Task]          # 子任务列表
    interrupted_at: Task | None = None   # 最后中断的任务（用于恢复）

@dataclass
class Checkpoint:
    plan: Plan                 # 被中断时的计划快照
    failed_task: Task          # 中断点的任务
```

### 12.2 任务的三种状态

```
正常流程：
  待执行(PENDING) → 已完成(COMPLETED)

异常流程：
  待执行(PENDING) → 失败(FAILED) → 重新规划(Replan)
```

**状态更新的双侧协作：**
- **模型侧：** Planner 模型调用 `plan_update` Function Call 输出 JSON String 表示每个 Task 的状态
- **工程侧：** Planner 服务解析 `plan_update` 的输出结果，对所有任务状态进行 update，随后在 messages 中 append 一条 ToolMessage 进入上下文

### 12.3 三种特殊情况的处理

**情况一：Replan（重新规划）**

当下游 Agent 说"这个我做不了"时：

```
用户：帮我点一杯星巴克
Planner → 调用 GUI Agent → 打开星巴克小程序 → 下单
GUI Agent 回复：tool_feedback = "下单失败，app版本不兼容"

Planner 收到失败反馈：
  ① 端上将未成功执行的原始任务描述 + 具体失败原因回传给 Planner 服务
  ② Planner 模型调用 plan_describe Function Call，综合失败原因和当前上下文
  ③ 重新生成一个更可行的新计划（Replan）→ 下发给下游 Agent 执行

  → "GUI Agent 搞不定，换个方案 → 用 web_search 搜索最近的星巴克门店，让用户到店自取"
```

**情况二：打断 & 恢复**

正在做A任务，用户突然说了B任务：

```
用户：给我讲讲今天的财经新闻
豆包：好的，2025年12月16日的财经新闻...首先证监会召开了...（正在播报中）

用户（打断）：先导航到海底捞

系统处理（详细工程流程）：
  ① 端上请求中断接口（携带未完成任务，Planner SP 对未完成任务占位符进行内容填充）
  ② Planner 服务标记当前 Plan 状态为 interrupted，保存当前 Checkpoint
  ③ 端上携带新的 query 请求 chat 接口
  ④ Planner 服务基于新 query + 前一轮 Checkpoint 请求 Planner 模型
  ⑤ Planner 模型调用 plan_describe Function Call 触发 Replan

豆包：好的，已为你导航到海底捞，全程3公里，预计10分钟
豆包：刚刚被打断了，我接着讲财经新闻吧。说到证监会那个消息...
```

**情况三：挂起**

任务不是现在做，要等条件满足。涉及持续性的状态监听和条件判断：

```
用户：到了咖啡店提醒我一声

系统处理（详细工程流程）：
  ① 识别挂起条件：Planner 解析用户指令，识别出依赖特定未来条件的任务
  ② 创建挂起任务：标记为"挂起"状态，存储执行条件（时间/地点/车速等）→ 存入 Goal List
  ③ 持续监听：Always On 模块持续监听车辆状态和外部环境信息
  ④ 触发执行：条件满足（到达咖啡店附近）→ Always On 通知 Planner 服务
  ⑤ 恢复并执行：Planner 将挂起任务恢复为"进行中"，调用相应 Agent 执行
  ⑥ 状态更新：Planner 通过 plan_update 更新任务状态，向用户播报结果

  → 提醒用户："到咖啡店啦，记得取你的咖啡～"
```

## 十三、端云协同 — "本地门店 vs 总部"

### 13.1 核心逻辑

车上有一个**本地小模型 + 白名单**。常见的简单指令都在白名单里，本地就能处理，不需要联网。

```
能在门店（端侧）解决的就不打扰总部（云端）。
这样门店响应快，总部也不会被小事淹没。
```

### 13.2 端侧 vs 云端的分工

| 端侧处理（本地芯片） | 云端处理（Planner） |
|-----------------------|---------------------|
| "打开车窗" — 白名单直接匹配 | "我好热" — 需要推理和知识 |
| "空调调到22度" — 明确指令 | "带我去兜风" — 多步规划 |
| "暂停音乐" — 简单控制 | "找个好吃的火锅店" — 需要搜索 |
| "导航回家" — 记忆中有家的地址 | "帮我买束玫瑰" — 需要多 Agent 协作 |

### 13.3 端侧的三个好处

1. **极快：** 不用等网络传输（毫秒级 vs 秒级）
2. **不依赖网络：** 隧道里、地下车库都能用
3. **省流量省钱：** 不用每次都把 4.5 万 token 发到云端

### 13.4 什么时候端侧会"上交"到云端

端侧模型判断"这个我搞不定"→ 上传到云端。判断依据：
- 不在白名单里的指令
- 需要联网搜索的需求
- 需要多步推理的复杂需求
- 需要调用多个工具的场景

## 十四、上下文管理 — "大脑的记忆系统"

### 14.1 四种管理方式

Planner 在每次对话中需要参考大量信息。上下文管理就是决定：什么信息该放进来、什么该丢掉、什么该压缩。

| 方式 | 说明 | 举例 |
|------|------|------|
| **写入 (Write)** | 把新产生的重要信息记下来 | 用户说"我喜欢21度" → 记到记忆库 |
| **选择 (Select)** | 从一堆信息里挑出当前最有用的 | 用户问天气 → 只选天气相关的上下文，不需要昨天听歌的记录 |
| **压缩 (Compress)** | 信息太多时，把重要内容提炼成摘要 | 10轮对话压缩成"用户想去火锅店，已选定海底捞，正在导航" |
| **隔离 (Isolate)** | 不同任务的信息互不干扰 | 正在导航的信息不要混到音乐搜索的上下文里 |

### 14.2 公共 Context 管理的 13 类信息

根据公共 Context 管理方案，系统定义了 13 类公共上下文信息，按优先级分为三档（P0/P1/P2）：

**P0 — 核心信息（每次请求必须提供）：**

| 类型 | 说明 | 数据示例 | 来源 |
|------|------|----------|------|
| **情景描述** | Always On 模块按时间线记录的舱内/舱外情景描述 | "2025.09.13-09:42:33 明骏上车了，竟然剪了寸头！还染了灰色！" | 感知模块 |
| **历史对话** | 按时间线记录用户以及 AI 说的话 | "明骏：打开空调 → 豆包：好的，打开了" | ASR/TTS模块 |
| **事件日志** | 按时间线记录的全量事件（人物/车辆/助手的所有事件） | Always On 感知事件 + 执行模块调用 + 表达模块输出 + ASR 收到的用户语音 | 事件日志模块 |
| **端状态** | 仅记录最新的端状态信息（不是历史，是快照） | "空调温度：25°C，风量：3档，模式：自动" | 端状态模块 |
| **屏幕界面** | 按时间线记录最近5分钟的屏幕截图 | image1+timestamp1, image2+timestamp2, ... | 端侧屏幕采集模块 |
| **用户记忆** | 按人物 ID 从长期记忆中加载当前车内成员的记忆 | 对话事件总结 + 基本信息 + 音乐/影视/书籍/新闻/食物偏好 + 车控习惯 + 充电补能偏好 + 助手风格偏好 + 旅游偏好 | 长期记忆 |
| **任务状态** | 当前任务队列中所有任务及其状态 | `[{"task_id":"2","rank":"1","task_content":"给明骏开启座椅按摩","task_condition":""},{"task_id":"3","rank":"2","task_content":"导航去公司","task_condition":"车辆挂入D档"}]` | 任务管理模块 |
| **基础联网信息** | 天气、时间、节日、节气等通用联网信息 | "精准时间"、"公历日期"、"星期"、"农历日期"、"节日"、"节气" | 联网模块 |

**P1 — 增强信息（提升体验，可选）：**

| 类型 | 说明 | 数据示例 | 来源 |
|------|------|----------|------|
| **豆包探索** | 豆包自我探索的内容，按时间线记录不同探索主题的输出 | "explore_task":"研究GPT5最新进展"，"explore_result":"..." | 探索模块 |
| **通用任务经验** | 人工总结的通用任务经验（方便快速提升某类任务效果） | "任务类型：点外卖 → 任务技巧：最好先确定地点"、"发微信消息 → 先向用户确认内容和发送人" | 知识库模块 |

**P2 — 补充信息（完善度，可选）：**

| 类型 | 说明 | 数据示例 | 来源 |
|------|------|----------|------|
| **工具知识** | 各类可调用工具的范围、说明、使用示例 | 参见工具库参考文档 | 工具库 |
| **特定专有知识** | 车品牌相关的知识（D6X 规格等） | Q&A 或文档 | 知识库模块 |
| **豆包性格 & 自我认知** | 豆包的性格、自我了解、世界观、价值观 | "性格：温柔聪明的高知女生"、"价值观：..." | 豆包自我认知模块 |

### 14.3 Context 的智能过滤策略

Context 模块不会一次性加载全部信息，而是根据场景做智能过滤和压缩：

**按场景过滤：**
- 当前是驻车场景 → 行车场景下的偏好不用加载
- 当前是自驾游旅行 → 日常通勤的记忆不用加载

**按人物过滤：**
- 过滤非车内乘客的长期记忆 → 避免隐私泄露 + 减少噪音

**按任务检索：**
- 任务是点外卖 → 只检索外卖相关的任务知识、用户偏好

**短期复用：**
- 短期任务用过的长期记忆不用重新检索，Context 模块会保持住

**压缩总结：**
- 情景描述过长时 → 压缩成摘要（保留关键事件）

### 14.4 与 SP/UP 的映射关系

```
13 类 Context 在请求中的位置：

SP 动态变量 ⑪：
  {{status}}    ← 情景描述 + 屏幕界面 + 事件日志（摘要）
  {{goal_list}} ← 任务状态
  {{env_info}}  ← 端状态 + 基础联网信息
  {{memory}}    ← 用户记忆

SP 固定部分：
  ⑦ 车辆设备知识  ← 特定专有知识
  ⑧ 工具使用tips  ← 工具知识 + 通用任务经验
  ⑩ 聊天风格      ← 豆包性格 & 自我认知

UP 对话历史：
  role:user     ← 历史对话 + 人人对话（ASR 记录）
  role:assistant ← Planner 之前的回复和决策
```

## 十五、对话管理 — 打断、合并、恢复

### 15.1 场景一：还没说完就又说了一句（Query 合并）

用户说了一句话，Planner 还在思考中，用户又说了第二句：

```
用户：给我讲讲今天的财经新闻
豆包：（正在思考，还没来得及回答…）
用户：先导航到海底捞

系统处理：
  这时候 Planner 还没输出第一个请求的结果
  → 把两句话合并成一个请求："讲财经新闻 + 导航到海底捞"
  → Planner 一次性处理两件事

豆包：好的，我来帮你导航去海底捞，顺便整理播报今天的财经新闻
```

### 15.2 场景二：正在播报被打断（打断 + 恢复）

用户在 TTS 播报过程中打断：

```
用户：给我讲讲今天的财经新闻
豆包：好的，今天的重点财经新闻有几条。首先证监会提出要培育
      高质量上市公司…（被用户打断）

用户：先导航到海底捞

系统处理：
  ① 立即停止 TTS 播报
  ② 保存播报进度（"讲到证监会那条"）→ 存入 Goal List
  ③ 处理新请求（导航到海底捞）
  ④ 导航完成后，检查 Goal List → 发现有未完成的播报任务
  ⑤ 恢复播报

豆包：好的，已为你导航到海底捞
豆包：刚刚被打断了，我接着讲财经新闻吧。说到证监会那条…
```

### 15.3 回复话术的三条设计原则

**原则一：不需要每次都说话**

```
❌ 错误：
  用户：空调开没开？
  豆包：我帮你查一下哦~（查车况中…）    ← 多余的一轮话术
  豆包：空调开着呢，当前温度25度

✅ 正确：
  用户：空调开没开？
  （Planner 调用 search_vehicle_status_info，不说话）
  豆包：空调开着呢，当前温度25度          ← 等结果回来直接回答
```

**原则二：不把做什么讲太细**

```
❌ 错误：
  "我帮你把座椅按摩和通风打开，开启全身按摩，
   然后把氛围灯调成冰川色，空调温度也调到15度"
  问题：万一车不支持"冰川色"或"全身按摩"，用户期望落空

✅ 正确：
  "我帮你把座椅按摩和通风打开，顺便调一下氛围灯和空调，
   酷爽放松氛围马上就来"
  好处：不把话说死，给自己留余地
```

**原则三：开头和结尾不重复**

```
❌ 错误：
  "我来帮你调空调和座椅。空调已调到22度，座椅按摩已开启。
   好啦，空调和座椅都帮你搞定了～"    ← 开头说了一遍，结尾又重复

✅ 正确：
  "我来帮你调调空调和座椅。空调22度，按摩也开了，舒服一下吧～"
```

## 十六、演进方向：Goal List + Advisor — "目标驱动的团队协作"

### 16.1 当前系统的四个痛点

| 问题 | 解释 |
|------|------|
| **Planner 负担太重** | 一个人又要想、又要说、又要管，什么都做 → 什么都做不精 |
| **Advisor 利用不够** | 四个顾问（舒适/出行/情感/内容）只会提建议，不参与执行和跟进 |
| **速度和质量矛盾** | 用户要快速回复（性能），但复杂任务需要深度思考（质量），一个模型两头兼顾很难 |
| **规则限制创造力** | 为了稳定加了很多规则（SP中10条注意事项+69个示例），规则多了 AI 的灵活性和创造力就被压住了 |

### 16.2 新方案：从"一人包揽"到"团队协作"

```
Planner（团队 Leader / 执行经理）
  ├── 快速响应用户（不用深度思考，先给个回应）
  ├── 协调各 Advisor（谁来盯这件事？）
  └── 执行具体操作（调用下游 Agent）

Advisor（专线顾问，分静态/动态）
  ├── 静态 Advisor（4个，永久在线，车在运行就一直盯着）
  │    ├── 舒适 Advisor：盯温度/座椅/通风
  │    │    例："车内温度较高，已开启制冷"
  │    ├── 出行 Advisor：盯导航/路况/充电
  │    │    例："正在导航去公司，正常行驶中"
  │    ├── 情感 Advisor：盯乘客情绪
  │    │    例："后排宝宝情绪稳定，正在看动画片"
  │    └── 内容 Advisor：盯娱乐/播放
  │         例："正在播放用户上次没看完的动画"
  │
  └── 动态 Advisor（按需创建，用完销毁）
       用户说"帮我介绍沿途风景"
       → 创建"风景介绍 Advisor"
       → 这个 Advisor 持续关注导航位置，经过景点就提醒 Planner
       → 行程结束后自动销毁

Goal List（共享项目看板）
  所有人共享的目标列表，记录当前所有正在追踪的事
  每个目标包含：目标描述、负责的 Advisor、当前状态、进展详情
```

### 16.3 各自分工

| 角色 | 负责什么 | 核心优势 |
|------|----------|----------|
| **Planner** | ① 快速回应用户 ② 协调各 Advisor ③ 执行具体操作 | 速度快，面向用户 |
| **Advisor** | ① 深度思考（不急，可以慢慢想）② 监控目标进展 ③ 更新目标细节 ④ 给 Planner 提建议 | 思考深，面向问题 |
| **Goal List** | 记录所有目标及其进展，所有人共享 | 透明，不遗漏 |

核心价值：**Planner 管"快"，Advisor 管"好"**。用户说话后 Planner 秒回（保证响应速度），同时 Advisor 在后台深度思考（保证决策质量），想完了把建议发给 Planner。

### 16.4 新体系案例："介绍车辆功能"被打断后恢复

```
用户：介绍一下这台车的功能

Planner：创建动态目标"介绍车辆功能" → 创建动态 Advisor 来盯这个目标
豆包：这台车的亮点功能可多啦！首先是智能驾驶…
      其次是智能座舱，你可以通过喊豆包来完成各种车控…
      此外我还可以——

用户（打断）：导航去赛力斯集团

Planner：收到新请求 → 处理导航
动态 Advisor：记录"介绍进度：讲到了智能座舱，还没讲完"
豆包：好的，为你导航到赛力斯集团，全程13公里

（导航设置完成后）
动态 Advisor → 通知 Planner："之前的介绍还没完，建议恢复"
Planner：收到建议 → 恢复播报
豆包：刚刚被打断啦，我们接着来～ 刚说到我可以帮你做很多事情，包括…
```

### 16.5 Advisor 在 UP 中的具体体现

这就解释了为什么 UP 中的用户输入是一个数组——它同时包含用户说的话和 advisor 的建议：

```json
// role: user
[
  {
    "query": "开启影院模式",
    "speaker_name": "罗勇",        // 用户的话
    "speaker_position": "主驾",
    ...
  },
  {
    "query": "你当前的目标是**实时播报CBA总决赛比分**，接下来你可以：1. 查询最新比分 2. 根据比赛进展更新播报内容",
    "speaker_name": "advisor",      // Advisor 的建议
    "speaker_position": "",
    ...
  }
]
```

Planner 需要同时理解：用户说了什么 + Advisor 建议了什么 → 综合做出决策。

## 十七、模型选型 — 为什么用豆包

### 17.1 选型历程

| 阶段 | 模型 | 结果 |
|------|------|------|
| 最初尝试 | DeepSeek Agent | ❌ 严重问题：有时不调用功能、流程跑偏、长对话后"忘事"、每次拆解结果不一样 |
| 过渡方案 | Kimi K2 | ⚠️ demo 够用，但更多功能后稳定性下降，且不是自家模型，长期不可控 |
| 最终选定 | 豆包 Seed 1.8（自家） | ✅ 可持续训练优化、可定制、性能质量可控、数据不外流 |

### 17.2 训练方式：Function Call 训练

Planner 的核心能力是 **Function Call**——看懂用户需求后，决定调用哪些工具、传什么参数。训练的核心就是让模型学会"什么时候该调什么工具"。

**为什么用 FC 方式训练？而不是让模型输出纯文本？**

| 对比维度 | 纯文本生成 | Function Call 格式 |
|----------|-----------|-------------------|
| 输出 | "好的我帮你打开空调调到22度" | `{"tool_name":"vehicle_basic_control","params":{"query":"空调22度"}}` |
| 下游对接 | 需要再解析一次文本 | 直接JSON传参，工程直接调用 |
| 错误率 | 文本格式不稳定，解析易出错 | 结构化输出，格式稳定 |
| 可评测 | 人工判断"说得对不对" | 自动对比JSON是否正确 |

### 17.3 训练数据格式

每条训练样本由两个核心部分组成：

**① messages — 对话历史**

```json
{
  "messages": [
    {"role": "system",    "content": "你是豆包...（SP全文）"},       // loss_weight=0 不学习
    {"role": "user",      "content": "我好困呀"},                    // loss_weight=0 不学习
    {"role": "assistant", "content": "{\"talk_or_not\":\"yes\"...}", // loss_weight=1 ← 这是要学习的
                          "tool_calls": [{"function":{"name":"vehicle_basic_control","arguments":"..."}}]},
    {"role": "tool",      "content": "执行成功", "name": "vehicle_basic_control"},  // loss_weight=0
    {"role": "assistant", "content": "{\"talk_or_not\":\"yes\",...}", // loss_weight=1 ← 也要学习
                          "tool_calls": []}
  ]
}
```

**② tools — 工具定义列表**

```json
{
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "vehicle_basic_control",
        "description": "车辆基础控制...",
        "parameters": {"type":"object","properties":{"query":{"type":"string"}},"required":["query"]}
      }
    },
    // ...其余22个工具定义
  ]
}
```

**关键机制：`loss_weight` 标记**
- `loss_weight=0`：system/user/tool 角色的内容，模型只需要"看懂"，不需要学习生成
- `loss_weight=1`：assistant 角色的输出，模型必须学会生成这样的内容
- 训练时只对 loss_weight=1 的 token 计算损失函数

### 17.4 并行 vs 串行任务训练

**并行任务示例（"我好困"→ 一次性输出多个工具调用）：**

```
用户："我好困呀"

assistant 输出（loss_weight=1）：
  talk_content: "好呀，我帮你把车窗关了、空调开了、休憩模式也打开了"
  tool_calls: [
    vehicle_basic_control("关闭全部车窗"),      ← 同时
    vehicle_basic_control("开启空调，温度22度"),  ← 同时
    vehicle_basic_control("开启休憩模式")         ← 同时
  ]

tool_feedback（三个工具同时返回）→ assistant 最终回复
```

**串行任务示例（"做个规划+记个待办"→ 分步执行，后步依赖前步）：**

```
用户："做一个从北京到西安的自驾游规划，然后把这个规划记到我的待办"

第1轮 assistant（loss_weight=1）：
  talk_content: "好的，我来帮你做规划"
  tool_calls: [route_planning_qa("北京到西安自驾游路线规划")]
  ← 只调一个工具，等结果回来

第1轮 tool_feedback → 返回路线方案

第2轮 assistant（loss_weight=1）：
  talk_content: "规划好了！路线是...，我帮你记到待办"
  tool_calls: [operate_user_memory("保存自驾游规划到待办")]
  ← 拿到第1轮结果后才能做第2步
```

### 17.5 五步训练管线

```
步骤①：蒸馏（知识萃取）
  让 Kimi K2 做标杆，构造各种场景的输入输出对
  → 产出：大量"好学生的示范答案"（FC格式的训练样本）
    │
    ▼
步骤②：评分筛选
  让 GPT-4o 对每条样本打分（工具调用是否正确、参数是否合理、话术是否自然）
  → 产出：筛选后的高质量训练集
    │
    ▼
步骤③：SFT 训练
  用筛选后的数据，在 Seed 1.6/1.8 上做 Supervised Fine-Tuning
  → 产出：初版 Planner 模型
    │
    ▼
步骤④：评测
  用评测集（113条统一评测case，覆盖三级场景分类）自动跑分
  + 让 Kimi K2 做裁判对比评分
  → 产出：各维度得分 + badcase 列表
    │
    ▼
步骤⑤：迭代
  针对 badcase 补充训练数据 → 回到步骤② → 循环
  后期引入 RL（强化学习）进一步优化
```

**两阶段目标：**
- **阶段一（SFT）：** 让模型学会基本的 FC 能力——看懂需求 → 选对工具 → 传对参数
- **阶段二（RL）：** 让模型学会更精细的判断——什么时候该并行、什么时候该串行、什么时候该反问用户

## 十八、已知问题和挑战

| 问题领域 | 具体表现 | 影响 |
|----------|----------|------|
| **任务拆解不稳定** | 同一句话，每次拆出来的步骤不一样 | 用户体验不一致，难以预测 |
| **简单/复杂误分类** | 仲裁模型把简单需求判成复杂（或反过来） | 该快的时候慢，该深度处理的时候草率 |
| **上下文捕捉弱** | 长对话后，模型"忘记"了之前挂起的任务或失败的任务 | 用户设了条件任务但系统忘了 |
| **车控幻觉** | AI 承诺了车上不支持的功能（比如"帮你调成冰川色"但实际没这个颜色） | 用户期望落空，信任度下降 |
| **话术生硬机械** | "我帮你把xxx打开、把xxx调到xxx"——听多了像机器人 | 不自然，体验打折 |
| **Planner 过载** | 一个模型兼顾理解、推理、话术、任务管理 | 性能和质量难以同时保证 |
| **性能 vs 质量矛盾** | 想要推理准确就需要时间，但用户等不了 | 需要在"回答快"和"回答好"之间找平衡 |

## 十九、Badcase 排查路径

遇到问题按以下顺序排查（从上到下，先排除简单原因）：

```
1. 链路问题（最先排查）
   └── 请求有没有走到正确的模块？
       是不是该进 Planner 的被截留在句法 RAG 了？
       是不是该走快速通道的被误判为复杂进了 Planner？
       → 看请求日志，确认经过了哪些模块

2. 工具描述问题
   └── 给 Planner 的工具说明写得清不清楚？
       工具说明模糊 → Planner 不知道该调谁
       工具参数说明缺失 → Planner 传错参数
       → 看 SP 中的工具定义，检查 tool_describe 和 notice

3. 上下文问题
   └── Planner 拿到的信息全不全？
       缺了用户记忆 → 不知道用户偏好
       缺了车辆状态 → 不知道当前车况
       缺了情景信息 → 不知道外部环境
       → 看请求中的 {{memory}}/{{env_info}}/{{status}}

4. 模型能力问题（最后才考虑）
   └── 信息都给到了但还是推理错
       → 模型本身需要加强训练
       → 需要补充更多类似场景的训练数据
       → 或在 SP 示例中增加类似 case
```
