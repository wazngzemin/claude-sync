触发器语义匹配（模型解析）· 链路与协议

以"后排宝宝哭了告诉我"为例 · 供 7.16 对齐 · 基于端侧真实链路（高频任务UP）
📌 v0.1 草案 · 2026-07-16 · 供对齐，非最终

这份材料就一个目的：把"触发器要做的语义匹配（模型解析）"这件事，用真实链路摆到桌上，逐段对齐输入、判断、输出、边界。会上顺着下面的流程图往下走，每停一处就对一处。

⚠️ 开场先钉这一颗钉子（头号待澄清，不澄清后面白聊）：
现在"判断是否达成用户 query（用户指令）的任务目标"这个动作，被画在了 Task Service（任务服务）名下。但 7.15 技评庆磊定的是"语义匹配归 Trigger（触发器）"，技术方案也白纸黑字写"Trigger 完成语义匹配"。
这个判断到底是 Task Service 做，还是我触发器的规则服务做？——必须今天定死。如果对外都算 Task Service，触发器就被架空成一个纯转发管道了。我的主张：判断这块是触发器规则服务的核心能力，归我。

一、端到端链路（会上投屏，指着这张讲）

① 用户 USER"后排宝宝哭了告诉我"
↓
② Task Service（任务服务）· 生成任务根据用户 query 生成一个 高频任务UP（编号如 044），下发端侧 VLM
↓ 下发观察任务（编号044 + 目标"宝宝哭了"）
③ 端侧感知模块 VLM（视觉大模型）舱内高频 AlwaysOn 每 5秒 拍一次，把结果拆成两框，分两个 Topic（订阅主题）上报 ↓
↓
🟥 红框 = 高频任务UP结果{"任务编号":"044","识别结果":"后排宝宝安静 / 在哭"}
← 这一框专门给你（触发器）做匹配
🟩 绿框 = Context 结果乘客列表（位置 / 动态神态）
← 走 Context（上下文）路线，不归你
↓ 红框经 AI Service 数据中心，订阅给触发器
④ 触发器 · 语义匹配模型　【你的核心 · 归属待澄清】判断 VLM 的"识别结果（在哭）" 是否 语义匹配 任务目标"哭了"
↺ 每 5 秒来一次就判一次，没匹配上就继续等
↓ 匹配命中 → callback（回调）
⑤ Task Service · 起执行 + 控反馈周期命中后，按用户反馈周期（如"每60秒告诉我一次"）叫 Planner 执行播报；达成后通知 VLM 结束高频任务UP
↓
⑥ Planner（规划器）→ 用户播报"后排宝宝哭了"
看这张图，你要在会上锁定的就一件事：第 ④ 步是你的。①②⑤是 Task Service、③是 VLM、⑥是 Planner——只有第 ④ 步"语义匹配判断"是触发器的核心能力，这是你这一期要交付、也要守住归属的东西。

二、VLM 的输出：为什么是"红框归你、绿框不归你"

这是你链路图里最关键的设计——VLM 主动把舱内高频结果拆成两框，分两个 Topic。别混：

框内容订阅方 / 走哪条线归谁
🟥 红框
高频任务UP结果		 {"任务编号","识别结果"}
识别结果 = user_query（用户指令）对应的回答		 触发器订阅（经 AI Service 数据中心） 触发器语义匹配
🟩 绿框
Context 结果		 乘客列表：位置、动态神态（"头靠肩膀睡觉"等） Context 订阅（context 路线） 不归你，走 Context
你的边界第二刀（会上要主张）：我的语义匹配只消费红框（高频任务UP结果的"识别结果"字段）。绿框、以及格式表里那些自由 caption（画面描述）、标签串（如"大件物品-否-否"）是别的任务（停车/迎宾/安全/Context），不进我的语义匹配。守住这条，你的模型才不会被无限加复杂度。

三、两个周期不能混（你链路图里的关键细节）

周期典型值谁定 / 干嘛
VLM 上报周期5 秒VLM 定，高频任务UP 每 5 秒吐一次识别结果给你判断
用户反馈周期如 60 秒用户 query 里指定（"每60秒告诉我宝宝在干嘛"），Task Service 控制多久播报一次给用户
会上要问清：这两个周期解耦——VLM 5 秒给我判断素材，但用户要的是 60 秒一次反馈。"多久判一次"（我）和"多久告诉用户一次"（Task Service）是两回事，别被合并到一个周期里。这也牵扯：命中一次后是立即报、还是等反馈周期到点报？今天定。

四、输入协议草案（VLM 红框 → 触发器）

这是你抢定义权的实物——你先写，让丁彬/西钺在你的字段上改。会上投这张，逐字段确认：

字段类型说明 / 待确认
任务编号 task_nostring（字符串）Task Service 生成高频任务UP时分配，VLM 原样带回
识别结果 resultstringuser_query 对应的回答，如"后排宝宝安静""后排宝宝在哭" 粒度待定
上报时间 tsint（整数时间戳）这一帧的时间
空值约定—无异常/未识别到时输出什么？空串=没触发，得约定死 待丁彬确认
置信度 confidencefloat（小数）建议加VLM 若能给置信度，我做阈值控误报更稳

五、输出协议草案（触发器 → Task Service）

字段类型说明
任务编号 task_nostring命中的是哪个任务
回调编号 callback_idstring本次触发的唯一标识
命中结果 matched_resultstring触发时 VLM 的原始识别结果（"在哭"），供播报用
触发时间 triggered_atint命中的时刻
我不回什么（边界声明）：我只回"匹配命中了、是哪个任务、原始识别结果"。我不生成播报话术、不管任务生命周期、不做二次判断——那些是 Task Service / Planner 的。命中之后的事，不归我。

六、今天要拍死的待确认项（会上逐条收结论）

#待确认找谁拍我的主张 / 倾向
1语义匹配判断的归属：Task Service 还是触发器规则服务？庆磊 / 晓伟归触发器（我）。这是我这一期的核心交付
2我的输入 = 红框"识别结果"字段，对吧？绿框/自由caption/标签串不进我丁彬 / 西钺只消费红框识别结果，划死边界
3识别结果的粒度：VLM 给"在哭 / 没哭"（近结论）还是"有泪珠"（纯描述）？丁彬越接近结论我越好判；但结论仍由我下
4空值约定：没识别到时输出什么丁彬空串=未触发，写进协议
5两个周期：上报周期(5s) vs 反馈周期(60s) 怎么解耦，命中后何时报庆磊 / 丁彬判断和反馈解耦，别合并
6匹配模型谁提供：task 模型侧(晓朝)给，还是触发器自起轻量模型？晓朝 / 刘杨问清，决定实现路径和成本归属
7准确率责任：badcase 谁扛晓朝 / 全场我定阈值兜底策略；准确率和算法共担；宁漏报不误报
8成本：5s一判的调用量级韩杰轻量模型，非常驻，和 Planner 切开；测算后补
9L2 泛化边界：像"照顾二排老人"那种每步重决策的，走不走我？庆磊 / 明骏不走我，走 Task 模型/React；我只做有明确目标的匹配
会上收尾你就念这句："今天结论我复述一遍——判断归触发器；输入是红框识别结果；粒度 X；空值 Y；两周期解耦；模型 Z 提供；准确率共担；成本我给测算；L2 不走我。协议草案我 X 日出正式版。有漏的吗？" —— 每条都落到白纸黑字，这场会你就赢了。
