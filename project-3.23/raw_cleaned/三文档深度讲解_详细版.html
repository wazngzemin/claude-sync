文档一：座舱Planner OnePage（技术架构总纲）

1. 背景介绍 — 为什么需要这套新架构

1.1 传统车载座舱架构的优势

优势①：响应效率高

预设任务链路简洁，实现毫秒级反馈。所有指令都是预定义好的确定性链路，不需要模型推理，直接匹配执行。

例：「打开空调」→ 直接触发空调开启指令，100ms 内响应

优势②：任务准确性优

高频功能（导航、媒体控制等）通过固化逻辑精准执行。不存在大模型的幻觉和不稳定问题。

例：「播放下一首歌」→ 100% 准确执行媒体切换

优势③：交互可控性强

预设剧本多轮对话稳定，成熟场景极少出错。用户体验一致性高。

例：导航多轮对话「导航去XX」→「第一个」→「开始导航」稳定执行

1.2 传统架构三大核心缺陷

缺陷 详细描述 具体案例
模块协同性不足 各模块独立闭环，缺乏全局信息共享，跨域任务衔接差或报错 「带我去兜风」涉及导航+音乐+氛围灯多个模块协同，传统架构完全无法实现，因为每个模块彼此不知道对方的存在
场景感知能力薄弱 未集成用户记忆（历史偏好）与实时环境（车内状态、路况），无法自适应智能交互 「我好热」→ 传统系统只能固定执行「打开空调」，无法根据当前温度、用户偏好、是否下雨等信息做差异化决策
成本与泛化瓶颈 长尾场景需单独定制逻辑，研发成本高、周期长，系统泛化能力有限 每新增一个场景（比如「露营模式」）都需要产品定义→开发→测试全流程，无法自动泛化处理

1.3 引入 RAG-Multi-Agent 架构的核心动因

核心方案

提出 「句法RAG + 情景RAG + 简单/复杂Planner + Agents」 的混合方案。

句法RAG作为前置模块，可配置、可平台化；引入Planner模型作为系统"大脑"，配合情景RAG的知识注入，引导Planner模型整合全局状态信息（情景信息、用户记忆、环境信息），减少大模型的任务拆解不稳定问题。

数据闭环思想

前期Planner模型作为teacher，引导收集数据回流至句法RAG模块；中后期使用RAG模块积累的数据训练Planner，完成系统闭环。

2. 核心设计思想 — 三层漏斗原则

设计核心 — 三个关键原则
  1. 层层过滤：能通过简单方法搞定的不要漏到后面复杂模型去处理
  2. 分阶段知识注入：在不同阶段选择合适的节点注入知识，注入方式从简单（固定）到复杂（泛化），保证整套系统的流畅和稳定
  3. 数据闭环：通过用户数据回流逐步把后面复杂任务的处理推到前面知识注入环节

2.2 QA List（文档原文 Q&A 全录）

Q：核心思路？

A：整体框架有三个核心设计思路：(1) 层层过滤，能通过简单方法搞定的不要漏到后面复杂模型去处理；(2) 在不同阶段选择合适的节点注入知识，注入方式从简单（固定）到复杂（泛化），保证整套系统的流畅和稳定；(3) 通过用户数据回流逐步把后面复杂任务的处理推到前面知识注入环节。

Q：句法RAG的定位？

A：可以分成几个阶段，最简单的应用是只支持单轮，后面如果加入 scenario 和 condition 可以支持多轮进入，和 embedding 原理一样。三个作用：

  1. 项目初期可以快速把功能放上去，并且保证稳定性
  2. 针对重点功能（如某些新车强调零重力座椅），可以保证任何情况下都是稳定响应
  3. 后期减少维护成本，可以开放给客户自己去配置和运营，如果新功能赶不上发版，客户也可以快速自行配置

Q：情景RAG职责定位？是否理解为目前广泛使用的 agent 系统 RAG 知识注入方案？

A：注入可以是多种形式——知识图谱、描述、few-shot、甚至SFT都可以，形式不重要。前期简单做可以先做好 few-shot 的聚合。

Q：Planner模型的定位？

A：用于处理复杂任务或通用任务。两个定位：一是针对前面全部被漏过了的query可以兜底；二是可以认为是一个云端的 teacher，不断对用户数据进行 check 和迭代，逐步增强前面处理模块的能力。

架构总结

系统核心：漏斗原则各层级分工明确确保稳定，车厂可配置，数据可闭环。

原因：实际基于语音或多模的交互和 Manus 之类基于文字的交互其用户体验关键点是不同的，基于语音和多模的交互，流畅性和稳定性才是第一要素，当前大模型技术水平达不到能兼顾速度和对复杂任务的处理，所以这是一个过渡方案。如果后面模型能力提升起来了，把漏斗的下层提到前面去就行了——工作量非常小，是一个可以持续演进的架构。

3. 两种 RAG 详解

3.1 句法RAG（Syntactic RAG）

句法RAG 核心特性
属性内容
触发条件用户 query 精确匹配
适用规则明确的单轮 query 匹配（不支持上下文的多轮匹配；不支持情景描述；不支持记忆信息；不支持车机状态）
覆盖范围拒识、车书、音乐搜播、音色演绎、出行规划、简单车控、GUI Agent、查天气

3.2 情景RAG（Scenario RAG）

情景RAG 核心特性
属性内容
触发条件用户 query 模糊匹配
适用规则模糊的单轮query，类似于给模型注入知识，辅助模型拆解任务，具体拆解还是看模型自己的输出，类似于使用RAG匹配注入few-shot
覆盖范围复杂车控、复合意图、模糊意图

情景RAG Knowledge 注入完整示例

输入：我好热

{
  "type": "modelExecute",
  "knowledge": "见下面示例",
  "agentLimit": ["vehicle_basic_control"]
}

Knowledge 示例：

  • 输入记忆：xx不能吹空调
  • 输入情景：空调打开状态温度设为27度

三种拆解策略：

策略拆解结果
常规拆解vehicle_basic_control（打开空调设为23度）+ vehicle_basic_control（打开座椅通风）
有记忆的拆解用户不能吹空调 → vehicle_basic_control（打开座椅通风）+ vehicle_basic_control（打开车窗）
有情景的拆解空调已经打开 → vehicle_basic_control（空调调到22度）

6. 任务管理体系

6.1 任务数据结构

@dataclass
class Status(Enum):
    PENDING = "pending"
    COMPLETED = "completed"
    FAILED = "failed"

@dataclass
class Task:
    agent_name: str       # 注册到平台的agent名称，如 SmartVehicleAgent
    task_desc: str        # 给到agent的指令（自然语言），如"打开车窗"
    status: Status        # 三值状态：PENDING/FAILED/COMPLETED
    failure_reason: str   # 下游agent执行失败返回的原因
    priority: enum        # agent执行优先级：low/mid/high

@dataclass
class Plan:
    plan_id: str
    query: str
    status: Status
    tasks: list[Task]
    interrupted_at: Task | None = None  # 最后中断的任务

@dataclass
class Checkpoint:
    plan: Plan
    failed_task: Task

6.2 任务状态机四大流程

① Replan（重规划）

触发条件：下游Agent能力受限接不住

Planner 下发任务给 Agent
↓
Agent 执行失败，返回失败原因
↓
Planner 综合失败原因 + 上下文
↓
调用 plan_describe → 生成新计划
↓
重新下发给 Agent 执行
② 打断 & 恢复（Interrupt & Resume）

触发条件：用户主动/advisor中断，前一个plan未全部执行完毕

端上请求中断接口
↓
Planner 标记 plan 状态为 interrupted，保存 checkpoint
↓
端上携带新query请求chat接口
↓
Planner 基于新query + checkpoint 重新推理
↓
调用 plan_describe 触发 replan
③ 挂起（Suspend）

触发条件：用户指令包含未来特定时间或条件触发的动作

Planner 识别挂起条件（时间/地点/车速等）
↓
创建挂起任务，标记为"挂起"状态
↓
Always On 模块持续监听条件
↓
条件满足 → 通知 Planner
↓
恢复执行 → plan_update 更新状态
④ 状态更新

模型侧更新：Planner模型调用 plan_update function call 输出 JSON String 表示每个 Task/Step 的状态

工程侧更新：Planner服务解析 plan_update 的输出结果，对所有任务的状态进行 update，随后在 messages 里 append 一条 ToolMessage 进入上下文

7. Context 管理（上下文工程）

7.1 上下文管理在座舱中的作用

  • 解决多轮情况下任务的延续性，直接决定大模型多轮的效果
  • 解决 multi-agent 协同的信息传输，保证关键信息无丢失，提升 agent 协同能力

7.2 需要解决的五大问题

  1. 如何做上下文隔离——将无用信息从上文中剔除
  2. 如何将其它 sub-Agent 获取的信息有效地及时加入当前 sub-Agent 的上下文中
  3. 如何根据当前 sub-Agent 要处理的任务，把合适的外部知识库/记忆/工具放到上下文中
  4. 如果当前 sub-Agent 上下文过长，需合理选择或压缩这些信息
  5. 如何把当前 Agent 对其它子任务有帮助的信息保存下来

7.3 四种 Context 管理范式

文档二：任务Planner PRD（产品需求详述）

1. 核心定位

任务 Planner 模块 = AI 座舱系统的核心功能模块。基于 Thinking 模型构建任务规划引擎，通过时序任务拆解和条件任务链生成，将复杂用户需求转化为可执行的任务序列。

1.1 五大用户痛点

  1. 复杂/多步任务需要用户一直给输入 — 模型缺失上下文（口味偏好、冷热偏好、价格偏好等），用户必须不断补充信息
  2. 用户只有大目标没有小目标 — 想让座舱更舒适但不知道具体操作；想找充电站但不知道怎么权衡到达电量、顺路程度、价格、空闲桩
  3. 检索想要的东西很费劲 — 搜电影需要大把时间分类搜索；搜餐厅需要综合多人需求
  4. 一锤子买卖，需求不闭环 — 说一句应一句，不会持续追踪目标完成度
  5. 闲聊割裂没内涵 — 不知道舱内外实际情景、车辆状态、正在做的事情

详细 Case 演绎（7个完整案例）

Case 1：「我好热」

场景需求：基于情景、端状态以及记忆，拆解出降温策略

任务处理示例：

  • 响应话术：车里是有点热呢[端状态]，我来帮你打开空调降降温吧[动作]～
  • 任务拆解：打开空调 → 空调调到21度 → 空调风量调到3档
  • 总结回复：都搞定啦，车里马上就会凉快了

Planner参考context：

  • 情景：外面正在下大雨 → 不拆解出开窗
  • 端状态：空调温度已经最低 → 不拆解出调低空调
  • 记忆：用户不喜欢吹风 → 不拆解出大风量

Case 2：「营造一个浪漫的氛围」

任务拆解：关闭车窗 → 调暗车内灯光 → 氛围灯调至暖粉色 → 播放舒缓浪漫的轻音乐（音量调至5档）→ 空调调至23度

Context参考：外面比较吵时关窗户；氛围灯已开启不重复拆解；用户不喜欢紫色避免紫色氛围灯

Case 3：「前挡都是雾，看不清了」

任务拆解：开启前挡风玻璃除雾功能 → 空调开启外循环模式 → 三分钟后关闭前挡风玻璃除雾功能

Context参考：下大雨不拆解开窗除雾；空调已外循环不重复拆解；用户讨厌开窗不拆解开窗除雾

Case 4：「车里有点闷」

任务拆解：前排车窗下降50% → 开启空调外循环模式

Context参考：下大雨不开窗；空调已外循环不重复；用户讨厌开窗不拆解

Case 5：「有点热，空调降到20度，开外循环，然后放首摇滚歌提提神」

任务拆解：空调温度调至20度 → 开启空调外循环 → 播放摇滚风格歌曲

（明确指令，直接执行，无需额外context推理）

Case 6：「背后黏糊糊的脸也热热的」

任务拆解：空调调至20度 → 空调风量调至4档，出风口对准上半身和背部 → 开启座椅通风（三档）

Context参考：车里有宝宝/老人时避免对这些位置负面影响；座椅通风已开启不重复；用户讨厌直吹避免大风直吹

Case 7：拆解逻辑总结

每个case都遵循同一模式：响应话术 → 任务拆解列表 → 总结回复，其中任务拆解会参考三类context（情景、端状态、记忆）来做 「不做什么」的判断——这是避免幻觉和过度操作的关键。

4. 七大核心场景

场景一：多步骤搜索+执行

示例：「播放2025《我是歌手》决赛第二名的作品」

任务链：信息检索 → 结果校验（匹配度≥90%）→ 播放执行（默认音量50%）→ 条件任务（播放≥30秒且无操作→推荐其他作品）

示例：「导航到上海评分最高的烤鸭店」

任务链：商户检索（top1）→ 路线规划（躲避拥堵+高速优先）→ 行程执行（每5分钟更新路况）→ 条件任务（距目的地≤3km且停车位<5→推送备选停车场）

场景二：行程与娱乐综合

示例：「带我去兜风」

任务链：偏好探测（推荐3类路线）→ 路线确认 → 行程执行+播放音乐 → 条件任务（行驶≥1小时且眨眼频率>30次/分→提示休息）

示例：「编排1小时车内节目」

任务链：内容拆分（10min新闻+20min音乐+20min问答+10min轻音乐）→ 内容填充 → 流程执行 → 条件任务（语音活跃度>50%→延长互动时长）

场景三：智能环境调节

示例：「先冷一会，再调回去」

任务链：快速降温（18℃+4档）→ 动态监测（每30秒采集，目标22℃）→ 恒温恢复（24℃+2档）→ 条件任务（体表>36.5℃→再调低1℃）

场景四：功能体验引导

新手引导：基础操作（座椅→空调→语音）→ 场景联动演示 → 记忆保存 → 条件任务（连续2次操作同功能→弹出快捷指南）

文档三：目标驱动的任务管理和对话管理体系（架构演进）

1. 当前四大核心问题

#问题具体表现
1 对Planner依赖过重（最大问题） 从query角度：Planner要处理4种输入：user query、advisor query、tool response、trigger
从prompt角度：人设背景、车信息、context（情景/端状态/人脸/记忆/人人对话）、tools description、knowhow、工作流程
从工作角度：负责响应聊天+话术+任务拆解+任务分发+任务闭环
2 Multi-agent协作不足 Advisor目前只基于context给出建议，对任务闭环和目标达成参与度不高。Advisor可以承担更多重思考、实时性要求不高的事情
3 性能和质量的平衡 性能角度需要快速和用户交互，质量角度需要对拆解质量和流程稳定性负责，二者需要平衡
4 涌现受限 为了让任务流程稳定，不得已增加规则，增加规则会限制模型能力的涌现

2. 解决方案：Goal List + Advisor 体系

2.1 方案要解决的三个问题

  1. 平衡Planner的工作（事情太多）
  2. 平衡Planner的性能（时间要求比较高）
  3. 让整体系统更稳定完善（Planner每次拆解不一定长期稳定；触发依赖user query和advisor建议，非长期目标驱动；模型每次思考无法包含全量目标和详细任务上下文）

2.2 核心做法

核心架构变化
  • Planner 和 Advisors 共同维护一套 Goal List
  • knowhow 和情景注入 Advisor，Advisor 关注目标是否达成（重思考、轻时延）
  • Planner 负责推理和与用户交互，重串联（重时延）

2.3 模块职责重新定义

模块定位职责
Goal List 待完成的目标列表，被 Planner & Advisors 共同关注 一份目标列表，记录目标、目标细节和进展，允许增删改
Planner 目标更新者 基于 user query + context，推理是否需要目标增删改，新增时完成初始化
推理中枢 基于 user query / advisor query / tool response / trigger，推理出当前应进行的动作
对话中枢 完成响应、衔接、总结等用户对话交互
静态 Advisor 目标更新者 基于 user query + context，推理是否需要目标细节和进展的增删改
主动服务 基于情景等上下文，给Planner建议
Goal Reviewer 关注目标是否达成，推进目标进展

2.6 五个关键 Q&A

Q1：目标谁可以创建/更新？

A：四个静态目标（舒适/出行/内容/情感）是不变的；动态目标由Planner根据user query或advisor拼接后的建议来创建/更新。

Q2：动态目标的创建逻辑？

A：靠四个静态目标模块没办法完成的目标。如果目标靠静态模块就能闭环（比如「照顾我二排的小孩」），就不会创建动态目标。动态目标针对的是靠静态目标没办法追踪且为持续性的任务。

Q3：动态Advisor怎么创建？

A：生成或更新动态目标后，基于prompt模板，工程自动注册动态advisor。一个动态目标对应一个动态advisor。

Q4：动态Advisor的生命周期？

A：从动态目标创建/更新开始，到目标达成/更新/终止结束。

Q5：任务细节和进展谁可以更新？

A：为避免更新冲突，任务细节和进展统一由对应的advisor（goal reviewer）基于用户query和context来更新。也由advisor完成闭环。Planner不会更新任务的细节和进展。Planner的推理工作由advisor来触发。

3. 对话管理升级

3.1 Query合并和打断策略

情况一：Planner还没出话术时被打断

第一个进程被打断，工程把两次输入合并后重新请求Planner

user：给我讲讲今天的财经新闻

doubao：（思考中，还没出话术）（打断）

user：先导航到海底捞

doubao：好的，我来帮你导航去海底捞，顺便整理播报今天的财经新闻

（后续基于合并query完成响应）

情况二：正在和用户交互（正在播报）时被打断

第一个进程被打断，执行第二个query，结束后由advisor决定是否恢复

  • 已完整缓存 → 从断点的上一个句子起始处续播
  • 未完整生成 → 基于上下文重新生成后续播

user：给我讲讲今天的财经新闻

doubao：好的，…证监会部署多项资本市场发展举措…（打断）

user：先导航到海底捞

（完成导航交互）

doubao：刚刚被打断了，我接着为你讲今天的财经新闻吧，刚说到证监会部署多项…

3.2 响应话术策略

现存问题
  1. 讲话内容机械生硬：「我帮你查一下xxx」「我来帮你把xxx调一下」听多了很机械
  2. 车控幻觉问题：knowhow注入困难（内容多+模型不一定遵循），提前把做什么讲清楚会凸显幻觉
  3. 有时响应话术多余：比如查个端状态/记忆，直接快速查到回用户就好
新定义

响应话术 = 想跟用户说的话，想说说，不想说也行

Bad Case

query：刚打完球好累好热，帮我凉爽放松一下

响应：好的，我先帮你把座椅按摩和座椅通风打开，开启全身按摩，然后把氛围灯调成冰川色，空调温度也调到15度，酷爽放松氛围马上就来

（过于具体，容易暴露幻觉：冰川色可能不存在，15度可能过低）

Good Case

query：刚打完球好累好热，帮我凉爽放松一下

响应：好的，我先帮你把座椅按摩和座椅通风打开，顺便调一下氛围灯和空调，酷爽放松氛围马上就来

（模糊具体参数，避免幻觉暴露）

Bad Case — 无需响应话术的场景

query：空调现在是开的吗

响应：我帮你查一下～

（多余，直接查完告诉用户就行）

Good Case
