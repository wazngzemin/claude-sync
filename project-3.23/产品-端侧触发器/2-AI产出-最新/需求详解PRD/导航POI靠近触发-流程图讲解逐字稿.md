# 导航 POI 靠近触发需求评审逐字稿

> 使用方式：这版是“先讲背景和目标，再拿着流程图讲”的稿子。前 3-5 分钟先把为什么做、做什么、不做什么讲清楚；然后把流程图投出来，从左下角开始，一路讲到右侧工具库和右上角动态插途经点。  
> 目标：让上汽地图、端侧研发、触发器、信号库、运算符、Advisor/Director 都知道自己在图上的位置、要评估什么、今天要给什么结论。

## 1. 开场：先讲今天为什么评审这个需求

大家好，我先用几分钟把这个需求的背景、目标和今天要大家评估的点讲清楚。后面我不会按 PRD 一页一页念，我会拿流程图从注册到触发完整走一遍。

这个需求叫“导航 POI 靠近触发”。它对应的是用户很自然会说的一类话：

```text
快到外婆家提醒我打电话
快到公司帮我找充电桩
离目的地还有 1 公里提醒我拿咖啡
路过徐浦大桥提醒我右转
```

这些话听起来都很像，都是“到某个地方附近时帮我做一件事”。但落到触发器系统里，现在是做不了的。

原因很简单：当前触发器还不认识导航里的几个核心概念。它不知道“导航中”是什么，不知道“目的地”是什么，不知道“途经点”是什么，也不知道“剩余距离”是什么。更关键的是，它不知道“快到了”这个状态应该怎么判断。

所以用户说“快到外婆家提醒我打电话”，系统可能能听懂这句话，但没有办法真正注册成一条可执行的触发规则。就算注册了，后面用户真的开车快到了，系统也不知道什么时候触发、触发几次、触发后怎么停止。

这就是这个需求的背景。

## 2. 背景：现在缺的不是一句播报，而是一类导航事件

我想先把问题讲准确。

这不是一个“加一句播报”的需求，也不是一个“加一个距离小于 500 米条件”的需求。我们真正缺的是一组导航事件能力。

比如：

```text
发起导航
接近目的地
到达目的地
离开目的地
目的地变更
导航回家 / 去公司
接近途经点
路过 POI
```

这些事件如果没有，后面的 Advisor、Director、工具调用都没有稳定入口。

举个例子，“快到公司帮我找充电桩”。真正的链路应该是：

```text
系统先知道用户注册了这个提醒
后面用户导航去公司
导航距离真的进入阈值
系统产出 approach 事件
触发器命中规则
Advisor/Director 再决定是否调工具找充电桩
最后再播报给用户
```

所以这个需求不是在补一个单点 case，而是在补“导航过程可以触发主动服务”的底层事件能力。

## 3. 目标：这次到底要补哪些能力

这次目标我建议收成四个。

第一，补导航信号。

至少要让系统能拿到导航状态、目的地/途经点、沿路线剩余距离、收藏地址匹配结果这些信息。

第二，补导航事件。

最小闭环是 approach、arrive、leave。也就是接近、到达、离开。再往后才是 pass_by，也就是路过 POI。

第三，补 one_shot 状态机。

“接近”不是 distance 小于 500 米。它必须是第一次从远处跨过阈值，只触发一次。否则车辆从 499 米开到 50 米，每秒都满足条件，用户会被刷屏。

第四，把这些事件供动态 Advisor 和 Director 消费。

触发器本身只负责把事件和规则匹配上。真正的播报、工具调用、话术生成，是后面的 Advisor、Director、工具库来完成。

所以今天这次评审，我们不是只评审一个文档，而是要一起确认这条链路能不能成立。

## 4. 范围：先保 A 类，B 类要看地图能力

这里我先把范围说清楚。

我把 case 分成两类。

A 类，是目的地/途经点距离。

也就是说，用户要靠近的 POI，本身就是当前导航目的地，或者已经是途经点。这个时候导航引擎天然就在算剩余距离。比如：

```text
离目的地还有 1 公里提醒我
快到途经点帮我找停车位
快到公司帮我找充电桩
```

这一类我认为是第一版最应该保的。

B 类，是任意 POI 或路过 POI。

比如：

```text
路过徐浦大桥提醒我右转
快到外婆家提醒我打电话，但外婆家不是当前导航目的地
```

这一类的难点是：POI 不在当前导航目标里，导航引擎默认不会算到它的剩余距离。除非地图侧支持动态插途经点，或者支持按当前路线计算到某个 POI 的沿路线距离，否则 B 类就不能按主方案落地。

所以我建议今天的范围口径是：A 类先做，B 类请地图和导航评估后再决定，不要一上来全部承诺。

## 5. 今天要大家拍板的问题

今天我希望大家不是泛泛说“方向可以”，而是按角色给明确结论。

地图和导航要确认：目的地、途经点的沿路线剩余距离能不能稳定输出；运行中能不能动态插途经点；高德是否有地理围栏 API，如果没有，端侧自建围栏需要什么。

端侧要确认：端侧信号采集、端侧信号库、POI 状态机、事件同步云端这条链路能不能做，工期是多少。

触发器要确认：如果端侧已经产出 approach、arrive、leave 事件，云端触发器能不能只做事件匹配和 push Advisor。

信号库要确认：nav_event 怎么注册，事件 payload 字段怎么落库。

运算符要确认：event_type 字符串匹配是否支持；如果后续做地理围栏，路线相交、点在区域内、距离小于半径这些运算是否已有。

Advisor 和 Director 要确认：收到事件之后，怎么生成建议、怎么决定播报、怎么调用工具。

意图模型要确认：“快到”和“路过”能不能区分。如果区分不了，第一版就只做更确定的“快到目的地/途经点”。

## 6. 过渡：接下来我们看流程图

这张图里我想讲清楚三件事。

第一，用户说“快到外婆家提醒我打电话”之后，系统到底是怎么从一句话变成一条可执行规则的。

第二，半小时之后用户真的导航快到了，这条规则怎么被端侧状态机触发，再怎么同步到云端，最后怎么走到 Advisor、Director、TTS。

第三，这条链路里最难的不是播报，也不是规则配置，而是两个判断：一个是“快到”不能只是 distance 小于 500 米，必须有 one_shot 状态机；另一个是，如果 POI 不是当前目的地或途经点，我们到底能不能把它变成导航可计算的点。

我先把我的结论说在前面：这张图里主方案不是传统地理围栏。主方案是基于导航沿路线剩余距离做触发。地理围栏只是备选。如果大家不同意这条主方案，那我们就要一起评估地理围栏，但地理围栏需要额外补很多能力，不能只说一句“换成围栏”。

现在我们开始看图。

## 7. 先看左下角：真实触发来源是导航，不是用户那句话本身

大家先看图的左下角，这里是“高德导航”。

这个需求很容易被误解成：用户说了一句话，系统马上就要播报。其实不是。

用户说“快到外婆家提醒我打电话”的时候，只是在注册一个未来任务。真正触发是在之后，可能是半小时后、也可能是明天，用户真的开始导航，并且导航距离真的靠近了某个目标。

所以左下角这个“高德导航”很关键。它会持续输出导航过程里的信号，比如：

```text
distance: 15000 -> 479
status: 导航中
目的地/途经点信息
```

这里我希望地图和导航同学先帮我确认：我们需要的不是直线距离，而是导航沿路线剩余距离。也就是说，高架旁边直线 300 米但沿路线还要 2 公里，这种情况不能触发。我们一定要用导航路线上的剩余距离。

这里第一个问题就抛给地图和导航：

你们现在能不能稳定输出目的地剩余距离？能不能输出途经点剩余距离？单位是什么？是整数米，还是字符串，还是公里？频率是多少？是不是能做到 1Hz？

## 8. 看 4 号端侧信号库：端侧要先把导航信号标准化

然后大家看图中下方的 4 号模块，端侧信号库。

高德导航出来的原始信号，不能直接丢给触发器用。它需要先被采集、标准化，然后写进端侧信号库。

比如地图侧可能给的是：

```text
"479m"
"0.48km"
479
```

但触发器和状态机最好只认一种格式，比如整数米：

```json
{
  "key": "navigation.destination.remaining_distance",
  "value": 479,
  "source": "amap_navi",
  "timestamp": 1234567890,
  "TTL": 3
}
```

这里沿用我们之前定过的信号库口径：`L8 -> ④端侧信号库`，字段最少要有 `key/value/source/timestamp/TTL`。

所以端侧这里不是简单透传。端侧至少要做三件事：

第一，把地图数据采集过来。

第二，把格式统一成触发器能消费的标准格式。

第三，给后面的 POI 状态机提供稳定的当前快照。

这里请金龙、汪帅评估：地图信号采集到 4 号端侧信号库这条链路，按 L8 跑有没有问题？格式标准化谁做？TTL 怎么定？如果导航信号断了，状态机是停判，还是用最后一次快照？

## 9. 再看左上角 1 和 2：用户语音只是注册入口

接下来我们回到图左侧，1 号“用户语音 query”，2 号“ASR 转文字”。

用户说的是：

```text
快到外婆家提醒我打电话
```

ASR 只是把它转成文字。真正要拆的是后面这几件事：

```text
触发条件：快到外婆家
动作：提醒我打电话
目标 POI：外婆家
阈值：默认 500 米，或产品配置
触发策略：one_shot
```

我这里想强调一点：注册阶段不能只存一句自然语言。必须把它变成规则。否则半小时后用户真的快到了，端侧不知道该盯谁，也不知道阈值是多少。

所以这一路进入 13 号规则配置平台。

## 10. 看 13 号规则配置平台：这里存的是“未来要触发什么”

大家看 13 号规则配置平台。

这里存的不是已经发生的事件，而是未来要监听的规则。

比如这条规则可以理解成：

```json
{
  "condition": {
    "type": "approach",
    "poi": "外婆家",
    "threshold": 500,
    "trigger_type": "one_shot"
  },
  "action": {
    "type": "callback",
    "target": "Advisor",
    "policy": "one_shot"
  }
}
```

注意，这里还没有真正触发。这里只是把“快到外婆家提醒我打电话”注册成一条未来规则。

这也是为什么我说流程图要拆成两段：

第一段是注册段，用户说话的时候发生。

第二段是触发段，用户真的导航快到的时候发生。

如果这两段混在一起讲，大家会误以为用户说完话就要立即算距离。不是这样。

## 11. 看 15 号云端触发器：云端负责规则匹配，不建议每秒跑状态机

然后看 15 号云端触发器。

我的建议是：B 方案下，云端触发器主要做规则匹配和事件分发，不要让云端每秒去跑 POI 状态机。

为什么？

因为导航距离是在端侧产生的。端侧离数据最近，能最快知道 distance 从 15000 变成 479，也能最快处理导航取消、路线重算、离车、弱网这些情况。

如果云端每秒等端侧上报所有距离，再云端跑状态机，会多一层网络延迟，也会把状态同步搞复杂。

所以这张图里我建议的链路是：

```text
端侧状态机先判断出 approach / arrive / leave
再把事件同步云端
云端 15 号触发器只做 event_type 匹配
命中后 push 给 Advisor
```

这里请刘杨帮忙判断：如果端侧已经产出事件，比如 event_type = approach，触发器只做事件匹配、命中规则、push Advisor，这个改动是否可控？eq 运算符能不能直接匹配 event_type 字符串？

## 12. 回到 4 号和状态机：这张图里真正的核心是端侧 POI 状态机

现在讲这张图最核心的地方：端侧 POI 状态机。

我知道图里状态机没有画成一个很大的单独模块，但它实际上夹在“端侧信号库”和“事件同步云端”之间。也就是端侧读 4 号信号库里的导航快照，判断是否要产出事件。

这里我一定要讲清楚：接近不是 distance 小于 500 米。

如果只是写：

```text
distance < 500
```

那车辆从 499 米、498 米、497 米一路开到目的地，每一秒都会满足条件。用户会被一直提醒。

所以“快到”必须是状态迁移：

```text
远 -> 已接近
```

只有第一次从远处跨过阈值，才产出 approach。

我建议端侧状态机至少维护这几个状态：

```text
远
已接近
已到达
已离开
```

判断逻辑是：

```text
state = 远 && d <= 480m && 导航中
=> 产出 approach
=> state = 已接近

state = 已接近 && d <= 50m && 档位=P
=> 产出 arrive
=> state = 已到达

state = 已接近/已到达 && d > 520m
=> 产出 leave
=> state = 已离开

导航取消 / 目的地变更 / 离车 / 重新导航
=> 状态重置
```

这里 480 和 520 是死区。不要用 500 米硬切。因为 GPS 或导航距离可能在 498 和 502 之间来回跳，硬切会反复触发。

这里请端侧同学重点评估四件事：

第一，这个状态机放在哪个端侧模块里。

第二，状态怎么持久化。比如用户今天注册了一个提醒，明天才导航，规则还在不在。

第三，什么时候重置。取消导航、目的地变更、离车、车机重启，各自怎么处理。

第四，端侧产出的事件怎么同步云端，是 L9 还是现有别的链路。

## 13. 看 12 号云端信号库：云端要不要存快照，边界要说清楚

图上中间有 12 号云端信号库。

这里我想请大家一起定边界：云端信号库可以存端侧上报的信号快照，也可以存事件结果，但不要让 12 号变成“每秒状态机”的主要执行点。

我的理解是：

```text
4 号端侧信号库：保存最新导航快照，供端侧状态机实时判断
12 号云端信号库：保存上报快照或事件，供云端匹配、回放、分析
15 号云端触发器：消费事件，做规则命中和分发
```

这里请韩天亮帮忙确认：nav_event 怎么注册？approach、arrive、leave、pass_by 是一个 nav_event 里的 event_type 枚举，还是拆成多个信号？payload 里 distance、threshold、poi_id、trigger_type、timestamp 这些字段要不要都进入云端信号库？

## 14. 看 14 号条件提醒解析：它负责把事件变成 Director 能理解的任务

接着看 14 号，条件提醒解析。

这个模块我理解是把触发条件和后续动作组织成 Director 能消费的任务，尤其是常识判断和任务拆解。

比如 approach 事件来了：

```text
event_type = approach
poi_id = 外婆家
distance = 479
user_action = 提醒我打电话
```

14 号要把它整理成：“用户快到外婆家了，之前注册过提醒打电话，现在应该把这个事件交给 Advisor/Director 判断是否播报。”

这里我希望确认：14 号是只在注册阶段解析一次，还是触发时也会参与解释？如果触发时也参与，那它输入的 payload 要和 15 号触发器、17 号 Advisor 对齐。

## 15. 看 17 号动态 Advisor：这里不是判断快到了，而是生成建议

再看 17 号动态 Advisor。

我希望大家不要把 Advisor 的职责和状态机混掉。

Advisor 不应该负责判断“是不是快到了”。这个判断应该在端侧状态机或触发器事件层完成。Advisor 拿到的应该已经是一个明确事件：

```text
用户快到外婆家了
这个事件已经 one_shot
用户之前注册的动作是提醒打电话
```

Advisor 要做的是生成建议，比如：

```text
建议提醒用户给外婆打电话
```

或者在“快到公司找充电桩”这个 case 里，Advisor 可以建议 Director 调工具找充电桩。

这里请施佳杰确认：Advisor 需要的 callback payload 里，至少要有哪些字段？event_type、poi_id、distance、原始 query、action，这些够不够？

## 16. 看 20 号 Director 和 21 号 Context：Director 负责决定说不说、怎么说、要不要调工具

然后看右侧 20 号 Director 和 21 号 Context。

Director 不是每秒被导航距离驱动。Director 收到的是事件之后的建议。

比如 Advisor 给出：

```text
用户快到外婆家了，之前让系统提醒他打电话
```

Director 再结合 21 号 Context，比如偏好、情绪、记忆、日志，决定：

```text
talk_or_not = true
talk_content = "快到外婆家了，要不要先给外婆打个电话？"
action_list = []
```

如果是“快到公司找充电桩”，Director 可能决定先调工具，再播报：

```text
action_list = [search_charging_station]
```

所以这张图右侧的重点是：触发事件进来以后，Director 才做推理和工具决策。不要把“是否快到”的实时判断放到 Director 里。Director 不适合每秒判断距离。

## 17. 看 22 号工具库：工具调用是触发后的动作，不是触发条件

再看 22 号工具库。

工具库这里包含车控、搜索、导航、音乐、记忆、播报等工具。

对这个需求来说，工具库是触发后的动作层。

比如：

```text
快到公司帮我找充电桩
```

触发条件是 approach。这个已经由端侧状态机完成。后面 Director 决定调工具：

```text
调用工具库搜索充电桩
拿到 tool_feedback
再决定播报
```

所以这里请工具和 Director 同学确认：触发后的 action_list 怎么表达？工具反馈怎么回到 Director？如果工具失败，是否还播报“快到了”？

## 18. 看右上角橙色链路：这是 B 类任意 POI 的最大卡点

现在看图右上角这条橙色链路。这里写的是：

```text
地址匹配
解析为坐标
动态插入途经点
```

这是我今天最想让导航和地图同学评估的地方。

A 类很清楚：POI 已经是目的地或途经点，导航已经在算距离。

B 类难在：POI 不是当前导航目标。

比如用户说“路过徐浦大桥提醒我右转”。徐浦大桥不是目的地，导航引擎默认不会给我一个“到徐浦大桥还有 500 米”的 distance。

那怎么办？

主方案是：先把“徐浦大桥”通过地址匹配或 POI 搜索解析成坐标，再动态插成导航可识别的途经点。只要它变成途经点，导航引擎就开始算它的沿路线剩余距离，B 类就退化成 A 类。

但这里有几个硬问题：

第一，运行中的导航能不能动态插入途经点？

第二，插入途经点会不会改变用户真实路线？

第三，插入途经点用户是否可见？如果用户看见路线被系统偷偷改了，这个体验不能接受。

第四，有没有一种“内部计算点”，只用于算距离，不真的改变导航路线？

第五，如果不能插途经点，有没有接口能计算当前路线到某 POI 的沿路线距离？

如果这些都不成立，那 B 类任意 POI 不要放进第一版承诺。

这里我想把话说直一点：动态插途经点是 B 类 case 的生死问题。请导航和地图今天给明确结论，不能只说“我们再看一下”。如果要再看，也请给时间点和负责同学。

## 19. 这里顺手讲清楚：为什么这不是传统地理围栏

讲到右上角这条链路时，很自然会有人问：为什么不做地理围栏？

我想在图上直接讲清楚。

传统地理围栏是这样的：

```text
给外婆家画一个 500 米圆
车辆 GPS 进入这个圆
触发
```

我们的主方案是这样的：

```text
车辆正在导航
外婆家是目的地或途经点
导航沿路线剩余距离第一次小于阈值
触发
```

差别非常大。

地理围栏看的是车在哪里。我们的方案看的是车沿着当前导航路线是不是真的快到目标。

高架旁边就是最典型的例子。车的直线距离可能只有 300 米，但沿路线还要 2 公里。地理围栏会误触发，我们的导航距离不会。

再比如用户只是路过外婆家附近，但目的地是机场。GPS 围栏可能触发“快到外婆家”，但导航距离方案会先判断外婆家是不是这次导航的目标。如果不是，就不触发“快到外婆家”。

所以我不是说地理围栏不能做。我是说它不适合作为“快到目的地提醒”这个场景的第一选择。

## 20. 如果大家不同意这张图的 B 方案，那地理围栏要补哪些能力

如果今天大家评估下来，认为右上角动态插途经点做不了，端侧状态机也做不了，或者地图距离拿不到，那我们可以退回地理围栏。

但请大家一起把地理围栏需要做的事情说清楚。

第一，地图或端侧要能创建围栏。

最简单是圆形围栏：

```text
POI 坐标 + 半径 500 米
```

复杂一点是多边形围栏：

```text
商圈、园区、桥梁、停车场边界
```

这里请地图同学确认：高德有没有现成围栏 API？如果没有，端侧是否自建？需要地图提供什么基础数据？

第二，端侧要持续拿 GPS 坐标，自己算进出。

这不是导航剩余距离，而是经纬度计算。圆形围栏要算当前位置到圆心距离，多边形围栏要算点是否在区域内。

第三，就算做围栏，端侧还是要做状态机。

进圈一次触发，不能每秒都触发。出圈也要防抖。也就是说，围栏只替代“距离来源”，不能替代 one_shot。

第四，如果要做“路过某地”，可能还不只是点进圈，而是路线和围栏相交。

这就需要路线 polyline 和围栏相交计算。请任军评估运算符有没有这类能力。没有的话，成本要单独算。

第五，围栏方案的体验风险要大家接受。

它会有高架、隔河、绕路、定位漂移的问题。适合“进入园区”“进入停车场”“进入服务区”这种区域事件，但不一定适合“真的快到目的地”。

所以如果切地理围栏，请大家不要只评估“能不能画圈”，还要评估误触发、漏触发、提前量、状态机、防抖、运算符和事件同步。
