Planner SP 结构分析
概述
Planner 的 System Prompt (SP) 是车载 AI 助手最核心的配置文件，定义了模型的角色、能力边界、行为规则和知识储备。本文从模块化视角分析 SP 的组成结构，并探讨 token 膨胀的问题根源。
SP 模块分解
Planner SP 可拆分为以下 7 个功能模块：
模块 1：角色与接口定义
内容：角色定义（"你是豆包，一台 AI 汽车的个人助手"）、三类输入格式（user_query / advisor / tool_feedback）的 JSON schema
大小估算：约 500 tokens
变更频率：极低（产品定位不变则不改）
膨胀风险：无
模块 2：工具定义（Tool Definitions）
内容：18 个工具的完整描述，每个工具包含 tool_name / tool_describe / param / notice 四个字段
大小估算：约 3,500-4,000 tokens
变更频率：中等（新工具上线或功能扩展时更新）
膨胀风险：高 — 每新增一个工具，增加约 150-250 tokens；工具描述冗长，notice 字段重复性高
工具定义是 SP 中 token 密度最高的模块之一。以 search_poi_qa 为例，仅 param 字段的示例文字就超过 100 tokens。
模块 3：输出格式约束与注意事项
内容：
输出 JSON 格式定义（talk_or_not / talk_content / action_list）
注意事项列表（v3 为 12 条行为规则）
大小估算：约 600-800 tokens
变更频率：高（随 badcase 修复持续增加条目）
膨胀风险：高 — 注意事项从 v1 的 10 条增长到 v3 的 12 条，但每条规则的平均长度在增加（GUI 状态管理相关的规则普遍较长）
模块 4：Few-shot 示例库
内容：60+ 个多轮对话示例，覆盖各类工具使用场景
大小估算：约 15,000-18,000 tokens（SP 中最大的单一模块）
变更频率：高（持续新增示例，v2 新增示例 61-64）
膨胀风险：极高 — 这是 SP token 膨胀的首要来源
示例模块的特殊问题：
每个 few-shot 示例包含完整的 input/output JSON，token 开销巨大
POI 查询、路线规划等工具的返回数据（tool_feedback）极其冗长，单个示例可达 300+ tokens
示例数量从 v1 到 v3 持续增长，且几乎不会删除旧示例
模块 5：知识库
包含三个子模块：
5a. 车辆设备知识
空调、座椅、氛围灯、音量、驾驶模式等硬件参数
约 1,500 tokens
膨胀风险：低（硬件参数相对固定）
5b. 工具使用 tips
各工具的使用技巧、边界说明、注意事项
v3 新增大量 GUI Agent 相关规则（type 规范、状态管理、播报规则、登录约束、延迟澄清、混合指令处理）
约 2,500-3,000 tokens
膨胀风险：高 — GUI Agent 规则块是 v1→v3 增长最快的部分
5c. 其他知识
导航技巧、冥想体验、互动游戏、隐私管理、身份推理等场景知识
约 1,200 tokens
膨胀风险：中等（场景知识持续积累）
模块 6：人设与聊天风格
内容：角色性格描述、语音情绪表达原则、其他补充规则
大小估算：约 800 tokens
变更频率：低
膨胀风险：低
模块 7：动态注入模板
内容：舱内视觉感知 {{status}}、目标队列 {{goal_list}}、车辆状态 {{env_info}}、用户记忆 {{memory}}
大小估算：模板本身约 200 tokens，但运行时注入内容可达 2,000-5,000 tokens
膨胀风险：高 — 动态内容随使用时长增长（特别是 goal_list 和 memory）
Token 膨胀问题根源分析
根源 1：Few-shot 示例库无上限增长
SP 从 v1 的 60 个示例增长到 v3 的 64+ 个示例，每个示例包含完整的多轮 JSON 交互。示例模块占 SP 静态 token 的约 50%。且修复 badcase 的主要手段就是"加示例"，形成恶性循环。
根源 2：GUI Agent 规则的叠加式增长
v1 到 v3 的核心差异几乎全部集中在 GUI Agent 相关规则上。v2 新增了 6 个独立规则块（type 规范、状态管理、播报规则、登录约束、延迟澄清、混合指令），每个规则块都包含完整的触发条件、处理规则和禁止行为。这种"发现一个问题就加一段规则"的模式导致工具知识模块膨胀。
根源 3：工具描述缺乏分层
18 个工具的描述全部内联在 SP 中，没有按使用频率或场景动态加载的机制。低频工具（如 image_generate、search_and_control_ai_broadcast）占用了与高频工具相同的 token 配额。
根源 4：注意事项列表的碎片化
注意事项从 v1 的 10 条增长到 v3 的 12 条，每条规则是独立的、非结构化的文本。缺乏统一的决策框架（如"状态机"或"决策树"），导致规则之间存在重叠和潜在冲突。

与 SP 压缩策略的关联
sp-compression-strategy 提出的四个压缩方向与上述分析高度对应：
字段精简 — 针对模块 1 和模块 4 中的 JSON 字段名
静态 + 动态示例拆分 — 直接解决根源 1（示例库膨胀）
知识动态注入 — 解决根源 3（工具描述不分层）和根源 2（GUI 规则叠加）
模块化 — 为上述所有优化提供架构基础
关键补充：除了 sp-compression-strategy 中已有的四个方向，还应考虑注意事项的结构化重构 — 将碎片化的注意事项列表重构为决策树或状态机模型，用更少的 token 表达更清晰的逻辑。
