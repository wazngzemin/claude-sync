一、背景：我们要做什么（1 个例子讲清）

用户说："后排宝宝哭了告诉我"。系统要做三件事：

步骤要做什么谁来做
① 盯着持续观察后排宝宝端侧 VLM（视觉大模型），舱内高频每 5 秒看一次
② 判断判断"宝宝到底哭了没有"本次要对齐的环节——模型解析
③ 反馈哭了就语音告诉用户Task Service（任务服务）触发 → Planner（规划器）播报

难点在第②步。因为 VLM 只输出对画面的客观描述，不直接下"哭了 = 是/否"的结论。同一个宝宝，VLM 每 5 秒可能这样上报：

第 1 次："后排儿童安静，闭眼休息"　→ 没哭
第 2 次："后排儿童睁眼，面部平静"　→ 没哭
第 3 次："后排儿童睁眼，脸上有泪珠，嘴角向下"　→ 这算哭了吗？谁来判？

把"有泪珠、嘴角向下"这类描述，变成"哭了/没哭"的判定——这就是"模型解析"要解决的事，也是本次讨论的全部范围。

二、整条链路（对照端云架构图逐步走，模块与真实图一一对应）

先认全两侧的模块（和架构大图对应），再按 下发 → 观察上报 → 判断反馈 三个阶段逐步走。每一步都写清"哪个模块、收到什么、做什么、给谁"。

侧本链路涉及的模块
端侧AIBox（含 任务优先级管理 + 端侧感知模块 VLM）、视频流处理、AI Service 消息处理模块（下行通道）、AI Service 数据中心（上行通道）、SLS 舱内 AlwaysOn 开关
云侧Task Service（任务服务，含动态感知模块）、云端端状态、二次组织、DT（对话推理引擎 Director）、判断环节（模型解析——本次讨论对象）

阶段 A · 任务下发（云 → 端）：用户一句话，变成 VLM 的观察任务

步骤 1 ·【云】用户 query（用户指令）进入"后排宝宝哭了告诉我" → 进入云侧，路由到 Task Service（任务服务）
↓
步骤 2 ·【云】Task Service · 动态感知模块 —— 生成高频任务UP根据用户 query 生成 高频任务UP（Task Service 职责 1），产出下发结构 dynamic_watch_tasks：任务编号 + 感知目标。举例：{"任务编号":"8","感知目标":"二排右侧的宝宝醒了吗？"}（真实下发样例格式，见本节末代码块）
↓ 动态感知模块下发（红色回路，跨云→端）
步骤 3 ·【端】AI Service · 消息处理模块 —— 接收并透传端侧接收下发的高频任务UP，透传给 AIBox（不做解析、不做改写）
↓ 透传高频任务UP
步骤 4 ·【端】AIBox · 任务优先级管理 —— 排队高频任务UP 和其他任务（SLS 应用触发的 oneshot 单次任务 / 周期推理）统一在这里排优先级，决定 VLM 先执行谁
↓
步骤 5 ·【端】端侧感知模块 VLM —— 挂载观察任务VLM 挂上编号 8 的感知目标，配合视频流处理送进来的画面，开始每 5 秒观察一次二排右侧。前提：SLS 舱内 AlwaysOn 开关处于开启（开关经 AI Service 传入）

阶段 B · 观察与上报（端 → 云）：VLM 拆两框，走两条订阅线

步骤 6 ·【端】VLM 每 5 秒输出一次，拆成两框、提供不同订阅 Topic（主题）🟥 红框 = 高频任务UP结果：动态任务[{"任务编号":"8","识别结果":"后排儿童睁眼，脸上有泪珠"}]
🟩 绿框 = Context 结果：乘客列表[{"位置":"副驾后方","动态":"头依靠在中间乘客肩膀上睡觉"}]
↓ 按 Topic 订阅
步骤 7 ·【端】AI Service · 数据中心 —— 两框都收订阅 🟥 高频任务UP结果（红框）+ 🟩 Context 结果（绿框），然后分两条路往云上走 ↓
↓
步骤 8a ·【端→云】context 路线：模型原始输出整包上传按端状态协议上报 → 云端端状态 → 二次组织（把整包拆成子类）→ 供 DT（对话推理引擎）/ Context（上下文）消费。沿用 7.8 评审已定的整包上传链路，不新增。
↓
步骤 8b ·【端→云】动态任务路线：透传高频任务UP结果（红框）红框沿专线（紫色通路）透传给云侧 Task Service——这是"判断达成没达成"的输入来源，不走二次组织

阶段 C · 判断与反馈（云）：本次讨论的主角在这里

步骤 9 ·【云】判断环节（模型解析）—— 拿红框比目标输入 = 红框"识别结果"（"睁眼，脸上有泪珠"）+ 任务的"感知目标"（"宝宝哭了吗"）。判断是否达成（Task Service 职责 4；A 标签路线=比对枚举值，B 语义路线=轻量模型判断，见第三节）。
没达成 ↺ 等下一条 5 秒上报，再判；达成 → 往下走。（该判断的具体承载模块，本次对齐，见第九节 #10）
↓ 达成 → callback（回调）
步骤 10 ·【云】Task Service —— 控制反馈节奏接收高频任务UP的周期返回结果（职责 2）；按用户 query 里的反馈周期反馈（职责 3）——比如"每 60 秒告诉我宝宝在干嘛"，就算 VLM 5 秒报一次，也只每 60 秒反馈用户一次。两个周期解耦：5 秒 = 喂判断素材的频率，60 秒 = 打扰用户的频率
↓
步骤 11 ·【云→用户】播报 + 收尾反馈用户"后排宝宝哭了"。任务目标达成后，Task Service 通知 VLM 结束其下发的高频任务UP（职责 5），端侧停止观察，链路闭环

两条与本链路并行、但不在本次讨论范围的支线（架构图里有，列出来防混淆）：
① SLS 闭环业务：SLS 应用触发 oneshot（单次任务，由调用方备图）或周期推理（由弋途拉流），触发调用结果返回给 SLS——赛力斯侧自闭环，只在 AIBox 的任务优先级管理处与我们的高频任务UP汇合排队。
② DT 的 VQA（视觉问答）分支：舱内 VQA 待定；舱外 VQA 和舱外长时任务暂不走端侧 VLM。本材料只覆盖"舱内高频 AlwaysOn + 动态任务"这一条。

三、"判断"怎么实现：两条路线 + 分层建议

对比路线 A：标签提取路线 B：语义匹配
做法预定义标签（如"哭闹/平静/睡眠"），VLM 从画面提取标签，判断环节比对枚举值即可不预定义。VLM 输出自然描述，判断环节用一个轻量模型判断"描述是否符合目标语义"
适合高频、稳定、边界清晰的场景（抽烟 / 哭闹 / 安全座椅误装）用户随口说的长尾需求（"毯子滑落了告诉我"）——没法提前穷举成标签
要准备标签清单（第四节初稿）+ caption 支撑核对（第五节）+ 种子数据（第六节）判断模型 + 少量样例
短板新需求要加标签、重新准备数据，不灵活模型有 badcase（误判样例），准确率要打磨
◎ 建议分层，两条路线共存：高频场景走 A（快、准、省），长尾场景走 B（保证用户想到什么都能试着支持）。第四节的标签清单初稿里，每个场景都标了建议走哪条。

四、问题① 初稿：要提取哪些标签（源自《视觉标签需求-触发器评估》31 条，逐条整理，未删减）

这张表怎么读（先看这个，把原表 6 类事件范式收敛成"标签 + 触发方式"两件事）：
原表的"开始X / 结束X / 出现X / 移除X / 存在X"，不需要各做一个标签。设计收敛为：
① VLM 只输出【状态标签】——每帧报当前状态（哭闹/无、睡着/清醒、有/无），VLM 的活最简；
② "开始 / 结束 / 出现 / 移除" = 触发引擎用事件运算符"变更为"抓跨帧变化——原表反复强调"非单帧判断、要监听变化"，而变化检测正是触发引擎的本职（现有条件 8 运算符 + 事件 2 运算符"变更为 / 有变更"）；
③ 瞬时动作例外（比 OK 手势 / 打哈欠 / 手伸出窗外）：单帧可判，检出即报事件；
④ 时长条件（"长时间分心""红灯超 90 秒"）：由触发引擎计时，不要求 VLM 判时长。
这样 31 条需求收敛成约 15 个状态标签 + 统一的"变更为"事件机制。

六、问题③ 初稿：种子数据怎么构建

种子数据 = 训练/调试判断能力用的样例集。初步定义每条种子数据长这样：

{ "场景": "儿童哭闹", "用户query": "后排宝宝哭了告诉我", "正例caption": ["后排儿童睁眼，脸上有泪珠，嘴角向下", "后排儿童张嘴大哭，面部涨红"], // 应判"达成" "负例caption": ["后排儿童闭眼安睡", "后排儿童睁眼，面部平静"], // 应判"未达成" "边界例caption": ["后排儿童揉眼睛，表情不明"], // 难判样例，用于定阈值 "期望判定": "正例=达成；负例=未达成；边界例=未达成（宁漏判不误判）" }
分工内容
产品提供每个场景的：用户 query 示例、期望判定口径（什么算达成/不算）、边界 case 的产品判断（如"揉眼睛不算哭"）
算法构建按上述口径生成/标注 caption 样例（正例/负例/边界例）；数量口径由算法定（建议每标签正负各若干十条起步 具体量算法拍）
研发使用用种子数据调试判断环节，跑准确率基线
判定策略的产品口径（初稿）：宁可漏判，不要误判。理由：车内误报（没哭说哭了）反复打扰用户，伤害大于漏报（晚一轮 5 秒再报）。边界例一律判"未达成"，等下一帧更明确的描述。——这条口径请大家确认。
