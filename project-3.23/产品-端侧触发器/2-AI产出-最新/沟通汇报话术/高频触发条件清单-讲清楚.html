= 周会"实车信号可获取清单，必须与触发器一致"（项4）＋ 需求1 总源头 ｜ 王泽民 · 火山侧

一句话：这张清单 = 告诉系统"什么情况下自动做什么"的总说明书。 它是触发器一切的地基 —— 配置平台照它录、弋途照它订阅、意图照它训模型、益红照它配主动服务、实车照它测。清单里少一条/编号错/单位错，后面五方全错。所以它是 P0 总源头。

② 一个真实 case 跑逻辑（像那张步骤图，分"配置期"和"运行期"两段）

就拿上面那条「车内温度 > 26℃ → 提醒/开空调」，看它从你清单一路怎么落地：

🅰 配置期：你这一行清单，喂给五方各自干活
  1. 你在清单写下这一行：车内温度 / temperature / SIG-T-012 / 数值阈值 / 大于 / 26℃ / temp>26
  2. 刘杨（火山产研）拿这行 → 火山配置平台：signals 注册 SIG-T-012 → 新建场景"温度过高"：动态条件 temperature > 26
  3. 意图·婉婷拿这行 → 当约束词表，训模型把"太热了""超过26度"解析成 {车内温度 > 26}
  4. 弋途·佳杰（动态advisor）拿整张清单 → 知道"车内温度可订阅" → 订阅这个条件事件（项4"必须与触发器一致"就是这一步：他能订的，必须是你清单里真有、配置平台真注册了的）
  5. 益红拿这行 → 配主动服务预设场景"温度过高关怀"
  6. 测试拿这行 → 实车测"温度真能触发吗"，标 真能触发 / 未部署
🅱 运行期：实车跑起来（这段就是那张步骤图的逻辑）
  1. 车身传感器测到舱内 35° → 写入 端侧信号库（SIG-T-012 = 35）
  2. 端侧信号库 增量同步到云端信号库（延迟 <1s）
  3. 云端触发器 每 5s 扫描 → 判定 35 > 26 命中 → 频控通过（上次触发已过 cooldown）→ 无冲突 → Callback
  4. Callback 路由：事件写入 Context + 透出给 advisor（弋途）
  5. advisor 消费 → 产出 advice "建议开空调26度"
  6. → Director 推理"没更高优先级" → 调 vehicle_basic_control 开空调 → 播报"帮你把空调调到26度了"
看懂没：你清单里这一行，是上面 12 步全部的地基 —— 编号错（SIG-T-012 写错）→ 第2步刘杨注册不上；单位错（26写成26档）→ 第3步运行期判错；清单里没这条 → 弋途第4步根本订不到。这就是为什么它叫"总源头"。
