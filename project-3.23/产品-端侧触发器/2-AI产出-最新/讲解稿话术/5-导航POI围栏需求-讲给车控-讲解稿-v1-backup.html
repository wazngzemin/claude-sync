01

开场：今天要对什么

1 min

"今天讲一个需求：导航 POI 靠近触发。一句话——用户在车里说'快到某个地方提醒我干点啥'，系统得能盯着、到了提醒。这里面要在端侧建一套地理围栏，这就是今天主要跟车控对的事。

"我按四块讲：需求是什么、为什么围栏放你们端侧、围栏具体做成什么样、最后要你们确认和评估什么。讲完大概 25 分钟，中间随时可以打断。"

提醒开场别铺背景，30 秒内把"要你们建围栏"这句话扔出去。技术团队先知道今天要谈的是什么，后面才听得进去。
02

需求是什么（缺口）

3 min

"现在触发器支持不了'快到目的地''路过某地'这一类。举几个真实用户会讲的话：

——'快到外婆家提醒我给她打个电话'。
——'快到公司提醒我找个充电桩'。
——'路过徐浦大桥提醒我待会右转'。

"这种话现在的状态是：没法注册、没法监测、没法触发，整个链路是断的。这期要补的就是'接近 / 到达 / 离开'这套判断。"

两类场景要分开说清楚

A 类 · 导航目的地 / 途经点B 类 · 任意 POI（随口说的地点）
POI 和导航的关系就是当前导航的目标不在导航路线上（"外婆家"可能压根不在路线上）
距离谁在算导航引擎已经在实时算剩余距离导航没在算，没人知道车离它多远
例子"离目的地还有 1 公里提醒我拿咖啡""快到外婆家提醒我打电话"
端侧要补什么只差一个"过程判断"（状态机）坐标解析 + 围栏 + 状态机，一整套

"这两类我用同一套围栏机制兜。A 类简单，导航已经把剩余距离算出来了，我在上面加一个过程判断就行。B 类是真正缺的——导航不管这个点，那就得给它画一个圈，端侧盯着车进没进这个圈。这个圈就是地理围栏。

"另外一个难点我要专门点出来：'接近'不是'距离小于 500 米'这么一句。它是个过程判断——得记住上一次的状态，得知道车在变近还是变远，触发过一次就不能再触发。这个就是后面讲的状态机加只触发一次的锁。"

03

为什么围栏要建在端侧（不是云端）

2 min

"先堵一条弯路——为什么这个判断不放云端做。云端做就意味着端侧把车的 GPS 坐标不停往上传，云端来算距离、判进出。这条路两个死结：

"第一，坐标出车。车的实时位置是敏感信息，要出车就得过数据安全合规审批，流程长、还不一定过得了。

"第二，频率。围栏要实时，就得每秒往云上传一次坐标。这要专门建一条高频上报链路，云端还得扛这个量。难度和成本都不值。

"所以围栏判断只能放端侧：实时位置本来就在你们车上，就地判断最省，坐标也不用出车。云端只在'真的到了'那一下收一个事件。这就是我找车控做这件事的原因——不是把活推给你们，是这活只能在你们身上。"

预期反应这条是说服车控接活的正当性，讲清楚"坐标不出车、只报一次事件"，他们最容易接受的点就是数据不用离开车。
04

围栏具体做成什么样（核心）

10 min

"这是今天的重点。端侧要建一个围栏模块，拆成五块。我一块块讲它干什么。

1
指令接收器
收云端推下来的围栏定义（一段 JSON），解析出来。JSON 里带任务 id、圆心坐标、半径、策略。
2
围栏存储
把围栏存在本地、持久化，车机重启不能丢。存坐标、半径、围栏 id、当前状态。
3
距离轮询器
一个定时器，每秒（或降频）去问一次"车离这个点多远"——调地图 SDK 的距离查询，或读导航引擎的剩余距离字段。
4
围栏状态机
核心。维护"远 → 已接近 → 已到达 → 已离开"，带只触发一次的锁。判断进出、防抖动。
5
事件上报器
进出圈那一下，把事件报给云端。失败要重试。上报量极小，一个围栏一辈子就报一两次。

① 云端推给端侧的围栏定义（下行）

{
  "cmd": "create_fence",
  "task_id": "t_20260720_001",
  "fence_id": "f_001",
  "center": [121.4737, 31.2304],   // 圆心经纬度（十进制度）
  "radius": 500,                          // 半径，单位米，纯数字
  "poi_id": "B001234",
  "poi_name": "外婆家(徐汇店)",
  "policy": "one_shot"                     // 只触发一次
}
格式要求半径用米、纯数字，不要 "500m" 这种带单位的字符串（剥单位是个老坑）。经纬度用十进制度、6 位小数。

② 端侧报回云端的事件（上行）

{
  "event_type": "fence_enter",   // fence_enter 进入 / fence_arrive 到达 / fence_leave 离开
  "fence_id": "f_001",
  "task_id": "t_20260720_001",
  "poi_id": "B001234",
  "timestamp": "2026-07-20T14:58:25Z"
}

③ 围栏状态机（这是和普通"距离判断"的区别）

远——d≤480m→fence_enter 已接近——d≤50m 且 P档→fence_arrive 已到达——d>520m→fence_leave 已离开

"状态机每秒跑一次判定，但注意——绝大多数时候它什么都不干。它干三件事：记住上一次状态、看距离有没有穿越阈值线、穿越了就翻转状态并报一个事件。翻转过一次就锁住，不再重复报。这就是 one_shot，只触发一次。

"具体怎么穿越，我拿阈值讲。'接近'不是一根 500 米的线，是两条线加一个死区：小于 480 米才算真的接近，大于 520 米才算离开，中间这 40 米是缓冲区、什么都不做。为什么——因为 GPS 会抖。等红灯的时候距离可能在 498、502 之间来回跳，要是阈值是一根线，就会'到了、没到、到了、没到'反复触发。死区把这个问题挡掉——必须真正穿过 480 才算到。"

④ 三个异常处理（车控一定会问，主动先讲）

问题场景处理
GPS 抖动路口等红灯，距离在 498↔502 反复跳迟滞：<480 才算接近、>520 才算离开，中间死区不动作
立体空间高架旁边直线才 300 米，沿路线要绕 2 公里一律用沿路线距离，禁用直线距离
导航中途取消导航中状态变"未发起"状态机前置检查不过，停止判定
阈值不一加油站 2km、咖啡 500m，不同场景阈值不同阈值按场景由产品配，给默认值，配置平台支持

⑤ 距离从哪来（要地图 SDK 给的）

"轮询器每一跳要拿一次'车到这个点多远'。这个数据从地图 SDK 来——你们有一个'查当前车到某个 POI 的距离'的接口，我传一个 POI 或坐标，你返回距离。导航中、并且这个点在路线上的时候，优先用导航引擎的沿路线剩余距离，更准。

"这里有几个接口问题我要跟地图同学确认，放到最后一节一起说。"

05

一次完整的触发走一遍

4 min

"我拿'用户从 15 公里外开去外婆家'这条时间线走一遍，大家看状态机在每个时刻干什么。重点看两点：平时零开销，到点不遗漏。

14:30:10围栏建好，定时器启动。d=15000m，状态=远，什么都不干。
14:30:11d=14980，状态=远，什么都不干。
…（中间 29 分钟，每秒查一次，都是"远"，都不干活）…
14:58:20d=530，状态=远，还没到，不干活。
14:58:22d=510，进死区（480~520），不动作。
14:58:25d=485，还在死区，不动作。
14:58:26d=479，≤480！穿越 → 状态"远"翻成"已接近"，产出 fence_enter，打 one_shot 锁，上报云端。
14:58:27d=470，状态=已接近，但已锁定，不再触发。
…（之后每秒还在看，但已锁定，不会再报第二次）…

"看清楚——用户开半小时，定时器跑了一千八百次，前一千七百多次都是'远'、什么都不干，只有最后穿过 480 那一下才真正干活。这个设计就是平时零开销、关键时刻不遗漏。

"事件上报到云端之后，后面那条链路——引擎匹配、生成话术、语音播报——完全是现有链路，一行不用改。你们端侧要动的，就只有'距离从哪来'和'谁来判进出'这两块。"

车控可能问d=479 那一下，判定发生在端侧还是云端？
全部在端侧。定时器、查距离、状态机判定，这三步都在端侧本地、不经过网络。只有最后"上报事件"才联网，而且整个围栏一生就报一两次，不是每秒都发。
车控可能问距离都是正数，怎么判"离开"？
看趋势。先减后增就是先靠近再远离——到了变成 0，离开又涨回 500，涨过 520 就是离开。不过这一期做的是 one_shot，只要"接近"这一个事件，触发一次就完，不需要判离开。离开是后续 repeat（重复提醒）场景才要的。
06

生命周期与边界情况

2 min
情况端侧怎么处理
正常触发闭环播报完，停掉这个围栏的定时器，从本地存储删掉围栏记录；云端同时清规则
用户中途取消云端下发 delete_fence，端侧删围栏、停轮询
车机重启围栏存在本地持久化里，重启后读一遍本地存储，发现活跃围栏就重启定时器继续盯，不丢用户的提醒
同时多个提醒围栏模块维护一个围栏列表，不是单个。定时器每跳遍历列表，每个围栏查一次、判一次。3~5 个完全没问题
顺带确认围栏数量特别多（10 个以上）每秒要调多次 SDK，可能有性能压力——这个归到下面"要确认的事"里问数量上限。
07

要车控确认和评估的事

3 min

"最后，要你们给我答复的几件事。尤其地图 SDK 那几个，会直接决定我这边能做到什么程度。

地图 SDK（距离查询接口）

  • 调用频率限制：每秒 1 次 / 5 次 / 更低？给个数字或范围就行
  • 入参传什么：poi_id？经纬度？要不要带当前车位置？
  • 返回的是直线距离还是沿路线距离？带不带单位？
  • 非导航状态下这个接口能不能用

端云两条通道

  • 云端推围栏 JSON 给端侧：复用现有端云长连接，还是新建？（数据很轻、一次性，倾向复用）
  • 端侧上报事件：能否复用现有信号上报链路？（上报量极小，一围栏 1~2 次）

端侧围栏模块

  • 五个模块的整体工作量和排期
  • 距离轮询器：现有导航 SDK 有没有暴露"剩余距离"字段？读取接口长什么样？
  • 同时生效的围栏数量上限？
  • 定位精度、从进圈到上报的延迟大概多少？（决定能不能用在高速场景）
  • 圈是不是只能正圆？以后要不要支持不规则形状？接口上要不要先留口子？
08

收尾（30 秒）

0.5 min

"总结一下。要补'到某个地方就触发'，导航已经在算的那种加个过程判断就行；导航不管的任意地点，就得在端侧建一套地理围栏——云端推定义下来，你们端侧存着、每秒查距离、状态机判进出、到了报一个事件。坐标不出车，后面播报链路全复用。

"要你们回去评估的是端侧那五个模块的工作量，还有地图 SDK 距离接口那几个问题。我们对个排期。有做不了或者更省的做法，现在提。"

收尾心法说完"现在提"就停住，别再自己往下补。让车控先开口——他们当场蹦出来的"这个 SDK 不支持""这块排期紧"，才是你今天真正要谈成的东西。
