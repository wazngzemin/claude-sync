AlwaysOn 开关链路详解

舱内 / 舱外 · 上电下电 · Button 开关 · 全流程逐段说清 · v1 · 2026-07-08
本文只讲一件事：端侧触发器怎么开关 AI-Box 上的 AlwaysOn 周期任务。

覆盖：舱外 AlwaysOn（只看上下电）+ 舱内 AlwaysOn（看上下电 AND Button）+ 开关机制（start/stop 怎么走）+ 开=AND 关=OR 的逻辑本质 + Button 的作用范围（关着时用户 query 和系统预设任务分别怎么走）。

§0 先读懂三根线

丁彬架构图上，跟"开关"直接相关的就三根线。所有开关动作都在这三根线上来回，先把它们认清楚，后面全程用它们：

    PRIO(("任务优先级
管理"))

    PRIO --> VLM
    VLM -->|"③ 触发&工作状态返回：状态回调"| TRIG

线方向是什么你要记住
①Button → 端侧触发器用户的舱内使能开关，作为一个信号进来你只读它，永远不去操作这个按钮
②端侧触发器 → 任务优先级管理你下发的 start / stop 指令你所有"开关"动作都走这根线——开就 start，关就 stop
③VLM → 端侧触发器VLM 回给你的状态（开没开成 / 每轮成功）你靠它知道任务到底跑起来没
第③行下面那条虚线（VLM → 数据中心 → 云端）是数据通道，走 VLM 的识别结果，跟"开关"是两码事。本文只聊开关（①②③实线），数据通道不展开。

§1 舱外 AlwaysOn

舱外前视摄像头 · 3 秒周期 · 只看一个条件：上下电。跟 Button 完全无关——这是它和舱内最大的区别。

逐步说明

🔌 上电（开）

1. 用户上车，整车上电 → 信号库里 power_state 变成 on
2. 【端侧触发器】读到 power_state = on
3. 判断舱外开条件 = power=on（单条件）→ 满足 ✅
4. 【端侧触发器】沿 ② 触发推理 线，向【任务优先级管理】下发：
start(task_id=舱外AlwaysOn, camera=前视, 周期=3s)
5. 【任务优先级管理】排好优先级 → 交给【端侧感知模块 VLM】
6. 【VLM】开始每 3 秒看一次前方（天气/路况/障碍物），进入周期循环
7. 【VLM】沿 ③ 触发&工作状态返回 线，回给你"已启动 / 本轮 success"

🔌 下电（关）

1. 用户熄火，整车下电 → power_state 变成 off
2. 【端侧触发器】读到 power_state = off
3. 判断舱外关条件 = power=off（单条件）→ 满足 ✅
4. 【端侧触发器】沿 ② 触发推理 线下发：stop(舱外AlwaysOn)
5. 【VLM】停止舱外周期任务，不再跑
舱外结论：开 = 上电，关 = 下电，一根 power 说了算，Button 完全不参与。

§2 舱内 AlwaysOn

舱内舱内摄像头 · 动态 5 秒 + 静态 5 分钟 · 要看两个条件：上下电 AND Button。这是全文最核心的一块。

规矩：开 = power=on AND button=on（两个都要满足）；关 = power=off OR button=off（任一不满足就关）。

AND / OR 决策流程图

    READ --> Q{"power=on 且 button=on ?
【开 = AND】"}
    Q -->|"是：两个都满足"| OPEN["开条件成立"]
    OPEN --> C1["下发 start（舱内动态·5s）
+ start（舱内静态·5min）"]
    C1 --> RUN["VLM 按 5s / 5min 周期跑"]
    Q -->|"否：缺任意一个【关 = OR】"| CLOSE["关条件成立"]
    CLOSE --> C2["下发 stop（舱内动态）
+ stop（舱内静态）"]
    C2 --> STOP["舱内不跑
（舱外只看 power，不受影响）"]

三状态机（一张图看全 5 个场景）

三个状态：全关（下电，什么都不跑）、仅舱外（上电但 Button 关：舱外跑，舱内不跑）、全开（上电且 Button 开：舱外+舱内都跑）。下面把每条转移线（情况 A~E）走清楚。

情况 A~E 逐个走清楚

情况 A：上电，且 Button 本来就是开的 → 开

1. 上车上电 → power=on
2. 【触发器】读 power=on，再读【SLS舱内always on开关】→ button=on
3. 开条件 = power=on AND button=on → 两个都满足 ✅
4. 沿 ② 触发推理下发两条 start：start(舱内动态,5s) + start(舱内静态,5min)
5. 【VLM】按 5s / 5min 两个节奏跑，状态沿 ③ 回给你

情况 B：上电，但 Button 关着 → 不开

1. 上车上电 → power=on
2. 【触发器】读 button=off
3. 开条件 = power=on AND button=off → AND 不满足 ❌
4. 【触发器】不下发舱内的 start，舱内两个任务都不启动（舱外该开还是开）

情况 C：行驶中，用户把 Button 打开（power 早已 on）→ 开

1. 当前 power=on，舱内没跑
2. 用户在设置页开 Button → 【SLS舱内always on开关】把 button: off→on 的变更通知你
3. 【触发器】重算开条件 = power=on AND button=on → 现在满足了 ✅
4. 下发 start(舱内动态) + start(舱内静态)

情况 D：行驶中，用户把 Button 关掉 → 关舱内（舱外不动）

1. 当前 power=on，舱内正在跑
2. 用户关 Button → button: on→off，通知你
3. 【触发器】算关条件 = power=off OR button=off → button=off 满足 OR ✅
4. 下发 stop(舱内动态) + stop(舱内静态)
5. 舱外不受影响——舱外只看 power，power 还是 on，继续跑

情况 E：下电（不管 Button 是开是关）→ 全关

1. 熄火下电 → power=off
2. 【触发器】算舱内关条件 = power=off OR button=off → power=off 满足 OR ✅
3. 下发 stop(舱内动态) + stop(舱内静态)
4. 同时舱外也因 power=off 被关
5. 三个 AlwaysOn 全停
短暂下电再上电（熄火重启）：先全关（下电时）→ 再次上电重走点火时序。若此时 Button 仍为 on，则三个全开；若 Button 为 off，则只开舱外。下电不管时长，一律先关。

§3 开=AND 关=OR 的逻辑本质

为什么开要 AND、关要 OR？不是随便定的，背后是"权限 + 授权"的逻辑：

含义为什么这么设计
开 = ANDpower 和 button 都满足才开要开始看舱内，得同时具备两样东西：前提（power=on，车在用）+ 授权（button=on，用户允许）。缺一不可——车没上电没法看，用户不允许也不能看
关 = OR任一不满足就关要停下来，只要一个理由就够：要么断电（车不用了）要么用户撤回授权（关 button）。满足一个就该立刻停，不能等两个都不满足
一句话记牢：开舱内，要凑齐两把钥匙（power + button）；关舱内，丢一把就够了。舱外只有一把钥匙（power）。

§4 开关机制（start / stop 时序）

前面反复说"下发 start / stop"，这里把这个动作在时间轴上展开，看清楚你和 VLM 之间来回几趟：

sequenceDiagram
    autonumber
    participant T as 端侧触发器（你）
    participant P as 任务优先级管理
    participant V as VLM
    participant D as 数据中心
    Note over T: 开条件成立（上电 / Button开）
    T->>P: ② 触发推理：start(task_id, camera, 周期)
    P->>V: 排优先级后启动任务
    V-->>T: ③ 状态返回：已启动
    loop 每个周期（3s / 5s / 5min）
        V->>V: 推理一帧
        V-->>T: ③ 状态回调：success / empty
        V->>D: 订阅结果返回（数据通道）
        D->>D: 整包透传云端
    end
    Note over T: 关条件成立（下电 / Button关）
    T->>P: ② 触发推理：stop(task_id)
    P->>V: 停止任务
    V-->>T: ③ 状态回调：已停止
动作你下发什么发生什么
开start(task_id, camera, 周期)VLM 启动该任务并进入周期循环，每轮回一个状态给你
关stop(task_id)VLM 中止该任务，回一个"已停止"状态
关键：这个开关是按任务（task_id）分别开关的，不是一个总闸。所以 Button 关的时候，你能精准只 stop 舱内那两个、留着舱外继续跑——因为你是对着具体的 task_id 喊 stop。
会议依据：斯华——VLM SDK 提供 init / start / stop 控制接口；丁彬 7.7——"他有一个接口就把这个任务停就行了，我任务注销了就可以了……他就不跑了"。

§5 Button 的作用范围

核心原则：Button 只是"舱内 AlwaysOn"这一个任务的开关，不是"舱内摄像头总闸"。其他舱内任务各有各的使能条件，Button 一律不管。

为什么？因为它就叫 AlwaysOn Button——名字写死了它管的是 AlwaysOn，不是"所有用舱内摄像头的事"。下面用决策树看它到底管到哪。

任务分流决策树

    S([一个要用舱内摄像头的任务来了]) --> Q{"这个任务是什么类型？"}
    Q -->|"系统自发背景扫描
舱内AlwaysOn 5s/5min"| B1{"button = on ?"}
    B1 -->|是| R1["跑"]
    B1 -->|否| X1["停 ★ Button 只在这里生效"]
    Q -->|"用户主动 query
『帮我看宝宝』"| R2["跑
用户开口=授权，不看 Button"]
    Q -->|"系统预设·儿童照看 4-6"| B2{"儿童照看开关 = on ?"}
    B2 -->|是| R3["跑（看它自己的开关，不看 Button）"]

整棵树里，只有最左边那条分支（舱内 AlwaysOn）会去问 Button。其余每条分支都有自己的判断，Button 管不着。

Button 关时：用户主动 query 怎么走

场景：Button 关着，用户说"帮我看看后排宝宝坐好没"

1. 用户开口 → 语音上云 → 识别为一次性视觉问答（VQA）
2. 系统判断：这是用户主动请求，不是系统自发的背景扫描
3. 走 VQA 链路 → 直接调 VLM：camera=舱内，看一眼宝宝
4. VLM 执行 → 结果返回给用户
关键：全程不看 Button，照样执行。 Button 管的是"系统偷偷一直看"，而这是"用户明确开口要你看一眼"。用户自己开口 = 自己授权，Button 管不着。总不能因为用户关了背景监控，就拒绝他"帮我看看宝宝"的直接请求。

Button 关时：系统预设任务怎么走

系统预设的舱内任务不止一个，每个有自己的使能条件。Button 关着时它们各走各的：

儿童照看（4-6）

使能条件 = 用户单独开启"儿童照看"功能（它有自己的开关）
Button 关 + 用户开了儿童照看 → 跑（用它自己的开关，跟 AlwaysOn Button 无关）
Button 关 + 用户没开儿童照看 → 不跑（是它自己没开，不是 Button 拦的）

遗留物品检测（1-5，P0 安全）

使能条件 = 用户离车（下电锁车）自动触发，一次性检查
这是 P0 安全兜底——防止把孩子/宠物/贵重物品落在车里
Button 关 → 建议照跑。安全类不该被隐私开关关掉（把孩子落车里是人命）

安全提醒（4-5）

使能条件 = 行车中首次 P→D 自动触发，持续观察舱内不安全行为
这个性质最接近 AlwaysOn（持续舱内监控），是唯一有争议的 → 见下方决策点

总表 + A/B 决策点

舱内任务谁使能它Button 关时为什么
舱内 AlwaysOn（动5s/静5min）就是 Button 自己停Button 就是它的开关
用户 query（看宝宝）用户开口跑用户主动 = 授权
儿童照看（4-6）它自己的开关看自己开关独立 toggle，与 Button 无关
遗留物品（1-5,P0）离车自动·安全建议跑安全兜底，防落孩子
安全提醒（4-5）P→D 自动·安全决策点性质接近 AlwaysOn

规律：表里只有第一行受 Button 管，因为只有它是"纯背景 always on 扫描"。其余每个都有自己的使能条件。

你要拍板的一个点：Button 是"AlwaysOn 专用开关"还是"舱内摄像头总闸"？

定义Button 关 = 影响
A. AlwaysOn 专用开关（推荐）只停那 3 个 AlwaysOn安全类（遗留物品/安全提醒）、用户 query、儿童照看照常
B. 舱内摄像头总闸舱内摄像头一律禁用连遗留物品（防落孩子）都不看了 —— 安全风险
强烈建议 A。若选 B，用户关了 Button 后，连"把孩子落车里"这种 P0 安全兜底都失效，是拿隐私换人命，不划算。Button 应只管"系统别一直闲着盯我"，不该把安全功能一起关掉。
唯一模糊的是"安全提醒（4-5）"——它是持续舱内监控，归"安全类不受约束"还是"持续监控类受约束"，建议评审时跟安全/隐私团队单独确认，写进 PRD。
