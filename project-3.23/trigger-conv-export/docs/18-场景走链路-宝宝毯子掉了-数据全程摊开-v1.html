场景走链路 · "宝宝毯子掉了提醒我" · 数据全程摊开

2026-07-05 · 用一个具体场景把整条动态链路走到底 · 每一步:发生什么 / 此刻数据长什么样(真实值) / 我做什么 / 交给谁

场景设定:14:00,爸爸(主驾)开车,后排安全座椅上坐着 5 岁女儿,盖着一条小毯子。爸爸不放心,说了一句:"宝宝毯子掉了提醒我"。
接下来就看:这句话进系统后,整条链路怎么走,你(触发器)在每一步具体干什么、数据怎么一步步变化,直到毯子真掉了、车里响起提醒。
怎么读:每一步一张卡。发生什么人话讲这步 · 此刻的数据这一刻数据长什么样(真实值,像运行日志)· 我做什么你(触发器)的动作 · 交给谁下一站。
绿色加粗的卡=你的动作。你会看到:这条链路 18 步里,第 5-8、11-16 步几乎全是你的活。灰色的结构名标签指向《17号数据契约》里对应的结构,待定标记的是还没定的字段。
1
用户说话 → ASR转文字丁彬·端
发生什么
爸爸说的那句话,车机的麦克风收到,端侧 ASR 转成文字。
此刻的数据 
{ "text": "宝宝毯子掉了提醒我",
  "speaker": "主驾", "position": "driver",
  "timestamp": "14:00:03" }
交给谁→ 第2步 拒识分流
2
拒识分流:这是哪种话?斯华·意图
发生什么
判断这句话属于四路里的哪一路。"毯子掉了提醒我"不是"开车窗"那种直接指令,是一个要持续观察的需求 → 归"非车控/条件话",送 Director。
此刻的数据 
{ "intent": "条件话·持续观测", "route": "→ Director" }
交给谁→ 第3步 Director
3
Director 拆解 → 创建动态Advisor仁杰/庆磊
发生什么
Director 判断这是"持续视觉观测任务"(要一直盯着毯子,不知道什么时候掉),于是创建一个专门管这件事的动态 Advisor。
此刻的数据 
{ "task_type": "持续视觉观测",
  "advisor_id": "dyn_adv_001", // 新建的动态Advisor
  "目标": "毯子掉了→通知用户" }
交给谁→ 第4步 动态Advisor自己先查一下
4
动态Advisor 先查:毯子是现成信号吗?庆磊
发生什么
动态Advisor 先看 Context 里有没有"毯子"这种现成信号(官方图第一个菱形"是否预对齐的 enum")。查下来——AIBox 没有"毯子掉没掉"这种枚举标签 → 走"拼SP"这条路(不是走 enum 直接监听那条)。
此刻的数据 
查Context: "毯子" → 未找到枚举信号
判定: "is_enum": false  // false=走拼SP路 / 如果是"宝宝睡了吗",睡觉是enum→true→走监听路
交给谁→ 第5步:把需求给你(云端触发器)
5
动态Advisor 把"人话需求"给你你·云端触发器
发生什么
动态Advisor 给你的是一句人话(不是SP!):"观察后排宝宝毯子状态"。拼SP是你的事。
此刻的数据 ObserveRequest 
{ "task_id": "task_blanket_001",   待定·谁生成找刘杨
  "request_text": "观察后排宝宝毯子状态",  // 人话,不是SP
  "is_enum": false,
  "position_hint": "后排",
  "advisor_id": "dyn_adv_001" }  // 记住是谁发起的,结果要还给它
我做什么接收这个人话需求,准备把它变成 VLM 能执行的 SP。注意:task_id 谁生成还没定——这是你要找刘杨定的。
6
你判:描述类 + 舱内你·云端触发器 15-16/15-17
发生什么
你先判"毯子"是描述类(不是enum)→ 要拼SP。再判方向:"毯子"在舱内 → 走端侧VLM(舱内图不上云,合规)。
此刻的数据 
15-16 枚举/描述分流: "type": "描述类"  // → 要拼SP
15-17 提槽判方向:     "target": "舱内VLM" // 毯子在舱内 → 端侧VLM
我做什么两个判断:是描述类(要拼SP)、在舱内(走端侧VLM)。判完进入拼SP。
7
⭐ 你拼SP + 定KV槽位你·云端触发器 15-18·核心
发生什么
这是你整条链路最核心的动作。VLM 现在正在跑一个 AlwaysOn SP(在看主驾、副驾状态)。你把"观察毯子"这句拼进去,并给它分配一个 KV 槽位(用任务名当 key),记住"毯子结果放在 answer_blanket 这个位子"。
此刻的数据 SPPackage 
// 拼之前,VLM正在跑的原生SP:
原生SP = "prompt: 观察主驾状态(aaaa), 观察副驾状态(bbbb)"

// 你拼入"观察毯子",变成:
{ "task_id": "task_blanket_001",
  "sp_content": "prompt: aaaa, bbbb, 观察后排宝宝毯子状态(cccc)",  ← 多了cccc
  "kv_key": "answer_blanket",   // 毯子结果放这个槽位(用任务名)
  "target": "舱内VLM",
  "period_sec": 5,
  "deploy_interface": "?" }  待定·下发接口找刘杨(T2)
我做什么三件事:① 把"cccc"追加到 SP 字符串尾部;② 定 KV 槽位 answer_blanket,记录"这个任务的结果放这个位子";③ 准备下发。你不理解毯子是什么,只做字符串拼接 + 记槽位。
8
你把SP包下发给端侧触发器你·云端触发器 → 端侧触发器
发生什么
拼好的 SP 包,从云端触发器下发给端侧触发器。
我做什么通过下发接口把 SPPackage 送到端侧触发器。用什么接口(push还是pull)就是张荣缺席时问你的"端侧触发器怎么跟云端拼接SP"——待定·找刘杨(T2)
交给谁→ 第9步 端侧触发器
9
端侧触发器触发VLM推理你·端侧触发器
发生什么
端侧触发器收到 SP,调 AIBox,让 VLM 按这个新 SP 每 5 秒看一次。
我做什么端侧触发器是 VLM 的"触发方"——调 AIBox 让它按合并SP推理。从这一刻起,VLM 每 5 秒除了看主驾副驾,还多看一眼毯子。
交给谁→ 第10步 VLM
10
VLM 第1轮推理:毯子还盖着丁彬·AIBox
发生什么
VLM 第一次看(14:00:15),毯子还好好盖着。它只输出客观描述,不判断"掉没掉"。
此刻的数据 VLMResult 
{ "task_id": "task_blanket_001",
  "kv_key": "answer_blanket",
  "answer": "毯子盖在宝宝身上",   // 客观描述,没说"没掉"
  "capture_ts": "14:00:15" }  待定·时间戳找丁彬
交给谁→ 第11步 回到你的端侧触发器
11
端侧触发器接收结果 + 透传AI service你·端侧触发器(紫线)
发生什么
VLM 结果先回到端侧触发器,你把它以端状态协议透传给 AI service → 上云进 Context。
我做什么① 接收 VLMResult;② 以端状态协议透传 AI service(紫线你负责这段)。注意:结果不是给"触发者自己",是给 AI service 上云——这就是丁彬第③点"触发方≠接收方",要跟斯华对齐。待定·找斯华
交给谁→ AI service → Context → 第12步 你云端监听
12
云端触发器监听槽位,存前帧你·云端触发器 15-13
发生什么
你的云端触发器盯着 answer_blanket 这个槽位。第1轮结果来了,你把它记成"前帧"(用来跟下一轮比)。第1轮无可比,不触发。
此刻的数据 PrevFrame 
{ "signal_key": "answer_blanket",
  "last_value": "毯子盖在宝宝身上",   // 存下来当前帧
  "capture_ts": "14:00:15" }
存储位置: AI service? 触发器自维护?  待定·前帧存哪找赵衍(C2)
我做什么存前帧。这里有个大坑:前帧存哪还没定(端不留存、透传不存)——这就是 C2,你要找赵衍/仁杰定。没有前帧,你根本没法比"变没变"。
13
几轮后…VLM 第N轮:毯子掉了丁彬·AIBox
发生什么
中间几轮毯子都还盖着(数据略)。到 14:05:20,孩子一动,毯子滑到脚底了。VLM 这一轮的描述变了。
此刻的数据 VLMResult 
{ "task_id": "task_blanket_001",
  "kv_key": "answer_blanket",
  "answer": "毯子滑落在脚底",   // ← 变了!
  "capture_ts": "14:05:20" }
交给谁→ 同样经端侧触发器透传上云 → 第14步 你检测
14
⭐ 你检测到变化(穿越判断)你·云端触发器 15-13·核心
发生什么
你拿这一轮的值,和你存的前帧比——变了!这就是"变更为"判断。
此刻的数据(你在比) 
前帧(last_value): "毯子盖在宝宝身上"
本轮(answer):     "毯子滑落在脚底"
→ 判定: 变了!触发!  // 防抖:连续K帧确认,防VLM抖动误判
我做什么① 前帧 vs 本轮对比 → 检测到变化;② 防抖(连续几帧一致才认,防 VLM glitch);③ 命中 → 把结果返回给发起的 Advisor。注意:你只判"变没变",不判"是不是掉了"——那是 Advisor 的事。
15
你把结果返回动态Advisor你·云端触发器 → Advisor
发生什么
你把"毯子滑落在脚底 + 变了"返回给当初发起的 dyn_adv_001。
此刻的数据 
→ 返回给 dyn_adv_001:
{ "task_id": "task_blanket_001",
  "answer": "毯子滑落在脚底", "changed": true }
我做什么把结果原样返回 Advisor。这一步走"绕Context"还是"原路返回",7.4有争议(T5)——要和丁彬西钺定。
交给谁→ 第16步 动态Advisor判断
16
动态Advisor 判断:满足退出了吗?庆磊
发生什么
动态Advisor 判断:"毯子滑落在脚底" = 毯子确实掉了 → 满足退出条件 → 通知你删SP。"掉没掉"的判断在 Advisor,不在你、不在 VLM。
此刻的数据 
Advisor判断: "毯子滑落在脚底" = 掉了 → 满足退出
→ 通知你: "删除 task_blanket_001 的SP"
交给谁→ 第17步 通知你删SP
17
你删SP + 还原AlwaysOn你·云端触发器
发生什么
Advisor 通知你任务完了,你把"观察毯子(cccc)"从合并SP里删掉,还原成原生SP。VLM 下一轮不再看毯子。
此刻的数据 
删除前: "prompt: aaaa, bbbb, cccc"
删除后: "prompt: aaaa, bbbb"   // cccc拿掉,还原
同时: 删掉槽位 answer_blanket、清前帧、任务状态→success
我做什么① 删 cccc;② 删槽位;③ 清前帧;④ 任务状态置 success。这一步漏了,VLM 会一直白看毯子——拼SP的收尾必须做。
18
Advisor出建议 → Director出话术 → TTS庆磊/仁杰
发生什么
动态Advisor 出建议,Director 生成话术,车里响起提醒。整条链路闭环。
此刻的数据 
{ "talk_content": "宝宝的毯子掉了哦,要帮您调高一下空调温度吗?" }
→ TTS 播报
🎯 走完这18步,回头看你(触发器)干了什么:
第5步接需求 → 第6步判描述/舱内 → 第7步拼SP定槽位 → 第8步下发 → 第9步触发VLM → 第11步接结果透传 → 第12步存前帧 → 第14步检测变化 → 第15步返回Advisor → 第17步删SP还原。
这10步就是"你要做什么"的全部。而你现在的待办,本质就是把这10步里几个"待定"的接口/字段定死:task_id谁生成(第5步)、SP下发接口(第8步)、触发方≠接收方(第11步)、前帧存哪(第12步)、回传绕不绕(第15步)。
这个场景里你碰到的数据结构(回指17号):ObserveRequest(第5步)· SPPackage(第7步)· VLMResult(第10/13步)· PrevFrame(第12步)· 还有没直接出现但一直在用的 Signal、TaskState。17号那份"字典"里的每个结构,在这个场景里都变成了带真实值的一步——那份字典是查的,这份场景是走的。
