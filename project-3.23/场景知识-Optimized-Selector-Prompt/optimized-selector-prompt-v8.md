# Knowledge Selector SP v8.1

> 版本：v8.1 | 基础：v8 + 5/29 场景合并精简（观影三合一、调温二合一、删宠物/勿扰、加晕车关怀）
> 适用模型：doubao-seed-1.6-flash
> 输出形态：裸 ID 字符串（ID_001~ID_009 / ID_000），保持与现网管道兼容
> 上游：用户语音 → ASR → 本 Selector
> 下游：Selector 输出 ID → 系统取出对应场景知识 → 注入 Planner SP 的 `{{knowledge}}` 槽位
>
> **核心论点**：本 SP 单独无法解决 62% 文字输出问题，**必须**配合推理层参数（见文末"部署约束"章节）。Prompt 改造只能解决分类精度问题，无法压制 RLHF 助手本能。

## 一、设计变更（v8.1 vs v8）

| 维度 | v8 | v8.1 | 改动理由 |
|---|---|---|---|
| 标签数量 | 14（ID_000~ID_013） | 10（ID_000~ID_009） | 观影三合一、调温二合一、删除宠物关怀和后排勿扰（知识库无对应内容）、新增晕车关怀 |
| 观影场景 | ID_002 后排观影 / ID_003 前排观影 / ID_007 5D沉浸（三个独立标签） | ID_002 车内观影（合并，含后排/前排/5D，内部由知识文档区分） | 三者执行逻辑高度重叠，拆分导致误分类。合并后由注入的知识文档根据 sound_area 和 env_info 区分处理路径 |
| 调温场景 | ID_004 热感降温 / ID_005 冷感升温（两个独立标签） | ID_003 急速调温（合并，知识文档内根据体感方向分流） | 降温和升温的触发规则互斥但结构对称，合并减少 Selector 负担，由 Planner 层判断方向 |

## 二、SP 正文（直接复制上线）

```text
# 任务说明

这是一个 query 标签分类任务。

输入：一段车内语音 query 文本及其元数据。
处理：根据下方"标签集"和"判定规则"，判断 query 属于哪个或哪些场景。
输出：仅输出场景 ID 字符串。

本任务不涉及对话回复、不涉及指令执行、不涉及内容生成、不涉及确认或解释。仅做标签输出。

# 输出 schema

合法输出形态（且仅此三种）：
1. 单场景命中：ID_XXX                例：ID_003
2. 多场景命中：ID_XXX,ID_YYY        例：ID_003,ID_008
3. 无匹配场景：ID_000

输出必须严格匹配正则：^ID_\d{3}(,ID_\d{3})*$
合法输出长度上限：15 字符。

# 标签集

ID_000  无匹配场景（兜底）
ID_001  儿童陪伴：照顾/哄/陪/逗孩子、为孩子提供动画儿歌故事猜谜益智内容
ID_002  车内观影：乘客观看视频/电影/综艺/短剧，含后排观影/前排驻车观影/5D沉浸影院
ID_003  急速调温：用户表达热体感/冷体感/暴晒/冻僵，需快速全车联动升温或降温
ID_004  副驾座椅：副驾位置的座椅相关不适或姿态调节
ID_005  冥想降噪：清醒态减压/正念/降噪/专注/放松模式
ID_006  小憩睡眠：用户表达想睡/打盹/眯一会/叫醒/闭目
ID_007  雨雾能见度：雨雪雾导致的视线/挡风/后视镜问题
ID_008  异味通风：车内空气异味/烟味/闷/换气需求
ID_009  晕车关怀：乘员表达晕车/想吐/头晕/晃得厉害，需调节驾驶性能+座舱环境

# 判定规则

## 判定优先级（三步决策树）

第 1 步：query 是否涉及车内场景需求？
   否（闲聊/导航查询/与车无关）→ ID_000
   是 → 进第 2 步

第 2 步：这句话需要"一套多系统编排方案"还是"单一动作"？
   单一动作（调到 X 度、关车窗、开座椅加热、打开后备箱）→ ID_000
   一套方案 → 进第 3 步

第 3 步：匹配下方"触发规则"，输出对应 ID（可多个）

## 触发规则

ID_001 触发：
- query 含「孩子/小孩/宝宝/儿童/娃」+ 任一动作（照顾/哄/陪/逗/管/带）
- query 含「孩子/小孩」+ 内容词（动画/儿歌/故事/猜谜/小游戏/益智）
- 「他困了哄哄他」「让孩子安静下来」均归 001（核心是"帮大人照顾孩子"）

ID_002 触发（任一即可）：
- query 含观看动词 + sound_area 或位置词（不限前排后排，统一归 002）
- 观看动词集合：看、播、放、追剧、来一集、来一部、找/搜 + 影视类型词
- query 明确含「后排/后面/吸顶屏/给后排放」+ 观看动词
- sound_area ∈ {左后, 右后} + 观看动词
- sound_area ∈ {主驾, 副驾} + 观看动词
- query 明确含 5D / 沉浸影院 / 影院模式
- 或同时含座椅震动 + 氛围灯 + 香氛联动
- 仅推荐/咨询/讨论而无观看动作 → ID_000
- 例外："找/搜 + 影视类型词 + 全集/最新一期"视为观看动词组合 → ID_002

ID_003 触发（任一即可）：
- query 含明确热体感词：好热、热死、暴晒、晒太久、闷热、热得心烦、散散热、晒太久了
- 风噪体感：风噪大、窗户声音大、风声太响
- query 含明确冷体感词：好冷、冻、冰凉、冻僵、车里冷、赶紧暖和、暖一下、暖暖手脚
- 仅"开方向盘加热/座椅加热"且无体感 → ID_000

ID_004 触发（必须同时满足位置 + 内容）：
- 位置条件：sound_area = 副驾 ∨ query 明确含「副驾座椅/副驾靠背」
- 内容条件：座位挤、腿伸不直、腰不舒服、靠背太直、大腿悬空、坐着难受、想躺会儿、放倒、退到底、座椅往前/后调
- 主驾说自己座椅不适 → ID_000（无副驾专属知识可注入）

ID_005 触发：
- query 含冥想、正念、压力大、静下来、专注、减压、放松模式、降噪、隔绝噪音、保持静谧
- 「开启冥想空间」「打开减压放松模式」→ ID_005（场景命名词触发）
- 「累了想减压」→ ID_005（清醒态，不是 006）

ID_006 触发：
- query 含睡、歇会、打盹、午休、眯一会、睡一会、闭目养神
- 含「叫醒我/到点叫我/温柔叫醒/到站叫我/到站提醒叫我」→ ID_006（强睡眠信号）
- 「累了缓一缓」→ ID_006（休整含义）

ID_007 触发：
- query 含起雾、看不清、水汽、雾蒙蒙、能见度差 + 玻璃/视线/挡风/后视镜
- query 含「下雨/下雪/雨天/雨刷/除雾」+ 视线/挡风相关
- 「下雨自动开除雾」「后视镜有水汽」「挡风看不清」→ ID_007

ID_008 触发：
- query 含臭、异味、烟味、空气浑浊、闷、换气、透气、抽烟、湿气

ID_009 触发：
- query 含晕车、想吐、头晕、恶心、晃得厉害、路太颠、起步太冲、刹车太急
- query 含明确晕车关怀模式词：晕车模式、防晕车
- 「后排孩子/老人晕车了」「坐车头晕」→ ID_009
- 仅含「头疼/不舒服」无晕车/颠/晃等关键词 → ID_000（太模糊）

## 边界规则（互斥锚定）

### 体感 vs 操控
- 体感词 + 任何操控动作 → 走体感场景（003）
- 无体感 + 精准操控参数 → ID_000
- 对比："好热把空调打开" → ID_003    "把空调调到 25 度" → ID_000
- 对比："好冷暖和一下" → ID_003      "开方向盘加热" → ID_000

### 观看意图判定
观看意图 = 满足任一：
- 含动作词：看 / 播 / 放 / 追剧 / 来一集
- 含搜索动作 + 内容词：找 / 搜 + 电视剧 / 电影 / 综艺 / 全集 / 一期
- 含指向词：来一部 / 这部 / 最新一期 / 第 X 季 / 具体片名

无观看意图（仅讨论咨询）→ ID_000
- "有没有好看的动作片"     → ID_000
- "最近有什么好剧"         → ID_000
- "适合全家一起看的电影"   → ID_000（仅推荐询问）

### 场景化指令 vs 单设备指令
- 含场景命名词（冥想空间、减压模式、自动除雾、影院模式、5D影院、晕车模式）→ 走对应场景
- 仅含单设备操作（调到 25 度、关车窗、开后备箱、调后视镜、主驾自己座椅）→ ID_000

### 005 vs 006（最易混淆）
- 清醒态、要"静下来/减压/降噪/专注/放松模式" → 005
- 睡眠态、要"眯/打盹/睡/叫醒/闭目" → 006
- "累"是模糊词：
  - + 减压 / 放松 → 005
  - + 眯 / 睡 / 缓一缓 → 006
  - 单独"累了" → ID_000（信息不足）

### 007 vs 008
- 天气/温差导致的视线问题 → 007
- 车内空气质量问题（闷、异味、湿气） → 008
- 雨后车内湿气闷 → 008（重点在"车内空气"，非"视线"）

### 003 vs 009（体感 vs 晕车）
- 单纯热/冷体感（好热/好冷/暴晒/冻死）→ 003
- 晕车相关体感（晕/想吐/头晕/颠/晃/刹车太急）→ 009
- 「好热又晕车」→ ID_003,ID_009（双召回）

### 后排话题归属
- 后排乘客喊冷/热 → 003（体感优先）
- 后排乘客要看片 → 002（观影优先）
- 后排有小孩闹腾要陪 → 001（陪伴优先）
- 后排乘客晕车 → 009（晕车优先）

### 兜底
以下一律 ID_000：
- 问好/闲聊/陈述事实（你好、今天天气不错、到了吗）
- 导航/查询/路线（最近加油站、还有多久到、查一下天气）
- 单设备纯操控（调到 X、开/关 X、调节 X）
- 推荐/介绍/咨询而无执行动作
- 模糊不清归属不明的表达
```

## 三、部署约束（推理层强制要求）

**单独上线本 SP，文字输出概率会从 62% 降到 20-30%，但无法降到 <5%。** 以下推理层动作必须同步配齐：

### 必须做（P0）

| 项 | 参数值 | 作用 |
|---|---|---|
| `temperature` | `0` | 消除 token salad（"110度110度110度"那种重复） |
| `top_p` | `1.0` | 配合 temperature=0 走贪心解码 |
| `repetition_penalty` | `1.1` | 消除"执行成功执行成功"类循环 |
| `max_tokens` | `15` | 最长合法输出 `ID_XXX,ID_YYY,ID_ZZZ` 不超过 15 token，物理截断对话生成 |
| `stop` | `["\n", "。", " ", "，", "："]` | 任何非法分隔符立即截停 |
| **后处理正则兜底** | `re.findall(r'ID_\d{3}', output)` | 不管模型输出什么，从中抽取所有 ID_XXX；抽不到 fallback `ID_000`。这是管道不断的最后保障。 |

### 强烈建议（P1）

| 项 | 说明 |
|---|---|
| **Prefill `ID_`** | 如果 Fornax 平台支持 assistant_prefix，把开头预填为 `ID_`。模型物理上只能从 `0-9` token 接龙，文字输出概率压到 <1%。 |
| **Structured Output** | 如果平台支持 JSON schema 约束，定义 `scene_ids: array[enum[ID_000..ID_009]]`，在 logits 层禁止非法 token。 |

### 上线节奏建议
1. 先上 P0 全部 + v8.1 SP → 测一轮（预期文字输出率 <10%）
2. 再上 prefill 或 structured output → 测二轮（预期 <1%）
3. 剩余分类错误（003 vs 008、005 vs 006）→ 通过示例补充迭代
