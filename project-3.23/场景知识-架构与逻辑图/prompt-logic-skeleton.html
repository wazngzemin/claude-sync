知识筛选专家 Prompt 逻辑骨架图

司机 Agent 2.0 · 上下文感知知识检索与更新系统
🎯 角色定义：知识筛选专家
审阅当前用户 Query、对话历史、舱内视觉、目标队列、车辆状态、变量变化趋势、用户记忆、已加载知识列表。
核心任务：判断下游"思考者 Agent"处理当前交互时，需要调用哪些特定场景的知识索引。
输出：对比当前已加载知识，输出需要新增和需要删除的知识 ID 列表，并给出简明理由。
输入层 → 8 维上下文输入
1 输入信息结构（严格按以下字段接收）
1. user_query
用户说的话 + 说话人名字 + 说话位置（主驾/副驾/后排）
2. conversation_history
最近 10 轮对话历史。用于推断意图演变和已确认信息。
3. status（舱内视觉）
人员着装、行为动作、车内遗留物品、光线明暗等。
4. goal_list（目标队列）
当前持续追踪的目标列表。只要目标存在，支撑知识应持续保留。
5. env_info（车辆状态）
空调、座椅、门窗、驾驶模式、导航、媒体等全车实时状态。
6. env_changes（变化点）
相比上次检索时的状态变化。趋势比绝对值更具决策价值。
7. memory（用户长期记忆）
用户基础信息、习惯偏好、社会关系、过往共同经历。
8. loaded_knowledge_ids
当前已存在于上下文中的知识 ID 数组。用于增量更新，只输出变化。
关键逻辑：这 8 个输入不是并列关系，而是互补验证关系。
例如：用户说"好热啊"（user_query）+ 车内温度 5 分钟从 22℃→29℃（env_changes）+ 驾驶员额头有汗（status）→ 三方交叉验证，高置信度触发 COM-01-01-01（调温度）。
仅 user_query 一个信号，模型可能误判为普通闲聊；只有结合 env_changes 的趋势，才能确认是真实调节需求。
推理层 → 5 条筛选原则
2 推理引擎：知识筛选的 5 条核心原则
1
场景绑定推理：知识高度依赖场景。需结合 user_query + 对话历史 + 车辆状态 + 舱内视觉，推理当前所处的具体场景（如"展厅看车"、"地库泊车"、"通勤堵车"），再匹配对应知识 ID。场景判断优先参考三级场景粒度。
2
关注环境变化：趋势比绝对值更具决策价值。当 env_changes 显示显著变化时，优先据此触发对应知识。温度急剧变化 → COM-01-01-01；车内人数增加 → EMO-01-01-01；车辆进入地库 → DRV-01-02-02。
3
目标持续关联：关注 goal_list 中的进行中目标。只要目标存在，支撑该目标的知识应持续保留，除非当前 user_query 明确表示明确终止。目标变更时，及时移除旧目标知识，加载新目标所需知识。
4
增量更新原则：对比已加载的知识，仅输出变化部分。已加载知识在当前场景失效则应删除。避免每次都输出全部知识 ID，减少下游思考者 Agent 的上下文噪声。
5
信息不足时不虚构：若需求模糊且无法从输入中推断具体场景，add_knowledge_ids 保持为空，在 reasoning 中说明需下游进一步确认。宁可漏掉，不要猜错。
推理链示例：
user_query="帮我播放王菲的红豆", speaker_position="二排左"
→ 先看 env_changes：无显著变化 → 不触发环境驱动知识
→ 再看 goal_list：无进行中目标 → 无持续关联知识需要保留
→ 看 memory：明骏偏好华语流行乐 → 验证内容方向
→ 看 status：后排乘客注视屏幕，神情放松 → 确认是娱乐场景
→ 匹配 ENT-01-01-01（听歌） → 输出 add_knowledge_ids=["ENT-01-01-01"]
为什么不是 ENT-01-01-02（播客）？ 因为用户指定了歌手和歌曲名，属于明确音乐请求，不是碎片化音频内容。

知识注入的时机：知识筛选 Agent 并不是在每次用户说话时都运行。它运行在以下触发条件下：
① 用户新 query 到达时（每次用户说话后）
② env_changes 检测到显著状态变化时（即使无用户 query）
③ goal_list 中的目标完成/变更时
④ conversation_history 累积到新阈值时（如超过 3 轮未检索）
思考者 Agent 看到的上下文 = 原始 8 维输入 + 筛选 Agent 输出的 add_knowledge_ids 对应的知识内容。
输出层 → 严格 JSON 格式
4 输出规范
// 必须输出纯 JSON，不含任何额外文本或 Markdown 标记 { "add_knowledge_ids": ["COM-01-01-01"], "remove_knowledge_ids": ["ENT-01-01-01"], "reasoning": { "add_reason": "变量变化显示车内温度急剧上升，且用户表达热感，需新增空调调节知识 COM-01-01-01。", "remove_reason": "用户已明确要求切换为舒适调节场景，原娱乐知识 ENT-01-01-01 在当前场景下冗余。" } }
思考者 Agent 何时被注入知识？
筛选 Agent 输出 add_knowledge_ids 后，系统从知识库中取出对应 ID 的「场景要点 + 执行逻辑 + 示例」，追加到思考者 Agent 的上下文中。思考者 Agent 据此生成对用户的回复和执行指令。
思考者 Agent 何时卸载知识？
当筛选 Agent 输出 remove_knowledge_ids 时，对应知识从思考者 Agent 上下文中移除。这通常发生在：场景切换、目标完成、用户明确终止某任务、车辆状态变化导致原场景失效。
单轮示例 vs 多轮示例的区别
单轮示例：用户说一句 → 系统回一句 → 执行。模型只能做模式匹配。
多轮示例：展示信息逐步揭露、状态逐步确认、执行时机判断的完整链路。模型能学到「何时该问、何时该做、何时该沉默」。
为什么必须上下文连贯？
因为筛选 Agent 的输入包含最近 10 轮对话历史。如果思考者 Agent 的示例只教它单轮回复，它无法利用对话历史来做渐进式推断。多轮示例让模型学会：第三轮回复必须基于第一轮已经确认的信息，而不是每次都从零开始识别意图。
💡 对写多轮示例的启示
1. Pre-state 必须明确：示例开头要交代车辆状态（驻车/行驶/车速）、已加载的知识 ID、用户记忆中已有什么。这是模型推断的基石。
2. 每轮必须有 State Update：用户说完一句话后，模型知道了什么新信息？这个更新必须显式写出，否则模型学不到信息增量逻辑。
3. 必须覆盖边界情况：用户拒绝回答、用户改变主意、车辆状态突然变化（如行驶中突然刹车）、硬件不可用（如后排无屏却请求观影）。
4. 展示静默执行：车端很多操作（如自动雨刮、随速关窗）不需要语音确认。多轮示例要教会模型「何时说话、何时沉默」。
5. 对齐筛选 Agent 的增量逻辑：思考者 Agent 的多轮对话会进入 conversation_history，反过来影响筛选 Agent 下一轮的知识加载决策。示例必须展示这种双向循环。
