为什么车企越造越亏，钱却被电池和华为赚走了

一篇讲车企利润率从 6.1% 掉到 3.2% 的文章。它真正讲的不是车，是一件对你更要紧的事：为什么最辛苦的人不一定最赚钱。这一版把它揉进了你的 Planner 和触发器架构，给到你该怎么做。

分诊结论
这篇 → 做成「解析文档」，不做工具。

它是框架/观点型，给的是一副判断透镜，不是一套你会反复运行的评分/诊断流程。价值在于想通它、吸收它、接到你的工作上，所以做深读解析。（那个「价值捕获四问」我放在第 05 节当一个能随手套的小清单，不单独做成交互工具。）

以后每篇的分诊标准：文章给的是「一副看问题的透镜/一套认知」→ 解析文档；文章给的是「一个你会反复填、反复算、反复判的流程」（评分卡、决策矩阵、可勾选诊断表）→ 工具。拿不准就先解析，工具当解析里的内嵌小件追加。
01

一句话讲透

这篇文章说的其实很朴素：你有多累，跟你能赚到多少，是两码事。钱不会因为谁最辛苦就流向谁，它流向那些别人绕不开、而且越做越强的位置。车企累死累活造车，利润却从 6.1% 掉到 3.2%，不是被对手打的，是它站的那个位置（总装、卖车）本来就存不住钱。

02

拆开看：它其实只做了三件事

第一件，换个角度看问题

大部分人看到利润跌，第一反应是「竞争太激烈了」。作者说，这是把症状当成了病根。你摸到的那个「卷」，只是表面。真正的问题在你站的位置。他的证据挺硬：同样在这个卷得要死的市场里，宁德时代和华为在闷声赚钱。一样的竞争，活法完全不同，说明胜负手不在卷不卷，在你站哪一层。

这一步为什么不容易想通？因为它反直觉。我们心里默认「我很辛苦」就等于「我在创造价值」。文章逼你把这两件事分开。

第二件，讲清楚什么位置存得住钱

位置例子为什么存得住 / 存不住钱
基础设施层宁德 · 电池你是绕不过去的那道关卡，谁造车都得过你这道门，全行业围着你转
能力层华为 · 智驾座舱你卖的是别人做不出、还决定人家能不能进高端市场的东西，你攥着入场券
组装层车企 · 造一辆卖一辆卖完就结束，谁都能组装，用户开走就断联。没锁定、没复利、可替代，被两头抽

「造一辆车、卖一辆车」这六个字是全篇的病灶。毛病是三样凑齐了：算一辆是一辆（线性）、交易完就断（一次性）、你的和他的差不多（可替代）。三样一齐，你就没有讨价还价的本钱。

第三件，给三条出路

既然组装层存不住钱，那就换位置。三条路放到第 05 节的方法论里一起讲，那里有决策图。

03

四个能从车企身上摘走、装到你自己身上的点

点 01先分清「现象层」和「结构层」

很多人一辈子在现象层治标：卷、加班、焦虑。会分层的人，先问一句「这是结构问题吗」。

怎么用：任何「我很努力但被架空 / 没结果」的场景，先画上面那张分层图，别急着在现象层加药。
点 02好位置不是占到就完，是会自己越滚越强

占到好位置只是开始。它有一个自我加强的循环转着，不转它，位置就会被人掀掉。这就是飞轮。

点 03判断一个能力值不值钱，别看好不好用，看是不是「入场必需」

文章那句「没有华为，某界连高端牌桌都上不了」很关键。判据不是「这功能做得好」，是「没有它，对方根本上不了台」。

怎么用：问「没有我这个东西，对方是体验差一点，还是压根上不了台？」前者可替代，后者才有定价权。这一问能砍掉一堆「我这个挺重要的」自我安慰。
点 04价值的重心，正在从「造东西」挪到「占关系」

文章说终极赢家是出行服务商，车只是交付服务的终端。谁占住用户的持续关系，谁就把一次性买卖变成长期收入。软件从卖 License 到卖订阅，走的是同一条路。

怎么用：看任何产品，问它在做「一锤子买卖」还是在「攒一段持续关系」。后者天花板高得多。记住这条，第 06 节讲你的触发器时会正面撞上它。
04

连问五个为什么，钻到最底下那块铁

  • 为什么1车企利润率为什么跌？
    因为被上游电池和上游智驾两头抽走了利润。
  • 为什么2为什么会被抽？
    因为组装层没有讨价还价的本钱。
  • 为什么3为什么没本钱？
    因为它可替代（谁都能组装），又没锁定（卖完就断）。
  • 为什么4为什么可替代、没锁定就没价值？
    因为能拿多少价值，看的是「对方离开你要付多大代价」。可替代，就等于对方随时能换掉你，代价约等于零。
  • 为什么5为什么「更努力」救不了它？
    因为努力只能把当前这层的活干得更好，改变不了「对方离不离得开你」。前者努力能动，后者位置决定。
钻到底的那块铁你能拿到多少价值，取决于别人离开你要付多大代价，不取决于你干了多少活。这是这篇脱掉车企外衣以后，剩下的那句话。
05

压成一套方法：价值捕获四问

对任何东西都能反复跑的诊断。对象可以是一个产品、你负责的一个模块、或者你自己。四个问题，逐个答。

1
别人绕得过你吗？绕过你有没有损失。测「关卡」。
2
你在攒一个越来越强的东西吗？每次交付是不是在积累资产。测「飞轮 / 复利」。
3
你是入场必需，还是锦上添花？没有你对方上不了台，还是只差一点。测「差异化」。
4
你的位置会漂移吗？下游能自己做出来平替你吗，新技术能掀你桌吗。测「护城河还能撑多久」。

答完这四问，就知道自己在哪一层、往哪条路走。三条路是这样分的：

制造驱动 · 成本领先比亚迪
靠规模、供应链、产线摊销把成本压到别人做不到。前提有规模经济。风险被规模追平。稳，天花板低。
技术驱动 · 数据飞轮华为 · 小鹏
押智驾，关键是统一数据 + 软件迭代 + 闭环，攒出别人追不上的差异。前提飞轮真转得动。风险智驾同质化。复利最实。
服务驱动 · 占用户关系出行服务商
从卖车变卖出行，车成了交付终端。前提L4 落地。风险最像愿景。最难，也最值钱。

我读了你的 planner 架构 和 advisor 架构 文档。把「生态位」这把尺子架上去，你系统里的价值分布是这样的。凡是我推断的、不是你文档原话的，都标 推断，你自己核。

你架构里的组件落在哪一层为什么（据你文档）
底层模型 Doubao-Seed-1.8基础设施层全系统的「电池」，Director/advisor/agent 都得过它。不归你。
Director / Planner 推理能力层推断座舱的差异化大脑，26 工具 + ReAct。但正在被稀释，见下面那颗钉子。
触发器 + 动态 advisor + 目标驱动层服务层你文档原话：它解决旧架构「我讲一句应一句、一锤子买卖、需求不闭环」的病。这就是「占用户持续关系」。你的主战场。
用户记忆（Context 13 类）复利 / 锁定越用越懂用户，是黏住用户的那根线。落地中。
badcase → SP → 前置筛选回流数据飞轮知识筛选专家按场景挑知识 ID 注入，越用越准。SP 优化实测首包 P50 从 2309ms 降到 998ms。

一颗你文档里藏着的钉子：Planner 这个能力层，是被设计成会缩小的

你的三层漏斗，第三条设计原则原话是：「通过用户数据回流逐步把后面复杂任务的处理推到前面知识注入环节」，后面还跟着一句：「随着时间推移，进入 Planner 的流量会越来越少」。

用生态位的话讲：你们亲手设计了一个让 Planner 推理这个能力层慢慢被「组装化」的机制。

句法 RAG、情景 RAG 这些便宜的层被训练得越来越强，Planner 这个贵的层，它的活被一点点搬到前面去。这是对的架构，便宜的活不该让贵模型干。但对你个人是个信号：别把你的价值锚在「Planner 推理调得好、动态示例修得勤」上，那是一个按设计在缩小的生态位。你越往那使劲，越像那个在组装层里拼命的车企。

真正在长大的生态位，是你 own 的那块

目标驱动层是这套系统里往「服务层」爬的唯一引擎，而它的底座就是你的触发器和动态 advisor。拿第 05 节的四问打一下（据你文档，判断是我的推断）：

四问你的触发器 / 目标驱动层
① 绕得过吗advisor 的持续任务、条件/定时/VLM 订阅都得经触发器注册和触发。中偏高。
② 有复利吗用户记忆 + 持续目标在累积，二期还上了 Skill 系统复用。中。
③ 决定牌桌吗没有持续照看，座舱还是个「一问一答的音箱」，上不了「陪伴式智能体」的牌桌。高。
④ 会漂移吗会。如果触发器只是「被动执行 advisor 下发的订阅」，它就是根管道，谁都能替；如果它掌握「什么时候触发、优先级怎么排、条件怎么定义」的判断权，它才不可绕过。这一条就是你的定义权战场。
07

你该怎么做，顺便把 PM 能力抬一档

你说自己现在偏初级、想提升。这篇正好给一个具体的抬升路径，不是道理，是落到你架构里的动作。

初级和高级 PM 的差，在你的架构里看得很清楚

在做什么落在哪一层
初级执行已经定义好的节点：把动态示例一条条维护、把某个 advisor 的 SP 调好组装层：必要，但可替代、不复利、谁做都差不多
高级定义还没定义的接口和边界：谁触发、边界在哪、字段怎么设能力层：稀缺，挂你名，别人绕不过

你之前几场评审吃亏（比如 7.10 条件表被端走），根子就是你在组装层出力，把「定义权」这个能力层的位置让出去了。这跟车企一模一样：活最重，值钱的位置被别人占。

这周就动：从你文档里的三个「还没定义」的空位挑一个

我从你文档里挑了三个明确「未定义 / 待对齐」的接口。每一个都是一次抢定义权的机会：

空位你文档里的原状
出行 advisor 与 Planner 的对接格式出行 advisor 还是 v0.4，文档写着输出格式和与 Planner 的对接「需要和 planner 详细沟通，还未完全定义」。谁定义，谁就框定了它怎么接大脑。
触发器的触发边界与优先级你有「触发优先级 3 级：订阅事件 > qa 结束 > 定时」，但触发器和 advisor「谁判断、谁下发、边界在哪」一直是评审里被端走的点。这是你能力层护城河的核心。
事件日志的字段拒识内容写入 context 事件日志、被 advisor 感知触发主动服务，但「事件日志不是新建是扩展」这事 7.10 提过、还没落成字段。

我押 出行 advisor 对接格式 那个，因为它最没主、阻力最小，适合你练手。这周做三步：

出一版接口定义草案。输入输出字段 + 边界，直接给「我认为应该这么定」，不是「这个该怎么定」。姿态是你在定，不是你在问。
找 planner 侧和出行 advisor 侧的人，当面对一次。用「这个接口的结果我来背」换「这个接口归我定」。
可以这么说「出行 advisor 跟 Planner 的对接格式现在还悬着，我出了一版定义。既然接口出问题我来背，那这块的定义和后续维护挂我这，行吗？」
把它写进你负责的模块文档，挂你名。让「归属」落到纸上，不是口头。

这三步就是你从「执行型」往「定义型」挪的最小一步，正好补你最贵的那个短板：定义权。

再给你一个能天天用的 PM 判断习惯

评估你手上任何一个需求或模块，先问它在价值捕获的哪一层：

把已定义的东西做好组装层该做，但别投入过多，更别拿它逃避评审
定义一个还没定义的接口 / 边界能力层高杠杆，优先抢
让系统占住用户的持续关系服务层天花板，往这使劲（你的触发器就在这）

这个「先分层再决定投入」的习惯本身，就是初级和高级 PM 思维方式的分界线。你每天问一遍，几个月后你看需求的眼睛就变了。
