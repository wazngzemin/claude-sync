Harness Engineering 对比分析

业界标准的 Harness 架构 vs 我们的豆包上车 Planner 方案 · 差距在哪 · 怎么补

一、Harness Engineering 到底是什么？

三代演进：Prompt → Context → Harness

2026 年业界公认的三层架构演进：

代际名称控制范围关注点时间
第一代Prompt Engineering单条消息级别怎么写指令让模型听话2022-2024
第二代Context Engineering上下文窗口级别往模型的"工作记忆"里塞什么信息2025
第三代Harness Engineering整个系统级别模型周围的一切：工具/记忆/护栏/重试/编排2026

来源：Atlan: Prompt vs Context vs Harness Engineering · AI Magicx: Why Harness Engineering Matters in 2026

Martin Fowler 的定义

Agent = Model + Harness

Model = LLM 本身（如 Seed 1.8 / Claude / GPT）
Harness = 模型周围的一切：
  - 执行循环（Agentic Loop）
  - 工具注册与调用（Tools）
  - 上下文组装（Context Assembly）
  - 记忆系统（Memory）
  - 安全护栏（Guardrails）
  - 输出验证（Output Validation）
  - 可观测性（Observability）

Phil Schmid 的类比：
  Model = CPU
  Context Window = RAM
  Harness = 操作系统
  Agent = 应用程序
一句话理解：Harness 不是模型本身，而是模型周围的"操作系统"——执行循环、工具管理、记忆、护栏、验证、可观测性。你可以换模型（换 CPU），但 Harness 不变。Meta 2025年底花 ~20 亿美元收购 Manus，买的不是模型，而是 Harness。

二、业界标准 Harness 的 7 大组件

综合 Anthropic、OpenAI、LangChain 和业界实践者的总结，一个生产级 Harness 包含 7 大核心组件：

#组件做什么为什么重要
1 Agentic Loop
（执行循环）		 Thought → Action → Observation 循环。组装 prompt → 调 LLM → 解析输出 → 执行工具 → 结果回传 → 重复直到完成。 这是 Agent 的"心跳"。没有 Loop，模型只能回答一次就结束了。
2 Context Assembly
（上下文组装）		 在每一步 Loop 前，决定往 prompt 里塞什么：system prompt + 工具定义 + 记忆 + 对话历史 + 当前消息。Anthropic 说目标是"找到最小的高信号 token 集合"。 模型的上下文窗口是有限的"注意力预算"。塞太多=注意力分散；塞太少=信息不够。
3 Tools & Execution
（工具与执行）		 暴露给模型的能力（搜索/代码执行/API/MCP），接收模型的工具调用请求并执行。 模型本身不能"做事"，只能"说话"。工具让模型能执行动作。
4 Memory
（记忆系统）		 跨轮次/跨会话的状态持久化。包括：短期记忆（对话历史）、长期记忆（用户偏好）、工作记忆（当前任务状态）。 裸模型没有记忆——上一个任务做完，下一个任务就忘了。记忆让 Agent "认识"用户。
5 Guardrails
（安全护栏）		 在三个点做校验：工具执行前（validateToolCall）、工具执行后（validateToolOutput）、最终输出前（validateFinalPlan）。 "护栏不是后处理，它是执行循环里的承重逻辑。"——没有护栏，Agent 就是一颗不知道什么时候爆的炸弹。
6 Output Validation
（输出验证）		 把模型输出解析成结构化类型（如 JSON schema），校验后再执行。用最强的模型专门做验证。 模型输出是不可靠的文本——可能格式错、参数错、幻觉。验证是防线。
7 Observability
（可观测性）		 每一步输出结构化数据：组装了什么 context、模型做了什么决策、调了什么工具、参数是什么、耗时多少、token 用了多少。 出了问题不能靠猜——需要 trace 每一步才能定位是哪里出错。

三、我们的方案 vs 业界标准 Harness：逐组件对比

Harness 组件 业界标准做法 我们的豆包 Planner 方案 差距
1. Agentic Loop
执行循环		 完整的 TAO 循环：思考→行动→观察→重复。模型可以多轮调用工具，每轮结果回传后继续推理，直到模型自己决定"完成了"。支持错误重试、超时退出。 Planner 有基本的 Loop：输出 action_list → 工具执行 → tool_feedback 回传 → Planner 可以继续推理（二次推理）。但是循环次数有限，通常 1-2 轮就结束了，不支持复杂的多步编排。 部分达标。有 Loop 但不够深。复杂场景（如"先查温度→判断要不要开窗→再查天气→决定开窗幅度"）需要 3-4 轮 Loop，当前只做 1-2 轮。
2. Context Assembly
上下文组装		 动态组装 context，包括：静态 SP + 动态检索的知识/示例 + 工具定义 + 记忆 + 对话历史 + 端状态。按优先级裁剪，确保不超过上下文窗口限制。Anthropic 说"找最小的高信号 token 集合"。 我们有完整的 Context Assembly：
- Selector SP（LLM 推理）检索动态知识
- vikingDB（向量检索）检索动态示例
- env_info / memory / 对话历史动态注入
- SP 模板变量 {{knowledge}}、{{examples}} 等
优化后从 32K 压到 ~16K tokens。		 做得不错！这是我们最强的部分。双路检索（LLM + 向量）的 Context Assembly 是比较先进的做法。但 Selector 延迟 1.8s 是瓶颈。
3. Tools & Execution
工具与执行		 工具注册表 + 工具描述 + 参数 schema + 执行引擎 + 结果解析。支持 MCP（Model Context Protocol）标准接口。工具数量可动态加载。 23 个下游工具（vehicle_basic_control / music / navi 等），每个有 tool_name / tool_describe / param / notice 四个字段。通过 SOA 车控指令下发到方向靠。 基本达标。工具数量和描述都有，但缺少动态工具加载（23 个全量塞在 SP 里占 4K tokens）。业界做法是只加载当前场景需要的工具。
4. Memory
记忆系统		 多层记忆：
- 短期：对话历史（最近 N 轮）
- 长期：用户画像 / 偏好 / 历史行为（持久化存储）
- 工作记忆：当前任务状态（Goal List）
- 跨会话记忆压缩/摘要		 有基础记忆：
- 对话历史（最近 10 轮）
- 用户记忆（search_user_memory / operate_user_memory）
- Goal List（goal_list_update）
- FaceID 身份识别（未就绪）		 差距明显。
- 10 轮后信息丢失（无会话摘要压缩机制）
- FaceID 没修好 → 记忆写不进去
- 无跨会话记忆持久化
- 场景编排结果无法持久化保存
5. Guardrails
安全护栏		 三阶段校验：
① 工具执行前（validateToolCall）：检查参数合法性
② 工具执行后（validateToolOutput）：校验返回值
③ 最终输出前（validateFinalPlan）：整体安全检查
"护栏不是后处理，是执行循环里的承重逻辑。"		 当前护栏极度薄弱：
- 层1：SP 注意事项（不能 100% 保证）
- 层2：Adviser 前置判断（部分）
- 层3：车控底层代码兜底（待定方案！）
没有结构化的三阶段校验。		 最大差距！
业界标准是在代码层面做硬校验（三阶段 validate），我们几乎完全依赖 SP 里的文字约束（LLM 推理层面，概率性的）。
小明的话："哪怕 99.9% 都规避了，有那万分之一可能车毁人亡。"
这正是业界说的"harness 的护栏是承重逻辑，不是建议"。
6. Output Validation
输出验证		 模型输出 → JSON schema 校验 → 类型检查 → 参数范围检查 → 通过才执行。不通过则让模型重新生成。 Planner 输出 action_list 后直接执行，没有结构化验证步骤。tool_name 写错了（如写成不存在的工具名）也会直接下发，到方向靠才报错。 缺失。需要加一层：Planner 输出 → 校验 tool_name 是否在 23 个合法工具里 → 校验 param 格式是否合法 → 通过才执行。这是纯代码逻辑，不需要 LLM，几 ms 就能做。
7. Observability
可观测性		 每一步输出结构化 trace：组装了什么 context / 模型决策 / 调了什么工具 / 参数 / 耗时 / token 用量。可查询、聚合、回放。 有基础 log（通过 logid 可追溯）。但不够结构化——排查 badcase 时需要手动翻 log，没有自动化 trace 和回放工具。 部分达标。有 log 但不够结构化。6/3 晚例会上晓伟排查 badcase 时说"具体的 logid 是哪个呀？"——说明 trace 不够便捷。

四、差距总结：哪些是最大的短板？

按严重程度排序

排名差距影响业界怎么做我们怎么补
🥇 Guardrails 缺失 安全红线无法 100% 保证。非 P 档调座椅 = 安全事故。 三阶段 validate，在代码层面硬拦截，不依赖 LLM。 ① SP 加约束（层1，已做）
② 车控底层写死拒绝逻辑（层3，待定方案，最关键）
③ 在 Planner 输出后加一层代码校验：检查 action_list 里的操作是否在当前条件下被允许
🥈 Output Validation 缺失 Planner 输出的 tool_name/param 可能是错的（如调不存在的工具），直接下发到方向靠才报错。 JSON schema 校验 + 类型检查 + 参数范围检查 → 不通过让模型重生成。 在 Planner 和工具层之间加一层代码校验：
① tool_name 是否在 23 个合法工具里？
② param 格式是否正确？
③ 不合法 → 返回给 Planner 让它重新生成
🥉 Memory 记忆弱 10 轮后信息丢失 / FaceID 未就绪 / 无会话摘要。 多层记忆（短期/长期/工作记忆）+ 跨会话压缩摘要。 ① 晓伟+超然本周定方案解决 10 轮丢失
② 等 FaceID 修复
③ 长期引入会话摘要压缩
4 Selector 延迟高 额外一次 LLM 调用 = 500ms-1.8s，是性能瓶颈。 向量预筛 + LLM 精选的两级漏斗，或缓存命中跳过。 ① 压缩 Selector prompt 里的索引表
② 同场景连续对话跳过 Selector（用缓存）
③ 长期引入 embedding 预筛
5 Agentic Loop 不够深 复杂场景需要 3-4 轮工具调用，当前只做 1-2 轮。 完整 TAO 循环，模型自己决定什么时候结束。 需要研发侧支持更深的 Loop，但会增加总耗时。权衡性能和能力。

五、业界的研究怎么说？

Princeton 研究结果

harness 配置可以提升 Agent 任务成功率 64%——对比基础配置（没有护栏/验证/记忆）和完整 harness 配置。说明 harness 的架构设计比模型本身更影响实际效果。

MIT 研究

约 95% 的企业 AI 试点项目未能产生可衡量的回报。核心原因不是模型不够好，而是 harness（系统集成、护栏、记忆、可观测性）没做好。

VS Code 团队的总结

"Model 是引擎，Harness 是车。"——引擎再好，车架子不行也跑不了。他们发现：在 SWE-bench 测试中，仅通过缩小工具空间（harness 优化），本地小模型从 2/10 提升到 10/10。不是模型变强了，是 harness 变好了。

对我们的启示

我们的 Planner 用的模型（Seed 1.8）足够强。当前的 badcase 大部分不是模型推理错误，而是：

  • Skill 知识写错了（Context Assembly 问题）→ 模型拿着错误信息推理
  • 没有输出验证（Output Validation 缺失）→ 错误输出直接执行
  • 没有安全护栏（Guardrails 缺失）→ 危险操作没拦住
  • 记忆不够（Memory 弱）→ 10 轮后忘了之前说的

这些都是 Harness 层面的问题，不是模型层面的问题。

六、我们应该怎么补？（优先级排序）

优先级补什么怎么补谁做工作量
🔴 P0 Output Validation
输出校验层		 在 Planner 输出和工具执行之间加一层代码：校验 tool_name 合法性 + param 格式 + 安全条件。不合法则返回给 Planner 重新生成。纯代码逻辑，不需要 LLM。 明骏/超然 中（几天）
🔴 P0 Guardrails 硬校验
安全护栏		 在车控底层代码写死：非 P 档 + 调座椅指令 → 直接拒绝。不依赖任何 LLM 推理。这是业界标准的"三阶段校验"的简化版。 车控开发 中（需协调）
🟡 P1 Context Assembly 优化
Skill 质量 + Selector 延迟		 ① 修复 Skill 质量（我做）② 压缩 Selector 索引表降延迟（我+明骏）③ 同场景缓存跳过 Selector 泽民+明骏 大（持续）
🟡 P1 Memory 增强 ① 解决 10 轮丢失（晓伟+超然本周定方案）② FaceID 修复 ③ 长期引入会话摘要 晓伟/超然 大（分阶段）
⚪ P2 Observability 结构化 每次 Planner 调用输出结构化 trace（什么 Skill 注入了 / 什么示例检索到了 / action_list 是什么 / 工具返回什么），可查询可回放。 研发 中
⚪ P2 Agentic Loop 加深 支持 3-4 轮工具调用循环，用于复杂编排场景。但要权衡延迟。 明骏 大
⚪ P2 工具动态加载 不把 23 个工具全塞 SP，按场景只加载需要的工具（减少 SP token，提升注意力聚焦度）。 明骏 中
