知识检索模块 — 延迟优化方案

司机 Agent 2.0 / 场景知识匹配 / 2026-05-18

1 问题在哪

当前知识检索模块（Knowledge Selector）单次调用耗时 1.8 秒，在整个对话链路中占比过高。用户从说话到得到回复，总延迟超过 6 秒。

为什么这么慢？

Input Tokens
~3,200
13 个场景的完整描述全塞在 prompt 里，模型每次请求都要读完。这是延迟主因。
Output Tokens
~150
输出就是一个 ID + reasoning。占比极小，压缩输出效果有限。

一句话诊断：Selector 做的是"分类"（这个 query 匹配哪个场景），但 prompt 里塞的是"说明书"（如何执行该场景）。模型读了 3000 tokens 的说明书，只为了输出一个分类编号。

2 怎么改

两步走，都只改 prompt，不动架构。

P0 必做省 800ms改 prompt 压缩场景索引

Selector 不需要知道"怎么处理晕车"，它只需要知道"这个 query 是不是在说晕车"。

把场景描述换成锚词表：每个场景一行，只保留匹配关键词。

现在：每个场景 200-400 字

名称ID: ID_004
名称场景：舒适用车-车内环境-急速降温-急速降温/智能调窗/风噪管控/动态风噪平衡调节
场景描述：1.用户在夏天车辆暴晒后上车需要快速降温，或行车中因车速变化需要调整车窗和天窗打开百分比以减少风噪。2.用户行车途中感知风量不适、想通风换气……

→
改后：每个场景 1 行

004 急速降温｜好热/暴晒/降温/风噪/开窗调节

13 个场景总计从 ~4000 字 → ~500 字，input tokens 直接砍掉 80%。

P0 必做省 150ms改 prompt 精简输出格式

输出从 JSON + reasoning 改为纯数字。

现在的输出

{"add_knowledge_ids": ["DRV-01-01-05"],
"remove_knowledge_ids": [],
"reasoning": {"add_reason": "用户当前Query为去医院…", "remove_reason": "无"}}

→

多命中：001,013
未命中：000

注意：reasoning 字段在调试阶段有价值。建议保留一个 debug 开关：正式环境只输出 ID，调试环境带 reasoning。

P0 必做保准确率加 5 行 保留互斥规则

场景描述可以砍，但互斥判断规则不能丢——这是分类准确率的关键。

3 改后的完整 Prompt

直接可以替换到 PromptHub，预估 ~400 tokens。

你是车载场景分类器。根据 query + context 判断命中哪个场景。

## 规则
- 命中输出场景 ID（如 004），多命中逗号分隔
- 未命中输出 000
- 仅输出数字，无任何其他内容

5 验证步骤

01

用精简 prompt 替换，跑现有 74 条测试用例，对比场景命中准确率
02
关注边界用例
重点看"易混淆"场景的准确率：001/013、008/009、003/002、004/012
03
延迟对比
对比新旧 prompt 的 TTFT 和总延迟，确认 tokens 下降比例
风险提示：锚词表的覆盖率取决于用户表达的多样性。如果测试集中有"隐晦表达"（如用户说"车里好闷"可能同时触发降温和通风），需要补充锚词或调整互斥规则。建议先跑测试集，收集 miss 的 case，再迭代补充锚词。

6 进阶优化（可选，不急）

P1 高频场景规则前置

对"抽烟通风""冥想"等高置信度场景，用正则关键词直接命中，跳过 LLM 调用。覆盖约 30-40% 请求，命中时延迟 < 10ms。

P2 Embedding 预筛 + LLM 精排

场景扩展到 50+ 时，用 embedding 相似度取 top-3 候选，再用 LLM 做 3 选 1 精排。延迟恒定在 ~350ms，不受场景数量影响。
