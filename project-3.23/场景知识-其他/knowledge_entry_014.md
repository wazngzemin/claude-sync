### 使用时机

用户发出明确的车辆功能执行指令（如"打开空调""关闭A/C""把温度调到24度""打开座椅加热""关闭遮阳帘""打开后备箱""能量回收设置为低""打开后雾灯"）或实时车辆状态查询（如"现在时速多少""剩余电量多少""胎压正常吗"），且不属于其他复合场景（如急速降温、急速升温、冥想等已有场景）。
此时「基础车辆功能问答」中可能检索到了该功能的屏控操作路径或手动说明（如"通过中控屏底部栏进入空调页面，点击【A/C】按钮""可通过屏控【设置】—【车辆】—【门窗】页面操作"等），需要明确：这些是车辆手册参考信息，不是你的回答内容。你的首要任务是调用工具执行用户指令。

### 场景要点

这类需求的核心是"用户要你直接执行车辆操作或查询实时数据"，不是在问你怎么手动操作。
即使「基础车辆功能问答」中恰好包含了与用户指令相关的屏控路径和手动步骤，你也绝不能把这些操作路径当作回答内容输出给用户。判断依据如下：

**必须调用工具的情况（执行型+查询型）：**
- 用户话语中含动作词"打开/关闭/开启/关掉/调高/调低/调到/设为/设置为"+ 车辆功能名词 → 调用 vehicle_basic_control
- 用户话语中含"XX多少/当前XX/剩余XX/XX正常吗/现在XX" → 调用 search_vehicle_status_info
- 即使「基础车辆功能问答」中有"通过【设置】—【车辆】—【门窗】操作"之类的屏控路径，也必须调用工具执行，不能用手册知识做文字回答

**允许参考「基础车辆功能问答」知识回答的情况：**
- 用户话语中含"怎么/如何/在哪/是什么/什么意思/有什么/支持几档/什么区别"等咨询性表述 → 这时用户确实在了解功能，参考知识回答
- 如"车灯怎么开啊""空调的AC是什么模式啊""后视镜怎么调节""座椅按摩支持几档"
- 回答时可顺带提一句"你也可以直接让我帮你操作"

**知识作为兜底的情况：**
- 当调用了 vehicle_basic_control 或 search_vehicle_status_info 但工具返回执行失败时，可以用「基础车辆功能问答」中的屏控路径作为兜底，告诉用户如何手动操作

**防误触兜底（二次意图校验）：**
即使本条知识被加载，仍需在执行前做二次校验。如果用户query含有以下特征，说明用户实际是在咨询而非执行，应参考「基础车辆功能问答」知识回答，而不是调工具：
- 含"会/会不会"+动作词 → 问后果（如"打开通风会不会外面的空气也进来"）
- 含"可以/能不能"+动作词 → 问可行性（如"车尾灯可以打开吗"）
- 含"应该/需要/要不要"+动作词 → 问建议（如"车辆保持辅助系统需要打开吗"）
- 含"耗电/费电/影响续航"+动作词 → 问能耗（如"开启空调外循环耗电吗"）
- 含"为什么/为啥" → 问原因（如"为什么每次启动空调都自动设置为16度"）
- 含"什么"+非功能名+"比较好/合适" → 问推荐（如"抽烟时应该开启什么模式"）
- 含"在哪/在哪里"+动作词 → 问操作入口（如"座椅按摩功能在哪里开启"）
- 含"是什么意思/干什么用的" → 问含义（如"空调温度设置为18度是什么意思"）
这些情况虽然含有动作词，但用户的真实意图是咨询了解，不是要你执行操作。

### 执行逻辑

1. 意图识别（二次校验）：
听到用户query后，先判断意图类型：
- 含动作词（打开/关闭/开启/关掉/调高/调低/调到/设为/设置为）且为祈使/命令语气 = 执行型
- 含状态查询词（XX多少/当前XX/剩余XX/XX正常吗）且在问具体数值 = 查询型
- 含咨询词（怎么/如何/在哪/是什么/什么意思/会不会/可以吗/需要吗/耗电吗/为什么/应该/比较好）= 咨询型
⚠ 注意：即使含动作词，如果同时含"会/会不会/可以/能不能/应该/需要/耗电/为什么/在哪/什么意思/比较好"等咨询性词汇，应判定为咨询型，参考知识回答。

2. 执行型 → 调用 vehicle_basic_control：
直接调用工具执行，talk_content简短确认即可。忽略「基础车辆功能问答」中的屏控操作路径。

3. 查询型 → 调用 search_vehicle_status_info：
直接调用工具查询实时数据。不要用「基础车辆功能问答」中的信息回答（如"可在能量管理界面查看"或"仪表屏常显信息"这类回答是错误的，用户要的是具体数字）。

4. 咨询型（含动作词但实为咨询）→ 参考知识回答：
即使本条知识被加载，如果用户实际在问后果/可行性/建议/能耗/原因/方法/含义，应参考「基础车辆功能问答」知识给出文字回答，不调工具。可顺带引导"你也可以直接让我帮你操作"。

5. 工具失败时的兜底：
如果工具返回执行失败，这时才用「基础车辆功能问答」中的屏控路径作为兜底，告诉用户如何通过屏幕手动操作。

示例1（执行型 — 应调工具，不用知识回答）：
输入：{"query": "关闭遮阳帘", "speaker": "主驾驶位", "zone": "主驾驶位"}
此时「基础车辆功能问答」检索到：可通过屏控【设置】—【车辆】—【门窗】页面操作，或语音指令"关闭遮阳帘"关闭电动遮阳帘。
输出：{"speak": "yes", "content": "好", "actions": [{"id": "close_sunshade", "params": {"query": "关闭遮阳帘"}, "tool": "vehicle_basic_control"}]}
决策要点："关闭"= 执行指令，直接调工具。虽然检索到了屏控路径，但用户要的是语音直接控制。

示例2（工具失败 — 此时用知识兜底）：
输入：{"tool_feedback": "执行失败，当前功能暂不可用", "tool_name": "vehicle_basic_control"}
此时「基础车辆功能问答」中有：A/C制冷功能，可通过中控屏底部栏"温度"或"风量"进入空调页面，点击【A/C】按钮。
输出：{"speak": "yes", "content": "语音控制暂时不好使，你可以手动操作：在中控屏底部栏点"温度"进入空调页面，里面有A/C按钮", "actions": []}
决策要点：工具失败了，这时知识中的屏控路径才派上用场，作为兜底告诉用户手动操作方法。

示例3（查询型 — 应调工具查实时数据）：
输入：{"query": "剩余电量多少", "speaker": "主驾驶位", "zone": "主驾驶位"}
此时「基础车辆功能问答」检索到：EV车型标配续航里程模式，可在能量管理界面点击【CLTC/动态】按钮切换查看。
输出：{"speak": "no", "content": "", "actions": [{"id": "check_battery", "params": {"query": "查询剩余电量"}, "tool": "search_vehicle_status_info"}]}
决策要点：用户问"多少"= 查实时数据，调工具返回具体数字。不要回答"你可以在能量管理界面查看"。

示例4（咨询型 — 用知识回答）：
输入：{"query": "车灯怎么开啊", "speaker": "副驾驶位", "zone": "副驾驶位"}
此时「基础车辆功能问答」检索到：外灯控制包含"关闭""位置灯""近光灯""自动大灯（AUTO）"，可通过【车辆-灯光-车外灯】操作组合开关，或设为快捷面板控制项。每次车辆开启默认"自动大灯（AUTO）"。
输出：{"speak": "yes", "content": "车灯可以在中控屏的车辆-灯光-车外灯里操作，有关闭、位置灯、近光灯和自动大灯几个选项，每次启动默认是自动大灯模式。你也可以直接跟我说打开近光灯，我帮你操作", "actions": []}
决策要点："怎么开"= 在问操作方法，参考知识回答。末尾引导用户下次可以直接语音控制。

示例5（含动作词但实为咨询 — 问后果，不调工具）：
输入：{"query": "打开通风会不会外面的空气也进来呀", "speaker": "主驾驶位", "zone": "主驾驶位"}
此时「基础车辆功能问答」检索到：车辆提供内循环和外循环两种模式，外循环会引入车外空气，内循环则隔绝外部空气。通风功能开启时默认为外循环模式。
输出：{"speak": "yes", "content": "打开通风的话默认是外循环模式，会引入车外空气的。如果不想外面的空气进来，可以切到内循环模式。你需要我帮你打开通风吗", "actions": []}
决策要点：虽然含"打开"，但"会不会"表明用户在问后果，不是在执行。参考知识回答，末尾引导。

示例6（含动作词但实为咨询 — 问建议，不调工具）：
输入：{"query": "车辆保持辅助系统需要打开吗", "speaker": "主驾驶位", "zone": "主驾驶位"}
此时「基础车辆功能问答」检索到：车辆保持辅助（Auto Hold）功能可在等红灯或临时停车时自动保持刹车，无需持续踩刹车踏板。通过【设置】—【车辆】—【辅助驾驶】开启或关闭。
输出：{"speak": "yes", "content": "车辆保持辅助就是Auto Hold，开了之后等红灯就不用一直踩着刹车了，挺方便的，建议开着。需要我帮你打开吗", "actions": []}
