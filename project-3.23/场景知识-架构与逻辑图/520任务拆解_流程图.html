背景：这件事到底是什么

项目定位

司机 Agent 2.0 是字节火山引擎 × 上汽 SAIC 的智能座舱 AI 助手项目。豆包大模型作为座舱 AI 核心，通过 planner 架构进行原子化工具调用（控车、播音乐、查天气等 11 个工具）。

你的角色：负责 13 个 planner 可调用 AI 场景的「场景知识」注入。当用户点击 HMI 上的场景按钮 → 系统给豆包一段 context（提示词）→ 豆包通过 planner 决定调哪些工具。

核心约束：13 条场景知识全部塞进 system prompt（不走 RAG）。模型每次对话面对 13 条堆在一起的 ~14K tokens——区分它们的唯一锚点是你写的「场景要点对照句」。

为什么这件事的本质是"上下文工程"而不是"产品写作"？
你已经懂这个领域（excel 13 条产品规格写满了）。你卡的不是"我不知道怎么做"，而是——怎么把你脑子里对 13 个场景的区分能力，转化成模型在 long-context 下也能稳定区分的结构。这是一个信号密度和形状设计问题，不是内容量问题。

4 组易混区：互斥锚定矩阵

为什么互斥锚定是最关键的新产出？ 因为你全塞 system prompt 不走 RAG，模型一次看到 13 条。当用户说"我想睡一会儿"，模型面前同时有 08 冥想 / 09 小憩 / 13 后排勿扰三个候选。如果场景要点里没有"区别于 X 条"的对照句，模型就是猜。互斥锚定矩阵的作用是——让你在写对照句时，确保每组易混场景的区分信号互相一致、不自相矛盾。

验收标准 + 风险矩阵

验收标准分三档的意思是：底线 = 520 不翻车；加分 = 领导觉得"这个东西好用"；锁版 = 可以量化的技术指标。你的时间分配应该是——先确保底线 100% 达到，再追加分项。

🔴 底线标准（必须达到）

  • 1.13 条按钮点击后，豆包都能识别出当前场景（不串台、不空转）
  • 2.每条的执行逻辑至少走完主流程（调对工具、发对指令）
  • 3.系统回复不出 AI 味废话（不说"已为您打开后排吸顶屏"）

🟡 加分标准（好用 vs 能用）

  • 4.多轮交互形态达真实版水准（短 query / 焦躁情绪 / 该沉默就沉默）
  • 5.相邻场景 disambig 稳定（"好热"不跑到通风）
  • 6.主驾被后排打扰时的冲突处理自然

🟢 锁版指标（D7 22:00 前）

  • 7.65 个种子 query 命中率 ≥ 90%
  • 8.4 组 disambig 陷阱 query 误召率 ≤ 5%
  • 9.3 个连续多轮不漂移跨条
