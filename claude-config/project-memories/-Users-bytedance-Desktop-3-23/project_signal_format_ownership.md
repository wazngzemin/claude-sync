端上报信号的**格式/单位/示例归"产品约束"**，不是端上(汪帅/金龙)决定。汪帅2026-06原话「端状态上报产品来约束，产品要什么就实现什么」。所以遇到"单位是米还是千米/时长什么格式/地址什么结构"这类问题，**别去问端上**——这是产品 PRD 缺口。

**人物链路（导航信号这块）：**
- 仁杰 = 导航 PRD 的产品 owner（写格式/示例的人）
- 汪帅、金龙 = 端上，只按 PRD 实现；金龙会把格式问题转给汪帅，汪帅再转仁杰
- 王泽民(你) = 触发器侧产品，最清楚要什么格式（要配阈值）

**正确打法：别等仁杰从零写，主动给"建议格式"反喂确认（快10倍）。** 触发器侧建议格式标准：剩余距离=整数·米；剩余时长=整数秒、到达时间=Unix时间戳（别用"X小时X分"字符串）；地址=字符串+经纬度，收藏地址带类型枚举(家/公司)。

**仁杰(吴仁杰) 2026-06-20 确认（上汽实际）：** 剩余距离**固定单位米**、时长**秒**、到达时间 **hh:mm**；当前端上报的是**带单位字符串**（如"200m"），因老协议不支持单位字段、客户端拼好上报。→ 距离/时长跟建议格式对上了；剩两坑：① 距离"200m"是字符串、触发器配"≤500"是数值比较，**需剥单位**（建议引擎解析剥"m"取整数，端上不改，找刘杨/任军）；② 到达时间 hh:mm 无法算时间差(无日期/跨天/字符串)，时间类要不要本期+给秒/时间戳，找潘伟/仁杰。

**Why:** 问错对象(端上)会被原地打回"产品没定我不报"，白耗一轮。
**How to apply:** 格式/单位问题 → 给仁杰建议格式+@汪帅知会，不是问端上确认。详见 `信号要不到-推进手册-手把手.html` 的「端上格式」卡。关联 [[project_nav_trigger_prd]]、[[project_trigger_architecture]]、[[reference_config_platform]]。
