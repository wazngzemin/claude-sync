AI Car 意图升级讨论
一、背景与目标
传统意图识别依赖预定义分类器（如 “音乐”“导航”），存在细颗粒度意图划分模糊、多任务并行支持不足等问题。本次重构旨在通过更灵活的意图处理逻辑，提升座舱交互的准确性与自然度，核心目标包括：
解决意图过死规则定义问题
意图和FC的压力平衡
意图更具备推理能力，和任务结合
参考端状态
多意图
自定义意图动态调整
参考意图问题整理基于COT的新快慢思考架构实验
二、方案一：基于 “简单 / 复杂意图” 的轻量化拆分方案
核心思路
打破预定义分类体系，仅将意图划分为 “简单意图” 和 “复杂意图”：
简单意图：无需深度拆解，可直接执行的明确指令；
复杂意图：包含模糊表达、多需求或需上下文补全的意图，交由任务模块处理。
详细链路设计

用户输入 → 意图初步判断 →  
  ├─ 简单意图 → 原有SFT链路 → 执行指令（如“打开车窗”）→ 反馈结果；  
  └─ 复杂意图 → 任务模块 →  
     ├─ 判断需求类型（需生成任务/需澄清/纯闲聊）→  
     │  ├─ 需生成任务 → 拆解需求要素（对象/行为/目标）→ 调用合并功能模块（如“空调调节”）→ 生成指令+解释话术 → 执行；  
     │  ├─ 需澄清 → 结合场景/历史生成澄清话术（如“是想调空调还是音量？”）→ 用户确认后执行；  
     │  └─ 纯闲聊 → 闲聊模块 → 陪伴式响应；  
     └─ 涉及信息补充 → 调用工具（如POI查询、电量查询）→ 整合结果 → 执行/反馈。  
（示意图说明：链路以 “简单 / 复杂” 为核心分支，复杂意图通过任务模块动态处理，支持多意图并行与模糊需求澄清。）
关键处理逻辑
功能模块合并：将细粒度函数（如 “空调调 22℃”“空调调 24℃”）合并为粗颗粒度功能（如 “空调调节”，包含调温、调风量等操作），降低任务模块理解成本；
多意图支持：复杂意图中若包含多个需求（如 “找充电桩，同时开空调”），任务模块自动拆解为并行任务，同步调用 POI 服务与车控模块；
模糊意图处理：通过实时场景（如当前播放音乐→“调一下” 指向音量）与用户历史（如常说 “冷了”→对应空调升温）补全信息。

方案一处理流程

用户说 “有点冷，找个附近的充电桩”


1. 判定为复杂意图（含模糊需求 + POI 服务需求）；2. 任务模块拆解：   - 模糊需求 “有点冷”→ 结合场景补全为 “空调调节（升温）”；   - POI 服务需求 “找充电桩”→ 调用地图工具；3. 并行执行：调节空调至 24℃，同时推荐 3 个附近充电桩，反馈话术：“已为你调高空调温度，附近 3 公里内有 2 个充电桩，需要导航吗？”

用户说 “换首歌，这歌不好听”

1. 判定为复杂意图（明确操作 + 情感表达）；2. 任务模块拆解：   - 明确操作 “换首歌”→ 调用媒体模块切换歌曲；   - 情感表达 “不好听”→ 关联用户听歌偏好（如偏好流行）；3. 执行后反馈：“已切换下一首，为你优先推荐流行风格歌曲～”
三、方案二：基于 “需求类型 + 互斥 / 多选规则” 的分类方案
核心思路
保留分类器框架，按 “需求清晰度 + 属性” 划分为五大类，明确互斥与多选规则：
互斥性：“清晰指令” 与其他类别（服务需求、信息需求、不明确需求、闲聊）互斥；
多选性：非清晰指令的四类可并行（如同时触发服务需求 + 信息需求）。
详细链路设计

用户输入 → 一级分类器 →  
  ├─ 清晰指令 → 直接执行（如“打开空调”）→ 反馈；  
  └─ 非清晰指令（可多选）→  
     ├─ 服务需求（如“调节车内舒适度”“查找POI”）→ 服务处理模块；  
     ├─ 信息需求（如“查续航”“查天气”）→ 信息查询模块；  
     ├─ 不明确需求（如“弄一下那个”）→ 澄清模块（结合场景猜需求）；  
     └─ 闲聊（如“讲个笑话”）→ 闲聊模块；  
  （非清晰指令并行处理时，按“服务＞信息＞不明确＞闲聊”优先级整合结果）  
（示意图说明：链路以分类器为起点，清晰指令独立执行，非清晰指令按类别分流并支持并行，最终整合反馈。）
关键处理逻辑
五大类需求定义（座舱场景适配）：


需求类型

定义示例（座舱场景）

清晰指令

明确操作（如 “导航到公司”“关闭天窗”）

服务需求

需系统主动提供服务（如 “调节车内舒适度”“查找附近充电桩（POI）”）

信息需求

需获取实时 / 场景信息（如 “查续航里程”“前方限速多少”）

不明确需求

模糊表达（如 “调一下那个”“不对，不是这样”）

闲聊

情感互动（如 “无聊了，聊聊天”“讲个开车笑话”）
多选处理：非清晰指令并行时，按优先级整合结果（如 “找充电桩（服务）+ 查续航（信息）”→ 先推荐充电桩，同步反馈续航）；
模糊需求澄清：不明确需求优先结合场景猜意图（如当前在导航→“弄一下” 指向路线调整），猜不中则用高频功能引导（如 “是想调空调、音乐，还是导航？”）。

方案二处理流程

用户说 “有点冷，找个附近的充电桩”

1. 分类器判定：服务需求（调节舒适度 + 查找 POI）（非清晰指令，可多选）；2. 并行处理：   - 服务需求 “调节舒适度”→ 调用空调模块升温至 24℃；   - 服务需求 “查找 POI（充电桩）”→ 调用地图工具；3. 按优先级整合反馈：“已为你调高空调温度，附近 3 公里内有 2 个充电桩，需要导航吗？”

用户说 “换首歌，这歌不好听”

1. 分类器判定：清晰指令（换歌）+ 闲聊（情感表达）（因含清晰指令，仅执行指令，闲聊作为附加响应）；2. 处理：   - 清晰指令 “换歌”→ 调用媒体模块切换；   - 附加闲聊响应：“已切换下一首，你好像不太喜欢这首歌，下次为你避开这类风格～”

若系统需明确区分需求优先级（如 “安全相关操作＞娱乐”），且分类器迭代成本可控，优先选方案二（规则清晰，可解释性强）。
