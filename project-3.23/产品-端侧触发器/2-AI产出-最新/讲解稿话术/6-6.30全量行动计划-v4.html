一、关键术语表（中英对照 + 人话解释）

为什么要先看术语表？
因为后面的所有分析都会用到这些术语。如果你在后面看到一个词不理解，可以回来查这张表。每个术语我都用最通俗的大白话解释了它在我们项目里到底是什么意思。
术语（英文）中文翻译人话解释（在我们项目里的具体意思）
SP
(System Prompt)		系统提示词 人话：给VLM的"工作指令"。你可以理解为一张"任务单"，上面写着"你这次推理要看什么东西"。

类比：就像你给实习生写的任务说明——"今天你帮我盯着后排宝宝，看毯子有没有掉"。这张任务说明就是SP。

在我们项目里：VLM每5秒拍一张舱内照片做一次推理。它推理的时候读的全部指令就是SP。SP越详细，VLM就看得越仔细；但SP太长，推理就慢。
AlwaysOn SP常驻提示词 人话：VLM"一直在干的基础活"。不管有没有用户特别要求，VLM每5秒都会按照AlwaysOn SP的指示做一遍基础观察：看看各座位有没有人、人是男是女、大概多大、在干什么。

类比：就像保安每5分钟巡逻一圈，不管有没有警报都会巡。AlwaysOn SP就是保安的"常规巡逻路线"。

动态SP就是在常规巡逻路线上临时加的"特别关注事项"——"保安你顺便看看后排那个宝宝毯子掉没掉"。
KV槽位
(Key-Value Slot)		键值返回位 人话：你给VLM布置了一个临时任务（比如"看毯子掉没掉"），VLM做完后把结果写在一个"指定的格子"里。这个格子就叫KV槽位。Key是格子的名字（比如"blanket_status"），Value是VLM填的答案（比如"毯子在地上"）。

为什么需要：如果同时有3个临时任务（看毯子+看小狗+看手势），VLM的回答需要分开放在3个不同的格子里，触发器才知道哪个答案对应哪个任务。

端状态表里的对应字段：动态查询结果——这就是那个"格子"。
VLM
(Vision Language Model)		视觉语言模型
（端侧）		 人话：装在车机AIBox里的一个"AI眼睛"。它每5秒拍一张舱内照片，然后"看图说话"——用自然语言描述它看到了什么。

它的输出长这样（真实JSON）：
{"乘客列表": [{"位置":"主驾区域", "动态":"闭眼休息", "性别":"男", "年龄":"中青年"}], ...}

关键限制：
① 只输出客观描述（"毯子在地上"），不做判断（不会说"毯子掉了"）
② 频率固定5秒
③ 舱内图片不能上传到云端（用户隐私合规）
④ 输出可能不稳定——同一个场景，前一次说"有小狗"，后一次可能漏掉（这叫glitch/抖动）
VQA
(Visual Q&A)		视觉问答
（云端）		 人话：部署在云端服务器上的"AI眼睛"。和VLM的区别是：VQA看的是舱外的图，因为舱外的图允许上传到云端。

什么时候走VQA：用户说"看到瑞幸咖啡提醒我"、"前面有沙漠景观拍照" → 这些需要看车外 → 图片传云端 → VQA模型推理。
Caption描述性文本 人话：VLM输出的那段描述文字。比如"后排有一名女性，手持手机，穿白色衬衫"。

和标签的区别：
标签 = 有限的固定值，像下拉框（男/女/未知）
Caption = 自由发挥的描述文字，像一段话（"这个人看起来很高兴地在聊天"）

问题：标签可以直接用==号判断（性别=="女"），Caption不能——你没法写"动态=='闭眼休息'"，因为VLM可能写"小憩"、"打盹"、"睡着了"等各种表述。
枚举(enum)枚举/有限选项 人话：值是"几个固定选项之一"的字段。就像下拉框——只能选预设的几个选项，不能自己打字。

在VLM输出里，真正的enum只有3个字段：
· 位置 = 6个选项（主驾区域/副驾区域/...）
· 性别 = 3个选项（男/女/未知）
· 年龄 = 4个选项（婴儿/儿童/青中年/老年）

为什么enum重要：enum的值是固定的、确定的，触发器可以直接用等号判断（年龄=="儿童"）。不需要语义理解，不会出错。
str(字符串)自然语言描述字段 人话：值是"一段自由描述文字"的字段。VLM想怎么写就怎么写，没有固定格式。

在VLM输出里，大部分字段都是str：
· 动态 = "闭眼休息，身体仅随呼吸轻微起伏"（自由描述）
· 物品和动物描述 = "有一只猫咪安静卧着"（自由描述）

问题：同一个意思，VLM每次可能用不同的词。"睡觉"可以写成"闭眼休息"/"小憩"/"打盹"/"睡眠状态"——至少5种不同表述。触发器没法用一个固定关键词去匹配。
SFT
(Supervised Fine-Tuning)		监督微调训练 人话：用大量标注好的训练数据，教VLM模型学会一个新技能。比如教它学会在输出里新增一个"睡眠状态"标签：每次看到有人睡觉就固定输出睡眠状态="sleeping"。

好处：输出稳定、准确，触发器可以直接==判断
坏处：① 需要训练周期（几周到几个月）②可能影响模型在其他任务上的表现（VLM学会了固定输出标签，但可能变得不会灵活描述了）

丁彬的原话："如果很重要的，可能就是要去加标签，去做SFT，导给方舟"
后处理模型
(Post-processing)		后置标签转换 人话：VLM照常输出它的描述文字，然后在云端再挂一个小模型，这个小模型专门把描述文字"翻译"成标签。

举例：VLM输出"闭眼休息，身体仅随呼吸轻微起伏" → 后处理模型 → 输出标签sleeping=true

好处：不用改VLM模型本身，几百毫秒出结果
坏处：多一步处理，可能不够精准

嘉锋原话："外挂一个模型去解析，用light model几百毫秒出标签"
穿越检测
(Threshold Crossing)		阈值穿越 人话：一直盯着一个信号的值，当这个值从"不满足条件"变成"满足条件"的那个瞬间，触发事件。

举例：
触发器一直监听sleeping字段。
上一帧：sleeping=awake（没睡）
当前帧：sleeping=sleeping（睡了）
从"没睡"变成"睡了" = 穿越！ = 触发"有人睡着了"事件。

只在变化的瞬间触发：如果已经睡了很久，每一帧都是sleeping=sleeping，不会重复触发。只有awake→sleeping那一下才触发。
防抖/连续帧确认抖动过滤 人话：VLM有时候会"看花眼"——前一帧说有小狗，后一帧突然说没有，再后一帧又说有了。如果触发器不做防抖，就会在"有→没有→又有"之间反复触发，给用户发一堆错误通知。

解决办法：设定"连续3帧（15秒）都是同一个值才算真的变了"。这样一次性的glitch就不会误触发。

张兰若原话："但凡出现一次错误，触发器就会有问题"
提槽工具
(Slot Extraction)		方向/位置提取 人话：一个小工具，作用就一个——判断用户的需求是看车里还是看车外。

为什么需要：用户说"帮我看看副驾"→车里→走VLM。用户说"看到瑞幸提醒我"→车外→走VQA。这两条路线完全不同，所以需要先判方向。

西钺原话："人人尽可用，触发器直接调，不用绕VQA"
动态Advisor临时顾问 人话：用户说了一个需求（"毯子掉了告诉我"），系统临时创建一个"小监工"专门盯这件事。等这件事完成了（毯子确实掉了→通知了用户），这个小监工就自动销毁。

和静态Advisor的区别：静态Advisor是4个"永久在岗的监工"（舒适/出行/情感/内容），系统一启动就存在，不需要用户发起。动态Advisor是临时工，用完就走。
Context上下文/情景池 人话：存储车辆当前状态的一个"数据池"——里面有：谁在车上、车速多少、空调多少度、导航去哪里、VLM看到了什么……

关键定义（6.30西钺确认）：Context给Advisor的方式是"固定拼接"——推理的时候自动带上，不是Advisor主动去查。只存状态/条件，不存事件。
动态查询结果SP按需查询的
返回字段		 人话：这是VLM输出JSON里的一个特殊字段。平时AlwaysOn推理不会用到它，只有当触发器临时拼了一个动态SP让VLM去"特别看一个东西"的时候，VLM的回答才会写在这个字段里。

端状态表定义：更新频率=非周期上报，按需调用
示例：拼了SP"观察毯子状态"→VLM回答写在这里→"动态查询结果":"主驾后排宝宝的毯子已滑落"

这就是6.30会议说的"KV槽位"在真实系统里的落地！

二、VLM真实输出长什么样？（Ground Truth）

为什么这一章是全文最重要的？
因为在拿到端状态表和VLM真实JSON之前，我们只能猜VLM输出什么。猜的结果（v2）和真实情况差别非常大。你后面所有的工作——梳理标签需求、填触发器可行性评估、找丁彬要什么——全部取决于你理解VLM真实输出的结构。

所以这一章你一定要仔细看，把VLM的真实输出结构搞懂。

2.1 VLM舱内AlwaysOn的10个字段（来自端状态表）

人话版解读：VLM每5秒推理一次，输出一个JSON。这个JSON里有10个字段。其中只有3个是"下拉框"（enum）——位置、性别、年龄，这3个的值是固定的几个选项。其他7个全是"自由输入框"（str）——VLM想怎么写就怎么写，没有固定格式。
#字段名类型值是什么样的？（真实示例）更新频率人话：触发器能怎么用？
1 位置 enum 6个固定值，只能是这6个之一：
"主驾区域" / "副驾区域" / "中控扶手区域" / "主驾后方区域" / "副驾后方区域" / "后方正中区域"		 每5分钟 触发器可以直接用==号判断
比如：位置 == "副驾后方区域" → 说明后排右侧有人/物品
2 性别 enum 3个固定值：
"男" / "女" / "未知"		 每5分钟 触发器可以直接用==号判断
比如：性别 == "女" → 有女性乘客
3 年龄 enum 4个固定值：
"婴儿" / "儿童" / "青中年" / "老年"

注意：真实输出还会带括号里的具体年龄，比如"儿童（3-4岁）"、"中青年（28-35岁）"。但枚举部分是固定的4个。		 每5分钟 触发器可以直接判断
年龄 == "儿童" or "婴儿" → 有小朋友
年龄 == "老年" → 有老人
4 动态（动作和神态） str 自然语言描述，每次表述都不一样：
· "端正坐姿，双手十点位握方向盘，注意力集中于驾驶，表情中性自然"
· "左手单手握方向盘，右手握持手机。头部低垂看手中手机超过5秒"
· "闭眼休息，身体仅随呼吸轻微起伏"
· "佩戴眼罩靠在座椅上小憩"
· "自行解开安全座椅卡扣脱离约束，站立在后排座位上"		 每5秒 触发器不能直接用==号判断！

这个字段里藏着很多重要信息（睡觉、打电话、玩手机、哭闹...），但全是自由文字描述。

要从里面提取"有没有睡觉"，要么做关键词匹配（不准确），要么SFT新增标签（准确但慢），要么挂后处理模型（折中方案）。

端状态表在这个字段的备注栏写了"睡觉、打电话"——暗示这两个可能计划做成标签。
5 造型描述 str "穿白色衬衫+深色薄外套"
"佩戴眼罩"
"身着夏季短袖T恤"
"身着羽绒服"		 每5分钟 自由文字，无法==判断
但对于"穿白衬衫的那个人告诉我"这类用户需求有用。走动态SP链路，不需要标签。
6 物品和动物描述 str "放置有水杯"
"灰色小型犬和毛毯"
"有一只猫咪安静卧着，状态温顺"
"安装有空置的儿童安全座椅"
"合规笼养宠物背包，内有猫"
"放置有妈咪包，散落着安全座椅挂件玩具"		 每5分钟 自由文字！小狗、小猫、儿童座椅全在这个str里。

"有没有小狗"不是一个has_dog=true的标签，而是要从这段文字里判断"含不含'狗'或'犬'这个字"。

问题：VLM可能写"小型犬"/"金毛"/"大型犬"/"狗"——表述不统一，关键词匹配容易漏掉。
7 不安全因素枚举值 str "儿童座椅安装位置隐患"
注意：字段名叫"枚举值"但端状态表标的类型是str。需要确认。		 每5分钟 可能有用，但类型待确认。如果是真enum就太好了——直接匹配。如果是str那还是需要解析。
8 不安全因素描述 str "儿童未被安全座椅约束，站立在后排座位上存在磕碰、甩飞风险"
"驾驶员长时间看手机，视线完全脱离路面，且仅单手握方向盘"		 每5分钟 安全相关的描述文字。可以从中提取"儿童不安全"/"分心驾驶"等信号，但需要语义解析。
9 整体描述 str "舱内仅主驾位有一名28-35岁男性驾驶员，情绪平稳无波动，无社交交互"
"全车5人都正确佩戴安全带，儿童使用专属安全座椅，均处于休息或睡眠状态"		 每5分钟 全局概览信息，可以辅助验证其他字段的判断。
10 动态查询结果 str "主驾后排宝宝的毯子已滑落"

更新频率：非周期上报，按需调用！
只有触发器通过15-18拼了动态SP让VLM去特别观察某个东西时，VLM的回答才会写在这里。		 按需 这是动态SP链路的核心返回字段！

6.30会议说的"SP1的KV值"就是指这个字段。

触发器拼SP"观察毯子状态"→VLM推理→结果写在"动态查询结果"里→触发器从这里取结果→做穿越检测。

六、4条处理路径详解

为什么要分4条路径？
因为30条事件不是都用同样的方式处理。有的可以直接从enum匹配（最简单），有的需要从str解析（需要SFT），有的需要临时拼SP问VLM（动态场景），有的根本不需要VLM（用车信号就行）。你必须知道每条事件走哪条路，才能正确填写你的产出清单。
路径A enum字段 → 直接条件匹配（仅2-3种事件可用）
什么时候走路径A：当你需要的信号正好对应VLM输出JSON里的一个enum字段时。

现在只有3个enum字段：位置(6值) / 性别(3值) / 年龄(4值)。
所以只有这3类事件可以走路径A：
· "有小朋友" → 年龄 == "儿童"/"婴儿" ✅
· "有老人" → 年龄 == "老年" ✅
· "副驾是女性" → 位置 == "副驾区域" 且 性别 == "女" ✅

触发器怎么处理：直接用 == 号判断，最简单、最可靠。不需要SFT，不需要解析str。
路径B str字段需解析 → 建议SFT新增标签（最多！约12条事件！）
什么时候走路径B：当你需要的信号藏在VLM输出的str描述字段里（主要是"动态"和"物品和动物描述"两个字段），但没有对应的enum标签时。

具体事件：睡觉、打电话、玩手机、打哈欠、哭闹、有小狗、有小猫、有儿童座椅……都是路径B。

为什么说"建议SFT"：
现在的方案（从str关键词匹配）不靠谱——"闭眼休息""小憩""打盹"都是睡觉，但关键词不统一。
如果做了SFT，VLM直接在JSON里固定输出睡眠状态:"sleeping"这样的enum标签，触发器就可以像路径A一样直接 == 判断。

这些路径B的事件，就是丁彬让你"梳理出来给他导入"的核心内容！

端状态表的暗示：动态字段的备注栏写了"睡觉、打电话"——这两个可能已经在计划SFT了。你去问丁彬确认。
路径C 动态SP → 使用"动态查询结果"字段（约3-4条）
什么时候走路径C：当用户的需求是完全开放的，没法预设标签，也没法用固定的str关键词匹配。比如"比OK手势告诉我""毯子掉了告诉我""看到遗留物品提醒我"。

触发器怎么处理：
① 收到Advisor的人话需求 → ② 15-18拼SP追加到AlwaysOn → ③ VLM推理 → ④ 结果写在"动态查询结果"字段 → ⑤ 穿越检测

这条路径就是6.30下午会议的核心产出——SP拼接引擎。
路径D 舱外VQA / CV端状态 / 非VLM信号（约11条）
什么时候走路径D：
· 看车外的事件（沙漠景观/海滨景观/雪山）→ 走VQA云端，不走端侧VLM
· 分心驾驶/疲劳驾驶 → 走DMS摄像头的CV端状态，不走VLM推理
· 电量低/导航/充电/红灯 → 走车辆信号，不需要视觉

这些事件不需要你找VLM要标签。在你的产出清单里标注"非VLM，信号来源=XXX"就行。
