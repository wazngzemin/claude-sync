§1 背景：围栏下架了（2分钟）

大家都知道之前6.25的会上，俊龙投屏高德auto SDK官网的时候发现围栏服务标了下架。后来上汽高德也正式回复了——围栏服务确实不可用了。

那天会上我们讨论了替代方案。帅哥和仁杰都觉得走端侧比较合理——仁杰原话是"走端上比较合规，代价比较小"。云端围栏帅哥说"难度指数级"，所以排除了。

今天我把具体方案拉出来了，跟大家对一下。整体思路一句话：端侧定时问高德SDK"我离这个POI多远"→ 拿到距离 → 自己判断进出 → 状态变化时告诉云端。

后半段——云端引擎匹配、Advisor、Director、TTS播报——完全不改，跟原来PRD一样。我今天主要讲改了的部分。
好，那我从流程第一步开始讲——用户说了句话之后，怎么选点、怎么拿到POI坐标。

§2 Phase A · 选点 + 注册（3分钟 + 对齐）

2.1 讲选点流程

用户说"快到外婆家提醒我打电话"。云端解析出来这是一个条件触发——条件是"靠近外婆家"，动作是"提醒打电话"。

云端从用户的话里提取出POI="外婆家"，但这只是个模糊名字。要建围栏得有经纬度——所以需要先搜索POI、拿到坐标。

搜POI这步是Director调AINAVI的搜索工具。搜到之后如果有多个结果——比如外婆家(徐汇店)、外婆家(浦东店)——就出卡片让用户选。用户选完了，我们就拿到了这个POI的经纬度 + POI ID + 名称。
讲到选点，需要AINAVI的搜索能力 佳杰 / 思晴
佳杰，选点这步依赖AINAVI的POI搜索。Director调你的搜索工具，传一个关键词比如"外婆家"，你返回匹配的POI列表。我需要每个POI带三个字段：POI ID、经纬度、名称。你那边现在能返回这三个吗？
佳杰可能问的
佳杰
POI ID是高德的ID吗？我返回的搜索结果里有没有这个字段？
你答
对，就是高德的POI ID。后续查距离的时候要用这个ID去调高德SDK。如果你返回的结果里本来就带高德POI ID那最好。如果不带，我需要知道你返回的POI标识是什么，我看看能不能映射到高德的。
佳杰
多结果出卡片的逻辑是谁做的？AINAVI出还是Director出？
你答
Director来判断要不要出卡片。AINAVI只管搜索+返回列表。列表到了Director手里，Director看到有多个结果就出卡片让用户选。选完之后Director拿到确定的那个POI，再调触发器注册工具。AINAVI只负责搜，不负责选。
思晴
这对AINAVI的搜索接口有什么新的要求吗？还是用现有的？
你答
用现有的就行。我需要的就是"传关键词、返回POI列表"——这个能力AINAVI已经有了。唯一要确认的是返回字段里有没有高德POI ID和经纬度。没有新接口需求，只是确认现有字段。
拿到：① AINAVI搜索结果是否带高德POI ID + 经纬度 ② 如果不带，怎么映射

2.2 讲注册 + 围栏指令下发

选完点拿到经纬度之后，云端做三件事：注册触发器规则 → 规则入库到引擎 → 给端侧下发一条围栏创建指令。

这条指令内容很简单，就是一个JSON：围栏ID、中心坐标、半径500米、POI名称、one_shot策略。几十个字节。端侧收到后，把这个围栏存到本地，启动距离轮询器。
讲到"云端给端侧下发指令"，涉及通道选型 帅哥 / 仁杰
帅哥、仁杰，这里有个问题需要你们定——云端注册完规则之后，怎么把围栏创建指令发到端侧？走哪条通道？

数据量很小，就一条JSON，几十个字节。频率很低——只有用户设提醒的时候才发一次。
帅哥/仁杰可能问的
帅哥
走现有的哪条通道？Context下发还是端状态同步？
你答
你来定。我的需求就是"注册完后，能把一条JSON发到端侧，端侧能收到并解析"。走哪条通道你和仁杰决定——哪条改造量最小就走哪条。 不要自己挑通道。让架构定，你只描述需求。
仁杰
这个下发是同步的还是异步的？用户说完话之后多久端侧能收到？
你答
异步就行，不需要毫秒级。用户说"到外婆家提醒我"，云端注册完规则后下发指令，端侧几秒内收到就够了。用户说完话到真正靠近外婆家，中间可能半小时——不差这几秒。
帅哥
删除围栏的指令也走同一条通道？
你答
对，同一条通道。创建指令和删除指令格式一样，就是cmd字段不同——一个是create_fence，一个是delete_fence。一条通道搞定，不用分。
拿到：① 围栏指令下发走哪条现有通道 ② 需不需要新建消息类型 ③ 改造量/工期
好，选点+注册讲完了。到这步为止，端侧已经拿到了围栏信息、启动了距离轮询器。接下来讲Phase B——端侧怎么监听距离、怎么判进没进围栏。

§3 Phase B · 距离监听 + 围栏判定（5分钟 + 对齐）

3.1 讲距离怎么来的

端侧启动一个定时器，每N秒触发一次，去调高德SDK查"我离这个POI多远"。高德SDK返回一个距离数值，单位米。

这个接口是被动请求式的——我问一次它答一次，不是主动push给我。所以端侧得用定时器去轮询。
讲到"调高德SDK查距离"，需要确认接口能力 俊龙 / 丽萍
俊龙，这里是整个方案最关键的依赖——高德SDK的距离查询接口。上次你说过"POI详情信息去请求，有一个当前距离"。我现在需要跟你确认几个具体细节。
接口入参
你主动问俊龙
第一个问题：我传给你什么，你返回给我什么？我传POI ID还是经纬度？你返回的距离是直线的还是沿路线的？
不管俊龙怎么答，你接
如果他说"传POI ID" → "行，我选点的时候就拿到了高德POI ID，直接传给你。"
如果他说"传经纬度" → "行，我也有经纬度。"
如果他说"两种都行" → "那我先用POI ID，不行再换经纬度。"

关于距离类型：能给路线距离优先给路线距离，更准。但如果路线距离算力高或者非导航没法给，直线距离也够用——我的阈值是500米，这个量级直线和路线差不太多。 关键：不要说"我需要路线距离"把话说死——直线距离也能兜底。
调用频率
你主动问俊龙
第二个问题：这个接口我需要定时调，你那边能支持的最高频率是多少？理想是每秒一次，但如果有压力，3秒5秒也行。
俊龙可能反问
为什么要每秒一次？这个接口不是设计来高频轮询的。
你答
因为我要做围栏判定——判断车到没到500米范围。距离更新越快，判定越及时。但不是硬性要求。车速60km/h每秒走17米，5秒走80米——对用户来说提前80米提醒还是提醒，体感差不多。你告诉我你能支持多少，我就按多少来。
俊龙可能追问
如果同时有多个POI在查呢？3个POI每秒3次调用。
你答
正常用户同时不超过3~5个。如果有压力，我端侧做调度——5个POI轮流查，每秒只查1个，每个POI 5秒更新一次。我来管频率，不给你加负担。
非导航状态
丽萍可能问
你这个是在导航状态还是非导航？因为非导航的话我没有路线信息。
你答
两种都有。用户可能在导航中说"路过外婆家提醒我"，也可能没在导航。非导航的话给直线距离就行，500米阈值下直线距离够用了。
丽萍可能追问
非导航状态下你怎么拿车的位置？
你答
车机本身有持续定位，不是只有导航开着才有GPS。精度可能从2~3米降到10~20米，但对500米阈值够了。 这题6.25会上丽萍问过类似的"它是在导航状态吗？"，你当时回头问了佳杰。这次你自己答。
6.25会上丽萍问"距离应该是米吧？整数还是浮点？"你含糊了。
这次如果被问到："整数就行，单位米，精度到米级完全够了。比如返回479就是479米。"
拿到：① 入参格式（POI ID / 经纬度）② 返回格式（距离数值+单位）③ 支持的最高调用频率 ④ 非导航能不能用

3.2 讲围栏状态机

拿到距离之后，端侧做判定。状态机很简单——只有两个状态："远"和"已接近"。

距离大于480米 → 状态=远，什么都不做，等下一秒。
距离 ≤ 480米，且当前状态是"远" → 首次穿越，状态翻成"已接近"，产出一个fence_enter事件。

中间480到520米是死区——防GPS抖动，在边界上来回跳不会反复触发。

第一版只做这两个状态。"已到达"和"已离开"是后续repeat场景才需要的，当前one_shot场景——"到外婆家提醒我打电话"——触发一次就完事了，不需要判到达和离开。
讲到围栏状态机，这是端侧新建的模块 刘杨
杨哥，这块是你要开发的——端侧新建一个围栏判定模块。我刚才讲的状态机就是核心逻辑。整个模块拆开就四个子模块：

围栏存储——收到指令后把围栏信息存本地，重启能恢复。
距离轮询器——定时器，每N秒调一次高德SDK。
状态机——距离vs阈值判定，刚才讲的那个。
事件上报——状态变化时给云端发事件。

你帮忙评估一下这四块加起来多少工期？
杨哥大概率会问的
杨哥
定时器跑在哪个线程？每秒触发会不会影响其他业务？
你答
这个你来定。从产品角度我只管"每N秒查一次距离"，跑在哪个线程、用什么定时器，你按现有架构来。我定频率和阈值，你定技术实现。 技术实现让开发定。你只定产品规格。
杨哥
你说调高德SDK查距离——这个接口文档有吗？长什么样？
你答
刚才跟俊龙确认过了（指向俊龙刚才的回答）。你这边先按"调一个函数、传入坐标、返回整数距离"来评估。具体的接口文档俊龙那边会提供，到时候我拉你们一起对一下。
这是6.25的原题——杨哥当时问过"距离都是正数，怎么判离开"。帅哥跟了一句"离开确实很难"。你当时绕了半天绕到one_shot。

如果杨哥这次又问"离开怎么判"：
离开判定（高概率被问）
杨哥
上次我问过你——距离都是正数，怎么判离开？接近的时候距离从大变小，离开的时候距离也是从0变大，跟接近的时候一样。
你答
杨哥你说得对，距离确实都是正数。但关键是状态机有状态记忆。

接近的时候：状态=远，距离从15000降到479——穿越480阈值，判定"已接近"。

到了之后离开：状态已经是"已接近"或"已到达"了。距离从0变回100、200、500——当距离涨到520以上，状态机知道你之前是"已到达"，现在距离又超了520——所以判"已离开"。

方向用状态推导，不用距离本身。

但是——第一版不做离开。当前one_shot场景只需要一个"已接近"事件，触发一次后自锁，围栏就删了。离开判定是repeat场景才需要的，后面再做。 先给完整答案证明你想清楚了，再说"但第一版不做"砍scope。
重启恢复（有概率被问）
杨哥
用户设了提醒，车熄火重启了，围栏数据还在不在？
你答
必须还在。围栏信息写本地持久化存储——SharedPreferences也好SQLite也好你选。重启后先读一遍本地存储，有活跃围栏就恢复、重启定时器继续监听。

状态的话，第一版简单处理：重启后统一重置为"远"。one_shot场景下，如果已经触发过了围栏早就删了。如果没触发，重置为远重新判就行——最多到了的时候重新触发一次，不会有副作用。
拿到：① 围栏模块整体开发工期评估 ② 第一版scope确认：只做远→已接近→锁定 ③ 他需要你提供什么（高德SDK接口文档 → 你拉俊龙和杨哥对接）
好，到这步端侧判定完了——距离穿越480米，状态机判"已接近"，产出一个fence_enter事件。接下来这个事件要上报到云端，触发后面的链路。

§4 Phase B→C · 事件上报 + 触发链路（3分钟 + 对齐）

4.1 讲事件上报

端侧状态机判定"已接近"之后，要把这个事件上报到云端。上报的内容是一个JSON——event_type、fence_id、poi_id、distance。不带经纬度，坐标不出车。

事件到了云端之后，走触发器引擎匹配——DSL规则命中了就触发Callback，后面就是标准链路了：Advisor生成建议 → Director推理出话术 → TTS播报。这段跟原PRD完全一样，不改。
讲到"事件上报到云端"，需要确认上报通道 帅哥 / 仁杰
帅哥、仁杰，刚才对齐了下发通道，现在反过来——端侧围栏事件怎么上报到云端？走现有的端→云链路吗？
帅哥/仁杰可能问的
仁杰
这个事件跟现在的端状态上报走同一条链路吗？
你答
不太一样。端状态上报是每秒一次、固定字段。围栏事件是偶发的——可能半小时才一次，但发了就得送到。不能丢——用户等了半小时快到了，结果事件丢了没触发，体验就坏了。

具体走哪条你们定。我的要求就是：能送到、别丢。
仁杰
事件到了云端，引擎⑮那边需要适配什么吗？
你答
需要引擎能识别event_type=fence_enter这个新事件类型。围栏事件和导航距离信号走的是同一个入口——都写入信号库⑫，引擎统一消费。区别只是event_type不同。你看引擎那边加一个类型适配的工作量大不大？
拿到：① 事件上报走哪条端→云通道 ② 引擎⑮适配fence_enter事件类型的工期

4.2 讲触发后的链路（不需要对齐，一笔带过）

事件到了引擎之后，后面全是原来的链路：引擎匹配DSL规则 → 命中 → Callback推给Advisor → Advisor生成建议 → Director推理话术 → TTS播报"快到外婆家啦～要不要先打个电话给外婆？"

端到端延迟大概3~5秒——引擎匹配20毫秒、Advisor半秒、Director推理2秒、TTS合成半秒。

这段不改，跟原PRD完全一致。我就不展开了。

§5 Phase D · 生命周期管理（1分钟）

最后说一下生命周期。

one_shot场景：触发一次后，端侧自动删围栏、停轮询，云端标记规则已完成。双侧清理，闭环。

用户取消：用户说"取消提醒"→ 云端下发delete指令 → 端侧删围栏。

异常：如果高德SDK查距离挂了，端侧降级用GPS + haversine公式算直线距离。精度低一点但能用。隧道没GPS的话暂停轮询，出来了继续。

Phase D没有新的对齐项——都是端侧和云端自己处理的逻辑。

"弱网/隧道怎么办"

你答
分两种：没GPS（隧道）——SDK查不到准确距离，状态机不做判定，等出来GPS恢复了继续。不会误触发。
没网（地下车库）——距离查询是端侧本地调SDK，不需要网络。唯一需要网络的是事件上报——如果没网就先缓存，有网了补发。
"跟原PRD是什么关系"
任何人

你答
不是推翻，是升级。只改了"距离从哪来"和"谁来判"。原来是导航引擎给距离、云端判；现在是高德SDK给距离、端侧判。

后半段——引擎匹配、Advisor、Director、TTS——完全没动。

而且新方案覆盖面更广——原方案只能在导航中用，新方案导航不导航都能用。是进步不是退步。
