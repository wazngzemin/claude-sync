§1 产品概述

1.1 产品定位与愿景

触发器是豆包上车（Driver Agent 2.0）智能座舱的"条件监控引擎"。用户不只想"我说你做"，还想要"车里热了自动开空调""10 分钟后关加热""副驾睡着了帮我调暗灯光""看到充电桩自动开盖"——这些都是"等某个条件出现，自动做件事"。

一句话定位：触发器 = 门口的保安 + 床头的闹钟。系统里会动脑子的东西（Director、Advisor、VLM）都是模型，模型只在"被叫的那一秒"活着。可世界一直在变（电量在掉、宝宝会醒）。谁来 7×24 盯着世界、在该出手那一刻捅模型一下？就是触发器。它自己不动脑子，只用便宜的"死规矩判断"（等于/大于/变更为）挡掉 99% 的噪音，让贵的模型只在条件命中那一秒被叫醒。西钺原话："Advisor 不做条件判断，所有条件判断类任务都交给触发器。"

1.2 用户研究（4 个真实痛点）

痛点说明
主动服务缺"条件监控"LLM 没有常驻监控能力，"车里热了自动降温"这类得有个中间层盯着
条件场景已 90+ 条静态 Advisor 45 + 动态视觉 35 + 预设条件 20+，各写一套逻辑不可维护，必须统一注册/判定/执行引擎
端云协同是硬需求定时器必须本地（断网不停摆）、VLM 在端侧（隐私+延迟）、复杂判定在云端（算力+仲裁），纯云或纯端都不够
Advisor 需要耳目动态 Advisor 的"照看宝宝"、静态 Advisor 的"低电量推荐"，都需要触发器做底座

§2 整体功能说明

2.1 云端干什么、端侧干什么、为什么这么分

逻辑上一个触发器，一个条件只判一次；物理上必须劈两半（6.22 西钺警告"别开发两个触发器"）。因为有物理约束绕不过：

只能在端侧的三件事只能在云端的三件事
① 掐表：本地倒计时才准，云端有延迟、可能没网
② 断网也得干：进隧道前设的"10分钟后开窗"云端够不着
③ 离摄像头近：舱内画面合规上不了云		① 解冲突：多任务撞车要全局视角排序
② 懂人话："毯子掉没掉"要过模型理解
③ 统一登记：校验/查重/拼 SP/审计一处收口
☁️ 云端触发器 = 前台 + 大脑🚗 端侧触发器 / AI Box = 手脚 + 眼睛 + 闹钟
干什么登记校验 · 条件判定/穿越/防抖/冷却 · 解冲突仲裁 · 懂人话过模型判 · 拼 SP 管槽位 · 叫醒下游本地定时器掐表 · 点火 AlwaysOn(舱外3s/舱内动态5s/舱内静态更慢) · 调度 VLM 看 · 判死规矩 · 断网本地直执车控 · 透传上报
拿什么料Context（记忆/车况/日历）、端状态、注册请求摄像头+端侧 VLM、本地定时器、CAN 车控、端侧配置平台
交接铁律：不许上游直接捅端侧。Advisor/Planner 想让端侧看点什么，必须先经云端触发器（6.30 西钺"不行，先到云端"）。下行：Planner/Advisor→云端触发器(校验+拼SP)→端侧(注入 AI Box)；上行（视觉结果回云端）有两条走法，定版=混合（实时走原路返回 + 异步落 Context 审计），见 §6 附录。
三个最容易串台的名词（6.30/7.3 西钺）：感知器=调度器（管调谁去看，VLM 每5秒是它的逻辑）；VLM=端侧模型，只客观描述、不做判断、不给时序（"宝宝闭眼安静"是描述，"哭了没"是判断，判断在 Advisor）；VQA=拍一张问一次答一次（能联网）。

2.2 主链路：一个请求怎么从入口走到执行

触发源四类入口运营配规则 / 用户对话 / 车况信号 / 视觉信号
→
云⑮ 注册校验查 iCanDo 能力清单 + 查重
→
云⑮ 判定闸门链条件判定→穿越→防抖→冷却→仲裁
→
出口三类出口白名单车控 / push Advisor / callback Director

为什么要一串闸门：叫醒模型很贵，每道闸门都在"能不能不叫醒模型"上再拦一次——用最便宜的判断挡掉 99% 噪音，这就是触发器省钱的地方。

§3 静态 Advisor 场景（配置平台预配 · 出口 push Advisor）

这一大类的通用逻辑（人话）：运营在配置平台把"什么条件 → push 哪个 Advisor"配好，触发器 7×24 盯着 Context 里的信号。条件成立，触发器不自己出主意，而是 push 对应静态 Advisor（出行/情感/舒适/内容），Advisor 查 Context 想一个方案（选哪个充电站、说什么话），再交 Director 执行。触发源是 Context + 触发器，不是 Advisor 自己轮询——Context 不会主动通知（7.4 三方证实），所以必须触发器盯到了 push。全类 45 条：出行 18 / 情感 13 / 舒适 9 / 内容 5。下面抽 6 条代表。

3.1 低电量推荐-未发起导航 出行 · P0

注：电快没了、又没在导航，车主动说"附近有充电站，帮你导过去？"。这是最典型的"车况信号 + 导航状态"双条件静态场景。

业务流

端电量下降battery 25→20% 经 CAN 上报
→
云⑮ 判定 ANDbattery≤X% 且 nav=未导航
→
云⑮ 穿越+冷却首次跨阈 · 本次上电未推过
→
云push 出行 Advisor+ 启 3min 计时器
→
云Advisor 查 POI找最近 3 个充电站选评分高的
→
执行Director 弹窗+TTS"附近有充电站，导过去？"

需求说明

项内容
触发条件（2 AND）battery_level lte X%（X=用户设的电量提醒值）　AND　nav.status eq 未导航
触发动作弹窗（标题"低电量提醒"/文案"附近有充电站，是否前往？"）+ TTS「电量偏低，附近有充电站，要不要帮你导过去？」；用户确认 → 发起新导航到充电站
频控/失效一次上电周期仅一次；push 后 3min 无响应自动失效
数据怎么流① 端侧 CAN 把 battery 上报到端状态（云端）② 触发器盯到 battery 变更 → 取信号快照做 AND 判定（云端，零模型）③ 命中 → push 出行 Advisor ④ Advisor 查 Context 里的 POI 库选站（云端推理）⑤ 交 Director 弹窗
研发要做什么触发器（庆磊/刘杨）：battery+nav 双条件判定、穿越、per_power_cycle 频控、3min 计时器（到期发 event_expired）；出行 Advisor（弋途/庆磊）：收 event 后查 POI 选站的推理；Director：弹窗+TTS+确认后调导航
边界 case① 附近无充电站（搜索空）→ Advisor 改播"附近暂无充电站"、不弹窗　② 用户拒绝 → 记 user_rejected，本行程 frozen 不再推　③ 用户突然发起导航 → 条件②变 False，不触发
异常3min 计时由触发器侧维护（Advisor 无状态不计时）；battery 信号超时/TTL 过期 → 场景冻结不误触发

3.2 低电量推荐-已发起导航 出行 · P0

注：电快没了、正在导航中，车说"沿途有充电站，顺路补个能？"。和 3.1 是同族，差别在——找的是沿途站不是附近站，动作是加途经点不是起新导航。

业务流

端电量下降导航中 battery→20%
→
云⑮ 判定 ANDbattery≤X% 且 nav=导航中
→
云push 出行 Advisor
→
云Advisor 查沿途站按路线 5km 内选顺路+评分
→
执行确认→加途经点ETA+3min 重算路

需求说明

触发条件battery_level lte X%　AND　nav.status eq 导航中
触发动作TTS「电量偏低，沿途有充电站，顺路补个能？」；功能2（用户确认后）= 把推荐站加为途经点
数据怎么流 / 研发与 3.1 同，差异：Advisor 查的是导航路线上的沿途站（要 nav 的 route 数据），动作是 route_add_waypoint。二段式：推荐与执行分离，用户拍板才加点
边界/异常无沿途站 → 降级纯提示；用户拒绝 → frozen；导航突然取消 → nav.status 变，场景自然失效

3.3 单人上车-有 Face ID 用户常规问候 情感 · P0

注：注册过脸的车主一个人上车，车叫出昵称、结合时间和近期记忆问候，如"早上好泽民，今天 9 点有产品评审会"。情感类的即时性最高——纯语音、无弹窗、不失效。

业务流

端上车+人脸VLM 出 seat + face_id
→
云⑮ 判定 AND仅一人 且 face_id 已识别
→
云push 情感 Advisor带 face_id
→
云Advisor 查记忆昵称+时间段+日历+近期记忆
→
执行Director TTS个性化问候（无弹窗）

需求说明

触发条件seat.occupancy eq 仅一个座位　AND　face_id.status eq 已识别
触发动作 / 出口push 情感 Advisor → 查用户记忆生成话术 → Director 纯语音播报。满足条件即触发、无失效
研发触发器：seat + face_id 双条件判定；情感 Advisor：结合 face_id 查记忆/日历生成话术；依赖 face_id 信号（VLM 已有）

3.4 低温接人暖手 舒适 · P0

注：大冷天有人上车，车主动开座椅加热 + 方向盘加热暖手。这条要重点讲触发器 vs 车控执行的边界——触发器不直接调车控，是 push 舒适 Advisor → Advisor 给建议 → Director 调车控。

业务流

端新乘员入座seat change_to 已占用
→
云⑮ 判定 AND新入座 且 外温≤5℃ 且 座椅加热=off
→
云push 舒适 Advisor
→
云Advisor 出建议识别座位→开哪个座椅加热
→
执行Director 调车控座椅+方向盘加热+TTS

需求说明

触发条件（3 AND）seat.states change_to 已占用　AND　outdoor_temp lte 5℃　AND　seat_heater.status eq off（避免重复开）
触发动作TTS「外面挺冷，帮你开了座椅和方向盘加热」+ 开对应座位座椅加热(中档)+方向盘加热
出口边界（评审高频问）座椅加热是触发器直执还是 Director 执行？→ Director 执行。触发器只 push 事件给舒适 Advisor → Advisor 给建议 → Director 调车控。触发器不直接调车控（车控白名单只给 A 类预设场景，见 §5）。这是分工边界，董均 7.2 专门问过。
边界/异常座椅加热已 on → 不重复触发；车控执行失败 → 记日志 + TTS 报错

3.5 情绪个性化问候 情感 · P1 · 视觉情绪

注：车看出你今天开心还是疲惫，说的话跟着情绪走。这条是静态 Advisor 里用视觉信号的代表——条件里带了 VLM 的情绪识别结果。

需求说明

触发条件（3 AND）seat.occupancy neq 无人　AND　face_id.status eq 已识别　AND　情绪识别结果 neq 中性
触发动作（按情绪分支）开心 → TTS「看你心情不错，有什么好事？」；疲惫/低落 → TTS「看起来有点累，听首轻松的歌？」
数据怎么流情绪是 VLM 舱内 AlwaysOn 出的枚举字段（开心/疲惫/中性），走端状态上云 → 触发器直接 eq 判（L1 零成本）→ push 情感 Advisor
边界情绪=中性 → 不触发（避免尬聊）；情绪字段需 VLM 精调为枚举（非 caption），否则走 L2

3.6 儿童 / 老人乘车 舒适 · P1 · 视觉+双分支

注：车看到有儿童或老人上车，自动调整舒适策略。一条场景两个分支，展示"同一视觉信号、不同人群、不同动作"。

需求说明

触发条件年龄（VLM 枚举：婴儿/儿童/青中年/老年）eq 儿童 或 老年　AND　seat.occupancy neq 无人
触发动作儿童分支 → 开儿童锁 + 后排空调避直吹 + 车窗锁定；老人分支 → 温度调 22-24℃ + 降风量
数据流 / 判定层年龄是 VLM 已精调枚举 → L1 直接 eq 判；儿童/老人走枚举，宠物那种走 caption 提槽（L2）
安全红线儿童锁属安全类，需二次确认（见 §6 安全红线）

§4 动态 Advisor 场景（用户对话即时拆解 · 5 种事件类型）

这一大类的通用逻辑（人话）· 这就是红线：和前后两类最大的不同——条件不是预置的，是用户临时张嘴说的开放需求（"照看后排宝宝，哭了告诉我"）。VLM 从没约定过"宝宝哭没哭"这个信号，所以：动态 Advisor 把这句人话拆解 → 触发器临时拼一句 SP 让 VLM 顺便去看 → VLM 每 5 秒如实描述 → 结果回传 → 云端 Advisor 判"算不算哭了"。全类 35 条，是"5 种事件类型 × N"的展开，不是固定枚举清单——所以按事件类型讲规则 + 代表 case。

4.0 动态任务完整业务流（以"照看宝宝哭了告诉我"为例）

注：这是动态 Advisor 的主链路，7.3 晓伟会讲透的。所有动态场景都走这条骨架，先把它讲清，后面 5 类只讲各自差异。
用户①张嘴"照看后排宝宝，哭了告诉我"→ASR→Director
→
云②Director 判长时任务调 goal_list_update 建目标 + 动态 Advisor
→
云③Advisor 拆解注册定时器(30s触发我) + 拼感知需求
→
云④触发器拼 SP"看宝宝情绪"拼进舱内 always-on
→
端⑤VLM 每5秒看只描述"安静/有泪"，不判断哭没哭
→
端⑥结果回传文本经端状态上云(走混合方案)
→
云⑦30s 到点 Advisor 判取 Context 最新文本判"算不算哭了"
→
执行⑧达成→出建议→DirectorTTS"宝宝在哭，回头看看？"+还原 SP
★ 两个频率是两条独立的循环，别混（7.3 庆磊反复确认，评审必问）：
· 端侧 5 秒 = 生产端：VLM 的 always-on 循环，不管云端要不要，SP 拼进来就每 5 秒跑一次、把文本上报，"没哭没哭…直到哭了"。
· 云端 30 秒 = 消费端：Advisor 是模型、自己不会跑，触发器注册一个 30 秒定时器踢它一下，它才去 Context 取最新文本判一次。
两者解耦：端按识别能力定 5 秒，云按任务需要定 30 秒。庆磊原话："触发器每 30 秒去检查最新宝宝情绪；VLM 按自己频率定时上报文本，就像空调温度、车窗状态那种。"
★ 硬约束（7.3 拍板）：视觉条件类任务必须先降级成"定时任务"。逻辑上"宝宝醒了告诉我"是个条件任务，但端上现在不支持条件任务，所以必须在 Director 侧把它注册成定时任务（每 30 秒轮询 Context），而不是条件任务。对比：车窗状态本就支持条件，"车窗开就播音乐"可正常注册成条件任务；但视觉相关的一律先降级成定时。
★ VLM 只描述不判断（7.3 既定策略）：VLM 像一只眼睛，只客观描述（"宝宝闭眼安静"），不做业务判断（"这算不算哭了要不要提醒"）。判断放云端 Advisor。反例（西钺）："帮我看乘客难不难受"——VLM 只能看"醒着"，看不到"难受"（对难受无先验），"难受↔醒着"的映射在云端结合记忆拆解，下发给 VLM 的是客观事实"是否醒着"。暂定：是否类(哭没哭)可让 VLM 试；难受这类抽象状态一律云端拆解。

4.1 出现类：{位置}出现{非持续行为} 如 比OK手势 / 打哈欠

注：用户说"我比 OK 手势时就开始三周年仪式"，之后每次比 OK 就触发一次。瞬时动作，单帧判断即可，天然无重复问题。

需求说明

触发条件gesture eq "ok_gesture"（+ position=副驾）· 判定层 L1 枚举（gesture 已精调）
触发动作 / 频控push 动态 Advisor → 拉起预设仪式(音乐+氛围灯+TTS)。Context 里有就给一次，不限次数（瞬时动作无重复问题）
边界用户比多次 OK → 每次都触发；用户说"取消" → 动态 Advisor 发取消订阅 → ⑮ 删注册
异常VLM 识别失败 → Context 写 gesture=null → 不触发（静默）

4.2 开始类：{位置}开始{持续行为} 如 开始睡觉

注：用户说"留意下后排宝宝"，宝宝真睡着（不是眨眼）时触发一次。关注行为状态变化（睁眼→闭眼），要监听多帧。

需求说明

触发条件eye_state change_to 闭眼（从睁→闭，监听变化非单帧）　AND　闭眼 duration_gte 3s（防瞬眼误判）· 判定层 L2 提槽
触发动作 / 频控push 动态 Advisor → 建议调低音乐/调暗灯 → Director+TTS「宝宝睡着了，帮你调了音乐和灯光」。change_to + duration → 仅触发一次（关键：不能单帧判 eye==闭眼，否则每周期都触）
边界 case（重要）① 瞬眼(<3s) → 不触发　② 宝宝上车就在睡（首帧已闭眼）→ 无 change_to 事件、不触发 → 需另配"存在睡觉"场景(§4.4 存在类)互补
异常VLM 误判 → 记日志 + 可选 TTS 提示

4.3 结束类：{位置}结束{持续行为} 如 结束睡觉

注：宝宝睡醒了，车把之前调低的音乐灯光调回来。和"开始睡觉"对称反向。
触发条件eye_state change_to 睁眼（闭→睁）　AND　body_movement neq 静止（确认真醒非短暂睁眼）
触发动作push 动态 Advisor → 恢复音乐/灯光 → Director+TTS「宝宝睡醒了，帮你调回来」。反向 change_to 触发一次
边界短暂睁眼又闭上（<阈值）→ 加 body_movement 条件排除，不误触发

4.4 存在类：{位置}存在{物品/动物/人物} 如 存在小狗

注：用户说"留意下后排小狗"，只要小狗在就持续保持关注。关注物体存在，是持续状态判定——和"开始/结束"关注行为变化不同。
触发条件object_list contains "宠物" → eq True（持续返回）
触发动作 / 频控push 动态 Advisor（每周期都推）→ 生成持续关注话术。存在类每周期都触发 → 需在 Advisor 侧做频控（周期内只播一次），避免重复打扰（与出现类天然单次不同）
边界与"出现"区别：存在类每检测到都上报；出现类只在从无→有触发一次。这条也用来兜"宝宝上车就在睡"的首帧问题（§4.2 边界②）

4.5 移除类：{位置}移除{物品/动物} 如 移除小狗

注：用户说"留意小狗下车"，小狗从车里没了就提醒。存在的反向。
触发条件object_list contains "宠物" change_to False（从有到无）
触发动作push 动态 Advisor → 建议检查小狗是否安全 → Director+TTS「小狗下车了，注意安全」。T→F 穿越触发一次
边界和"结束睡觉"都是反向 change_to，但监听信号源不同：移除小狗看 object_list（物体存在），结束睡觉看 eye_state（行为状态）

§5 预设条件任务场景（车企预配 · A 车况 / B 视觉标签 / C 组合）

这一大类的通用逻辑（人话）：车企（赛力斯/上汽）出厂就配好的任务，不是用户说的，配在端侧配置平台。分三小类：A 车况（条件=CAN 信号，触发后白名单车控直执，全类唯一能直执的）；B 视觉标签（条件=VLM 精调枚举，push Director）；C 组合（多信号源，push Director）。全类 20 条。

5.1 充电桩-自动开充电口盖 A 车况 · P0 · 链式触发

注：停车挂 P 后，视觉确认真有充电桩，自动开充电口盖免下车。这条是 7.4 庆磊拿来把丁彬问懵的经典 case，要讲透它踩的三个点：配在端侧、车况+视觉两条件、端侧只出感知结果决策在上面。

业务流

配置端侧配置平台预配PRESET-A-CHARGING-001
→
端①挂 P 档vehicle.states R→P
→
云②判定①+穿越change_to P（首次）
→
端③oneshot 调 VLM看车位有没有充电桩
→
云④判定②charging_pile eq 有（原路返回）
→
执行白名单车控open_charging_port

需求说明

触发条件（2 条链式）① vehicle.states change_to P（R→P 穿越，不是 ==P，避免每 5s 扫描都触）　② charging_pile_visible eq 有（oneshot 调 VLM 确认，非周期）
触发动作 / 频控白名单车控直执 open_charging_port + 可选 TTS「已为您打开充电口盖」；一次停车周期仅一次
数据怎么流（三个点）① 配在端侧配置平台（不是云端）② 条件是"车况(档位)+视觉(充电桩)"两个一起 ③ 端侧只出"有没有充电桩"结果，开不开的综合决策不在端上（要参考别的信息+语音确认）
研发要做什么触发器：档位穿越判定 + oneshot 调 VLM + 回证结果判定②（走原路返回 §6）+ 白名单校验；VLM：舱外元素识别"充电桩有/无"；性能：oneshot 按需调用约 1.5s（1080P，丁彬 7.2）
边界本周期冷却用完后即使再 R→P 也不触发；熟客/生客无关
异常① VLM timeout → 写 charging_pile:timeout、不开盖、可选 TTS「未确认到充电桩，如需请说一声」② VLM 返回"无" → 条件②False 不触发 ③ 白名单查不到 → 降级 push Director 问"要不要开盖" ④ 车控失败 → 重试 1 次仍失败 → TTS 报错 + push Director
评审接招：庆磊问"这不就是静态 watcher 吗"→ 答：原来静态 watcher 管这类预设条件，现在把"带视觉的预设条件"从 watcher 拆出来给端侧触发器管，所以它既不是纯 watcher、也不是动态 Advisor，是端侧触发器的系统预设任务。

5.2 天气路况保护-雾天后雾灯 A 车况 · P0 · 双向触发

注：识别到雾天自动开后雾灯，雾散了自动关。要讲清一条场景其实是"开启 + 恢复"两个独立的双向触发。
触发条件（拆两个独立场景）开启：weather_type change_to 含[雾,大雾]；恢复：weather_type 变更为不含[雾,大雾]
触发动作 / 频控开启 → open_rear_fog_light + TTS「已开启后雾灯」；恢复 → 关后雾灯（均白名单直执）。双向各自独立，默认 per_day=2（避免持续雾天反复触发）
边界（用户意图优先）用户手动关雾灯 → 检测到 fog_light_switch=off → 标"用户意图"、本周期 frozen 不再自动开 → 直到下次 weather_type 变更
异常 / 多源冲突天气数据源失效 → 降级用 VLM 舱外 AlwaysOn（"天气描述"enum 含"浓雾"，3s 周期）；外部天气为主、VLM 为备，多源融合不在触发器层解决（触发器只看配置指定的一个信号源）

5.3 AI 展销-陌生访客打招呼 B 视觉标签 · P1 · 多条件 AND

注：展车模式下有陌生人（没见过的脸）上车，主动打招呼 + 功能引导。展示多条件 AND 怎么判。
触发条件（3 AND）① exhibit_mode eq on（🟠N3 待新增）② seat.states change_to 已占用 ③ face_id eq 无匹配（不认识）
触发动作 / 频控push Director → 打招呼+功能引导话术(基于短期记忆)+可选氛围灯。每位新访客一次（按 face_id 去重）
判定说明（评审必问）3 条件是同时判还是串行？→ ⑮ 事件驱动+状态快照：任一信号变更触发候选 → 取当前所有信号快照做 AND → True/False，不是串行查询
边界① 熟客（face_id 匹配）→ 条件③不满足、不重复打招呼 ② 多访客同时上车 → 按座位逐个判定、每位新访客各触发一次 ③ exhibit_mode 未上线 → 本场景 frozen 等信号

5.4 AI 展销-试驾临终点小结 B 视觉标签 · P1 · 导航距离

注：试驾快到终点时，车主动小结刚才的试驾体验。用来展示"导航距离信号触发"，复用导航"接近"事件。
触发条件（3 AND）① test_drive eq 进行中（🟠N4）② nav.status eq 导航中 ③ nav.distance lte 300m
触发动作 / 出口push Director（带任务原文"总结用户刚才的体验/需求"）→ Director 查短期记忆生成小结话术 → TTS。触发器只 push、不生成话术（嘉锋 7.2 原话："director 直接总结，你只需要发给 director 一句话"）
边界用户中途取消导航 → nav.status 变、条件②不满足；距离跳变(350→280) → 防抖：连续 2 帧≤300 才触发；test_drive 未上线 → frozen

5.5 熟路 PLUS C 组合 · P1 · 位置信号

注：走到常走的通勤路（家-公司）时，推荐智驾接管或智舱服务。组合类代表——两类独立触发方式 OR。
触发条件（两类 OR）① familiar_route change_to 是（赛力斯识别上报 🟠N1）② current_address 与记忆点距离 <阈值（=电子围栏到达，复用触发器端侧围栏）
触发动作 / 出口push Director → 推荐智驾/智舱 + TTS「进入 XX 路，是否开启智驾？」。同一熟路一次行程仅一次（按 route_id 去重）。智驾部分归汇文承接，其余原子化座舱任务归触发器（嘉锋 7.2）
边界familiar_route 未上线 → frozen；用户拒绝 → 本路线本行程不再推；多条熟路重叠 → 按 route_id 各自独立频控

5.6 用车助手（页面反复翻找） C 组合 · P1 · 埋点事件

注：用户在某设置页反复翻找找不到功能时，车主动讲解这个页面。组合类里"埋点信号触发"的代表。
触发条件page_id eq 指定页 AND behavior eq 反复翻找 AND count ≥ 阈值（🟠N2，建议 5 次点击/30 秒内）
触发动作push Director → 讲解该页面功能与用法，如「座椅设置可以调节…」。同一页面同一行程仅一次
边界 / 供方埋点信号未上线 → frozen 等赛力斯研发；用户主动退出页面 → 不触发；阈值有争议 → 拉群定协议。信号供方=赛力斯研发埋点（带 page_id + 反复翻找次数）

注：迎宾模式下识别明确的视觉标签（大件行李、怀抱婴儿、行动不便等），触发对应迎宾动作。这是"一条链路、只换 contains 关键词和动作"的批量类。
