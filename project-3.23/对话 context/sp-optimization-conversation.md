# 动态示例 vs 动态知识:详解、做法与优化目标

## 一、先看背景:为什么要"动态"

当前 Planner 的 System Prompt 已经膨胀到 **~32K–34K tokens**,其中第⑥块"参考示例"就占了 ~15K+,约 SP 的一半。这种"什么都往里塞"的模式带来了一连串问题:

| 症状 | 触发场景 |
|------|----------|
| 注意力分散 | 一个 query 里 4 件事,中间 1–2 件被丢 |
| 幻觉 | 编造用户从未说过的偏好 / 记忆 |
| 推理不结合 context | 拿到了 env_info 但不用 |
| 性能不稳定 | 越塞越长,回答越没准 |
| KV Cache 命中率下降 | 长对话(>20 轮) 容易被挤出缓存 → 更慢 |
| 规则压死创造力 | 10 条注意事项 + 69 个示例把模型框成"流水线工人" |

核心思路:**SP 里只留高频、通用、教主逻辑的内容;剩下的全部转成"按需投喂"。** 这就分出了两条动态化路线——动态示例、动态知识。

---

## 二、两者的本质区别

很多人会把这两个混着说,实际上它们是**两种不同形态的"教学材料"**:

| 维度 | 动态示例 (Dynamic Examples) | 动态知识 (Dynamic Knowledge) |
|------|------------------------------|------------------------------|
| **本质** | Few-shot 演示 | 陈述性知识 / 规则 / Q&A |
| **教什么** | "怎么做" —— 通过模仿来学 | "是什么 / 边界在哪" —— 通过规则来学 |
| **形态** | 一个完整 case:用户 query + Planner 完整 JSON 输出(含 talk_content + action_list) | 一句话或一段说明:概念、参数、规则、tips |
| **来源** | SP 第⑥块 69 个参考示例 | SP 第⑦车辆设备知识 / ⑧工具使用 tips / ⑨其他知识 / 通用任务经验 |
| **典型例子** | "我好热" → 调 search_vehicle_status_info + vehicle_basic_control 的完整 JSON | "D6X 天窗最大开度 50%" / "点外卖任务最好先确认地点" / "发微信前先向用户确认内容和收件人" |
| **检索 key** | 当前 query 改写 + 当前 tool_feedback | 任务类型 / 领域分类 |
| **典型粒度** | 200–500 tokens / 条 | 50–200 tokens / 条 |

> 一个类比:动态知识 = 教科书的"知识点/定理";动态示例 = 教科书的"例题"。两者互补——光给定理没例题,模型学不会怎么用;光给例题没定理,模型学不会泛化。

## 三、怎么做(实施路径)

### 1. 拆分:静态保留 + 动态外置

**示例侧**:把现有 69 个示例做分类(多步导航、车控+闲聊、记忆相关、GUI 操作、条件判断、TTS 状态、打断恢复 …),挑出 **10–15 个最具代表性的核心 case 留在 SP 常驻**,标准是:

- 优先选第一轮交互(轮数少、tokens 短)
- 必须覆盖四个能力维度:多工具协同、串/并行判断、困难场景灵活处理、个性化理解
- 是"教主逻辑"而非"教某个工具用法"的示例

剩下 50+ 示例全部下沉到**示例库**。

**知识侧**:除了 D6X 设备规格这种几乎所有 query 都用得到的留 SP,其他(POI 处理 tips、外卖任务经验、发消息 tips …)全部下沉到**知识库**。

### 2. 建库:可检索的存储

用 VikingDB 这类向量库分别建:
- 示例库(按场景类别分桶 + 向量索引)
- 知识库(按领域知识 / 通用任务经验 / 工具知识 / 专有知识 分桶)

### 3. 检索:两条触发路径

**路径 A:基于 query 检索**
- 用户当前 query 做向量召回
- 必要时先做一轮 **query 改写**(口语化 query 召回率低 → 改写成更"知识库语言"的形式)
- 改写要多一次模型调用 → 用**并行化**(与其他 context 模块并行)解决延时

**路径 B:基于 tool_feedback 检索**
- 上一轮调了哪个 tool,就拉这个 tool 相关的示例
- 例:上一轮调了 `search_poi_qa` → 下一轮自动注入 POI 相关示例
- 这条路径**几乎零额外延时**(不需要 query 改写)

### 4. 注入:SP 留动态槽位

SP 不能是一坨写死的字符串,要做**模块化**:

```
SP = ① 主逻辑(Director 核心,几乎不改)
   + ② 工具描述模块
   + ③ 知识 & Context 模块  ← 动态知识塞这里
   + ④ 示例模块             ← 动态示例塞这里
   + ⑤ 动态变量(status / goal_list / env_info / memory)
```

工程侧:在调 Planner 之前,由前置筛选层(情景 RAG / Advisor)把检索结果填到对应槽位,再发给 LLM。

### 5. 延时账:能不能扛得住?

字段精简(`speaker_name → speaker`、`sp_position → 音序`、`tool_name → tool`)大约能省 **~90ms**——刚好够做一次 VikingDB 检索。最终**总延时持平**,但 prompt 减少 50%+。

## 四、到底在优化什么问题

这是最关键的部分——动态化不是为了"看起来高级",是奔着具体痛点去的:

### 问题 1:SP 长度直接导致的模型不稳定
- 注意力 O(n²),输入越长焦点越散
- 多任务 query 漏任务、推理不结合 context、编造记忆
- **动态化 → SP 从 34K 压到 < 15K,注意力重新聚焦**

### 问题 2:示例之间互相干扰
- 69 个示例放一起,模型看到一个"打开车窗"示例可能去模仿一个完全不相关 case 的格式
- **动态化 → 当前 query 只看到相关的 3–5 个示例,干扰消失**

### 问题 3:Badcase 修复路径堵死
- 现状:每发现一个 badcase,往 SP 里加一条规则或一个示例 → SP 越来越长 → 修一个 bug 引入两个
- 动态化后:badcase 进检索库,**只对触发该场景的 query 生效**,不影响全局
- 这是李强反复强调"动态示例一定要做,不做将来 badcase 没法修"的根本原因

### 问题 4:KV Cache 命中率
- SP 固定部分越短,跨请求复用的概率越高
- 长对话不容易把固定 SP 挤出缓存
- **动态化 → 命中率提升 → 同等硬件下更快、更省**

### 问题 5:规则压制创造力
- 当前为了稳定性塞了 10 条注意事项 + 69 个示例 → AI 灵活性被压死
- **动态化 → 主逻辑回归到最小约束集,示例按需出现 → AI 在该灵活的地方灵活**

### 问题 6:知识覆盖度 vs Prompt 长度的死结
- 静态塞:覆盖度上不去(车型/工具/场景越多越塞不下)
- 动态拉:覆盖度无上限,长度由 top-k 控制
- **这是唯一可扩展的架构**

---

## 五、一句话总结

> **动态示例**教模型"在这种情况下应该输出这样的 JSON";**动态知识**告诉模型"这个领域 / 这个工具 / 这个任务有这些规则和边界"。两者都是把原本写死在 34K SP 里的内容,按"当前 query / 当前工具 / 当前任务"按需检索注入,目的是**同时拿下 SP 压缩、注意力聚焦、badcase 可修复、KV Cache 命中率、扩展性**这五个收益。

## 二、关键新决策点

这次方案里有 3 个之前没有定的工程决策被定下来了,值得重点记录:

1. **域分类小模型登场** — 不再是单纯向量检索,而是先经过 `query → 域分类(导航/车控/视觉/智驾/娱乐/行程规划)→ top-k 检索`。这是个**两阶段架构**:粗筛(域分类)+ 精排(向量召回)。好处是精度高、可解释;代价是要训/维护一个分类小模型。

2. **top-k = 2~3 而不是 5–10** — 这意味着注入预算很紧,每个域的示例必须**高度精炼**,否则 2~3 个示例可能覆盖不全场景。这反过来强约束了"示例压缩"任务的质量。

3. **并行化彻底分模块** — 知识注入、示例注入、context 模块全部并行执行。这要求 SP 必须先做模块化改造(槽位化),工程链路才能"分别填、最后拼"。

# 三、怎么做(6 个优化项展开)

### ① 输出格式精简(最容易)

字段重命名,省 token:

| 原始 | 优化后 | 省 |
|------|--------|-----|
| `user_query` | `query` | 1 |
| `speaker_name` | `speaker` | 1 |
| `speaker_position` | `zone` | 2 |
| `talk_or_not` | `speak` | 2 |
| `talk_content` | `content` | 1 |
| `tool_name` | `tool` | 1 |
| `action_id` | `id` | 1 |
| `action_content` → params | `param` | 1 |

**收益:首 token 延时减少 ~90ms**(这 90ms 就是后面留给 vikingdb 检索用的)。

### ② 静态 + 动态示例拆分

这是核心改造。文档里给了一个完整的"我去虹桥机场吃日料"的 6 轮对话例子,每一轮注入策略不同:

| 轮 | 输入 | 注入策略 |
|----|------|---------|
| 1 | 用户 query | 静态 10 个 + query 检索动态 5 个 |
| 2 | tool_feedback(poi) | 静态 10 + tool 检索动态 5 + 上轮 query 检索动态 5 |
| 3 | 用户后续 query | 静态 10 + 上轮 query 5 + 本轮 query 5 |
| 4 | tool_feedback(poi) | 静态 10 + tool 检索 5 + 上轮 query 5 |
| 5 | 用户确认 | 同上策略 |
| 6 | tool_feedback(route) | 同上策略 |

**关键约束**:静态保持精简(10 个核心示例),动态按需注入(5+5)。

### ③ 检索加载(vikingdb 工程)

需要解决两个问题:
- **query 改写** — 口语化 query 召回率低,需要先改写
- **多轮检索** — vikingdb 要支持基于 tool_feedback、基于历史 query 的多种检索 key

### ④ 示例压缩(内容侧)

现状:"裸示例",比如 tool_feedback 里塞了一整段 POI 详情(电话、营业时间、驾车信息……),其实模型只需要看到核心字段。

文档给的对照例:

## 关键的新增信息(以前没说清楚的)

**1. 输入是怎么打包的**
每轮 Planner 拿到的是完整 UP(user prompt):前几轮的 user/assistant 历史 + 这轮的新输入。tool_feedback 也走 user 角色——对 Planner 来说"工具返回"就是"外部新信息进来了"。

**2. 输出 actions 数组的两种状态**
- `actions: [...]` 非空 → 调工具,挂起等返回
- `actions: []` 空 → 不调任何工具,纯反问/推荐用户,等用户回复

**3. 每一轮的注入策略都标了**
每张卡片底部的小铭牌写着这轮 SP 注入了什么:
- P1 静态 10 + query 检索 5
- P2 静态 10 + tool 检索 5 + 上轮 query 检索 5
- ...

**4. 末尾抽象了"通用模式"**
6 轮看似复杂,其实只有 2 种处理模式:
- `user.in` 来了 → 想想信息够不够,够就拍板,不够调工具
- `tool.return` 来了 → 看数据能不能自己决定,不能就反问用户

**5. 点破 Agent 循环本质**
Planner 就是个永远在等下一个事件的状态机。6 不是固定的——简单 query 可能 2 轮,复杂多任务 10 轮 +。
