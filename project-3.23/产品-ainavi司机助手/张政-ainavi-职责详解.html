张政 — Ainavi 交付职责详解

基于 2026年5月11日 ainavi-产品交付SOP分享 会议讨论

角色定位：各交付项目的 Ainavi 技术方案 Owner（Across Projects）

一、核心定位

张政 ≠ 单一项目 owner，而是跨项目的 Ainavi 方案总负责人。
和思晴的分工：思晴 owner 共性产品问题（TTS、可见可说、版本差异文档）；张政 owner 分项目实施落地（tool 定义、skill 配置、能力评估、项目对齐）。
维度思晴（Jojo）张政
范围 共性产品问题、SOP 制定、需求池管理 分项目实施、技术方案配置、能力评估
产出 版本差异文档、必过集、产品需求评估 各项目 tool/skill 配置、能力差距分析报告
对接人 Ancher、研发、行业解决方案 各项目 XPM、思晴、复杂推理研发

二、六大职责模块

模块 1：Tool 定义与 Skill 配置（Per Project）

原文：Ancher — 「正好让张政进来把详细的 tool 的定义…我怎么去配置它对应的 skill？怎么来去区分？」
子任务具体内容说明
1.1 梳理每项目 tool 清单 基于客户 SDK 版本、项目诉求，确定该项目可用 tool 列表 高德 SDK 版本不同 → 可封装的 tool 能力不同 → tool 列表不同
1.2 定义 tool 能力边界 每个 tool 的入参、出参、搜索策略（云端/端侧）、支持场景 参考「Navi 工具定义 & UE 集合」文档
1.3 配置 skill 文件 为不同 OEM / 车型编写专属 skill，声明 tool 集合 + 品牌偏好 同一套 1.0 代码 + 不同 skill 文件 = 差异化交付
1.4 区分版本 skill Navi 1.0 全量 skill vs 奔驰定制版 skill vs 其他 OEM skill 如奔驰版 skill 内：酒店搜索 → 五星以上；餐厅搜索 → 人均 ¥200+
关键认知：同一套云端代码，通过不同 skill 文件（tool 集合声明 + 品牌偏好规则）适配不同 OEM、不同车型、不同 SDK 版本。这是智斌在会上一再确认并认为可对外宣传的「卖点」。

模块 2：版本兼容性适配（0.5 / 1.0 / 1.0+ / 2.0）

原文：Ancher — 「复杂导航也就是0.5版本，1.0版本，1.0加的版本在用户体验层的差异就是你的工具一个、两个、三个、7个或者17个，它在用户体验层到底意味着什么？」
原文：智斌 — 「奔驰项目第一期就先不交 AI Navi…我要怎么样去用一套方案去交两个不同的版本？」
子任务具体内容
2.1 定义版本 → tool 数量映射 0.5 = 基础复杂导航 tool（~3个）；1.0 = 完整 Navi tool 集；1.0+ = 1.0 + Agent 扩展 tool
2.2 定义版本 → 用户体验差异 每个版本能支持哪些场景？哪些场景做不好？哪些根本不支持？
2.3 版本间代码共用方案 证明「一套代码 + 配置」可同时支持旧 SDK（810）和无 Agent 版本以及新 SDK（910+Agent）
2.4 端侧适配层设计 不同 SDK 版本的接口差异由端侧映射适配，云端下发保持一致
2.5 能力缺失时的降级策略 老版本 SDK 缺少某能力 → Skill 层面告知用户「不支持」，非链路报错

模块 3：分项目能力差距分析（Gap Analysis）

原文：思晴 — 「我们先整体地盘一下，我们现在的交互项目都有哪些？然后针对于不同的项目客户的诉求是什么？然后再结合我们现有的能力，让张政这边分析一下我们哪些项目当中的体验我们有没有达到，然后对应的他的诉求是…」
项目客户诉求现有能力张政需分析
奔驰 可能第一期不交 AI Navi，仅复杂导航；后期升级 1.0；品牌偏好（五星酒店、高端餐厅） 1.0 tool 集 + 奔驰专属 skill ① 0.5→1.0 升级路径 ② 品牌 skill 是否覆盖所有场景 ③ MQTT 协议适配
比亚迪 已和博洋做了一轮能力评估表（FC/Navi/Planner 分工） FC + Navi + Planner ① 已有评估表的 tool 是否完整 ② 单地图方案下的能力交付范围
沃尔沃 待确认 待对齐 ① 需求梳理 ② 现有能力覆盖度
AI car（赛力斯/DNA） 赛力斯后续采购 SDK Agent → 新增 tool；DNA 无 Agent 1.0 + 1.0+ ① 赛力斯 tool 扩展方案 ② DNA 无 Agent 下的能力边界

模块 4：端云数据流适配（Per Project）

原文：Ancher — 「奔驰它本身链路里面，因为它是从端侧去拿一些信息上来的，链路可能也会跟我们现在这个链路不一样，比如说他们那个 MQTT 的协议是通过思必驰另外一种方式端云同步来去做的」
子任务具体内容
4.1 梳理每项目的端云协议 奔驰 = MQTT（思必驰）；其他项目可能有不同协议
4.2 确认 tool 结果回传路径 端侧 tool 执行结果如何回到云端 Mind？走什么协议？时延能否接受？
4.3 区分单向 vs 双向链路 复杂导航（0.5）= 云→端单向；Navi（1.0+）= 云↔端双向

模块 5：拉通 XPM 对齐

原文：Ancher — 「拉上对应的项目那边的 XPM，就是比如说奔驰上面就是你自己（智斌），比亚迪上面是博洋，然后沃尔沃上面是健成和宇淇」
项目XPM张政对齐内容
奔驰 智斌（谢智斌） 0.5→1.0 分期交付策略；品牌 skill 定制；MQTT 链路；老版本 SDK 兼容方案
比亚迪 博洋 已有能力评估表复核；单地图方案下的分工；语音主链路对接到位
沃尔沃 健成、宇淇 需求梳理；交付方案制定
AI car（赛力斯/DNA） 思晴（暂代） 赛力斯 Agent 扩展方案；DNA 基础版本维护

模块 6：上手熟悉（Onboarding）

原文：Ancher — 「你正好借这些项目来去熟悉一下」

张政作为新接手跨项目方案的负责人，需要通过上述工作完成以下知识域的熟悉：

  • Navi 三层架构：导航业务 / 沿途问答 / 行程规划三大模块的工具定义
  • 端云数据链路：FC 下发 → 高德 SDK 调用 → Navi 云端处理 → Planner 推理 → 用户交互
  • Planner（复杂推理）交互协议：Navi 工具 input/output 的数据格式，与复杂推理模块的对接方式
  • 各项目差异化变量：SDK 版本、语音供应商、端云协议、品牌偏好、交互形态（双地图/单地图）
  • PRD + Navi 交付方案（研发） + 复杂推理交付 SOP：三份核心参考文档

五、优先级建议

优先级事项原因
P0 梳理 0.5 / 1.0 / 1.0+ 版本 tool 差异表 所有项目的 skill 配置都依赖这张表；是思晴被 Ancher 要求输出的「版本体验差异文档」的技术基础
P0 奔驰项目 0.5→1.0 分期方案 + 品牌 skill 草稿 奔驰是第一个要交付的外部 OEM，且变量最多（MQTT、老 SDK、品牌偏好、可见可说白名单）
P1 比亚迪能力评估表复核 已有博洋做的基础，复核 + tool 化即可
P1 沃尔沃需求对齐 + 方案初稿 需要先跑通项目对接流程
P2 赛力斯 Agent tool 扩展方案 思晴已经在看，张政后续接手
P2 端侧 tool 点击事件上云方案 依赖思晴 + 研发的共性方案先行落地
