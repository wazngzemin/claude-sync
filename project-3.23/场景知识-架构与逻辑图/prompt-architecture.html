1
输入层：8 维上下文输入
8 个输入不是并列关系，而是互补验证关系
user_query
用户说的话 + 说话人名字 + 说话位置（主驾/副驾/后排）
conversation_history
最近 10 轮对话历史，推断意图演变和已确认信息
status（舱内视觉）
人员着装、行为动作、车内遗留物品、光线明暗
goal_list（目标队列）
当前持续追踪的目标。目标存在 → 知识保留
env_info（车辆状态）
空调、座椅、门窗、驾驶模式、导航、媒体等全车实时状态
env_changes（变化点）
相比上次检索的状态变化。趋势比绝对值更具决策价值
memory（用户长期记忆）
用户基础信息、习惯偏好、社会关系、过往共同经历
loaded_knowledge_ids
当前已加载知识 ID 数组，用于增量更新，只输出变化
互补验证，推理场景
2
推理层：5 条核心筛选原则
从 8 维输入中推理出需要加载/卸载的知识
1
场景绑定推理：结合 query + 对话历史 + 车辆状态 + 舱内视觉，推理当前具体场景（展厅看车 / 地库泊车 / 通勤堵车），再匹配对应知识 ID
2
关注环境变化：趋势比绝对值更具决策价值。温度急剧变化 → COM-01-01-01；车内人数增加 → EMO-01-01-01；车辆进入地库 → DRV-01-02-02
3
目标持续关联：关注 goal_list 中的进行中目标。只要目标存在，支撑该目标的知识应持续保留，除非用户明确表示终止
4
增量更新原则：对比已加载知识，仅输出变化部分。已失效的删除，新增的追加，减少下游上下文噪声
5
信息不足时不虚构：需求模糊且无法推断时，add_knowledge_ids 保持为空。宁可漏掉，不要猜错
匹配专家编码

知识注入的 4 个触发时机：
① 用户新 query 到达时    ② env_changes 检测到显著变化时（即使无用户 query）
③ goal_list 目标完成/变更时    ④ conversation_history 累积到新阈值时（如超过 3 轮未检索）
增量输出 JSON
4
输出层：严格 JSON 格式
必须输出纯 JSON，不含任何额外文本或 Markdown 标记
// 示例：车内温度急剧上升，用户表达热感 { "add_knowledge_ids": ["COM-01-01-01"], "remove_knowledge_ids": ["ENT-01-01-01"], "reasoning": { "add_reason": "变量变化显示车内温度急剧上升，且用户表达热感，需新增空调调节知识 COM-01-01-01。", "remove_reason": "用户已明确要求切换为舒适调节场景，原娱乐知识 ENT-01-01-01 在当前场景下冗余。" } }
注入/卸载知识内容
5
下游：思考者 Agent
接收筛选结果 + 知识内容，生成对用户的回复和执行指令
📥
知识注入
筛选 Agent 输出 add_knowledge_ids 后，系统取出对应 ID 的「场景要点 + 执行逻辑 + 示例」，追加到思考者 Agent 上下文中
💬
生成回复
思考者 Agent 根据注入的知识内容和当前对话状态，生成对用户的自然语言回复和执行指令
🔄
历史回流
思考者 Agent 的多轮对话输出进入 conversation_history，反过来影响筛选 Agent 下一轮的知识加载决策
对写多轮对话示例的 5 条核心启示
  • Pre-state 必须明确：示例开头交代车辆状态、已加载知识 ID、用户记忆，这是推断基石
  • 每轮必须有 State Update：用户说完后模型知道了什么新信息？必须显式写出增量
  • 必须覆盖边界情况：用户拒绝、改主意、车辆突变（急刹）、硬件不可用（后排无屏）
  • 展示静默执行：自动雨刮、随速关窗等无需语音确认，教会模型"何时说话、何时沉默"
  • 对齐增量逻辑：思考者 Agent 的对话进入 history，影响下一轮筛选，示例必须展示这种双向循环
