1.2 佳杰用「副驾打电话→调音量」走链路

佳杰把第二个例子换成了"副驾打电话时把音量调小"，因为这个case更清楚地展示了事件驱动的逻辑：

#步骤谁干的说明

丁彬指出："打电话这其实是一个周期性的任务，它一直在跑，所以就不需要触发器来触发VLM。触发器只要去看有没有这个信号就行了。"
佳杰确认："对我来说你这边是封装好的东西，我只在乎开始和结束这两个事件。你从哪拿数据是你封装的。"

1.3 充电桩case的链路推演——核心讨论

丁彬指出充电桩和打电话本质不同——"它不是周期性的，它得有人来触发一下才能调AIBOX做一个看充电桩的任务"。

#步骤谁干的和"打电话"case的区别
1充电桩场景提前配置在配置平台⑬上（系统预设条件任务，不是用户说出来的）产品/运营在火山后台配区别点①：打电话的事件是Advisor运行时订阅的；充电桩是提前配好的固定场景
2端侧㉔轮询信号库④，监听gear_current信号端侧触发器和打电话一样，都是轮询信号库
3检测到P档 → 触发器主动调VLM任务："看后面有没有充电桩"端侧触发器 → AIBOX区别点②：打电话不需要调VLM（AlwaysOn已经在看了）；充电桩需要主动发起一次VLM任务
4VLM看后面 → 结果写回信号库④AIBOX → 信号库VLM结果和AlwaysOn一样都写信号库
5触发器重新扫描信号库 → P档✓ + 充电桩在后面✓ → 条件满足触发器区别点③：充电桩需要"两个条件同时满足"（P档+看到充电桩），打电话只需要一个事件
6push给Advisor/Director → Director发指令开充电口盖触发器 → Director—
三人确认的关键结论——断网=做不了：
丁彬问："如果没有云端怎么办？"
佳杰："这个任务就做不了，就不开盖了。"
丁彬确认："端上没有去拆解任务的能力。"

这比v4的说法更精确：v4强调"端侧触发器的价值=断网能跑"，但6.24明确了——只有简单场景（倒计时、白名单车控）断网能跑。复杂场景（需要VLM+云端推理的）断网就是做不了。

1.5 两种信号消费模式——周期性 vs 触发式

丁彬当场帮你理清了——触发器和AIBOX之间只有两种通信方式：

模式含义触发器怎么做典型case和v4的关系
①周期性
（Periodic）		AlwaysOn已经在持续输出信号了，触发器只需要轮询信号库看有没有变化不需要调VLM任务，只需要扫描④信号库中已有的AlwaysOn输出副驾打电话（AlwaysOn已经在看舱内动作）、车内有小狗（AlwaysOn已覆盖舱内物品检测）v4的15-3扫描、15-10穿越检测覆盖了这种模式
②触发式
（On-demand）		AlwaysOn不覆盖这个观测目标，需要触发器主动发起一次VLM任务条件A满足（如P档）→ 触发器调AIBOX执行一次VLM任务 → VLM结果写回信号库 → 再扫描条件BP档看充电桩（不在AlwaysOn范围内，需要专门看后方）v4没有明确区分这两种。24-6 VLM调度写的是"云端下发的视觉观测任务"，但没有说"端侧触发器自己也能主动发起VLM任务"
丁彬原话："你那边给我注册的信号就两种，一种是周期的我就直接往上报。另外一种是你要触发一下调用我一次我返回一个结果。这些咱们都提前约定好就行了，分好类就行了。比如打电话就是周期类的，看后面充电桩就属于你要来调用我的。"

1.8 AlwaysOn舱外 vs 室内巡车——互斥问题（全新议题）

丁彬提了一个v4完全没覆盖的问题：

  • 舱外AlwaysOn（天气/路况/前方障碍物等）在行驶中默认开启，持续占用前摄
  • 室内巡车（找停车位的应用）也需要用前摄 → 两者互斥
  • 谁来做这个互斥排队？
方案谁做怎么做状态
优先级排序AIBOX的意图模块（任务调度器）所有任务统一排队，高优先级先跑。意图能感知到"有人调我了"→做排队丁彬确认意图能做
互斥逻辑需要工程层面实现进地下停车场 → 停掉舱外AlwaysOn → 启动巡车。出停车场 → 停巡车 → 恢复AlwaysOn。这是确定性策略（工程层面的，不是模型的）丁彬说"可以做"，但具体返回什么（暂不可用？重试？）需要和赛力斯对
保底机制赛力斯端 + 我们AlwaysOn被停了 → 给巡车应用返回"暂不可用" → 超过N次 → 需要用户重新触发唤醒佳杰提议，未定
和v4的关系：v4的§12写了"AlwaysOn调度归属——在AIBOX意图模块，不在触发器"。但6.24暴露了互斥的具体细节是v4没有的：谁触发停止、怎么恢复、给对方返回什么。而且这不是仅仅"排队"的问题——舱外AlwaysOn是周期性持续跑的（不会主动释放资源），巡车也是周期性持续跑的，两个周期任务互斥需要显式的切换策略，不是简单排队能解决的。

为什么：丁彬对毯子case的态度是"还不一定能做下来"。你v4的18步链路建立在VLM能提供"毯子位置"观测的前提上。前提不稳，链路就是空中楼阁。

怎么做：

  1. 拿佳杰的事件清单（行动③拿到的），和丁彬逐条过
  2. 每个事件问丁彬三件事：
    • AlwaysOn现在能不能输出这个信号？（能→周期性，不需要额外开发）
    • VLM能不能通过追加prompt做到？（能→触发式，需要开发下发通道）
    • 完全做不了？（做不了→这个事件先砍或走VQA降级方案）
  3. 特别要验证的重点case：毯子掉落检测、充电桩识别（如果归触发器的话）

做到什么程度：每个事件有三选一的结论（AlwaysOn已有/VLM可做/做不了），标在清单上。

行动⑤ 【定了调用方身份后】更新PRD，补3块新内容

为什么：6.24暴露了v4/现有PRD缺少的3块内容。但要等行动①的结论再写——scope不定，PRD写了也白写。
