端侧触发器 · 概念与架构讲透

触发事件 vs 触发条件
用你天天配的信号,一个一个讲穿

你之前卡住,不是因为笨,是因为没人拿档位、车速、车窗这些你真实系统里的信号,一条一条把"事件"和"条件"摆给你看。这一版就干这件事:先把两个概念钉死,再用十几个真实信号列全,再讲三个最容易搞混的地方,最后把三层架构拿真实场景从头走一遍。

基于 7.10 信号中枢评审 + 你的车态信号表 · 2026-07-13
01

先把地基钉死

两个词,一个测试就能分。看它盯的是"变化"还是"此刻"。

触发事件 = 动态条件
变化的那一下
测试:它变没变?
状态从一个值,跳到另一个值的那一瞬间。

它是个动作、一声发令枪。响一下就过去了。系统平时不管它,等它"响"。

档位从 P 切到 D · 主驾座椅从空 变成 有人 · 车窗从关 变成 开

触发条件 = 静态条件
此刻的状况
测试:它是不是?
当前这个状态,是不是某个值。

它是个状态、一张随时能查的快照。不管怎么变来的,只问现在是什么样。

车窗此刻 是不是 有一扇没关 · 车速此刻 是不是 0 · 温度此刻 是不是 高于 28

只记一句:事件问"变没变",盯的是那一下;条件问"是不是",盯的是这一刻。

"动态是不是就理解成状态从 A 到 B,静态就是当前的状况是 B?" 韩杰:"对,是的。"
02

用你的真实信号,全摆开

这是这份文档的核心。同一个信号,写成"变没变"就是事件,写成"是不是"就是条件。逐条看,你会发现规律。

顺带说一句取值: 枚举型信号的"值"是几个固定选项,比如档位是 P / R / N / D、驾驶模式是 经济 / 舒适 / 运动、氛围灯是几种颜色。数值型信号的"值"是一个范围,比如车速 0–220、车内温度、电量百分比。事件盯的是这些值之间的跳变(P→D、越过 28℃),条件盯的是此刻落在哪个值 / 哪个区间。
看出规律了吗: 每一行,左边都在描述"从…到…"的一次变化,右边都在描述"此刻是不是…"的一个状态。一个信号,你想抓它变化的那一下,就写成事件;你想查它此刻什么样,就写成条件。信号本身不分事件条件,是你用它的方式决定的。

顺带说清:事件那句"人话"是怎么拼出来的

触发事件不只是"响一下"。它响的同时,会按一个固定规则拼出一句描述,给事件日志和 Advisor 用。拼法是:功能名 + 切换类型 + 目标值,需要的话前面再加"谁、几点"。

档位+ 变更为+ D 档= "档位变更为 D 档"
信号运算符(真实)值拼出来的那句话
档位变更为D 档"档位变更为 D 档"
主驾座椅变更为有人"主驾落座"
隐私开关变更为开"隐私开关打开"
档位有变更—"档位有变化"(不限定变成哪档)
前面加上"谁、几点""14:03 主驾把档位变更为 D 档"
这句话不是拼给自己看的,是给下游用的。 它就是你要交给明骏(Director / 事件日志)的东西,也是给 Advisor 的触发理由。所以要不要在前面带"谁、几点",得按明骏事件日志的格式来定,这是你俩得当面对齐的一件事。拼接规则你不定清楚,事件表就落不了地。
03

三个最容易搞混的地方

这三个点,是你之前反复没绕明白的真正原因。搞定它们,你以后看任何一条规则都不会再懵。

1同一个信号,既能当事件又能当条件
这是最容易卡的一点。你问过"车速为啥是静态条件,不是动的吗"。答案:车速这个信号本身会动,但你怎么用它决定了它是事件还是条件。
当事件用:车速从 0 变成非 0  →  这是"起步"这件事发生了
当条件用:车速此刻是不是 0  →  这是"停着没"这个状态判断
同一个车速信号,前者抓的是"变的那一下",后者查的是"此刻的值"。别看信号动不动,看你要它的"变化"还是"此刻"。
2数值信号的"阈值穿越":连续的值怎么变成一次事件
温度、车速、电量这些是连续变化的数值,它们一直在动,不像档位那样"啪"地跳一下。那它们怎么当事件?靠穿过一条线。

事件:温度从 27 爬到 29,越过 28 的那一下 → "车里热起来了"这件事发生
条件:温度此刻是不是 >28 → 无论怎么到的,现在高于 28 就算满足
区别在:事件只在穿越的那一瞬间响一次(27→29 触发一次);条件是只要现在 >28 就一直为真(29、30、31 都算)。电量跌破 20%、车速超过 120,都是这个道理。
补一句,免得跟运算符对不上: "越过阈值"是概念说法,配置平台里没有这个运算符。数值信号(温度、车速、电量)的阈值,你是用条件运算符「大于 28」「小于 20」配的;那个"越过的瞬间",是触发引擎在这个条件从不满足变成满足时自动帮你抓的边沿。所以数值型的"事件感",是靠条件从假变真实现的。真正用「变更为」的,是档位、开关这种有明确取值的枚举信号。
3运算符:平台就这 10 个,分两拨;"切入/切出"根本不在里面
配置平台的运算符,就是你那张图里的这 10 个。它们其实分成两拨,正好对应事件和条件:
属于运算符(平台真实的)干嘛用的

那"切入/切出"在哪?不在。 那是韩杰 7.10 开会时的口头说法,不是平台里的运算符。真能用的变化型运算符只有两个:变更为 和 有变更。"切入某个值"最接近「变更为」;"切出某个值"(离开某个值)平台压根没有对应运算符,得用「有变更」再配一个判断去凑。
刘杨 · 7.10 · 49:28
"我们现在不支持切入切出啊。你是把原来'变更为/变更'这种运算符,又改了个名字叫切入切出,还是我们要再实现一下切入切出?"
刘杨怼的就是这个:韩杰想按"切入/切出"来配,但平台只有「变更为/有变更」。你要么接受用 变更为 来表达,要么就得让研发真去实现一套"切入/切出"运算符。这个口径得你来定,定不清楚,你的事件表就落不了地。
4一条规则里,事件配几个条件、几个动作?
一条规则的骨架永远是这个样子:1 个触发事件 + N 个条件 + M 个动作。关键是 N 和 M 都可以是 0,也可以是好几个。
情况什么意思
N = 0 没有条件事件一响就直接执行,不过滤。韩杰原话:"最少的情况就是有一个触发事件,但可能没有条件。"比如"主驾落座"直接就打招呼,不用再判断什么。
N > 1 多个条件串联或并联过滤(下一节讲)。比如"档位切D"后面跟"车窗未关"和"车速为0"两道。
M > 1 多个动作一个事件同时喂几个下游:调 Advisor + 写事件日志 + TTS 播报,一次做完。
那为什么大多数事件还是要挂条件?因为事件的量太大了。 韩杰原话:"触发事件量级特别大,所以想着通过条件去过滤掉一些无效的噪音,给到云端就比较精准。"车上每秒都在产生一堆状态变化,全报上去云端会被噪音淹没。条件就是给事件降噪的闸门,把没意义的挡在漏斗上层,只让真正该处理的往下走。
04

为什么非把它们拆开不可

拿"挂 D 档提醒关窗"这个真实场景,对比旧的并列和新的漏斗,你一眼看出差别。

旧:三个条件并列

档位=D  且  车窗有未关  且  车速=0  →  触发

三个条件地位平等,一起判断。全满足就调 Advisor。

要写事件日志时,系统傻了:这三个里,哪个是"刚发生的事"?它不知道,只能写一堆状态,Director 用不了。

新:一个事件领头

档位变更为 D  →  车窗有未关?车速=0?  →  动作

档位变更为 D 是"发生的事",领头;车窗车速是过滤条件,在后。

写事件日志时清清楚楚:"档位变更为 D 档"就是那句话,Director 直接用。

拆开解锁的,是旧结构做不到的四件事。都拿真实信号说:

拆开之后能做旧的并列为什么做不了
写一句人话事件日志"档位变更为D档"要求系统知道主语是档位。并列里三个条件平等,认不出主语。
带先后/时延的流程比如"档位切D后延时3秒,再看车窗"。并列只能"同时满足",没有先后。
拖拽式 workflow 画布画布要事件、条件、动作三种节点才拖得起来。一堆平级条件画不成流程图。
一个事件喂多个下游"档位切D"同时给 Advisor、事件日志、TTS。旧结构只有调 Advisor 一条出口。

条件之间怎么接:串行、并行,还能插延时

一个事件后面挂好几个条件时,它们之间有两种接法。这不是细枝末节,它决定了漏斗省不省算力、能不能表达先后。

串行:一道一道过
条件①→ 过了才查条件②
前一道不过,后一道根本不查。好处有两个:一是省算力,后面的信号平时不用一直盯着;二是天然有先后,适合"先看 A,A 成立再看 B"。韩杰改成串行,图的就是这个。
并行:几道一起判
条件① 条件② 全为真 → 才继续
没有先后,就是纯粹的"A 且 B 且 C 同时成立"。这其实就是旧配置平台的老逻辑,几个条件平摊着一起判。适合那些本来就没有先后关系的判断。
还能在中间插一个延时。 比如"档位切 D → 等 3 秒 → 再看车窗"。这种带时间、带先后的流程,正是旧的并列条件做不了、非拆成漏斗不可的原因(回到 §4 第二条)。串行、并行、延时,这三样加起来,才让触发器从"一次性判断"变成能编排的流程。
三个场景一对比你会发现: 每条规则都是"一个事件领头 + 若干条件过滤 + 若干动作执行"。事件是唯一的入口(发令枪),条件是漏斗的每一层(不过就停),动作在最后(可以一次做好几个)。这就是三层漏斗的全部。
06

架构全景:信号在三张表里怎么流

韩杰把端状态拆成了三张表。看一个真实信号(档位)怎么在三张表里各出现一次,以及数据怎么从车上流到动作。

① 状态查询表
给 Context 查"此刻什么样"
档位 = D
车速 = 0
车窗 = 左前未关
② 触发事件表
监控"变化",生成事件
档位变更为 D 档

给漏斗做过滤判断

同一个"档位"信号,在三张表里各有一个身份:在①是"当前档位值",在②是"档位变更为D档这个事件",在③是"当前是不是D档这个判断"。三张表都来源于同一批端上报的状态。
