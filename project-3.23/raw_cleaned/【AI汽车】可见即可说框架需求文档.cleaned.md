# 【AI汽车】可见即可说框架需求文档

## 需求背景

传统语音交互框架需用户记忆固定指令，且指令需提前配置，适配成本高；“可见即可说” 框架旨在解决该问题，通过自动解析界面元素、动态生成语音指令，实现 “屏幕显示什么，用户说什么就能操作”，降低用户使用门槛，同时为开发者提供轻量化、易集成的接入方案

## 可见即可说完整流程

## 核心模块

### 1. 界面元素解析模块

- 通过无障碍服务界面（等技术方式）变化实时扫描当前屏幕，提取所有可交互元素（如：按钮、文本框、列表项、菜单等）的关键信息，仅将最顶层页面进行扫描。
- **责任方：车企或各业务模块**

### 2. 语音指令生成模块

- 将界面元素信息转化为自然语言语音指令，让用户能通过“说文字”触发操作，同时避免指令冲突。
- 关键逻辑：
  - 指令提取：直接将元素的“显示文本”作为核心指令（如按钮文本“打开相机”→ 语音指令“打开相机”）。
  - 冲突处理：若多个元素文本相同（如列表中多个“删除”按钮），需补充位置描述（如“删除第3个”）或上下文信息（如“删除当前联系人”）。
  - 指令生成时需过滤掉符号、标点等信息。
  - 动态更新：界面变化时（如切换页面、弹出弹窗），实时更新指令库，确保“所见即所说”的实时性。
- 输出：动态维护的“语音指令-界面元素”映射库，包括热词指令和对应的泛化词指令 以及与它对应的ID。
- **责任方：车企或者各业务模块**

### 3. 语音识别模块

- 将用户的语音输入转化为文本，为后续指令匹配提供基础。
- ASR选择：
  - 在线ASR：将可见即可说注册的「语音指令集」作为「动态热词」进行在线ASR增强。
  - 离线ASR：将可见即可说注册的「语音指令集」作为「动态热词」进行离线ASR增强。
  - ASR选择策略与免唤醒多音区交互策略相同。
    - 免唤醒时走纯离线
    - 唤醒后离在线并行处理，离线有结果执行离线结果，离线无结果等待在线结果。
- **责任方：字节语音模块**

### 4. 指令匹配与意图理解模块

将ASR识别出的文本与注册的“语音指令库”进行匹配，确定用户意图对应的界面元素和操作。

- 关键逻辑：
  - 精确匹配：直接比对识别文本与指令库中的核心指令（如“确认”→ 匹配文本为“确认”的按钮）。
  - 模糊匹配：在全局指令中支持在指令前增加（点击/点一下/帮我选/选择+指令集）作为前缀信息，指令后可增加，吧，啦，等语气词。
  - 歧义消解：若匹配到多个元素（如两个“返回”按钮），将多个匹配的结果，返回给“注册方”，由注册方自行选择使用。
  - 局部匹配：6个以上，需连续匹配4个字以上。
  - 当一页面注册词：不超过500个词（随系统能力更新）
- 输出：匹配的指令集以及指令集对应的ID信息。
- **责任方：字节语音模块**

### 5. 操作执行反馈模块

- 根据匹配结果，通过模拟用户操作（点击、输入、滚动等），与应用/系统交互。
- 输出：操作执行结果（成功/失败），反馈给用户。
- 反馈方式：
  - 语音反馈：
    - 成功播报：好的/搞定了。
    - 失败：抱歉，没执行成功。
    - 需将执行结果“成功”“失败”以及需要播报的内容，传给豆包。由豆包统一播报。
  - 视觉反馈：
    - 模拟点击时需要有UI点击动效。
    - 失败时：按照系统提示方式呈现，如toast文字提示等，失败的文言不进行播报。
- **责任方：车企/各业务模块**

## 主链路关系

### 系统优先级

可见即可说属于在语音系统面向 “视觉界面交互场景” 的定制化交互模式，为了让用户看到什么就能说什么，少推理快响应，因此它的优先级应该比较高。

下面列出语音系统各个模块的优先级

| 优先级顺序 | 内容 | 描述 |
|---|---|---|
| P0 | 用户自定义任务 | 用户自定义创建的任务场景 |
| P1 | 可见即可说 | 针对界面上的文字，进行模拟点击，仅支持简单表达，无推理能力 |
| P2 | FAQ | 高频的问答且回答不能出错的指令 |
| P3 | 端侧LLM | 端侧能理解的功能 |
| P4 | 云端LLM | 云端大模型能理解的功能 |

### 历史对话

可见即可说的内容，需进行对话历史中。需在历史对话中标记出来自模拟点击事件，防止下一轮模型推理错误。
