0 核心思维模型 — 为什么要看 Trace

在看任何 trace 之前，先建立正确的思维框架

Agent 是一条数据变换链

用户的一句话进来，经过一个个模块的加工，最终变成一段回复 + 一组动作出去：

用户说话 ──→ [拒识判断] ──→ [变体选择] ──→ [Planner 推理] ──→ [Tool 执行] ──→ [二轮推理] ──→ 回复 + 动作
💡 核心原则 badcase 意味着这条链里某一步的变换出了错。你的工作就是找到是哪一步。方法只有一个：在每个节点查看 input 和 output，找到第一个 output 开始出错的节点 — 那就是根因所在。

这就像查电路故障：不是猜哪里断了，而是拿万用表从电源端一个节点一个节点往下量，找到第一个电压跌落的位置。

⚠️ 最常见的错误 拿到一个 badcase 就直觉猜"肯定是 prompt 的问题"或"模型太笨了"。不要猜，看数据。像五哈那条 case，看起来像 prompt 问题，实际是 tool 返回穿透。数据不会骗你，直觉会。

你的 Fornax trace 树就是这条链的可视化。树上的每个节点就是一个变换步骤。下面按你应该看的顺序逐个展开。

① 第一站：Planner 第一轮 Output

每条必看 树上位置：text_planner → ChatModel → Output

为什么第一个看这里

Planner 是整个 Agent 的中枢大脑。它的 output 决定了三件事：

决定什么对应 output 字段
要不要和用户说话talk_or_not（yes / no）
说什么话、用什么语气talk_content（回复文本）
要不要执行动作、调哪个工具action_list（工具 + 参数列表）
💡 为什么先看 output 不先看 input 因为你的目标是快速判断问题在 Planner 还是在 Planner 之后。如果 output 已经错了，就不用往下游查了，直接转向查 input（第三站）来找原因。如果 output 是对的但最终回复是错的，说明问题在下游（tool 执行或二轮推理）。

先看结果，再追原因。就像医生先看症状再做检查，不是反过来。

逐字段检查指南

🔍 action_list — 最关键的字段

这个字段记录了 Planner 决定调用哪些工具、传什么参数。它是 badcase 的重灾区。你要看三件事：

检查点 A：tool 选对了没有 你们有 24 个 tool。Planner 需要从里面挑对的。常见错法：
• 该用 vehicle_basic_control（车控）却用了 search_and_control_short_video（搜视频）
• 该用 web_search（搜信息）却用了 route_planning_qa（导航问答）
• 该同时调两个 tool 却只调了一个

为什么 tool 会选错？因为 Planner 是靠 system prompt 里的 tool description 来理解每个 tool 能干什么的。如果 tool description 写得有歧义，或两个 tool 的功能描述有重叠，模型就可能选错。所以当你看到 tool 选错时，不要只记录"选错了"，要去看 tool description 是不是写得有问题。
检查点 B：params 传对了没有 即使选对了 tool，传的参数也可能有问题。常见错法：
• 多意图揉成一个 query — 五哈那条 case 就是这个问题
• query 太模糊导致搜索跑偏
• query 太精确导致搜不到结果（写了全名"哈哈哈哈哈"搜不到，写"五哈"反而能搜到）

为什么 params 会传错？一是 prompt 里没有教 Planner 怎么拆分多意图请求；二是 tool 的 param 描述不够清晰，模型不知道该传什么粒度的内容。
检查点 C：action 的顺序和依赖 如果 Planner 输出了多个 action，执行顺序对不对？有没有该串行的写成了并行？比如"先激活屏幕，再搜索视频"应该是两个 action 按顺序执行，不能合成一个。

🔍 talk_content — 回复文本

逐字读，问自己三个问题：

问题异常信号
内容对不对？事实错误、竞品品牌、不该出现的信息
语气对不对？该关切时在开玩笑，该简短时太啰嗦
完整不完整？说了一半截断，遗漏用户问的某个点

🔍 talk_or_not — 是否说话

有一类 badcase 就卡在这里：用户问了问题，Planner 判断 "no" 不说话。

常见原因：prompt 注意事项第一条写了"若用户说的话听不懂，或感觉不是对你说的，不需要说话也不用调用工具"— 模型误判了用户的话不是对它说的。

🔍 emoji_id — 表情选择

只在情感类 badcase 关注。要和 emotional_state_change 对照 — 如果情绪从"欣喜"变成"关切"但 emoji 还在用 cheer（欢呼），说明情绪状态机和表情选择脱节了。

📋 实战案例：五哈 badcase 在这一站发现了什么

查看 Planner 第一轮 output，发现 action_list 里的 query 是 "激活后排吸顶屏并播放综艺《哈哈哈哈哈》"。

问题：两个意图（车控：激活屏幕 + 搜索：搜五哈）被塞进了一个工具调用。应该拆成 vehicle_basic_control 激活屏幕 + search_and_control_short_video 搜"综艺 五哈"。

判断：Planner output 已经有问题了 → 继续查 tool 返回（第二站）确认影响范围。

② 第二站：Tool 的原始返回

每条必看 树上位置：text_planner → Tools → [具体tool名]

为什么看这里

Planner 不是全知的 — 它的第二轮回复完全取决于 tool 返回了什么。

⚠️ 很多新人会忽略的关键认知 你看到最终回复里有一句错误的话，第一反应可能是"prompt 写得不好"或"模型理解力不行"。但实际上，大量 badcase 的真正原因是 tool 返回了脏数据，模型只是忠实地引用了它。

模型对 tool 返回有天然的信任倾向 — 它会认为"工具返回的就是事实"。如果搜索工具返回了一条理想汽车的视频，模型会认为这是合理结果并引用它，不会想到"这是竞品品牌我不能提"。
💡 这是 badcase 归因的分水岭 • 如果脏数据在 tool 返回里就有 → 根因在工具层，修 prompt 没用
• 如果 tool 返回是干净的但最终回复是错的 → 根因在 Planner 推理层

具体怎么看

展开 tool 节点，找到最底层的原始返回数据。比如 search_and_control_short_video 下面会有 video_search → mixer_search，你要看的是 mixer_search 返回的完整结果列表，不是上层封装过的摘要。

检查项看什么异常信号
结果相关性 返回内容和用户意图匹配吗？ 搜"五哈"出来全是吸顶屏教程 = query 跑偏
敏感内容 逐条看 desc、title、author 竞品品牌、敏感词、不合规内容
结果条数 返回了多少条？ 只有 1-2 条 → Planner 被迫引用差结果
工具报错 tool 有没有执行失败？ 超时、服务异常 → Planner 可能复述错误信息
📋 实战案例：五哈 badcase 在这一站发现了什么

展开 tool 返回，看到 9 条抖音搜索结果。没有一条是《哈哈哈哈哈》的正片，全是各品牌吸顶屏操作教程。

其中 rank 3（doc_id: 7448638465375309113）的 desc 里有：

"理想的超大娱乐屏，丝滑又好用...追五哈真的太嗨了！
#我和理想L7的日常 #理想汽车 #娱乐屏"

这就是"理想车机"的污染源。一条理想 L7 车主看五哈的 vlog，desc 里带了竞品品牌。

判断：脏数据在 tool 返回里就有 → 根因在工具层（搜索结果无品牌过滤），不是 prompt 问题。

③ 第三站：Planner 的 Input Arguments

①②定位不到时必看 树上位置：text_planner → ChatModel → Input → arguments

为什么看这里

🧠 Context Engineering 的核心理念 模型的表现不只取决于它的能力，更取决于你在它的 context window 里放了什么。同一个模型，给它完整的上下文它能做对，给它残缺的上下文它就会犯错。

Arguments 是每一轮动态注入到 Planner context 里的实时信息。如果这些信息是错的、缺的、或格式不对的，Planner 就是在基于错误前提做推理 — 结论自然是错的。

3.1 knowledge（动态知识）

必查 重要性 ★★★

这个字段是什么

上游有一个知识路由模块，它根据用户意图从你们的知识库（Excel 原表 场景知识（saic）_数据总览.xlsx 里的 13 条场景知识）中选一条注入。注入后，知识成为 Planner 推理时的"专家指导"。

为什么重要

知识注入是你们系统的差异化能力 — 没有知识的 Planner 只能靠通用 prompt 里的 21 个示例来推理，有知识的 Planner 能获得场景化的精准指导。如果知识加载链路有问题，这个差异化能力就形同虚设。

检查四件事

1
有没有加载？ — knowledge 字段是空字符串还是有内容？如果测试标注了"有知识"但 trace 里是空的，说明知识路由模块出 bug — 问题不在 Planner。
2
加载的对不对？ — 把 trace 里的 knowledge 内容和 Excel 原表对比。比如用户说"用吸顶屏看五哈"，应该加载 DRV-01-01-02 后排车内观影。如果加载了 DRV-01-01-07 5D沉浸观影，那是路由选错了。
3
内容有没有被截断？ — 知识在传输过程中有没有被截断或乱码？把 trace 全文和 Excel 原文逐段对比。
4
格式兼容不兼容？ — 知识示例用的 JSON schema 和主 prompt 要求的 schema 是否一致。这是五哈 case 发现的问题：知识用 speak/content/actions，主 prompt 用 talk_or_not/talk_content/action_list，两套 schema 打架，降低知识的实际约束力。
📋 实战案例：五哈 badcase 在这一站发现了什么

✅ 加载了：knowledge 字段有内容，场景名称 "内容娱乐-乘客娱乐-后排车内观影"

✅ 加载对了：和 Excel 原表 DRV-01-01-02 完全匹配

✅ 没有截断：全文一致

❌ 格式不兼容：知识示例用 {"speak","content","actions"}，主 prompt 用 {"talk_or_not","talk_content","action_list"}

✅ 知识里无竞品词：全表 13 条扫描，零竞品品牌

3.2 env_info（车辆环境状态）

必查 重要性 ★★★

这个字段是什么

车辆当前的物理状态快照：车速、档位、屏幕状态、导航状态、音量等。

为什么重要

Planner 需要知道车辆当前状态才能做正确决策。如果 env_info 说吸顶屏"息屏待激活"，Planner 就知道需要先激活。如果实际已经开了但 env_info 说没开，Planner 就会做多余操作。

检查三件事

检查项异常信号会导致什么
车速、档位实际驻车但显示行驶中驻车功能被错误禁用
屏幕状态实际已开但显示待激活Planner 多发一次激活指令
导航/媒体状态和实际不符错误的上下文推理

3.3 memory（用户记忆）

条件看 重要性 ★★

这个字段是什么

分三部分：短期记忆（近期对话总结）、长期语义记忆（用户偏好）、长期情景记忆（具体事件）。

为什么重要

如果记忆里有错误信息，模型会基于错误记忆做推荐。比如长期记忆标签写了"喜欢科幻电影"，但用户实际喜欢喜剧 — 推荐就全偏了。

检查三件事

1
短期记忆有没有把上一轮的错误回复也记进去了 — 这会形成"错误滚雪球"，一个错误污染后续所有轮次。
2
长期记忆里的偏好标签对不对 — 标签错误 → 推荐方向全偏。
3
记忆是不是空的 — 有些知识示例写了"查询用户记忆中的观影历史偏好"，如果记忆空了，知识引导就无法完整执行。

3.4 goal_list（目标队列）

条件看 重要性 ★★

这个字段是什么

当前正在追踪的用户目标列表，比如 "1. 帮小明点肯德基外卖（进行中）"。

为什么重要

如果有未完成的旧目标，新请求进来时 Planner 需要决定：是切换目标还是继续追踪。如果旧目标和新请求冲突，可能导致行为混乱。

什么时候看

大部分单轮 case 的 goal_list 是空的，可以跳过。只有在多轮连续对话中出现目标跟丢或目标冲突时才需要检查。

3.5 speaker_name + speaker_position

辅助看 重要性 ★

为什么重要

多人车舱场景下，Planner 需要知道谁在说话、在哪个位置。如果主驾说了话但 speaker_position 标成了"二排左"，Planner 可能错误启动后排设备。

什么时候看

只有在回复对象错误（回答了错的人）或设备操作方向错误（操作了错的座位/屏幕）时才需要检查。

④ 第四站：对话历史 History

①②定位不到时必看 树上位置：text_planner → ChatModel → Input → history

为什么看这里

🧠 Transformer 架构的本质特性 LLM 的注意力机制不区分"当前轮"和"历史轮" — 它对所有 context 一视同仁地 attend。

这意味着：即使当前轮次完全没提到"理想汽车"，如果历史对话里有一条 tool 返回曾经包含过这个词，模型在生成回复时仍然可能被历史中的这个词影响。这不是 bug，是 Transformer 架构的工作方式。

具体检查四件事

1
历史里有没有竞品品牌词 — 前面的对话里用户或系统有没有提到过竞品？可能是之前某次 tool 返回带回来的，一直留在 context 里影响后续轮次。
2
历史的轮次数和总长度 — 你们的 system prompt 已经 195K 字符。如果再加上长对话历史，总 context 可能接近模型窗口上限。越靠近上限，模型对中间位置信息的注意力越弱（"lost in the middle" 现象），你的知识注入和 tool description 可能正好处于注意力洼地。
3
历史中有没有格式错乱 — 正常应该是 user/assistant 交替的 JSON 格式。如果拼接出问题（两轮 user 消息连在一起、assistant 的 JSON 被截断），模型解析上下文的能力会急剧下降。
4
历史里之前的 tool_feedback — 之前轮次的 tool 返回也在历史里。如果某个搜索结果包含了脏数据，这些脏数据就一直留在 context 里，影响后续所有轮次。
📋 实战案例：五哈 badcase 在这一站发现了什么

历史显示前面几轮都是罗勇在给宝宝找娱乐内容（游戏、动画片），然后才说"我想用吸顶屏看五哈"。

✅ 历史里没有出现"理想"这个词 → 排除历史污染

ℹ️ 儿童/家庭娱乐的上下文影响了 Planner 的场景理解，但这不是 bug 的直接原因

⑤ 第五站：System Prompt 变体选择

条件看：整体风格跑偏时 树上位置：get-system-prompt-vari → ArkChatModel

为什么有这个模块

你们的架构在 Planner 主推理之前，先用一次额外的 LLM 调用（1.42 秒、36 个 token）来动态选择 system prompt 的变体。这意味着你们有多套 system prompt，针对不同场景用不同版本。

为什么重要

⚠️ Single Point of Failure（单点故障源） 这 36 个 token 决定了 Planner 用哪套"大脑操作系统"。如果选错了变体，后面整条链路用的就是错误的 system prompt，所有行为逻辑从根上就偏了。但因为它只有 36 个 token，非常容易被忽略。

什么时候必须看

症状可能原因
Planner 回复整体风格跑偏（闲聊 vs 任务执行混淆）选了错误的 prompt 变体
Planner 调用了不该有的 tool不同变体的 tool 列表可能不同
同一个 query 有时候对有时候错变体选择不稳定，同一输入选了不同变体

怎么看

展开这个节点，看 output 那 36 个 token 到底是什么 — 是变体编号？一段 JSON？一个路由标签？把它记下来，然后和后端确认这个编号对应的 system prompt 是哪个版本。

什么时候不用看

如果 badcase 只是某个具体细节错了（比如说错一个词、选错一个 tool），而整体风格和能力范围是对的，那大概率不是变体选择的问题，跳过这一站。

⑥ 第六站：拒识模块

条件看：沉默或误拒时 树上位置：intent-and-refusal → reject-analyse → ArkChatModel

这个模块做什么

拒识模块和 Planner 是并行跑的（从 trace 树上可以看出 intent-and-refusal 和 planner_analyze_task_text 在 aidv-main-flow 下面是同级节点）。它的职责是判断用户的请求是否应该被拒绝。

为什么重要

💡 拒识模块有"否决权" 即使 Planner 规划正确了，如果拒识模块误判"这个请求应该拒绝"，用户也会得到拒绝回复。反过来，即使用户请求应该被拒绝，如果拒识模块漏判了，Planner 就会照常执行。

所以拒识模块可以"覆盖"Planner 的决策，这使得它在特定 badcase 中是关键排查点。

什么时候必须看

badcase 症状排查方向
用户合理请求被拒绝了（"我不太明白"、"暂时无法处理"）拒识模块误杀了 → 看 reject-analyse 的判断逻辑
危险请求没被拦住（行车中调大屏幕亮度等）拒识模块漏了 → 看 reject-analyse 是否正确识别了风险
回复被截断或突然中止可能是拒识模块中途拦截

怎么看

展开 reject-analyse 的 input 和 output：

1
Input（5780 tokens）— 拒识模块的输入是什么？是只有用户原话？还是把完整对话历史都塞进去了？如果是后者，大部分 token 是浪费的，拒识只需要当前轮 query + 少量上下文。
2
Output— 结论是什么？"通过"还是"拒绝"？拒绝理由是什么？理由和用户实际意图匹配吗？

什么时候不用看

绝大部分 badcase（回复内容错误、tool 选错、语气不对等）跟拒识模块无关。只有当 Planner 完全没有正常输出（沉默、拒绝模板话术）时才需要排查。五哈那条 case 就不需要看，因为 Planner 确实产出了回复和 action，只是内容有问题。

⑦ 第七站：direct_return（最终交给用户的内容）

必看 — 这才是用户真正听到的话 树上位置：text_planner → direct_return（在树的最底部）
⚠️ 重要更正：ChatModel Output ≠ 用户听到的话 很多人会以为 ChatModel 的 Output 就是最终回复 — 这是错的。

ChatModel Output 是大脑那一刻写下的草稿。从大脑写完到用户听到，中间还有 TTS 合成、文本拼装等后处理模块（比如 sami_tts 链路），这些模块可能会修改、补充、拼接最终文案。

direct_return 才是经过所有后处理之后、真正送给用户的最终版本。

怎么区分这两个

ChatModel Outputdirect_return
是什么 大脑思考的那一刻写下的草稿 经过所有后处理之后，最终交给用户的成品
对话轮数 N 轮（不包含本轮新生成的内容） N+1 轮（包含本轮新生成 + 后处理结果）
什么场景下不一样 中间有 TTS、文本拼装、内容补充等模块时，两者可能不一致
用户实际听到 ❌ 不一定 ✅ 是的

为什么必须看 direct_return

用户投诉的所有 bug，都是基于他们实际听到的话。如果你只看 ChatModel Output，会漏掉一类很隐蔽的 bug：大脑写的没问题，但后处理改坏了。

💡 用电视台节目做类比 ChatModel Output = 主持人写的稿子
direct_return = 观众电视上看到的最终成片

中间可能经过：剪辑、配音、字幕、片头片尾拼接。如果观众反映节目内容有问题，你不能只看主持人的原稿，必须看最终成片 — 因为问题可能出在剪辑环节。

看的时候要做的对比

1
把 direct_return 的最后一条（第 N+1 轮）和 ChatModel Output 做比对。如果两者一致 → bug 在大脑思考阶段（看第二站 tool 返回 + 第一站 Planner Output）；如果两者不一致 → bug 在后处理阶段（找研发查中间的处理模块）。
2
把 direct_return 的内容和第二站 tool 返回做比对。后处理模块有没有把 tool 返回的某些片段拼进了最终文案？这是隐蔽的"内容穿透"路径。
3
问研发：从 ChatModel 到 direct_return 中间有哪些模块在动这段文本？你看到的 trace 里有 sami_tts:connect、sami_tts、sami_tts:first_token — 这些只是合成语音的，还是会改写文本？
📋 实战案例：五哈 badcase 的关键发现

"理想车机"这四个字只在 direct_return 的第 29 轮里出现，ChatModel Output 里没有。

这说明 ChatModel 那一刻并没有说"理想车机" — 是大脑写完之后，下游的某个后处理模块把搜索结果里"理想的超大娱乐屏"相关内容拼进了最终回复。

归因结论要修正：

  • 之前我们以为：bug 在 ChatModel 二轮推理引用了脏数据 → 这个判断不准
  • 实际情况：bug 在 ChatModel 之后的后处理模块，可能直接从搜索结果拼接了内容
  • 修复方向也要变：除了第二站说的"tool 加品牌过滤"，还要查 ChatModel 之后的后处理逻辑

这条 case 教会我们：永远以 direct_return 为准。

⑧ 第八站：Timing 性能数据

辅助参考 树上位置：每个节点的耗时标注

为什么有时候要看

性能数据本身不是 badcase 归因的直接线索，但有两种情况需要关注：

1
某个节点耗时异常长 — 比如 tool 调用花了 10 秒+，可能是超时后返回了兜底结果（质量通常很差），模型基于兜底结果生成的回复自然也差。
2
并行模块的完成时序 — 如果 collect-tools-info 应该在 Planner 之前完成（提供可用 tool 列表），但因为延迟 Planner 在没有完整 tool 列表的情况下就开始推理了，可能导致 tool 选择不全。

✓ 必看清单

每条 badcase 至少看完前 5 项才能下结论。可以打印贴工位。
━━ 必看项（每条 badcase 都要看）━━
1. Planner Output — talk_content + action_list 的完整 JSON
2. Tool Output — 工具返回的原始数据（全部结果，不只第一条）
3. Tool Input — Planner 传给工具的 param / query 是什么
4. Knowledge — 有没有加载？加载了哪条？和 Excel 原表对得上吗？格式兼容吗？
5. History — 对话历史完整吗？有没有竞品污染？有没有格式错乱？
━━ 按需检查项 ━━
6. env_info — 车辆状态信息和实际一致吗？
7. memory — 记忆有没有误导？有没有错误滚雪球？
8. System Prompt 变体 — 36 token 选的变体对不对？
9. reject-analyse — 有没有误杀或漏杀？
10. 第二轮 Output — tool_feedback 之后的最终回复 vs tool 返回的比对
⏱ 时间基准 按这个清单走，一条 badcase 从打开 trace 到得出归因结论，应该控制在 5 分钟以内。
如果超过 5 分钟还没头绪，大概率是缺数据 — trace 平台没有记录你需要的那个节点的 input/output，需要找研发补数据。

五哈 badcase 的归因模板示例

Bug 现象：用户说"我想用吸顶屏看五哈" → 回复中出现竞品品牌"理想车机"

排查结果：
├ Planner Output:  异常，两个意图塞一个 tool call
├ Tool 选择:       部分对，用了 search_and_control_short_video，
│                  但缺少 vehicle_basic_control 激活屏幕
├ Tool Param:      异常，query 写了"激活后排吸顶屏并播放综艺《哈哈哈哈哈》"
│                  应该只写"综艺 五哈 最新一期"
├ Tool 返回:       异常，9条结果无一相关，rank3 含竞品品牌"理想汽车"
│                  (doc_id: 7448638465375309113)
├ Knowledge:       已加载 DRV-01-01-02 后排观影，内容无竞品词，
│                  但格式不兼容（speak/content/actions vs talk_or_not/talk_content/action_list）
├ History:         干净，无竞品品牌词
└ Prompt 约束:     缺失，无竞品品牌过滤规则

根因层：工具层 + Planner层
根因描述：Planner 将车控意图混入搜索 query 导致搜索跑偏，
         搜索工具返回竞品品牌内容且无过滤，Planner 忠实引用了竞品描述

修复建议：
  P0: 搜索工具增加竞品品牌过滤（理想/蔚来/小鹏/比亚迪等）
  P1: Prompt 增加多意图拆分规则 + 品牌约束兜底规则
  P2: 知识示例 JSON 格式统一为主 prompt 的 talk_or_not/talk_content/action_list
