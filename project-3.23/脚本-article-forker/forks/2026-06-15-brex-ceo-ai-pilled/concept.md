# 把 Agent 从"富士康工人"改造成"同事"：撞墙优先的 AI 采用法

> 从 Brex CEO Pedro Franceschi 提取的可迁移运营模型，适配到 Driver Agent 2.0 Planner / SP / Agent 工作。

## 核心问题

大多数人（尤其工程师）在用 AI 时，本能地把模型当成"昂贵又脆弱的零件"——写规则、裁上下文、加围栏、省 token，于是把一个有行动能力的同事用成了流水线工人。结果是：你以为模型不行，其实是你的 harness 不让它行。

Pedro 给出的解法不是"换个更大的模型"，而是一套**重写自己、再重写组织**的方法：先把 token 打到墙、亲手撞出模型的真实边界，再围绕这条边界重排分工、重写成本口径、重织上下文。

本概念页把这套方法落成 6 个可执行模块，每个都带 SOP / 清单 / 评分 / 诊断，并明确映射到你的 Planner 工作。

## 方法框架：6 个模块

### 1. 电的时间感 —— 把"渐进升级"换成"地基置换"

**原理**：当基础设施换代（推理模型+工具调用+harness 协同），按旧节奏"等成熟、等共识"就是在点蜡烛。判断标准从"这个功能稳不稳"变成"六个月后这套该怎么搭"。

**对你**：今天因为"豆包 Seed 做不到"而在 SP 里写死的规则、裁掉的能力，要打上"半年后复检"的标签——能力基线在抬升，写死的约束会变负债。

### 2. Free the Claw —— 区分"必要围栏"和"过度 harness"

**原理**：高级 AI 产品拆到底只有模型+工具+循环。过度 harness（多余规则、过度裁剪上下文、压制推理的围栏）会把同事用成工人。但**安全围栏不是过度 harness**。

**对你（关键边界）**：
- ✅ 属于 free the claw（可放开）：为省 token 而过度裁剪的历史上下文、限制模型只能走某条死板链路的规则、把本可让模型自己判断的事写成穷举 if。
- ❌ 不属于（必须保留）：车控类工具的安全围栏、金融/隐私红线、明确的兜底动作。这是责任底线。

### 3. 撞墙优先 —— 边界靠撞不靠想

**原理**：模型能做什么，推演不出来，必须把真实任务灌进去用经验试，直到墙出现。墙会移动（模型升级），要定期重撞。

**对你**：把"我觉得模型做不到 → 写死规则"的链路，换成"先丢给 Planner 跑 → 看它在哪一步崩 → 只在崩点加约束"。

### 4. Token ROI 记账 —— 从"省 token"升级到"算 ROI"

**原理**：token 会成为最大开支之一（单价跌但用量涨）。Magpie 追每一美元去向：客户产品 / 内部工具 / 员工工作流。AI 从创新项目变成标准经营科目。

**对你**：不要只在"省 token"层面优化。建立账本：每类调用烧的 token × 换回的决策质量。比如 69 个参考示例占 SP 50%——哪些示例真在提升输出，哪些只是在烧缓存却没改变决策。

### 5. 世界模型 —— 把离散触点织成连续上下文

**原理**：不是做几个孤立 agent，而是把按钮点击/邮件/电话/会议织进同一个上下文，让 AI 从"回答问题"变成"提前看见事情"。

**对你**：{{status}}/{{goal_list}}/{{env_info}}/{{memory}} 就是驾驶舱世界模型的雏形。下一步是让 Goal List + Advisor 共享看板演化成持续维护的"用户/车辆世界模型"，支撑 S9-S10 的持续任务（"提前看见用户下一步需求"）。

### 6. 人的位置 = 供给稀缺上下文

**原理**：只要 RAM 有限、模型装不下所有上下文，人就还有位置。模型还带"构建者世界观偏差"。人的不可替代价值 = 持续把模型看不见的现场信号灌进去 + 用自己场景的世界观校准它。

**对你**：豆包 Seed 看不见 D6X 车况、这位用户的记忆、此刻的视觉感知——你这个 PM 的护城河就是把这些"驾驶现场信号"工程化灌进 Planner，并用"驾驶场景世界观"纠偏通用模型的默认行为。

## 操作 SOP

### SOP-A：Free the Claw 体检（SP/工具约束审查）

- **输入**：当前 SP 全文（特别是 10 条注意事项、工具描述、上下文裁剪策略）+ 一组真实 badcase。
- **步骤**：
  1. 列出 SP 里所有"硬约束"（规则、围栏、裁剪、穷举 if）。
  2. 对每条打两个标签：①目的是【帮模型聚焦】还是【压制推理】；②是否属于【安全/合规底线】。
  3. 安全底线一律保留，不进入候选。
  4. 对"压制推理"且"非安全底线"的约束，做 A/B：放开它，重跑那组真实任务，看输出变好还是变乱。
  5. 变好 → 删/放松；变乱 → 保留并记录"为什么这条约束是必要的"。
- **输出**：一张"约束去留表"（保留 / 放松 / 删除 + 证据）。
- **失败模式**：把安全围栏误删（高危）；只凭直觉删而不跑真实任务验证。
- **怎么算成功**：删掉约束后，目标 badcase 的输出质量不降反升，且没有触发安全风险。

### SOP-B：撞墙日记（主动找边界，前置于 SP 迭代）

- **输入**：一个你"觉得模型做不到、准备写死规则"的需求。
- **步骤**：
  1. 先别写规则。把这个需求当成真实 query，灌进 Planner 完整跑一遍（含工具调用、多轮循环）。
  2. 记录它具体在哪一步崩：是链路走错、工具描述不清、上下文缺失、还是真的推理不动？（对齐你的四级 badcase 分诊）
  3. 只在"真的推理不动"的崩点加最小约束；其余三类不加规则，改修链路/工具描述/上下文。
  4. 标注这堵墙的日期——下个版本模型升级后重撞一次。
- **输出**：撞墙日记一条（需求 / 崩点类型 / 最小修复 / 重撞日期）。
- **怎么算成功**：你加的规则数量下降，而 Planner 的覆盖能力上升——说明你在删旧世界的"防御性规则"。

### SOP-C：Token ROI 账本（Magpie-lite for SP）

- **输入**：一次完整请求的 token 构成（SP 固定段 / 动态变量 / UP 历史 / 当轮输入）+ 输出质量评分。
- **步骤**：
  1. 把 SP 拆成成本块：角色定义 / 工具定义 / 注意事项 / 69 示例 / 设备知识 / 风格…标注各块 token 量。
  2. 对每个成本块问：删掉它，输出会变差吗？（用真实 badcase 验证，不靠直觉）
  3. 重点审示例区（占 SP ~50%）：哪些示例真在改变决策，哪些是"陪跑"。
  4. 把"高 token 低贡献"的块标记为优化候选；把"低 token 高贡献"的块固化到缓存前缀。
  5. 与缓存策略联动：动态变量是缓存断裂源，评估"精度收益 vs 缓存命中损失"。
- **输出**：SP 成本-收益表（每块：token / 是否改变决策 / 去留）。
- **怎么算成功**：总 input 下降的同时，输出质量不降；缓存命中率上升。

### SOP-D：世界模型织入清单

- **输入**：当前注入 Planner 的所有上下文源（status/goal_list/env_info/memory + 对话历史）。
- **步骤**：
  1. 列出驾驶舱里所有"离散触点"：语音 query、车况、视觉感知、用户记忆、历史目标、tool_feedback…
  2. 标注每个触点是【已织入同一上下文】还是【孤立的】。
  3. 对孤立触点问：把它织进来，能不能让 Planner"提前看见下一步"？（信号 vs 噪声判断）
  4. 高信号 → 织入并验证；低信号/噪声 → 不织（避免稀释）。
  5. 设计"持续维护"机制：世界模型不是单轮拼接，要随 Goal List 更新。
- **输出**：织入决策表 + 一个"提前看见"用例（如：连续车况+记忆 → 预判用户要开空调）。
- **怎么算成功**：出现 Planner"未被直接询问就给出有用预判"的 case，且预判准确率可衡量。

## 检查清单：你是把 Planner 当工人还是当同事？

- [ ] SP 里有没有"为省 token 而裁掉、其实模型需要"的上下文？
- [ ] 有没有"穷举 if / 死板链路"在替模型做本可让它自己判断的事？
- [ ] 我是先撞墙再加规则，还是凭"我觉得模型做不到"先写死？
- [ ] 安全围栏和过度 harness 我分清了吗？（车控类有没有被误放开）
- [ ] 我有没有 token 的"去向 × ROI"账本，还是只在"省 token"？
- [ ] 69 个示例里，我知道哪些真在改变决策、哪些是陪跑吗？
- [ ] 驾驶现场的离散触点，是织成了世界模型，还是各管各的？
- [ ] badcase 里我有没有分出"模型带通用世界观偏差"这一类？
- [ ] 作为 Planner 的"首席 AI 官"，我是亲手跑 trace/调 SP，还是只在需求层指挥？
- [ ] 我写死的约束，有没有"半年后随模型升级复检"的标签？

```text
你是 SP 约束审查官。下面是一段 Planner System Prompt 的约束/规则/上下文裁剪策略。
对每一条，输出：
1. 目的判定：【帮模型聚焦】还是【压制推理】
2. 安全判定：是否属于【安全/合规底线】（车控、隐私、兜底）
3. 建议：保留 / 放松 / 删除（安全底线一律保留）
4. 验证方案：用哪个真实任务 A/B 测试这条约束的去留
约束清单：
{{粘贴 SP 约束}}
```

```text
需求：{{描述一个你觉得模型做不到的需求}}
任务：不要先写规则。假设我把它当真实 query 灌进 Planner 完整跑一遍。

- 链路问题（走错模块）
```

## 60 分 vs 95 分对照

| 维度 | 60 分（还在把模型当工人） | 95 分（已经当同事用） |
|---|---|---|
| 加约束 | "我觉得模型做不到" → 直接写死 if | 先撞墙，只在真崩点加最小约束 |
| 上下文 | 一刀切省 token，远区全删 | 按"信号 vs 噪声"织入，安全/记忆类信号优先 |
| 成本 | 只看总 token，越省越好 | 看 token 去向 × 换回的决策质量 ROI |
| 围栏 | 安全围栏和过度 harness 混在一起 | 两者分层：安全底线保留，过度 harness 放开 |
| 现场信号 | 等线上 badcase 才补 | 主动把模型看不见的驾驶信号工程化灌入 |
| 偏差 | 模型答错 = 模型笨 | 分出"通用世界观偏差"，用驾驶场景校准 |
| 角色 | PM 在需求层指挥 | PM 是 Planner 的首席 AI 官，亲手撞边界 |

## 做对了的信号

- 你加的规则数量在**下降**，而 Planner 覆盖能力在上升。
- 出现 Planner"没被直接问就给出有用预判"的 case（世界模型生效）。
- SP 总 input 下降的同时输出质量不降，缓存命中率上升。
- badcase 复盘时你能说出"这条是过度 harness / 这条是必要围栏 / 这条是世界观偏差"。
- 你的迭代节奏从"等线上出问题"变成"主动撞墙找边界"。

## 仍停留在表层的信号

- 一遇到不稳就加规则、加示例、加围栏，SP 越来越长。
- 优化只停在"省 token"，说不清每块 token 换回了什么。
- 车控围栏被当成"过度工程"误放开（危险），或所有约束都不敢动。
- 把模型答错一律归为"模型能力问题"，从不区分上下文缺失和世界观偏差。
- 你只在需求文档层面工作，从不亲手跑 trace / 调 SP / 撞 badcase。
