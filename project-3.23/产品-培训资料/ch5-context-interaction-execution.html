5.1公共 Context 服务——所有 Agent 的"共享记忆仓库"

Director / Advisor / 感知模块都需要知道"现在车里发生了什么"，但各自去采集会重复且不一致。Context 服务是唯一的中间层：收集、存储、预处理、按需分发。

骨架图：多源写入 - 存储预处理 - 多方读取

写入/感知 存储/推理 预处理/执行 目标/Advisor 筛选入口

讲解：公共 Context 到底在干什么

第一步：理解 Context 的定义。Context = 传给模型的上下文信息。每个 Agent（Director、各 Advisor、工具 Agent）需要的 Context 不同，但很多内容是公用的——比如"现在车里有几个人""外面在下雨吗""用户刚才说了什么"。与其让每个 Agent 自己去采集，不如统一做一套公共 Context 服务来管理维护。它的核心功能就两个：按结构存储公共 context 信息，以及做预处理（压缩总结、打标分类、提前过滤）。
第二步：13 类 Context 内容全表。下面这张表是系统中所有公共 Context 类型的完整清单。按优先级分 P0（每次都带）、P1（场景需要时带）、P2（辅助/低频）：
类型说明来源优先级
情景描述感知模块 Always on 持续输出的舱内/舱外情景描述，按时间线记录感知模块P0
历史对话用户(ASR)和AI(TTS)的对话记录，按时间线排列ASR/TTS 模块P0
事件日志全量事件：人/车/助手完成的各类事件事件日志模块P0
端状态仅保留最新一帧——空调温度、车速、档位等端状态模块P0
屏幕截图5 分钟窗口内的屏幕截图序列端侧采集P0

基础联网信息时间/日期/星期/农历/节日/节气/天气联网服务P1

第三步：Context 的优化机制——三维过滤 + 情景压缩 + 按需抽取。13 类内容全部注入 SP/UP 会爆 token。所以 Context 服务不是"全塞"，而是做了三层优化：

（1）三维过滤——按场景/人物/任务三个维度裁剪：
- 场景过滤：当前是驻车场景，行车场景下的记忆偏好就不加载；当前是去外地自驾游，日常通勤的记忆就不用带；
- 人物过滤：车里只有主驾一个人，就不加载副驾或后排乘客的隐私记忆，减少噪音同时保护隐私；
- 任务过滤：当前任务是点外卖，就检索外卖相关的任务知识和用户偏好，不带导航知识。

（2）情景压缩——情景描述是 Always on 感知模块持续输出的，时间一长就会很长。做法是对远程情景做总结压缩（把"09:42 明骏上车，09:43 明骏系安全带，09:44 明骏拿出手机"压缩成"09:42-09:44 明骏上车后正常就座"），只保留近期的原始细节。

（3）按需抽取——短期用过的长期记忆会被 Context 服务"保持住"，不用重新检索；而长期记忆不会一次性全部加载，而是根据场景变化提前取一部分。
用户说"去昨天那家游泳馆"。Context 服务此时做的事：（1）场景过滤确认当前是行车场景；（2）人物过滤确认当前说话人是主驾"超然"；（3）任务过滤识别到"昨天那家"涉及记忆，从长期记忆中按人物 id 抽取"超然"的地点偏好和最近事件总结；（4）组装成 Context 注入 Director SP 的动态变量槽位 {{memory}} 和 {{env_info}}。Director 读到后才能决策"先调 search_user_memory → 拿到地址 → 再调 route_planning_qa"。
第四步：任务队列数据结构。任务状态是 P0 Context 中最核心的运行时数据。每个任务是一个 JSON 对象，包含五个字段：
- task_id：唯一标识
- rank：执行优先级（数字越小越先执行）
- task_content：任务描述自然语言
- task_condition：触发条件（空=立即执行；"车辆挂入D档"=等条件满足）
- task_nlg：预设话术（空=让 Director 自己组织语言）
一个早晨通勤场景的任务队列：rank=1 是"给明骏和雨晴开座椅按摩"（立即执行），rank=2 是"导航去雨晴公司再去明骏公司"（条件=挂D档），rank=3 是"推荐小宇宙播客最新一期"（条件=车辆开始行驶后，且有预设 NLG 话术），rank=4 是"调研GPT5做PPT"（无条件但最低优先级，随时可能被更高优先级打断）。
为什么 Context 是"公共服务"而不是各 Agent 自己管？因为多个 Agent 同时运行（Director + 4 个 Advisor + 若干动态 Advisor），如果各自采集，会出现数据不一致（A 认为空调 24 度，B 认为 22 度）和重复采集浪费资源的问题。统一服务确保 single source of truth。
"远近区"优化是 1.0 时代的做法（远区删 tool_feedback/action_list、近区完整保留）。2.0 架构中 Context 优化的正确描述是"三维过滤 + 情景压缩 + 按需抽取"，不要混淆。三维过滤是面向整个 Context 空间的，不只是对话历史。
P0 类 Context（情景描述、历史对话、事件日志、端状态、屏幕截图、用户记忆、任务状态）每次调用都会注入，加起来可能就 15K-20K tokens。如果 SP 固定部分再有 30K+，整个 input 很容易超过 50K，命中 KV Cache 的概率会急剧下降。这是"长对话越来越慢"的根因之一。

5.2前置筛选 / 知识筛选专家——"Director 调用前先塞对的知识"

Director 的 SP 有 30K+ tokens，不可能把所有场景知识都塞进去。前置筛选是一个轻量 Selector 模型，在 Director 推理之前，根据当前状态判断"该给 Director 加载哪些专家知识"，做增量注入。

骨架图：8字段输入 - 6类专家知识库 - 增量输出 - 注入动态专业知识

输入字段 Selector 推理 EXP销售 DRV司机 ENT娱乐 EMO情感

讲解：为什么需要前置筛选，它到底在做什么

核心问题：Director SP 的 {{knowledge}} 槽位大小有限。Director 的 SP 已经 30K+ tokens，不可能把所有场景的编排方案都塞进去。13 个场景（急速降温、急速升温、冥想、小憩、后排观影、宠物关怀......）的完整知识加起来可能有上万 tokens。所以需要一个"前置筛选器"：在 Director 推理之前，判断当前 query 可能涉及哪些场景，只把命中的场景知识注入。
Selector 的输入是 8 个字段。它不只看 query 本身，还看近 10 轮对话（判断上下文连续性）、视觉感知（后排有小孩？）、目标队列（正在执行什么？）、车辆状态（驻车 or 行车？）、变化点（什么刚刚发生了变化？）、用户记忆（这人有什么偏好？）、以及已加载的知识 ID（避免重复注入）。
输出格式是增量式的 JSON：{"add_knowledge_ids": ["COM-01-01-01"], "remove_knowledge_ids": ["ENT-02-01"], "reasoning": {...}}。add 是这次需要新加载的场景知识 ID，remove 是上一轮加载了但本轮不再需要的。增量更新而非全量替换，确保知识不会无谓膨胀也不会丢失。
6 类专家知识库对照表：
专家代码角色覆盖场景典型知识 ID
EXP销售专家功能介绍、产品亮点、新手引导、试驾陪伴EXP-01-xx
DRV司机专家驾驶安全、路况播报、临停、泊车、驾驶风格DRV-01-xx
COM管家专家温控（降温/升温）、座椅、空气、灯光、关怀场景COM-01-xx
ENT娱乐专家音乐、短视频、播客、前排/后排/5D 观影ENT-01-xx
LIFE生活助理专家外卖、订座、日程、充电、记忆管理LIFE-01-xx
EMO情感伙伴专家情绪安抚、深度聊天、冥想空间、放松模式EMO-01-xx
5 条筛选原则——Selector 不是搜索引擎，是"守门员"：
（1）场景绑定：知识必须跟当前场景强相关。驻车场景不加载行车安全知识，行车场景不加载驻车观影知识。
（2）关注变化：优先关注"什么刚刚发生了变化"。温度骤降 10 度、有人上车、雨刚开始下——这些变化点才是触发场景知识的信号。
（3）目标关联：如果目标队列里有"正在导航去西湖"，就应该加载 DRV 的沿途信息相关知识。
（4）增量更新：只 add 新命中的、remove 不再需要的，不做全量替换。
（5）不虚构：如果没有匹配的场景，输出 ID_000（无匹配），绝不强行塞知识进去。宁可不加也不加错。
完整链路："好热啊" 从 query 到知识注入：
1. 用户说"好热啊"，ASR 转文本。
2. Selector 接收 8 字段输入：query="好热啊"、sound_area=主驾、近 10 轮无温控相关话题、视觉感知=阳光强烈、车辆状态=外温 38 度/内温 29 度/空调 AUTO 模式、变化点=外温 5 分钟内升了 4 度、已加载知识 ID=空。
3. Selector 三步判定：步骤 1 是车内场景需求 → 步骤 2 "好热啊"需要一套方案而非单一动作 → 步骤 3 匹配 COM-01-01-01（急速降温）触发规则"含明确热体感词"。
4. 输出 {"add_knowledge_ids":["COM-01-01-01"],"remove_knowledge_ids":[]}。
5. 系统把 COM-01-01-01 的完整编排方案（查空调当前状态 → 调低温度到 22 度 → 风量调到最大 → 开内循环 → 关天窗遮阳帘）注入 Director SP 的 {{knowledge}} 槽位。
6. Director 读到后，就知道这不是简单的"调空调到某某度"，而是一套多步骤编排，输出相应的 action_list。
为什么不直接让 Director 自己判断该用什么知识？因为 Director 的 SP 已经 30K+ tokens，如果还要它在推理时"先想想该加载什么知识"，等于把分类任务和执行任务混在一起，注意力分散会导致分类不准且推理变慢。Selector 用的是轻量 flash 模型（doubao-seed-1.6-flash），延迟目标 < 500ms，跟 Director 推理并行执行，总链路不增加等待时间。
Selector 当前最大的坑是延迟。初版 Selector 延迟 1,805ms（input tokens 3,355，大部分是场景描述），优化方向是把场景描述压缩成"关键词锚定"格式（4000 字压到 600 字），input tokens 降到 ~800，预计 TTFT 降低 40-60%。更进一步可以做"规则前置 + LLM 兜底"——高频场景（抽烟/通风/冥想/小憩）用正则直接命中 < 10ms，覆盖约 40% 请求。
易混淆场景的互斥锚定是 Selector 出错的重灾区：
- "孩子睡了别打扰" → 013（后排勿扰）而非 001（儿童陪伴）
- "累了减压" → 008（冥想）而非 009（小憩）
- "好热好臭开窗" → 004+012（双命中，降温+异味通风）
- "把空调调到25度" → 000（单一动作，不需要场景知识）
区分的关键在于第 2 步："需要一套方案"还是"单一动作"。有体感词（好热/好冷）就走场景，有精准参数（25度）就走 000。

5.3Goal List 共享目标看板——Director 和 Advisor 的"协作白板"

从"Planner 一人包揽"进化到"团队协作"的关键结构。Goal List 是一个所有角色都能读写的共享看板，让 Director 管快速响应+执行，Advisor 管深度思考+目标监控。

骨架图：中心看板被 Director / Advisor / HMI 读写

Director (创建/删除/读取) 静态Advisor (更新/review/读取) 动态Advisor/HMI (读取/更新)

讲解：Goal List 是怎么运转的

第一步：理解为什么需要 Goal List。1.0 架构中，Planner 一人包揽所有事情——理解意图、规划任务、调工具、跟踪进度、恢复被打断的任务。这导致 Planner 过载：SP 太长、注意力分散、容易"忘事"。Goal List 的引入就是把"任务管理"从 Planner 脑子里拿出来，变成一个外部化的、所有角色共享的数据结构。
第二步：读写权责。不同角色对 Goal List 的权限不同：
角色创建目标更新进展Review 达成删除目标读取
Director可以--可以可以
4 静态 Advisor-可以可以-可以
动态 Advisor-可以(仅绑定目标)可以(仅绑定目标)-可以
HMI 卡片----可以
第三步：数据结构。Goal List 是一个 JSON 数组，每个元素三个字段：
- "目标"：目标描述（如"介绍车功能"）
- "目标状态"：进行中 / 已完成 / 暂停 / 取消
- "目标细节和进展"：自由文本，记录目标的当前进展和中间信息

分两区存储：static 区对应 4 个永久 Advisor（舒适、出行、情感、内容），这些 Advisor 是"永久合同监工"，只要车在开就在运行；dynamic 区对应动态注册的 Advisor，目标达成/取消后 Advisor 生命周期结束，目标从 dynamic 区移除。
第四步：生命周期完整流程。
（1）创建：用户说"介绍一下这台车的功能"→ Director 判断这是长时任务 → 调用 goal_list_update 工具（type=add_goal）→ Goal List 新增一条{"目标":"介绍车功能","目标状态":"进行中","目标细节和进展":""}→ 工程动态注册一个"功能介绍 Advisor"。
（2）更新：Director 开始介绍智驾功能，介绍到一半被用户打断 → 对应 Advisor 更新进展字段为"已介绍智驾部分，待介绍座舱和车控"。
（3）Review：用户打断处理完后，静态 Advisor（内容类）读取 Goal List，发现有"介绍车功能"还没完成 → 发出 advisor 建议"可以接着介绍了" → Director 收到后决定是否恢复。
（4）删除：介绍完成 → Advisor review 判断目标达成 → Director 调用 goal_list_update（type=delete_goal）→ 目标移除，动态 Advisor 生命周期结束。
用户说"介绍一下这台车的功能"。Director 创建目标，开始介绍智驾。说到"此外我还可以帮你做很多别的事情——"，用户打断说"导航去赛力斯集团"。Director 立即处理导航。导航完成后，内容 Advisor 读 Goal List 发现目标"介绍车功能"状态为"进行中"、细节为"已讲智驾，待讲座舱"，于是发出 advisor 建议："刚被打断了，可以继续介绍车功能"。Director 收到后恢复："刚刚被打断啦，我们接着来介绍吧～刚说到，我可以帮你做很多别的事情，包括......"
为什么 Director 只负责创建和删除，进展更新交给 Advisor？因为 Director 是"快响应"角色——收到 query 就要尽快出 action_list，没有余力去细致跟踪每个目标的进展。而 Advisor 是异步运行的"后台监工"，有时间慢慢 review 目标的完成度。这就是"分工协作"的精髓：Director 管执行，Advisor 管跟踪。
"目标细节和进展"是自由文本，这带来一个隐患：不同 Advisor 写入的进展格式不统一、信息质量参差不齐。比如一个 Advisor 写"已完成第一步"，另一个写"导航已设置，目的地赛力斯集团龙兴基地，全程13公里"——信息密度差很远。当 Director 需要读进展来恢复任务时，如果进展描述太模糊，恢复质量就会很差。这是当前架构的一个已知软肋。
Goal List 的大小没有上限控制。如果用户在一次长对话中创建了十几个目标、大部分处于"进行中"状态，整个 Goal List 作为 Context 注入 Director SP 的 {{goal_list}} 变量，会占用大量 tokens，挤压其他信息的空间，且容易导致 Director 注意力分散、"忘掉"某些目标。

5.4任务与交互管理——"新 query 来了，该排队还是打断？"

车载语音是单通路管道：同时只能有一个 TTS 在播报、一个 Director 在推理。当新输入到达时，系统必须在毫秒级决定：打断还是排队？合并还是丢弃？这套决策逻辑是交互管理的核心。

流程图：新输入到达 - 首 token 出没出 - 打断/合并/排队

讲解：交互管理的核心决策树

第一步：理解"单通路管道"约束。车载语音交互不像聊天软件可以开多个对话窗。同一时刻只能有一个 TTS 在播报、一个 Director 在推理。所以当新输入到达时，系统必须做出选择：当前正在做的事情怎么处理？新来的事情怎么安排？
第二步：核心分叉——"首 token 出了没？"
情况 A：首 token 还没出（Director 还在思考中）。这时候最优策略是"打断合并"——把旧 query 和新 query 合并成一个请求，重新提交给 Director。因为 Director 还没产出任何结果，打断无成本。
情况 B：首 token 已出（TTS 正在播报或已有输出）。这时候要看新输入的来源：
- P0 用户打断式：用户亲自说了话，立即停止当前播报，执行新 query。用户永远最优先。
- P1 非用户排队式：Advisor 或系统事件触发的，排队等当前播报完再处理。可以利用等待时间做"预推理缓存"——先偷偷推理好，播完了直接出结果。
第三步：六种冲突策略。
策略说明典型场景
扔弃直接丢弃旧任务的结果旧任务还在思考，用户说了完全不相关的新指令
排队新任务排在旧任务之后Advisor 建议在用户对话中间到达
静默新任务静默执行，不打断播报简单车控（开车窗）只发叮声，不停止当前 TTS
整合合并旧 query 和新 query 重新提交首 token 未出时两个 query 合并
插队新任务强制优先执行安全类告警、紧急导航变更
停止旧立即终止旧任务执行新任务用户打断正在播报的内容
第四步：优先级矩阵。当多个任务冲突时，按以下优先级排序（从高到低）：
安全（碰撞预警/紧急制动）> 导航（路线变更/到站提醒）> 普通用户指令 > 配置类（调空调/座椅）> 条件触发类（到站叫醒）> 非配置类（播音乐）> 简单查询 > 复杂多步任务
注意最后一个：复杂多步任务优先级最低。因为它耗时长，如果让它占住管道，短任务就得排很久队。
第五步：三条合并特殊规则。
（1）长程任务不参与合并：如果旧任务是"介绍沿途风景"这种 Always on 长程任务，新 query 来了不要把两者合并，而是暂停长程任务，处理完新 query 后由 Advisor 决定是否恢复。
（2）AIGC 内容走通知卡，不走 Director 管道：AI 播客生成完成、图片生成完成这类 AIGC 产物，不要插入 Director 的推理管道，而是直接以通知卡片形式 push 到 HMI，不抢占语音通路。
（3）同一个 Advisor 只保留最新：如果舒适 Advisor 连发了两条建议（第一条"温度有点高，建议调低"、第二条"已检测到你开了车窗，取消降温建议"），只保留最新的那条。
第六步：任务工程管理——打断后怎么恢复？
（1）打断 → 保存 checkpoint：被打断的任务状态标记为 interrupted，同时保存当前的执行进度（已调了哪些工具、tool_feedback 是什么、Goal List 里的进展记录）作为 checkpoint。
（2）恢复 → Advisor 读 Goal List 重触发：新任务执行完后，对应的 Advisor 读 Goal List 发现有 interrupted 的目标，发出 advisor 建议"可以恢复介绍车功能"，Director 收到后带着 checkpoint 上下文继续。
（3）挂起 → Always on 监听：长时任务（如"帮我介绍沿途风景"）不是被打断就结束，而是挂起到后台，感知模块持续监听，条件满足时重新激活。
（4）失效兜底：端侧超时（端侧等不到云端响应）或云端工具超时（工具调用无返回）→ 先告知用户"稍等"→ 重试一次 → 仍失败则告知用户并结束任务。
用户说"给我讲讲今天的财经新闻"。Director 开始推理、调用 web_search、拿到结果、开始 TTS 播报。播报到一半，用户打断说"先导航到海底捞"。
首 token 已出 + P0 用户打断 → 立即停止 TTS → 状态=interrupted + checkpoint（已播报到"证监会部署多项..."）→ 执行导航任务。
导航完成后，内容 Advisor 读 Goal List 发现"讲财经新闻"还没完，发出 advisor 建议 → Director 恢复："刚刚被打断了，我接着为你讲今天的财经新闻吧，刚说到证监会部署多项资本市场发展举措......"
合并场景：用户说"给我讲讲财经新闻"，Director 还在思考（首 token 未出），用户又说"先导航海底捞"。→ 打断旧进程 → 合并两个 query → 重新请求 Director → 输出："好的，我来帮你导航去海底捞，顺便整理播报今天的财经新闻。"一次性解决两个需求。
为什么"首 token 出没出"是核心分叉点？因为首 token 出了意味着已经产生了用户可感知的输出（TTS 开始播了），打断会造成"说了一半突然停"的不良体验，需要更慎重地处理（保存 checkpoint、后续恢复）。而首 token 还没出时，用户还在等待中，合并重发对用户体验零损失。
"预推理缓存"听起来很美——P1 排队时偷偷推理好，播完直接出结果。但实际上有坑：如果当前 TTS 播报的内容会改变 context（比如 Director 正在和用户确认某件事，用户回答了"好"），那"偷偷推理"的时候 context 还没包含这个回答，推理出来的结果可能已经过时。所以预推理只适用于"不依赖当前播报结果"的场景。

5.5执行层：26 工具 + HMI 卡片——"action_list 出来之后，谁干活？"

Director 输出 action_list，交互管理决定执行顺序，最终工具执行器逐条调用。成功就把 tool_feedback 回灌给 Director，失败就触发 Replan。同时，结果要通过 HMI 卡片呈现给用户。

讲解：执行层的工作机制

第一步：action_list 怎么执行。Director 输出的 JSON 中 action_list 是一个数组，每个 action 包含 action_id、tool_name、params。工具执行器拿到后判断依赖关系：无依赖的工具并行调用（如 search_poi_qa 和 route_planning_qa 可以同时发），有依赖的串行调用（如先 search_user_memory 拿到地址，再 route_planning_qa 导航）。Director 在 SP 注意事项第 6 条已经约束了："如果工具有串行依赖，不要一次性输出多个工具请求"——这就是让模型自己管好串行依赖。
第二步：3 个易混点——新人最容易搞错的地方。
（1）媒体播控不是独立工具，归 vehicle_basic_control。上一首/下一首/暂停/播放这些播控指令不走 search_and_control_music，而是走 vehicle_basic_control（它管导航、播控、空调、座椅等所有车辆基础控制）。search_and_control_music 只管"搜索/推荐"。
（2）记忆分两类：search_user_memory（只读检索）和 operate_user_memory（写入更新）。查"超然爱吃什么"用前者，记住"超然今天点了日料"用后者。Director SP 中两个工具的 tool_describe 差异很细微，模型经常搞混。
（3）4 个导航类工具不要并行调用。search_poi_qa（搜 POI）、route_planning_qa（规划路线）、navi_basic_control（导航控制）、search_weather（天气查询，影响路线选择）——它们之间往往有隐含依赖。比如搜完 POI 才能导航去那个地方，所以不要并行。
第三步：播报归属——谁 Director 统一回？谁 Agent 自己播？
Director 统一播的工具：vehicle_basic_control、search_and_control_music、search_and_control_short_video、gui_agent_operation——这些工具返回的 tool_feedback 是结构化数据（POI 列表、路线信息、车况数据），需要 Director 阅读后用自然语言总结播报给用户。
Agent 自己播的工具：本地生活类（实时语音交互）、search_visual_info（VQA 有自己的回答模板）、vehicle_manual_qa（车书查询有自己的回答格式）、S2S（端到端语音模型）——这些 Agent 有自己的 TTS 通路。
特殊：web_search——返回的是原始网页摘要，由 Director 阅读后总结成用户友好的 talk_content 播报。
第四步：HMI 三类卡片。
（1）小卡（AI 语音卡）：最多同时显示 1 张。展示 Director 的 talk_content 文字版——就是语音播报的文字对照。我方（字节）容器控制。
（2）大卡（AI 画布卡）：最多同时显示 1 张，且与小卡互斥（显示大卡时小卡消失）。用于展示富内容——POI 列表、搜索结果、图表、图片生成结果等。上汽容器控制（因为涉及车机大屏 UI）。
（3）通知卡（Push 通知）：可同时存在多张。独立通道，不占用语音管道。用于 AIGC 内容完成通知、条件触发提醒、Advisor 建议卡片等。
用户说"去虹桥机场附近找个好吃的日料"。Director 并行调用 route_planning_qa（规划路线）和 search_poi_qa（搜日料店）。route_planning_qa 先返回"路线规划成功"——Director 看到后判断不需要说话（界面已显示路线）。search_poi_qa 返回 3 家日料店——Director 阅读后总结："沿途有家�的一日本料理，评分 4.8，就在 T2 附近，要不要设为途经点？" 同时 HMI 显示大卡展示 3 家店的详情列表。

为什么"成功回灌"而不是"成功直接结束"？因为很多任务是多步的。Director 调 search_user_memory 拿到"昨天去的是XX游泳馆"，这只是中间步骤——还需要拿着这个地址调 route_planning_qa 去导航。所以 tool_feedback 必须回灌给 Director 作为下一轮的 input，让 Director 决定"是继续调工具还是可以对用户说话了"。
为什么大卡和小卡互斥？因为车机大屏空间有限。如果同时显示 AI 语音卡的文字和 POI 列表的富内容卡，界面会很拥挤，用户注意力分散。互斥确保同一时刻屏幕上只有一个重点信息在展示。
Replan 的死循环陷阱：如果工具调用失败，Director 收到失败的 tool_feedback 后会尝试 replan。但如果 replan 后再次调用同一个工具还是失败（比如网络断了），Director 可能会无限重试。SP 注意事项第 8 条约束了这一点："工具反复调用失败时，可暂时放弃。" 但实际上模型不一定能严格遵守，需要工程侧加上最大重试次数（通常 2-3 次）的硬限制。
26 个工具的 SP token 占比问题：26 个工具的完整 tool_describe + param + notice 在 SP 中占据约 5K-8K tokens。加上参考示例（占 SP 约 50%），SP 总量很容易超过 30K。这直接影响 KV Cache 命中率和推理延迟。优化方向之一是动态工具加载——根据前置筛选的结果，只注入当前场景需要的工具子集，而不是全量 26 个。
