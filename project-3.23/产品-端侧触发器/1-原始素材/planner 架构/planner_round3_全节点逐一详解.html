感知/信息层推理层执行/交互层回灌闭环
三层一句话：左边「感知/信息层」负责知道现在车里啥情况（context 13类）；中间「推理层」负责决定干什么（三个模型分流 + Director 拆解）；右边「执行/交互层」负责把事做了并告诉用户（18工具 + 卡片/TTS）。下面逐层逐节点拆。

1感知 / 信息层：6 个节点（系统的"眼睛和记忆"）

① user（用户输入）PRD图

是什么用户的语音/手势/屏幕操作，经 ASR/DMS-OMS/屏幕采集转成 query。
输入输出入=语音/动作；出={"query","speaker_name","speaker_position"}（带说话人和座位）。
怎么工作声学→vad→asr→query；手势→DMS/OMS→query；屏幕操作→事件日志。
产品影响speaker_position 决定"给谁服务"（主驾/二排），是多音区个性化的根。

② advisor（建议输入）PRD图

是什么4个静态advisor（舒适/出行/情感/内容）+ 动态advisor 给 Director 的主动建议。
输入输出出={"advisor":"晓伟上车了，车内温度高，可调空调降温"}。
怎么工作advisor 读 context 情景，判断时机合适就给 Director 提议，由它来决定说不说/做不做。
产品影响这是"主动服务"的源头——用户没开口，系统也能先关心一句。

④ 触发器（离线触发器 + 在线触发器）PRD图PRD文

5.3 四个动态模板变量（每次调用注入的"实时现场"）SP原文

占位符标题注入内容
{{status}}舱内视觉感知车内摄像头：人员着装/行为/物品/光线
{{goal_list}}当前目标队列持续追踪的目标list+进展

5.4 14 条注意事项（Director 的"行为宪法"）SP原文

  1. 听不懂/不是对你说的 → 不说话也不调工具
  2. 需求模糊缺信息 → 可以询问
  3. 偏闲聊 → 只说话不一定行动
  4. tool_feedback 不一定都回话（关窗成功不必再说）
  5. 工具失败 → 不必道歉，想办法处理
  6. 工具有串行依赖 → 不要一次性输出多个（先调一个等返回再下一个）
  7. 不重复说相同的话
  8. 工具反复失败 → 可暂时放弃
  9. 连续重复相似输入 → 不说话
  10. 仅输出JSON
  11. 每次任务独立执行，不受历史失败影响；用户重发同类任务必须重新new_task，不得预判失败
  12. 新任务关键参数(地址/收货人)不得默认沿用历史，除非用户说"和上次一样"
  13. 任务被打断终止后重发 → 用new_task，不用add_info
  14. GUI等待状态路由优先级：有进行中GUI任务且feedback要求补信息时，用户下一句(哪怕"确认"/验证码)必须优先路由为对GUI任务的add_info回复

7.3 ★ 把所有节点串成一条活链路：「照顾二排宝宝」时序图（图7，11步）PRD图

泳道：user · director · goal list · 动态advisor · 感知触发模块 · 视觉感知器 · context。这11步把上面所有节点连起来了：

1
user → director：帮我照顾一下二排宝宝
2
director → goal list：新增目标：照顾二排宝宝（调 goal_list_update）
3
director → 动态advisor：新建动态advisor（按模板注册临时监工）
4
动态advisor → 感知触发模块：感知和触发需求分发：每10s告诉我一次二排宝宝视觉状态
5
感知触发模块 → 视觉感知器：感知注意力影响：重点关注二排宝宝状态（=调sp/attention）
6
感知触发模块 ↻ 自循环：定时器
7
感知触发模块 → 动态advisor：每5s触发（唤醒监工）
8
动态advisor ↻ 自循环：目标完成情况review
9
动态advisor → goal list：更新目标细节和进度（"宝宝睡着了，持续监测中"）
10
动态advisor → director：给出建议，推进目标（如"宝宝醒了在哭，建议放安抚音乐"）
11
视觉感知器 → context：视觉感知信息上报（流式写回）
看懂这11步=看懂整套目标驱动：Director只做了1-3步（建目标+建监工）就"撒手"了；之后4-11步是监工(advisor)+订阅+视觉在自循环跑，只有真要推进时(第10步)才回头找Director。这就是"减轻Director过载"的具体体现——Director不盯进度，监工盯。

8.1 VLM 怎么把"图像"变成"语言"PRD文

"识别方案：大小LLM融合，且大LLM的识别目标(SP)是动态生成的。小模型识别既定高优标签(兜底)；大模型不训练只引导(如识别人是否睡着)。支持adviser/Planner的自然语言调用，并输出目标自然语言描述结果。"

所以"视觉转语言"= 大模型按一段 System Prompt（识别目标）去看画面，输出一段自然语言描述，图像变成 token 进 context。例："明骏坐主驾，穿粉色皮卡丘T恤戴黑墨镜；后排有一只柯基趴座椅上，未系宠物安全带。"

9前置筛选模块（知识筛选专家）：真实存在的优化模块PRD文

你 Schema 提的"上下文优化/前置筛选"，我找到了它真实的 SP。它是一个独立的小模型，专门在 Director 之前裁剪要注入的知识，治"50+示例冗余拖慢推理"的病。

输入（8个字段）

用户Query / 近10轮对话 / 舱内视觉{{status}} / 目标队列{{goal_list}} / 车辆状态{{env_info}} / 状态变化点{{env_changes}} / 用户记忆{{memory}} / 已加载知识ID列表

输出（增量更新知识ID）

{"add_knowledge_ids":[...], "remove_knowledge_ids":[...], "reasoning":{...}}

知识库结构：6类专家角色 × 四段式知识ID 专家代码-一级-二级-三级：

专家代码角色覆盖场景举例
EXP销售/体验官购车体验/试驾伴行/离车送别
DRV司机/导游通勤/接送/自驾游/充电/泊车
COM随侍/管家调温/调气味/调光/晕车缓解/小憩/冥想
ENTDJ/陪乐/主播听歌/播客/新闻/游戏/节目编排
LIFE管家/助理停车缴费/查天气航班赛事金融快递百科
EMO伙伴迎宾/惊喜彩蛋/闲聊

5条筛选原则：①场景绑定推理 ②关注环境变化（趋势比绝对值更重要） ③目标持续关联（目标在则知识保留）④增量更新（只输出变化部分）⑤信息不足不虚构。

这就是 token 优化的真身：不是把所有知识/示例一股脑塞给 Director，而是先用这个小模型按当前场景挑出几条相关知识ID，只注入这几条。它甚至专门盯"变化点"（温度骤升→调温知识；进地库→泊车知识）。你做 Planner 优化，这个模块是直接抓手。
