# Query — 知识查询

核心理念：好的回答本身就是知识，应回流到 wiki 中。一次对比分析、一个发现的关联、一段综合论述——这些不应消失在聊天记录里，而应像摄入的源文档一样在知识库中持续积累。

## 前置条件

- 已 fetch AGENTS 文档查看规范

## 步骤

1. **从 `~/.llm_wiki.setting.json` 读取配置**（含 storage_type、space_id），获取 INDEX → 解析注册表和 token

2. **搜索相关页面**
   - 在注册表中按标题/关联匹配，可选 docs +search 补充

3. **fetch 3-5 个最相关页面**，深入阅读

4. **综合回答**（引用 mention-doc），回答形式根据问题类型灵活选择：
   - Markdown 文档（默认）
   - 对比表格（Comparison）
   - 幻灯片（Marp）
   - 图表（matplotlib）
   - 画板（飞书白板）

5. **归档判断与推荐**

   根据回答类型判断是否推荐归档：

   | 回答特征 | 归档推荐 | 页面类型 |
   |---------|---------|---------|
   | 对比了 2+ 实体或概念 | 推荐归档 | Comparison |
   | 综合了 3+ wiki 页面进行综述 | 推荐归档 | Overview |
   | 简单事实查找（答案来自单页） | 不归档 | — |
   | 解释已有页面内容 | 不归档 | — |

   当判断为「推荐归档」时：
   - 主动向用户推荐，展示拟创建的页面标题和存放目录
   - 用户确认后立即创建页面、补充交叉引用、更新 INDEX 和 LOG
   - Comparison → 参照 `adapter/<STORAGE_TYPE>.md`「创建文档」，使用 COMPARISONS_TOKEN 作为父文件夹
   - Overview/综述 → 参照 `adapter/<STORAGE_TYPE>.md`「创建文档」，使用 OVERVIEWS_TOKEN 作为父文件夹
   - 归档后补充交叉引用，让新页面融入已有知识网络
   - 用户拒绝则跳过，在 LOG 中记录「归档: 无（用户跳过）」

6. **丰富已有页面**（无论是否归档都执行）

   检查回答是否为已有页面带来了新信息：
   - 发现了 Entity A 和 Entity B 之间的新关联 → 更新双方的「相关实体」段落
   - 为某个 Concept 补充了新的来源或解释 → 更新该 Concept 的「来源」和「详细说明」
   - 修正了某页面的过时信息 → 更新该页面内容和 callout 中的「最后更新」

   更新方式：使用 `docs +update --mode replace_range` 精确替换相关段落，避免全量覆盖。
   更新前向用户简要说明要更新哪些页面及原因。

7. **更新 INDEX 和 LOG**
