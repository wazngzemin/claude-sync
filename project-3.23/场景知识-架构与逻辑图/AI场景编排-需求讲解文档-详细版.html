W4：Skill 知识质量修复 P0 · 最高优

① 这个问题是怎么来的？

Skill 是我写给 Planner 的"场景知识"——告诉 Planner "用户想开宠物模式，你应该调哪些车控"。第一版 Skill 是 5 月中旬粗写的，没有和上汽的实际车控能力做对齐。

问题暴露过程：

  1. 5/23 第一版上线 → 小明在备用车上做走查
  2. 5/29 走查会 → 小明展示了一批 badcase，发现很多场景 Planner 调不通
  3. 6/3 上午会 → 和小明/左晨/明骏深入分析，确认根因是 Skill 里写了车厂不具备的能力

具体 badcase（小明实车走查发现的）

Badcase 1：离车保电

小明："周六我去看的时候，他相当于他没有去调，他就直接他去自己去 plan 了，plan 之后说不行。"

原因：Skill 里写了"调用离车保电模式"，但方向靠车控协议里没有这个指令。Planner 找不到这个 function call，只能自己瞎编排，编排完发现也不对，就说"不支持"。

Badcase 2：宠物模式

小明："他下给方先下发的是'使用宠物模式'，你又没有嘛。"

原因：Skill 把宠物模式写成了编排逻辑，引导 Planner 给方向靠下发"使用宠物模式"指令。但方向靠不认识这个指令。Planner 退而求其次"手动模拟"——调了空调温度和关窗，但漏掉了大屏显示"主人马上回来"、物理按键禁用、高压维持等关键动作。

Badcase 3：观影模式

小明："这个知识咋写的？我没细看啊。"

原因：Skill 引导 Planner 给车控下发"观影模式"一级指令，但车控只有空调、座椅、车窗等原子操作，没有"观影模式"这个一级指令。

Badcase 4：新手模式

小明："按道理讲，他所有这种罗列的都对应下头会有一个操作，但实际上什么都没有。对，就是假的。"

原因：Planner 罗列了一堆要执行的操作（空调、座椅等），但实际没调工具或工具没返回结果。Skill 引导不正确 + SP 里缺少"工具调用失败时怎么办"的规则。

Badcase 5：通风模式

小明原话提到通风模式在某些情况下失败，以及"左后小气"这种自创表达 Planner 不理解。

根因总结（6/3 上午会上小明的话）：

小明："有好多是车厂不具备的能力，也在里边写。对，那你就要给 planner，他肯定返回不回来。"

② 不做会怎样？

Skill 不修复 → Planner 继续拿着错误知识编排 → 调不存在的指令 → 用户看到"先说能做，又说不行，又说手搓一个，最后没执行"的混乱体验 → 6/8 演示直接翻车。

③ 具体怎么做？（精确到每一步操作）

第一步：拿三份参照材料

材料找谁要怎么要（话术）当前状态
小明的 bug 走查表 杨麟（小明） "小明，上次说好的走查 bug 表能发一下吗？我需要逐条分析 root cause，修复 Skill。6/8 要演示，比较急。" 还没发——他承诺发群里但一直没发
方向靠 SOA 完整能力清单（约 700 条原子功能） 左晨 "左晨，能把方向靠 SOA 的完整能力清单发我一下吗？我需要逐条对照 Skill，确认哪些功能车上实际有、哪些没有，避免 Planner 调不存在的指令。" 还没提供——6/3 会上确认车上有 9-10 个模式但协议里很多没注册
上汽《AI场景编排评估》文档（20 张产品功能结构图+流程图） 已有 — 已拿到——在 场景知识-架构与逻辑图/AI场景编排 评估.html

第二步：逐场景对照重写 Skill

拿着上面三份材料，打开每一个场景的 Skill 文件，逐条对照：

  1. 打开上汽 UC 文档中该场景的产品功能结构图（思维导图）——看上汽原来设计了哪些执行项
  2. 打开 SOA 清单——看 Skill 里写的每一条执行项，SOA 里有没有对应的原子功能
  3. 有的保留，写成 SOA 的原子操作格式（如"空调开关:ON"而不是"打开空调模式"）
  4. 没有的删掉，并在 Skill 注意事项里写"禁止调用XX，该功能不存在"
  5. 对照上汽 UC 的前置条件——Skill 的前置条件和 UC 是否一致？不一致的改
  6. 对照上汽 UC 的退出条件——补上退出方式

第三步：重写后发给明骏

明骏收到后做两件事：① 配置到动态知识检索模块（绑定场景 ID）② 配几条动态示例（few-shot）

第四步：上分支测试

6/4 已提测到分支（非主线），罗丽验证 + 产品上车体验。小明 6/5 下午来现场走查。

ID_004 / ID_005 → 极速调温（合并后）

场景ID: ID_004 (降温) / ID_005 (升温) → 合并为"极速调温"
场景名: 极速调温
触发词:
  降温: 好热/暴晒/快速降温/晒太久了散散热/闷死了/车里像蒸笼
  升温: 好冷/冻僵/急需升温/赶紧暖和一下/太冷了/方向盘冰凉
前置条件: 无（行驶中也可触发）
降温执行项:
  - vehicle_basic_control: 空调开关 ON
  - vehicle_basic_control: 空调温度 主驾LO/副驾LO
    （如有用户记忆偏好温度，用偏好温度）
  - vehicle_basic_control: 空调风量 11（最大）
  - vehicle_basic_control: 空调AC 开启
  - vehicle_basic_control: 空调循环模式 外循环
  - vehicle_basic_control: 主驾电动出风口 自动风向
  - vehicle_basic_control: 主驾车窗 打开（前1分钟通风）
  - vehicle_basic_control: 右后车窗 打开（对角线通风）
  （延时60秒后）
  - vehicle_basic_control: 主驾车窗 关闭
  - vehicle_basic_control: 右后车窗 关闭
  （车内温度达到设定值21°C后）
  - vehicle_basic_control: 空调AUTO 打开
  - vehicle_basic_control: 空调温度 主驾22°C/副驾22°C
升温执行项:
  - vehicle_basic_control: 空调开关 ON
  - vehicle_basic_control: 空调温度 主驾HI/副驾HI
  - vehicle_basic_control: 空调风量 11
  - vehicle_basic_control: 空调AC 开启
  - vehicle_basic_control: 空调循环模式 外循环
  - vehicle_basic_control: 座椅加热 主驾高/副驾高
  - vehicle_basic_control: 后排座椅加热 高（如有配置）
  - vehicle_basic_control: 方向盘加热 打开（如有配置）
  - vehicle_basic_control: 空调除霜 前除霜开/后除霜开/后视镜加热开
  （延时60秒后）
  - vehicle_basic_control: 空调除霜 关闭
  （再延时60秒后）
  - vehicle_basic_control: 空调温度 主驾22°C/副驾22°C
可编辑项: 温度(LO~16~28~HI) / 风量(1~11) / 循环模式 / 座椅加热档位
退出:
  手动: 语音"退出" / 点击场景列表 / Personal卡片
  自动: 车内温度达到设定值（默认21°C，可标定）
注意事项:
  - 禁止调用"极速降温模式"或"极速升温模式"等一级指令，必须拆成原子操作
  - 温度值必须在 LO~HI 范围内，不要编造不存在的温度值
  - 座椅加热和方向盘加热要先判断车辆是否有该配置（通过env_info）
  - 如果用户说"我好热"但env_info显示车内温度已经很低，要提醒用户而不是继续降温

ID_002 / ID_003 / ID_007 → 观影模式（合并后）

场景ID: ID_002(后排) / ID_003(前排) / ID_007(5D) → 合并为"观影模式"
场景名: 观影模式
触发词:
  后排: 后排想看片/给后排放点综艺/后排吸顶屏播放
  前排: 我想看个电影/帮我找个剧看/播一下XX剧
  5D: 开启5D影院/沉浸式观影/座椅震动+氛围灯
前置条件:
  后排观影: 电源≥20% · 能源管理等级>0（无档位限制，行驶中后排也能看）
  前排观影: 必须P档!!! 行驶中前排禁止观影!!!（安全红线）
  5D观影: 同前排（需要P档） + 需要腾讯视频有5D片源
位置判定逻辑:
  - speaker_position = 二排左/二排右 → 后排观影
  - speaker_position = 主驾/副驾 → 前排观影
  - query明确提"后排/吸顶屏/给后排放" → 无论说话人在哪都走后排
  - query明确提"5D/沉浸/影院模式" → 走5D
后排观影执行项:
  - vehicle_basic_control: 后排屏 展开
  - vehicle_basic_control: 座椅位置 移动到选择的位置（支持场景内记忆）
  - search_and_control_short_video 或相关工具: 打开腾讯视频
  - vehicle_basic_control: 天窗遮阳帘 关闭
  - vehicle_basic_control: 阅读灯 关闭
  - ambient_light_control: 设置为影院氛围
  - vehicle_basic_control: 场景音效 力影院模式
  - vehicle_basic_control: 最佳听音位 设为后排
前排观影执行项:
  （与后排类似，但不需要后排屏展开，且必须先检查P档）
  - 首先检查env_info中的档位，非P档直接拒绝并TTS提示
  - vehicle_basic_control: 主驾座椅 调整到舒适位置
  - 其余同后排（腾讯视频/遮阳帘/灯光/音效）
5D观影特殊说明:
  - Planner只负责: 调起腾讯视频 + 播放5D片源
  - 3分05秒座椅震动/氛围灯变色等时间点联动由腾讯视频协议控制，不是Planner做的
可编辑项: 关联座椅按摩模式/强度 / 关联空调A/C / 关联风挡灯 / 关联车外灯
退出:
  手动: 触控 / 语音
  自动: 档位变化 / SOC≤15%
注意事项:
  - 前排观影必须P档，这是安全红线，Planner必须检查env_info的档位信息
  - 禁止调用"观影模式"一级指令，方向靠没有这个指令
  - 禁止在行驶中给主驾播放视频（后排不受此限制）

ID_009 → 车内小憩

场景ID: ID_009
场景名: 车内小憩
触发词: 休息/歇会/打盹/午休/眯一会/睡一会/叫醒我/累了缓一缓/闭目养神
前置条件:
  必须P档!!（行驶中禁止进入小憩，安全红线）
  电量≥20% · 高压正常
Planner应做的交互:
  1. 先问用户"你想休息多久？"（用于设定闹钟）
  2. 再问用户"你坐在哪个位置想休息？"（用于调对应座椅）
  ← 6/3 小明明确说"如果它再能问一下是哪个位置要小憩，更好"
执行项:
  - vehicle_basic_control: 对应座椅 调至零重力位
  - vehicle_basic_control: 遮阳帘 关闭
  - vehicle_basic_control: 车窗 全部关闭
  - vehicle_basic_control: 阅读灯 关闭
  - ambient_light_control: 氛围灯 调暗/关闭
  - search_and_control_music: 播放白噪音/轻音乐（可问用户偏好）
  - vehicle_basic_control: 空调 维持舒适温度（参考env_info当前温度+用户偏好）
  - goal_list_update: 设定闹钟（用户指定时长）
退出/收尾:
  - 闹钟到时 → TTS温柔叫醒 → 停止BGM → 座椅无法自动回滚（已确认D7）
  - 语音说"退出" → 停止BGM → 恢复座椅到记忆位置（依赖记忆工具）
注意事项:
  - 非P档绝对禁止进入小憩模式（安全红线）
  - 冥想(ID_008)和小憩(ID_009)是互斥的：用户清醒想放松→冥想；用户想睡觉→小憩
  - 结束后BGM必须停止（6/3小明反馈冥想模式BGM不停的问题）
  - 无回滚机制，退出后仅恢复座椅记忆位置

ID_008 → 冥想空间

场景ID: ID_008
场景名: 冥想空间
触发词: 冥想/正念/压力大/情绪舒缓/深度专注/静下来/让自己平静/减压放松模式/降噪/隔绝噪音
前置条件: P档（驻车时使用）
与小憩的区别: 用户是清醒的，想安静/减压/专注，不是要睡觉
执行项:
  - vehicle_basic_control: 座椅调至舒适位（不是零重力，用户是坐着的）
  - vehicle_basic_control: 车窗关闭 / 遮阳帘关闭
  - ambient_light_control: 氛围灯调至冥想模式（柔和暖光）
  - search_and_control_music: 播放冥想引导音频/自然白噪音
  - vehicle_basic_control: 空调维持舒适温度
收尾逻辑（重要！6/3小明反馈的问题）:
  - 用户说"结束" → 必须：① 停止BGM ② 座椅恢复 ③ TTS语音收尾
  - 6/3走查发现：结束后BGM还在播、座椅没恢复、没有收尾语音 → 必须修复
注意事项:
  - "打开减压放松模式"走ID_008冥想，即使用户说"我有点累了"也走冥想（场景命名词优先）
  - 单独说"累了"无场景命名词 → 走ID_009小憩

ID_012 → 快速通风

场景ID: ID_012
场景名: 快速通风
触发词: 抽烟/通风/异味/闷/换气/车里臭/空气浑浊/车内湿气重/好闷打开窗透透气
前置条件: 无
执行项:

  - vehicle_basic_control: 右后车窗 打开（对角线通风）

  - 根据实际情况调整通风时长（5分钟/10分钟）
  ← 6/3小明原话："Planner可能可以有机会能做成，而且通风时长也可以按实际情况调整"
注意事项:
  - 通风模式需要端状态支持（温度/PM2.5/天气），目前部分端状态信号未接入
  - "又热又臭" → 同时命中ID_004(降温)+ID_012(通风)，两份Skill都要注入

W9：安全红线兜底方案 P0

① 这个问题是怎么来的？

上汽原来的预制场景在代码层面硬写了前置条件校验——比如洗车模式进入前检查"P档✓ 电量≥18%✓ 充电枪未插✓"，不满足直接拒绝，不存在"概率性"。

切换到 Planner 编排后，判断逻辑从"代码硬写"变成了"LLM 推理"——我在 SP 里写了"非P档禁止调座椅"，Planner 大概率会遵守，但不能100% 保证（LLM 有幻觉风险）。

6/4 上午小明举了致命场景：

行驶中 Planner 误调座椅放平 → 驾驶员失去对车辆的控制 → 车毁人亡。这不是体验问题，是安全问题。

层 1：Planner SP 约束（我做 → 给明骏）

Planner SP 注意事项新增:
安全红线（绝对禁止，任何情况下都不得违反）:
  1. 非P档状态下，禁止调整任何座椅位置（前排/后排）
  2. 非P档状态下，禁止放倒任何座椅靠背

W7：防晕车模式（新增场景）P1 · 测试中

① 这个问题是怎么来的？

6/4 上午小明提出，防晕车模式是一个特殊场景——"一半确定+一半发挥"。

小明（6/4会上）："防晕车模式它涉及到动力形式（雪地/舒缓/经济）、能量回收的等级，这一部分是确定的。但还涉及空调、开窗、香氛、音乐，这些又希望 Planner 自由发挥。"

小明问"一半一半 Planner 能不能做？"泽民回答：

泽民："一半一半我感觉也不太好做。链路很麻烦，就是你靠车端执行一半，然后万一链路很麻烦。"

最终结论："你可以测一下，都可以测一下。" + 杨麟 6/4 纪要第 4 条："今天补充晕车舒缓模式知识。"

② 怎么做？

  1. 杨麟 补充晕车模式的具体知识（确定部分是什么、发挥部分是什么）
  2. 我写 Skill 先测： 
    确定部分（必须严格执行，不发挥）:
  - 动力模式: 设为舒缓模式
  - 能量回收等级: 设为最低
发挥部分（结合env_info和用户偏好编排）:
  - 空调: 根据车内温度和用户偏好调节
  - 车窗: 适当开窗通风（但不要太大风噪）
  - 香氛: 如有配置，开启清新香氛
  - 音乐: 播放舒缓音乐或关闭音乐
  3. 如果 Planner 做不好（确定部分也发挥了），就走分子级预置应用（需要研发搭链路，工作量大）
