# Claude Code 里的 HTML：不只是网页，而是更好的工作文档

- **原文链接**：https://x.com/trq212/status/2052809885763747935
- **作者**：Thariq，Anthropic Claude Code 团队成员
- **剪存时间**：2026-05-09 14:57:34

---

Markdown 已经成为 agent 与我们沟通时最常用的文件格式。它简单、可移植，具备一些富文本能力，也很容易由你来编辑。Claude 甚至已经非常擅长在 Markdown 文件里用 ASCII 画图。

但随着 agent 变得越来越强，我越来越觉得 Markdown 成了一种受限的格式。超过一百行的 Markdown 文件，我读起来就会觉得困难。我想要更丰富的可视化、更丰富的色彩和图表，也希望这些内容能更容易分享。

我自己也越来越少亲手编辑这些文件，而是把它们当作规格说明、参考文件、头脑风暴产物等等。即使我确实要修改，通常也是提示 Claude 帮我改，这也削弱了 Markdown 最大的优势之一。

我已经开始更偏好让 Claude 输出 HTML，而不是 Markdown，并且也越来越常看到 Claude Code 团队里的其他人这么做。下面就是原因。

## 为什么是 HTML？

### 1. 信息密度

相比 Markdown，HTML 可以承载丰富得多的信息。它当然可以表达标题、格式这些简单文档结构，但它也可以表达各种其他信息，例如：

- 用表格表达表格数据
- 用 CSS 表达设计数据
- 用 SVG 表达插图
- 用 script 标签表达代码片段
- 用 HTML 元素配合 JavaScript 和 CSS 表达交互
- 用 SVG 和 HTML 表达工作流
- 用绝对定位和 canvas 表达空间数据
- 用 image 标签表达图片

我甚至会说，几乎不存在某类 Claude 能读懂、但无法用 HTML 相当高效地表达的信息。这让 HTML 成为一种非常高效的媒介：模型可以用它向你传达深入信息，你也可以用它来审阅。

### 2. 视觉清晰度与阅读便利性

随着 Claude 能完成越来越复杂的工作，它也会写出越来越大的规格说明和计划。实践中我发现，超过 100 行的 Markdown 文件，我往往就不会真的读完；我当然也很难让组织里的其他人去读它。

但 HTML 文档读起来容易得多。Claude 可以用视觉化的方式组织结构，让它更适合通过标签页、插图、链接等方式导航。它甚至可以做成移动端响应式，让你在不同设备形态下以不同方式阅读。

### 3. 易于分享

Markdown 文件其实相当难分享，因为大多数浏览器无法原生很好地渲染它们。你通常不得不把它们作为附件放进邮件或消息里。

而 HTML 只要上传文件即可，例如上传到 S3，就能很方便地分享链接。同事可以在任何地方打开它，也可以很容易地引用它。

如果你的规格说明、报告或 PR 说明是 HTML 格式，那么别人真正去读的概率会高得多。

### 4. 双向交互

HTML 可以让你与文档交互。例如，你可能希望让它加一些滑块或旋钮，用来调整设计；或者让你调试算法中的不同选项，看看会发生什么。你也可以让它把这些改动复制成一段 prompt，方便粘回 Claude Code。

### 5. 数据摄取

为什么用 Claude Code 来生成 HTML 文件，而不是用 ClaudeAI 或 Claude Design？其中一个最大原因是 Claude Code 能摄取的上下文非常多。

例如，在写这篇文章时，我让 Claude Code 读取我的代码文件夹，找出我生成过的所有 HTML 文件，对它们分组和分类，然后做一个 HTML 文件，用图表展示每种类型。你在这篇文章里看到的图表，就是直接由这个过程产生的。

除了文件系统，Claude Code 还可以通过 MCP 获取更多上下文，比如 Slack、Linear 等；也可以读取你的浏览器上下文，比如 Claude in Chrome；还可以读取你的 git 历史等等。

### 6. 它很有乐趣

用 Claude 制作 HTML 文档就是更有意思，也让我在创作过程中更有参与感和投入感。单凭这一点就足够了。

## 如何开始

我有点担心，大家读完这篇文章后会把它做成一个 /html skill 之类的东西。这样或许有些价值，但我想强调的是，你其实不需要做很多准备就能让 Claude 做这件事。你只要直接让它「做一个 HTML 文件」或「做一个 HTML artifact」就可以。

关键在于你要知道这个 artifact 应该做什么，以及你可能如何使用它。随着时间推移，你也许会做一个 skill，但现在我建议先从零开始提示，摸索它在不同场景里的用法。

## 六大使用场景

### 场景一：规格说明、规划与探索

HTML 是一个丰富的画布，适合 Claude 深入一个问题。当我开始处理一个问题时，相比一个简单的 Markdown 计划，我会预期产出一组互相关联的 HTML 文件。例如，我可能先让 Claude Code 进行头脑风暴，并针对不同选项创建一些探索稿。然后我会让它进一步展开其中一个方向，也许做一些 mockup 或代码片段。最后，当我觉得方向不错时，我会让它写一份实现计划。等我对计划满意后，我会开一个新会话，把所有这些文件都传进去，让它开始实现。

在验证时，我也会让验证 agent 读取这些文件，这样它就能拥有更完整的上下文，知道需要验证什么。

**示例 prompt**：

> 我不确定 onboarding 页面该走哪个方向。生成 6 种明显不同的方案，布局、语气和信息密度都要有差异，并把它们排成一个 HTML 文件里的网格，方便我并排比较。为每个方案标注它所做的取舍。

> 创建一份完整的 HTML 实现计划，务必加入一些 mockup、展示数据流，并添加我可能需要审阅的重要代码片段。让它易读、容易消化。

**适用场景**：
- 探索某个代码功能的不同实现方式
- 探索多种视觉设计方向

### 场景二：代码审查与理解

代码放在 Markdown 文件里可能很难读。但用 HTML，我们可以渲染 diff、注释、流程图、模块关系等等。你可以用它理解 agent 写出的代码，做代码审查，或者向审查你代码的人解释一个 PR。我发现这通常比默认的 GitHub diff 视图更好用，现在我会给自己每个 PR 都附上一份 HTML 代码解释器。

**示例 prompt**：

> 帮我审查这个 PR，创建一个描述它的 HTML artifact。我不太熟悉其中的 streaming/backpressure 逻辑，所以重点关注这一块。请渲染实际 diff，并在行内边栏加注释，按严重程度给发现的问题标颜色；还可以加入任何有助于清楚解释概念的内容。

**适用场景**：
- 创建 PR
- 审查 PR
- 理解代码里的某个主题

### 场景三：设计与原型

Claude Design 基于 HTML，是因为 HTML 在设计表达上非常强大，即使你的最终交付界面并不是 HTML。Claude 可以先用 HTML 勾勒设计，然后再用你选择的语言来实现，比如 React、Swift 等。

你也可以用它来原型化交互，比如动画、动作等等。可以考虑让 Claude 做一些滑块、旋钮等控件，帮你精确调出想要的效果。

**示例 prompt**：

> 我想原型化一个新的结账按钮，点击时它会播放一个动画，然后快速变成紫色。创建一个 HTML 文件，里面放几个滑块和选项，让我尝试这个动画的不同参数；再加一个复制按钮，能复制调好的参数。

**适用场景**：
- 创建设计系统 artifact
- 调整组件
- 可视化组件库
- 原型化有趣的动画

### 场景四：报告、研究与学习

Claude Code 非常擅长综合多个数据源的信息，并把它们转换成一份易读的报告。你可以提示 Claude 搜索 Slack、代码库、git 历史、互联网等等，然后用它为自己、领导或团队生成极其易读的报告。

你可以把它组织成长篇 HTML 文档、交互式解释器，甚至幻灯片或 deck。可以让 Claude 用 SVG 做图表，帮助可视化。

> 我不理解我们的 rate limiter 到底是怎么工作的。读取相关代码，并产出一个单页 HTML 解释器：包含 token-bucket 流程图、3 到 4 段关键代码片段及注释，并在底部加一个 "gotchas" 部分。请针对只读一遍的人进行优化。

**适用场景**：
- 总结某个功能的工作方式
- 解释一个概念
- 给老板的周状态报告
- 给领导的事故报告
- SVG 插图、流程图、技术图表

### 场景五：自定义编辑界面

有时候，只靠文本框很难描述你想要什么。这种情况下，我会让 Claude 为我正在处理的具体事项做一个一次性的编辑器。它不是产品，也不是可复用工具，而是一个单独的 HTML 文件，专门服务于这一份数据。

诀窍是最后一定要有导出能力：一个「复制为 JSON」或「复制为 prompt」的按钮，把我在 UI 里做的操作转换回某种可以粘贴到 Claude Code 的内容。

**示例 prompt**：

> 我需要重新排列这 30 个 Linear ticket 的优先级。给我做一个 HTML 文件，每个 ticket 都是一张可拖拽卡片，可以在 Now / Next / Later / Cut 四列之间移动。先按你的最佳判断预排序。加一个「复制为 markdown」按钮，导出最终排序，并为每个 bucket 附上一句理由。

> 这是我们的 feature flag 配置。为它做一个基于表单的编辑器，按领域分组展示 flag，显示它们之间的依赖关系；如果我开启了某个前置条件关闭的 flag，要给出警告。加一个「复制 diff」按钮，只输出变更过的 key。

> 我正在调这个 system prompt。做一个左右并排的编辑器：左边是可编辑 prompt，并高亮变量槽；右边有三个示例输入，会实时重新渲染填充后的模板。加一个字符数/token 数计数器和复制按钮。

**适用场景**：
- 重新排序、分诊、分桶（ticket、测试用例、反馈）
- 编辑结构化配置（feature flag、环境变量、带约束的 JSON/YAML）
- 调整 prompt、模板或文案，并实时预览
- 策展数据集、批准或拒绝行、标注样本、导出选择结果
- 注释文档、转录稿或 diff，并导出注释
- 选择那些很难用文本表达的值：颜色、缓动曲线、裁剪区域、cron 调度、正则表达式

## 常见问题

**Q: 这不是更低效地消耗 token 吗？**

虽然 Markdown 往往使用更少 token，但我发现 HTML 增加的表达力，以及我更有可能真正去读它这一点，会让整体输出更好。随着 Opus 4.7 拥有 1MM 上下文窗口，增加的 token 用量在上下文窗口里其实不太明显。

**Q: 现在什么时候还用 Markdown？**

老实说，我几乎已经完全不用 Markdown 做大多数事情了，不过我可能是比较偏 HTML 最大化主义的一边。

**Q: 我怎么查看 HTML 文件？**

我通常直接在本地浏览器里打开它，你也可以让 Claude 帮你打开；如果你想要可分享链接，就上传到 S3。

**Q: 这不会比生成 Markdown 花更久吗？**

确实会更久！HTML 可能比 Markdown 多花 2 到 4 倍时间，但我发现结果值得。

**Q: 版本控制怎么办？**

坦白说，这是 HTML 最大的缺点之一。相比 Markdown，HTML diff 很嘈杂，也很难审查。

**Q: 怎么让 Claude 匹配我的品味，不要做得很丑？**

前端设计插件能帮助 Claude 做出不错的 HTML 文件。但如果要匹配你们公司的风格，你可以让 Claude 指向你的代码库，创建一个单独的设计系统 HTML 文件。之后你就可以把这个设计系统文件作为其他 HTML 文件的参考。

## 结语

以上所有内容其实都是在说，我使用 HTML 的真正原因，是它让我感觉自己更深地处在与 Claude 协作的回路中。我曾经开始担心，因为我已经不再深入阅读计划，我是不是只能放手让 Claude 自己做选择。

但我很高兴地说，使用 HTML 之后，我反而比以往任何时候都更在回路中。希望你也会如此。
