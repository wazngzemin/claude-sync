把 Agent 从"富士康工人"改造成"同事"

大多数人把 LLM 当成昂贵又脆弱的零件——写规则、裁上下文、加围栏、省 token。结果你以为模型不行，其实是你的 harness 不让它行。
"很多人把 agent 放进一座富士康工厂里管理，可它真正想待的是 Esalen。"

Pedro Franceschi（Brex 联合创始人 / CEO）的解法不是换更大的模型，而是一套先重写自己、再重写组织的方法：把 token 打到墙、亲手撞出模型真实边界，再围绕这条边界重排分工、重写成本口径、重织上下文。

这个工具箱给你 6 个可运行模块

每个模块都能直接产出可用的表 / 分数 / 诊断结论 / 可复制 Prompt，并映射到你的 Planner / SP / Agent 工作。

模板 1 · Free the Claw 约束审查

你是 SP 约束审查官。下面是一段 Planner System Prompt 的约束/规则/上下文裁剪策略。 对每一条，输出： 1. 目的判定：【帮模型聚焦】还是【压制推理】 2. 安全判定：是否属于【安全/合规底线】（车控、隐私、兜底） 3. 建议：保留 / 放松 / 删除（安全底线一律保留） 4. 验证方案：用哪个真实任务 A/B 测试这条约束的去留 约束清单： {{粘贴 SP 约束}}

模板 2 · 撞墙日记（写死规则前先跑）

需求：{{描述一个你觉得模型做不到的需求}} 任务：不要先写规则。假设我把它当真实 query 灌进 Planner 完整跑一遍。 请预测它最可能在哪一步崩，并按四级分诊归类： - 链路问题（走错模块） - 工具描述问题（SP 工具定义不清） - 上下文问题（缺记忆/车况/情景） - 模型能力问题（信息全但推理错） 只有归类为"模型能力问题"时才建议加最小约束；其余三类给出非规则的修复方向。

模板 3 · Token ROI 拷问

下面是 SP 的一个成本块（约 {{N}} tokens）： {{粘贴成本块}} 请回答： 1. 删掉它，哪类 badcase 会变差？给具体场景。 2. 它是在"改变决策"还是"陪跑"？ 3. 它属于"应固化进缓存前缀的固定段"还是"导致缓存断裂的动态部分"？ 4. 去留建议 + 一句话理由。

一句话收束

"只要上下文和内存还有边界，工作就不会凭空消失；你要做的，是把模型没有的信号灌进去。"

你作为 Planner 的"首席 AI 官"，不可替代的价值就是持续把豆包 Seed 看不见的驾驶现场信号工程化地灌进去，并用驾驶场景的世界观给它纠偏。
