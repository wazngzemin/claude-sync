一、架构分层

六层结构：从车内现实世界到 Planner 执行闭环

产品解释

这套系统不是“语音识别后调接口”，而是先判断车内现实、用户语境和任务状态，再决定该说、该做、该问、该等。

关键变化

从只看 query 的意图分类，升级成基于 Context、Knowledge、Goal 的多源推理。这样才能处理“我好热”“周杰伦”“好暗”这类依赖场景的表达。

文档依据

来自《豆包incar1.0-长期架构梳理》《对话推理引擎(Director) PRD》《公共Context管理》《动态知识检索&使用》《基于COT的新快慢思考架构实验》。

二、Director / Planner 是什么

它的核心输出不是“意图标签”，而是可执行计划

输入
  • User query：用户直接说的话
  • Advisor input：主动服务建议
  • Tool feedback：工具返回结果
  • Trigger：定时/条件/感知触发
推理
  • 是否需要和用户交互
  • 是否需要调用工具
  • 是否需要创建或更新目标
  • 工具返回后是否继续二次推理
输出
  • talk_or_not：是否说话
  • talk_content：说什么
  • action_list：执行序列
  • goal_list_update：目标状态变化
产品经理要抓住这一点：Planner 的产物是“下一步怎么推进用户目标”，不是一个静态分类结果。
三、Goal List 与 Advisor

长任务不是一直压在 Planner 身上，而是交给目标体系持续跟进

为什么要拆出 Advisor

文档明确说系统对 Planner 依赖过重。Planner 既要快，又要保证拆解质量、状态管理、打断恢复和目标闭环，压力过大。

新的职责分工

Planner 重实时串联：处理用户、Advisor、工具、Agent。Advisor 重目标跟进：关注目标是否达成，合适时给 Planner 建议。

产品例子

“接下来行程持续介绍沿途风景”会创建动态目标和动态 Advisor。Advisor 订阅路线/位置/时间，合适时再建议 Planner 播报。

四、任务与交互管理

车内是单线程语音管道，所以必须管理打断、排队和合并

用户体验原则

用户主动说话优先，车内助手不能为了主动服务随意打断人。

工程策略

首 token 未出时更适合合并重推理；已经播报时更像“打断当前输出，处理新输入”。

HMI 约束

小卡、大卡、通知卡按任务类型使用，不是所有任务都要语音播报。

五、动态知识与示例

不要把所有知识塞进 SP，而是按场景加载

动态知识

根据 query、近期对话、goal list、车辆状态和已加载知识，决定本轮该新增或删除哪些知识 ID。

动态示例

静态示例教通用能力，动态示例解决低频、明确、难抽象的单点 bug。它们是 Planner 的“按需经验包”。

性能意义

SP 优化文档提到参考示例 token 很大，拆分静态/动态加载可以降低噪音和时延。

用户说一句模糊话，比如“好热” 需要哪些状态才能判断？ Context + Planner 不能只做意图分类，要看端状态、位置、偏好和当前任务。
主动服务建议 现在适不适合打扰用户？ Advisor + 任务管理 Advisor 负责建议，Planner 负责最终交互仲裁。
长任务或持续任务 是不是需要后续跟进？ Goal List + 动态 Advisor 不能当作一次性 react，要创建目标并持续更新进展。
低频工具调用 badcase 能否抽象成通用规则？ 动态示例 / 静态示例 能抽象放静态，不能抽象且明确的用动态示例。
工具返回结果 是否已经满足用户目标？ Tool feedback + 二次推理 工具结果不是终点，Planner 还要决定继续问、继续查、执行还是总结。
