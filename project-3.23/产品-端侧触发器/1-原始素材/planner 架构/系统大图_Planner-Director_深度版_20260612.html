A1空间关系理解错误产品(你)出文案 · 算法收顽固
根因（代入真实SP）：数值依据《AI汽车能力边界》里其实都有——"车窗 0~100% 精度5%"、"座椅靠背正常位置 20~30%"、"遮阳帘 0%全关/100%全开"。但 vehicle_basic_control 现有 tips 只写了"可结合说话人位置推理…脚冷用热风吹脚，手冷用方向盘加热"，没把这些静态数值翻译成"动作语义"：满开时"调高"该往哪、"靠背抬起来"该回到 20~30% 而非最前 0%。模型有数值、没动作规则（机制④）。
思路：让 tips 显式指向能力边界并补动作语义（不是新造常识，是把已有数值连成规则）；唯独"主驾右=副驾"能力边界里没有 → 干预(L1)+训练(L6)。
  1. L3给 vehicle_basic_control.tips 追加动作语义并援引能力边界原文（见⑨-1/⑭）：「车窗已100%(能力边界:满开)时'调高'=调低开度；'靠背抬起来/我要起来'→回到能力边界定义的正常坐姿20~30%，不是最前0%；遮阳帘已开时'向前'=减小开度；'调高X%'是相对量裁剪到[0,100]」。
  2. L1"主驾右=副驾、副驾左=主驾"能力边界里没有方位换算 → 先写干预规则(见⑨-4)兜高频必现项。
  3. L6V-13/15/16 方位 case 整理成训练样本（输入含 speaker_position，输出正确部位），交算法纳入下轮 SFT。
  4. 验收构造 10 条：车窗满开"调高"、遮阳帘"向前"、"靠背抬起来"、副驾说"左边"——核对是否落在能力边界定义的合理区间。
✅ 验收标准：援引能力边界后 L3 可补类 ≥ 9/10（V-19 应回到 20~30% 而非 0%）；顽固方位干预后必现项消除，训练后裸模型 ≥ 7/10。

⑮被忽略的关键一环：下游 SFT / FC 链路 + 怎么训 SFT

怎么看：Planner 输出改写后，不是直接执行，中间还要过一道 SFT 路由模型 → FC 匹配 → 意图二次分类。C 类 bug 就发生在这一段（Planner 改写合理，但 SFT/意图接不住）。这也是"以 Planner 为中心改造下游"的落点——训 SFT 去适配 Planner 的输出习惯。

SFT 到底是什么（给初级 PM）

SFT = Supervised Fine-Tuning（监督微调）。用大量"输入→正确输出"的标注样本去训练一个模型，把某种固定的输入输出习惯"教"给它。在本链路里，下游有一个 SFT 训出来的"分流模型"，负责把 Planner 改写后的 query 路由到正确的工具/意图。

关键点：它现在主要是按"人类怎么说"训的（接人话），而 Planner 的改写措辞和人不完全一样 → 接不住 → 产生 C 类 bug。解法就是用 Planner 的改写样本重训它。

SFT vs FC vs 意图 三者区别

SFT 模型下游的"分流/承接模型"，决定 query 走哪条路、能不能接住 Planner 的表达。可训练。
FCfunction call，最终填工具名+参数的结构化调用格式。"FC 不对"= 调用格式/参数错（转 FC 团队，属 D 类）。
意图弋途负责的二次分类器，把 query 再细分到具体函数（算路 vs 开启导航）。
把 SFT 接回解法体系：它就是第 ⑦/⑧/⑩ 节里 L5（下游/弋途协商）+ L6（训练） 的具体抓手——C 类（N-04/05 开始导航、车控接人改成接 Planner）的根治路径。你（产品）的活是① 出 Planner 真实改写样本 ② 提需求给弋途/算法；训练由下游/算法做。短期先用 ⑭ 的干预 + navi notice 兜，长期靠 SFT 收敛。
