1.2 为什么是三层漏斗

车载场景对时延极敏感——"打开车窗"期望 150ms 响应，但"帮我规划周末自驾游"用户能接受等几秒。一个大模型处理所有请求，简单的也要等 500ms+，不可接受；只用规则引擎，复杂的处理不了。所以系统被设计成三层漏斗，越简单的越早被拦住。

设计三原则（原文）：① 层层过滤——能简单搞定的不漏后面；② 分阶段注入知识——从固定规则到动态检索；③ 数据回流——Planner 处理的难 case 回流训练前面的层，让漏斗自己变聪明。

原文对这个设计的评价："这是一个过渡方案。如果后面模型能力提升起来了，把漏斗的下层提到前面去就行了——工作量非常小，是一个可以持续演进的架构。"

1.3 三层各自是什么

第一层：句法 RAG

本质：规则匹配 + 工程直接下发，不涉及大模型。处理"明确的单轮 query"。

输入：单轮 query 文本。

输出（原文 JSON 格式）：

{"type":"directExecute", "reject_ornot":"react/reject", "talk_content":"", "task_list":[...]}

type="directExecute" 表示不需要模型，直接执行。reject_ornot="reject" 表示拒识（如"妈妈帮我拿水杯"→reject）。时延：毫秒级。

第二层：情景 RAG + 仲裁模型

情景 RAG 不直接给答案，而是给仲裁模型"补课"——把场景知识以 few-shot 注入。输出格式：

{"type":"modelExecute", "knowledge":"当空调关闭时，'我好热'通常意味着需要开空调...", "agentLimit":[...]}

type="modelExecute" 表示需要模型处理。knowledge 是 few-shot 示例，agentLimit 是能力边界约束。注意：情景 RAG 只注入知识，不给结论。

仲裁模型（Seed1.6 flash SFT）接收注入了知识的 query，判断简单/复杂。为什么用轻模型？因为只做二分类，不需要深度推理，大模型做这事太浪费时延。

判定规则："未拒识 + 仲裁为车控 + 命中6大意图"三个同时满足 → 简单快车道（FC.SFT 300ms → 端控 150ms）；任何一个不满足 → 复杂 → Director（思考 500ms + 回答 500ms）。设计逻辑：宁可让复杂的走慢路，不可让简单的走错路。

仲裁误判三大根因（原文）：① 输入太单一——同一句"我好热"因车况不同结论完全不同，如果情景 RAG 没注入正确的车况 knowledge 仲裁就判错；② 靠造数据学习，case 造不完——"仍然是 NLU 的人肉解 bug 方法"；③ 意图处理信息量太少——车控有上千函数，粗分类后从千函数里选效果很差。原文类比："人说到热，就只在开窗、空调、加热几个选项里想，不会想到车控这个大类然后在1千个范围内搜索。"

第三层：Director / Planner

漏斗最底层，云端大模型深度推理。详见第二章。它有双重身份：兜底（前面都接不住的它接）+ 云端 teacher（它处理完的数据回流训练前面的层）。

3.1 ReAct 流程图

3.2 什么是 ReAct

ReAct = Reason + Act（推理+行动）的循环。传统的 LLM 调用是"一次性想完所有事然后输出"，ReAct 不同——它是想一步→做一步→看结果→再想下一步。为什么车载场景要用 ReAct？因为工具的返回是不确定的——导航可能找不到地址、记忆可能没有、联网可能超时。"一次性想完再做"在不确定环境下容易出错；"想一步做一步"每步都基于真实反馈，稳定性高得多。

3.3 ReAct 怎么运行（逐步）

Step ①：第一次推理。Director 收到三种输入之一（user_query / advisor / tool_feedback），结合 SP（13块系统提示词）做推理。它做的第一件事是先快速回一句话（首 token 在 500ms 内出来），让用户感到"它听到了"——这是体验优化，哪怕后面还要花几百毫秒调工具，用户已经不焦虑了。然后输出 5 字段 JSON。

Step ②：工具执行。action_list 里指定了调哪些工具、传什么参数。工具按 FC（Function Call）方式被调用，一个 agent 只执行一个任务。如果多个工具有串行依赖（先查记忆拿地址→再导航到该地址），不能一次性全输出（注意事项第6条）——必须先调第一个、等 feedback 回来，再决定是否调第二个。但无依赖的可以并行（如调空调+放音乐）。

Step ③：二次推理（ReAct 的核心）。工具返回 tool_feedback（如"导航成功"或"地址未找到"），Director 读取 feedback 做二次推理，判断三种情况：

  • A. 任务完成 → 总结告诉用户（或不说话——车窗关了用户能感受到，不必多嘴）
  • B. 需要更多信息 → 继续调另一个工具（回到 Step①再出一个 JSON）→ 这就是 ReAct 循环
  • C. 失败 → Replan——端上回传"未成功的原始任务 + 具体失败原因"，Director 综合当前上下文重新生成计划

3.4 什么时候结束

ReAct 的循环没有固定的最大轮次上限（至少在 PRD 层面没有明确写）。它的终止条件是：

  1. 任务完成：Director 判断所有 action 都成功完成了 → 输出最终的 talk_content（或静默）→ 结束
  2. 失败且无法 Replan：工具反复调用失败 → 注意事项第8条"工具反复调用失败时，可暂时放弃" → Director 输出告知用户的话术 → 结束
  3. 用户打断：用户在 Director 推理/播报期间说了新的话 → 触发交互管理的打断逻辑（详见第八章）→ 当前 ReAct 被打断，新 query 开始新的推理
  4. Director 判断不需要行动：有时候 Director 读完 feedback 后决定"不需要再做什么了"→ action_list: [] + 可能说一句话或静默 → 结束
"去昨天游泳馆"走两轮 ReAct：
第一轮：Director 发现需要查记忆才知道"昨天游泳馆"是哪 → 输出 action_list:[{tool:"search_user_memory", params:{query:"去过的游泳馆"}}] → 同时 talk_content:"好的，我帮你查一下"（先快回一句）。
工具返回：tool_feedback:"昨天去了虹桥游泳馆"。
第二轮（feedback回灌触发二次推理）：有地址了 → 输出 action_list:[{tool:"route_planning_qa", params:{query:"导航到虹桥游泳馆"}}] → talk_or_not:"no"（不说话，直接导航）。
工具返回：tool_feedback:"导航成功"。
第三轮（再一次feedback）：导航成功了 → action_list:[] → talk_or_not:"no"（静默，用户能看到导航画面）→ 结束。
整个过程：一次调用、三轮推理、两次工具调用。

user_query（7个字段）：

{
  "query": "去昨天游泳馆",
  "speaker_name": "明骏",
  "speaker_position": "主驾驶位",
  "timestamp": "2026-04-13T14:30:05Z",        // ISO8601，用于推算时间间隔
  "assistant_emotional_state": "平静",          // 上一轮情绪，保持连贯
  "goal_list": "1.帮小明点肯德基(进行中)",       // 当前目标队列，直传不用再调工具查
  "env_info": "车速45km/h，导航未设置，位于上海中山东一路"  // 最新车况，直传
}

timestamp 让 Director 能做自我时间感知：对比当前时间戳与记忆中的"上次见面时间" → 超 30 天 → "好久不见"开场；超 5 分钟 → 主动确认上个任务是否继续。goal_list 和 env_info 随输入直传，原文说"可直接参考它们来编排行动，避免不必要的工具调用"——能从输入读的车况/目标别再调 search_vehicle_status_info 查。

advisor：{"advisor":"车内温度35度，建议开启空调强力制冷和座椅通风"}。不回应 advisor，直接面向用户做行动和表达。这不是用户说的话。

tool_feedback：{"tool_feedback":"导航成功","tool_name":"route_planning_qa"}。触发二次推理。对于简单成功（"车窗已关闭"），Director 通常不说话（talk_or_not:"no"）——用户能感受到结果。

3.6 五个输出字段

{
  "talk_or_not": "yes",                          // 可以选择不说话
  "talk_content": "[轻声地] 后排左侧还没系安全带",  // 可带情绪标签[开心地说]/[唱]
  "emoji_id": "speaking",                         // 17选1形象动画
  "emotional_state_change": "关切",               // 仅情绪变化时输出，相同则省略
  "action_list": [{"action_id":"1","params":{"query":"后排安全带状态"},"tool_name":"search_visual_info"}]
}

情绪状态机（6态）：平静(默认) / 欣喜 / 关切 / 低落 / 生气 / 好奇。转换规则：

  • 不能突变：低落→不能直接欣喜，需经平静过渡
  • 关切惯性：关切状态下即使中性任务也保持关切，不能秒变欣喜
  • 用户明确表达情绪好转 → 消退为平静
  • 被责骂 → 生气/低落，道歉后消退
  • 久别重逢 → 平静可直接进欣喜
情绪连贯示例：用户说"豆包，我被裁员了" → Director 输出 emotional_state_change:"关切"，talk_content:"[轻声地] 雨晴……这事来得太突然了"。下一轮用户说"放首安静的歌" → Director 保持关切（不输出 emotional_state_change 因为没变），talk_content:"[轻声地] 好。放一首没有歌词的"。直到用户说"好多了，谢谢" → 消退为平静。

3.7 Director 的职责边界

  • P0 目标增删：据 query 判断是否需要新建/删除目标（调 goal_list_update 工具）
  • P0 动态 advisor 增删：据 query 判断是否需要创建/删除动态 advisor
  • P1 合理性仲裁：增删时校验——感知条件能不能监测到？触发频率合不合理（"每200ms播报NBA"不合理）？动作是否矛盾（"同条件下又开又关空调"）？是否违反价值观？
核心约束：Director 明确不更新目标的"细节和进展"——这个写权限收归 advisor。Director 只管"建/删目标+快速响应"，"盯进度/改进展"交给 advisor。这是为了避免 Director 与 advisor 双写冲突——如果两边都能改进展，Director 写"宝宝在睡觉"的同时 advisor 写"宝宝醒了在哭"，谁的为准？

4.2 为什么"固定在前动态在后"（KV Cache原理）

Director 每次推理=读一份 SP+UP。SP 的 ①~⑫ 是固定的——角色定义、工具定义、示例等每次调用完全一样。因为前缀完全匹配，KV Cache（Prompt Cache）可以复用已算好的注意力向量，不需要重新计算这几万个 token——只要算 ⑬ 的动态部分和 UP 的增量，推理时间大幅缩短。

⑬ 放在最后，因为它每轮都变（车况不同、记忆不同、筛选出的知识不同）。一旦某个位置的 token 变了，后面所有的 KV Cache 都作废。所以把"变"的放最尾巴，"不变"的前缀尽可能长，缓存命中率最大化。

同一请求内多轮调用（ReAct 第2/3轮）：前序全部 token 的 KV 已算好，第2轮只算增量（几百 token 的 tool_feedback + 新的 output），显著更快。这就是为什么 ReAct 多轮推理不意味着时延翻倍——只有增量需要计算。

4.3 三套加载体系详解

A. 静态示例（常驻 SP）

是什么：10 个完整的多轮对话示例，常驻在 SP 的 ⑥ 区块里，每次推理都会读到。

为什么叫"授模型以渔"：静态示例教的是通用能力——不是教模型"遇到这个 query 调什么工具"，而是教它"遇到这一类问题怎么思考"。比如一个示例教模型"当 query 涉及导航+顺路搜时，要先导航再沿途搜"，模型学会后能泛化到所有类似的导航+顺路场景。

覆盖 10 大类场景（原文分类表）：导航与路线规划类 / 车辆座舱控制类 / 娱乐与互动类 / GUI代理生活服务类 / 用户记忆与身份管理类 / 目标任务管理类 / 通用信息搜索类 / AI图像生成类 / 车辆信息查询类 / 情感关怀与场景适配类。

格式：每个示例是一段完整的多轮对话——input1（用户说什么）→ output1（Director 输出什么 JSON）→ input2（工具返回什么）→ output2 → ……，少则 3 轮多则 12 轮。

B. 动态示例（按 query 检索注入）

是什么：存在 VikingDB 向量检索引擎里的示例库（数量远多于10个），每次请求时根据当前 query 做语义检索取 top-5，注入 SP。

为什么叫"授模型以鱼"：动态示例解决的是低频且无法归类抽象的单点 bug。比如"切换壁纸"这种指令太少见了，不值得为它写一个教通用能力的静态示例，但确实需要告诉模型"这个指令调 vehicle_basic_control"。所以动态示例只含 query + 执行答案（调什么工具），不需要完整的多轮 input/output。

格式（原文示例）：

query: 前挡都是雾，看不清了
决策分析: 首先结合端状态的车内车外温差判断是车内有雾还是车外有雾，
         其次结合端状态判断是否已经打开除霜除雾，未打开的情况下
         如果车内起雾开空调除雾模式，如果车外起雾开雨刮器。

检索机制：用 VikingDB 做向量检索。难点是多轮 query 改写——对话到第二轮时 query 可能只是"好啊"，需要结合对话历史改写成完整意图（如"好啊，就去那家日料"→改写为"导航到鮨一日本料理虹桥店"）再做检索，否则检索结果不相关。

和静态示例的区别（原文总结）：

静态示例动态示例
目的教通用能力（渔）解单点bug（鱼）
加载方式常驻 SP按 query 检索注入
格式完整多轮 input/output只有 query + 决策分析/执行答案
数量10个每轮检索5个
适用场景可被提炼共性的一组例子简单明确无法归类的单点

哪些问题不应该放动态示例（原文）：可被提炼共性的一组例子，应该放静态示例。比如"离6月18日还有几天""春节期间有什么电影""下一个是什么节气"这类都涉及时间计算→应提炼成一条静态的决策分析规则（"查询时间日期先用 timestamp，其他用 web_search"），而不是为每个 query 都加一条动态示例。

C. 动态知识（前置筛选/知识检索注入）

是什么：66 条场景知识（结构化为"6类专家 × 四段式 ID"），由一个独立的知识筛选小模型根据当前场景从中挑选几条，注入 SP 末尾的 {{动态专业知识}} 块。

5.3 Context 的三维过滤（真实的优化抓手，不是"远近区"）

7.1 条件任务完整流程图

7.2 触发器在架构中的位置

图2三层泳道里，触发器位于推理层，分离线触发器和在线触发器两个子框。它们的下游有两条路：① 触发advisor推理（唤醒动态advisor做目标review）和 ② 触发条件任务下发（条件满足后通知Director执行）。

7.2 条件触发器（operate_conditional_task / CTA）

为什么需要它：用户说"10分钟后帮我关座椅加热"——Director不可能"挂起等10分钟"。它需要一个独立的模块来持续监控条件、到时候再唤醒Director。

工作流程（完整5步时序）：

  1. Director判断需要创建条件任务 → 调operate_conditional_task以自然语言下发："十分钟后，关闭主驾座椅加热"
  2. 工具回执："条件任务已注册"（Director收到回执后继续处理别的事，不挂起等待）
  3. 工具内部拆出条件="定时器10分钟"，开始监控
  4. 10分钟后条件满足 → 工具给Director下发"[任务原文]+条件已触发"
  5. Director收到触发通知做二次推理——结合当前最新上下文决定是否仍然执行（因为10分钟后车况可能已变——用户可能已下车、座椅加热可能已被别的指令关掉了）
为什么条件满足后不直接执行而是回Director？因为10分钟后的车况可能已变。让Director拿到"条件已触发"后做一次推理、结合当前env_info决定是否仍然执行——这比盲目执行安全得多。

合法性校验（5条，都过了才注册）：能否被感知（"世界毁灭"不能）/ 触发可能性（"温度1000度"不现实）/ 价值观（"200km/h时关安全气囊"违反安全）/ 可执行性 / 条件与动作是否矛盾（"下雨时打开天窗"）。非法→不注册，转聊天回复。

7.3 离线触发器 vs 在线触发器

离线触发器：端侧运行，处理不依赖网络的触发条件（如端侧传感器信号：温度阈值、face id检测到某人上车、车速超阈值）。优势是弱网也能触发。

在线触发器：云端运行，处理需要云端能力的条件（如导航POI距离判断"距离外婆家<2km"、联网信息变化"航班延误了"）。

二期新增：端侧定时器（弱网/断网时也能触发AI PUSH，端侧倒计时展示——不再显示"30分钟后"一直不变）、导航场景感知（距离某POI<指定距离时触发事件）。

第八章 · 交互管理：单通路治理 + 6种冲突策略 + advisor排队规则

8.0 交互治理流程图：新输入到达时的决策

8.1 为什么是"单通路"

车里的语音交互是单线程、单通路管道——原文说"同一时间只能对一次请求做出回应"。不像手机App可以弹多个通知让你逐个处理，车上你只能听到一个声音、看到一个卡片。所以多个输入同时到达时，只能在输入端做治理。

8.2 四种输入 + 优先级

输入类型优先级调度策略
用户主动说的话（即时任务）P0 最高打断式
非用户主动（定时/条件任务触发）P1排队式
Tool feedback（车控/aigc/长程返回）—分流
Advisor建议—排队，同一advisor只保留最新

8.3 核心判据：首token出了没

新输入到达时Director的首token还没出来（正在推理中）→ 打断当前推理，新旧query合并后重推理。合并后话术自然串联："车里是有点热呢我来调一下，顺便帮你找今天的热点新闻。"

新输入到达时Director已经在播报（首token已出/TTS正在播）→ 分情况：

  • P0用户说话：直接打断当前播报，按新query推理。被打断的播报不恢复。
  • P1非用户（定时/条件触发）：排队，等上一条TTS播完再处理。两种做法：A等播完再推理；B预推理+结果缓存（边播边在后台算好答案，播完直接出结果省时延）。
  • Advisor建议：在"用户正在讲话"或"任务流进行中"时（图8的红线）→ 丢弃该建议（后续advisor推理后重新下发）。在"任务流结束后"（绿线）→ 允许下发。

8.4 六种冲突策略（已在播报时的新任务怎么处理）

#策略行为例子
①扔弃新任务新的到了直接丢正回用户话，advisor下发→丢弃
②排队新的等旧的播完正回话，推送主动服务→排队
③静默执行新的悄悄做不打断旧的正找景点，"打开车窗"→静默开窗不打断找景点
④整合（核心）暂停旧的，整合上下文重新规划正规划行程，"再加个点"→暂停当前规划，整合后重新规划
⑤插队暂停旧的，新的插入播完，旧的继续播新闻中，"小心右侧行人"→安全提醒插播→继续新闻
⑥停止旧任务旧的直接丢弃正说"帮你开窗"，"下个路口左转"→导航优先→丢弃开窗

决策依据是优先级矩阵——行=正在进行的旧交互，列=新来的交互，交叉格填具体用哪种策略。优先级从高到低：安全通知 > 导航通知 > 普通系统消息 > 配置类主动服务 > 条件任务 > 非配置类主动服务 > 简单query > 复杂query。

8.5 合并三特殊规则

  1. 长程任务（订座/停车/点咖啡/外卖）不参与合并——它们有自己的GUI界面，合并会乱。
  2. aigc（生图/ailight）不走Planner、无语音播报，完成后走通知卡系统通知。
  3. 同一advisor排队期间的多条建议只保留最新一条——防旧建议堆积。

8.6 任务工程管理模块：打断/恢复/挂起

打断：端上请求中断接口（携带未完成任务）→ 标记plan为interrupted + 存checkpoint（当前哪些task完成了、中断在哪一步）→ 新query进来处理。

恢复：新query处理完后，由advisor（goal reviewer）决定是否恢复——不是Director自己记住。恢复 = 带上下文（含目标"细节和进展"）重新触发任务。续播规则：已完整缓存→从断点上一个句子起始处续播；未完整生成→话术衔接后基于上下文重新生成续播。

挂起：定时/条件任务注册后进入"挂起"态，Always on模块持续监听车辆状态，条件满足时通知Planner恢复为"进行中"调Agent执行。

replan阻塞结果播报（图2中Director下方的节点）：当工具执行失败需要Replan时，如果此时正在播报别的内容，Replan的结果会被阻塞等播报完再出。这个节点管的就是"Replan产生的新结果什么时候能被播报出去"的排队。

8.7 失效兜底

  • 端侧上报超时：白名单召走→采纳端侧不等云端；端侧有结果非白名单→超时采纳端侧；端侧无结果→超时播报兜底
  • 云端工具超时（ReAct链路）：①工程判断超时，tool_feedback反馈超时 → ②告知用户后执行重试 → ③重试失败告知用户后结束任务
  • 被端侧召走的query：云端并行在跑但结果被丢弃，被合并的任务恢复排队；云端历史对话保留端侧处理信息、不保留被打断的云端推理历史

  1. 体感温度舒适性——车内温度是否在人体适宜范围（约22-26度），乘客有没有表现出热/冷的体征
  2. 空气质量舒适性——PM2.5、CO2浓度、异味（如有人抽烟）是否超标
  3. 光线舒适性——有人在阅读/找东西/化妆时光线够不够亮，有人在睡觉时光线是不是太亮
  4. 声音舒适性——有人在睡觉时音量是不是太大，有人在听歌时音量是不是太小
  5. 乘坐舒适性——座椅角度/按摩/通风/加热是否匹配乘客当前状态（如长途驾驶后需要按摩放松）
  6. 空间舒适性——人员变动时（有人上车/下车）座椅空间是否够用（如后排坐了三个人是否太挤）
  7. 其他舒适性——上述6类覆盖不了的（如冥想场景需要综合调光+音乐+空调静音）

对应知识专家 COM（Comfort/随侍管家）。PRD v1.0。关键约束：舒适advisor不直接控车——它只输出"分析+建议"给Director，Director决定要不要执行。这意味着如果Director此时正在处理更高优先级的用户query（交互管理的排队/丢弃策略），舒适advisor的建议可能被延迟执行甚至丢弃。

10.2 触发机制：事件触发 + 定时review

事件触发（P0）

事件由"触发器"模块生成（产品负责人吴仁杰）。能触发舒适advisor的事件包括但不限于：

  • 有人上车/下车——上车时需要判断当前车内环境对新来的人是否舒适（如车内38度很热）；下车时需要判断剩余乘客的空间是否需要调整
  • 温度变化——车内/车外温度发生显著变化（如出了地库进入暴晒区域温度骤升）
  • 人员状态变化——VLM观察到有人开始阅读（需要检查光线）、有人睡着了（需要检查音量和灯光）、有人擤鼻涕（可能需要调小风量避免直吹）

定时目标Review（P1）

触发条件："1分钟内无新事件触发推理 + 当前存在未完成的舒适子目标"。两个条件必须同时满足——如果没有子目标，不会做无意义的轮询。

为什么需要定时review：因为舒适调节通常不是一次就完成的。比如用户说"我好热"→舒适advisor建议"开空调20度风速7档极速制冷"→Director执行→但极速制冷开了之后不应该一直保持7档强风——车内降到舒适温度后应该自动把风速降下来。如果没有定时review，空调就会一直以极速制冷的状态运行，用户从"太热"变成"风太大太冷"。

定时review的完整时序：
T=0：事件触发——"车内38度，空调关闭"→舒适advisor建议"空调20度+风速7档极速制冷"→Director执行→同时写入舒适子目标"极速制冷中"到goal list
T=60s：1分钟过去，无新事件+子目标"极速制冷中"仍在→触发定时review
T=60s review结果：端信号显示"车内已降到26度但风速还在7档"→舒适advisor分析"温度已接近舒适但风速过大"→建议"风速降到3档，空调温度调到24度"→Director执行→更新子目标进展"温度已接近舒适，降低风速中"
T=120s：又过1分钟→再次review→"车内24度，风速3档"→分析"温度舒适，风速适中"→建议"/"（无需调整）→子目标标记完成→从goal list删除

舒适advisor有自己独立的SP（不是Director的SP），其中注入以下5类模板变量：

变量具体内容数据来源什么时候用

记忆的三个时间层级（这是舒适advisor做个性化判断的关键）：

  • 瞬时记忆（20分钟内）：刚才发生的事。如"3分钟前用户说了一句'有点冷'"——即使端信号显示温度在舒适范围，用户主观感受冷也需要响应。
  • 短期记忆（20分钟~上次夜间更新）：今天的事。如"今天早上用户把空调调到了25度"——说明用户今天偏好25度。
  • 长期记忆（最长半年）：用户的长期偏好。如"明骏总是把空调设到23度""雨晴夏天喜欢开车窗通风不开空调"。这些在每晚更新（系统离线总结当天的舒适数据）或用户主动要求时记录。

10.4 内部推理逻辑：SP内置5条knowhow（完整原文）

舒适advisor的SP中写了5条决策树式的推理规则（if-then结构），原文如下：

  1. 温度舒适（原文）："当车内温度超出人体适宜温度，思考如何快速的使车内温度达到适宜温度；当车内人员表现出温度不适感（比如说话打颤、反复擦汗），思考如何调整局部车辆控制使对应人员舒适"——注意"局部"二字：不是把全车温度改了，而是只调不舒服的那个人附近的控制（如只调主驾空调）。
  2. 空气舒适（原文）："当车内空气质量超出正常范围，思考如何快速的使车内空气质量达到适宜值"——比如有人抽烟→开窗通风+空气净化。
  3. 空间舒适（原文）："当车内出现人员变动时，判断车内空间是否满足所有人员舒适条件"——后排上了第三个人→座椅可能要调整腾出空间。
  4. 光线舒适（原文）："当车内人员出现对光线有诉求的行为时（比如阅读、找东西、化妆、睡觉等），判断当前光线是否满足舒适要求"——有人看书→检查阅读灯是否开了、亮度够不够；有人睡觉→检查灯是不是太亮要关掉。
  5. 声音舒适（原文）："当车内人员出现对声音有诉求的行为时（比如睡觉、听歌等），判断当前车内音量是否满足舒适要求"——有人睡觉→音量要调低；有人在听歌但音量太小→建议调大。

核心推理模式是一个三步链：① 感知异常（通过{{attention}}事件 + {{watcher}}视觉 + {{car_condition}}端信号，综合发现"有什么不对"）→ ② 诊断维度（判断是温度/空气/光线/声音/空间哪个维度的问题）→ ③ 生成调节方案（结合{{memory}}里的用户偏好，输出具体的调节建议）。

这里的SP还引用了一份外部knowhow文档：「AI car-舒适knowhow-baseline v0.4」，里面有更详细的规则（如"人体适宜温度范围是22-26度""风量与温差的推荐对应关系""不同活动对光照度的要求"等）。

推理模式是一个四步链条：

  1. 多因素融合——把上面10+个数据源的信息综合在一起，不是看单一因素。"周二8:30+在家附近+带书包"三个因素叠加后置信度远高于单看任何一个。
  2. 概率排序——可能有多个候选目的地。比如"周二8:30在家"可能去公司也可能送孩子上学。出行advisor要给出排序：最可能=送孩子(因为带了书包)+上班(因为工作日)。
  3. 输出预测+置信度——每个预测附带一个置信度分数。高置信（如95%确定是通勤）→直接推荐路线不问用户。低置信（如只有60%）→需要确认。
  4. 低置信时主动确认——置信度低于阈值时，出行advisor给Director的建议里会包含"建议向用户确认目的地"，Director用一句自然的话问用户："你是要去公司还是送依依上学？"而不是生硬地弹选择框。
PRD原文完整推理示例：
输入：12月3日周二8:31 / 益红和依依已上车 / 车在长江小区 / 导航未设置
推理过程（原文）："根据当前信息（12月3日周二早上8:31），益红和依依已上车，车辆位于长江小区。结合两人的日常安排：
- 依依应在8:30-8:35出发去光启小学上学，当前时间已接近最晚出发时间，且她带了书包，说明今天需要上学。
- 益红通常8:45-9:00出发去平金中心上班，当前时间稍早，但两人已上车，很可能计划先送依依上学，再去公司。
因此，本次出行的目的是：1.送依依上学 2.益红上班"
置信度：高（多因素一致指向同一结论）→ 直接推荐"先去光启小学再去平金中心"的路线
注意：这里"带了书包"是VLM（watcher）看到的视觉信息——如果VLM不工作或识别不到"书包"，这条推理就少了一个关键因素，置信度会下降，可能需要问用户确认。

11.4 输出格式与流向

出行advisor的输出是结构化字段（不是每个场景都涉及所有字段）：

字段示例值给谁什么时候有
出行目的预测"去上班/车内午休/NA/预测用户等会儿8点钟要去平金中心上班"内部（供后续场景推理用）有人上车时

11.5 真实case（完整链路）

Case 1：停车缴费（行前场景）
触发：事件"车在停车场 & 车辆起步/驶离"或"车机下电"
输入：端信号"车辆位于万达广场地下停车场，已停放3小时12分钟"
推理：停车场 + 停放时间>0 → 存在缴费需求
输出：需求分析="停在万达广场停车场已3小时，存在缴费需求" / 建议="主动询问是否帮缴，用户同意后通过GUIAgent帮缴费"

Case 2：堵车绕行（行中场景）
触发：事件"堵车"（导航监测到前方拥堵>N分钟）
输入：历史出行偏好"平时走中环内圈上班" / 实时路况"中环今天严重拥堵" / 替代路线"外环高速顺畅但过路费10元"

Case 3：充电推荐（行中场景）
触发：事件"能耗不足"（剩余电量不足以到达目的地）或"接近目的地但电量低"
输入：端信号"剩余电量15%，剩余里程50km，目的地距离80km" / 记忆"用户习惯快充，对费用不敏感"
推理：50km < 80km → 到不了 → 需要中途充电 → 检索沿途充电站 → 结合用户偏好（习惯快充）推荐
输出：建议="电量不够到目的地，推荐前方2km处XX快充站，有20-5优惠，预计充30分钟可续航200km"
扩展：如果充电站排队 → "下一出口2km处有另一个空闲站，节省40分钟等候"
Case 4：最后一公里指引（行后场景）
触发：事件"接近目的地(100m)" 或 "到达目的地 & P档"
输入：目的地"安又胖餐厅" / 搜索结果"安又胖在3楼"
输出：建议="安又胖在三楼，建议找最近的扶梯上楼，排队大概需要20分钟"
注意：这里用到了SP⑩其他知识里的导航技巧——"人下车的地方和车停的地方要分开想"——先在目的地门口放人下车，再去找停车位。

11.8 Badcase排查

  • 目的预测错了：查10+数据源哪个缺了或错了→最常见的是VLM没识别到"携带物品"（如书包），导致少了一个关键因素→或历史出行偏好数据不足（新用户/新地点没积累）

第十二章 · 内容 Advisor——"克制比推荐更重要"

12.0 内容Advisor L0-L3判定流程图

12.1 定位

"成为车内最懂场景、最懂你的首席内容建议官。不做内容搬运工，只专注于生成让你无法拒绝的'下一段时光'内容提案。"成功指标是建议采纳率，不是播放量。对应知识专家ENT。PRD V0.4。

12.2 核心设计原则

默认不工作。没有Trigger，就没有内容。保持系统资源与用户注意力的双重零占用。

12.3 触发机制（三种，含独特的"协同信号"）

  • 事件触发：场景切换（起步/停车/路况剧变）、用户行为变化（对话结束/打断）
  • 定时Review：长时间静默后的状态复查（如已静默30分钟）
  • 协同信号：情感advisor发出"适合陪伴"的信号——这是4个advisor之间唯一明确的跨advisor协同接口

12.4 核心推理：L0-L3介入等级（状态机）

触发后第一件事不是"推什么"而是"该不该推"。输出唯一的介入等级：

等级定义认知占用触发条件（信号源）
L0 隐形不推荐不打扰0%愤怒/烦躁/高压力；暴雨冰雪驾驶负荷极高；社交冲突/明确拒绝。信号源：DMS&OMS/语音/ADAS/车身域/交互历史
L1 氛围白噪声/轻音乐<5%疲惫/低落/发呆；单调且安全环境。信号源：DMS/座舱/光照/雨量
L2 碎片笑话/脱口秀/新闻30%通勤/短途(导航<30min)；正常/轻度焦虑。信号源：导航/用户画像/DMS
L3 沉浸专家解读/历史讲坛>80%长途巡航/驻车休息(NOA/充电)；高接收度/好奇。信号源：智驾域/车身域/推荐引擎
"要像人一样察言观色——车上夫妻正在吵架，即使满足了Trigger条件，也不要推送。"

12.5 8步推理Flow

  1. 静默态（默认不工作）
  2. Trigger唤醒（事件/定时/协同信号触发，但Trigger≠必须输出）
  3. Context汇聚（拉取完整状态快照）
  4. 介入判断（决策核心，输出L0-L3）
  5. 策略参数生成（仅L1+时，生成内容形态+表达风格）
  6. 分发给Director
  7. 反馈感知（持续监听显隐性反馈，负反馈→立即降级）
  8. Review与回收（内容结束后策略校准，回到Step1）

12.6 反馈闭环

满意度校准：连续跳过3次→归因分析（针对歌手 or 曲风？）→策略修正。时空/负荷校准：路况从畅通变拥堵且持续>5min→降低认知门槛（L3→L2）。每20分钟后台扫描一次。

12.7 做到什么程度

L0-L3判定逻辑和8步Flow已完整定义。内容源对接（QQ音乐/网易云/喜马拉雅/小宇宙API）是工程依赖。AIGC能力（讲解/串场词/知识胶囊生成）依赖实时生成。Benchmark方案"暂不支持"。

13.0 情感Advisor Event Prompt 4步推理流程图

13.1 定位

纯情感层察言观色。不控车、不推内容、不提醒物品——只管"用什么语气说话、要不要主动搭话、该不该关心一下"。对应知识专家EMO。PRD v1.0。有两套完整SP（Event Prompt + Review Prompt）。

13.4 Event Prompt 4步推理链

第一步：事件解析+情感影响评估

解析event类型→结合context识别当前可建议的情感方向→明确受影响对象。核心是多情感优先级规则（决策树）：

启动车辆方式 > 安全相关 > 节日/特殊日期 > 人物状态 > 情绪 > 人物 > 车内场景状态 > 车外场景状态 > 人物+时间 > 人物着装

安全相关强制执行安全驾驶知识中的建议，不管其他情感目标怎么说。

第二步：即时建议生成

格式：「语气/语气+表情+对象+情感表达方向+禁止提示」。6条红线（绝对禁止）：

  1. 只输出情感建议，不输出具体内容（✘"建议播放轻音乐"→越界给内容advisor管）
  2. 不控车（✘"建议调低空调"→越界给舒适advisor管）
  3. 不对未发生的事猜测
  4. 不对曾发生的事预测
  5. 不提醒物品（✘"别忘了带伞"→越界）
  6. 禁止"维持现状"类建议（✘"建议不做任何事"不算建议）

第三步：子目标生命周期管理（完整状态机）

两种子目标类型：

  • A 目标达成型："一直用温柔的语气，围绕消消气等方向安抚主驾情绪，直到主驾不再愤怒"
  • B 条件响应型："从现在开始一直用轻柔的语气，直到右后排的乘客睡醒"

操作：create / update / delete。Create需同时满足5条件（有服务对象+情感可建议+非即时闭环+可判定+语义不重复）。子目标上限10个。

第四步：输出整合+冲突检查

即时建议 vs goals冲突→以即时建议为准。新增子目标 vs goals重复→合并。

13.5 Review Prompt 3步推理链

触发："10分钟没关注+有未完成子目标"。三种情况：

  • A 进行中需调整：重度→轻度疲惫→update goal
  • B 触发条件消失：乘员已下车→恢复建议+delete goal
  • C 目标已完成：仅删除，不输出任何建议——克制！完成了就安静离场

15.3 阈值穿越详解

VLM不是"看到宝宝就报告"而是"看到状态变化才报告"。每轮VLM只输出一行true|reason或false|reason，系统对比上次(last)和这次(this)——只有last==False and this==True才触发（状态从"未发生"变"发生"的上升沿）。持续True不重复触发。

"照顾宝宝"→VLM query:{"type":"state","query":"二排存在宝宝"}
第1轮:false|宝宝没睡→不触发。第2轮:true|宝宝睡着了→last=F,this=T→触发！第3轮:true|还在睡→不触发(没变化)。

15.4 event vs state两种VLM判定模式

类型定义例子
event有明确触发动作，瞬时视觉特征，一帧可判"用户打哈欠了"
state无明确触发瞬间，长期持续关注"二排存在宝宝"

15.5 视觉感知advisor的3类策略

类型策略例子
风险监控有异常才说，只说一次，同一问题不重复照顾宝宝(哭了才报)
信息讲解必须有新内容，间隔≥60-90秒介绍沿途风景(每次讲新景点)
状态陪伴明显状态信号才触发，冷却≥5分钟缓解疲劳(打哈欠才提醒)

16.0 VLM视觉转语言流程图

16.1 Watcher三大功能

端侧部署、自带算力、Always on的观察者。

  • ① 细致理解描述场景：输出结构化流式"情景描述"——人/位置/衣着/动作/物品/精神状态/舱外天气路面
  • ② 全时多模聆听：视觉+声音持续感知
  • ③ 发现值得关注事件：以自然语言注册事件（小孩哭闹/咳嗽/前方大货车/OK手势），发生时触发advisor主动服务。一期纯工程触发，二期多模态+工程混合

16.2 VLM原理：大小LLM融合

输入：OMS摄像头(前/后视镜) video 10-15s。

处理：小模型把图像编码成token（兜底识别高优标签）→ 大模型按一段SP(识别目标)看画面输出自然语言。SP是动态的——由感知订阅模块按需拼。

输出：结构化流式的"情景描述"文字写回context的{{舱内视觉感知}}块。

"影响感知器sp/attention" = 改VLM大模型的SP。同一VLM拼不同SP切换关注对象：无订阅→既定SP通用观察；收到"关注二排宝宝"→拼新SP重点关注后排婴儿状态。不需要为每个需求各起一个模型。

约束：VLM视觉持续任务同时仅1个。演示阶段定时5s/次，量产30-60s。
