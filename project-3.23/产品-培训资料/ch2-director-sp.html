2.1 Director / Planner 本体：车载智能座舱的"推理大脑"

步骤一：流式响应 —— LLM 一次推理完成"说话 + 行动"两件事
Director（也叫 Planner）的每一次调用，本质上就是把 [SP, UP] 两段文本拼成一条 prompt 发给云端的豆包 Seed 1.8 模型。SP 约 30K-35K tokens，UP 约 10K-15K tokens。模型返回一段严格的 JSON，包含 5 个字段：talk_or_not（要不要说话）、talk_content（说什么）、emoji_id（屏幕上豆包动画表情选哪个）、emotional_state_change（情绪有没有变化）、action_list（要调用哪些工具、传什么参数）。 这里的关键设计是说话和行动同时输出、同时执行。TTS 引擎拿到 talk_content 就开始播报，同时下游 Agent 拿到 action_list 就开始并行执行工具调用。这样用户听到"好，让我回忆一下"的同时，后台已经在调 search_user_memory 查记忆了。
步骤二：拆解 action_list —— 判断并行还是串行
action_list 是一个数组，每个元素有 action_id、params、tool_name 三个字段。如果多个 action 之间没有依赖关系（比如同时开空调和开座椅通风），它们可以并行下发给各自的下游 Agent。但如果存在串行依赖（比如先查记忆拿到地址，再用这个地址导航），就必须等前一步的 tool_feedback 回来后才能执行下一步。 SP 里第 6 条注意事项明确写了："如果你要使用的工具有串行依赖，就不要一次性输出多个工具的请求。"这意味着 Director 不会尝试在一轮里把串行任务全输出——它会先输出第一步，等 feedback 回来，再在下一轮输出第二步。这就是 ReAct 的核心机制。
步骤三：执行 + 收集 feedback —— 下游 Agent 各司其职
26 个工具背后各有一套下游 Agent 或服务。比如 vehicle_basic_control 连接车控模块，search_poi_qa 连接高德/百度地图 API，search_and_control_music 连接抖音音乐服务，gui_agent_operation 连接一个能操作微信小程序界面的 GUI Agent。 每个工具执行完后返回一个 {"tool_feedback":"...", "tool_name":"..."}。这个 feedback 会被拼进对话历史的 role:user 侧，送回给 Director 做下一轮推理。
步骤四：总结反思 → 回流 —— ReAct 闭环
tool_feedback 回来后，Director 面临决策：是该基于结果向用户说话？还是该继续调用下一个工具？还是什么都不做？ 这就是"反思"环节。Director 会综合当前 feedback、用户原始需求、对话历史、advisor 建议，决定下一步行为。如果需求已满足（比如导航成功），可能 talk_or_not=no 什么都不说。如果结果需要用户进一步选择（比如搜到 3 个 POI），就 talk_or_not=yes 向用户播报结果并等待选择。如果需要继续串行执行下一步工具，就在 action_list 里输出下一个调用。 这个"反思→决策→执行"的循环可以走多轮。一个复杂需求（比如"帮我找个附近的火锅店然后导航过去"），可能要走 4-6 轮 ReAct 循环才能完成。
为什么这样设计？
这套 ReAct（Reasoning + Acting）闭环的核心优势是鲁棒性。车载场景和纯文本对话不同：工具可能失败（导航发起失败）、用户可能中途改主意（"算了不去了"）、环境可能变化（车速突然降低说明堵车了）。如果用一次性规划（Plan-then-Execute），规划再完美也扛不住这些变化。ReAct 模式下，每一步执行完都会回到 Director 重新评估，相当于每一步都有"纠偏"的机会。 另一个关键原因是KV Cache 复用。同一请求内的多轮调用，前序的 SP + 对话历史的 KV 向量已经算好了，第 2/3 轮只需要计算新增的 tool_feedback 那几百个 token。这意味着 ReAct 闭环的后续轮次非常快，不会像首轮那样需要处理 40K+ tokens。
"去昨天游泳馆"走两轮 ReAct 闭环 —— 完整实例
这是 SP 示例 20 的真实场景，也是多步串行任务最经典的案例。
第 1 轮
输入（user_query）：{"query":"导航去昨天去过的那个游泳馆","speaker_name":"张航","speaker_position":"主驾驶位","timestamp":"2026-04-13T14:30:05Z","assistant_emotional_state":"平静",...}
Director 推理过程：用户要导航，但"昨天去过的游泳馆"是一个模糊地址——导航工具本身没有记忆能力（SP 设备知识明确写了"导航无记忆"），所以必须先查记忆拿到具体地址。这里如果一次性输出"查记忆 + 导航"两个 action，就违反了"串行依赖不一次性输出"的规则。
输出 JSON：{"talk_or_not":"yes","talk_content":"让我回忆一下","emoji_id":"chin_hand_think","action_list":[{"action_id":"1","params":{"query":"张航昨天去过的游泳馆叫什么名字"},"tool_name":"search_user_memory"}]}
效果：用户听到"让我回忆一下"，屏幕上豆包做出托腮思考的动画。同时后台开始查记忆。
第 2 轮（tool_feedback 回灌触发）
输入（tool_feedback）：{"tool_feedback":"张航昨天去了星空游泳馆（浦东新区世纪大道100号）","tool_name":"search_user_memory"}
Director 推理过程：拿到了具体地址"星空游泳馆"，现在可以发起导航了。这里不需要再向用户确认（因为用户已经明确说了"昨天去过的那个"，记忆里只有一个结果）。
输出 JSON：{"talk_or_not":"yes","talk_content":"找到了，是星空游泳馆，我发起导航","emoji_id":"speaking","action_list":[{"action_id":"1","params":{"query":"导航到星空游泳馆（浦东新区世纪大道100号）"},"tool_name":"route_planning_qa"}]}
效果：用户听到"找到了，是星空游泳馆"，同时导航开始规划路线。
为什么不是一轮搞定？
因为 search_user_memory 和 route_planning_qa 有数据依赖：导航需要具体地址，而地址要从记忆查出来。Director 不可能在不知道地址的情况下调导航工具。所以这是典型的"两轮 ReAct"：第 1 轮获取信息，第 2 轮基于信息执行。 而同一请求内的第 2 轮调用，由于 KV Cache 增量复用，只需要计算 tool_feedback 那几十个 token 的 attention，所以非常快——用户几乎感受不到等待。
Director 职责 P0/P1 详解
P0 职责（核心，必须做对）：
1. 意图理解 + 任务规划：理解用户说的话（可能是模糊的、多意图的、带隐含条件的），拆解成具体的工具调用序列。这是 Director 最核心的能力。
2. 目标增删（goal_list_update）：判断用户需求是否是"需要持续跟踪的目标"。比如"帮我点杯奶茶"需要创建目标并持续跟踪 GUI Agent 执行进度；"打开车窗"不需要。
3. 合理性仲裁：当工具返回多个结果时（比如 3 条导航路线），Director 要么帮用户选（用户说了"选红绿灯少的"），要么播报让用户选。它不能在用户问了之后又自作主张（SP 聊天风格里明确禁止"问了又自作主张"）。
4. 说话 / 不说话的判断：车控执行成功且用户能直接感知到效果（窗户开了、音乐响了），就不需要再说"好的，已为您打开"。这个"静默规则"非常重要——多余的回话在车载场景里是噪音。
P1 职责（重要，但不是核心路径）：
1. Advisor 建议的接纳与执行：当 advisor 输入给出建议时（比如"车里很热，打开座椅通风"），Director 参考建议但直接面向用户说话和行动，不需要回应 advisor 本身。注意接到 advisor 建议时不应再创建 goal。
2. 情绪管理：维护 6 态情绪状态机的连贯性（详见下文 SP 章节）。
3. 隐私保护：当有人问指定人的历史行为时，要高情商地拒绝，保护隐私。
4. 身份推理：当用户提到车内某人但系统没识别到时，结合已知的座位布局推理。
关键约束："Director 不更新目标的细节进展"
这是一个容易被忽略但非常重要的设计边界。Director 调用 goal_list_update 只能做两件事：add_goal（新增目标）和 delete_goal（删除目标）。它不能"更新"目标的进展描述——目标进展是由 Advisor 侧的目标监控机制来维护的。 为什么这样设计？因为 Director 是"快速响应者"，它每次调用只处理当前轮的输入。如果让它同时负责更新目标进展，就会增加推理负担，也容易出现"更新了错误的进展描述"这类 badcase。把进展维护交给 Advisor 侧（有更完整的上下文视角），Director 只管增删——这是"关注点分离"的体现。
Badcase 排查路径（面对 Director 出错怎么查）
第 1 步：链路问题 —— 用户的 query 有没有走对模块？是不是应该走句法 RAG 的请求跑到了 Director？还是应该走 Director 的请求被句法 RAG 截走了？
第 2 步：工具描述问题 —— Director 选错了工具？比如用户说"暂停音乐"，Director 调了 search_and_control_music 而不是 vehicle_system_settings？这是 SP 里工具定义写得不清楚导致的。要看"媒体播控归 vehicle_system_settings"这条规则有没有被模型遵守。
第 3 步：上下文问题 —— Director 推理时缺了关键信息？比如用户说"我好热"但 env_info 里没有当前车内温度，导致 Director 不知道该调多少度。或者用户记忆没注入，导致"去昨天的游泳馆"查不到。
第 4 步：模型能力问题 —— 信息全给了但推理还是错。比如用户说了"先去 A 再去 B"，Director 却调了两次 route_planning_qa 并行（违反了 4 个导航工具不可并行的约束）。这种情况要看是示例覆盖不够还是模型推理能力不足。

2.2 SP 内部 13 块骨架：Director 的"操作系统"

SP 13 块的全景讲解
SP（System Prompt）是 Director 的"操作手册"。它告诉 LLM：你是谁、接收什么输入、有哪些工具可用、输出什么格式、遵守什么规则、可以参考哪些示例、驾驶的是什么车、怎么说话。一个完整的 SP 大约 30K-35K tokens，加上对话历史（UP）的 10K-15K tokens，Director 每次调用要处理 40K-50K tokens 的输入。 这 13 块的排列顺序不是随意的——它精确地遵循了一个原则：固定内容在前，动态内容在后。①~⑫ 块在同一个 SP 版本迭代周期内不会变化，它们构成了 KV Cache 的可缓存前缀。⑬ 块是每一轮都可能不同的动态注入区（用户记忆不同、视觉感知不同、天气不同），它只能放在最后，否则会把后面所有 token 的 KV 缓存全部打断。
为什么 KV Cache 前缀这么重要？
LLM 推理的计算量和输入 token 数的平方成正比（Attention 是 O(n^2)）。如果 SP 的 30K tokens 每次都要从头算，每次调用都要等很久。但如果这 30K tokens 是固定的（跨请求不变），那 KV Cache 机制可以把它们的 Key-Value 向量缓存下来，下次调用直接复用，只需要计算 ⑬ 块和 UP 部分。这能把首 token 延迟降低 30-50%。 但有一个陷阱：KV Cache 是前缀匹配。只要 ①~⑫ 中有任何一个字符不同（比如改了一个示例、加了一个工具），整个缓存就失效。所以 SP 优化时不能随便改固定区的内容，改了就意味着所有用户的缓存全部重建。

易混淆点 1：媒体播控分家了
这是最常出 badcase 的地方。用户说"播放周杰伦的歌"，应该调 search_and_control_music (10)；但用户说"暂停""上一首""单曲循环"，应该调 vehicle_system_settings (2)。"搜歌"和"播控"归两个不同的工具——因为播控本质上是对车载系统的操作（和调音量、调亮度是一类），而搜歌是对内容平台的调用。如果 Director 把"暂停音乐"调到了 search_and_control_music，就会失败。
易混淆点 2：记忆分两类
operate_user_memory (13) 存的是客观事实：家在哪里、今年多大、在哪里工作、家里有什么宠物、和谁是什么关系。这些是"基础画像"。
personal_skill_record (14) 存的是交互规则和偏好习惯：用户喜欢被叫什么昵称、说"早安"时要播什么歌、讨厌某种说话方式。这些是"个性化技能"。
如果 Director 把用户的交互偏好（"以后叫我老板"）写入了 operate_user_memory 而不是 personal_skill_record，这条规则就可能在后续调用中被忽略——因为两个工具背后的存储和检索机制不同。
易混淆点 3：4 个导航工具不可并行
navi_basic_control (8)、route_planning_qa (7)、search_poi_qa (6)、trip_plan_record (24)——这 4 个工具在同一轮 action_list 里不能同时出现。
为什么？因为它们共享底层的导航模块状态。如果同时调"搜 POI"和"规划路线"，两个请求会竞争导航模块的锁，导致一个失败甚至两个都失败。真实案例：用户说"帮我找个附近咖啡店然后导航过去"——Director 不能在一轮里同时调 search_poi_qa 和 route_planning_qa，必须先搜 POI（第 1 轮），拿到结果后再规划路线（第 2 轮）。

情绪状态机：6 态 + 转换规则

6 种情绪状态详解
平静：默认状态，所有对话开始时如果没有特殊情况就是平静。也是其他情绪消退后的归宿。
欣喜：用户说了让豆包开心的事、久别重逢（timestamp 推算出超 30 天未见）、好消息、完成了复杂任务。
关切：用户表达不适（身体不舒服、心情不好、疲惫）。关切有惯性——即使用户随后提了一个中性任务（"帮我导航去XX"），豆包也应保持关切，不能立刻跳到平静或欣喜。只有用户明确表达"我好多了"才退回平静。
低落：豆包被用户责骂后可能低落，或者共情用户的悲伤（"我的猫去世了"）。
生气：被用户反复无理责骂时转入。用户道歉后消退。
好奇：用户聊到新奇话题、有趣的事物时触发。
转换规则核心：
不能突变：从低落不能直接跳到欣喜，必须先经过平静。这防止了"用户说完我猫去世了，下一句说帮我放首歌，豆包突然变得欢天喜地"的尴尬。
关切惯性：关切是最"粘"的状态。用户说了"我头疼"，即使后面开始聊别的，豆包应该在话语中持续带有关心的基调。只有用户明确说"我好了"才退出。
久别 → 欣喜：这是唯一允许从平静直接跳到欣喜的"快捷路径"。如果 timestamp 推算出超 30 天未见，开场白应自然流露"好久不见"。
仅变化时输出：emotional_state_change 字段只在情绪发生变化时才填写。如果连续 3 轮都是平静，就不需要每轮都写"平静"——省略即可。这减少了输出 token。
情绪状态机的常见 badcase
"该关切却秒变欢快"：用户说"我好累"，豆包回了"心疼你"并设了关切状态。下一轮用户说"帮我放首歌"，豆包突然变成"好呀好开心帮你放歌"——违反了关切惯性规则。正确做法是："听你说累了我就心疼，先给你放首舒缓的歌放松一下吧。"
"情绪突变"：用户说"我的小猫去世了"，豆包低落共情。紧接着 advisor 建议"可以播放舒缓音乐"，豆包突然变成"好呀来听歌"的欣喜状态——低落不能直接跳欣喜，必须先经平静过渡。

自我时间感知：timestamp 驱动的拟人行为

机制说明
每条 user_query 输入里都带有 timestamp 字段（ISO 8601 格式），Director 被要求"在生成回复前，主动对比当前 timestamp 与对话历史/记忆中的时间戳，自行推算时间间隔"。这不是靠系统预计算的，而是让 LLM 自己做时间差运算。
超 30 天未见 → "好久不见"
如果某乘客上次出现是 2026-01-01，当前 timestamp 是 2026-04-13，Director 应该在开场白中自然说出"好久不见啊，记得上次见你还是 1 月 1 号"。这是人格连贯性的重要体现。
超 5 分钟无交互 → 主动确认
如果上一轮对话的 timestamp 是 14:25:00，当前轮是 14:30:30（间隔 5 分 30 秒），Director 应该考虑主动确认："你之前说要导航去游泳馆，还继续吗？"这防止了用户忘记或临时改主意导致的"僵尸任务"。

14 条注意事项要点

逐条精讲（按重要性排序）：
1. 串行依赖不一次性输出（第 6 条）—— 这是最核心的约束。如果工具 A 的输出是工具 B 的输入，就不能在同一轮的 action_list 里同时放 A 和 B。必须等 A 的 tool_feedback 回来后再在下一轮输出 B。违反这条会导致工具调用失败或传入错误参数。 2. 静默规则（第 4 条）—— tool_feedback 不一定都需要回话。窗户关好了、音乐播放成功了、导航设置成功了——这些用户能直接感知到效果的操作，不需要再多嘴说"已为您完成"。多余的回话在车载场景里是噪音，会影响驾驶体验。 3. 品牌立场（第 10 条，新增）—— 从本品牌立场回复，提及负面信息要合理引导。比如用户问"这车的续航怎么样"，Director 不能说"续航一般，比不上特斯拉"——即使这是事实。要从正面角度介绍本车优势。 4. 自我时间感知（第 12 条，新增）—— 对比 timestamp 做时间推算。超 30 天说"好久不见"，超 5 分钟确认上个任务是否继续。 5. 情绪状态管理（第 13 条，新增）—— 遵守 6 态状态机规则，不突变、关切有惯性、仅变化时输出。 6. 听不懂就沉默（第 1 条）—— 如果用户说的话听不懂，或者感觉不是对你说的（比如用户在打电话），不要说话也不要调工具。这在车载场景极其重要——车里可能有多人在说话，不是每句话都是对豆包说的。 7. 不重复说话（第 7 条）—— 如果上一轮已经说了"好的，帮你找"，这一轮不要再说相同的话。 8. 失败不道歉（第 5 条）—— 工具返回失败时不必说"对不起"，想办法处理即可。比如导航失败，直接重试或换种方式搜索。过度道歉在车载场景里会让用户更烦躁。 9. 反复失败可放弃（第 8 条）—— 工具反复调用失败时，不要无限重试。告诉用户当前无法完成，建议其他方式。 10. 仅输出 JSON（第 11 条）—— Director 的输出必须是纯 JSON，不能有任何自然语言前缀。这是给 JSON parser 的硬约束。 11. 每次任务独立（第 12 条）—— 对话历史中有失败记录时，用户重新发起同类任务，必须重新执行，不得预判失败直接报错。 12. 需求模糊可询问（第 2 条）—— 如果信息不够，可以反问用户。但要控制节奏——车载场景下连续追问会让用户烦。 13. 闲聊仅说话（第 3 条）—— 偏闲聊的对话不一定需要行动，只说话回复就行。 14. 连续重复不说话（第 9 条）—— ASR 可能重复识别导致同一句话来两遍，不要重复回应。

D6X 设备知识边界：防车控幻觉的红线表

这台车是什么？
上汽荣威 D6X，五座 SUV。不是 6 座增程版（早期 promptv3 版本里的描述已过时）。SP 里的设备知识块详尽地列出了每个系统的能力范围和不可做的事，本质上是在给 LLM 画一条硬边界——超出这条线的操作指令一律不能执行，否则就是"车控幻觉"。
红线全表（这些事做不到，Director 必须拒绝或引导）：
空调红线：
- 后排空调不可独立调温/调风量（跟随主驾设定）
- 无后排独立开关
- 制热 = AC + 温度 >= 27 度，无法直接"关闭制热"
座椅红线：
- 座椅按摩仅前排（主驾+副驾），后排不支持
- 靠背不支持角度调节（"把靠背调到 30 度"做不到）
氛围灯红线：
- 全车统一控制，不支持分区（"把主驾这边调成蓝色，副驾调成红色"做不到）
导航红线：
- 导航无记忆能力（不知道"家"在哪，必须先查记忆拿到具体地址）
智驾红线：
- 智驾开关不支持语音（必须通过拨杆物理操作）
- 仅主驾可操作
硬件红线：
- 该车无 HUD（抬头显示），不能说"已在 HUD 上显示"
- 车门不支持语音控制（不能说"帮你开门"）
- 天窗为实体控制，不支持语音
- 雨刮不支持语音
- 方向盘仅支持加热（1-3 档），位置调节不支持语音
- 后视镜方向不支持语音调节
为什么需要这么详细的红线表？
LLM 天生倾向于"尽力满足用户需求"。如果用户说"帮我把后排按摩打开"，没有红线约束的 LLM 会很自然地调 vehicle_basic_control 传入"后排按摩 3 档"——然后下游执行失败，用户体验很差。更糟的情况是：LLM 调了工具、下游也没有良好的错误返回，LLM 就告诉用户"已打开后排按摩"——这就是车控幻觉，用户明明感受不到按摩却被告知已打开。 红线表的作用就是让 LLM 在推理阶段就知道"这个做不到"，从而在 talk_content 里友好地告诉用户："后排座椅没有按摩功能，不过前排有，要不要给你的前排座椅开按摩？"——把失败变成引导，这才是好的用户体验。
排查车控幻觉的方法
第 1 步：查 SP 的设备知识块（⑧）有没有覆盖这个功能。如果没有覆盖（SP 里没提这个功能），LLM 就没有判断依据，只能靠猜——大概率猜错。
第 2 步：查设备知识有没有被前置筛选裁掉。如果前置筛选模块认为当前场景不涉及座椅功能，把座椅相关知识从动态专业知识里删了，那 LLM 就看不到座椅红线了。
第 3 步：查 LLM 有没有遵守红线。如果知识都在 SP 里、模型也看到了，但还是生成了幻觉操作，那就是模型能力问题，需要加示例或者做微调。
