Director / Planner 架构逐节点讲义

这份不是"图+表格"，而是像一篇长文章一样讲。每个节点的格式：先看一张骨架图（它在整个系统里长什么样、内部有几块），配一段讲解把图说透；再看一张流程图（一条具体请求怎么流过它），配一段逐步走读带真实举例。读完每个节点你应该能：① 向别人画出来讲清楚 ② 遇到 badcase 知道到这个节点里查什么。

1多级拒识和召回——整条链路的第一道闸门

用户说了一句话，系统做的第一件事不是"理解它"，而是"判断该不该接"。这个节点就是那道闸。

骨架图：它在链路中的位置 + 内部三级结构

端侧/快链路拒识推理/Directorcontext/advisor

骨架图讲解：为什么是三级拒识，不是一级？

你可以把这个节点想象成一个三道安检的机场。用户说了一句话（就是一个"旅客"要进来），系统不是直接让它进 Director（"登机口"），而是先过三道关：

第一道：端侧白名单+LLM（安检门）——车上本地跑的一个小模型，配了一张"白名单"。如果用户说的是"打开车窗""空调调到25度"这种明确、高频、简单的车控指令，白名单直接命中，端侧就地执行，根本不上云。时延约 150ms，用户几乎感觉不到等待。这是性能的关键——越多流量在这里拦住，系统整体越快。
第二道：云侧前置拒识（海关）——白名单没命中的 query 上到云端，先过一个"云端拒识模型"。如果判定这句话不该由车机处理（比如"妈妈帮我拿一下水杯"——这不是对车说的），就 reject 掉。但 reject 不是直接丢弃！拒识的内容会写进 context 事件日志，被 advisor 感知。为什么？因为"妈妈帮我拿水杯"虽然车做不了，但 advisor 可能据此推断"用户可能是小孩/需要帮助"，从而触发主动服务。
第三道：Director 内部兜底拒识（登机口最后查票）——即使过了前两道，Director 在 plan_describe 阶段之前还会再做一次拒识判断。这是最后的兜底，防止少量漏网的非车机 query 消耗 Director 的推理资源。
用户说"帮我叫一辆滴滴"。① 端侧白名单没有"叫滴滴"→未命中；② 上云，云侧拒识模型判定"叫网约车不是车机能力"→ reject，写入 context 事件日志；③ advisor 感知到这条日志，可能后续主动提醒"要不要我帮你导航到打车点？"。这就是拒识不丢弃、转主动服务的设计精妙处。
为什么不只做一级拒识？因为约 30% 的车控指令是模糊/复杂表达（吉利量产数据），单看 query 文本分不准。比如"我好热"——空调没开时该控车降温，空调已开到最冷时该对话回复"已经开最低了"。同一句话因车况不同结论不同，所以端侧靠白名单接简单的，云侧靠模型+情景接复杂的，Director 内部再兜底一次——三级冗余换稳定。

流程图：一句"空调调低点"怎么走过这个节点

流程图逐步走读："空调调低点"这句话发生了什么

让我们跟着这句话走一遍完整链路：

Step 1：ASR 把语音转成文本——用户在车里说"空调调低点"，ASR 模块把它转成文字 query。这一步纯语音处理，耗时约 200ms。注意 query 里还带了 speaker_name（说话人名字）和 speaker_position（说话人位置，如"主驾"），这两个字段后面会决定"给谁的座椅调"。
Step 2：端侧离线 fc 查白名单——query 到了端侧的 fc 模型（function call 模型），它拿着一张白名单比对。"空调调低"是高频车控指令，大概率命中白名单。如果命中了，接下来就不上云了——端侧直接把"空调温度-2"的指令发给车控模块，约 150ms 完成。用户几乎秒响应。
Step 3（如果没命中）：上云——假设用户说的不是"空调调低点"，而是"我好热啊"（模糊表达）。这时白名单没有"我好热"→未命中→query 上到云端。规则是"优先云端结果"：云端先跑拒识+仲裁模型，如果云端太慢超时了，退回来看端上有没有结果（如果端侧的 fc 也猜了个结果的话），端上也没有→做超时兜底播报。
Step 4（云端）：拒识+仲裁判别——"我好热啊"到了云端，先过拒识模型：这句话是不是不该由车机处理的？不是，它是合理需求→放行。然后仲裁模型判断：这是简单车控（直接给 fc 处理）还是复杂任务（需要 Director 思考）？"我好热"看起来简单，但要判断"空调是否已开、当前温度多少"才能决定是调温度还是回复"已经开最低了"——所以仲裁可能判为"复杂"，交给 Director。
对比两个 case 理解分流的难度：case A "打开车窗"→白名单直接命中→端侧 150ms 搞定；case B "我好热啊"→白名单没有→上云→仲裁判复杂→Director 读取 env_info（空调 26 度已开）→决定"再降 2 度"→调 vehicle_basic_control→完成。同样是车控，A 走 150ms 快链路，B 走 500ms+ 慢思考。这就是"层层过滤"设计的价值——简单的别让贵模型干。
Badcase 高发区：约 30% 的车控是模糊表达（"我好热""太吵了""灯太亮"），这些在端侧/仲裁层很容易分错。排查路径：① 先看 query 走了哪条路（端侧命中？云端召回？reject？）→ ② 看是不是被误 reject（该进 Director 却被拒了）或误放行（该 reject 却进了 Director）→ ③ 看仲裁的简单/复杂分流是否正确。reject_ornot 字段是排查第一抓手。

2仲裁模型——决定这句话走"快车道"还是"慢思考"

拒识放行后，仲裁模型做最关键的一个判断：这个需求简单还是复杂？简单走快车道（300ms级），复杂才进 Director（500ms+）。

骨架图：仲裁模型在链路中的位置

召回/端侧仲裁/Director快车道

骨架图讲解：仲裁模型到底做了什么

经过节点1放行的 query，带着情景 RAG 注入的知识（以 few-shot 形式，不是直接给答案），来到仲裁模型面前。仲裁模型是一个轻量级判别模型（Seed1.6 flash SFT），它只做一件事：这句话简单还是复杂？

简单的标准很直接：如果这句话是"明确的、能直接映射到一个车控指令"的，就是简单——交给云侧 LLM 快速生成指令，走 FC 快车道。复杂的标准是"需要多步推理、需要查信息、需要拆任务、模糊不明确"——交给 Director 慢思考。

情景 RAG 在这里的角色很关键：它不直接给仲裁答案，而是注入 few-shot（少量示例）和 agentLimit（能力边界）。格式是 {"type":"modelExecute","knowledge":"...","agentLimit":[...]}。这些 few-shot 帮仲裁模型理解"这种场景下类似的 query 应该走哪条路"，但不替它做决定。

为什么仲裁模型选 Seed1.6 flash SFT 这么轻的模型？因为它只需要做二分类（简单/复杂），不需要深度推理能力。用大模型做这件事太贵太慢——它的时延就是白花的等待。轻模型+few-shot 注入，能在很低的时延内完成路由判断。
仲裁误判的三大根因（原文论证）：① 输入太单一，仅有 query——同一句"我好热"，空调没开该控温，空调已开最冷该回话，单看 query 分不清；② 靠造数据学习，case 造不完——NLU 的人肉解 bug 方法；③ 意图处理信息量太少，车控有上千函数，效果差。COT 快慢思考实验（⚠️未上线）想解决的就是这个问题——在意图阶段就引入状态/记忆/视觉信息做"执行结论前置"。
链路时延走什么
简单快车道FC.SFT 300ms → 端控 150ms"打开车窗""导航去公司"
复杂慢思考Director 思考 500ms + 回答 500ms"我好热""帮我规划个周末"

3Director / Planner 本体——整个系统的"大脑"

Director = Planner = 对话推理引擎。它是三个名字、一个东西。漏斗最深一层，兜底+云端 teacher，跑 Doubao-Seed-1.8。

骨架图：Director 的内部四块循环 + 在链路中的位置

骨架图讲解：一次调用 = 多次推理的 ReAct 闭环

Director 不是"想一次就完事"的模型调用，它是一个循环——想一点、做一点、看结果、再想。这就是 ReAct（Reason + Act） 范式。图里四个方块从上到下就是一次循环的四步：

① 流式对话响应——Director 收到输入后，第一件事是先快速回一句话（首 token 要在 500ms 内出来），让用户感到"它听到了、在处理了"。比如用户说"帮我导航去公司"，Director 先回一句"好的，我来帮你规划"。这是体验优化——哪怕后面还要花几百毫秒调工具，用户已经不焦虑了。
② 任务拆解——回完话，Director 开始拆任务。"帮我导航去公司"可能拆成：先查记忆找到"公司在哪"→ 再调 route_planning_qa 导航。"我好热还想听歌"拆成：调空调降温 + 放音乐，两个可并行。这是 Director 的核心推理能力。
③ 结构化任务序列 → action_list——拆好的任务变成 JSON 格式的 action_list，每个 action 包含 action_id（序号）、params（参数）、tool_name（调哪个工具）。这个列表发给下游 26 个工具之一去执行。
④ 总结与反思——工具执行完返回 tool_feedback（比如"导航成功"或"未找到该地址"），Director 读取反馈，决定下一步：成功→总结一句话告诉用户（或静默不说话）；失败→重试或换方案（Replan）；需要更多信息→继续调另一个工具。然后回到①——这就是虚线画的ReAct 回流。
用户说"去昨天游泳馆"。第一轮推理：Director 发现需要查记忆才知道"昨天游泳馆"是哪，先调 search_user_memory 查。工具返回："昨天去了虹桥游泳馆"。第二轮推理（回流到①）：有地址了，调 route_planning_qa 导航到虹桥游泳馆。工具返回："导航成功"。Director 判断任务完成，不说话（静默，因为用户能直接看到导航画面）。这就是一次调用里的两轮 ReAct。
为什么要 ReAct 而不是一次想完？因为车载场景的工具返回是不确定的——导航可能找不到地址、记忆可能没有、联网可能超时。"想完再做"在不确定环境下会大量出错；"想一步做一步看结果"虽然多调几次 LLM，但每一步都基于真实反馈，稳定性远高于一次性规划。

流程图：三种输入 → 五个输出字段

流程图讲解：输入输出到底长什么样

三种输入，同一时刻可能同时到达（以数组形式传入）：

user_query——用户说的话，是最常见的触发源。真身比你想象的丰富，它不只有 query 文本，还带了 7 个字段：query（话本身）、speaker_name（说话人）、speaker_position（座位位置）、timestamp（ISO8601 时间戳，用于推算时间间隔）、assistant_emotional_state（豆包上一轮情绪，保持连贯）、goal_list（当前正在追踪的目标列表）、env_info（最新车辆状态）。后两个字段是降时延的关键设计——车况和目标直接随输入传给 Director，不用再调工具查。
advisor——{"advisor":"晓伟上车了，车内温度高，可调空调降温"}。这不是用户说的话！是 advisor（主动服务模块）给 Director 的建议。Director 不回应 advisor，直接面向用户做行动和表达。也就是说 advisor 在幕后提议，Director 在台前执行。
tool_feedback——{"tool_feedback":"导航成功","tool_name":"route_planning_qa"}。工具执行完回来的反馈，触发 Director 的二次推理。对于成功的简单车控（"车窗已关闭"），Director 通常选择不说话（talk_or_not: no），因为用户已经能感知到结果了。

五个输出字段（这是真身SP，比旧版多了两个）：

talk_or_not = "yes" 或 "no"。Director 可以选择不说话——这是很多新人会忽略的点。车控成功、音乐开始播、导航画面出来了，用户已经能看到/听到结果，Director 就不必多嘴。但该说时要说——失败要解释，需要澄清要问，有趣要互动。
emoji_id（★真身新增）——从 17 个预定义动画表情里选一个，驱动车机屏幕上豆包的形象动画。默认用 speaking（嘴巴动），听音乐用 listening_music，思考用 chin_hand_think，欢呼用 cheer。规则：除 speaking 外不连续使用。
emotional_state_change（★真身新增）——仅在情绪发生变化时输出，相同则省略。6 种状态：平静（默认）、欣喜、关切、低落、生气、好奇。有转换规则：不能突变（低落→不能直接欣喜，需经平静过渡）；关切有惯性（用户情绪没好转就不能秒变欣喜）；久别重逢可直接进欣喜。这套机制让"温柔高知女生"人设有连贯的情绪线。
示例：用户说"豆包，我被裁员了"。Director 输出：talk_or_not:"yes", talk_content:"[轻声地] 雨晴……我不知道该说什么。这事儿来得太突然了。", emoji_id:"speaking", emotional_state_change:"关切"。接下来用户说"放首安静的歌"——Director 保持【关切】（惯性），不会突变成【欣喜】：talk_content:"[轻声地] 好。放一首没有歌词的，不打扰你。"，情绪不输出 emotional_state_change（因为还是关切，没变化）。
情绪状态机是 badcase 新维度。常见问题："情绪突变"——用户刚哭完，Director 下一轮突然"[开心地说]好呀！"；"关切不持续"——用户说不舒服，Director 关切了一句，下一轮就回到平静跟没事人一样。排查时看 assistant_emotional_state 输入 vs emotional_state_change 输出，是否遵守了"不突变/关切惯性"规则。

4SP 内部骨架——Director 脑子里装了什么（13 块）

Director 的"脑子"就是 System Prompt。看懂 SP 的 13 块结构 = 看懂 Director 怎么思考、能力边界在哪。

骨架图：SP 的 13 块纵向堆叠（标注固定/动态）

骨架图讲解：为什么"固定在前、动态在后"——这是缓存友好的关键

Director 每次推理 = 往模型里喂一条 [SP, UP] 请求。SP 就是这 13 块的拼接体，UP 是当前轮的三种输入 + 历史对话。

核心设计精妙处在于分区：①~⑫ 是固定的——角色定义、工具定义、示例、设备知识这些东西每次调用都一样。因为前缀完全匹配，KV Cache（Prompt Cache）可以复用已算好的注意力向量，不需要重算这几万个 token——只要算增量（⑬ 的动态部分 + UP 的当前输入），推理时间大幅缩短。

⑬ 放在最后，是因为它每轮都变——车辆状态变了、用户记忆不同了、前置筛选注入了不同的知识。一旦某个位置的 token 变了，后面所有的 KV Cache 都作废。所以把"变"的放在最尾巴，"不变"的前缀尽可能长，缓存命中率最大化。

⑥ 参考示例是 SP 的体积主力——示例 1~21，每个示例是一段完整的多轮对话（input1→output1→input2→output2…），少则 3 轮多则 12 轮。它们是 Director 学会"拆解任务、选工具、控情绪、判断 talk_or_not"的核心 few-shot 教材。也正因为它太大，前置筛选模块在想"能不能按场景只注入相关的几个示例"——这是 token 优化的最大潜力空间。
⑬ 里的「动态专业知识」= 前置筛选模块的注入点——上一轮讲的"知识筛选专家"（6类专家×四段式ID），它挑好的知识 ID 对应的内容，就塞到这里。所以 Director 每轮看到的"动态专业知识"都是当前场景最相关的那几条，不是全量。
某一轮 user_query 是"帮我照顾后排宝宝"。前置筛选判断这是 COM（管家/舒适）+ EMO（伙伴/情感）场景，注入 COM-01-01-01（调温度）和 EMO-01-01-04（新朋友仪式）相关知识到 {{动态专业知识}}。下一轮用户说"去机场接人"，筛选器删掉 COM-01-01-01，添加 DRV-01-01-03（接机），Director 看到的知识集实时跟着场景变。
goal_list 和 env_info 为什么不走 ⑬ 而是随输入直传？因为 SP 原文说"可直接参考它们来编排行动，避免不必要的工具调用"。如果目标列表和车辆状态每轮都摆在输入里，Director 就不需要每次都调 search_vehicle_status_info 去查"当前空调几度"——输入里已经有了。这又省了一次工具调用的时延（约 300ms 起步）。
常见 badcase 与对应的 SP 区块：话术生硬/人设跳脱→查⑪聊天风格+情绪状态机；工具误并行(如 navi 与 route 同时调)→查⑨tips"4个不可并行"+⑤注意事项6"串行依赖"；车控幻觉（说"已打开HUD"但该车无HUD）→查⑧设备知识；重复调 search_* 取已知信息→查 goal_list/env_info 是否直传到位；拆解错→对照⑥示例是否覆盖该意图类型。

5goal list（共享目标看板）——从"一锤子买卖"到"持续照看"的载体

旧架构的病：用户说一句，Director 应一句，做完就忘。goal list 就是治这个病的——把"正在为用户做的事"显式写成一张看板，Director 和 advisor 共同盯着它。

骨架图：看板在中心，被谁读、被谁写

骨架图讲解：为什么 Director 故意"不改进展"

goal list 的读写权责是整套目标驱动架构最反直觉、但最重要的设计。它的规则只有一条：谁建谁删（Director），谁盯谁改（advisor），两者不交叉。

Director 能做的：创建一个新目标（如"照顾二排宝宝"）、删除一个完成/取消的目标。它不更新目标的"细节和进展"。为什么？因为 Director 是快速响应者（重时延），它没有时间也不应该去"盯进度"；而 advisor 是深度思考者（重质量），它专门负责持续 review 目标进展。如果两边都能改"细节和进展"，就会出现并发写冲突——Director 写"宝宝在睡觉"的同时 advisor 写"宝宝醒了在哭"，谁的为准？

所以设计者做了一个硬约束：Director 只管"建/删"这个低频且确定的操作，advisor 管"改进展"这个高频且需要持续盯的操作。

goal list 的数据结构很简单，就是一个 JSON 数组，每条三个字段：
[
  {"目标":"照顾二排宝宝", "目标状态":"进行中", "目标细节和进展":"当前宝宝情绪平稳，看起来睡着了，持续监测中"},
  {"目标":"每5分钟播报NBA赛况", "目标状态":"进行中", "目标细节和进展":"当前三场比赛进行中，已完成第三次播报"}
]
"目标细节和进展"是一段自由文本——不是结构化的状态枚举，而是 advisor 用自然语言写的一句话。这设计灵活但有隐患：如果 advisor 写的含糊（"差不多了"），Director 下一轮读到这个进展时可能误判目标是否达成。
"细节和进展"是自由文本是已知的 badcase 源。典型问题：① advisor 写得含糊→Director 误判目标已完成→提前删除→用户发现"系统忘了我让它做的事"；② advisor 写得自相矛盾（上一次"宝宝睡着"，这一次"宝宝一直很安静"但没说醒没醒）→Director 无法判断是否需要推进。优化方向之一是把进展从自由文本改为结构化字段（status枚举+描述），但目前还是自由文本。

6goal_list_update 工具——防止目标爆炸的守门员

Director 不直接改 goal list，而是通过这个工具。它干两件事：增删目标 + 数量仲裁。

流程图：一个目标从创建到仲裁

流程图讲解：为什么要"守门"——资源不是无限的

你可能会问：为什么不让 Director 随便建目标？因为持续目标会持续消耗资源：每个定时任务要跑定时器、每个条件任务要开监控、每个 VLM 视觉任务要占用端侧摄像头处理管线。如果不限量，用户说十句话建了十个持续目标，系统就要同时跑十个监控——性能崩溃。

数量上限（硬约束）：定时任务≤6、条件任务≤6、长程 GUI≤1（因为 GUI 独占屏幕）、长时任务（除 vlm）≤2、vlm 视觉持续任务同时仅 1 个。最后这条特别重要——它意味着"照顾宝宝"（需要 VLM 持续看后排）和"介绍沿途风景"（需要 VLM 看车外）不能同时持续运行，只能二选一。这是产品功能边界的真实来源。
tips 约束（软规则）：① 只有"需要持续观察跟踪/较长步骤"的任务才该用这个工具——"打开车窗"这种一次性的不该建目标；② 如果输入是 advisor 的建议，不用这个工具（advisor 建议已有目标管着，不需要另建）；③ 已有或相似的目标不重复建；④ 建 GUI 长时任务时必须同步建 goal（确保它在看板上可追踪）。
用户说"帮我照顾后排宝宝"→Director 调 goal_list_update({type:"add_goal", goal_content:"照顾二排宝宝"})→工具检查：是持续任务吗？是。vlm 视觉任务当前有几个？0个，未超限→写入 goal list→同时创建动态 advisor 实例。后来用户又说"帮我也介绍沿途风景"→Director 再调一次→但 vlm 已有 1 个（宝宝），如果也需要 vlm 持续看→超限→返回 error。这时 Director 需要告知用户"目前已经在照顾宝宝了，两个视觉任务不能同时跑，要不要先停一个？"

7静态 advisor ×4（永久监工）——舒适/出行/情感/内容

4 个永远在岗的"值班员"，各盯一条线（舒适/出行/情感/内容），不随对话结束而消失。

骨架图 + 讲解

像一家公司的四个常驻部门经理

如果把整个系统比作一家公司，Director 是 CEO（快速决策、对外沟通），那这 4 个静态 advisor 就是四个常驻部门经理：舒适经理盯着车内温度/座椅/气味有没有让乘客不舒服；出行经理盯着导航/充电/路况有没有要提醒的；情感经理盯着乘客情绪/迎宾/生日彩蛋；内容经理盯着音乐/视频/游戏/节目编排。

它们原本只做一件事——主动服务（基于情景给 Director 建议，比如"车内温度高了，建议开空调"）。在新框架里，它们被新增了两项职责：

职责①目标更新者：基于 query/context，更新自己绑定的那条 static 目标的"细节和进展"。比如舒适 advisor 把"当前已开启极速制冷，车内温度正在下降"写进 goal list。
职责③goal reviewer：持续盯着自己绑定的目标是否达成。比如出行 advisor 发现"导航已到达目的地"→标记出行目标达成→推进进展。

这两项新增职责就是"从 Planner 一人包揽到团队协作"的具体体现——盯进度的活从 Director 身上卸下来，分给了这四个常驻经理。

四个 advisor 都往 Director 抛建议，会不会造成"CEO 被建议淹没"？会的。这就是交互管理里"advisor 建议排队/丢弃"机制存在的原因（见节点14）。在用户讲话或 Director 推理中，advisor 的建议会被丢弃或排队，不是谁都能随时插嘴。

8动态 advisor 实例（临时监工）——照顾宝宝的那个"值班员"

静态 advisor 管的是固定四维度，但"帮我照顾宝宝""介绍沿途风景"这种临时冒出来的持续任务，需要动态创建一个专属监工。

流程图：照顾二排宝宝的 11 步时序

流程图逐步走读：Director 只做了前两步，后面全是"监工+感知"在自循环

这 11 步时序是理解整套"目标驱动"最好的入口。让我一步步走：

Step ①②（Director 主导）：用户说"帮我照顾一下二排宝宝"，Director 做了两件事就"撒手"了——调 goal_list_update 建了一个目标"照顾二排宝宝"，同时基于 prompt 模板工程自动注册了一个动态 advisor 实例。从这一刻起，Director 不再主动盯这个任务。
Step ②③（advisor 接管）：新建的动态 advisor 向感知订阅模块发出订阅："每 10s 告诉我一次二排宝宝的视觉状态"。感知订阅模块把这句自然语言拆成三维度——动态感知=宝宝视觉状态，条件=na，定时=每10s——然后去配置 VLM：调整视觉感知器的 sp/attention，让 VLM 重点关注后排宝宝。
Step ④（VLM 开始看）：VLM（端侧摄像头+视觉模型）按照新的 SP（识别目标="关注二排宝宝"）开始看画面，持续输出结构化情景描述，比如"后排左座婴儿闭眼平稳呼吸，疑似睡眠中"。这些描述流式写回 context。
Step ⑤⑥⑦（自循环，最核心的闭环）：定时器每 5s 触发一次动态 advisor。advisor 醒来后做两件事：A）目标 review——读取 VLM 最新的感知信息，判断宝宝状态有没有变化，把进展写回 goal list（如"宝宝安睡，持续监测中"）；B）推进目标——如果发现宝宝哭了，就给 Director 一条建议（如"宝宝醒了在哭，建议播放安抚音乐"）。Director 收到建议后做响应（"我放首轻柔的摇篮曲"+ 调音乐工具）。

关键洞察：Director 在 Step①②之后就再也没主动参与过——所有的"持续盯"都是 advisor + 感知订阅 + VLM 在自动循环跑。只有 advisor 觉得需要推进时（Step⑦），才回头找 Director。这就是"减轻 Director 过载"的具体体现——大脑不盯进度，监工盯。

恢复机制也靠这个闭环。假设中途用户说了别的（"帮我导航去公司"），Director 去处理导航了。宝宝的监控没有中断——因为它不靠 Director 运转，advisor+定时器+VLM 还在自循环。等用户的导航处理完，advisor 依然会按时给 Director 发建议。"恢复"不靠 Director 记忆，靠 advisor 读 goal list 里的"细节和进展"重新触发。
动态 advisor 不是随便建的。判断逻辑：靠 4 个静态目标（舒适/出行/情感/内容）就能闭环的（比如"照顾二排小孩"落在舒适维度），不建动态目标。只有靠静态无法追踪的、需要专属监控的持续任务（如"点咖啡送公司"要盯外卖进度）才建。建多了撞数量上限（vlm 仅 1 个），建少了任务没人盯。
