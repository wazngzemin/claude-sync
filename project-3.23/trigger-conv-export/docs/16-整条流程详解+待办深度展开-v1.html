一整条流程走一遍:"宝宝毯子掉了提醒我"(17步)

这是最复杂的一类任务(动态视觉观测),经过你最多的节点。绿色圆圈=你的动作,蓝色=别人。

1
用户说话 → ASR转文字丁彬·端
用户在车里说"宝宝毯子掉了提醒我",端侧ASR转成文字。
2
拒识分流斯华·意图
⑨判定这句话是"非车控query/条件话"(不是"开车窗"那种直接指令),送给Director。
3
Director拆解 → 创建动态Advisor仁杰/庆磊
Director判断这是"持续视觉观测任务"(要一直盯着毯子),创建一个动态Advisor专门管这件事。
4
动态Advisor先查:毯子状态是现成的吗?庆磊
动态Advisor先看Context里有没有"毯子"信息(官方图的第一个菱形"是否预先对齐的enum")。毯子状态不是enum(AIBox没有"毯子掉没掉"这种现成标签)→ 走"拼SP"这条路(路2)。
// 对比:如果是"宝宝睡着了吗",睡觉是enum → 走路1,你直接监听信号,不用拼SP // 毯子不是enum → 走路2,要拼SP让VLM去看
5
动态Advisor把"人话需求"给你的云端触发器你·云端触发器
动态Advisor给你的是一句人话:"观察毯子状态/毯子掉没掉"——不是SP。拼SP是你的事。
你的云端触发器接收这个观察需求,准备把它变成VLM能执行的SP。
6
15-16 判枚举/描述 → 15-17 提槽判舱内/舱外你·云端触发器
你先判:毯子是"描述类"(不是enum)→ 需要拼SP。再判方向:"毯子"在舱内 → 走端侧VLM(舱内图不上云,合规)。
7
15-18 拼SP + 定KV槽位你·云端触发器·核心动作
你把"观察毯子"这句需求,拼进当前正在跑的AlwaysOn SP里,并给它分配一个KV槽位(用任务名称当key)。
原生SP: "prompt: aaaa, bbbb" (舱内AlwaysOn本来在看的) + 你拼入: "cccc = 观察后排宝宝毯子状态" = 合并SP: "prompt: aaaa, bbbb, cccc" KV槽位: answer_毯子任务 ← cccc的结果 (key=任务名,不是SP1/SP2)
拼字符串 + 定槽位 + 记录"这个任务对应哪个answer"。这是你整个流程里最核心的技术动作。
8
合并SP下发给端侧触发器你·云端触发器→端侧触发器
拼好的新SP,从云端触发器下发给端侧触发器。
⚠ 这里就是张荣缺席时点名你回答的"端侧触发器怎么跟云端拼接SP"——你要在PRD讲清:云端触发器用什么接口把SP下发给端侧触发器(见T2)。
9
端侧触发器触发VLM推理你·端侧触发器
端侧触发器接到SP,调AIBox让VLM按这个新SP去推理(绿线)。
端侧触发器是VLM的"触发方"——它调AIBox,让VLM每5秒按合并SP看一次。
10
VLM推理,输出客观描述丁彬/兰若·AIBox
VLM每5秒看一次,输出客观描述(不判断"掉没掉")。
{ "answer_毯子任务": "毯子滑落在脚底" } // 注意:VLM只说"在脚底",不说"掉了"
11
端侧触发器接收VLM结果你·端侧触发器
VLM结果回到端侧触发器(紫线的起点)。
⚠ 这里是丁彬第③点、你截图问的T3:VLM结果不回给"触发者"了,而是要给AI service。触发方(端侧触发器)≠最终接收方(AI service)。你要和斯华对齐这个改动(见T3)。
12
端侧触发器透传给AI service → 云端你·端侧触发器
端侧触发器以端状态协议,把结果透传给AI service数据中心 → 上云进云端端状态/Context。
你负责"接收VLM结果 + 以端状态协议透传AI service"这一段(紫线)。AI service→云端是韩杰的(蓝线)。
13
云端触发器监听新SP的结果,检测变化你·云端触发器
你的云端触发器一直盯着那个KV槽位(answer_毯子任务),检测它变了没。
上一次: "毯子盖在宝宝身上" → 这一次: "毯子滑落在脚底" // 变了!
⚠ 这一步就是回传通道争议(T5):官方图画的是"结果绕到云端触发器监听"(现在这样);7.4研发提"原路返回,不绕"。要和丁彬西钺定守图还是改。
14
把结果返回给动态Advisor你·云端触发器→Advisor
你把"毯子滑落在脚底"这个变化,返回给动态Advisor,让它判断。
15
动态Advisor判断:满足退出条件吗?庆磊
动态Advisor判断"毯子滑落在脚底"= 毯子确实掉了 → 满足退出条件。"掉没掉"的判断在Advisor,不在你、不在VLM。
16
Advisor通知你删SP → 你还原AlwaysOn你·云端触发器
动态Advisor告诉你"任务完成,删掉观察毯子的SP"。你把cccc从合并SP里删掉,还原成原生SP(aaaa,bbbb)。
删除前: "prompt: aaaa, bbbb, cccc" 删除后: "prompt: aaaa, bbbb" // 还原,VLM下一轮不再看毯子
你负责"接收删除通知 + 删SP + 还原AlwaysOn"——这是拼SP的收尾,漏了VLM会一直白看毯子。
17
Advisor出建议 → Director出话术 → TTS庆磊/仁杰
动态Advisor出建议 → Director生成话术"宝宝的毯子掉了哦,要不要帮您调高空调?" → TTS播报。流程闭环。
🎯 看完这17步,"你要做什么"就具体了:运行时你负责第5-9步(接需求→判枚举描述→提槽→拼SP定槽位→下发→触发VLM)+ 第11-14步(接结果→透传→监听变化→返回Advisor)+ 第16步(删SP还原)。整条链路一半的步骤是你的动作。而你现在要做的"待办",本质就是把这些运行时动作背后的接口、协议、归属定下来——下面5件待办,就是把第7、8、11、13步的技术细节拍死。

二核心待办深度展开(拆到验收清单)

T2 · 端侧触发器怎么触发云端、怎么拼SPP0·评审核心·对应流程第7-9步
① 要解决什么(拆成4个子问题)
子问题为什么要定
a. 云端触发器用什么接口把拼好的SP下发给端侧触发器?push还是端侧pull?流程第8步。张荣缺席时点名你回答的正是这个
b. 端侧触发器接收的SP数据结构长什么样?(SP文本 + KV槽位定义 + 任务ID)流程第9步,端侧要照这个结构解析
c. 端侧触发器用什么接口调AIBox触发VLM推理?触发方式,复用现有VLM SDK还是新接口
d. 弱网/断网时,端侧触发器怎么处理已下发的SP?本地缓存吗?断网兜底是端侧触发器的核心价值
② 现状 vs 目标
子问题现状目标(你要定成什么)
a.下发接口官方图画了"云端触发器→端侧触发器"有这条线,但接口未定义定义接口:如 deploySP(task_id, sp_text, kv_slot),云端push
b.SP数据结构只有底部示意图(aaaa+cccc)定死JSON结构:{task_id, sp_content, kv_key, ttl}
c.调AIBoxVLM SDK现有"谁调谁拿"接口确认复用哪个接口 + 结果回哪(见T3)
d.断网无定:SP本地持久化,断网继续按SP推理,结果缓存待网络恢复上传
③ 具体做法
  1. 画一张时序图:云端触发器 →(下发SP接口)→ 端侧触发器 →(调AIBox接口)→ VLM →(结果)→ 端侧触发器 →(透传)→ AI service,每条箭头标接口名+数据结构
  2. 找丁彬确认c(调AIBox用哪个接口)、找斯华确认结果回传(见T3)
  3. 把这张时序图放进端侧触发器PRD,作为下周评审的核心页
端侧触发器PRD里的一张"端侧触发器↔云端触发器"完整时序图 + 4个接口的数据结构定义
✅ 完成 = 以下全部有明确答案并落进PRD:
a. 下发SP的接口名 + 方向(云端push/端侧pull) + 触发时机定义了
b. 端侧触发器接收的SP数据结构(JSON字段)定死了
c. 端侧触发器调AIBox的接口确认了(复用现有 or 新增),和丁彬对齐
d. 断网时SP本地缓存+结果续传的处理写清了
时序图能在评审上直接回答张荣缺席时挂起的"端侧触发器怎么触发云端/拼SP"
丁彬:端侧触发器调AIBox触发VLM,复用现有VLM SDK的哪个接口?结果同步返回还是异步回调?

T3 · 触发方≠接收方(你截图问的那个)P0·对应流程第11步
① 要解决什么

丁彬第③点:原来VLM SDK是"谁触发谁接收"(端侧触发器调VLM,结果就回给端侧触发器自己用)。现在变成:端侧触发器触发VLM,但结果不是给触发器自己,而是要透传给AI service上云。这个"触发方和接收方分离"的改动,落在你的端侧触发器和斯华的意图SDK之间。要解决3个子问题:

子问题说明
a. VLM结果回来,先回到端侧触发器,还是直接进AI service?官方文档:先回端侧触发器,再由它透传AI service
b. 斯华的意图SDK在这条链路的位置?它原来"返回调用方"的逻辑要改成什么?意图SDK包着VLM,原来谁调返回给谁
c. oneshot任务(如充电桩)和周期任务(如毯子),回传走不走同一条路?两类任务回传机制可能不同
② 现状 vs 目标
现状(原VLM SDK)目标(AIVA新要求)
触发端侧触发器调VLM不变:端侧触发器调VLM
结果回哪回给调用方(端侧触发器自己)回端侧触发器 → 透传AI service上云(给下游所有消费方,不只自己)
意图SDK谁调返回给谁要支持"结果导向AI service"这条新出口
③ 具体做法
  1. 拉斯华(意图SDK)+ AI service链路同学(赵衍/董均那边)开个小会
  2. 对齐上面a/b/c三个子问题,画出"触发方(端侧触发器)→VLM→结果→端侧触发器→AI service"的时序,标清意图SDK的改动点
  3. 把结论并进T2的时序图(它们是同一张图的两半)
"触发-接收分离"的时序图(可并入T2)+ 斯华意图SDK的改动点清单
✅ 完成 = 以下全部有结论:
a. 结果回传路径定了:VLM→端侧触发器→AI service(每跳协议标清)
b. 斯华意图SDK的具体改动点列出来了(原"返回调用方"→新"导向AI service")
c. oneshot vs 周期任务是否同一条回传路,明确了
三方(你/斯华/AI service)在文档或群里确认了这个时序,丁彬能看到
斯华:意图SDK现在"谁调返回给谁",要改成"结果导向AI service",具体改哪个回调?工作量多大?
赵衍/董均:AI service接收端侧触发器透传的结果,用什么协议?你们准备好接了吗?
T1 · 紫线指向(会上挂起,最快能关的)P0·对应流程第11-12步
① 要解决什么

董均缺席时问、张荣点名你回答:VLM上报的结果,紫线指向AI service数据中心,还是返回端侧触发器?——这个官方文档已经答了,你只需确认+回复,是5件里最快能关的。

③ 具体做法

群里/会上明确回复:"紫线=端侧触发器接收VLM结果→透传AI service,见文档上云通道章",挂起问题关闭
这条路径体现在T2时序图里
丁彬/张荣:确认我这个回复口径对不对——紫线就是"端侧触发器接收→透传AI service",对吧?

T2b · 黄红线优先级(荣姐要你确认)P0
① 要解决什么

荣姐挂你名下,要你确认黄红线是P0还是P1、什么时候能确认。庆磊的技术方案在等这个。

线整条你负责的部分不归你的

  3. 拿到结论回复荣姐+丁彬:黄红线X月能确认,紫线的依赖X时间点解
✅ 完成 =
嘉锋给了黄红线(触发器部分)P0/P1的明确结论
