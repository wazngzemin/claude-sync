## 三、P 级定义共识（本次明确）

| 级别 | 定义 |
|---|---|
| **P0** | 阻塞用户流程、需求无法闭环，且发生在**高频重要域** |
| **PE** | 同样阻塞、无法闭环，但发生在**低频域** |
| **P1 / PR** | 需求能闭环，但体验不够好（弱一些） |
| **P2 / P3** | 任务能完成，只是体验还不够好（更弱） |

> 判定红线：**只有不可用才算 bug**。不是四分的 case 不等于 bug；钓鱼式/极端表述的 case 不算问题。

## 四、Bug 分类体系（核心产出）

### A. 不是 Planner 的问题（要翻走 / 转需求 / 转其他模块）

| 类型 | 说明 | 处理 |
|---|---|---|
| **手动操作类** | 用户手点路线后语音"开始导航"，Planner 拿不到点击信息无从判断 | 单列一类"对话流程中含手动操作"，非 Planner |
| **下游承接能力不足 / 需协商** | Planner 改写合理（如"开始导航到浦东机场"），但下游工具/系统地图接不住 | 与下游、弋途协商；标准：只要 Planner 改写逻辑合理且在工具可承接范围内，就不算 Planner 问题 |
| **多工具同时输出** | 意图不支持同时出两个工具（如录音 + 图片 I） | 交互/工程承载能力问题，非 Planner |
| **UI / 界面问题** | 走联网无结构化 POI 卡片、导航卡片不消失出现空白页 | 客户端 / 结构化数据需求 |
| **FC 不对** | 有输出但 function call 错误 | 转 FC 团队，非 Planner |
| **低质量 bug** | 无 log ID、无描述、描述中文都读不懂 | 无法分析，转回 / 要求补 log |
| **钓鱼 case** | "屁股凉""车里像冰窖"等极端表述，本身是 good case | 不算问题，标注说明 |
| **ASR 问题** | 历史 ASR 识别错导致上下文错乱 | 非 Planner |
| **低频冷门功能** | 演示厅、一般录像/录音开关、蓝牙、泊车辅助、AI Light、智驾、行程规划 | 走干预平台 / 知识补充，普遍低优 |

### B. 是 Planner 的问题（按类型归类）

#### B1. 幻觉问题
- **典型**：QQ 音乐播周杰伦，实际放《搁浅》却播报《晴天》（工具 feedback=0 仅表成功，Planner 自行幻觉出歌名）；用户骗"我没有氛围灯"，Planner 被引导跟着承认。
- **解法**：工具返回具体信息（如歌名）来遏制幻觉；执行成功类只回"已开始播放"即可，不强行编造细节；长期需强化模型对自身能力的坚持。

#### B2. 推理问题（三个子类）
| 子类 | 典型 case | 解法 |
|---|---|---|
| **空间关系** | 车窗全开后"调高一点"未反向调低；遮阳帘"往前一点"理解错；主驾右侧=右后（1.8 怎么试都不行，纯模型）；副驾说"打开左边座椅通风"开成副驾；座椅"往前"调成往后 | 工具描述/知识里补空间关系说明（部分纯模型问题难解） |
| **端状态不清晰** | 风量档位读错（表格式端状态人都读不懂）；温度调节读错 | **改端状态结构**，从表格改为自然语言（"现在空调 5 档、XX 度"） |
| **上下文过度参考 / 端状态参考** | 基于历史对话回档位、不再调工具取实时值；上下文优先级压过端状态 | 端状态 vs 上下文权重，长期方向问题 |

#### B3. 不调工具（重复指令时基于历史直接答）
- 重复/相似 query 时 Planner 直接用历史对话作答，不调工具取最新结果。
- **实验结论**：在注意事项里加约束后仍不稳定，10 次里 3~4 次失败 → 约等于不可用；模糊/抽象要求放 SP 中间位置必然不稳。
- 候选解法：把"依赖外部信息（检索地图/媒体）时优先调工具"写入；或把这类工具的历史 answer 从对话历史里抹掉（偏 trick，争议大）。

#### B4. 话术 / 执行顺序
- "已将 XX 添加为途径点"在工具尚未返回成功时就播报（完成时态用错）；播报内容与工具返回顺序不一致。
- **解法**：与弋途协商方案 / 转需求；系统稳定后此类"先说我帮你做"反而是好体验。

#### B5. 改写问题
- "开始导航" → Planner 改写时带出具体地址（"开始导航到浦东机场"），下游意图接不住。
- "调高 10%" → 被理解成"加 10"，超过 100%。
- **解法**：把单目的地导航函数转向意图并向弋途提训练需求；或在干预平台直接拦截（"开始导航"无论如何理解为开启、不需要目的地）。

## 五、重点域案例归档

### 导航线
- **"开始导航到 XX 地址"**：Planner 改写超出工具承接范围 → 转弋途协商 / 训练；可干预兜底。
- **手点路线后"开始导航"**：手动操作类，Planner 无信息，非 Planner。
- **行程卡片 / 行程规划（西安七日游、兵马俑等）**：低优暂不解；疑似产品间未拉起（行程管理模块），与 Director 无关；个别属用户常识问题（如兵马俑游玩时长）非 bug。
- **POI vs 联网（web_search）选择标准**（参考豆包 POI 查询 与 web_search 意图定义）：
  - 有"附近"/指定地点诉求 → 必须出 POI（导航）。
  - "上海 top1 的烤鸭/炸酱面"等极致排名 → 必须走联网。
  - 区级以上的模糊"找好吃的" → 联网 OK；精确到路段/附近评分高 → POI 更好。
  - **无明确地点诉求时，导航和联网都能闭环 → 不算 P0/PE。**

### 车控线（现存约 36~40 条，真正问题集中在推理）
- **车窗调高一点（已全开）应反向调低** → 推理错误，**高优、明天就解**。
- **车窗"调高 10%"→ 加 10** → 改写/推理。
- **右车窗打开 10% 却开了右后** → 第二代改写 + 纯模型空间问题。
- **遮阳帘"往前一点"→ 直接调到 20%（关 80%）** → 空间关系，可加知识改善。
- **风量档位播报错（端状态读不懂）** → 改端状态结构。
- **屁股凉/冰窖 → 开方向盘加热/座椅加热**：钓鱼 case，本身是 good case，**不算问题**。
- **座椅通风 2→3→1 反复横跳**：用户神经病式测试，非问题。
- **吹脚吹窗（没关吹脸模式）**：改写没问题，转需求（空调可同时设吹窗/吹人/吹脚）。
- **AI Light / 氛围灯颜色调节**：判定为 AI Light / DU 侧，非 Planner。

## 六、解法方向汇总

1. **Planner 侧**：补工具描述、注意事项、知识（空间关系、车载 APP list）。
2. **端状态**：改结构，人都读不懂的表格 → 自然语言表述。
3. **下游 / 弋途**：协商方案、转需求、SFT 增强承接（以 Planner 为中心改造下游、扩大兼容性）。
4. **干预平台**：低频冷门功能（演示厅、录音开关、蓝牙、泊车辅助、AI Light）直接干预。
5. **动态示例**：高频 case 补 2 个示例兜底。
6. **工具返回**：返回具体信息以遏制幻觉。
