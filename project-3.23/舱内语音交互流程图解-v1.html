01

唤醒与唤醒抑制

流程说明

唤醒是用户与豆包交互的第一道门。用户说出唤醒词"豆包豆包"后，端侧唤醒引擎在 350ms 内完成检测并激活 AI 伙伴。

但并非所有包含"豆包"的声音都应触发唤醒。系统需要识别并抑制两类误唤醒源：

  • 车外人声抑制：车外有人喊"豆包豆包"时，通过声源隔离 + 舱内摄像头视觉感知，判定声源在车外 → 抑制唤醒，HMI 显示 Toast "车外人唤醒已抑制"
  • 电子设备音频抑制：手机/平板播放的视频或音乐中包含唤醒词时，系统需识别这类音源并拒绝唤醒
供应商当前不支持唤醒抑制能力，此功能为 TBD 状态。

02

主驾免唤醒

流程说明

主驾免唤醒是全时免唤醒的前身/子集。当开关打开后，主驾音区的用户无需说唤醒词即可直接下达指令。核心判断逻辑如下：

  • 自动开启条件：当 Watcher/端状态模块检测到车内仅主驾一人时，默认开启免唤醒
  • 手持电话抑制：舱内摄像头识别到主驾正在手持电话通话 → 强制拒识，不把音频送云端
  • 车外人/电子设备抑制：与唤醒抑制逻辑一致，需将其判定为非用户对话并 Toast 提示
  • 退出语音 ≠ 关闭免唤醒：用户说"退出语音""别播报"只打断当前 TTS，豆包仍继续监听。要完全退出需手动关闭开关
  • 被抢占：主驾免唤醒交互中，其他音区用唤醒词唤醒 → 打断主驾交互，转向响应新音区

03

降噪与回声消除

流程说明

在车内复杂声学环境下，用户语音需经过两级处理后才能被 ASR 识别：

  • 噪声降噪：抑制车辆机械噪音（空调运转、轮胎噪声、风雨声）和电子设备声音（手机视频/音乐）。系统根据不同噪声强度动态调节抑制阈值。每个座位区域独立配置高灵敏度麦克风。
  • 回声消除（AEC）：车载扬声器播放的音乐、导航语音、电话通话等音频会被麦克风再次采集形成回声。系统实时采集 4 路参考信号（多个扬声器输出），从麦克风信号中精准抵消回声，覆盖从最小音量到最大音量的全范围。

两级处理总延迟要求 ≤50ms，处理后信噪比 ≥20dB。

04

音区识别

流程说明

音区识别让系统知道"谁在说话"以及"从哪个座位说的"。核心能力：

  • 基础音区划分：5 座车 = 5 个音区，6 座车 = 6 个音区。用户语音指令到达后，系统输出"主驾/副驾/二排左/二排右/二排中"等位置信息
  • 声源定位精度：定位误差 ≤±20cm，可以区分同音区内不同乘客
  • 多模态融合：不仅靠麦克风阵列的声学特征，还结合舱内摄像头识别人体位置（锥形特征），二者融合做更精准的收音和定位
  • 特殊场景：座椅靠背放倒 >120° 时，不能把主驾误判为二排左侧，仍需正确识别为前排左侧音区
  • 音区开关：用户可单独关闭某个音区的交互；被关闭的音区有人说话时 Toast "该音区已关闭"

语音识别（离在线仲裁）

流程说明

语音识别是将用户声音转文字的核心环节。系统采用 离在线并行 + 仲裁 架构：

  • 在线 ASR：音频送云端，走云云对接接入供应商/字节内部在线 ASR。针对车载场景做定向优化——支持车辆专有名词、通讯录人名、POI 地名的热词增强（≥3000 热词）。还支持地域 POI 增强（同一个发音，浙江识别为"云和"，黑龙江识别为"云河"）
  • 离线 ASR：端侧本地运行，支持中文普通话和中英混合，兼容口音
  • 仲裁策略：不论网络好坏，离线场景用离线 ASR，在线场景用在线 ASR。上屏时先流式展示离线结果（速度快），最终用在线 ASR 的 final 结果刷新。断网时完全使用离线结果
  • 方言二遍识别：第一遍追求低延迟先上屏（普通话为主），第二遍侧重高精度修正方言/强口音（粤语字错率约 7%，口音约 10%）
  • 识别纠错：不需要修改 ASR 输出，而是由后续模型根据上下文理解最终语义（"虹煤路"→"虹梅路"）

声纹识别

流程说明

声纹识别让豆包知道"是谁在说话"，而不仅仅是"从哪里在说话"。分为注册和应用两个阶段：

  • 显性注册：用户在界面主动录入，说 3 次唤醒词+指令（如"豆包豆包，空调调到 23 度"），系统提取声纹特征。注册上限 5 个声纹。注册后设置昵称/年龄/性别
  • 隐性注册(TBD)：锁定某音区，该音区语音交互 ≥10 次后，豆包主动询问"你的声音很好听诶，可以告诉我你的名字么？"，用户回答昵称后绑定声纹
  • 应用场景：
    • 声纹识人：唤醒时叫出用户昵称（"你好晓晓~"）
    • Context 丰富：将声纹属性（昵称/年龄/性别）传给 Context 模块，做个性化交互（如识别为儿童 → 更简短的对话风格）
    • 安全校验：识别为儿童 → 禁止操作车窗/车门/驾驶类等安全功能
    • 车外语音安全校验（跨舱）
  • 隐私：声纹特征仅存储在本地，录入时需用户授权

全时免唤醒交互

流程说明

全时免唤醒是舱内语音的终极形态——所有音区的所有乘客都无需唤醒词，随时随地直接和豆包说话。

核心机制分三层：

  • 持续监听 + 拒识：豆包始终在听，但大部分话是人人对话（聊天/唱歌/自言自语）。端侧拒识模型区分人机对话和人人对话，只召回真正对豆包说的话
  • 强召回：用户语句中出现"豆包"或指令句式时，不管拒识模型怎么判，强制召回为人机对话
  • 强拒识：摄像头识别到用户正在手持电话通话 → 该音区所有语音强拒识，不上云

上下文继承机制：豆包将被拒识的对话做话题分类和总结（每 5 轮一次），当后续有正例召回时，将前 5 个话题总结作为上文补充，辅助理解用户意图。

激活后进入连续对话模式（30 秒超时）。30 秒内全是无效文本 → 退回免唤醒未激活状态；有有效指令 → 重新计时 30 秒。

多音区并发交互

流程说明

一台车只有一个豆包伙伴，它可以同时听到所有人说话，但同时只能回答一个人。当多个音区同时或先后发起指令时，系统需做冲突仲裁和排队调度。

核心设计原则：能不播报就不播报。执行结果用视觉反馈（图标/卡片）就够的，不额外占用 TTS 通道。必须播报的，按紧急程度排队或插队。

系统将多音区并发拆分为两大场景：

  • 同时说：多人的 query 还没分发出去，合并文本给 Planner 统一处理（"主驾驶说XXX，副驾驶说XXX"）
  • 任务执行中：某个任务正在执行/播报，新指令到达 → 根据 5 种类型决定打断/排队/合并

09

免唤醒上云策略

流程说明

全时免唤醒意味着用户随时可能对豆包说话，如果所有音频都送云端，会产生巨大的流量和 token 消耗。因此端侧需做预拦截：

  • 端侧预处理：端侧模型先做一轮判断——能处理的（明确指令/明确拒识）直接在端侧完成，不上云
  • 端侧无法处理的 query：需要上云。上云有两条路径同时走：
    • 文本上云：端侧 ASR 识别出的文本作为 soft VAD 结果，提前触发云端 LLM 请求（但只跑到拒识意图模块，不做全量推理）
    • 音频流式上云：用户语音音频流式传到云端，云端做在线 ASR 得到 hard VAD 结果
  • 端云 ASR 对比：soft VAD 和 hard VAD 结果一致 → 不重新请求；不一致 → 用 hard VAD 结果重新请求云端 LLM
  • 激活后：免唤醒激活进入连续对话后，后续指令通常是继续和豆包对话，此时端云并行同步处理以保证速度和准确性
注意：S08 未完成通过座舱云接入在线 ASR，需要将端上获取到的在线 ASR 上云做端云 ASR 对比。
