0.3 三类用户意图的信号特征与正确处理

执行型
信号词：打开/关闭/调高/调低/开启/设为/调到

示例："打开空调""关闭车窗""温度调到24度"

→ 正确：调用 vehicle_basic_control

✗ 错误：用QA知识做文字回答

查询型
信号词：XX多少/当前XX/剩余XX/XX正常吗

示例："现在时速多少""剩余电量多少""胎压正常吗"

→ 正确：调用 search_vehicle_status_info

✗ 错误：回答"去仪表盘看"

咨询型
信号词：怎么用/如何设置/步骤/方法/咋调

示例："定速巡航怎么用""雨刮器咋调""怎么设置自动大灯"

→ 正确：参考QA知识回答操作步骤

✗ 错误：调工具执行（矫枉过正）

0.4 泽民的两个操作杠杆 — 所有方案的基础

杠杆一：{{knowledge}} 内容注入
能力：在 {{knowledge}} 中注入任意内容（知识/规则/示例），Director每轮推理时可见
权重：SP引导语定位为"灵活参考"（参考级，非指令级）
触发：可配置为始终加载或条件触发
限制：不是"硬约束"，Director可能不完全遵循
杠杆二：QA数据内容
能力：修改现有QA对的Q/A文本，或新增QA对
影响：直接改变检索结果 → 改变Director看到的context内容
触发：全量触发（每轮都检索top3）
限制：需要数据入库流程；向量库更新可能有延迟

方案E — Knowledge注入few-shot示例（推荐首选 · 泽民自主完成）

在 {{knowledge}} 中注入具体的输入输出示例，通过模式匹配引导Director正确决策

E.1 机制原理图 — 方案E如何改变Director的决策过程

对比：没有方案E时 vs 有方案E时，Director的决策路径差异
❌ 当前（无方案E）
1. Director看到 {{car_function_simple_qa}} 中有操作步骤
2. 引导语只说"可能会用上"
3. 没有任何参考示例告诉它怎么处理
4. Director默认用知识做文字回答
5. → 错误：没调工具
VS
✅ 有方案E后
1. Director看到 {{car_function_simple_qa}} 中有操作步骤
2. 同时看到 {{knowledge}} 中有处理示例
3. 模式匹配：当前case像示例1（执行型）
4. 跟着示例1做：调用 vehicle_basic_control
5. → 正确：调了工具
核心原理：LLM的in-context learning = 模式匹配
当context中有具体的输入→输出示例时，LLM会自动做模式匹配：找到与当前输入最相似的示例，然后模仿该示例的输出。这比理解抽象规则更简单、更可靠。

与方案B（规则版）的区别
方案B注入规则："执行指令 → 调工具" → Director需要：理解规则 → 判断当前case是否属于"执行指令" → 决定调哪个工具（三步推理）
方案E注入示例："用户说'打开空调' → 调vehicle_basic_control" → Director只需要：模式匹配（一步）

条目正文内容（复制这段）

【车辆功能问答知识正确使用指南】 你的context中可能包含「基础车辆功能问答」部分的参考知识（来自车辆手册的检索结果）。 这些知识描述了功能的手动操作方法，但不代表你应该直接用它们回答用户。 请根据用户的真实意图选择正确的处理方式。以下是具体示例： ━━━ 示例1：执行指令 → 调工具执行（不用知识回答）━━━ 用户说："打开空调" 此时「基础车辆功能问答」中有："空调可通过中控屏底部空调按钮开启，点击后选择温度和风量" ✅ 正确做法：调用 vehicle_basic_control({"query":"打开空调"})，说"好的，正在为你打开空调" ✗ 错误做法：回答"空调可以通过中控屏底部的空调按钮开启，点击后..." 判断依据："打开"是执行动作词 → 用户要的是系统帮他执行，不是教他手动操作 ━━━ 示例2：实时查询 → 调工具查数据（不用知识回答）━━━ 用户说："现在时速多少" 此时「基础车辆功能问答」中有："车速信息是仪表屏常显信息，非P挡时可直接查看" ✅ 正确做法：调用 search_vehicle_status_info({"query":"查询当前车速"}) ✗ 错误做法：回答"车速信息可以在仪表屏上查看" 判断依据："现在XX多少"是问实时数值 → 用户要具体数字，不是告诉他在哪看 ━━━ 示例3：操作咨询 → 用知识回答（这是知识的正确使用场景）━━━ 用户说："定速巡航怎么用" 此时「基础车辆功能问答」中有："定速巡航开启方法：进入车辆设置-辅助驾驶，打开自适应巡航辅助..." ✅ 正确做法：参考知识回答"定速巡航可以这样开：进入车辆设置，找到辅助驾驶..." 判断依据："怎么用"是咨询操作方法 → 知识中的步骤正是用户想要的答案 ━━━ 示例4：工具失败 → 知识兜底 ━━━ 用户说"打开定速巡航" → 调用vehicle_basic_control → 返回：执行失败 ✅ 正确做法：告知"暂时无法语音开启，你可以手动操作：进入车辆设置的辅助驾驶..." 判断依据：工具失败后，知识中的手动步骤就是最好的兜底方案

设计要点说明（不需要复制，帮你理解为什么这么写）

  • 四个示例覆盖四种场景：执行/查询/咨询/兜底。Director面对任何case都能找到最近的模式。
  • 每个示例都有"正确做法"和"错误做法"：正反对比是LLM行为引导中效果最强的格式。Director不仅知道该怎么做，还知道不该怎么做。
  • 每个示例都有"判断依据"：告诉Director用什么信号判断意图类型。"打开"=执行、"多少"=查询、"怎么"=咨询。
  • 示例中的场景刻意与车书QA的典型内容匹配：空调、定速巡航、车速 —— 这些是最容易出问题的高频场景。

5

记录：方案E单独修复率 = XX%。

方案D — 竞争性QA检索对（检索层解决 · 与E叠加效果最佳）

新增执行型QA对，利用向量检索排序让"系统直接执行"的answer占据top-1位置

D.1 机制原理图 — 方案D如何改变检索结果

关键原理：向量检索按相似度排序。"打开定速巡航"与"打开定速巡航"的匹配度 >> "打开定速巡航"与"定速巡航怎么开"的匹配度。
新增的QA对与执行型query完全匹配，会自动抢占top-1位。
❌ 当前检索结果
用户query："打开定速巡航"
Top-1: Q"定速巡航怎么开"
→ A"进入车辆设置-辅助驾驶..."
Director看到操作步骤 → 直接回答
VS
✅ 方案D后的检索结果
用户query："打开定速巡航"
Top-1（新增）: Q"打开定速巡航"
→ A"该操作由系统语音直接执行"
Top-2（原有）: Q"定速巡航怎么开"
→ A"进入车辆设置..."
Director首先看到"系统直接执行" → 调工具

D.2 为什么咨询型query不会被干扰 — 向量距离分析

新增的QA对只会被执行型query检索到，不会被咨询型query检索到。原因：向量语义距离。
执行型query的检索结果
用户说："打开定速巡航"

相似度排序：
1. "打开定速巡航"（新增QA）→ 相似度 0.98（几乎完全匹配）
2. "定速巡航怎么开"（原有QA）→ 相似度 0.72（关键词匹配）
3. "巡航功能介绍"（原有QA）→ 相似度 0.55

结论：新增QA稳居Top-1
咨询型query的检索结果
用户说："定速巡航怎么用"

相似度排序：
1. "定速巡航怎么开"（原有QA）→ 相似度 0.91（语义高度匹配）
2. "巡航功能介绍"（原有QA）→ 相似度 0.65
3. "打开定速巡航"（新增QA）→ 相似度 0.58（"怎么用"≠"打开"）

结论：原有QA仍是Top-1，新增QA被推到底
为什么方案A做不到这一点？方案A修改了原有QA的answer，所以不管是执行型还是咨询型query，检索到的都是修改后的answer（带前缀）。咨询型用户看到"该功能支持语音控制..."的前缀会感到多余。而方案D的新增QA和原有QA是独立的——各自被各自类型的query检索到，互不干扰。

D.5 效果预估与风险分析

优势（5条）

  • 从检索排序层面解决：不是教Director"怎么处理知识"，而是直接控制它"看到什么知识"。上游治理比下游修补更彻底。
  • 现有数据完全不动：只新增，不修改。删掉新QA就完全回滚。
  • 咨询型天然免疫：向量语义距离保证了新QA不会被咨询型query检索到。
  • 信号最直接：Director看到的top-1就是"由系统执行"，不需要"理解规则"或"解读前缀"。
  • 可脚本化生成：模板固定，映射表+脚本10分钟生成。

劣势（4条）

  • 需要入库流程：泽民可能没有QA数据的直接入库权限。需确认。
  • 长尾说法覆盖不全："帮我把那个巡航开了"这种口语化表达可能不会匹配到新QA。
    → 缓解：由方案E的示例兜底。
  • 向量库容量/质量影响：新增56条QA，确认不会影响其他功能的检索。
  • 新增QA的answer较简短：Director可能觉得top-1的answer"太短了，不够回答用户"，转而看top-2的详细步骤。
    → 缓解：方案E的示例明确说了"不要用步骤回答"。
