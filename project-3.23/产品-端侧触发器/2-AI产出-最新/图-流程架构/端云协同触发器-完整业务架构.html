01触发器一句话定位 + 边界

触发器是什么：触发器是 Driver Agent 系统里"事件订阅 + 条件匹配 + 动作回调"的中间件。它不在主链路上（用户说话→Planner→执行不经过触发器），但所有"持续/条件/定时任务"必须靠它。

🟡 触发器吃进来的

  • ① 后台配置的静态预设事件
  • ② Advisor 注册的订阅条件
  • ③ Planner 注册的条件任务（经条件解析模型）
  • ④ 端信号（车身/外设/视觉）
  • ⑤ 定时器（端侧倒计时）

🔵 触发器内部做的事

  • ① 接收注册 + 持久化
  • ② 云→端同步条件 + 策略
  • ③ 信号变化时做两级条件判断
  • ④ 命中后做优先级仲裁 + 互斥检查
  • ⑤ 路由到 notify or direct
  • ⑥ 处理冷却、超时、抢占恢复
  • ⑦ 回传执行状态

🔴 触发器吐出去的

  • ① 给 Planner 触发推理（S8）
  • ② 给 Advisor 条件命中通知
  • ③ 给端侧工程执行（车控/TTS/媒体）

触发器明确不做的事（边界）

触发器不负责归谁
用户意图理解 / 口语解析Planner / Director
条件任务的语义解析（"任意位置" → 4 个具体位置）条件解析模型（吴仁杰）
Advisor 的话术决策Advisor 内部
VLM 内部推理端侧 VLM 模块
信号原始采集（读 CAN/蓝牙）端信号库
记忆 / Context 存储Context 模块
端云仲裁（离线/在线结果择优）端云仲裁模块（左晨）
用户身份识别身份识别模块

🎯 上汽 DNA 6/10 SOP 范围圈定（必读）

上汽 DNA SOP 主流程：用户说话 → ASR → Planner → Agent/Tool 执行 → TTS 反馈。
触发器在主流程中只承担「Advisor 事件订阅」——命中后 notify 给 Advisor 即可。
"直接执行 / 策略路由 / 协议改造 / VLM / 动态语义"全部是优化项，主流程不需要，统统推到赛力斯阶段。

📋 7 个候选需求点的分级判断

#需求点上汽 SOP 必要性影响主流程决策
1 端侧触发器直接接执行能力 🔴 低 不影响 砍 → 推赛力斯。主流程靠云端推理+执行，弱网下豆包整体不可用，单独优化触发器没意义
2 端侧策略模块（事件路由 notify/direct） 🔴 低 不影响 砍 → 推赛力斯。不做 1，策略路由只有 notify 一种，不需要单独模块
3 端侧接受云端下发的执行策略 🔴 低 不影响 砍 → 推赛力斯。跟 1、2 配套，云端下发的目的是支持端上直接执行
4 定时器做在端上（+时间信息透出） 🟡 看需求 部分 看上汽 DNA 用户场景。如果用户有"10 分钟后..."这类需求 → 必做；如果没有 → 推迟。需要找廖婉婷确认
5 云端/端侧事件分工（语义/联网在云，端信号/定时器在端） 🔴 架构 间接 不是独立任务。DNA 阶段事件都从端信号生成；云上生成的事件（语义/联网）是赛力斯阶段能力
6 "任一位置"协议表达 🔴 低 不影响 砍协议改造 → 用条件解析模型展开为 4 条具体位置规则。短期方案就够用
7 上汽协议没对清楚 🟢 高（前置） 前置阻塞 必须立刻做。找韩天亮拿最新协议表，对不清楚则所有需求都建在沙堆上
上汽 6/10 SOP 触发器真实最小范围（结论）：
  1. ✅ Advisor 注册接入（益红主动服务 6 个点）
  2. ✅ Planner 条件任务注册接入（经条件解析模型）
  3. ✅ 端信号事件触发（车身信号为主）
  4. ✅ 命中后 notify → Advisor / Planner（不做 direct 路径）
  5. ✅ 上汽协议对清楚（韩天亮的协议表）
  6. ❓ 定时器（看上汽是否有定时任务需求 → 找廖婉婷）
其他能力（弱网直接执行、策略路由、云端策略下发、协议改造、VLM 调度、机器人事件、动态语义、模型类事件触发）全部推到赛力斯阶段。
这个结论意味着 6/10 SOP 你只需要做对外承诺的 4 件事：
① 触发器能接 Advisor 订阅 → ② 触发器能接 Planner 条件任务 → ③ 端信号能触发规则 → ④ 命中后能 notify 给云端业务方。
这 4 件事的复杂度，对应到原来你担心的"整个系统"，只是其中一小块。范围一旦圈住，6/10 节点就不再恐怖。

03云端触发器内部子模块详解

🔧 C1 · 平台配置

云端S7 已实现
维护事件库的元数据：哪些事件可被订阅、每个事件的条件 schema、支持的 action 类型。
输入：运营/PM 在配置后台填表（韩天亮维护）
输出：可订阅事件列表给 C2

📥 C2 · 注册中心

云端S7 已实现
接收所有订阅请求（来自后台配置、Advisor、Planner）。校验格式 → 校验条件枚举值合法 → 校验是否冲突 → 持久化。
输入：注册请求 JSON
输出：注册成功/失败回执 + 内部存储

🧠 C3 · 条件任务解析

云端S8 在做
和"条件解析模型"协作（吴仁杰 own 这个模型），把 Planner 给的口语化条件任务解析成结构化规则。"任意位置车门打开"→ 4 条位置具体规则就在这里展开。
输入：Planner 的条件任务
输出：结构化条件 JSON

⚖️ C4 · 任务管理

云端S8 在做
条件任务上限管控（Wiki 说明硬上限 6 个）；多任务冲突仲裁（10分钟后开窗 vs 5分钟后关窗）；任务生命周期管理（创建/激活/触发/完成/过期/撤回）。
输入：注册的条件任务
输出：超限返回 error 给 Planner

🤖 C5 · 模型类事件触发

云端S9-S10 演进
端上没模型能力，所以这类事件必须在云端生成：
• VLM 语义匹配（"宝宝在哭"匹配到"宝宝哭闹"规则）
• 动态语义事件（用户对话中识别"心情不爽"）
• 外部数据事件（股价、天气、节日）
输入：VLM 描述/对话语义/外部 API
输出：触发事件 → C2

📤 C6 · 同步下发

云端S7 已实现
把云端注册好的规则下发到端侧。下发内容：触发条件 + action 策略 + 优先级 + 冷却参数。支持热更新（云端改了 → 推到端 → 端侧热加载）。
输入：C4 的最终规则
输出：JSON → 同步通道 → 端侧 E1

📨 C7 · 中转 notify

云端S8 在做
接收端侧触发后上报的 notify。根据规则注册方分发：给 Planner（启动推理）/ 给 Advisor（启动建议）。这是触发器对外输出 S8 路径的出口。
输入：端侧 notify
输出：分发给 Planner / Advisor

04端侧触发器内部子模块详解

📥 E1 · 平台配置同步

端侧S7 已实现
接收云端下发的规则，本地持久化（断网仍可执行）。启动时全量拉取，运行中增量订阅。车机重启后从本地恢复。
输入：同步通道
输出：规则给 E2、任务给 E3

🔍 E2 · 触发条件管理

端侧S7 已实现
规则库管理 + 两级条件判断：
• 第一级：运行时段（行车中/停车时/上车前/离车后）
• 第二级：触发条件（具体信号匹配）
第一级不满足时，不订阅第二级信号——省算力。
输入：信号库变化通知
输出：命中事件给 E4

📦 E3 · 触发任务管理

端侧S7 已实现
任务队列 + 状态跟踪。定时任务集成在这里（10 分钟倒计时）。每个任务的状态：pending → running → success / timeout / interrupted。
输入：注册任务 + 定时器到达
输出：任务执行请求给 E4

⚖️ E4 · 触发判断（核心引擎）

端侧S7 已实现
核心判断逻辑：

🔑 触发流程的 5 个关键判断点（E4 内部）

  1. 信号快照对齐 500ms：CAN 信号到达有先后（档位变化先到、车门解锁后到），等 500ms 内信号都齐了再判断，防漏触发
  2. 两级条件判断：先看运行时段（停车时/行车中），再看触发条件。第一级不满足直接退出，省算力
  3. P0-P4 优先级仲裁：同时命中多个事件时，P0 抢占 P1-P4；同优先级 FIFO
  4. 互斥组检查：同摄像头组同时只能跑一个 VLM 任务
  5. 冷却去重：60s 内同一规则不重复触发；P0 安全任务不冷却
