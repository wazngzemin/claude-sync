2. 背景与目标

2.1 背景

触发器现在不认识"导航目的地""剩余距离"这些概念，导致一类"快到某地就主动服务"的需求一个都做不了。这些导航信号在《触发器条件清单》里本来就有定义（序号 119~133），只是之前被删了 / 没开放。本期是把它们作为新增重新开放，并设计出"接近/到达/离开"的判定逻辑——不是从零开发。

2.2 目标（本期要达成）

  • 让触发器能取到导航状态、目的地/途经点位置（地址 + 剩余距离）、收藏地址等信号；
  • 产出 发起导航 / 接近 / 到达 / 离开 / 目的地变更 / 回家去公司 / 接近途经点 等事件，供动态 advisor（弋途）订阅；
  • 解决核心难点："接近/到达/离开"不是简单的"距离 < 阈值"，而是 one shot 过程判定（详见 §8）；
  • （v1.1）开放座舱/车况信号（座位占用、空调、电量、续航、胎压、媒体播放/音量、路过 POI、围栏），覆盖佳杰必过集 19 个未支持 case，建立 4 型边界 case 全集（阈值穿越/状态变更/空间 POI/事件型）和单位归一标准（详见 §6.5/§7/§17）。

2.4 非目标与能力边界（分两档，别一刀切"做不了"）

场景靠什么触发能否做归属

3. 术语

术语含义
one shot条件满足时只触发一次，之后即使持续满足也不再触发（门铃式，非警报器式）
POI 状态机给每个被监控目的地/途经点维护的状态：远 → 已接近 → 已到达 → 已离开
接近阈值 / 到达阈值 / 离开阈值判定接近/到达/离开的距离门限（默认 500m / 50m / 500m，按场景可配）
缓冲死区（dead_zone）为防数值抖动设的双阈值区间。距离：<480m 算接近、>520m 算离开，中间不动作；座舱信号同理（如电量 48/52、续航 98/102、音量 80/82）
动态 advisor弋途（施佳杰）维护的主动服务消费方，订阅触发器事件后产 advice
导航剩余距离沿导航路线的剩余距离，非直线距离
单位归一端上报带单位字符串（"200m""50%""1.8bar"），引擎按 DSL unit 字段剥尾缀取纯数值再比较。固定制式：m(距离)、%(电量/音量)、km(续航)、bar(胎压)
trigger_source事件触发来源标记：user（用户主动操作）/ system（系统自动恢复），用于区分切歌、播放器重启等场景
changed_to状态变更运算符，仅在值从旧态跳到目标态时触发一次（如 off→on），持续保持不重复触发；需持续 ≥2s 才认（防瞬时闪变）
4 型边界 casev1.1 归纳的 4 种易错边界类型：① 阈值穿越（数值抖动）② 状态变更（语义模糊）③ 空间 POI（路过/靠近/围栏）④ 事件型（切歌/暂停/结束）

4. 需求范围

4.1 导航信号范围（6/16 会议拍板）

档内容说明
✅ 做导航状态 + 目的地/途经点位置（地址、剩余距离）+ 收藏地址用「目的地位置 + 剩余距离」实现接近/到达/离开
❓ 待评估剩余时长（duration）、到达时间（time）等时间类不是砍——先确认端上报格式能否转统一单位 + 是否真有必要（见 §7/§11）
❌ 真做不了完全无目标的即兴兜风（"随便讲讲路过的楼"）无目标可盯，转 Director 持续任务 / VLM。注：能预置 POI / 设围栏的景点 → 走电子围栏（需10/11），仍属触发器

4.2 座舱/车况信号范围（v1.1 · 6/23 佳杰必过集扩入）

档内容说明
✅ 做座位占用、空调电源/风量、电池电量、剩余续航、媒体播放/音量、胎压（10 类信号，详见 §6.5）复用同一套状态机 + dead_zone + one_shot 机制；覆盖佳杰 19 个未支持 case
✅ 做路过 POI（poi.pass_by）：经过不停 ≠ 靠近新增【路过】事件，需路径与围栏相交判定（§8.6 序 13）
❓ 待确认无导航围栏兜底（poi.geofence）未开导航时靠近/离开收藏地址，端侧是否提供围栏能力（§11 #16）
⚠ 前置10 类信号编号锁版 + 单位归一口径编号 200+ 为占位，待韩天亮锁版替换正式编号；单位标准见 §7

5.2 单次判定时序（以"接近公司"为例）

  • ① 上报：端上每隔几秒上报 nav_status=导航中, distance=15000(m) → 写端侧信号库 → 同步云端
  • ② 扫描：云端触发器每 5s 扫一遍该 POI 的 one_shot 条件，读取最新 distance
  • ③ 判定：套 §8 状态机——distance 从 15000 递减，>接近阈值时不触发
  • ④ 命中：某帧 distance 首次 ≤480m（含死区）→ 触发【接近】，状态 远→已接近（自锁，本程不再触发接近）
  • ⑤ 透出：事件 Callback 给弋途 advisor
  • ⑥ 服务：advisor 产 advice"提醒拿咖啡" → Director 播报

地址比对标准："导航回家/去公司" = 目的地 address 等于 收藏地址 address。两边必须同源、同格式（建议统一以经纬度做主判、字符串做辅），否则等值比对失败。

单位归一标准（v1.1 新增·座舱/车况信号）：端上报多为带单位字符串（"50%""1.8bar""200m"），引擎一律按 DSL unit 字段剥尾缀取纯数值再比较。固定：%(电量/音量)、km(续航，mi÷1.609)、bar(胎压，kpa÷100、psi×0.0689)、m(距离·已有)。数值型阈值均配 dead_zone 防穿越抖动（沿用 §8.4 死区思路，±2~10%）。

8. 核心判定逻辑：接近 / 到达 / 离开（重点）

8.1 为什么不能简单配"距离 < 阈值"

用一个完整场景看（周一早 8:00 从家导航去公司，出发 15km，做"距公司 500m 提醒拿咖啡"）：

时刻剩余距离"距离<500"满足?实际该不该提醒
8:00–8:2515km → 501m不满足不提醒 ✅
8:25:30500m（第一次进）满足🔔 提醒一次 ✅
8:25:31 起499、498…50m一直满足！不该再提醒 ❌ 崩在这
💥 直接配"距离<500"，从 499m 到公司每帧都满足，触发器无记忆不知道提醒过了 → 连续触发几十上百次刷屏。所以必须是「从远到近第一次跨过 500m → 触发一次 → 然后闭嘴」= one shot，这正是刘杨说"现有协议做不了"的点。

8.2 解法：给每个 POI 维护一个状态机

远
→距离≤接近阈值(死区<480m)触发【接近】one shot
已接近
→距离≤到达阈值(50m) 且 档位变更为P触发【到达】
已到达
→距离>离开阈值(死区>520m)触发【离开】
已离开

关键：状态一旦从「远」翻到「已接近」，接近事件就再也不会触发——这就是用"记状态"换来 one shot 的机制。

8.3 判定伪代码

state = 该POI当前状态          # 初始：远
d     = 当前剩余距离（沿路线·米）

if state == 远  and  d <= 接近阈值(500m):
    触发【接近】;  state = 已接近   # ← one shot 关键：状态一变，接近不再触发
if state == 已接近  and  d <= 到达阈值(50m)  and  档位 变更为 P:
    触发【到达】;  state = 已到达
if state in (已接近, 已到达)  and  d > 离开阈值(500m):
    触发【离开】;  state = 已离开

10. 方案 B 落地详解（端上算好事件，触发器只收事件）

10.1 决策结论

✅ 选定方案 B：端侧算好「接近/到达/离开」事件后上报，触发器引擎不改状态机、只收事件匹配场景。原因：A 方案改引擎工期顶不上 630。策略：本期 B 兜底上量产，A 排下版统一。
方案做法优点缺点
A · 触发器侧加过程判定状态机放触发器，引擎存每个 POI 的"远/已接近/已到达"状态统一、所有距离类需求复用；端上不改引擎要存状态、改动大、工期顶不上 630

# ============================================================
# 端侧接近事件计算模块（每秒执行一次，distance 变化驱动）
# ============================================================

# 全局：每个 POI 一张状态卡片（内存常驻）
poi_states = {}   # {poi_id: {state:"远", last_d:None}}

def on_distance_update(signals):
    # ① 前置检查：必须在导航中且有目的地
    if signals.status != "导航中" or not signals.address:
        return   # 无导航 → 不处理（围栏场景走 poi.geofence）

    poi_id = hash(signals.address + signals.lat_lng)
    card   = poi_states.get(poi_id, {state:"远", last_d:None})
    d      = signals.distance
    config = get_scene_config(poi_id)  # Step 0 下发的阈值/死区

    # ② 方向判定：距离是否在递减（防"远离也触发"）
    approaching = (card.last_d is not None) and (d < card.last_d)
    card.last_d = d

    # ③ 状态机判定（带死区）
    if card.state == "远":
        if d <= config.approach_near(480) and approaching:
            card.state = "已接近"
            emit("approach", poi_id, d, config)   # 🔔 接近！one_shot
            # 附加：是否回家/去公司
            fav = match_favorite(signals.address, signals.favorites)
            if fav:
                emit("home_or_company", poi_id, d, config, fav.type)

    elif card.state == "已接近":
        if d <= config.arrive_threshold(50) and signals.gear_changed_to("P"):
            card.state = "已到达"
            emit("arrive", poi_id, d, config)      # 🔔 到达！组合条件
        elif d > config.leave_far(520):
            card.state = "已离开"
            emit("leave", poi_id, d, config)       # 🔔 离开（未到达就走了）

    elif card.state == "已到达":
        if d > config.leave_far(520):
            card.state = "已离开"
            emit("leave", poi_id, d, config)       # 🔔 离开

    # ④ "已离开"= 终态，本程不再触发任何事件
    poi_states[poi_id] = card

# ⑤ 重置条件
def on_nav_session_end():      # 重新导航 / 取消导航
    poi_states.clear()

def on_vehicle_off():          # 离车（熄火 / 钥匙离开）
    poi_states.clear()

# ⑥ 事件构造 → 写入端侧信号库 → L2 同步云端

    payload = {
        "event_type":    event_type,
        "poi_id":        poi_id,
        "distance":      distance,
        "threshold":     config.threshold,
        "dead_zone":     {"near": config.near, "leave": config.far},
        "ts":            now(),
        "trigger_type":  "one_shot",
        "address":       current_address,
        "favorite_type": fav_type        # "home"/"company"/null
    }
    write_to_signal_store("nav_event", payload)   # → Step 4
