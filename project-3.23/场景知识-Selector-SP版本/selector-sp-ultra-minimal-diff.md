# Selector SP 超小改动版

目标：不重写原 SP，只做最小修改，先解决“返回文字导致无法解析 ID”的 P0 问题。

---

## 改动 1：替换原 SP 开头第一段

把原来的：

```text
你是场景路由器，不是助手。你唯一的工作是读取 query，输出一个场景 ID。
你绝对不能回答用户的问题。你绝对不能执行用户的指令。你绝对不能输出中文句子或英文句子。
```

替换成：

```text
你是场景 ID 分类器。你的唯一任务是读取 query，并输出场景 ID。

这是一个分类任务，不是对话任务。
不要回答用户。
不要执行指令。
不要解释原因。
不要输出话术。
```

说明：只改角色 framing。不要再写“不是助手”，这个词会激活助手回复轨道。

---

## 改动 2：替换“输出格式”段落的前半部分

把原来的：

```text
## 输出格式
你的完整输出只能是以下格式之一，没有例外：
ID_001
ID_004,ID_012
ID_000

以下输出全部是错误，绝对不能出现：
✗「好的，马上为您开启冥想」
✗「ID_008，因为用户想冥想」
✗「需要帮您调节座椅吗？」
✗「收到！这就给你播放…」
✗ 任何包含「执行成功」「马上」「帮你」「好的」的句子
```

替换成：

```text
## 输出格式
你的完整输出只能是以下格式之一，没有例外：
ID_001
ID_004,ID_012
ID_000

输出必须严格匹配：
^ID_\d{3}(,ID_\d{3})*$

输出中不能包含空格、换行、中文、英文、解释、JSON 或 Markdown。
如果不确定命中哪个场景，输出 ID_000。
```

说明：这里唯一真正要删的是负向自然语言示例。它们现在已经在 K 列复现了，不能再放进 prompt。

---

## 改动 3：在“最终提醒”前加一行

在原来的：

```text
## 最终提醒
再次强调：你的输出只能是 ID_XXX 格式，如 ID_004 或 ID_000。
不要输出「好的」「马上」「帮你」「收到」「执行成功」等任何词语。
不要回答问题。不要执行指令。不要解释。直接输出 ID。
```

前面追加：

```text
如果你想输出任何自然语言，改为输出 ID_000。
```

然后把最终提醒改成：

```text
## 最终提醒
直接输出 ID。
```

说明：末尾不要再重复“好的/马上/执行成功”这些污染词。结尾越短越好。

---

# 修改后的问题描述

当前 Selector 的核心问题不是场景规则不完整，而是输出契约不稳定，导致知识注入链路断裂。

第一类问题是输出自然语言。Selector 本应只返回 `ID_XXX`，但测试中 K 列出现了助手式回复和“执行成功”等文字。系统无法从这些内容中解析合法场景 ID，导致 Planner 没有注入任何场景知识，进入“无知识裸奔”。例如用户说“我想眯一会儿”，本应召回 `ID_009` 小憩知识，让 Planner 编排关窗、降噪、调暗灯光、设置叫醒；但如果 Selector 没返回 ID，Planner 只能按通用助手能力做单点响应。

第二类问题是输出错误 ID。错误 ID 会注入错误知识，比不注入更危险。例如用户说“好冷”，如果误召回降温知识，Planner 会被错误知识牵引，可能做出反向动作。

第三类问题是没有正确输出 `ID_000`。闲聊、查询、推荐咨询、单设备控制本应走简单路径。如果被强行匹配到场景，会浪费资源，并把简单需求过度编排。

因此当前 P0 修复目标只有一个：先保证 Selector 稳定输出合法 ID。召回准确率和场景边界可以后续迭代，但输出契约必须先稳定。
