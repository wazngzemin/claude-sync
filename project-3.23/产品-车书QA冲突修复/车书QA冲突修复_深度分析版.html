零、问题本质与根因定位

问题现象

用户说 打开定速巡航（执行指令），Director没有调用 vehicle_basic_control 工具，而是用车书QA知识做了文字回答："定速巡航可以这样开启：进入车辆设置..."

根因分析

直接原因：SP中 {{car_function_simple_qa}} 的引导语太弱

当前引导语："可能会用上的一些基础车辆问答信息"

这句话对Director来说等于说"这是可以用的答案"。Director在决策时天然倾向于用已有的context回答，而不是去调用一个工具——调工具需要"主动判断"，用现成知识只需要"跟随"。当引导语没有明确告诉它"不要直接用"，它就会用。

深层原因：LLM的"context跟随倾向"

LLM有一个被广泛观察到的行为模式：当context中包含了一个看起来直接可用的答案时，模型倾向于使用它，而不是走额外的工具调用路径。这是因为：

  • 工具调用需要模型"认为当前context不够"才会触发
  • 当QA知识已经提供了一个"看起来合理"的回答，模型的"需要调工具"信号被抑制
  • SP中的注意事项有14条，但没有一条明确说"工具优先于知识"

为什么之前没出问题？

车书一期没有 {{car_function_simple_qa}} 的全量检索注入。二期上线后，每条query都会触发向量检索top3，将车书QA结果注入Director context。这是一个新引入的"噪声信号"，而SP没有为它建立防护机制。

问题发生链路

用户："打开定速巡航"
→
QA向量检索（全量top3）
→
注入："定速巡航开启方法：进入车辆设置..."
↓ 引导语仅说"可能会用上的一些基础车辆问答信息"
Director内部决策
→
context中已有答案 → 直接用它回答 → 不调工具

一、方案A — 定向修改QA数据（在answer中标注"支持语音控制"）

方案概述

不改500-2000条全量QA，只改与车控功能重叠的80-150条。在answer开头加标准前缀："该功能支持语音直接控制..."

核心逻辑：为什么加一句话就能改变Director行为？

当前Director看到的context是：

// 当前QA answer内容 "定速巡航开启方法：进入【车辆设置】-【辅助驾驶】，打开【自适应巡航辅助】，然后按方向盘PILOT按键激活。" // Director的"思考"过程： 用户要打开定速巡航 → context里有操作方法 → 直接告诉用户怎么操作

加了前缀后Director看到的context变成：

// 修改后的QA answer内容 "该功能支持语音直接控制，用户说'打开定速巡航'即可由系统执行。若需手动操作：进入【车辆设置】-【辅助驾驶】..." // Director的"思考"过程： 用户要打开定速巡航 → context说"支持语音控制，由系统执行" → 我应该调工具执行

关键机制：LLM在决策时，会优先采信context中最前面的信息（primacy effect）。当answer开头就说"支持语音控制"，这个信号比后面的手动操作步骤更先被模型处理和采信。

优势分析（5条）

  • 改在"信息源头"：不是在下游修补，而是在上游数据层直接把信息改对了。Director看到的context本身就包含了正确的引导，不需要在其他地方再加规则。这是最干净的修复方式。
  • 对LLM行为影响确定性高：学术研究和工程实践都表明，context中包含显式的处理指引（"该功能支持语音控制"）比抽象的规则（"执行型指令要调工具"）对LLM行为影响更大。因为前者是具体事实，后者需要模型做推理。
  • 不侵入任何系统组件：不改SP、不改工程、不改检索逻辑、不改Knowledge Selector。只改了数据内容，回滚也简单（把前缀去掉即可）。
  • 不增加任何token开销：QA对的总长度略微增加（每条多20-30字），但对检索结果注入的token量影响很小（top3 answer总共也就300-500字，加前缀后变400-600字）。
  • 与会议结论方向一致：会上李明骏提出的就是"在answer中加入语音可控信息"的方向。此方案是对这个方向的系统化执行。

劣势分析（4条）

  • 不是100%保证有效：加了"支持语音控制"后，Director 大概率会选择调工具，但LLM行为有不确定性。在某些复杂context下（比如多轮对话、用户指令模糊），模型仍可能选择用手动操作步骤回答。预估单靠方案A的修复率在70-85%。
  • 需要人力标注和审核：需要逐条判断"哪些QA描述的功能支持语音控制"。虽然可以参照SP中的"车辆设备知识"表，但仍需人工对照确认。如果映射错误（标注了一个实际不支持语音控制的功能），会导致新的badcase。
  • QA数据的长期维护成本：未来每次新增QA对，都需要判断是否需要加前缀。如果没有建立标注规范和流程，问题会随新数据再次出现。
  • 需要产品/标注侧配合：泽民不能独立完成——需要有人导出QA数据、执行批量修改、重新导入向量库。如果产品侧排期紧张，可能无法在1-2天内完成。

资源成本精算

  • 人力：
    - 制作"功能→是否语音可控"映射表：2-3小时（需熟悉SP车辆设备知识 + 逐条QA核对）
    - 编写批量修改脚本：1小时（标准前缀 + 映射表 → 自动拼接）
    - 人工审核修改结果（抽查20-30条）：1小时
    - QA数据重新导入向量库：0.5小时（取决于工程流程）
    - 验证测试（跑10-15个case）：1-2小时
    总计：约6-8小时，1人/天
  • token开销变化：
    - 每条QA answer增加约20-30字（~30-40 tokens）
    - top3检索结果总共增加约90-120 tokens
    - 对Director单次调用的context长度影响：<1%
    - 可忽略不计
  • 长期维护成本：
    - 每次新增QA批次时需同步标注（每批次约30分钟-1小时）
    - 如果不建立规范，3-6个月后新QA会再次引起问题

可行性评估

  • 技术可行性：高 — 只是文本拼接，技术上没有难点
  • 组织可行性：中 — 需要产品侧配合导出数据+重新导入。如果产品侧排不出人，泽民要自己做数据处理。确认问题：泽民是否有QA数据的直接编辑权限？
  • 效果可行性：中高 — 预估修复率70-85%。原因：前缀信息改变了context的"语义倾向"，但不是硬性约束，模型仍有选择空间。
  • 时间可行性：紧张 — 如果泽民有数据编辑权限，1天可完成。如果需要走数据导入流程，可能需要1.5天。

可能的失败场景与应对

  • 失败场景1：Director忽略前缀信息
    概率：15-25%
    原因：Director SP中有14条注意事项+20多个示例，如果执行类的few-shot示例中没有类似场景，模型可能仍按原有模式处理
    应对：叠加方案B，用 {{knowledge}} 从另一个维度强化规则
  • 失败场景2：映射表不准确
    概率：5-10%（映射出错的那些QA对会出问题）
    原因：标注"支持语音控制"的功能实际上不支持（如智能驾驶开关是拨杆控制）
    应对：严格参照SP"车辆设备知识"表核对，不确定的宁可不标注
  • 失败场景3：前缀和原answer语义冲突
    概率：5%
    原因：某些QA的answer本身就包含"不支持语音控制"或限定条件，前缀和正文矛盾
    应对：人工审核抽查，发现矛盾的手动调整

方案A综合判定

适用场景：作为"数据层修复"，是从源头改善问题的正路方向。中长期必须做。
单独使用的局限：修复率预估70-85%，不够"保险"。因为前缀信息是"事实陈述"不是"行为指令"，Director可能知道这个功能支持语音控制，但仍选择用文字回答。
建议：不建议单独使用。应与方案B组合。

二、方案B — 通过 {{knowledge}} 注入处理规则

方案概述

在动态专业知识 {{knowledge}} 中创建一条"车辆功能问答知识使用规则"（RULE-01），教Director区分执行/查询/咨询三种意图

核心逻辑：为什么通过 {{knowledge}} 注入规则能起作用？

机制1 — SP赋予的权重

SP中 {{knowledge}} 的引导语是："根据当前情景、任务，动态加载的专业知识，灵活参考能有效帮助你把事情做得更好"。这句话告诉Director要"灵活参考"这里的内容。所以这里注入的规则不会被忽略——Director会把它当成"专业建议"来处理。

机制2 — 规则的具体性

我们注入的不是抽象的原则（如"要正确使用知识"），而是具体的 if-then 规则："用户说'打开XX' → 调 vehicle_basic_control"。LLM对具体规则+示例的遵循度远高于抽象原则。

机制3 — 与 {{car_function_simple_qa}} 同时可见

Director在同一轮推理中会同时看到两块内容：QA知识和我们的规则。当两者在同一个attention窗口内时，规则可以直接"override"QA知识的误导效应。

要注入的知识条目完整内容（可直接复制使用）

知识ID: RULE-01 (或 COM-99-01-01) 知识名称: 车辆功能问答知识的使用规则 触发策略: 始终加载（先验证效果，后续可改为条件触发） ──────────────────────────────────── 【车辆功能问答知识使用规则】 你的context中可能包含「基础车辆功能问答」知识（从车辆手册中检索的参考信息）。这些知识描述了功能的手动操作方法。你必须根据用户的真实意图决定是调用工具还是使用知识回答： 规则1 — 执行指令 → 调用工具，不用知识回答 判断标准：用户说出带动作词的指令："打开XX""关闭XX""调高XX""调低XX""开启XX""设为XX" 处理方式：调用 vehicle_basic_control 执行操作 反面要求：即使「基础车辆功能问答」中有该功能的操作步骤，也不要用它做文字回答 例：用户说"打开定速巡航" → 调 vehicle_basic_control，不要说"进入车辆设置..." 规则2 — 实时状态查询 → 调用工具，不用知识回答 判断标准：用户问实时数据："现在XX多少""当前XX""剩余XX""XX正常吗" 处理方式：调用 search_vehicle_status_info 查询 反面要求：不要用知识中"去仪表盘看"类的回答替代实时查询 例：用户说"现在时速多少" → 调 search_vehicle_status_info，不要说"可以在仪表盘查看" 规则3 — 操作方法咨询 → 使用知识回答 判断标准：用户问方法："怎么用""如何设置""步骤是什么""咋调" 处理方式：参考「基础车辆功能问答」中的知识回答 例：用户说"定速巡航怎么用" → 用知识回答操作步骤 规则4 — 工具失败兜底 当工具调用失败时，用「基础车辆功能问答」知识告诉用户手动操作方法

Knowledge Selector 触发策略：深度对比

策略1：始终加载（推荐先用）

做法将 RULE-01 设为基线知识，不走 Knowledge Selector 的条件判断，每轮都注入
原理因为 {{car_function_simple_qa}} 本身就是全量触发的（每轮都检索），所以规则也应该每轮都在。两者同频出现才能对冲。
优点最简单、最稳定、不会漏匹配、泽民完全自主可控、2小时内可完成
缺点非车控场景也会注入（如用户问天气、放歌时也会看到这条规则）。每轮多占约150 tokens。
缺点的实际影响 150 tokens vs Director单次调用通常消耗 3000-8000 tokens，占比 2-5%。
非车控场景注入规则：Director看到"车辆功能问答知识使用规则"但context中没有车辆功能问答知识时，这条规则不会产生任何效果（因为规则是条件式的："如果有车辆功能问答知识，则..."）。
结论：缺点影响极小，可接受。

策略2：关键词触发

做法在 Knowledge Selector 中加条件：query含动作词（打开/关闭/调高/多少/剩余...）+ 车控设备词时加载
优点精准匹配，不浪费token
缺点 1. 需要维护关键词表（动作词+设备词的组合数很大）
2. 用户说"帮我把巡航打开"或"调一下那个温度"这种非标准表述可能漏匹配
3. Knowledge Selector 本身有延迟问题（当前1.8s，目标<500ms），加复杂规则会进一步增加延迟
4. 调试和维护成本高——每次有新的漏匹配case都要更新关键词
结论效果验证后，如果token开销确实是问题，再切换到这个策略。现阶段不推荐。

策略3：与 {{car_function_simple_qa}} 联动

做法工程侧实现：当 {{car_function_simple_qa}} 非空时，自动同时注入 RULE-01
优点最精准——有QA知识时才有规则，完美对冲
缺点需要工程改动，短期不可行（约束条件排除）
结论中期优化方向。短期用策略1代替。

优势分析（6条）

  • 泽民完全自主可控，不需要任何人配合：这是方案B最大的优势。创建一条知识条目+配置触发策略，全在泽民的职责和权限范围内。不需要产品侧、工程侧、标注侧任何人参与。
  • 速度最快：从动手到验证完成，预计2-3小时。今天上午就能出结论。
  • 不改任何数据：QA数据完全不动。如果效果不好，删掉知识条目就回滚了，零风险。
  • 不改SP：符合团队约束。
  • 规则写一次、长期有效：不管QA数据怎么变、新增多少条，这条规则始终适用。不需要跟着数据迭代。
  • Director对 {{knowledge}} 中的内容有明确的参考权重：SP引导语说了"灵活参考能有效帮助你把事情做得更好"，Director不会忽略这里的内容。

劣势分析（4条）—— 这是你需要重点关注的

  • {{knowledge}} 的权重是"参考"级，不是"指令"级：这是方案B最大的不确定性。SP引导语说的是"灵活参考"，不是"必须遵循"。Director可能参考规则，也可能在某些情况下"灵活"地不参考。对比：SP注意事项中的规则是"硬约束"（如"不要说脏话"），{{knowledge}} 中的内容更像是"建议"。
    预估单靠方案B的修复率：60-80%。比方案A的70-85%略低，因为规则的"绑定力"弱于数据层的直接信息。
  • 始终加载策略有token浪费：每轮多约150 tokens。月度来看，假设日均100万次调用：
    - 每次多150 tokens → 日增1.5亿 tokens → 月增45亿 tokens
    - 按Claude/GPT-4级别模型 $15/M tokens计算 → 月增约 $67,500
    - 按国内模型 ¥2/M tokens计算 → 月增约 ¥9,000
    - 实际影响取决于调用量和模型单价，需要跟李浩然确认是否可接受
  • "规则"放在"知识"的位置，语义上有点违和：{{knowledge}} 的设计初衷是注入"场景化的专业知识"（如接机场景的注意事项、晕车场景的应对方法），不是注入"通用处理规则"。这条RULE-01更像是一个"元规则"而非"场景知识"。
    实际影响：对Director来说没区别——它只看内容不看"这是知识还是规则"。但对团队的知识管理体系来说，这是一个"架构上不那么优雅"的做法。
  • 无法处理"规则和QA知识都缺失"的情况：如果某轮对话中 {{car_function_simple_qa}} 为空（检索没命中），但 {{knowledge}} 中有规则，规则不会产生任何效果（也不会有副作用）。但反过来，如果因为触发策略问题导致规则没注入，而QA知识注入了，问题就会复发。所以始终加载策略是必须的，不能省。

资源成本精算

  • 人力：
    - 编写 RULE-01 知识条目内容：30分钟（上面已给出完整内容，微调即可）
    - 在知识管理系统中创建条目并配置触发策略：30分钟
    - 在FORNAX上验证3条badcase：1小时
    - 扩展验证（10-15条测试case）：1小时
    总计：约3小时，泽民一人完成
  • token开销：
    - RULE-01 内容约 200字 ≈ ~250 tokens（含标签格式）
    - 始终加载 → 每轮多 250 tokens
    - vs Director单次调用总token通常 5000-10000 tokens
    - 占比 2.5-5%，可接受
  • 长期维护成本：
    - 基本为零——规则是通用的，不需要随QA数据更新
    - 只有当车控工具的名称或能力发生变化时才需更新

可行性评估

  • 技术可行性：高 — 创建知识条目是泽民的日常工作，完全在能力范围内
  • 组织可行性：高 — 不需要任何人配合。这是泽民可以"今天就做"的方案
  • 效果可行性：中 — 预估修复率60-80%。核心不确定性在于 {{knowledge}} 的"参考"权重是否足以override QA知识的"直接回答"吸引力
  • 时间可行性：高 — 3小时可完成全流程

可能的失败场景与应对

  • 失败场景1：Director不遵循规则
    概率：20-35%
    原因：{{knowledge}} 权重不够，或在特定context组合下规则被"淹没"
    应对：叠加方案A，从数据层提供另一个信号源；或者退而推动SP最小改动（改引导语那一句话）
  • 失败场景2：规则导致矫枉过正
    概率：5-10%
    具体表现：本该用QA知识回答的咨询型问题（如"定速巡航怎么用"），Director强行去调了工具
    原因：规则中对"咨询型"的判断标准不够清晰，或用户表述介于执行和咨询之间
    应对：在规则中加强咨询型的判断标准描述；测试验证矫枉过正case
  • 失败场景3：token预算问题
    概率：10%
    原因：产品侧或工程侧认为每轮多250 tokens不可接受
    应对：切换为关键词触发策略（策略2），但需要投入更多时间做关键词维护

方案B综合判定

最大价值：速度快、泽民自主可控、零依赖。是"今天就能出结论"的最快路径。

三、方案C — A+B组合双层保障（推荐）

方案概述

同时执行方案A（改QA数据）和方案B（注入Knowledge规则），从数据层和推理层双重保障

核心问题：为什么单独的A或B不够？

方案A单独使用的薄弱点：

A在数据层说了"该功能支持语音控制"，但这是一个事实陈述，不是行为指令。Director可能的"思考"：
"嗯，这个功能支持语音控制，我知道了。但用户问的是怎么操作，我来告诉他步骤吧。"
模型知道了事实，但没有被告知"知道这个事实后应该怎么做"。

方案B单独使用的薄弱点：

B在知识层给了明确规则"执行指令→调工具"，但规则是抽象的。Director面对具体的QA知识（详细的操作步骤）和抽象的规则时，具体内容的"吸引力"可能更强。
"规则说要调工具，但context里已经有详细的操作步骤了，直接说不是更高效吗？"
规则没有足够的context支持来抵抗具体知识的吸引力。

A+B组合为什么能弥补：

当两层信号同时存在时，Director的"思考"变成：
"QA知识说'该功能支持语音控制，由系统执行'（方案A信号）→ 知识规则也说'执行指令要调工具'（方案B信号）→ 两个独立信息源都指向同一结论 → 调工具"
这就是"信号叠加"效应：两个独立的弱信号叠加后变成强信号。

效果概率分析

方案 预估修复率 概率逻辑
方案A单独 70-85% 数据层信号强度高（primacy effect），但不是行为指令，模型有选择空间
方案B单独 60-80% 规则明确但权重是"参考"级，具体知识可能override抽象规则
方案C (A+B) 90-95% 两个独立信号同时失效的概率 = 失效A × 失效B
保守估计：(1-0.70) × (1-0.60) = 0.30 × 0.40 = 0.12 → 修复率88%
乐观估计：(1-0.85) × (1-0.80) = 0.15 × 0.20 = 0.03 → 修复率97%
取中间值：约90-95%修复率

注意：上述概率是粗估，基于"两个信号独立"的假设。实际上A和B不完全独立（它们在同一个context中被Director同时处理），但信号叠加效应仍然成立。

优势分析（5条）

  • 修复率最高：预估90-95%，远高于任一方案单独使用。双层保障的本质是"冗余设计"——即使一层失效，另一层仍能工作。
  • 覆盖了两种失效模式：
    - 方案A失效（Director忽略数据前缀）→ 方案B的规则仍在起作用
    - 方案B失效（Director不遵循规则）→ 方案A的数据信号仍在引导
  • 可并行执行：泽民做方案B（3小时），产品侧同步做方案A（1天）。不需要串行等待。
  • 阶段性验证：上午先验证B的效果，下午A就位后验证A+B的组合效果。两个数据点帮助判断每层的贡献度。
  • 为中长期优化积累数据：通过对比"只有B"vs"A+B"的效果差异，可以量化数据层改动的增量价值，为后续是否大规模改造QA数据提供决策依据。

劣势分析（3条）

  • 资源消耗是A和B的总和：需要泽民做方案B（3小时）+ 产品侧做方案A（6-8小时/1人天）+ 联合验证（2小时）。如果只有泽民一个人，且产品侧无法配合，那实际上只能执行方案B部分。
    关键依赖：方案A是否能在1-2天内完成取决于产品侧的配合度。
  • 调试复杂度增加：如果出现badcase，需要判断是A层还是B层的问题。比如出现矫枉过正（咨询型问题被调了工具），是规则写得太强还是前缀引导有误？需要分别关闭A/B来定位。
  • 长期维护有两个"维护面"：QA数据的标注要持续维护（方案A），Knowledge规则也需要在工具变动时更新（方案B）。虽然各自维护成本不高，但有两个面意味着两个可能出错的点。

可行性评估

  • 组织可行性：中 — 取决于方案A部分是否有人配合。如果只有泽民一人，实际方案C退化为"方案B + 方案A的部分"
  • 效果可行性：高 — 预估修复率90-95%
  • 时间可行性：中 — 并行1天 / 串行1.5天
