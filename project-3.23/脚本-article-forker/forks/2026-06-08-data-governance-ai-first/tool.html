触发条件清单：从"老表迁移+永久清洗"到"DSL 先行+源头合规"

旧·先产后治
  • 拿赛力斯/上汽旧表迁移
  • 事后补运算符、改 key、对口径
  • 每次改版重清一遍（台账"行1"5条）
新·AI First
  • 先定触发器 DSL 结构规范
  • 新条目只能按 DSL 结构产生
  • 人填结构化表单 / AI 从需求生成草案
触发器 DSL 字段规范 v1（每条必含，缺字段不让进库）
signal_key 唯一英文 key（全局不重复，禁 states/status 混用） signal_name_cn 中文显示名 value_type bool | number | enum | str（禁把枚举标成 str） operator ==,!=,>,<,>=,<=,changed_to（number 必须配齐后四个） value_domain 取值域（number 给范围 [0,100]；enum 给全集） unit number 必填（℃ / % / km） source 一手实车 | 二手老表迁移 verified_date 最后实车验证日期（二手未验证 = null + 标红） lib_version 信号库版本（如 v2.5），锁版后不随意变
真实例子：你台账"温度>26℃ 配不出" → DSL 化后天生可配
signal_key: cabin_temp value_type: number operator: [==,!=,>,<,>=,<=,changed_to] ← number 强制配齐，"温度>26℃"天生可配 value_domain: [16,32] unit: ℃ source: 二手老表迁移 verified_date: null ← 标红待韩天亮确认 lib_version: v2.5
从需求文本生成合规 DSL 草案
你是触发器 DSL 生成助手。我给你一句需求，请严格按下列字段输出一条触发条件，字段缺一不可，number 类型必须配齐 6 个运算符，拿不准的字段标"待确认"，不要编造取值域： 字段：signal_key（唯一英文,小写下划线）/ signal_name_cn / value_type(bool|number|enum|str) / operator / value_domain / unit / source / verified_date / lib_version 【需求】：（例：电量低于 10% 时提醒充电） 输出一个 DSL 代码块 + 一行"我不确定、需你确认的字段"。
结果：运算符缺失、key 不统一、类型标错从源头不再发生，你不用逐条救火。
常见错误：① 为兼容历史保留多个 key（实体歧义温床，宁可一次性统一改）；② 把 DSL 做成又一张人工大表（它的价值是"校验+生成自动化"，不是多一张表）。

Skill / 示例生产：从"人工粗写"到"白名单+模板生成"

关键

定一张车厂能力白名单 + Skill 写作模板。新 Skill 由 AI 按模板生成、自动对照白名单校验，人只审定。白名单直接堵死"写了车厂没有的能力（如观影模式一级指令）"。

车厂能力白名单（节选示例）
| 能力 | 是否支持 | 调用路径 | 可控参数 | | 座舱温度调节 | ✅ | vehicle_basic_control | temp[16-32] | | 车窗开合 | ✅ | vehicle_basic_control | window,position[0-100] | | 观影模式(一级指令) | ❌ | — | 车厂无此一级指令，禁写成可直接下发 | | 宠物模式 | ⚠️走预置应用 | 预置应用,非编排 | Skill 必须写明走预置应用 |
真实例子：观影模式被白名单拦下
旧：Skill 写"看电影 → 下发观影模式一级指令" → 车厂没有 → 调用失败 新：AI 生成时 capability_ref 查白名单"观影模式"→ ❌不支持 → 自动打回， 提示"需拆成 调暗氛围灯+降座椅+开屏 的编排，或走预置应用"
按白名单生成/校验 Skill
你是 Skill 写作助手。下面是【车厂能力白名单】和一条 Skill 需求。请按模板生成 Skill，并强制校验：capability_ref 必须能在白名单查到；call_path 必须和白名单一致；若涉及白名单中"❌不支持/⚠️走预置应用"的能力，不要硬写成可直接下发，明确指出并给替代方案。 模板字段：skill_name / trigger_intent(2-3 真实 query) / capability_ref / call_path / params / negative_examples / verified 【车厂能力白名单】：（粘贴） 【Skill 需求】：（例：用户说"我要看电影"） 输出：Skill 模板代码块 + 一段"白名单校验结论（通过/打回，原因）"。
结果：把"治理 Skill 质量"前移成"Skill 生产即合规"，不靠上线后 badcase 反推。
常见错误：白名单不全 = 校验形同虚设。先覆盖高频能力再补全；白名单本身要锁版、标来源。

badcase 闭环：从"人工逐条排查"到"AI 主导归因+你裁决"

让 AI 按你 CLAUDE.md 的四层路径（实体歧义→陈旧→检索→模型）自动归因 + 给修改草案，你只做裁决（提示词见底部 📋）。

结果：你从"救火队员"变"裁判"，省下的时间投到重设计 1/2 和护城河。

一手 context 治理：这个不能绕，是你唯一值得深耕的

文章判据"规则在人之上→数据精度极高"。你的车端正是：车况 {{env_info}}、用户记忆 {{memory}}、视觉感知 {{status}} 是被传感器/系统充分数据化、别人拿不到、AI 也替代不了的一手数据。

把精力挪到这
前 3 个重设计把脏活从源头砍掉；这一个把唯一干净的一手数据治理好——这才是抗 AI 折旧、接"认知大脑→具身智能"的能力。

工具 · 提示词

📋 AI 自动归因提示词（重设计 3）

把 badcase 处理从"人工救火"改成"AI 主导归因+你裁决"——这是把救火工作量从源头砍掉的第一步。

badcase 四层自动归因 + 修改草案
你是我的 Planner badcase 归因助手。按下面四层顺序归因，命中一层就停，不要直接归到"模型不行"，最后给出修改草案，由我裁决。 【粘贴 badcase】 - 用户输入： - 当时车况/记忆/目标（env_info / memory / goal_list）： - Planner 实际输出（action_list / talk_content）： - 预期行为： 请输出： 1. 实体歧义：相关工具/能力描述有没有"多种叫法 / key 不唯一 / 模糊"，可能让 Planner 走错工具？指到具体哪句。 2. 数据陈旧：用到的信号/枚举/参数，有没有过期或从老表迁移、口径不对？ 3. 检索失败：正确处理示例是否其实在库里、但这条 query 没检索到？ 4. 只有前三层都排除，才判为模型推理问题，并说明依据。 最后给：| 归因层 | 证据(具体到哪句Skill/哪个字段) | 修改草案 | 这是"治理止血"还是可"重设计根治" | 像审计一样冷酷，不要替模型开脱。
进阶：最后一列"能不能重设计根治"，是帮你把一次次救火，沉淀成重设计 1/2 的输入——这样你修的 badcase 越多，未来要修的越少。
