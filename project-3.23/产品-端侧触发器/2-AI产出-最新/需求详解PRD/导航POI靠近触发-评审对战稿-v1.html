1. 开场（2分钟）

开场白
大家好，今天评审的是导航POI靠近触发需求，也就是"快到某地主动服务"这类场景。

先说一下方案从 v3 到 v4 变了什么——核心变化就一件事：高德auto SDK的围栏服务确认下架了，陈俊龙6.25会后投屏发现的，上汽高德正式确认不可用。

所以v4做了一个调整：我们不依赖高德围栏SDK，端侧自建围栏判定。用的是高德SDK已有的"POI距离查询"接口——陈俊龙在6.25会上确认过的，你传POI ID，它返回当前距离。我们端侧自己拿着这个距离做状态机判定。

今天需要各位确认三件事：
① 方案能不能做——各方技术可行性
② 各方要做什么——分工和依赖关系
③ 630能不能上——工期评估
开场要点：先把v3→v4的核心变化讲清楚（围栏SDK下架→端侧自建），然后明确今天的目标。不要上来就讲细节。

2. 方案讲解要点（5分钟）

不用把PRD从头讲到尾，抓住四步架构 + 一个case + 开发链路图就够了。

四步架构
方案很简单，四步：

第一步，云端理解用户意图、完成选点——用户说"快到外婆家提醒我打电话"，意图模型拆出条件和动作，AINAVI选点工具拿到外婆家的经纬度。思晴6.25确认这个工具可以复用。

第二步，云端注册规则，然后下发一个围栏任务给端侧——告诉端侧：盯着这个坐标，500米范围，进了告诉我。

第三步，端侧定时查距离——A类用导航沿路线距离，B类用高德SDK距离查询接口。距离小于阈值了，端侧自己判"进入围栏"，上报一个enter事件给云端。坐标不出车，合规没问题。

第四步，云端触发器引擎匹配事件，push给Advisor，Director说人话，TTS播报。这一段和现有链路完全一样，不用改。
讲完四步后，翻到PRD的§6.2 Phase B泳道图，走一遍"快到外婆家"的case。重点是让大家看到：d=15000→...→479，每秒判一次，479≤480的那一秒触发，然后one_shot自锁不再触发。

3. 按角色拆解：他们会问什么，你怎么接

陈俊龙 / 宫丽萍 · 地图团队
高德SDK距离查询接口 · 选点POI搜索 · 你最核心的依赖方
陈俊龙"每秒调一次距离查询接口，QPS顶得住吗？有频率限制的"
这个确实需要你帮我确认一下QPS上限。我目前设计的是1Hz——每秒查一次。但如果SDK限频，比如5秒一次也能接受。60km/h的车5秒走85米，不会错过480米的阈值。如果限到30秒一次那确实就很紧了——我们可以聊一下你那边实际的限制是多少。
60km/h ≈ 17m/s → 5秒=85m → 在死区范围(40m)外，可以接受
120km/h ≈ 33m/s → 5秒=167m → 仍在合理范围，但精度下降
如果限频30秒 → 120km/h走1000m → 可能跳过整个围栏 → 方案不可用
陈俊龙"距离查询返回的是直线距离还是路线距离？"
你6.25会上说过"POI详情信息去请求，有当前距离和预计到达时间"。如果返回的有路线距离那最好，直线距离也够用——对于"快到某地"这种场景，直线500米已经是"确实近了"的程度。高架旁的误触发是已知限制，但在"提醒打电话"这种轻量场景下可以接受。
陈俊龙"非导航状态下这个接口能用吗？"
你6.25也确认过——"POI详情请求"不依赖导航引擎，非导航状态也能调。我再跟你确认一下：非导航时返回的距离，是直线距离对吧？
陈俊龙6.25原话："POI详情信息去请求，有当前距离和预计到达时间"
宫丽萍"围栏SDK都下架了，距离查询接口不会也下架吧？"
围栏是一个独立的服务模块，标注"已下架"。距离查询是POI详情接口的一部分，属于基础能力——和围栏不是一回事。不过这个风险确实要考虑，俊龙你帮看一下距离查询接口在高德auto SDK roadmap上的状态？
陈俊龙"你说的能订阅吗？还是只能被动查？"（汪帅6.25会上也问过）
如果能支持订阅那更好——端侧不用自己轮询，省资源。但目前方案按"被动查询"设计的，也能跑。所以这个是P1，不是阻塞项。如果能订阅，我们后续优化。
汪帅 / 仁杰 · 端侧 + 导航产品
端侧围栏模块的核心Owner · 工作量最大的一方
仁杰"这不就是人肉实现围栏逻辑吗？"
对，你说得对。你6.25原话就是"围栏本身的逻辑其实比较简单"——给个坐标，建个围栏，以一定频率查在不在。高德SDK围栏服务帮你包装了这层，现在它下架了，我们自己做。核心就是一个状态机：远→已接近→已到达→已离开，加一个防抖死区。代码量不大，逻辑也清楚。
仁杰6.25原话（00:38:59）："围栏本身的高德提供的围栏能力本身其实是比较简单的，你给他一个坐标，创建一个围栏，你告诉他半径是多少"
汪帅"端侧围栏模块5个子模块，工作量怎么评？630来得及吗？"
5个子模块听起来多，但拆开看：指令接收器就是解析一个JSON；围栏存储就是SharedPreferences存个列表；距离轮询器就是一个定时器每秒查一次；状态机就是PRD里那个伪代码；事件上报器就是发个HTTP请求+失败缓存。每个模块都不复杂。你帮评估一下，这个量大概几天？
汪帅"云端→端侧的围栏任务下发走什么通道？"
这个需要你定。两种路径：复用现有端云长连接（如果有的话），或者走推送通道。核心需求就是云端能把一个JSON包下发到端侧指定模块。你看哪种现有通道最合适？
汪帅"端侧→云端的事件上报走什么链路？"
和上一个问题成对——围栏事件上报到云端触发器引擎。上报量很小，整个过程就1-2次（进入一次、离开一次），不是每秒上报。看能不能复用现有的端→云信号上报链路？
汪帅"有必要绕到云端吗？端侧直接播TTS不行吗？"（6.25他问过这个）
你6.25提过这个——"端上其实也能播出来的，可以不经过Director"。纯播报确实可以。但实际场景里，用户说"快到公司帮我找充电桩"——这种需要Director调工具的，端侧做不了。而且走Director能保证统一的话术风格、能结合Context做个性化。所以端侧判定归端侧，播报还是走云端链路。
汪帅6.25原话（00:48:18）："端上其实也能做一些简单的触发器...我们端上其实也能播出来的，可以不经过director"
仁杰"坐标不出车你怎么做到的？"
端侧自己调高德SDK查距离，拿到一个数字（比如479米），自己判"进了"——然后上报给云端的只有4个字段：task_id、event_type、poi_id、timestamp。不传坐标、不传距离。云端拿task_id匹配注册规则就够了。
仁杰"多围栏并发，比如用户同时设了3个POI，性能扛得住吗？"
3个围栏就是每秒3次SDK距离查询。如果SDK没有QPS限制，这个量级很小。如果有限制——比如只允许1Hz——那3个围栏就是每秒轮流查一个，3秒一轮。60km/h走3秒=50米，在死区范围内还是能接受。具体数字等俊龙确认QPS后再定。
刘杨 · 触发器引擎
引擎适配 · v4对他是利好——不用开发云端状态机了
刘杨"我这边需要做什么改动？"
v4对你是好消息——不需要开发云端状态机了。判定下沉到端侧，你这边只需要做引擎适配：支持fence_event类信号的场景匹配，收到端侧上报的fence_enter事件后做15-11 Callback路由，push给Advisor。和现有事件匹配逻辑一样，只是多一种事件类型。
刘杨"fence_event和现有信号上报有什么区别？"
区别是：现有信号是端侧每秒上报原始数据（distance=479等），你这边要读数据做判断。fence_event是端侧已经判完了，直接告诉你"进了"——你只需要匹配task_id、检查event_type是不是fence_enter，命中了就走Callback。工作量比之前的状态机方案小很多。
施佳杰 · 弋途Advisor
Advisor订阅fence事件 + 生成advice
施佳杰"Advisor需要订阅一种新的事件类型？改动大吗？"
对，新增订阅fence_enter事件。payload里有poi_name和action（比如"提醒打电话"），你拿这两个字段组装advice就行。和现在订阅其他事件的逻辑一样，不是新模式。
施佳杰"B类Case用户没开导航，Context里导航信息缺失怎么办？"
没影响。Advisor拿到的是fence_enter事件的payload——里面有poi_name和action，这是生成advice的核心信息。导航状态是Context的一部分，有就用、没有Director也能推理。"快到外婆家了，提醒打电话"这种场景，有没有导航信息不影响话术。
思晴(Jojo) / 李明骏 · AINAVI + 记忆模块
选点工具返回经纬度 · 阻塞项
思晴"AINAVI选点工具现在返回poiname和poiid，加经纬度需要改什么？"
你6.25确认过AINAVI选点工具可以复用。现在需要在返回字段里加上lat和lng两个字段。具体要看经纬度存在哪——如果记忆模块里有，直接取；如果没有，需要调地图接口用poiid换坐标。这块需要和李明骏确认：记忆模块里存经纬度会不会影响现有功能。
李明骏"记忆模块加经纬度字段会影响现有功能吗？"
这个需要你评估。两种路径：① 记忆模块的POI记录直接带经纬度（改存储结构），② 不改记忆模块，选点时用poiid实时查地图接口换坐标。第二种对你零改动，但多一次网络调用。你看哪种更合适？
韩天亮 / 任军 · 信号库 + 运算符
前置依赖 · 需要先完成才能开工
韩天亮"fence_event的信号编号怎么定？"
需要注册一个fence_event主信号，下面3个子类型：fence_enter、fence_arrive、fence_leave。另外payload里要支持task_id字段——这是匹配围栏任务的唯一标识。和现有nav_event信号格式类似，你看能不能这周先注册？这是引擎适配的前置。
任军"eq运算符支持fence_event字符串，需要改吗？"
需要确认一下现有的eq运算符是否已经支持对event_type和task_id做字符串匹配。如果已经支持字符串eq，那不用改；如果只支持数值，需要扩展。你帮确认一下？

任何人"为什么不直接在云端做围栏判定？不用端侧改那么多"
会上讨论过。仁杰原话："坐标出车，合规可能会麻烦一点"。汪帅原话："难度就指数级的了"。云端围栏意味着端侧每秒上报GPS坐标到云端——合规风险、网络依赖、上报量都是问题。端侧自建围栏，坐标不出车，判完了只上报一个事件，干净。

汪帅6.25（00:44:39）："难度就指数级的了"
任何人"B类高架旁误触发怎么办？"
两层处理：第一，A类（POI是导航目的地）用导航沿路线距离，高架不会误触发。第二，B类用直线距离，高架旁确实可能触发。但想一下场景——"路过迪士尼提醒我拍照"，高架旁直线500米内确实离得很近了，误触发一次也不是灾难，最多就是早提醒了。真要精确，后续版本可以加Z轴高度过滤。本期先不做。
任何人"如果高德SDK距离查询接口也不可用怎么办？"
有兜底方案：端侧自己用GPS坐标 + Haversine公式算直线距离。仁杰说过"围栏逻辑本身比较简单"——核心就是拿两个坐标算距离，和阈值比大小。没有SDK包装，自己算也行。精度降低但满足围栏判定需求。
任何人"v3到v4变化太大了，方案稳不稳？会不会v5又推翻？"
v3到v4的变化看着大，但核心架构没变——还是"云端下发任务、端侧判定、事件上报"这个四步模型。变的只是端侧判定的数据来源：从"围栏SDK"换成"距离查询接口+自建状态机"。这次改方案是因为高德围栏SDK下架这个外部变化，不是我们自己推翻自己。新方案反而更稳——不依赖第三方服务的开关，自己控距离查询+自建状态机，可控性更强。
