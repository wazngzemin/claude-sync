7.6下午会定版核心结论
1. VLM数据不经触发器，VLM SDK 直送数据中心（数据通道），触发器只收状态回调（控制通道）
2. 数据中心（李鑫）订阅VLM SDK topic → 整包缓存 → 按端状态格式透传云端
3. 触发器保证"发送成功"，VLM SDK保证"收到后执行成功"（AlwaysOn由SDK内部轮询保证）
4. OneShot数据回传路径待后续评审（丁彬原话: "one shot 等需求导入再聊"）
5. 端云必须基于同一份规则，否则就是bug（汪帅原话）

对照丁彬原图的变更点（7.6下午定版）
• 原图：VLM结果 → 触发器 → 数据中心（串行，触发器承担数据透传）
• 定版：VLM SDK → ③数据直送数据中心（数据通道）+ ②状态回调给触发器（控制通道）→ 并行双通道，触发器不碰VLM数据
• 原因：触发器是信号处理模块，不该承担数据透传职责；VLM数据量大、频率高，经触发器中转增加延迟且触发器无法解析利用

AlwaysOn 关键逻辑（7.6会议确认）
环节机制会议原话 / 结论
谁负责周期调度 触发器只负责"上电开、下电关"，3s/5s/5min的周期循环由VLM SDK内部控制 丁彬："端侧开关，但不负责里头的小的周期处理"
斯华/Walter："周期性我们肯定是内部保证的，只要你告诉我让我干活，我就自闭环"
被高优打断 P0 oneshot（如VQA、下车安全）到达时，AlwaysOn(P3)自动挂起让出摄像头；oneshot完成后VLM SDK自行恢复AlwaysOn 斯华："有那个状态回调，中间一直被打断，也会有结果的返回"
丁彬："它是个低优的任务"
数据通道 VLM SDK将原始输出发布到topic → 数据中心(李鑫)订阅 → 整包缓存 → 整包透传云端。触发器全程不经手VLM数据 汪帅："AI Box产生的事件，结果返回是直接到数据中心"
丁彬确认："数据中心订阅斯华...斯华就把结果给到李鑫，李鑫整包全部给到云端"
控制通道 VLM SDK同时回调触发器㉔-9，只传status（success/empty/timeout/failed），㉔-9据此更新任务状态、安排下一周期 张兰若总结："VLM应该既返回给触发器，又返回给AI Service，只是返回的信息不同——一个是任务成功性保证，一个是结果下传"
AIBox异常 如果AIBox坏了/未启动，VLM SDK内部做鲁棒处理（如初始化时AIBox未起来，SDK内部等它起来后自动启动任务） 汪帅："AI Box内部要保证鲁棒性，自己做重试。不能要求触发器不断重试，我只通知一遍"
丁彬："内部轮询保证"
订阅方式 数据中心通过具体topic订阅，不是全量订阅。目前只订阅3个AlwaysOn任务的topic 斯华："有topic的，具体的topic"
丁彬："数据中心就订阅这三个就行，别的就不要订阅了"

触发失败重试 OneShot触发失败 → 触发器进行一次重试 → 二次失败停止触发 丁彬："触发失败由触发器进行一次重试"
汪帅确认："触发器进行一次重试就行"
同步 vs 异步 调用VLM SDK接口：发送状态同步返回（立刻知道发送成功/失败），推理结果异步回调（模型推理需要1-2秒） 斯华："调用成功会给你结果的，最后的结果是异步回调回去"
丁彬："结果异步，状态是同步的"
OneShot抢占 P0 oneshot到达 → AlwaysOn(P3)自动挂起让出摄像头 → oneshot完成后AlwaysOn恢复 斯华："oneshot来了，周期性的就会被搁置"
丁彬："它是个低优的任务"
数据回传路径 待评审。AlwaysOn双通道已定版，但OneShot的数据如何到云端还没定 丁彬："one shot等需求导入再聊"
汪帅："先定一种逻辑，AI Box数据直接给数据中心，触发器找他订阅"
empty不是异常 VLM"看了没看到"是有效结论（如后方确实没有充电桩），不算执行失败 产品逻辑：empty上报云端后，云端不触发开充电口盖动作
