司机 Agent 2.0 双层架构全图

筛选 Agent（领班）+ 思考者 Agent（服务员）协作机制与数据流向
一句话理解：把系统想象成一家餐厅。 筛选 Agent 是前厅领班，站在暗处观察全场，决定"这桌客人现在需要什么菜谱"（加载/卸载知识）。 思考者 Agent 是直接面对客人的服务员，拿着领班给的菜谱，结合自己看到的、记住的，去跟客人聊天、动手干活。 你 Excel 里写的多轮示例，就是服务员手里的实战菜谱。

推理层 5 条筛选原则
1
场景绑定推理：结合 query + 历史 + 状态 + 视觉，推理当前具体场景，再匹配知识 ID
2
关注环境变化：趋势比绝对值更具决策价值。温度骤变、雨天、入地库都可能自动触发知识
3
目标持续关联：只要 goal_list 中的目标存在，支撑该目标的知识应持续保留
4
增量更新原则：对比已加载知识，仅输出变化部分（新增/删除），减少下游噪声
5
信息不足时不虚构：需求模糊且无法推断时，宁可漏掉，不要猜错

对话历史回流（关键闭环）
思考者 Agent 的每一轮输出（说的话 + 执行的动作）都会进入 conversation_history，成为筛选 Agent 下一轮判断的输入。这意味着：你写的多轮示例不是孤立的，而是在持续影响系统的知识调度。

这个架构如何约束多轮示例的写作？

1. Pre-state 必须对齐 8 维输入
示例开头要交代车辆状态（驻车/行驶）、用户记忆、目标队列、已加载知识。因为筛选 Agent 就是用这 8 个维度判断场景的，思考者 Agent 的多轮回复也必须基于同样的上下文基石。
2. 每轮必须有 State Update
真实对话是信息逐步揭露的。每轮示例必须显式写出"这轮之后模型知道了什么新信息"。如果某轮结束后状态没变化，这轮就是冗余的，删掉。
3. 必须覆盖工具调用链路
思考者 Agent 不是只说话，它要查车辆状态、查天气、查记忆、调空调。多轮示例要展示"先查后做"的完整链路，而不是直接假设所有信息已知。
4. 必须展示边界情况
用户会改主意、会拒绝、会提供矛盾信息。车辆状态会突变（急刹、入地库）。硬件会不可用（后排无屏）。示例必须包含这些异常路径，否则模型只学了"顺风局"。
5. 静默执行 vs 语音反馈
车端很多操作不需要语音确认（自动雨刮、切歌）。多轮示例必须区分"需要确认的操作"和"静默执行的操作"，教模型何时说话、何时沉默。输出 JSON 中 talk_or_not 就是控制这个的。
6. 对齐聊天风格与情绪
思考者 Agent 是"温柔聪明的高知女生"，会生气、会好奇、会磕巴。多轮示例里的系统回复不能是机器人腔，而要像真人服务员。用 [开心地说]、[轻声说] 等情绪标注。
