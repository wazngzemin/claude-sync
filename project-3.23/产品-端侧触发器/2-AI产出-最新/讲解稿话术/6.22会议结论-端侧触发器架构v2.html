一、最大的结论：端侧闭环场景全部回CDC 6.22会议结论

会上仁杰提了一个判断标准，所有人都认了，这是整场会最大的scope变更：

仁杰 13:14：像那种车辆安全、上车安全、下车安全，它是所有车一个标配的功能，不会变，不涉及条件变化或用户改配置，那这种直接底软做掉，不需要走触发器去绕。像那种有生命周期的、用户临时创建、消费完就没了、条件可能变化的，那走触发器。
仁杰 15:32：那种可运营的，比如车企突然想出一个场景，他可以放在端侧做，但他是临时的，动态的，会变化的，这种你可以通过触发器做。

丁彬确认：

丁彬 16:17：那看来原来端侧闭环的，现在还继续是闭环的，就几乎不需要触发器去做什么。

西钺也拍了：

西钺 16:26：对，我觉得这样也很清晰，相当于所有的这种触发都在客户那边嘛，我们就其实只用云端的那些，然后到时候跟他做一个系统的仲裁就行了，这样也很干净。
判定标准（一句话版）：

固定 / 不变 / 标配 / 车辆整个生命周期都不会变 / key-value纯规则 → CDC底软闭环
动态 / 有生命周期 / 用户创建 / 可运营 / 条件会变化 → 走触发器

哪些场景回CDC——逐个对照

场景归属为什么会上谁说的
起步安全CDC所有车标配，纯key-value判定，不涉及动态配置仁杰 14:32
上下车安全CDC固定规则，不需要用户改仁杰 13:14
睡眠提醒CDC常驻固定功能，"没必要做触发器去找"仁杰 16:01
无感迎宾CDC系统feature，不是场景引擎会上隐含
充电大师CDC赛力斯自己的业务逻辑会上隐含
哨兵模式CDC赛力斯端侧闭环会上隐含
温控CDC固定温度阈值，不需要AI参与会上隐含
—以上全部归赛力斯CDC底软，触发器不做
圣诞播放圣诞音乐触发器运营临时配的，动态的，有时效丁彬 15:46 举例
"宝宝毯子掉了提醒我"触发器用户临时创建，有生命周期丁彬 14:07
"10分钟后关座椅加热"触发器用户设的条件任务之前已定
"到公司了提醒我拿外卖"触发器动态地理围栏 + 用户query之前已定

这个结论带来的行动项

  • 产品(我) 把之前PRD里"端侧任务②简单条件判定"、"端侧任务③白名单车控直执"删掉，这些全归CDC了
  • 产品(我) 跟赛力斯确认：他们内部那些固定闭环场景（哨兵、迎宾、温控等）的仲裁机制——西钺提醒了（17:14），赛力斯各业务之间同一时机多个提醒没做仲裁，体验有问题，要推他们理
  • 开发 端侧触发器不需要接CDC闭环场景的信号了，scope收窄，只做云端下发的动态任务

逐步拆解（带case）

1
⑳Director 接到用户query"宝宝毯子掉了就提醒我"，识别为规则型任务（"当X就Y"），在 ⑱GoalList 建目标"监控后排宝宝状态"，创建 ⑰动态Advisor 实例 [云端]
↓
2
⑰动态Advisor 拆解query，发现需要视觉类条件，改写出一条注意力SP：比如 "持续观测后排座椅区域，上报毯子/被子/覆盖物的位置状态"。然后调 ⑮云端触发器 注册这个视觉场景 [云端]
仁杰 33:53 讲了这段链路的起点："Director会先去创建一个动态Advisor，动态Advisor会根据query去拆解，比如毯子掉了这个事需要用到视觉模型，就需要去生成对应的注意力SP"
↓
3
⑮云端触发器 做端云分类（步骤h），判断这个视觉任务需要端侧VLM执行，通过 L14 把动态SP下发给 ㉔端侧触发器 [云→端]
↓
4
㉔端侧触发器 收到场景，发运行指令给 ㉕感知器（VLM调度器）。㉕把这条新的注意力SP 增量追加到端模的prompt里（不是覆盖，不然原来的观测项就没了）[端侧]
丁彬 01:03:12 强调："你本质上是对当前的prompt做一个增量，如果不是增量，就会导致原来的prompt就没了，原来感知的内容也没了"
↓
5
㉕端模(VLM) 按周期观测（每3-5秒），输出纯客观状态描述，写入 ④端侧信号库。比如每次输出："后排蓝色毛毯在座椅上"、"后排蓝色毛毯在座椅上"...某次变成"后排蓝色毛毯在地板上" [端侧]
丁彬 40:10 讲端模定位："端侧现在目前模型，它跟业务都是解耦的，它现在就是作为眼睛去观测...目前还没有融入逻辑"
西钺 46:12 确认输出方式："我们还是纯客观地描述此时此刻的状态，不描述它是或否，坛子是否掉地上不描述，我们只说坛子当前在地上"
↓ L9 端信号同步上云
6
⑮云端触发器 通过 ⑫云端信号库（④的镜像）监听信号变化。检测到阈值穿越："座椅上→地板上"，执行步骤j——组装成语义化事件"毯子位置变化" [云端]
佳杰 24:05-24:31 描述这段："视觉类的信号从端模来，进入信号库，触发器会监听这些信号的变化情况，也许有一些逻辑，可能是阈值穿越，可能是首次出现...组装成事件，作为高度简化的语义，Advisor就可以消费这些事件了"
↓ push（不是轮询！）
7
⑮ 把事件 push 给 ⑰动态Advisor。⑰消费事件，可能调VQA进一步确认（"毯子是否掉地上？宝宝还盖着吗？"），然后给 ⑲Director 发advice："毯子掉了，建议提醒用户" [云端]
↓
8
⑳Director 收到advice + Context("后排儿童3岁") → 推理 → ㉖TTS："检测到宝宝的毯子掉到地上了哦，要不要在下一个服务区停一下帮宝宝盖好？" [云→端]

端模能力边界——会上讨论的三个卡点

VLM链路的关键前提是端模能做到什么程度。会上丁彬讲了现状（40:10-43:21）：

问题现状会上谁说的
端模能不能按prompt限制只看特定东西？ 有限集内（毯子、毛巾等固定几类）通过prompt限制，效果基本能保证。发散了靠基模能力，效果不确定。"通过prompt能限制住，得试一下，目前还没试过" 丁彬 40:58
端模能不能做到"只在看到X时才上报"（条件式注意力）？ 仁杰问了（40:38），丁彬说理论上可以但没试过。如果能做到，后面链路会简单很多 仁杰 40:38, 丁彬 40:58
端模自己做判断（"毯子掉了"）还是只报状态？ 目前结论：只报客观状态。"坛子当前在地上"——掉没掉是上层判 西钺 46:12, 丁彬 46:24
卡点预警：以上三个问题中，"条件式注意力"直接决定VLM链路的复杂度。如果端模能做到"看到毯子掉了才上报"，就不需要云端持续判断每次输出；如果做不到，就需要每3秒的输出都同步上云，⑮每次都做阈值穿越检测，计算量和带宽都大很多。丁彬说得试一下。

两类视觉事件的走法区别

西钺和丁彬在后半段（01:00:14-01:02:35）讨论清楚了这个分类：

有限集事件（可穷举）

睡醒、驶出地库、打电话...这些能提前列出来的。

走法：预置在端模SP里，端模每次观测都带着这些观测项，周期上报。

优点：稳定，可预训练调优。

丁彬 01:00:19 说得很清楚："如果是静态的、我们预训练好的，不需要动态Advisor了，静态Advisor也能满足。"

通用视觉事件（泛化兜底）

"帮我看宝宝毯子掉没掉"——用户临时提的，事先不知道要看什么。

走法：动态Advisor改写SP → ⑮触发器下发 → ㉔端侧触发器 → ㉕端模增量追加prompt → 新增观测项。

风险：效果不一定好，佳杰说可以让豆包先问"这个我可能做得还不是那么好，你确定要让我试试吗？"（01:04:33）

VLM链路的行动项

  • 产品(我) 约佳杰+益红+丁彬的会，专门聊VLM链路的工程细节：SP怎么下发、数据怎么拿、和AIBox怎么打通（佳杰 47:57 主动提了这个会）
  • 产品(我) 等佳杰和益红的视觉事件清单（周内），拿到后对照触发器支持的信号类型看缺口

  • 开发 ㉔端侧触发器要支持接收云端下发的动态SP，转发给㉕感知器（增量追加，不是覆盖）
  • 开发 丁彬那边要验证端模的"条件式注意力"能力——能不能做到"看到X才上报"

四、事件 vs 条件（Context）：核心设计选择 6.22会议结论

这是6.22最核心的架构决策，贯穿全篇。佳杰用了一个例子把这事讲得特别清楚（29:34）：

Case："用户倒完车之后，如果车里有小朋友，就把儿童锁打开"
  • "用户倒完车" = 事件（event） → 等 ⑮ push。不能24小时轮询"用户在不在倒车"，太浪费
  • "车里有小朋友" = 条件（condition） → 存在 ㉑Context 里，Advisor要用的时候去读
  • "把儿童锁打开" = 动作（action） → Advisor做出决策后执行
事件 vs 条件 的数据流向
  ④端侧信号库                    ㉑Context
       │                             ▲
       │ L9同步                      │ 按策略上报
       ▼                             │
  ⑫云端信号库 ──────────────────→ 存condition/状态
       │                          "车里有小朋友"
       │                          "车内温度32度"
       │                          "坛子当前在地上"
       │
       │ ⑮监听变化

  j.事件组装

佳杰的原话，把这个设计选择说到位了：

佳杰 26:35：我需要一个用户睡醒了的这样一个事件，我不会去轮询你的Context，我是注册了这个事件之后，我就等待你的云端触发器给我发这个事件。
佳杰 29:34：Context拿的是另外一部分的信息，它是作为condition存在的...第一趴"车在倒车"这个事件，我不可能无限制的24小时在轮询，这就产生了触发器的需求。

  5. ⑰拆解：需要持续视觉观测 → 改写注意力SP → 向 ⑮ 注册"后排覆盖物位置变化"事件

八、任务中心对接需求 6.22会议结论

这块是佳杰在51:07-55:08提出来的，之前没聊过，会上第一次摆到台面。

佳杰 51:59-52:33：你需要第一个需要建设的能力就是端云的这种任务同步的一个机制...比如说我配置了一些定时任务，或者是我预设了一些条件任务，这部分任务它能够在端上有个数据库做持久化。

佳杰需要触发器提供什么

需求说明佳杰原话
端云任务同步 云端配置的任务同步到端侧持久化，弱网也能用 "你必须第一个需要建设的能力就是端云的这种任务同步的一个机制"（51:59）
任务级 vs 感知级字段 触发器的数据分两种：用户能看到的"任务"（充电桩自动开盖）vs 底层"感知"（注册了某个信号订阅）。任务中心只显示任务级的 "你可能额外会有一些字段来区分它是任务级的数据，还是只是一个感知级的数据"（52:33）
enable / disable 接口 用户在任务中心UI上关掉某个任务，触发器要真的停掉 "某个充电桩的任务我关掉，你要去能够真的把这个任务给关掉"（55:08）
删除 / 找回 从待办移除 / 从场景中心找回 "删除就是我不在待办里显示，添加回来就是我从场景中心又点了找回"（56:05）

九、AlwaysOn调度归属 6.22会议结论

西钺问了（57:25）：AlwaysOn的调度是触发器做吗？

佳杰 57:47：AIBOX上有一个模块是专门去对所有的请求做优先级的排序的。AlwaysOn优先级大概是多少、VQA的优先级如何、其他视觉类需求的优先级如何，如果优先级低的，它可能会排队，会等。
结论：AlwaysOn调度在佳杰的AIBOX任务调度模块里，不在触发器里。触发器负责的是"什么时候触发"，不负责"谁先谁后执行"。

但是㉔端侧触发器内部也有优先级（P0-P4），这个是管触发器自己的场景之间谁先执行，跟AIBOX的任务调度是两层不同的优先级。
