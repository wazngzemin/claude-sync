触发器真实机制 · 注册与触发两块

按真实架构讲透 · 以"后排宝宝哭了告诉我"贯穿 · 只用图里真实字段 · v1
先立个纲，后面所有细节都挂在这上面：
触发器就干两块事：①注册（把 task 注册进来）　②触发（判断这个 task 的条件达没达成）。
注册解决"我要盯什么"，触发解决"盯到了没、盯到了就通知"。下面一块一块讲，讲到每一步的输入、输出、判断逻辑。
第一块 · 注册（task 注册进来）

1. 什么叫"注册"

用户提一个需求，比如"后排宝宝哭了告诉我"。这句话没法直接拿去感知，得先变成一个结构化的任务（task），登记进触发器——这个登记动作，就叫注册。注册完，触发器手里就有了一张"要盯的任务清单"。

2. 一个 task 注册进来长什么样

注册的 task 里，几个关键字段（图里真实字段）：

// 注册动态任务 { "user_goal": "后排宝宝哭了告诉我", // 用户目标（原始诉求） "感知目标": "后排宝宝哭了", // 要判断达成的那个目标 "execute_text": "告知用户：后排宝宝哭了", // 触发后要执行什么 // 还有 task 的类型：[条件] [时序] [事件] [定时] }
字段干嘛的
user_goal用户的原话诉求，留底
感知目标触发判断时要比对的目标——触发器就是拿 VLM 结果去比它
execute_text一旦判断"达成了"，要执行的动作内容（告知用户）
类型（条件/时序/事件/定时）这个 task 是靠什么方式触发的

3. 谁把它注册进来的

1
用户说 "后排宝宝哭了告诉我"（query）
2
经 Director / Advisor 拆解，识别出这是个"要持续感知的动态任务"
3
Task Service（任务中心）创建 task：填好 user_goal、感知目标、execute_text
4
task 注册进云端触发器（触发器做"动态任务注册"）；同时下发端侧，端侧 VLM 开始周期感知后排
注册这块记一句就够：用户诉求 → 变成带"感知目标 + execute_text"的 task → 登记进触发器的待盯清单。感知目标是后面触发判断的"标准答案"，execute_text 是达成后要干的事。
第二块 · 触发（判断是否达成条件）

注册完，触发器就开始盯。盯的过程就是：不断拿 VLM 的识别结果，去比对 task 的感知目标，判断达没达成。这是触发器最核心、最该讲细的地方。分三步讲：结果怎么来（两条链路）→ 怎么判断达成 → 达成后怎么办。

1. VLM 的识别结果，两条链路进触发器

端侧 VLM 周期性输出识别结果（描述性文本），到达云端触发器有两条链路：

链路 A：直接从云端 Context 拿绿框

端侧 VLM 的常规周期感知（AlwaysOn），由数据中心整包缓存、按格式上传到云端 Context。触发器订阅 Context，直接拿这里的结果。Context 里长这样（绿框那部分）：

{ "乘客列表": [ { "位置": "", # 位置枚举值 "动态": "" } # 动态描述，含动作和神态 ] }

这是"常规感知"——不管有没有动态任务，AlwaysOn 一直在描述车里乘客状态，整包传上来放 Context，触发器订阅着看。

链路 B：task 任务管理，云端接收后转发给 Trigger红框

针对具体注册任务的识别结果，走"动态任务"这条——端侧上报云端后，由云端接收、转发给 Trigger。数据长这样（红框那部分）：

{ "动态任务": [ { "任务编号": "xx", # 对应哪个注册的 task "识别结果": "xx" } # user_query 对应的回答内容 ] }

这条链路的流程（图 18）：

10
端侧定时上报云端（动态任务表：任务编号 + 识别结果，如"后排儿童蜷缩，趴上有羽绒"）
11
云端接收并转发 Trigger
12
Trigger 接收，筛选该车辆活跃的 VLM 条件 Task（这辆车现在有哪些注册着的、正在盯的任务）
13
条件匹配：这条识别结果，满不满足"后排宝宝哭泣"？
两条链路的区别（关键）：链路 A 是触发器被动订阅 Context 里的常规乘客感知（绿框，一直在传）；链路 B 是动态任务的识别结果走task 管理专门转发给 Trigger（红框，带任务编号，是冲着某个具体注册 task 来的）。一条是"泛感知放公共 Context 里等你看"，一条是"针对性结果专门送到 Trigger 手上做匹配"。

2. 拿到结果，触发器怎么判断"达成没"（最核心）

这一步是触发器的心脏。它不是"信号 = 某个值"那种硬比对（那是车态信号的玩法）。VLM 给的是描述性文本（"后排儿童蜷缩，趴上有羽绒"），要判断这段描述满不满足"后排宝宝哭了"——这是语义判断。逻辑是这样：

// 触发判断（图里真实写法） 输入 = "识别结果"（"后排儿童蜷缩，趴上有羽绒"） + 任务的"感知目标"（"后排宝宝哭了"） 判断条件 = 从「Task Service」找（这个 task 的达成标准在哪，去 Task Service 取） 判断是否达成：语义 + 轻量模型判断 ├─ 没达成 → 等下一条（5 秒周期）上报的结果，再判 └─ 达成 → 通知 Task Service

把这段掰开讲：

  1. 输入是两样：VLM 这一帧的"识别结果" + 这个 task 的"感知目标"。一个是"现在看到啥"，一个是"要看到啥才算数"。
  2. 标准去哪找：判断条件（达成标准）从 Task Service 取——Task Service 存着每个 task 的判定规则。
  3. 怎么判：用语义 + 轻量模型判"识别结果"是否命中"感知目标"。因为是文本比文本、含义比含义，不能用等于号，得靠语义模型判。（"儿童蜷缩趴着"算不算"哭了"？轻量模型来判。）
  4. 没达成不放弃：这一帧不满足，不结束，等 5 秒后端侧上报的下一帧结果，再判一次——因为宝宝可能这一秒没哭下一秒哭。周期性地反复判，直到达成或任务被撤。
  5. 达成了：通知 Task Service，说"这个 task 达成了"。

3. 达成之后怎么办

1
触发器判定达成 → 通知 Task Service
2
取出这个 task 的 execute_text（"告知用户：后排宝宝哭了"）
3
交给 Director 推理，拆解成执行动作：talk_or_not、talk_content（含情绪/emoji）、action_list
4
执行 → 告知用户"后排宝宝哭了"；任务达成后可注销（不用一直盯了）
完整走一遍："后排宝宝哭了告诉我"
1
【注册】用户说"后排宝宝哭了告诉我" → Director/Advisor 拆解 → Task Service 建 task（感知目标="后排宝宝哭了"，execute_text="告知用户…"）→ 注册进触发器 + 下发端侧
2
【端侧感知】端侧 VLM 周期性看后排，输出识别结果（"后排儿童蜷缩，趴上有羽绒"…每 5 秒一帧）
3
【上报·两链路】常规乘客描述走 A 进 Context（绿框）；这个任务的识别结果走 B（红框）定时上报→云端转发 Trigger
4
【触发判断】触发器拿 识别结果 + 感知目标，去 Task Service 取标准，语义+轻量模型判：像"哭了"吗？
5
【没达成】这帧是"蜷缩趴着"，判不像哭 → 等 5 秒下一帧，再判
6
【达成】某一帧识别结果=脸涨红张嘴带哭腔 → 判达成 → 通知 Task Service
7
【执行】取 execute_text → Director 推理拆 action_list → 播报"后排宝宝哭了" → 注销任务
一句话收总：触发器 = 把用户诉求注册成带"感知目标+execute_text"的 task；再不断拿 VLM 识别结果（两条链路来）跟感知目标做语义判断，达成就通知 Task Service 执行。核心难点在触发判断是语义的、周期反复的，不是硬比对——这正是它跟普通车态条件（信号=值）最不一样的地方。
