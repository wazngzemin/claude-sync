# 7.10 SLS端状态接入Context + 触发器需求评审

**时间**：2026-07-10 (周五) 16:00–17:05
**主讲**：韩杰
**参会**：赵衍(端侧)、刘杨(触发器研发)、向超然(云端)、于庆磊、李睿、陈民瑞、王泽民、李明骏

---

## 一、核心议题：信号中枢三表统一

韩杰将原来分散的端状态、触发器条件表统一为"信号中枢"(Signal Hub)，用一张大表的三个 sheet 统管所有信号相关能力：

| Sheet | 内容 | 消费方 |
|-------|------|--------|
| **状态查询** | 端侧信号原始值（112条，从数百条缩减） | Context（车辆状态+事件感知） |
| **触发事件** | 状态变更瞬间生成的事件（原"动态条件"） | Advisor / 事件日志 / 规则引擎 |
| **条件** | 静态条件判断（当前态是/否） | 作为漏斗过滤触发事件 |

三表通过关联字段互相映射：触发事件关联来源状态、条件关联来源状态。

## 二、架构变化：并行→串行漏斗

**旧架构**（配置平台）：N个条件并行判断，全部满足→执行动作
**新架构**：触发事件（状态突变检测）→ 条件判断（串行过滤）→ 执行动作

```
状态上报 → 订阅状态突变 → 生成触发事件
                              ↓
                     条件判断（串行/并行/延时）
                              ↓
                     执行动作（可挂多类型）
                     ├── 触发 Advisor
                     ├── 生成事件日志 → Context → Director
                     ├── TTS 播报
                     └── [未来] 规则引擎 / 条件任务 / 任务大师
```

**拆分四个理由**（韩杰陈述）：
1. Advisor和事件日志都需要"明确的触发理由"——谁在几点做了什么事，触发事件天然提供这个描述
2. 原来并行轮询所有信号算力和总线压力大；串行逐步过滤减少长时监控
3. 并行只适合简单规则（5条件同时满足）；车载场景需要延时、串行、并发等复合条件
4. 规则引擎要标准化为火山座舱产品（配置平台→运营平台），workflow画布拖拽需要原子化节点

**举例**：档位切入D档(触发事件) → 判断是否有车窗未关(条件) → TTS播报"当前有车窗打开，是否关闭？"(动作)

## 三、事件日志链路澄清

关于"事件日志"概念的争论：

| 发言人 | 立场 |
|--------|------|
| 韩杰 | 明骏下周要评一个新的"事件日志 for Director"需求 |
| 向超然 | 事件日志去年就有了，不是新概念，记忆功能就是基于它 |
| 李明骏 | "我不是要重做事件日志，而是要挑选特定事件插入Director上下文" |
| 刘杨 | 核心问题：触发器也要生成事件日志，那来源就有两个了，会不会重复？ |
| 赵衍 | 确认存在两条技术链路：①端状态变化→触发器→生成事件日志 ②端侧直接从三方拿到→上云。第二条刘杨干预不到，有重复可能 |

**韩杰的回答**：产品侧用一张表统管所有触发事件，不管来源端状态还是埋点/客户透传，需求管控统一收口。技术链路重复问题通过严格约束来源方解决。

**于庆磊追问多订阅方冲突**：触发事件+条件满足后，事件日志→Director、Advisor→Director，多条信息同步到Director会不会冲突？
**韩杰**：事件日志给Context是参考信息，Advisor是任务信息，两条并行链路。Director自己闭环仲裁。

## 四、端侧触发器定位争议

**赵衍（端侧负责人）提出三个产品层面必须定义的问题：**

1. **端侧触发器要不要有？** 离线可用=客户诉求，不是技术问题
2. **预置多少场景脚本？** 端侧内置的规则，必须产品定义
3. **运营脚本是否下发端侧执行？** 涉及配置平台→端侧的流程

**韩杰的回应**：
- 产品不会提"端侧触发器"概念，只提能力需求和性能要求
- 哪些放端侧自闭环=技术实现，由研发判断
- 但承认"哪些能力离线也要运行"是产品需求，可以输出
- 730时间周期短，当前scope全部云端，端侧场景后续单独拉会

**赵衍引用的先例**：韩杰在极氪的"场景工坊"经验——基本所有能力在端侧，云端只管API查询和大模型；配置平台配完→SOTA下发端侧→端侧本地判断执行。但当前豆包的情况不同：端状态已全量上报，再基于全量数据生成部分变化数据重新上报=流量浪费。

## 五、信号协议改动细节

| 改动项 | 旧 | 新 | 争议 |
|--------|-----|-----|------|
| 信号范围 | 数百条 | 112条 | 无争议，缩减合理 |
| 信号ID | 无规则（灯000几） | 小驼峰命名，按功能 | 超然：收益不明确，端侧全改一遍工作量大。建议云端映射 |
| 变化频率/时效 | 有字段 | 删除 | 云端只需最新值，不关注刷新频率 |
| 单位 | 无独立字段 | 韩杰答应补 | 刘杨再次提出老问题：参数单位(档/度/km/h等)缺失导致触发器使用困难 |
| 协议示例 | 无 | 新增列（待填） | 方便端云对接 |
| Query示例 | 有 | 删除 | 韩杰认为意义不大 |
| 参数关联性 | 旧表有 | 新表缺 | 陈民瑞：必须补回，要统一一个表生成配置脚本 |

**超然总结立场**：缩减和删除可以做，但信号ID重定义需要再评估收益，找时间单独聊。
