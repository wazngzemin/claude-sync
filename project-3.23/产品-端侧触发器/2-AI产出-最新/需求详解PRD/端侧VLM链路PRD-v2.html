一、需求背景

1.1 为什么需要这份 PRD

端侧VLM（视觉语言模型）是Driver Agent 2.0在上汽荣威D6X/赛力斯车型上的核心能力：通过AIBox搭载的端侧VLM引擎，持续感知车辆内外环境（天气、路面、乘客状态、充电桩等），为云端Planner/Advisor提供视觉信息。

问题是：VLM从"被触发"到"数据到达云端"这条链路，涉及4个团队5个模块（端侧触发器/VLM SDK/数据中心/云端端状态/Context），之前没有一份文档把整条链路的产品需求从头到尾写清楚。7.6下午的专项会上，链路设计发生了根本性变更——从单通道串行变成了双通道并行，必须有一份独立PRD把定版结论落地成可执行的产品需求。

四、7.6 定版变更：从串行到并行双通道

4.1 变更前（旧链路）

旧链路（7.6上午之前的理解）：

触发器调VLM → VLM执行 → 结果返回给触发器 → 触发器整包透传给AI Service → AI Service上报云端

这条链路有四个问题：
① 触发器不该碰VLM数据——触发器本质是信号处理模块（运算符匹配、穿越检测），它拿到VLM的原始JSON既不解析也不消费，纯粹当管道。但管道就意味着延迟和故障点
② VLM数据量大、频率高——AlwaysOn每3-5秒一帧完整JSON，经触发器中转增加延迟和内存开销
③ 异常处理模糊——VLM执行失败了，到底谁负责重试？触发器觉得"我只管触发"，VLM SDK觉得"你调的你负责"
④ "谁调返给谁"改造成本高——斯华的VLM SDK通过topic发布结果，不区分调用方。要实现"谁调的返回给谁"需要SDK改造，增加了工期和复杂度

4.2 变更后（双通道分离）

新链路（7.6下午定版）：VLM执行完后，结果分两条路走——

🔵 控制通道：VLM SDK → 触发器 ㉔-9
　　只传4个终态（success / empty / timeout / failed），触发器据此管任务调度状态

🟢 数据通道：VLM SDK → 数据中心 → 云端
　　原始JSON通过topic发布 → 数据中心订阅后整包透传 → 云端自行解析消费

核心原则：触发器管调度不管数据，数据中心管数据不管调度。两条通道并行，互不依赖。

张兰若总结 ~00:40："一个是任务成功性保证（控制通道），一个是结果下传（数据通道）"

十三、产品红线（6 条）

红线1 · 超时不写伪结果：VLM执行超时不能写"无障碍"/"无异常"——模型没看完，结果不可信。写了就是欺骗下游消费方，可能导致安全判断错误。
红线2 · 触发器不碰VLM原始数据：VLM的原始JSON只走数据通道到数据中心。触发器只收状态回调（4个status值），不接收、不解析、不透传VLM原始数据。7.6下午定版核心决策。
红线3 · 端云必须基于同一份规则：端侧触发器和云端触发器对同一条规则的判定结果必须一致，否则就是bug。汪帅 ~00:48原话："端云必须基于同一份规则，否则就是bug"。规则由云端下发，端侧执行。
红线4 · 数据中心整包透传，不解析：在产品明确提出拆字段需求之前，数据中心对VLM原始输出不做任何解析/拆解/过滤/聚合。丁彬原话："直接拿来就给了就行"。
红线5 · 空结果是有效结果：empty = "看了没看到"，与 timeout/failed = "没看" 有本质区别。empty必须正常写入信号库、正常透传给云端、正常参与下游决策。不能丢弃。
红线6 · P0任务不设冷却：安全类任务（下车安全/起步安全/充电桩/遗留物品）每次条件满足必须触发，不能因为"刚触发过"就跳过。延迟或漏触可能造成人身安全风险。
