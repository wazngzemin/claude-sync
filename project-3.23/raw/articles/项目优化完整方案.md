## 文档信息
- **创建日期：** 2026-03-27
- **会议类型：** 项目优化方向讨论会
- **文档版本：** v1.0

## 一、背景与现状

### 1.1 核心问题
当前系统面临的主要挑战是**模型注意力分散问题**：
- 模型在处理复杂任务时注意力会丧失或不聚焦
- 无法有效参考和利用其他context信息
- 导致任务执行质量下降，错误率上升

### 1.2 问题影响范围
- GUI Agent任务执行频繁出错
- 复杂工具调用混淆（如planner工具）
- 用户个性化需求处理不当
- 整体系统稳定性和可靠性下降

---

## 二、优化方案详解

### 方案一：示例浓缩提纯

#### 2.1.1 方案目标
通过精简和优化示例库，提升模型的学习效率和注意力集中度，减少冗余信息对模型判断的干扰。

#### 2.1.2 核心策略
**保留10+个最核心示例**
- 从现有大量示例中筛选出最具代表性的10-15个核心case
- 每个示例都需要经过抽象和优化
- 确保示例质量高于数量

#### 2.1.3 核心示例必须覆盖的能力维度

**能力1：多工具协同使用**
- 示例需展示如何在单个任务中组合使用多种工具
- 体现工具之间的配合逻辑
- 展示工具调用的先后顺序和依赖关系

**能力2：串行与并行判断**
- 教会模型识别哪些工具调用可以并行执行
- 哪些必须串行执行（存在依赖关系）
- 如何权衡执行效率和正确性

**能力3：困难场景的灵活处理**
- 遇到错误时的重试策略
- 遇到阻塞时的替代方案
- 遇到不确定情况时的决策逻辑

**能力4：个性化需求理解**
- 如何理解用户的隐含需求
- 如何处理模糊或不完整的指令
- 如何在多个可行方案中做出选择

#### 2.1.4 实施方式

**第一步：核心示例筛选**
- 从现有示例库中识别高频场景
- 评估每个示例的教学价值
- 选出10-15个最具代表性的示例

**第二步：示例优化改造**
- 对选中的示例进行重写和优化
- 增强示例的清晰度和可学习性
- 添加必要的注释和说明

**第三步：动态知识库构建**
- 将其他示例转化为"动态知识+示例"模式
- 建立知识检索机制
- 根据任务类型动态加载相关知识

**第四步：前置筛选层设计**
- 设计筛选算法，根据用户任务特征
- 从动态知识库中检索最相关的知识和示例
- 控制注入到prompt中的信息量

#### 2.1.5 预期效果
- 减少prompt长度，降低注意力分散
- 提高核心能力的学习效果
- 保持知识覆盖的完整性（通过动态检索）

### 方案二：工具复杂度简化

#### 2.2.1 问题背景

**GUI任务的特殊性**
- GUI任务本身难度较高，涉及复杂的界面交互
- 需要理解界面元素、布局、操作流程
- 错误容忍度低，一步错误可能导致整个任务失败

**Planner工具的复杂性**
- 当前planner工具要求输出两种不同类型的参数：
  1. **创建任务参数**：用于创建新的子任务
  2. **补充信息参数**：用于向现有任务补充上下文
- 模型经常混淆这两种输出模式
- 导致任务创建错误或信息补充失败

**下游工具难度分析**
- 工具本身的复杂度成为模型的负担
- 模型需要同时理解任务目标和工具使用规则
- 认知负荷过高导致错误率上升

#### 2.2.2 优化思路

**核心理念：职责分离**
- 让planner专注于任务规划，只输出自然语言
- 用专门的小模型处理工具调用细节
- 通过工程分解降低单个模型的复杂度

#### 2.2.3 具体实施方案

**改造Planner工具**

*改造前：*
```
Planner输出 → {
  type: "create_task" | "supplement_info",
  task_description: "...",
  context: {...},
  dependencies: [...]
}
```

*改造后：*
```
Planner输出 → "我需要创建一个任务来处理用户的登录请求，需要先验证用户名和密码"
```

**引入Flash模型做二次处理**

*Flash模型的职责：*
1. 接收planner的自然语言输出
2. 分析判断这是"创建任务"还是"补充信息"
3. 将自然语言转换为结构化的工具参数
4. 注入GUI相关的专业知识

*Flash模型的优势：*
- 模型更小，推理更快
- 专注于单一任务，准确率更高
- 可以独立优化和迭代
- 降低主director的认知负担

**工程架构调整**

```
用户请求
  ↓
Director (主模型)
  ↓
Planner (输出自然语言规划)
  ↓
Flash模型 (参数转换 + 知识注入)
  ↓
工具调用 (结构化参数)
  ↓
执行结果
```

#### 2.2.4 适用范围扩展

**生图工具优化**
- 生图工具同样存在双参数输出问题
- 应用相同的优化思路：自然语言描述 + Flash模型转换

**通用原则**
- 所有需要输出复杂参数的工具都考虑此方案
- 所有容易混淆的工具都在前面加思考层
- 用小模型处理细节，用大模型做决策

#### 2.2.5 预期效果
- 降低planner的错误率
- 提升GUI任务的成功率
- 架构更清晰，易于维护和优化
- 可以针对性地优化各个环节

### 方案三：补充知识和示例

#### 2.3.1 Badcase驱动的优化

**优化流程**
1. 收集和分类badcase
2. 分析每个badcase的根因
3. 判断是缺知识还是缺示例
4. 针对性地补充内容

**知识补充场景**
- 模型对某个领域概念理解不足
- 缺少特定场景的处理规则
- 需要明确的边界和约束

**示例补充场景**
- 模型不知道如何处理某类任务
- 需要展示具体的思考过程
- 需要演示工具的正确使用方式

#### 2.3.2 动态检索机制

**为什么需要动态检索**
- 不可能把所有知识都放在prompt中
- 需要根据任务类型加载相关知识
- 平衡知识覆盖度和prompt长度

**检索策略**
1. 任务分类：识别用户任务的类型和领域
2. 知识匹配：从知识库中检索相关知识
3. 相似度排序：按相关性排序
4. 动态注入：将top-k知识注入prompt

**知识库结构**
```
知识库 {
  领域知识: {
    GUI操作: [...],
    文件处理: [...],
    网络请求: [...]
  },
  示例库: {
    多工具协同: [...],
    错误处理: [...],
    个性化需求: [...]
  },
  规则库: {
    安全规则: [...],
    业务规则: [...],
    交互规则: [...]
  }
}
```

#### 2.3.3 实施步骤
1. 建立badcase收集和分类机制
2. 构建结构化的知识库
3. 实现检索算法
4. 持续迭代和优化

## 三、GUI Agent专项优化

### 3.1 紧急程度
- **截止日期：** 2026-03-30
- **优先级：** 最高
- **原因：** 需要交付成果，问题较多

### 3.2 问题详细分类

#### 问题类型1：指令理解错误

**具体表现**
- 用户发出澄清类指令（如"我是想让你帮我打开设置页面"）
- 模型误判为闲聊，回复"好的，我明白了"但不执行任务
- 导致任务流程中断

**根本原因**
- 缺少对"澄清指令"的明确定义
- 没有足够的示例教会模型识别这类指令
- 闲聊和任务指令的边界模糊

**解决方案**

*方案A：知识补充*
- 在prompt中明确定义什么是澄清指令
- 列举澄清指令的常见模式
- 强调澄清指令需要转化为任务执行

*方案B：示例强化*
- 添加3-5个澄清指令的正确处理示例
- 展示如何从澄清中提取任务信息
- 演示澄清后的任务执行流程

*方案C：规则约束*
- 添加明确的判断规则："当用户在任务上下文中发言时，优先理解为任务相关指令"
- 设置闲聊的触发条件（如明确的问候、无关话题）

**实施计划**
1. 第一周：补充知识定义和规则
2. 第二周：添加示例并测试
3. 第三周：收集反馈并迭代

#### 问题类型2：工具参数混淆

**具体表现**
- Planner需要输出两种参数：创建任务 vs 补充信息
- 模型经常输出错误的参数类型
- 导致任务创建失败或信息丢失

**根本原因**
- 工具设计复杂，要求模型做多重判断
- 两种模式的触发条件不够清晰
- 模型的认知负荷过高

**解决方案**

*核心方案：架构重构（详见方案二）*
- Planner只输出自然语言
- Flash模型做参数转换
- 分离规划和执行

*短期缓解方案*
- 在prompt中强化两种模式的区别
- 添加判断流程图
- 提供更多对比示例

**实施计划**
1. 短期（1周内）：添加强化说明和示例
2. 中期（2-3周）：实施架构重构
3. 长期：持续优化Flash模型

#### 问题类型3：任务难度高

**具体表现**
- GUI任务涉及复杂的界面理解
- 需要多步操作的精确协调
- 错误率明显高于其他类型任务

**根本原因**
- GUI任务本身的复杂性
- 缺少GUI领域的专业知识
- 界面元素识别和操作的不确定性

**解决方案**

*方案A：GUI知识注入*
- 构建GUI操作知识库
- 包含常见界面元素的识别方法
- 包含标准操作流程和最佳实践

*方案B：分步执行策略*
- 将复杂GUI任务分解为小步骤
- 每步执行后验证结果
- 支持回滚和重试

*方案C：专家模型*
- 训练或微调专门的GUI理解模型
- 用于界面元素识别和操作规划
- 与主模型协同工作

**实施计划**
1. 第一阶段：构建GUI知识库
2. 第二阶段：实现分步执行
3. 第三阶段：探索专家模型方案
