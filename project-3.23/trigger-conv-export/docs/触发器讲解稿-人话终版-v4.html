一、这套系统是干嘛的 + 云端（cloud）车端（edge）怎么分

触发器（trigger，满足条件就去干活的机器）解决的是——用户没开口、但车该主动做事的那些场景（scenario，具体的使用情境）。

用户14:00说"十分钟后关座椅加热"→ 这不是让车现在就关，是让车等10分钟（condition，条件）→ 14:10到了（条件满足）→ 自动关（action，执行动作）。
用户说"看到我妈上车提醒我"→ 条件是"FaceID（人脸识别）识别到妈妈"→ 满足的那一刻 → 自动TTS（Text-to-Speech，文字转语音）播报"妈妈上车了"。
用户说"副驾睡着了把音乐调小"→ 条件是VLM（Vision Language Model，端侧视觉语言模型）检测到"副驾闭眼低头超过30秒"→ 满足的那一刻 → 自动把音量从60调到20。
一句话分工：
云端（cloud，远程服务器） = "登记（register） + 拍板（decide）"——规则（rule，触发条件+执行动作的组合）怎么定、冲突（conflict，两个规则打架）谁优先（priority，优先级）、复杂判断（需要大模型推理的）。
车端（edge，车上本地处理器） = "感知（sense） + 执行（execute）"——等不了云端的事（断网、太慢、太频繁），本地办掉。
铁律：同一条规则，绝不能云端和车端两边都判（duplicate judging，重复判定）——否则结论打架。比如云端判"温度没超标"、车端判"超标了"，到底听谁的？所以只能一个地方判。
端侧触发器㉔的断网能力边界（6.22+6.24综合结论）：6.24修正

断网能跑的（简单条件任务）：本地定时器/倒计时、白名单车控（TTS播报/座椅加热/空调调温等）、简单数值条件判定（温度>30等）。这些全在车端㉔本地完成，不需要云端。

断网做不了的（复杂场景）：需要VLM+云端推理的任务（如"P档看后面有没有充电桩→开盖"）、需要调云端API的（天气/POI搜索/web search）、需要大模型语义判断的（"副驾是不是在睡觉"）。

6.24 丁彬+佳杰确认原话："如果没有云端怎么办？""这个任务就做不了，就不开盖了。端上没有去拆解任务的能力。"

所以更精确的说法是：端侧触发器的价值 = "简单条件任务断网能跑" + "本地定时器断电不丢" + "本地VLM持续观测写信号库"。不是所有任务都能断网跑。

二、一句话进来怎么走（主链路 + 四路分流详解）

用户说一句话，⑨意图分流模块（Intent Router，意图路由器——判断用户想干嘛）做四路分流（4-way routing）——判断这句话属于哪一类，走哪条路：

分流结果走哪条路举例为什么分开实际case
①车控指令
（Vehicle Control）		端侧⑤离线fc（function call，函数调用）+白名单（whitelist，允许清单）→ ⑥命中（match，匹配上了）? → ㉘直接执行"打开车窗"、"空调调到24度"这种不需要大模型（LLM），白名单直接映射（mapping，一一对应）执行，最快——延迟（latency）<200ms用户："开车窗"→ASR转文字→⑨判断是车控→⑤查白名单命中"open_window"→⑥确认→㉘发CAN指令给车身控制器→车窗打开。全程不过云端，200ms内完成
②复杂需求
（Complex Request）		L5 → ⑲Director（决策者）汇入 → ⑳Director推理（inference，模型计算）"帮我照看后排宝宝"需要大模型拆解（decompose，把一件复杂的事拆成多个步骤）、创建监工（Advisor实例）、多步骤用户："帮我照看后排宝宝，毯子掉了提醒我"→⑨判断是复杂需求→送Director→Director拆解成：1.建动态Advisor（临时监工）2.注册VLM观测任务3.设阈值穿越检测→最终18步链路（见§7）
③条件任务
（Conditional Task）		⑩意图解析（Intent Parse，理解用户说的条件和动作分别是什么）→ ⑬配场景（Configure Scenario，在配置平台生成规则）→ ⑮注册（Register）"10分钟后关座椅加热"、"到公司提醒我拿外卖"要等条件满足再执行，需要注册（register，登记）到触发器用户"到公司提醒我拿外卖"→⑩解析出 condition（条件）=GPS距离公司<500m，action（动作）=TTS播报"别忘了外卖"→⑬生成规则DSL（Domain Specific Language，领域专用语言，写规则的格式）→⑮校验+下发→㉔本地监听GPS信号→满足→播报
④拒识成功
（Rejection）		写入㉑Context事件日志（event log，记录发生了什么事）车外噪音、无意义语音不是对车说话，过滤掉（filter out，筛掉不要的）高速上风噪触发了ASR，转出"嗖嗖嗖"→⑨判断不是有效指令→拒识→记日志→不做任何事

注意：以上四路分流只覆盖"用户开口说话"的情况。系统预设条件任务（§1b的第①类）不走这四路——它们是提前配好的，不需要用户说话就一直在运行。

二·b、两种信号消费模式：周期性（Periodic）vs 触发式（On-demand）6.24新增

6.24丁彬帮我理清了：触发器和AIBOX（AI计算盒子，车上跑VLM模型的硬件）之间的信号交互只有两种模式，必须提前约定清楚每个事件走哪种。

丁彬6.24原话："你那边给我注册的信号就两种，一种是周期的我就直接往上报。另外一种是你要触发一下调用我一次我返回一个结果。这些咱们都提前约定好就行了，分好类就行了。比如打电话就是周期类的，看后面充电桩就属于你要来调用我的。"
模式含义（白话）触发器怎么做AIBOX怎么做典型case和v4原有架构的关系
①周期性
（Periodic）
"AlwaysOn已经在看了，我只要读信号库就行"		 AlwaysOn（常驻观测，一直在后台跑的基础视觉任务）已经在持续输出这个信号了。触发器不需要调VLM，只需要轮询（poll）④信号库看有没有变化 15-3扫描⑫云端信号库 → 15-10穿越检测 → 15-11组装event → push给Advisor AIBOX什么都不用额外做——AlwaysOn本来就在跑，信号本来就在往④里写 副驾打电话 / 副驾放下电话 / 后排有小孩 / 主驾打哈欠 / 乘客上下车 / 车内有宠物 v4的15-3扫描、15-10穿越检测覆盖了这种模式，不需要改
②触发式
（On-demand）
"AlwaysOn没在看这个东西，我得主动调一次VLM"		 AlwaysOn不覆盖这个观测目标（比如"后方有没有充电桩"不在AlwaysOn的观测范围内）。触发器需要先满足前置条件A（如P档）→ 主动发起一次VLM任务 → VLM看完后结果写回④ → 触发器再扫描条件B 24-3扫描到条件A满足 → 24-6发VLM运行指令（7字段）给㉕感知器 → 等VLM结果写回④ → 24-3重新扫描条件B AIBOX㉕感知器收到运行指令 → 调度VLM执行一次观测任务 → 结果写④ P档看后方充电桩 / 停车后看周围有没有停车位标识 / 特定条件下看车外路面积水 v4的24-6写的是"接收云端下发的VLM任务"，但6.24明确了端侧触发器自己也能主动发起VLM任务（不是只从云端被动接收）

充电桩case——触发式模式的完整walkthrough

场景："P档时看到后面有充电桩→开充电口盖"（系统预设条件任务，配置平台配好的）

步骤1 产品在配置平台⑬配好规则：{condition_A: "gear=P", condition_B: "rear_charger=detected", action: "open_charge_port"} → ⑮校验入库 → L14下发㉔

步骤2 用户开车到充电站附近，挂P档

步骤3 ㉔端侧扫描④信号库 → gear_current="P" → 条件A满足！

步骤4 ㉔查规则：还有条件B"后方有没有充电桩"→ 这个信号AlwaysOn没在看（AlwaysOn看的是舱内+前方，不看后方）→ 触发式模式启动：㉔发VLM运行指令给㉕感知器 → "用后摄看一下后方有没有充电桩"

步骤5 ㉕收到指令 → 调AIBOX的VLM → VLM看后摄画面 → 输出"后方5米处有充电桩" → 结果写回④ {key:"rear_charger", value:"detected", source:"VLM"}

步骤6 ㉔重新扫描④ → gear="P"✓ + rear_charger="detected"✓ → 两个条件都满足！

三、信号库④（Signal Library，信号仓库）：6字段逐个展开 + 上下游

车上有各种来源的信息（车门开没开、车速多少、温度、摄像头看到啥），全汇总到一个地方叫信号库④（Signal Library，信号仓库）。它存的是快照（snapshot，某一时刻所有状态的完整记录）——就像给车的所有状态拍一张照片，每500ms（毫秒，千分之一秒）更新一次。

6类信号源（signal source，产生信号的来源）往④里写数据：
① CDC车身总线（Controller Domain Controller，车身域控制器，管所有硬件传感器）：档位/车速/温度/安全带/车门/车窗/雨刷/灯光/电量...
② VLM视觉模型输出（Vision Language Model，端侧视觉语言模型）：如"副驾在低头看手机"、"后排座椅上有蓝色毛毯"
③ App埋点（app event tracking，记录用户在车机App里点了什么）：用户切歌/调音量/打开导航...
④ 电子围栏（geofence，虚拟地理边界）：进出某区域（如进入公司500米范围）
⑤ 音频（audio signal）：环境音量/切歌事件/语音唤醒...
⑥ 时间（timer signal）：定时器到点/日期变化/节日...
④的数据有三条出路：
① → ⑫云端信号库（Cloud Signal Library，云端信号仓库）：全集同步（full sync，所有数据复制一份）到云端做镜像（mirror，完全一样的副本）
② → ㉑Context（上下文记事本，全局统一事实源）：按策略（policy，规则）上报关键状态，作为Advisor（顾问）/Director（决策者）的推理依据
③ → ㉔端侧触发器引擎（Edge Trigger Engine）：本地读/订阅（subscribe，注册监听某个信号的变化），做端侧判定（evaluate，判断条件是否满足）

四、event（事件，刚刚发生的变化）vs condition（条件/状态，当前持续存在的情况）：6.22 最核心的设计决策

9 原子逐个展开

24-1 规则接收（Rule Reception，从云端收规则）

【是什么】把云端⑮打包好的规则接进车端。一条规则包含：scene_id（场景编号）、condition（什么条件）、action（干什么）、priority（优先级P0-P4）、valid_phases（生效阶段）、expire_at（过期时间）。

【三种下发方式】全量下发（新车激活）/ 增量下发（日常改一条规则）/ 灰度下发（先发10%车试3天）。

【支持回滚（rollback，撤回）】发现有问题→发revoke指令→㉔删除该规则。

运营在火山后台配了"充电桩靠近自动提醒"：{scene_id: "SCN-GEO-012", condition: {signal: "charger_distance", op: "<", value: "50m"}, action: {type: "tts", content: "附近有充电桩，要去充电吗？"}}→ ⑮校验→灰度下发10%车→3天无问题→全量下发→㉔收到存本地
L14从⑮下发的规则包。包含 scene_id/conditions[]/actions[]/priority/valid_phases/expire_at/gray_ratio + 6.24新增 signal_mode（信号消费模式）："periodic"周期性 或 "on_demand"触发式——告诉㉔这条规则要不要主动调VLM
规则存入本地持久化存储（24-2），供后续扫描匹配使用
24-2 本地持久化（Local Persistence，断电不丢的存储）

【是什么】把规则存到车机硬盘，重启不丢。用轻量级KV存储（如LevelDB/SQLite）。重启后从本地恢复+向云端全量同步校验。冲突时以云端为准。

Day1 22:00设了3条规则→存SQLite。23:00熄火→内存清空但SQLite不受影响。Day2 07:30上电→从SQLite恢复3条→同步⑮校验→如果用户昨晚在App删了一条→云端少一条→以云端为准删掉本地。
24-3 扫描+简单匹配（Scan & Simple Match，周期性检查条件是否满足）6.24扩展

【是什么】每500ms读一次④信号库快照，逐条和本地规则做匹配。

【6.24细化——两种匹配结果的不同后续动作】：

  • 规则的signal_mode=periodic（周期性）：条件匹配成功 → 直接走24-5防抖冷却 → 24-9执行。不需要调VLM。
  • 规则的signal_mode=on_demand（触发式）：前置条件A匹配成功 → 24-6发VLM任务 → 等VLM结果写回④ → 下一个扫描周期再匹配条件B。
周期性case：规则"belt_fr=unbuckled AND speed>10 → TTS提醒"，signal_mode=periodic。AlwaysOn已经在输出belt_fr和speed。㉔每500ms扫④：belt_fr="unbuckled"✓, speed=45>10✓ → 全匹配 → 走24-5防抖 → 24-9执行TTS。

触发式case：规则"gear=P AND rear_charger=detected → 开充电口盖"，signal_mode=on_demand, trigger_condition=gear=P。㉔扫④：gear="P"✓ → 触发条件满足 → 发VLM任务给㉕"看后方有没有充电桩"→ 等结果。下一个周期扫④：rear_charger="detected"✓ → 全匹配 → 走后续流程。
④信号库快照 + 24-2存储的规则列表
周期性规则匹配成功 → 24-5防抖冷却。触发式规则前置条件满足 → 24-6发VLM任务
24-4 本地定时器（Local Timer，车上的闹钟）

【是什么】断网也能倒计时的闹钟。4种模式：倒计时（"10分钟后"）/ 定时（"下午3点"）/ 重复（"每天早8点"）/ 日期（"6月22号"）。

【关键设计】存的是目标时间点不是剩余时间——重启后用"目标-当前"重算。已过期→立即触发。

14:00设"30分钟后关空调"→ 存 {target_time: "14:30:00"}。14:15重启→读到target=14:30:00→还剩15分钟→继续。如果14:35才恢复→已过期→立即触发（晚了5分钟但总比不触发好）。
24-5 防抖冷却（Debounce + Cooldown，防止重复触发的两道保险）

防抖（debounce）= 信号在阈值附近跳来跳去时不反复触发，要求"连续N秒都满足"才算。
冷却（cooldown）= 触发后N秒内不再触发，防持续骚扰。

防抖case：temp=29.8→30.1（开始计时）→29.9（重置！抖动）→30.3（重新计时）→连续5秒>30→确认触发。没有防抖→空调一分钟开关十几次。

冷却case："安全带提醒"cooldown=120秒,daily_cap=5→14:30第1次→14:32第2次→...→14:40第5次→今天不再提醒。没有冷却→每半秒提醒一次。

注意（6.22结论）：阈值穿越检测（threshold crossing detection）上收到了云端⑮-10，端侧㉔只做防抖+冷却，不做穿越判定。

24-6 VLM调度+资源仲裁（VLM Scheduling + Resource Arbitration）6.24扩展

【是什么】把VLM观测任务翻译成㉕感知器能理解的7字段运行指令发给㉕。同时管摄像头资源冲突。

【6.24明确的两种触发来源】：

  • 来源A——云端下发的动态视觉任务：用户说"帮我看毯子"→ ⑰动态Advisor → ⑮编译7字段 → L14下发㉔ → 24-6转发㉕。（这是v4原有的链路）
  • 来源B——端侧自主发起的触发式VLM任务：㉔扫描到前置条件满足（如P档）→ 24-6按规则里预定义的VLM任务模板生成指令 → 发给㉕。（6.24新增链路——丁彬确认"你要触发一下调用我一次"）

15-10 阈值穿越·变化检测（Threshold Crossing Detection）

【是什么】持续监听⑫信号变化，检测"状态从A变到B"的那个瞬间。只认 last=A → this=B 且 A≠B 的穿越。
