Case 1：快到外婆家 → 提醒打电话

用户Query

"我快到外婆家的时候提醒我给她打个电话"

触发条件：靠近{POI}（外婆家，收藏地址） · 阈值 500m · one_shot

第一段：注册（用户说完这句话的10秒内）

1
①用户语音 ②ASR L1

用户在车里说了这句话。车机麦克风拾音，ASR（在端侧AIBox上跑的）把语音转成文字："我快到外婆家的时候提醒我给她打个电话"。

2
⑨拒识仲裁 L2端→云

文字上传云端，进拒识仲裁。仲裁模型一看——有"的时候"这个标志词，不是简单车控，也不是闲聊，判成"②条件话"。走L4进条件链路。

3
⑩意图解析 L4 需2·意图模型

条件提槽，拆出"条件"和"动作"。

意图模型把这句话拆成结构化的两部分：

  • 条件：靠近"外婆家"（从收藏地址里匹配，或者从当前导航目的地匹配）
  • 动作：提醒用户打电话给外婆

拆完之后变成DSL结构，准备往配置平台写。

找谁 意图模型的解析能力——"的时候""快到"能不能准确识别成条件触发？这是需2·意图模型要cover的。

4
⑬配置平台 L11 刘杨·任军

条件任务注册到火山后台。配置平台收到DSL，生成一条场景规则：

  • Condition：event_type eq "approach"（B方案下，等端上报的事件，不是直接配距离条件）
  • poi_match：地址 = 外婆家（收藏地址匹配）
  • Action：callback → 动态Advisor → "提醒打电话"
  • execute_policy：one_shot，单程触发1次
5
⑮触发器引擎 L13

云端触发器引擎收到这条规则。走内部流水线：

  • 15-1 注册校验：检查iCanDo——这个条件能做吗？靠近POI，能做（导航信号已支持）
  • 15-2 场景入库：这条规则存进场景库，带版本号和灰度标签
6
㉔端侧触发器 L14云→端 金龙

规则从云端下发到端侧。㉔端侧触发器收到：

  • 24-1 规则接收：接到⑮经15-12下发的规则包
  • 24-2 本地持久化：写进SQLite，掉电不丢——车熄了火重新启动，这个"快到外婆家提醒打电话"的任务还在

同时，端侧状态机模块知道了：我要盯"外婆家"这个POI的距离，阈值500m，死区480/520。

注册完成。到这一步，系统已经记住了"快到外婆家就提醒打电话"这件事。接下来就是等——等用户真的导航去外婆家、真的快到了。

7
①地图模块 ③端信号源

用户导航去外婆家，地图模块在车上跑着。地图模块每秒算一次沿路线剩余距离（注意是沿路线，不是直线距离），推给端侧采集层。比如现在还剩15000m。

上汽/高德 提供这个API。仁杰 对齐格式：要整数米、沿路线、~1Hz。

8
④端侧信号库 L8 汪帅

采集层把地图模块的数据标准化后写入端侧信号库。信号库里现在是这样的：

{distance: 15000, unit:"m", status:"导航中", gear:"D", address:"外婆家地址"}

distance每秒在变：15000 → 14800 → 14500 → ... 一路递减。

9
B方案核心 金龙/汪帅

端侧状态机模块在跑——这是B方案的心脏。

每秒钟，它读一次④信号库的distance，然后做这几件事：

  1. 前置检查：status是"导航中"吗？有目的地吗？→ 是的，继续
  2. 取状态卡片："外婆家"这个POI的状态现在是"远"（初始状态）
  3. 方向判定：distance比上一秒小了吗？→ 是的，在靠近
  4. 死区判定：distance ≤ 480m了吗？

前面一路都是：15000 → 14800 → ... → 600 → 520 → 501。到这里，480都没到，状态一直是"远"，什么都不触发。安静等着。

然后某一秒：distance = 479。≤ 480，进死区了！

  • 状态翻转：远 → 已接近
  • 产出事件：{event_type: "approach", poi_id: "外婆家", distance: 479, threshold: 500, trigger_type: "one_shot"}
  • 自锁：状态已经翻到"已接近"了，接下来478、477、476...再也不会触发第二次接近事件。这就是one_shot——门铃按一下响一次，不是警报器一直叫。
10
④信号库 ⑫云端信号库 L9端→云

事件写进端侧信号库，通过L9同步到云端。云端信号库⑫现在多了一条事件信号：nav_event: approach, poi: 外婆家。

11
⑮触发器引擎 L12

云端触发器引擎扫到了这个事件。走内部流水线：

  • 15-3 扫描：发现⑫里出现了新的approach事件
  • 15-13 事件组装（6.22新增）：把approach事件和Step 4注册的场景匹配上了——"外婆家" + "approach" = 命中！
  • 15-10 标准事件打包：把事件payload打包
  • 15-11 Callback路由：这条规则的Action说"callback → 动态Advisor"，走L18 push
12
⑰动态Advisor L18 push事件 施佳杰

动态Advisor（弋途）收到push的事件。它不是自己去轮询的——6.22铁律④：event → ⑮组装push给Advisor，"我不轮询你的Context，注册事件后等你push"。

Advisor收到事件后，产出一条建议advice："用户快到外婆家了，场景：提醒打电话。建议话术：提醒用户打电话给外婆。"

13
⑲Director汇入 ⑳Director推理 L20

建议进Director，大脑开始说人话。

  • ⑲汇入：Advisor的建议通过L20进来，同时L26注入㉑Context（当前车况、用户记忆、情景）
  • ⑳推理：SP~30K tokens的大模型推理，生成talk_content。风格=温柔高知女生。
  • 输出："快到外婆家啦～要不要先打个电话给外婆说一声呀？"
14
㉖TTS L29云→端

话术下发到车里，TTS播报。车里的喇叭响了："快到外婆家啦～要不要先打个电话给外婆说一声呀？"

用户听到了。任务完成。

整条链路走完。从用户说出"快到外婆家提醒打电话"到最终播报，经过了14个模块、8条连线、两段时间。核心：注册阶段靠意图解析+配置平台+规则下发，触发阶段靠端侧状态机算事件+云端匹配+Advisor+Director。

这个case为什么特殊

"路过"和"靠近"是两回事，别搞混了：

  • 靠近（approach） = 徐浦大桥是你的目的地，你一路开过去，距离越来越近 → 用导航剩余距离判，状态机走"远→已接近"
  • 路过（pass_by） = 徐浦大桥不是你的目的地，你的目的地是别的地方，只是路线会经过徐浦大桥 → 用的不是"距离≤阈值"，而是"路线与POI围栏相交且你还没到那个点"

这句话说的是"路过"——用户要去别的地方，只是路上会经过徐浦大桥。所以不能用Case 1/2的approach事件，得用新的poi.pass_by事件。

第一段：注册（一样的链路，但解析出来的条件不同）

①→②→⑨→⑩ 这一段和Case 1/2一样。区别在⑩意图解析的结果：

3'
⑩意图解析 需2·意图模型

"路过"这个词很关键。意图模型要能区分"快到"和"路过"——"快到"对应approach（靠近目的地），"路过"对应pass_by（经过某个点但那不是目的地）。

拆出来的结构：

  • 条件：poi.pass_by eq true，POI = "徐浦大桥"
  • 动作：提醒准备右转

后面④→⑬→⑮→㉔的注册链路一样，略过。

第二段：触发（这里逻辑完全不同）

9'
路过判定·新增能力 金龙·任军

"路过"怎么判？不是看距离递减，是看"路线经过了那个点"。

Case 1/2的approach用的是"距离≤阈值"——因为目的地就是那个POI，导航一直在算到那里的剩余距离。

但"路过"不一样：徐浦大桥不是目的地，导航不会算"到徐浦大桥还有多远"。所以要换一种方式：

判定方式怎么做谁来做
方式A
途经点距离		把徐浦大桥作为一个途经点加入导航路线，这样地图就会算到途经点的剩余距离，复用approach逻辑仁杰确认地图SDK是否支持动态插入途经点
方式B
路线与围栏相交		给徐浦大桥设一个电子围栏（圆形，半径~500m），判断当前导航路线是否会穿过这个围栏。如果会穿过，在穿过前~1km时触发pass_by事件金龙确认端侧能力 · 任军确认运算符
方式C
GPS围栏进入		不依赖导航路线，直接用GPS实时位置判断是否进入徐浦大桥的围栏范围。最简单但精度低、提前量不够端侧GPS + 围栏判定

B方案下推荐方式A（如果地图支持动态途经点）——这样可以完全复用Case 1/2的状态机逻辑，只是把"目的地"换成"途经点"。

如果地图不支持，退到方式B，需要新增围栏相交判定能力——这是§11 #15待确认的。

判定之后的链路（⑫→⑮→⑰→⑲→⑳→㉖）和Case 1一样：事件同步云端 → 引擎匹配 → push Advisor → Director说话 → TTS播报。

最终播报："马上要路过徐浦大桥了哦，记得准备右转～"

Case 3是3个case里唯一有硬卡点的：

  • poi.pass_by这个事件现有协议没有——§6.5编号208，待锁版
  • 路线与围栏相交判定——端侧能不能做？谁来做？→ §11 #15 找金龙·任军
  • 徐浦大桥的POI坐标和围栏半径——从哪来？高德POI库还是预设？→ §11 #5 找金龙

Case 1/2可以直接做（信号和状态机都ready），Case 3要等卡点落地才能做。
