新方案探索 — 超越A/B/C的思路

重新审视系统架构后发现的新可能性

本文档是对前版三方案（A/B/C）的补充和升级。重新分析SP后发现了被忽略的系统设计和新的操作杠杆。

零、重新审视后的关键发现

被忽略的关键事实：SP中已经有一个 vehicle_manual_qa 工具

SP的第7个工具是 vehicle_manual_qa —— 一个专门用于回答车辆手册问题的独立工具。
这意味着系统设计中，车书知识有两条路径到达Director：预检索注入（被动）和工具调用（主动）。
当前问题的本质是：被动注入的知识"抢跑"了主动工具调用。

重新理解问题：不只是"知识误导"，还有"路径抢占"

之前的理解（不完整）

Director看到了QA知识中的手动操作步骤 → 直接用它回答 → 没调工具。

补充的理解：

{{car_function_simple_qa}} 的预注入不仅"提供了一个现成答案"，还抑制了Director对 vehicle_manual_qa 工具的调用动机。如果context中没有车书知识，Director在面对咨询型问题时本来会调用 vehicle_manual_qa 工具获取信息。但现在context已经有了，Director认为"信息已充足，不需要再调工具"。

这意味着 {{car_function_simple_qa}} 的全量注入实际上产生了两个副作用：

  1. 执行型query：用手动操作步骤替代了 vehicle_basic_control 工具调用
  2. 咨询型query：用预注入知识替代了 vehicle_manual_qa 工具调用（这个副作用可能是设计意图——预注入是为了减少工具调用延迟）

一、新方案D — 竞争性QA检索对（从检索机制层面解决）

核心思路：不改现有QA，新增QA对"抢占"检索top位

利用向量检索的相似度排序机制，让执行型query检索到"引导工具调用"的答案，而非"手动操作步骤"

原理说明

当前向量检索是 top-3。当用户说"打开定速巡航"时，检索匹配到的是：

当前检索结果（问题根源） Q: "定速巡航怎么开" ← 匹配关键词"定速巡航"，相似度中等 A: "进入【车辆设置】-【辅助驾驶】，打开..." ← Director拿这个做了回答

如果我们新增一条QA对：

新增的竞争性QA对 Q: "打开定速巡航" ← 与用户query"打开定速巡航"完全匹配，相似度极高 A: "该操作由系统直接执行，无需手动操作。"

检索结果变成：

修改后的检索结果 Top-1（新增）: Q:"打开定速巡航" → A:"该操作由系统直接执行，无需手动操作。" Top-2（原有）: Q:"定速巡航怎么开" → A:"进入【车辆设置】-【辅助驾驶】..." Top-3（原有）: Q:"巡航功能介绍" → A:"自适应巡航辅助是..."

Director看到Top-1说"由系统直接执行"，自然倾向调用工具。

关键机制：向量检索的相似度排序意味着"打开定速巡航"与"打开定速巡航"的匹配度远高于与"定速巡航怎么开"的匹配度。新QA对必然排在top-1。

与方案A的本质区别

维度方案A：改现有QA answer方案D：新增竞争性QA对
操作对象 修改现有80-150条QA的answer文本 新增80-150条全新QA对，现有数据完全不动
作用机制 改变answer内容，但检索排序不变（手动操作步骤仍是top-1） 改变检索排序本身——"执行引导"answer跃升为top-1，手动操作步骤被挤到top-2/3
Director看到的信号强度 answer开头有前缀但后面仍跟着详细操作步骤 top-1结果本身就是"系统直接执行"，操作步骤被推到次要位置
对现有数据的风险 修改了原始answer，有改错的风险 零风险——原始QA完全不动，新增QA删掉就回滚
对咨询型query的影响 咨询型query（"怎么用"）检索到的answer也被改了前缀，多了一句赘言 不受影响——"定速巡航怎么用"与"打开定速巡航"向量距离远，新QA不会被咨询型query检索到

优势（5条）

  • 从检索机制层面解决问题：不是在Director端"教它怎么用知识"，而是直接控制它"看到什么知识"。这是上游治理，比下游修补更彻底。
  • 完全不修改现有数据：原有500-2000条QA一字不动。新增的QA对删掉就完全回滚，零残留风险。
  • 对咨询型query天然免疫：用户说"定速巡航怎么用"，向量检索会匹配到原有的"定速巡航怎么开"而非新增的"打开定速巡航"。两者的向量距离足够大，不会互相干扰。这是方案A做不到的——方案A改了answer后，咨询型query检索到的也是带前缀的answer。
  • 可批量脚本化生成：模板固定（"XX可通过语音直接控制，由系统执行"），只需要一张"功能→说法变体"的映射表，Python脚本10分钟生成。
  • 效果确定性最高：Director看到的top-1就是"由系统执行"，不需要它去"理解规则"或"解读前缀"。这是最直接的信号。

劣势（4条）

  • 新增QA数量较多（70-100条）：虽然可以脚本生成，但需要逐条审核"说法变体"是否合理。如果用户的真实说法没被覆盖（如方言、口语化表达"把那个巡航给我打开"），新QA对不会被检索到，回退到原有问题。
  • 向量数据库的管理成本：新增100条QA对意味着向量库多了100个embedding。需要确认向量库是否有容量或质量问题（如新QA对是否会影响其他query的检索结果）。
  • 说法覆盖的长尾问题：用户说"帮我定速巡航一下""把巡航开着""那个定速的给我开了" 等非标准表达，可能不会匹配到新QA对。这个问题可以通过每个功能增加3-5种说法变体来缓解，但无法100%覆盖。
  • 需要产品/数据侧配合：新增QA对需要入库到向量数据库。泽民不一定有直接操作权限。

与方案A/B的关系

方案D可以替代方案A：两者都是数据层操作，但D不改现有数据、对咨询型query无干扰、信号更直接。
方案D可以与方案B叠加：D（检索层）+ B（推理层）= 更强的双保险。
推荐组合：D + B 替代原来的 A + B（方案C）。

二、新方案E — Knowledge注入few-shot示例（方案B的增强版）

核心思路：在 {{knowledge}} 中注入的不是规则，而是具体的输入输出示例

LLM对few-shot示例的遵循度远高于抽象规则。用示例替代规则，效果更强。

为什么示例比规则更有效？

方案B注入的RULE-01是抽象规则：

// 方案B的规则格式 "当用户说出带动作词的指令 → 必须调用 vehicle_basic_control" "当用户询问实时数据 → 必须调用 search_vehicle_status_info"

Director需要先理解规则，再判断当前case属于哪类，再执行对应动作。三步推理，每步都可能出错。

方案E注入的是具体示例：

// 方案E的示例格式 "用户说'打开空调'（context中有'空调可通过中控面板开启'的知识） → 正确：调用vehicle_basic_control ← Director直接模式匹配"

Director只需要做模式匹配：当前case和哪个示例最像？找到了就直接跟着示例做。一步完成，出错概率大幅降低。

学术支撑：大量Prompt Engineering研究表明，few-shot示例对LLM行为的引导效果比instruction-only（纯规则）高出20-40%。这在GPT-4、Claude等模型上都被验证过。

知识条目完整内容（替代RULE-01）

知识ID: RULE-01-V2 知识名称: 车辆功能知识使用指南（示例版） 【车辆功能问答知识正确使用示例】 你的context中可能包含「基础车辆功能问答」中检索到的车辆手册参考知识。以下示例展示了正确的处理方式： --- 示例1：执行指令 → 调工具（不用知识回答）--- 用户说："打开空调" context中有知识："空调可以通过中控屏底部的空调按钮开启，也可以通过..." 正确做法：调用 vehicle_basic_control("打开空调")，说"好的，正在为你打开空调" 错误做法：回答"空调可以通过中控屏底部的空调按钮开启..." 判断依据："打开"是执行动作词 → 用户要的是执行，不是操作说明 --- 示例2：实时查询 → 调工具（不用知识回答）--- 用户说："现在时速多少" context中有知识："当前车速是仪表屏常显信息，非P挡时可直接在仪表屏查看" 正确做法：调用 search_vehicle_status_info("查询当前车速") 错误做法：回答"车速信息在仪表屏上可以查看" 判断依据："现在XX多少"是查实时数据 → 用户要的是数值，不是在哪里看 --- 示例3：操作咨询 → 用知识回答（这时才该用）--- 用户说："定速巡航怎么用" context中有知识："定速巡航开启方法：进入【车辆设置】-【辅助驾驶】..." 正确做法：参考知识回答"定速巡航可以这样开：进入车辆设置..." 判断依据："怎么用"是咨询操作方法 → 知识中的操作步骤正是用户想要的 --- 示例4：工具失败 → 用知识兜底 --- 用户说："打开定速巡航" → 调用vehicle_basic_control → 工具返回失败 正确做法：告诉用户"暂时无法语音开启，你可以手动操作：进入车辆设置..." 判断依据：工具失败时，知识中的手动操作方法就是最好的兜底方案

方案B（规则版）vs 方案E（示例版）效果对比

方案B（RULE-01规则版）预估修复率：60-80%
规则需要Director做三步推理（理解规则→分类意图→执行动作），每步都有误差。

方案E（RULE-01-V2示例版）预估修复率：75-90%
示例只需Director做一步模式匹配（当前case像哪个示例→跟着做），效果显著提升。

增量收益：~10-15个百分点的修复率提升，但token开销从约250 tokens增加到约500 tokens。

额外收益：示例中的"错误做法"是显式的负面示例（"不要这样做"），这种正反对比对LLM的约束效果非常强。方案B的纯规则没有负面示例。

相比方案B的额外优势

  • 模式匹配 > 规则推理：LLM的in-context learning本质就是模式匹配。示例直接提供了pattern，模型只需copy，不需要think。
  • 包含负面示例：每个示例都有"错误做法"，这是一种"对比学习"信号。Director不仅知道该怎么做，还知道不该怎么做。
  • 四种场景全覆盖：执行/查询/咨询/兜底四个示例，覆盖了所有可能的决策场景。Director面对任何case都能找到最相似的示例参照。
  • 与SP中已有示例格式一致：SP已有21个示例，Director对"示例→模仿"的模式已经建立了强关联。在 {{knowledge}} 中也给示例，Director会自然地将其视为"额外的行为参考"。

劣势

  • token开销翻倍：从方案B的~250 tokens增加到~500 tokens。始终加载的话，月度成本也翻倍。
  • 知识条目较长：500 tokens的"知识"在 {{knowledge}} 中算长了。如果同时有其他场景知识需要注入，总token可能较高。
  • 权重仍是"参考"级：虽然示例比规则更强，但 {{knowledge}} 本身的"参考"定位没变。Director仍有一定概率不完全遵循。

方案E判定

方案E应该替代方案B，而非与B共存。它们占用同一个位置（{{knowledge}} 中的知识条目），但E的效果显著优于B。
建议：直接用RULE-01-V2（示例版）替换RULE-01（规则版）。如果token预算紧张，可以精简示例到3个（去掉兜底示例）。

三、新方案F — QA答案结构化重塑（"双通道"格式）

核心思路：将QA answer改为"语音控制通道 + 手动操作通道"双格式

不是简单加前缀，而是重新组织answer的信息结构，让"语音控制"成为第一推荐路径

与方案A的区别

方案A：在answer开头加一句话（"该功能支持语音控制"），但answer的主体结构不变，手动操作步骤仍然是主要内容。

方案F：彻底重组answer结构，把answer变成两个并列的"通道"，语音通道放在第一位。

answer格式示例

方案A的做法（加前缀）：

// 方案A：一句前缀 + 原始answer A: "该功能支持语音直接控制，用户说'打开定速巡航'即可由系统执行。若需手动操作：进入【车辆设置】-【辅助驾驶】，打开【自适应巡航辅助】，然后按方向盘PILOT按键激活。" // 问题：手动操作步骤仍然是answer的主体内容（占80%篇幅），Director可能"跳过前缀"直接采用步骤

方案F的做法（双通道结构化）：

// 方案F：结构化双通道格式 A: " [语音控制] 对车机说"打开定速巡航"即可自动执行，推荐使用。 [手动操作] 进入【车辆设置】→【辅助驾驶】→ 打开【自适应巡航辅助】→ 按方向盘PILOT键激活。 " // 效果： // 1. [语音控制]标签在前，且标注"推荐使用"，Director天然优先采信第一通道 // 2. [手动操作]标签明确告诉Director这是"备选方案" // 3. 结构化标签让Director的决策更"有据可依"，而非需要从连续文本中提取信号

不支持语音控制的功能（不改结构）：

// 雨刮器等不支持语音控制的功能，只保留手动操作通道 A: " [手动操作] 通过方向盘右侧拨杆调节雨刮器速度。上拨增速，下拨减速。 " // 没有[语音控制]通道 → Director看到只有手动操作方案 → 自然用知识回答

相比方案A的额外优势

  • 结构化标签比自然语言前缀更强：[语音控制] 和 [手动操作] 是显式的分类标签，LLM对标签化信息的解析精度远高于从段落中提取信息。
  • "推荐使用"标注的引导效果：明确写了"推荐使用"，Director在决策时会倾向于被推荐的路径。
  • 有无[语音控制]通道本身就是信号：支持语音控制的功能有两个通道，不支持的只有一个。Director看到只有[手动操作]通道时，自然知道"没法调工具，用知识回答"。
  • 手动操作步骤被降级为次要位置：方案A中步骤仍是主体内容；方案F中步骤变成了[手动操作]通道下的子内容，视觉和语义权重都降低了。

劣势

  • 改造工作量比方案A大：不是简单加前缀，需要重组每条answer的结构。即使有脚本辅助，也需要更多人工审核。
  • 仍需改现有数据：和方案A一样需要修改原始QA对的answer字段。
  • 格式可能影响咨询型回答质量：用户问"定速巡航怎么用"时，Director看到结构化格式后的回答可能也带上了[语音控制]/[手动操作]标签，不如自然语言回答友好。

方案F判定

方案F是方案A的"升级版"。如果要走"改QA数据"这条路，F比A更有效。
但方案D（竞争性QA）比F更优：D不改现有数据、对咨询型免疫、回滚更安全。
建议：如果团队决定改QA数据，优先考虑F而非A。但如果可以新增QA对，首选D。

四、新方案G — 利用 vehicle_manual_qa 工具重定向

核心思路：在Knowledge规则中引导Director对咨询型问题调用vehicle_manual_qa工具

减少Director对 {{car_function_simple_qa}} 预注入知识的依赖，让咨询型问题也走工具路径

思路说明

当前SP中 vehicle_manual_qa 工具的描述是：

"当用户询问如车辆功能（如OTA更新说明、故障排查指导、保养维护建议...）等时，可使用此工具帮助用户充分了解和使用此车。" 参数示例："如何进行OTA更新""空调怎么设置最省油""座椅记忆功能怎么设置""雨刮器快慢咋调"

注意看参数示例——"座椅记忆功能怎么设置""雨刮器快慢咋调"这些本来就应该走vehicle_manual_qa工具，但因为 {{car_function_simple_qa}} 预注入了相关知识，Director觉得"不需要再调工具了"。

方案G的逻辑：在 {{knowledge}} 注入的规则/示例中，不仅处理"执行型→调vehicle_basic_control"，还处理"咨询型→引导调vehicle_manual_qa获取更详细信息"。这样做的效果：

  • 执行型query → vehicle_basic_control（不用QA知识）
  • 查询型query → search_vehicle_status_info（不用QA知识）
  • 咨询型query → 可以参考QA知识，也可以调vehicle_manual_qa获取更完整信息

这个方案的巧妙之处：它把 {{car_function_simple_qa}} 的定位从"可以直接用的答案"变成了"初筛参考"。Director看到QA知识后会想："这只是初步信息，我可以调vehicle_manual_qa获取更详细的"——这个思维模式也会让它对执行型query更倾向于调工具。

优势

  • 一石二鸟：不仅解决了执行型query的问题，还提升了咨询型query的回答质量（vehicle_manual_qa工具返回的信息比预检索top3更完整）
  • 降低了 {{car_function_simple_qa}} 的整体"权威性"：当Director形成了"这只是初筛"的理解后，所有类型的query都不会过度依赖预注入知识
  • 不改任何数据，通过 {{knowledge}} 注入即可

劣势

  • 增加工具调用次数：如果咨询型query也走vehicle_manual_qa工具，每个咨询问题多一次工具调用。增加延迟和成本。
  • 与预检索的设计初衷冲突：{{car_function_simple_qa}} 预检索的目的可能就是为了减少vehicle_manual_qa的调用延迟。如果又引导Director调工具，相当于绕开了优化。
  • 效果不确定：Director可能不会仅因为规则说"可以调vehicle_manual_qa"就去调——当context中已经有信息时，模型倾向不做额外工具调用。

方案G判定

有价值但不建议单独使用。可以作为方案E（示例注入）的一个附加策略——在示例的"咨询型"部分提到："也可以调用vehicle_manual_qa获取更完整信息"。
不建议把所有咨询型query都引导去调工具，因为这违背了预检索的性能优化初衷。
