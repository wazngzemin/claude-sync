§1 背景 · 痛点 · 决策记录

1.1 现状与痛点

豆包上车系统里会"动脑子"的东西（Director / Advisor / VLM）都是模型，模型有个物理限制——只在你叫它的那一秒活着。你不问 Director 不想，不触发 Advisor 不出主意，不给画面 VLM 不看。可现实世界一直在变：电量在掉、车在接近目的地、宝宝随时会醒、用户 10 分钟前说的事到点了要办。

§3 技术方案：云端触发器 + 端侧触发器

3.1 核心原理

触发器 = 门口的保安 + 床头的闹钟。便宜的、不用模型的判断（等于/大于/变更为）由它 7×24 做；贵的、要模型的语义推理只在条件命中那一秒被叫醒。用"运算符"挡住 99% 的噪音，让模型只处理 1% 值得处理的时刻。

§4 核心难点：为什么不能"信号一变就触发"

4.1 数字走查：低电量场景的真实数据

假设直接配"battery_level ≤ 20%"就触发，看会怎样：

时刻电量如果配"≤20%就触发"正确行为
8:0025%不触发 ✅不触发 ✅
8:1521%不触发 ✅不触发 ✅
8:2020%（首次跨阈）触发 ✅触发 ✅
8:2120%（持续）又触发 ❌ 重复弹窗不触发（穿越自锁）
8:2519%又触发 ❌不触发
8:3022%（充了点）不触发不触发
8:3520%（又降回来）触发 ❌ 二次骚扰不触发（冷却：本次上电已推过）
所以不能只判"≤20%"，要有完整闸门链：① 条件判定（≤20%）② 穿越（只认 F→T 边沿，持续为真不重触）③ 冷却/频控（本次上电推过一次就不再推）。三道闸门缺一不可。

4.2 闸门链状态机（云端运行，每次信号变更执行）

当前状态转换条件目标状态产出云端动作
未命中（初始）条件求值=True 且 last_result=False候选—进穿越判定
候选last=F 且 this=T（首次 F→T）已穿越—进防抖判定
候选last=T 且 this=T（持续为真）未命中—不触发（穿越自锁）
已穿越连续 K 帧稳定 True防抖通过—进冷却判定
已穿越第 n<K 帧变 False未命中—计数清零（排除误识别）
防抖通过频控未触顶（如 per_power_cycle 0/1）触发事件出口路由 → push 下游
防抖通过频控已触顶未命中—本周期忽略，下次上电重置
触发下游已消费未命中（last=T）—等下次 F→T

4.3 三个必须处理的坑

坑具体场景处理方案
坑1 · 持续为真反复触发电量持续 1 小时低于 20%穿越闸门：只认 F→T 边沿，T→T 不触发（见上表"候选→未命中"）
坑2 · 单帧误识别VLM 某一帧误报"宝宝哭了"，下一帧又"安静"防抖闸门：连续 K=3 帧 True 才算，单帧误报被清零
坑3 · 频控维度不一"本次上电1次"vs"本次行程1次"vs"每日1次"冷却四维：per_power_cycle / per_day / per_lifecycle / min_interval，配置表里每个场景标清（找益红定口径）

§6 完整链路：四阶段闭环

阶段什么时候干什么在哪里
Phase A · 注册+配置运营配规则 / 用户说话那一刻（几秒～十几秒）配置平台配场景 / 动态 Advisor 拆解需求 → 云端注册校验 → 入库云端（动态含端侧 ASR）
Phase B · 信号监听+判定持续运行（可能很久）信号变更驱动扫描 → 闸门链判定（穿越/防抖/冷却/仲裁）云端（视觉信号来自端侧）

6.5 回传通道（上行）定版：混合方案

★ 定版 = 方案 B（原路返回）为主 + 异步落 A 档（Context 审计）
方案 A（6.30）：端侧结果→信号库→Context→云端轮询监听 → 被仁杰/超然/庆磊三人推翻（"绕"、两小时无效计算）
方案 B（仁杰7.4）：谁下发 SP 谁原路返回 → 实时低延迟，但纯 B 不落 Context 丢审计
混合：实时触发走原路返回（回应"绕"），同时异步落 Context（守铁律②审计+多消费方）。关键话术：原路返回的终点仍是云端触发器，判断权没变，变的只是不再轮询 Context。要找仁杰+意图 SDK 侧定"谁调返回给谁"的改造。

§7 四个 Case 完整走查（核心 · 参照 POI-PRD v4 §7 颗粒度）

Case 1：低电量推荐-已发起导航（静态 Advisor · 出行）

静态 Advisor · 出行服务 · P0
"电量低于 20% 了，车主动推荐沿途充电站作途经点"
分类理由：条件=车况(battery)+导航(nav.status)双信号，已在配置平台预配 → 静态类
触发条件：battery ≤ X% AND nav=导航中 · L1 枚举零成本
触发后动作：push 出行 Advisor → 查沿途站 → Director 弹窗+TTS → 用户确认 → 加途经点
频控：一次上电周期仅一次 · 3min 失效

第一段：配置 + 注册（运营配场景的那一刻，一次性）

Step模块/谁做什么 + 输出
1⑬ 配置平台
益红		运营在火山后台配场景"低电量推荐-已发起导航"：功能1-条件=`battery≤X% AND nav=导航中` / 功能1-执行=`TTS+弹窗` / 功能2-条件=`用户确认` / 功能2-执行=`加途经点` / 推荐间隔=`一次上电1次` / 失效时长=`3min`
2⑮-1 注册校验
云端触发器		校验：battery_level 在 CAN 表 ✅ / nav.status 在端状态 ✅ / 加途经点动作在能力清单 ✅ → 放行
3⑮-2 入库
云端触发器		持久化场景规则，初始 status=1 / last_result=false / per_power_cycle=0/1。反向索引建立：key=battery_level → 本场景；key=nav.status → 本场景

配置完成，触发器持有一条生效规则，等信号变化。

第二段：判定 + 推理 + 执行（用户开车导航中，电量持续下降，半小时后）

Step模块/谁做什么 + 输出
4端侧 CAN
CDC		用户开车导航中，电量从 25% 持续下降。CDC 每秒上报 battery_level 到云端端状态：
25% → 24% → ... → 21% → 20% @t=0
5⑮-3 信号扫描
云端触发器		检测到 battery_level 变更（变更驱动，非轮询）→ 反向索引命中本场景 → 取当前所有信号快照
6⑮-4 条件判定
云端触发器		取快照做 AND：① battery_level=20 ≤ X(20) ✅ ② nav.status="导航中" ✅ → raw_result=True。L1 枚举直接比较，零模型成本
7 ⭐⑮-5 穿越
云端触发器		查 last_result：25%时=False，20%时=True → F→T 首次穿越！状态翻转 → 进防抖。
（若 21%→20%→21%→20% 抖动，防抖 K=1 帧即过；车况信号不做 K=3 防抖，因为 CAN 信号本身稳定）
8⑮-7 冷却
云端触发器		查 per_power_cycle 计数：0/1 → 未触顶 → 通过，计数变 1/1。本次上电即使电量再降也不再触发
9⑮-11 出口路由
云端触发器		按 source=static_advisor → push 出行 Advisor，同时启动 3min 计时器（到期发 event_expired）
事件 payload：
{event_type:"low_battery_navigating", payload:{battery:20, dest:"浦东机场", remain_km:68}, scene_id:"..."}
10⑯ 出行 Advisor
云端推理		Advisor 收到事件 → 查 Context（导航路线上 5km 内的充电站 POI + 用户偏好 + 历史充电记录）→ 选评分 4.8 + 顺路 +3min 的站 → 生成建议：
