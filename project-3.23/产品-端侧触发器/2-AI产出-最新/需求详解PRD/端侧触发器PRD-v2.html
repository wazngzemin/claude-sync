一、需求目标

1.1 建设目标

端侧需要同时建设两个基础能力：

模块定义
端侧信号库统一存储端状态、OEM/用户信号、导航/场景信号、VLM 结果和任务状态。是触发器的前置依赖——信号库不成立，触发器规则就无法稳定运行。所有信号必须经过信号库，触发器不直接监听原始信号源。
端侧触发器基于规则和信号判断任务是否触发，并负责任务排队、优先级、互斥组、冷却、下发指令、接收终态、结果透传上报。7.6 新增：管理 AlwaysOn 三任务的开关 + VLM 结果整包透传。

1.2 产品定位

端侧触发器 = AI Box 上的手脚 + 眼睛 + 闹钟

跑在车机（AI Box）上，做四件事：

  1. 开关 VLM：管 AlwaysOn 三任务的启停，按条件点火/熄火
  2. 掐表：本地定时器，断网不停，到点叫醒
  3. 硬条件判定：挡位变P / 温度超30度 / 天气包含雾——认死规矩，不判语义
  4. 透传：VLM 结果整包透传给 AI Service（不解析），"谁调返给谁"

不是什么：6.22 西钺警告"不要开发两个触发器"——逻辑上它和云端触发器是一个触发器的两半，一个条件只判一次。端侧做的事 = 物理上只能在车端做的事。

二、整体架构

2.1 三模块组成

端侧由三个模块组成：信号库、触发器、VLM 模型。三者的关系如下：

关系链说明
信号库 → 触发器触发器读取/订阅信号库中的信号快照，用于规则条件匹配。触发器不直接监听原始信号源，所有信号必须经过信号库。信号库是唯一事实源。
触发器 → VLM条件命中后，触发器向 VLM 下发运行指令。运行指令包含四个核心信息：任务标识、识别目标描述、优先级、超时时间。触发器不关心 VLM 内部如何推理，只管下发和接收结果。
VLM → 触发器
7.6 新增		VLM 执行完成后将原始输出返回给触发器（"谁调返给谁"）。触发器不解析内容，整包透传给 AI Service 数据中心。数据中心自行解析分发。
VLM → 信号库VLM 执行结果由 VLM 业务写回信号库（任务状态类信号），供触发器做续接调度。注意：任务状态给触发器做调度（实时性要求高）；识别结果给整个系统消费（需持久存储 + 标准化上报）。两条路分开，不混。
为什么任务状态和识别结果要分两条路？如果合成一条路，存储会拖慢调度，或者触发器不得不承担结果存储职责，导致模块职责不清。任务状态走信号库（快·调度用）；识别结果走透传上报（大·消费用）。

2.2 调用关系变更（7.6 定版）

核心变更：VLM 结果流向从 "VLM → AI Service 直接" 改为 "VLM → 端侧触发器 → AI Service（数据中心）"
原因：AI Box 架构统一原则——"谁调用我，结果返回给谁"（丁彬 7.6 ~10:21）
影响：端侧触发器从"只管开关"变成"开关 + 结果透传中继"

三、端侧信号库

3.1 设计原则

信号库不是"建一个缓存表"，而是把端侧所有会影响任务触发的状态先收口成一个统一事实源（Single Source of Truth）。后续触发器只认信号库，不再各自去车辆、APP、云端、VLM 结果里散读状态。

原则说明
唯一入口所有信号源（车辆底层、导航、天气、VLM、用户输入）统一写入信号库，触发器只从信号库读取
标准化字段每条信号统一为 key / value / source / timestamp / ttl 五字段，不管来源是什么，格式一致
实时性分级瞬态信号（挡位变化）默认 TTL 5s，需按 key 可配。持久信号（天气状况）TTL 可达 30min
只存最新同一 key 只存最新值——信号库不是时序数据库，不存历史。历史存储是 Context/云端的职责
空 ≠ 不存在信号值为空（如 VLM 看了但没看到目标）是有效数据，和"从未写入过"不同
被动写入信号库自己不主动采集信号——由各信号源主动 push 写入。信号库只负责存储、索引、TTL 过期
信号库是触发器的前置依赖——信号库不成立，触发器规则就无法稳定运行。先建信号库，再建触发器。

3.4 信号写入规则

规则说明
覆盖写入同一 key 新值覆盖旧值，只保留最新。timestamp 必须单调递增，旧时间戳的写入被拒绝
变更通知值发生变化时（旧值 ≠ 新值），信号库向所有订阅该 key 的触发器规则发送变更事件
空值写入允许写入空值（value: null）。空值是有效数据（"看了但没看到"），不等于删除信号
批量写入VLM 一次输出可能包含多个信号（位置+性别+年龄...），支持原子化批量写入，保证同一帧数据的时间戳一致
时序脏数据防护人换座位时：先写入原座位空数据（"无人"）→ 再写入新座位新数据。一空一实，保证数据干净。超然 7.4 提出的方案。

3.5 信号订阅与读取

方式说明使用场景
快照读取触发器周期性获取信号库中所有相关 key 的当前值快照，一次性读取周期性条件扫描（如 AlwaysOn 每 3s/5s 一轮）
事件订阅订阅特定 key 的变更事件，值变化时立即收到通知瞬态条件触发（如挡位 R→P 变更）
条件订阅订阅"满足特定条件时通知"——如 vehicle.gear_state 变更为 P充电桩 oneshot 触发

3.6 信号 TTL 与过期处理

信号类型建议 TTL过期后行为
瞬态信号（挡位、车速）5s信号作废→依赖该信号的规则暂停匹配（frozen），直到收到新值
中频信号（座位占座、FaceID）30s同上
低频信号（天气状况）30min同上
VLM 任务状态与任务周期一致过期后任务标记为 timeout
永久信号（展销模式开关）0（不过期）—
产品红线：过期信号不允许被用于条件判定。触发器匹配前必须检查 TTL，current_time - timestamp > ttl 的信号视为无效。

3.7 VLM 结果回写规则

VLM 执行完成后，结果分四类写回信号库：

结果类型写入内容是否有效处理规则
成功识别标签 / 自然语言描述✅写入信号库，供下游消费和透传上报
空结果"空"✅空 ≠ 没看。空是有效结论——对下游有意义（如下车安全空结果代表安全）
失败 / 超时失败或超时状态❌产品红线：不允许写伪结果。超时不能写"无障碍"，因为模型没看完，结果不可信
被打断不出现—由 VLM 业务内部管理，触发器只收最终结果
为什么"空结果"要单独区分：空结果和失败/超时有本质区别。空是"模型正常执行、仔细看了、确实没有目标"，可以作为下游决策依据（如告诉用户"四周没有障碍物"）。而失败/超时是"模型没正常完成"，不能基于此做任何决策。如果混为一谈，会导致下游误判。

增量更新运行中支持增量更新；更新后需即时生效云端触发器下发新规则/修改规则 → 端侧热加载 → 不中断正在运行的任务 → 新规则下一轮生效

异常回滚加载失败/触发率异常/VLM调用失败率超阈值时自动回滚回滚到上一版本，同时告警上报云端。阈值 [待研发定义]
接收动态注册接收云端触发器下发的 Advisor/Planner 动态任务动态任务=有生命周期的临时规则，带 expire_at 过期时间。一期只接收定时任务和硬条件任务

4.3 业务流程（完整生命周期 15 步）

触发器在整个车辆运行期间持续执行以下循环，不是一次性的：

启动初始化：上电 → 加载全量预置规则 → 恢复持久化的定时器状态 → 初始化信号订阅
AlwaysOn 点火：根据上电/Button 状态 → 开启满足条件的 AlwaysOn 任务 → 订阅 VLM 结果 topic
获取信号快照：从信号库读取所有相关 key 的当前值 → 组成本轮快照 → 检查每个信号的 TTL 有效性
条件匹配：遍历所有已加载的规则 → 用运算符逐条比对信号快照 → 复合条件做 AND/OR 组合
穿越检测：对匹配成功的规则做 F→T 穿越检测——上一轮为 False、本轮为 True 才算穿越。防止持续满足条件时反复触发
防抖：穿越成功后进入防抖窗口——连续 N 帧（可配，默认 1）都穿越才算真正命中。防止信号抖动误触
命中确认：防抖通过 → 正式命中 → 进入命中后处理流程
优先级标记：为命中的任务标记优先级 P0-P4。P0 = 安全类不可延迟，P4 = 最低
互斥组检查：检查任务所属互斥组（VLM_CABIN / VLM_FRONT / 同时占用）→ 组内冲突时高优先级抢占
冷却检查：检查该任务的冷却状态——冷却期内不重复触发。P0 任务不设冷却（产品红线：每次满足必须触发）
下发运行指令：向 VLM 业务或车控执行模块下发运行指令——包含 task_id、task_goal、priority、timeout、mutex_group
VLM / 车控执行：VLM 按 SP 执行识别任务 / 车控直接执行动作（如开雾灯、切雪地模式）。端侧触发器不关心内部执行细节
接收 VLM 结果：VLM 完成后返回结果给触发器（"谁调返给谁"）。触发器接收原始输出字符串
透传上报 + 终态处理：① VLM 结果整包透传给 AI Service ② 读取终态（success/empty/timeout/failed）→ VLM 业务将任务状态写回信号库
续接：根据任务类型做续接——周期任务回到步骤3等下一周期；单次任务标记完成；动态任务检查 expire_at → 回到步骤 3，循环继续
关于循环回路：从步骤 15 会回到步骤 3（获取信号快照），形成触发器的持续运行循环。触发器在整个车辆运行期间（上电→下电）持续执行这个循环，而不是一次性的。

4.4.1 支持的运算符（12 种）

4.4.2 复合条件

组合方式说明示例
AND（并）所有子条件同时满足才算匹配舱内 AlwaysOn 开启 = power_state 等于 on AND awake_button 等于 on
OR（或）任一子条件满足即算匹配舱内 AlwaysOn 关闭 = power_state 等于 off OR awake_button 等于 off

4.4.3 穿越检测（F→T 边沿触发）

穿越检测是闸门链的核心——防止"条件一直满足就一直触发"。

上一轮结果本轮结果是否穿越说明
FalseTrue✅ 穿越条件从不满足变为满足——触发
TrueTrue❌ 不穿越条件持续满足——不重复触发
TrueFalse❌ 不穿越条件从满足变为不满足——复位
FalseFalse❌ 不穿越条件持续不满足——等待
为什么不能"条件满足就触发"：假设直接配 battery_level ≤ 20% 就触发——
8:20 电量首次降到 20% → 触发 ✅
8:21 电量仍然 20% → 又触发 ❌ 重复弹窗
8:25 电量 19% → 又触发 ❌
8:35 充了点又降回 20% → 又触发 ❌ 二次骚扰
所以必须有穿越检测：只认 F→T 那一刻，之后持续满足不再触发，直到条件复位（T→F）后再次 F→T。

4.5 命中后：优先级 + 互斥组 + 冷却

命中后，触发器对每条候选任务依次完成三件事：

4.5.1 优先级标记

级别定义示例冷却

P1用户体验·高优安全提醒（驾驶中不安全行为）/ 儿童照看 / 停车寻车按自然周期
P2增值服务舱外迎宾 / 增强哨兵 / 抢车位可配

P4最低（动态任务）动态 Advisor 创建的临时任务可配

4.5.2 互斥组标记

4.5.3 冷却判断

4.7 收尾：终态接收与续接

4.7.1 终态类型

VLM 业务 / 车控执行完成后回传终态，触发器接收并处理：

终态含义触发器处理
success正常完成·有结果① 整包透传给 AI Service ② VLM 业务写回信号库 ③ 进入续接
empty正常完成·空结果同 success 处理——空是有效结论 ② 进入续接
timeout执行超时释放资源 → 不写伪结果 → 进入续接（周期任务等下一轮，单次任务可选重试）
failed执行失败同 timeout → 告警上报
产品红线：触发器只接收终态，中间状态不感知。VLM 执行过程中的进度、中间帧等不回传给触发器。信号库写回和 Context 上报都由 VLM 业务负责，触发器不承担。

4.7.2 续接规则

任务类型续接行为说明
周期任务回到步骤3（信号快照），等待下一个执行周期如 AlwaysOn 高频每 3s 一次，完成一次后等 3s 再回到步骤 3
单次任务标记完成，不再触发如下车安全跑完一次就结束，除非重新满足触发条件（如重新 D→P）
动态任务按 expire_at 判断是否失效；未失效继续按条件命中过期则标记为 expired 并清理
被打断任务不出现在触发器侧由 VLM 业务自行恢复（闲时常驻原则，见 §4.8.1）

4.8 关键策略（横切关注点）

4.8.1 AlwaysOn 闲时常驻

AlwaysOn（舱内 + 舱外）是行车中 VLM 的默认工作状态，不存在"VLM 空闲"这个状态：

  • 被高优任务打断后，由 VLM 业务自行恢复，触发器不参与恢复调度
  • 分高频（每 3s / 5s）和低频（每 5min），低频共享高频的视频流
  • AlwaysOn 是兜底任务——"没有别的事干，就做 AlwaysOn"

4.8.2 合并下发策略

安全提醒（4-5）与儿童照看（4-6）同属 VLM_CABIN，本期采用合并下发：

场景处理方式
同时触发触发器下发合并指令：task_goal = "全舱安全观察 + 儿童状态观察"，VLM 一次运行返回两组结果，触发器分别写入两个任务状态
仅触发其一按单任务下发，不合并
合并运行中其一结束如用户关闭儿童照看 → 触发器下发更新后的 task_goal 退回单任务模式。建议本轮 VLM 跑完后再切换，避免一次调用浪费

不缓存不聚合每周期收到一帧就推一帧，不做攒批

7.2 P0 场景详细走查

P0 · 充电桩 · oneshot链式
充电-自动开充电口盖

场景描述：用户停车挂P档后，视觉确认当前停车位有充电桩，自动打开充电口盖，免用户下车手动开盖。

链路类型：oneshot 链式触发——先硬条件判R→P，再 oneshot VLM 确认充电桩存在。

条件拆解

条件#一级分类信号key类型运算符写法来源
①车控功能-驾驶车辆行进状态vehicle.gear_stateenum: D/P/R/N变更为gear_state 变更为 P（R→P）端状态✅
②舱外元素识别(VLM)充电桩有无vlm.charging_stationenum: 有/无等于oneshot→结果 等于 有VLM✅

完整流程（逐步）

信号变更：端状态写入信号库 vehicle.gear_state = P（从 R 变更而来）
硬条件命中：㉔-J 检测到 gear_state 变更为 P → F→T 穿越 → 命中
冷却检查：检查 cooldown_scope = trip → 本次行程未触发过 → 通过
下发 oneshot：㉔-9 向 VLM 下发 oneshot 指令：camera=rear, task_goal="检测后方充电桩", priority=P0, timeout=5000
VLM 执行：后置摄像头拍一帧 → VLM 推理 → 输出结果（"有充电桩"/"无充电桩"）
结果返回：VLM → 端侧触发器（"谁调返给谁"）→ 触发器接收原始输出
透传上报：㉔-U 整包透传给 AI Service（附 task_type="vlm_oneshot" + timestamp）
云端消费：AI Service → 数据中心解析 → 云端触发器事件判断：充电桩="有" → 条件满足 → 下发执行：开充电口盖
执行：车控指令 → 赛力斯 CDC → 打开充电口盖

频控：一次停车周期仅触发一次（cooldown_scope = trip）
恢复策略：天气恢复不适用（oneshot 无恢复概念）

九、异常处理与产品红线

9.1 异常处理策略（7.4 评审通过）

异常类型处理策略设计原则补充说明
网络波动积压周期类：只传最新帧，压包丢弃中间帧
oneshot：暂存本地，网络恢复后续发一次		周期 = 时效优先（过期没用）
oneshot = 可靠优先（丢了无法重现）		AlwaysOn 断网：本地缓存最新一帧，恢复后推最新
信号 TTL 过期信号作废 → 依赖该信号的场景 frozen防过期数据误触发不同信号 TTL 不同（见 §3.6）
端侧不留存端侧不做数据持久化，只管透传。断网积压的续发完即清端侧算力/存储有限唯一持久化的是：定时器状态 + 任务列表（掉电恢复用）
时序脏数据人换座位 → 先上报一帧空数据清掉前值 → 再上报新值防旧 face_id 残留导致误判超然 7.4 提出：一空一实保证数据干净
VLM 超时单次超过 timeout → 释放资源 → 跳过本帧 → 不写伪结果不阻塞后续周期超时阈值待与丁彬对齐
规则加载失败自动回滚到上一版本 + 告警上报不能用坏规则运行回滚后继续用旧规则匹配
VLM 连续失败连续 N 次 failed/timeout → 暂停该任务 → 告警上报 → 等待人工介入或新规则防止无效重试消耗资源N = [待研发定义]

9.2 产品红线（不可逾越）

#红线说明
1P0 任务不设冷却安全类任务每次满足条件必须触发。延迟或漏触可能造成人身安全风险
2超时不写伪结果VLM 执行超时不能写"无障碍"——模型没看完，结果不可信。写了就是欺骗用户
3过期信号不判定TTL 过期的信号不允许用于条件匹配。宁可 frozen 等新值，不能用过期数据误触发
4一条一判不双判同一个条件只在端侧或云端的一处判定。两边都判 = 逻辑混乱 + 可能重复触发
5不解析不改写透传数据端侧触发器透传的 VLM 输出必须是原始字符串，不得解析、修改、截断、格式转换
6空结果是有效结果"看了没看到"和"没看"是两回事。空结果必须正常写入、正常透传，不能丢弃
