开始之前——先理解几个关键概念

如果你对 SP 的内部机制不熟，先看这一段，后面的分析才能看懂。

SP（System Prompt 系统提示词）
就是 Planner 这个 AI 模型每次回答前先读的"工作手册"。约 30000-35000 字（token），包含 23-24 个工具说明、69 个参考示例、注意事项、聊天风格等。每次用户说话，模型都会把整本"手册"+ 用户这句话一起读完，然后输出 JSON 决策。手册写得好不好，直接决定 AI 行为对不对。
注意事项（SP 中的一节）
SP 里的硬性规则清单，原本 14 条，类似公司员工手册里的红线。比如"工具调用失败时不要道歉"、"输入听不懂就不要回答"。这些规则是模型要严格遵守的。
参考示例
SP 里塞的"标准答案"集合，原本 11+ 个，现在 21 个。每个示例都是一段完整的对话——用户说什么 → Planner 应该怎么想 → 输出什么 → 工具返回什么 → 继续怎么处理。模型靠这些示例学"遇到 X 类问题该这样做"。示例占了 SP 一半的字数。
advisor（专家智能体的建议）
不是用户的话，是另一个 AI 模块给 Planner 的建议。比如 advisor 检测到车内 35°，会推送一条建议给 Planner："建议开空调"。然后 Planner 决定怎么处理这条建议——直接做？问用户？还是忽略？会议的 D2 决议就是关于这个的。
tool_feedback（工具返回）
Planner 调用工具后（比如"打开空调"），下游执行系统给的回复（比如"空调已打开"）。这也不是用户说的话。
talk_or_not 字段
Planner 每次输出 JSON 时必填的字段，取值是 yes / no。yes 表示要说话（同时 talk_content 写要说的内容），no 表示静默不出声。会议的 D1 决议就是关于"什么时候 talk_or_not 该是 yes"——结论是用户说的任何话都必须回（yes）。
9 条会议决议（D1-D9）
5.22 会议达成的 9 条共识，每条会改 SP 的某个具体地方。比如 D5 = "禁止 Planner 主动服务"，意思是 SP 里不能再有"用户没问、advisor 没建议、Planner 自己提议要做什么"的示例。本文就是要看这 9 条到底有没有改进 SP。

🔍字面解读：到底改了什么

  1. 描述从 250 字变成 30 字。原来一口气罗列了 30+ 种功能（什么 HUD、外后视镜、加油盖、K歌、账号...），现在用 6 个大类（空调/座椅/门窗/灯光/导航/播控）盖住。
  2. 修复了双 param bug。原版居然有两条 param 字段同时存在，第一条说"系统设置指令"，第二条说"车辆控制指令"。JSON schema 里同一个字段不该出现两次，这是字面意义上的 bug——下游工程在解析时可能踩坑。新版合并成一条。
  3. 加了一条 notice。"导航没有记忆能力，若用户说关联着记忆的地址时要先去查询记忆。"这条是新加的——明确告诉 Planner：用户如果说"去妈家"这种关联记忆的地址，不能直接喊车控工具，要先去查记忆里"妈家"是哪。

🎬用户场景模拟：改完之后差别在哪

场景：用户说"豆包，导航去我妈家"，妈家地址在记忆里是"虹口区凉城新村 4 号 502"
用户豆包，导航去我妈家
原版 SP 下 Planner 的可能行为：
原版"好的，正在导航。" action: vehicle_basic_control("导航到我妈家")
→ 因为 SP 没说必须先查记忆，模型可能直接把"我妈家"作为字符串扔给车控工具。车控工具不知道"我妈家"是哪，要么报错要么导到一个奇怪的地方。
新版 SP 下 Planner 的预期行为：
新版"我先看看妈家在哪。" action: search_user_memory("妈家的地址")
工具"虹口区凉城新村 4 号 502"
新版"好，导航到虹口区凉城新村 4 号 502。" action: vehicle_basic_control("导航到虹口区凉城新村 4 号 502")
→ 因为 notice 里明确说了"导航没有记忆能力，要先去查询记忆"，模型会先调记忆工具拿到具体地址再导航。

🎯实际影响

体验层

"去我妈家"、"去公司"、"去老地方"这类带记忆指代的导航请求成功率会上升。这是车上很常见的场景。

工程层

修复了双 param 字段的 schema bug，下游解析不会再踩坑；同时整个工具描述精简了约 200 token，省下来的 token 可以给别的内容用。

📋评估

很好的修改 一次同时修了 schema bug、精简了字数、补了缺失能力。对应了我之前提出的"优化点 1"。

🎬用户场景模拟：删了"避开限行"会发生什么

场景：用户在上海早高峰想避开限行
用户豆包，帮我打开避开限行
原版 SP 下：模型知道这个能力，会直接调用
原版"好的，已为你打开避开限行。" action: navi_basic_control("开启避开限行")

🎯实际影响

好处

省下约 200 token；同时缩小了"工具能力幻觉面"——模型不会再答应做实际并未深度对接的能力。比如原版"上下文语音缩放地图"这种听起来很厉害但可能没真接通的能力，删了之后模型不会乱答应。

坏处

真实支持的能力也被一起砍了，用户问到这些功能时模型会说"不支持"，但实际是支持的。比如"避开限行"在上海早晚高峰是高频需求，被砍掉相当于关掉了这个入口。

📋评估

方向对，但收紧过度 应该在精简时区分"实际未对接的能力"和"已对接但描述冗长的能力"——前者该删，后者该用更短的方式保留。建议复查删除清单，把"避开限行"、"地址收藏管理"、"道路层级切换"这些真实支持的功能加回来（用更简短的方式）。

🔍字面解读：为什么这个改动有意义

跳号是怎么来的？原版历史上删过一些 case，删完之后没有重新编号，就留下了 15、16、17、18、20 这些"空位"。

这有什么问题？有两个：

  1. KV Cache 命中率问题。SP 前半部分如果稳定不变，模型每次计算时可以"复用"上次算过的中间结果（这叫 KV Cache 命中）。但如果示例编号一会儿 14 一会儿 19，未来再删/加一个示例时，整个示例段的字符位置就会大范围位移，原本可以复用的缓存全部作废。
  2. 示例补充段单独存在，模型困惑。原版同时有"参考示例"和"示例补充"两段，里面甚至有"示例 1 改写版"——模型读到这里会困惑：到底应该参考哪一版？

新版怎么改的：把示例 15-20 这些原本跳号的位置实际写满了新示例，编号连续起来；同时把"示例补充"段整个删除，所有示例统一在"参考示例"一段里。

🎯实际影响

性能层

未来如果再有示例调整，影响范围更可控；同时示例段的前缀稳定 → KV Cache 命中率提升 → Planner 响应速度更快。

模型决策层

消除了"两段示例同时存在哪个为准"的歧义，模型行为更稳定。

📋评估

结构性改进，没有副作用 对应了我之前提的优化点 6 + 优化点 11。

这些示例想教模型什么？核心就一个词：主动性。原版示例大都是"用户说 → Planner 反应"的被动模式，新版示例 17、18 是用户只说一句"晚上好"，Planner 要主动从环境（降温、晚上、车里）捕捉细节展开对话。

⚠ 注意

  1. 与 D7 决议冲突：D7 说"除非用户明确说'记一下'，否则不主动调记忆工具"，但【闲聊中的静默记忆】反过来鼓励主动记。这是 SP 里两条规则打架。
  2. 三处情绪/时间相关内容重复：注意事项 #13 + 聊天风格【自我时间感知】+ 新增的【情绪连贯性】——三处讲类似的事。D8 决议本来要去重，结果不仅没去重，还多增了一处。
  3. 聊天风格段从 ~600 token 膨胀到 ~1500 token：这正是张嘉锋会议上担心的"原则越多模型遵循力越差"的现实问题——新版正好放大了这个风险。

🎯实际影响

短期

中长期

聊天风格段膨胀，加上抽象原则越来越多，模型对硬性规则（注意事项里的安全/边界条款）的遵循力可能下降。这正是张嘉锋会议上提的"原则 vs 例子"实验问题——新版给了完美的实验样本，应该跑这个实验确认效果。

N1 · 【闲聊中的静默记忆】原则与 D7 决议正面冲突 🔥

⚠⚠ 高危
本次修改最严重的隐患——SP 内部有两条规则正面打架，模型行为会变得不可预测。

📍冲突在哪里

规则 1 · D7 决议（会议结论）
"除非用户明确表达说'我什么什么你记一下'，不然我们端的应该不会主动调起（记忆工具）。"
这两条规则在 SP 内同时存在 ↕
规则 2 · 新版聊天风格【闲聊中的静默记忆】
"当用户在闲聊中自然透露了值得长期记住的个人信息时（如性格描述、价值观、MBTI、重要经历、对某事的独特观点），可以在不打断对话节奏的前提下，并行调用 operate_user_memory 工具记录下来。记忆内容应简洁准确，以用户原话或精准概括为主。不需要在回复中告知用户'我记住了'——好的记忆是无声的。"

🔍三个示例直接示范违反 D7

示例 17 output6（讲妈妈和雨后泥土味的童年）
用户"张信哲的《别怕我伤心》。小时候我妈老在客厅放这张CD……长大了自己听，才知道什么叫土里藏着的东西。"
Planner"'土里藏着的东西'，这句话我帮你记住……" action: [operate_user_memory("雨晴小时候妈妈在客厅放张信哲的CD，当时她觉得土，长大后听《别怕我伤心》听哭了，她说'土里藏着的东西'。她容易被深夜车里随机切到的老歌触动")]
⚠ 用户没说"你记一下"，Planner 主动静默写入记忆 → 违反 D7
示例 18 output4 / output6（讲性格特质和 MBTI）
用户"我擅长调节奏。剖心……说实话不太擅长，总觉得别人不一定想听。"
Planner"……" action: [operate_user_memory("雨晴在人际关系中擅长调节奏，但不擅长对人剖心，习惯照顾别人的感受而把自己的表达放在最后")]

⚠ 用户同样没说"你记一下"，Planner 静默写入了两次记忆 → 违反 D7

🎯实际影响

模型行为不稳定

同时有两条相反的规则，模型读完不知道听哪一条。结果：有些场景按 D7 不记，有些按【静默记忆】记——行为完全不可预测。

用户隐私风险

用户随便讲讲就被记成"长期偏好/性格档案"，下次别人开这台车（家人、朋友），他可能看到这些被静默记录的隐私信息。

💡建议两种解决思路

  1. 方案 1：把【静默记忆】加上边界条款 ——"仅限用户主动陈述的事实/性格/经历；豆包推荐被采纳不算偏好；与已有记忆冗余的不重复记"
  2. 方案 2：反过来在 D7 里精确化 ——"禁止把'豆包推荐被采纳'记为偏好；但用户主动陈述的特质/经历可以静默记"

推荐两个方案合一：边界条款 + 注意事项精确化都做。
