MODULE 2上下文四分法

关键洞察：文章说"AI不强 = 你给的上下文太少"。映射到你的工作：Planner 出 badcase = SP 中的上下文不够好。你每天维护 SP 的工作，就是在"给 Planner 上下文"。

MODULE 3三件套升级路径
学习目标：掌握"工具 + 上下文 + Skill"三件套，知道每一件在你的工作中对应什么。

Prompt: 让 AI 帮你写 Skill

我是车载AI助手（豆包上车）的产品经理，负责Planner模块。 我有一个反复做的工作：Planner的badcase排查。 每次的步骤大概是： 1. 看用户query和Planner的action_list输出 2. 判断是否选对了工具 3. 如果选错，判断根因（链路/工具描述/上下文/模型能力） 4. 根据根因决定改SP的哪个部分 5. 改完后找类似case验证 请帮我把这个工作流写成一个可复用的Skill，包括： - 触发条件：什么情况下执行 - 输入：需要哪些信息 - 步骤：每步具体做什么 - 输出：产出什么 - 检查点：怎么判断做对了

Checkpoint 3

能说出三件套的递进关系（基础设施→燃料→引擎）。能列出你已有的Skill。能用Prompt模板让AI帮你形式化一个Skill。

MODULE 410倍效率 = 回测循环
学习目标：把"10倍效率"从"干活快10倍"重新理解为"试错成本降10倍"，并掌握SP回测的具体方法。

10倍效率 ≠ 干活快10倍。= 试错成本降10倍。 意味着你可以快速验证更多假设，在需求没定死之前就发现方向错了。对SP修改来说：以前改一条等两周看效果，现在改三条当天回测完。

Checkpoint 4

能区分"旧循环"和"新循环"。能说出回测操作的5个步骤。理解10倍效率的本质是试错成本而非速度。

MODULE 6自测 + 本周行动
学习目标：用自测题确认理解程度，选出本周第一个行动。

"AI不强，很多时候不是模型不行，而是 ___"

正确。上下文 = AI输出质量的决定因素。对Planner来说，SP中工具描述模糊 = "丢一句话让AI猜"。
答案是B。不是硬件也不是Prompt长度，而是上下文——业务场景、架构、历史经验。

10倍效率的真正含义是什么？

正确。10倍效率 = 可以快速验证更多假设，在方向定死之前就发现错了。
不是速度快10倍。是试错成本降10倍——能快速出多套方案、离线回测、提前验证方向。

本周行动（选一个开始）

  1. 结构化你的 badcase 日志（上下文资产建设）— 建一个表格，每个 badcase 记录：user_query、Planner 输出、错误类型、根因、修复方案
  2. 建 SP changelog（历史上下文积累）— 从今天开始，每次改 SP 记一行：日期、改了什么、为什么改、解决了哪个 badcase
  3. 用 Claude Code 写一个 badcase 分类脚本（L2→L3 跳跃）— 用上篇 fork 的 Prompt 模板
  4. 审计 3 个工具定义（架构上下文优化）— 用上篇 fork 的 6 项检查清单

最后提醒：文章说"未来12个月PM岗位快速分层"。AGI路线图说"壁垒窗口0.5-1年"。Dan说"骑在模型上"。三篇文章从不同角度指向同一个结论——窗口在关闭，现在开始做比明天开始做好。
