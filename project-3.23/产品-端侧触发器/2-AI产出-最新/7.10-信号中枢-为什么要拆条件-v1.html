PART 2

为什么要拆？— 产品视角追问链

Q
老的并列条件用了这么久，为什么现在突然要拆？

因为触发器这个产品的职责扩大了。

老触发器只做一件事：条件满足 → 调Advisor。五个条件并列判断完全够用。

但韩杰把触发器升级成了"信号中枢"，现在不止调Advisor，还要生成事件日志给Director，未来还要支撑规则引擎画布。产品职责变了，数据结构必须跟着变。

Q
职责扩大了，跟拆不拆有什么关系？不拆也能新增功能啊？

不能。问题出在事件日志。

事件日志要给Director一句人话："14:03 主驾切入D档"。这需要系统知道"到底发生了什么事"。

但老的并列条件里，五个条件地位平等——档位P→D、车窗未关、车速=0 都叫"条件"。系统不知道哪个是起因（发生了什么事），哪个是背景（当时什么情况）。写事件日志的时候，该写谁？

老架构尝试写事件日志
1
条件①：档位 P→D ✅
2
条件②：车窗未关 ✅
3
条件③：车速=0 ✅
4
三个条件同时满足 → 触发
❌ 事件日志写什么？"档位P→D + 车窗未关 + 车速=0"？——这不是"发生了什么事"，这是一堆状态。Director没法用。
新架构写事件日志
1
触发事件：档位 P→D（自带事件描述："档位切入D档"）
2
条件：车窗未关？ ✅ → 继续
3
动作：① TTS播报 ② 写事件日志
✅ 事件日志："14:03 档位切入D档" ——触发事件天然就是事件描述，Director直接可用。
Q
所以拆开的本质是什么？

给每条规则定义一个"主角"。

老平台五个条件都是群演，没有主次之分。拆开之后，触发事件是"主角"（这条规则为什么被激活），条件是"配角"（激活之后还要不要继续走）。

有了主角，事件日志就有了主语；有了先后，复杂流程就能编排了。

Q
就为了事件日志，值得把产品架构改了？

不止事件日志。拆开同时解锁了三个老架构做不了的产品能力——见下一节。

PART 3

拆开解锁了三个产品能力

📋
能力一：事件日志 — 让模型知道"发生了什么"
老架构被卡住：五个并列条件同时满足就触发，但系统不知道谁是"原因"谁是"背景"，没法生成一句话的事件描述。
拆开后解锁：触发事件自带拼接规则（"档位 + 切入/切出 + D档"），自动生成事件描述，喂给Director的事件日志。
韩杰原话（08:06）："信号的变化而导致我们有一些突变值，而突变值我们把它叫做一个触发的事件，而这些触发事件会生成对应的事件日志。"
🔗
能力二：复杂场景编排 — 有先后、有时延、有分支
老架构被卡住：只能做"A 且 B 且 C → 执行"。做不了"先等A发生，延时3秒，再查B，满足走分支1、不满足走分支2"这种有顺序的流程。
拆开后解锁：触发事件是起点，后面可以串行接多个条件、支持时延和分支。从"并行AND"变成了"流程图"。
韩杰原话（14:39）："原来是并行条件判断，现在改成了复合条件判断……对于车载场景，我们会涉及到一些时延，串行改并行、同时并发。"
🎨
能力三：运营画布 — 非技术人员拖拽配规则
老架构被卡住：画布需要不同类型的节点（触发/判断/执行）来拖拽连线。老平台五个条件都是同一种节点，画不出流程图。
拆开后解锁：三种节点类型（🔶触发事件 → 🟡条件 → 🟢动作），运营拖拽即可编排workflow，作为火山座舱标准化产品对外输出。
韩杰原话（15:51）："拖拽一个触发事件，它是档位切入D档，再拖几个条件，串行或并行，再拖一个动作……快速通过画布的方式把能力建设起来。"
PART 4

对你的触发器产品意味着什么？

老定位

触发器 = 条件判断器

我监控一堆条件，都满足了我就调Advisor。
  • 被动响应（轮询等结果）
  • 逻辑简单（只有AND）
  • 输出单一（只能调Advisor）
  • 名字：触发器
新定位

信号中枢 = 事件驱动流程引擎

车上发生了一件事 → 我判断当时情况 → 我执行一系列动作。
  • 主动感知（事件驱动）
  • 逻辑可编排（串行/并行/时延/分支）
  • 输出多元（Advisor + 事件日志 + TTS + 规则引擎）
  • 名字：信号中枢
拆条件不是技术优化，是产品升级。
从"条件满不满足"升级到"发生了什么 → 情况如何 → 该做什么"。
韩杰不叫"触发器"了，叫"信号中枢"——因为产品定位变了，名字跟着变。
PART 5

这件事对你（触发器PM）的三个影响

1
配置平台要改版。韩杰明确说第二阶段要"基于触发事件、条件、动作把平台做重构，包含前端和后端"。你的配置平台产品需求要跟着改——老的"动态条件/静态条件"表单要拆成三段式workflow配置。
2
你的信号清单要和韩杰的三张表对齐。韩杰把端状态拆成了"状态查询 / 触发事件 / 条件"三张表。你之前梳理的端侧信号格式、触发器条件表，都要映射到这三张表里。向超然在会上说"这个表可能后面还得再重新拉齐review"。
3
"触发器"这个产品名可能要淡化。韩杰在会上明确说"没有触发器的概念了，整个系统里面就是触发事件和条件"。你需要判断：你的产品叙事是继续叫"端侧触发器"，还是融入"信号中枢"的框架？这影响你在评审里怎么讲故事。
