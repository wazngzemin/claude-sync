一、触发器是什么？为什么需要它？

你在做的产品叫"豆包上车"（Driver Agent 2.0 驾驶代理2.0版本，简称DA2），是装在上汽荣威D6X车上的AI助手。

这个AI助手有一个"大脑"叫 Director 导演/总决策者。用户说话的时候，Director负责理解意思、做决策、调工具执行。

但Director有一个致命的缺陷：它只有用户说话的时候才工作。

用户不说话的时候呢？比如：
· 副驾驶睡着了——谁来自动调小音量？
· 车里温度超了28度——谁来自动开空调？
· 用户10分钟前说了"10分钟后关座椅加热"——10分钟到了谁来执行？
· 车后面有只小狗——谁来通知系统调整环境？

Director做不了这些事。它需要一个"值班哨兵"——24小时不停地盯着车上的所有传感器和摄像头，一旦发现某个预设的条件满足了，就主动去通知该通知的人。

这个哨兵 = 触发器 = Trigger 触发器 = 你负责的产品。

触发器干3件事，对应英文缩写 CTA：
C = Condition（条件）：盯着信号——车速多少？温度多少？有人在睡觉吗？
T = Trigger（触发）：条件满足了吗？温度>28度了吗？→ 满足了就触发
A = Action（执行）：触发了就通知人去做事——通知Advisor（顾问）调温度/播音乐/提醒用户
类比：如果把整个系统比作一个公司

· Director（导演） = CEO。用户说话的时候CEO亲自接待。
· 4个静态 Advisor（顾问） = 4个VP。舒适VP管温度灯光、出行VP管导航充电、情感VP管打招呼安慰、内容VP管音乐视频。
· 动态 Advisor = 临时项目经理。CEO安排的临时任务（"盯着宝宝毯子"），做完就走。
· 触发器 = 7×24小时值班的安保监控中心。盯着公司所有的监控摄像头和传感器，发现异常（有人闯入/温度过高/烟雾报警）就立刻通知VP或CEO。

安保不做决策（"要不要报警"不是安保决定的，是VP/CEO决定的），安保只负责发现+通知。触发器也一样——它不决定"要不要给用户开空调"，它只负责"温度超了28度我通知舒适VP，VP你自己决定怎么办"。
触发器在整个系统里的位置

用户说话时走的是一个"三层漏斗"：
第1层：RAG（Retrieval-Augmented Generation 检索增强生成）——用已有的规则快速匹配，处理"打开车窗"这种指令
第2层：情景RAG + Arbitration Model（仲裁模型）——注入用户记忆和车况，判断是简单需求还是复杂需求
第3层：Director（导演）——云端大模型深度推理，处理复杂需求

触发器不在这个漏斗里。它是漏斗之外的一个独立系统。

唯一和漏斗交叉的地方：用户说了一句带条件的话（"10分钟后关座椅加热"）→ 漏斗处理 → Director把这个条件注册到触发器 → 触发器自己计时 → 10分钟到了通知Director执行。

6.22会议最大的变化：端侧scope（范围）大幅收窄

这是什么意思？

6.22之前的方案：端侧触发器㉔什么都做——信号扫描、条件判定、防抖、优先级仲裁、穿越检测，一整套复杂逻辑都在车上的AIBox里跑。

6.22之后：这些复杂逻辑全部搬回云端⑮做。端侧只保留3件最简单的事：
① 接收云端下发的规则包 + 存在本地
② 本地定时器（断网也要能倒计时）
③ VLM结果转发（把VLM看到的东西传上去）

为什么要收窄？
仁杰原话："像起步安全这种，放在触发器里面专门去配一下，反而还要让触发器去接所有底层信号，还要考虑稳定性和延时，不如直接底软就做了。"

翻译成人话：端侧AIBox算力有限，塞太多复杂逻辑进去又慢又不稳定。简单的让CDC底软做，复杂的传到云端做，端侧只当"中转站"。

三、信号系统

触发器要知道车上发生了什么，靠的是信号（Signal 信号）。

什么是信号？就是"把车上发生的一件事变成一条数据"。

比如"车门开了" → 变成一条数据：{key: "door_fl", value: "open", source: "CDC", timestamp: "15:30:45"}
比如"温度28度" → 变成：{key: "temperature_cabin", value: 28, source: "CDC"}
比如"VLM看到有人闭眼" → 变成：{key: "动态", value: "闭眼休息，身体仅随呼吸轻微起伏", source: "VLM"}

车上能产生信号的地方一共有6类。但触发器不直接去读这些原始数据——原始数据格式乱七八糟（CAN总线是二进制、VLM是JSON、App是埋点格式），触发器不想一个一个去适配。所以中间有一个"信号库"把所有原始数据转成统一格式，触发器只读信号库里标准化好的数据。

3.2 信号库：统一格式详解

信号库就是一个"翻译中心"。不管信号从哪来（CAN总线的二进制？VLM的JSON？App的埋点？），写入信号库之前都翻译成同一种格式。

为什么要统一格式？因为触发器需要做条件判断（温度>28？安全带==系上？），如果每种信号格式都不一样，触发器就要写N套判断逻辑——太复杂。统一格式后，触发器只需要一套通用的判断逻辑就够了。

信号库有两个：
· ④ 端侧信号库：在AIBox上。存端侧产生的信号（CAN/VLM/定时器）。
· ⑫ 云端信号库：在字节服务器上。是信号的"全集"——端侧的数据通过L9连线同步上来，再加上云端自己产生的信号。

触发器引擎⑮读的是⑫（云端全集），不是直接读④（端侧）。

① enum（枚举）= 固定的几个选项。就像一个下拉菜单，只能从预定义的选项里选一个。
比如：性别只能是"男/女/未知"，车门只能是"开/关"

② number（数值）= 一个具体的数字。

③ str = string（字符串/自由文本）= 一段自由写的文字。VLM的大部分输出都是这种。
比如："闭眼休息，身体仅随呼吸轻微起伏"

source
来源		 这条信号是从哪里来的。来源决定了信号的可信度——CDC来的信号基本100%准确（物理传感器），VLM来的信号可能有误差（AI判断可能出错）。 7种来源：CDC / VLM / App / GPS / Audio / Timer / Seres_CDC（赛力斯底软事件）
timestamp
时间戳		 信号产生的时间。注意是"产生"不是"写入信号库"——中间可能有网络延迟。

格式：ISO 8601（国际标准时间格式）= YYYY-MM-DDThh:mm:ss.mmm		 2026-07-01T15:30:45.123
ttl
= Time To Live
存活时间/保质期		 这条信号的"保质期"（单位：秒）。过期了就失效，触发器跳过不用它做判断。

为什么需要保质期？
想象一下：VLM正常每5秒出一次结果。但某一帧因为算力不够，延迟了15秒才出。如果这帧的TTL设成5秒，那它到达信号库的时候就已经"过期"了——触发器会忽略这个过时的结果，避免用10秒前的旧信息做判断。

不同信号的TTL不同：
· 车速=1秒（变化极快，1秒前的车速没参考价值）
· 车门=5秒
· 安全带=10秒
· VLM=3~5秒
· 座椅占座=30秒（变化慢）		 5（5秒后过期）

四、DSL到底是什么

DSL = Domain Specific Language 领域特定语言

听着很吓人，其实很简单：DSL就是一套"填表格的规范"。

你在配置平台上创建一个触发场景的时候，你填的每一个条件、每一个动作，后台都不是存成一段文字，而是存成一个结构化的格式。这个格式就叫DSL。

为什么不直接存文字？因为机器看不懂文字。"超25度开空调"人能看懂，但机器需要的是：
"条件 = 温度 大于 25" + "动作 = 通知舒适顾问"
每一部分都有固定的字段名、固定的值类型。

DSL一共有5个部分（5个结构）。下面我用一个真实的例子"超过25度自动开空调"来走一遍，让你理解每个部分是什么。
🌰 真实例子：把"超过25度开空调"变成DSL

· 条件：舱内温度 > 25度
· 动作：通知舒适Advisor（它会去决定怎么开空调）
· 策略：每天最多触发10次，两次间隔至少5分钟

后台把它翻译成这样的DSL结构（我用人话标注每个字段）：

· duration（持续时长）—— 比如"温度超过25度持续5分钟"。为什么需要：防止瞬间波动误触发。

七、CTA框架

CTA = Condition-Trigger-Action 条件-触发-执行

CTA是触发器的核心工作框架——整个触发器系统的一切设计都是围绕这3步展开的：

C = Condition（条件）：定义"什么情况下触发"。比如"温度>28度"。
T = Trigger（触发）：持续监控信号，判断条件是否满足。满足了就触发。
A = Action（执行）：触发后做什么。比如"通知舒适Advisor"。

但实际执行的时候，不是条件满足了就直接触发。中间还有一层"门禁"——叫两级匹配。

7.1 两级匹配是什么意思

就像进一栋大楼：先刷工卡进大楼（第一级），再刷指纹进具体的房间（第二级）。

触发器也一样：先检查"车辆当前处于什么阶段"（第一级），再检查"具体条件满足了吗"（第二级）。

为什么要两级？因为同一个条件在不同阶段下意义不同。比如"温度>28度"：行车中触发→开空调很合理；停车锁车后触发→车都锁了开空调没意义。

7.2 两种信号消费模式（6.24确认）

VLM的信号有两种消费模式——触发器主动去读，还是被动等通知？
模式什么意思什么时候用举例
周期性消费
（AlwaysOn "永远开着"）		 VLM的AlwaysOn SP一直在跑，每5秒自动输出一次结果写入信号库。触发器只需要定期读信号库，看看有没有变化。

人话：VLM就像一台一直在拍照的监控摄像头，每5秒自动拍一张照发到信号库。触发器就是监控室里看屏幕的保安——定期扫一眼有没有异常。		 AlwaysOn已经在监测的信号。触发器不需要做额外的事，只需要读。 "有人打电话"——AlwaysOn每5秒输出一次动态描述，里面可能含"握持手机"。触发器读到就知道了。
按需触发
（On-demand "按需"）		 AlwaysOn没在监测某个东西。触发器需要主动让VLM去看——通过15-18 SP拼接引擎给VLM追加一条新的观察指令。

人话：保安看监控屏幕发现看不清某个角落，于是打电话给摄像头操作员："帮我把3号摄像头转过去看看那个角落有没有人。"		 AlwaysOn没在看的东西，需要临时追加观察任务。 "毯子掉没掉"——AlwaysOn不会主动去看毯子。触发器要拼一个SP让VLM特别去观察毯子状态。

八、云端触发器引擎⑮：18个子模块

⑮是整个触发器系统的心脏——你作为触发器PM，这就是你负责的核心模块。

它有18个子模块，按功能分成6组。你可以把⑮想象成一条流水线：

注册组（规则进来→检查→存起来→开始盯信号）
↓
判定组（信号变了→条件满足了吗？→防止误触发→控制频率）
↓
仲裁组（多个场景同时触发→按优先级排队→决定在哪里执行）
↓
事件感知组（状态变化的那一瞬间→打包成事件→推送给Advisor）
↓
输出组（打包→路由→下发端侧）
↓
VLM处理组（枚举/描述分流→判方向→拼SP→下发VLM）
由于18个子模块在v6中已经逐一详细讲解过，这里不再重复每个模块的表格。重点补充v6里没讲透的几个概念：

8.1 "穿越检测"（15-10）到底是什么

穿越（Threshold Crossing 阈值穿越）是6.22会议新增的最重要的概念。

举一个生活中的例子：

你家有一个温度报警器，设了"超过30度就报警"。

没有穿越检测的情况：温度从29度升到31度，报警器响了。然后温度一直在31度不动——报警器每秒钟都在响，因为每秒钟它都检查一次"31>30？是的→报警"。你会被烦死。

有穿越检测的情况：温度从29升到31度的那一瞬间（从"不满足"→"满足"），报警器响了一次。之后温度一直是31度，报警器不再响——因为它知道"你已经超过了，我已经通知过了，不需要重复通知"。只有当温度降到29度以下，再重新升到31度以上的时候，才会再响一次。

技术表述：
F→T = False→True（从不满足到满足）= 穿越发生 = 触发事件
T→T = True→True（一直满足）= 没有穿越 = 不重复触发

施佳杰的定义："事件（event）是唯一时刻的触发；条件（condition）是Context中的持续状态。"

翻译：
· Context里存的是"当前有小朋友在车上"——这是状态（持续的）
· 触发器push的是"小朋友刚上车"——这是事件（那一瞬间的）
· 穿越检测就是把"持续的状态"变成"瞬间的事件"的机制

8.2 "防抖"（15-6）到底是什么

什么是抖动（Jitter 抖动）：信号在阈值附近来回跳。

生活例子：你设了一个"车速>130就提醒"的条件。高速上车速在128~132之间来回波动（路面不平/风阻变化）。如果不防抖：
128→131→提醒！→129→取消→132→提醒！→128→取消→……
10秒钟内提醒了3次——根本不是真正超速，是信号在波动。

怎么防抖：设一个"死区"（dead zone）。比如设125~135的死区：
· 只有从125以下涨到135以上，才算真正超速（穿越了整个死区）
· 在125~135之间来回跳的不算（在死区里波动）

VLM信号的抖动更严重：VLM可能前一帧说"有小狗"，下一帧突然说"没有"（因为小狗换了个姿势VLM没认出来），再下一帧又说"有"。这种抖动叫 glitch（闪烁/短暂错误）。

张兰若原话："但凡出现一次错误，触发器就会有问题。"

所以VLM信号需要"连续N帧一致才算数"的防抖策略。N取多少，取决于VLM的实际抖动概率——这就是为什么你需要问兰若要抖动数据。

8.3 "互斥组"是什么意思

互斥（mutex = mutual exclusion 互相排斥）：同一组里的任务不能同时执行。

举例：VLM舱内摄像头同一时刻只能做一个推理任务（算力不够同时做两个）。所以"看后排有没有小狗"和"看主驾有没有睡觉"不能同时跑——它们在同一个互斥组 VLM_CABIN（舱内VLM互斥组）里。

如果两个同时触发了怎么办？按优先级。高优先级的先跑，低优先级的暂停等待——等高优的做完了再恢复低优的。不是取消，是暂停。

两个互斥组：
· VLM_CABIN：舱内VLM任务互斥
· VLM_FRONT：舱外VLM任务互斥
舱内和舱外之间不互斥——可以同时一个看舱内一个看舱外。

九、端侧触发器引擎㉔

㉔的9个子模块在v6中已经逐一讲过。这里补充一个v6没讲清楚的概念：白名单执行。
白名单（whitelist）执行是什么意思？

大部分触发后的动作需要上云让Director决策——因为Director是"大脑"，要由它判断"这个时候该不该做这个事"。

但有些动作太简单了，不值得跑一趟云端。比如"定时到了→关座椅加热"——这有什么好犹豫的？直接关就行了。跑一趟云端浪费时间（几百毫秒到几秒的网络往返），而且断网了就做不了了。

所以㉔端侧有一个"白名单"——列出哪些动作可以直接在端侧执行，不需要请示Director：
· TTS播报（播一句话）
· 音量调节
· 温度调节
· 车窗开关
· 氛围灯
· AI PUSH（推一条通知）

不在白名单里的（安全相关的车控、复杂交互、需要工具调用的）→ 必须上云让Director决策。比如"关闭安全气囊"这种不可能放在白名单里——必须让大脑决定。

十、四条铁律

这4条是6.22评审会上定下来的绝对不能违反的架构约束。你在写任何方案、评审任何设计的时候，第一件事就是对照这4条检查。
铁律一：所有动态注册必须经过⑮中转

规矩：不管谁要注册触发规则（Director注册条件任务 / Advisor注册感知需求），路径只能是 → ⑮ → ㉔。
不许：Director直接给㉔注册（绕过⑮）、Advisor直接给VLM下SP（绕过⑮）
为什么：⑮是唯一的"规则管理者"。如果绕过，⑮不知道有多少规则在跑，就没法做优先级仲裁和冲突检测。就像公司所有合同必须经过法务审核——如果各部门直接签，法务管不了风险。
铁律二：端侧结果不直推Advisor，必须走信号库

规矩：端侧产生的结果（VLM/定时器）只能走 ㉔ → ④信号库 → ⑫ → ㉑Context
不许：㉔直接push给Advisor
为什么：信号库是唯一的"事实源头"（Single Source of Truth 单一事实来源——所有人查数据都去同一个地方查，避免"你说有小狗他说没有"的矛盾）。绕过就会出现不一致。另外断网时端侧结果先存在本地队列，等联网后再同步——直推就没法做断网缓存。
铁律三：触发器只读信号库，不直接监听原始信号源

规矩：⑮读的是⑫云端信号库里标准化后的数据。不直接读CAN总线、不直接调VLM。
为什么：解耦（Decoupling 让两个模块互不依赖，改一个不影响另一个）。原始信号格式千差万别，信号库统一翻译了，触发器只读翻译后的标准格式。
铁律四：condition→Context，event→⑮主动push

规矩：
· 状态/条件（"车里有小朋友"）→ 写入Context → Advisor推理时自动带上
· 事件（"小朋友刚上车"）→ ⑮组装后主动push给Advisor
不许：Advisor自己轮询（polling 反复问"有变化吗？有变化吗？"）Context来发现事件
为什么：轮询太慢太浪费。触发器的核心价值就是在事件发生的那一瞬间主动push——Advisor不需要自己盯着看。

十一、VLM（视觉模型）

VLM是你当前工作中最复杂、最重要、最需要你推动的一块。

v6里已经讲了VLM的10个输出字段、4条处理路径、12条路径B事件。这里补充v6没讲透的两个关键概念。

11.1 "AlwaysOn SP"是什么

AlwaysOn SP = Always On System Prompt "永远开着的系统提示词"

什么是System Prompt（系统提示词）？

你用ChatGPT的时候，你输入的是"用户消息"。但在你看不到的地方，还有一段"系统消息"在告诉ChatGPT"你是一个什么角色、你应该怎么回答"。这段你看不到的指令就叫System Prompt（SP）。

VLM也一样。它每5秒拍一张舱内照片做推理的时候，是按照一段SP指令来工作的。这段SP告诉VLM：
"你是一个舱内观察员。请描述画面中每个人的位置、性别、年龄、动作、物品、不安全因素。按以下格式输出JSON……"

这段一直在运行的基础SP就叫AlwaysOn SP。它不会被关掉，车只要启动了VLM就按这个SP在工作。

"动态SP"是什么？

在AlwaysOn SP后面追加一段额外的指令。比如Advisor说"帮我看看毯子掉没掉"→ 触发器的15-18模块把这个需求翻译成一段SP → 追加到AlwaysOn SP后面 → VLM下一次推理的时候就会额外去看毯子了。

追加，不是替换！动态SP加上去之后，VLM同时执行基础观察 + 毯子观察。任务完成后，把动态SP删掉，恢复成只有AlwaysOn SP。

最多同时追加3个动态SP（6.30会议确认）。

11.2 "KV槽位"是什么

KV = Key-Value 键值对——一个名字(Key)对应一个值(Value)

什么是KV槽位？

你给VLM布置了一个临时任务（"看看毯子掉没掉"），VLM看完了要把答案写在一个指定的位置。这个指定位置就叫"槽位"。

在VLM真实输出的10个字段里，第10个字段叫"动态查询结果"——这就是那个"指定位置"。VLM的临时任务答案就写在这里。

Key = 这个槽位的名字（比如"毯子检测"）
Value = VLM写进来的答案（比如"后排宝宝的毯子已滑落"）

有一个问题：端状态表里只有1个"动态查询结果"字段。如果同时有3个动态SP在跑（比如同时看毯子+看小狗+看手势），3个答案都要写在这一个字段里——怎么区分哪个答案对应哪个任务？

这就是你在SP拼接方案里要解决的问题之一。可能的方案：用task_id作为前缀区分，或者扩展成多个返回字段。你需要问丁彬确认。

十二、Advisor（顾问）系统

Advisor = 触发器的"客户"。触发器检测到事件后推送给Advisor，Advisor决定怎么响应用户。

v6已经详细讲了静态×4 + 动态×N + 事件类型。这里补充v6没讲清楚的概念。

12.1 "固定拼接"是什么意思

6.30会议确认：Advisor读Context的方式是"固定拼接"——不是Advisor主动去查Context，是推理的时候自动把Context里的信息带上。

什么意思？

每次Advisor要做推理的时候，系统会自动把㉑Context里的相关状态信息"拼接"到Advisor的输入里。Advisor不需要自己说"我要查一下Context"——系统自动带上了。

就像：你每天上班前，秘书已经把今天的日程表放在你桌上了。你不需要自己去日程系统里查——桌上就有。这就是"固定拼接"。

和"订阅"的区别：
· 订阅（subscription）= 你告诉系统"温度变了通知我"，然后你等通知。是主动注册+被动接收。
· 固定拼接 = 你不需要注册任何东西，系统每次都自动给你带上所有状态。更简单但也更"粗暴"（不管你需不需要，都给你）。

12.2 动态Advisor的6.30行为规范（和触发器直接相关）

这5条规范定义了动态Advisor和触发器之间的关系——谁做什么，边界在哪。你需要记住这些，因为你的接口方案要按这个来设计。
#规范人话解释对你的影响
1创建前先查ContextAdvisor不是一收到任务就立刻找触发器——它先看看Context里有没有已有的信息。有就不重复发请求了。减少了触发器的负载——不是每次都要发VLM任务
2给⑮的是"人话需求"，不是SPAdvisor说"毯子掉没掉"（人话），不说"请在AlwaysOn SP后追加以下JSON指令……"（技术语言）。把人话翻译成SP是触发器的活。你的15-18 SP拼接方案需要处理"人话→SP"的翻译逻辑
3SP管理全归触发器⑮SP的创建、追加、合并、删除——全是⑮负责。Advisor不管SP的生命周期。你要在SP拼接方案里设计完整的CRUD（Create/Read/Update/Delete 增删改查）流程
4任务结束→通知⑮清理Advisor判断任务完成后→通知⑮删掉那段动态SP→AlwaysOn恢复原状→Advisor自动销毁你需要设计清理接口（Advisor怎么通知你"我做完了请删SP"）
5不做条件判断Advisor只做"决策"（要不要开空调），不做"判断"（温度有没有超过28度）。判断全归触发器。边界清晰：触发器=判断，Advisor=决策

十六、导航触发器

导航触发器是触发器系统的一个具体应用场景——用户说"快到外婆家时提醒我买水果"，系统在车辆接近目的地时自动提醒。

这个功能需要用到围栏（第三章讲的第4类信号源），并且有一个关键的设计决策：用状态机来管理。

16.1 三种接近状态

车辆和一个POI的关系有3种状态：

① 已接近（approaching 接近中）—— 车辆进入了围栏范围（比如距离外婆家<500m）
② 已到达（arrived 已到达）—— 车辆到达目的地附近（比如<50m）
③ 已离开（left 已离开）—— 车辆离开了围栏范围

每种状态变化都是一次穿越——可以触发不同的事件：
· outside→approaching：提醒"快到了，记得买水果"
· approaching→arrived：通知"已到达，导航结束"
· arrived→left：通知"已离开外婆家"

16.2 one-shot状态机

one-shot 一次性 状态机 = 每个状态变化只触发一次，触发后就标记为"已消费"。

为什么需要one-shot？

如果不做one-shot：车辆在围栏边界来回开（比如围栏边界正好是一个红绿灯），每次穿越边界都会触发——"快到了买水果" / "已离开" / "又快到了买水果" / "又离开了"……

做了one-shot：第一次approaching触发后标记"已消费"。后面即使再次穿越边界也不再触发。除非用户重新注册一个新的围栏任务。

16.3 围栏技术方案（6.25会议决定）

关键决策：优先走端侧围栏，不走云端围栏。

理由①合规：端侧围栏 = 坐标数据不出车。如果走云端围栏，需要高频向云端上报车辆GPS坐标——涉及位置隐私。

理由②实时性：端侧本地计算"车辆在不在围栏范围内"比传到云端算快得多。

风险：高德SDK的Auto版围栏部分下架了（750版本）。端侧围栏的SDK能力不确定——丽萍/俊龙在调研。

备选方案：如果端侧围栏SDK不行，就用导航SDK的ETA（Estimated Time of Arrival 预计到达时间）来估算——但精度不如围栏。
