二、产品定位与愿景 — 1.0 到 2.0 的跨越

2.1 先说 1.0 做了什么

你来之前可能了解过一些背景。我们字节有个 AI 助手叫豆包，就是那个能聊天的、能帮你搜信息的 AI。豆包汽车 1.0 呢，就是把豆包放到了车里的座舱系统上——所谓座舱，就是车里面你能看到的、能摸到的那些东西：中控大屏、语音交互、空调、音乐、导航、车窗车灯这些。

1.0 做得怎么样呢？说实话还挺不错的。用户可以用非常自然的语言跟车聊天，说"帮我开空调到 24 度"、"放一首周杰伦的歌"、"搜一下附近的加油站"，豆包都能处理得很好。

但是——这里有一个非常关键的"但是"——1.0 有一个本质性的边界：它只能管座舱，管不了智驾。

什么意思呢？我举个例子。你可以对豆包说"帮我把空调温度调低"，它能做到；你说"帮我导航去公司"，它也能做到。但是，如果你说"前面那个星巴克靠边停一下"——它做不到。为什么？因为"靠边停"这个动作涉及到方向盘和油门刹车的控制，这属于智驾系统的范畴，而 1.0 的豆包跟智驾系统之间是没有打通的。

你可以这么理解：1.0 的时候，座舱和智驾是两套完全独立的系统。座舱有座舱的大脑（豆包），智驾有智驾的大脑（智驾公司的算法）。它们各管各的，互相不说话。

（你可以理解这个割裂感吗？就好比你家里有一个管家和一个司机，管家能帮你安排生活琐事，司机能帮你开车，但管家和司机之间从来不交流。你想让管家转告司机一句话，做不到。）

2.2 2.0 要做什么——舱驾融合

好，那 2.0 要解决的就是这个问题。四个字：舱驾融合。

什么叫融合？就是让豆包不光能管座舱的事儿，还能指挥智驾。你对它说"前面星巴克靠边停一下"，它能理解你想干什么，然后它去调用智驾系统，智驾系统真的把车开到星巴克门口停下来。你说"开快点"，它能调整智驾的驾驶风格。你说"帮我盯着对面的充电车位"，它能调用摄像头去监视。

PRD 里有一句话我觉得说得很好，说 2.0 的目标是让汽车成为"像人一样思考和行动的物理世界的具身智能体"。

一句话记住 1.0 vs 2.0 的区别：
1.0 = 数字世界的 Agent，只管座舱（能说不能动）
2.0 = 物理世界的具身智能体，管座舱 + 指挥智驾（能说也能动）

2.3 愿景架构图讲解

PRD 第一页有一张大的愿景架构图。这张图很重要，它定义了整个 2.0 系统的分层结构。我来跟你逐层讲。

整张图从上到下分三层：

最上面一层：用户交互层。用户通过什么方式给系统输入信息呢？主要是：语音（自然语言，就是说话）、手势（比如手指着某个方向）、手机 App（远程发指令）、还有车机屏幕上的触摸操作。在 2.0 里，语音是最核心的交互方式。

中间一层：AI 大脑。这是整个系统最核心的部分，也是我们字节负责的部分。它包含了好几个模块——Director、Adviser、Planner、Context、VLM、触发器、信号库——后面第五章我会一个一个讲。你现在先有个大概印象：这一层负责"思考"。

最下面一层：执行层。包括三个主体：智驾系统（由智驾供应商提供，控制车辆的物理运动）、座舱 HMI（给用户反馈）、导航 App（路径规划和 POI 搜索）。

所以整个系统的核心逻辑就是：用户在最上层说了一句话 → 中间的 AI 大脑理解意图、规划任务 → 调用最下层的智驾/导航/HMI 去执行和反馈。

（这三层的结构清楚吗？我再打个比方：用户是"老板"，AI 大脑是"管家"，智驾是"司机"，导航是"地图"，HMI 是"对讲机"。老板说了一句话，管家听懂了、想好了怎么做、然后通过对讲机告诉司机照着地图开。）

三、用户研究

在写 PRD 之前，我们做了一轮用户调研。我简单讲一下我们找了什么人、发现了什么。

我们一共找了三类人：第一类是 11 位网约车和专车的司乘用户，做的是 workshop。第二类是 1 位私家司机，做的是一对一深度访谈。第三类是 2 位 Robotaxi 深度用户，也是一对一访谈。

调研下来，最核心的发现是：不同出行方式之间存在巨大的体验断层。快车服务一般、专车参差不齐、私人司机太贵、Robotaxi 不会主动服务。这几种体验之间的落差，就是我们的机会——做一个 AI 私人司机，有自动驾驶能力、有主动服务意识、每个人都用得起。

四、五个典型场景故事

用研做完之后，我们提炼了五个最有代表性的场景故事。

故事 1：上班路上顺路接娃

小王上车说"智驾去公司，中间经过本溪路幼儿园靠边停下接小宝"。车自动导航，设幼儿园为途经点、公司为目的地。快到幼儿园时自动减速靠边停。接完人说"好了，继续开"。到公司后自动找停车位泊入。考验核心能力：长时任务规划——一句话包含"导航+途经点临停+继续导航+泊车"四个子任务。

故事 2：接人时视觉找人

小张去火车站接朋友，说"停在前面那个穿红衣服拉行李箱的人旁边"。车必须用摄像头"看"——通过 VLM 在画面里找到目标并停靠。考验核心能力：VLM 视觉识别+语义对齐。

故事 3：停车场找位泊车

小李说"把车停到负二层离电梯口最近的充电车位"。考验核心能力：停车场环境极其复杂——有楼层、有区域、有各种参照物，很多停车场没有地图。

故事 4：高速上调整驾驶风格

小赵觉得车开得太慢了，说"开快点"。考验核心能力：模糊表达的理解+"开快点"到底是快多少？

故事 5：下车后自动蹲车位

小周停好车说"帮我盯着对面充电车位，空出来就自动开过去"。然后下车走了。车持续监控，有车走了就自动泊出+泊入。考验核心能力：VLM 长时监控+离车后自动驾驶。

五个故事 → 五大核心能力：
故事 1 → 长时任务规划 + 途经点临停
故事 2 → VLM 视觉识别临停
故事 3 → 停车场泊车
故事 4 → 驾驶风格控制
故事 5 → VLM 长时监控 + 离车自动泊车

后面每一个章节的详细需求都是从这五个故事里拆出来的。

五、系统架构 — 六大区域逐模块拆解

5.1 为什么有两套方案

先说一个背景。我们字节做 AI 能力，不造车。目前有两条并行合作线：

  • 上汽（SAIC）+ MMT + 字节。上汽造车、MMT 做智驾、字节提供 AI。
  • 赛力斯（Seeres）+ 元戎（YuanRong）+ 字节。赛力斯造车、元戎做智驾、字节提供 AI。

最大差异：MMT 有停车场建图能力（有图方案），元戎主要是无图方案。对我们字节侧，核心七大模块两套方案共用，差异只在智驾接口适配层。

5.2 架构图逐区域拆解 — 对照真实架构图来讲

好，现在我打开 PRD 里那张上汽+MMT+字节的系统架构图，跟你从右到左、从上到下把每个模块和每条编号数据流走一遍。你看到这张图被分成了好几个色块区域，我来一个一个讲。

【右侧】用户交互入口 — User 端

图最右边画的是用户（User）和输入输出设备。整条链路从这里开始。

输入链路：

1
用户对着 Mic 说话 → 产生①语音输入（编号 1）
2
Mic 把声音转成②音频信号（编号 2），传给 ASR（语音识别模块）
3
ASR 把音频转成③文本（编号 3），传给意图分类模块

输出链路：

  • TTS（文本转语音）把 AI 生成的文字 → Speaker → 用户听到"好的，即将停在前面 98 米的星巴克"
  • 座舱 HMI → 交互屏显示信息（SR 渲染、文字提示等）

另外右上角有手机端入口：⑳语音+图片（编号 20）→ ㉑ Agent 调用（编号 21）进入系统。

【上方蓝色大区域】座舱云 — 字节的 AI 大脑（全部跑在云端服务器上）

图上方那一大片蓝色区域标注着"座舱云"。这里面所有模块都跑在字节的云端服务器上——不是跑在车机上的！为什么在云端？因为 LLM（大语言模型）需要巨大的算力，车机芯片跑不动完整的大模型。

意图分类：ASR 输出文本后的第一个节点。它的两个输出方向：

  • ④简单任务（编号 4）→ 直接给 Action 分发 Service
  • ④复杂任务（编号 4）→ 交给 Planner 规划

Planner：复杂任务时接手。㉒规划（编号 22）拆解子任务 → ⑤工具调用（编号 5）调 Agent/Tools。

Agent/Tools（工具箱）：车控 Agent、GUI Agent、联网搜索、条件任务。执行完通过⑥工具执行（编号 6）返回结果。

Action 分发 Service：把指令分发给正确的执行模块。输出：⑬Command（编号 13）→ VLA SDK；⑦修改目的地/途经点（编号 7）→ 导航 APP。

VLA SDK：座舱云跟智驾的统一接口层。输出 ⑬⑭Command/Response（编号 13、14）与智驾通信；㉓泊车指令（编号 23）→ 泊车 APP。

导航 APP：接收 ⑦ 修改目的地/途经点，产生 ⑧⑨导航信息（编号 8、9）→ 智驾侧。

统一车辆入口：不管请求从哪来都统一接入，负责 ㊹Context 传输（编号 44）给 Planner。

【中间蓝色区域】座舱端 — 跑在车机芯片上的端侧能力

注意跟"座舱云"的区别：座舱云跑在云端服务器上，座舱端跑在车机芯片上。

Camera Service：管理车上摄像头。两个摄像头输入：

  • ㉚ 前视摄像头图像（编号 30）→ 拍车外的画面
  • ㊵ 舱内摄像头图像（编号 40）→ 拍车内乘客

AlwayOn 框架（VLM）：跑在端侧芯片上的视觉语言模型。接收 Camera Service 的 ㉛图像数据（编号 31），做实时视觉识别。输出 ㊶图像信息（编号 41）给 HMI Service。

HMI Service：管理车机屏幕和语音交互。接收 VLM 的 ㊶ 和来自智驾的反馈信息。同时把 ㊷舱内环境信息（编号 42）上报给座舱云。

座舱云的舱内外感知数据模块通过 ㊸数据同步（编号 43）跟 VLA SDK 同步，让智驾也能获得视觉感知结果。

【左侧蓝色区域】智驾域 — Momenta 的自动驾驶

MFF（Multi-Feature Fusion）：多传感器融合模块，接收 ⑩TBT（编号 10，Turn-by-Turn 导航指引）。

EBM SDK：智驾接口层，接收来自 VLA SDK 的 ⑭Command/Response（编号 14）。

Control：控制决策模块，接收 ⑮⑯结构化控制指令（编号 15、16），输出最终控制信号。

Vehicle Service：车辆服务，输出 ⑪控制指令（编号 11）和 ⑫控制报文（编号 12）→ 底盘域。

Map Provider：地图数据提供。智驾云的 云端融图 模块提供高精地图。

【下方两个区域】底盘域 + 车身域

底盘域：车辆运动控制最底层——制动控制、转向控制、驱动控制。接收智驾 Vehicle Service 的 ⑫控制报文 → CAN/Ethernet 总线 → 制动/转向/驱动执行器。这就是车轮真正转起来、刹车真正踩下去的地方。

车身域：车门控制、车窗控制、空调控制等。"打开车窗"或系统开双闪时，指令通过车身域到达执行器。

六大区域一句话总结：
① 用户端：Mic/Speaker/交互屏/手机 — 人机接口
② 座舱云（云端）：意图分类 + Planner + Agent/Tools + Action 分发 + VLA SDK + 导航 APP + 座舱 HMI + ASR + TTS — 字节的 AI 大脑，跑在云端服务器
③ 座舱端（车机芯片）：Camera Service + AlwayOn VLM + HMI Service — 跑在车机芯片上的端侧能力
④ 智驾域：MFF + EBM SDK + Control + Vehicle Service + Map Provider — Momenta 的自动驾驶
⑤ 底盘域：制动/转向/驱动控制 → 执行器 — 车辆物理运动
⑥ 车身域：车门/车窗/空调控制 — 座舱硬件控制

5.3 七大核心模块逐个讲

模块 ①：Director（对话推理引擎）

Director 是整个系统的大脑中枢，CEO——所有决策都经过它。它干三件事：

第一：意图理解。例："前面的星巴克靠边停一下"→ 分析出：靠边临停 + 参照物=星巴克 + 类型=地图POI + 方位=前面。再例："停在前面拉行李箱的人旁边"→ 靠边临停 + 类型=视觉目标 + 目标属性=人物+行李箱。同样是"靠边停"，参照物类型不同，后续链路完全不一样。

第二：任务编排。无参照物→直接调智驾；POI→先调导航搜POI→再调智驾；视觉目标→先调VLM识别→再调智驾；复杂长时→调Planner拆解。

第三：对话管理。信息不够时多轮对话澄清："前面没找到星巴克，是否就近靠边？"

模块 ②：Adviser（主动服务 / 顾问）

Director 是被动的（用户说了才响应）。Adviser 负责主动说话。

静态 Adviser（4 个）：系统预设的固定规则，覆盖最高频主动服务场景。

动态 Adviser：运行时根据用户需求动态注册。例："到幼儿园帮我找阴凉地方停"→ Director 注册一个动态 Adviser，监听导航剩余距离，距离 ≤ 200m 时自动触发 VLM。

模块 ③：Planner（任务规划器）

处理复杂多步骤任务。例："导航去公司，顺路去幼儿园接小宝，到公司后停到负二层电梯口"→ 三个子任务：① 设导航 ② 添加途经点临停 ③ 泊入。Planner 负责拆解+管理执行状态。

模块 ④：Context（上下文模块）—— 非常重要！

Context 是整个系统的信息中心。Director 每次做决策之前都要先查 Context。它维护 8 种类型的信息：

#类型包含什么谁在用什么时候被查
1用户画像偏好、历史行为、习惯Director做个性化理解时（如判断"开快点"调几档）
2车辆状态车速、挡位、电量、智驾模式Director每次 query 进来第一步就查——判断当前能不能执行
3环境信息天气、时间、GPS 位置Adviser判断触发条件时
4导航状态目的地、途经点、ETA、偏航Director + Adviser判断离目的地多远、有没有途经点等
5对话历史之前的多轮对话内容Director做上下文理解（如"不去了"指的是哪个途经点）
6任务状态当前在执行的任务、排队中的Planner + 触发器判断有没有冲突任务、能不能打断
7智驾状态ACC/LCC/NOA 模式、可用功能Director判断智驾当前能力边界（如能不能靠边停）
8传感器数据摄像头画面、雷达信号VLMVLM 执行视觉识别时
Context 是贯穿所有流程的"大白板"。后面讲每个业务流的时候，你会反复看到"Director 先查 Context"这一步。记住：没有 Context 提供的背景信息，Director 就是个瞎子——它只知道用户说了什么，不知道车在干什么、在哪里、能干什么。

模块 ⑤：VLM（视觉语言模型）— 跑在端侧！

VLM 是 2.0 的全新核心能力。重要：VLM 跑在座舱端（车机芯片上），不是云端！为什么？因为视觉识别需要实时处理摄像头画面，如果传到云端再回来，延时太大（光上传图片就要几百毫秒），车都开过目标了。

三种工作模式：

① 舱外识别：前视摄像头画面 → VLM 找目标 → 返回 2D bounding box [y1,x1,y2,x2] + label。

② 舱内识别：舱内摄像头 → 检测乘客状态（化妆/喂奶/用电脑）→ 自动调整驾驶风格。

③ 长时监控：持续盯着看（如豆包猎手监视充电车位几十分钟）。

模块 ⑥：端侧触发器 + 模块 ⑦：端侧信号库

触发器：管理 VLM 任务的调度——判断触发条件、优先级仲裁（P0-P4）、互斥组（VLM_FRONT / VLM_CABIN）、打断与恢复。跑在端侧。

信号库：统一信号存储中心。格式：key/value/source/timestamp。触发器从信号库读信号判断触发条件。跑在端侧。

云端 vs 端侧 总览：

跑在云端的：Director（意图理解）、Planner（任务规划）、Adviser（主动服务）、Context（上下文）、Action 分发、ASR、TTS、导航 APP、VLA SDK

跑在端侧（车机芯片）的：VLM（视觉识别）、触发器（VLM 调度）、信号库（信号存储）、Camera Service、HMI Service

为什么这么分？需要大算力的 LLM 推理 → 云端；需要低延时的实时视觉处理 → 端侧；需要实时读取车辆传感器 → 端侧。

六、主链路四条路径 — 四个示例逐步骤走读

好，模块讲完了，现在来看它们怎么串起来工作。PRD 里有一张主链路流程图。这张图的核心是：根据 query 类型不同，系统会走完全不同的路径。我用四个具体的 query 示例，分别走一遍四条不同的路径，让你看到数据在每一步经过了谁、传了什么、Context 在哪里被查、智驾在哪里被调。

6.1 路径 A：简单直接指令（无参照物临停）— "前面靠边停"

这是最短的链路。我来逐步走：

用户 Query："前面靠边停"
1
【端侧】Mic → ASR：用户说"前面靠边停"，Mic 收到声音，传给 ASR。ASR 跑在云端（因为语音识别模型也需要算力），把音频流转成文本"前面靠边停"。耗时约 300-500ms。
2
【云端】意图分类：收到文本，做第一层分类——这是个跟智驾相关的指令，不是闲聊、不是查天气、不是放歌。分类结果：④简单任务（不需要 Planner 拆解，直接执行）。传给 Action 分发 Service。
3
【云端】Director 查 Context（关键步骤！）：在 Action 分发之前，Director 必须先查 Context 确认当前状态。它查的具体内容：
— 查智驾状态：当前在智驾模式吗？→ 是（NOA 激活中）→ 可以执行
— 查车辆状态：当前车速？→ 45kph → OK（无参照物没有车速限制）
— 查任务状态：当前有在执行的靠边停任务吗？→ 没有 → 不冲突
— 查导航状态：当前在哪条路？→ 浦东大道 → 不是高速（高速不能随便停）
结论：可以执行。
4
【云端】Action 分发 → VLA SDK：Director 生成指令包：
{action: "pullover", reference: null, direction: "front"}
注意这里 reference=null，因为没有参照物。通过 ⑬Command 发给 VLA SDK。
5
【云端→智驾域】VLA SDK → EBM SDK：VLA SDK 把指令翻译成智驾能理解的格式，通过 ⑭Command 发给智驾的 EBM SDK。智驾收到的数据：action=靠边停，无具体位置（由智驾自主选择停靠点）。
6
【智驾域】EBM SDK → Control → Vehicle Service → 底盘域：智驾开始执行——自主选择安全停靠点（避开消防栓、公交站、斑马线等禁停区域）→ Control 模块计算减速曲线 → Vehicle Service 输出 ⑫控制报文 → 通过 CAN 总线 → 底盘域的制动/转向执行器。车辆开始物理减速和变道。
7
【云端】同时：HMI 反馈：与步骤 5 几乎同时，Director 让 TTS 生成语音"好的，正在靠边停车" → Speaker 播出。座舱 HMI 同步在屏幕上显示 SR 渲染（智驾的行驶路径和停靠目标）。
8
【智驾域→云端】状态回传：智驾停稳后，挂 P 挡、开双闪。智驾通过 ⑭Response 回传"靠边停完成"状态 → VLA SDK → Director → Director 更新 Context 中的任务状态（任务=已完成）。HMI："已经靠边停好了。"
路径 A 关键特征总结：
— 全程主要在云端处理（ASR + Director + Action 分发 + VLA SDK）
— 不经过 Planner（简单任务不需要拆解）
— 不经过 VLM（没有视觉参照物）
— 不经过 导航 APP（没有 POI 需要搜索）
— Context 在步骤 3 被查询，提供车辆状态和智驾状态做可执行性判断
— 智驾收到的数据只有 action + direction，具体停在哪由智驾自主决定
— 端到端耗时：ASR ~400ms + Director ~300ms + 通信 ~200ms ≈ ~1s（最快的链路）

6.2 路径 B：POI 参照指令 — "前面星巴克靠边停一下"

用户 Query："前面星巴克靠边停一下"
1
【端侧→云端】Mic → ASR：音频 → 文本"前面星巴克靠边停一下"。~400ms。
2
【云端】意图分类：判断为④简单任务（虽然有 POI 搜索，但不需要 Planner 拆解多步骤）。
3
【云端】Director 查 Context：
— 智驾状态：NOA 激活中 → OK
— 车辆状态：车速 30kph → OK
— 导航状态：当前在浦东大道，距离目的地还有 2km → OK（不是高速、不是隧道）
— 任务状态：无冲突任务 → OK
结论：可以执行，且参照物"星巴克"是 POI 类型 → 需要调导航 APP 搜索。
4
【云端】Director → 导航 APP 搜索（⑦修改目的地/途经点）：
搜索条件：{keyword: "星巴克", direction: "front_same_side", range_m: 150, road: "current", limit: 1}
— direction=front_same_side 表示搜同向前方（不搜对面车道）
— range_m=150 表示视距 150 米内
— 取最近的 1 个结果
5
【云端】导航 APP 返回结果：
{poi_name: "星巴克咖啡（浦东大道店）", distance_m: 98, lat: 31.2345, lng: 121.5678}
6
【云端】Director → VLA SDK → 智驾（⑬⑭Command）：
{action: "pullover", reference: {type: "poi", name: "星巴克", lat: 31.2345, lng: 121.5678, distance_m: 98}}
注意跟路径 A 的区别：这里 reference 不是 null，而是带了 POI 的经纬度坐标。智驾知道精确的停靠位置。
7
【云端→导航 APP→智驾域】导航信息同步：与步骤 6 同时，导航 APP 也会把 POI 位置通过 ⑧⑨导航信息 同步给智驾。这就是"两条并行链路"：豆包→智驾（直接发指令）+ 豆包→导航→智驾（更新导航信息）。两条路的数据必须一致。
8
【智驾域→底盘域】智驾执行：EBM SDK 收到指令 + 导航信息 → Control 计算路径（减速 + 变道到最右侧 + 在星巴克门口停靠）→ Vehicle Service → ⑫控制报文 → 底盘域执行器。
9
【云端】HMI 反馈：TTS："好的，即将停在前面 98 米的星巴克。" + 屏幕 SR 渲染。
10
【智驾域→云端】完成回传：停稳 → P 挡 + 双闪 → ⑭Response 回传 → Director 更新 Context → HMI："已停好。"
路径 B 与路径 A 的关键差异：
— 多了步骤 4-5：调导航 APP 搜索 POI。路径 A 不需要搜索。
— 多了步骤 7：两条并行链路。路径 A 只有一条直接链路。
— 智驾收到的数据带了精确的 POI 经纬度坐标，路径 A 只有方向。
— 端到端耗时更长：多了导航搜索的时间（~300ms），总计约 1.3-1.5s。
— Context 的作用相同：在步骤 3 提供可执行性判断。

6.3 路径 C：VLM 视觉指令 — "停在前面拉行李箱的人旁边"

这是最复杂也最有特色的链路——涉及云端+端侧协同。

用户 Query："停在前面拉行李箱的人旁边"
1
【端侧→云端】Mic → ASR：音频 → 文本。~400ms。
2
【云端】意图分类：判断为④简单任务（虽然要调 VLM，但不需要 Planner 多步拆解）。
3
【云端】Director 查 Context（这一步很关键！）：
— 智驾状态：NOA 激活中 → OK
— 车辆状态：当前车速 = 28kph → ≤ 35kph 阈值 → VLM 可用！
如果这一步查出来车速 = 50kph → 超过 35kph → Director 直接回复用户"车速有点快，看不清参照物，是否就近靠边停？"→ fallback 到路径 A
— 任务状态：当前有没有其他 VLM 任务在跑？→ 没有 → 不冲突
— 传感器数据：前视摄像头是否可用？→ 可用
结论：可以执行，参照物是视觉目标 → 需要调 VLM。
4
【云端→端侧】Director → VLM（关键转换！从云端到端侧！）：
Director 把语义描述传给端侧 VLM：
{target_description: "拉行李箱的人", target_type: "person", attributes: {carrying: "行李箱"}, direction: "front"}
注意数据流方向：这是从云端发到端侧！因为 Director 在云端，VLM 在端侧。这个通信通过车机的网络连接完成。
5
【端侧】VLM 执行视觉识别：
— Camera Service 提供前视摄像头当前帧（㉚㉛图像数据）
— VLM 在画面中搜索匹配"拉行李箱的人"的目标
— 这一步完全在端侧完成，不上传图片到云端
— 耗时约 1.25 秒（端侧芯片推理速度有限）
— 找到了！返回：{box_2d: [120, 340, 280, 520], label: "拉行李箱的人", confidence: 0.87}
6
【端侧→云端】VLM 结果回传 Director：
VLM 把识别结果（2D box 坐标 + label）回传给云端的 Director。Director 收到后确认识别成功。
7
【云端→智驾域】Director → VLA SDK → 智驾：
{action: "pullover", reference: {type: "visual", box_2d: [120,340,280,520], label: "拉行李箱的人", original_query: "停在前面拉行李箱的人旁边"}}
跟路径 B 的关键区别：这里传的不是经纬度坐标，而是 2D 像素坐标！智驾系统需要自己把 2D 像素坐标转换成 3D 世界坐标（通过摄像头标定参数+深度估计）。
8
【智驾域→底盘域】：智驾把 2D 坐标转 3D → 计算停靠路径 → Control → Vehicle Service → ⑫控制报文 → 底盘域 → 车辆减速靠边停。
9
【端侧+云端】HMI 反馈：VLM 识别过程中 → 屏幕显示豆包的"思考中"动效；识别到后 → TTS："好的，即将停在拉行李箱的人旁边。"
10
状态回传：同路径 A/B。
路径 C 的核心特征——云端+端侧协同：

① 数据流经了两个域：用户语音 →【云端】ASR → Director 意图理解 + 查 Context → 【跨域通信】 → 【端侧】VLM 视觉识别 → 【跨域通信】 → 【云端】Director 收结果 → 【云端→智驾域】发指令

② 为什么 VLM 不在云端？如果把摄像头图片上传到云端做 VLM，光上传一帧 1080p 图片就要 200-400ms，加上 VLM 推理时间，总延时会大大增加。端侧直接在本地处理图片，省掉了图片上传的时间。

③ Context 在这个链路中的关键作用：如果 Context 里的车速 > 35kph → 整个 VLM 链路不会被触发，直接 fallback 到简单临停。Context 是"守门员"。

④ 智驾收到的数据跟前两条路径完全不同：
路径 A：只有 action + direction（无位置）
路径 B：action + POI 经纬度坐标（精确 GPS）
路径 C：action + 2D 像素坐标（需要智驾自己转 3D）

⑤ 端到端耗时最长：ASR ~400ms + Director ~300ms + 跨域通信 ~100ms + VLM ~1250ms + 跨域通信 ~100ms + 通信 ~200ms ≈ ~2.3s（挑战 2s 目标）

6.4 路径 D：复杂多步骤任务 — "导航去公司，顺路去幼儿园接人，到公司停到 B2 电梯口"

用户 Query："导航去公司，顺路去幼儿园接小宝，到公司后停到负二层电梯口旁边"
1
【云端】ASR + 意图分类：文本进来，意图分类判断为④复杂任务——因为一句话里有三个子任务，需要 Planner 拆解。传给 Planner。
2
【云端】Director 查 Context：
— 智驾状态：当前是否可以激活智驾？（可能还没激活，用户这句话可能隐含了"激活智驾"的意思）
— 用户画像："公司"在哪？→ 查用户设置的常用地址，公司=浦东新区XXX
— 导航状态：当前有没有正在导航？→ 没有 → 需要新建导航
— 车辆状态：当前位置在哪？→ 在家附近
3
【云端】Planner 任务拆解（㉒规划）：
Planner 把这一句话拆成三个有序的子任务：
子任务 1：设导航（目的地=公司，途经点=幼儿园，幼儿园临停属性=true）→ 立即执行
子任务 2：到幼儿园时靠边停（临停）→ 触发条件：距幼儿园 ≤ 300m
子任务 3：到公司后泊入 B2 电梯口 → 触发条件：到达公司停车场
注意：子任务 2 和 3 不是现在执行，而是注册为"待触发"状态！
4
【云端】Planner 执行子任务 1：
— 调导航 APP：设目的地=公司，添加途经点=幼儿园（临停=true）
— 导航 APP 算路 → 下发路线+途经点信息给智驾和 Context
— 激活智驾 NOA
5
【云端】Planner 注册子任务 2 为动态 Adviser：
注册条件：{trigger: "nav.remaining_distance_to_waypoint_1 <= 300m"}
这个动态 Adviser 会持续监听 Context 中的导航状态。
6
【云端】Planner 注册子任务 3 为另一个动态 Adviser：
注册条件：{trigger: "nav.arrived_at_destination == true && location.type == 'parking_lot'"}
7
HMI 反馈："好的，帮你导航去公司，途经幼儿园接小宝，到公司后停到负二层电梯口。智驾已激活。"

…… 车行驶 20 分钟后，快到幼儿园 ……

8
【云端】动态 Adviser 触发：Context 中的 nav.remaining_distance_to_waypoint_1 变为 280m → ≤ 300m → 触发！
Adviser 通知 Director → Director 调智驾执行靠边停 → HMI："快到幼儿园了，正在靠边停车。"
9
用户下车接人，回来说"好了，继续开"：
Director 理解 → 继续导航去公司 → 智驾恢复 NOA → Planner 更新任务状态：子任务 2 = 已完成。

…… 又开了 30 分钟，到公司停车场 ……

10
【云端】第二个动态 Adviser 触发：到达公司停车场 → 触发子任务 3。
Director 查 Context：公司停车场有图吗？→ 假设有图有 POI → 直接调智驾导航到 B2 电梯口泊入。
如果无图 → 走 VLM 链路引导（这就串联到了后面十一章泊车的流程）。
路径 D 的核心特征——Planner + 动态 Adviser + Context 持续监听：

① 跟路径 A/B/C 的本质区别：前三条路径是"一说就做"——用户说完马上执行。路径 D 是"说了先记着，条件到了再做"——任务被拆解成多个阶段，每个阶段由动态 Adviser 监听 Context 来触发。

② Context 在这个链路中的作用升级了：不再只是在 query 进来时被查一次。而是持续被监听——动态 Adviser 每隔一段时间就去读 Context 中的导航剩余距离，判断是否到了触发阈值。Context 从"被动查询"变成了"持续数据源"。

③ Planner 的价值体现：如果没有 Planner，Director 只能处理"一说一做"的简单指令。有了 Planner，系统能理解"先做A，到了B做C，到了D做E"这种条件分支的长时任务链。

④ 这个链路横跨了几十分钟甚至更长时间。从用户上车说那句话，到最后泊入 B2 电梯口，可能过了一个小时。整个过程中 Planner 持续管理任务状态。

8.3 性能延时图拆解

PRD 旁边配了一张性能延时拆解图。目标：端到端 2 秒（从 query 结束到智驾开始动）。

延时预算拆解：
ASR 语音转文字：~400ms
Director 意图理解 + 查 Context：~300ms
导航 APP 搜索 POI：~300ms
VLA SDK 通信：~200ms
—— 总计：~1200ms ≈ 1.2 秒，在 2 秒目标内

剩余 800ms 是什么？是"安全余量"——网络波动、云端负载高峰时 LLM 推理慢、导航 APP 搜索慢等情况的 buffer。

2 秒的物理意义：30kph 时 2 秒走 16 米。如果超过 2 秒才开始减速，用户体感就会觉得"它反应好慢"，甚至可能开过目标。

总停车距离（端侧 LLM）：6.4m + 10m（VLM 可并行）+ 35~50m ≈ 56.4m

总停车距离（云端 LLM）：17.6m + 10m + 35~50m ≈ 67.6m

而用户能看清并描述参照物的有效距离：50~80m。

所以：端侧 LLM 方案（56m）刚好在 80m 范围内 ✅；云端 LLM 方案（68m）也在范围内但余量更小。如果车速更快（如 50kph = 13.9m/s），所有数字都要乘以 1.67 倍，总距离远超 80m → 车会开过目标。

性能图的核心结论：

① ≤ 35kph 才支持 VLM 临停（35kph 这个值还是 TBD，需要实车验证）
② 端侧 LLM 比云端 LLM 省 11m 停车距离（56m vs 68m）→ 推荐端侧方案
③ VLM 跑在端侧的必要性：如果 VLM 也放到云端（上传图片+推理+下发结果），Stage 2 耗时会从 1.25s 变成 3s+，总距离直接爆表
④ 以上计算不包含用户说话的时间和说话期间车行驶的距离——实际更紧张

完整走读 — 从发起到执行的全生命周期
1
【云端】Director 意图理解："智驾去幼儿园，找个阴凉的地方停" → 拆解为：
子任务 A：导航去幼儿园（立即执行）
子任务 B：到幼儿园附近时用 VLM 找"阴凉的地方"（条件触发）

— 找到了！→ 返回坐标给 Director

7
异常：VLM 没找到：车到了目的地附近，VLM 始终没找到阴凉处 → 智驾自主靠边停（兜底）→ HMI："没找到阴凉的地方，先帮你靠边停了。"

十一、业务流：泊车（含决策树 + 有图/无图两条流程）

11.1 决策树走读 — 进停车场前的三个判断

PRD 里泊车场景有两张决策树图：一张是"有图 vs 无图"判断，一张是"有 POI vs 无 POI"判断。这两张图决定了后续走哪条业务流。

用户："停到 P7 沃尔沃店旁边"— Director 怎么做三个判断
1
判断 1：停车场有图吗？
Director 查 Context 的智驾状态——智驾系统是否有该停车场的建图数据？
— MMT 方案：可能有图（MMT 有众包建图+单车建图能力）
— 元戎方案：大概率无图
→ 假设结果：无图
2
判断 2：有地图 POI 吗？
Director 查导航 APP——停车场地图上有"沃尔沃店"这个 POI 吗？
→ 假设结果：无 POI（沃尔沃店不是地图标注的 POI）
3
判断 3：智驾有 VLA 漫游能力吗？
无图情况下，智驾能不能在停车场内自主漫游（识别入口/出口/楼层/区域）？
— 如果有 VLA 漫游 → 智驾能自己在场内走，VLM 负责找 POI
— 如果没有 VLA 漫游 → VLM 要承担全部导引（楼层+区域+POI）
→ 假设结果：有基础 VLA 漫游

11.2 四象限矩阵

有 POI无 POI
有图最简单：智驾直接导航到 POI 泊入。不需要 VLM。中等：智驾到目标楼层/区域，最后几百米 VLM 识别+指引。
无图最复杂：有 VLA 漫游→智驾场内走+VLM找POI；无 VLA 漫游→VLM 全程导引。

11.3 有图有 POI 业务流走读（最简链路）

"停到 P7 沃尔沃店旁边" — 假设有图有 POI
1
Director 查 Context：智驾有 P7 地图？→ 有。地图上有沃尔沃 POI？→ 有。
2
Director → VLA SDK → 智驾：{action: "park", reference: {type: "poi", name: "沃尔沃", parking_lot: "P7", floor: 2}}
3
智驾执行：基于停车场地图导航到 2F 沃尔沃位置 → 在附近找空车位 → 泊入。全程不需要 VLM。
4
HMI："为你找到沃尔沃门店在 2 层，即将前往。" → "正在泊入。" → "已泊入。"

11.4 无图无 POI 业务流走读（最复杂链路）

"停到 P7 沃尔沃店旁边" — 假设无图无 POI
1
【云端】Director 查 Context：无图、无 POI → 需要 VLM 帮忙。
2
【云端】Director 调联网搜索（Agent/Tools 里的联网搜索工具）：
搜索"虹桥 P7 沃尔沃店" → 返回：在 T2-P7 2 楼。
为什么要联网搜？因为车机导航只有路面 POI，停车场内部的店铺位置导航不知道。但互联网上可能有大众点评、高德地图的评论里写了"在 2 楼"。
3
【云端】Director 拆解任务 → Planner：
子任务 1：智驾进入 P7 入口（导航 APP 能导到入口）
子任务 2：智驾 VLA 漫游到 2 楼
子任务 3：在 2 楼唤醒 VLM 识别"沃尔沃" → 找到后在附近泊入
4
【智驾域】子任务 1-2 执行：智驾开进 P7 → 通过坡道上 2 楼。这一步靠智驾自己的 VLA 能力（识别坡道、入口、出口）。
5
【端侧】子任务 3：VLM 长时检测：
到 2 楼后，VLM 开启持续模式，通过前视摄像头识别"沃尔沃"——包括：
— 指引标识（导视牌上的"沃尔沃→"）
— 门店招牌（"VOLVO"字样）
— 门店外观特征（沃尔沃 4S 店的装修风格）
找到后返回位置信息给 Director → Director 告诉智驾"在这附近泊入"
6
【智驾域】泊入：智驾在沃尔沃附近找空车位 → 泊入。
7
异常：VLM 找不到沃尔沃怎么办？
→ Director 回到 2 楼漫游状态 → 让智驾继续巡航 → VLM 继续检测
→ 如果巡航完 2 楼还是没找到 → HMI："在 2 楼没找到沃尔沃门店，是否在 2 楼找个空车位停好？"
有图 vs 无图的核心对比：

有图有 POI：Director → 智驾（一步到位）。云端完成，不需要 VLM。
无图无 POI：Director → 联网搜（查楼层）→ Planner 拆解 → 智驾 VLA 漫游 → VLM 持续检测 → 找到后智驾泊入。云端+端侧深度协同，链路最长。

Context 在泊车中的作用：Director 的第一步就是查 Context 中的智驾状态——有没有图、有没有 POI。这个判断决定了后续走哪条链路。如果判断错了（以为有图其实没图），整个链路都会走错。

十二、业务流：豆包猎手（含监控 + 抢位两条子流程）

PRD 里豆包猎手有一张大的业务流程图，分成两条子流程：监控子流程和抢位子流程。旁边还有一张监控范围图。我来分别走读。

12.1 监控子流程走读

"帮我盯着对面充电车位，空出来去抢" — 监控阶段
1
【云端】Director 意图理解：
— 两个子需求：① "盯着"=VLM 长时监控 ② "空出来去抢"=自动泊入
— 目标：对面的充电车位
2
【云端】Director 查 Context 做前置条件检查：
— 电量：够吗？（VLM 长时运行 + 后续自动泊入都需要电）→ 电量 > 最低阈值（TBD）→ OK
— 摄像头：前视摄像头没被遮挡？→ OK
— 智驾：后续能激活吗？（VPA/VLA 功能可用？）→ OK
如果任一条件不满足 → 直接拒绝："电量不足，无法长时间监控。"
3
【云端→端侧】Director → VLM 开启长时监视：
{mode: "long_term_monitor", target: "充电车位", target_status: "occupied→empty", camera: "front"}
VLM 持续盯着，不是拍一帧就完！
4
【端侧】VLM 持续执行：
— 每隔几秒处理一帧前视画面
— 识别对面车位的状态：有车（occupied）/ 无车（empty）
— 监控范围：前视广角 120°，约覆盖对面 6~7 个车位（PRD 的监控范围图画的就是这个扇形覆盖区域）
— 持续监控可能长达几十分钟甚至更久
— 同时通过手机 App 推送实时状态给用户
5
VLM 检测到车位空出：某辆车开走了！→ VLM 返回：{event: "spot_available", spot_id: 3, box_2d: [...]} → 通知 Director

12.2 抢位子流程走读

监控检测到车位空出 → 自动抢位
6
【云端】Director 收到 VLM 通知：对面 3 号充电车位空出。
7
【云端】Director 再次查 Context：
— 智驾状态：VPA 可激活？→ OK
— 车辆状态：电量还够？→ OK（虽然监控消耗了一些电，但仍在阈值以上）
— 任务状态：没有其他冲突任务？→ OK
8
【云端→智驾域】Director → VLA SDK → 智驾：
{action: "grab_spot", steps: [ {step: 1, action: "unpark_current"}, // 先从当前车位泊出 {step: 2, action: "drive_to_target_spot"}, // 开到目标车位 {step: 3, action: "verify_spot_available"}, // 到了之后再验证一次车位确实空的 {step: 4, action: "park_in"} // 泊入 ]}
9
【智驾域】智驾执行步骤 1：泊出当前车位
10
【智驾域】智驾执行步骤 2+3：开到目标车位 + 验证
为什么要步骤 3 验证？因为从 VLM 检测到车位空出到车开过去，中间可能有几十秒到几分钟。这段时间可能有别的车先抢了。智驾到了目标车位后，用自己的感知（雷达/摄像头）再确认一次。
11
分支 A：车位确实空的 → 智驾泊入 → 完成 → HMI + 手机 App："已成功泊入充电车位。"
12
分支 B：车位被别人抢了 → 智驾回传失败状态 → Director 有两个选择：
选择 1：调智驾回原车位（如果原车位还空着的话）
选择 2：调智驾在附近漫游找其他空车位
同时：Director 重新调 VLM 继续监控 → 回到监控子流程步骤 4
豆包猎手的链路特征总结：

① 横跨时间最长的链路：从用户下车说出指令到最终泊入充电车位，可能经过 30 分钟到几小时。

② 唯一的"离车后自动驾驶"场景：用户不在车里，车自己动。这是 L4 级别能力。

③ 两个域深度协同：端侧 VLM 持续监控（几十分钟）+ 云端 Director 决策 + 智驾域自动泊出泊入。

④ Context 的作用：前置条件检查（电量、摄像头、智驾可用性）+ 抢位前再次确认状态。

⑤ 监控范围限制：一期只支持前摄，120° 覆盖约 6-7 个车位。不能看后面、侧面的车位。

总结

好，今天的内容就是这些。我最后帮你理一遍主线：

  1. 大背景：1.0 只管座舱 → 2.0 舱驾融合 → 物理世界具身智能体
  2. 架构六大区域：用户端 → 座舱云（云端 AI 大脑）→ 座舱端（端侧 VLM+触发器）→ 智驾域 → 底盘域 → 车身域

  6. 云端 vs 端侧：LLM 推理/意图理解/任务规划 → 云端；VLM 视觉识别/触发器/信号库 → 端侧

  8. 泊车：四象限矩阵——有图有 POI（最简）/ 有图无 POI / 无图有 VLA / 无图无 VLA（最复杂）
