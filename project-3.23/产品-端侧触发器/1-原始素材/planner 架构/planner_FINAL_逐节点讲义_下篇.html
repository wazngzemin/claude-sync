9感知订阅模块——把"帮我盯着"翻译成"看什么、什么条件、多久一次"

动态 advisor 说"每10s告诉我宝宝状态"，这句自然语言要变成能执行的配置：让 VLM 去看二排、设一个10s定时器、条件满足时唤醒 advisor。干这件事的就是感知订阅模块。

骨架图：三个推理子模块 + 组装触发

动态advisor(上游)三个推理子模块组装触发/触发器/定时器VLM感知器

骨架图讲解：一句自然语言怎么变成三个"槽位"

你可以把感知订阅模块想象成一个"翻译官"：advisor 说的是人话（"每10s告诉我宝宝状态"），底层的 VLM/触发器/定时器只认配置参数。翻译官把人话拆成三个正交的"槽位"：

槽位①动态感知需求——"看什么"。如果要看，就去调 VLM 的 sp/attention（让视觉模型重点关注特定对象）。不是所有订阅都需要视觉——"温度超20度开空调"不需要看，只需要条件信号，动态感知填 na。只有"二排宝宝状态""沿途风景"这种需要摄像头画面的才填。
槽位②条件需求——"什么条件下触发"。比如"温度>20度""face id 检测到 xx 上车""车速>120"。这些条件被交给条件触发器持续监控。没有条件的填 na。
槽位③定时需求——"多久一次"。比如"每10s""每5min""20min后（一次性）"。这个交给定时器去跑。没有定时的填 na。

第四能力"组装+触发"是把三个槽位的输出拼在一起落地：比如"照顾宝宝"拆出来是{感知=宝宝视觉, 条件=na, 定时=每10s}→配置 VLM sp 关注二排 + 设10s定时器→每10s到点时，取 VLM 最新的宝宝描述，拼成信号，唤醒动态 advisor 并把最新感知带给它。

5 个真实 case 的拆解结果，帮你理解"哪些用了视觉、哪些没用"：
用户需求动态感知条件定时
20分钟后关座椅加热nana20min(一次性)
温度超20度开空调na温度>20na
照顾二排宝宝宝宝视觉状态 ★na每10s
每5min播报NBAnana每5min
介绍沿途风景周边视觉状态 ★na每10s

注意：只有宝宝和风景用了视觉感知（带★的），其他全是纯条件/定时，不需要摄像头。而视觉持续任务同时只允许 1 个——这也是为什么表里只有一种视觉任务能跑。

Badcase 排查：①拆解错（"每10s"拆成了一次性→只触发一次就停了）→查三维度槽位填充；②该调 VLM 却没调（宝宝需要视觉但动态感知填了na）→感知需求推理失误；③触发了但感知信息是旧的（定时到了但 VLM 描述还是5分钟前的）→组装时取到了过期感知，查 VLM 写回 context 的流式更新是否正常。

10视觉感知器 / Watcher / VLM——让文本大脑"看见"车内发生了什么

Director 是一个文本模型，它看不见图像。VLM（Vision-Language Model）就是它的"眼睛"——把摄像头画面转成一段自然语言描述，写进 context 让 Director 读。

骨架图：Watcher 的三大功能 + VLM 大小模型融合

骨架图讲解："视觉转语言"到底是什么原理

你可以把 VLM 想象成一个坐在车里的观察员，TA 眼睛盯着 OMS 摄像头（前后视镜），每隔几秒看一段 10-15 秒的视频，然后用文字描述看到了什么。这段文字就是"情景描述"，写进 context 让 Director 读。

技术上怎么做的？大小模型融合：

小模型——纯视觉编码器，把摄像头画面变成一堆"视觉 token"（可以理解为"图像的压缩表示"）。它不理解语义，只负责把像素变成模型能读的格式。它是训练过的，能识别一些既定的高优标签（如"有人""安全带未系"），相当于兜底。
大模型——语言模型，拿到视觉 token 后，按照一段System Prompt（SP）来看画面。SP 写的是"识别目标/关注对象"，比如"重点关注二排乘客的状态""关注车外是否有特殊车辆"。大模型不训练、只引导——靠 SP 告诉它该看什么，它就输出一段自然语言描述。
输出——一段结构化的、流式更新的自然语言。比如："主驾明骏，穿粉色T恤戴黑墨镜，在开车；后排左座婴儿闭眼平稳呼吸，疑似睡眠中；车外左侧白色轿车，右侧黑色越野。"这段话写进 context 的 {{舱内视觉感知}} 块，Director 每轮推理都能读到。
"影响感知器 sp/attention"到底是什么？就是改 VLM 大模型的 SP。比如没有订阅时，VLM 用"既定 SP"做通用观察；当感知订阅模块收到"关注二排宝宝"的需求后，就给 VLM 拼一段新的 SP："在既定识别基础上，重点关注二排乘客（婴儿）的睡眠/清醒状态、情绪、肢体动作"。同一个 VLM，拼不同 SP 就切换关注对象——不需要为每个需求各起一个模型。
三种不同需求下 VLM 拼的 SP：
· 无订阅："既定SP"——通用观察舱内外
· 持续关注女儿睡眠："既定SP + 条件任务SP（重点关注后排女性乘客睡眠/清醒状态）"
· 即时查后方车辆："生成的即时任务SP（重点关注后方车辆品牌型号）"
这就是"同一VLM拼不同SP"的真实形态。
VLM 视觉持续任务同时只允许 1 个——这是整套感知层最硬的约束。意味着"照顾宝宝"（盯后排）和"介绍沿途风景"（盯窗外）不能同时跑。用户如果同时提了两个，数量仲裁会拒绝第二个。另外，VLM 运行在端侧，处理链（拍摄→编码→大模型推理→输出描述）有延迟。如果定时器设得太短（如每1s触发），VLM 可能来不及刷新——advisor 拿到的描述还是上一次的。

Always on 事件发现——被动触发主动服务

除了被 advisor 订阅的"定向感知"，Watcher 还有第三功能：自己发现值得关注的事件。这些事件由云端 Agent 以自然语言注册（比如注册"小孩哭闹""前方大货车""乘客 OK 手势"），Watcher 发现后触发 advisor 做主动服务。一期是纯工程触发（基于视觉文本记录做字符串匹配），二期会升级为多模态模型+工程混合。

识别到"后排乘客擤鼻涕+按太阳穴持续10s"→触发舒适 advisor→建议 Director 调空调/降噪→Director 执行。这就是用户没开口、系统主动关心的体验来源。

11条件触发器（operate_conditional_task）——"10分钟后帮我关座椅加热"的执行者

Director 判断需要创建条件任务时，调这个工具用自然语言下发条件。工具负责解析条件、持续监控，条件满足后通知 Director 二次推理执行。

流程图："十分钟后关座椅加热"怎么走完全程

流程图逐步走读：为什么条件满足后不直接执行，而是回 Director？

这 5 步链路有一个最容易被误解的设计：为什么 ④ 不直接调 vehicle_basic_control 关座椅加热，而是要回 Director 做"二次推理"？

Step ①：Director 判断"10分钟后关座椅加热"是一个条件任务，调用 operate_conditional_task，用自然语言下发条件"十分钟后，关闭主驾座椅加热"。注意是自然语言，不是结构化的 {delay:600, action:"turn_off_seat_heating"}——条件解析是工具的事。
Step ②③：工具回执"条件任务已注册"。内部拆出条件=定时器10分钟，开始倒计时。Director 收到回执后就继续处理别的事了，不会挂起等待。
Step ④⑤（10分钟后）：定时器到点，工具给 Director 下发"[任务原文]+条件已触发"。为什么不直接执行？因为10 分钟后的车况可能已经变了——也许用户已经下车了、也许用户刚才又说了"算了不关了"、也许座椅加热已经被别的指令关掉了。让 Director 拿到"条件已触发"后做一次推理，结合当前最新上下文决定是否仍然执行——这比盲目执行安全得多。
对比两个 case：
case A"10分钟后关座椅加热"→正常case，10分钟后 Director 确认用户还在车上且座椅加热还开着→执行关闭。
case B"世界毁灭时打开空调"→非法条件，合法性校验拦截：能否被感知？不能（没法监测"世界毁灭"）→不注册，Director 转聊天回复："这个我可做不到，但空调现在就能帮你打开。"
合法性校验 5 条（都过了才注册）：① 能否被感知（"世界毁灭"感知不了）② 触发可能性（"温度达到 1000 度"不现实）③ 价值观（"车速 200 以上关闭安全气囊"违反安全）④ 可执行性（"把车门飞出去"做不到）⑤ 条件与动作是否矛盾（"下雨时打开天窗"自相矛盾）。另外，SP tips 里条件任务曾标"还不会"——该功能实际是否已上线需内部确认。

12公共 Context 服务——所有模块共享的"现场与记忆"

Director 推理时知道"车内温度35度、用户叫明骏、上次去过游泳馆"——这些信息不是 Director 自己查的，而是公共 Context 服务提前准备好注入给它的。

骨架图：多来源写入 → 存储+预处理 → 多方读取

骨架图讲解：Context 不是一坨字，而是一个有"预处理管线"的服务

很多人以为 context 就是"把所有信息拼成一个大字符串塞给模型"。错了。公共 Context 是一个独立的服务模块，它做三件事：

第一，按结构存储——13 类内容，每类有明确的来源和优先级。P0（必须有）：情景描述、历史对话、事件日志、端状态（只保留最新）、屏幕截图（近5分钟）、用户记忆（按人物id加载）、任务队列、基础联网。P1：豆包探索、通用任务经验。P2：工具知识、专有知识、性格自我认知。
第二，预处理（这才是价值所在）——不是全量塞给模型，而是先处理：
· 压缩总结：情景描述太长，压缩成要点。
· 打标分类：给每条信息打上类型标签，方便检索。
· 三维过滤：① 按场景——你在停车充电，就不需要加载"行驶中的路况记忆"；② 按人物——车上有两个人，A 的私人行程记忆不给 B 看（隐私红线）；③ 按任务——你在点外卖，就只检索外卖相关的知识偏好，不塞"洗车攻略"进来。
第三，组块注入——处理完的内容分成几个模板变量，注入 SP 末尾的 ⑬ 动态 context 块：{{舱内视觉感知}}（VLM 写回的描述）、{{用户记忆}}（记忆库加载的）、{{动态专业知识}}（前置筛选挑出的知识）。另外 goal_list 和 env_info 不走 Context 块，而是随 user_query 直传——目的是让 Director 能直接读到最新车况和目标，免得每次都调工具查。
这套预处理是 token 优化的真实抓手——不是"远区精简近区完整"（那个在这批PRD里没有），而是"按场景/人物/任务过滤 + 情景压缩"。做得好，Director 拿到的就是干净、相关、紧凑的上下文，推理更快更准，token 消耗更少，KV Cache 也不容易被挤出。
Badcase 排查：① Director 回答时"不知道车况"→查 env_info 是否直传到位，或端状态模块是否上报延迟；② 泄露了同车人的隐私记忆→检查"按人物过滤"是否生效；③ 停车场景答出行车记忆→"按场景过滤"失效；④ token 超标/推理变慢→情景压缩或按需抽取退化成了全量加载。

13前置筛选模块（知识筛选专家）——token 优化最清晰的落点

SP 里的示例、工具描述、knowhow 太多了（50+条），全塞给 Director 会拖慢推理。前置筛选是一个独立的小模型，先跑一遍，只挑当前场景相关的几条知识注入。

流程图："温度骤升"如何触发知识筛选

流程图讲解：一个温度变化怎么让 Director 变得"更聪明"

这是整套架构里token 优化最优雅的闭环。走一遍：

Step ①：筛选专家的 8 个输入字段里有一个特别的——{{env_changes}}（车辆状态变化点，不是绝对值）。这一轮的变化是"车内温度过去5分钟从22度上升到29度"。
Step ②：筛选原则第②条说"趋势比绝对值更有决策价值"——29度的绝对值不算高，但"5分钟升了7度"这个趋势很猛。原则②被触发。
Step ③：哪个专家领域？温度→舒适→COM（随侍/管家）。在 COM 的知识树里，"调温度"的 ID 是 COM-01-01-01。
Step ④：输出增量——{"add_knowledge_ids":["COM-01-01-01"], "remove_knowledge_ids":[], "reasoning":{"add_reason":"env_changes 显示温度骤升，命中 COM 调温知识"}}。注意是增量：只说"加什么""删什么"，不重算全集。
Step ⑤：COM-01-01-01 对应的知识内容（"分区温控按乘员位置独立调节，用户记忆偏好联动，调节后闭环确认"等）被注入到 SP 末尾的 {{动态专业知识}} 块。Director 下一次推理时，就能读到这条知识——它不需要"从零理解空调分区"了，知识已经给它准备好了。

为什么不把所有知识都塞给 Director 让它自己挑？因为token 多→注意力稀释→推理变差（LLM 的已知问题：context 越长性能越降）。加上 KV Cache 有容量上限，塞太多容易被挤出导致缓存失效→更慢。前置筛选用一个轻量小模型先做一遍粗过滤，把 50+ 条知识压到场景相关的 3~5 条，Director 只看这几条——更少的噪音 = 更快更准。
5 条筛选原则要记住：① 场景绑定推理（展厅→EXP，通勤→DRV）② 关注环境变化（趋势比绝对值重要）③ 目标持续关联（目标在则知识保留，别删）④ 增量更新（只输出变化部分）⑤ 信息不足不虚构。第⑤条特别重要——宁可不注入也不编。Badcase：该注入的知识没注入→Director 回答得"不专业"→排查筛选专家有没有拿到完整的输入8字段。

14任务与交互管理——车上为什么只能"一个一个来"

车里的语音交互是单通路管道——同一时间只能回应一次请求。所以多个输入同时到达时，必须在入口处做"交通管制"：谁先说、谁排队、谁被丢弃。

流程图：新输入到达时的决策树

流程图讲解：为什么"首 token 出没出"是分水岭

整套交互管理的核心判据就一个：Director 的推理结果（第一个 token）开始输出了没有？

还没出（推理中）→ 可以打断：Director 还在想，结果还没开始往 TTS 送。这时如果用户说了新的话，旧的推理被打断，新旧 query 合并在一起重新推理。合并后的话术自然串联，比如用户先说"帮我找今天的热点新闻"→(正推理中)→用户又说"先把空调调下，车里热死了"→合并后重推理→"车里是有点热呢，我来调一下，顺便帮你找今天的热点新闻。"——"顺便"就是合并的痕迹。
已经出了（正在播报）→ 分优先级：Director 的话已经在说了（TTS 正在播），这时处理就看新来的是谁：
· P0 用户主动说话——直接打断当前播报，处理新 query，被打断的播报不恢复。因为用户的话最大，系统在说什么都没用户说的重要。
· P1 非用户输入（定时任务触发、advisor 建议）——排队，等当前 TTS 播完再处理。可以"预推理+缓存结果"——在等的过程中先让模型想好答案，播完后直接出结果，省时延。

六种冲突策略（交互层）

当"旧任务正在播报"+"新任务来了"，按优先级矩阵选一种策略：

策略行为例子
① 扔弃新任务新的到了就丢planner正回话，advisor下发→丢弃advisor
② 排队等旧的播完再播新的正在回话，主动服务推送→排队
③ 静默执行新的悄悄做，不打断旧的正在找景点，打开车窗→静默执行开窗
④ 整合(核心)暂停旧的，整合上下文重规划正在规划行程，"再加个点"→暂停重规划
⑤ 插队暂停旧的，新的插入播完，旧的继续播新闻中，"小心右侧行人"→插队播完→继续新闻
⑥ 停止旧任务旧的直接丢弃正开窗，"下个路口左转"→停止开窗指令，处理导航

优先级从高到低：安全通知 > 导航通知 > 普通系统消息 > 配置类主动服务 > 条件任务 > 非配置类主动服务 > 简单query > 复杂query。

合并的三条特殊规则（很容易踩坑）：
① 长程任务（订座/停车/点咖啡/外卖）不参与合并——它们有自己的 GUI 界面，合并会乱；
② aigc（生图/ailight）不走 Planner、无语音播报，完成后走通知卡系统通知；
③ 同一 advisor 排队期间的多条建议，只保留最新一条——防止同一个 advisor 的旧建议堆积。

15任务工程管理模块——打断后怎么恢复、挂起后怎么唤醒

用户说"帮我介绍这台车的功能"（长时任务），说到一半又说"先导航去公司"→被打断了。打断后能不能接着说？靠的就是这个模块。

三个核心机制：打断 → 恢复 → 挂起

打断——端上请求中断接口（携带未完成任务）→ 标记 plan 为 interrupted 并存 checkpoint → 新 query 进来处理。checkpoint 保存的是当前 plan 的状态快照（哪些 task 完成了、哪些没完成、中断在哪一步）。
恢复——新 query 处理完后，由 advisor（goal reviewer）决定是否恢复，不是 Director 自己决定。恢复的动作是：带上下文（含目标的"细节和进展"）重新触发任务。续播规则：已完整缓存的播报→从断点的上一个句子起始处续播；未完整生成的播报→基于上下文重新生成续播。
挂起（定时/条件任务）——定时/条件任务注册后就进入"挂起"状态，Always on 模块持续监听车辆状态，条件满足时通知 Planner 服务，恢复为"进行中"调 Agent 执行。
场景演绎：用户说"介绍这台车"→Director 开始播报功能介绍（长时任务，有目标在 goal list 上）→说到"手势识别可通过挥手…"被用户打断"先导航去公司"→
①任务管理标记 interrupted，存 checkpoint（播报进度="手势识别可通过挥手"）
②Director 处理导航
③导航完成后，advisor（goal reviewer）检查 goal list："介绍车功能"目标还在、状态"进行中"、进展"被打断，播报进度为手势识别"→决定恢复
④advisor 给 Director 建议："刚才的功能介绍被打断了，可以继续"→Director 话术衔接后从"手势识别"上一句续播。
为什么恢复靠 advisor 而不是 Director 自己记住？因为 Director 是无状态的 LLM——每次调用它读的是 SP+UP，它没有"持续记忆"。但 goal list 上的"目标细节和进展"是持久化的，advisor 读它就知道"上次说到哪了"。这就是目标驱动架构的核心价值——恢复不靠模型记忆，靠看板上的结构化进展。
失效兜底：
· 端侧上报超时：白名单召走→采纳端侧结果；端侧有结果→超时采纳；无结果→超时播报兜底。
· 云端工具超时（ReAct）：①工程判断超时，tool_feedback 反馈超时 → ②告知用户后执行重试 → ③重试失败后告知用户结束任务。
· 被端侧召走的 query：云端并行在跑但丢弃云端结果，被合并的任务恢复排队；历史对话保留端侧信息、不保留被打断的云端推理。

16执行层：26 个工具 + HMI 卡片显示——Director 的"手"和"嘴巴"

Director 想好了（action_list），最终要有人去做（工具执行）和有人去告诉用户（卡片+TTS）。这一层就是"做了并告诉用户"。

流程图：action_list → 工具执行 → 成功/失败 → 卡片+TTS

流程图讲解：工具执行 + 失败处理 + 结果呈现

工具执行的基本原则：一个 Agent 只执行一个任务，Director 靠 Function Call 输出 action_list。每个 action 包含 action_id（序号）、params（参数）、tool_name（调哪个工具）。如果多个工具之间有串行依赖（先查记忆拿地址，再导航），不能一次性全输出——注意事项第6条明确禁止。但无依赖的可以并行（如调空调+放音乐）。
成功路径：tool_feedback 回灌给 Director → Director 做二次推理。很多成功场景 Director 选择不说话（talk_or_not:"no"）——车窗关了用户能感受到，音乐响了用户能听到，不必多嘴。
失败路径 → Replan：下游 Agent 无法成功执行 → 端上将未成功执行的原始任务描述 + 具体失败原因回传给 Director → Director 综合失败原因和当前上下文，调 plan_describe 重新生成一个更可行的新计划。注意事项说"不必道歉，想办法处理"——失败不是终点，是 Replan 的起点。

26 个工具的三个易混淆点

易混1：媒体播控归谁？"搜歌/搜视频"用 search_and_control_music(10) / search_and_control_short_video(9)。但"上一首/暂停播放/单曲循环"这种播放控制，归 vehicle_system_settings(2)——不是音乐工具。这是最常见的误调。
易混2：记忆分两类。客观事实（家在哪/年龄/工作/宠物）→ operate_user_memory(13)。交互习惯偏好（"放歌时我要听创作背景"）→ personal_skill_record(14)。后者能 add/update/delete 个性化技能规则。
易混3：4个导航工具不可并行。navi_basic_control(8) / route_planning_qa(7) / search_poi_qa(6) / trip_plan_record(24) 这四个绝不能在同一轮 action_list 里同时出现。即使没有语义依赖也不行——工程限制。
