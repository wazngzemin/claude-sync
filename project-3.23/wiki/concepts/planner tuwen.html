1 它在系统里的位置 — 三层漏斗回顾

豆包上车的语音理解采用漏斗（Funnel）架构：能用简单方法解决的不要漏到后面复杂模型去处理。情景RAG + 仲裁模型共同组成第二层。

第一层：句法RAG（规则匹配） ≤ 500ms · 拒识 / 车书 / 音乐 / 简单车控
匹配不了 ↓
第二层：情景RAG + 仲裁模型（知识注入 + 分流） ≤ 1s · 注入情景/记忆/知识 → 判断走车控还是Planner
判定为"推理" ↓
第三层：Planner（云端 LLM 深度推理） 1-3s · 多步规划 · 多工具调度
核心设计思路（三个）：
(1) 层层过滤：简单方法能搞定的不漏到后面
(2) 分阶段注入知识：从简单（固定）到复杂（泛化），保证流畅稳定
(3) 数据回流：用户数据逐步把复杂任务的处理推到前面知识注入环节

2 情景RAG：做什么、怎么做

定位

情景RAG 不是一个独立做决策的模块，而是知识注入层。它的职责是：在 query 进入 Planner 之前，把模型做决策所需的外部信息（记忆、车况、情景、参考方案）提前注入，让 Planner 看到足够的 context 后再做任务拆解。

句法RAG（第一层）

  • 明确的单轮 query 匹配
  • 不支持上下文多轮
  • 不支持情景描述、记忆信息、车机状态
  • 输出：directExecute（直接执行）
  • 例如："打开车窗" → 直接调用车控

情景RAG（第二层）

  • 模糊的单轮 query，需要推理
  • 给模型注入知识，辅助拆解
  • 注入方式：few-shot、知识图谱、描述、SFT
  • 输出：modelExecute（交给 Planner 处理）
  • 例如："我好热" → 注入车况/记忆 → Planner 决策

注入方式

注入可以是多种形式，形式不重要，前期简单做先做好 few-shot 的聚合：

注入形式 说明 阶段
Few-shot 示例 匹配相似 query，注入参考拆解方案（最核心） 当前已用
知识图谱 结构化的工具/设备/场景知识 规划中
自然语言描述 直接描述 context（车况、记忆） 当前已用
SFT 微调 将知识直接训进模型权重 长期目标

输出格式

{ "type": "modelExecute", // 交给 Planner 模型处理 "knowledge": "注入的情景知识（few-shot 参考方案）", "agentLimit": ["vehicle_basic_control"] // 可选：限制可调用的 agent 范围 }

3 知识注入的四种信息源

情景RAG 注入的信息来自四个维度。每种信息通过不同的 Tool 或规则触发获取：

1. 用户记忆（Memory）
内容：用户画像、偏好、高频控制记录（空调/后视镜设置）、对话历史
触发：Planner 的 search_user_preferences Tool 判断是否需要记忆
调用：Planner 告诉记忆模块"需要查什么"，记忆模块自行匹配返回
示例："去常去的那家咖啡店" → 查记忆 → 返回"常去的咖啡店=星巴克南京西路店"
2. 车辆状态（Env Info）
内容：温度、空调状态、座椅功能、车窗/车门/锁、胎压、车速、充电续航、音量等
触发：Planner 的 search_vehicle_status_info Tool 判断是否需要环境信息
调用：Planner 告诉环境信息模块"需要查什么"，模块自行匹配返回
示例："我好热" → 查车况 → 返回"空调已开，温度27°C"
3. 当前情景（Scenario）
内容：当前行驶场景、乘客状态、时间/天气/位置、视觉感知信息
触发：SP 中的动态模板变量 {{status}} 和 {{env_info}}
注入方式：自动填充到 SP 的动态区域，每次请求都更新
示例："车里好吵" → 注入情景：当前车窗开启，高速行驶80km/h → 建议关车窗
4. 参考方案（KnowHow）
内容：基于知识库匹配的任务拆解参考方案（few-shot）
触发：情景RAG 根据 query 相似度匹配
注入方式：以 knowledge 字段传给 Planner，附带多种条件下的拆解方案
示例："我好热" → 匹配 knowHow → 返回常规/有记忆/有情景三种拆解方案

4 情景RAG 实战示例："我好热"

一个完整的端到端示例，展示情景RAG如何根据不同信息源产生不同的任务拆解：

用户说："我好热"
1 句法RAG 无法匹配 — "我好热"不是明确指令，没有具体的车控对象
2 进入情景RAG — 匹配到"体感温度不适"类 knowHow
3 查记忆 — 发现用户记忆："xx不能吹空调"
4 查车况 — 发现情景："空调已打开，温度设为27°C"
5 注入 knowledge 给 Planner（见下方三种拆解）
A
常规拆解（无额外信息）
Planner 按默认逻辑拆解：
action_list:
① vehicle_basic_control（打开空调设为23°C）
② vehicle_basic_control（打开座椅通风）
B
有记忆的拆解
发现用户不能吹空调，调整方案：
action_list:
① vehicle_basic_control（打开座椅通风）
② vehicle_basic_control（打开车窗）
⚠ 跳过空调，避免触发用户身体不适
C
有情景的拆解
发现空调已开且27°C，微调即可：
action_list:
① vehicle_basic_control（空调温度调到22°C）
✓ 不需要"打开空调"，只需降温
关键洞察：同一句 "我好热"，因为注入的 context 不同（无额外信息 / 有记忆 / 有车况），Planner 产出完全不同的执行方案。这就是情景RAG的核心价值 — 让 Planner 从"猜"变成"知道后再做"。

5 仲裁模型：车控 vs 推理

核心职责

仲裁模型负责分流判断：每条用户 query 到底应该走车控（简单）还是推理（复杂/Planner）。

Seed 1.6
基础模型（Flash 版本）
SFT
训练方式
PE + SP
核心控制方式
2分类
车控 or 推理

判断标准

重要区分：仲裁模型的判断标准是句式的简单和复杂，并非通过意图区分。所以 Face ID、记忆增删改查等需要单独由意图模块处理的问题，需要特殊设计。长期方案中，需要对问题进行分类：意图分类 → 意图模块处理，简单/复杂分类 → 仲裁模型处理。

仲裁模型的 SP 结构

仲裁模型使用一个完整的 System Prompt 来定义什么是"车控"、什么是"推理"：

sp = """你的任务是判断用户的指令应使用车控模型还是推理模型。 车控模型用于处理和执行较为明确的指令。 推理模型用于处理和执行较为复杂的指令、模糊的指令、带条件的指令， 或者依赖用户记忆信息、当前情景、联网搜索信息的指令， 需要通过推理得出用户的真实意图。 <车控模型介绍开始> // 一、车控指令场景（明确/反问/否定/单实体/不完整/上下文继承） // 二、媒体控制场景（明确/反问/否定/不完整/上下文继承/多媒体搜索） // 三、导航指令场景（普通导航/设置/查询） // 四、组合指令场景（以上三类简单指令的组合） <车控模型介绍结束> <推理模型介绍开始> // 一、车控指令场景（模糊/带触发条件/表达感受/需要额外信息推理/其他） // 二、媒体控制场景（模糊/带触发条件/表达感受/需要额外信息推理/其他） // 三、导航指令场景（带触发条件/需要额外信息推理） // 四、组合指令场景（包含任意一种复杂指令的组合） <推理模型介绍结束> 要求：请根据上下文，判断用户最后一轮的指令属于车控还是推理。 如果是推理，直接说：推理；如果是车控，直接说：车控。不要输出其他内容。"""

6 仲裁 SP 的四版演进

仲裁模型的 System Prompt 经历了 4 个版本的迭代，每版解决不同的分类错误：

V0.1 — TOB 原版 SP
直接复用 TOB 简单复杂仲裁模型
问题：条件任务（"十分钟后提醒我下高速"）、记忆任务（"导航去上次那家川菜馆"）大量被错分为"车控"
车控定义：只覆盖明确/反问/否定/单实体/不完整/上下文继承
推理定义：只覆盖模糊指令/表达感受/需要额外信息推理
V0.2 — 产品 12.24 更新 · Seed 1.6 Flash 基模
新增：
• 推理侧增加"带触发条件的指令"类别
• 推理侧增加"基于用户记忆和偏好推理"
• 车控侧增加"组合指令场景"（简单+简单=简单）
• 推理侧增加"组合指令场景"（含复杂=推理）
改善：条件任务、记忆任务大部分修复；但"导航到我孩子的学校"、"导航到爸妈家"仍误分
V0.3 — 产品 12.30 更新 · Seed 1.6 Flash 基模
新增：
• 推理侧增加"基于当前情景推理"（如"车里好吵调安静点"）
• 推理侧增加"基于联网信息推理"（如"导航去上海最高的建筑"）
• 导航侧增加"基于用户记忆和偏好推理"独立分类
改善：绝大多数 case 已修复；个别仍有问题（"我老婆那边的出风口" 1230 回退为车控）
新问题："来点歌听听"等简单 query 被过度召回为推理
V0.4 — 算法 01.06 更新 · SFT 精调模型
训练方式从 PE → SFT 精调
• 不再依赖 Prompt Engineering，直接在 Seed 1.6 Flash 上做 SFT
• 新增"任务操作（模糊）"类别：取消/删除所有任务

7 车控模型 vs 推理模型 — 判断维度全景

以 V0.3（12.30 版本）为基准，仲裁模型分别在三个场景域中定义了车控和推理的判断规则：

一、车控指令场景

→ 车控模型

  • 明确指令：开一下空调、调低音量
  • 反问表达：能不能给窗开一下
  • 否定表达：不用给我开窗了
  • 单实体：车窗、大钟寺2号楼
  • 不完整表达：调高一档、向左调一点
  • 上下文继承：历史"座椅加热几档"→ 当前"换3档"

→ 推理模型

  • 模糊指令：打开头顶上的布、给我捏捏背
  • 带触发条件：十分钟后打开座椅加热
  • 表达感受：声音怎么那么轻啊、我腿伸不开了
  • 需要位置推理：开我女朋友那边的空调
  • 需要记忆推理：把座椅调到我喜欢的档位
  • 需要情景推理：车里好吵调安静点

二、媒体控制场景

→ 车控模型

  • 明确指令：暂停播放、继续播放
  • 多媒体搜索：基于语种/榜单/年代/风格/标签搜索
  • 上下文继承：正在播歌 → "调大一点"

→ 推理模型

  • 模糊指令：放点好听的、来点音乐
  • 带触发条件：车里人睡着后把音乐关了
  • 基于歌词/影视剧/场景搜索："风吹啊吹啊是什么歌"
  • 基于记忆推理：播放我喜欢的歌手的歌
  • 基于情景推理：找点符合车外风景氛围的歌

三、导航指令场景

→ 车控模型

  • 普通导航：导航到xxx、去xxx（明确地址/家/公司）
  • 导航设置：避开高速、避开收费
  • 导航查询：还有多远、剩余时间

→ 推理模型

  • 带触发条件：早上上车后导航去公司
  • 路线体验推理：前面堵死了，有没有别的路
  • 基于记忆推理：去我女儿辅导班的地方
  • 基于联网推理：导航去上海最高的建筑

四、组合指令场景（关键判断规则）

→ 车控模型

  • 简单 + 简单 = 车控
  • 打开空调并打开方向盘加热
  • 导航回家顺便播放林俊杰的小酒窝
  • 播放音乐并把音量调小一点

→ 推理模型

  • 含任何一种复杂指令 = 推理
  • 播放林俊杰并导航到爸妈家（爸妈家=记忆）
  • 导航去公司，顺便去常去的包子店（常去=记忆）
  • 打开空调，顺便找附近有什么好吃的（联网搜索）

8 8 类问题分类 — 仲裁必须覆盖的场景

通过实际评测和 badcase 分析，归纳出 8 大类需要仲裁模型正确处理的问题。这是仲裁升级的核心需求清单：

1
条件/定时任务
带触发条件的指令，不是即时执行，需要 Planner 创建 Goal 追踪。
"十分钟后帮我关掉座椅加热"
"副驾上车后给她唱小星星"
"车内温度超过28度之后打开空调"
"把那个关座椅加热的任务删了吧"
归属：仲裁 需区分增删改查
2
持续任务
持续运行的长期任务，需要 Planner 持续管理状态。
"接下来的行程里给我介绍沿途的风景"
"讲个笑话逗到我笑为止"
"5星以下的就别讲了，只介绍5星的吧"
"不用再介绍风景了"
归属：仲裁 需区分增删改查
3
Face ID 操作
人脸识别相关的注册/修改/查询，涉及人名和座位映射。
"认识一下坐在副驾的明骏，他是我的朋友"
"白宇是我的好朋友，不是我同事"
"你知道副驾是谁么？"
归属：意图 需增加 Face ID 增删改查意图
4
记忆增删改查
用户主动操作记忆的存储/修改/删除/查询。
"帮我记一下我公司地址是漕河泾中心D栋"
"把我妈妈的手机号改成13999999999"
"删掉我前女友相关的记忆"
"我上次和你说的很好吃的那家店叫啥来着"
归属：意图 需增加记忆增删改查意图
5
需要记忆推理
Query 本身看似简单导航/车控，但实际需要调用记忆才能执行。
"去常去的那家咖啡店"
"导航到爸妈家"
"播放我上次没听完的有声书"
"把氛围灯调到我喜欢的颜色"
归属：仲裁 TOB 原版全部误分为车控
6
需要情景推理
需要结合当前车况、环境状态才能正确理解的模糊指令。
"车里好吵啊，调安静点"
"外面好臭，帮我处理一下"
"后排有小孩，帮我把相关设置打开"
归属：仲裁 短期方案未覆盖
7
需要联网推理
需要联网搜索才能获取关键信息后再执行。
"导航去上海最高的建筑附近的停车场"
"播放我是歌手2024夺冠的那首歌"
"我想看沈腾最新演的电影"
归属：仲裁 需分到推理
8
组合指令（含复杂）
组合指令中包含任何一种复杂指令，整体应判为推理。
"播放音乐，导航去常去的那家咖啡店"
"导航去爸妈家，顺便给我放点周董的歌听"
"放个我喜欢的音乐再导航去洗车店"
归属：仲裁 关键判断：含复杂=推理
意图 vs 仲裁的分工：
• Face ID + 记忆增删改查 → 由意图模块处理（需要增加对应意图）
• 其他 6 类 → 由仲裁模型处理（通过 SP 升级 + SFT 训练）

11 一页总结

情景RAG 是什么

  • • 漏斗第二层的知识注入层
  • • 在 query 进 Planner 前注入记忆、车况、情景、参考方案
  • • 核心注入方式：few-shot 匹配（当前）→ 知识图谱/SFT（长期）
  • • 让 Planner 从"猜"变成"知道后再做"
  • • 通过 search_user_preferences 和 search_vehicle_status_info Tool 触发信息获取

仲裁模型 是什么

  • • 漏斗第二层的分流判断器
  • • 基于 Seed 1.6 Flash + SFT，二分类：车控 or 推理
  • • SP 定义了三大场景域 × 车控/推理的判断规则
  • • 经历 4 版 SP 迭代（TOB → 12.24 → 12.30 → 01.06 SFT）
  • • 必须覆盖 8 类问题场景，错误率从 ~40% 降至 ~8%
  • • 长期：句式+意图联合判断，统一由仲裁决定是否进 Planner
