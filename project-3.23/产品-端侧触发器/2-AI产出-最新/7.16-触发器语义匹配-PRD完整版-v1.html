VLM 条件任务的"判断"环节：链路、协议、判定策略与落地分工 · 评审稿
v1 · 2026-07-16 · 产品：王泽民

一、背景与目标

用三句话说清楚这个需求为什么存在：

  1. 任务系统（Task）落地后，用户可以说"后排宝宝哭了告诉我""有人抽烟提醒我"这类持续观察类需求（VLM 条件任务）——系统要一直盯着，条件到了才反馈。
  2. 负责"盯"的端侧 VLM（视觉大模型）只输出画面描述、不下结论：它会说"后排儿童睁眼，脸上有泪珠"，但不会说"哭了"。因为它是通用感知，不预知每个用户会提什么任务。
  3. 因此链路上必须有一个环节，把"描述"判成"条件到了没到"——这个判断环节（本文叫"语义匹配 / 模型解析"）就是本 PRD 要定义的东西。它是所有 VLM 条件任务的公共底座：这一环不落地，"宝宝哭了告诉我"这类需求一个都跑不起来。
本期目标：① 定义判断环节的归属、输入输出与判定策略；② 定义与 VLM、Task Service（任务服务）之间的完整协议；③ 产出高频场景标签清单与种子数据构建方案，支撑 0818 首批 VLM 条件任务上线。

名词含义

callback（回调）判断命中后，触发器通知 Task Service 的消息

边沿触发只在状态发生变化的那一刻触发（未达成→达成），连续处于"达成"不重复触发。触发引擎现有"变更为"运算符的机制

四、用户场景

4.1 主场景（贯穿全文）："后排宝宝哭了告诉我"

驾驶员独自带娃，想专心开车又怕照顾不到。说一句话，系统持续观察后排，宝宝一哭立刻播报提醒。VLM 每 5 秒的观察可能是：

第 1 条："后排儿童安静，闭眼休息" → 没哭
第 2 条："后排儿童睁眼，面部平静" → 没哭
第 37 条："后排儿童睁眼，脸上有泪珠，嘴角向下" → 判断环节判定：哭了 → 触发播报

4.2 场景全集

完整场景清单见附件《7.16-高频场景标签清单-初稿-v2.xlsx》：源自《视觉标签需求-触发器评估》31 条逐条整理 + 增补候选 Z1–Z6（吸烟 / 安全座椅误装 / 儿童滞留 / 晕车 / 后排争吵 / 长尾类）。结论摘要：

  • 31 条需求按"状态标签 + 变更为事件"收敛为 约 15 个状态标签（睡眠 / 哭闹 / 玩手机 / 通话 / 分心 / 疲劳 / 小狗存在 / 遗留物 / 舱外景观等）；
  • caption（画面描述）支撑逐条核对：✅ 10 条现有描述可支撑、⚠️ 10 条待算法确认、❌ 6 条现有输出无相关描述（吸烟 / OK 手势 / 舱外景观 / 红灯 / 景色优美 / 拥堵——每条三个出路：增强 prompt 提示词 / 转 B 语义 / 主走非视觉信号）。

五、整体方案：端到端链路（三阶段 11 步）

对照架构图从左到右。模块名与图一致：端侧 = AIBox（任务优先级管理 + 端侧感知模块 VLM）、视频流处理、AI Service 消息处理模块（下行）、AI Service 数据中心（上行）；云侧 = Task Service（任务服务，含动态感知模块）、云端触发器（现有规则服务的能力扩展，非新建模块）、云端 Context（上下文）、Director（对话推理引擎）。

阶段 C · 两层判断与反馈（云）

步骤 9 ·【云端触发器】判断① 条件命中 —— 本 PRD 核心输入=识别结果（"睁眼，脸上有泪珠"）+ 感知目标（"二排的宝宝哭了吗？"）。A 标签路线=比对枚举值；B 语义路线=轻量模型判断（见第六章）。没命中 ↺ 等下一条 5 秒上报再判；命中 → 边沿去重后 callback（回调），带 task_id（任务ID）+ callback_id（回调ID）+ triggered_at（触发时刻）
↓ callback
步骤 10 ·【Task Service】判断② 任务达成 —— 生命周期决策收到 callback（触发器已判过条件，不做二次条件匹配）→ 按 EndPolicy（结束策略）决定：播报一次即关（close_after_success）/ 持续任务继续挂着；并按用户反馈周期控制播报节奏（如"每 60 秒告诉我一次"）
↓
步骤 11 ·【云→用户】播报 + 收尾Director（对话推理引擎）执行播报"后排宝宝哭了"。任务关闭后：注销触发器规则 + 通知 VLM 结束高频任务UP（任务编号 8），端侧停止观察
★ 本方案最重要的一条边界——"判断"有两层，各归各：
判断① 条件命中判断② 任务达成
判什么这条识别结果符不符合感知目标任务要不要结束、什么时候反馈用户
频率每 5 秒一次，循环命中后一次
归属云端触发器（规则服务）Task Service
为什么这么分：①《Task 任务管理设计》技术方案原文："规则服务（现有 Trigger）管理延时、条件、周期和 VLM 语义匹配，命中后产生 callback""Trigger 完成语义匹配并判定命中后 callback 到 Task，Task 不负责二次条件匹配""Context 不做 VLM 目标匹配"；②第一性原理：判断应发生在存着判断标准的地方——感知目标注册时就存进触发器规则库；③工程复用：边沿去重、one_shot、执行策略都是触发引擎现成能力，放别处需要重造。

六、判断①怎么实现：A / B 两条路线，分层共存

6.1 两条路线对比（同一句"宝宝哭了"，两种数据形态）

路线 A · 标签比对路线 B · 语义匹配
VLM 给什么结论选项（枚举值）：
{"哭闹状态":"无"} → {"哭闹状态":"哭闹"}		自由文字描述：
"后排儿童睁眼，脸上有泪珠，嘴角向下"
判断动作字符串比对：来的值 == "哭闹"？与现有"档位 等于 P"同机制，零成本零误差轻量模型读文字判含义："这段描述符合'宝宝哭了'吗？"答 是/否
判断的智能在哪端VLM 端（打标签时已判完）触发器端（模型现场判）
覆盖仅预定义过的标签场景用户任意说的长尾需求
成本零模型调用每条约百级 token（词元）的轻量模型调用
短板新场景要预置标签+种子数据，不灵活存在误判样例（badcase），准确率需打磨

6.2 分层原则（怎么定一个场景走 A 还是 B——三问判断器）

顺序问题不过 → 走 B
第 1 问这个事能收敛成有限选项吗？收敛不了（"景色优美"主观开放）→ B
第 2 问高频吗？值得预置标签+种子数据的成本吗？长尾低频（"毯子滑落"）→ B
第 3 问VLM 的 caption 描述范围撑得住吗？没相关描述且改 prompt 不划算 → B
三问全过 → A 标签附件 xlsx 每行"建议路线"列即按此生成
为什么两条共存而不是二选一：全走 A——用户随口说的话永远预置不完；全走 B——高频场景每 5 秒烧模型、还有误判率，明明枚举比对又准又免费。A 管头部（降成本提准确），B 兜长尾（保覆盖）；判断器入口统一，按注册单形态自动分流，对上下游透明。分层本身就是成本方案（见第十章）。

6.3 状态标签 + 引擎抓变化（"开始/结束"不做独立标签）

原始需求里的"开始哭闹 / 结束哭闹 / 出现小狗 / 移除小狗"，不为每个变化单独做标签：VLM 只输出状态（哭闹/无、有/无），"开始/结束"由触发引擎用事件运算符"变更为"抓跨帧变化。瞬时动作（OK 手势 / 打哈欠 / 手伸出窗外）例外：检出即报。时长条件（"红灯超 90 秒""长时间分心"）由引擎计时，VLM 不判时长。

为什么：VLM 单帧无记忆，判"开始"需要跨帧比对——而跨帧变化检测正是触发引擎本职（现有条件 8 运算符 + 事件 2 运算符"变更为/有变更"）。VLM 的活最简（每帧报状态），变化逻辑全在引擎，31 条原始需求由此收敛成约 15 个状态标签。

七、判定策略（B 路线的产品口径）

策略内容
三态输出达成 / 未达成 / 无法判断（目标超出视觉可判范围时显式暴露，如"宝宝发烧了吗"），并输出置信度与依据（引用识别结果关键原文——供播报话术与 badcase 排查）
从严原则宁漏判，不误判。证据不足（"揉眼睛，表情不明"）一律判未达成，等下一帧更明确的描述
边沿去重判断结果视为布尔状态，引擎抓"未达成 → 达成"的边沿才触发（与"变更为"同构）；配合 execute_policy 的 one_shot（单次）策略，连续 10 帧"达成"只回调 1 次
位置约束感知目标带位置时（"二排右侧"），只依据对应位置（"副驾后方"）的观察对象判断；位置映射规范见 8.5
空值约定VLM 未识别到 / 无异常时输出空串，空串 = 未达成 待丁彬确认
为什么宁漏判不误判：代价不对称——误判 = 车里凭空播报"宝宝哭了"，打扰用户并摧毁信任；漏判 = 晚 5 秒（下一帧再判）。判定策略已落成可运行的判断模型 SP（系统提示词）初稿，含 12 条测试用例：见附件《7.16-触发器语义匹配SP-v1.md》。

八、接口协议（四条消息 + 一个转写规范，全靠"任务编号"串联）

8.1 注册：Task Service → 云端触发器（任务创建时一次）

复用现有场景注册协议三件套（conditions / actions / execute_policy），Trigger Adapter（触发器适配器）负责转换并补齐 callback、scope（范围）、source（来源）等传输字段——与手工在配置平台"新建场景"结构一致，只是程序化生成。

字段B 语义路线示例A 标签路线示例

8.5 感知目标转写规范（Spec Builder 规格构建器职责，新增待确认）

规则内容与示例
① 劈条件 / 动作原话"宝宝哭了告诉我" → 条件"宝宝哭了"进感知目标；动作"告诉我"进 execute_text。
反例（不劈的后果）："哭了先别吵她、睡熟了再告诉我"直接喂判断必乱
② 疑问句化"宝宝哭了" → "二排的宝宝哭了吗？"（正向疑问，避免"没系安全带吗"这类否定反转歧义）
③ 位置标准化口语位置映射到 VLM 位置枚举：二排左侧→主驾后方｜二排右侧→副驾后方｜后排中间→后方正中
④ 时长与频率剥离"超过一分钟""每 60 秒"不进感知目标——时长归引擎计时，反馈频率归 Task Service

8.6 两个周期解耦

周期示例值含义
VLM 上报周期5 秒给判断环节喂素材的频率（VLM 侧定）
用户反馈周期60 秒打扰用户的频率（用户 query 定，Task Service 控制）
为什么解耦：按 5 秒反馈=骚扰用户；按 60 秒判断=漏事件。判断密、反馈疏，各自独立设定。
