开场白（30秒，先把结论扔出来）

大家好，今天要对一件事：高德auto SDK的围栏服务确认下架了。上汽高德地图团队正式回复的，不是"部分下架"了，是我们要用的那个围栏能力确实没有了。

上次6.25的会上，我们讨论的方向是走端侧高德SDK围栏——就是我给SDK一个坐标一个半径，它告诉我车进没进这个圈。现在这条路走不通了。

但是上次会上俊龙确认了一件事——高德SDK虽然没有围栏了，但是有一个"查到某个POI的距离"的接口。你传一个POI给它，它能返回当前车的位置到这个POI有多远。

所以我的方案是：我们自己在端侧建一个围栏判定模块。距离数据从高德SDK查，判定逻辑我们自己写。说白了就是——高德给距离，我们自己判进出。

我现在把整个流程从头到尾走一遍，大家看看有没有问题。
6.25会上的坑：你开场时说的是"需要导航给到端侧的信号库，可能是一块是需要给到的，我理解是需要给经纬度坐标的...我不知道我有没有讲清楚这条逻辑"

你自己都说了"不知道有没有讲清楚"，那铁定没讲清楚。

纠正：开场30秒内必须说清三件事——①发生了什么（围栏下架）②我们怎么办（自建围栏）③核心依赖（高德距离查询接口还在）。不要先铺背景，先扔结论。

排除云端方案（1分钟，先把弯路堵死）

在讲具体方案之前，先说一条我们不走的路——把围栏判定放在云端做。

什么意思呢？就是说端侧把车的GPS坐标不停地传到云端，云端来算你离外婆家多远，云端来判断你是不是进了围栏。

这条路上次会上仁杰分析过，两个问题：

第一个，坐标出车。你要把车的实时位置一直往云上传，这个涉及到数据合规。车的位置属于敏感信息，要出车就得过数据安全审批，这个流程很长而且不一定过得了。仁杰原话："坐标出车，合规可能会麻烦一点。"

第二个，频率问题。围栏要实时，你就得高频上报——每秒传一次GPS坐标到云端。这意味着你得建一条专门的高频上报链路，云端还得扛住这个量。帅哥的原话是："难度就指数级的了。"

所以云端围栏方案排掉。我们走端侧。端侧判定，坐标不出车，只有到了的时候才通知云端一次。
帅哥可能会说：我上次说的是"做不了"啊，你现在又要在端侧自己建，和我说的不矛盾吗？
你这样接：

"帅哥，你上次说的'做不了'，是在说当时认为高德SDK有围栏，所以我们没必要自己造轮子。那个判断在当时的语境下是对的。但现在情况变了——SDK围栏下架了，轮子没了。而且仁杰上次也说了，围栏逻辑本身其实挺简单的——'给他一个坐标，创建一个围栏，告诉他半径是多少，以一定频率查当前位置在不在围栏内'——仁杰原话。我们现在做的就是把这段逻辑自己写出来。"

1用户说话

用户在车里说了一句："快到外婆家的时候提醒我给她打个电话。"

这就是我们的输入。注意这句话里有三个关键信息："快到外婆家"是条件，"提醒我打电话"是动作，"的时候"说明这不是现在就要干的事，是未来满足条件时才干。

23ASR转文字 → 仲裁判定

端侧的麦克风收到语音，ASR把语音转成文字，通过L2链路传到云端。

云端收到文字后第一件事是仲裁——这句话应该走哪条链路？是直接车控（"开车窗"）？还是闲聊？还是条件触发？

仲裁模型看到"的时候"这个标志词，判定：这不是让我现在就干什么，这是一句条件话——用户在设定一个"以后满足XX条件时再干XX事"的任务。走条件触发链路。
// L2 端→云 上传的内容 { text: "快到外婆家的时候提醒我给她打个电话", speaker_name: "王泽民", timestamp: "2026-06-26T14:30:00Z" } // 仲裁结果 → 判定为"条件话" → 走条件触发链路（不是车控、不是闲聊）

4意图提槽：把自然语言拆成结构化数据

判定是条件话之后，意图解析模型对这句话"提槽"——就是把一句自然语言拆成机器能理解的结构化字段。

拆出来三个东西：

条件（condition）：什么时候触发？——"快到外婆家"，翻译成机器语言就是：类型是approach（靠近），目标POI是"外婆家"，阈值是500米（默认值）。

动作（action）：触发了干什么？——"提醒我打电话"，翻译成：播报一段提醒话术。

策略（policy）：触发几次？——用户没说"每次都提醒"，所以默认是one_shot，只提醒一次。
// 提槽输出 { condition: { type: "approach", // 靠近 poi: "外婆家", // 目标POI（自然语言，还不是坐标） threshold: 500 // 默认500米 }, action: { type: "talk", // 播报 content: "提醒打电话" // 要干的事 }, policy: "one_shot" // 只触发一次 }
注意：这时候"外婆家"还只是一个文字，不是坐标。系统不知道外婆家在哪。所以下一步得去搜。

567搜POI → 选点 → 拿到经纬度

系统现在知道用户说的是"外婆家"，但"外婆家"在哪？

所以系统要调AINAVI的POI搜索接口，拿"外婆家"这三个字去搜。搜出来可能有好几个结果：外婆家徐汇店、外婆家静安店、外婆家虹桥店……

如果只搜到1个——直接用，不用问用户。

如果搜到多个——在车机屏幕上出一张选点卡片，列出来几个结果让用户选："您说的是哪个外婆家？"用户点了第2个，系统就确认了。

用户选完之后，我们拿到了两个关键数据：
- 经纬度：(121.4737, 31.2304)——这是外婆家徐汇店的准确坐标
- POI ID：高德体系里这个店铺的唯一标识

上次会上仁杰说的很清楚："出个卡片，让他去选一个点，然后如果用户点了，确认是第几个，然后再把这POI ID去建围栏。"必须有这个选点过程，系统不能替用户选。
// AINAVI搜索返回 [ {poi_id: "B001234", name: "外婆家(徐汇店)", lat: 31.2304, lng: 121.4737, distance: "12.3km"}, {poi_id: "B005678", name: "外婆家(静安店)", lat: 31.2290, lng: 121.4485, distance: "8.7km"}, {poi_id: "B009012", name: "外婆家(虹桥店)", lat: 31.1944, lng: 121.3367, distance: "15.1km"} ] // 用户选了第1个 → 确认结果 → poi_id: "B001234" → center: [121.4737, 31.2304] → name: "外婆家(徐汇店)"
6.25会上的坑：汪帅问"外婆家是目的地还是途经点还是什么都不是？"你说"外婆家是途经点，外婆家是，我理解外婆家是一个途经点吧"，然后被刘杨和陈俊龙纠正。

用自建围栏方案后，这个问题不存在了。你应该说："外婆家是目的地、途经点还是什么都不是——在我们的方案里不影响。因为我们不靠导航引擎来算距离了。不管用户在不在导航、外婆家在不在路线上，只要选完点拿到经纬度，我们端侧就能开始盯着。这就是自建围栏比'导航剩余距离'方案好的地方。"
宫丽萍可能问：这个搜索是端侧搜还是云端搜？
"AINAVI这个搜索能力是端侧的地图SDK提供的，搜索请求在端侧发起，返回结果也在端侧。整个选点过程不经过云端。选完之后，经纬度和POI ID会跟着触发器注册请求一起传到云端。"
施佳杰可能问：如果用户说的是"外婆家"但没有收藏过、也搜不到呢？
"如果搜不到，说明'外婆家'这个名字在高德POI库里匹配不上。这时候系统会反问用户：'我没找到外婆家，能说一下具体地址吗？'或者从用户记忆里找——比如用户之前去过一个地方叫'外婆家'，记忆里有坐标，直接用。总之最终一定要拿到一个确定的经纬度才能往下走，拿不到就停在这一步。"

8云端注册规则

选完点了，现在系统手上有了全部信息：条件是什么（靠近哪个点、多近算靠近）、动作是什么（说什么话）、策略是什么（触发几次）。

这些信息打包成一条DSL规则，发到云端触发器引擎，注册入库。

什么是DSL呢？就是一个机器能读的条件表达式。翻译成人话就是：当收到一个类型是fence_enter、目标POI是外婆家徐汇店的事件时，通过Callback通知动态Advisor。
// 注册到触发器引擎的DSL规则 { rule_id: "r_20260626_001", condition_dsl: "event_type eq 'fence_enter' AND poi_id eq 'B001234'", action: { type: "callback", target: "dynamic_advisor", payload: { scene: "approach_remind", poi_name: "外婆家(徐汇店)", user_action: "提醒打电话" } }, policy: "one_shot", status: "active" }

910下发围栏指令 → 端侧创建围栏

规则在云端注册好了之后，云端要通知端侧：你帮我盯着这个点。

云端通过下行通道给端侧发一条围栏创建指令——告诉端侧三个核心信息：盯着什么坐标、半径多大、围栏ID是什么。

端侧收到指令后做两件事：
第一，把这个围栏信息存到本地——坐标、半径、围栏ID、POI名字，都写到本地存储里。这样万一车机重启了，还能恢复。
第二，启动一个定时器，每秒触发一次，开始持续查距离。

到这里，注册段就完成了。整个过程大概10秒。从用户说完话到围栏建好、开始监听——10秒。然后系统就在后台默默盯着了，用户该干嘛干嘛。
// 云端→端侧 下发的围栏指令 { cmd: "create_fence", fence_id: "f_001", center: [121.4737, 31.2304], // 外婆家(徐汇店) 的经纬度 radius: 500, // 半径500米 poi_id: "B001234", poi_name: "外婆家(徐汇店)", policy: "one_shot" } // 端侧本地存储的围栏记录 { fence_id: "f_001", center: [121.4737, 31.2304], radius: 500, state: "远", // 初始状态 locked: false, // 未锁定 created_at: "2026-06-26T14:30:10Z" }
帅哥/仁杰可能问：云端给端侧下发指令这条通道，走的是什么链路？
"这个要跟帅哥和仁杰确认——是复用现有的端→云Context下行通道，还是需要单独建一条。从数据量来说，这条消息很轻，就是一个JSON，几十个字节，一次性的，不是高频。所以我倾向于复用现有的下行链路就行。"
Phase A讲完了。回顾一下：
用户说了句话 → ASR转文字 → 云端仲裁判定是条件话 → 提槽拆出条件/动作/策略 → 搜POI → 选点拿到经纬度 → 云端注册规则 → 下发指令给端侧 → 端侧本地建围栏 → 启动定时器。

10步，10秒完成。现在端侧在后台盯着这个点了。

⑪定时器每秒触发一次

Phase A最后我们启动了一个定时器。这个定时器每秒触发一次——就像一个闹钟，每秒响一下，响了就去干活。

干什么活呢？去问高德SDK：我现在离外婆家多远？

⑫调高德SDK查距离

每次定时器触发，端侧就调一次高德SDK的POI距离查询接口。

这个接口是俊龙在6.25会上确认过的。他原话说的是："高德SDK可以提供，你当前自车位到你选的POI点，它的一个直线距离，或者是它的粗略的一个路线距离。"另外他还说过："我直接从POI的详情信息去请求，它是有一个当前距离的，还有一个预计到达时间。"

调用方式：端侧传入POI ID（就是Phase A拿到的那个B001234），高德SDK返回当前距离。

注意几个细节：
- 这个是端侧内部调用——端侧直接调本地的高德SDK，不经过网络，不出车。坐标不出车。
- 返回的距离可能是直线距离（两点之间直线距离）或者粗略路线距离（沿路大概有多远），具体返回哪个要和俊龙确认。
- 如果正在导航并且外婆家在导航路线上，可以优先用导航引擎输出的沿路线剩余距离，更精准。
// 端侧调用高德SDK request: queryPOIDistance(poi_id: "B001234") response: { distance: 15000, unit: "m", type: "direct_line" } // 下一秒 request: queryPOIDistance(poi_id: "B001234") response: { distance: 14800, unit: "m", type: "direct_line" } // ……每秒一次，距离逐渐变小
仁杰可能问：每秒调一次，SDK扛得住吗？有没有频率限制？
"这个要和俊龙确认具体的QPS限制。如果每秒一次太频繁，我们有两个降级选项：

降级1：把频率降到每3秒或每5秒查一次。判定逻辑不变，只是可能晚3~5秒触发。对于500米阈值来说，车速60km/h的话1秒走17米，5秒走80多米——差距在可接受范围内，因为用户说的是'快到的时候'，不是说精确到某一秒。

降级2：如果SDK距离查询接口不好用或者性能有问题，端侧直接用GPS坐标自己算。端侧本身有GPS定位能力，能拿到当前经纬度(lat, lng)，然后用Haversine公式算到目标点的直线距离。精度差一点，但对'500米范围内'的判定完全够用。仁杰上次说的'人肉实现围栏逻辑'就是这个。"
帅哥可能问：能不能不用轮询，让高德SDK主动推送距离变化？
"帅哥你上次会上提了这个——'能订阅吗？订阅距离某一个POI的距离。'

俊龙当时说的是'可能得看一下能不能支持，我们之前做的只是一个由你们来请求这个接口'。所以目前是被动查询式的，我们主动去调接口，不是SDK主动push。

如果后续俊龙那边能确认支持订阅模式——就是我注册一个POI，高德SDK在距离变化时主动回调通知我——那就更好了，我们连轮询都不用做了。但当前方案先按轮询设计，订阅能力作为优化项。"

⑬⑭拿到距离 → 状态机判定（每秒都在跑，但大部分时候什么都不干）

拿到距离d之后，状态机做一次判定。判定逻辑是五步检查，我一步步说：

第一步：有没有活跃围栏？
查本地围栏列表，如果没有围栏就直接跳过，什么都不干。有围栏才继续。

第二步：当前围栏的状态是什么？
围栏有四个状态：远、已接近、已到达、已离开。刚创建时状态是"远"。

第三步：距离d跟阈值比，看看要不要切状态。
这一步要分当前状态讨论——

如果当前是"远"：
- d > 520米 → 还在远处，什么都不干，等下一秒。
- d 在 480~520之间 → 进了死区，也不干事。为什么？因为这40米的缓冲带是防GPS抖动的。如果阈值是一根线比如500米，GPS在498和502之间跳，就会反复触发"到了→没到→到了→没到"。死区把这个问题解决了——必须穿过480这根线才算真的到了。
- d ≤ 480米 → 穿越了！状态从"远"翻成"已接近"，产出一个fence_enter事件。

如果当前是"已接近"：
- d ≤ 50米 且 档位是P（停车了） → 状态翻成"已到达"，产出fence_arrive事件。为什么要加P档条件？因为高架上你可能直线距离不到50米但其实你在高架上面飞过去，加P档是确认你真的停下来了。
- d > 520米 → 你又走远了（比如你绕路了），状态翻回"远"。

如果当前是"已到达"：
- d > 520米 → 你离开了，状态翻成"已离开"，产出fence_leave事件。

第四步：如果产出了事件，检查one_shot锁。
这个围栏是one_shot策略，只能触发一次。如果已经触发过了，locked=true，哪怕满足条件也不产出事件。

第五步：如果确实产出了事件且没被锁住，上报给云端。
99%的时间，状态机什么都不干。
用户从家出发到外婆家，可能开了半小时。在这半小时里定时器触发了1800次，状态机跑了1800次——但前1790次都是d>520，状态是"远"，什么都不干。只有最后那几次，d穿过480那根线的时候，才真正干活。这个设计就是为了保证：平时零开销，关键时刻不遗漏。

举个具体的例子——用户从15公里外开过来

我用一个时间线走一遍，大家看一下状态机在每个时刻的行为：

14:30:10 — 围栏建好，定时器启动。d=15000米。状态=远。什么都不干。
14:30:11 — d=14980。状态=远。什么都不干。
…（中间29分钟，每秒查一次，都是"远"，都不干事）…

Director是最终"说人话"的那个环节。

它拿到Advisor的建议之后，还会同时看当前的Context——就是L26注入的那些动态信息：用户记忆（这个用户的外婆叫什么、以前打过电话没有）、车况（现在在开车、车速多少）、情景（是白天还是晚上）。

其他生命周期场景

除了正常触发闭环之外，还有两种情况要处理：

场景1：用户中途取消。
用户说了"算了，不用提醒了"。云端识别到取消意图，给端侧下发一条delete_fence指令，端侧收到后删围栏、停轮询。云端标记规则失效。系统回复用户"好的，已取消提醒。"

场景2：车机重启。
熄火、重启后定时器会丢失。但围栏信息存在本地持久化存储里，重启后围栏模块读一遍本地存储，发现有一个活跃围栏f_001——重新启动定时器，继续监听。不会因为重启就丢了用户设的提醒。
帅哥可能问：如果用户同时设了好几个提醒呢？比如"快到外婆家提醒我打电话"+"快到加油站提醒我加油"？
"端侧围栏模块维护的是一个围栏列表，不是单个围栏。用户设了两个提醒，端侧就有两个围栏，定时器每秒跑的时候遍历列表，每个围栏都查一次距离、判一次。

如果围栏数量特别多——比如10个以上——每秒要调10次SDK查距离，这时候可能需要考虑性能。但正常场景下用户不会同时设太多提醒，3~5个完全没问题。"
