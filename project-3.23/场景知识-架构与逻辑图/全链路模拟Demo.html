⭐ 你写的内容
⚠️
关键理解：13 条知识是全量注入，不是按需召回。也就是说，不管用户点了"车内小憩"还是"急速降温"，豆包同时看到全部 13 条。所以：
① 每条必须有清晰的"场景要点"让豆包快速定位
② 易混的场景必须有"互斥锚定"帮豆包区分
③ 总 token 量必须控制（14K 以内），否则影响推理质量
📋 你在这一步该做什么
★
理解你的内容在整个 prompt 中的位置：你写的 14K tokens 占总 prompt 的约 34%（41K 总量）。豆包先读 27K 的角色定义和工具说明，最后才读到你的知识。这意味着你的知识必须自包含——不能假设豆包"记得"前面说过什么。
✓
写路由原则（200字）：放在 13 条知识的最前面。内容 = 当多条场景都匹配时，按什么信号选？信号包括：说话人位置（主驾/副驾/后排）、车辆状态（驻车/行驶/温度）、视觉信号（闭眼/哭闹）、关键词差异（"我要睡" vs "别吵到他"）。
!
token 预算管理：13 条 × ≤1500 字符 + 路由原则 200 字 = 约 14K tokens。每条场景知识不能超过 1500 字符，否则总量超标。已有案例（R10/R14）在范本中约 1200 字符，可以作为参考。

Planner 推理：豆包怎么选场景

用户说了一句话，豆包拿着完整 Prompt（41K tokens）开始推理。它要在 13 条场景中选出正确的一条。下面用 4 条真实 query 模拟推理过程。

🧠
这一步你不写代码——这是豆包在运行时自己做的。但推理质量 100% 取决于你在 Step 3 写的内容。你写的"场景要点"= 豆包的眼睛；"互斥锚定"= 豆包的裁判；"路由原则"= 豆包的规则书。
Query 1 · 简单情况 · "太困了给我定个20分钟闹钟" ▸ 点击展开
❶
解析 user_query
输入：{"query":"太困了给我定个20分钟闹钟","speaker_name":"明骏","speaker_position":"主驾"}
提取关键词："困" + "20分钟闹钟" + 说话人主驾
❷
扫描 13 条场景要点
"困"+"闹钟" → 匹配睡眠三角组：
  • 车内小憩："主驾/乘客主动想要自己短暂休息"✓ ← 高匹配
  • 后排勿扰："前排表达后排有人需要安静" ✗ 不匹配
  • 冥想空间："清醒地放松" ✗ 用户说"困"，不是清醒状态
❸
交叉验证互斥锚定
小憩的互斥锚定里没有列出这条 query → 不需要排除
后排勿扰的触发样本里也没有 → 双向确认
✓
最终判断
→ DRV-01-01-09 车内小憩（置信度：高）
依据：说话人=主驾 + "困"=睡眠意图 + "给我"=自己要睡 + "闹钟"=需叫醒
Query 2 · 易混情况 · "安静点别吵到后面" ▸ 点击展开
❶
解析 user_query
输入：{"query":"安静点别吵到后面","speaker_name":"明骏","speaker_position":"主驾"}
关键词："安静" + "后面" → 可能匹配多条！
⚠
歧义检测
"安静"在三条场景都出现：
  • 车内小憩的触发词有"安静"
  • 后排勿扰的触发词有"安静"
  • 冥想空间的触发词有"安静"
→ 仅靠 query 文本无法区分，需要额外信号！
❸
查路由原则
路由原则说："当无法区分时，检查以下信号"：
  1. speaker_position = 主驾 → 说话人在前排
  2. query 提到"后面" → 关心的对象是后排
  3. 查 env_info：后排有1名儿童乘客
→ 前排人关心后排人 → 排除"自己要睡"
❹
查互斥锚定
车内小憩的互斥锚定写了：
"小声点别吵到后面"→ DRV-01-01-13 后排勿扰
直接命中！互斥锚定明确指示走后排勿扰。
✓
最终判断
→ DRV-01-01-13 后排勿扰
依据：前排说+关心后排+互斥锚定明确指向。如果没有互斥锚定，这条 query 大概率会走错。
Query 3 · 陷阱 query · "我想放松一下" ▸ 点击展开
❶
解析 user_query
"放松" 是高歧义词——可以是冥想（清醒放松）、可以是小憩前奏（累了想休息）。
没有"困""睡""闹钟"等睡眠关键词。
❷
互斥锚定命中
车内小憩的互斥锚定明确写了：
"帮我冥想/我想放松一下"（非睡眠意图）→ DRV-01-01-08 冥想空间

冥想空间的场景要点："用户想要清醒地放松——没有叫醒/闹钟需求"

"放松"+ 没有睡眠关键词 = 清醒放松 = 冥想
✓
最终判断
→ DRV-01-01-08 冥想空间
依据：无睡眠信号 + 互斥锚定双向确认。如果用户说的是"我想放松一下然后睡一觉"——有"睡"字，就会走小憩。
Query 4 · 多场景碰撞 · "好热，开窗透透气" ▸ 点击展开
❶

"原子化调用"的意思：每个工具调用是独立的一步。调一次→等返回→再决定下一步。不是批量执行。

你写的"执行逻辑"直接决定这个调用链。如果你写"先查温度再开空调"，豆包就会先调 search_vehicle_status_info 再调 vehicle_basic_control。如果你没写，豆包会自己猜——可能猜对，也可能猜错。

✓
执行逻辑里写清楚工具名。用 vehicle_basic_control 不用"调整车辆设置"，用 search_user_memory 不用"查用户喜好"——贴近工具的真实名称，减少豆包的翻译步骤。
✓
写清调用顺序和优先级：用户说"热死了"，先开空调（体感最快）还是先开窗（换气最快）？你的执行逻辑决定了这个顺序。建议：先做体感最强的动作（空调/座椅通风），再做辅助动作（开窗/遮阳帘）。
★
写好条件分支和失败降级：每个工具调用都可能失败。你的执行逻辑里必须写"如果 X 失败 → 做 Y"或"如果 X 失败 → 告知用户具体的手动操作方法"。不写=豆包只会说"操作失败请手动"。
!
别忘了记忆调用：在执行逻辑里加一步"查记忆"可以让豆包提前知道用户偏好（不吹脸/喜欢恐龙），实现"超预期服务"。这是 520 演示的加分项。

6
多轮对话：用户最终看到/听到什么

前面 5 步都是后台机制，用户只感知到这一步——豆包说了什么、做了什么、是不是像个人。

🎯
520 演示的评审标准：评审最关注的不是功能完整度，而是"豆包像不像一个人"。一个自然的沉默 > 十个正确但机械的回复。你写的对话示例直接决定了这个"人感"。

每一轮的设计意图
第1轮：用户 13 个字包含全部信息（困+20分钟+闹钟）。系统一句话搞定，不逐项播报。
第2轮：用户只回"嗯"——已进入休息状态。
第3轮 · 沉默 🤫：这是最关键的一轮。豆包选择不说话。"祝您休息愉快"= 打扰。沉默 = 最好的服务。
第4轮：叫醒用轻声+叫名字，不用闹铃声。像人叫人起床。
第5轮：用户"好了好了"=我醒了别啰嗦。系统自动恢复座椅，一句话收尾。
✅
7 轮对话，豆包只说了 3 句话。其余时间沉默或执行。这就是"人感"。

每一轮的设计意图
第1轮："后面吵死了"是情绪宣泄。系统翻译情绪→后排儿童哭闹→需要安抚。不问"您需要什么帮助"。
第2轮：连说两个"放吧"+"快"=焦躁。不追问"哪一集"，直接执行最可能成功的方案（恐龙——来自记忆）。
第3轮：同时服务两个人——给孩子放内容 + 主动给家长降噪（"不吵你开车"）。
第4轮 · 沉默 🤫："总算清净了"是自言自语。搭话 = 打断清净 = 反效果。
第5轮：儿童直接提需求。系统用"好不好"商量而非命令，同时争取缓冲（看完再换）。
✅
关键技巧：系统在和儿童对话时，用拟人化语气（"恐龙看完这集"），不像机器（"当前内容播放完毕后将切换"）。对家长用效率（一句话搞定），对孩子用陪伴（商量、鼓励）。
