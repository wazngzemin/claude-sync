1. 背景

1.1 现状与痛点

触发器系统目前不认识"导航目的地""剩余距离"这些概念。用户说"快到外婆家提醒我打电话"，系统无法注册、无法监测、无法触发。整类"快到某地就主动服务"的场景一个都做不了。

这些导航信号在《触发器条件清单》里本来就有定义（序号 119~133），只是之前被删了/没开放。本期是把它们作为新增重新开放，并补上核心缺失的"接近/到达/离开"判定逻辑。

1.2 地图侧确认结论（2026-06-24 宫丽萍）

高德地图SDK没有现成的地理围栏API。但可以根据导航距离判断。
注意：围栏可以在端侧自建（用GPS坐标自行计算进出），只是高德不提供现成接口。
产品决策：优先走导航剩余距离方案（方式A：动态插途经点 → 复用距离判定），工作量最小、复用度最高。端侧自建围栏作为备选。

3. 两类Case的本质区别

看起来都是"快到XX提醒我"，但触发条件的性质完全不同，必须分开处理：

A类 · 目的地/途经点距离（能做）

  • POI就是当前导航的目的地或途经点
  • 导航引擎已经在实时算到它的剩余距离
  • 信号现成（124/128/130），只缺one_shot状态机
  • 例："离目的地还有1km提醒我拿咖啡"
  • 例："快到途经点帮我找停车位"

B类 · 靠近任意POI（有前置依赖）

  • POI不是当前导航目标（"外婆家"可能不在路线上）
  • 导航引擎没有在算到它的距离
  • 需要先把它变成导航系统认识的途经点
  • 然后问题退化为A类
  • 例："快到外婆家提醒我打电话"
  • 例："路过徐浦大桥提醒我右转"
核心洞察：既然高德不支持围栏，B类的唯一路径是把任意POI转成途经点 → 退化为A类。所以"地图SDK能否动态插入途经点"（仁杰确认中）是B类case的生死问题。

4. 核心难点：为什么不能直接配"距离<500m"

4.1 数字走查：一个真实场景

周一早8:00，用户导航去公司（出发距离15km），产品想做"距公司500m提醒拿咖啡"。

时刻剩余距离如果配"距离<500m"正确行为
8:00~8:2515km → 501m不触发 ✅不触发 ✅
8:25:30首次到500m触发 ✅触发一次 ✅
8:25:31499m触发（第2次）❌不触发 ✅
8:25:32498m触发（第3次）❌不触发 ✅
......一路刷屏到50m ❌安静 ✅
8:30:00≈0m，挂P档还在触发 ❌触发"到达" ✅
问题：从500m开到公司（499、498...50m），每帧都满足"<500m"，会触发几十上百次——用户被"该拿咖啡了"刷屏。所以必须是"从远到近、第一次跨过500m → 触发一次 → 然后闭嘴"——这就是 one_shot，也正是刘杨说"现有信号协议做不了"的原因。

4.2 现有协议为什么做不了

现有协议判断的是"当前值满不满足"（距离<500m？），它是无记忆的——每次刷新独立判断。而"接近"需要三样东西：

  1. 记住上一次的状态（有记忆）
  2. 判断方向（在变近还是变远）
  3. 触发后标记"已触发、别再发"（one_shot自锁）

4.3 解决方案：POI状态机

给每个被监控的目的地/途经点维护一个状态：远 → 已接近 → 已到达 → 已离开。

→

state = 该POI当前状态           # 初始：远
d     = 当前剩余距离(沿路线·米)

if state == 远  and  d <= 接近阈值(480m):        # 死区下沿
    触发【接近】;  state = 已接近                  # one_shot：状态翻了就不再触发

if state == 已接近  and  d <= 到达阈值(50m)  and  档位变更为P:
    触发【到达】;  state = 已到达                  # 组合条件

if state in (已接近, 已到达)  and  d > 离开阈值(520m):
    触发【离开】;  state = 已离开                  # 反向判定

17. 新方案：云端下发任务 + 端侧判定（6.25会后演进）

for task in 任务队列:
    poi   = task.poi                   # POI经纬度
    r     = task.radius                # 判定半径（米）
    state = task.state                 # 初始：outside

    # ---- 选判定方式 ----
    if 导航中 and poi == 当前导航目的地:
        d = 导航沿路线剩余距离         # A类：高精度
    elif 围栏SDK可用:
        d = 围栏SDK.query(poi)         # B类：围栏返回距离或在/不在
    else:
        d = haversine(当前GPS, poi)    # B类降级：自算直线距离

    # ---- enter判定 ----
    if state == outside and d <= r:
        触发【enter】                  # 进入围栏
        state = inside
        上报事件: {task_id, event:"enter", poi_id, timestamp}
        if task.policy == "one_shot":
            标记task.enter_locked = true  # one_shot：不再触发第二次enter

        触发【exit】                   # 离开围栏
        state = outside
        上报事件: {task_id, event:"exit", poi_id, timestamp}
        if task.policy == "one_shot":
            从任务队列移除task          # one_shot：enter+exit各一次，任务结束

    if task.判定方式 == "导航距离" and 导航已取消:
        切换到围栏判定(task)            # 信号源断了，降级到围栏

这一秒：isInsideFence(GPS={31.1510, 121.6580}) → true（距迪士尼约940m）

① 状态翻转前：state = outside ✅
② 围栏查询结果：true（在围栏内）
③ false → true = 进入围栏！
④ 状态翻转：outside → inside
⑤ 产出事件：{task_id:"task_nav_002", event_type:"enter"}

17.9 新方案如何解决 6.25 会上三个问题

问题会上谁提的现方案为什么解不了新方案怎么解
① 离开判不了 刘杨、汪帅 距离永远是正数，d > 520 无法区分"还在靠近但没到"和"已经走了" 围栏天然有"在内/不在"两个状态。从"在"变"不在" = 离开。不需要比较方向
② B 类不应依赖导航 施佳杰、陈俊龙 B 类需要"动态插途经点"，地图接口待确认；用户可能根本没开导航 围栏只需要 POI 经纬度 + 半径。不需要导航状态，不需要插途经点
③ 坐标出车的合规问题 仁杰 信号每秒上报云端，坐标频繁出车 围栏判定在端侧，只上报 enter/exit 事件。坐标不出车
