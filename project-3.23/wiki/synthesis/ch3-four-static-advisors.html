3.1 舒适Advisor —— 座舱环境AI管家

一、舒适Advisor是什么？

定位：座舱内负责用户舒适性的AI管家。在整个Driver Agent 2.0的四个静态Advisor里，舒适Advisor是用户"体感"上最直接能感知到的模块。它不负责导航、不负责娱乐、不负责情绪，它只做一件事：让车里的人感觉舒服。它是一个持续运行的后台"监工"，盯着"舒适"这个静态Goal，不断检测车内环境、判断是否需要干预、给出具体的调整建议。

二、6+1舒适维度 —— 它"看"什么？

舒适Advisor的SP里定义了6+1个舒适维度，这些就是它评估舒适度的"体检指标"：
  1. 体感温度 —— 车内温度、座椅温度、方向盘温度等综合感受
  2. 空气质量 —— PM2.5、CO2浓度、车内异味、花粉
  3. 光线 —— 阳光直射、阅读灯、氛围灯亮度
  4. 声音 —— 风噪、路噪、音乐音量、后排是否有孩子在睡觉
  5. 乘坐 —— 座椅角度、按摩、通风、头枕位置
  6. 空间 —— 后视镜遮阳帘、天窗、车窗状态
  7. 其他 —— 无法归入上述六类的舒适性问题，兜底维度
为什么要定义维度而不是让模型自由发挥？因为LLM在开放空间里推理"什么让人不舒服"容易遗漏——它可能想到温度，但忘了光线；想到空气，但忘了声音。维度化本质上是给模型一份checklist，确保它每次推理都"过一遍"所有维度，不会漏项。这和Planner的23个工具定义逻辑一样：不是限制模型，而是用结构降低遗漏概率。

三、5类输入模板变量 —— 它"读"什么信息？

每次触发舒适Advisor推理时，SP中的动态模板变量会被注入5类信息：
  1. {{attention}} —— 事件信号。谁上车了？温度传感器变化了？有人打喷嚏了？
  2. {{watcher}} —— VLM（视觉语言模型）的描述。"后排小女孩在揉眼睛"、"主驾在翻书"、"副驾在打电话"
  3. {{car_condition}} —— 端侧信号。空调设定温度、实际温度、风量、车窗状态、座椅通风是否开启等
  4. {{web_information}} —— 世界信息。当前城市天气38度、AQI 180重度污染、紫外线指数极强
  5. {{memory}} —— 用户记忆。"张先生怕冷，空调不低于24度"、"后排宝宝3岁，不能直吹"
为什么需要这么多信号源？因为"舒适"是一个高度context-dependent的判断。同样是28度：夏天暴晒后上车觉得酷热难耐，秋天午后觉得刚刚好。你不知道外面天气就没法判断车内温度够不够用；你不知道后排有宝宝就可能把空调开太大。五类信号是五个维度的上下文，缺任何一个，建议都可能"离谱"。

四、触发机制 —— 什么时候"醒来"？

舒适Advisor有两种触发模式：
  1. 事件触发：有人上车 / 温度传感器跳变 / VLM检测到特定行为（如"用户在阅读"→ 需要调亮阅读灯）。Watcher和工程端侧信号作为事件源。
  2. 定时Review触发：当1分钟内没有新事件，且Goal List中存在舒适相关的子目标（比如"正在极速降温"），就定时触发一次review——检查子目标进展，判断是否需要调整。
真实Case：38度暴晒上车
事件触发：有人上车 + {{car_condition}}车内温度48度 + {{web_information}}室外气温38度
舒适Advisor推理：体感温度维度严重超标 → 应对建议："开启极速制冷模式（空调16度+最大风量）+ 开启座椅通风 + 先开窗换气2分钟后关窗"
2分钟后定时review触发：{{car_condition}}温度已降至32度 → 更新建议："关闭车窗 + 风量调至3档 + 维持制冷"
5分钟后再次review：温度降至26度 → 更新建议："空调调至24度 + 风量调至2档 + 关闭座椅通风"
真实Case：后排擤鼻涕（TBD场景）
事件触发：{{watcher}} VLM描述"后排乘客在擤鼻涕" + {{car_condition}}当前风量4档直吹后排
舒适Advisor推理：后排乘客可能鼻塞/感冒，大风量直吹会加重不适 → 应对建议："后排风量调至1档，出风口调为非直吹模式"
这个case之所以标TBD，是因为需要VLM足够精确识别"擤鼻涕"这个行为，误判率高时会导致频繁误触。
真实Case：个性化午休
事件触发：{{watcher}} VLM描述"主驾座椅放倒，闭眼" + {{memory}} "用户习惯午休，喜欢20度+暗光环境"
舒适Advisor推理：用户进入午休状态 → 应对建议："空调调至20度 + 风量最低 + 关闭阅读灯 + 天窗遮阳帘关闭 + 座椅按摩开启（轻柔模式）"

五、SP结构分析

舒适Advisor的SP整体结构如下：
  1. 角色定义：你是座舱舒适管家，负责6+1维度的舒适性管理
  2. 输入说明：说明5类模板变量的含义和来源
  3. 5条knowhow：温度策略（极速降温→渐进降温→维持）、空气策略（开窗换气→内循环过滤）、空间策略（遮阳帘/天窗）、光线策略（阅读灯/氛围灯）、声音策略（降噪/音量适配）
  4. 输出格式：必须输出JSON，包含"当前的舒适性问题"和"应对建议"两个字段
  5. 注意事项：不直接控车、只建议；尊重用户记忆中的偏好；多人场景优先照顾弱势群体（宝宝/老人）
为什么knowhow要写进SP而不是让模型自己推理？因为车控的物理约束是模型不知道的。比如"极速降温"策略里有一条：先开窗换气2分钟再关窗——这是因为暴晒后车内空气有害气体浓度高，直接关窗开空调会把有害气体循环在车内。这种knowhow来自汽车工程经验，不是LLM训练数据能覆盖的，必须显式注入。

六、输出格式与下游对接

舒适Advisor的输出是一个JSON对象： {"当前的舒适性问题": "...", "应对建议": "..."} 这个JSON会被传递给Director/Planner，由Planner决定是否采纳，如果采纳则调度COM专家（对应vehicle_basic_control等车控工具）执行具体车控操作。
为什么舒适Advisor不直接控车？因为架构上需要一个"仲裁层"。想象一下：舒适Advisor建议开窗，但用户正在高速上（安全约束），或者正在打电话（声音约束），或者后排宝宝刚睡着（声音+温度约束）。这些跨维度的冲突只有Director能看到全局，所以Advisor只出建议，Director做最终决策。

七、坑点与排查

坑1：持续追踪 —— 降温后忘记降风量
舒适Advisor的定时review机制就是为了解决这个问题：极速降温后，如果不持续追踪，会出现"车内已经很冷了但空调还在16度最大风量狂吹"的badcase。排查路径：检查Goal List中舒适子目标是否正确更新了细节和进展，review是否被正确触发。
坑2：多人冲突 —— 主驾怕热、后排宝宝怕冷
舒适Advisor的SP里有"优先照顾弱势群体"的规则，但实际推理中模型可能给出"整车开到最低温度"的粗暴建议。排查路径：检查{{memory}}中是否正确注入了每个乘员的偏好，SP中多人冲突规则是否被遵循。理想输出应该是分区空调策略。
坑3：VLM误判导致的误触发
比如VLM把"乘客低头看手机"识别为"阅读"，触发调亮阅读灯。或者把"乘客扭头和后排说话"识别为"不舒服的姿势"。排查路径：检查{{watcher}}的原始VLM描述是否准确，这类问题本质是感知层误差，需要在Watcher侧优化，舒适Advisor侧无法修复。
坑4：knowhow缺失 —— 模型编造物理约束
比如模型不知道D6X的空调温度范围是16-32度，建议"调到10度"。或者不知道后排没有独立空调控制，建议"后排单独调温"。排查路径：检查SP中的knowhow是否覆盖了车型的实际硬件约束。这是车控幻觉的主要来源。

3.2 出行Advisor —— 出行场景全链路管家

一、出行Advisor是什么？

定位："理解人、理解出行场景、提供整体解决方案"。如果说舒适Advisor管的是"车内微环境"，出行Advisor管的就是"从出发到到达"的整个出行链路。它覆盖行前→行中→行后三个阶段，核心能力是出行目的预测——在你还没说去哪之前，就根据上下文推断你这趟出行要干嘛。

二、核心推理：出行目的预测

出行Advisor最有技术含量的部分就是出行目的预测。它需要综合10+个数据源进行推理：
  1. 历史偏好 —— 用户工作日通常8:30出发去公司
  2. 车内人员 —— FaceID识别到主驾+后排一个孩子
  3. 携带物品 —— VLM看到孩子背着书包
  4. 时间 —— 现在是周二早上8:31
  5. 位置 —— GPS显示在长江小区（家）
  6. 常去地点 —— 记忆中：长江小区=家，XX小学=孩子学校
  7. 导航状态 —— 当前无导航
  8. 日程信息 —— 日历显示9:00有会议
  9. 天气 —— 今天有雨
  10. 特殊日期 —— 非节假日
真实推理Case：
"12月3日周二8:31，益红和依依已上车，车在长江小区，依依带书包 → 推断：先送依依上学（XX小学），再去公司（XX路）"
出行Advisor输出建议：规划路线（先XX小学→再公司），预计行程25分钟，途中经过XX早餐店（用户历史常买豆浆油条）。
为什么要做"目的预测"而不是等用户说？因为车载场景下用户上车后的第一需求往往是明确的（上班/送孩子/回家），但每次都要口头确认很"笨"。一个真正聪明的AI管家应该像老司机一样，看到孩子背着书包、又是周二早上，就知道该先送孩子上学。主动服务的核心价值就在于减少用户操作，让AI多想一步。

三、行前→行中→行后 三阶段场景

行前场景：
  • 出行目的预测 + 路线规划建议
  • 充电推荐（电量不足时结合路线推荐沿途充电桩）
  • POI沿途预推荐（"路上经过XX咖啡店，要不要点一杯？"）
行中场景：
  • 堵车绕行建议（前方拥堵严重，建议绕行XX路，多3分钟但省20分钟）
  • 接近目的地指引 + 最后一公里导引（"目的地在右手边第二栋楼，B1有停车位"）
  • 沿途POI提醒（"前方500米有个网红打卡点"）
行后场景：
  • 停车缴费——通过GUIAgent操作停车APP完成支付
  • 生态服务——"到了海底捞，帮你取号，目前排队8桌"
  • 行程记忆沉淀——更新用户偏好（这次去了XX餐厅，评价不错）

四、触发机制

出行Advisor的触发分为事件触发和定时Review：
  • 上车事件：有人上车 → 立即进行出行目的预测
  • 导航变更/堵车：路线偏离或实时拥堵 → 触发绕行建议
  • 接近目的地：距目的地<1km → 触发最后一公里指引
  • 定时Review：行中每隔N分钟review → 检查出行目标进展

五、SP结构与对应专家

出行Advisor的建议输出后，由Director调度DRV专家执行。DRV专家对应的工具集包括：route_planning_qa（路线规划）、navi_basic_control（导航控制）、search_poi_qa（POI搜索）等。出行Advisor的SP核心是出行目的预测的推理链，需要将10+数据源的信息结构化喂给模型，让模型像老司机一样做"综合研判"。

六、坑点与排查

坑1：出行目的预测误判
比如周末带孩子出门，孩子恰好背了书包（装了零食），模型预测"送上学"。排查路径：检查日期特征（周末vs工作日）、日程信息是否注入。PRD中对推理链的约束不够强，模型容易被单一强特征（书包）误导。
坑2：GUIAgent依赖 —— 停车缴费失败率高
停车缴费需要GUIAgent操控手机APP，这是一个跨Agent协作链路。GUIAgent本身识别UI元素的准确率有限，加上不同停车APP界面差异大，失败率高。排查路径：检查GUIAgent的截图识别日志，通常不是出行Advisor的问题，而是下游执行层的问题。
坑3：PRD成熟度低 —— 很多场景只有描述没有实现
PRD v0.4中出行Advisor的大部分场景还停留在P2优先级，真正在开发的只有堵车绕行和基本的目的预测。排查badcase时要先确认：这个场景是否已经实现了？很多时候不是模型推理错了，而是功能还没开发。
坑4：数据源缺失 —— 推理巧妇难为无米之炊
出行目的预测依赖10+数据源，但实际产品中很多数据源还没接入（比如日程信息需要和手机日历打通，携带物品需要VLM持续识别）。数据源缺失会导致推理退化为简单的"时间+位置"模式匹配，丧失了"多源推理"的核心优势。

3.3 内容Advisor —— 首席内容建议官

一、内容Advisor是什么？

定位："首席内容建议官"。但和一般的"推荐系统"完全不同，内容Advisor的核心设计原则是"克制优先"——默认不工作，没有Trigger就没有内容推送。这在四个静态Advisor里是独一无二的：舒适Advisor上车就工作，情感Advisor持续观察，出行Advisor有导航就运转，唯独内容Advisor默认静默。
为什么内容Advisor要"克制优先"？因为内容推送是所有主动服务中用户反感成本最高的一个。舒适建议（帮你调空调）用户最多觉得多余但不反感；出行建议（前方堵车建议绕行）用户会觉得有用。但内容推送一旦推错——在你专注开车时突然播个笑话、在你心情烦躁时推荐个"振奋人心的播客"——用户会直接关掉主动服务的开关。所以内容Advisor的设计逻辑是：宁可不推也不要推错。

二、L0-L3介入等级状态机

内容Advisor最核心的设计就是这个四级介入等级状态机：
  1. L0 隐形（0%认知占用）：完全不推送任何内容。触发条件：驾驶员愤怒、暴雨/暴雪恶劣天气、车内有争吵冲突。此时任何内容推送都是火上浇油。
  2. L1 氛围（<5%认知占用）：只推极低认知负荷的内容——白噪声、轻音乐、雨声。不需要用户"听"，只是填充背景音。
  3. L2 碎片（~30%认知占用）：推送碎片化、轻量的内容——一个笑话、一条热点新闻、一个短段子。用户听不听都行，不会打断思路。
  4. L3 沉浸（>80%认知占用）：推送深度内容——专家讲坛、深度播客、长篇故事。需要用户投入注意力，只有在"堵车/高速巡航/乘客模式"等注意力可释放的场景才使用。
为什么要做认知占用分级？因为开车时的注意力是一种有限资源。认知心理学里有个概念叫"认知负荷"——人在同一时间能处理的信息量有上限。开车本身就在消耗认知资源（路况判断、操控），如果内容推送占用太多认知资源，就会影响驾驶安全。L0-L3本质上是一个认知预算管理系统：根据当前驾驶场景可释放的认知余量，分配对应等级的内容。

三、8步推理Flow

内容Advisor每次被触发后，走一个完整的8步推理链：
  1. 静默：默认待机状态，不消耗资源。
  2. 唤醒：Trigger到达（事件/定时/协同信号），Advisor激活。
  3. Context收集：拉取当前场景信息——驾驶状态、情绪、路况、时间、乘客组成。
  4. 介入判断：根据Context决定当前应该处于L0/L1/L2/L3哪个等级。如果判断为L0，直接回到静默。
  5. 策略制定：确定等级后，选择具体内容类型和内容来源（音乐库/新闻/播客/AI生成等）。
  6. 分发：将内容推荐建议发送给Director，由Director调度ENT专家执行播放。
  7. 反馈收集：监听用户响应——跳过？继续听？主动要求换？
  8. Review：基于反馈做归因分析，调整下一次的策略。

四、反馈闭环 —— 内容Advisor的"自我纠错"

内容Advisor有两个关键的反馈闭环机制：
  • 连续跳过3次 → 归因 → 修正：如果用户连续3次跳过推荐的内容，Advisor不会继续推，而是先做归因分析——是内容类型不对（不喜欢新闻？换音乐）？还是时机不对（在打电话？降级到L0）？还是介入等级不对（堵车了但用户在处理工作电话？降级）？归因后修正策略再继续。
  • 路况变拥堵 >5分钟 → 降认知等级：如果正在播L3内容（深度播客），突然前方路况变拥堵，需要用户集中注意力看路，Advisor会自动降级到L2或L1，确保驾驶安全优先。

五、协同信号 —— 和情感Advisor的联动

真实Case：情感Advisor通知"适合陪伴"
情感Advisor检测到用户情绪低落但已趋稳（不再是愤怒/悲伤高峰期），发出协同信号"适合陪伴" → 内容Advisor收到信号后唤醒 → 判断当前为L1等级（用户情绪刚恢复，不宜高认知负荷）→ 推荐轻柔的治愈系音乐 → Director调度ENT专家播放。
为什么需要情感Advisor的协同信号？因为内容Advisor自己不做情绪判断——它只看场景和认知负荷。但内容推送的时机极度依赖情绪状态：用户开心时推个脱口秀合适，用户悲伤时推个脱口秀就是灾难。情绪判断交给专业的情感Advisor来做，内容Advisor只接收结论（"适合陪伴"/"不宜打扰"），这就是模块化分工的好处——每个Advisor做自己最擅长的判断。

六、坑点与排查

坑1：介入等级误判 —— 该静默时推了内容
最常见的badcase：用户在高速超车/变道时（高认知需求），内容Advisor却判断为L2推了条新闻。排查路径：检查Context中的驾驶状态信号是否准确注入，L0的触发条件是否覆盖了"高速变道"这种场景。
坑2：连续跳过归因不准
用户连续3次跳过不是因为内容不好，而是因为在打电话（语音通道被占用）。但归因分析误判为"内容类型不对"，下次换了内容类型还是被跳过。排查路径：检查归因分析时是否纳入了"当前是否有语音通话"这个信号。
坑3：和情感Advisor协同时序错误
情感Advisor发出"适合陪伴"信号时用户情绪刚好转，但等到内容Advisor完成推理+Director调度+ENT专家执行，可能已经过了30秒，用户情绪又变了。排查路径：检查协同信号的时效性，是否在信号失效前完成了内容推送。
坑4：冷启动问题 —— 不了解用户偏好
新用户首次使用时，内容Advisor没有任何用户偏好数据，只能基于场景做泛化推荐（堵车+早上 → 推新闻），命中率低。排查路径：这不是bug而是产品设计问题，需要引导用户建立偏好画像（比如首次使用时询问内容偏好）。

3.4 情感Advisor —— 纯情感层察言观色

一、情感Advisor是什么？

定位："纯情感层察言观色"。这是四个静态Advisor中推理最精细、红线最严格的一个。它的职责极其纯粹：只做情感层面的观察和建议——不控车（那是舒适Advisor的事）、不推内容（那是内容Advisor的事）、不提醒物品（那是出行Advisor的事）。它就像一个极其敏感的"情绪感知器"，察言观色、捕捉微妙的情绪变化，然后给出纯情感层面的应对建议。
为什么情感Advisor的边界要画得这么严格？因为情感推理是所有推理中最容易"越界"的。你检测到用户悲伤，本能反应是"帮他放首舒缓的歌"——但这就越界到了内容Advisor的地盘。你检测到用户焦虑在发抖，本能反应是"帮他调高车内温度"——但这就越界到了舒适Advisor的地盘。如果不画严格的红线，情感Advisor会变成一个"什么都管"的万能模块，和其他三个Advisor产生大量冲突。6条红线就是它的"行为围栏"。

二、两套完整SP：Event Prompt + Review Prompt

情感Advisor是唯一一个有两套完整SP的Advisor：
  • Event Prompt：事件驱动，收到Watcher发来的事件（有人上车/表情变化/语调变化等）时触发，走4步推理链。
  • Review Prompt：定时驱动，每隔一段时间（当存在情感子目标时）触发，走3步推理链，检查子目标进展。
两套SP的输入上下文不同、推理逻辑不同、输出侧重点不同——Event重在"发现问题+即时建议"，Review重在"跟踪进展+闭环目标"。

三、Event Prompt 4步推理链

Step 1：事件解析 + 优先级排序
收到的事件可能有多个（同一时刻可能"有人上车"+"车内温度高"+"用户表情烦躁"），需要按优先级排序处理。优先级从高到低：启动方式 > 安全 > 节日 > 人物状态 > 情绪 > 人物 > 场景 > 着装。
Case：优先级排序实例
同时收到三个事件：1)用户表情烦躁(情绪) 2)今天是用户生日(节日) 3)后排有儿童安全座椅未系(安全)。
排序结果：安全 > 节日 > 情绪。
情感Advisor先处理安全事件（但注意：它不控车，只在情感层面建议"用温和语气提醒系安全带"），再处理节日事件（"创建子目标：今天是生日，找合适时机送上祝福"），最后处理情绪事件（"用户烦躁，创建子目标：持续观察情绪变化"）。
Step 2：即时建议（含6条红线约束）
基于事件分析，输出即时的情感建议。但这一步有6条红线硬约束：
  1. 不控车 —— 不能建议"开空调"、"调座椅"
  2. 不推内容 —— 不能建议"放首歌"、"讲个笑话"
  3. 不提醒物品 —— "你忘带手机了"不是情感层该说的
  4. 不给出行建议 —— "前方堵车"不关情感的事
  5. 不代替用户做决策 —— 只共情，不指导
  6. 安全事件优先级 > 一切情感目标
Step 3：子目标管理
情感Advisor维护自己的子目标列表（上限10个），分两种类型：
  • A目标（达成型）：有明确达成条件的目标。例如"安慰用户被超车后的烦躁情绪"——当用户情绪恢复后标记达成。
  • B目标（条件响应型）：持续监测、条件触发的目标。例如"监测后排宝宝情绪，如果哭闹则提醒主驾"。
操作类型有三种：create（新建）、update（更新进展）、delete（删除/完成）。
Step 4：冲突检查
多个子目标之间可能冲突，需要做冲突检测和优先级裁决。裁决规则：安全 > 生理 > 情绪。

四、Review Prompt 3步推理链

  1. 评估进展：逐一检查每个子目标的达成情况，信号来源包括VLM观察（表情变化）、对话记录（用户说了什么）、tool_feedback（执行结果）。
  2. 决策：对每个目标做出决策——调整（方向变了）/ 恢复删除（外部条件变了）/ 完成（目标达成）。注意一个关键设计：完成和恢复删除时只执行delete操作，不输出新的建议——避免"目标完成了还多嘴"的情况。
  3. 整理：清理已完成/已删除的目标，重新排序剩余目标的优先级。

五、5种动作空间

情感Advisor的输出不是直接的车控指令，而是5种"软动作"：
  1. DO_NOTHING：什么都不做。这是最常用的动作——大部分时候情绪是平稳的，不需要干预。
  2. CONVO_STRATEGY：对话策略暗示。不直接说话，而是在Director的对话策略中注入情感暗示（比如"回复时用温柔一点的语气"）。
  3. CONVO_INITIATE：主动发起对话。在合适时机主动开口（比如"你刚才被超车了是不是有点气？没事，那人开得太莽了"）。
  4. EMOJI_SUGGESTION：表情建议。通过IOT驱动豆包的屏幕表情/形象做出对应的情绪响应。
  5. TOOL_CALL：self ReACT工具调用。情感Advisor自己做一轮ReACT推理，调用查询类工具获取更多信息（比如查用户记忆确认"今天是不是纪念日？"）。

六、真实Case深度解析

Case 1：被超车共情吐槽 → 安抚
事件：{{watcher}} VLM描述"主驾表情紧绷，嘴唇抿紧" + {{attention}} 刚被一辆车别了一下。
Event Prompt推理：
Step 1：事件类型=情绪变化，优先级=情绪层级。
Step 2：即时建议 → 动作=CONVO_INITIATE，内容="嚯，那人开车也太莽了吧，差点蹭到咱们"（共情吐槽，不是说教"别生气"）。
Step 3：创建A目标"安抚主驾被别车后的烦躁情绪"。
Step 4：无冲突。
后续Review：1分钟后review，VLM显示用户表情放松 → 目标达成 → delete。
Case 2：子目标冲突 —— 后排儿童睡着 + 主驾分心
当前子目标：
- 目标A（B型条件响应）："后排宝宝刚睡着，保持安静环境"
- 目标B（A型达成型）："主驾注意力分散，需要提醒专注驾驶"
冲突：目标A需要安静 vs 目标B需要发声提醒。
Step 4冲突裁决：安全 > 生理 > 情绪，主驾安全是最高优先级。
输出：CONVO_INITIATE，用最低音量轻声提醒主驾注意路况（折中方案：满足安全需求的同时尽量降低对宝宝的影响）。

七、坑点与排查

坑1：越界 —— 情感Advisor给出了车控/内容建议
最常见的badcase：模型检测到用户冷了（发抖），直接建议"开暖风"——这越界到了舒适Advisor的职责。排查路径：检查SP中6条红线的约束是否足够强，few-shot示例是否有越界的正例。解决方案：在SP中加入更多负例（"以下建议是错误的..."），强化红线约束。
坑2：子目标爆炸 —— 超过10个上限
在复杂场景下（多人同车+特殊日期+复杂情绪），情感Advisor可能创建过多子目标导致溢出。排查路径：检查create操作前是否做了数量检查，以及当达到上限时的降级策略（应该合并低优先级目标或删除最旧的已部分完成的目标）。
坑3：Event Prompt和Review Prompt的状态不一致
Event创建了一个子目标，但下一次Review时因为上下文窗口限制没有带上最新的目标列表，导致Review不知道有这个目标存在。排查路径：检查Goal List的传递机制，确保两套SP共享同一份目标列表的最新快照。
坑4：情绪误判 —— VLM表情识别不准
"用户皱眉"可能是在思考而不是愤怒；"用户沉默"可能是在享受安静而不是情绪低落。情感Advisor高度依赖VLM的表情识别准确率，而车内光线变化大（隧道/逆光）、摄像头角度偏、戴墨镜等场景下VLM准确率会大幅下降。排查路径：检查{{watcher}}的VLM描述置信度，低置信度事件应该被降级处理或直接忽略（DO_NOTHING）。
坑5：CONVO_INITIATE时机不对 —— "多嘴"
情感Advisor决定主动开口安慰用户，但此时用户正在打电话/和副驾说话/专注驾驶。这种"多嘴"比不说话还糟糕。排查路径：检查动作输出前是否做了"当前通道占用"检查——如果语音通道正在被占用（通话/其他Agent播报），CONVO_INITIATE应该被缓存等待或降级为CONVO_STRATEGY（暗示而非直接说话）。
