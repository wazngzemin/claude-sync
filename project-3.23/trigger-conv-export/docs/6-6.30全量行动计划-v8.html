一、触发器是什么？为什么需要它？

你在做的产品叫"豆包上车"（Driver Agent 2.0 驾驶代理2.0版本，简称DA2），是装在上汽荣威D6X车上的AI助手。

这个AI助手有一个"大脑"叫 Director 导演/总决策者。用户说话的时候，Director负责理解意思、做决策、调工具执行。

但Director有一个致命的缺陷：它只有用户说话的时候才工作。

用户不说话的时候呢？比如：
· 副驾驶睡着了——谁来自动调小音量？
· 车里温度超了28度——谁来自动开空调？
· 用户10分钟前说了"10分钟后关座椅加热"——10分钟到了谁来执行？
· 车后面有只小狗——谁来通知系统调整环境？

Director做不了这些事。它需要一个"值班哨兵"——24小时不停地盯着车上的所有传感器和摄像头，一旦发现某个预设的条件满足了，就主动去通知该通知的人。

这个哨兵 = 触发器 = Trigger 触发器 = 你负责的产品。
图1：触发器干的3件事（CTA）
类比：如果把整个系统比作一个公司

· Director（导演） = CEO。用户说话的时候CEO亲自接待。
· 4个静态 Advisor（顾问） = 4个VP。舒适VP管温度灯光、出行VP管导航充电、情感VP管打招呼安慰、内容VP管音乐视频。
· 动态 Advisor = 临时项目经理。CEO安排的临时任务（"盯着宝宝毯子"），做完就走。
· 触发器 = 7×24小时值班的安保监控中心。盯着公司所有的监控摄像头和传感器，发现异常（有人闯入/温度过高/烟雾报警）就立刻通知VP或CEO。

安保不做决策（"要不要报警"不是安保决定的，是VP/CEO决定的），安保只负责发现+通知。触发器也一样——它不决定"要不要给用户开空调"，它只负责"温度超了28度我通知舒适VP，VP你自己决定怎么办"。
触发器在整个系统里的位置（对应原图中间偏左的橙色区域）

用户说话时走的是一个"三层漏斗"：
第1层：RAG（Retrieval-Augmented Generation 检索增强生成）——用已有的规则快速匹配，处理"打开车窗"这种指令
第2层：情景RAG + Arbitration Model（仲裁模型）——注入用户记忆和车况，判断是简单需求还是复杂需求
第3层：Director（导演）——云端大模型深度推理，处理复杂需求

触发器不在这个漏斗里。它是漏斗之外的一个独立系统。

唯一和漏斗交叉的地方：用户说了一句带条件的话（"10分钟后关座椅加热"）→ 漏斗处理 → Director把这个条件注册到触发器（L33连线）→ 触发器自己计时 → 10分钟到了通知Director执行。

四、DSL到底是什么

DSL = Domain Specific Language 领域特定语言。就是一套"填表格的规范"——你在配置平台上填的每一个条件和动作，后台存成的结构化格式就叫DSL。DSL一共有5个部分。
图4："超25度开空调" → DSL 5个结构的关系
DSL不是编程语言，就是一套"填表格的规范"——告诉机器"什么条件触发"、"触发后做什么"、"做多少次"。详细字段说明见v7第四章。

运算符需要加 change_to（变更为，"从D变成P的那一瞬间"）/ range（区间）/ duration（持续时长）→ 写进配置平台改造需求文档。

七、CTA框架

对应原图：从⑮引擎内部的橙色子模块区域——15-3(条件判定) → 15-6(防抖) → 15-5(频控) → 15-10(穿越) → 15-7(仲裁) → 15-8(事件组装) → 输出。
图5：CTA两级匹配流程
穿越检测（Threshold Crossing 阈值穿越）：你家温度报警器设"超30度报警"，29→31度的那一瞬间响一次。之后一直31度不再响（T→T不重复触发）。只有降到29以下再升到31以上，才再响一次。

防抖（Jitter Filtering）：车速在128~132之间波动，不是真超速——设"死区"（dead zone），只有穿越整个死区才算。VLM抖动更严重：前一帧"有小狗"后一帧"没有"——需要"连续N帧一致才算数"。

十、四条铁律

铁律一：所有动态注册必须经过⑮中转
不管谁注册规则，只能 → ⑮ → ㉔。不许绕过⑮直接给㉔。为什么：⑮是唯一的规则管理者，绕过就没法仲裁和冲突检测。
铁律二：端侧结果不直推Advisor，必须走信号库
㉔ → ④信号库 → L9 → ⑫ → ㉑Context。不许㉔直推Advisor。为什么：信号库是Single Source of Truth（单一事实来源），绕过会出不一致。
铁律三：触发器只读信号库，不直接监听原始信号源
⑮只读⑫云端信号库标准化后的数据。为什么：解耦（Decoupling 让两个模块互不依赖），原始格式千差万别。
铁律四：condition→Context，event→⑮主动push
状态（"车里有小朋友"）→写Context。事件（"小朋友刚上车"）→⑮push给Advisor。Advisor不许自己轮询（polling 反复问"有变化吗？"）。

十一、VLM（视觉模型）

VLM是你当前最复杂最重要的一块。对应原图：左下角的㉕VLM节点 + ⑮里面的第6组VLM处理子模块。
图7：VLM输出的4条处理路径
AlwaysOn SP = 永远开着的系统提示词，VLM按它每5秒自动观察舱内。动态SP = 追加（不是替换！）一段额外指令让VLM临时去看某个东西（"毯子掉没掉"）。最多同时3个动态SP。

KV槽位 = VLM输出的第10个字段"动态查询结果"。临时任务答案写在这里。只有1个字段，3个SP同时跑时怎么区分→你需要问丁彬。

VLM真实JSON输出详细标注、12条路径B事件清单、5种VLM结果状态 → 见v7第十一章。
