Case 1："十分钟后关座椅加热" —— 定时任务

场景：用户上车，觉得座椅有点烫，说了一句"十分钟后帮我关掉座椅加热"。然后开上高速，第 8 分钟车进了隧道，断网了。问题：这十分钟到了，座椅加热还能不能准时关？

步模块在哪干什么（含上下游）为什么在这侧

这个 case 证明的：定时器放车端，断网照样准时关。要是放云端，进隧道就废，出隧道才关，早烫够了。这就是㉔ 端侧触发器存在的全部理由——四个字：断网能跑。

② 链路模块逐个展开（前因→不防会怎样→怎么干→上下游）

⑨ 意图分流（端侧）

【前因】用户说一句话，系统得先判断它是哪类，才能决定走哪条线——是马上控车（"打开车窗"）、还是回答问题（"今天天气怎样"）、还是设个条件任务（"十分钟后关加热"）。三类走的链路完全不同。

【不分流会怎样】所有话都一股脑送给云端 LLM 推理 → 云端扛不住、延迟高、还浪费（"打开车窗"这种明确指令根本不需要 LLM）。

【怎么干】⑨ 用句法 RAG（规则匹配，毫秒级）做第一道筛——含车控关键词走蓝线、含查询走绿线、含时间/条件词（"十分钟后""到了…时候"）走橙线·条件任务。筛不了的才往上送第二层、第三层。

上游：①② ASR 转出来的文字串
下游：橙线·条件任务 → 送 ⑩ 意图解析；蓝线 → ⑤ 车控；绿线 → ⑲ Director

⑩ 意图解析（云端）

【前因】"十分钟后关座椅加热"是人话，端侧看不懂，得拆成机器能懂的结构化条件+动作。

【不拆会怎样】端侧触发器只吃规则包（结构化 JSON），不吃自然语言。不拆，规则包生成不了。

【怎么干】⑩ 用云端 LLM 拆解：识别出时间条件"十分钟后"→ delay:600秒；识别出动作"关座椅加热"→ cmd:seat_heater_off, seat:driver。

上游：⑨ 分流过来的条件任务（橙线）
下游：拆好的 condition + action → 送 ⑬ 配置平台生成场景

⑬ 配置平台（云端）

【前因】拆好的条件+动作，得变成一条"可注册、可下发、可执行"的场景记录。

【怎么干】⑬ 生成 scene_id（唯一编号），配执行策略（这条是一次性，max_count=1，触发后自动清理）。

⑬ 也是产品同学在火山后台手动配预设场景的地方（"新建场景"页：基础信息/发布范围/条件/动作/执行策略）。

上游：⑩ 拆好的 condition+动作；或产品手动配置
下游：场景记录 → 送 ⑮ 注册校验

⑮ 云端触发器（15-1 校验 → 15-2 入库 → 15-9 分类 → 15-14 下发）

【前因】场景生成后，得校验合法性、决定下发到哪、打包成端侧能执行的格式。这套"登记+拍板"的活必须在云端。

【为什么必须在云端】① 校验要查全局能力清单（这车有没有这个硬件），清单在云端；② 分类要全局信息（这条该云端还是车端跑）；③ 自然语言解析要 LLM。

【怎么干】

  • 15-1 iCanDo 校验：查"这车支持 seat_heater_off 吗"→ 支持 ✅
  • 15-2 入库：规则存进云端规则库（带 active 状态）
  • 15-9 端云分类：纯定时任务 → 判定必须车端（断网要能跑）→ 标 target=edge
  • 15-14 打包下发：打成规则包 JSON，经 L14 链路下发到端侧
上游：⑬ 的场景记录
下游：规则包 JSON → 经 L14 链路 → ㉔ 24-1 接收

㉔ 24-1 接收 + 24-2 持久化（车端）

【前因】规则到了车端，得接住、还得存住——因为车每次熄火上电都会重启，内存会清空。

【不存会怎样】规则只放内存，用户设完关车走人，明天上车发现规则没了。

【怎么干】24-1 接收规则包；24-2 存进车机硬盘（flash/eMMC），存的是"到期时间点"（不是剩余秒数，这样重启后倒计时不会从头开始）。

上游：L14 下发的规则包
下游：存好的规则 → 24-4 定时器读取

㉔ 24-4 本地定时器（车端）—— ㉔ 的头号价值

【前因】倒计时如果放云端：车进隧道断网，云端到点了想发"关加热"指令，发不下来。

【不本地跑会怎样】用户进隧道，十分钟到了也不关，出隧道五分钟后才关，早烫够了。

【怎么干】倒计时在车机本地跑，纯本地计时，不依赖网络。断网也照倒数，到点自己触发。

上游：24-2 存好的规则（含到期时间）
下游：到期命中 → 24-9 执行
支持能力：倒计时（秒级）/ 定时（"每天早 8 点"）/ 日期（"周末"）/ 节日。最小间隔 1 分钟、最多同时 30 个、倒计时可显示在车机屏。

㉔ 24-9 白名单执行（车端）

【前因】触发的动作，有的风险低（关座椅加热、调空调）可以本地直接干；有的风险高（解锁车门）必须大脑确认。

【不分会怎样】全绕云端 → 断网时低风险动作也干不了；全本地直执 → 误触发危险动作就是安全事故。

【怎么干】查白名单：seat_heater_off ∈ 低风险白名单 ✅ → 端侧直接发车控指令执行。涉及安全/确认的，送回 Director。

上游：24-4 命中的 scene_id
下游：执行结果 → 24-8 回写

㉔ 24-8 结果回写（车端 → 云端）

【前因】端侧干完活，上面（云端 Director/Advisor）得知道结果，否则云端不知道"已经关了"。

【断网时怎么办】结果先存本地缓存队列（最多 100 条、单条<1KB、超 24 小时丢弃），网络恢复后按时间戳排序补传。

上游：24-9 的执行结果
下游：写信号库④ → 同步 ⑫ 云端信号库 → ㉑ Context 更新 → Director 消费
铁律②：㉔ 结果绝不直接捅给 Advisor/Director，统一写信号库→上报 Context。为啥？直接通知这个那个，十个模块各收各的，状态就乱了。统一进一个记事本（Context），永远一致。

Case 2："帮我照顾一下二排宝宝" —— 动态 Advisor + VLM + 触发器，三者怎么协作

场景：用户开车带娃，上车说了一句"帮我照顾一下二排宝宝"。然后专心开车。系统要自己持续盯二排——宝宝睡着了主动调暗灯光降音量，宝宝哭了主动提醒家长。这个 case 会把动态 Advisor（⑰）、触发器（⑮）、VLM（㉕）三者到底是什么关系，通过每一步的数据流彻底拆开。

① 先把三个角色的关系讲死（这是整个 case 的地基）

角色是谁在这个 case 里干啥一句话定位
动态 Advisor ⑰Director 临时派出来的"专属监工"，用完销毁提需求："我要盯二排宝宝" + 消费结果："宝宝哭了→该降音量调暗灯"。但它自己没眼睛，看不见二排提需求的 + 拿到结果想怎么办的
VLM ㉕端侧视觉模型（看图说话的小模型），装在 AIBOX 上只负责"看 + 描述"：每 5 秒看一眼二排，输出"宝宝在哭/在睡/醒了"。不判断该不该报警、不决定降不降音量干活的眼睛（只看，不判断）
触发器 ⑮云端触发器中间的中介·翻译·路由：把 Advisor 的人话翻译成 VLM 能懂的指令；把 VLM 吐的状态、在发生变化那一刻变成"事件"推给 AdvisorAdvisor 和 VLM 之间的翻译+事件化层
为什么 Advisor 不能直接调 VLM（两个原因，都要靠触发器兜）：
① 语言对不上。Advisor 说人话"盯二排宝宝"，VLM 要的是结构化指令（7 字段：看哪个摄像头/哪个座位/查什么/多久看一次/超时多久…）。触发器 15-5 就是这个翻译层。
② Advisor 要"事件"、VLM 只吐"状态"。VLM 每 5 秒输出"宝宝哭/没哭"（状态）。但 Advisor 不可能每 5 秒被叫醒问一次"现在哭没哭"。Advisor 要的是"宝宝刚哭"这个变化（事件）——把状态变事件（阈值穿越：上次没哭→这次哭了，那一刻才触发）是触发器 15-10 干的，然后 15-11 把事件 push 给 Advisor。

12⑰ 推理
→ ⑲ → ⑳		云端【动态 Advisor 的活】⑰ 收到"宝宝刚睡着"事件 → 推理："该降低音量、调暗氛围灯、关掉副驾直吹" → 产出 advice → 进 ⑲ Director 汇入 → ⑳ 拍板：降音量到 20% + 调暗灯 + 对用户说"宝宝好像睡着了，帮你把音乐调小了~" → 执行。⑰ 在 ⑱ Goal List 更新进展"已识别宝宝入睡"建议是 Advisor 推理，执行是 Director 拍板

（循环）⑰ 继续监控——下次有新事件（宝宝醒了、哭了）再走第 8~12 步。直到用户说"不用管了" → Director 删 ⑱ 目标 → ⑰ 自我注销 → 取消感知订阅 → 触发器删除对应 VLM task 规则 → ㉕ 停止看二排。

③ 三者在这个 case 里的协作，用数据流再串一遍（谁给谁、什么格式）

动态 Advisor ⑰ → 触发器 ⑮：给的是人话需求"盯二排宝宝"（自然语言）。
触发器 ⑮ → VLM ㉕：翻译成结构化指令（7 字段 VLM task：camera_group/seat/query/interval/timeout/priority/mutex_group），经 L14 链路下发。
VLM ㉕ → 信号库 ④：每 5 秒吐状态（sleeping/awake/crying），写进信号库。
信号库 ④ → 触发器 ⑮：⑮ 监听状态变化。
触发器 ⑮ → 动态 Advisor ⑰：状态穿越那一刻，组装成事件（"宝宝刚睡着"），push 给 ⑰。
动态 Advisor ⑰ → Director：基于事件推理出建议（advice）。
看这条数据流就明白了：Advisor 和 VLM 从头到尾没有直接对话。Advisor 给的是人话、VLM 吐的是状态，中间全靠触发器翻译+事件化。这就是三者的关系——触发器是 Advisor（人话/要事件）和 VLM（指令/吐状态）之间不可省的中间层。

④ 链路模块逐个展开

⑰ 动态 Advisor（云端）—— 提需求 + 消费结果，自己没眼睛

【是什么】Director 在对话中临时创建的独立推理实体，带一个具体目标。不是预置的（静态 Advisor 是预置 4 个），是"派活"派出来的。一个目标建一个，用完销毁。

【和静态 Advisor 的区别】静态 Advisor（⑯ 舒适/出行/情感/内容）常驻、被动消费 Context 里已有的信号；动态 Advisor 是临时、主动注册新的感知订阅（"订阅"行为 = 注册一个触发器）。

【它干两件事】① 提视觉需求（第 5 步，人话）；② 收到事件后推理出建议（第 12 步）。它不直接调 VLM、不直接控车、不直接对用户说话——建议要送 Director 拍板。

上游：Director 创建它、给目标+SP；事件由触发器 push 给它
下游：视觉需求 → 触发器注册；建议 advice → ⑲ Director 汇入

⑮ 触发器（云端）—— Advisor 和 VLM 之间的中介

【在视觉里干两件翻译+路由的活】

  • 15-5 编译（人话→指令）：Advisor 说"盯宝宝"，编译成 VLM task 7 字段。没有这层，Advisor 的人话下发不到 VLM。
  • 15-10 阈值穿越 + 15-11 事件 push（状态→事件）：VLM 吐的是状态（一直在睡），Advisor 要的是事件（刚睡着）。这是 6.22 最大的纠正——原来 Advisor 要轮询 Context 等事件，6.22 改成触发器组装事件后 push。

【为什么不能让 Advisor 直接轮询 Context】（6.22 佳杰 25:48 原话逻辑）：Advisor 不可能 24 小时反复去翻记事本等"宝宝醒了没"——浪费算力、不实时。事件必须是发生那一刻被推过来的。

上游：Advisor 的视觉需求（人话）；VLM 的状态（经信号库）
下游：VLM task（经 L14）→ ㉔；事件 → push 给 Advisor

㉕ VLM（车端）—— 只管看的眼睛

【是什么】端侧视觉语言模型，装在 AIBOX 上。被调用方——不主动看，要外部指令告诉它看什么。

【它干一件事】收到 24-6 的运行指令后，按 query（"宝宝状态"）持续看指定摄像头（CABIN 舱内）、指定座位（二排），每 interval（5 秒）输出一次状态描述。不判断该不该报警、不决定调不调音量——只描述"宝宝在哭/在睡"。

【6.22 丁彬补充·端模输出规范】端模应纯客观描述此时此刻的状态（"坛子在地上"），不描述"是或否"（不说"坛子掉没掉"）。判断交给上层（触发器阈值穿越 + Advisor 推理）。

上游：24-6 的运行指令（7 字段翻译来的）
下游：状态 → 24-7 归一 → 写信号库④（不直接给 Advisor）
端模能力边界（6.22 丁彬 40:58，最大不确定性）：端模能不能做"看到 X 才告诉我"的条件式注意力？① 有限集内（只看毯子/毛巾）prompt 限制应该行；② 发散了（手绢/小方巾）是问号——没测过；③ prompt 有渠道改到端上就 OK。这条链路能不能成立，前提是先验证端模能力，7 月中拿结论。

⑤ 两条视觉路的对比（为什么你会乱）

动态 Advisor 要"看"东西，有两条路，都叫视觉，但链路完全不同：

对比路 A·即时视觉（云端 VQA）路 B·持续视觉（端侧 VLM）← 本 case 走的
场景偶尔看一眼："副驾笑了吗""前面有加油站吗"一直盯着："盯二排宝宝""盯副驾打瞌睡"
经过触发器吗不经过。Advisor 自己有定时器，直接调云端 VQA经过。Advisor→触发器编译→下发端侧 VLM
用的是 VLM ㉕ 吗不是，用云端 VQA（取一帧图问一个问题）是，端侧 VLM ㉕ 持续看
在哪跑全程云端端侧（仅赛力斯有端侧算力）
上汽能用吗能（只要有摄像头抓帧上传）不能（没端侧 VLM，上汽走路 A 顶）
所以你乱，是因为："VLM（端侧㉕）"和"VQA（云端）"都叫视觉能力，动态 Advisor 都能用，但一条 Advisor 直连 VQA、一条经触发器调 VLM。本 case（照顾宝宝，要持续盯）走的是路 B。

这个 case 的核心设计点——阈值穿越为什么管用：

② 端模能力卡点（6.22 未定，7 月中必须验证）

• 能做的：输出"毯子在地上"（客观状态，有限集内，prompt 限制可保证）
• 做不了的："如果毯子掉了就上报"（条件判断+主动上报 ← 这步在触发器/云端做）
• 不确定的：如果要看的东西太发散（小方巾、手绢、围巾…），识别准确率未知 ← 需要 7 月中验证

附录 C：VLM · 动态 Advisor · 触发器 — 三者关系一页纸

① 一句话定性

三者不是并列模块，是同一条链路上的三个角色——动态 Advisor 是大脑（决定看什么、看到了怎么办），触发器是神经系统（注册/翻译/信号监听/事件组装+push），VLM 是眼睛（实际去看，输出客观状态）。没有触发器，Advisor 的视觉需求下不去端侧；没有 Advisor，触发器不知道该看什么。

② 数据流全链路（每一步给什么、什么格式）
