关键澄清内容（评审原话）
事件 vs 状态/condition佳杰(29:34)：Context 解决的是 condition（"车里有小孩"），触发器给的是 event（"用户在倒车""用户睡醒"）。倒车=事件、有小孩=condition、做什么=Advisor 给。事件是"高度简化的语义"，advisor 订阅事件
为什么事件放触发器不放 Advisor佳杰(29:03)+西钺(30:27)：事件比状态更有价值；从推理 ROI 角度，不可能让 Advisor 24h 轮询不相关的事 → 才有了触发器的需求
Director 不直接到触发器仁杰(33:53)：Director 先创建 ⑰动态Advisor，由动态Advisor 拆解 query、改写注意力 SP，再去调触发器注册——不是 Director 直连触发器
SP 下发链路（待定）仁杰(34:59)：动态 SP 是"经触发器下发给端模" 还是 "注册触发器场景再给端模" 还是 "动态版本直接给端模"？——没定，7 月中要定
端模能力（关键不确定性）端模能不能做"看到 X 就告诉我"的条件式判断（动态注意力）？丁彬(40:58)：靠 prompt 限制有限集，指令遵循好应该行，但没试过。能 → 链路简单；不能 → 端模只能固定周期输出，需云端模块从每次输出里判断
所以"链路是否清晰"的答案：评审把主链路捋清了（上面这条），但三个卡点没定——① SP 下发走哪条 ② 端模有没有条件式动态注意力能力 ③ 事件组装这层放云端触发器还是端侧。这三个是你 7 月中前要推动定下来的。

四、⚠ 端侧触发器 scope 的变化（评审前 vs 评审后）

这是这次复盘最该让你警醒的一块：你 PRD-v2 和五方关系文档里假设的端侧触发器职责，评审后大半被划走了。

你 PRD 里的"端侧 7 类任务"评审后归属依据
① 本地定时器保留 端侧触发器仁杰/西钺明确：端侧触发器做云端下发的任务，正因为它有定时器
② 简单条件端侧执行大半划走 → 赛力斯 CDC固定类（如赛力斯 CDC 已做的充电桩开盖、温控）归赛力斯；仅动态/会变的留触发器
③ 白名单车控直执划走 → 赛力斯 CDC 底软丁彬(16:17)/仁杰：端侧闭环执行继续走赛力斯业务逻辑，不耦合触发器
④ VLM 视觉持续监控重塑 → "动态视觉感知事件"执行在端模/AIBox，触发器做 SP 下发 + 事件组装。这是你新的主战场
⑤ 弱网/断网降级执行部分保留固定闭环类归赛力斯；但"任务中心端侧持久化（弱网可用）"保留在你这
⑥ 信号采集标准化归属待定你会上甩给了仁杰；实际 ④信号库∈你 scope，采集上报由端模/业务做
⑦ 上报 Context保留但要改成"触发器组装事件后推送"，不是 Advisor 轮询 Context
🆕 评审新增压给你的活：(a) 动态视觉感知事件链路（含 SP 下发 + 事件组装）；(b) 通用视觉事件 / 端侧泛化感知通道（兜底长尾，透传动态 query 给 AIBox，分舱内静态/动态/舱外三类、prompt 增量更新）；(c) 任务中心端云打通（端侧持久化 + 任务/场景字段区分 + enable/disable 反向写入接口）
结论：端侧触发器从你 PRD 的"7 类全包"，收窄成 { 本地定时器 + 动态视觉感知事件 + 通用视觉事件通道 + 任务中心端侧同步 }。固定闭环类（哨兵/上下车/起步安全/迎宾/简单车控）让给赛力斯 CDC。你的 PRD-v2 和五方关系文档需要按这个结论重新划一遍 scope——否则下次评审还是会被打回。
