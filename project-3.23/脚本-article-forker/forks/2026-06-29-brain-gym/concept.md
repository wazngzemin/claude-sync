# 能力位移商业化模型：从"标配→选配"到"刻意重新使用"

## 核心问题

当AI把一种重要的认知能力从"每天必须用"降级为"可以不用"，如何识别这种位移、评估其商业价值、并设计对应的"训练产品"？

## 场景映射

| 文章里的角色/机制 | AI产品经理的工作场景 | 可迁移动作 |
|---|---|---|
| 工业革命替代体力 → 健身房 | AI产品替代用户的思考 → ？ | 用"位移模型"审视自己正在构建的产品对用户认知的影响 |
| 哑铃=剥离用处、只留动作 | SP/Prompt设计=剥离实现细节、只留接口 | "哑铃原理"作为解耦设计的检验标准 |
| 25%→5%的能源占比量化 | 用户在AI介入前后的认知参与度变化 | 设计可量化的"认知位移度"指标 |
| 最闲的人最先进健身房 | 最重度AI用户最先感知依赖 | 从重度用户的反馈中寻找"补偿训练"需求信号 |
| 记忆/推理/专注/创造/判断五区 | AI产品替代的具体认知子能力 | 用五区框架做产品的"认知影响评估" |
| 标配→选配→刻意训练市场 | 能力被AI接管→用户退化→新产品机会 | 能力位移的四步诊断流程 |

## 方法框架

### 框架1：能力位移四步诊断

这是全文核心定律"标配→选配→刻意训练"的操作化版本。

**Step 1 识别位移：** 列出你的产品（或AI工具）替代了用户的哪些动作。不要用抽象词（"帮用户思考"），要用具体动词（"替用户记住常去地点""替用户规划先查天气再定路线""替用户判断现在该不该开窗"）。

**Step 2 分解子能力：** 每个被替代的动作背后，涉及五种认知能力中的哪几种？

| 认知子能力 | 定义 | AI接管方式 | 退化信号 |
|---|---|---|---|
| 记忆 | 存储和提取信息 | AI帮你存取，你不再主动记 | 脱离AI后回忆不出常用信息 |
| 推理 | 从已知推导未知 | AI直接给结论，跳过推导过程 | 面对多步问题无法自行拆解步骤 |
| 专注 | 持续集中注意力 | （非AI直接导致，算法+推送更早在做） | 离开手机10分钟就焦躁 |
| 创造 | 从白纸开始生成 | AI替你写第一稿、想第一个方案 | "从零开始"时长时间空白 |
| 判断 | 在不确定中做决策 | AI替你做推荐、排序、取舍 | 面对开放选项时依赖"AI推荐" |

**Step 3 评估位移深度：** 对每种子能力，判断位移处于哪个阶段：

| 阶段 | 特征 | 举例 |
|---|---|---|
| 辅助期 | AI提供参考，用户仍做最终决策 | 导航建议两条路线，用户选 |
| 替代期 | AI默认执行，用户偶尔干预 | Planner直接导航到最优路线 |
| 依赖期 | 用户丧失脱离AI完成该任务的能力 | 离开导航无法到达熟悉的地方 |
| 焦虑期 | 用户意识到依赖并开始不安 | "我是不是太依赖AI了？" |

当用户群体大规模进入"焦虑期"时，"刻意训练"市场就到了。

**Step 4 设计"器械"：** 对进入焦虑期的子能力，设计"去掉用处、只留动作"的训练产品。设计原则：
- 去结果化：训练的目的是过程本身，不是达成某个外部目标
- 可量化：有明确的训练指标（时长、正确率、完成度）
- 有反馈：训练效果必须可感知（这是健脑房比健身房难的地方）
- 有社交：排名、竞技、分享——借鉴健身房的社交激励机制

### 框架2：哑铃原理——解耦设计的检验标准

"剥离掉原来的用处，只留下动作本身"不只是健身原理，更是一种通用的解耦设计哲学。

**在产品设计中的应用：**

一个好的工具定义/API设计/模块划分，应该像哑铃一样——把"做什么"和"怎么做"彻底分开。检验标准：

| 检验维度 | 哑铃式（好） | 粘连式（差） |
|---|---|---|
| 接口 | "播放音乐(歌名)" | "打开QQ音乐App然后搜索歌名然后点播放" |
| 工具描述 | 描述能力边界，不描述实现路径 | 把调用步骤写进描述 |
| 模块职责 | 一个模块做一件事 | 一个模块同时理解意图+调用工具+生成话术 |
| 测试 | 可以脱离上下游独立测试 | 必须整条链路跑通才能测 |

**在SP维护中的应用：**

SP里的工具tips和参考示例，本质上就是"训练器械"——它们的目的是让模型"练会"一种能力（正确使用工具），而不是在运行时真的被执行。检验一条示例是否好用的标准：

- 哑铃式：示例展示了"用这个工具解决问题"的推理过程，模型看完能迁移到新问题
- 粘连式：示例只展示了特定场景的特定答案，模型只能pattern match，换个问法就不会了

### 框架3：认知位移度评估——产品自检清单

用五区框架评估你的AI产品对用户认知的影响：

**自检问题：**

1. **记忆维度：** 你的产品让用户不需要记住哪些信息？这些信息中，有哪些是"记住了对用户有独立价值"的？（例：常去餐厅的地址 vs 昨天的导航参数——前者记住有社交价值，后者记住无意义）
2. **推理维度：** 你的产品跳过了用户的哪些推导步骤？用户是否能看到推导过程？（例：Planner直接输出action_list vs 展示"我先查了天气→发现要下雨→建议带伞"的推理链）
3. **专注维度：** 你的产品是否增加了打断用户的频率？（例：Advisor主动推送建议 vs 等用户问）
4. **创造维度：** 你的产品是否替用户做了"从零开始"的工作？（例：AI主动生成行程安排 vs 提供素材让用户自己规划）
5. **判断维度：** 你的产品是否减少了用户需要做选择的次数？（例：自动选最优路线 vs 展示三条路线让用户选）

**评估矩阵：**

对每个维度打分（1=几乎不替代，5=完全替代），然后判断：
- 得分高的维度 = 你的产品在此处创造的用户价值大，但也意味着用户在此处的能力退化风险高
- 得分高且该能力"记住了有独立价值"= 需要考虑是否保留一些用户参与

## 操作 SOP：对任意AI产品做"认知位移审计"

**输入：** 一个AI产品（或功能模块）的用户使用场景列表

**步骤：**

1. 列出该产品的Top 10高频使用场景
2. 对每个场景，写出"没有AI时用户怎么做"（旧动作流）
3. 对每个场景，写出"有AI后用户怎么做"（新动作流）
4. 对比新旧动作流，标注被AI接管的步骤
5. 将被接管的步骤按五区分类（记忆/推理/专注/创造/判断）
6. 对每个子能力的位移程度打分（辅助→替代→依赖→焦虑）
7. 识别处于"替代→依赖"过渡期的能力——这些是最可能引发用户反弹的点
8. 对可能引发反弹的能力，设计"保留认知参与"的产品方案

**输出：** 一份"认知位移报告"，包含：场景列表、位移热力图、高风险能力标注、产品建议

**失败模式：**
- 场景列举太抽象（"帮用户导航"而非"用户说'去昨天那个饭店'→Planner查记忆→获取地址→自动导航"）
- 只关注"AI替代了什么"，不关注"替代后用户感受如何"——位移本身不是问题，用户因位移而丧失有价值的能力才是问题
- 把所有替代都视为正面（效率提升），不考虑长期依赖风险
- 混淆"减少无意义的重复劳动"和"剥夺有价值的认知过程"——前者是好的位移（AI替你记密码），后者需要警惕（AI替你做路线判断导致用户丧失空间感）
- 把分析停在"发现了高位移维度"就结束，没有给出具体的"保留认知参与"的产品方案

**怎么判断做对了：**
- 能具体到"XX场景下用户的推理步骤被跳过了3步"这个粒度
- 能区分"替代了无聊的重复劳动"和"替代了有价值的认知过程"
- 报告能指导出至少一个具体的产品设计决策——不是泛泛的"要注意"，而是"在XX场景把自动执行改为展示两个选项让用户选"
- 团队内能用"位移度"这个词快速对齐讨论——"这个功能的判断维度位移度太高了，我们要不要加一个确认步骤？"

## Prompt 模板

### 用"位移模型"分析一个AI功能的认知影响

```text
我正在评估一个AI功能对用户认知的影响，请帮我做"认知位移审计"。

功能描述：[具体描述这个功能做什么]

请分析：
1. 没有这个功能时，用户完成同样目标的完整动作流（每一步）
2. 有了这个功能后，用户的动作流变成什么样
3. 被AI接管的步骤涉及哪些认知子能力（记忆/推理/专注/创造/判断）
4. 每种被接管的认知子能力目前处于什么阶段（辅助/替代/依赖/焦虑）
5. 哪些接管是"好的"（减少无意义负担），哪些可能导致"不好的退化"（用户丧失有价值的能力）
6. 对"不好的退化"，建议保留多少用户认知参与，以及如何在产品设计中实现
```

### 用"哑铃原理"检验一个工具描述/API定义

```text
请用"哑铃原理"检验以下工具描述/API定义是否做到了良好的解耦：

[粘贴工具描述]

哑铃原理：好的工具定义应该"剥离掉实现细节，只留下能力接口"。

请检查：
1. 描述中是否混入了实现路径？（应该只描述"能做什么"，不描述"怎么做"）
2. 边界是否清晰？（能做什么 + 不能做什么）
3. 是否可以脱离上下游独立理解？
4. 示例是否展示了推理过程（可迁移）还是只展示了特定答案（只能pattern match）？
```

## 检查清单

### 读完一篇"AI对XX的影响"类文章后的快速诊断

- [ ] 文章是否识别了具体的"标配→选配"位移？（还是只泛泛说"AI会改变XX"）
- [ ] 位移的量化证据是什么？（有没有类似"25%→5%"的数据）
- [ ] 文章讨论的是"辅助期"的位移还是"替代期/依赖期"的位移？
- [ ] 文章假设了位移的速度——这个假设是否合理？
- [ ] 文章提出的"应对方案"是"保持旧能力"（健脑房逻辑）还是"培养新能力"（人机协作逻辑）？
- [ ] 哪些认知子能力被提到了？哪些被遗漏了？
- [ ] 文章的结论在"AI做得比人差"和"AI做得比人好"两种情况下是否都成立？

## 判断标准

### 做对了的信号

- 你能用"位移模型"在3分钟内分析一个新AI功能对用户认知的影响，并给出"保留/替代/增强"的产品建议
- 你在设计产品时会主动问"这个功能替代了用户的哪种认知？这种替代是好事吗？"
- 你能区分"减少无意义的认知负担"（好）和"剥夺有价值的认知参与"（需要谨慎）
- 在写SP工具描述时，你会用"哑铃原理"检查是否做到了接口和实现的分离
- 在评估badcase时，你会考虑"是不是Planner替代得太彻底，以至于用户不理解为什么这样做"

### 仍然停留在表层的信号

- 读完文章只记住了"AI时代要保持思考"这个鸡汤结论
- 把"标配→选配"当成一个比喻而非一个可操作的诊断框架
- 只在"AI会不会让人变笨"的层面讨论，没有转化为产品设计决策
- 在设计产品时只关注"如何让AI替代更多"，从不考虑"是否应该保留一些用户认知参与"
- 用"哑铃原理"只是作为一个优雅的说法，没有真正用来检验工具描述的解耦程度
