核心逻辑：豆包每次工作时，不是只听到用户说的话。她同时会收到3类动态输入（用户说的话、专家建议、工具返回）+5份背景资料（舱内画面、任务清单、车辆状态、用户档案、动态专业知识）。

「动态专业知识」就是你 Excel 里 I 列写的内容。系统判断当前是什么场景（比如儿童娱乐），就把对应行的知识塞进这里，告诉豆包"现在该怎么服务"。

二、完整案例走查：「后排有点吵」→ 安抚儿童
案例背景
场景：用户开车带两个孩子出门，老二坐在后排左，不肯坐安全座椅，开始哭闹。用户（主驾）说"后面有点吵"。
目标：通过这个案例，完整展示豆包是怎么一步步确认信息、调用工具、生成回复的。
对话开始前的状态（Pre-state）
• 车辆状态：行驶中，车速 45km/h，后排屏幕关闭，空调 24℃
• 舱内视觉：主驾（用户），后排左（儿童，正在扭动哭闹），后排右（儿童，安静）
• 用户记忆：家里有老二（3岁，喜欢恐龙）、老三（1岁）
• 目标队列：空
• 已加载知识：无
• 动态专业知识（来自 Excel I 列）：儿童娱乐场景知识
  - 场景要点：后排儿童哭闹/无聊时，可播放视频/音乐安抚
  - 执行逻辑：确认儿童位置 → 确认内容偏好 → 检查后排屏幕 → 播放 → 调节环境（空调/灯光）

舱内视觉感知（她"看到"的）
主驾：用户；后排左：儿童（3岁左右），正在扭动、嘴巴张开；后排右：婴儿（1岁），安静。
车辆状态
车速 45km/h（行驶中），后排屏幕关闭，空调 24℃，媒体未播放。
动态专业知识（来自 Excel I 列）— 这是关键
系统判断当前场景为「儿童娱乐」，把 Excel 里 DRV-01-01-01 的知识塞进来了：
• 场景：后排儿童哭闹/无聊时，可通过播放视频分散注意力
• 注意：需先确认儿童位置和内容偏好；行驶中前排不能播放视频；后排无屏时需降级为音频
豆包的思考过程
1. 用户说"后面有点吵" + 视觉看到后排左儿童在扭动哭闹 → 推断是儿童不舒服/无聊
2. 动态专业知识说"儿童哭闹可播放视频安抚"
3. 但我现在还不知道：用户是否需要我帮忙？儿童想看什么？后排有屏幕吗？
4. 决策：先询问确认，不直接执行。因为直接放视频可能打断用户处理，也可能放错内容。
{ "talk_or_not": "yes", "talk_content": "[轻声说]是小朋友在闹吗？需要我帮忙安抚一下吗？", "action_list": [] }
这轮结束后的 State Update
• 已知信息 += {后排有儿童哭闹, 说话人是主驾, 车速45km/h（行驶中）}
• 缺失信息 -= {位置已确认: 后排左}
• 仍缺失：是否需要安抚？儿童想看什么？后排是否有屏幕？

1. 用户确认了：是老二（3岁）在闹，原因是不肯坐安全座椅

三、Excel I 列的内容，在这个案例里起什么作用？

上面的案例中，红色高亮框标注的「动态专业知识」就是来自你 Excel 里 I 列的内容。

如果 I 列是单轮示例（现在的情况）：
内容可能是：「用户说安抚儿童 → 系统播放动画片」
结果：豆包只会背这一句，用户说"后面有点吵"它就懵了，因为它没学过怎么从"有点吵"推断出"需要安抚"。

如果 I 列是多轮示例（你要改成的）：
内容应该包含上面案例的完整流程：
1. 听到模糊表达（"后面有点吵"）→ 先确认是不是儿童在闹
2. 确认后 → 询问内容偏好，同时查硬件（后排是否有屏）
3. 信息齐全 → 执行播放（并行打开屏幕+搜视频）
4. 目标存续期间 → 继续处理子任务（调空调）
5. 边界情况 → 无屏时降级为音频，行驶中前排禁视频

这就是多轮示例的价值：不是给豆包一个"标准答案"，而是给它一套"见招拆招的思维链"，让它学会信息怎么一步步确认、工具什么时候查什么时候做、意外情况怎么降级。

四、从这个案例里提炼的 6 条写作规则

1. 每轮必须有 State Update（状态更新）
案例里每一轮结束后，我都写了「已知信息 += ...」。这是最关键的部分，教模型这轮之后知道了什么新东西。如果某轮结束后状态没变，这轮就是废话，删掉。
2. 先查后做，不要假设
案例第2轮，豆包没有直接播放视频，而是先调用 search_vehicle_status_info 查"后排是否有屏"。查完确认有才执行。多轮示例必须展示这种"先查后做"的链路。
3. 信息不全时，询问而不是瞎猜
第1轮用户只说"后面有点吵"，豆包没有直接放视频，而是问"是小朋友在闹吗？需要我帮忙吗？"。这对应 prompt 里的原则：信息不足时不虚构，宁可问也不要猜错。
4. 目标存续期间，对话围绕目标推进
第3轮创建了目标「安抚后排儿童老二」，第4轮用户说"有点热"，豆包知道这个目标还在，所以调空调是"安抚目标下的子任务"，不是独立的新场景。示例要展示这种目标关联。
5. 能并行就并行，减少用户等待
第3轮同时做了三件事：打开后排屏幕 + 搜恐龙视频 + 创建目标。这三个互不依赖，一起发最快。示例要展示 action_list 里有多个工具的情况。
6. 必须覆盖边界情况（降级/拒绝/异常）
案例中专门加了一个「如果后排无屏」的分支：行驶中前排不能放视频 → 降级为音频。这种异常路径必须在示例里出现，否则豆包只学了"顺风局"，遇到意外就崩了。
