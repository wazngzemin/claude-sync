先说整体

这张图画的事情很简单：一个请求进来，或者一个传感器的值变了，经过哪些模块、做了什么判断、最后车子动了或者豆包说话了。从左到右就是这个过程。

6.22之后定下来的是三方架构：

谁的在哪跑管什么
字节云端服务器⑨意图判断 → ⑬规则配置 → ⑮触发器引擎 → ⑯⑰Advisor → ⑲⑳Director → ㉒工具
字节端侧AIBox①②语音输入 → ③④信号库 → ⑤⑥白名单 → ㉔端侧触发器 → ㉕感知器
赛力斯CDC底软起步安全、迎宾、充电大师这些固定功能，直接读信号直接执行

图中间那条虚线就是端云分界线。每条穿过它的橙色粗线都走网络，弱网会卡。

四条铁律（6.22会上定的）
  1. 动态注册必须经⑮云端触发器中转
  2. 端侧的结果先写④信号库，再同步到㉑Context，不能直推、不能跳级
  3. 触发器只读④信号库，不碰③原始信号源
  4. condition写到㉑Context让Advisor去读；event由⑮组装好了push给Advisor。这条是6.22最关键的结论。
一
信号从哪来
①② 语音

用户说了句话，①拿到原始音频，②在端侧做ASR转成文字，然后L2这条橙色线把文字送到云端⑨去判断意图。

③④ 非语音信号

车上除了语音，还有6类信号一直在产生：

类别具体的
端状态四门两盖、车速、档位、座椅、安全带、上下电、充电状态
视觉VLM端模输出的人物动作、物品位置
埋点App上的点击、滑动
电子围栏进出某个区域
音频音量、切歌、循环
时间定时器、日期

这些信号全部通过L8写进④端侧信号库，统一格式存储：key、value、source、timestamp、ttl。

6.22新加的一条原则：端模只报客观状态。输出"坛子当前在地上"，不说"坛子掉了"。掉没掉是上层逻辑判断的事。
④信号库往哪走

信号库有三条出路：

L9（橙色粗线）同步到⑫云端信号库，这是全量同步。L15给㉔端侧触发器读，做本地判定。还有一条隐含的路是往㉑Context上报，供Advisor和Director做决策。

铁律③管的就是这里：触发器只读④信号库，不许直接去听③原始信号源。

二
用户说了句话，走哪条路

文字到了云端⑨，第一件事是分流。四条路：

连线什么情况去哪
L3（橙色）明确的车控指令，比如"打开车窗"回端侧⑤白名单
L4带条件的话，"10分钟后关加热"⑩意图模型解析
L5复杂需求，"帮我照看宝宝"⑲Director汇入
L6跟车无关的，拒掉㉚日志
⑤⑥ 白名单这条路

⑤是端侧的函数映射表，"打开车窗"这种直接映射API。⑥判断白名单里有没有：有就走L7直接⑦执行⑧出卡片，全程不过大模型，延迟很低。没有就回云端⑲走完整推理。

举个例子："把车窗打开" → ASR → ⑨判断是车控 → L3回端 → ⑤命中 → ⑦开窗。用户体感就是说完窗就开了。
三
条件任务怎么注册进来

用户说"超25度开空调"、"到公司了提醒我拿外卖"这类带条件的话，⑨分到第二条路，走⑩→⑬→⑮。

⑩ 意图模型

⑩的工作就是把一句自然语言拆成condition加action。"超25度开空调"拆出来就是{温度>25, 开空调}。拆完通过L11送到⑬。

⑬ 火山后台

⑬是触发器的管理后台。里面有6个API（signals、actions、scene、update、query、status），有一套DSL（Condition、Option、Action、Param、ExecuteInfo），场景定义里包含静态条件、动态条件、执行动作、优先级1-10、作用范围、执行策略。频控默认是总100次、每天100、每次上电10、间隔60秒。

注册来源有五种：用户App、产品预设、运营配置、静态Advisor、Planner。

配好之后L13送到⑮触发器引擎。

四
⑮ 云端触发器引擎

⑮是整个触发器系统的核心，6.22之后又多了一个身份：事件组装中心。

内部是一条流水线，a到j：

a.场景注册
→
b.信号扫描
→
c.规则判定
→
d.模型推理
→
e.条件达成?
→
f.抖动过滤
→
g.冲突仲裁
→
h.端云分类
→
i.场景下发
→
j.事件组装+push

a到i跟之前一样：收场景、扫信号（5秒一轮或信号更新时扫）、规则判定（温度>45这种死规则）、模型推理（VLM输出的非结构化信息）、判断条件是不是全达成了、过滤数值抖动（车速在125-135之间晃不重复触发）、多场景冲突按优先级仲裁、分端云、下发。

重点讲j，这是6.22最关键的变更。

之前的理解（错的）

Advisor自己去轮询Context，看到状态变了，自己判断算不算一个事件

6.22纠正的

⑮来做状态变化检测和事件组装，组装好了push给Advisor。Advisor就等着接。

具体怎么做的：⑮持续监听⑫云端信号库的快照，发现某个值从F变成了T（比如档位从R变成P），就把这个变化组装成一个语义化的事件"倒车完成"，然后push给订阅了这个事件的Advisor。

⑮组装完事件往四个方向输出：

连线去哪干什么
L17（粗线）⑯静态Advisorpush事件
L18（粗线）⑰动态Advisorpush事件
L21（虚线）⑲Director条件已触发的通知
L14（橙色粗线）㉔端侧触发器场景下发到端侧执行
佳杰在会上说的："我不会去轮询你的Context。我注册了事件之后，就等待你的云端触发器给我发这个事件。"
铁律④怎么理解

佳杰举了一个例子把这事讲清楚了：

佳杰："倒完车之后，如果车里有小朋友，就把儿童锁打开。"

这句话里三个成分：

哪个部分是什么数据怎么来
"用户倒完车"事件等⑮检测到档位变化，组装成事件push过来
"车里有小朋友"condition去读㉑Context里的乘客信息
"把儿童锁打开"actionAdvisor综合判断后给出的建议

事件是推过来的，condition是自己去读的。这两个东西的获取方式不一样，6.22之前我们没区分清楚。

五
Context、Advisor、Director
㉑ Context

Context是所有决策的事实依据，6.22定的规矩是：只存condition，不存event。

什么是condition？持续存在的状态。"车里有一名儿童"、"当前车速80"、"坛子在地上"。什么是event？状态变化那一瞬间的通知。"用户刚睡醒了"、"车辆刚驶出地库"。后者不写Context，由⑮push。

L26：㉑→⑲，Context注入Director，包括情景信息、端状态、用户记忆。

⑯ 静态Advisor

4个常驻的：舒适、出行、情感、内容。一直在。

6.22之后它们的工作方式：从⑮接收push事件（比如"用户上车了"），同时去Context读condition（"车内32度"），综合判断完给⑲Director发advice。L19这条线。

举个例子：⑮ push "用户上车"事件，同时Context里写着"车内32度"。舒适Advisor综合这两个信息，给Director发advice："建议开空调24度、打开座椅通风"。
⑰ 动态Advisor

一个目标建一个实例，完成了就删。有上限：长时任务最多2个、持续GUI 1个、条件订阅6个、定时（含VLM）6个。

跟触发器之间有一套接口（P0）：订阅/取消、callback回调、事件触发、能力描述、文本转事件类型、优先级、兜底。

6.22也是一样：订阅事件等push，不轮询Context。

L20：⑰→⑲发advice。L22：⑰→㉕下发感知订阅。L23：㉕→⑰回传结果。

⑱ GoalList

Director和Advisor之间的共享任务板。L24注入⑲，⑳通过goal_list_update建目标删目标，⑱给⑰建监工实例。

⑲⑳ Director

⑲是汇入口，6路输入都在这里汇合：用户的话、两种Advisor的建议、工具返回的tool_feedback、⑮的触发通知、GoalList的目标。再加上L26注入的Context，经过前置筛选（挑相关工具、示例、knowhow），打包成prompt送到⑳。

⑳是Seed 1.8，SP大概30K tokens：角色定义、23个工具、69个示例、D6X车辆知识、"温柔高知女生"的风格设定、加上动态模板变量。

推理完输出三样东西：要不要说话(talk_or_not)、说什么(talk_content)、做什么(action_list)。

输出往五个方向走：

连线去哪什么内容
L27㉒工具库action送到23个下游工具执行
L28㉓任务中心条件/定时任务变成Widget卡片
L29（橙色）㉖TTStalk_content送到端侧播报
L30（橙色）㉗㉘车控action送到端侧执行
L33（虚线）⑮触发器对话中产生的条件任务回注册
六
VLM和VQA两条视觉链路

视觉这块有两条路，容易混，拆开讲。

链路一：VLM，端侧持续跑

㉔端侧触发器给㉕感知器发运行指令，㉕调度摄像头抓帧，端侧VLM（跑在AIBox上）持续出文本描述。这些描述通过L16回写到④信号库，再通过L9同步到⑫云端信号库。⑮监听⑫的变化，一旦发现阈值穿越就组装成事件push给Advisor。

㉔ → 指令 → ㉕ → 摄像头抓帧 → 端侧VLM → 客观状态文本 → ④ → L9 → ⑫ → ⑮ 检测穿越 → push Advisor
举个例子：端模每3秒输出"副驾女性，睡眠状态"。一直输出，一直没变化，⑮不管。某一次变成了"清醒状态"，⑮检测到穿越，push "副驾醒了"给舒适Advisor。
链路二：VQA，云端按需问

VQA不是持续跑的，是⑰动态Advisor需要确认某个细节的时候才调一次。⑰生成一个prompt，通过视频通话接口下发抓帧指令到端侧（L_V3），端侧摄像头拍一帧传回云端（L_V4），云端VQA模型拿图加prompt做推理，结果回给⑰（L_V5），同时注入㉑Context（L_V6）。

举个例子：⑰收到"毯子位置变化"的事件，但它不确定具体什么情况，就生成prompt"毯子是否掉地上了？宝宝还盖着吗？"，调一次VQA，拿到"毯子在地板上，宝宝身上没有覆盖物"，确认了，再给Director发advice。
七
㉔ 端侧触发器

6.22之后scope收窄了不少。先说内部流水线，也是a到j：

a.接收云端下发
→
b.读信号库快照
→
c.本地规则判定
→
d.优先级P0-P4
→
e.冷却控制
→
f.下发运行指令
→
g.VLM业务
→
h.终态
→
i.回写信号库
→
j.动作执行

几个关键点：P0级别的任务（下车安全、VQA、遗留物检测）不冷却，其他的高频3秒低频5秒。互斥组分VLM_CABIN（舱内）和FRONT（舱外），同一时刻一个互斥组里只跑一个任务。下发运行指令有7个字段：task_id、task_goal、priority、timeout、mutex_group、trigger_source、expire_at。

i这步有个循环：VLM识别结果通过L16回写信号库，写进去之后可能触发新的判定，形成二次触发。j是端本地执行车控或TTS，弱网的时候㉔可以独立干活。

6.22 scope收窄

6.22最大的变化：一批功能从端侧触发器划给了CDC底软。仁杰给了一个判定标准：

归CDC底软

  • 常驻的、固定的、标配的、纯key-value判定
  • 起步安全、上下车安全
  • 睡眠提醒、无感迎宾
  • 充电大师、哨兵模式

留在端侧触发器

  • 有生命周期的、动态的、可运营的、临时的
  • 云端下发的定时/条件任务
  • 动态视觉感知事件
  • 通用视觉通道（兜底长尾）
仁杰的原话："像起步安全，你放在触发器里面专门去配一下，反而还要让触发器去接底层信号、考虑稳定性和延时，不如直接在底软就把事做了。"
弱网怎么办

车进隧道断网了，㉔可以独立执行。比如云端之前已经下发了"到目的地500米提醒我"，㉔本地定时器到时，直接走㉖TTS说"快到目的地啦"，不需要云端参与。

八
CDC底软做什么

6.22之后CDC底软接了一大块功能，全是赛力斯那边的：

功能为什么归CDC
起步安全所有车标配，纯key-value
上下车安全固定规则，不需要动态配置
睡眠提醒系统设置里的button，常驻
无感迎宾系统feature
充电大师赛力斯自己的业务逻辑
哨兵模式赛力斯端侧闭环

数据流很简单：直读④端侧信号库，直接执行㉘车控或㉖TTS。不走触发器引擎，不走Advisor，不走Director。

西钺的原话："不要开发两个触发器，不管从开发工作还是部署资源上都是浪费。端侧触发器只在CDC，VLM在AI Box。"
九
6.22新加的几个模块
任务中心（端侧）

佳杰的依赖。端侧需要一个持久化的任务库，要区分任务级和感知级字段。用户看到的是"帮我照看宝宝"（任务级），底层注册的是"观测毯子位置"（感知级）。用户在任务中心UI上点了disable，端侧要真的把对应任务关掉。端侧任务库跟㉓云端任务中心要双向同步。

AIBox任务调度

优先级排序在佳杰的AIBox里做，不在触发器。always-on、VQA、其他视觉需求，优先级低的排队等。

通用视觉事件通道

兜底长尾用的"万能通道"。预设事件集不够用的时候，透传动态query给AIBox。分三类：舱内静态（性别、年龄这些长时间不变的）、舱内动态（动作、表情这些快速变化的）、舱外（只用前摄）。

一个要注意的点：prompt是增量追加的，不是覆盖。加了新的观测项，原来的感知不能丢。

⊕
卡点和回环
3个卡点，7月中之前必须定
① SP下发路径
动态视觉任务的prompt（"看到毯子就上报"），走触发器下发给端模，还是绕开触发器直接给？还没定。
② 端模条件式注意力
端模能不能做到"看到X才上报"？丁彬说通过prompt应该能限制住，但还没试过。
③ 视觉判定放哪
端模自己判断"毯子掉了"，还是只报"坛子在地上"让上层判？结论是只报客观状态，但工程链路没定。
6条关键回环

图上不是单向的，有好几个地方形成了闭环：

连线从哪到哪什么意思
L32㉒工具 → ⑲Director工具执行完了把结果反馈回来，Director二次推理
L33⑳Director → ⑮触发器对话过程中产生了条件任务，注册回触发器
L16㉔VLM → ④信号库识别结果写回信号库，可能触发新的判定
L19/L20⑯⑰ → ⑲DirectorAdvisor的建议回到大脑，这是主动服务的核心
L9④端信号库 → ⑫云信号库端云信号同步
L17/L18⑮ → ⑯⑰信号变化组装成事件push过去，6.22定的
