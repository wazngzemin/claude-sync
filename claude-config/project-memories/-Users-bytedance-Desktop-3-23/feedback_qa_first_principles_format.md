用户 2026-07-01 明确的讲解稿格式要求（v8 起生效，见 触发器讲解稿-人话终版-v8.html）：

1. **追问式 Q&A**：每节标题是一个真实会被问的问题（"为啥做在端侧？""确定不是云端吗？""我还没懂高频低频"），回答开头先给结论，然后表格论证。模仿终端对话里被用户称赞"讲的很透彻"的那种风格。
2. **第一性原理**：每个设计决策必须回答"到底解决什么问题"，用"没有它会怎样"反证（列替代方案逐个证死路），不接受"惯例如此"。
3. **对抗式审查（交付前必做）**：写完切换挑剔审查者视角，列出素材中的矛盾/模糊点（3-6处），逐条给证据A/证据B/核对结论，作为文档最后一节交付。
4. **真实 case 带入**：用用户提供的需求表（xlsx）逐条织进讲解，含原话使用场景 + 触发器运算符映射（等于/变更为/无变更为有/有变更为无）。
5. 原话引用标注来源会议（会①②③色块标记），素材来源表放文档开头供核对。

**Why:** 用户要拿这些讲解稿内化 + 对外讲链路，矛盾点不提前标出来会在对外场合翻车；Q&A 式比平铺目录式更贴合他理解方式。

**How to apply:** 之后所有 讲解稿/逻辑梳理类 HTML 都用这个骨架；版本仍走 人话终版-vN 递增。相关：[[fork-internalization-format]]、[[html-numbering]]
