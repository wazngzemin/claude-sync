## 0. 先纠正两个认知（很重要，影响你怎么定位这条需求）

**纠正① 这是上汽 D6X（DNA1.0），不是赛力斯。**
PRD 标题就是 `【DNA1.0】..._D6X_`，场景都带"第1/2/3批"交付批次 → **这是当期在推的上汽产品，不是"赛力斯阶段、不排期"那一档。** 我之前面聊手册按"赛力斯不排期"定调是错的。
→ **行动**：跟航哥/吕壮确认——机器人触发器接入是不是要进上汽 610 节奏？如果是，需求4 要从"收料归档"升级为"当期排期"。

**纠正② 这个机器人没有摄像头，它对触发器主要是「执行端（动作输出）」，不是「感知输入」。**
PRD V3.0 明确"取消摄像头配置及相关功能"。所以它几乎不产生"它看到了什么"这类感知事件。它的本体能力是 **屏（表情/配饰/背景）+ 水平电机（转向）+ 复用座舱麦克风/扬声器**。
→ 对触发器而言：**机器人 ≈ 一个可被调用的「动作执行端」**，触发器/Planner 判断条件满足后，让它做表情、转头、播报。

## 1. 一句话定性：机器人和触发器的三种关系

| 关系 | 说明 | 对应你哪条需求 |
|------|------|----------------|
| **① 机器人当「执行端」（主）** | 触发器/Planner 决策后，调机器人原子能力（表情/电机/形象）执行 | 喂 **需求9 端云分工**（端侧执行能力） |
| **② 触发条件大量来自「车辆信号」** | 机器人 27 个场景里，一半以上由车辆信号触发（智驾/泊车/充电/疲劳…） | 并进 **需求1 条件清单 + 信号库** |
| **③ 机器人自身产生少量「输入事件」** | 听声辨位（说话人方位）、外力扭头检测、唤醒位置 | 需登记为新信号源 |

## 5. 机器人作为「端侧执行端」——喂给需求9（端云分工）

机器人怎么被驱动，正好是需求9 端云分工的一个具体样本：

```
云端(Planner/Advisor)：决策"现在该演哪个场景"（上车欢迎/天气/节日…）
      └─▶ 下发"调用哪些原子能力"（表情标签 + 电机标签 + 配饰/背景 + TTS文案）
            └─▶ 端侧：机器人本体执行（转电机 + 渲染屏幕形象 + 播报）

端侧自闭环（不绕云，实时）：
  • 听声辨位转向（唤醒后转向说话人，要快，24°/快速聚焦）
  • 多人交互转向状态机（见 §6）
  • 外力扭头的即时反馈
```

**给需求9 的两条原则输入：**
1. **「演什么」在云、「怎么转怎么显」在端**——和你需求9"云端注册+仲裁、端侧判断+执行"完全一致。机器人是这条原则最干净的例子。
2. **听声辨位/多人转向必须端侧闭环**——毫秒级反应，绕云会卡，正是需求9"简单/实时任务端侧直接闭环"要支撑的。

## 6. 一个被低估的需求：多人交互转向状态机（端侧）

PRD V11.0 新增的"多人交互转向策略"其实是**一个端侧状态机需求**，值得单拎出来给研发：

| 场景 | 声源 | AIBOT 表现 |
|------|------|------------|
| 首次触发 | 本轮第一说话者(A) | 转向 A，**且仅转一次** |
| 同位追问 | A 持续 | 保持不动（不重复转） |
| 异位插话 | 出现第二个不同位置(B/C/D) | **转向正中间**并保持到本轮结束 |
| 唤醒打断 | 任一位置说"豆包豆包" | 强制结束本轮、转向唤醒位、规则重置 |
| 新轮首句 | 新一轮首个发声者 | 规则重新生效，转向他 |

> 这是"一轮对话内的有状态逻辑"，不是简单的条件→动作。**归端侧（实时、不绕云）**，和需求5 触发机制的"状态/时机"概念相关。建议作为需求4 的一个子项单独排。
