深度讲义 1/5：系统全景与入口分流

这份只讲两件事：① 整套系统的骨架是什么、为什么这样搭；② 一句话进来后怎么被分流的——三级拒识的每一级怎么判、句法RAG和情景RAG各自的输入输出格式、仲裁模型的判定逻辑、端云协同超时降级的完整链路。每个细节都讲到你能给技术团队复述的程度。

一、整套系统的骨架：三层漏斗 + 目标驱动层

1.1 为什么是三层漏斗

车载场景对时延的要求比手机/PC 严苛得多。用户说"打开车窗"，期望的是瞬间响应——如果等了两秒窗户才动，用户会觉得"这车是不是坏了"。但用户说"帮我规划一个周末去莫干山的自驾行程，中间要充电和休息"，这种请求天然需要时间思考，用户能接受等几秒。

问题在于：用一个大模型处理所有请求，简单的也要等大模型跑完（至少500ms），这对"打开车窗"是不可接受的；但只用规则引擎，复杂的处理不了。所以系统被设计成一个三层漏斗——越简单的请求越早被拦截、用越快越轻的方式处理；只有真正需要"思考"的请求才进入大模型。

这个思路的原文表述是三条设计原则：

"(1) 层层过滤，能通过简单方法搞定的不要漏到后面复杂模型去处理；(2) 在不同阶段选择合适的节点注入知识，注入的方式也是从简单（固定）到复杂（泛化）；(3) 通过用户数据回流逐步把后面复杂任务的处理推到前面知识注入环节。"

第三条尤其重要——它意味着这个漏斗会自己进化。Planner 处理的那些难 case 会被回流成训练数据，反过来训练前面的句法 RAG 和情景 RAG，让它们能接住更多的请求。随着时间推移，进入 Planner 的流量会越来越少，系统整体会越来越快。

1.2 三层各自是什么

第一层：句法 RAG

句法 RAG 是最轻最快的一层，本质是规则匹配 + 工程直接下发，不涉及任何大模型。它处理的是"明确的单轮 query"——用户说的话能直接映射到一个确定的动作，不需要任何推理。

输入：单轮 query 文本。

处理：规则匹配（类似正则/模板匹配），判断这句话是否在已知的指令集里。

输出（原文给出了完整的 JSON 格式）：

{
  "type": "directExecute",        // 直接执行，不需要模型
  "reject_ornot": "react/reject", // react=放行执行，reject=拒识
  "talk_content": "",             // 要说的话（通常为空，直接执行不说话）
  "task_list": [...]              // 要执行的任务列表
}

时延：毫秒级。因为是规则匹配没有模型推理，几乎瞬间完成。

能处理什么：如"打开车窗""空调调到25度""关闭氛围灯"这类明确的单指令。

不能处理什么："我好热"（模糊，需要结合车况判断是调空调还是回话）、"帮我规划周末行程"（复杂，需要多步推理）。

拒识能力：句法 RAG 也能做拒识。比如用户说"妈妈帮我拿一下水杯"，规则匹配发现这不是车机能执行的指令 → 输出 "reject_ornot": "reject"。

用户说"打开主驾车窗"→ 句法 RAG 规则匹配命中 → 输出 {"type":"directExecute","reject_ornot":"react","talk_content":"","task_list":[{"action":"打开主驾车窗","tool":"vehicle_basic_control"}]} → 直接下发给车控执行 → 车窗打开 → 全程不到 200ms，用户几乎感觉不到延迟。

第二层：情景 RAG + 仲裁模型

句法 RAG 处理不了的 query（模糊的、复杂的）会漏到第二层。这一层做两件事：① 情景 RAG 注入场景知识 ② 仲裁模型判断简单/复杂。

情景 RAG 的作用不是直接给答案，而是给仲裁模型和 Director "补课"——把当前场景下可能用到的知识以 few-shot 的形式注入，帮模型理解"这种情况应该怎么处理"。

情景 RAG 的输出（原文格式）：

{
  "type": "modelExecute",          // 需要模型处理
  "knowledge": "...",              // 注入的场景知识（few-shot 示例）
  "agentLimit": [...]              // 能力边界约束
}

注意 type 的区别：句法 RAG 输出 "directExecute"（直接执行），情景 RAG 输出 "modelExecute"（交给模型）。这就是两层的分水岭。

仲裁模型接收情景 RAG 注入了知识的 query，做一个核心判断：这个 query 简单还是复杂？

  • 简单（"未拒识 + 仲裁为车控 + 命中6大意图"）→ 交给云侧 FC 模型快速生成指令下发，走快车道（FC.SFT 解析 300ms → 端控执行 150ms）
  • 复杂（上述条件不同时成立）→ 交给 Director/Planner 做深度推理，走慢思考（思考 500ms + 回答 500ms）

仲裁模型的选型：Seed1.6 flash SFT。为什么用这么轻的模型？因为它只需要做二分类（简单 or 复杂），不需要深度推理能力。用大模型做这件事太浪费——它的推理时延就是白花的等待。轻模型 + 情景 RAG 注入的 few-shot，足以在低时延内做出准确的路由判断。

用户说"我好热啊"。句法 RAG 匹配不到直接指令（"我好热"不是"打开空调"这种明确表达）→ 漏到第二层。情景 RAG 注入当前车况知识：{"knowledge":"当车内温度>28度且空调关闭时，'我好热'类表达通常意味着需要开空调降温；当空调已开到最冷时，可能只是在抱怨。","agentLimit":[...]}。仲裁模型拿到这个 few-shot 后判断：如果当前空调关闭 → 简单车控 → 快车道；如果空调已开到最冷 → 需要思考怎么回应 → 复杂 → Director。
仲裁模型为什么经常判错？原文分析了三大根因：

① 输入太单一——仲裁模型只能从 query 和注入的 knowledge 判断，但同一句话因车况不同结论完全不同。"我好热"在空调没开时和空调已开到最冷时是两回事，如果情景 RAG 没注入正确的车况 knowledge，仲裁就会判错。

② 靠造数据学习，case 造不完——仲裁的 SFT 训练数据需要人工标注"这句话是简单还是复杂"，但真实场景无穷无尽，标注永远追不上长尾。原文说这"仍然是 NLU 的人肉解 bug 方法"。

③ 意图处理信息量太少——仲裁只对粗 domain 分类（如"这是车控类 query"），但车控下面有上千个具体函数（开哪个窗？调哪个座椅？调到多少度？），粗分类后交给 FC 模型从上千函数里选，效果远差于少量函数的情况。原文用了一个很好的类比："人说到热，就只会在开窗、空调、加热几个选项里思考解决，不会想到车控这个大类然后在1千个范围内搜索。"

第三层：Director / Planner

这是漏斗的最底层，也是最"重"的一层——云端大模型深度推理。它接收的都是前两层处理不了的复杂请求。Director 的详细机制在下一份讲义（第2份）里展开，这里只讲它在漏斗中的角色定位。

Director 有双重身份（原文）：

  1. 兜底——前面全部漏过来的、任何模块都接不住的 query，Director 兜底处理。哪怕它处理得慢一点，至少不会让用户"完全没有回应"。
  2. 云端 teacher——Director 处理完的 case 不是用完就扔，而是回流成训练数据，用来训练前面的句法 RAG、情景 RAG 和仲裁模型。这就是第三设计原则"数据回流"的落地——前期 Director 作为 teacher 引导数据回流，中后期用积累的数据训练，让漏斗自己变聪明。

原文对这个设计的评价是："这是一个过渡方案。当前的大模型技术水平达不到能兼顾速度和对复杂任务的处理，所以用漏斗分层。如果后面模型能力提升起来了，把漏斗的下层提到前面去就行了——工作量非常小，是一个可以持续演进的架构。"

二、三级拒识：每一级怎么判、判的依据是什么

2.1 整体架构：为什么是三级

拒识的目的是把不该由车机处理的 query 拦住——比如"妈妈帮我拿水杯"（不是对车说的）、"世界毁灭吧"（无意义的）、"帮我叫个滴滴"（车机能力范围外的）。但拒识本身也有漏判和误判：漏判（该拒的没拒）会浪费 Director 的推理资源；误判（不该拒的拒了）会让用户觉得"车没反应"。所以系统用三级冗余：三道关各有不同的判断视角，一道漏了还有下一道。

2.2 第一级：端侧白名单 + 离线 fc 模型

位置：在端侧（车机本地），不上云。

组成：一个离线的 function call 小模型 + 一张白名单（已知的高频车控指令集合）。

判定逻辑（原文完整规则）：

  1. query 进来后，离线 fc 模型对它做 function call 匹配
  2. 如果匹配结果在白名单里 → 端侧直接执行，不上云，约 150ms 完成
  3. 如果匹配结果不在白名单里 → 优先云端结果：同时上报给云端处理
  4. 如果云端超时无结果 → 看端侧有没有结果：有 → 采纳端侧；无 → 做超时播报兜底

白名单里有什么：简单高频的车控指令——"打开车窗""关闭空调""座椅加热3档""打开阅读灯"这类。白名单由车厂可配置，不需要改模型。

为什么要白名单而不是让 fc 模型自己判断：因为 fc 模型可能把模糊表达也匹配到某个车控函数上（比如把"我好热"匹配成"打开空调"），但这不一定是对的。白名单起到双保险的作用——即使 fc 模型匹配了，也要在白名单里才端侧直接执行，否则交给云端做更精确的判断。

端云协同超时降级链路（完整时序）：

T=0ms：用户说"帮我查下油耗"→ ASR 转文字
T=200ms：端侧 fc 模型匹配 → 匹配到 search_vehicle_status_info（查车辆状态）→ 不在白名单里（白名单只有简单车控如开关窗/调温度，不包含查询类）
T=200ms：同时上报给云端
T=200~700ms：等云端结果
情况A：云端 500ms 内返回了仲裁结果 → 按云端结果走（简单→FC快车道 / 复杂→Director）
情况B：云端超时了（比如弱网）→ 回看端侧 fc 匹配结果：端侧匹配到了 search_vehicle_status_info → 采纳端侧结果，执行查询
情况C：云端超时 + 端侧也没匹配到 → 超时播报兜底（系统说"抱歉没听清，能再说一次吗"）

2.3 第二级：云侧前置拒识

位置：云端，在 Director 之前。

组成：云端拒识模型（和仲裁模型配合工作）。

判定逻辑（原文）：

  1. 端侧白名单没命中的 query 上到云端
  2. 云端拒识模型判断：这句话是否该由车机处理？
  3. 如果该拒识（不是对车说的、非车机能力范围的）→ 拒识。但拒识不是丢弃——拒识的内容会做两件事：
    • 写入 context 事件日志
    • 被 advisor 感知（advisor 会扫描事件日志，发现有用信息时触发主动服务）
  4. 如果不拒识（是合理的车机请求）→ 进入仲裁判定（简单/复杂分流）
  5. 如果拒识超时（拒识模型没来得及返回结果）→ 交给 Director 处理（宁可让 Director 多花点时间判断，也不能因为拒识模型慢就让用户没有响应）
为什么拒识不丢弃而是写入 context？这是一个精妙的设计。比如用户说"妈妈帮我拿一下水杯"——车机确实做不了，应该拒识。但这句话透露了一个信息："车里可能有小孩"。拒识的内容写入事件日志后，情感 advisor 扫描到这条日志，可能触发主动服务——比如判断"车内有儿童，当前温度偏高"→ 给 Director 建议"调低温度、注意儿童舒适"。所以拒识的 query 不是没用的，只是不该由 Director 直接处理。

2.4 第三级：Director 内部兜底拒识

位置：在 Director 的推理流程内部，plan_describe 阶段之前。

判定逻辑（原文）：

"在 Plan Describe 阶段之前，引入拒识思考。content 直接输出 reject，不进行后续的拆解等其他逻辑。若是，则直接输出 reject，结束回复，不再进入后续 plan 流程。"

这是最后一道防线。即使前两道拒识都没拦住，Director 在开始拆解任务之前还会再想一下"这个 query 我到底该不该处理"。如果此时判定不该处理 → 直接输出 reject，不浪费后续的工具调用资源。

原文也承认这一级的问题："该方式存在一些缺点，包括格式输出不符合规定等，暂时没有想到好的解决办法。"——也就是说 Director 偶尔会在本该 reject 的时候输出格式不对的内容（既不是正常 JSON 也不是干净的 reject），这是已知 bug。

2.5 云端仲裁的完整判定树

过了拒识的 query 进入仲裁。完整规则（原文逐条）：

条件结果走哪条路
离线 fc 命中白名单端侧直接执行不上云，约 150ms
离线 fc 未命中白名单上云，优先云端结果云端超时→端上→兜底（三级降级）
云端未拒识 + 仲裁为车控 + 6大意图直接给 fc 模型处理简单快车道：FC.SFT 300ms→端控 150ms
上述条件不同时成立给 Director 处理复杂慢思考：思考 500ms + 回答 500ms
云端拒识超时给 Director 处理兜底（宁可慢不可无响应）
云端拒识成功写入 context 事件日志，被 advisor 感知不走 Director，可能触发主动服务

注意"未拒识 + 仲裁为车控 + 6大意图"这三个条件必须同时满足才走简单快车道。任何一个不满足就交给 Director。这个设计的逻辑是宁可让复杂的走慢路（多花几百ms），不可让简单的走错路（执行错误的指令）。

三、时延预算：每一段花多少时间

这是优化性能时的"标尺"——每一段的预算是多少，超了就是问题。

段时延说明
ASR（语音转文字）200ms端侧处理
NLU（自然语言理解）300ms端侧初步理解
端侧 fc + 白名单命中~0ms（规则匹配）命中后端侧执行 150ms
云端拒识 + 仲裁含在 Planner 首 token 内—
FC.SFT 解析（简单快车道）300ms云侧轻量 FC 模型
端控执行150ms车控指令下发到执行
Planner 思考（复杂慢思考）500msDirector 第一次推理
Planner 回答（含工具调用后二次推理）500ms总结/继续/结束
TTS 生成200ms文字转语音

总目标（原文）：

  • P50 首 token ≤ 500ms（50% 的请求在 500ms 内出第一个字）
  • P90 首 token ≤ 800ms（90% 的请求在 800ms 内出第一个字）
  • 推理 ACC ≥ 90%（推理正确率）
  • 端状态回复 ACC ≥ 93%（查车辆状态的回复正确率）

SP 优化前后的实测数据（来自 Sp 优化专项文档）：

指标优化前（Baseline）优化后（PE3-0415）变化
首包时延 P502309 ms998 ms降 57%
首包时延 P903439 ms1317 ms降 62%
总耗时 P5012231 ms6179 ms降 49%
拒识+待确认 ACC79.38%88.38%升 9pp

优化前首包时延 P50 = 2309ms 意味着一半的请求要等 2.3 秒才出第一个字，这在车载场景完全不可接受。优化后降到 998ms（不到 1 秒），接近了 500ms 的目标但还没完全达到。

四、Badcase 排查：入口分流出了问题怎么查

当一个 case 的结果不对时，第一步永远是确认它走了哪条路。排查路径：

  1. 先看 reject_ornot 字段：query 是被 reject 了还是被 react（放行）了？
    • 被 reject 了但不该被 reject → 哪一级拒识拒错了？（端侧白名单误拒？云侧拒识模型误拒？Director 内部误拒？）
    • 该 reject 但被 react 了 → 三级拒识都没拦住，说明这类 query 在训练数据/规则里缺覆盖
  2. 再看仲裁的简单/复杂分类：
    • 复杂的被判成简单 → 走了快车道，FC 模型从上千函数里选，大概率选错或执行不全 → 回看情景 RAG 注入的 knowledge 是否缺失
    • 简单的被判成复杂 → 走了 Director 慢思考，时延爆炸但结果可能是对的 → 性能问题不是正确性问题
  3. 看端云协同降级：
    • 云端超时 → 降级到端侧结果 → 端侧结果对吗？（端侧 fc 模型匹配的准确度本身就低于云端）
    • 端侧白名单该命中没命中（白名单覆盖不全）/ 不该命中却命中了（白名单过宽）→ 找白名单的配置问题
