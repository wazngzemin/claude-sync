给佳杰讲 SP 拼接 · 为什么必须触发器做

2026-07-03 · 沟通对象：佳杰（动态Advisor/常驻任务 owner）· 目的：让佳杰理解并认可"SP 拼接由触发器负责"

讲之前先对齐一件事：你面对佳杰，不是要说服他"你该干活"。你要让他看到——他的 Advisor 有一个下不去端模的问题，而你的触发器恰好能帮他解决。站在他的利益上讲。

一、讲法策略：从佳杰的痛点入手，不要从你的方案入手

佳杰关心的不是"触发器怎么拼字符串"，而是：
① 我的 Advisor 说了"看毯子"，端模怎么才能看？
② 端模看了之后结果怎么回到我的 Advisor？
③ 如果有三个 Advisor 同时要看不同的东西，端模怎么都顾上？

所以讲法是：先用一个 case 把他的问题暴露出来，然后自然引出触发器是解法。

二、逐字话术（可以照着念）

2.1 开场：抛问题，不讲方案

你说："佳杰，我想跟你对齐一个事——你的常驻任务要让端模去看东西，这个链路现在通不通？我举个例子。"
你说："比如你有一个常驻任务，用户说了'帮我看后排宝宝毯子掉没掉'。你的 Advisor 知道要看，但问题来了——你怎么让端模去看？你直接调 AIBox 吗？"
预判他的回答：佳杰大概率会说"我订阅你触发器的事件啊"或者"丁彬那边的接口"。
关键点：不管他说什么，你要把他引到下面这个矛盾上——

2.2 核心矛盾：三个 Advisor 同时要看，端模只有一个窗口

你说："好，假设现在同时有三个 Advisor 各自要看不同的东西："
Advisor它想看什么它给端模的指令
Advisor A后排宝宝毯子掉没掉"看后排毯子在不在宝宝身上"
Advisor B副驾有没有打电话"看副驾有没有手持手机"
Advisor C主驾有没有闭眼"看主驾眼睛闭合状态"
你说："端模 5 秒一个窗口，一次只吃一个 SP。这三条指令谁来合并成一个 SP 喂给端模？"
这就是要让佳杰意识到的矛盾：
Advisor A 不知道 B 和 C 存在（互相看不见）→ A 没法把 B 和 C 的指令拼进自己的 SP
端模自己不能合并（它只吃 SP，不知道哪条来自谁）
必须有一个"全局角色"看得到所有 Advisor 的需求，合并成一个 SP，再拆开返回。

2.3 解法：这个全局角色就是触发器

你说："所以我这边触发器来做这件事。逻辑很简单，就三个动作——"
#动作干什么给佳杰的白话
1合并追加三个 Advisor 各给我一句话，我按槽位拼到 always-on SP 的尾部"你给我人话，我往端模 SP 里追加一行"
2对位读取端模返回一坨结果，我按预定好的 key 把每个 Advisor 要的值捞出来，分发回去"结果回来我帮你挑，你只拿到你要的那条"
3清理删除某个 Advisor 的任务结束了，我把它那行从 SP 里摘掉，还原"你说不看了，我帮你撤"
你说："对你来说，你只管给我一句人话+告诉我你要什么 key 的返回值。拼接、下发、拆结果、清理——全是我的活。你不用碰端模的 SP 格式，也不用知道还有几个 Advisor 在跑。"

2.4 回应"为什么 Advisor 自己不能拼"

佳杰可能会问：为什么不是我自己拼？或者兰若 6.30 那个方案——Advisor 自己拼、端模自己拼？

你说："兰若 6.30 一开始也想让 Advisor 自己拼，后来她自己推翻了。原因就一句话：端模返回的是一坨混在一起的结果，Advisor 自己挑不出自己要的那部分。"
你说："而且技术上，端模一次只吃一个 SP。三个 Advisor 如果各自拼各自发，端模吃谁的？吃了 A 的就漏了 B 和 C。只有一个人做合并，才能保证都喂进去。丁彬也说了：'谁拼的谁才知道从哪去拿。'"
一句话总结给佳杰：
"你的 Advisor 负责说'看什么'和'看到了怎么办'。我的触发器负责'怎么让端模去看'和'把结果分发回你'。中间拼 SP 这件事，必须在我这——因为只有我看得到所有 Advisor 的需求，能做全局合并。"

三、你和佳杰需要对齐的接口协议

讲完"为什么是我做"之后，佳杰下一个问题一定是"那我给你什么、你给我什么"。提前准备好：

3.1 Advisor → 触发器（注册观测需求）

字段含义示例
advisor_id哪个 Advisor 发的"advisor-007"
observe_query人话描述"看什么""后排毯子是否在宝宝身上"
camera_group用哪组摄像头CABIN（舱内）
return_keyAdvisor 要的返回值 key"rear_blanket_status"
lifecycle持续型 / 一次性持续型（用户说停才停）
对佳杰说："你就给我这五个字段。SP 怎么拼、端模怎么调、slot 怎么定——都是我的事，你不用管。"

3.2 触发器 → Advisor（返回结果）

字段含义示例
advisor_id发给谁"advisor-007"
return_key对应注册时的 key"rear_blanket_status"
value端模输出的值"毯子在地上"
event_type状态（持续）/ 事件（穿越那一刻）event（从"盖着"变成"在地上"）
timestamp发生时间1751529600
对佳杰说："你收到的就是干净的、只属于你这条任务的结果。不用从一坨混合信息里自己挑。"

四、全链路走一遍（讲给佳杰看的版本）

用"三个 Advisor 同时要看"的场景走一遍，让佳杰看清触发器在中间干了什么。

步谁干什么佳杰的视角
1Advisor A/B/C分别向触发器注册观测需求：
A: "看后排毯子" key=blanket_status
B: "看副驾手机" key=copilot_phone
C: "看主驾闭眼" key=driver_eyes		我只管提需求
2触发器合并：把三条人话按槽位拼入 always-on SP 尾部：
[slot-A] 请描述后排毯子位置状态
[slot-B] 请描述副驾是否手持手机
[slot-C] 请描述主驾眼睛闭合状态		我看不到这一步
（也不需要看到）
3触发器 → 端模拼好的完整 SP 下发到 AIBox 端模
4端模每 5 秒按 SP 输出一坨混合结果：
{slot-A: "毯子在宝宝身上", slot-B: "副驾右手持手机", slot-C: "主驾眼睛睁开"}
5触发器拆分发：
• 按 key=blanket_status 捞 slot-A 的值 → 发给 Advisor A
• 按 key=copilot_phone 捞 slot-B 的值 → 发给 Advisor B
• 按 key=driver_eyes 捞 slot-C 的值 → 发给 Advisor C		我只收到我要的那条
6触发器阈值穿越：如果 slot-A 从"在宝宝身上"变成"在地上"→ 组装事件 push 给 A我收到"毯子刚掉了"事件
7Advisor A收到事件 → 推理 → advice 给 Director我拿到事件就开始干活
画重点给佳杰看：
步骤 2-5 是触发器的活。佳杰的 Advisor 只出现在第 1 步（提需求）和第 7 步（收结果）。
中间"怎么拼、怎么喂端模、怎么拆结果"——全部透明，佳杰不用关心。

五、佳杰可能的反问 + 你的回答

佳杰可能问你的回答
"SP 的模板谁定？你能定吗？""模板是丁彬/兰若定的，端模能理解什么格式他们说了算。我不生成 prompt、不理解语义——我只做字符串追加、按 key 读值、任务结束删除。纯机械操作。"
"你拼的东西端模能理解吗？""能不能理解是丁彬要保证的事。我和丁彬对好 SP 模板格式（哪个位置放什么 slot），然后我严格按模板往里填你给我的人话。我不改写、不润色、不判断语义。"
"如果我只有一个 Advisor 要看，还需要走你吗？""一样走。因为即使只有一个，你的指令也需要拼进 always-on 的 SP 里——always-on SP 是一直在跑的，你不能覆盖它，只能追加。而且将来随时可能来第二个、第三个，提前走通不返工。"
"我能不能自己直接调 AIBox？""技术上能调，但你调了之后结果是一坨混合的——你怎么从里面挑出你要的？而且你调的时候不知道别的 Advisor 在不在跑，你可能把别人的 SP 覆盖掉。兰若 6.30 就是想到这一步才推翻了'Advisor 自己拼'的方案。"
"那你对我的依赖是什么？""我对你的依赖就一个：你告诉我看什么、要什么 key 的返回值。五个字段，很轻。我不需要你改任何现有逻辑。"

六、一页纸总结（可以截图发给佳杰）

SP 拼接分工：一句话版

角色职责不做什么
动态 Advisor
（佳杰）		说"看什么" + 定 return_key + 收结果 + 推理不碰 SP 格式、不调 AIBox、不拆结果
触发器
（泽民）		合并多条需求 → 按模板拼 SP → 下发端模 → 拆结果按 key 分发 → 阈值穿越 → 事件 push → 任务结束清理不理解语义、不生成 prompt、不判断结果对不对
端模/AIBox
（丁彬）		定 SP 模板格式 + 吃 SP + 按 slot 输出结果不知道有几个 Advisor、不管合并
为什么只能触发器做合并：
① Advisor 互相看不见 → 没法合并别人的指令
② 端模一次只吃一个 SP → 三个 Advisor 各自发 = 覆盖冲突
③ 返回结果是混合的 → Advisor 自己挑不出自己的
④ 触发器是唯一一个"看得到所有需求 + 看得到所有结果"的全局角色

兰若 6.30 结论（她自己推翻了"Advisor 自己拼"）："只能触发器那边合并成一个任务。"
丁彬结论："谁拼谁才知道从哪去拿。"
