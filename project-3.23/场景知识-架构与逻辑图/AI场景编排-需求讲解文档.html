一、这个需求是什么？

各位好，我先花5分钟讲清楚这个需求到底是什么、为什么要做。

1.1 一句话定义

AI 场景编排 = 把上汽原来"按钮点击 → 固定规则执行"的预制场景，升级为"用户说人话 → Planner 理解意图 → 动态知识检索 → 智能编排车控"的 AI 原生方式。

1.2 背景

4月16日徐汇，Daniel 提出：车上现在有 9-10 个预制场景（宠物模式、洗车模式、小憩模式等），用户要点按钮或者说精确的模式名才能触发。能不能让豆包直接听人话就能理解？比如用户说"我车里有只猫，我要临时下车"，豆包就知道这是要开宠物模式。

1.3 新旧模式对比

❌ 旧模式（预制场景）

  • 入口：情景空间 APP / 场景大师 APP / 桌面卡片（多个按钮入口）
  • 触发：用户手动点按钮，或者必须说准确的"打开XX模式"
  • 执行：写死的规则化逻辑，每个模式 10-20 条固定执行项
  • 退出：手动点退出 / 满足自动退出条件（如车速>15km/h）
  • 回滚：退出后所有设置项自动恢复到进入前的状态
  • 持续监控：运行期间实时监测异常（高压异常/SOC低/档位变化），自动退出+APP短信通知
  • ✅ 优点：确定性100%，安全有完整兜底
  • ❌ 缺点：不灵活，不理解自然语言，不能结合环境个性化编排

✅ 新模式（AI场景编排）

  • 入口：唯一入口 = 语音（APP页面只做引导，不是真正入口）
  • 触发：用户自然语言表达（"我想休息一下"/"车里有只猫帮我看着"）
  • 执行：Planner 理解意图 → 匹配场景知识 → 动态编排车控组合
  • 退出：仅支持语音说"退出"（无自动退出监控）
  • 回滚：无回滚，仅恢复到座椅记忆位置
  • 持续监控：暂无（需Adviser+触发器配合，未建）
  • ✅ 优点：理解自然语言，结合环境个性化编排，泛化query覆盖
  • ❌ 缺点：不能100%保证执行正确，无回滚，无持续监控
关键认知：新模式不是完全替代旧模式，而是两条路径并存——对于执行项100%确定、有安全边界的场景（如宠物/洗车），走旧的预制方式更安全；对于需要结合环境灵活编排的场景（如小憩/观影/调温），走新的 Planner 编排更灵活。

二、完整链路讲解：一个请求从进来到出去经过了什么？

好，接下来我拿两个真实案例，一步一步走给大家看，每一步系统里到底发生了什么。大家可以对着那张架构图看。
🔴 案例 A（分子级路径）：用户说 "我车里有只猫，我要临时下车，帮我照顾一下"
🟢 案例 B（编排级路径）：用户说 "我好热好闷，赶紧帮我降降温通通风"

Step 1：用户语音输入

对应架构图：LAYER 1 · 用户输入

用户开口说话，这是整条链路的起点。唯一入口就是语音——6/4 会上和小明确认过，所有预制场景的按钮入口后续都会删掉，APP 页面只做引导（放提示词教用户怎么说），不是真正的入口。

语音有两种形式我们都要支持：

  • 泛化型：用户不知道有宠物模式，只说需求——"我车里有只猫帮我照顾一下"
  • 直接型：用户知道模式名——"打开宠物模式"

语音输入会带上附加信息：谁说的（speaker_name）、坐在哪个位置（position）、时间戳（timestamp）。这些对后面的场景分类很重要——比如主驾说"我热"和副驾说"我热"，要调的是不同区域的空调。

Step 2：ASR 语音转文字

对应架构图：LAYER 2 · ASR

ASR = Automatic Speech Recognition，语音转文字。这一步在端侧（车上本地）做，速度快，不依赖网络。

转完之后，文字 + speaker信息 一起从端侧上传到云端。在架构图上，这是 L2 橙线，方向是端→云。

两个案例到这一步：

  • 🔴 案例A：文字 = "我车里有只猫我要临时下车帮我照顾一下"
  • 🟢 案例B：文字 = "我好热好闷赶紧帮我降降温通通风"

Step 3：拒识仲裁——第一个分流器

对应架构图：LAYER 3 · 路由分流（菱形决策⑨）

这是整条链路最关键的第一个分流器，叫拒识仲裁意图模型。"拒识"是行业黑话，意思是判断"这句话该不该接、归谁接"。

它会把用户这句话分成四条路：

路径判定条件去向举例
① 简单车控 (L3)明确的单一车控指令端侧直接执行，不进大脑，毫秒级"打开车窗""调到25度"
② 条件话术 (L4)带条件/定时的话意图解析⑩ → 触发器注册⑬"十分钟后关座椅加热"
③ 复杂/场景类 (L5) ★复杂的、需要理解意图的Selector SP 场景分类 → 这是我们的主线"我好热""帮我照顾猫"
④ 拒识 (L6)不该接的话存到 Context 事件日志自言自语、无意义的话

两个案例到这一步：

  • 🔴 案例A："猫...照顾" → 不是简单车控，不是条件话术 → 判定为 ③ 复杂/场景类
  • 🟢 案例B："好热好闷...降温通风" → 有体感+多诉求 → 判定为 ③ 复杂/场景类

两个案例都走 L5 主线，进入 Selector SP。

Step 4：Selector SP 场景分类——第二个分流器（我负责的核心模块）

对应架构图：LAYER 4 · 场景分类

这是我（泽民）负责的核心模块。Selector SP 是一个分类 prompt，它的任务很简单：输入一句 query，输出一个场景 ID。它不是对话任务，纯粹做分类。

分类逻辑（4步）

  1. ❶ ID_000 预判过滤：先排除掉不走场景的 query——闲聊/问好/纯控制指令/推荐咨询/问路导航/单设备操控 → 直接输出 ID_000
  2. ❷ 上下文辅助：query 本身关键词不明确时，结合 speaker_position、env_info 辅助判断。比如副驾+座椅不适 → ID_006 副驾调节
  3. ❸ 场景特殊规则：观影场景的前排/后排判定、副驾座椅的严格限定、情绪+温度的归属等
  4. ❹ 互斥锚定：最后一步确认归属。比如儿童相关的 ID_001/ID_009/ID_013 三个怎么区分：陪孩子玩→001，孩子困了要睡→009，孩子睡了别打扰→013

两个案例的分类过程

🔴 案例A：
❶ 不是闲聊/问路/纯控制 → 继续
❷ 无特殊上下文 → 继续
❸ 无特殊规则命中 → 继续
❹ 互斥4.5："猫""照顾" → 输出 ID_011（宠物关怀）
→ ID_011 是分子级场景 → 走预置应用！
🟢 案例B：
❶ 有体感词"热""闷"，不是纯控制 → 继续
❷ 无特殊上下文 → 继续
❸ 命中4.3："热" → ID_004(降温)，"闷/通风" → ID_012(通风)
❹ 多场景命中 → 输出 ID_004,ID_012
→ 编排级场景 → 走 Planner！
⚠ 当前问题：Selector SP v11 的命中率只有 60-70%，意味着约三分之一的 query 被错分或漏分。目标是提到 ≥85%。

★ 关键分流点：分子级路径 vs 编排级路径

对应架构图：分流点

Selector SP 输出场景 ID 后，系统根据 ID 走两条完全不同的路径：

  • 分子级场景（ID_011 宠物、洗车、露营、离车保电、大床、一键躺平）→ 不注入 Skill 给 Planner，直接调用预置应用
  • 编排级场景（ID_004 降温、ID_012 通风、以及观影/小憩/冥想/关怀/副驾/雨天）→ 检索 Skill 知识注入 Planner，由 Planner 动态编排

为什么要分？因为 6/4 上午和小明确认过：Planner 做不了 100% 保证。宠物模式有 15 条固定执行项必须全部准确执行（包括高压维持、空调维持、屏蔽按键等安全操作），Planner 有幻觉风险，可能调不存在的功能。所以执行项 100% 确定的场景必须走预置应用。

Step 5A：分子级路径（以案例A"宠物模式"为例）

对应架构图：LAYER 5A · 分子级

第一步：不注入 Skill，直接调方向靠

Selector 命中 ID_011 后，系统识别出这是分子级场景，不会把 Skill 知识注入 Planner，而是直接通过方向靠（车控协议层）下发 Function Call："开启宠物模式"。

⚠ 当前卡点：方向靠车控协议里还没有注册宠物模式等 6 个预置场景的指令！需要我整理 query 示例提给 左晨。这是我的待办。

第二步：预置应用执行前置条件校验

预置应用收到指令后，先检查前置条件：

前置条件校验:
  P档        → ✓ 满足
  电量 ≥ 20% → ✓ 满足（当前68%）
  高压正常    → ✓ 满足
  整车电源    → ✓ run状态
→ 全部通过 → 弹出确认弹窗

第三步：二次确认弹窗

弹窗内容：空调温度设置（默认22°C，用户可编辑）→ 用户点"确认" → 正式进入宠物模式。

第四步：执行 15 条固定项（全部写死的规则化逻辑）

宠物模式执行项（共15条，全部确定性100%）:
 ① 设置弹窗 → 空调温度（仅开启前可编辑，默认22°C）
 ② 确认弹窗 → 用户确认后执行
 ③ 中控屏   → 进入专属占屏画面
                （显示空调温度 + 文案"不用担心，主人马上回来"）
 ④ 后排屏   → 黑屏屏蔽，用户不可操作
 ⑤ 多媒体   → 所有多媒体/视频/小程序 暂停
 ⑥ 车窗     → 关闭所有车窗
 ⑦ 氛围灯   → 关闭
 ⑧ 阅读灯   → 关闭
 ⑨ 空调     → 打开A/C · 温度22°C
 ⑩ 空调循环 → 打开外循环
 ⑪ 语音     → 屏蔽（防止宠物误触发语音助手）
 ⑫ 高压维持 → 整车电源变为off后，仍维持高压
 ⑬ 空调维持 → 整车电源变为off后，仍可维持空调运行
 ⑭ 电动控键 → 外部锁车后屏蔽车内电动控键，解锁后解除
 ⑮ 空调硬开关 → 屏蔽（若有配置）

第五步：运行期持续监控

进入模式后，系统持续监控以下异常：

  • 高压异常 → 车窗通风 + APP消息及短信 + 车机toast提醒
  • SOC < 18% → 提醒退出
  • SOC < 15% → 自动退出 + APP消息+短信通知
  • 挡位非P档 → 自动退出 + toast提醒

第六步：退出 + 回滚

退出方式：屏幕软按键 / 自动退出条件触发。退出后 15 条设置项全部自动回滚至进入前状态。

总结：分子级路径 = 确定性100% + 完整回滚 + 持续监控 + 安全兜底。代价是不灵活、不能理解自然语言（需要方向靠注册精确指令）。

Step 5B：编排级路径（以案例B"降温+通风"为例）

对应架构图：LAYER 5B · 编排级

第一步：检索 Skill 知识

Selector 输出 ID_004(降温) + ID_012(通风) 后，系统去知识库里检索这两个场景对应的 Skill 知识：

检索到的 Skill 知识:

极速降温 Skill (ID_004):
  执行项: 空调ON / 温度调低 / 风量最大 / AC开启 / 外循环
  特殊逻辑: 前1分钟打开车窗 → 1分钟后关窗 → 空调继续
  达到目标温度后 → 风量降挡 → 空调AUTO
  可编辑: 温度(LO~28°C~HI) / 风量(1~11) / 循环模式

快速通风 Skill (ID_012):
  执行项: 打开主驾车窗 + 对角线(右后)车窗 → 快速通风
  特殊逻辑: 如果下雨 → 减小开窗幅度
  可根据实际情况调整通风时长

第二步：获取端状态 + 用户记忆

端状态 env_info:
  车内温度: 32°C    车外温度: 35°C
  档位: P档         SOC: 78%
  PM2.5: 正常       天气: 晴，未下雨
  车窗: 全部关闭    空调: 关闭状态

用户记忆 memory:
  偏好温度: 24°C
  习惯: 喜欢开窗通风，不喜欢太冷
  FaceID: 张三（主驾）

第三步：全部注入 Planner SP

以上 Skill 知识 + env_info + memory + 动态示例（few-shot参考），全部注入到 Planner SP 的 {{knowledge}} 动态模板变量中。此时 Planner 收到的完整输入约 40K tokens。

第四步：Planner 推理（对应 LAYER 6）

Planner 收到这些信息后开始推理，输出三样东西：

Planner 推理输出:

talk_or_not: true    // 需要开口说话
talk_content: "好的，我帮你降降温通通风~先给你开窗换换气"

action_list: [
  {
    tool_name: "vehicle_basic_control",
    param: "空调ON, 温度24°C, 风量8, AC开启"
    // 注意: 温度选了24°C——因为记忆里用户偏好24°C
    // 风量选了8不是最大——因为记忆里用户不喜欢太冷
  },
  {
    tool_name: "vehicle_basic_control",
    param: "主驾车窗打开50%"
  },
  {
    tool_name: "vehicle_basic_control",
    param: "右后车窗打开50%"
    // 对角线通风——Skill知识里写了这个策略
  }
]
这就是 Planner 编排的价值：它结合了 Skill（知道对角线通风策略）+ env_info（知道没下雨可以开窗）+ memory（知道用户喜欢24°C不喜欢太冷），做了个性化编排。传统预制场景做不到这个。

第五步：23个下游工具执行（对应 LAYER 7）

action_list 里的 3 条指令依次下发给工具层：

工具执行:
  ① vehicle_basic_control("空调ON, 24°C, 风量8, AC")
     → SOA车控指令 → 方向靠(端侧) → 执行成功
     → tool_feedback: {status: "success"}

  ② vehicle_basic_control("主驾车窗打开50%")
     → SOA车控指令 → 方向靠(端侧) → 执行成功
     → tool_feedback: {status: "success"}

  ③ vehicle_basic_control("右后车窗打开50%")
     → SOA车控指令 → 方向靠(端侧) → 执行成功
     → tool_feedback: {status: "success"}

3条全部成功 → 不需要二次推理 → 直接输出结果

如果有一条失败了（比如车窗控制不了），tool_feedback 会回传给 Planner，Planner 做二次推理决定怎么处理（这就是架构图上那条虚线回环箭头）。

第六步：呈现层输出（对应 LAYER 8）

  • 🔊 TTS："好的，我帮你降降温通通风~先给你开窗换换气"
  • 📺 AI画布：展示3条执行项（空调已调至24°C / 主驾车窗已打开 / 右后车窗已打开）
  • 🤖 AIBOT表情：切换为服务态，1分钟后自动恢复默认桃子形象
  • 🚗 车控：空调、车窗已实际执行
编排级的短板：
  • ❌ 无回滚——退出后空调和车窗不会恢复到之前的状态，仅恢复座椅记忆位置
  • ❌ 无持续监控——不会像预置场景一样持续检测异常
  • ❌ 无安全弹窗——不会在执行前弹窗让用户确认
  • ❌ 不能100%保证——Planner可能出现幻觉（如调不存在的功能）

W1：Selector SP 维护（场景分类器）

为什么会有这个问题？

用户说一句话，系统第一件事是判断"这句话属于哪个场景"。Selector SP 就是干这个的——输入一句 query，输出一个场景 ID。但当前 v11 的命中率只有 60-70%，意味着用户说"我好热"有 30% 的概率不会被识别为"极速降温"场景，导致场景知识没被检索、Planner 没拿到 Skill、最终只能瞎猜着回答。

前因后果

Selector SP 是我从零写的一个分类 prompt。v11 包含 4 步判定逻辑 + 13 个场景 ID + 35 条参考示例 + 互斥锚定规则。问题是写的时候场景还没合并、测试用例不够、边界 case 没覆盖（比如"我累了"到底走小憩还是冥想？"方向盘冻得握不住开加热"到底算纯控制还是升温场景？）。

怎么做？完整流程

  1. 构建评测集（我做）：每个场景写 ≥20 条 query（正例/反例/边界 case），格式为飞书表格或 CSV： 
    | query                          | 期望输出   | 类型   |
|--------------------------------|-----------|--------|
| 我好热赶紧降降温                  | ID_004    | 正例   |
| 帮我调到25度                     | ID_000    | 反例   |
| 好热好臭赶紧开窗                  | ID_004,ID_012 | 多命中 |
| 方向盘冻得握不住开加热              | ID_000    | 边界   |
  2. 跑评测（我+罗丽）：把评测集里的 query 逐条输入 Selector SP，对比输出 vs 期望，量化每个场景的 precision/recall。
  3. 针对低分场景调 SP（我做）：改判定逻辑、改互斥锚定规则、加参考示例。每改一版跑一次评测集验证。
  4. 提交给明骏部署（我→明骏）：改好的 Selector SP 文本文件发给明骏，他部署到云端分类器模块。

给谁？他拿到什么？

交付物给谁格式他拿到后做什么
Selector SP prompt 文本明骏.md 纯文本部署到云端场景分类器

W4：Skill 知识质量修复（P0 最高优）

为什么会有这个问题？

Skill 是我写给 Planner 的"场景知识"——告诉 Planner"小憩模式应该调哪些车控"。但第一版 Skill 是粗写的，没有和上汽的实际车控能力做对齐。比如我在 Skill 里写了"调用观影模式"这个一级指令，但方向靠车控协议里根本没有这个指令，只有空调、座椅、车窗等原子操作。Planner 拿到我写的 Skill 后真的去调"观影模式"，方向靠收到后不认识这个指令，直接失败。

前因后果

5/23 第一版上线 → 小明实车走查发现大量 badcase（离车保电失败/宠物模式调不通/新手模式罗列操作但没执行）→ 追查根因发现全部指向 Skill 质量 → 6/3 上午会上小明明确说"有好多是车厂不具备的能力，也在里边写" → 必须重写。

怎么做？完整流程

  1. 拿三份参照材料（我去催）：
    • 小明的 bug 走查表（他承诺发群里，还没发）——知道哪些场景出了什么问题
    • 左晨的方向靠 SOA 完整能力清单（约 700 条原子功能，还没提供）——知道车上到底能调什么

  2. 逐个场景对照重写（我做）：拿着上汽 UC 文档，每个场景逐条对：
    • Skill 里写的每一条执行项，SOA 清单里有没有？→ 没有的删掉
    • Skill 里写的前置条件，跟上汽 UC 文档一致吗？→ 不一致的改
    • Skill 里有没有引导 Planner 去调不存在的指令（如"观影模式"一级指令）？→ 改成实际存在的原子操作组合
  3. 重写后的 Skill 交给明骏（我→明骏）：他配置到动态知识检索模块。
  4. 上分支测试（明骏+罗丽）：6/4 已提测到分支，产品上车体验验证。

W5：Skill 合并重写（3 份新替代 7 份旧）

为什么会有这个问题？

6/3 上午会上小明明确要求"尽量合，能少就少，到二级就行"。原因是场景太多会导致 Selector 分类困难（相似场景分不清）、Skill 冗余（内容重复）、Planner 注意力分散。合并后场景数减少，Selector 更容易判准，Planner 收到的 Skill 更聚焦。

怎么做？

  1. 观影合并（我做）：后排观影(ID_002) + 前排观影(ID_003) + 5D观影(ID_007) → 写 1 份新 Skill，内部做分支：
    • 如果 speaker_position=后排 或 query 提到"后排/吸顶屏" → 后排观影逻辑
    • 如果 speaker_position=前排 → 前排观影逻辑（必须P档，这是关键前置条件）
    • 如果 query 提到"5D/沉浸/影院模式" → 5D 逻辑（Planner只管调腾讯视频，时间点联动由腾讯协议控制）
  2. 调温合并（我做）：极速降温(ID_004) + 极速升温(ID_005) → 写 1 份新 Skill，根据 query 中"热/冷"判断升/降。
  3. 用车关怀合并（我做）：儿童陪伴(ID_001) + 后排勿扰(ID_013) → 写 1 份新 Skill，覆盖儿童+大人睡觉等场景。
  4. 同步更新 Selector SP 互斥规则（我做）：步骤 4.1 儿童相关、4.2 观影相关、4.3 温度相关的规则都要改。
  5. 交给明骏配置（我→明骏）。

W8：Query 示例整理 → 提给左晨注册方向靠

为什么会有这个问题？

6 个分子级场景（宠物/洗车/露营/离车保电/大床/一键躺平）不走 Planner，直接调预置应用。但是预置应用是通过方向靠（车控协议层）的 Function Call 触发的，方向靠里目前没有注册这些指令。就好比你想打电话给一个人，但通讯录里没有他的号码——你怎么都打不通。

6/3 上午会上左晨确认："车控车控工具里边是没有的（离车保电等）"。需要我整理 query 示例提给他，他才能在方向靠里注册。

怎么做？

  1. 整理 query 示例（我做）：每个场景写标准 query + 泛化变体 query（至少 5 条）： 
    场景: 宠物模式
标准 query:
  - "打开宠物模式"
  - "开启宠物关怀"
泛化 query:
  - "我车里有只猫，帮我照顾一下"
  - "我要临时下车，车里有只狗"
  - "启动宠物安全锁"
  - "帮我照看一下车里的宠物"
  - "我下车买个东西，狗在车里别热着"
  2. 发给左晨（我→左晨）：飞书表格，每行一个 query，标注触发哪个预置应用。
  3. 左晨注册（左晨做）：在方向靠车控协议里给每个预置场景注册一组语音触发指令，配置 NLU（让它能识别泛化 query）。
  4. 联调测试（左晨+我+测试）：在车上实测，确认用户说这些 query 能正确触发预置应用。

W9：安全红线兜底方案

为什么会有这个问题？

上汽原来的预制场景有完整的安全机制——每个场景进入前都有前置条件校验（必须P档、电量够、高压正常），运行期持续监控异常，异常时自动退出。但 Planner 编排没有这些机制。Planner 是一个 LLM，它"大概率"会遵守 SP 里写的规则，但不能"100%保证"——LLM 有幻觉风险。

6/4 上午小明举了一个致命场景："用户正在开车（D档），说'我想打会盹'，Planner 如果调座椅放平就出大事。"小明原话："哪怕 99.9% 都规避了，有那万分之一，可能就是车毁人亡。"

杨麟 6/4 纪要也确认："字节反馈保底策略仍需要车端完成。"

前因后果

这个问题在旧的预制场景里不存在——因为预制场景在代码层面硬写了前置条件校验，不满足就直接拒绝，不存在"概率性"。切换到 Planner 编排后，判断逻辑从"代码硬写"变成了"LLM 推理"，引入了不确定性。这是 AI 编排 vs 传统规则的本质矛盾。

怎么做？三层兜底设计

  1. 层1 Planner SP 约束（我做，已做）：在 SP 注意事项里加规则"非P档禁止调座椅/禁止放倒靠背/禁止进入小憩模式"。这是"告示牌"——告诉 Planner 不要做，但 Planner 可能"视而不见"。
  2. 层2 Adviser 前置判断（弋途做，部分已做）：Adviser 持续监控端状态（通过 env_info 拿到当前档位），非P档下直接拦截小憩等模式的触发请求。这比层1可靠一些，但 Adviser 本身也是基于模型的，不是代码硬写。
  3. 层3 车控底层写死兜底（车控开发做，待定方案）：在车控底层代码（方向靠/SOA 层）里硬写逻辑："如果当前档位≠P档，且收到的指令=调座椅/放倒靠背 → 直接拒绝执行，返回错误"。这是唯一的确定性兜底——不管上游是 Planner 还是 Adviser 还是手动操作，底层代码层面就不允许这个操作。

怎么推进？

  1. 我整理一份"安全红线清单"（我做）：列出所有"什么条件下禁止什么操作"，比如： 
    安全红线清单:
  非P档 → 禁止调座椅位置
  非P档 → 禁止放倒座椅靠背
  非P档 → 禁止进入小憩/冥想/观影(前排)模式
  行驶中(车速>0) → 禁止主驾屏幕全屏占用
  充电枪已插 → 禁止进入洗车模式
  ...
  2. 发给明骏加 SP 约束（我→明骏）：他在 Planner SP 注意事项里加层1的规则。
  3. 发给车控开发加硬兜底（我→车控开发）：他们在方向靠/SOA 底层代码里加层3的逻辑。
  4. 周五下午沟通（我+明骏+车控开发）：确认层3的可行性和排期。

给谁？

交付物给谁格式他拿到后做什么
安全红线清单明骏飞书文档层1：加到 SP 注意事项
安全红线清单车控开发飞书文档层3：底层代码写死拒绝逻辑

W10：预制场景弹窗整理

为什么会有这个问题？

上汽原来的预制场景在进入前都有二次确认弹窗——比如观影模式会弹窗说"后排有人，座椅将移动，请确保安全"，用户确认后才执行。这是安全设计，防止误操作伤人。

但 Planner 编排是"直接执行"的（6/4 会上确认不需要用户中间确认），没有弹窗环节。小明 6/4 提出："涉及到安全的前置条件弹窗是可以保留的。OMS 识别到后排有小朋友→告诉用户座椅不能调→这样更好。"

杨麟 6/4 纪要第 3 条也明确："分享预制场景进入前的提示弹窗，考虑补充至 Planner 执行前展示。"

前因后果

预制场景的弹窗内容在上汽《AI场景编排评估》文档里有完整记录（每个场景的 UC 文档里都有"二次确认弹窗"的内容描述）。我需要把这些弹窗的触发条件和内容整理出来，作为 Planner 执行前安全提醒的参考。

怎么做？

  2. 我补充 Planner 侧的映射（我做）：哪些弹窗对应 Planner 编排的哪些场景？比如观影弹窗"后排有人注意座椅"→ 对应 Planner 编排观影模式时的前置检查。
  3. 给雪晴设计交互方案（我+小明→雪晴）：雪晴（交互设计）来决定——Planner 执行前是弹弹窗让用户确认？还是 TTS 语音提醒？还是直接在 AI 画布里展示安全提示？
  4. 明骏实现（雪晴出方案→明骏开发）。

W11：上汽预期对齐（生成式卡片/持久化/分享/UI）

为什么会有这个问题？

上汽（杨麟）对 AI 场景编排有一些超出当前 SOP 能力的期望，需要我跟他对齐"哪些能做、哪些放二期"。这些期望来自 5/29 走查会和 6/4 沟通会：

问题拆解（4 个子项）

11a. 生成式卡片展示

  • 杨麟的期望：Planner 编排出来的场景，在画布里不要用现在工程化的纯文字列表展示，要有设计感的生成式卡片——就像原来预制场景那样有沉浸态页面、背景音、专属视觉。
  • 为什么现在做不了：SOP 的 Planner 是动态编排的，每次结果不同，做不了固定的沉浸态页面。需要有一个"生成式页面引擎"，根据 Planner 输出的 action_list 动态生成好看的页面——这是一个独立的前端工程需求。
  • 怎么办：我跟杨麟对齐预期，明确 SOP 内只做画布文字展示 + 卡片聚合（6/9 版本芮雪优化），生成式页面放二期。
  • 找谁：先找 杨麟 对齐预期 → 后续找 芮雪（UI 设计）出生成式方案。

11b. 场景持久化（放入"我的场景"）

  • 杨麟的期望（5/29 纪要 4.1）：用户用 Planner 编排出一个场景后，想保存下来下次直接用，不用每次重新说。放到场景大师 APP 的"我的场景"里。
  • 为什么现在做不了：Planner 是一次性调用，执行完就没了。要持久化就需要——把 action_list 序列化保存 → 建一个"我的场景"数据库 → 用户下次打开直接加载 → 还要支持编辑/删除。这涉及服务端存储+客户端 UI，工作量大。
  • 当前临时方案：编排结果存在记忆里（通过 memory 工具），用户问豆包"我之前创建了什么"可以文字回忆。但"找不到"——小明原话"时间长了我是不知道我到底创了多少了"。
  • 怎么办：Daniel 确认放二期（SOP+3，年底前）。一期先做好单个用户自己的体验。

11c. 场景分享（分享码）

  • 杨麟的期望（5/29 纪要 4.2）：用户觉得自己编排的场景特别好，想分享到车友群——生成一个分享码，其他车主输入分享码就能同步。场景大师 APP 已有一键生成分享码的能力。
  • 为什么现在做不了：依赖场景持久化（11b）先做好。没有存下来的场景就没法分享。
  • 怎么办：Daniel 确认放二期。

11d. 执行项展示 UI 太粗犷

  • 杨麟的期望（5/29 纪要 4.4）：AI 画布里展示的执行项"太工程化"——一堆文字罗列，占好几页要滑动。应该设计更紧凑，一页展完。
  • 为什么现在这样：画布本身是通用的 AI 对话展示框架，不是为场景编排专门设计的。每调一个工具显示一个卡片，调 5 个就 5 个卡片。
  • 怎么办：6/3 晚例会令平确认——车控 UI 卡片聚合展示目标 6/9 版本（把同类车控卡片合并到一个聚合页）。其他优化 芮雪 跟进。

W7：防晕车模式（新增场景）

为什么会有这个问题？

6/4 上午小明提出——防晕车模式比较特殊，它是"一半确定+一半发挥"：动力模式（雪地/舒缓/经济）和能量回收等级必须确定地设置到特定值（这些关乎行驶安全），但空调、开窗、香氛、音乐等舒适项可以让 Planner 结合用户偏好灵活编排。

前因后果

小明问"一半一半 Planner 能不能做？"——泽民回答"链路很难做，车端执行一半+Planner 编排一半，需要研发搭链路"。但可以先写 Skill 测试看效果。

怎么做？

  1. 杨麟补充晕车模式知识（杨麟做，6/4 纪要第 4 条）——告诉我"确定的那一半"具体是什么。
  2. 我写 Skill 先测试（我做）：在 Skill 里把确定部分写死（"动力模式设为舒缓，能量回收设为低"），发挥部分写引导（"根据用户体感调整空调/开窗/香氛/音乐"）。
  3. 上分支测试效果（明骏+我）：看 Planner 能否正确执行"确定部分严格遵守+发挥部分灵活编排"。
  4. 如果效果不行：还是走分子级预置应用（需要研发搭一半确定+一半编排的链路，工作量大，可能放后面做）。

W-补充：AIBOT 表情 loop 结束信号

为什么会有这个问题？

上汽的 AIBOT（车机屏幕上的虚拟机器人）在执行场景时会播放循环表情动画——比如小憩模式时显示"睡觉"表情，宠物模式时显示"照顾"表情。这些表情是循环播放的（loop），从模式开始一直到模式结束。

但 Planner 是一次性调用——它只管"触发开始"，不管"什么时候结束"。所以 Planner 触发后表情开始 loop，但没有人告诉 AIBOT "模式结束了该停了"，导致表情一直循环，直到下一个场景触发新表情才被覆盖。

6/3 下午婉婷在群里明确说："不太能接受实际小憩模式已经关了表情还在 loop。"令平确认："Director 没有给结束逻辑。"

怎么解决的？

婉婷提了两个方案：A. 触发单次 emoji 不循环 B. SOP 前走预制模式。最终决策（6/4 会上泽民+明骏）：方案 A 变体——触发后 1 分钟自动恢复默认桃子形象。不需要退出信号。

  • 6/5 上线：打招呼 + 天气表情
  • 6/12 上线：小憩 + 宠物表情

吕壮的《触发事件清单》还没发（6/1 面聊时承诺的），到手后需要逐条梳理确认。

八、研发怎么开发？（技术实现路径）

最后讲一下研发侧拿到我们的交付物后，他们的开发路径。

8.1 明骏拿到 Skill 后怎么做？

  1. 配置知识库：把每个 Skill 文件录入动态知识检索模块。每个 Skill 绑定一个场景 ID（如 ID_004 绑定"极速降温 Skill"）。
  2. 配置检索逻辑：Selector 输出场景 ID → 用 ID 去知识库检索 → 拿到对应 Skill → 注入 SP 的 {{knowledge}} 变量。
  3. 配置动态示例：每个场景配几个 few-shot 示例（用户输入 → Planner 期望输出的 action_list），帮助 Planner 理解这个场景该怎么编排。
  4. 测试验证：在 Flux 环境上自测 → 提交到测试分支 → 罗丽用评测集验证。
  5. 上线流程：分支验证通过 → 产品上车体验确认 → 合入主线 → 发版。

8.2 左晨拿到 Query 示例后怎么做？

  1. 注册指令：在方向靠车控协议里，给每个预置场景注册一组语音触发指令。
  2. NLU配置：配置 NLU（自然语言理解）模型，让它能识别泛化 query 映射到对应的预置应用。
  3. 联调测试：在车上实测，确认用户说这些 query 能正确触发预置应用。

8.3 整体技术架构（研发视角）

  ASR → 语音转文字 → 上传云端
