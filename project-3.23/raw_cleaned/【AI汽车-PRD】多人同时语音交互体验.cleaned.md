# 【AI汽车-PRD】多人同时语音交互体验

# 整体策略（粗看）

> - 同时说场景
>   - 同时说的定义：任务的FC还没下发/输出，其他人还在和豆包讲话。
>   - 体验：将多个人的对话query合并给到planner处理（不区分具体用户的任务是什么类型），planner内考虑上下文和任务相关性处理。
>     - 合并的文本是：主驾驶说xxxx，副驾驶说xxxx。
> - 任务执行中的场景
>   - 任务执行中的定义：意图FC输出了但是任务没有执行完/播报完成。
>     - 文本/控制任务+控制任务
>       - 控制任务需要NLG播报则打断，打断后恢复，不需要播报的则轻量化反馈（提示音）
>     - 文本+文本
>       - 后面的文本，打断之前的文本回复。
>     - 控制任务+文本任务
>       - 排队任务，等控制任务执行播报完，再进行文本类任务播报。「这类场景可能比较少」
>       - 排队任务：要在界面显示排队信息，让用户可感知。

# 详细定义

## 术语与定义

为了确保沟通的精确性，我们首先对核心术语进行统一定义。

| 术语 | 定义 | 示例 |
|---|---|---|
| **文本任务 (Text Task)** | 指系统的回复以一段完整的、信息承载为主的自然语言播报（NLG）为核心的交互。这类任务通常用于回答问题、提供信息、讲故事等。 | “今天天气怎么样？”、“讲个笑话”、“介绍一下北京故宫” |
| **控制任务 (Control Task)** | 指用户通过语音指令直接操作设备或软件功能的交互。这类任务的核心是执行一个动作，而非获取长篇信息。 | “调高音量”、“打开空调”、“导航去公司” |
| **NLG 播报** | 指由系统生成的，用于语音回复的自然语言内容（Natural Language Generation）。 | “好的，已为您打开空调。”、“今天北京晴，最高温度 25 摄氏度。” |
| **打断 (Interrupt)** | 指在一个正在进行的 NLG 播报或任务执行过程中，被一个新的、更高优先级的用户请求或系统事件所中断的行为。 | 正在播报新闻时，用户说“导航回家”，新闻播报被中断。 |
| **恢复 (Resume)** | 指在“打断”事件处理完毕后，系统能够回到之前被中断的任务或播报流程，并以一种自然的方式继续。 | 导航指令执行后，系统询问“还需要继续收听刚才的新闻吗？”or 系统不询问直接恢复。 |
| **提示音 (Earcon)** | 一种简短、非语言的音效，用于提供即时、轻量的状态反馈，避免打扰当前的主要交互流程。 | 在播放音乐时，用户说“下一首”，系统播放一个轻快的“叮”声以示操作成功。 |
| **排队任务 (Queued Task)** | 当多个任务无法同时执行时，系统将后到的、优先级较低的任务放入一个队列中，待当前任务完成后再依次执行。 | 正在执行“播放音乐”的控制任务时，用户又发起了“讲个笑话”的文本任务，后者将进入队列。 |

## 同时说场景：合并与处理【P0】

当多个用户几乎同时对系统说话时，为了避免指令丢失和混乱，我们采用合并处理的策略。

> **核心思想和规则：** 豆包正在聆听或思考一个任务，有其他人也要发起任务。用户A已说完指令，豆包思考中暂未为下发函数功能或调用工具执行，用户B也说完了指令（VAD end，语音聆听态结束），豆包也即将思考。系统必须将这些指令合并为一条结构化的内部请求，交由 Planner 进行统一处理。

### 合并格式与原则

- **合并格式**：系统应以角色标注的方式，将不同用户的 query 拼接成一个文本字符串。
  - **格式**：“主驾驶说'XXXX'，副驾驶说'YYYY'。”
  - **合并指令：**仅合并有效指令（非拒识query）
  - **合并数量上限**：暂定仅支持4个合并。
- **处理原则**：将合并的指令全部交给Planner进行处理，无需进行简单复杂仲裁。
  - Planner 在收到合并后的请求时，应遵循以下原则进行决策：<text bgcolor="light-yellow">【⚠️这里期望体验，详细信息需planner来整体定义 </text>@用户】
  - **上下文与相关性优先**：Planner**必须** 结合当前的对话历史、车辆状态以及各个指令本身的意图相关性，来判断最终执行哪个任务或如何响应。例如，如果当前正在导航中，那么与导航相关的指令（如“切换路线”）其相关性通常高于一个全新的闲聊指令（如“讲个笑话”）。
  - **角色标注的用途**：角色（如“主驾驶”、“副驾驶”）信息主要用于清晰地表述请求来源，不应该 作为默认的优先级判断依据。除非在特定场景下（如安全相关的车辆控制），产品规则明确定义了角色的优先级。
  - **冲突处理：**如果合并的指令存在明显冲突（如一个说“开窗”，另一个说“关窗”），Planner 应**该** 优先发起澄清式询问，而不是默认执行任何一个。

### 示例

| 场景 | 用户话术 | 系统内部合并 | 系统响应（示例） |
|---|---|---|---|
| 任务互补 | 主驾驶：“导航去最近的加油站。” 副驾驶：“顺便找找附近有充电桩的地方。” | "主驾驶说'导航去最近的加油站'，副驾驶说'顺便找找附近有充电桩的地方'。" | TTS 播报：“好的，为您寻找带有充电桩的加油站。已找到三处，您想去哪一个？” |
| 任务冲突 | 主驾驶：“空调开到最大。” 副驾驶：“有点冷，把空调关了吧。” | "主驾驶说'空调开到最大'，副驾驶说'有点冷，把空调关了吧'。" | TTS 播报：“抱歉，我同时收到了开大空调和关闭空调的指令，请问具体要执行哪一个？” |

### 界面展示

**显示策略：**

识别结果：识别结果【同时】显示主驾驶和副驾驶的指令，主驾驶显示在前，副驾驶显示在后。

**卡片展示：**

执行结果中输出多个结果或多个结果替换显示（依赖Planner的输出顺序）

**特殊场景说明：**

单人 / 多人语音交互场景下，若用户已向系统发送有效指令，且系统处于处理中、尚未完成指令执行与反馈的阶段，此时若接收到拒识指令，**不得打断当前正在处理的有效指令**，需继续完成该有效指令的正常执行与结果下发。

示例：

单人场景：主驾驶下达指令 “给我讲个笑话吧”，系统未输出笑话内容前，主驾驶又说出无意义语音（拒识），系统仍需继续处理并输出笑话内容。

多人合并场景：主驾驶：今天上海天气。 副驾驶：今天北京天气。左后：吧啦吧啦（拒识），系统应继续下发：今天上海和北京的天气
