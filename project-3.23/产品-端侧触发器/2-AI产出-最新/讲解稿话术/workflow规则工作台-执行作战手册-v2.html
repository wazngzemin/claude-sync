第一部分

一、workflow 是什么：30 秒看懂那张图

一句话：让你不写代码，像搭积木一样配出一条"车该在什么情况下做什么"的规则。你那张图配的就是一条——挂进 D 档、还有车窗开着，就语音提醒关窗省电。

图里的区域是什么
左边一竖条积木库：触发 12条件 18动作 9（就是第二部分那三张库）
中间一大块你拼好的规则：Start → 触发（挂 D 档）→ 条件（车窗开着）→ 满足就动作（提醒），不满足就结束
右边一竖条点中某块积木，配它的参数（选 D 档、去抖 300ms、只在点火时触发…）
底下一条自动把整条规则翻译成人话，谁都看得懂
一条规则永远三段：触发什么时候开始看 → 条件还得满足啥 → 动作那就做点啥。这就是韩杰说的"触发+条件+动作"，也是"事件+条件+执行"漏斗。下面所有场景，都是往这三个格子里填东西。

第三部分 · 成品②

三、8 条场景剧本（每条配到底）

每条都按"触发 + 条件 + 动作 + 离线要求"填好了。"离线要求"那栏是精华——它直接决定这条规则该落端侧还是云端（第五部分讲原理）。你把这 8 条录进平台，就是韩杰要的"先怼一批场景"。

① D 档车窗未关提醒 （图里那条）
触发档位 变更为 D（去抖 300ms，仅点火时）
条件车窗 至少 1 个 = 打开
动作TTS 播报"行车中关闭车窗可以降低能耗哦"
离线要（档位、车窗、TTS 都是纯端侧信号，断网也该提醒）→ 落端侧
② 挂 P 档看充电桩（充电管家）
触发档位 变更为 P
条件充电口盖 = 关 且 电量 < 30%
动作触发出行 Advisor 推荐"要不要找个充电桩"
离线不强求（推荐充电桩要查云端 POI）→ 走云端
③ 导航接近目的地提醒
触发导航事件 变更为 接近（距离 < 500m）
条件导航状态 = 导航中 且 POI 类型 = 目的地
动作TTS 播报 + 生成事件日志
离线不强求（导航数据在云端/高德）→ 走云端，端侧配合 one_shot 防重复
④ 驻车时舱外感知降频（省电）
触发档位 变更为 P
条件舱外 AlwaysOn = 运行中
动作车控/VLM：把舱外感知频率降到低频
离线要（纯端侧任务调度，断网也得省电）→ 落端侧
⑤ 国道超速提醒
触发车速 大于 120（引擎抓从 ≤120 到 >120 的边沿）
条件道路类型 = 国道（高速开 120 正常，不提醒）
动作TTS 播报"开慢一点" + 生成事件日志
离线半（车速端侧有，道路类型要导航/云端）→ 研发定，倾向云端判道路
⑥ 雨雪天气切驾驶模式
触发舱外视觉 变更为 雨雪（依赖舱外 VLM）
条件当前 ≠ 泥泞模式 且 电量足（避免运动模式更费电）
动作车控：切驾驶模式为泥泞模式
离线不能（舱外 VLM 端侧现在没有）→ 走云端 AI Service，现阶段先搁置
⑦ 宝宝看护（动态场景）
触发用户设置（动态 Advisor 下发"看二排宝宝"任务）
条件舱内 VLM 描述 = 命中"宝宝醒了/哭了"（Advisor 自判）
动作通知用户 + 达成后注销任务
离线舱内 VLM 端侧有 → 可落端侧（7.7 焦点描述任务链路）
⑧ 节日彩蛋
触发时间触发器 = 农历节日（端午/春节…，穷举）
条件上电 且 首次上车
动作请求云端 API 拿节日内容 → 执行特定效果
离线不能（节日 API 在云端）→ 走云端
把这 8 条摆一起你会发现一个规律：①④⑦（纯端侧信号 + 要断网跑）落端侧，②③⑥⑧（要查云端数据/API）走云端，⑤看研发怎么切。这个"离线要求 → 端云"的对应，就是你跟研发对接的全部秘密（下面第五部分）。
第四部分 · 成品③

四、workflow 画布产品方案（目录 + 每节点规格）

这份是你抢 owner 的凭证。你不用画界面，把下面这个目录填成文档就行——它证明"画布的产品设计是我在做"。

方案目录（照这个写）

  1. 定位：workflow 规则工作台 = 配置平台的下一代，整个系统（Director/Advisor/主动服务）的规则统一在这里配
  2. 节点类型：Start / Trigger / Condition / Action / Branch(分支) / End（规格见下表）
  3. 连线逻辑：Condition 出 Yes/No 两条边；支持多条件串联、分支
  4. 自然语言预览：规则自动翻译成一句人话（就是图底下那行）
  5. 草稿 / 试运行 / 发布 / 版本：右上角那三个按钮的产品逻辑 + 版本号 + 命中率统计
  6. 端云同步：每条规则打"离线要求"标记 → 下发端侧 or 留云端（研发据此实现）
  7. 原子能力库管理：三张库怎么增删、怎么接信号中枢的信号

每种节点的参数规格（照图整理的）

节点要配的参数
Trigger 触发显示名、信号源、运算符（变更为/有变更）、目标值（可多选）、去抖动 ms、仅点火时触发(开关)、记录事件日志(开关)
Condition 条件信号、位置（至少 1 个/全部，多选 key）、运算符（等于/大于…）、目标值、多条件的 且/或
Action 动作动作类型、类型对应参数（TTS 配音色+文本、车控配指令、Advisor 配场景…）
Branch 分支Yes → 下一步；No → 结束 或 另一条动作
Start / End入口 / 结束（无参数）
第五部分 · 成品④

五、端云分工：判断表 + 研发问你时怎么答

你一直卡在"怎么跟研发不预先定端云、可研发又一定问哪些端侧做"。核心就一句：你不预先拍端云，但你给"离线要求"，研发拿这个自己推。端侧存在的唯一硬理由是"断网也要能跑"（丁彬 7.15 自己提的）。

判断表（你标左边，研发填右边）

这条规则…那它大概率…
断网也要能跑 + 信号都是端侧本地的（档位/车窗/车速/舱内 VLM）落端侧触发器
要查云端数据/API（POI/天气/节日/舱外 VLM）走云端触发器
信号端侧有，但判断逻辑复杂/要问用户研发权衡，你别替他拍

研发问"哪些是端侧触发器做的"，你就这么答（背下来）

研发："你这批场景，哪些端侧触发器做，哪些云端？"
你："我不预先替你们拍端云。我给你们标了每条的离线要求——哪些断网也得跑。断网要跑的、信号又都在端侧的，你们落端侧；要查云端数据的，走云端。具体怎么切，你们看端侧有没有数据、性能够不够，你们定，定完告诉我落哪，我回填。"
研发："那这条雨雪切模式呢？"
你："这条我标了'断网不强求'，而且雨雪要舱外 VLM，端侧现在没有——所以它走云端，现阶段先搁置，等舱外 VLM 能力上来再说。"
为什么这么分你占理：你没甩锅（你给了离线要求这个依据），也没越界（端云实现是研发的专业）。7.15 韩杰、汇文都是这个意思——技术协议本不该产品来拍，产品只定业务场景 + 关键约束（离线/实时性）。你以前 PRD 越写越多，就是因为把端云实现全拆进了产品逻辑。这次卸掉，你轻装，研发也没话说。
