车书QA与车控冲突修复方案

基于实际FORNAX SP分析 · 不改SP · 三套可行方案详细说明
角色 泽民 · 场景知识/Skill注入 约束 不修改SP固定内容 时间 1-2天出结论 数据 QA对500-2000条 · 车控功能80-150个

一、问题是什么

核心问题：车控指令被车书QA知识"劫持"

用户说 打开定速巡航（执行指令），系统检索到车书QA知识"定速巡航的手动操作步骤..."，
Director看到这段知识后，直接用它做了文字回答，而没有调用 vehicle_basic_control 工具执行操作。

当前链路：问题是怎么发生的

用户说："打开定速巡航"
→
QA向量检索（全量触发，top3）
→
检索到："定速巡航开启方法：进入车辆设置..."
↓ 注入到 SP 的 {{car_function_simple_qa}} 变量中
Director 看到知识 + 用户query
→
❌ 用知识做文字回答："可以这样开启定速巡航：进入车辆设置..."
根因：SP中 {{car_function_simple_qa}} 的引导语只写了"可能会用上的一些基础车辆问答信息"，
没有告诉Director"这只是参考，执行指令时应该优先调工具"。Director默认把它当成了"可以直接用的答案"。

期望链路：修复后应该怎么走

用户说："打开定速巡航"
→
QA向量检索 → 检索到知识
→
知识注入（含"该功能支持语音控制"标注）
↓ 同时 {{knowledge}} 中有处理规则："执行指令→调工具"
Director 综合判断：用户意图是执行
→
✅ 调用 vehicle_basic_control 执行操作
如果工具失败 →
✅ 用QA知识做兜底："你也可以手动操作..."

二、三类用户意图的判断逻辑

修复的核心是让Director能区分三种不同的用户意图，对每种意图采取不同的处理方式。

意图类型 用户说法特征 示例 正确处理 错误处理
执行型 带动作词："打开""关闭""调高""调低""开启""设为" "打开定速巡航"
"关闭车窗"

三、方案A — 定向修改QA数据

方案A

在车控功能相关的QA对answer中加入"支持语音控制"标注

执行人：产品侧标注 + 泽民验证  |  工作量：1天  |  不改SP，不改工程

核心原理

不改全部500-2000条QA，只改和车控功能重叠的80-150条。方法是：在answer开头加一句标准前缀，告诉Director"这个功能支持语音控制"。当Director看到这个信息时，遇到执行型指令就会倾向于调用工具而非复述手动操作步骤。

第一步：确定哪些QA对需要改

  • 判断标准：该QA描述的功能是否支持语音控制（即 vehicle_basic_control 或 search_vehicle_status_info 能处理的功能）
  • 数据来源：SP中的"车辆设备知识"部分已经列出了所有功能的语音控制能力，直接参考
  • 参考清单（从SP提取）：

第二步：怎么改 — 两种标准前缀模板

模板1：功能支持语音控制的QA对

❌ 改之前
Q:
定速巡航怎么开
A:
进入【车辆设置】-【辅助驾驶】，打开【自适应巡航辅助】，然后按方向盘PILOT按键激活。
✅ 改之后
Q:
定速巡航怎么开
A:
该功能支持语音直接控制，用户说"打开定速巡航"即可由系统执行。若需手动操作：进入【车辆设置】-【辅助驾驶】，打开【自适应巡航辅助】，然后按方向盘PILOT按键激活。

模板2：功能支持实时状态查询的QA对

❌ 改之前
Q:
怎么看车速
A:
当前车速是仪表屏常显信息，非P挡时你可以直接在仪表屏查看当前车速。
✅ 改之后
Q:
怎么看车速
A:
该信息支持语音实时查询，用户说"现在时速多少"即可由系统查询并返回实时数据。此外，当前车速也是仪表屏常显信息，非P挡时可直接在仪表屏查看。

模板3：功能不支持语音控制的QA对 — 不改

如"雨刮器怎么调""智能驾驶怎么开"等只能手动操作的功能，保持原样不改。这类QA不会引起冲突，因为Director本身也没有对应的工具可调。

第三步：标准前缀（可直接批量拼接）

# 对于"支持语音控制"的功能QA： 前缀 = "该功能支持语音直接控制，用户说\"{对应的语音指令示例}\"即可由系统执行。若需手动操作：" 新answer = 前缀 + 原answer # 对于"支持实时查询"的功能QA： 前缀 = "该信息支持语音实时查询，用户说\"{对应的语音指令示例}\"即可由系统查询并返回实时数据。此外，" 新answer = 前缀 + 原answer # 对于"不支持语音控制"的功能QA： # 不改，保持原样

这个拼接可以脚本化：准备一份 功能名 → 是否语音可控 → 对应指令示例 的映射表，批量对QA数据执行字符串拼接。

为什么这个方案可行？

1. LLM对context中的显式声明非常敏感 — 当answer开头就说"该功能支持语音直接控制"，Director在做决策时会把这个信息纳入考量，大幅降低"用手动操作步骤回答执行指令"的概率。
2. 工作量可控 — 不是改全部500-2000条，而是只改与车控重叠的80-150条。对照车辆设备知识表即可快速判断。
3. 不侵入任何系统组件 — 只改数据内容，不改SP、不改检索逻辑、不改工程代码。
4. 可脚本化 — 标准前缀+映射表+批量拼接，人工只需要审核映射表的准确性。
5. 会上李明骏认可的方向 — "此功能支持直接使用XX工具来控制，也可以告知用户手动操作"。

风险和局限

  • 不是100%保证：加了"支持语音控制"后Director大概率会调工具，但不是绝对（LLM行为有随机性）。需要实验验证通过率。
  • 映射表准确性：需要仔细核对哪些功能真的支持语音控制。SP中"车辆设备知识"已经标注了，但要逐条确认。
  • 新增QA对时需要同步：未来新增QA数据也要遵循这个标注规范，否则问题会复发。

四、方案B — 通过动态专业知识注入处理规则

方案B

在 {{knowledge}} 中注入"车书知识使用规则"

执行人：泽民独立完成  |  工作量：2-3小时  |  不改SP，不改工程，不改QA数据

核心原理

SP中 {{knowledge}} 的引导语写着："根据当前情景、任务，动态加载的专业知识，灵活参考能有效帮助你把事情做得更好"。这意味着Director会认真参考这里注入的内容。我们在这里注入一条"处理规则"类的知识，教Director如何正确处理 {{car_function_simple_qa}} 中的知识。

要创建的知识条目内容（可直接复制）

知识ID建议：RULE-01（或 COM-99-01-01） 知识名称：车辆功能问答知识的使用规则 【车辆功能问答知识使用规则】 当你收到「基础车辆功能问答」中的参考知识时，必须根据用户的真实意图决定处理方式，而非直接使用知识内容作为回答： 规则1 — 执行型指令 → 调用工具，不用知识回答 当用户说出带动作词的指令（"打开XX""关闭XX""调高XX""调低XX""开启XX""设为XX"）， 必须调用 vehicle_basic_control 工具执行操作。 「基础车辆功能问答」中的手动操作步骤不能替代工具调用。 → 例：用户说"打开定速巡航" → 调用 vehicle_basic_control，不要回答"进入车辆设置..." 规则2 — 实时状态查询 → 调用工具，不用知识回答 当用户询问实时数据（"现在XX多少""当前XX""剩余XX""XX正常吗"）， 必须调用 search_vehicle_status_info 工具查询真实数据。 「基础车辆功能问答」中"去仪表盘看"类的指引不能替代实时数据查询。 → 例：用户说"现在时速多少" → 调用 search_vehicle_status_info，不要回答"可以在仪表盘查看" 规则3 — 操作方法咨询 → 使用知识回答 仅当用户明确询问操作方法（"怎么用""如何设置""步骤是什么""咋调"）时， 才参考「基础车辆功能问答」中的知识进行回答。 → 例：用户说"定速巡航怎么用" → 参考知识回答操作步骤 规则4 — 工具失败兜底 当工具调用失败时，可以用「基础车辆功能问答」中的知识作为兜底方案， 告诉用户手动操作的方法。

Knowledge Selector 触发策略：三种选择

策略做法优点缺点推荐度
始终加载 把 RULE-01 设为基线知识，每轮都注入 {{knowledge}} 最简单，不会漏匹配；泽民直接控制 每轮多占 ~150 tokens；非车控场景也会注入（浪费） 推荐先用这个
关键词触发 在 Selector 中加一条规则：query含"打开/关闭/调高/调低/开启/多少/剩余" + 车控设备词时加载 精准，不浪费token 需要维护关键词表；可能漏匹配边缘case 效果验证后可切换
与 car_function_simple_qa 联动 工程侧：当 {{car_function_simple_qa}} 非空时自动注入 RULE-01 最精准——只有检索到QA知识时才注入规则 需要工程侧配合（短期不可行） 中期优化方向

为什么这个方案可行？

1. {{knowledge}} 的权重高 — SP引导语明确说"灵活参考能有效帮助你把事情做得更好"，Director会认真对待这里的内容。
2. 规则明确、有示例 — 不是抽象的原则，而是具体的"如果用户说X→做Y"的规则+示例，LLM对这种格式的遵循度高。
3. 泽民完全自主可控 — 不需要任何其他人配合，不改SP、不改工程、不改数据。创建一条知识+配置触发条件就完成了。
4. 可快速验证 — 2小时内可完成创建+配置+测试3条badcase。

风险和局限

  • knowledge是"参考"不是"指令"：{{knowledge}} 在SP中的定位是"专业知识参考"，不如注意事项/示例那样是"硬性规则"。Director可能不会100%遵循。
  • "始终加载"策略有token浪费：每轮多150 tokens，在token预算紧张时可能需要切换为条件加载。
  • 无法覆盖所有边缘case：如果用户的表述很模糊（既像执行又像咨询），规则可能不够精确。

五、方案C（推荐） — A + B 组合双保险

推荐方案

QA数据标注 + Knowledge规则注入 = 双层保障

执行人：泽民（方案B）+ 产品标注（方案A）并行  |  工作量：1天  |  不改SP，不改工程

为什么组合方案效果最好？

单靠方案A：QA数据说了"支持语音控制"，但Director不一定会因此改变行为 — 它可能仍然选择用后面的手动步骤回答。

单靠方案B：{{knowledge}} 中有处理规则，但它的权重是"参考"级别，不如SP中的"指令"级别强，Director可能不完全遵循。

A + B 组合：两层信号同时告诉Director同一件事 —— 数据层说"这个功能支持语音控制"，知识层说"执行指令要调工具"。两个独立信号指向同一方向，Director做出正确决策的概率大幅提升。这就像考试时同一道题从两个不同角度给了提示，答对概率比只有一个提示高得多。

八、备选建议：最小SP改动（如果团队后续愿意微调）

仅改一句引导语（5分钟，~80字）

如果后续团队愿意做一个极小的SP调整，只需要把 {{car_function_simple_qa}} section的引导语从：

可能会用用上的一些基础车辆问答信息

改为：

以下是从车辆手册检索到的参考知识。执行指令时优先调用工具，仅在咨询操作方法或工具失败时参考以下内容回答

这不是结构性改动，只是改了一句描述性文字。但效果会比方案A+B更直接，因为它在知识注入的"最近点"告诉了Director怎么用这些知识。如果A+B验证后修复率还不够理想，建议推动这个最小SP改动作为补充。
