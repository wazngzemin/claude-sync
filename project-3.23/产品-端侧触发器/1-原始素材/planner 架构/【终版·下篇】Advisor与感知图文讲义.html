豆包上车架构讲义 · 下篇：Advisor 体系与感知订阅全拆解

每个 advisor 都讲清：为什么做 → 怎么想的（推理逻辑/SP结构）→ 输入输出 → 真实 case → 做到什么程度 → 坑点。动态 advisor 的 9 套 prompt、生命周期状态机、感知订阅的 VLM 阈值穿越全部覆盖。

一、舒适 Advisor——车内的"体感管家"

关注6+1个舒适维度（温度/空气/光线/声音/乘坐/空间/其他），不直接控车只给Director建议。对应知识专家COM。PRD v1.0。

1.1 流程图：舒适advisor的推理链

1.2 讲解

为什么做：车内环境时刻在变，靠用户说"我好热"太被动。好的体验是系统自己发现不舒适、主动建议调整。

触发：两种——① 事件触发（有人上车、温度变化、VLM看到有人阅读→检查光线）；② 1分钟内无新事件+有未完成舒适子目标→自动定时review（如"刚才开了急速制冷，1分钟后看降到没"）。

输入（5类模板变量注入SP）：{{attention}}事件 / {{watcher}}VLM描述 / {{car_condition}}端信号 / {{web_information}}世界信息 / {{memory}}记忆。

推理逻辑：SP内置5条knowhow（决策树式if-then）——温度超标→快速调温；空气超标→恢复空气质量；人员变动→检查空间；有人阅读/睡觉→检查光线/声音。核心模式：感知异常→诊断维度→生成方案。

输出：JSON{当前舒适性问题, 应对建议}→给Director。不直接控车。

38度上车(P0)：VLM"乘客额头有汗"+端信号"车内38度空调关闭"→"温度不舒适"→"开空调20度+座椅通风3档+通风2分钟后关窗"→Director执行。1分钟后定时review："降到26度但风速还在7档"→"降到3档"→Director再调。

做到什么程度：核心场景（上车高/低温、多人分区、睡眠调节）P0已定义。结合视觉的高感场景（如擤鼻涕→调小风量）标TBD依赖VLM。反思机制P1待讨论。

二、出行 Advisor——"理解你去哪、为什么去"

覆盖全出行生命周期（行前/行中/行后），核心能力=出行目的预测。对应知识专家DRV。PRD v0.4。

2.1 流程图：出行目的预测链

2.2 讲解

为什么做：用户上车还没说去哪，系统如果能猜出来并推荐路线、顺手帮缴停车费、提前订座，就从"导航工具"升级成"出行管家"。

核心推理：出行目的预测，靠10+数据源——历史偏好、车内人员、携带物品（带书包→上学）、时间（周二早上→通勤）、位置、常去地点、导航、日程。推理模式：多因素融合→概率排序→输出预测+置信度→低置信主动确认。

"12月3日周二8:31，益红和依依已上车，车在长江小区。依依带了书包应在8:30出发去光启小学，益红通常8:45去平金中心上班。预测：1.送依依上学 2.益红上班。"

场景覆盖：行前（目的预测/缴停车费通过GUIAgent/路线推荐）→行中（堵车绕行/充电推荐/POI推荐/订座取号）→行后（最后一公里指引/停车记位）。

做到什么程度：PRD v0.4成熟度偏低。核心场景已定义但生态服务（订座/取号）、POI推荐等标P2未实现。输出格式"需要和planner详细沟通"。

三、内容 Advisor——"克制比推荐更重要"

核心设计哲学："先判断该不该推，再判断推什么"。默认不工作，没Trigger就没内容。对应知识专家ENT。PRD V0.4。

3.1 流程图：L0-L3介入等级判定

3.2 讲解

核心机制：被触发后第一件事不是"推什么"而是"该不该推"，输出唯一的介入等级L0-L3：L0隐形（愤怒/暴雨/冲突→0%认知→一个字不说）/ L1氛围（疲惫/发呆→白噪声）/ L2碎片（通勤→笑话/新闻）/ L3沉浸（长途巡航→专家讲坛）。原文设计哲学："车上夫妻正在吵架，即使满足Trigger条件也不要推送"。

8步推理Flow：静默→唤醒→Context→介入判断→策略→分发→反馈感知→Review回收。反馈闭环：连续跳过3次→归因（不喜歌手还是曲风？）→修正；拥堵>5min→自动降认知。

独特协同：情感advisor判断"用户适合陪伴"时发信号给内容advisor触发推荐——4个advisor之间唯一明确的跨advisor协同接口。

四、情感 Advisor——推理最精细、红线最严格

纯情感层察言观色，不控车不推内容不提醒物品。有两套完整SP（Event+Review），含大量正反例。对应知识专家EMO。PRD v1.0。

4.1 流程图：Event Prompt 的4步推理链

4.2 讲解

两套SP：Event Prompt（事件触发用，4步推理链）+ Review Prompt（10分钟定时复查，3步推理链：评估进展→调整/恢复/完成仅删除不建议）。

Step1 多情感优先级：启动方式>安全>节日>人物状态>情绪>人物>场景>着装。安全类强制执行——后排儿童睡着(子目标B"轻柔语气")+主驾分心(安全事件)→安全优先用严肃语气提醒。

Step2 即时建议6条红线（4个advisor里最严格）：不推具体内容(越界给内容advisor)/不控车(越界给舒适advisor)/不猜测未发生的/不预测曾发生的/不提醒物品/禁止"维持现状"。SP里有大量反例："建议播放轻音乐"→✘越界；"建议调低空调"→✘越界。

Step3 子目标状态机：A目标达成型（"安抚主驾直到不再愤怒"）和B条件响应型（"轻柔语气直到右后排睡醒"）。create需同时满足5条件（有服务对象+情感可建议+非即时闭环+可判定+不重复）。上限10个。

输出：{reason, advice, goal_operations:[{action,fields,reason}]}。动作空间：DO_NOTHING/CONVO_STRATEGY(沟通策略)/CONVO_INITIATE(主动搭话)/EMOJI_SUGGESTION/TOOL_CALL(self ReACT查车况)。

被超车：用户"卧槽"→事件触发→判定"愤怒/受挫"→输出：语气=共情吐槽→Director："那个开车也太横了吧。不过你没事就好。"

做到什么程度：v1.0，两套完整SP全文已写。打扰预算自适应(P2)未实现。主动搭话默认3次/天。

五、动态 Advisor——9套Prompt · 4类任务 · 完整生命周期

临时冒出来的持续任务（照顾宝宝/介绍风景/点咖啡），静态advisor管不了，需要动态创建专属监工。

5.1 流程图：创建流程（Director→分类→4条流水线）

5.2 讲解

分类优先级：① 调用点单/缴费→type=2(持续GUI) ② 明确时间/周期→type=4(定时) ③ 条件表达→type=3(条件) ④ 其他→type=1(长时)。

9套prompt模板：任务分类 / VLM分类+query生成 / 通用长时任务(SYSTEM+USER) / 持续任务GUI / 视觉感知advisor(3类策略：风险监控/信息讲解/状态陪伴) / 事件定时触发advice(≤20字口语化) / 定时器生成(interval/cron/once/none) / 视觉理解(true|reason或false|reason) / 触发器脚本生成。

长时任务输出JSON（最完整）：{goal, goal_status(active/achieved/terminated_by_user/blocked_by_system), goal_progress, need_advice, advice_timing_confidence(0~1), goal_achieve_advise, completion_rate(0~1), exec_state(normal/suppressed/recoverable/short_circuit)}

exec_state状态机：normal(可输出)→suppressed(抑制,强制不输出)→suppressed_attention_shift(用户注意力偏移)→recoverable(可解除)→short_circuit(上轮有效跳过本轮)。

销毁：max_execution达上限/用户删除/goal_status=terminated或blocked/触发器注册失败。

数量上限：定时≤6·条件≤6·GUI≤1·长时≤2·VLM视觉仅1（照顾宝宝和介绍风景不能同时）。循环最大30次；最小间隔1分钟；>365天不创建。

二期变化：Skill系统+Workflow编排(长时从特调到通用)；端侧定时器(弱网也能触发)；策略预生成(触发时不再现场推理)；导航POI距离触发。

六、感知订阅——把"帮我盯着"翻译成可执行的触发器

把advisor的自然语言订阅拆成"看什么/什么条件/多久一次"三个槽位，配置到VLM/触发器/定时器。

6.1 流程图：VLM阈值穿越检测

6.2 讲解

三个子模块：① 动态感知("看什么"→调VLM sp/attention让视觉模型关注特定对象) ② 条件("什么条件"→注册触发器监控) ③ 定时("多久一次"→APScheduler四种格式interval/cron/once/none)。第四能力：信号组装+触发advisor带最新感知。

VLM阈值穿越（触发的真实机制）：VLM每轮只输出一行true|reason或false|reason。对比上次(last)和这次(this)——只有last==False and this==True才触发（状态从"未发生"变"发生"的上升沿）。持续True不重复触发。

"照顾宝宝"→VLM query:{"type":"state","query":"二排存在宝宝"}
第1轮:false|宝宝没睡→不触发。第2轮:true|宝宝睡着了→last=F,this=T→触发！第3轮:true|还在睡→不触发(没变化)。

3类视觉策略：风险监控(异常才说一次)/信息讲解(新内容间隔≥60-90s)/状态陪伴(明显信号才触发冷却≥5min)。

触发优先级：订阅事件(最高)>qa结束>定时(最低)。高优打断低优。

七、VLM/Watcher——让文本大脑"看见"

端侧Always on，大小LLM融合，把摄像头画面转成自然语言写回context。

7.1 流程图

7.2 讲解

Watcher三功能：①细致描述场景(人/位置/衣着/动作/物品) ②全时多模聆听 ③事件发现(小孩哭闹/咳嗽/前方大货车/OK手势→触发advisor主动服务)。

VLM原理：大小LLM融合。小模型把图像编码成token(兜底)；大模型按SP(识别目标)看画面输出自然语言。"影响感知器sp/attention"=改VLM的SP——同一VLM拼不同SP切换关注对象（无订阅→通用观察；收到"关注二排宝宝"→拼新SP重点关注后排婴儿）。

约束：VLM视觉持续任务同时仅1个。事件发现一期纯工程触发，二期多模态+工程混合。
