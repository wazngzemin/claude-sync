泽民反复说"还是不够详细"时,不要靠再出一版更长的文档或更抽象的解释去满足他。2026-07 这轮连出五版他都说不够,直到用他系统里真实的车态信号(档位/车速/车窗/车门/座椅/座椅加热/温度/天窗/隐私开关/电量/驾驶模式/雨量/充电/安全带)把"触发事件 vs 触发条件"逐条摆开、把三层漏斗拿真实场景从头走一遍,才对。

**Why:** 他是把概念挂到自己天天配的东西上才能真懂的人(执行型 PM 的具体思维)。抽象定义 + 三个例子对他等于没讲。他要的"详细"= 颗粒度(真实信号/真实字段/真实case)+ 穷尽(每一类都举例,不留概括),不是篇幅。呼应 [[feedback_table_first]] 单元格写详细、[[project_710_signal_hub_review]] "听不懂就守不住"。

**How to apply:** 讲概念/架构先问自己"能不能换成他系统里的真实信号和真实场景"。能就换,并且每一类穷举(事件运算符 vs 条件运算符、阈值穿越、同一信号两用都要单独举例)。遇到他连着说"不够详细",别默认再加长——用 AskUserQuestion 一次性问清是要"起草真交付物/概念讲穿/逐case推演/每条摊开"哪一种,别猜第五次。产物:2-AI产出-最新/7.10-信号中枢-为什么要拆条件-v3.html(真实信号全展开版)。

**同理适用于架构/链路讲解(2026-07-15 补，连踩两次):** 讲模块/数据流一律用真实模块名。**不要任何通俗类比**——保姆/婴儿监视器/备忘录/车主/摄像头这类生活比方,哪怕只当"可选直觉铺垫"他都反感(我第一次保留作铺垫,他直接发火"我说了只写保姆什么的要换成我的模块词啊",连着两次要求删干净)。"哨兵/闹钟/仪表盘"这类轻比喻若他没点名也尽量少用、能换模块词就换。他明确要求"用各个模块的真实名字、根据图讲、和会议评审一样详细"。做法=拿真实case(如"后排宝宝哭了告诉我")对着流程图,按评审级逐步走:每步标 `模块名 | 收到什么→做什么→给谁`,关键步补"为什么",并圈出他的交付边界(哪几步是触发器域)。真实模块名要用全:Planner/Task Agent/Task Service·(Ingress/Spec Builder/Manager/Trigger Adapter/Runtime Dispatcher/Message Dispatcher)/规则服务Trigger/端侧VLM。产物:7.15-动态adv评审-触发器任务梳理-v6.html(图下评审级模块详解)。关联 [[project_715_task_management_trigger]]。
