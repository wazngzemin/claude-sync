座舱Planner OnePage

架构总纲 - 整套 Planner 系统是怎么搭的, 最重要的一份

1.1 为什么要搞 Planner? - 传统架构的三个死穴

文档开篇先喷了一把传统车载座舱架构。传统架构的优势是"稳、快、准"——预设任务链路简洁, 高频功能固化执行, 交互可控。但随着智能化需求升级, 三个核心缺陷暴露了:

模块协同性不足
各模块独立闭环, 缺乏全局信息共享。用户说"带我去兜风", 涉及导航+音乐+车控多个模块, 传统架构根本搞不定, 直接报错。
场景感知薄弱
没有集成用户记忆(历史偏好)和实时环境(车内状态、路况), 不能自适应。每次都当用户是第一次上车。
长尾场景成本爆炸
每个新场景都要单独定制逻辑, 研发成本高、周期长, 系统泛化能力有限。100个场景写100套规则, 1000个呢?
这三个问题本质上就是一个问题: 传统架构是"固化的 if-else", 而用户需求是"开放的自然语言"。这个 gap 只能靠 LLM 来填。但你不能全靠 LLM, 因为 LLM 太慢太不稳定。所以需要一个混合架构——确定性的事情用规则, 不确定的事情用模型。这就是这份文档的核心思路。

1.2 核心架构: 三层漏斗

整套系统的灵魂就一句话——"层层过滤, 能用简单方法搞定的不要漏到后面复杂模型去处理"。

具体来说, 这是一个从简单到复杂的三层漏斗:

三层漏斗架构 (核心中的核心)
用户 Query 进入
"播放周杰伦的歌" / "我好热" / "带我去兜风" / ...
第一层: 句法 RAG (Syntactic RAG)
定位: 硬编码规则匹配, 不过模型
原理: 用户 query 直接匹配预设模板
特点: 稳定 / 快 / 车厂可自行配置
适用: 明确的单轮指令, 不需要上下文
匹配失败, 漏到下一层
第二层: 情景 RAG (Scenario RAG) + 仲裁模型
定位: 知识注入, 辅助模型拆解任务
原理: 根据场景检索 few-shot 示例, 注入给 Planner
仲裁: Seed 1.6 Flash SFT 判断"简单指令 / 复杂任务 / 拒识"
适用: 模糊意图(如"我好热"), 需要结合情景/状态/记忆
复杂任务, 漏到下一层
第三层: Planner (Thinking Model)
定位: 复杂任务的核心大脑, 最终兜底
模型: doubao-seed-1.8 (SFT 后)
能力: 任务拆解 / 工具调用 / 状态管理 / 对话生成
适用: 多步任务, 条件任务, 持续任务, 所有前面接不住的
下游 Agents 执行 (车控/音乐/导航/联网/...)
三个核心设计原则 (文档原话)
  1. 层层过滤: 能通过简单方法搞定的不要漏到后面复杂模型去处理
  2. 分阶段知识注入: 在不同阶段选择合适的节点注入知识, 从简单(固定)到复杂(泛化)
  3. 数据闭环: 通过用户数据回流, 逐步把后面复杂任务的处理推到前面知识注入环节
第三条"数据闭环"是最聪明的设计思路: Planner 现在当 teacher, 它处理的 case 产生的数据, 回流去训练句法 RAG 和情景 RAG, 让前面的层越来越强, 后面的层压力越来越小。这不是一个静态架构, 是一个会自己进化的架构。但前提是——你得有 eval 来验证前面的层接得住了没有。

情景RAG的知识注入具体长啥样?
文档给了个例子: 用户说"我好热", 情景 RAG 输出三种拆解:
- 常规: 打开空调设为23度 + 打开座椅通风
- 有记忆(用户不能吹空调): 打开座椅通风 + 打开车窗
- 有情景(空调已开27度): 空调调到22度
这些作为 few-shot 塞给 Planner, 引导它根据实际上下文选择正确的拆解。

已知的仲裁误分类 (1224实车测试发现的)

1.5 Planner 的四个特殊工具

Planner 不只是调下游 agent, 它自己有四个"内部工具", 用来增强思考、感知和管理能力:

plan_describe
定位: 思考/铺垫工具
在调度 agent 之前做深度思考, 输出计划方案的自然语言描述给用户。
参数: tts (话术文本)
注意: description 把它写成了"展示"工具, 但实际应该强调"思考"和"推理"。这会影响模型的行为模式。
search_user_preferences
定位: 记忆查询工具
查询用户长短期记忆: 导航偏好、音乐喜好、常用地点、驾驶模式等。
参数: query (自然语言查询指令)
由 Planner 判断何时需要触发记忆查询。
search_vehicle_status_info
定位: 环境/状态查询工具
查询车辆信息: 温度、空调、座椅、胎压、车速、充电续航等全方位状态。
参数: query (自然语言查询指令)
由 Planner 判断何时需要查端状态。
plan_update
定位: 任务状态管理工具
更新任务状态: 哪些完成了、哪些在进行中、哪些待做、哪些失败了。
参数: update_data (JSON 格式任务状态)
工程端解析结果后存入全局 State, 并追加到 messages。

Planner 可以调用的下游 agent, 每个 agent 负责一个专业领域:

1.7 任务管理: 状态机 + Replan + 打断恢复 + 挂起

任务的数据结构

每个任务是一个 Task 对象, 包含: agent_name / task_desc / status / failure_reason / priority。多个 Task 组成一个 Plan, 绑定到一个 user query。

任务状态流转

任务状态机
PENDING
待执行
执行中
COMPLETED
已完成
执行中
FAILED
失败 → 触发 Replan

四种任务管理场景

Replan (重新规划)
下游 agent 执行失败 → 端上把失败原因回传 → Planner 调用 plan_describe 重新思考 → 生成新计划再下发。

例: "调低空调温度" 被 agent 拒绝(已经最低了) → Planner replan: "空调已经最低了, 要不要开座椅通风?"
打断 & 恢复
当前 plan 没执行完, 被新 query 打断 → 工程标记当前 plan 为 interrupted, 保存 checkpoint → 处理新 query → 新 query 结束后, 带 checkpoint 请求 Planner 决定是否恢复。
挂起 (Suspend)
条件任务/定时任务: Planner 识别出需要延迟执行的任务 → 标记为挂起 → Always On 模块持续监听条件 → 条件满足后通知 Planner → 恢复执行。
状态更新
模型侧: Planner 调用 plan_update function call 输出 JSON。
工程侧: 解析结果, 更新全局 State, 追加 ToolMessage 进入上下文, 让 Planner 在下一轮能看到最新状态。

1.9 Context 管理: 上下文工程

文档引用了 Lance Martin 的 Context Engineering 框架, 提出四种管理方式:

Write Context
写入上下文: agent 的中间结果、tool response 写入供后续使用
Select Context
选择上下文: 根据当前任务, 选择性地放入相关信息
Compress Context
压缩上下文: 长情景描述做摘要, 控制 token 预算
Isolate Context
隔离上下文: 多人场景下隐私过滤, 不同乘客看不到对方的记忆
OnePage 总结: 你需要记住的关键架构决策
  1. 三层漏斗: 句法RAG → 情景RAG + 仲裁 → Planner
  2. Planner 有 4 个特殊内部工具 + N 个下游 agent
  3. 任务状态机支持 replan / 打断恢复 / 挂起
  4. 模型选型: doubao-seed-1.8, 从 Kimi K2 蒸馏数据做 SFT
  5. 仲裁模型 (Seed 1.6 Flash SFT) 是已知的薄弱环节
02

任务Planner PRD [豆包 in car 1.0]

产品需求文档 - Planner 要做什么, 四类任务的详细定义

2.1 Planner 到底是什么?

工程视角
基于 Thinking Model 构建的任务规划引擎, 通过时序任务拆解和条件任务链生成, 将复杂用户需求转化为可执行的任务序列。系统通过任务状态机实现多任务并发管理, 并与 Watcher 模块实时同步上下文。
用户视角
一位"智能需求管家"——你说一句模糊的话, 它帮你想清楚要做什么、怎么做、按什么顺序做, 然后一件件帮你搞定。做到一半你插嘴改主意, 它也能接住。

2.2 Planner 解决的五大车上痛点

1. 复杂任务需要用户一直给输入
口味偏好、冷热偏好、价格偏好... 模型缺失上下文, 每次都要用户主动说。应该模型自己去查记忆和状态。
2. 用户只有大目标, 没有小目标
"想让座舱更舒适" → 具体该怎么操作? "OTA更新了智能光毯" → 具体怎么用? 用户不知道, 需要 Planner 帮他拆。
3. 检索想要的东西很费劲
"搜一个我想看的电影" 要用户自己分类搜索。应该结合偏好和语音指令一步到位。
4. 一锤子买卖, 需求不闭环
用户说一句, 系统应一句, 然后就没了。没有持续跟进, 没有状态追踪, 没有闭环。
5. 闲聊聊不下去, 割裂没内涵
不知道舱内外实际发生的情景, 不知道车辆实际状态, 不知道正在进行的任务。聊天和车割裂了。

2.3 四类任务: 这是整份 PRD 的核心

PRD 把 Planner 要处理的任务分成四大类, 每类有不同的优先级(S7/S8/S9)、不同的能力要求、不同的评测指标:

四类任务总览 + 迭代节奏
S7 优先 单步复杂任务
定义: 用户说一句模糊的话, Planner 拆出一组无依赖的并行车控动作

典型 query:
"我好热" / "营造浪漫氛围" / "前挡都是雾" / "背后黏糊糊的" / "车里有点闷"

核心挑战: 要结合情景+端状态+记忆来推理, 不是简单翻译

指标: 推理 ACC ≥ 90%, 控制完成 P90 ≤ 2000ms

badcase 归因大概率是: 上下文没喂对
S8 多步任务
定义: 有依赖关系的串行任务, 前一步的结果是后一步的输入

典型 query:
"播放我是歌手第四季夺冠的那首歌" (先查→再播)
"去昨天那个游泳馆" (先查记忆→再导航)
"导航到上海评分最高的烤鸭店" (先搜→再导航)

核心挑战: 无依赖步骤不要串行(Kimi K2 的问题)、依赖步骤要正确串联

badcase 归因大概率是: 控制流设计 + 并行/串行判断
S8 条件任务
定义: 注册一个"条件→动作"的触发规则

典型 query:
"十分钟后关座椅加热" (明确条件+明确动作)
"车内温度超28度让我凉快下" (明确条件+模糊动作)
"我累的时候给我放首歌" (模糊条件+明确动作)

核心挑战: Planner 要做四重验证:
1) 条件是否可观测
2) 条件是否有可能触发
3) 动作是否合法合规
4) 条件和动作是否矛盾

badcase 归因大概率是: 约束验证逻辑 + knowhow 注入
S9-S10 持续任务
定义: 长时间运行的任务, 能被打断、能恢复、能动态调整

典型 query:
"新手引导" / "介绍沿途风景" / "扮演猪八戒" / "跟副驾聊天"

核心挑战: 打断恢复、上下文保存、动态调整内容。需要显式状态机, 不能靠 LLM 在 context 里"记住"做到哪了

这是最难的一类, 需要 Goal List + Advisor 体系支撑

2.4 单步复杂任务: 深入看 case 是怎么拆的

这是 S7 的重点, 也是你现在 badcase 最可能集中的地方。PRD 给了大量 case, 核心逻辑是结合三种 context (情景/端状态/记忆) 来做拆解:

Case: "我好热"

响应话术: "车里是有点热呢, 我来帮你打开空调降降温吧~"

任务拆解: 打开空调 → 空调调到21度 → 空调风量调到3档

总结回复: "都搞定啦, 车里马上就会凉快了"

Context 怎么影响拆解:

  • 情景: 外面正在下大雨 → 不拆出"开窗"
  • 端状态: 空调温度已经最低 → 不拆出"调低空调"
  • 记忆: 用户不喜欢吹风 → 不拆出"大风量"
看到了吗? 同一个 query "我好热", 在不同 context 下的正确拆解是完全不同的。如果 context 没喂对——比如下雨天但 context 里没有天气信息——Planner 可能会拆出"开车窗", 用户会觉得模型脑子有问题。所以这类 badcase, 你第一步永远是看: context 里到底喂了什么? 该有的信息有没有? 有没有过期的脏数据?

2.5 对话策略: 一个很聪明的产品决策

PRD 对响应话术做了一个重要的策略调整——不要把具体动作讲太清楚:

Bad Case
"好的, 我先帮你把座椅按摩和座椅通风打开, 开启全身按摩, 然后把氛围灯调成冰川色, 空调温度也调到15度, 酷爽放松氛围马上就来"

问题: "冰川色"可能不存在, "15度"可能不合理 → 车控幻觉
VS
Good Case
"好的, 我先帮你把座椅按摩和座椅通风打开, 顺便调一下氛围灯和空调, 酷爽放松氛围马上就来"

策略: 模糊处理具体参数, 避免幻觉被用户感知到
Bad Case (查询场景)
用户: "空调现在是开的吗"
豆包: "我帮你查一下~"

问题: 多余的响应话术, 直接给结果就行
VS
Good Case
用户: "空调现在是开的吗"
豆包: "空调现在是开着的~"

策略: 查到结果直接回复, 不需要铺垫话术
这个产品决策本质上是用产品策略回避技术问题——模型不知道车支持哪些氛围灯颜色、空调范围是多少, 所以让它"别说太具体"。短期能减少用户感知到的错误, 但长期正解是把功能清单和参数范围结构化注入上下文(knowhow 注入)。

3.1 当前 Planner 的四大问题 (文档原话)

这份文档开篇就把问题摆出来了, 非常坦诚:

问题1: Planner 负担太重 (最大的问题)
从 query 角度: 要处理 user query、advisor query、tool response、trigger 四种输入。
从 prompt 角度: 要塞人设、车信息、context(情景/状态/记忆)、tools、knowhow、工作流程。
从工作内容: 要负责响应聊天、任务拆解、意图分发、任务闭环。

一个 LLM 干这么多事, attention 是有限的, 关注十件事每件事的质量就只有十分之一。
问题2: Multi-agent 协作不够
Advisor 目前的职责只是基于 context 给建议, 对整体任务闭环和目标达成参与度不高。

Advisor 可以承担更多重思考、实时性要求不高的事情, 减轻 Planner 压力。
问题3: 性能和质量的平衡
性能要求: Planner 需要快速和用户交互。
质量要求: Planner 需要对拆解质量、流程稳定性负责。

二者需要平衡和同步精进, 不能顾此失彼。
问题4: 规则 vs 涌现的矛盾
为了让任务流程稳定, 不得不增加规则约束模型行为。
但增加规则会限制模型能力的涌现——模型本来能自己想出一个好方案, 但被规则卡死了。
问题1是核心。我在 LangChain 社区见过无数次——单 agent 架构的天花板就在这。你往一个 agent 的 system prompt 里塞的职责越多, 每一项的执行质量就越差。这不是模型能力问题, 是信息论问题。解法只有一个: 拆。

3.3 Goal List 长什么样?

Goal List 就是一张目标列表, 记录了每个目标的当前状态和进展。四个静态目标永远存在, 动态目标按需创建:

类型 目标 示例内容 由谁关注 是否会创建动态Advisor
静态 舒适 "当前车内温度较高, 需极速降温, 已开启极速制冷" 舒适 Advisor 否 (永久)
出行 "当前目标是去公司, 正常导航中" 出行 Advisor 否 (永久)
情感 "当前目标为呵护宝宝情绪, 正在讲故事, 宝宝情绪稳定" 情感 Advisor 否 (永久)
内容 "当前目标为播放宝宝喜欢的动画片, 播放中" 内容 Advisor 否 (永久)
动态 点咖啡 "目标为点一杯咖啡送到公司, 等待用户确认杯型" 动态 Advisor (临时) 是
介绍风景 "目标为介绍行程中的风景, 正在介绍前方的中南百草园" 动态 Advisor (临时) 是
功能介绍 "目标为介绍车辆功能, 被其他内容打断, 播报进度为..." 动态 Advisor (临时) 是
关键设计决策: 谁更新 Goal List?

为了避免更新冲突, 文档明确规定: 任务细节和进展统一由对应的 Advisor (Goal Reviewer) 来更新, Planner 不更新。

  • Planner 只负责: 创建/删除目标, 推理和工具调用
  • Advisor 负责: 更新目标的细节和进展, 关注目标是否达成

这个分工很清晰, 但实际执行中有个数据流向问题: Planner 调了 agent, agent 返回成功/失败, 这个状态怎么流到 Advisor? 如果靠工程侧中转, 延迟和一致性怎么保证?

3.4 四类任务和 Advisor 的关系

任务类别 有单独卡片? 会创建动态 Advisor? 典型场景
简单即时任务 没有 不会 做完就完, 不需要持续跟踪
多步任务 没有 不会 步骤间有依赖, 但总体时间短
条件任务 有 不会 注册后等触发, 不需要 Advisor 持续关注
长时任务 有 会 功能介绍 / GUI agent / 新手引导 / 照顾宝宝 / 介绍风景
注意这个对应关系: 只有长时任务才会创建动态 Advisor。因为只有长时任务需要持续跟踪目标进展、支持打断恢复、动态调整内容。简单任务和多步任务做完就完, 不需要一个 Advisor 盯着。条件任务虽然有卡片, 但触发逻辑交给 Always On 就行, 也不需要 Advisor。

3.5 对话管理: Query 合并和打断策略

这部分定义了两种打断场景的处理方式:

场景1: Planner 还在思考, 用户又说话了
用户: "给我讲讲今天的财经新闻"
Planner 思考中... (还没出话术)
用户打断: "先导航到海底捞"
工程: 废弃第一次请求, 把两个 query 合并
→ 重新请求 Planner
豆包: "好的, 我来帮你导航去海底捞, 顺便整理播报今天的财经新闻"
场景2: Planner 结果正在播报, 用户打断了
用户: "给我讲讲今天的财经新闻"
豆包正在播报: "2025年12月16日的财经新闻... 证监会部署多项..."
用户打断: "先导航到海底捞"
停止播报 → 处理导航 → 导航完成后...
Advisor (Goal Reviewer) 决定: 是否恢复上个任务?
恢复 → 带上下文(含目标细节和进展)重新触发
→ 从断点的上一个句子起始处续播
豆包: "刚刚被打断了, 接着讲财经新闻吧, 刚说到证监会部署多项..."
!
打断策略的工程隐患:
场景1中, 如果第一次请求已经产生了 tool call 但还没拿到结果, 合并后重新请求会不会导致重复调用? 会不会丢失中间状态?

场景2中, "已完整缓存"和"未完整生成"是两种不同处理: 已缓存的直接续播, 未完整的要基于上下文重新生成。工程侧需要判断当前是哪种情况。

Agent 自己播报

  2. [车控执行完成] "车内灯光和车窗都调好啦, 怎么样, 是不是很浪漫~ 马上我就为你高歌一曲"
  3. [s2s自己播] "我终于看到, 所有梦想都开花..."
目标驱动体系总结: 你需要记住的关键演进方向
  1. 核心变化: 从"Planner 一个人扛所有"演进到"Planner + Advisors 分工协作"
  2. Goal List 是 shared state, Planner 和 Advisors 共同维护
  3. Planner 职责收窄: 快速响应 + 推理 + 串联 (重时延)
  4. Advisor 承担更多: 目标跟踪 + 主动服务 + Goal Review (重思考)
  5. 只有长时任务才创建动态 Advisor, 其他任务不需要
  6. 对话管理有两种打断模式: 思考中打断(合并query) vs 播报中打断(Advisor决定是否恢复)
