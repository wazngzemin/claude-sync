## 回答 1

我来详细解释这张图——这是 **Driver Agent 2.0 升级版架构图**（Goal List + Advisor 协作模式），从左到右是一个 **"原始信号 → 上下文切片 → 注意力筛选 → 多 Advisor 建议 → Planner 决策"** 的流水线。

---

### 一、整体流程：5 个阶段

```
[上游信号源] → [Context 切片] → [Attention 筛选] → [Advisors 建议] → [Planner 决策]
   VLM/记忆/探索      8个子模块         吴仁杰          4个 Advisor      嘉锋
```

每个模块旁标注的姓名是 **产品经理责任人**（明骏、婉婷、芮婷、白宇、吴仁杰、弋逋、嘉锋）。

---

### 二、阶段一：上游信号源（最左 3 个蓝框）

| 信号源 | 作用 | 触发示例 |
|---|---|---|
| **VLM** | 视觉语言模型，感知舱内画面 | "有人上车" |
| **长期记忆** | 持久化的用户画像/偏好/历史对话 | "依依喜欢爵士乐" |
| **探索模块** | 豆包自主探索的内容（类似 Agent 后台任务） | "今天研究了 GPT5 进展" |

这三个是**原始数据生产者**，但格式各异、信号量大，不能直接喂给 LLM。

---

### 三、阶段二：Context 模块（灰色大框，8 个子模块）

**职责：把原始信号"切片+结构化"，每个子模块产出特定类型的 context 片段。**

| 子模块 | 负责人 | 内容 | 示例 |
|---|---|---|---|
| **watcher** | 明骏 | 视觉感知结果（人物、动作、外貌变化） | "依依上车坐在主驾，今天画了一个美美的妆" |
| **端信号** | 婉婷 | 车辆端实时状态快照 | "空调温度:xxx, 风量:xxx, 模式:xxx" |
| **事件日志** | 芮婷 | 55 分钟内的事件流（人/车/助手所有事件） | 时间线日志 |
| **基础联网信息** | 明骏 | 精准时间/日期/星期/节日/节气 | "公历日期"、"农历"、"节气" |
| **用户记忆** | 白宇 | 长期记忆的格式化输出（画像/对话/偏好） | "user:依依, 音乐偏好:..." |
| **短期记忆** | 芮婷 | （图中标注 ???，尚未确定） | 待定 |
| **豆包探索** | 明骏 | 探索任务的结果集 | explore_task / timestamp / explore_result |

> 关键理解：Context 是**全量信息池**，把"VLM/记忆/探索"等异构信号统一成 LLM 可消费的结构化片段。

---

### 四、阶段三：Attention 模块（吴仁杰，核心）★

这是整张图**最关键的创新点**，对应你 CLAUDE.md 里讲的"远区精简+近区完整"思路的**下一代升级**。

**职责：** 不是 Transformer 里那个 attention，而是一个**业务层的"注意力路由器"**。

**工作机制：**
1. 输入：Context 模块的全量信息（可能 50K+ tokens）
2. 处理：判断"此刻什么信息重要"+ "该召唤哪个 Advisor"
3. 输出：**带 @ 标签的浓缩 brief**，例如图中那段红字：

> "依依上车了，她今天化了个很美的妆，心情很不错的样子 **@情感advisor**，现在车里温度 42 度，空调没开 **@舒适advisor**，看看依依等会会去哪里 **@出行advisor**，最近依依关注的话题有没有什么更新 **@内容advisor**"

**为什么需要它？**
- 解决 SP 30K+UP 15K 的 token 爆炸 → 由 attention 做**信息蒸馏**
- 解决 Planner 一人包揽过载 → 由 attention 做**任务分发**
- 解决 KV Cache 命中率低 → 把动态信息聚焦到一个固定位置

---

### 五、阶段四：Advisors 模块（弋逋，绿色框）

**职责：4 个专业领域 Advisor 并行思考，各出建议。**

| Advisor | 输出 | 关注维度 |
|---|---|---|
| **舒适 Advisor** | 舒适空间建议 | 温度/空调/座椅/光照（对应你 CLAUDE.md 的"舒适"静态 Advisor） |
| **出行省心 Advisor** | 省心出行建议 | 导航/POI/路况（对应"出行"） |
| **内容 Advisor** | 内容消费建议 | 音乐/视频/新闻/电台（对应"内容"） |
| **情感 Advisor** | 情感互动建议 | 用户情绪/陪伴/对话风格（对应"情感"） |

这是从"Planner 一人包揽"演进到"团队协作"的具体实现——**Goal List 是共享看板，Advisors 是常驻专家**。

---

### 六、阶段五：Planner 决策（嘉锋）

**职责：**
- 输入：4 个 Advisor 的建议 + 反馈回路（图中底部有箭头从 Advisor 区回到 Context）
- 处理：决策、任务编排、生成 action_list
- 输出：**回复 + 任务**（即 `talk_content` + `action_list`）

注意图中底部有一条**反馈线**从 Advisor 区域回流到 Context 模块——这意味着 Advisor 的建议会**沉淀回上下文**，形成下一轮的输入，构成闭环。

---

### 七、与现有架构的对应关系

对照你 CLAUDE.md 里描述的演进路径：

| 旧版（单 Planner） | 新版（图中架构） |
|---|---|
| SP 里塞 11 类动态变量 | 拆给 8 个 Context 子模块 |
| Planner 自己做意图识别+拆解+执行 | Attention 做筛选，Advisors 做专业建议，Planner 只做最终决策 |
| 上下文优化靠"远区精简" | 上下文优化靠"Attention 蒸馏 + Advisor 分工" |
| 1 个 LLM 调用搞定 | 多 LLM 协作（Attention + 4 Advisor + Planner） |

---

### 八、一句话总结

> **Context 把世界切成片，Attention 挑出此刻重要的片并 @ 对应专家，Advisors 各出建议，Planner 拍板执行。**

这是一个把"Planner 单体大脑"拆成"专业分工的 AI 团队"的架构，**Attention 是新引入的指挥官/秘书角色**，承担了原本 Planner 要做的"看一眼全场决定关注什么"的工作。

# 🚗 Driver Agent 2.0 升级版架构：超详细逐模块拆解

## 📊 全景理解：这张图在解决什么问题？

旧版架构（你 CLAUDE.md 里描述的当前线上版本）的**4 个核心痛点**：

```
痛点1：Planner 一人包揽 → SP 30K+UP 15K 塞爆，注意力分散
痛点2：Context 杂乱无章 → 11类动态变量混在 SP 里，缓存命中率低
痛点3：响应速度 vs 决策质量 → 既要秒回又要深度思考，两难
痛点4：Advisor 利用不足 → 4个顾问只是 SP 里的文字描述，没实体
```

这张图的本质：**把 Planner 这个"超人"拆成一个"AI 团队"**，每个角色专精一件事。

---

## 🏗️ 五大阶段总览（数据流向）

```
┌─────────┐    ┌──────────────┐    ┌─────────┐    ┌──────────┐    ┌─────────┐
│ 原始信号 │ →  │ Context 切片 │ →  │Attention│ →  │ Advisors │ →  │ Planner │
│  3个源  │    │   8个子模块   │    │ 注意力路由│    │ 4个专家  │    │ 决策中枢 │
└─────────┘    └──────────────┘    └─────────┘    └──────────┘    └─────────┘
   生产者           结构化层          蒸馏+路由      并行专业建议        最终拍板
                                       ↑                                  ↓
                                       └──────── 反馈闭环 ─────────────────┘
```

# 🟦 阶段一：上游信号源（最左侧 3 个蓝框）

这是**最原始的数据生产者**，相当于一个城市的"传感器网络"。

## 1.1 VLM（视觉语言模型）

| 维度 | 内容 |
|---|---|
| **是什么** | 多模态大模型，看摄像头画面（舱内+舱外）并产出自然语言描述 |
| **输入** | 车内 RGB 摄像头 + 红外 + 摄像头朝向（DMS 驾驶员监测） |
| **输出** | "有人上车"、"主驾位有人坐下"、"后排小孩在哭"…… |
| **频率** | 持续监听，事件触发上报 |
| **图中触发** | 检测到"有人上车" → 发送给 watcher |

**为什么重要：** 它是"无声的眼睛"，让系统在用户**还没开口**之前就知道车里发生了什么——这是实现"主动服务"的前提。

## 1.2 长期记忆

| 维度 | 内容 |
|---|---|
| **是什么** | 跨 session 持久化的知识库（数据库形式存储） |
| **存什么** | ① 用户画像（年龄/性别/职业）② 偏好（音乐/口味/驾驶习惯）③ 高频指令记录 ④ 历史对话摘要 ⑤ POI 收藏 |
| **写入时机** | 用户主动说"帮我记一下…" 或 系统自动总结 |
| **读出时机** | 用户提到"我常去的"、"我喜欢的"…… |

**对应你 CLAUDE.md 的 Tool：** `search_user_memory` / `operate_user_memory`

## 1.3 探索模块

| 维度 | 内容 |
|---|---|
| **是什么** | 豆包的"自主后台进程"——用户不在时它也在思考 |
| **干什么** | 比如自己上网搜"GPT5 最新进展"、"今天用户可能感兴趣的新闻" |
| **产出** | `{explore_task, timestamp, explore_result}` 三元组 |
| **作用** | 让豆包在用户问"有什么新鲜事吗"时有"存货"，而不是临时去搜 |

**类比：** 像一个秘书在你出门时帮你整理好今日简报。

# 🟫 阶段二：Context 模块（灰色大框，8 个子模块）

**核心定位：** 不做决策，只做**"切片 + 格式化"**。把上游异构信号 → 转成统一可消费的结构化片段。

> 对应你 CLAUDE.md 里说的 **"公共 Context 管理"**——把原本散落在 Planner SP 里的 11 类动态变量，提到一个独立模块来统一管理。

## 2.1 watcher（产品：明骏）

| 字段 | 详情 |
|---|---|
| **数据源** | VLM 的视觉描述 |
| **格式化输出** | "依依上车坐在了主驾，今天画了一个美美的妆" |
| **特点** | 重点关注**变化**（剪发了/化妆了/穿新衣了），不是描述每一帧 |
| **更新频率** | 事件触发（人物进出、姿态变化） |
| **典型坑** | 不能太啰嗦，否则 SP 爆炸；不能太省略，否则 Planner 没感知 |

**为什么叫 watcher 不叫 vision：** 因为它的责任是"持续观察+总结"，不是简单的视觉识别 API 包装。

## 2.2 端信号（产品：婉婷）

| 字段 | 详情 |
|---|---|
| **数据源** | CAN 总线 / 车机端实时状态 |
| **格式化输出** | `{"空调温度": 22, "空调风量": 3, "空调模式": "自动", "车窗": "关闭", ...}` |
| **特点** | **快照式**（不是历史），只保留最新值 |
| **更新频率** | 高频轮询（秒级） |
| **覆盖范围** | 空调/座椅/车窗/车门/胎压/电量/驾驶模式/音量…… |

**对应你 CLAUDE.md 的 Tool：** `search_vehicle_status_info`——但现在前置到 Context 里直接提供，**Planner 不用再主动 Tool Call 去查**，省一次往返。

## 2.3 事件日志（产品：芮婷）

| 字段 | 详情 |
|---|---|
| **数据源** | 全量事件流（人物/车辆/助手的所有事件） |
| **格式化输出** | 时间线日志，55 分钟滚动窗口 |
| **覆盖事件** | Always On 感知事件 + Planner Tool 调用 + TTS 输出 + ASR 收到的语音 |
| **特点** | **按时间排序**，可以回溯 |

**典型用途：** 用户说"刚才那首歌是谁唱的？"——靠事件日志查最近播放的歌。

## 2.4 基础联网信息（产品：明骏）

| 字段 | 详情 |
|---|---|
| **数据源** | 联网 API |
| **格式化输出** | `{"精准时间": "...", "公历日期": "...", "星期": "...", "农历日期": "...", "节日": "...", "节气": "..."}` |
| **更新频率** | 每分钟 / 进入新一天时 |
| **为什么单独一类** | 时间是**几乎所有任务都需要**的全局变量，单独放方便所有模块引用 |

## 2.5 用户记忆（产品：白宇）

| 字段 | 详情 |
|---|---|
| **数据源** | 长期记忆数据库（注意区别于上游"长期记忆"——这里是**格式化好的快照**） |
| **格式化输出** | "user: 依依, 对话事件总结: 总结历史对话内容, 基本信息: 姓名/性别/年龄/生日/职业, 音乐偏好: 喜爱的歌手/作者" |
| **关键设计** | 不是 dump 全部记忆，而是**根据当前场景挑出相关 10-20 条**（防止 token 爆炸） |

**对应你 CLAUDE.md：** Tool `search_user_preferences` 的前置化版本。

## 2.6 短期记忆（产品：芮婷，图中标 ???）

| 字段 | 详情 |
|---|---|
| **数据源** | 本次启动到现在的会话记忆 |
| **状态** | **未完成设计**（图中 ??? 是 PM 留的疑问） |
| **可能定义** | 本次行程内的"上下文摘要"——区别于事件日志的细颗粒度 |
| **挑战** | 如何与"事件日志"和"长期记忆"不重叠？|

> 这是这张图里**最值得讨论的留白**，建议你接下来重点对齐。

## 2.7 豆包探索（产品：明骏）

| 字段 | 详情 |
|---|---|
| **数据源** | 探索模块的产出队列 |
| **格式化输出** | `[{"explore_task": "任务名1", "timestamp": "...", "explore_result": "..."}, ...]` |
| **使用时机** | 用户问开放问题（"今天有什么新鲜事？"）时由 Attention 选用 |

# ⭐ 阶段三：Attention 模块（产品：吴仁杰）—— 全图最关键创新

这个模块**完全是新设计**，旧版本不存在。它解决了**"Context 多了之后塞不进 Planner SP"**的核心矛盾。

## 3.1 它的本质：业务层的"注意力路由器"

⚠️ **不要和 Transformer 的 Attention 混淆**。这里的 Attention 是一个**独立的 LLM 调用**，职责是：

```
INPUT:  Context 模块产出的全量信息（可能 50K+ tokens）
        + 当前用户 query（或事件触发）
        
处理:    1. 判断哪些 Context 片段与当前情境相关
        2. 写成一段浓缩的自然语言 brief
        3. 在 brief 中用 @ 召唤对应的 Advisor

OUTPUT: 带 @ 标签的浓缩 brief（图中红字示例）
```

## 3.2 图中的实际输出示例

```
"依依上车了，她今天化了个很美的妆，心情很不错的样子 @情感advisor，
 现在车里温度 42 度，空调没开 @舒适advisor，
 看看依依等会会去哪里 @出行advisor，
 最近依依关注的话题有没有什么更新 @内容advisor"
```

**逐句拆解 Attention 是如何工作的：**

| 输入信号 | Attention 的判断 | 输出片段 |
|---|---|---|
| watcher: "依依上车坐在主驾，今天画了美美的妆" | 情绪+外貌变化 → 情感话题 | "心情很不错的样子 @情感advisor" |
| 端信号: "空调温度: 42°C, 空调: 关闭" | 温度异常 → 舒适问题 | "车里温度42度，空调没开 @舒适advisor" |
| 用户记忆: "依依工作日通常去公司" | 上车场景默认 → 出行话题 | "看看会去哪里 @出行advisor" |
| 探索模块: "依依关注 GPT5 进展" | 用户兴趣点 → 内容话题 | "关注的话题有没有更新 @内容advisor" |

## 3.3 它解决的三大问题

| 问题 | 旧版 | Attention 方案 |
|---|---|---|
| **Token 爆炸** | SP 30K 塞死动态变量 | Attention 把 50K Context 蒸馏到 1-2K brief |
| **Planner 过载** | 一个模型同时做"看全场+决策+执行" | 分工：Attention 看全场，Planner 只决策 |
| **KV Cache 命中率低** | 动态变量散落 SP 各处，前缀经常变 | Attention 输出聚焦在 UP，SP 保持稳定，**前缀缓存命中率大幅提升** |

## 3.4 为什么是它，不是 Planner 自己做？

因为这两个职责的**模型需求不同**：

- **Attention**：需要"看得广但不深"——轻量模型/中等模型即可（小参数+快推理）
- **Planner**：需要"想得深但范围窄"——大模型（豆包 Seed 1.8）

分开后，**Attention 可以用更便宜的模型，频繁调用，做信息蒸馏**；Planner 只在需要时调用，专注决策。

# 🟩 阶段四：Advisors 模块（产品：弋逋，绿色框）

## 4.1 整体定位

> **专科医生 vs 全科医生**：Planner 是全科医生（什么都看一眼），Advisor 是专科医生（一个领域看到底）。

每个 Advisor 是一个**独立运行的 LLM Agent**，**永久在线**（车一启动就启动），各自持有：
- 独立的 SP（专科知识库，比 Planner SP 简洁得多）
- 独立的上下文窗口（只关注自己领域的事件）
- 独立的输出格式（建议给 Planner）

## 4.2 四个静态 Advisor 详解

### 🛋️ 舒适 Advisor

| 维度 | 详情 |
|---|---|
| **盯什么** | 温度/湿度/空调/座椅加热通风/氛围灯/车窗/天窗 |
| **触发** | 端信号变化（温度突变/座椅异常） + Attention @ |
| **输出建议** | "车内温度较高，建议开启制冷"、"后排座椅按摩可启用" |
| **典型场景** | 用户没说热，但温度计 42°C → 主动建议开空调 |

### 🗺️ 出行省心 Advisor

| 维度 | 详情 |
|---|---|
| **盯什么** | 导航/POI/路况/充电桩/油量/路线偏离 |
| **触发** | 用户记忆中有目的地 + 上车 / 路况突变 |
| **输出建议** | "正在导航去公司，前方拥堵 5 公里，建议绕行" |
| **典型场景** | 推断用户大概率去哪 → 主动设置导航预案 |

### 📺 内容 Advisor

| 维度 | 详情 |
|---|---|
| **盯什么** | 音乐播放/视频/电台/新闻/有声书 |
| **触发** | 用户偏好 + 时段 + 心情信号 |

### ❤️ 情感 Advisor

| 维度 | 详情 |
|---|---|

| **典型场景** | 用户语气低落 → 切换为温柔陪伴模式 |

## 4.3 动态 Advisor（图中没画，但是体系一部分）

按需创建、用完销毁。比如：
- 用户："帮我介绍沿途风景" → 创建 **风景介绍 Advisor**
- 该 Advisor 持续监控 GPS → 经过景点提醒 Planner → 行程结束销毁

## 4.4 Advisor 建议如何回到 Planner

看你 CLAUDE.md 引用的代码：

```json
// role: user 是个数组，同时包含用户话和 advisor 建议
[
  {"query": "开启影院模式", "speaker_name": "罗勇", ...},
```

**关键设计：** Advisor 建议**伪装成 user role 输入**给 Planner，这样 Planner 可以同时看到"用户说什么"和"专家建议什么"。

# 🎯 Part 1：Context 到底是个什么东西？

## 1.1 先破除一个常见误解

很多人第一反应：**Context = Planner SP 里的一段文本**。

❌ **错。** 那是**旧版**的做法（把 11 类动态变量直接拼到 SP 里）。

✅ **新版 Context 是一个独立运行的「内存数据结构」**，**24 小时常驻**，不是某次调用时才生成的字符串。

## 1.2 Context 的本质：一个常驻的「黑板系统」

打个比方：

> 把 Context 想象成办公室里那块**白板**。
> - 不同的人（VLM/长期记忆/探索模块）随时上来写字、擦字、更新
> - Attention 想要信息时，**走过去看一眼白板**当前的状态
> - 白板不主动找人，但永远保持"最新可用状态"

技术术语：这叫 **"Blackboard Architecture"**（黑板架构），是经典 AI 系统的常见模式。

## 1.3 Context 内部的数据结构（脑补一下代码）

```python
class ContextManager:
    def __init__(self):
        # 8 个子模块各自维护自己的状态
        self.watcher_state = WatcherSlot()           # 最新视觉描述
        self.vehicle_signal = VehicleSlot()          # 端信号快照
        self.event_log = EventLogRing(window="55min") # 滚动窗口
        self.network_info = NetworkSlot()            # 时间/日期
        self.user_memory = UserMemoryCache()         # 用户记忆缓存
        self.short_term = ShortTermSlot()            # 短期记忆
        self.exploration = ExplorationQueue()        # 探索结果队列
    
    def render(self, scope="all"):
        """被 Attention/Planner 调用时，渲染当前快照"""
        return {
            "watcher": self.watcher_state.read(),
            "vehicle": self.vehicle_signal.read(),
            ...
        }
```

**3 种数据形态：**

| 形态 | 含义 | 例子 |
|---|---|---|
| **Slot（槽位）** | 只存最新值，新数据覆盖旧数据 | 端信号（空调温度 = 22°C，新数据来了就覆盖） |
| **Ring（滚动窗口）** | FIFO 队列，超时自动淘汰 | 事件日志（保留最近 55 分钟） |
| **Queue（队列）** | 累积式，按需消费 | 探索模块产出 |

## 1.4 Context 的两个关键时刻

```
时刻 A：上游推送 → Context 更新自己（白板被改写）
时刻 B：下游查询 → Context 渲染快照给查询者（有人看白板）
```

这两个时刻**完全异步**，互不阻塞。这就是为什么 Planner 的延迟可以做到很低——它不用等 Context 现去拉数据。

# 🎯 Part 2：上游数据如何到达 Context？

**核心结论：3 种模式并存**（不是统一一种）。

## 2.1 模式一：PUSH（推送）—— 事件驱动

**适用场景：** 状态变化稀疏 + 必须及时响应

```
VLM 检测到有人上车
    ↓ （立即推送）
watcher 槽位收到事件 → 更新内存
    ↓ （顺便）
事件日志记录一条
```

**实现机制：** 上游订阅事件总线（如 Kafka / 内部 EventBus），Context 模块监听特定 topic。

**典型例子：**
- VLM → watcher（人物进出/动作变化）
- ASR → 事件日志（用户开口说话）
- 探索模块 → 豆包探索（后台任务完成）

## 2.2 模式二：POLL（轮询）—— 高频快照

**适用场景：** 状态连续变化 + 不能漏掉

```
车机 CAN 总线
    ↓ （每 500ms 轮询一次）
端信号槽位 → 永远存最新值
```

**实现机制：** Context 模块自己起一个定时器线程，定期去车机硬件接口拉数据。

**典型例子：**
- 端信号（空调温度、车速、电量这些必须高频刷新）
- 基础联网信息（时间每秒走，但每分钟刷一次就够）

## 2.3 模式三：PULL ON DEMAND（按需拉取）—— 懒加载

**适用场景：** 数据量大 + 大部分场景用不到

```
触发条件：Attention 判断当前需要用户偏好
    ↓
Context.user_memory 模块发起查询
    ↓
长期记忆 DB 返回相关切片（比如 20 条）
    ↓
缓存到内存槽位（下次同场景免查）
```

**实现机制：** 类似数据库的 lazy loading，第一次访问时去查，之后缓存命中。

**典型例子：**
- 用户记忆（长期记忆 DB 太大，全量加载没意义）
- 短期记忆（按需从 session DB 拉）

## 2.4 三种模式的对照表

| 上游 | 模式 | 频率 | 为什么这么设计 |
|---|---|---|---|
| VLM → watcher | **PUSH** | 事件触发 | 视觉事件稀疏但重要 |
| CAN 总线 → 端信号 | **POLL** | 500ms | 连续变化必须高频 |
| 时间服务 → 基础联网信息 | **POLL** | 60s | 时间走得慢，省资源 |
| ASR/TTS → 事件日志 | **PUSH** | 事件触发 | 对话天然事件化 |
| 长期记忆 DB → 用户记忆 | **PULL** | 按需 | 数据量大，全量加载浪费 |
| Session DB → 短期记忆 | **PULL** | 按需 | 同上 |
| 探索任务 → 豆包探索 | **PUSH** | 后台完成时 | 异步任务结果 |

## 2.5 关键工程问题：谁来保证一致性？

3 种模式混用会出问题：
- PUSH 可能丢消息（网络抖动）
- POLL 可能滞后（500ms 间隙内的变化看不到）
- PULL 可能查到老数据（缓存没失效）

**解决方案：每个 Context 子模块自己定义 staleness 容忍度**

```python
class VehicleSlot:
    max_staleness = 1.0  # 秒，超过就标记为"可疑数据"
    
    def read(self):
        if time.now() - self.last_update > self.max_staleness:
            return {"data": self.value, "warning": "stale"}
        return {"data": self.value, "warning": None}
```

# 🎯 Part 3：Attention 是触发器吗？

## 3.1 直接回答：**部分是，但不只是。**

更准确说，**Attention 同时扮演 3 个角色**：

```
Attention = 触发器（Trigger） + 蒸馏器（Distiller） + 路由器（Router）
```

我一个个拆给你看。

## 3.2 角色 ① 触发器（Trigger）—— 你问的那个

**什么是触发器？** 监听某个条件，条件满足时启动下游流程。

Attention 的"触发"体现在：

```
事件：用户开口说话 / Context 发生重大变化 / 定时心跳
    ↓
Attention 被启动（这一步 Attention 本身被触发）
    ↓
Attention 决定 @ 哪些 Advisor → 等于触发了 Advisor（这一步 Attention 触发别人）
```

所以 Attention **既是被触发者，又是触发者**。它在事件链上承上启下。

**但是：** 如果只把它当触发器，会漏掉它最核心的价值。下面两个角色才是它的本职工作。

## 3.3 角色 ② 蒸馏器（Distiller）—— 信息压缩

这是 Attention 最核心的价值。

```
输入：Context 渲染的快照（可能 50K tokens）
         + 当前用户 query
         + 历史对话
         
        ↓ Attention 的 LLM 调用

输出：1-2K tokens 的浓缩 brief
```

**它在干的事 = 一个高级"摘要 + 推断"**：
- 从 50K 信息里**挑出现在最相关的部分**
- 用自然语言重新组织（不是机械拼接）
- 加入**推断**（比如 watcher 说"妆很美"，Attention 推断"心情不错"）

**为什么这一步必须有：**
- LLM 输入越长，注意力越分散（O(n²) 复杂度，长上下文质量下降）
- KV Cache 命中率：SP 稳定 + UP 集中变化 → 缓存复用率高
- Planner 不用看 50K 原始数据，只看 1K 蒸馏版，决策更准

## 3.4 角色 ③ 路由器（Router）—— 任务分发

这是 @ 符号的真实作用。

```
Attention 在 brief 中写：
"……心情不错的样子 @情感advisor，温度 42 度 @舒适advisor……"
                       ↑                          ↑
                  这是路由指令              这又是路由指令
```

系统层面看到这些 @ 标签，会做：

```python
for tag in extract_tags(attention_brief):
    if tag == "@情感advisor":
        emotion_advisor.activate(brief)
    elif tag == "@舒适advisor":
        comfort_advisor.activate(brief)
    ...
```

**这就是路由器**：Attention 不仅说了"现在重要的是什么"，还指明了"该谁来处理"。

## 3.5 三个角色的协同时序

```
T0: 用户上车（VLM 推送事件）
     ↓
T1: watcher 更新，触发 Context 变化通知
     ↓
T2: Attention 被启动 ← 【触发器角色：被触发】
     ↓
T3: Attention 调用 LLM，蒸馏 Context ← 【蒸馏器角色】
     ↓
T4: Attention 输出带 @ 的 brief ← 【路由器角色】
     ↓
T5: 系统解析 @，激活对应 Advisor ← 【触发器角色：触发别人】
     ↓
T6: Advisor 们并行思考，给 Planner 出建议
     ↓
T7: Planner 综合决策，输出 talk_content + action_list
```

## 3.6 一个更精确的比喻

如果一定要类比，Attention 更像**「秘书 + 调度员」的合体**，而不是单纯的触发器：

```
触发器 = 烟雾报警器（只会响）
Attention = 公司前台秘书
   ├── 收到来访者（被触发）
   ├── 判断这事归哪个部门（路由）
   ├── 写一份简报递给老板（蒸馏）
   └── 同时打电话叫对应部门来人（触发下游）
```

**所以你说"Attention 是触发器"，对一半——它**包含**触发逻辑，但本职工作是"信息加工 + 任务分发"。**
