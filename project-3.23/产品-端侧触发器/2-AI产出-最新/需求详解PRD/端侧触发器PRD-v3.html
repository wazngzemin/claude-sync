一、需求背景

1.1 项目背景

当前状态

豆包上车（Driver Agent 2.0）的车机 AI Box 上已经部署了端侧 VLM（Vision Language Model，方舟端 VLM），具备通过舱内/舱外摄像头实时感知车辆内外环境的能力。VLM 可以输出两种结果：精调标签（结构化字段，如"座位占用=主驾""挡位=P"）和自然语言 caption（如"后排左侧婴儿闭眼安静睡觉"）。硬件和模型能力已经就位。

缺什么

但 VLM 本身不具备自主决策能力——它不知道什么时候该开机、什么时候该关机、该跑什么任务、跑完结果给谁。目前没有一个模块来管理 VLM 的生命周期。具体表现为以下四个缺失：

#缺失能力当前状态导致的问题
1VLM 生命周期管理没有模块负责 VLM 的启停。AlwaysOn 三任务（舱外/舱内动态/舱内静态）什么时候开、什么时候关、用户关了 Awake Button 怎么办——都没有明确的管理方VLM 不知道自己该不该跑 → 要么一直跑（浪费算力+隐私风险），要么不跑（安全类任务漏触）
2统一信号源车端信号散落在各个底层服务中：挡位在车辆端状态服务、天气在外部 API、导航在导航模块、FaceID 在视觉模块。格式不统一（有的是 enum，有的是字符串，有的带单位有的不带），TTL 不可控，时序不可靠触发器如果各自去散读信号，会出现：① 格式不匹配导致条件误判 ② 过期信号导致误触发 ③ 人换座位时 FaceID 残留导致错人
3条件触发引擎一批"不需要用户主动发起、系统自动执行"的安全/体验类任务（下车安全、起步安全、遗留物品检测、天气路况保护等）需要基于信号条件自动触发，目前没有这个条件匹配和调度引擎安全类任务无法自动执行 → 用户停车开门时不会自动检测障碍物 → 需要用户自己注意安全
4VLM 结果回传链路VLM 跑完后结果直接给了 AI Service，触发器（调用方）不知道任务完没完、结果是什么触发器无法做续接调度（不知道上一轮结果）、无法释放互斥资源（不知道任务结束了）、AlwaysOn 被打断后无法恢复

1.2 需求来源

需求动因具体说明
VLM 需要调度器端侧 VLM（方舟端 VLM）不具备自主决策能力——它不知道"什么时候该开始观测""用哪个摄像头""跑完结果给谁"。需要一个触发器模块来管理 VLM 的生命周期：按条件开启/关闭 AlwaysOn 三任务，按事件触发 oneshot 任务，按优先级抢占互斥资源。
端侧信号散落目前车辆状态（挡位、车速、电源）、导航状态（剩余距离、是否地下停车场）、用户状态（FaceID、座位占用）等信号散落在各个底层服务中。触发器如果各自去散读，信号格式不统一、TTL 不可控、时序不可靠。需要先建一个信号库作为统一事实源，触发器只认信号库。
云端不能做的事三件事只能在端侧做：① 掐表——断网时本地定时器仍然准确，云端掐不了本地闹钟；② 开关 VLM——VLM 跑在车机上，开关信号（上电/下电/Button）也在本地，不需要绕云端；③ 隐私合规——舱内摄像头数据不允许上云存储，处理必须在本地完成。
系统预设任务一批"不需要用户主动发起、系统自动执行"的安全/体验类任务（下车安全、起步安全、遗留物品、天气路况保护等）需要基于信号条件自动触发，这是触发器的核心产品场景。
7.6 新增：结果透传7.6 会议确认 VLM 结果流向变更为"谁调返给谁"——端侧触发器调用了 VLM，VLM 结果必须先返回给触发器，再由触发器整包透传给 AI Service 数据中心。端侧触发器从"只管开关"升级为"开关 + 结果中继"。

二、需求目标

1.1 建设目标

端侧需要同时建设两个基础能力：

模块定义
端侧信号库统一存储端状态、OEM/用户信号、导航/场景信号、VLM 结果和任务状态。是触发器的前置依赖——信号库不成立，触发器规则就无法稳定运行。所有信号必须经过信号库，触发器不直接监听原始信号源。
端侧触发器基于规则和信号判断任务是否触发，并负责任务排队、优先级、互斥组、冷却、下发指令、接收终态、结果透传上报。7.6 新增：管理 AlwaysOn 三任务的开关 + VLM 结果整包透传。

1.2 产品定位

端侧触发器 = AI Box 上的手脚 + 眼睛 + 闹钟

跑在车机（AI Box）上，做四件事：

  1. 开关 VLM：管 AlwaysOn 三任务的启停，按条件点火/熄火
  2. 掐表：本地定时器，断网不停，到点叫醒
  3. 硬条件判定：挡位变P / 温度超30度 / 天气包含雾——认死规矩，不判语义
  4. 透传：VLM 结果整包透传给 AI Service（不解析），"谁调返给谁"

不是什么：6.22 西钺警告"不要开发两个触发器"——逻辑上它和云端触发器是一个触发器的两半，一个条件只判一次。端侧做的事 = 物理上只能在车端做的事。

2.3 一期目标与边界

✅ 一期做（7 项）

#功能名称对应模块功能描述交付节点
1AlwaysOn 三任务开关管理㉔-8管理舱外 AlwaysOn（3s 周期）、舱内动态（5s 周期）、舱内静态（5min 周期）三个 VLM 常驻任务的启停生命周期。根据 power_state（上电/下电）和 awake_button（用户隐私开关）两个信号做开关判断——开 = AND（两条件同时满足才开），关 = OR（任一条件不满足就关）。开启时向 VLM 下发启动指令并订阅结果 topic；关闭时下发停止指令并取消订阅。舱外只看 power，舱内看 power + button。818
2VLM 结果整包透传㉔-9 + ㉔-U"谁调返给谁"——VLM 执行完成后将原始输出先返回给端侧触发器（因为是触发器发起的调用），触发器收到后分两路处理：① 调度用：读取任务终态（success/empty/timeout/failed），更新任务队列，释放互斥组资源，决定续接策略；② 透传用：㉔-U 将 VLM 原始输出字符串不解析、不改写、不截断，附上 task_type（区分是哪个任务产出）+ timestamp（端侧时间戳）+ seq（序列号），整包推送给 AI Service 数据中心，由数据中心自行解析分发。818
3天气类端侧自闭环
（雾灯 / 湿滑 / 雪地）		㉔-J三个场景的完整自闭环：信号→条件判断→车控执行→恢复，全程不上云。
• 雾天：weather_type 包含 [雾, 大雾] → F→T 穿越 → 直接下发车控指令开启后雾灯；天气恢复后 → 下发关闭后雾灯
• 雨天：weather_type 包含 [小雨, 中雨, 大雨, 暴雨] → 开启湿滑模式；恢复后关闭
• 雪天：weather_type 包含 [小雪, 中雪, 大雪] → 切换雪地模式；恢复后切回
三个场景均支持频控配置（周期内几小时/几天/自然日一次），防止重复触发。信号来源：外部天气 API + VLM 舱外 AlwaysOn 天气标签。		818
4充电桩链式触发
（R→P + oneshot VLM）		㉔-J + ㉔-9两步链式触发：① ㉔-J 检测到 gear_state 从 R 变更为 P（硬条件，F→T 穿越）→ ② ㉔-9 向 VLM 下发 oneshot 任务（camera=rear，task_goal="检测后方充电桩"，priority=P0，timeout=5000ms）→ VLM 看一帧出结果 → 结果返回触发器 → 整包透传给 AI Service → 数据中心解析"有充电桩"→ 云端触发器下发执行"打开充电口盖"。一次停车周期只触发一次（cooldown_scope=trip）。818
5本地定时器基础能力㉔-7支持两种定时模式：倒计时（注册 delay_seconds，到期触发一次）和循环定时（注册 interval_seconds + max_repeat，最多循环 30 次）。全程基于本地系统时钟，断网不停。定时器状态（注册任务 + 剩余时间 + 循环次数）持久化到本地存储，上电后读取持久化状态：已过期的立即触发，未过期的继续倒计时。精度 ≤ 1 秒。定时器到期后向信号库写入一个事件信号，由 ㉔-J 做条件匹配。818
6规则接收 + 本地持久化㉔-J启动时从云端规则配置平台拉取全量预置规则（按车型/VIN 维度）→ 加载到内存 → 开始匹配循环。运行中支持增量更新（云端下发新/改规则 → 热加载 → 下一轮生效，不中断正在运行的任务）。支持灰度下发（配置平台按 VIN 维度灰度，端侧自动获取本 VIN 适用版本）。加载失败时自动回滚到上一版本并告警上报。接收云端触发器下发的动态注册任务（带 expire_at 过期时间）。规则本地持久化，掉电后可恢复。818
7端侧信号库基础建设信号库建设端侧的统一事实源——所有信号源（车辆端状态/导航/天气/FaceID/OEM/用户输入）统一写入，触发器只从信号库读取。每条信号标准化为 key/value/source/timestamp/ttl 五字段。支持覆盖写入（同 key 只存最新值）、变更通知（值变化时通知订阅方）、空值写入（空 ≠ 不存在）、批量写入（VLM 多信号原子写入）。TTL 过期管理：过期信号自动作废，不参与条件判定。支持快照读取（一次拿全部相关 key）和事件订阅（值变化时通知）两种消费方式。818

❌ 一期不做（6 项）

#功能名称为什么不做归属/计划
1SP 拼接/注入接收云端合并后的 System Prompt → 注入 AlwaysOn 的 VLM 槽位，使 VLM 按动态 Advisor 要求额外关注特定事物。7.6 丁彬明确"没聊清，拆开不阻塞"——SP 从哪来、怎么合并、和 AlwaysOn 默认 SP 怎么共存，整条链路没达成共识。"由上游业务方推动下游沟通"，端侧不主动做。二期；上游推动
2动态 Advisor 视觉任务下发动态 Advisor 创建的临时 VLM 观测任务（如"帮我观察后排小孩状态"）需要 SP 注入能力来告诉 VLM 关注什么。依赖 SP 注入（上一条）+ 黄线（动态 Advisor）+ 红线（云端触发器），三个前置依赖都没完全就绪。二期；依赖黄线+红线
3VLM 结果解析端侧触发器不解析 VLM 输出内容——不拆字段、不取标签、不判"看到了什么"。7.6 定版"整包透传不解析"，解析是 AI Service 数据中心的事。端侧触发器只管传，传完即丢。永远不做（归数据中心）
4语义条件判断用自然语言理解来判断条件是否满足（如判断 caption "后排有人闭眼" 是否意味着"在睡觉"）。这需要过模型推理，端侧触发器只做死规矩式运算符判定（=、≠、>、<、包含、变更为），永远不过模型。语义判断归云端触发器的事件判断 L1/L2/L3。永远不做（归云端触发器）
5冲突仲裁多个触发器任务之间的冲突仲裁（如"天气建议开雪地模式"和"用户手动设了运动模式"谁优先）。仲裁需要全局视角和业务理解，归云端触发器⑮负责。端侧触发器只做互斥组内的优先级抢占，不做跨场景的语义冲突仲裁。归云端触发器⑮
6事件判断 L1/L2/L3VLM caption 的语义解析三层：L1 枚举精调（零成本~80%）/ L2 提槽关键词（<10ms~15%）/ L3 模型兜底（200-500ms~5%）。这是云端触发器消费 VLM 结果后做的事，不在端侧做。端侧只管把 VLM 原始输出透传上去，判断是云端的活。归云端触发器⑮

二、整体架构

2.1 三模块组成

端侧由三个模块组成：信号库、触发器、VLM 模型。三者的关系如下：

关系链说明
信号库 → 触发器触发器读取/订阅信号库中的信号快照，用于规则条件匹配。触发器不直接监听原始信号源，所有信号必须经过信号库。信号库是唯一事实源。
触发器 → VLM条件命中后，触发器向 VLM 下发运行指令。运行指令包含四个核心信息：任务标识、识别目标描述、优先级、超时时间。触发器不关心 VLM 内部如何推理，只管下发和接收结果。
VLM → 触发器
7.6 新增		VLM 执行完成后将原始输出返回给触发器（"谁调返给谁"）。触发器不解析内容，整包透传给 AI Service 数据中心。数据中心自行解析分发。
VLM → 信号库VLM 执行结果由 VLM 业务写回信号库（任务状态类信号），供触发器做续接调度。注意：任务状态给触发器做调度（实时性要求高）；识别结果给整个系统消费（需持久存储 + 标准化上报）。两条路分开，不混。
为什么任务状态和识别结果要分两条路？如果合成一条路，存储会拖慢调度，或者触发器不得不承担结果存储职责，导致模块职责不清。任务状态走信号库（快·调度用）；识别结果走透传上报（大·消费用）。

3.2 "谁调返给谁"——VLM 结果流向变更（7.6 定版）

一句话原则

"谁调返给谁" = 是你下的单，结果就先回到你手上，不能绕过你直接给别人。

端侧触发器是 VLM 任务的唯一发起方（不管是 AlwaysOn 还是 oneshot，都是触发器告诉 VLM "去看"的），所以 VLM 看完的结果必须先返回给触发器，然后由触发器决定结果怎么分发。

用充电桩场景讲清楚：变更前 vs 变更后

变更前（有问题的做法）
① 端侧触发器 → 对 VLM 说："帮我看看后面有没有充电桩"
② VLM 看完了，输出 "后方3米处有充电桩"
③ VLM 直接把结果发给了 AI Service
④ 端侧触发器：❓"我是下单的人，结果没告诉我？我怎么知道任务完没完？怎么做续接调度？"

问题在哪：触发器不知道 VLM 跑完了没、结果是什么，无法做超时释放、互斥组抢占恢复、周期任务续接——调度链断了。
变更后（7.6 定版，正确做法）
① 端侧触发器 → 对 VLM 说："帮我看看后面有没有充电桩"
② VLM 看完了，输出 "后方3米处有充电桩"
③ VLM → 结果先回给端侧触发器（因为是触发器下的单——"谁调返给谁"）
④ 端侧触发器收到结果后做两件事：
　　a. 调度用：读取任务状态（success/empty/timeout/failed），做续接调度
　　b. 转发用：把 VLM 的原始输出整包透传给 AI Service 数据中心（不拆、不改、原样转发）
⑤ AI Service 数据中心收到后自行解析、分发给云端各消费方

三、端侧信号库

3.1 设计原则

信号库不是"建一个缓存表"，而是把端侧所有会影响任务触发的状态先收口成一个统一事实源（Single Source of Truth）。后续触发器只认信号库，不再各自去车辆、APP、云端、VLM 结果里散读状态。

原则说明
唯一入口所有信号源（车辆底层、导航、天气、VLM、用户输入）统一写入信号库，触发器只从信号库读取
标准化字段每条信号统一为 key / value / source / timestamp / ttl 五字段，不管来源是什么，格式一致
实时性分级瞬态信号（挡位变化）默认 TTL 5s，需按 key 可配。持久信号（天气状况）TTL 可达 30min
只存最新同一 key 只存最新值——信号库不是时序数据库，不存历史。历史存储是 Context/云端的职责
空 ≠ 不存在信号值为空（如 VLM 看了但没看到目标）是有效数据，和"从未写入过"不同
被动写入信号库自己不主动采集信号——由各信号源主动 push 写入。信号库只负责存储、索引、TTL 过期
信号库是触发器的前置依赖——信号库不成立，触发器规则就无法稳定运行。先建信号库，再建触发器。

4.3 业务流程（完整生命周期 15 步）

触发器在整个车辆运行期间持续执行以下循环，不是一次性的：

启动初始化：上电 → 加载全量预置规则 → 恢复持久化的定时器状态 → 初始化信号订阅
AlwaysOn 点火：根据上电/Button 状态 → 开启满足条件的 AlwaysOn 任务 → 订阅 VLM 结果 topic
获取信号快照：从信号库读取所有相关 key 的当前值 → 组成本轮快照 → 检查每个信号的 TTL 有效性
条件匹配：遍历所有已加载的规则 → 用运算符逐条比对信号快照 → 复合条件做 AND/OR 组合
穿越检测：对匹配成功的规则做 F→T 穿越检测——上一轮为 False、本轮为 True 才算穿越。防止持续满足条件时反复触发
防抖：穿越成功后进入防抖窗口——连续 N 帧（可配，默认 1）都穿越才算真正命中。防止信号抖动误触
命中确认：防抖通过 → 正式命中 → 进入命中后处理流程
优先级标记：为命中的任务标记优先级 P0-P4。P0 = 安全类不可延迟，P4 = 最低
互斥组检查：检查任务所属互斥组（VLM_CABIN / VLM_FRONT / 同时占用）→ 组内冲突时高优先级抢占
冷却检查：检查该任务的冷却状态——冷却期内不重复触发。P0 任务不设冷却（产品红线：每次满足必须触发）
下发运行指令：向 VLM 业务或车控执行模块下发运行指令——包含 task_id、task_goal、priority、timeout、mutex_group
VLM / 车控执行：VLM 按 SP 执行识别任务 / 车控直接执行动作（如开雾灯、切雪地模式）。端侧触发器不关心内部执行细节
接收 VLM 结果：VLM 完成后返回结果给触发器（"谁调返给谁"）。触发器接收原始输出字符串
透传上报 + 终态处理：① VLM 结果整包透传给 AI Service ② 读取终态（success/empty/timeout/failed）→ VLM 业务将任务状态写回信号库
续接：根据任务类型做续接——周期任务回到步骤3等下一周期；单次任务标记完成；动态任务检查 expire_at → 回到步骤 3，循环继续
关于循环回路：从步骤 15 会回到步骤 3（获取信号快照），形成触发器的持续运行循环。触发器在整个车辆运行期间（上电→下电）持续执行这个循环，而不是一次性的。

4.4.3 穿越检测（F→T 边沿触发）

穿越检测是闸门链的核心——防止"条件一直满足就一直触发"。

上一轮结果本轮结果是否穿越说明
FalseTrue✅ 穿越条件从不满足变为满足——触发
TrueTrue❌ 不穿越条件持续满足——不重复触发
TrueFalse❌ 不穿越条件从满足变为不满足——复位
FalseFalse❌ 不穿越条件持续不满足——等待
为什么不能"条件满足就触发"：假设直接配 battery_level ≤ 20% 就触发——
8:20 电量首次降到 20% → 触发 ✅
8:21 电量仍然 20% → 又触发 ❌ 重复弹窗
8:25 电量 19% → 又触发 ❌
8:35 充了点又降回 20% → 又触发 ❌ 二次骚扰
所以必须有穿越检测：只认 F→T 那一刻，之后持续满足不再触发，直到条件复位（T→F）后再次 F→T。

4.7.1 终态类型

VLM 业务 / 车控执行完成后回传终态，触发器接收并处理：

终态含义触发器处理
success正常完成·有结果① 整包透传给 AI Service ② VLM 业务写回信号库 ③ 进入续接
empty正常完成·空结果同 success 处理——空是有效结论 ② 进入续接
timeout执行超时释放资源 → 不写伪结果 → 进入续接（周期任务等下一轮，单次任务可选重试）

4.7.2 续接规则

任务类型续接行为说明
周期任务回到步骤3（信号快照），等待下一个执行周期如 AlwaysOn 高频每 3s 一次，完成一次后等 3s 再回到步骤 3

动态任务按 expire_at 判断是否失效；未失效继续按条件命中过期则标记为 expired 并清理

4.8 关键策略（横切关注点）

4.8.1 AlwaysOn 闲时常驻

AlwaysOn（舱内 + 舱外）是行车中 VLM 的默认工作状态，不存在"VLM 空闲"这个状态：

  • 被高优任务打断后，由 VLM 业务自行恢复，触发器不参与恢复调度
  • 分高频（每 3s / 5s）和低频（每 5min），低频共享高频的视频流

九、异常处理与产品红线

9.1 异常处理策略（7.4 评审通过）

异常类型处理策略设计原则补充说明
网络波动积压周期类：只传最新帧，压包丢弃中间帧
oneshot：暂存本地，网络恢复后续发一次		周期 = 时效优先（过期没用）
oneshot = 可靠优先（丢了无法重现）		AlwaysOn 断网：本地缓存最新一帧，恢复后推最新
信号 TTL 过期信号作废 → 依赖该信号的场景 frozen防过期数据误触发不同信号 TTL 不同（见 §3.6）
端侧不留存端侧不做数据持久化，只管透传。断网积压的续发完即清端侧算力/存储有限唯一持久化的是：定时器状态 + 任务列表（掉电恢复用）
时序脏数据人换座位 → 先上报一帧空数据清掉前值 → 再上报新值防旧 face_id 残留导致误判超然 7.4 提出：一空一实保证数据干净
VLM 超时单次超过 timeout → 释放资源 → 跳过本帧 → 不写伪结果不阻塞后续周期超时阈值待与丁彬对齐
规则加载失败自动回滚到上一版本 + 告警上报不能用坏规则运行回滚后继续用旧规则匹配
VLM 连续失败连续 N 次 failed/timeout → 暂停该任务 → 告警上报 → 等待人工介入或新规则防止无效重试消耗资源N = [待研发定义]

9.2 产品红线（不可逾越）

#红线说明
1P0 任务不设冷却安全类任务每次满足条件必须触发。延迟或漏触可能造成人身安全风险
2超时不写伪结果VLM 执行超时不能写"无障碍"——模型没看完，结果不可信。写了就是欺骗用户
3过期信号不判定TTL 过期的信号不允许用于条件匹配。宁可 frozen 等新值，不能用过期数据误触发
4一条一判不双判同一个条件只在端侧或云端的一处判定。两边都判 = 逻辑混乱 + 可能重复触发
5不解析不改写透传数据端侧触发器透传的 VLM 输出必须是原始字符串，不得解析、修改、截断、格式转换
6空结果是有效结果"看了没看到"和"没看"是两回事。空结果必须正常写入、正常透传，不能丢弃
