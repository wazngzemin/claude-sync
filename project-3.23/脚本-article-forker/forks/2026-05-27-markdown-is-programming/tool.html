MODULE 2手写量 = 控制颗粒度
学习目标：量化理解SP中每一部分的"控制力"，识别你的"遗留代码"盲区。

图2：SP Token分布 — 示例占50%，是控制Planner行为最关键的杠杆

自检方法（5分钟快速自测）

随机抽取以下各一条，问自己能否说出它的"来源"：

  • 一条注意事项 → 它是因为哪个badcase加上去的？解决了什么问题？
  • 一个工具描述的某一句 → 为什么这么写？不这么写会出什么错？
  • 一个参考示例 → 它测试的是什么边界条件？代表哪类用户行为？

如果说不出来 → 那部分不是"你的源代码"，是"别人的遗留代码"——你需要重新理解它，或者删掉它。

思考题：王建硕说"越偷懒，AI给你的就是它的缺省值"。对你的SP而言，"缺省值"是什么？
正确。SP越模糊 → 模型越依赖训练数据中的"平均"模式 → 输出通用但不精确。这就是为什么示例越多、描述越详细，Planner越"像你想要的"。
不完全对。SP模糊时，模型不会不工作（C），也不会完全没有车载知识（A有一部分对）——但核心是：它会给出"平均"响应而非"精确"响应。
MODULE 3四个行为转变：从写文档到写程序
学习目标：掌握将SP维护工程化的四个具体行为——commit、测试、review、重构。每个转变都有可复制的SOP。

图3：SP工程化流程 — 六步循环 + 定期重构，从"试试看"变为"精确迭代"

转变1：SP修改 = commit

SOP 每次SP修改必须记录changelog，格式如下：

## SP变更日志 ### [YYYY-MM-DD] 修改位置：具体描述 - **触发badcase**: 用户说了什么？Planner做错了什么？ - **根因分析**: 链路/工具描述/上下文/模型？具体哪里不对？ - **修改内容**: 改了SP的哪个部分？怎么改的？ - **影响范围**: 这个改动会影响哪些场景？ - **回测结果**: X个历史case中Y个修复，Z个无关，W个回归 - **上线时间**: YYYY-MM-DD HH:mm

转变2：回测 = 单元测试

SOP 每次改SP前跑回测（3步，共50分钟）：

步骤做什么时间
1. 准备测试集从badcase库选10个相关case的user_query + 预期输出15分钟
2. 构造调用用新SP + 历史UP构造Planner请求20分钟
3. 对比判断逐个比对新输出 vs 预期，标记通过/失败15分钟

转变3：SP评审 = Code Review

SOP 6项检查清单（改SP前过一遍）：

#检查项通过标准
1内部一致性：新修改是否和现有规则矛盾？无逻辑冲突
2影响范围：会影响哪些场景？已列出并验证
3示例覆盖：改了规则后，现有示例还能覆盖吗？至少1个示例对应
4边界条件：极端case下新规则表现如何？不导致更差行为
5Token开销：增加了多少token？ROI合理？ROI > 1
6可回滚性：上线后有问题怎么快速回滚？changelog有前一版本

转变4：清理技术债 = 重构

框架 SP技术债五种类型：

债务类型表现影响清理方法
老化示例示例中参数格式已更新模型学到错误行为定期审查时效性
规则冲突两条注意事项互相矛盾模型行为不确定统一规则优先级
冗余描述同一行为多处描述浪费token+冲突风险去重统一到一处
过度规则为1个case加的规则影响100个正常case矫枉过正缩小适用范围
知识腐烂车型更新但SP没同步幻觉或遗漏同步硬件更新

Checkpoint

你能否用一句话描述你上一次SP修改的"commit message"？如果说不出来，从今天开始建changelog。

MODULE 45分钟闭环愿景——你的迭代可以多快？
学习目标：分析当前迭代的瓶颈环节，找到可以立即加速的3个点。

图4：三条时间线对比 — 你不需要达到5分钟，达到1-2天就已经是10倍提速

Prompt模板：Badcase自动分类

模板 复制下方Prompt，每天早上跑一次：

以下是今天的Planner badcase日志（共N条）： {粘贴badcase日志} 请按以下四分法分类每条badcase： - 【链路问题】：请求根本没到Planner（走错了层级RAG/仲裁） - 【工具描述问题】：到了Planner但选错工具/传错参数（SP工具定义不清晰） - 【上下文问题】：选对了工具但缺关键信息（缺记忆/车况/目标） - 【模型能力问题】：信息完整但推理错误（需升级模型或加更多示例） 输出格式： | # | Badcase摘要 | 分类 | 严重度(P0-P3) | 建议修改方向 | 最后给出今日总结： - 最高优先级应修复的1-2个问题 - 是否有可以合并处理的同类问题

Prompt模板：SP修改方案生成

模板 确定了要修什么后，用这个生成3套方案：

我是车载AI助手的产品经理，负责Planner模块的System Prompt维护。 当前问题（badcase描述）： {用户说了什么 → Planner做了什么 → 预期应该做什么} 当前SP中相关部分： {粘贴SP中你认为需要修改的片段} 请生成3套SP修改方案： - 方案A：最保守修改（改动最小，只解决这一个case） - 方案B：中等修改（解决这一类case，可能需要改规则描述） - 方案C：根本性修改（重新组织这部分SP逻辑，彻底解决问题） 每个方案请给出： 1. 具体修改内容（精确到哪句话改成什么） 2. 预计影响范围（可能影响哪些其他场景） 3. Token变化（增加/减少约多少token） 4. 风险评估（可能引入什么新问题）

Checkpoint

下一次你需要改SP时，尝试用方案生成模板跑一次——对比"凭感觉改"和"3套方案对比后选最优"的体验差异。

MODULE 5聊天 vs 编程——你的AI使用在积累吗？
学习目标：区分"消耗型"和"积累型"AI使用，把更多工作从"聊天"转化为"编程"。

图5：聊天 vs 编程 — 判断标准："同类问题下次出现时，Planner能自动处理吗？"

王建硕的两个金句

给普通人："少聊天，多写程序" — 别停在对话框，把交互固化成可复用流程

给程序员："少写代码，多写程序" — 别停在代码层，在更高层（.md）编程

给你："少在脑子里推演，多往SP里写" — 每一条分析结论都应该变成SP中的一行规则

判断题：你上周分析了一个badcase（用户说"帮我调低空调"，Planner调用了vehicle_basic_control但参数写反了），你在群里说了分析结论。这是"聊天"还是"编程"？
正确。"说了"不等于"改了"。只有落地到SP中——比如在vehicle_basic_control的描述里加一句"decrease = 数值递减（如温度从26降到24）"——下次才不会再错。这就是"编程"。
不完全对。关键不是你分析了什么或谁知道了，而是SP有没有被修改。只有写进SP的东西才能被Planner"执行"。

代码工程实践映射到SP管理你可以做的
