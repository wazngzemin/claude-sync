文档定位：本文档聚焦 VLM 视觉感知在端云协同触发器体系中的 5 个核心场景。每个场景包含：流程图 → 链路拆解 → 关键节点说明 → 缺口与待拍板项。

1. 场景总览

VLM 在 Driver Agent 2.0 中有两种部署形态、两条调用路径，覆盖 5 类典型场景：

#场景VLM部署调用路径优先级车型
1即时视觉问答云端VQA⑳Director → VQA → DirectorP0上汽+赛力斯
2端侧持续感知（条件触发）端侧VLM⑰→⑮→㉔→㉕→④→㉑P0赛力斯
3持续监控+主动服务端侧VLM完整七阶段闭环P1赛力斯
4Always-on 安全感知端侧VLM㉔常驻→㉕→④→㉑P0赛力斯
5VLM 视觉参照物临停端侧VLMDirector→司机Agent→VLM→智驾P0赛力斯
判定规则：有端侧VLM硬件 + 持续感知 = 端侧路径；即时问答 = 云端VQA路径；有端侧VLM但query不支持 = fallback到云端VQA。

2. 场景01 — 即时视觉问答（云端VQA）

云端P0 典型话术："前面是不是有个加油站" / "车后面有没有人" / "旁边那个牌子写的啥"

2.2 链路拆解

#模块干什么输入→输出
1① 语音 → ② ASR用户说"前面是不是有个加油站"，ASR转文字语音 → query
2⑨ 意图分流判为纯对话/查询（不是车控、不是条件任务）→ 绿线→ ⑲⑳ Director
3⑳ Director推理判断需要视觉辅助 → 调用 search_visual_info 工具→ 云端VQA
4☁ 云端VQA取一帧前向摄像头图片 → 送云端VLM模型（纯文本prompt + 图片）→ 返回 {result: true, description: "前方200米左侧有中石化加油站"}→ Director
5⑳ Director回答"前面200米左边确实有个中石化~要不要帮你导航过去？"→ TTS播报
关键点：全程不经触发器⑮、不经㉔、不经㉕。Director直接调云端API，云对云闭环。上汽和赛力斯都走这条路。

4. 场景03 — 持续监控+主动服务（完整七阶段）

端侧P1 典型话术："帮我照顾二排宝宝，哭了或者踢被子告诉我"

4.1 流程图

4.2 链路拆解（详细）

阶段①：用户开口 → Director创建目标

  1. 用户说"帮我照顾二排宝宝，哭了或者踢被子告诉我" → ASR转文字
  2. ⑨判为复杂对话（不是车控、不是条件话）→ 绿线进Director
  3. ⑳Director识别为持续监控任务，需要VLM → 三个输出：① TTS "好的帮你盯着宝宝~" ② goal_list_update 创建目标 ③ create_dynamic_advisor 建专属监工

阶段②：⑰注册视觉订阅 → ⑮编译下发

  1. ⑰向⑮注册两条视觉订阅：{query: "baby_crying", interval: 5s} 和 {query: "baby_kick_blanket", interval: 10s}
  2. ⑮ 15-1校验能力 → 15-5编译成VLM task（补全camera_group/seat/priority/mutex_group等7字段） → 15-9判赛力斯有端侧VLM→下发端侧 → 15-12打包规则包增量下发㉔

阶段③：㉔调度 → ㉕持续感知

  1. 24-1接收规则包 → 24-7仲裁摄像头资源（如Always-on P2占着 → P1抢占） → 24-8转运行指令下发㉕
  2. ㉕开始工作：每5秒拉一帧舱内图 → 送端侧VLM推理 → "宝宝在哭吗？" → false...false...
  3. 车上本地跑，断网不影响

阶段④：宝宝哭了 → 结果回传

  1. ㉕检测到宝宝持续哭泣>10秒 → result=true
  2. 24-9 归一五态=success → 24-10 阈值穿越 last=false/this=true → F→T 触发！
  3. 24-11 回写④信号库 {signal: "rear_baby_crying", value: true} → ④同步⑫ → ㉑Context更新
  4. 不直接通知⑰（铁律②），统一走信号库→Context

阶段⑤：Advisor出建议 → Director拍板

  1. ⑰从㉑Context读到"宝宝在哭" → 推理产advice："降音量+温柔提醒+调高温度"
  2. ⑳Director拿到advice + Context（高速开车中/音量80%/温度24°） → 拍板：音量→20%、温度→25°、TTS"宝宝好像哭了，帮你把音乐调小了~要不要靠边看看？"

阶段⑥：任务结束

  1. 用户说"不用盯了" → Director撤销目标 → ⑰销毁 → ⑮规则标记deleted → ㉔清理本地task → ㉕停止拉图 → 24-7恢复Always-on

端侧P0 典型场景：驾驶员疲劳检测 / 驾驶员注意力分散 / 安全带未系 / 二排儿童状态

5.1 流程图

5.2 链路拆解

#模块干什么与场景02的区别
1㉔ Always-on上电即启动，不需要注册，常驻运行。P0优先级，不可被任何P1/P2任务抢占场景02需要⑰注册→⑮下发；Always-on是预置的
2㉕ 感知器持续拉图推理，检测驾驶员疲劳/注意力分散/安全带等相同
324-9 五态success/empty/failed/timeout/interrupted相同，但 timeout 必须标记 unknown，不能写"无异常"（安全红线）
424-10 阈值穿越F→T 才触发（防重复报警）相同
524-11 → ④ → ㉑回写信号库 → Context → Advisor/Director消费相同
Always-on vs 普通VLM任务的关键区别：① 不需要注册，上电即跑 ② P0优先级，不被抢占 ③ 被P1抢占时暂停不删除，P1完成后自动恢复 ④ timeout 不写"无异常"（模型没返回 ≠ 环境安全）

6. 场景05 — VLM视觉参照物临停（舱驾融合）

端侧P0 典型话术："停在前面拉行李箱的人旁边" / "停在那个蓝色雨棚下面" / "看到幼儿园门口停一下"

6.1 流程图

6.2 链路拆解

#模块干什么输入→输出
1⑳ Director理解用户有"视觉参照物靠边停车"需求 → 调用司机Agentuser query + context(智驾状态) → 司机Agent
2司机Agent提槽：提取视觉目标自然语言描述（"拉行李箱的人"）+ 方位（前方右侧）→ 端侧VLM + 智驾
3端侧VLM前向摄像头拉图 → VLM推理 → 第1帧：未检测到 → 第2帧(2s后)：检测到目标 → 返回2D坐标→ 智驾
4智驾（并行）收到指令后先减速靠边（不等VLM结果），VLM返回坐标后精确停靠到参照物旁→ 停车完成
5⑳ Director回复"已经停在那个拉行李箱的人旁边了~"→ TTS

6.3 ODD约束（使用范围）

约束项规则超出约束时的处理
车速≤ 30kph（实际支持~35kph含抖动）提示"车速有点快，看不到{参照物}，请降到30再试"
方向仅支持前向、向右靠路边停靠不支持向左/对向车道靠边
智驾状态必须在智驾行车模式下非智驾模式不触发
与触发器体系的关系：场景05不走触发器链路（不经⑮/㉔/㉕），而是Director直接调用司机Agent→端侧VLM→智驾。这条链路属于舱驾融合的即时任务，和场景01（云端VQA即时问答）类似，区别是需要端侧VLM做实时检测+智驾执行物理动作。
