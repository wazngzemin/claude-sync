## 定义

**SP即源代码——用编程思维重新定义你的Planner维护工作**：王建硕提出".md是新时代的编程语言，LLM是编译器"，本教程将其精确映射到你的Planner SP维护工作中。核心转变：你不是"写文档的PM"，你是"用自然语言编程的工程师"——SP是源代码，Planner的JSON输出是编译产物，badcase回测是单元测试，SP changelog是版本控制，SP评审会是code review。

## 详细说明

### 一、核心范式转换：你已经在"编程"了

王建硕的框架：

```
传统编程：    程序员 → 写C/Python → 编译器 → 机器码 → 执行
新编程：      PM → 写.md → LLM（编译器）→ 代码/行为 → 执行
你的工作：    你 → 写SP → Seed 1.8（编译器）→ Planner JSON决策 → 下游Agent执行
```

**为什么这不是比喻，是精确描述：**

| 编程概念 | 传统编程对应 | 你的SP维护对应 |
|---------|------------|--------------|
| 源代码 | `.c` / `.py` 文件 | SP文档（~30K-35K tokens） |
| 编译器 | gcc / python | 豆包 Seed 1.8 模型 |
| 编译产物 | 可执行文件 / 字节码 | Planner的JSON输出（talk_or_not/talk_content/action_list） |
| 运行时环境 | OS + 内存 + CPU | UP（对话历史）+ 动态变量（车况/记忆/目标） |
| Bug | 运行时错误 | Badcase — Planner输出与预期不符 |
| 修Bug | 改源代码 → 重新编译 | 改SP → 下次调用自动生效 |
| 单元测试 | pytest / jest | Badcase回测 — 拿历史case验证SP修改 |
| 版本控制 | git log | SP changelog — 每次修改记录原因和效果 |
| Code Review | PR review | SP评审 — 多人审查SP修改的合理性 |
| 重构 | 不改行为，优化结构 | SP精简 — 不改效果，减少token消耗 |
| 技术债 | 写死的逻辑、过时的注释 | SP中的老化示例、过时规则、互相矛盾的注意事项 |
| 依赖管理 | package.json | 23个工具定义的版本管理 — 下游接口变了SP要同步更新 |

**关键判断**：当你的SP出了badcase，王建硕的框架告诉你——不要去Planner的输出层修（那是在改"汇编"），要回到SP层修（改"源代码"）。你已经在这么做了。但你可能缺的是：**像对待代码一样对待SP的工程化纪律**。

### 二、"手写自然语言量 = 控制颗粒度"——量化你的SP

王建硕："你亲手一个字一个字敲的自然语言的数量，约等于你对项目的控制颗粒度。"

**映射到SP的具体含义：**

| SP组成部分 | 你亲手写的量 | 控制到什么颗粒度 | 如果你偷懒会怎样 |
|-----------|------------|----------------|----------------|
| ①角色定义 | ~200字 | 整体人设 | Planner回答风格飘忽不定 |
| ②输入类型定义 | ~500字 | 三种输入的处理方式 | 混淆user_query和tool_feedback |
| ③23个工具定义 | ~8000字（核心） | 每个工具何时调用、怎么调用、什么参数 | 工具误调、参数遗漏、时机错误 |
| ④输出格式 | ~500字 | JSON结构的每个字段 | 格式错乱、字段缺失 |
| ⑤注意事项 | ~1000字 | 10条边界case的处理 | 踩已知的坑 |
| ⑥参考示例 | ~15000字（占50%） | 69个具体场景的标准答案 | 模型只能靠"感觉"推测你想要的行为 |
| ⑦车辆设备知识 | ~3000字 | D6X的硬件约束 | 幻觉（说出不存在的功能） |
| ⑧工具tips | ~1500字 | 微妙的使用技巧 | 工具调用"能用但不好用" |
| ⑨聊天风格 | ~500字 | 语气、用词、人设一致性 | 有时温柔有时冷漠 |
| ⑩动态变量 | 模板，运行时注入 | 实时状态感知 | 不知道当前车况/位置/对话状态 |

**SP总量 ~30K tokens ≈ 你的控制颗粒度。**

但这30K中，你"亲手一字一字敲"的有多少？如果大部分是从别人那里继承的、从模板复制的——那些部分的控制颗粒度可能不够。

**自检方法：**
- 随机抽一条注意事项 → 你能说出它是因为哪个badcase加上去的吗？
- 随机抽一个工具描述 → 你能说出它的每一句话为什么这么写吗？
- 随机抽一个示例 → 你能说出它测试的是什么边界条件吗？
- 如果你说不出来 → 那部分SP不是"你的源代码"，是"别人的遗留代码"

### 三、从"写文档"到"写程序"——四个关键行为转变

#### 转变1：SP修改 = commit（不是随手改）

**旧模式**（写文档心态）：
- 发现badcase → 直接改SP → 上线 → 不记录
- 下次发现类似问题 → 不确定之前改过什么 → 可能重复改或覆盖

**新模式**（写程序心态）：
- 发现badcase → 记录问题 → 写SP修改 → 记录changelog → 回测 → 上线

**SP Changelog 模板：**

```markdown
## SP变更日志

### [2026-05-27] 工具描述：search_poi_qa 参数优化
- **触发badcase**: 用户说"附近有加油站吗"，Planner未传递location参数
- **根因分析**: 工具描述中未明确"location为必传参数，从env_info.location获取"
- **修改内容**: 在search_poi_qa描述中增加"必须从env_info.location获取当前位置作为搜索起点"
- **影响范围**: 所有POI查询类请求
- **回测结果**: 10个历史location相关badcase中8个修复，2个无关
- **上线时间**: 2026-05-27 14:00

### [2026-05-26] 注意事项：新增"多步任务确认规则"
- **触发badcase**: ...
```

#### 转变2：badcase回测 = 单元测试（不是"上线看看"）

王建硕的5分钟闭环之所以可能，是因为有自动化测试。你的SP修改目前缺少"测试"环节：

**旧循环**：改SP → 上线 → 等1-2周看日志 → 发现新问题 → 再改
**新循环**：改SP → 用10个历史badcase回测 → 确认修复/无回归 → 上线

**回测操作SOP（3步）：**

| 步骤 | 做什么 | 工具 | 预计时间 |
|------|--------|------|---------|
| 1. 准备测试集 | 从badcase库中选10个相关case的user_query + 预期输出 | Excel/飞书表格 | 15分钟 |
| 2. 构造调用 | 用新SP + 历史UP构造Planner请求，获取输出 | API调用脚本 | 20分钟 |
| 3. 对比判断 | 逐个对比新输出 vs 预期输出，标记通过/失败 | 手动 or AI辅助判断 | 15分钟 |

**Prompt模板——让AI帮你判断回测结果：**

```
我修改了Planner的System Prompt中的一条规则：
【修改前】：{旧规则文本}
【修改后】：{新规则文本}

以下是10个测试case的Planner输出（用新SP跑出来的）：

Case 1:
- 用户输入：{user_query}
- Planner输出：{new_output}
- 预期行为：{expected_behavior}

Case 2: ...

请逐个判断：
1. 这个case的Planner输出是否符合预期行为？（是/否/部分）
2. 如果否，哪里不符合？可能的原因？
3. 总结：10个case中通过了几个？有没有引入新问题（回归）？
```

#### 转变3：SP评审 = Code Review（不是一个人闷头改）

**Code Review的核心价值不是"找错"，是"防止盲区"。**

你一个人维护SP时最大的风险：
- 改了A工具的描述，没意识到影响了B工具的调用逻辑
- 加了一条注意事项，和第3条注意事项矛盾
- 优化了某个示例，破坏了另一个示例试图教的行为模式

**SP Review Checklist（6项）：**

| 检查项 | 检查方法 | 通过标准 |
|--------|---------|---------|
| 1. 内部一致性 | 新修改是否和现有规则矛盾？ | 无逻辑冲突 |
| 2. 影响范围 | 这个改动会影响哪些场景？ | 已列出并验证 |
| 3. 示例覆盖 | 改了规则后，现有示例还能覆盖吗？ | 至少1个示例对应新规则 |
| 4. 边界条件 | 极端case下新规则表现如何？ | 不会导致更差的行为 |
| 5. Token开销 | 这个改动增加了多少token？值得吗？ | ROI合理 |
| 6. 可回滚性 | 如果上线后有问题，怎么快速回滚？ | changelog记录了前一版本 |

**Prompt模板——让AI帮你做SP Review：**

```
我准备修改Planner SP的以下部分：

【当前SP全文（或相关片段）】：
{sp_section}

【我的修改】：
{my_change}

【修改原因】：
{reason_and_badcase}

请帮我做Code Review，检查以下6项：
1. 内部一致性：这个修改是否和SP中其他规则矛盾？
2. 影响范围：可能影响哪些场景？
3. 示例覆盖：现有示例是否还能覆盖新规则？
4. 边界条件：极端case下表现如何？
5. Token开销：增加了多少token？值得吗？
6. 可回滚性：回滚方案是什么？

特别注意：我的SP中有23个工具定义和69个示例，请检查我的修改是否和其中任何一个产生冲突。
```

#### 转变4：SP技术债清理 = 重构（不是不动）

王建硕说"屎山不用管"——但那是说**编译产物**（AI写的代码）不用管。你的SP是**源代码**，源代码的技术债必须管。

**SP技术债的典型表现：**

| 债务类型 | 表现 | 影响 | 清理方法 |
|---------|------|------|---------|
| 老化示例 | 示例中的工具参数格式已更新，示例没同步 | 模型从错误示例学到错误行为 | 定期审查示例时效性 |
| 规则冲突 | 注意事项第2条说"优先响应"，第7条说"先确认再执行" | 模型困惑，行为不确定 | 统一规则优先级 |
| 冗余描述 | 同一个行为在3个地方重复描述，措辞不同 | 浪费token + 增加冲突风险 | 去重，统一到一处 |
| 过度规则 | 为了解决1个case加了1条通用规则，影响了100个正常case | 矫枉过正 | 缩小规则适用范围 |
| 知识腐烂 | D6X新增了某功能，SP中的设备知识没更新 | 幻觉或遗漏 | 同步硬件更新 |

**重构原则（类比代码重构）：**
- 重构不改行为——改了SP结构但Planner输出不变
- 有测试保障才能重构——先有badcase回测集，再动SP
- 小步前进——一次只改一处，验证后再改下一处

### 四、5分钟闭环的愿景——你的Planner可以多快迭代？

王建硕展示了：用户反馈 → 5分钟 → APP更新。

你的当前迭代速度：badcase产生 → ~1-2周 → SP修改上线

**卡在哪里？**

| 环节 | 当前耗时 | 瓶颈 | 可以加速到 |
|------|---------|------|-----------|
| 发现badcase | 1-3天（手动看日志） | 没有自动告警 | 实时（自动分类+告警） |
| 分析根因 | 1-2天 | 需要人工判断是链路/工具/上下文/模型问题 | 2小时（AI辅助排查） |
| 写SP修改 | 几小时 | 不确定怎么改最好 | 1小时（AI生成3套方案） |
| 回测验证 | 目前不做 | 没有回测机制 | 30分钟（历史case自动跑） |
| 上线部署 | 走发布流程 | 流程固定 | 取决于团队流程 |

**短期（1-2周）可以做到的加速：**

1. **Badcase自动分类脚本**：每天定时扫描日志，按"链路→工具→上下文→模型"四分法自动分类，输出优先级排序
2. **SP修改生成器**：给定badcase描述，AI生成3套SP修改方案供你选择
3. **简单回测**：准备一个20条核心case的测试集，每次改SP前跑一遍

**Prompt模板——Badcase自动分类：**

```
以下是今天的10条Planner badcase日志：

{badcase_logs}

请按以下四分法分类每条badcase：
- 【链路问题】：请求根本没到Planner（走错了层级RAG/仲裁）
- 【工具描述问题】：到了Planner但选错工具/传错参数（SP中工具定义不清晰）
- 【上下文问题】：选对了工具但缺少关键信息（缺记忆/车况/目标）
- 【模型能力问题】：信息完整但推理错误（需要升级模型或加更多示例）

输出格式：
| # | Badcase摘要 | 分类 | 严重度(P0-P3) | 建议修改方向 |
```

### 五、"聊天 vs 编程"——你的两种AI使用模式

王建硕定义的区分：

| | 聊天（消耗型） | 编程（积累型） |
|---|---|---|
| 特征 | 一次性对话，每次从零开始 | 写成.md，可被反复执行 |
| 产出 | 一个答案 | 一条永久规则 |
| 半衰期 | 对话关闭就没了 | 直到主动删除一直生效 |
| 在你的工作中 | "这个badcase怎么解决？"在对话框里问AI | 把解决方案写进SP的注意事项/示例中 |

**你当前的"聊天"行为（可能没意识到）：**
- 每次遇到badcase在脑子里推演"可能是什么原因" → 这是聊天（跟自己聊）
- 写了一个badcase分析但放在飞书文档里没同步到SP → 这是聊天（分析了但没积累）
- 在群里说"这个case我们加一条规则" → 这是聊天（说了但没落地）

**转换成"编程"的做法：**
- 每次badcase分析的结论 → 写进SP changelog + SP本身 → 变成永久规则
- 每次SP讨论的共识 → 当天就改进SP → 不隔夜
- 每次发现的模式 → 写成一条注意事项或一个示例 → 永久生效

**判断你是否在"编程"的标准：** 如果你今天分析了一个badcase，一个月后同类问题出现了，Planner能自动处理得对吗？如果能 → 你做了编程。如果不能 → 你只是聊了个天。

### 六、SP的"目录结构"——像管代码项目一样管理

王建硕展示了他的.md目录：角度.MD / VI.MD / voice.MD / platform/ / generation/。

你的SP也是一个有目录结构的"代码项目"：

```
SP（30K-35K tokens）
├── ① 角色定义（200字）           — 相当于 README.md
├── ② 输入类型（500字）           — 相当于 types.d.ts（类型定义）
├── ③ 工具定义 ×23（8000字）      — 相当于 api/ 目录（23个接口定义）
├── ④ 输出格式（500字）           — 相当于 schema.json（输出规范）
├── ⑤ 注意事项 ×10（1000字）      — 相当于 .eslintrc（lint规则）
├── ⑥ 参考示例 ×69（15000字）     — 相当于 tests/ 目录（69个测试用例）
├── ⑦ 车辆设备知识（3000字）      — 相当于 config/hardware.json
├── ⑧ 工具tips（1500字）          — 相当于 docs/best-practices.md
├── ⑨ 聊天风格（500字）           — 相当于 styles/theme.css
└── ⑩ 动态变量模板                — 相当于 .env（运行时环境变量）
```

**这个类比的实用价值：**

| 代码项目的最佳实践 | 映射到SP管理 | 你可以做的 |
|------------------|------------|-----------|
| 模块解耦 | SP各部分职责清晰，不交叉重复 | 检查：有没有同一个行为在多处重复定义？ |
| 接口文档清晰 | 每个工具描述有完整的输入/输出/何时调用 | 审查23个工具描述是否都够精确 |
| 测试覆盖率 | 69个示例覆盖了多少种场景？ | 列出所有可能场景 → 检查示例覆盖率 |
| 依赖更新 | 下游Agent接口变了，SP要同步 | 每次下游更新时同步检查SP |
| 性能预算 | 总token有上限 | 每个部分的"token预算"是否合理？ |
| 发布流程 | PR → Review → CI → Deploy | SP修改 → 评审 → 回测 → 上线 |

### 七、与前四篇fork的综合——从"认知"到"行动"的完整路径

五篇fork组合成一个完整的升级体系：

| 层级 | 问题 | 答案来源 | 行动 |
|------|------|---------|------|
| **认知层** | PM的核心价值是什么？ | Dan Shipper：判断力 + 养Agent | 你的价值 = SP的质量 × 迭代速度 |
| **定位层** | 我在什么水平？ | PM清洗篇：L0-L3 | 你在L2-L3之间，向L3的"编程"迈进 |
| **框架层** | 怎么定义"做得好"？ | 王建硕：.md=源代码 | 用代码工程标准对待SP |
| **方法层** | 具体怎么做？ | 本教程：4个转变+SOP | changelog + 回测 + review + 重构 |
| **度量层** | 怎么知道做对了？ | 王建硕：Token是GDP / 手写量=控制力 | SP覆盖度 + badcase修复速度 |
| **战略层** | 长期壁垒在哪？ | AGI路线图：场景know-how | 你积累的SP know-how不可被模型替代 |
| **风险层** | 什么会让我变得无关？ | 王建硕：抬轿子vs开车 | 如果你只是"维护SP"，会沦为熟练工；如果你是"用SP编程来解决用户问题"，你是驾驶员 |

**本周行动清单（从最容易开始）：**

| 优先级 | 行动 | 预计时间 | 来源 |
|--------|------|---------|------|
| P0 | 建一个SP changelog文档，记录接下来每次SP修改的原因和效果 | 30分钟 | 本教程：转变1 |
| P0 | 准备一个20条核心badcase的回测集（覆盖最重要的场景） | 2小时 | 本教程：转变2 |
| P1 | 随机抽5条SP规则，验证你能说出每条的来源badcase | 1小时 | 本教程：自检方法 |
| P1 | 用AI跑一次SP内部一致性检查（把SP喂给Claude，让它找矛盾） | 1小时 | 本教程：转变3 |
| P2 | 列出SP中"继承来的"部分 vs "自己写的"部分，标记理解深度 | 2小时 | 王建硕：手写量=控制力 |
| P2 | 尝试用AI生成一次badcase分类（用Prompt模板） | 1小时 | 本教程：5分钟闭环 |

### 八、判断标准

**你已经在"用.md编程"的信号：**
- SP每次修改都有changelog记录（含原因、badcase编号、回测结果）
- 你能在30分钟内回测一个SP修改是否有效（有测试集）
- 你把SP当成"我的代码"，有保护欲——别人想改要经过你review
- 你能说出SP每一部分的"为什么这么写"，没有"遗留代码"的盲区
- 每次SP修改是一次精确的commit，不是"试试看"

**你还在"写文档"的信号：**
- SP改动不记录，下次想不起来上次改了什么
- 没有回测——改完直接上线，等线上反馈
- 不清楚SP某些部分为什么这么写（继承来的，没深究）
- SP中有互相矛盾的规则但没发现
- 分析了badcase但结论没落地到SP中（聊了天但没编程）

**终极判断**：王建硕说"你写得越多.md，控制得越细；你越偷懒，AI给你的就是它的缺省值。" 对你而言——SP写得越精确，Planner表现越好；SP越模糊，Planner就只能靠模型的"缺省理解"来猜你想要什么。
