Planner 模型训练全流程详解

从"为什么需要训练"到"怎么训练" — Function Call 数据范式 + 蒸馏管线 + 迭代闭环

基于：座舱 Planner OnePage 训练方案 + 数据范式构建

一、为什么需要训练？— 光靠 Prompt 搞不定

最初的方案是只用 System Prompt（SP）来指导模型。给模型一份超长的说明书（3万+ tokens），告诉它该怎么做。但即使是最强的模型，纯靠 SP 也有严重问题：

问题严重程度具体表现
不调用工具严重模型直接输出文字"我帮你打开空调"，但实际没有调用任何工具去执行
不遵循流程较严重跳过"先查车况"直接执行，或跳过"先想清楚"直接拆解
上下文捕捉弱较严重长对话后"忘记"挂起的任务、失败的任务
拆解不稳定较严重同一句话，每次拆出的步骤不同
幻觉需解决回复中总结了模型没做过的事
核心结论

SP（提示词工程）能力的天花板很低。只靠 SP 相当于只给员工发了一份操作手册——手册再详细，员工不经过培训也记不住、做不好。训练 = 让模型"内化"这些能力。

二、训练的核心思路 — "抄好学生的作业来教自家学生"

一句话理解

让强模型（Kimi K2）做大量"标准答案" → 人工/GPT-4o 筛选高质量数据 → 用这些数据训练自家小模型（豆包 Seed）。这个过程叫蒸馏（Distillation）。

蒸馏 = 知识迁移

没有蒸馏的情况

小模型直接学SP →

看不懂复杂指令、不会调工具、拆解不稳定

就像让小学生直接读博士论文

有蒸馏的情况

强模型先"做题"生成标准答案 →

小模型跟着标准答案学习 → 学到正确的拆解方式

就像老师先做例题，学生跟着例题练习

每一步的详细拆解

Step 1：蒸馏数据生成

以产品集（真实用户场景）为核心，把各种用户 query 输入给 Kimi K2（强模型），让它输出完整的 Function Call 序列。

例如：输入"我好困呀" → Kimi K2 输出：先查车况 → 开休憩模式 + 关车窗 + 开空调 → 回复用户

然后对数据做泛化——把"我好困呀"扩展为"好累啊""有点犯困""眼皮好沉"等变体，制作训练集和开发集。

Step 2：GPT-4o 质量打分

Step 1 生成的数据不是全都好用的。让 GPT-4o 作为评审员，给每条数据打分：

✅ 满足质量要求 → 加入训练集

❌ 不满足 → 舍弃（工具选择不对、参数错误、流程不合理等）

Step 3：SFT 精调

用筛选后的高质量数据，对 Doubao-Seed-1.6 模型进行 SFT（Supervised Fine-Tuning，监督微调）。

SFT 的本质：告诉模型"在这种输入下，你应该输出这样的 Function Call"，让模型记住正确的行为模式。

Step 4：Kimi K2 评测

微调后的模型对开发集做预测，然后让 Kimi K2 作为裁判给结果打分。

注意：生成数据用 Kimi K2，评测也用 Kimi K2。因为它是当前能力最强的参考标杆。

Step 5：Bad Case 迭代

如果评测不达标 → 分析哪些 case 做得不好 → 针对 bad case 泛化更多类似数据 → 回到 Step 1 重新走一遍。

形成模型迭代闭环：训练 → 评测 → 找不足 → 补数据 → 再训练 → 再评测…

SFT 之后还有 RL 强化学习

SFT 教会模型"正确答案长什么样"，但模型可能只是死记硬背。RL（Reinforcement Learning，强化学习）进一步让模型学会：

- 在没见过的新场景中也能做出合理决策（泛化能力）

- 在多种可行方案中选择最优的那个（决策质量）

- 避免已知的错误模式（从惩罚中学习）

四、Function Call 数据格式 — 训练数据长什么样

核心概念

Function Call 训练 = 教模型输出的不是自然语言文字，而是结构化的"调用哪个函数 + 传什么参数"指令。这样模型的输出可以直接被工程系统解析执行，不需要二次翻译。

4.1 训练数据的完整结构

每条训练数据是一个 JSON 对象，包含两个核心部分：

messages
对话序列：system → user → assistant → tool → assistant → ...
模型要学习的完整对话流程
tools
可用工具列表：[{name, description, parameters}, ...]
告诉模型有哪些工具可以调用

4.2 messages 中的五种角色

role: system
"#火山车载助手Volcano。"
系统指令，定义模型身份
role: user
"我好困呀"
用户输入，模拟真实用户说的话
role: assistant
(纯文字)
"好呀，我来帮你操作一下"
loss_weight: 1
模型的文字回复
loss_weight=1 表示这条要被学习
role: assistant
(tool_calls)
content: ""
tool_calls: [{function: {name: "vehicleAgent", arguments: ...}}]
loss_weight: 1
模型的工具调用输出
这就是 Function Call 的核心
role: tool
"休憩模式已经打开了"
工具执行返回结果
训练时写死，推理时由真实工具返回

4.3 关键字段：loss_weight

loss_weight 是什么？

loss_weight = 1 的行 → 模型需要学习"在这个上下文中应该输出这段内容"，会计入训练损失函数。

没有 loss_weight 的行（比如 role:user 和 role:tool）→ 只作为上下文输入，模型不需要学习如何生成这些内容。

类比：一道数学应用题，"题目"和"已知条件"不需要学生写出来（loss_weight 缺失），但"解题步骤"和"答案"是学生必须学会的（loss_weight=1）。

五、真实训练数据样例 — 两种任务模式

5.1 并行任务："我好困呀"

并行执行 3个工具同时调用 单步复杂任务

这个案例教模型：在查到车况后，一次性输出多个不依赖彼此的工具调用（并行执行）。

对话流程
① system
"#火山车载助手Volcano。"
② user
"我好困呀"
③ assistant loss=1
"好呀，我来帮你操作一下"
④ assistant → tool_call loss=1
调用：get_extra_info(need_env=true)
⑤ tool 返回
"车速:0km/h | 车窗:开 | 空调:关"
⑥ assistant → tool_calls ×3 loss=1
调用1：vehicleAgent("打开休憩模式")
调用2：vehicleAgent("关闭车窗")
调用3：vehicleAgent("打开空调")
⑦ tool 返回 ×3
"休憩模式已打开" / "车窗关了" / "空调打开了"
⑧ assistant loss=1
"休憩模式、车窗、空调都调好了"
模型要学到什么
用户说了模糊的需求
学习点1：先回应用户（不要闷头做事）
学习点2：遇到模糊需求先查车况，而不是直接猜
这是环境信息，模型不需要生成
学习点3：根据车况决定做什么
学习点4：三个操作互不依赖，并行输出
这是工具返回，模型不需要生成
学习点5：工具都执行完后，简短总结告知用户

5.2 串行任务："做北京出行规划，好的话记录成待办"

串行依赖 先做完A才能做B 多步任务

这个案例教模型：当后续步骤依赖前一步的结果时，不要一次性全输出，要等结果回来再决定下一步。

对话流程
① user
"做一个北京的出行规划，有好的结果帮我记录成待办"
② assistant loss=1
"别急，这就帮你做个北京的出行规划"
③ assistant → tool_call loss=1
调用：travel_agent("做一个北京的出行规划")
⚠️ 注意：只调了一个工具！没有同时调 task_agent
④ tool 返回
"北京的行程规划xxxxx"
⑤ assistant loss=1
"搞定啦"
模型要学到什么
用户说了两件事：① 做规划 ② 好的话记成待办
第②件事依赖第①件事的结果
学习点1：先回应，让用户知道在处理
学习点2（关键）：虽然用户说了两件事，但只先调 travel_agent。
因为不知道规划结果好不好，不能直接记待办。
这就是"有串行依赖不一次性输出多个"
规划结果回来了，模型可以判断是否要记待办
学习点3：根据返回结果决定下一步
（源数据此处省略了后续的 task_agent 调用）

5.3 并行 vs 串行 的核心区别

并行：一次输出多个 tool_calls

"我好困呀"

查完车况后 → 同时调3个工具：开休憩+关窗+开空调

因为这3个操作互不依赖，谁先完成都行

tool_calls: [{开休憩}, {关窗}, {开空调}]

串行：每次只输出一个 tool_call

"做规划+好的话记待办"

先调 travel_agent → 等结果 → 看结果好不好 → 才决定是否调 task_agent

因为第2步依赖第1步的结果

第1轮 tool_calls: [{规划}]
第2轮 tool_calls: [{记待办}]

六、Tools 定义 — 模型的"工具箱说明书"

训练数据中的 tools 字段告诉模型有哪些工具可用。每个工具包含三个信息：

name
工具名称，如 "vehicleAgent"
模型在 tool_calls 中引用这个名称
description
工具描述，如 "车控agent"
帮模型判断什么时候该用这个工具
parameters
参数定义：{type, properties, required}
规定模型传什么参数、格式是什么

训练阶段使用的工具集

阶段工具覆盖范围
第一阶段
5个典型Agent		 get_extra_info记忆或环境信息查询
vehicleAgent车辆控制
travel_agent / task_agent / ...出行、定时任务等
第二阶段
全覆盖		 量产版本全部 Agent + 座舱大模型所需全部工具（23个下游工具 + 4个内部工具）

七、两阶段训练目标

第一阶段 — 跑通全流程

目标：

• 确定座舱典型 Agent（5个）+ 典型工具（plan_describe, search_info 等）

• 搭建数据蒸馏管线，从强基模蒸馏数据，保证数据质量

• 外部车企真实线上数据收集

• 走通"数据生成 → 模型训练 → 评测迭代"全流程

交付物：可跑通的 Planner 模型 + 迭代管线

第二阶段 — 量产上线

目标：

• 确定量产版本 Agent 全覆盖 + 座舱大模型所需全部工具

• 训练可量产上线的 Planner 模型

• 模型提优、量化、压缩（适配车载芯片算力限制）

交付物：可部署到车端/云端的生产级 Planner 模型

八、为什么选 Function Call 模式而不是纯文本模式

文档中明确写了"Function call 模式（选用该方案）"。两种模式的对比：

❌ 纯文本模式

模型输出：

"请帮用户打开休憩模式，同时关闭车窗，并且打开空调"

问题：

① 还需要额外的解析器翻译成指令

② 格式不稳定，可能漏参数、错格式

③ 幻觉风险大（可能说出不存在的功能）

④ 不确定调的是哪个工具

✅ Function Call 模式

模型输出：

tool_calls: [

  {name: "vehicleAgent", args: {query: "打开休憩模式"}},

  {name: "vehicleAgent", args: {query: "关闭车窗"}},

  {name: "vehicleAgent", args: {query: "打开空调"}}

]

优势：

① 直接解析 JSON 执行，无需二次翻译

② 工具名、参数都是预定义的，格式严格

③ 可并行调用，语义清晰

④ 可验证、可审计、可回放

十、一页纸总结

问题答案
为什么要训练？纯靠 SP 提示词，模型不调工具、流程跑偏、拆解不稳定
训练数据从哪来？Kimi K2（强模型）蒸馏生成 → GPT-4o 打分筛选
数据格式是什么？Function Call 模式：messages（对话流）+ tools（工具定义）
模型学什么？loss_weight=1 的 assistant 行：什么时候说话、调哪个工具、传什么参数
怎么教并行任务？一条 assistant 里放多个 tool_calls，模型学会同时调多个不依赖的工具
怎么教串行任务？每轮只输出一个 tool_call，等结果回来再决定下一步
训练方法是什么？SFT（学标准答案）+ RL（学举一反三）
怎么评测？Kimi K2 给微调模型的输出打分
不达标怎么办？分析 bad case → 泛化更多数据 → 重新训练 → 闭环迭代
为什么选 FC 模式？输出结构化工具调用，可直接解析执行，比纯文本格式稳定可靠
