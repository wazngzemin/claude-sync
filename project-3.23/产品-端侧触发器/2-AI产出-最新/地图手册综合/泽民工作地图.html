一、触发器——我的核心产品

触发器是什么（用白话说）

触发器 = "如果…就…"引擎。

"如果温度超过25度，就开空调"——这句话里，"温度超过25度"是条件，"开空调"是动作。触发器就是盯着条件、条件满了就执行动作的东西。

它分云端和端侧两半：

  • 云端触发器⑮：大脑。负责注册（你的条件合不合法）、判定（条件满了没）、仲裁（多个条件同时满了听谁的）、分类（这件事在云端做还是端侧做）、下发（把规则发到车上）。
  • 端侧触发器㉔：手。负责接收云端下发的规则、在车上本地执行。价值 = 断网能跑（本地定时器/弱网车控/VLM调度/白名单直执）。

为什么要做触发器

  1. 用户需要"设条件"的能力。"10分钟后关加热""到公司提醒我拿快递""看到副驾睡着调暗灯"——这些都是条件任务，没有触发器就没法做。
  2. Advisor 需要"盯着外界"的能力。Advisor 是动态的——它在跟进一个目标（"照看后排宝宝"），但它不能自己一直轮询信号，它需要一个底座帮它盯着、条件满了通知它。这个底座就是触发器。
  3. 端侧需要断网兜底。进隧道/地库断网了，云端触发器够不着车，端侧触发器能本地倒计时、本地执行白名单动作。
  4. 上汽610 P0 需求。"高频条件任务清单"是上汽 P0——不做触发器，610 交不了。

五、VLM——我调用的感知执行器

这是什么

VLM = Visual Language Model，端侧视觉语言模型，跑在 AI Box 上。它能看摄像头画面 → 输出结构化描述（"副驾闭眼""前方有充电桩""二排有小孩"）。

和我有什么关系

VLM = 触发器的眼睛。触发器想判定视觉条件（"副驾睡没睡"），自己看不了摄像头，需要调VLM来看。
我不管VLM怎么识别，我管"让它看什么、什么时候看、结果怎么用"。

我要做什么

  1. 定义 VLM task 7字段：camera_group（舱内/前向）/ seat（哪个座位）/ query（看什么）/ interval（多久看一次）/ timeout（超时多久放弃）/ priority（P0-P4）/ mutex_group（互斥组）
  2. 和丁彬对齐"VLM能识别什么"：拿他的VLM能力清单，和益红的视觉需求清单交叉 → 能做的配query模板，不能做的标"待模型升级"
  3. 定义 VLM 结果的五态处理（24-9）：running / success / failed / timeout / empty → 每种状态触发器怎么处理（timeout不伪果、failed不触发主动服务、empty是有效"未发现"）
  4. 定义阈值穿越逻辑（24-10）：只有从F→T（"没睡"→"睡着了"）才触发，持续为T不重复触发
⚠ 未定论：617会上有分歧——VLM的attention下发（"让VLM关注什么"）是在触发器里做（我的24-8），还是应该是一个独立模块？佳杰说"我还没太捋明白"，丁彬说"我也第一次看"。这个需要再拉会对齐。
