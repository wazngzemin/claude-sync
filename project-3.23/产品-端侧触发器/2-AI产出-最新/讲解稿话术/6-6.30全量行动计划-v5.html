一、触发器是什么？在整个系统里干什么？

一句话定位：触发器 = 豆包上车（Driver Agent 2.0）的"值班哨兵"。

用户跟豆包说话的时候，有Director（大脑）来处理。但用户不说话的时候呢？谁来盯着车里的情况、发现变化、主动服务？——触发器。

具体干3件事：
① 采集信号：盯着车上所有的传感器和模型输出，把信号标准化写入信号库
② 判断条件：信号变了 → 是不是满足了某个预设的触发条件？
③ 驱动执行：满足了 → 通知Advisor/Director去做事（调温度、播音乐、提醒用户）

类比：如果把整个系统比作一个公司——
· Director = CEO（用户说话时决策）
· 4个静态Advisor = 4个VP（舒适VP、出行VP、情感VP、内容VP）
· 触发器 = 7×24小时值班的监控中心（盯着所有仪表盘，发现异常就通知VP/CEO）
· 动态Advisor = 临时项目经理（CEO安排的临时任务，做完就走）
在三层漏斗架构中的位置

Driver Agent 2.0 有3层处理用户语音的漏斗（句法RAG → 情景RAG → Planner/Director）。但触发器不在这个漏斗里——触发器是漏斗之外的一个独立系统，它有自己的输入（信号）和自己的输出（事件push）。

唯一和漏斗产生交集的地方是：
· 用户说了一句条件任务（"10分钟后关座椅加热"）→ 漏斗处理 → Director解析出条件 → 注册到触发器
· 触发器检测到条件满足 → 通知Director → Director回复用户

（±区间）		 防止信号在阈值附近来回跳（抖动）导致反复触发。

抖动周期：30秒内的波动不算（默认）。

人话：就像你设了一个"体温超过37.5度报警"的闹钟，但体温在37.4和37.6之间来回波动。如果不做防抖，闹钟就会一直响-停-响-停。防抖就是"连续超过30秒还在37.5以上才报警"。

② daily_count：每天最多100次
③ power_count：每个上电周期最多10次
④ execute_interval：两次触发之间最少间隔60秒

举例："有人打哈欠"这个事件，如果每5秒VLM都检测到在打哈欠，不做频控就会5秒通知一次——用户会疯。设了"间隔10分钟"就变成最多10分钟通知一次。 		输入：15-6过滤后的触发

八、四条铁律

铁律 = 绝对不能违反的架构约束。这4条是6.22评审会定下来的，后续所有设计都必须遵守。违反任何一条都会导致系统混乱（信号不一致、事件丢失、模块耦合）。
铁律一：动态注册必须经⑮云端触发器中转

所有动态注册（Director注册条件任务、Advisor注册感知需求）都必须走：发起方 → ⑮ → ㉔
不允许：Director直接给㉔注册（绕过⑮）、Advisor直接给VLM下SP（绕过⑮）

为什么：⑮是唯一的"规则管理者"。如果允许绕过，⑮不知道有哪些规则在跑，就没法做优先级仲裁、频控、冲突检测。
铁律二：端侧结果不直推Advisor，必须走信号库

端侧（VLM/定时器等）的结果必须走：㉔ → ④信号库 → ⑫云端信号库 → ㉑Context
不允许：㉔直接push给Advisor、㉔直接推给Director

为什么：①信号库是唯一的"事实源头"（Single Source of Truth），如果允许绕过就会出现"信号库说没有小狗，但Advisor说有小狗"的不一致。②断网时端侧结果先缓存在本地队列，联网后再同步——如果直推就没法做断网缓存。
铁律三：触发器只读信号库，不直接监听原始信号源

⑮读的是⑫云端信号库的标准化快照，不是直接监听③端信号源。
不允许：⑮直接读CAN总线、⑮直接调VLM

为什么：解耦。信号源的格式/协议/频率千差万别（CAN总线是二进制、VLM是JSON、App是埋点），如果每个都去适配，触发器就变成了"万能适配器"。信号库做标准化，触发器只读标准化后的快照。
铁律四（6.22最核心）：condition→Context / event→⑮push

状态/条件（"车里有小朋友"）→ 写入Context → Advisor推理时自动拼接进去
事件（"小朋友上车了"）→ ⑮组装push → 主动推给Advisor

不允许：Advisor轮询Context去发现事件（太慢、太浪费）
不允许：⑮把事件写进Context让Advisor自己去读（事件是瞬时的，放在状态池里不合理）

施佳杰原话："事件应该是更有价值的信息，不等同于状态。"Advisor不能24小时轮询Context等变化，所以需要触发器在事件发生的那一瞬间主动push——这就是触发器存在的核心价值。

九、CTA框架

CTA = Condition-Trigger-Action，是触发器的核心工作框架。用大白话说就是：

采集信号（Condition） → 判断是否满足条件（Trigger） → 满足了就执行动作（Action）

整个触发器系统的一切设计都是围绕这3步展开的。
