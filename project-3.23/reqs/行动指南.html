3今天 / 本周：立刻做的 5 件事

原则：先抢"原料"（缺了开不了工的东西）→ 再做"产出" → 最后才谈"排期"。

抢原料：要信号库 + 开后台权限

找 韩天亮（信号库） · 文涛/芭比（权限）· 今天上午

找韩天亮要《信号库 v2.5》最新版，并问清楚锁不锁版本号；同时找文涛/芭比发起后台配置器登录权限申请。

为什么先做：需求1/3 都靠信号库当字典，它一直在变，不钉死版本就做到一半全废；权限审批有时滞，今天不发起到时卡你。

起需求1 草表（你的核心产出）

你 · 今天下午

新建 Excel/飞书表，一行一条件，6 列固定：条件名 / 取值判断方式 / 信号编号 / DNA能否支持(✓✗?) / 归属任务类型 / 备注。

取值方式只有三种，背下来：布尔开关数值阈值变更为/从无变有

范围 20~30 条，对标"理想任务大师"，精选高频、不贪全——初级 PM 最容易犯"想做全"的错。

并行做需求3：益红事件表评估

你 → 赵益红 · 本周内

对表里每个标 ? 的事件给准信：能做 / 回掉 / 视觉排除，回填进益红的表，每条附理由。

判定规则极简：信号库查得到端信号=能做；查不到=回掉；强依赖视觉=直接划掉（DNA 端侧视觉只有 Face ID）。

面聊吕壮：收机器人料（需求4）

吕壮（壮哥）· 漕溪北路 · 明天

只问两件事：① 机器人会产生哪些事件；② 期望触发器怎么接（端/云、什么格式）。拿到事件表后归档、打标"赛力斯阶段"，不进 610、不排期。

这是半小时的活：约时间 + 列两个问题。别展开。

清单初稿出来 → 找廖婉婷算工期（需求2）

你 → 廖婉婷（意图）

把定稿清单 + 协议范围说明发意图团队，核心动作：问"模型调试要几天"，然后倒推——她要 X 天，你就必须在 6/15 − X 之前交清单。

5跟研发沟通：怎么说才专业、不露怯

心法一句话：初级 PM 不靠"懂实现"显专业，靠"把边界和接口说清楚"显专业。你不需要会写代码，你需要会问对问题。

① 四条心法

1 · 永远用"输入→输出→工期"三段式，别说"你去做个 X"。
✗ "刘洋你把触发器改一下" ｜ ✓ "我产出这版清单(输入)，希望'超25度开空调'能被建成条件任务(输出)，你大概要几天(工期)？"
2 · 用系统黑话，研发听到就默认你懂： 端侧/云端状态变更 vs 当前态协议拍平信号库触发时机/瞬间
3 · 不懂实现就大方说"实现你定，我对齐边界" —— 这不丢人，反而专业。"具体怎么实现你们定，我只确认：这个判断放端侧还是云端？"
4 · 每次对话结尾收三件事：① 前置齐没 ② 工期几天 ③ 验收标准是啥。收齐再走。

② 分人话术模板（直接抄着说）

▸ 韩天亮 要信号库
"天亮，我在做触发器高频条件清单，需要《信号库 v2.5》最新版当字典。能给我一版、并帮我确认这版本号近期会不会再变吗？我想锁定一个版本开工，免得做到一半底层信号变了。"
专业点：主动提"锁版本"，研发会觉得你懂工程节奏。
▸ 廖婉婷 意图 · 需求2
"婉婷，这是定稿条件清单和协议范围说明。我希望最终效果是：用户说'超25度开空调'，模型解析成『条件:温度>25，动作:开空调』。想确认两件：① 这清单作约束够不够；② 模型调试要几天？我拿工期倒推交付时间，赶 6/15。"
▸ 刘洋 端侧研发 · 需求5/6/7/8/9
(需求9 端云分工) "赛力斯这版想做端云分工：云端管注册+冲突仲裁，端侧管判断触发+直接执行，冲突在云上消解完丢回端、端侧'没脑子只执行'。先对一下——目前端侧 SDK 能不能直连车控/TTS/播报？这块是定时器、距离事件的地基。"
(需求7 定时器) "'10分钟后关座椅加热'倒计时必须做在端上，云上跑会被隧道弱网搞挂。把'时间'当端状态一个维度、定时器挂端侧，可行吗？工作量多大？"
(需求6 协议拍平) "车门套娃结构、position 当参数，'任意门开且X'配不出。赛力斯想把每个门拆成独立一级条件、position 和端状态拍平、加'任意位置'表达式。上汽这版先用模型把'任意位置'解析成具体几个门顶过去。帮我评估真改造工作量、要不要供应商配合？"
▸ 赵益红 主动服务 · 需求3
"益红，你那张事件表标 ? 的我按信号库 v2.5 逐条评了：有端信号标'能做'，没有的'回掉'，强依赖视觉直接划掉（DNA 端侧视觉只有 Face ID）。结论回填进你表里、每条附了理由，你看着定这版上哪些事件、配成预设事件。"

▸ 文涛 / 芭比 · 吕壮 权限 · 收料
"想申请后台配置器登录权限，触发器条件清单要录进去上线，走什么流程、需要我提供什么？"
"壮哥，今天就想了解两件事：① 机器人会产生哪些事件；② 期望触发器怎么接——端侧还是云端、什么格式。我先收料归档，这块归赛力斯阶段，不占你 610 节奏。"

③ 三个万能句式 + 千万别说

✓ 任何研发对话都能用：

对齐边界："这块我只确认放端侧还是云端，具体实现你定。"
要工期："你大概要几天？我要倒推我的交付时间。"
确认前置："开工前你还缺我什么？我先给你备齐。"

✗ 千万别说（瞬间露怯）：

  • "这个很简单吧，应该很快" —— 研发最烦 PM 估工期
  • "你随便弄一下就行" —— 显得你不知道自己要什么
  • 当场对实现方案拍板 —— 越界，把实现还给研发
  • "我也不知道要什么，你看着办" —— 宁可说"我回去把边界写清楚再来对"

动态更新 需求4 · 吕壮面聊后的真实结论 已面聊 · 2026-06-01
⚠️ 原假设要修正：机器人不是"海量新触发事件的源头"，而主要是 表情形象的展示消费方。真正与触发器相关的，只是机器人《触发事件清单》里 「Planner 触发」的一小部分。先别按"海量事件"给赛力斯排资源，等表来看真实量级。
① 机器人的接口 = 三元素
新模式下上游不再说"显示哪个表情"，而是给：形象标签内容标签(晴天)TTS起止时间机器人自己跑逻辑展示。注意这三元素目前在"意图"侧产出、不是 Director → 边界待对齐。
② 边界：你只要"Planner 触发"那部分
机器人那张表已标好 Planner触发 / 信号触发，吕壮会把表发你。⚠️ 待复核：触发器本是信号驱动，他却说"信号触发的与你无关"——别现在下结论，等表到手看字段再定。
③ 暴露真痛点：TTS↔表情不同步
现在串行——TTS 都播完了天气表情还没出，用户没看到。吕壮反问你"能不能解"。这是 Director/Planner 执行链路问题，非触发器层。你欠他一个回话：回去找 Director/意图确认"串行→并行"可行性与排期。
④ 顺带澄清：Director = 班纳改名
Director = 原 Planner+Actor 拆分后只做规划、不做执行那部分；旧名"班纳/Banner"，只是改名、原理不变。AIOT SDK = 机器人车机端软件，存所有表情形象、打包在 APP。
