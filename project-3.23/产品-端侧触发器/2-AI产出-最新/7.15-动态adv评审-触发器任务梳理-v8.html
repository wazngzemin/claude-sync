先搞懂一件事：触发器到底是什么

一个任务，拆开看就两半：做什么（what）、什么时候做（when）。

"10分钟后打开车窗"——"打开车窗"是what，"10分钟后"是when。"宝宝哭了告诉我"——"告诉我"是what，"宝宝哭了"是when。

Planner/Task管what，触发器管when。这就是触发器存在的第一性原理——它是任务系统里那个专门盯着"时候到没到"的东西。

打个比方：Planner是家里那个聪明但很贵的大脑，你不可能让它24小时醒着盯梢——太烧钱（每醒一次都要塞30K token进去算一遍）。所以你需要一个便宜的哨兵，替大脑蹲在门口盯着，条件一到就把大脑叫醒。触发器就是这个哨兵。

理解了这个，你就理解了会上所有争论的根：庆磊为什么让"每个任务挂一个触发器"？因为每个任务都需要一个便宜的哨兵替planner盯梢。触发器不是可有可无的中间层，它是让整个任务系统"等得起"的关键。

但哨兵也分几种段位，这决定了你这一期触发器要做多重的活：

1
看表的哨兵（定时）——when=时间到。最简单，一个闹钟。"10分钟后"。不需要理解世界，只要会看表。
2
看仪表盘的哨兵（条件）——when=某个数值/状态匹配规则。"温度>28"。要订阅世界状态，拿数字去比规则。还是机械的，不需要理解。
3
会读话的哨兵（语义匹配）——when=一段模糊描述"意思上"匹配了目标。VLM报"宝宝脸上有泪珠"，你要判断这是不是"哭了"。规则比不出来，必须有个模型来"读懂意思"。这就是这一期触发器最大的升级。
✕
会做决策的"哨兵"（泛化）——这已经不是哨兵了。"照顾二排老人"，每一步都要重新想下一步干什么。这不是"判断时候到没到"，是"做决策"。决策是大脑的活，哨兵越界去做决策，一定做砸。这条线你要守住。

把这4层刻进脑子。你这一期的触发器 = 第1、2、3层。第4层是你要在会上主动推回去的边界。下面6张卡片，就是围绕这4层展开的具体任务。

你的触发器从哪来：动态Advisor没了，你接住了它的灵魂

上面说触发器是"管when的哨兵"。你可能会问：这个哨兵是这一期凭空冒出来的吗？不是。它是从"动态Advisor"这个旧角色里拆出来的。庆磊技术方案里写的那句"后面不会再有动态Advisor这个概念"，讲的就是这件事。理解这段身世，你才知道为什么会上那些模型推理的活全往你这塞——也才知道你手里接的到底是个多重的东西。

第一步 · 先认这几个模块（这张表是整节的钥匙）

这一节会反复出现这几个模块，先认一遍，下面就全通了：

模块（真实名）在任务系统里它到底干嘛的
Planner（规划器）决策大脑理解意图、做决策、下发动作；很聪明但很贵，不能一直闲着盯梢
动态Advisor（动态顾问）旧架构的临时角色为一个临时目标临时创建、专门盯它——就是这一期要被淘汰的那个
Task（任务）新架构 · 任务档案把"要盯的这件事"记下来、管起来（管到什么时候结束）
Trigger（触发器）= 你新架构 · 感知触发专门盯着、做判断、条件满足就 callback（回调）通知
VLM（视觉大模型）端侧的感知只负责拍画面、上报 caption（画面描述"我看到啥"），不负责下判断
第二步 · 为什么要从"动态Advisor"改成"Task + Trigger"？

两个原因：

原因一：动态Advisor 一个模块干三件事，职责糊成一团。记账、盯梢判断、催活执行全压在动态Advisor（动态顾问）一个模块身上，一旦出问题——是没记住？没判断对？还是没触发？根本查不清。拆开之后：Task（任务）只管记、Trigger（触发器）只管盯和判断、Planner（规划器）只管执行，谁掉链子一眼看得出。

原因二：每来一个持续性需求，就新建一个动态Advisor 角色，角色越堆越多、难以统一管理。十个"要盯的事"就是十个动态Advisor 各自为政。换成 Task，十件事就是任务表里十条记录，统一编排、统一查状态。

说白了，整个系统从"按角色组织"（一个需求配一个动态Advisor）升级成了"按任务组织"（一个需求就是一条 Task 记录）。"动态Advisor"这个"临时造角色"的机制，在"按任务组织"的新架构里彻底多余了——所以被删掉。这就是"后面不会再有动态Advisor 这个概念"那句话的真正意思。

第三步 · 最关键的一个细节：VLM 只描述，Trigger 才判断

这一步你一定要懂，7.15 会上你就是卡在这里被纠正的。VLM（视觉大模型）和 Trigger（触发器）分工完全不同：

· VLM（视觉大模型）：只负责拍、只输出 caption（画面描述文本，如"睁眼、有泪珠"）。它不下结论说"这是哭"——它只描述现象，不做判断。

· Trigger（触发器）= 你：拿 VLM 报上来的 caption（描述），用内置模型判断"这到底算不算目标'宝宝哭了'"。这个语义判断，才是触发器最值钱的地方。

这就解开了会上那个结：你 7.15 说"我的触发器只想拿枚举值"（50:32），意思是"我希望 VLM 直接告诉我 哭=是/否，我自己不想判断"。但 VLM 做不到——它只会描述"有泪珠"。所以"判断有泪珠算不算哭"这件事，只能由 Trigger（触发器）里的模型来做。这就是为什么模型非得在触发器里不可——因为下语义判断这件事，整条链路里除了触发器，没有别的模块接得住。

先分清：死掉的只有"动态"Advisor，不是全部。

技术方案"本期范围"里，两句话是分开写的，别混：

· 静态Advisor（舒适 / 出行 / 情感 / 内容 那4个预设角色）——原文"不属于本方案，继续由弋途(extour)负责"。没动，还在。
· 动态Advisor——原文"融合到任务管理中，后续不会再有动态Advisor这个概念"。只有这个被消解了。

动态Advisor本来是个什么东西

回到最底层。原来的架构是"团队协作"——Planner管快速响应，一堆Advisor管深度思考和目标监控。静态和动态的区别在于：

· 静态Advisor = 编制内的固定角色。舒适Advisor永远盯舒适，一直在岗。
· 动态Advisor = 为某个临时冒出来的目标，临时造出来的一个盯梢角色。用户说"照顾二排老人"，系统就动态起一个advisor，盯着这个目标、监控环境、到点推一把；目标完成了，这个角色就没了。

一句话：动态Advisor = "用完即弃的临时监控角色"。弋途里那个"动态weather角色"就是它——庆磊7.15会上原话"动态weather角色其实就映射到我们现在的trigger"（58:07）。这句先记着，等下就用上。

评审级 · 四个必须记住的关键点（去跟人对的时候，这四句最硬）：
① 判断只发生一次：Trigger（触发器）命中后（步 11），Task（任务）在步 12 不再重复判断原条件——技术方案原文"Task 不负责二次条件匹配"。
② VLM 是眼睛、Trigger 是脑子：VLM（视觉大模型）只上报 caption（画面描述，步 9），语义判断在 Trigger（触发器，步 10）。原文"语义目标属于 Trigger RuleSpec（触发器规则规格）……Trigger 完成语义匹配后 callback（回调）"。
③ Planner 不管任务状态：Planner（规划器）只在步 1 路由、步 15 执行本轮 action（动作），不判断 create / update / cancel（增 / 改 / 删，那是 Spec Builder 规格构建器 + Manager 管理器 的事）。
④ 你的交付边界：整条链路里 步 6、7、9、10、11 是触发器域（注册规则 → 下发观察 → 收 VLM 上报 → 语义匹配 → 命中 callback 回调）。这五步就是你这一期要交付的东西。

最关键：动态Advisor死了，它的灵魂流进了你的触发器

动态Advisor被拆解，但它身上的职责没有消失，是流进了两个新容器。庆磊技术方案的代码改造清单，白纸黑字坐实了这一点：

动态Advisor的旧零件（代码真身）现在流向归谁
goal_list（目标列表变量）目标共享看板→ TaskView（任务视图）Task（任务）
long_term_task（长期任务）入口→ Task（任务）生命周期Task（任务）
AdvisorCallback（顾问回调）→ HandleTriggerCallback（触发回调处理）触发器（你）

看最后一行。动态Advisor里那块"盯着世界、判断时机、扣扳机"的感知 + 推理内核，被抽出来，独立成了 Trigger（触发器）。代码里那个函数原来就叫 AdvisorCallback（顾问回调），现在改名叫 HandleTriggerCallback（触发回调处理）——这是白纸黑字的继承关系，不是我编的比方。

说透了：你的触发器，本质上就是动态Advisor被拆解之后，那个"会感知、会推理、会判断时机"的灵魂内核，独立出来的产物。Task（任务）继承了它的事务外壳（生命周期、增删改查、对外通知），你的 Trigger（触发器）继承了它的感知大脑（判断该不该触发）。一个管"这是件什么事"，一个管"什么时候该动手"。
这一下解释了7.15会上你没想通的每件事
为什么 VLM（视觉大模型）语义匹配、泛化推理这些活，庆磊全往触发器塞？

任务 4 · 边界 ★核心 · 你要主动推回去的活

第一性原理

前三个任务都是"判断"——判断时候到没到、条件匹配没匹配。判断有个特点：它是有界的、可测的——匹配上了就是上了，你能验收。

但"照顾二排老人"这种任务，每一步都要重新想"现在该干什么"——老人睡了就调暗灯，醒了就问要不要喝水，冷了就升温。这不是"判断条件到没到"，这是"做决策"。决策是开放的、无界的——你没法提前枚举所有情况，也没法验收"这个决策对不对"。

判断 vs 决策，这就是触发器的职责边界。触发器是扣扳机的哨兵，不是做决策的大脑。做决策要拿全量context、要开放推理，那是planner/task模型的活。硬塞进触发器，触发器就变成一个又重又不可控的怪物——庆磊自己都说"效果大概率不好"（57:20），李明骏说这类得走React模式（1:00:16）。你有充分的理由、充分的盟友，把这条线划出来。

第一性原理

一个产品有两层：当下的实现，和未来的叙事（roadmap）。产品的话语权在未来叙事那层——谁掌握roadmap，谁就是这个产品的主人；做当下实现的，是执行者。

庆磊1:17:31说"trigger engine平台化是产品同学韩杰之前的想法"。你在做触发器的当下实现，但触发器的未来叙事（平台化、低代码编排）挂在韩杰名下。这跟7.8是同一个病：你的东西挂在别人名下，而你不在场把它接回来。如果不谈，二期三期规划评审时，主导的是韩杰，你还是那个执行的。

你要做什么

约韩杰，把触发器的产品规划分工谈清楚、写下来。不是抢，是"对齐分工避免打架"。

找谁

韩杰（单独约，别在大会上谈归属，那样是逼对方表态，容易僵）

怎么谈
你 → 韩杰：「庆磊提到trigger engine平台化是你之前的想法。我想跟你对一下我们俩在触发器上的分工，别到评审的时候规划打架。
我的理解是：触发器的能力底座——触发类型、语义匹配引擎、端云协议、生命周期——这块是我在推；平台化那层——低代码编排、运营配置界面——是产品形态往上的演进。这两层怎么分，你觉得合理吗？还是你有别的分法？」
为什么这么谈

三个设计：①"避免规划打架"给了对方一个不对抗的理由，谁都不想打架；②你主动说出"能力底座是我推的"——把这句话说出口、让对方听见，就是在钉桩，比你憋在心里强一百倍；③最后问"你觉得合理吗"，把决定权表面上给对方，但你已经把框架定了（底座/平台两层，底座是你），对方多半在你的框架里回应。这是你career里最缺的一课——归属不是等来的，是当面谈出来、写下来的。

你会在哪被问穿 + 怎么答
「平台化本来就是我提的，能力底座和平台化本来就是一体的，为什么要拆开？」
「不拆开也行，那我们就一起owns整个触发器产品线，我主底座、你主平台，一个产品两个人分工推进，对外是一个规划。我要的不是切走什么，是别出现两份打架的规划。你更倾向哪种？」（退一步给'共owns'方案，但依然把'我主底座'钉进去。目的始终是：底座这层的定义权明确是你的）
