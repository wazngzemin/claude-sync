## 一、需求背景与目标

### 1.0 为什么现在要做？

当前端侧 VLM 任务触发采用短期方案：在 AI BOX SDK 中硬编码订阅状态，满足条件直接调用识别任务。存在三个瓶颈：

1. **不可扩展**：每增加任务或适配新车型，都需改代码发版；
2. **无统一信号源**：每个任务各自读 CAN 信号，VLM 结果不能复用，云端 Adviser 拿不到标准化端侧状态；
3. **无调度能力**：VLM 任务已增至 15+，缺乏优先级仲裁、打断恢复和超时释放。

现需升级至长期方案，建设端侧信号库 + 端侧触发器。

### 1.0.1 本期目标

- 已有 15 个 VLM 任务全部转为规则驱动（脱离硬编码）
- 支撑任务优先级仲裁、抢占恢复、超时降级
- 支持 Adviser/Planner 动态注册条件任务
- VLM 结果统一写入信号库，供云端 context 订阅

### 1.0.2 模块职责边界

| 模块 | 职责 | 不负责 |
|---|---|---|
| 端侧信号库 | 统一存储所有状态（端状态/OEM/场景/VLM结果/任务状态）；对外提供写入接口和订阅接口；按策略上报云端 context | 不负责信号的原始采集 |
| 端侧触发器 | 加载规则 → 读取信号 → 两级条件匹配 → 任务调度 → 下发运行指令；接收云端触发器转发的动态任务 | 不负责 VLM 内部推理；不负责 Adviser 思考逻辑；**不直接与 Adviser/Planner 通信** |

### 1.1 端侧触发器模块是什么？

端侧触发器是 AI 汽车端侧感知任务的调度中枢，负责把"端侧信号库里的实时状态"进行匹配，在条件满足时触发对应的 VLM任务，并管理任务的优先级、执行状态、超时、打断与恢复。

是端侧任务执行前的"规则判断 + 任务调度 + 生命周期管理"模块。

> 信号库负责存"车知道了什么"，触发器负责判断"现在该跑哪个任务"，VLM/小模型负责执行"具体看什么"，结果再回到信号库并上报云端 context。

目前要同时建设两个基础能力：

1.  **端侧信号库**：统一存储车辆端状态、外设/IoT 设备信号、视觉信号、应用状态信号、用户 Query、VLM 模型结果和任务状态；
2.  **端侧触发器**：基于规则和信号判断任务是否触发，并负责任务排队、优先级、打断、超时和结果回写。

### 1.2 目前问题点？

#### 端侧任务触发是写死逻辑，无法进行扩展

当前状态是车上的端侧识别逻辑是固定写死：在 AI BOX SDK 中订阅几了个状态，满足条件后直接调用识别任务，识别完把结果给云端。但是当不同车型规则不一致时，无法统一管理。

#### 端侧没有统一信号库，任务判断缺少稳定输入

-   **触发器没有稳定输入**：每个任务自己去读 CAN 信号，读的方式都不一样
-   **VLM 结果无法复用**：下车安全跑完的结果，停车位识别任务想用，拿不到
-   **云端 Advisor 订阅不到标准化端侧状态**：Advisor 想知道"刚才 VLM 检测到什么"，没有统一接口
-   **任务失败没有记录**：VLM 超时了、报错了，没有任何地方记录，后面排查问题找不到日志

#### VLM 任务数量已经明显增加，需要管理任务

没有统一调度会出现：

| 问题 | 场景 |
| --- | --- |
| Always-on 占用算力，用户 VQA 响应变慢 | P3 阻塞 P0 |
| 安全任务被低优任务挡住 | 下车安全 P0 被 Always-on P3 占位 |
| 低优任务覆盖高优任务结果 | Always-on 旧 caption 覆盖 VQA 结果 |
| 任务超时后队列不释放 | VLM 推理卡住，后续任务全部阻塞 |

## 二、整体架构

### 2.1 架构链路图

![](images/NuCfuS2T3VU1W2B1IpxgEH4F648rWCwL_v2.png)

#### 2.1.1 信号来源层（架构图最上方）

六类信号统一写入端侧信号库：

| 信号分类 | 包含内容 | 写入说明 |
|---|---|---|
| 车辆端状态 | 车速、档位、车门锁状态、座椅占位、安全带、电源状态等 | 由车辆底层服务持续写入，是大多数任务运行时段判断的基础信号 |
| 外设/IoT 设备信号 | 蓝牙钥匙距离、远程备车信号等 | 由手机或车外 IoT 设备通过 OEM 适配层写入，属于非车辆本体的外部设备状态 |
| 视觉信号 | 舱外 FaceID 识别结果、基础哨兵检测结果、地下停车场视觉判断等 | 由端侧视觉模块或 OEM 哨兵模块写入，属于需要通过摄像头/模型才能获取的状态 |
| 应用状态信号 | 导航结束、进入地下停车场（定位判断）、增强哨兵开关状态等 | 由端侧 APP/导航服务等应用写入，属于应用层产生的事件或状态变更 |
| 用户 Query | VQA 问题、泊车指令等 | 由语音模块或云端下发写入，属于用户主动发起的即时任务 |
| 模型结果信号 | VLM 识别标签、描述、任务状态等 | 由 VLM/小模型执行后写入，供下游消费和二次触发 |

> **设计说明**：信号来源层只负责"把信号写进来"，不关心触发器会怎么用这些信号。新增信号来源时只需要适配写入接口，不需要修改触发器逻辑。
>
> **分类说明（2026-05-09 需求对齐确认）**：原四分类中"OEM/用户信号"混合了外设设备信号和视觉信号，"导航/场景信号"混合了应用状态和视觉判断。现拆分为六类，使每类信号的来源和性质更明确：车辆端状态来自 CAN 总线，外设信号来自蓝牙/手机，视觉信号来自摄像头和模型，应用状态来自端侧 APP，用户 Query 来自语音/云端，模型结果来自 VLM 执行后的回写。

#### 2.1.2 端侧核心链路（架构图中间区域）

端侧由三个模块组成：信号库、触发器、VLM 模型。数据在三者之间形成闭环：

**正向链路**（触发→执行）：

- **信号库 → 触发器**：触发器读取/订阅信号库中的信号快照，用于规则条件匹配。触发器不直接监听原始信号源，所有信号必须经过信号库。
- **触发器 → VLM**：条件命中后，触发器向 VLM 下发运行指令。运行指令包含四个核心信息：任务标识、识别目标描述、优先级、超时时间。触发器不关心 VLM 内部如何推理，只管下发和接收结果。

**反向链路**（执行→回传）——VLM 执行完之后有两条独立的数据回路：

| 回路 | 方向 | 传递内容 | 消费方 | 设计目的 |
|---|---|---|---|---|
| 任务状态反馈 | VLM → 触发器 | 排队中/执行中/超时/被打断 | 触发器 | 触发器据此做抢占恢复、超时释放、队列管理等调度决策。这条回路的实时性要求高，触发器需要尽快知道任务状态变化 |
| 识别结果写入 | VLM → 信号库 | 标签/描述/空结果/失败 | 信号库（供下游消费和云端上报） | 结果存入信号库后，可被其他任务读取复用（如下车安全结果被停车寻车任务参考），同时按策略上报云端 |

> **为什么分两条回路**：任务状态是给触发器做调度的，实时性要求高；识别结果是给整个系统消费的，需要持久存储和标准化上报。如果合成一条路，存储会拖慢调度，或者触发器不得不承担结果存储职责，导致模块职责不清。

#### 2.1.3 端侧到云端的上报链路

- **信号库 → Context 模块**：信号库按策略将端侧结果上报至云端 Context 模块。上报内容包括任务结果、车辆状态快照等。上报是信号库的职责，触发器不参与上报。
- **Context → Adviser/Planner**：Adviser 和 Planner 从 Context 模块订阅消费端侧结果。这是 Adviser/Planner 获取端侧信息的**唯一通道**。

#### 2.1.4 云端到端侧的下发链路

| 链路 | 路径 | 说明 |
|---|---|---|
| 静态规则下发 | 云端规则配置 → 端侧触发器 | 产品预设的任务规则，全量/增量下发。触发器启动时拉取，运行中支持增量更新 |
| 动态任务下发 | Adviser/Planner → 云端触发器 → 端侧触发器 | 运行时注册的条件任务。Adviser/Planner 不直接与端侧触发器通信，必须经过云端触发器转发 |

> **核心边界（2026-05-07 研讨会确认）**：端侧触发器与 Adviser/Planner 之间**没有直接通信**。触发器只负责接收云端触发器转发的规则和动态任务（单向接收），不反向推送任何信息。Adviser/Planner 获取端侧信息的唯一通道是云端 Context 模块。这一设计保证了触发器不需要感知云端业务逻辑，只做端侧的规则匹配和任务调度。

### 2.2 主流程图

![](images/My9rirRbBpUgUdgr7U5Zy2cW1rM11BE0_v2.png)

#### 2.2.1 阶段一：规则加载（步骤 1-2）

| 步骤 | 节点 | 说明 |
|---|---|---|
| 1 | 任务规则下发 | 来源有两类：云端配置下发的静态规则，用户实时输入的即时任务（如 VQA） |
| 2 | 确认匹配规则 | 从规则库中找到当前可能适用的规则集。此步骤是规则的预筛选，不是最终匹配 |

> **说明**：规则在触发器启动时全量加载到本地，运行中支持增量更新。规则的加载和更新不阻塞正在进行的条件扫描。

#### 2.2.2 阶段二：两级条件匹配（步骤 3-4，流程图核心逻辑）

**这是触发器的核心调度逻辑，分两级判断，不是一级。**

| 步骤 | 节点 | 说明 |
|---|---|---|
| 3 | 触发器获取信号快照 | 从信号库读取当前时刻所有信号的状态切面（不是实时流），作为本轮匹配的输入 |
| — | 判断运行时段 | 根据信号快照判断当前处于哪个运行时段：离车后 / 上车前 / 停车时 / 行车中 |
| — | ◇ 运行时段满足？ | 否 → 回到信号监听，等待时段变化后重新获取快照 |
| 4 | 判断触发条件 | 时段满足后，检查具体的触发事件：下电锁车 / D→P 档位切换 / 用户 Query 等 |
| — | ◇ 触发条件满足？ | 否 → 回到步骤 3，继续监听信号变化 |

> **为什么分两级**：端侧任务目前 15 个、未来更多。如果每次信号变化都扫全部规则的全部条件，性能开销大且逻辑混乱。分两级后：第一级用运行时段做粗筛，把当前时段不可能触发的规则整组排除（比如行车中时，离车后的 5 个任务规则完全不扫描）；第二级只在匹配的时段内做精准事件匹配。既降低了扫描开销，也让规则的逻辑结构更清晰。
>
> **举例**：「下车安全」任务的第一级条件是"停车时"（电源上电 + P 档 + 解锁），第二级条件是"档位从 D/R/N 切到 P"。如果当前车在 D 档行驶，第一级就不通过，无需检查第二级。

#### 2.2.3 阶段三：任务生成与调度（步骤 5-9）

| 步骤 | 节点 | 说明 |
|---|---|---|
| 5 | 生成候选任务 | 两级条件均命中，创建一个待执行的候选任务。此时任务还未执行，需经过后续检查和仲裁 |
| — | 检查任务状态 | 检查该任务当前是否已在运行、已完成、或处于冷却期中。避免重复执行 |
| — | ◇ 允许执行？ | 否 → 已在运行则跳过本次触发；冷却期中则等待冷却结束；已完成的单次任务不再触发 |
| 7 | 加入任务队列 | 候选任务通过检查后进入调度队列 |
| 8 | 优先级标记 | 触发器按优先级规则（P0 > P1 > P2 > P3 > P4）为任务标记优先级。触发器只负责标记优先级，标记完成后将任务连同优先级信息一并下发给 VLM 业务侧。实际的任务并发调度、资源分配由 VLM 业务侧管理 |
| 9 | 生成运行指令 | 标记完成后，触发器组装运行指令，包含四个核心信息：任务标识、识别目标描述、优先级、超时时间 |

> **说明（2026-05-09 需求对齐确认）**：触发器的职责是"标记优先级 + 下发指令"，不是"排队等候串行处理"。触发器标记完优先级后即下发给 VLM 业务侧，由 VLM 业务侧根据当前并发能力和资源状况决定如何调度执行。这意味着如果同时有两个 P0 任务，触发器会同时下发两个 P0 指令，由 VLM 侧决定是否并行处理。
>
> **待研发讨论**：触发器的优先级标记与 VLM 业务侧的任务仲裁之间的职责划分，需在需求评审时与研发确认（触发器标记优先级，VLM 侧做实际调度，避免两边重复仲裁）。

#### 2.2.4 阶段四：模型执行与结果处理（步骤 10-11，需重点关注结果分流）

| 步骤 | 节点 | 说明 |
|---|---|---|
| 10 | 调用 VLM 模型 | 触发器将运行指令下发给端侧 VLM 或小模型 |
| 11 | 结果返回 | 模型执行完成后，结果分四条路处理（见下方详细说明） |

**结果分流——四种情况的处理规则**：

| 结果类型 | 写入内容 | 是否可作为有效结果 | 处理规则 |
|---|---|---|---|
| 成功 | 写入识别标签和/或自然语言描述 | 是 | 正常写入信号库，供下游消费和云端上报 |
| 空结果 | 写入"空" | 是（表示"看过了，确认无目标"） | 空≠没看。空是一个有效的识别结论，对下游有意义（如下车安全空结果代表安全） |
| 失败/超时 | 写入失败或超时状态 | 否 | **产品红线：不允许写伪结果。** 超时了不能写"无障碍"，因为模型没看完，结果不可信。必须如实记录失败/超时状态 |
| 被打断 | 写入打断状态 | 否 | 等高优任务完成后由触发器决定是否恢复。恢复时需重新检查触发条件是否仍然满足 |

> **为什么"空结果"要单独区分**：空结果和失败/超时有本质区别。空是"模型正常执行、仔细看了、确实没有目标"，可以作为下游决策依据（如告诉用户"四周没有障碍物"）。而失败/超时是"模型没正常完成"，不能基于此做任何决策。如果混为一谈，会导致下游误判。

#### 2.2.5 阶段五：结果流转与任务续接（步骤 12-15）

| 步骤 | 节点 | 说明 |
|---|---|---|
| 12 | 结果回写信号库 | 四种结果统一写入信号库，更新对应任务的状态和结果字段 |
| 13 | 上报云端 Context | 信号库按策略上报。P0/P1 任务立即上报；Always-on 可合并上报减少流量；断网恢复后批量补报 |
| 14 | Adviser/Planner 消费 | 云端 Adviser/Planner 从 Context 模块订阅消费结果。**注意：不是触发器推送给他们的，是他们主动从 Context 读取的** |
| 15 | 队列继续/任务恢复 | 三种续接情况见下方 |

**步骤 15 的三种续接情况**：

| 任务类型 | 续接行为 | 说明 |
|---|---|---|
| 周期任务 | 回到条件扫描，等待下一个执行周期 | 如 Always-on 高频每 3s 一次，完成一次后等 3s 再回到步骤 3 |
| 单次任务 | 标记完成，不再触发 | 如下车安全跑完一次就结束，除非重新满足触发条件（如重新 D→P） |
| 被打断的任务 | 高优完成后恢复 | 恢复时需重新检查触发条件是否仍满足。特别地，Always-on 被打断后必须恢复（闲时常驻原则） |

> **关于循环回路**：从步骤 15 会回到步骤 3（获取信号快照），形成触发器的持续运行循环。触发器在整个车辆运行期间持续执行这个循环，而不是一次性的。

## 三、端侧信号库

### 3.1 业务流程

![](images/WnfCcuTJB4QhCsAhevA8U1aqWHZzcHpN.png)

**流程说明**（对应信号库流程图）：

信号产生 → 判断信号类型（任务状态 / 端状态信号 / OEM用户信号 / 导航场景信号 / 模型结果）→ 标准化为统一字段（key/value/source/timestamp）→ 写入端侧信号库 → 更新信号版本/有效期 → 分两路输出：**通知触发器或等待扫描** ｜ **按策略上报云端 Context**

### 3.2 信号写入规范

**产品要求**：所有信号必须标准化后写入信号库，至少包含信号标识（key）、信号值（value）、写入来源（source）、写入时间（timestamp）四个维度，确保触发器和云端可以统一消费。

> 具体字段命名规范和数据结构由研发设计接口协议。

### 3.3 功能列表

| 分类 | 信号 | 示例取值 | 来源 | 写入方 | 使用任务 |
| --- | --- | --- | --- | --- | --- |

### 3.4 VLM 结果写入规范

**产品要求**：VLM 执行完成后，必须将结果写入信号库，需包含：任务标识、任务类型、优先级、结果类型（标签/描述/空/失败）、执行状态（success/empty/timeout/failed/interrupted）、时间戳等信息，确保触发器可据此管理任务生命周期，云端可据此消费结果。

> 具体字段定义和数据结构由研发设计接口协议。

### 3.5 VLM 两条回路说明

对应架构图，VLM 执行后有两条独立的数据回路：

- **任务状态反馈**（VLM → 触发器）：VLM 将执行状态回传触发器，触发器据此管理抢占、超时和恢复
- **识别结果写入**（VLM → 信号库）：VLM 将识别结果写入信号库，供下游消费和云端上报

### VLM 结果要怎么写

| 状态 | 含义 | 能不能当有效结果用 |
| --- | --- | --- |
| success | 识别成功，有效标签/描述 | 可以 |
| empty | 模型正常执行，但没有发现目标 | 可以，表示无目标 |
| failed | 模型执行失败 | 不可以 |
| timeout | 模型超时未返回 | 不可以 |
| interrupted | 被高优任务打断 | 不可以，等待恢复或终止 |

| 结果 | 写入 | 注意事项 |
| --- | --- | --- |
| 成功识别 | 写标签和/或描述 |     |
| 未发现目标 | 写 empty | 表示模型正常执行但无目标 |
| 超时 | 写 timeout | **不允许写成"无异常"** |
| 失败 | 写 failed | 不触发下游主动服务 |
| 被打断 | 写 interrupted | 等待恢复或终止 |
| 低优覆盖限制 | — | P3 Always-on 不得覆盖 P0/P1 刚写入的关键结果 |

## 四、端侧触发器

端侧触发器就是端侧任务的"调度员"。

它做这几件事：

`看规则 看信号 判断任务是否命中 多个任务同时命中时排优先级 告诉 VLM/小模型去看什么` `处理任务成功、失败、超时和打断`

### 4.1 业务流程

![](images/Ym5PlJUAi5N1h7isN32kEyGCXelua4en.png)

**流程说明**（对应触发器流程图）：

加载任务规则 → 读取/监听信号库 → **判断运行时段** → ◇运行时段满足？→ 否：回到监听 → 是：**判断触发条件** → ◇触发条件满足？→ 否：回到监听 → 是：生成候选任务 → 检查任务状态（是否已运行/冷却/完成）→ ◇允许执行？→ 生成运行指令 → 调用 VLM/小模型 → 结果回写信号库

> 关键设计：条件匹配为**两级判断**——先匹配运行时段（离车后/上车前/停车时/行车中），再匹配具体触发条件。运行时段不满足时不检查触发条件，避免无效扫描。

### 4.2 功能列表

| 一级功能 | 二级功能 | 产品诉求 |
| --- | --- | --- |
| 规则管理 | 规则加载与更新 | 支持启动时拉取全量规则，运行中支持增量更新；更新后需生效，异常时可回滚 |
| 规则管理 | 灰度 | 支持按车型/VIN 维度灰度下发规则 |

### 4.3 优先级与打断恢复

| 优先级 | 任务类型 | 抢占规则 | 示例 |
| --- | --- | --- | --- |
| P0 | 用户即时/安全强相关/离车即时提醒 | 立即抢占任意低优 | 下车安全、遗留物品、VQA |
| P1 | 安全/场景增强类 | 抢占 P2-P4，不抢 P0 | 停车寻车、地下停车场识别 |
| P2 | 持续服务类 | 仅 FIFO + 排队 | 增强哨兵、舱外迎宾、儿童照看 |
| P3 | Always-on 描述类 | 仅 FIFO，可被丢弃；**闲时常驻** | 舱内/舱外 Always-on |
| P4 | 动态 Adviser/Planner 任务 | 弱触发，可静默 | Adviser 条件任务、Planner 定时任务 |

### 4.4 Always-on 闲时常驻原则

> **会议确认（2026-05-07）**：端侧 VLM 的设计目标是"一直在干活"。没有更高优先级任务时，VLM 应始终在执行 Always-on 任务（舱内 4-7 / 舱外 4-8）。

- Always-on 是行车中 VLM 的**默认工作状态**，不存在"VLM 空闲"的情况
- 被高优任务打断后，高优完成即**立即恢复** Always-on
- Always-on 分为高频任务（每 3s 执行一次）和低频任务（每 5s 执行一次），低频任务共享高频 3s 的视频流内容并额外增加 2s
- 产品诉求：按任务优先级表执行；技术上的冲突裁剪、并发限制由研发出方案

### 4.8 抢占恢复规则

**产品原则**：

- 高优先级任务到达时，低优先级任务应被暂停
- 高优先级任务完成后，低优先级任务应自动恢复（特别是 Always-on，闲时必须常驻运行）
- 运行时段切换时（如从行车中变为停车时），当前时段任务终止，新时段规则开始扫描

> 恢复的具体条件判断（触发条件是否仍满足、超时判断等）由研发出技术方案，产品确认。

**被打断任务的处理边界（2026-05-09 需求对齐确认）**：

- 任务在 VLM 执行过程中被高优任务打断时，打断和重连的管理由 VLM 业务侧负责，触发器不感知中间打断状态
- 触发器只会收到最终的执行结果（成功/空/失败/超时），不会收到"被打断"的中间状态通知
- 高优任务完成后，VLM 业务侧会自行恢复被打断的任务并继续执行，触发器收到结果的时长可能会变长，但不影响触发器逻辑
- 触发器侧的"抢占恢复"指的是：触发器在下发层面管理任务优先级标记和恢复意图，实际的模型资源抢占和恢复执行在 VLM 业务侧完成

> **总结**：触发器只管"下发指令 + 接收最终结果"，中间的打断/重连/恢复由 VLM 业务侧管理。如研发评审时有不同方案，以研发方案为准。

### 4.9 冷却与队列策略

**产品诉求**：

- P0 任务（安全/用户即时）每次满足条件都必须触发，不设冷却
- 周期任务按执行周期自然间隔（Always-on 高频 3s / 低频 5s）
- 队列需有容量上限保护，超出时低优先级任务丢弃

> 冷却时长、队列上限等具体参数由研发出方案，产品确认。

**并发任务数量限制（2026-05-09 需求对齐确认）**：

- 当前端侧 VLM 大约支持同时并发处理 3 个类型的任务（具体数值以模型推理服务研发评估为准）
- 超过并发上限时，低优先级任务（如 Always-on P3）会被暂停，待并发数降下来后自动恢复
- 并发能力的最终数值由模型推理服务研发团队评估确认，触发器侧按该数值做队列容量参考

> **说明**：并发限制属于模型推理服务的性能约束，不属于触发器产品逻辑。触发器只负责标记优先级并下发，VLM 业务侧根据并发能力做实际调度。

### 4.11 运行时段信号状态定义（2026-05-09 需求对齐新增）

> **背景**：运行时段（离车后/上车前/停车时/行车中）是产品给出的业务描述，研发实际判断时需要读取具体的信号状态组合。以下表格定义每个运行时段对应的信号条件，触发器据此做第一级条件匹配。

| 运行时段 | 电源状态 | 档位 | 车门锁 | 车速 | 补充说明 |

## 五、动态任务（Adviser/Planner 注册）

### 5.1 概述

Planner/Adviser 可在运行时通过**云端触发器**向端侧触发器注册条件任务。端侧触发器将其作为普通规则纳入两级条件扫描，命中后下发 VLM 执行。

> **会议澄清（2026-05-07）**：动态任务的下发路径是 Planner/Adviser → 云端触发器 → 端侧触发器。端侧 VLM 对 Planner/Adviser 来说是一个"tool"（工具），调用路径通过云端 VLM → 端侧 VLM 下发。执行结果通过 context 模块回到云端，Adviser/Planner 从 context 消费。

### 5.2 注册要素

动态任务注册时需提供以下产品要素（具体接口协议由研发设计）：

- 任务名称、注册方标识
- 运行时段和触发条件（格式同静态规则）
- 识别目标描述（task_goal）
- 优先级（默认 P4）、互斥组
- 过期策略（行程结束自动过期 / 指定时间 / 永不过期）

**注册结果**：注册方应能收到注册成功或失败的反馈（如条件格式错误、队列已满等）。

### 5.3 动态任务生命周期

**产品要求**：动态任务从注册到执行完成需经历完整的生命周期管理，包括：注册生效、条件命中触发、执行中、执行完成/过期/撤回。周期任务执行完成后应能重新进入条件扫描。

> 具体状态流转设计由研发出方案。

### 5.4 核心规则

- 接收并保存动态任务（本地持久化，弱网可执行）
- 按注册条件纳入正常两级条件扫描
- 执行前检查上下文有效性（如用户是否仍在车内）
- 支持注册方主动撤回（传入 rule\_id）
- 车机重启后自动恢复未过期的动态任务
- 动态任务总数上限：建议 16 条

### 5.5 动态任务执行链路

> **会议澄清**：端侧触发器不直接与 Adviser/Planner 通信。动态任务的完整链路为：

1. Planner/Adviser 通过云端触发器向端侧触发器注册条件任务（action\_type: run\_vlm）
2. 端侧触发器条件命中后，下发运行指令给端侧 VLM（端侧 VLM 作为 Planner/Adviser 的"tool"）
3. VLM 执行结果写入信号库 → 上报云端 context
4. Adviser/Planner 从 context 模块消费结果，自行决定后续动作

### 5.6 示例场景

**Adviser 需要 VLM 观测**：舒适 Adviser 想知道"副驾是否在打哈欠"，通过云端触发器注册动态任务（action\_type: run\_vlm, task\_goal: "观察副驾乘客面部状态", mutex\_group: VLM\_CABIN），端侧触发器将其作为 P4 VLM 任务调度，执行结果通过 context 回到 Adviser。

**Planner 需要 VLM 识别**：Planner 注册"看到 XXX 上车就告诉我"（action\_type: run\_vlm, task\_goal: "持续关注是否有特定人上车", mutex\_group: VLM\_FRONT），触发器将其作为 P4 VLM 任务调度，结果通过 context 回到 Planner。

## 六、Fail 定义与验收标准

### 6.1 Fail 定义

| Fail 类型 | 定义 | 典型案例 | 严重程度 |
|---|---|---|---|
| 漏触发 | 条件满足但任务未触发 | D→P + 解锁了，下车安全没跑 | P0-致命 |
| 误触发 | 条件未满足但任务触发 | 还在 D 档开着车，下车安全触发了 | P0-致命 |
| 重复触发 | 同一事件触发多次 | 一次锁车，遗留物品检测跑了 3 次 | P1-严重 |
| 优先级错误 | 高优被低优挡住 | 用户问了 VQA，Always-on 没让路 | P0-致命 |
| 信号库污染 | 超时/失败写成伪结果 | 下车安全超时，信号库写了"无障碍" | P0-致命 |
| 抢占不恢复 | 高优完成低优不恢复 | VQA 跑完了，Always-on 没恢复 | P1-严重 |
| 队列阻塞 | 超时不释放 | VLM 卡住，后续全排队 | P0-致命 |
| 规则未生效 | 热更新后规则没起效 | PM 改规则 5 分钟后端侧还在跑旧规则 | P1-严重 |
| 动态任务丢失 | 重启后动态任务未恢复 | 注册"10分钟后关加热"，重启后丢了 | P1-严重 |
| 动态注册无反馈 | 注册失败但注册方不知道 | Adviser 注册条件任务，格式错误但没返回失败 | P1-严重 |
| 运行时段误判 | 时段判断错误导致触发/漏触发 | 行车中误判为停车时 | P0-致命 |

### 6.2 产品体验指标

| 指标 | 目标值 | 说明 |
|---|---|---|
| P0 任务端到端响应 | ≤ 5s | 从触发条件满足到结果写入信号库 |
| P1 任务端到端响应 | ≤ 8s | 同上 |
| P2 任务端到端响应 | ≤ 10s | 同上 |
| Always-on 高频周期 | 3s | 每 3s 执行一次 |
| Always-on 低频周期 | 5s | 每 5s 执行一次 |
| 规则更新生效 | 用户无感 | 更新后应尽快生效，不影响当前任务 |
| 崩溃恢复 | 用户无感 | 触发器崩溃后应自动恢复，不丢失正在运行的任务 |

> 触发器内部的调度延迟、队列释放等工程指标由研发定义和验收。

### 6.3 线上验收用例

| 用例 | 输入 | 预期 |
|---|---|---|
| 下车安全 + Always-on | D→P + 解锁 + Always-on 在跑 | Always-on interrupted，3-1 执行，完成后恢复 |
| VQA 抢占 | Always-on 在跑 + 用户 VQA | Always-on interrupted，VQA 执行，完成后恢复 |
| 超时 | 触发 1-5 + VLM 超时 | 写 timeout，不写伪结果，队列释放 |

| 动态 VLM 任务 | Adviser注册 run\_vlm "副驾观测" + P0 任务抢占 | 动态任务被挂起，P0 完成后恢复，结果通过 context 回到 Adviser |
| 动态任务过期 | 注册 expire\_at=current\_trip\_end + 行程结束 | 任务自动清理，不再触发 |
| 动态任务撤回 | Planner 注册后主动撤回 | 状态变为 cancelled，从扫描中移除 |
| 4-5+4-6 合并 | 安全提醒在跑 + 儿童照看触发 | 合并 task\_goal 下发，两组结果分别写入 |
| 4-6 单独结束 | 合并运行中 + 用户关闭儿童照看 | task\_goal 退回单任务模式，4-5 继续 |
| 运行时段切换 | 行车中任务在跑 + 用户停车（D→P） | 行车中任务终止，停车时任务规则开始扫描 |

## 七、Context 上报策略

### 7.1 上报内容

| 字段 | 是否上报 | 说明 |
|---|---|---|
| task_id, task_name, status | 是 | 基础信息 |
| label_result / desc_result | 视任务 | P0/P1 全量上报；P3 Always-on 只上报变化 |
| timestamp | 是 | 端侧生成时间 |
| 车辆状态快照 | 是 | 上报时刻车速/档位/车门（用于云端追溯） |

### 7.2 上报时机

对应主流程图步骤 17-18：

- 任务状态变为 Success/Empty/Failed/Timeout 时立即上报
- Always-on 周期任务可合并上报（减少流量）
- 动态任务状态变更（triggered/done/expired/cancelled）上报
- 支持网络恢复后批量补报
- 上报至云端 context 模块后，Adviser/Planner 从 context 订阅消费（步骤 18）

### 7.3 脱敏规则

- 人脸/人体图像不上传原始图，只传标签和描述
- 车牌号不上传或脱敏处理
- VQA Query 文本可上报，关联语音不上传

## 八、下电后行为与离车后任务

### 8.1 下电后触发器与信号库状态

- 车辆下电后，触发器和信号库进入**低功耗模式**，仅保留离车后任务（1-1/1-2/1-3/1-4/1-5）的运行能力。
- 电源由车辆常电提供，需与整车厂确认功耗预算。
- 下电后信号库不再接收 CAN 信号（CAN 休眠），仅保留已缓存的信号和 VLM 结果。
- 下电后动态任务中 expire\_at = current\_trip\_end 的自动清理。

### 8.2 离车后任务最长运行时间

| 任务 | 建议最长运行时间 | 超时处理 |
|---|---|---|
| 1-1 增强哨兵 | 8h 或电量 < 20% 时停止 | 记录日志，通知用户 |
| 1-4 豆包抢车位 | 30min 或用户取消 | 标记 Cancelled |
| 1-5 遗留物品 | 下电后 5min 内完成 | 超时标记 Timeout |
| 1-2 / 1-3 停车寻车 | 下电后 2min 内完成 | 超时标记 Timeout |

## 九、本期范围与长期方向

### 9.1 本期不做

1. 不建完整云端规则管理后台 → 用简单配置方式，后台后续补（需找仁杰要配置平台权限）
2. 不定义 VLM 内部算法 → VLM 是黑盒，触发器只管调接口
3. 不做用户自定义场景 → 用户不能说"帮我盯一下这个"
4. 不做复杂自然语言规则解析 → 规则用结构化 JSON，Planner 侧负责解析后再注册
5. 不定义车控安全策略 → 触发器只管"触发感知任务"
6. 不管长时任务的结束条件 → 由云端 go list 管理（意图侧晓伟负责）
7. 动态任务本期仅支持 action\_type: run\_vlm → 不支持触发器直接通知 Adviser/Planner

### 9.2 长期方向

- 云端按 VIN/车型/版本灰度下发规则
- Adviser/Planner 大规模动态注册
- 端侧算力预算管理（多模型并发调度）
- 自学习触发阈值（按用户行为微调）
- 触发可解释 UI（向用户展示"为什么触发"）
