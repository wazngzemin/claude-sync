5 · 6.30 会议整理：动态 Advisor 视觉识别目标下发链路

会议时间：2026-06-30 14:00–15:01  |  参会人：陈西钺、丁彬、张兰若、吴仁杰  |  来源：1-原始素材/会议纪要/6.30 端侧vlm会议纪要.md

§0 一句话总结

这次会议定了动态 Advisor 视觉感知的完整下行链路——从 Advisor 发出感知需求，到触发器拼 SP、下发端侧 VLM、拿到结果、还原 SP 的全流程。核心共识：舱内走端侧 VLM（合规不上云）、舱外走云端 VQA；提槽工具公共化给触发器调；SP 拼接/删除全由触发器管理；枚举类信号直接匹配、描述类才走动态 SP。

§1 会议背景与核心议题

维度 说明
起因 上汽项目中，动态 Advisor 的视觉感知全部走云端 VQA。赛力斯项目因合规要求，舱内图片不能上云，必须走端侧 VLM。因此下行链路需要重新对齐。
核心议题 动态 Advisor 发起视觉感知需求后，谁判断舱内/舱外？谁拼 SP？SP 怎么管理生命周期？枚举和非枚举信号怎么分流？
前置背景 6.29 西钺已与泽民初步沟通方案：VQA 工具内部区分舱内/舱外 → 舱内的 SP 交给触发器拼接下发。本次会议是扩大到端侧 VLM 团队（丁彬、张兰若）一起对齐。

§2 九大核心结论

结论 ① 舱内 / 舱外分流方案 合规驱动

方向 链路 原因 调用方式
舱内 端侧 VLM（AIBox） 舱内图片含乘客隐私信息，不能上云，只能在端侧推理 周期性 5 秒
舱外 云端 VQA 舱外图片可上云，且 VQA 有联网信源（懂车帝、LBS 等）能力 按需调用
西钺 00:12:16："仓外的图是持续上云的吗？不是，是有调用才会上云。"

结论 ② 提槽工具公共化 架构变更

现状：提槽模型（判断 direction：舱内/舱外/舱外什么位置）包裹在 VQA 工具内部。

变更：西钺将提槽工具从 VQA 中拆出来，独立部署，"人人尽可用"。触发器直接调提槽工具，无需绕一圈走 VQA。

西钺 00:55:02："现在我把这个提槽工具放外面，人人尽可用。"

触发器使用场景：当 Advisor 下发的感知需求不在枚举范围内时，触发器调提槽工具判断该需求属于舱内还是舱外，再决定走端侧 VLM 还是云端 VQA。

结论 ③ SP 拼接由触发器负责 职责确认

角色 职责
动态 Advisor 给出感知需求的人话描述（如"看毯子掉没掉"、"副驾在不在睡觉"）+ 开启/关闭任务
触发器（泽民） 接收人话描述 → 拼接到 always-on SP 中 → 下发端侧 → 任务结束时删除对应 SP 片段
端侧 VLM（丁彬/兰若） 接收拼好的 SP → 每 5 秒推理 → 按 KV 槽位返回结果 → 不做条件判断，只输出客观描述
丁彬 00:34:13："拼 SP 放到泽民那拼…就是云端触发器那拼。"

讨论过的备选方案：张兰若提议由 VLM 端自己拼（"接收一个新的就放进去一个"），但最终因触发器本身就是下发触发方、且需要管理 SP 的生命周期，确定由触发器拼。

结论 ④ SP 生命周期管理 关键流程

SP 的增删由 Advisor（决策方）+ 触发器（执行方） 配合完成：

  1. 创建：Advisor 创建感知任务 → 通知触发器 → 触发器把新 SP 片段拼入 always-on SP
  2. 运行：端侧 VLM 每 5 秒用完整 SP（原始 + 动态追加）推理 → 结果写入对应 KV 槽位
  3. 完成/取消：Advisor 判断任务达成（或用户取消）→ 通知触发器停止 → 触发器删除对应 SP 片段 → 还原到原始 always-on SP
西钺 00:31:07："你这个触发场景取消了，你就得同时给端模发一下，这个里面如果包含 SP，那就把 SP 也删掉。"
丁彬 00:35:29："adviser 就会告诉触发器任务停止，然后云端触发器就要把那个 SP 还原，就把新加的 SP1 给去掉。"

结论 ⑤ VLM 只输出客观描述，不做条件判断 能力边界

VLM 能做的 VLM 不能做的 谁来做
每 5 秒看一次，输出
"毯子在脚底" / "毯子盖着"
"副驾闭眼" / "副驾看手机"		 "毯子掉了告诉我"（条件触发式）
"看到X就怎么样"（带触发逻辑）		 触发器做条件判断
（阈值穿越 / 状态变更）
西钺 00:18:43："我们只说毯子的状态在哪，掉不掉应该是应用方来判断。"

关键设计原则：VLM 就是个"眼睛"，只看、只描述；"大脑"的判断逻辑全部在触发器层。和端侧车况信号的处理模式完全一致——VLM 输出 = 原始信号，触发器做"变更为"/"等于"等运算。

结论 ⑥ 枚举类 vs 描述类信号处理 分流规则

类型 定义 举例 处理方式
枚举类 VLM 已固化的标签输出，值域固定 性别（男/女）、位置（主驾/副驾/后排）、有无小狗（有/无）、在不在打电话（是/否） 已在 Context 中 → 触发器直接条件匹配，等同于"空调开没开" → 不需要额外拼 SP
描述类 没有固定标签，需要 VLM 根据 SP 临时回答 毯子掉没掉、穿什么衣服、小狗项圈在不在、看到瑞幸告诉我 Context 中没有 → 需要拼动态 SP 下发 VLM，走完整的下行链路
西钺 00:49:38："所有枚举类的都是当成直接条件，你可以直接用等号来处理。不是枚举类的，都要去拼 SP。"

推论：枚举类信号对触发器来说，和车况端状态（空调/座椅加热/车窗）的处理方式完全一致——都是已有的、可枚举的值，用"等于"/"变更为"运算符匹配。描述类信号才是本次新增的动态 SP 链路需要处理的对象。

结论 ⑦ 多任务必须合并到同一 SP 性能约束

如果同时有多个动态感知任务（如"观察后排宝宝" + "观察后排小狗" + "看窗外红绿灯"），不能分别下发到不同推理周期，必须全部拼到同一个 SP 中，让 VLM 一次推理全部回答。

方式 说明 结论
分别下发 每个任务一个独立推理周期 → 推理周期变长，5 秒可能不够 → 延迟叠加 否决
合并下发 所有任务拼入同一 SP → 每 5 秒一次推理拿到所有结果 → 各任务用 KV 槽位区分 采纳
西钺 00:28:59："你不能把多个任务放到不同的循环里面，不然每个循环的推理周期就变长了，所以你要把它放在一起去做推理。"

结论 ⑧ Advisor 先查 Context 已有信息再决定是否下发 过滤机制

动态 Advisor 在创建感知任务前，必须先检查 Context 里是否已有足够信息：

  • 如果 Context 中已有枚举标签（如"副驾=女性"）能回答 Advisor 的问题 → 不下发，直接用已有信息
  • 如果 Context 中信息不足（需求不在枚举范围内）→ 才走动态 SP 下发链路
西钺 00:44:35："宝宝哭了就立刻告诉我，你肯定要先过滤一次，先去看当前的 context 里面有没有这个信息，有的话你就不应该再下发任务了。"

设计意义：避免已有标签的信息被重复下发成动态 SP，浪费 VLM 推理资源。

结论 ⑨ 静态 vs 动态 Advisor 定义确认 概念对齐

类型 特征 举例 生命周期
静态 预设好的，系统启动即存在 舒适 Advisor、情感 Advisor、安全 Advisor、出行 Advisor（共 4 个） 始终存在，不销毁
动态 由用户 query 触发创建 "宝宝毯子掉了告诉我" → 创建一个毯子监控 Advisor 任务完成即销毁
西钺 00:51:34："静态就是预设的，一定会有的。动态就是根据用户的 query，实时创建、用完就毁的那种。"

§3 完整数据流（动态 Advisor 视觉感知链路）

3.1 核心链路（舱内-描述类信号）

用户："宝宝毯子掉了告诉我" │ ▼ Planner 理解意图 → 创建动态 Advisor（毯子监控） │ ▼ 动态 Advisor 检查 Context： ├── "毯子状态" 在枚举标签里吗？→ 不在 │ → 需要动态 SP ▼ 动态 Advisor → 云端触发器：注册感知需求 │ 需求内容（人话）："帮我看毯子掉没掉" │ ▼ 云端触发器 调提槽工具：判断 direction │ → 结果：舱内（后排） │ ▼ 云端触发器 拼 SP： │ 原始 always-on SP + "观察后排毯子状态，回答在 key=blanket_status" │ → 完整 SP 下发到端侧触发器 │ ▼ 端侧触发器 → 端侧 VLM（AIBox） │ 每 5 秒推理一次，SP 中含动态追加部分 │ ▼ VLM 推理输出（KV 格式）： │ { ..., blanket_status: "毯子盖在宝宝身上" } ← 第 1 个 5 秒 │ { ..., blanket_status: "毯子盖在宝宝身上" } ← 第 2 个 5 秒 │ { ..., blanket_status: "毯子滑落在脚底" } ← 第 3 个 5 秒 ⚡ │ ▼ 结果写入端侧信号库 → 上报 Context │ ▼ 云端触发器 订阅 Context，检测到状态变更： │ blanket_status: "盖着" → "滑落" （false→true 阈值穿越！） │ ▼ 云端触发器 push 事件 → 动态 Advisor │ ▼ 动态 Advisor 判断任务达成 → 触发用户提醒 │ → 同时通知云端触发器：任务结束 │ ▼ 云端触发器 删除 blanket_status 相关 SP 片段 │ → 还原到原始 always-on SP │ ▼ 端侧 VLM 下一个 5 秒周期开始用还原后的 SP 推理 │ → 链路恢复到动态任务前的状态 ✅

3.2 快捷链路（舱内-枚举类信号）

用户："副驾打电话的时候把音量降低" │ ▼ Planner → 创建动态 Advisor（副驾通话监控） │ ▼ 动态 Advisor 检查 Context： ├── "是否在打电话" 在枚举标签里吗？→ 在！ │ → 不需要拼 SP，直接走触发器条件匹配 ▼ 动态 Advisor → 云端触发器：注册条件 │ 信号：phone_call = true（枚举值） │ 运算符：变更为 │ 动作：降低音量 │ ▼ 云端触发器 等待 Context 中 phone_call 字段变更 │ （数据来源：VLM always-on SP 的固有标签输出） │ → phone_call: false → true ⚡ │ ▼ 触发动作 → 完成 ✅ （全程不需要拼任何动态 SP，和车况信号的处理完全一致）

3.3 舱外链路

用户："看到瑞幸咖啡的时候提醒我" │ ▼ 动态 Advisor 检查 Context → 不在枚举范围内 │ ▼ 云端触发器 调提槽工具 → direction = 舱外 │ ▼ 云端触发器 直接调云端 VQA（图片可上云） │ 按需取图 → VQA 推理 → 结果回 Context │ ▼ 云端触发器 检测结果 → 匹配到"瑞幸" → 触发提醒 ✅

§4 与上汽方案的差异对比

维度 上汽方案（旧） 赛力斯方案（本次确认）
舱内视觉 图片上云 → 云端 VQA 推理 图片不上云 → 端侧 VLM 推理（合规）
舱外视觉 云端 VQA 仍可走云端 VQA（不变）
提槽判断 VQA 工具内部自带 提槽工具公共化，触发器可独立调用
SP 来源 VQA 工具内部处理 Advisor 给人话需求 → 触发器负责拼接/管理/删除
条件判断 VQA 或 Advisor 各自处理 触发器统一做（阈值穿越/变更为/等于）
动态 Advisor 链路 Advisor → 直接调 VQA → 拿结果 Advisor → 触发器 → 拼 SP → 端侧 VLM → 信号库 → Context → 触发器 → Advisor

§6 关键讨论过程还原

6.1 "VLM 该不该做条件判断"的争论

围绕"宝宝毯子掉了告诉我"这个 case，会上出现了两种理解：

方案 内容 结论
条件式 给 VLM 的 SP 写成"毯子掉的时候告诉我" → VLM 自己判断掉/没掉 → 只在掉的时候返回结果 否决
观察式 给 VLM 的 SP 写成"帮我看毯子状态" → VLM 每 5 秒都客观描述 → 触发器做变更判断 采纳

核心理由：端侧 VLM 没有专门做触发逻辑的能力（不像豆包 APP 可以 prompt 里写"当什么什么的时候告诉我"），只能做"看一看，说出来"。条件判断、时间窗口、穿越检测都是触发器的职责。

6.2 "拼 SP 在哪里做"的争论

方案 支持者 理由 结论
VLM 端自己拼 张兰若 "接收一个新的就放进去一个，不需要上游帮我们拼好" 未采纳
触发器拼 西钺、丁彬 触发器本身就是下发触发方，触发时带上 SP 是自然动作；且触发器负责 SP 生命周期管理 采纳
西钺 00:30:18："触发是泽民来触发嘛，他触发的时候带上 SP，是一个比较自然的动作。"
丁彬 00:34:02："拼 SP 放到泽民那拼…谁发谁拼好像比较好，不然返回去的时候是一堆信息，挑不了。"

6.3 "佳杰的场景表"争议

会中看了佳杰之前整理的一个动态 Advisor 场景表（含"存在小狗"/"出现小狗"/"移除小狗"等事件描述），但参会者对其定义和使用方式存在困惑：

  • 丁彬：这些应该是静态的（已有标签），为什么标成动态？
  • 西钺：不清楚佳杰的场景是什么，"聊半天没什么场景可以用"
  • 张兰若：佳杰可能想了一些动态 Advisor case，但没跟 VLM 团队对齐

结论：暂不依赖佳杰的表，触发器侧自行定义枚举范围，和端侧 VLM 团队直接对齐。

6.4 VLM 错误容忍度问题

张兰若 00:39:39："我们但凡出现一次错误，它发现两次。比如狗这个事突然没有了，第二次可能有个 glitch 又出现了，触发器就会有问题。VLM 不能出错，错了他这边也会跟着错。"

启示：触发器的阈值穿越检测对 VLM 输出的稳定性有强依赖。如果 VLM 输出抖动（一会儿有狗一会儿没狗），会导致触发器误触发。这是后续需要端侧 VLM 团队保证输出稳定性的重要约束。
