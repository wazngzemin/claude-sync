# 触发器系统 · 业务流程讲解稿

> 对着《触发器-业务流程-全模块大图》讲。上半是云端，下半是端侧，橙粗线是端云之间传数据。从左到右看，按四条路径把整个流程串一遍。

---

## 先说整张图的结构

两个入口，一个出口：

- **入口一**：用户开口说话（左下角 ①）
- **入口二**：车辆自己产生信号（左下角 ③）
- **出口**：用户听到/看到结果（右下角 ㉙）

所有路最终都汇到右边的大脑 ⑲⑳ Director，由它决定说什么、做什么。

---

## 路径一：简单车控（最快，不进大脑）

> 例子："打开车窗"

① 用户说话 → L1 → ② ASR转文字 → L2（端→云）→ ⑨ 拒识仲裁，判断是简单车控 → L3（云→端）甩回端侧 → ⑤ 离线fc+白名单匹配 → L7 → ⑥ 命中白名单？→ 命中 → ⑦ 原子车控执行 → ⑧ 车控生效+卡片 → 用户看到"车窗已打开"。

**特点**：全程不进大脑、不绕触发器，端侧毫秒级搞定。

---

## 路径二：对话/复杂请求（走大脑）

> 例子："我好热" "附近有招商银行吗"

① → L1 → ② ASR → L2 → ⑨ 拒识仲裁，判断是复杂/对话类 → L5（长线直达右边）→ ⑲ Director汇入。

⑲ 同时注入 ㉑ Context（车内温度/用户记忆/端状态），做前置筛选（只挑相关工具和示例喂模型）→ L25 → ⑳ Director推理。

⑳ 输出三样：talk_or_not（说不说话）、talk_content（说什么）、action_list（动作清单）。然后扇出：

- L29（云→端）→ ㉖ TTS播报+卡片 → L31 → ㉙ 用户听到
- L27 → ㉒ 工具库（23个工具：车控/搜索/导航/音乐/记忆…）执行后 → L32 tool_feedback **回 ⑲ 二次推理**（不是一锤子，大脑会接着想）
- L30（云→端）→ ㉗ 机器人（转头/表情）+ ㉘ 车控执行
- L28 → ㉓ 任务中心（widget卡片/任务岛/AI PUSH）

**特点**：大脑结合情景推理，不只应一句——"我好热"它知道车内28度、你平时喜欢24度，自己想好调几度。

---

## 路径三：条件/定时任务（走触发器）

> 例子："超过25度就开空调" "十分钟后关座椅加热"

① → ② → L2 → ⑨ 拒识仲裁，判断是带条件的话 → L4 → ⑩ 意图模型解析，把人话翻成触发器协议（"超25度开空调" → condition:温度>25, action:开空调）→ L11 → ⑬ 火山后台注册成一个"场景"。

⑬ 火山后台里存着所有场景，一个场景 = conditions + actions + priority + 频控。6个API管增删改查（signals/actions/scene/update/query/status），走HMAC-SHA256签名。→ L13 场景库给 ⑮。

同时，信号那边一直在跑：③ 端信号源（车速/温度/车门/VLM视觉/围栏/音频/时间）→ L8 → ④ 端侧信号库（标准化快照 key/value/timestamp）→ L9（端→云同步）→ ⑫ 云端信号库全集（也接 ⑪ 云信号源的 L10）→ L12 信号快照给 ⑮。

⑮ 触发器引擎（云端）拿到场景库 + 信号快照，内部流水线：场景注册 → 信号扫描（每5秒或信号更新时）→ 规则判定 → 模型推理 → 条件全达成？→ 抖动过滤 → 冲突仲裁（优先级1-10，Planner默认最高）→ 端云分类 → 场景下发 → 命中输出。

条件达成后 → L21 通知 ⑲ Director"条件触发了" → Director真正去执行车控。

**注意**：触发器只负责"时机到了喊一嗓子"，真正执行是Director做的。

另外 ⑳ Director 也会反过来注册条件任务：L33 → ⑮（对话中产生的条件任务注册进触发器，等信号满足再回来）。⑭ 条件提槽负责拆条件项 + 常识拦截（"世界毁灭时提醒我"直接注册失败）。

---

## 路径四：主动服务（用户不用开口，车自己动）

> 例子：上车自动打招呼、副驾睡着调小音乐、快到目的地提醒找充电桩

信号一直在跑（路径三前半段一样）：③ → ④ → L9 → ⑫ → L12 → ⑮ 触发器引擎。

⑮ 条件命中后，往两个方向发：

**→ L17 → ⑯ 静态Advisor（永久监工）**：固定4个——舒适/出行/情感/内容。靠预设场景触发（上车→情感打招呼、擤鼻涕→舒适关怀、抽烟→提醒开窗）。想完 → L19 建议回 ⑲ Director。

**→ L18 → ⑰ 动态Advisor（临时监工）**：一个目标配一个，用完删。"帮我照看宝宝"就建一个专门盯着。内部管4类任务（长时2/持续GUI 1/条件6/定时6），超上限返error。想完 → L20 建议回 ⑲ Director。⑰ 还会 → L22 下发感知订阅给 ㉕ 感知器（端侧），VLM定时捞图做VQA（"宝宝睡了没"→ 返回result+description）→ L23 结果回传 ⑰。

⑱ GoalList 是 Advisor 和 Director 共用的目标看板 → L24 → ⑲。

Director 收到建议后推理、输出、执行——跟路径二的后半段一样。**用户全程没开口，车自己主动做了。**

---

## 端侧触发器（㉔，还没建起来）

⑮ 通过 L14（云→端）把场景下发到 ㉔ 端侧触发器。㉔ = VLM视觉任务调度器，内部也有一套流水线：场景更新 → 信号扫描 → 规则判定 → 优先级P0-P4 + 互斥组（VLM_CABIN舱内/VLM_FRONT舱外）→ 冷却（P0不冷却，Always-on高频3秒/低频5秒）→ 下发运行指令（7字段）→ VLM业务执行（起步安全P0/下车安全P0/VQA P0/遗留物品P0/哨兵P2/迎宾P2等）→ 终态（success/empty/timeout/failed，铁律：空≠没看、超时不写假结果）→ 结果 L16 回写信号库做二次触发 → 端本地动作执行（车控/TTS，弱网闭环）。

**现状**：这套端侧能力几乎没建，全堆在云，弱网就废——这是需求9要解决的。

---

## 两个特殊节点

**㉚ 拒识后→Context事件日志**：⑨判断"不该接"的话（L6），不丢掉，进Context事件日志让Advisor能感知到。

**铁律**：触发器只读信号库，绝不直接监听原始信号源——信号源怎么变触发器不用改。（图上黄色便签写的）

---

## 五条回环（图上虚线，系统"活"起来的关键）

1. **L32** ㉒工具 → ⑲ Director：工具结果回大脑二次推理（多步任务靠它）
2. **L33** ⑳ Director → ⑮ 触发器：对话产生的条件任务注册进触发器
3. **L16** ㉔ 端侧VLM → 信号库：识别结果当新信号二次触发
4. **L19/L20/L21** Advisor+触发器 → ⑲ Director：建议和触发通知回灌大脑
5. **L9** ④ 端信号库 → ⑫ 云信号库：端云信号同步

---

## 一句话总结

从"你说一句它应一句"升级成"懂情景、会规划、能主动、说做一致"。触发器就是让车"能主动"的引擎。
