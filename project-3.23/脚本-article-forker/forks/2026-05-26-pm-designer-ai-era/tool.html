MODULE 1核心认知转变

对你的意义：你的产品感 + 车载场景 know-how + AI 工具的编码能力 = 比纯工程师更快的"洞察→行动"循环。你不需要成为工程师，你需要成为一个"会用 AI 构建的 PM"。

MODULE 2Reach Test
学习目标：掌握"reach test"方法，用来评估你的产品功能是否真正有价值。

具体操作步骤

  1. 从用户交互 log 中提取"自发唤醒率"（用户主动 vs 系统提示触发）
  2. 按功能/场景分组：导航、音乐、车控、搜索、闲聊……
  3. 自发唤醒率高的场景 → 用户已经"自然伸手"→ 优化体验、保障稳定性
  4. 自发唤醒率低的场景 → 追问为什么不伸手 → 不是简单加提示语

Checkpoint 2

能用reach test评估一个功能。能说出"自发唤醒率"怎么计算。

MODULE 3养 Agent SOP
学习目标：掌握 Agent 产品经理的养护节奏——日/周/Sprint/月各做什么，形成可持续的质量改进循环。

Dan 说："为了让一个 AI Agent 有用，它现在真的需要一个在乎它的人。" 你就是 Planner 的那个人。

关键理解：这不是"额外工作"，这就是Agent PM的核心工作。Dan说"自动化是谎言，每个Agent都需要一个在乎它的人"——你对Planner的养护质量，直接决定了产品质量。

MODULE 4工具定义优化
学习目标：用"Agent-Friendly检查清单"审计 Planner 的 23 个工具定义，找出拖累决策质量的模糊定义。

核心洞察

Dan 说 SaaS 要同时服务人和 Agent。对你的意义：23 个下游工具就是 Planner 的"SaaS"。工具定义的质量直接决定 Planner 的决策质量——这不是模型的锅，是"SaaS"（工具）的锅。

检查清单详解

检查项好的定义示例差的定义示例影响
功能描述"查询当前车辆空调温度、风量、模式""查询车辆状态"模糊→Planner可能选错工具
参数说明type:string, 取值:["high","mid","low"], 示例:"high"只有参数名缺说明→参数填错→报错
返回格式{"temp":25,"mode":"cool","fan_level":3}"返回查询结果"不结构化→Planner不知怎么用
错误场景"设备离线时返回error_code:OFFLINE"无错误说明遇错不知该重试还是放弃
使用场景"用户询问空调相关问题时使用"无场景说明模糊场景下不知该不该调
组合规则"通常先查状态，再调vehicle_basic_control"无组合提示错过应连续调用的工具

MODULE 5PM 做实验
学习目标：掌握4种"Marcus式"工作方法——用 Claude Code / Cursor 直接验证判断，而非等工程排期。

四种实验场景

Prompt 模板（复制即用）

我是一个车载AI助手（豆包上车）的AI产品经理，负责Planner模块。 Planner 输入 = 用户语音query + 车辆状态 + 用户记忆 Planner 输出 = JSON（talk_or_not, talk_content, action_list） action_list 里是要调用的工具名和参数。 请帮我写一个Python脚本，实现以下功能： 1. 读取 Planner 的 log 文件（每行一条JSON记录） 2. 提取每条记录的 user_query 和 action_list 3. 自动分类 badcase 类型： - 选错工具（action_list中的工具与query意图不匹配） - 参数错误（参数值不合理） - 理解错意图（talk_content与query不相关） - 幻觉（action_list调用了不存在的工具） 4. 输出分类统计表 + 每类的top3典型case
我有两版Planner的System Prompt（旧版SP_v1.txt和新版SP_v2.txt）。 请帮我写一个回测脚本： 1. 读取一组历史badcase的user_query（badcases.jsonl） 2. 分别用SP_v1和SP_v2构造prompt，调用模型API 3. 对比两组输出的差异： - 选择的工具是否一致 - 参数是否一致 - talk_content的语义相似度 4. 输出对比报告：哪些case在新SP下变好了、变差了、没变化

Checkpoint 5

能说出4种实验场景。能修改prompt模板来做你具体的实验。核心理解：PM的产出不再是文档，而是"验证了什么判断"。

MODULE 6住进未来
学习目标：建立个人AI工具探索习惯，让"试新模型/新工具"成为你的肌肉记忆。

Dan 的方法

"你不该做预言家。你该做的，是一起住进未来。"

不做会议室预测，而是让自己在新工具中生活。关键动作：试用 → 写下观察 → 命名变化。没有被命名的变化不会被组织学习。

你的"住进未来"清单

频率做什么为什么具体怎么做
每次新模型发布用固定badcase测试看新模型能否解决旧问题准备10个"Planner解决不了的case"，每次新模型出来跑一遍
每月试一个新AI工具找到提升效率的工具组合Claude Code / Cursor / Codex / Windsurf 轮着试
每季度重审"做不到"清单以前受限的现在可能可以了翻出上季度的"因模型能力不够所以放弃"的需求
随时写下观察（一句话就行）不命名的变化不会被组织学习飞书文档/笔记/这个知识库都行

"AI的边缘不在旧金山，而在AI遇到一个正在做事的人那里。"

你就在这个"边缘"上——车载智能座舱 × AI 的交叉点。你积累的场景know-how（什么时候用户想开空调、什么情况需要多步规划、什么话术让人舒服）就是Dan说的"边缘"。这种领域知识不会被模型更新取代。
