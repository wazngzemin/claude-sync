# 云端触发器 · 定时任务 Trigger Timer PRD

## Problem Statement

当前 Driver Agent 2.0 的执行模型以「同步会话」为主：用户说完一句话，Planner 在本轮对话内推理并执行，事情结束后不会留下持续运行的任务载体。然而车内存在大量「现在不必做、未来到点才做、或需要反复做」的需求，例如：

- 「十分钟后关闭座椅加热」
- 「明天 9 点提醒我出发」
- 「每小时开一次座椅按摩」
- 「每分钟帮我看看后排的宝宝有没有哭」

这些需求今天没有统一的产品载体：

1. **Planner 无法承载**：Planner 是单轮同步推理，任务一旦结束就不会持续跟踪时间。
2. **条件触发器无法覆盖**：已上线的条件触发器只响应车态信号变化（如车速、温度），无法表达「时间到了」这一维度。
3. **历史方案碎片化**：5.29 的动态 Advisor 设计曾让 Advisor 自建定时器，导致条件与定时两条点火路径割裂，上下电、上限、判重等逻辑各做一套。

结果是：时间类用户请求要么被当场错误执行，要么直接丢失，体验不完整；研发侧也缺乏统一接口与边界定义，产品口径容易被技术方案替代。

## Solution

在云端触发器（Cloud Trigger）中新增第三种触发类型 **Timer（定时触发）**，作为时间维度的点火器。

- **定时器只负责「到点通知」**：创建后挂在云端等待，到点向 Task 服务发送回调，告知「该执行了」。
- **动作执行复用已有链路**：到点后的执行前检查、车端唤醒、语音播报、卡片更新、生命周期管理，全部复用 Task 服务与 Runtime Dispatcher 已有能力。
- **统一收口**：条件触发、定时触发、语义触发三种类型统一走云端触发器，Task 服务只对接一个订阅方。
- **下电不丢**：定时器部署在云端并持久化到数据库，车辆下电后跨天任务继续计时，本趟任务按业务规则关闭。

用户获得的能力：

- 相对延时一次性任务（十分钟后…）
- 绝对时间一次性任务（明天 9 点…）
- 固定间隔循环任务（每小时…）
- 定时 + 语义组合任务（每分钟看宝宝是否哭了）
- 任务中心统一查看、取消、暂停

## User Stories

1. 作为驾驶员，我想说「十分钟后关闭座椅加热」，让系统在十分钟后自动关闭座椅加热并语音告诉我，避免我手动操作或忘记。
2. 作为驾驶员，我想说「明天 9 点提醒我出发」，让系统在明天 9 点整语音提醒我，防止我错过约定时间。
3. 作为驾驶员，我想说「每小时开一次座椅按摩」，让系统在行车过程中每小时自动帮我开启座椅按摩，缓解疲劳。
4. 作为带宝宝的家长，我想说「每分钟帮我看看后排的宝宝有没有哭」，让系统在检测到宝宝哭泣时及时提醒我并给出安抚建议。
5. 作为通勤用户，我想说「每个工作日 8 点提醒我开会」，让系统在工作日早上固定时间提醒我当日日程。
6. 作为驾驶员，我想要在任务中心看到我创建的所有定时任务，清楚知道哪些正在等待、哪些已完成、哪些已取消。
7. 作为驾驶员，我想要通过语音或任务中心卡片取消一个定时任务，确保后台真的停止计时，不会继续打扰我。
8. 作为驾驶员，我想要循环任务可以临时暂停（关开关）并在需要时恢复，而不是只能彻底取消。
9. 作为驾驶员，我希望车辆下电后，跨天定时任务（如明天 9 点提醒）不会被清空，下次上车照常生效。
10. 作为驾驶员，我希望车辆下电后，本趟车内定时任务（如十分钟后关加热）自动结束，下次上车不会莫名其妙执行。
11. 作为驾驶员，我希望当我重复说同一个定时任务时，系统更新原来的任务而不是叠加出多个闹钟。
12. 作为驾驶员，我希望定时任务达到上限后，系统明确告诉我并让我删除旧任务，而不是无声挤掉重要提醒。
13. 作为任务中心产品经理，我希望任务卡片的状态由 Task 服务驱动，任务中心不自行判断到点，避免状态不一致。
14. 作为 Planner 产品经理，我希望 Planner 只识别「这是任务类请求」并路由给 Task 服务，不需要解析具体时间。
15. 作为云端触发器研发，我希望只接收结构化时间规则（触发时间 / 间隔 / 循环类型），不处理自然语言解析，保持模块边界清晰。
16. 作为 Task 服务研发，我希望触发器到点后只回调任务 ID 和触发时间，不带执行细节，由我决定是否执行、如何执行。
17. 作为 QA，我希望定时触发精度达到秒级，能够验收「创建 600 秒后触发」这类用例。
18. 作为 QA，我希望循环任务在条件不满足时（如座椅没人）本轮跳过，但不影响下一轮正常触发。
19. 作为 QA，我希望 missed 触发（如下电期间到点）不补做，避免下电后集中爆发执行。
20. 作为项目经理，我希望定时任务的研发排期与任务中心对齐，确保同版本上线。

## Implementation Decisions

### 1. 触发器定时任务的定位

- 定时器是云端触发器内部的一个子模块，与条件触发、语义触发并列。
- 定时器只提供三个能力：**注册定时规则**、**更新/取消定时规则**、**到点回调**。
- 定时器不执行任务、不管理任务生命周期、不访问车端状态。

### 2. 时间规则结构化

触发器只接收结构化规则，不接受自然语言。规则包含：

- `task_id`（任务唯一标识）
- `trigger_type`：relative_delay / absolute_time / fixed_interval / calendar_cycle
- `trigger_param`：
  - relative_delay：delay_seconds（如 600）
  - absolute_time：trigger_at（如 2026-07-21T09:00:00）
  - fixed_interval：interval_seconds（如 3600）、max_runs
  - calendar_cycle：cron-like 规则（二期）
- `expire_at`（任务过期时间，可选）
- `close_on_vehicle_offline`（下电是否关闭，布尔值）

自然语言解析由 Task 服务的 Spec Builder 完成。

### 3. 统一回调契约

到点后，触发器向 Task 服务发送回调：

```
TriggerCallbackRequest {
  channel: "timer",
  vehicle_id: string,
  task_id: string,
  callback_id: string,
  triggered_at: timestamp
}
```

回调只携带身份信息和触发时间，不携带动作细节。Task 服务收到后决定执行什么、是否执行。

### 4. 定时器内部调度机制

- 所有时间规则统一归一化为「下一次触发时间戳 next_fire_at」。
- 使用最小堆（Min-Heap）管理所有等待中的定时器，保证取下一个待触发任务为 O(1)。
- 调度线程只关注堆顶任务，未到点时休眠，避免空转。
- 循环任务触发后，基于规则重新计算 next_fire_at，再次入堆等待。

### 5. 持久化与下电不丢

- 定时规则写入云端数据库（DB），车辆下电后任务不丢失。
- 触发器服务本身无状态，可水平扩展；数据库单点故障时依赖数据库高可用方案。
- 重启后从数据库恢复未触发的定时规则，重建内存堆。

### 6. 上下电策略

- 触发器收到车辆下电事件时，遍历该车辆所有等待中的定时任务。
- 若 `close_on_vehicle_offline = true`，关闭任务并记录关闭原因为 vehicle_offline。
- 若 `close_on_vehicle_offline = false`（跨天任务），继续计时；到点时若车辆不在线，本次跳过不补做。

### 7. 数量上限与判重

- 单车同时存在的定时任务上限暂定 20 个（一期），后续可按车型/用户等级扩展。
- 创建新任务时先检查上限，超限则明确拒绝并提示用户。
- 判重规则由 Task 服务定义（如相同 vehicle_id + 相同任务语义），重复创建视为更新原任务，重置定时器触发时间。

### 8. 循环任务的暂停与恢复

- 循环任务支持「关闭开关」操作：暂停计时，任务保留，下次打开开关时恢复。
- 「取消」操作：彻底删除定时器，任务结束。
- 暂停期间不计入 max_runs。

### 9. 定时 + 语义组合

- 「每分钟看宝宝有没有哭」拆分为：
  - 定时部分：每 60 秒触发一次回调。
  - 语义部分：Task 服务收到回调后，取最新视觉描述进行语义判定。
- 产品规则：只在「状态跳变」（从没哭变成哭）时提醒一次；持续哭不重复提醒；看不清不更新记忆。

## Testing Decisions

### 测试 seams

本功能涉及三个主要集成 seam，优先在最高 seam 做端到端测试：

1. **Trigger Adapter ↔ 云端触发器接口**：验证注册/更新/取消定时规则的请求与响应。
2. **云端触发器 ↔ Task 服务回调接口**：验证到点回调的字段完整性与幂等性。
3. **Task 服务 ↔ 车端执行链路**：验证到点后执行前检查、车端唤醒、卡片更新全流程。

理想情况下，以 seam 2（回调接口）为主要验收 seam：只要回调在正确时间带着正确 task_id 到达，即证明定时器工作正常；执行逻辑由 Task 服务已有测试覆盖。

### 好测试的标准

- 只测外部行为：任务是否在指定时间触发、触发后是否正确回调、取消后是否不再触发。
- 不测内部实现：不验证最小堆结构、不验证数据库行数、不验证线程休眠时长。
- 幂等性：同一任务重复注册只更新不创建；同一到点事件不会触发两次回调。

### 验收用例

1. 创建相对延时任务 600 秒，验证 600 秒后收到回调。
2. 创建绝对时间任务到未来某时刻，验证该时刻收到回调。
3. 创建固定间隔任务 3600 秒，验证连续 3 个周期均收到回调。
4. 循环任务条件不满足时验证本轮跳过、下轮继续。
5. 用户取消后验证后续不再收到回调。
6. 车辆下电后本趟任务关闭、跨天任务继续计时。
7. 达到上限后创建新任务被拒绝并提示。
8. 重复创建同一任务只更新原任务。

## Out of Scope

- **自然语言时间解析**：由 Task 服务的 Spec Builder 负责，触发器只接收结构化规则。
- **端侧定时器**：本期所有定时器部署在云端，不支持车端本地定时。
- **农历 / 节假日日历**：二期评估，依赖日历服务接入。
- **跨时区处理**：本期按车辆当前时区处理，不考虑跨时区迁移。
- **复杂日历表达式**：如「每月最后一个工作日」等复杂规则二期支持。
- **任务执行失败后的重试策略**：由 Task 服务定义，不在触发器侧实现。

## Further Notes

- **归属演进**：定时器归属经历了从 Planner → Advisor 自建 → 云端触发器的三次搬家。庆磊《Task 任务管理设计》已明确分工：Trigger 负责延时、周期、确定性条件和 VLM 语义匹配。本 PRD 基于该分工进一步细化产品规则与边界。
- **与任务中心的关系**：定时任务跟任务中心同版本上线。任务中心只负责展示与用户入口，状态由 Task 服务消息驱动。
- **关键风险**：触发器侧定时引擎的研发排期需刘杨/庆磊确认；「执行前检查」所需的车况字段清单需王泽民/韩杰补充。
- **二期展望**：日历周期规则、定时+语义组合的边沿判定、更细粒度的时间表达式。
