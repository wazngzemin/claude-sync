# 信号中枢（Signal Hub）

## 定义

信号中枢是韩杰在 7.10 评审会上提出的统一产品框架，将原来分散的端状态管理、触发器条件、事件日志生成整合为一个三表架构。它是端侧信号从上报→消费的统一需求管控入口。

## 三表结构

```
端侧信号上报
    ↓
┌─────────────────────────────────────────┐
│           信号中枢（一张大表）              │
│                                         │
│  Sheet 1: 状态查询                       │
│  · 112条信号（从数百条缩减）               │
│  · 消费方：Context（车辆状态+事件感知）      │
│                                         │
│  Sheet 2: 触发事件                       │
│  · 状态突变检测（原"动态条件"）             │
│  · 消费方：Advisor / 事件日志 / 规则引擎    │
│  · 每条含事件描述拼接规则                   │
│  · 支持非信号来源（埋点、客户透传）          │
│                                         │
│  Sheet 3: 条件                           │
│  · 静态条件判断（当前态）                   │
│  · 作为漏斗过滤触发事件                    │
│  · 串行/并行组合                          │
└─────────────────────────────────────────┘
    ↓
触发事件 → 条件过滤 → 执行动作
                      ├── 触发 Advisor
                      ├── 生成事件日志
                      ├── TTS 播报
                      └── [未来] 规则引擎 / 任务大师
```

## 与旧架构的对比

| 维度 | 旧（配置平台） | 新（信号中枢） |
|------|---------------|---------------|
| 条件模型 | N个条件并行，全满足→执行 | 触发事件→条件串行过滤→动作 |
| 需求管控 | 端状态一张表 + 触发器条件一张表 | 统一大表三个sheet |
| 动作类型 | 仅Advisor | Advisor + 事件日志 + TTS + 可扩展 |
| 事件描述 | 无 | 每个触发事件有拼接规则生成描述文本 |
| 目标形态 | 静态配置 | workflow画布拖拽（触发事件→条件→动作原子节点） |

## 四个设计理由

1. **触发理由原子化**：Advisor和事件日志都需要"谁在几点做了什么"，触发事件天然提供
2. **算力优化**：并行轮询→串行过滤，减少长时信号监控
3. **复合条件支持**：延时、串行、并发等，不再局限于全部满足
4. **标准化产品化**：配置平台→运营平台，作为火山座舱标准产品给第三方用
