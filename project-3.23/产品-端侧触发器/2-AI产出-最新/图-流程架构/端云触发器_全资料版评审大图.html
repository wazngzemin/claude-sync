一、在原图基础上的细化结论

这部分不是复刻原图，而是回答评审最核心的四个问题：触发器需求是什么、为什么做、怎么做、如何落地到端云和 Advisor / Director / VLM。

原图位置 应该细化成什么 结合资料后的真实需求 评审时必须讲清楚
13 后台 / 规则配置平台 规则注册入口 + 能力查询 + DSL 生成前置 P0不是简单后台表单。它要提供可订阅信号、可执行 Action、场景注册、更新、查询、禁用/删除。资料里已经有 API：/trigger/signals、/trigger/actions、/trigger/scene。 研发要确认 API 字段与动态 Advisor、Planner 条件任务是否对齐；否则任务创建成功但触发器跑不起来。
15 云端触发器 云端 CTA 引擎，不只是透传 P0负责注册、校验、条件 DSL、执行策略、端云动作分类、任务冲突治理。会议明确“所有注册在云端，端侧只执行已校验策略”。 云端触发器可以 callback 云端 Advisor，但端侧触发器不能直接通知 Advisor；端侧物理链路必须走信号库/Context。
16 / 17 Advisor 静态 Advisor + 动态 Advisor 两种消费方 风险上汽 DNA 当前静态 Advisor 可能缺失；动态 Advisor 616 明确依赖条件任务、定时任务、VLM 任务三类订阅。 如果没有静态 Advisor，预设主动服务短期要走 Director 或动态 Advisor，不要在图里画一条不存在的链路。
18 / 19 / 20 / 21 Goal List + Director + Context 的闭环 已确认Director 负责 talk_or_not、talk_content、action_list；Context 是端侧结果、事件日志、偏好、记忆的统一入口。 触发器命中不等于直接执行复杂动作。需要 Director 二次推理结合当前车况判断还要不要说/做。
24 端侧触发器 / VLM 端侧本地履约 + VLM 调度 + 结果写回 P0必须做端侧定时器、弱网履约、规则持久化、条件扫描、VLM互斥/抢占/超时、五态写回。否则端侧触发器只是路由上云，没有独立价值。 会议里质疑点非常明确：如果端侧触发器不能离线闭环，那直接端状态上报云端触发器即可。
26 / 27 / 28 / 29 端侧触达与执行层 待拍板端侧 AI Push、TTS、本地车控、机器人动作都可能是命中后的承接方式，但不同动作安全等级不同。 端侧直执必须有白名单和安全边界；需要语音确认的主动推荐通常上云走 Director。

端侧触发器：本地履约层

  • 规则接收：云端下发已校验规则，本地持久化。
  • 条件扫描：持续状态 + 瞬间事件，不只扫当前态。
  • 信号对齐：多信号几十到几百毫秒延迟，用窗口合并快照。

  • 动作策略：AI Push、本地 TTS、白名单车控、notify 云。
  • 恢复同步：网络恢复后同步执行状态，避免重复播报。

  • 任务类型：VQA、状态监控、风险监控、Always-on。

  • 红线：timeout 不能写“无异常”；failed 不触发主动服务。

三、关键流程补细

这几条是会议里反复出现、研发一定会问的时序。放在主图下方，避免主图被线条塞爆。

流程 A：用户创建条件任务

1. 用户说条件话副驾上车时讲笑话 / 车速超过80提醒我。
2. Director识别判定为非即时任务，进入条件/定时注册。
3. 条件解析标准条件库匹配；不支持则失败回告。
4. 云端注册生成 DSL 和执行策略，写任务中心。
5. 下发端侧端侧持久化，监听变化与防抖。
6. 命中事件端侧触发并写 Context / notify 云端。
7. 二次决策Director 结合当前车况决定说/做。

流程 B：端侧定时弱网履约

1. 用户说定时5分钟后关闭空调。
2. 云端生成策略确认动作、时间、是否可端侧直执。
3. 端侧倒计时展示剩余时间，本地持久化。
4. 弱网/无网不依赖云端倒计时。
5. 到点触发AI Push / TTS / 白名单车控。
6. 状态记录写任务状态和事件日志。
7. 恢复同步网络恢复后同步云端，避免重复执行。

流程 C：VLM 视觉触发

1. Advisor订阅Workflow 等待“用户是否笑了/宝宝是否醒了”。
2. 下发感知目标云端触发器转成 VLM query / state。
3. 端侧调度检查互斥、优先级、采样频率。
4. VLM判定true/false + reason 或五态结果。
5. 阈值穿越false→true 才生成事件。
6. 写Context结果和任务状态可追溯。
7. Advisor推进生成 advice，Director 决定表达/动作。

待拍板项 影响的设计决策 推荐默认方案
P0 上汽没有静态 Advisor 时谁消费预设事件？ 决定原图 16 是否存在、15 命中后走 17/19/20 哪条线。 短期走云端触发器 → Director / 动态 Advisor；图上标注静态 Advisor 为可选/待部署。

P1 任意位置车门打开由谁展开？ 决定 DSL 复杂度和意图解析稳定性。 后台提供虚拟条件，研发展开为多条 OR；不要完全依赖模型临场生成。
