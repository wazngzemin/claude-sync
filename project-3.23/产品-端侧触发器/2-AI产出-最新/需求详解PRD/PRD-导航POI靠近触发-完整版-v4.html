方式适用场景数据来源精度限制
① 导航沿路线距离
（优先）		 导航中 + POI为目的地/途经点 导航引擎每秒native输出的沿路线剩余距离 高 — 沿实际路线计算，高架不误触发 仅导航状态可用；POI必须在路线上
② 高德SDK距离查询
（通用兜底）		 任何场景（导航中 + 非导航 + 任意POI） 调高德SDK接口：传入POI ID/经纬度 → 返回当前距离 + 预计到达时间 中 — 直线距离或粗略路线距离 需端侧定时调用；调用频率需和陈俊龙确认上限

4. 核心难点：为什么不能直接配"距离<500m"

4.1 数字走查：一个真实场景

周一早8:00，用户导航去公司（出发距离15km），产品想做"距公司500m提醒拿咖啡"。

时刻剩余距离如果配"距离<500m"正确行为
8:00~8:2515km → 501m不触发 ✅不触发 ✅
8:25:30首次到500m触发 ✅触发一次 ✅
8:25:31499m触发（第2次）❌不触发 ✅
8:25:32498m触发（第3次）❌不触发 ✅
......一路刷屏到50m ❌安静 ✅
8:30:00≈0m，挂P档还在触发 ❌触发"到达" ✅
问题：从500m开到公司（499、498...50m），每帧都满足"<500m"，会触发几十上百次——用户被"该拿咖啡了"刷屏。所以必须是"从远到近、第一次跨过500m → 触发一次 → 然后闭嘴"——这就是 one_shot。

4.2 解决方案：围栏状态机（端侧运行）

给每个被监控的POI维护一个状态：远 → 已接近 → 已到达 → 已离开。状态机在端侧运行，不在云端。

d ≤ 480m
→

d ≤ 50m 且 P档
→
触发【fence_arrive】
已到达

→
触发【fence_leave】

state = 该POI当前状态           # 初始：远（端侧维护）
d     = 当前到POI距离(米)       # A类=导航沿路线距离 / B类=高德SDK距离查询

if state == 远  and  d <= 入栏阈值(480m):        # 死区下沿
    产出 fence_enter;  state = 已接近              # one_shot：状态翻了就不再触发
    → 上报云端

if state == 已接近  and  d <= 到达阈值(50m)  and  档位变更为P:
    产出 fence_arrive;  state = 已到达             # 组合条件
    → 上报云端

if state in (已接近, 已到达)  and  d > 出栏阈值(520m):
    产出 fence_leave;  state = 已离开              # 距离从小变大 → 离开
    → 上报云端

5. A/B两类Case的区别

看起来都是"快到XX提醒我"，但端侧选取的距离获取方式不同：

A类 · 导航中 + POI是目的地/途经点

  • POI就是当前导航的目的地或途经点
  • 导航引擎已经在实时算到它的沿路线距离
  • 端侧判定用导航沿路线距离（精度高，高架不误触发）
  • 例："离目的地还有1km提醒我拿咖啡"
  • 例："快到途经点帮我找停车位"

B类 · POI非导航目标 或 未导航

  • POI不是当前导航目标（"外婆家"可能不在路线上）
  • 导航引擎没有在算到它的距离
  • 端侧判定用高德SDK距离查询（通用兜底）
  • 不再需要"动态插途经点"——围栏天然支持任意POI
  • 例："快到外婆家提醒我打电话"
  • 例："路过徐浦大桥提醒我右转"
v3 → v4 关键变化：原PRD中B类Case是"卡点"——需要"动态插途经点"（仁杰待确认），不能做→全砍。现在端侧自建围栏天然支持任意POI坐标的距离查询，B类不再是卡点。只要选点拿到经纬度，A类和B类用同一个围栏框架，区别仅在距离获取方式（导航距离 vs SDK查询）。
