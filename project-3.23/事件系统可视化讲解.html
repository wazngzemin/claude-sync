📋 一、核心问题（点击展开）

❌ QQ音乐登录bug
▼
用户手动登录后说"放歌"，Planner还说"你未登录"
🔧 解决方案：
音乐模块发送事件 → {event_type: "user_login", target: "qq_music"}
Planner看到事件 → 知道用户已登录 → 直接播放音乐
❌ 导航重复启动
▼
用户点了"开始导航"，Planner又说"我帮你导航"
🔧 解决方案：
导航模块发送事件 → {event_type: "user_action", action: "start_navigation"}
Planner看到事件 → 知道已启动 → 取消上一轮语音指令
❌ 切歌混乱
▼
用户说"换一首"又点了屏幕，结果连切两次
🔧 解决方案：
音乐模块发送事件 → {event_type: "user_action", action: "switch_song"}
Planner看到事件 → 知道已手动切歌 → 不再执行切歌指令
❌ 音乐切换感知
▼
用户问"这首歌谁唱的"，但系统刚切了歌
🔧 解决方案：
音乐模块发送事件 → {event_type: "song_changed", song: "凤凰传奇-月亮之上"}
Planner看到事件 → 知道当前歌曲 → 回答正确的歌手信息

🔄 二、三种处理类型（点击展开）

① Query关联事件

📝 存入上下文

事件存入事件队列，用户提问时检索注入

▼

处理逻辑：

  • 事件发生 → 存入事件存储（按时序、按session）
  • 用户提问 → 检索相关事件（关键词/向量）
  • 插入对话历史（作为event role）
  • Planner推理时参考事件信息

典型场景：

手动调节空调后，问"空调多少度"能回答准确值；手动切歌后，问"现在放什么"能回答正确的歌

✓ 不主动说话，只作为上下文信息

② 主动服务事件

💬 立即触发对话

作为独立请求，立即触发AI互动

▼

处理逻辑：

  • 事件发生 → 立即创建新的Planner请求
  • 携带事件信息作为输入
  • Planner决定是否说话 + 说什么
  • 输出语音/执行动作

典型场景：

主驾上车时主动打招呼"王先生你好"；检测到副驾入睡时说"副驾睡了，我会保持安静"；天气预警时说"外面可能结冰，开慢点"

✓ 主动说话，提供服务

③ 目标关联事件

🎯 触发目标闭环

事件发生时触发目标/提醒

▼

处理逻辑：

  • 用户设定目标（如"下车提醒带手机"）
  • Planner创建目标 + 订阅相关事件
  • 事件发生（如"副驾解安全带+车减速"）
  • 触发目标 → 执行提醒

典型场景：

乘员下车时触发"下车提醒任务"；后排入睡时创建"安静目标"；后排醒来时终止"安静目标"

✓ 让持续跟踪目标真正"活"起来

🚗 端侧负责的事件

  • 用户操作：点屏幕、手动登录、手动切歌、调节空调
  • 系统事件：音乐自动切歌、卡片弹出、任务状态变化
  • VLM识别：人对话、行为识别、表情识别、手势识别
  • 传感器数据：车门开关、座椅压力、温度变化、光照变化

☁️ 云端负责的处理

  • 事件存储：按时序存储、按session分组、保留时长管理
  • 检索模块：根据query检索相关事件（关键词/向量）
  • 总结模块：生成情景注意力描述（当前状态总结）
  • 筛选模块：选出必插事件（最近1条关键事件）

🤖 Planner接收的信息

  • 最近1条关键事件：插在query前（最强信号）
  • 情景注意力总结：SP底部变量（背景理解）
  • 完整事件流：SP变量（完整记录，参考用）
  • 推理结果：知道"发生了什么" → 做出正确决策

📝 六、核心总结

🎯 核心逻辑（点击展开/收起）

事件系统 = 让Planner知道"发生了什么"

现在：Planner只知道"用户说了什么"
加入事件后：Planner知道"用户说了什么"+"用户做了什么"+"系统发生了什么"

结果：Planner不再"瞎子"，能给出更准确、更贴合情境的回应

三种处理方式：
① Query关联：存入上下文，用户提问时参考（不主动说话）
② 主动服务：立即触发对话（主动说话）
③ 目标关联：触发目标闭环（让目标"活"起来）
