# 数据治理 > 模型能力：模型是最后一层接口

## 命题

> 在一个成熟的 LLM 应用里，**模型能力是"最后一层接口"，系统质量的主导变量是数据治理（定义/元数据/文档/检索/执行纪律）。** 当模型已足够强，再投入"更好的模型/更好的技术"边际收益递减；真正决定上限与稳定性的是数据层与组织执行层。

来源：黄钊 hanniman 对 Anthropic《How Anthropic enables self-service data analytics with Claude》的解读。详见摘要 [[anthropic-data-governance-ai-first]]。

## 论证骨架

### 1. 横跳曲线 = 数据治理的指纹

```
无 Skills ──── 21%
有 Skills ──── 95%      ← +74pt 全部来自领域知识/操作规范，不是换模型
停止维护 ───── 65%（1月）  ← 模型没变，纯靠"不维护"就掉 30pt
```

若瓶颈在模型，准确率应随模型增强单调上升，不可能 21→95→65 横跳。横跳证明：**系统表现由数据/治理层主导，模型只是接口。**

### 2. 难度错位：技术难度 ≈ 0，现实难度指数级

| | 技术问题 | 数据问题 |
|--|----------|----------|
| 形态 | 知道怎么做 → 投资源 → 能做出来 | 知道怎么做 ≠ 能长期做到 |
| 难点 | 一次性攻克 | 持续维护 / 执行纪律 |
| 例子 | "Claude 写 SQL" | "什么是 Revenue / Active User / 权威表 / 当前有效指标" |

类比：**治理河流污染**——难的不是找到污染源，而是持续监管/执法/维护，否则迅速反弹。

### 3. 三类系统性错误（治理失败的表现）

- **实体歧义 Entity Ambiguity**：一个概念对应多表多算法，定义不唯一。
- **数据陈旧 Staleness**：业务/表结构变了，文档/逻辑没跟上。
- **检索失败 Retrieval Failure**：答案存在但 Agent 没找到。

### 4. "人是污染源"的判据

- **数据精度高的场景**：人被充分数据化、规则在人之上（外卖骑手/网约车司机/主播）。
- **注定腐烂的场景**：人是独立组织体系、数据被后置规则治理。**足够的活性 = 足够的数据混乱**。
- 推论：越是高自主性、高价值的人（天才科学家/程序员），越不愿循规蹈矩贡献结构化数据 → 治理与活性天然两难。

### 5. 出路：AI First 重新设计业务（而非先产数据再治理）

- 错误路径：先按人的旧流程产数据 → 事后做 3 年数据治理（治理难度不随时间下降，世界早变了）。
- 正确路径：**一开头就问"这活能否完全由 AI 干、人辅助 AI"**。达成后数据本就按规则产生，治理成本指数级下降。
- 出自《无人公司》/《缝隙理论》——AI First 是"生产关系上 AI 优先"，不是"工具上用 AI"。

## 钉入 Planner / 场景知识（业务应用）

> 这条命题对 [[planner]] 与场景知识 Skill 知识库高度同构，是 AI PM 可直接复用的论据与方法。

| 文章概念 | 本项目对应物 | 行动启示 |
|----------|--------------|----------|
| 21%→95%→65% Skill 曲线 | 场景知识 Skill「质量差/未实车验证/写了车厂没有的能力」 | Skill 治理纪律是准确率主因；需"实车验证→回灌→版本锁版"的持续运维，不是一次性生产 |
| 实体歧义 | SP 工具定义 / Skill 描述不清，误导调用路径 | 见 [[planner-tool-simplification]]、[[planner-sp-structure]]；统一工具/能力的"权威定义" |
| 数据陈旧 | 触发条件清单从赛力斯老表迁移、口径漂移、信号库版本不锁 | 见 [[trigger-condition-taxonomy]]；建立"当前有效版本"机制（信号库 v2.5 锁版+回灌） |
| 检索失败 | 情景 RAG / 动态知识注入检索不到正确示例 | 见 [[scenario-rag-arbitration]]、[[query-routing]]；检索机制是准确率瓶颈之一 |
| "没有执行力的治理会迅速失效" | Skill 库无持续运维即腐烂 | 在评测方法论里把"治理纪律"列为一等公民 |
| AI First 重设计 | 触发条件清单/场景库工程化 | 别拿人工旧表迁移再清洗；让数据从源头按 Planner/触发器可消费结构产生 |

**对外论据用法**：在 [[planner-eval-methodology]]「场景库工程化/可落地评分卡」处，可引用 21/95/65 曲线，论证"评测分数的主导变量是 Skill/场景知识的治理质量，而非模型能力"——这正是过程归因（而非只看结果分）的底层依据。
