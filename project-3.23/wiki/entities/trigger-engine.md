# 触发器引擎（Trigger Engine）

## 定位

触发器是 Driver Agent 2.0 的"条件任务执行底座"——不直接面对用户，而是在后台持续监听车端信号，当预设条件满足时自动触发动作（车控/TTS/通知 Advisor）。产品负责人：王泽民（你）。研发负责人：刘洋。

7.10 评审后，触发器被纳入韩杰主导的"信号中枢"（[[signal-hub]]）统一框架，原独立条件表拆分为触发事件+条件两表。

## 在系统中的位置

### 旧架构（至7.10前）

```
用户语音 → ASR → 意图模型（解析条件话术）→ 触发器协议
                                                ↓
                              触发器引擎（云端注册 + 端侧判断触发）
                                                ↓
                              条件满足 → 执行动作 / 通知 Advisor / Planner 兜底
```

### 新架构（7.10 信号中枢）

```
端侧信号上报 → 信号中枢（三表统管）
                  ↓
              状态突变检测 → 生成触发事件
                              ↓
                     条件判断（串行/并行过滤）
                              ↓
                     执行动作（可挂多类型）
                     ├── 触发 Advisor（现有）
                     ├── 生成事件日志 → Context → Director（新增）
                     ├── TTS 播报（现有）
                     └── [未来] 规则引擎 / 条件任务 / 任务大师
```

关键变化：从"N个条件并行满足→执行"变为"触发事件→条件串行过滤→动作"，支持复合条件组合。

## 核心能力

| 能力 | 旧状态 | 7.10后目标 |
|------|--------|-----------|
| 条件判断 | 每帧全量扫描当前态 | 触发事件（突变检测）+ 条件（当前态过滤） |
| 条件模型 | N条件并行全满足 | 串行逐步过滤 + 串行/并行/延时复合 |
| 动作类型 | 仅Advisor | Advisor + 事件日志 + TTS + 可扩展 |
| 协议结构 | 套娃嵌套 | 拍平，各门独立一级 + 任意位置 |
| 时间条件 | 不支持 | 端侧定时器（定时/日期/节日） |
| 距离条件 | 不支持 | 端侧实时定位 < N米触发 |
| 地理围栏 | 不支持 | 圆心+半径+进出判定+防抖 |
| 端云分工 | 全绕云端 Planner | 云端注册仲裁 + 端侧判断执行（端侧scope待产品定义） |
| 事件描述 | 无 | 每个触发事件含拼接规则，生成"谁在几点做了什么" |
| 配置形态 | 静态配置平台 | workflow画布拖拽（触发事件/条件/动作原子节点） |

## 核心协议：五段式

条件在后台的标准表达格式：

```
条件类型 → 信号 → 字段 → 运算符 → 值
例：温度   → SIG-T-012 → temp → > → 26
```

7.10后触发事件新增：事件描述拼接规则（如"档位+切入/切出+D档"→"档位切入D档"）

## 端侧触发器定位争议（7.10未决）

赵衍在7.10评审中提出三个产品必须回答的问题：
1. 端侧触发器要不要有？（离线=客户诉求）
2. 预置多少场景脚本？
3. 运营脚本是否下发端侧执行？

韩杰回应：产品只提能力需求+性能要求，不提端侧/云侧分工。但承认"离线也能用"是产品需求。730 scope全部云端，端侧后续拉会。
