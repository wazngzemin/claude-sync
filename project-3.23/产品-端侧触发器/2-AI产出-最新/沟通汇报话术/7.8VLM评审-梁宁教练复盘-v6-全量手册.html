第二章 · 根因诊断（深度版）

闭环怎么转的：

你不带目标进会(D) → 不知道该守什么 → 定位被攻击时不知道怎么接(A) → 只能在技术层给弱回应 → 不自觉缩小scope来降低攻击面(B) → 对方用你自己说的话论证"不需要" → 你接受删除时什么都没锁(C) → 输了 → 回去用AI写更多文档来获得安全感 → 下次评审依然没有目标(D) → 循环

从哪里切入效率最高：D。如果你带了目标进会（"今天要拿到0818 scope确认"），你就知道该守什么（0818）。知道该守什么，定位被攻击时你就有锚点（"0804我同意，但0818另评"）。有锚点就不会自缩scope。不自缩就不会被论证"不需要"。即使被否，你也知道该锁什么条件（你的目标就是你的撤退锁定条件）。

所以练习的优先级是D → A → B → C，不是A → B → C → D。v4/v5把A（攻击面识别）排在第一位是错的。应该先练D（目标设定），因为D是闭环的驱动轮。

第4层：你从来没见过别人在被否之后锁条件。你在评审桌上观察到的模式是：有人提方案 → 有人反对 → 要么坚持要么放弃。你没有观察到"接受但提条件"这个中间路径的示范。

问题D "没带目标进会" 是怎么一步步形成的

第0层：你走进7.8这场会脑子里没有"要拿走什么"。

第1层：你把评审的目标定义为"讲清楚方案"，不是"拿到决策"。

第2层：你的工作流里没有"会前设定目标"这个环节。你的会前准备是：检查PPT/PRD有没有问题 → 过一遍要讲的内容 → 进会议室。你的准备流程里没有"写下要拿走的决策"这一步。

第3层：你把评审当成"被安排的事"，不是"你发起的事"。被安排的人来执行任务（"讲清楚方案"就是你的任务）。发起的人来拿结果（"定下0818做不做"是你的结果）。

第4层：你一直在做的是"响应需求"而不是"定义需求"。别人说"来评审一下"，你就准备去讲。你从来没有主动发起过一场评审说"这个问题今天必须定"。

第5层（触底）：这是执行型PM和高级PM的核心分界线。执行型PM的工作方式是：有人给我任务 → 我做好 → 交付。高级PM的工作方式是：我定义什么需要被做 → 拉人讨论 → 拿到决策 → 推进执行。你在第一种模式里待太久了，第二种模式你没有练过。

场景1：汪帅打定位（3条分支路径）

汪帅开枪："0818触发器我觉得也没必要。动态任务切换可以在SDK内部做。"

你的第一句（固定，不分支）：
"帅哥，0804你说的对，我同意了也删了。0818场景不一样——进停车场关舱外、行驶中切SP、驻车降频。这些不是开关，是条件触发。SDK内部做也可以，但我有个问题：0904再加新场景的时候，是每次都改SDK代码还是有个配置化的入口？"

分支点：汪帅会怎么回？三种可能：

分支A：他说"SDK内部就能配置化"

汪帅："SDK内部也可以做配置化啊，不一定要在外面加一个模块。"

你接："可以。那我们定义一下：SDK内部的配置化入口，产品定义谁来做？场景规则谁来配？如果还是我来定义规则、你们来实现，那它叫'触发器'还是叫'SDK内部配置模块'对我来说不重要——重要的是这个入口存在、有人管。名字可以改，产品owner不能没有。"

这条路径的策略：不争名字争实质。如果汪帅愿意在SDK内部做配置化入口，你的核心利益（有一个统一入口、你是产品owner）其实保住了。名字让步，实质不让。

分支B：他说"这几个场景不多，不值得"

汪帅："就三个场景，不至于做一个配置化的东西。"

你接："三个是现在的scope。但我翻了一下需求池，0904还有夜间模式切换、低电量降频、高速巡航增加感知频率——加起来已经快10个了。10个场景的触发逻辑还散着做？如果今天我们定不下来数量，那就换个判断标准：在第几个场景的时候我们要从散装切到统一入口？帅哥你觉得是5个？还是不管多少个都散着做？"

这条路径的策略：用数量反驳"不多"。如果你有需求池的数据，用数据说话。如果没有精确数据，现场列几个场景（夜间模式/低电量/高速巡航）——这些不需要100%确定，只需要让"未来会有更多"变得可信。

分支C：他说"那0904到时候再看"

汪帅："行，那到0904再说吧，0818先不做。"

你接："推到0904可以。但今天定一条：0904的评审议程里包含触发逻辑的统一入口方案评审。写进纪要。荣姐这条OK吗？"

这条路径的策略：他推你就锁。"0904再说"如果只是口头说，到了0904没人记得。写进纪要=你有了一张入场券。

场景3：有人质疑你的专业性

韩杰（或任何人）："泽民，你这个PRD里的字段定义跟端上实际的不一样，你有没有跟端上对过？"

你接："具体哪个字段？你告诉我，我现在记下来，会后跟端上确认。今天先把链路和scope定了，字段细节我补。"

为什么不解释/不辩解：字段对不对是L1问题（缺一个信息），不需要解释"我为什么不知道"。你承认不知道、承诺补上、把讨论拉回主线。评审桌上最贵的资源是时间，你不要让一个字段问题占掉十分钟。

场景4：有人挂名给你

张荣："这个问题先挂泽民这边跟一下。"

你接："可以跟。两件事先确认：①这个问题的上游信息谁给我、什么时候给？②deadline是哪天？这两个我清楚了就可以开始。"

如果对方说"你自己去了解"：

你接："了解谁？给我一个对接人。我可以主动找，但得知道找谁。"

公理4：挂名=风险转移。接之前必须索取对价。你的对价不是钱，是信息和时间。没有信息来源和deadline的挂名=无限责任，不接。

场景5：你的方案被质疑有技术漏洞

任何人："你的方案里这个状态恢复没有考虑AI Box崩溃的情况。"

"考虑了。崩溃后SDK重连会同步最新状态，丁彬上次确认过这个链路。"——一句话堵住。
如果你确实没考虑到：
"这个点我确实没覆盖到。你提的对——崩溃恢复需要一个机制。这个我记下来，补进PRD。今天先把主链路定了，异常处理我后面补一个方案再拉你们对。"

两种情况的区别在于会前预审。你如果用v5的"预审三刀"练法，大部分技术漏洞你会前就能堵。堵过了的你一句话就能回应，没堵过的也不用慌——承认不知道、标记补上、把会拉回主线。

场景6：会开到一半你突然不知道怎么接

任何时候你被问住了、脑子一片空白——不用慌。这是正常的。你需要的是一个"保命句式"。

三个保命句式（背下来，任何时候都能用）：

① "这个问题我没有现成的答案。但我判断它不影响今天要定的结论，我标记下来会后补。今天先继续？"
→ 承认不知道+判断影响范围+控制节奏。你在说"我不知道细节但我知道全局"

② "帅哥你的论点我听到了。我需要五分钟想一下再回你——荣姐我们先过下一个话题？"
→ 你在买时间但不是沉默。你告知了全场你在思考，不是放弃了。五分钟后如果想出来了就回来说

③ "这个点我现在不确定。但我确定的是——"然后转到你确定的一个论点上
→ 承认不确定的同时锚定一个你确定的事。你不是在逃避问题，是在选择战场

修正后的优先级
