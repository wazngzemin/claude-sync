为什么车企越造越亏，钱却被电池和华为赚走了

赛力斯上半年由盈转亏，亏 15 到 18 亿，去年同期还净赚 29 亿。这篇文章讲的不只是车。这一版我读了你的 7.10、7.15 会议纪要，把它照进了你触发器现在的真实处境，里面有你自己说过的话。

分诊结论
这篇 → 做成「解析文档」，不做工具。

它是观点/框架型，给的是一副看问题的透镜，不是一套你会反复运行的评分或诊断流程。价值在于想通它、接到你的工作上。第 05 节的「价值捕获四问」放这儿当随手能套的小清单，不单独做交互工具。

以后每篇的分诊标准：给「一副看问题的透镜/一套认知」→ 解析；给「一个你会反复填、反复算、反复判的流程」→ 工具。拿不准先解析。
01

一句话讲透

你有多累，跟你能赚到多少，是两码事。钱不会因为谁最辛苦就流向谁，它流向那些别人绕不开、而且越做越强的位置。赛力斯这些整车厂拼命造车却由盈转亏，单车毛利从 2.3 万掉到 1.4 万，不是被对手打的，是它站的那个位置（总装、卖车）本来就存不住钱。原材料涨价只是表面，真问题是：没掌握关键生态位的整车厂，在给生态主导者打工。

02

拆开看：它其实只做了三件事

第一件，换个角度看问题

看到亏损，大部分人第一反应是「原材料涨价」「竞争太激烈」。作者说这是把症状当病根。你摸到的那个「卷」和「涨价」，只是表面。真问题在你站的位置。证据挺硬：同样的市场里，宁德和华为在闷声赚钱。一样的环境，活法完全不同，说明胜负手不在卷不卷，在你站哪一层。

这一步为什么不容易想通？因为它反直觉。我们心里默认「我很辛苦」就等于「我在创造价值」。文章逼你把这两件事分开。

第二件，讲清楚什么位置存得住钱

位置例子为什么存得住 / 存不住钱
基础设施层宁德 · 电池你是绕不过的关卡，谁造车都得过你这道门，全行业围着你转，正向飞轮
能力层华为 · 智驾座舱你卖的是别人做不出、还决定人家能不能进高端市场的东西。问界靠深绑华为才上了高端牌桌
组装层车企 · 造一辆卖一辆卖完就结束，谁都能组装，用户开走就断联。没锁定、没复利、可替代，被两头抽

「造一辆车、卖一辆车」六个字是全篇病灶：算一辆是一辆（线性）、交易完就断（一次性）、你的和他的差不多（可替代）。三样一齐，就没有讨价还价的本钱。

第三件，给三条出路

既然组装层存不住钱，那就换位置。三条路放到第 05 节的方法论里一起讲。

03

四个能摘走、装到你自己身上的点

点 01先分清「现象层」和「结构层」

很多人一辈子在现象层治标：卷、加班、焦虑。会分层的人先问一句「这是结构问题吗」。

怎么用：任何「我很努力但被架空」的场景，先画分层图，别急着在现象层加药。
点 02好位置不是占到就完，是会自己越滚越强

占到好位置只是开始。它有一个自我加强的循环转着（宁德成主流标准 → 全链配套 → 规模涨 → 更强），不转它就会被掀掉。这就是飞轮。

怎么用：看你做的东西，有没有「用得越多、你越强」的循环。没有，你就是在原地出力。
点 03判断一个能力值不值钱，看是不是「入场必需」

「没有华为，问界连高端牌桌都上不了」。判据不是「这功能做得好」，是「没有它，对方根本上不了台」。

怎么用：问「没有我这个东西，对方是体验差一点，还是压根上不了台？」后者才有定价权。第 06 节这一问会正面照到你的触发器。
点 04价值重心正从「造东西」挪到「占关系」

终极赢家是出行服务商，车只是交付终端。Robotaxi 一旦跑通，现在这点单车毛利之争都成了旧时代的问题。谁占住用户持续关系，谁把一次性买卖变成长期收入。

怎么用：看任何产品在做「一锤子买卖」还是「攒一段持续关系」。后者天花板高得多。你的触发器 + 持续任务，正是座舱里往这条路爬的引擎。
04

连问五个为什么，钻到最底下那块铁

  • 为什么1车企为什么亏？
    被上游电池和上游智驾两头抽走了利润。
  • 为什么2为什么会被抽？
    因为组装层没有讨价还价的本钱。
  • 为什么3为什么没本钱？
    因为可替代（谁都能组装），又没锁定（卖完就断）。
  • 为什么4为什么可替代、没锁定就没价值？
    因为能拿多少价值，看的是「对方离开你要付多大代价」。可替代 = 对方随时换掉你，代价约等于零。
  • 为什么5为什么「更努力」救不了它？
    努力只能把当前这层的活干得更好，改变不了「对方离不离得开你」。前者努力能动，后者位置决定。
钻到底的那块铁你能拿到多少价值，取决于别人离开你要付多大代价，不取决于你干了多少活。记住这一句，第 06 节全靠它。
05

压成一套方法：价值捕获四问

对任何东西都能反复跑。对象可以是一个产品、你负责的一个模块、或者你自己。

1
别人绕得过你吗？绕过你有没有损失。测「关卡」。
2
你在攒一个越来越强的东西吗？每次交付是不是在积累资产。测「飞轮 / 复利」。
3
你是入场必需，还是锦上添花？没有你对方上不了台，还是只差一点。测「差异化」。
4
你的位置会漂移吗？下游能自己做出来平替你吗。测「护城河还能撑多久」。
制造驱动 · 极致性价比比亚迪
靠零部件复用、规模、供应链、产线摊销把成本压到别人做不到。前提有规模。风险被追平。稳，天花板低。
技术驱动 · 数据飞轮华为 · 小鹏
押智驾，关键是统一数据系统 + 软件迭代 + 智驾闭环，攒出别人追不上的差异，才有定价权。前提飞轮真转得动。风险同质化。
服务驱动 · 出行服务商Robotaxi 终局
从卖车变卖出行，车成交付终端。前提L4 落地。风险最像愿景。最难，也最值钱。
06

把这把尺子，架到你自己身上（据 7.10 / 7.15 纪要）

前面都是车。这一节讲你。我读了你 7.10 和 7.15 两场纪要，发现你部门里正在发生的，就是这篇文章的活体版。角色对得上：

文章里的角色你部门里对应谁凭什么这么对（据纪要）
宁德（基础设施 · 卡关卡）韩杰的信号 / 端状态大表7.10 他把你触发器的条件表并进了他的端状态大表。所有条件都得从他的信号来，他成了绕不过的卡口。
华为（能力层 · 承重墙）于庆磊的任务（task）系统7.15 他把 task 定成「顶层唯一抽象」，还说以后「不会再有单独的动态 advisor 这个概念了，它作为任务的一个子集」。所有东西都被他的任务系统包住。
车企（组装层 · 被两头夹）你的触发器上游条件定义归了韩杰，下游被于庆磊的任务系统「复用」。你在中间做注册、做匹配的活，可两端定义价值的位置都不在你手里。

第一次：归属被端走（7.10）

韩杰把你触发器的条件表，融进了他的信号大表。这是把你的定义，收编进他的卡口。他的原话：

韩杰 · 7.10 纪要原话「我们去把原来的触发事件条件，就是原来的那个表，融合到了整个端状态的大表里面…把原来的触发器的相应的条件的那个大表也融入进来，因为它们相关性很高。」

翻译成生态位：他让「信号 / 端状态表」成了不可绕过的基础设施，你的触发器条件表从一张独立的表，变成了他大表里的一部分。你的归属，被并进了别人的卡口。

第二次，更疼：能力层递到你手上，你自己推了出去（7.15）

7.15 讨论「后排宝宝哭了告诉我」这种 VLM 任务时，谁来做「把 VLM 上报的描述，匹配成任务条件」这个模型推理判断，成了焦点。于庆磊和刘杨明确把它划给触发器：

于庆磊 · 7.15「这肯定是触发器模块，我们是希望触发器有这个能力。从职责划分上，我觉得也更是触发器的能力。」
刘杨 · 7.15「触发器最开始其实是希望支持推理能力的，当时没有场景，现在有了。」

这是什么？这是有人把「能力层」亲手递到你手上。「用模型判断 VLM 描述有没有达成条件」这件事，是所有视觉类、语义类任务都绕不过的一道判断。谁做这道判断，谁就是那个不可绕过的关卡。可你当时的反应是：

王泽民（你）· 7.15「视觉条件，哎？…如果说做语义解析的话，不是在触发器这块做吧？我的触发器只需要拿枚举值啊。」
而且那一刻「什么？我刚上来，没太听。」

别人把能力层递到你手上，你的本能是推掉它、退回去当一根「只拿枚举值」的管道。

「只拿枚举值」是什么？是组装层。枚举值谁都能拿，你就是根管道，可替代。「做模型推理判断」是什么？是能力层，是那道别人绕不过的判断。你亲手把自己往组装层推了一把，而且当时没在听。这跟赛力斯犯的是同一个错，只不过赛力斯是被动被夹，你是主动往下站。

07

你该怎么做，顺便把 PM 能力抬一档

你说自己偏初级、想提升。这篇给的抬升路径，正好压在你这两场纪要的伤口上。

初级和高级 PM 的差，在这两场会里看得清清楚楚

在做什么对应纪要里的动作
初级执行已定义的活：拿枚举值、做注册、做匹配你 7.15「我的触发器只需要拿枚举值」= 主动选组装层
高级定义还没定义的判断和边界：谁触发、条件语义谁解析、边界在哪于庆磊 7.15 把 task 定成「顶层唯一抽象」= 定义了整个抽象，别人都得接他

于庆磊为什么强？不是他代码写得多，是他定义了「任务」这个顶层抽象，让所有人都得在他的框架里说话。这就是能力层。你要学的就是这一手：别只接活，去定义那个别人绕不过的东西。

这周就动：把 7.15 递给你的那道能力，接住并焊死

先认下来，别再往外推。7.15 于庆磊、刘杨已经把「触发器做模型推理判断（VLM 描述 → 条件匹配）」划给你了。你上次的反应是缩。这次主动认领，而且往大了认：不只是宝宝哭了这一个，是所有「VLM / 语义描述 → 条件是否达成」的判断，都走触发器这道。
出一版定义草案，把边界画死。触发器的推理判断层：输入是什么（VLM caption / 语义事件）、输出是什么（条件达成 true/false + reason）、和韩杰的信号表怎么分工（他给原子信号，语义匹配归你）、和于庆磊的 task 系统怎么对接。给「我认为应该这么定」，不是「这该怎么定」。
可以这么说「VLM 和语义类条件的匹配判断，7.15 已经定在触发器了。我出一版这层的定义：输入输出、和端信号表的分工、和 task 的对接。这道判断出问题我来背，那它的定义和边界挂我这。」
顺手把 7.10 被端走的那块，争一点回来。韩杰把条件表并进了他的信号大表，但「原子信号」和「语义条件匹配」是两回事。原子信号归他没问题，语义这层的定义权，你据理拿回来，别整张让掉。

这三步补的正好是你最贵的短板：定义权。而且这次不用你去抢，是有人递到手上，你只要不再往外推。

再给你一个天天能用的 PM 判断习惯

评估你手上任何一个需求或模块，先问它在价值捕获的哪一层：

拿枚举值、做注册、执行已定义的活组装层该做，但别拿它当全部，更别用它逃避「定义」这件难事
定义一道别人绕不过的判断 / 一个顶层抽象能力层高杠杆，优先抢（于庆磊就干这个）
让系统占住用户的持续关系服务层天花板，你的触发器 + 持续任务在这

你每接一个活，先问一句「这是让我更不可替代，还是让我更像根管道」。问多了，你在评审桌上的位置会变。这就是初级和高级的分水岭。
