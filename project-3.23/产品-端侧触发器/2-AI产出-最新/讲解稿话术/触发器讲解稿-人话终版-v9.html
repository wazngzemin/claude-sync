二、左下输入区：① 用户语音 → ② ASR → ③ 端信号源

2.1 ① 用户语音 query

① 用户语音 query（蓝色·x=40, y=860）

是什么：用户对着车机说的那句话。图上显示它携带三个附加信息：speaker_name（谁说的，比如"主驾"）、position（哪个座位）、timestamp（什么时候说的）。

为什么要带这三个字段？因为车里可能同时坐着好几个人。"帮我调高温度"——这个"我"是谁？是主驾还是副驾？position字段解决这个歧义。

① → ② ASR，通过 L1
主驾说"十分钟后关座椅加热" → ①收到的数据是：{speaker_name: "主驾", position: "driver", timestamp: "14:00:03", text: "十分钟后关座椅加热"}

2.2 ② ASR 语音转文字（端侧）

② ASR（蓝色·x=40, y=930）

是什么：ASR = Automatic Speech Recognition，自动语音识别。把用户的语音变成文字。跑在端侧 AIBox 上——不需要联网就能把语音转成文字。

① 用户语音，通过 L1
⑨ 云端拒识分流，通过 L2（端→云，橙粗线）
用户语音 "帮我看看后排宝宝的毯子掉没掉" → ②ASR 输出文字 "帮我看看后排宝宝的毯子掉没掉" → 通过 L2 上传云端。

③ 端信号源（橙色·x=40, y=1060）

三、端侧基础链路：④ 信号库 → ⑤ 离线 fc → ⑥ 命中? → ⑦ 车控 → ⑧ 生效

3.1 ④ 端侧信号库

④ 端侧信号库（绿色·x=300, y=1060）

是什么：车上本地的一个标准化数据仓库。所有传感器的原始数据都先写到这里，格式统一为标准快照。

每条信号包含的字段：

  • key — 信号名（如 seat_heat_fl、gear、speed）
  • value — 当前值
  • source — 谁产生的（哪个传感器/VLM/定时器）
  • timestamp — 写入时间
  • ttl — 过期时间（有些信号会过期，比如VLM的"毯子在地上"过了5秒就该刷新）
  • 编号 — 标准编号如 SIG-T-012

还维护任务状态机：pending → running → success/timeout/failed。

v6 新增标注：图上④节点下面多了一个青色小条："任何触发结果都上传至此(v6)"——意思是不管触发源是谁（用户语音产生的、系统预设的、SLS触发的），只要有触发结果，全部先写进④信号库，再上传云端Context。没有例外。
③ 端信号源，通过 L8。VLM端模推理结果也回写到这里，通过 L16
⑫ 云端信号库（端→云同步），通过 L9。㉔端侧触发器读取信号，通过 L15。CDC底软也直读④
VLM推理输出 "后排左座有毯子在地上" → 写入④：{key: "vlm_rear_left_object", value: "毯子在地上", source: "VLM", timestamp: "14:05:10", ttl: 5} → 5秒后下一轮推理会刷新这条。

3.2 ⑤ 离线 fc + 白名单

⑤ 离线 fc + 白名单（绿色·x=580, y=860）

是什么：fc = function call（函数调用）。这是端侧的一张快速指令映射表——用户说的话如果能直接匹配到一条确定的车控指令，就不用上云走大模型，端侧直接执行。

为什么存在？省时间。"打开车窗"这种指令，不需要LLM推理，直接映射到 window_open() 函数就行。白名单里列着能这么干的所有指令。

② ASR 的输出先上云到⑨做分流，⑨判定是"车控"后通过 L3 回端侧到⑤
⑥ 命中判断

3.3 ⑥ 命中?（菱形决策点）

⑥ 命中?（橙色菱形·x=580, y=940）

是什么：一个分支判断。问的是：这句话在⑤的白名单里有没有对应的函数？

  • 是（命中）→ 端执行：直接走 ⑦ 车控执行 → ⑧ 生效
  • 否（没命中）→ 等云：等云端⑲Director的决策结果下来再执行
"打开主驾车窗" → ⑥命中 → ⑦直接发CAN指令开窗 → ⑧用户看到窗户开了+屏幕弹卡片"已打开主驾车窗"。
"帮我规划去最近加油站的路线" → ⑥没命中 → 等云端⑳Director推理完下发动作。

3.4 ⑦ 原子车控执行（端）+ ⑧ 车控生效+卡片→用户

⑦ 原子车控执行（红色）+ ⑧ 车控生效（蓝色）

⑦ 是什么：端侧直接发 CAN 指令给车辆 ECU 执行动作（开窗、调温、开灯等）。叫"原子"是因为每个动作不可再分。

⑧ 是什么：车控执行完了，用户感知到的结果——窗户真的开了 + 屏幕弹一张卡片告诉用户"已执行"。

四、CDC 底软 + 任务中心 + 通用视觉通道

4.1 赛力斯 CDC 底软 · 固定闭环区

CDC 底软（粉色·x=580, y=1180）

是什么：赛力斯自己的固定闭环系统——直接读④信号库、直接执行车控/TTS，完全不经过触发器链路。

里面有什么：

  • 起步安全(P0) — 所有车标配，纯信号判定（档位+车速+门状态→判断是否安全起步）
  • 上下车安全(P0) — 固定规则，不需要动态配置
  • 睡眠提醒 — 系统设置里的一个button，常驻功能
  • 无感迎宾 — 系统feature，不是场景引擎
  • 充电大师 — 赛力斯业务逻辑，直读信号执行
  • 哨兵模式 — 赛力斯端侧闭环
佳杰区分标准："无感迎宾"=系统功能（设置button）vs "靠近公司推荐咖啡"=场景引擎（有生命周期）→后者归触发器。
仁杰判定法：常驻/固定/标配/key-value/整个车生命周期不变 → CDC。

15-18 SP拼接引擎 详解

这是6.30定版最核心的新增能力。之前⑮只做"条件判断+事件push"，6.30之后⑮还要做"SP拼接+管理+删除"。

SP 是什么？SP = System Prompt。VLM端模每5秒推理一次，它推理时读的指令就是SP。SP告诉VLM"这一轮你要看什么"。

怎么拼？

  1. Advisor给⑮人话需求（不是SP！）——比如"毯子掉没掉"
  2. ⑮把人话需求工程拼接成标准SP格式
  3. 每个SP有对应的KV槽位——SP1对应answer1，SP2对应answer2，VLM推理时每个任务的结果写到各自的answer位
  4. 如果同时有多个动态任务，合并到同一个SP——不是每个任务一个SP循环，而是一个循环里合并所有任务的SP
  5. 合并后的完整SP = AlwaysOn基础SP + 动态SP1 + 动态SP2 + ...
  6. 任务完成后，Advisor通知⑮删掉对应的动态SP → AlwaysOn还原为基础SP
完整示例：
1. 用户说"宝宝的毯子掉了告诉我" → Director创建动态Advisor → Advisor给⑮人话需求："毯子掉没掉"
2. ⑮ 15-18 拼接：
原始 AlwaysOn SP = "prompt: aaaa, bbbb"（基础感知任务）
+ 动态SP1 = "cccc"（观察毯子状态）
= 合并SP = "prompt: aaaa, bbbb, cccc"
KV映射：answer1←aaaa结果, answer2←bbbb结果, answer3←cccc结果
3. 经15-12 + L14下发给㉔ → ㉔下发VLM → VLM每5秒推理一次，返回 {answer1:"...", answer2:"...", answer3:"毯子在地上"}
4. answer3 写入④信号库 → 同步⑫ → 15-13检测穿越 → push ⑰
5. 任务完成 → ⑰通知⑮ → 15-18删掉"cccc" → 还原为"prompt: aaaa, bbbb"
生命周期完整回路：Advisor创建 → ⑮拼SP → ㉔下发 → VLM推理 → ④回写 → ⑫ → ⑮穿越检测 → push⑰ → ⑰判结束 → 通知⑮删SP → 还原

十一、v6 新增：动态 Advisor 两条路

图上有一个青色虚线框（x=1700, y=586），标题是 "v6: 动态Advisor两条路"。这个框解释的是：当动态Advisor要去感知一个东西时，有两条不同的路径——取决于感知目标是"枚举类"还是"描述类"。

路1: enum 信号订阅（绿色子框）

适用：枚举类需求——性别（男/女）、位置（主驾/副驾/后排左/后排右）、有无（有小狗/没小狗）
特点：这些都是已有标签，AIBox当前VLM能力已经支持输出这些标签。
处理方式：订阅结果直接返回 → 不需要拼SP → 不需要动态VLM推理链路 → 快速
丁彬原话："枚举类直接去用，直接条件匹配"
动态Advisor需要知道"副驾是不是在打电话" → 打电话状态=枚举类（正在打电话/未打电话）→ 走路1 → 直接读④信号库里的 passenger_phone_status 标签 → 等号匹配 → 条件满足就触发。
不需要额外下发SP给VLM，因为VLM的AlwaysOn已经在输出这些标签了。

路2: SP 拼接路（橙色子框）

适用：描述类需求——着装（穿白色衬衫/穿裙子）、物品状态（毯子掉没掉/项圈掉没掉）、动作描述（宝宝在哭/在笑）
特点：这些不是标签，不可穷举，VLM的AlwaysOn不会默认输出这些内容。
处理方式：Advisor给人话需求 → ⑮拼SP → 下发VLM → VLM推理 → 穿越检测 → push回 → 满足就删SP → 还原AlwaysOn
西钺原话："描述类需要去声明SP"
动态Advisor需要知道"毯子掉没掉" → 毯子状态=描述类（VLM没有"毯子掉/没掉"的标签）→ 走路2 → Advisor告诉⑮"毯子掉没掉" → ⑮拼SP（"观察后排毯子的状态"）→ 下发VLM → VLM每5秒看一次 → 某轮返回"毯子在地上" → 穿越检测 → push回Advisor → 任务完成 → 删SP还原。

十八、SP 生命周期 8 步流程——逐步详解

图下半部分（y=1610–2410）有一个红色大框，标题是 "6.30 SP生命周期完整回路 · 8步可视化流程"。这是整张图最详细的流程可视化——用"毯子掉了告诉我"这个case走一遍完整链路。

流程中有4个决策菱形（黄色/蓝色/绿色菱形），每个是一个分支判断。下面逐步讲：

Step ① ⑰动态Advisor创建任务

用户说"宝宝毯子掉了告诉我" → Director解析这是一个持续监测任务 → 创建一个动态Advisor来专门盯这件事。

关键动作：创建前先查㉑Context已有信息！如果Context里已经有"毯子"相关的信息（比如之前已经在监测了），就不重复下发。

西钺："必须过滤，已有的不能再下发动态任务"

→ 决策菱形"Context有信息?"：

  • 有 → 直接用已有信息，不下发新任务
  • 没有 → 进入Step②

Step ② ⑮收到人话需求

⑰给⑮的是人话——"毯子掉没掉"。注意：不是SP！ Advisor不做SP拼接，它只告诉⑮"我想知道什么"。

丁彬："advisor给的是想要感知的事"

⑮收到人话后，交给15-16做枚举/描述信号分流。

→ 决策菱形"枚举 or 描述?"：

  • 枚举类 → 直接条件匹配④信号库已有标签（快速，不需要VLM动态推理）
  • 描述类 → 进入Step③

"毯子掉没掉"是描述类（"掉没掉"不是一个固定标签）→ 走描述路径。

Step ③ 调提槽工具（15-17）

描述类需求来了，⑮调公共提槽工具判断direction：这个东西在舱内还是舱外？

  • "毯子" → 舱内（毯子在车里）
  • "瑞幸门店" → 舱外（门店在车外）

西钺："不用走VQA绕一圈，直接调我推理"

→ 决策菱形"舱内 or 舱外?"：

  • 舱外 → 走VQA云端推理链路（图可以上云）
  • 舱内 → 进入Step④（合规要求图不能上云→走端侧VLM）

"毯子"=舱内 → 走舱内路径。

Step ④ ⑮拼接SP（15-18引擎）

⑮把人话需求"毯子掉没掉"工程拼接成标准SP格式：

AlwaysOn基础SP
prompt: aaaa, bbbb + 动态SP1
prompt: cccc ("观察毯子状态") = 合并SP
prompt: aaaa, bbbb, cccc

KV槽位映射：SP1→answer1 / SP2→answer2 / 新增SP3("毯子状态")→answer3

多任务合并到同一5秒周期——丁彬说"不能把多个任务放到不同循环里，否则每个循环推理周期变长"。

合并后的SP通过L14（云→端，橙粗线）下发给㉔。

Step ⑤ VLM端模5秒推理

SP经㉔→24-8'下发到VLM端模。VLM以固定5秒周期运行，每轮推理输出：

{answer1: "主驾正常", answer2: "副驾正常", answer3: "毯子在地上"}

关键：VLM只输出客观描述。"毯子在地上" ≠ "毯子掉了"。VLM不做判断——它不知道毯子"在地上"意味着"掉了"，那是后面触发器做的事。

丁彬："无非往里插一个多感知一个任务"

Step ⑥ 结果回传链路

VLM推理结果的回传路径是一条严格的链：

VLM输出 → ④端侧信号库 → L9 → ⑫云端信号库 → ⑮ 15-13 穿越检测

铁律②：不直推Advisor，先经信号库链路。

⑮的15-13对answer3做阈值穿越检测：上一轮是"毯子在沙发上"（F状态），这一轮变成"毯子在地上"（T状态）→ F→T穿越！

→ 决策菱形"穿越命中?"：

  • NO → 继续等下个5秒周期（毯子还在沙发上，没变化）
  • YES → 进入Step⑦

Step ⑦ push⑰ · ⑰判断

穿越检测命中 → ⑮通过L18 push事件给⑰动态Advisor。

⑰收到事件后做判断：毯子确实掉了 → 满足退出条件吗？

丁彬："毯子当天没掉，advisor不退出，持续跑"

→ 决策菱形"任务完成?"：

  • NO → 继续持续监测（比如毯子从地上被捡起来了→条件又不满足了→继续等）
  • YES → 进入Step⑧

case里："毯子在地上"= 条件满足 → 任务完成 → YES。

Step ⑧ 通知⑮清理SP · 还原

⑰通过L34通知⑮：任务结束，清理SP。

⑮删除SP3（"毯子状态"）→ AlwaysOn还原为基础SP（prompt: aaaa, bbbb）→ VLM下一轮不再看毯子状态。

西钺："取消任务时触发器需同步把SP给删掉"

后续：⑰→⑲Director建议 → ⑳推理 → 输出话术"宝宝的毯子掉了哦~要不要帮您调高一下空调温度？" → L29下发㉖TTS播报给用户。

完整链路一句话总结：
Advisor给人话 → ⑮判枚举/描述 → 描述→调提槽判舱内/舱外 → 舱内→⑮拼SP → 下发VLM → 5秒推理→结果回④→同步⑫→⑮穿越检测 → 命中→push⑰ → ⑰判完成→通知⑮删SP → 还原AlwaysOn → ⑰出建议→Director出话术→TTS播报

二十、SP 拼接怎么拼？——完整示意

图最底部（y=2630–2830）有一个青色大框，标题是 "v6新增: SP拼接组合示意"。这是整张图最直观的一张"公式图"：

AlwaysOn基础SP
prompt: aaaa, bbbb
（始终在跑的基础感知任务） + 动态任务SP
prompt: cccc
（Advisor下发的新任务） = 合并后 AlwaysOn SP(新)
prompt: aaaa, bbbb, cccc
KV槽位: answer1(aaaa) answer2(bbbb) answer3(cccc)

合并后的SP经过这条链路下发和执行：

合并SP → ㉔ 端侧触发器
24-8' SP下发管理 → VLM端模
5S周期推理 → KV返回 → ④ 端侧信号库
结果全量上传 → ⑫云端

KV槽位机制详解

什么是KV槽位？每个SP任务有一个固定的"answer位"。VLM推理时，每个任务的结果写到对应的answer位，互不干扰。

当前有3个任务同时在跑：
- SP1: "观察主驾状态" → answer1: "主驾正常驾驶"
- SP2: "观察副驾状态" → answer2: "副驾闭眼低头"
- SP3: "观察毯子状态" → answer3: "毯子在地上"

VLM每5秒推理一次，返回 {answer1:"主驾正常驾驶", answer2:"副驾闭眼低头", answer3:"毯子在地上"}。

SP3任务完成 → Advisor通知⑮删除SP3 → 下一轮VLM只推理SP1+SP2 → 返回 {answer1:"...", answer2:"..."}。

为什么要合并到同一周期？

丁彬："不能把多个任务放到不同循环里，否则每个循环推理周期变长"
西钺："舱内都得在舱内5秒周期，每个周期都得带，直到拿到想要的结果"

如果3个任务各跑一个循环：循环1(5s)→循环2(5s)→循环3(5s)= 15秒才能轮一圈。合并到同一循环：5秒全部搞定。

AlwaysOn SP生命周期

  1. 初始状态：只有基础感知SP（aaaa, bbbb）
  2. 动态任务来了 → 拼入cccc → 变成（aaaa, bbbb, cccc）
  3. 又来一个任务 → 拼入dddd → 变成（aaaa, bbbb, cccc, dddd）
  4. cccc任务完成 → 删除cccc → 变成（aaaa, bbbb, dddd）
  5. dddd任务也完成 → 删除dddd → 还原为（aaaa, bbbb）= 初始状态

二十一、关键争论与约束汇总

图下半部分（y=2020–2400）右侧有多个讨论框，记录了6.30会议的关键争论。这里汇总讲：

21.1 张兰若3种query类型

#类型示例说明
①条件触发"毯子掉了告诉我"持续监测，满足条件才通知
②一次性"帮我看毯子掉没掉"看一次给结果就完了
③间接需求"宝宝冷不冷"需要先拆解为"看毯子掉没掉"再执行

结论：给VLM的都是同一个任务（"帮我看毯子掉没掉"）。持续/一次性的区分在⑰Advisor做，不在VLM。

丁彬："至于长期还是一次，本质上是Advisor去看"

21.2 SP写法争论：条件式 vs 客观任务式

写法示例谁支持
条件式"如果毯子掉了就告诉我"（含判断逻辑）丁彬："可以试试带推理的·yes or no回答"
客观任务式"帮我观察毯子的状态"（只看·不判断）西钺："我们现有能力=客观描述，没有让他做推理"

当前结论偏向客观任务式——VLM只管看，判断留给触发器。但丁彬觉得可以探索条件式（待验证 = 卡点②）。

21.3 SP拼接谁做的争论

张兰若提议："VLM自己拼·接一个放一个·不需要上游拼好"
西钺反驳："触发的时候带上SP是自然动作" → 触发器拼
丁彬补充："关键是结果返回后谁把SP1/SP2去掉" → 西钺："取消任务时同步删SP"

最终结论：SP拼接/管理/删除全归⑮触发器。Advisor只给人话需求。VLM只管推理。
丁彬确认："⑮既能判断事件，也能更新prompt" · 西钺："对，这就是要加的"

21.4 VLM输出抖动/glitch风险

张兰若警告："我们但凡出现一次错误，它发现两次"——比如VLM第N轮输出"没有小狗"→第N+1轮glitch输出"有小狗"→第N+2轮又"没有小狗"。如果触发器直接用，就会误触发。

影响：VLM误判 → 触发器误触发 → 误通知用户（"你家狗项圈掉了"，其实没掉）
对策：需⑮ 15-6抖动过滤配合——±区间过滤连续翻转。

21.5 仁杰语义判断结论

仁杰问：端侧触发器需要语义判断能力？
西钺纠正："端侧不需要做语义判断·做条件判断·大于等于/变更为"
语义判断在云端·不在端侧。端侧触发器只用运算符（等号/大于/变更为），不做"理解含义"这种事。

舱外也需要前置检查：西钺推演——舱外也需触发器先判断现有信息是否足够，不够才调VQA。
