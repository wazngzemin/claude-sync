一、当前架构是什么？（Harness 工程的实际结构）

Planner SP 的实际组成（优化前 vs 优化后）

从 SP 优化专项文档提取的真实数据：

组成部分优化前优化后（当前目标）
车辆知识914 tokens保持不变
工具描述（23个）2,769 tokens保持不变
参考示例（64条!）26,164 tokens（占80%!）拆分：静态10条 + 动态检索
其他（角色/格式/注意事项/人设）~2,959 tokens字段压缩（如 talk_or_not→speak）
总计32,806 tokens目标 ~16,000 tokens
核心发现：64条参考示例占了 SP 的 80%！这就是为什么 SP 这么大（32K tokens）。优化的核心就是拆示例——把64条静态示例拆成"少量静态 + 动态检索"。

优化后的示例加载架构（3层）

SP 里的示例 = 3 层加载：

第1层：静态示例（10条，始终在SP里）
  高频 + 通用的示例，不管什么场景都需要
  如：导航、车控、音乐播放等基础对话模式
  ~4000 tokens

第2层：动态示例 - 基于 query 检索（5条，每轮替换）
  用户当前 query 的 embedding 向量 → vikingDB 检索 → Top-5 最相似的示例
  如：用户说"我好热" → 检索到降温场景的示例
  问题：多轮对话时需要 query 改写
  （用户第3轮说"好的那就这家"，原始query检索不到有用示例，
   需要改写为"去鮨一日本料理用餐"再检索）

第3层：动态示例 - 基于 tool_feedback 检索（5条，工具返回后加载）
  上一轮工具返回了什么 → 检索该工具对应的多轮示例
  如：search_poi_qa 返回了 POI 列表 → 加载"POI选择对话"的示例
  让 Planner 知道拿到 POI 结果后应该怎么回复

总计：10（静态）+ 5（query检索）+ 5（tool检索）= 20 条
从64条减到20条 → SP从26K tokens减到~8K tokens → SP总量减半!

对比：动态知识 vs 动态示例的检索架构

动态知识（Skill）动态示例（Few-shot）
检索技术LLM 推理（Selector SP）纯向量检索（vikingDB）
检索什么场景 Skill 全文（执行项/前置条件）query-answer 的对话样本
检索依据语义推理：query + env + 历史 + 目标 + 已加载ID文本相似度：query embedding + tool_feedback
检索数量1-3 条 Skill（按场景命中）5+5=10 条示例（query检索5 + tool检索5）
耗时500ms - 1.8s几十 ms
向量库不用（LLM推理检索）vikingDB
注入SP位置{{knowledge}}替换/补充原来64条静态示例
负责人徐燃（已做好）4月做过一次，需更新

二、当前架构的好处和坏处

✅ 好处

1. 动态知识：LLM 检索能理解语义

"我车里有只猫" → LLM 能推理出 = 宠物场景。这是纯向量检索做不到的——"猫"和"宠物模式"的文本向量相似度很低，但 LLM 能理解它们的语义关系。

2. 动态示例：向量检索快且准

几十 ms 就能找到最相似的示例，对性能零影响。而且基于 embedding 的相似度对于"找相似 query"这种任务天然适合——"我好热好闷"和"好热啊赶紧降温"的向量相似度很高。

3. 示例拆分大幅压缩 SP

从 64 条全量加载（26K tokens）→ 20 条按需加载（~8K tokens）= SP 减半。直接带来：

  • 首 token 延迟从 2309ms → 998ms（下降 57%！）
  • Director 总耗时从 12231ms → 6179ms（下降 50%！）

4. 两层检索互相独立，各管各的

知识检索管"做什么"，示例检索管"怎么输出"——职责分明，一个出问题不影响另一个。

5. 动态示例支持基于 tool_feedback 检索

这是一个巧妙的设计——上一轮工具返回了 POI 列表，系统自动检索"拿到 POI 后怎么回复"的示例注入 SP。这让 Planner 在多轮对话中每一轮都能拿到最相关的示例参考。

❌ 坏处 / 当前瓶颈

1. Selector SP 是最大的性能瓶颈

额外一次 LLM 调用 = 500ms-1.8s。当前目标是压到 500ms 以内，但 13 个场景就已经 1.8s 了。PRD 目标 64 个场景，如果不改架构只会更慢。

2. 示例的 query 改写问题没解决

SP 优化文档明确写了：多轮对话时原始 query 检索不到有用示例。如用户第 3 轮说"好的那就这家"——这句话本身的 embedding 和任何场景示例都不相似。需要先做 query 改写（结合上下文把"那就这家"改写为"去鮨一日本料理"），再拿改写后的 query 去检索。这个改写本身又需要一次 LLM 调用或规则处理。

3. 知识检索和示例检索可能不一致

知识检索（LLM 推理）判断是"降温场景"，但示例检索（向量检索）可能因为 query 的表达方式不同，检索到的是"通风场景"的示例。两者不一致 → Planner 拿着降温知识但看着通风示例 → 输出混乱。

4. 示例库需要持续维护

Skill 改了，对应的示例也要跟着改。但示例是存在 vikingDB 里的向量，更新比较麻烦——需要重新 embedding、重新入库。不像 Skill 改个文本就行。

5. 示例压缩不够

SP 优化文档指出："目前示例都是裸示例，没有经过抽象。"比如 tool_feedback 里的 POI 数据包含大量无用信息（地址、电话、经纬度），这些不需要保留在示例里，应该压缩成关键信息。

6. 缺乏评测闭环

知识检索的准确率只有 60-70%，但没有自动化评测体系持续监控。每次改了 Selector SP 或 Skill，需要手动跑测试验证。

三、和传统 Harness 工程的区别

"Harness 工程"在这个语境下指的是不做动态检索，把所有信息全量塞进 SP的做法——也就是优化前的状态（64 条示例全加载、知识全量注入）。

维度传统 Harness（全量加载）当前方案（动态检索）
SP 大小~32K tokens（固定不变）~16K tokens（动态部分按需加载）
示例加载64 条全在 SP 里10 条静态 + 10 条动态检索
知识加载要么全塞（太大）要么不塞（没知识）Selector 选出最相关的 1-3 条注入
首 token 延迟2309ms998ms（↓57%）
总耗时12231ms6179ms（↓50%）
准确率79.38%（所有示例都在，模型选择困难）88.38%（只注入相关的，模型更聚焦）
扩展性不可扩展——示例越多 SP 越大，迟早超限可扩展——示例库无限大，每次只取 Top-K
额外延迟0（没有检索步骤）Selector ~1.8s + embedding ~30ms
适合场景数≤20 个（再多 SP 装不下）理论无上限（64+也行）
本质区别：传统 Harness = "把整个图书馆搬到桌上"——信息全但桌子太小、找东西慢。当前方案 = "只把这次要用的书拿过来"——桌子清爽、找东西快，但需要一个"图书管理员"（Selector + 向量检索）帮你找书，这个管理员本身也有成本。

四、优化方向

4.1 已在做的优化（SP优化专项里的）

优化1：输出格式压缩

把 JSON 字段名缩短：talk_or_not → speak（少2个token），talk_content → content（少1个token）。看似小但每次输出都省，首token延迟降了 ~90ms。

优化2：示例拆分静态+动态

64条 → 10静态+5动态(query)+5动态(tool) = 从26K tokens降到~8K tokens。

优化3：示例压缩

把示例里的裸数据（如 POI 的完整JSON）压缩成关键信息：

压缩前（裸示例）:
tool_feedback: "poiId:B0IA61DPAA 名称:上海虹桥国际机场T2航站楼
地址:上海市闵行区申昆路1500号 标签:航站楼,机场,交通枢纽
电话:021-96990 位置:121.335000,31.195000 距离:0km
商圈:虹桥商务区 评分: 营业状态..."   ← 400+ tokens/条

压缩后（抽象示例）:
tool_feedback: "路线1: 预估25分钟, 4.5km, 10红绿灯
路线2: 预估27分钟, 5.0km, 6红绿灯"   ← 50 tokens/条

优化4：基于人工反馈的纠错示例

把实际 badcase 转化为纠错示例。如：查天气时用户没说地点 → 示例展示从 context 推断位置的做法。

4.2 还没做但应该做的优化

优化5：Selector SP 延迟压缩

当前 1.8s 太慢。文档里的建议路径："先用压缩锚词表把当前 13 场景的延迟压到 500ms 以内"。锚词表 = 每条知识的摘要从 ~100 字压缩到 ~30 字关键词，减少 Selector 需要处理的 token 数量。

优化6：解决多轮 query 改写问题

用户第 N 轮的 query 可能是"好的那就这家"——这种短 query 的 embedding 检索不到有用示例。需要一个 query 改写模块，结合对话历史把短 query 扩展成完整表达。

优化7：知识-示例一致性保证

Selector 判断是"降温场景"（注入降温 Skill），但向量检索的示例可能是"通风场景"。需要在两套检索结果之间做一致性校验——或者让示例检索也参考 Selector 的输出 ID 做过滤。

优化8：自动化评测闭环

构建一个自动化管线：改了 Skill/SP/示例 → 自动跑评测集 → 量化命中率和准确率 → 低于阈值告警。避免"改了但不知道效果变好还是变差"。

五、可借鉴的其他架构模式

5.1 RAG（Retrieval-Augmented Generation）

它是什么

RAG 是当前业界最主流的"外部知识增强 LLM"的架构。流程：用户 query → embedding → 向量库检索最相关的文档片段 → 拼到 prompt 里 → LLM 生成回答。

和我们的方案对比

标准 RAG我们当前的方案
检索方式纯向量检索混合：知识用LLM检索 + 示例用向量检索
检索对象文档片段（同质的）两种不同的东西：Skill知识 + few-shot示例
检索粒度文档 chunk（~500字/条）完整 Skill（~2000字/条）+ 完整示例（~800字/条）
注入位置通常在 user message 里在 system prompt 的不同模板变量里

可以借鉴什么

  • Reranker 重排：向量检索 Top-20 → 再用一个小模型做精排取 Top-5。可以提升示例检索的准确率。
  • Chunk 优化：RAG 领域有大量关于"怎么切文档效果最好"的研究，可以参考优化 Skill 的长度和结构。
  • HyDE（假设文档嵌入）：先让 LLM 生成一个"假设的理想回答"，用这个假设回答去做检索，而不是用原始 query。可以解决我们的"短 query 检索不准"问题。

5.2 Self-RAG（自反思 RAG）

它是什么

标准 RAG 的升级版——LLM 生成回答后，自己检查"我检索到的内容和我生成的回答是否一致？如果不一致，重新检索"。引入了一个反思（reflection）步骤。

可以借鉴什么

当前问题：Planner 用了检索到的 Skill 编排，但可能漏掉了某些执行项。Self-RAG 的思路是——Planner 生成 action_list 后，检查"我的 action_list 是否覆盖了 Skill 里所有的执行项？"如果没有，补充。但这会增加一轮推理 = 增加延迟，在车载场景下不一定可行。

5.3 Agentic RAG（多 Agent 协作检索）

它是什么

不是一个 Agent 做所有检索，而是多个专业 Agent 各管一块：一个 Agent 管知识检索，一个 Agent 管示例检索，一个 Agent 管环境感知...然后一个 Orchestrator 汇总所有结果给主 Agent。

和我们的方案对比

我们已经在做这个模式了——Selector 是一个专业的"知识检索 Agent"，vikingDB 是"示例检索引擎"，env_info/memory 是"环境/记忆模块"，它们各自独立工作，结果汇合到 Planner SP。

可以进一步借鉴什么

加一个Router Agent在最前面——不是所有 query 都需要走场景知识检索。简单问题（"现在几点"）直接走 Planner，不触发 Selector。这样可以避免 Selector 的 500ms-1.8s 延迟浪费在不需要知识的 query 上。其实拒识仲裁⑨就是在做这件事——它先判断 query 是否需要走场景，不需要就不触发 Selector。

5.4 Tool-Augmented LLM（工具增强）

它是什么

不在 prompt 里塞知识，而是让 LLM 在需要时自己调用"知识查询工具"——就像 ChatGPT 的 function calling，LLM 自己决定什么时候需要查知识、查什么。

和我们的方案对比

Tool-Augmented我们的方案
知识获取时机LLM 推理过程中自己决定要不要查在 LLM 推理之前就预先注入
优点不需要 Selector → 省掉检索延迟LLM 一开始就有知识 → 推理更准
缺点LLM 可能忘了查 → 没有知识就瞎编需要额外的检索步骤 → 有延迟

可以借鉴什么

对于紧急场景（如安全红线判断），不需要等 Selector 检索完。可以把安全规则做成一个"安全检查工具"——Planner 在生成 action_list 之前先调用这个工具检查"当前条件下是否允许这个操作"。这比在 SP 注意事项里写"非P档禁止调座椅"更可靠。

5.5 Cache-Augmented Generation（缓存增强）

它是什么

既然每次调 Selector 都要 500ms-1.8s，能不能把 Selector 的结果缓存下来？用户连续说了 3 句关于"热"的话，第 1 次 Selector 判断了"降温场景"，第 2、3 次直接用缓存结果，不再调 Selector。

可以借鉴什么

其实 Selector 的"增量更新"设计（add + remove）就是这个思路的雏形——loaded_knowledge_ids 记录了上一轮已加载的知识，Selector 只做差异判断，不是每次全量重新检索。但当前实现似乎还是每轮都完整执行 Selector，可以优化为"如果用户还在同一个场景里，跳过 Selector"。

六、总结：我应该做什么？

基于以上分析，你（泽民）在写 Skill 和推进优化时应该注意：

1. Skill（动态知识）写作

  • 每条 1500-2500 字，不要太长（稀释注意力）也不要太短（模型会幻觉）
  • 执行项用 SOA 工具名而不是自然语言
  • 关键信息重复写（开头+结尾）
  • 同时写好"Selector 索引摘要"——30字触发词+场景描述，帮助 Selector 准确分类

2. 动态示例需要更新

  • 4月做过一次，Skill 改了示例也要同步改
  • 需要和明骏配合：你提供"改了哪些 Skill"→ 明骏更新 vikingDB 里对应的示例和向量
  • 注意示例压缩——不要裸数据，抽象关键信息

3. 推进 Selector SP 延迟优化

  • 压缩知识索引表（每条从 ~100字 → ~30字锚词）
  • 构建评测集量化回归
  • 长期：讨论是否引入向量预筛（先 embedding 选 Top-5 候选，再 LLM 精选）

4. 关注知识-示例一致性

  • 和明骏讨论：示例检索能否参考 Selector 的输出 ID 做过滤？
  • 避免"Skill 说降温，示例教通风"的不一致情况
