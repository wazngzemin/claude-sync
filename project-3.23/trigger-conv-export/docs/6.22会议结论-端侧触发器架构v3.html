一、最大的结论：端侧闭环场景全部回CDC（底软） 6.22会议结论

会上仁杰提了一个判断标准，所有人都认了，这是整场会最大的scope（范围）变更：

仁杰 13:14：像那种车辆安全、上车安全、下车安全，它是所有车一个标配的功能，不会变，不涉及条件变化或用户改配置，那这种直接底软做掉，不需要走触发器去绕。像那种有生命周期的、用户临时创建、消费完就没了、条件可能变化的，那走触发器。
仁杰 14:32：像这种起步安全，你放在触发器里面专门去配一下，你反而还要让触发器去接它所有的底层信号，然后还要考虑触发器本身的稳定性和延时，不如直接在底软就把事做了。
丁彬 16:17：那看来原来端侧闭环的，现在还继续是闭环的，就几乎不需要触发器去做什么。
西钺 16:26：我觉得这样也很清晰，相当于所有的这种触发都在客户那边嘛，我们就其实只用云端的那些，然后到时候跟他做一个系统的仲裁就行了，这样也很干净。
判定标准（一句话版）：

固定 / 不变 / 标配 / 车辆整个生命周期都不会变 / key-value（键值对）纯规则 → CDC底软闭环，不走触发器
动态 / 有生命周期 / 用户创建 / 可运营 / 条件会变化 → 走触发器

逐个场景对照表

场景归属为什么会上谁说的

边界情况：CDC（底软）和触发器职责交叉怎么办？
  • 同一个时机两边都想响应：比如起步时CDC要做起步安全检查，触发器也注册了一个"起步时播放音乐"的任务。这两个不冲突，各做各的——CDC管安全，触发器管体验。但如果CDC的安全检查阻止了起步，触发器的任务也不该执行。→ 需要一个仲裁机制，西钺 16:26 说了"到时候跟他做一个系统的仲裁就行了"
  • 赛力斯内部多个业务同时提醒没仲裁：西钺 17:14 发现的——赛力斯各个业务场景之间内部协同有问题，同一个时机多个业务都会做提醒，没有统一仲裁。→ 需要推赛力斯他们自己把业务梳理好
  • 赛力斯CDC（底软）能力不够怎么办：有些场景虽然是"固定"的，但CDC（底软）做得粗糙（比如温控只有开/关没有智能调节）。这种情况下触发器不越界，CDC（底软）做到什么程度是赛力斯自己的事

这个结论带来的行动项

  • 产品(我)做 把之前PRD（产品需求文档）里"端侧任务②简单条件判定"、"端侧任务③白名单车控直执"删掉，这些全归CDC（底软）了
  • 产品(我)做 跟赛力斯确认他们固定闭环场景的仲裁机制——西钺17:14提醒了这个问题
  • 开发做 端侧触发器㉔不需要接CDC（底软）闭环场景的信号了，scope（范围）收窄，只做云端下发的动态任务

四、event（事件）vs condition（条件）：核心设计选择 6.22会议结论

这是6.22最核心的架构决策。佳杰用了一个经典案例把这事讲清楚了（29:34）：

案例："用户倒完车之后，如果车里有小朋友，就把儿童锁打开"

这句话拆成三个部分：

  • "用户倒完车" = event（事件） → 等 ⑮ push（主动推送）。不能24小时轮询"用户在不在倒车"，太浪费资源

  • "把儿童锁打开" = action（执行动作） → Advisor（顾问）做出判断后交给Director（导演）执行

会上原话，把这个设计选择讲到位了：

佳杰 26:35：我需要一个用户睡醒了的这样一个事件（event），我不会去轮询你的Context（上下文），我是注册了这个事件之后，我就等待你的云端触发器给我发这个事件。
佳杰 29:34：Context拿的是另外一部分的信息，它是作为condition（条件）存在的...第一趴"车在倒车"这个event（事件），我不可能无限制的24小时在轮询，这就产生了触发器的需求。
铁律④（6.22版）：
condition（条件/状态）→ ㉑Context（上下文）供读取
event（事件/变化）→ ⑮ 组装后 push（推送）给 Advisor（顾问）

两条路，不混。Advisor不轮询Context拿事件，触发器不存condition。
边界情况汇总：event（事件）和 condition（条件）分不清怎么办？
  • "车速超过120"——event还是condition？ 取决于使用方式。如果Advisor需要"车速超120时提醒我"，那"超过120"是一个阈值穿越event（事件），⑮组装后push。如果Director推理时需要知道"当前车速是多少"，那"车速=125"是condition（条件），存Context。同一个信号可以同时产生event和condition。
  • "充电桩在附近"——event还是condition？ 如果是第一次进入范围，是event（"首次检测到充电桩"）；进入范围后持续存在，就变成condition。⑮在阈值穿越时发一次event，Context同时更新condition。
  • 一个信号既是event又是condition的处理：⑮组装event push给Advisor，同时信号值更新到㉑Context。两条路各走各的，不冲突。

案例全流程："帮我照看后排宝宝，毯子掉了就提醒我"

  4. ⑱⑰ GoalList建监工 → 创建Dynamic Advisor（动态顾问）实例 [云端]
  5. ⑰ Dynamic Advisor拆解：需要VLM（视觉模型）持续观测 → 改写注意力SP（提示词）："持续观测后排区域，上报毯子/被子/覆盖物位置" → 调⑮注册"后排覆盖物位置变化"event（事件） [云端]

  9. ⑮ 持续监听⑫，发现某次输出变为"后排蓝色毛毯在地板上" → 阈值穿越（"座椅上→地板上"）→ 过抖动过滤 → 步骤j组装event："覆盖物位置变化-从座椅到地板" [云端]

这个案例的边界情况
  • 用户说完就下车了：任务已注册但用户不在车上了。→ 端模继续观测（可能需要），如果检测到所有人都下车了，触发器可以挂起任务，用户回来自动恢复
  • 毯子根本不在摄像头视野内：后排座位可能在舱内摄像头盲区。→ VLM持续输出"未观测到毯子"，⑮组装"观测受阻"event → Advisor建议Director告诉用户
  • 宝宝把毯子拉到地上又拉回去：阈值穿越检测 + 30秒抖动过滤 → 30秒内只触发一次。如果用户觉得太频繁，可以在任务中心调整提醒频率或disable（停用）
  • 弱网场景：信号库→云端同步断了。端侧VLM还在继续观测写入④信号库，但⑮在云端收不到新数据。网络恢复后补发积压的信号，⑮批量处理可能瞬间触发多个event → 需要去重
  • VQA二次确认失败：摄像头被抢占或VQA超时。→ Advisor可以选择只根据VLM的阈值穿越结果做决策，不等VQA确认，直接提醒用户（宁可误报不漏报）

佳杰需要触发器提供的接口（API）

接口说明举例边界情况
read（读取） 拉取当前所有"任务级"的数据列表 ["充电桩靠近自动开盖", "到公司提醒拿外卖", "帮我看着宝宝"] 只返回任务级，不返回感知级（比如底层的"监听车速变化"用户看了也没意义）

delete / restore（删除/找回） 从待办列表移除 / 从场景中心找回 用户删了"充电桩"任务 → 任务中心不再显示。后来又从"推荐场景"里找回来 delete是软删除（标记不显示），不是真删场景定义。restore恢复显示
sync（端云同步） 云端配置的任务同步到端侧持久化，弱网也能用 云端新配了一个"下雨自动关窗" → 同步到端侧 → 断网了也能执行 弱网时云端有更新但端侧拿不到 → 网络恢复后全量对账 + 增量补发

任务级 vs 感知级 的区分

举例说明这两种数据的区别

任务级（用户能理解，要在任务中心显示）：

  • "充电桩靠近自动开盖" — 用户能看懂，能开关
  • "帮我照看后排宝宝" — 用户自己建的
  • "10分钟后关座椅加热" — 定时任务

感知级（底层信号订阅，用户看了没意义，不显示）：

  • "订阅信号：后排覆盖物位置" — 这是"照看宝宝"任务的底层实现
  • "订阅信号：充电桩距离" — 这是"充电桩自动开盖"的底层实现
  • "监听端状态：档位变化" — 系统级监听，跟用户无关

触发器里需要一个字段来区分这两类，比如 display_level: "task" | "signal"，AIBOX任务中心只拉 display_level = "task" 的数据。

十一、AlwaysOn（常驻观测）调度归属 6.22会议结论

西钺问了（57:25）：AlwaysOn（常驻观测）的调度是触发器㉔做吗？

佳杰 57:47：AIBOX上有一个模块是专门去对所有的请求做优先级的排序的。AlwaysOn优先级大概是多少、VQA的优先级如何、其他视觉类需求的优先级如何，如果优先级低的，它可能会排队，会等。
结论：AlwaysOn（常驻观测）的调度在佳杰的AIBOX任务调度模块里，不在触发器㉔里。

触发器负责的是"什么condition（条件）满足了就触发什么action（动作）"——也就是"什么时候触发"。
AIBOX任务调度负责的是"VLM同时有多个任务要跑时，谁先谁后"——也就是"谁先执行"。

但㉔端侧触发器内部也有优先级P0-P4（见第九章），那是管触发器自己内部场景之间谁先执行的，跟AIBOX的任务调度是两层不同的优先级。
