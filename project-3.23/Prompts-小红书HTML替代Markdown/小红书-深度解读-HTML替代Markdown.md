# 当 Claude Code 团队开始用 HTML 替代 Markdown：一次 AI 协作媒介的范式转移

> 原文作者：Thariq，Anthropic Claude Code 团队成员
> 原文链接：https://x.com/trq212/status/2052809885763747935

---

Thariq 发了一篇长文，说他和 Claude Code 团队的许多人已经让 AI 默认输出 HTML 而非 Markdown。

如果是普通人说这话，可能只是一次工具偏好分享。但说这话的人是 Claude Code 的核心开发者——他每天的工作就是和 AI 协作。他的选择背后，是对 AI-人类协作模式的深刻理解。

这篇文章不是翻译，不是信息图素材，是逐层拆解：他到底发现了什么，为什么这对每一个用 AI 工作的人都很重要。

---

## 一、先理解「前 AI 时代」Markdown 为什么赢

要理解 Thariq 为什么离开 Markdown，得先理解 Markdown 为什么成为标准。

Markdown 解决的是一个时代命题：**人类作者需要一种既能在编辑器里写、又能渲染成网页的轻量格式。** 它的核心价值是「人类可读可写 + 跨平台渲染」。

注意这个前提：**人类是作者。**

Thariq 在文章里说了一句非常关键但容易被扫过去的话：

> 我自己也越来越少亲手编辑这些文件了……通常也是提示 Claude 帮我改，这也削弱了 Markdown 最大的优势之一。

这句话是整个论证的起点。他观察到的事实是：**当 AI 成为文档的主要生产者，人类退居为审阅者和决策者后，Markdown 的核心价值——「方便人写」——变得无关紧要。**

而真正重要的变成了什么？**方便人审阅、理解、做决策。**

这就是他转向 HTML 的根本原因，不是 HTML 更新更酷，而是使用场景变了，评估标准也得跟着变。

---

## 二、HTML 真正提供的是什么：不是美化，是信息带宽

Thariq 列举了 HTML 的能力：表格、CSS、SVG、script 标签、绝对定位、canvas、image……

很多人读到这里会觉得「哦，就是比 Markdown 好看」。这是严重的误读。

他真正在说的是：**HTML 是一种更高信息带宽的媒介。**

什么是信息带宽？单位时间内你能从文档中获取的有效信息量。

Markdown 的信息带宽受限于纯文本线性排列——你只能一行一行往下读。HTML 可以同时传递多层信息：

- **空间信息**：并排对比、网格排列、绝对定位表达物理关系
- **视觉编码**：颜色标注严重程度、大小表达优先级、位置表达分组
- **交互信息**：折叠/展开、标签页切换、拖拽重排——读者可以主动探索而非被动接受
- **执行信息**：代码片段可以高亮、diff 可以内联标注、流程图可以直接可交互

Thariq 说他「几乎不存在某类 Claude 能读懂、但无法用 HTML 高效表达的信息」。这句话不是夸张，是真话。

因为 Claude 能理解的信息类型远不止文本——它能理解结构、关系、层级、流程、对比。而 HTML 是唯一一个能让这些信息类型全部「可视化呈现」的格式。

**Markdown 被迫用文字描述关系；HTML 可以直接展示关系。**

---

## 三、最被低估的洞察：文档变成了一次性工具

Thariq 讲了六个使用场景。其中第五个——「自定义编辑界面」——我认为是最重要的，也是最容易被忽略的。

他举了三个例子：

**例 1：优先级看板**
> 给 30 个 Linear ticket 做一个 HTML 文件。每张卡片可拖拽，在 Now / Next / Later / Cut 四列间移动。Claude 预排序。加「复制为 markdown」按钮。

**例 2：配置编辑器**
> 为 feature flag 配置做一个表单编辑器。按领域分组，显示依赖关系。开启一个前置关闭的 flag 时给警告。加「复制 diff」按钮。

**例 3：Prompt 调试器**
> 左右并排编辑器。左边是可编辑的 system prompt，高亮变量槽。右边三个示例输入，实时渲染填充后的模板。加 token 计数器和复制按钮。

这三个例子揭示了一个全新的工作模式：

**传统模式**：你告诉 AI 需求 → AI 写代码 → 你部署 → 你使用工具
**新模式**：你告诉 AI 需求 → AI 直接生成一个交互界面 → 你用这个界面完成当前任务 → 界面用完即弃

关键洞察在于：**这些 HTML 文件不是产品，不是可复用组件，不是 skill，而是一次性的专用工具。** 它们只为当前这一个任务存在。

这意味着什么？

**AI 不仅可以在「代码层面」帮你写东西，还可以直接在「交互层面」为你定制工作环境。** 你不再受限于现有工具的 UI，你可以随时让 AI 为你造一个趁手的临时工具。

这才是真正改变工作流的地方。不是「AI 写了文档我来看」，而是「AI 造了一个界面我来操作」。

---

## 四、导出按钮的真正含义：闭环设计

Thariq 反复强调一个细节：每个交互式 HTML 必须有一个「复制为 prompt」或「复制为 JSON」的导出按钮。

这不是一个顺手加的功能。它是一个设计原则。

**这个导出按钮的作用，是把你在图形界面里的操作，转换成可以喂回给 AI 的文本。**

这意味着整个交互流程是这样的：

你描述需求 → Claude 生成交互式 HTML → 你在 HTML 里调整/决策/排序 → 导出为文本 → 粘回 Claude Code → Claude 基于你的决策继续工作

注意这个结构中发生了什么：**人类的价值被精确地定位在了「做判断」这个环节上。** 你不需要写代码，不需要写文档，甚至不需要写 prompt——你的工作是拖动卡片、调整参数、审视结果。这些判断被导出按钮捕获，转换成 AI 能理解的格式。

**交互式 HTML 在这里扮演的角色，不是一个「输出结果」，而是一个「人类意图的捕获界面」。**

这个设计思想深到可以单独写一篇文章。它解决的是 AI 协作中最难的问题：如何让人类在正确的颗粒度上介入，既不被琐碎操作淹没，也不被完全排除在决策之外。

## 五、「回路感」：一个诚实的困境

文章结尾，Thariq 写了一段非常坦诚的话：

> 我曾经开始担心，因为我已经不再深入阅读计划，我是不是只能放手让 Claude 自己做选择。但我很高兴地说，使用 HTML 之后，我反而比以往任何时候都更在回路中。

这段话值得每一个重度 AI 用户停下来想一想。

**随着 AI 能力变强，人类面临的核心风险不是「AI 会取代我」，而是「我逐渐退出了思考回路」。** 当 AI 输出的内容越来越长、越来越复杂，你发现自己读得越来越快、越来越敷衍。最终你成了批准按钮——看一眼标题，说「行」，然后下一个。

这不是 AI 的错，也不是你的错。**这是媒介的问题。**

纯文本 Markdown 的线性阅读体验，天然不适合审阅复杂内容。100 行以上的 Markdown，即使是作者自己也很难逐字消化。

Thariq 的解决方案不是「减少用 AI」，不是「回去自己写」。他是**把媒介从纯文本换成了可交互的 HTML**。通过视觉编码、空间排布、交互控件，让审阅行为从「被动阅读」变成了「主动探索」。

他没有退出回路。他升级了回路。

## 六、对 AI 从业者的实操启示

以上是分析。以下是你可以立刻用的东西：

**1. 重新思考 AI 的输出格式（不要只接受默认）**

当前绝大多数 AI 产品默认输出 Markdown 或纯文本。这不是因为 Markdown 是最佳格式，而是因为它是历史惯性。下次让 AI 做事时，可以实验：

- 「用 HTML 做一个交互式的……」
- 「做一个单页 HTML 报告，用颜色编码严重程度」
- 「用网格并排展示这几个方案」

你会发现同样的 AI，产出质量完全不同——不是 AI 变了，是媒介变了。

**2. 善用「一次性工具」这个模式**

不需要做 skill、不需要做产品、不需要可复用。让 AI 为当前这一个任务做一个专用的交互界面。

适用场景远不止 Thariq 提到的那些：
- 对比多个 PR 方案，拖拽排序
- 调试一个算法，滑块调参 + 实时图表
- 给数据标注，界面化判读
- 决策矩阵，可视化各选项的权衡
- 调试 prompt，并排对比不同版本的效果

**3. 在自己的工作流里加导出按钮**

如果你让 AI 生成交互式界面，一定要加导出功能。这是闭环的关键。

格式可以是：复制为 Markdown、复制为 JSON、复制为 diff、复制为 prompt。核心逻辑是把「UI 里的操作」转化成「AI 能消费的文本」。

**4. 关注你的「回路感」**

定期问自己：我真的是在思考，还是只是在批准？如果发现自己越来越敷衍，检查你的协作媒介。可能不是你和 AI 的问题，是 Markdown 在拖后腿。

## 最后

Thariq 的文章标题是「Claude Code 里的 HTML」。但他在讲的远不止 HTML。

他描述的是一个更根本的转变：

**当 AI 成为协作者，文档就不应该只是「文字容器」。文档应该是一个界面——让人可以审视、理解和决策的界面。**

Markdown 是人类作者时代的最优解。HTML 可能是 AI 协作者时代的起点。

Thariq 说他已经在想下一个媒介是什么了。

你也不该停在 Markdown 里。
