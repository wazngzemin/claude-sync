先搞懂一个根本问题：触发器为什么需要一份"条件清单"？

我们这台车里有个 AI 大脑（就是 Planner）。但你注意过一件事没有——它只有在你开口说话的时候才会动，你不说，它就一直干等着。这就像一个只会"你问一句、我答一句"的客服，谈不上聪明。

可我们真正想要的，是一台"懂事"的车：你还没开口，它看见你上车了就主动招呼你；它发现车里都 30 度了，不等你吩咐就先把空调开上。这种"不用你说、它自己动"的能力，就叫主动服务，靠的就是触发器。

但这里藏着一个关键点：大脑本身并不知道"30 度该开空调"这回事，它甚至不知道车里有"温度"这个数据可以读。这些它统统不知道，得有人一条一条喂给它——什么数据能读、读到什么值、该做什么动作。这份"喂给它"的东西，就是《高频触发条件清单》。你写一条，系统就多懂一个场景；你不写，这个场景它这辈子都不会主动。

所以你做的根本不是"整理一张 Excel"这么轻。你是在给这台车定义"它能主动感知、主动反应的全部场景边界"。这就是为什么这件事被排成所有需求的总源头（P0）：意图模型要拿它去训练、主动服务要拿它去配置、研发要拿它去注册——四方全都在等你这一份。

那"信号"到底是什么？这些信号从哪来？

信号，就是车辆每时每刻在往外冒的数据。车门是开是关、车速多少、车内多少度、座椅上有没有人、现在放的歌叫什么——每一个都是一个信号，车上有上百个这样的信号在实时刷新。

而一个"条件"，就是架在信号之上的一句判断。比如"车内温度 > 26 度"这个条件，用的就是"车内温度"这个信号，再加一句判断"大于 26"。再比如"主驾上车"这个条件，用的是"主驾座椅占用"这个信号——从"没人"变成"有人"。

所以你填这张清单的本质，是从这上百个信号里，挑出那些能拼成有用场景的，把"信号 + 怎么判断 + 触发后干嘛"配成一条一条。这就是清单里每一行在干的事。

为什么会上定了"以刘杨的表为准"？你之前那张表到底错在哪？

这段背景你必须知道，不然你不理解为什么要推倒重来。最早我们用的是一张"赛力斯"的信号表——赛力斯是另一个车厂的项目。后来我们做上汽 D6X，图快，直接拿赛力斯那张表来改。

问题就出在这：上汽和赛力斯是两台不同的车，信号根本对不齐。赛力斯有的信号，上汽这台车上可能压根没有；上汽新增的信号（比如"上下电状态""收藏地址"），赛力斯老表里又根本没列。你拿着一张赛力斯的底稿去配上汽，自然处处"对不上"——这就是你之前那张表的病根，不是你填得不认真，是底稿本身就是别人家的车。

而刘杨手里那张 master 表不一样：它是直接从上汽当前线上系统里拉出来的，是"此刻真实存在的信号"的全集，系统里有什么它就有什么，不多不少。所以它才是唯一可信的底稿，会上定"以他为准"就是这个道理。

那张表里有些行标了红色——意思是这些信号上汽已经删掉、不存在了；还有黄色的——有，但这一期不要。所以你拿到表第一件事，就是把红、黄两色的行删干净，剩下的才是上汽这期真实能用的信号底子。

为什么你只填"名称 + 参数"两列？刘杨说的那个"脚本"是什么？

刘杨说"你给我名称和参数就行，剩下的我脚本来"。这个"脚本"是他写的一个小程序：你把信号的"名称 + 参数"给它，它会自动跑去数据库里，把这个信号对应的内部 ID、完整定义全查出来、自动补齐。

这对你意味着两件事：① 那些复杂的 ID 字段你不用填，省事；但② 你填的名称必须和他系统里的一字不差。打个比方——他系统里这个信号叫"车内温度"，你要是图省事写成"温度"，脚本拿着"温度"两个字去数据库里找完全匹配的，找不到，就报错，这一行直接作废，他会打回让你重改。所以"照他 master 表上的原名抄"不是死板，是脚本能跑通的硬前提。

最关键的一格：运算符为什么是任军说了算，不是你？

你会上问刘杨"这些信号支持什么表达式、等于大于这些，是你定的吗"，刘杨说"不是，是任军写的，他会告诉我每个信号支持哪些逻辑表达式"。这一下你可能有点懵——为什么一个"大于小于"还得专人管？

因为不是每个信号都能用每种方式去判断。举个例子你立刻就懂：「车内温度」是个数字，你可以判断它"大于 26""小于 10""在 20~25 之间"——大于小于对它有意义。可「车门开关」这种信号只有"开/关"两个值，你对它说"大于"就是胡话——门没有"比开还开"这一说。所以每个信号到底支持哪些运算符，是引擎底层写死的能力，这个能力由任军定义和维护。

这也就解释了：我之前帮你把那 33 个数值信号的运算符补全（补上大于、小于、≤），现在只能算"草稿"——我是按常理推断"温度总该支持大于小于吧"，但引擎实际支不支持，得任军点头。所以你不是白做：你拿着我补好的这份去找任军，等于带着具体答案去对，而不是空手问"你们到底支持啥"——这恰恰是显你专业、又最省事的姿势。

那"做完"到底长什么样？为什么是那 5 条自检？

先搞懂：这件事到底在解决什么？

用户从来不会说机器话。他会说「超过 25 度就帮我开空调」「等会儿下了高速提醒我加油」——你听得出，这句话里其实藏着一个「什么条件 → 做什么」的结构。但机器听不出来，它只收到一串文字。

把这串文字翻译成机器能执行的结构 {条件:车内温度>25, 动作:开空调}，这个翻译工作，就是意图模型干的。而需求2 这件事，就是教会意图模型怎么翻译我们触发器这套话。

关键概念：什么是「意图模型」、什么是「解析」？

意图模型，就是一个专门猜「用户到底想干嘛」的 AI。「解析（parse）」这个词你会反复听到，意思很简单——把一句囫囵的话，拆成一个个零件。

用户说「超 25 度开空调」，解析之后变成四个零件：条件信号=车内温度、运算符=大于、阈值=25、动作=开空调。模型能不能拆得准，直接决定用户这句话能不能被执行。

为什么你要给它一份「约束词表」？

如果你什么都不给、放任模型自己猜，会出乱子：用户说一个「热」，它该解析成「车内温度」还是「车外温度」还是「发动机水温」？它可能瞎蒙。

所以你把需求1 那份清单丢给它，当成一张白名单，告诉它「你只准在这些信号、这些运算符的范围里解析，超出的别碰」——这就叫约束词表。清单越干净，模型越不会乱来。这也是为什么需求2 死死依赖需求1：没有清单这本教材，模型没法教。

「协议要对齐」是什么意思？为什么会上这条松了口气？

你写清单用的格式，必须和后台那套机器格式（DSL）长得一模一样。因为意图模型解析出来的东西，是要直接塞进后台去执行的。要是意图按 A 格式吐、后台只认 B 格式，中间就得有人翻译、就会错——这种两边格式对不上的事，行话叫「协议漂移」。

会上刘杨说了一句关键的：给弋途的接口和我们后台用的是同一套，只是加了鉴权。这就意味着对内对外本来就同源，不存在两套漂移，你这块的风险一下就小了。

你不是「交完就完事」——你是节奏的控制点

这里有个你容易忽略的点：你得问廖婉婷「调一版要几天」。因为意图模型训练、调试是要时间的。她说要 X 天，你就得倒推——清单必须在某个日子前交到她手上，否则 615 节点赶不上。所以你交清单不是终点，你是在替整条链卡时间点。

先搞懂：现在的引擎「笨」在哪？

触发器引擎现在的干活方式，是每隔一会儿给所有信号拍一张「快照」，看当前是什么值。问题就出在这：它只看得到「门现在是开的」，却不知道这门是「刚从关变成开」的，还是「已经开着十分钟」了。

这个区别要命。用户上车，是「门：关→开」加「人坐下」；用户下车，是「门：开→关」加「人离开」。要是引擎只看「门开着」这一张快照，它根本分不清你是刚上车还是要下车——主动服务就会闹笑话：该说「欢迎」时它说了「再见」。

「状态变更 + 条件组合」是要改成什么样？

重构的方向，是让引擎不光看「现在是什么」，还能看「从什么变成了什么」（这就是状态变更），并且能把好几个变更组合起来一起判断。

「上车」= 「门:关→开」并且「座椅占用:无→有」两个变更同时发生。这样引擎才能精准认出「这是有人上车」，不再瞎猜。

为什么这版先用「变更为」顶着？

完整重构是个大工程。所以上汽这一版，先用一个简化的「变更为」运算符顶一下——它能判断「某个信号变成了某个值」，但还做不了复杂的多条件组合。完整的状态机重构，是中长期的活，不在这一期。

你不用懂他底层怎么改引擎。你要交的是业务定义——把「上车」「下车」这些场景，用「哪个信号、从什么、变成什么」精确描述出来，他照着实现。这就是产品和研发的分工：你定义「是什么」，他实现「怎么做」。

🔴【你忘了的那件 · 最重要】想清楚"接近/离开/到达"怎么实现 —— 刘杨点名要你整理的逻辑

会上刘杨说了一句关键的：「这个泽民还得想想怎么做啊」「根据现有的信号协议是做不了的，看想想有什么别的方案可以做」。意思是——光有"目的地位置 + 剩余距离"两个信号还不够，"接近/离开/到达"这三个事件到底怎么用这俩信号拼出来、判定逻辑是什么，要你来设计。这就是你忘的那件事，也是这场会你最该交的东西。

你要整理的判定逻辑（建议按这个定）：

· 接近 = 剩余距离从远到近、第一次 ≤ 阈值（如 500m）→ 触发一次（one shot）。施佳杰强调：从远到近到某阈值"只触发一次"。

· 离开 = 剩余距离 > 阈值（或距离持续变大）→ 触发。

· 到达 = 剩余距离 ≈ 0（或 ≤ 到达阈值）且 档位变更为 P。

· 触发类型两种：one shot（满足一次触发一次，接近/到达用它）/ 循环（满足就反复触发）。

· 必须处理的三个坑：① 阈值谁定、要不要展示给用户；② 防抖——车在阈值边界反复进出别重复触发；③ 立体空间的距离 corner case（施佳杰提的，不高频但碰到要有策略）。
