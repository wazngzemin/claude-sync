## 导读：一句话 + 三个认知锚点

**一句话**：端侧触发器是**动态 Advisor（及主动服务 / Director）实现"主动感知—条件触发—主动服务"的底层执行引擎**。它对外提供三种"订阅"能力——条件订阅（端信号组合 / 状态变更）、定时订阅（端侧定时器）、VLM 视觉订阅（阈值穿越）——在条件命中时把事件交回上层，驱动车"主动做点什么"。

**锚点①——别被名字骗了，"触发器"是三个东西**：
1. **CTA 引擎 + 信号库底座**（Condition-Trigger-Action，Agent 侧、云为主）；
2. **端侧 VLM 触发器**（感知任务调度器，车端）；
3. **云端触发器**（Advisor/Planner 动态任务注册的中转校验层，云端）。

**锚点②——它有两条线**：一条是"条件→动作"的事件引擎线（高频条件清单是它的源头），一条是"调度 15+ 个 VLM 视觉任务"的感知调度线（完整 PRD 那条）。你最初问的"短期→长期、含 adviser、含 VLM"，正是这两条线的交汇。

**锚点③——它不是孤立产品，是 Advisor 的底座**：动态 Advisor 把 Director 派的目标拆成 4 类任务，其中条件 / 定时 / VLM 三类都要向触发器注册（"感知订阅"）。**没有触发器，动态 Advisor 二期（DNA-616）的条件任务、定时任务、导航 POI、VLM 视觉全部落不了地。**

# 第一部分：是什么

## 1.1 触发器的本质定位

触发器是端侧任务执行前的"**规则判断 + 任务调度 + 生命周期管理**"模块。一句话分工：

> 信号库存"车知道了什么"，触发器判"现在该跑哪个任务"，VLM / 下游 Agent 执行"具体做什么"，结果回到信号库 → 上报 Context → Advisor/Planner 消费。

它**不负责**：VLM 内部推理算法、Advisor 的思考逻辑、信号的原始采集。它只做"看规则 → 看信号 → 判命中 → 排优先级 → 下指令 → 处理成功/失败/超时/打断"。

## 1.2 必须厘清的"三个触发器"

| 叫法 | 实际是什么 | 在哪 | 谁负责 |
|---|---|---|---|
| CTA 引擎 + 信号库底座 | 「条件→触发→动作」事件引擎，覆盖舒适/出行/情感/安全全场景 | 云为主 | 王泽民 |
| 端侧 VLM 触发器 | 端侧感知任务调度中枢，调度 15+ 个 VLM 视觉任务，管优先级/抢占/超时 | 车端 | 王泽民 + 西钺(技术) |
| 云端触发器 | Advisor/Planner 动态任务注册的中转校验层，把条件任务下发到端侧 | 云端 | 兰若 / 云端团队 |

会议里这三者长期串台，是所有混乱的根源。记住：**注册在云端触发器、判断在端侧触发器、感知调度由端侧 VLM 触发器、上报统一走 Context。**

## 1.3 触发器在主架构中的位置（挂回 Director / Advisor）

```
Director(CEO·快决策对外) ──派目标──▶ 4 静态 Advisor(部门经理·深度思考) + 动态 Advisor(临时任务)
                                          │  舒适/出行/内容/情感 各管一条线
        Goal List(共享看板) ◀──读写──── 所有 Advisor
                                          │ advice(JSON)
                              全部 advice ──▶ Director 统一编排执行
                                          │
                            动态 Advisor ──感知订阅──▶ 【触发器 / 定时器 / VLM】 ← 本需求
```

- **Director = CEO**，快速对外决策、管对话队列与状态控制。
- **4 个静态 Advisor = 部门经理**（舒适 / 出行 / 内容 / 情感），各管一条线、深度思考，输出 advice 给 Director，不直接控车。
- **动态 Advisor** 处理临时持续任务（照顾宝宝 / 介绍风景 / 点咖啡），一目标一实例，独立生命周期。
- **Goal List 是共享看板**，恢复靠 Advisor 读看板、不靠 Director 记忆——这与触发器"动态任务重启恢复"是同一套思路。

## 1.4 双线：CTA 引擎线 + 端侧 VLM 线

- **CTA 引擎线**：高频条件清单（需求1）是源头 → 意图解析"条件话"（需求2）→ 主动服务配预设事件（需求3）→ 机制重构状态变更（需求5）→ 协议拍平（需求6）→ 端侧定时器（需求7）→ 距离/POI/地理围栏（需求8/10/11）→ 端云分工（需求9）→ 事件接口透出（需求12）。
- **端侧 VLM 线**：15+ 个 VLM 视觉任务从硬编码转规则驱动，加优先级仲裁 / 抢占恢复 / 超时释放（完整 PRD 全篇）。

# 第二部分：为什么要做

## 2.1 端侧 VLM 线的三大瓶颈（短期硬编码之痛）

**短期方案现状**：在 AI BOX SDK 里硬编码订阅几个状态，满足就直接调 VLM 识别任务。三个瓶颈：

1. **不可扩展**：每加一个任务、每适配一款新车，都要改代码发版。VLM 任务已涨到 15+。
2. **无统一信号源**：每个任务各自读 CAN 信号，同一个档位信号被三个任务各读一遍；VLM 结果不能复用（下车安全跑出的结果，停车寻车想用拿不到）；云端 Advisor 拿不到标准化端侧状态。
3. **无调度能力**：没有优先级仲裁 → 安全任务 P0 被 Always-on P3 挡住；低优旧 caption 覆盖高优 VQA 结果；VLM 卡死后队列不释放，后面全堵。

→ **长期方案 = 建端侧信号库 + 端侧触发器**，15 个任务规则驱动。

## 2.2 CTA 引擎线的痛点（底座 / 引擎 / 链路 / 契约）

来自名下问题梳理（6/3 上报）：

- **底座数据不规范（总源头）**：《高频触发条件清单》（需求1）没定稿，20–30 条标准条件还没录后台，产品/研发/意图/运营各写各的。**它不定稿，需求2/3/5 全线塌。**
- **引擎只看"当前态"**：只扫当前信号值，看到"门是开的"但不知是"刚从关变开"，分不清上车/下车（需求5 要重构成"状态变更 + 条件组合"）。
- **协议两套配不出"任意位置"**：车门是套娃结构，配不出"任意一个门开"（需求6 要拍平）。
- **下游链路断了**：原设计 `触发器→静态Advisor→Director`，但上汽实际没部署静态 Advisor → 命中事件不知道发给谁 → 主动服务跑不通、多事件合并丢失（主驾上车打一次招呼、副驾 3 秒后又打一次）。
- **意图接不住"条件话"**：用户说"超25度就开空调""到家附近找充电桩"，意图模型解析不出来。

## 2.3 上层驱动：动态 Advisor 二期（616）的硬需求 ★ 最硬的"为什么"

6/2 动态 Advisor 二期内审（施佳杰主讲，张航/李明骏/郝晓伟/廖婉婷/王泽民）当场对触发器提了一串需求——**这才是触发器最直接的需求来源**：

1. **长时任务新方案**：Skill + Workflow + 端侧 low-code runtime + AI 参与决策（类比 A2UI）。三种模板（顺序 / 分支 / 循环）可嵌套。Workflow 节点会**等待触发器的事件**（如 VLM/ASR 返回"用户是否笑了"）。
2. **导航类需求**：触发器要支持"接近某精确 POI（距离 < 指定值）抛事件、离开也抛事件" → 泽民评估云端能否拿到信号。
3. **定时任务一串**：① 端侧定时器（弱网/无网也能提醒 + 持久化）；② 相对时间倒计时显示（不能一直显示"5 分钟后"）；③ 超高频循环拒绝创建（最小间隔 1 分钟）；④ 分钟级合并播报；⑤ 模糊时间澄清；⑥ 指定某天 → 转成"当天首次上车"的条件任务；⑦ 循环任务最大 30 次。
4. **端侧 AI push**：端侧主动推卡，不依赖 TTS / advice 链路（张航："push 不用走 planner，直接推；在讲话就直接推出来"）。
5. **任务执行策略预生成**：不再命中时现场推理（省几百毫秒），提前把 advice / widget 卡片 / 清理策略盘好，"像条件反射一样"。
6. **条件任务交互优化**：中性回复 → 异步创建（实测 P90 8–9 秒）→ 显示具体条件字段卡片 → 成功/失败播报 → 二轮澄清循环。
7. **生态闭环类 Advisor**（美团/Nebby/弋途/上汽 AI light）：任务推进提醒 + 状态跟踪 + 任务分类，做成通用方案。

> 关键引子（张航）：**Planner 自己也会创建条件/定时任务**——"我说车里太热"→ Planner 开极速制冷 + 开窗 + 设一个"2–3 分钟后关窗"的定时器 + "5 分钟后 review 温度"。所以触发器的注册方不止 Advisor，还有 Planner 自身。
>
> 节点：施佳杰明确**导航 / 定时任务 / 条件任务交互这三个需求要在 616 节点完成，有明确验收**；长时任务重点推但不保证赶得上。

## 2.4 端云必要性论证（5/28 会议反复吵的核心）

为什么必须有端侧触发器、不能全放云上？

- **定时器**："10 分钟后关座椅加热"——云上跑倒计时有延迟，也没法做端上倒计时 → 必须端上。
- **弱网**："10 分钟后开窗"——10 分钟后车在隧道/地库，云端触达不了 → 端上预置策略、本地直接触发执行。
- **简单车控**："上车就开窗"——弱网下不该绕云走 Planner，端上直接闭环最快。

但 6/11 会上兰若/董均提出质疑：**端侧触发器若只做"路由上云"就没意义，必须能脱离云端独立离线闭环才有存在价值。** → 益红标为"泽民来盘"的遗留课题。本说明的回应见 4.6。

# 第三部分：怎么做（架构与机制）

## 3.1 长期方案总架构（三统一）

5/7 西钺评审定调：旧架构每根烟囱自己读信号、自己调 VLM、自己扔结果；新架构 **信号统一入库、触发器统一调度、结果统一上报 Context**。

```
                         云端
  规则配置后台 ──规则下发──┐
  Planner ──注册──┐       │
  Adviser ──注册──┤       │
              云端触发器(校验+中转)
                    │下发          ┌── Context(端侧上报的统一消费入口)
                    │              │     ↑按策略上报
══════════════════ 车端 ═══════════│════════════
  信号来源              端侧信号库 ──┘
  ├ 车身(车速/档位/车门/座椅)  ↑写入  ↓读取/订阅
  ├ OEM(蓝牙/FaceID)      ──→ 端侧信号库 ──→ 端侧触发器(两级匹配+调度)
  ├ 导航/场景(地库/导航结束)                     │运行指令
  └ 用户Query(VQA/泊车)                       端侧 VLM/小模型
                                          │结果回写信号库
```

## 3.2 信号库

- **统一写入规范**：所有信号统一为 4 个字段——信号名 / 信号值 / 来源 / 时间戳。后写覆盖（同名取时间戳最新）。
- **存储内容**：车辆状态（电源/档位/档位变化/车门锁/车速/座椅占位/安全带）、用户识别（舱外 FaceID）、OEM（蓝牙距离/远程备车/哨兵）、用户输入（Query）、场景（进地库/导航结束）、模型结果（VLM 标签/描述）、任务状态（生命周期）。
- **两条输出**：① 通知触发器"信号变了"；② 按策略上报云端 Context。
- **数据契约体检（名下问题梳理已修）**：补全 31 个数值信号的"大于/小于"运算符；14 条音频信号一级分类归位；13 个枚举信号 str→enum；与上汽 DNA《对不上清单》9 处（车门锁全车 vs 整车、续航口径、座椅通风 on/off vs 1-3 档、温度范围 None 等）。

## 3.3 触发器核心六机制

1. **两级条件判断（核心）**：第一级只看运行时段（电源/档位/车门锁，判断在"离车后/上车前/停车时/行车中"哪个阶段）；时段满足才激活第二级订阅具体触发信号。收益：离车后只有 5 条规则活跃，平均每次信号变化只有 1/3~1/2 规则要求值。
2. **信号对齐窗口**：触发条件里多信号到达差几十~几百毫秒（档位先到、车门后到）。收到任一触发信号后等一个对齐窗口（默认 500ms），窗口内信号合并成逻辑快照统一求值——防漏触发。
3. **优先级仲裁 + 互斥组**：P0–P4 五级（P0 立即抢占任意低优；P1 抢 P2–P4；P2 排队；P3 Always-on 闲时持续 3s/5s 可被抢占；P4 动态任务弱触发可静默）。共用同一摄像头的任务互斥（车内摄像头同时只跑一个；下车安全特殊，同占车内+车外）。
4. **五态结果**：success（写标签描述）/ empty（正常跑了没发现，有效信息非失败）/ failed（模型挂了，不触发下游）/ timeout（**禁止写成"无异常"**，没结果≠没危险）/ interrupted（被抢占，等恢复或终止）。
5. **抢占恢复 + 冷却**：高优到达→低优挂起保存恢复点→高优完成→检查恢复条件（触发条件仍满足？时段没变？没超时？）→能恢复就从断点续跑。冷却按规则粒度；P0 安全不冷却；P1–P4 VLM 类 ≥60s；通知类 ≥120s。
6. **运行指令**：触发器向 VLM 下发——执行哪个任务、识别目标、优先级、超时时间。

## 3.4 感知订阅：Advisor 如何接触发器（4 类任务 → 3 种订阅）★

动态 Advisor 拿到目标后做任务分类，三类要向触发器/定时器注册：

| 动态 Advisor 任务类型 | 注册到哪 | 对应触发器的什么需求 |
|---|---|---|
| **条件任务**（副驾有人+连续打哈欠） | 触发器模块（5–8s 注册脚本 / 选预设事件） | CTA 条件引擎、需求1 条件清单、需求5 状态变更 |
| **定时任务**（10 分钟后关加热） | 定时器（APScheduler / 端侧定时器） | 需求7 端侧定时器 |
| **VLM 视觉任务**（盯着宝宝睡没睡） | VLM 阈值穿越检测（定时拍照判定） | 端侧 VLM 触发器（完整 PRD 那条线） |
| 长时 / 持续任务 | 不订阅，持续推理 | 不归触发器 |

- 订阅请求体：task_id / type(=3 条件) / trigger_script(结构化脚本或预设事件名) / max_execution(1 单次 或 -1 无限) 等。
- 触发器响应：iCanDo(True/False) / action(select 选已有事件 | create 生成脚本) / eventSelect 或 scriptCreate。
- 数量上限：条件 ≤6 / 定时 ≤6 / VLM 视觉 =1 / 长时 ≤2 / 持续 =1。

## 3.5 VLM 触发的真身：阈值穿越检测 ★（PRD 没讲透，Advisor 架构补全）

- VLM 定时拍照（演示 5s，量产 30–60s），每次输出 `true|reason` 或 `false|reason`。
- **只有 `last==False AND this==True`（上升沿）才触发**——状态从"没发生"变"发生"那一刻。持续 True 不重复触发（防轰炸）。
- 两种判定模式：**event**（瞬时可判，一帧定，如"打哈欠"）/ **state**（持续状态，无明确瞬间，如"二排有宝宝"）。
- 视觉感知 Advisor 三类输出策略：风险监控（异常才说、只说一次）/ 信息讲解（新内容间隔 ≥60–90s）/ 状态陪伴（明显信号才触发，冷却 ≥5min）。

> 这与端侧 VLM 触发器 PRD 的 Always-on(3s/5s)、互斥组、五态结果是**同一件事两个视角**：端侧管"怎么调度/抢占/超时"，Advisor 管"怎么判定算触发/什么策略输出"。

## 3.6 动态任务注册链路 + "callback vs Context"的两层解释 ★

**两份文档看似打架，实为两层**：

```
逻辑层（云端，Advisor 视角）：
  动态Advisor ──注册条件任务──▶ 云端触发器 ──命中callback──▶ Advisor ──advice──▶ Director

物理层（车端，端侧触发器视角）：
  端侧触发器 ──调度──▶ VLM ──结果──▶ 端侧信号库 ──上报──▶ Context
```

- 西钺"Adviser 不直连**端侧**触发器、全走 Context"约束的是**端侧物理实现**：端侧只管把结果写信号库、上报 Context，不许端侧直接捅云上的 Advisor。
- Advisor 架构里"callback 给 Advisor"发生在**云端触发器 / 感知订阅层**。
- **结论：notify_adviser 不在端侧做，在云端做。** 不用推翻西钺，两层各管各的。你要做的是打通"端侧结果→Context"，让云端触发器 / Director 去 callback Advisor。

## 3.7 端云分工铁律（5/28 吵出来的边界）

| | 云端触发器 | 端侧触发器 |
|---|---|---|
| 职责 | 注册（动态+静态都在云端注册）、任务管理/仲裁（解"10 分钟后开窗"vs"5 分钟后关窗"的冲突）、模型生成类事件(VLM/语义) | 触发判断（命中与否完全端侧判）、定时器、弱网下本地执行 |
| 一句话 | 登记 + 解冲突 + 改策略 | 判断 + 执行，没脑子只管执行 |

共识：云端负责注册，端侧负责"是否触发"；不可能两边同步判断。仲裁因要结合语义、过模型，放云端；端上只跑写死规则。**端侧 action 策略模块**（佳杰提出）：事件发生时有两类 action——notify（给云）/ 直接执行（打通 TTS/车控/语音播报）；策略云上可改。

## ① 4/29 台架最小链路（张荣/董均/Walter/陈治宇/兰若/西钺）

## ③ 5/28 端云分工 + action 策略（王泽民/赵益红/施佳杰）
- **议题**：动态 advisor 链路及触发器分工。
- **关键结论**：① 端云分工——云端负责注册（动态+静态都在云端注册）、端侧负责"是否触发"；② **定时器在端上**（云上做不了倒计时）；③ 端侧需要 **action 策略模块**（notify / 直接执行，策略云上可改）；④ 任务管理/冲突仲裁在云上（开窗 vs 关窗矛盾）；⑤ 静态 Advisor = 主动服务 = 预设事件（一回事，换名而已）；⑥ 暴露上汽 DNA 信号缺失——**下车信号没有**（只能靠车门+座椅，副驾座椅传感器可能没有）、**"任意位置车门"配不出**（套娃结构，只支持指定主驾/副驾）。

## 附：术语表

- **CTA**：Condition-Trigger-Action，条件→触发→动作。
- **感知订阅**：动态 Advisor 把"帮我盯着 XX"翻译成"看什么(VLM SP)+什么条件(触发器)+多久一次(定时器)"的可执行订阅。
- **阈值穿越**：VLM last==False & this==True 的上升沿触发。
- **运行时段**：离车后 / 上车前 / 停车时 / 行车中，由电源+档位+车门锁组合判定。
- **互斥组**：共用同一摄像头的任务互斥。
- **五态**：success / empty / failed / timeout / interrupted。
- **三统一**：信号统一入库、触发器统一调度、结果统一上报 Context。
- **Director**：对话推理引擎，CEO 角色，快决策对外。
- **静态 Advisor**：舒适 / 出行 / 内容 / 情感，= 主动服务（预设事件）。
- **动态 Advisor**：处理临时持续任务，一目标一实例，向触发器感知订阅。
