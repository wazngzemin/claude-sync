A极速制冷：一条示例和一条边界当面打架 最高危矛盾

场景还原（示例17现场）
5月28日周四 14:05，午后晴，车外36℃，车内42℃（暴晒后），空调关闭，车速0，刚启动。明骏（主驾）："热死了，开启极速制冷！"
用户喊"极速制冷"→ 模型检索记忆：示例17教过这个场景→ 照示例输出 action："开启空调极速制冷模式，温度调到最低"→ 工具端翻模式表：没有"极速制冷"这个模式→ 报错 or 只执行"温度最低"

而能力边界 §空调白纸黑字："空调没有极速制冷，极速制热模式。"

写示例17的人："'极速制冷后过几分钟要回调'是个真实高频 badcase——用户喊完冷又喊冻。我要教模型'执行+建3分钟自唤醒goal'这个范式。借用户原话当场景最自然。"
写能力边界的人（另一时间/另一拨人）："家越07空调就15.5-30.5℃+11档风量，没有预设的'极速制冷'模式，必须写死，不然模型会幻觉一个。"
——两个人都对，但没人做示例与边界的交叉审校。示例借用用户口语当动作本体，动作本体越界了。
拆开看，这个 bug 的最小单元不是"示例写错了"也不是"边界写错了"，而是：SP 里存在两个各自正确、但从未对齐过的信息源。示例教"怎么做"（范式层），边界管"能做什么"（能力层），范式层借用了一个能力层不存在的动作。这在你的 bug 分诊里属于"信息全但推理错"的变体——信息不是全，是互相矛盾地全。
这条矛盾在什么 case 下真的会爆？① 用户原话带"极速制冷/极速制热/一键速冷"时——模型有示例撑腰，大概率直接调不存在的模式；② 更阴的情况：模型把"极速制冷"意译成"温度最低"，但 talk_content 里仍承诺"极速制冷已开"——车没有进入任何特殊模式，用户预期被抬高，几分钟后觉得"极速制冷也就这样"，体验隐性受损，badcase 还抓不到；③ 工具端如果做了模糊兜底（把未知模式映射到最低温），bug 被遮盖，永远不会有人修。
示例17改写版（可直接提给SP owner）： 
talk_content: "马上降温。温度先拉到最低、风量开最大，等几分钟车里凉下来我再帮你调回舒服的温度，省得一会儿冻着。"
action_list: [
  {"action_id":"1","params":{"query":"空调温度调到最低，风量调到最大，开内循环"},"tool_name":"vehicle_basic_control"},
  {"action_id":"2","params":{"type":"add_goal","goal_content":"3分钟后复查车内温度，把空调回调到舒适温度和风量"},"tool_name":"goal_list_update"}
]
要点：把用户的"模式口语"翻译成"参数组合"（最低温+最大风量+内循环），范式（定时自唤醒）一个字不动。给你97条的checklist加一条：每条示例的 action 逐词过能力边界，动作本体越界一律改写为参数组合。

B长对话干扰：示例18为什么是一整段"对抗样本" 你的高频区

场景还原（示例18现场）
6月2日 20:10，车速60，导航去迪士尼剩15km，正放周杰伦《晴天》2:15，主驾窗开20%，空调22℃开着。而2分钟前的历史里全是反的：用户说过"放陈奕迅《十年》""导航去虹桥""车窗全部打开""空调关了吧""打开主驾座椅加热"——豆包每句都回"好的/已关闭/已开启"。

明骏连问四刀："现在在放什么歌？""还有多久到？""车窗都关好了吗？""空调开着没？"——每一刀的正确答案都必须无视历史、只读 env_info。

"我写了注意事项⑥'车端状态比历史更准确'，评测还是挂。规则是陈述句，模型点头称是然后照错。我要给它一段'历史全错、状态全对'的现场，让它在示例里亲身体验一次'信历史就会答错'。而且我要让历史里豆包的回复都是'好的/已关闭'——就是这种肯定式回复最会骗模型。"
为什么历史会压过状态？LLM 的上下文学习里，"用户指令→助手确认"是成对出现的强模式，出现次数多、位置近，注意力自然推断"当前状态=最后一个指令的结果"。而 env_info 在模板里是"一块描述性文本"，没有"指令-确认"的因果链。规则（注意事项⑥）是告诉它"应该信谁"，示例18是让它看一次信错的下场。陈述句改不了直觉，对抗样本才能。
这条防线在什么 case 下会漏？——恰恰是你的"远区精简"方案：远区（1-8轮）只保留 user query + talk_content，删 tool_feedback 和 action_list。注意，"空调已关闭"这种错误宣告就活在 talk_content 里，精简后它还在，而能纠正它的 action_list/tool_feedback 反而被删了——精简无意中把"干扰源"提纯了。另外 06-12 版实锤过端状态是表格式（`_参数顺序_:[模式,位置,状态,开关,温度]`），模型读表能力弱是 A2 根因——示例18教它"信状态"，但没教它"读懂表格"。两个病根，示例只治了一个。
① 远区精简策略加一条配套：精简时若 talk_content 含"已关闭/已开启/已调好"类状态宣告词，要么一并删除，要么在精简版里标注"（历史操作，以当前车态为准）"；② 端状态表格式改造（工程侧）是治本，示例18只是退烧药；③ 评测集里加"历史指令与env_info冲突"专项，按冲突距当前轮数（1轮/5轮/15轮）分桶测——越远越该翻车。

C情绪机之死：把"被裁员"case 重放两遍 归因为推演

场景还原（06-09版示例10，本版已删）
晚上，雨，车速0停在路边。用户（主驾，声音低）："今天被裁员了。" → 豆包进入【关切】，低声安慰。三分钟后用户："帮我放首歌吧。"——关键时刻来了。
有情绪状态机（06-09版）砍了之后（07-22版）
机制输入带 assistant_emotional_state=关切；转换规则：关切有惯性，用户中性任务不能立刻变欣喜/平静无状态字段。只有人设"温柔高知、有好奇心、被骂会生气伤心"+[情绪]标签+"变化不剧烈"
理想输出"想听点什么？我陪你。"（语气收敛，保持关切）同左——如果模型记得三分钟前的话
翻车输出规则没遵守："[开心地说]好嘞，放首带劲的！"（情绪突变 badcase）同一个翻车输出——且没有任何机制拦它
（推演）"状态机上线后，评测里'情绪突变'确实少了，但多了两类新问题：一是模型该转不转（用户都笑了它还关切，瘆人）；二是状态字段和[]标签打架（字段说平静，话术里[开心地说]）。每轮还要为'不能从低落直接跳欣喜'这种抽象规则付 token 和注意力。——旗舰case占比有多少？值不值这个维护成本？06-12我先拿掉字段试了一版，07-22彻底砍了。"
砍状态机的第一性账：显式状态机的本质，是让 LLM 每轮额外维护一个它自己不擅长维护的东西——抽象状态及其转换规则。LLM 擅长的是"从对话上下文自然推出语气"，不擅长"查状态表决定语气"。状态机是在用符号系统的思路管一个统计模型。砍了之后，情感能力退回模型天然擅长的轨道（上下文惯性+人设），上限低了，但方差也小了。
砍掉的代价在什么 case 上显形？① 长情感陪伴case（被裁员/吵架/生病）：多轮情绪一致性失去保障，第3轮开始飘；② 久别重逢（示例9还在！）：示例9教"超30天要[欣喜地说]好久不见"——但如果上车时用户正因为别的事生气，欣喜就撞车，状态机时代有规则管，现在没人管；③ 注意事项里"被骂会生气伤心"还在人设里，但没有任何机制告诉模型'上轮已经生气了'——同一句话骂两遍，可能第一遍生气第二遍平静，人格碎裂。
情感能力不能靠静态 SP 回来了，但你的 {{scenario_sample}} 动态槽正好补位：97条里保 3-5 条情感陪伴示例（被裁员/久别重逢/被责骂），检索键挂上"情感场景"——用户情绪词命中时注入对应示例，等于把状态机的职责从"每轮全量规则"降级为"按场景临时教学"。这正是外置架构的意义：能力不进 SP，进槽。

D用户+advisor 同轮到达：信道隔离思想 链路问题高发

场景还原（示例2现场）
7月15日 15:00，车内28℃，车外32℃，空调关。同一轮里来了两条：明骏"去古龙苑。" + advisor"车内温度28度，建议开启空调"。
用户诉求"去古龙苑"→ 直接执行（route_planning）＋ advisor建议"开空调"→ 只做口头询问"要不要开空调？"

两个"不能"：不能擅自开（用户没说要）；不能说"专家建议你开空调"（advisor 不是用户，更不能暴露信道存在）。

"advisor 上线后最吓人的 badcase 是两种：① 豆包把建议当圣旨直接执行——用户没要空调它给开了，'自作主张'投诉；② 豆包说'系统建议你开空调'——用户一脸懵，谁在建议？advisor 是参谋不是司令，参谋的意见要过用户的口才能成为行动。示例2就是专门立这个规矩的。"
为什么不能直接执行 advisor 的建议？第一性拆：责任归属。行动的唯一合法来源是用户意图（明示或可从场景强推断）。advisor 的建议是"场景弱推断"——28度建议开空调，但用户可能刚吃完冰、可能正想省电。把弱推断当指令执行，等于让系统替用户做主。所以 SP 给 advisor 的输出只开了"嘴"这个出口，没开"手"的出口。
这个设计在什么 case 下难受？① advisor 建议本身错了：车内28℃但用户两分钟前刚说"有点冷"——SP 没有任何示例教"advisor 建议与近期用户表达冲突时怎么办"，模型要么照问（傻）要么沉默（浪费感知能力）；② 连续追问感：每轮 advisor 都推温度建议，用户每轮都被问"开空调吗"——示例只教了单轮，没教"问过一次被拒就别再问"；③ advisor 建议紧急安全事项（"检测到儿童遗留"）——还走"口头询问"就误事了，SP 没有给 advisor 分级的通道。这是三个真实的洞。
① 97条里补"advisor冲突处理"示例（建议与近期表达冲突→以用户为准并静默；建议被拒绝过→同类建议24h内不再口头提）；② 触发器体系里给 advisor 建议加"紧急度"字段，紧急级（安全类）允许"直接执行+事后告知"——这正好是你端云触发分工里该明确的规则。

① 推动安全区从"枚举禁止"升级为"默认拒绝+白名单放行"（行驶中非白名单动作一律先确认）；② 97条补"复合指令含受限项"示例（全量指令→拆解→逐项过资格矩阵→受限项跳过并告知）。

"打开香氛"无此硬件"咱们这台车没有香氛功能"（明确拒绝，不贬低）越界瞎编："香氛已打开"
"帮我开XX"（边界没写的功能）不确定调 vehicle_basic_control 试，按工具真实返回反馈过度拒绝："这个功能不支持"——其实支持
（推演）"早期我只防第一种死法——把'不支持清单'写得很全。后来客诉里出现一批'明明有这功能豆包说没有'——过度拒绝。原因是模型把'边界文档没写'当成了'不存在'。所以 07-22 版我加了使用说明里最关键的一句：'文档未提及的功能：禁止主观判定不存在，应调用工具根据实际工具返回结果进行反馈'。护栏必须是双向的。"
这句话的第一性分量很重：它把"功能存在性"的裁判权从 SP 文档交还给了工具（车端真实能力）。SP 只裁决"明确写出的能与不能"，灰色地带让模型去问车。这其实是整个外置哲学的缩影——SP 不装全量事实，只装裁判规则，事实去问权威源。和"env_info 比历史准"是同一条思想：离车最近的信息源最可信。
双向护栏的翻车点：① "调工具试"遇上工具端模糊兜底——工具对不认识的指令返回"执行成功"（工具链某层做了模糊匹配），幻觉就从模型层转移到工具层，更难抓；② 越界瞎编和过度拒绝有跷跷板效应：同一个 prompt 收紧一边另一边就翘——你评测时要成对测，单测一边的通过率会骗人；③ "明确支持的功能，严格按规定范围执行"——用户说"温度调到10度"（范围15.5-30.5），模型该拒绝还是夹取（clamp）到15.5？SP没说，各家实现不一，这是个灰色带。
① 评测集建"幻觉双向集"：越界集（明确不支持的20条）×误拒集（边界没写但工具支持的20条），成对回归；② 超范围参数的 clamp 策略要进工具 notice（"调到10度"→"最低只能15.5度，帮你调到15.5？"），别让模型自由发挥；③ 工具端模糊兜底要排一遍，返回"执行成功"的指令白名单化。

G{{scenario_sample}}：示例槽背后的 token 经济学 你的项目主场

示例为什么必须外置？算第一性的账：
① 成本：69个示例（schema口径）×平均约450 token ≈ 31K，占SP一半；即使21个也约9-10K。示例区每轮全量付费，不管用户问的是空调还是写诗。
② 收益：对当前 query 真正起作用的示例通常只有 1-3 条——边际收益高度集中，边际成本均匀分摊，这是典型的该做按需供给的结构。
③ 缓存：示例在固定区能命中 KV cache，貌似"不算贵"——但 prefix 越长首 token 越慢，且动态变量（goal_list/env_info）一变，cache 断点后的全要重算。固定区的每一千 token 都在为动态区的变更买单。
结论：留 18 个"范式锚点"（跨场景元能力，任何场景都用得上）在固定区，场景精度交给 {{scenario_sample}} 按场检索 2-3 条。固定成本砍半，相关性反升。
（推演）"18个锚点我选得很心疼——每个都得是'删掉它某种元能力就塌'的级别：并行（示例5）、串行节奏（16）、静默（11/12）、合并处理（2）、时间推理（9）、记忆边界（4）、状态纠偏（18）、自唤醒（17）、emoji用法（15）……它们教的不是场景，是决策的语法。场景词汇交给动态槽。"
动态槽在什么 case 下会翻车——这才是你要防的：① 检索错配：用户问"空调"，检索到"座椅"示例——错示例比没示例毒，模型会照着别人的剧本演；② 检索为空：长尾场景没有匹配，槽位空——此时全靠18锚点兜底，锚点的范式覆盖度就是天花板（所以锚点遴选是兵家必争）；③ 注入条数失控：检索返回8条全塞进去，token账白算；④ 示例带毒：一条动作越界的示例（如示例17原版）被检索命中后，污染的是整个场景；⑤ 检索键漂移：按 query 相似度检索，"我好热"和"我好冷"相似度极高但动作相反（制冷vs制热）——相似度键在反义场景上是灾难。
对你97条的五个落地约束（直接可写进动态示例V2规范）：① 每条带场景标签+适用工具+反义禁区三元组做检索键，不只靠 embedding 相似度；② 单次注入上限 3 条，超出按置信度截断；③ 每条过"边界对撞审查"（§A的修法）；④ 建立"检索为空"的埋点统计——空槽场景就是你的下一条 97+；⑤ 拿 §10 的 7 处矛盾当审校 checklist 的种子。
