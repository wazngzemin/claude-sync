# "撞墙-分诊-重排"：从 Pedro 的 AI 电力隐喻中提取的组织重写操作系统

## 核心问题

当AI从"可选插件"变成"基础设施"，一个产品团队/PM该如何系统性地找到人与模型的分工线，避免既不过度放手也不过度管控？

Pedro在这期播客中给出的不是"用AI吧"的态度喊话，而是一套可拆解的操作路径：撞墙训练→暴露边界→重排分工→Token记账→持续迭代。下面把这套路径从CEO场景映射到Planner/SP/Agent PM的日常工作。

## 场景映射

| 文章里的角色/机制 | Planner/SP/Agent 场景 | 可迁移动作 |
|---|---|---|
| CEO每天问"为什么不能用AI解决" | PM每天问"这个Badcase是模型能力问题还是产品设计问题" | 撞墙训练法：把真实badcase喂入模型，区分边界 |
| Free the Claw = 少harness多上下文 | SP维护：69个示例是否过度约束？远区精简是否丢信号？ | 做ablation实验：去掉部分示例/上下文，看质量变化 |
| 一人公司思想实验 = 找人类不可替代的工作 | Planner PM的不可替代职责：分工边界/安全vs体验取舍/bug是谁的 | 列出你过去一周的决策，标记哪些模型能替代 |
| Magpie = token ROI追踪 | KV Cache命中率/上下文token预算分配 | 对SP每个section做"贡献度审计" |

## 方法框架

### 原则一：撞墙训练法——不是学AI，是用AI找边界

Pedro的核心操作：不是读AI报告、不是听播客，而是把模型拉进真实工作流，持续打到极限，直到墙出现。墙才是最有信息量的产物。

**为什么不是"学AI"而是"用AI"？**

学AI给你的是通用知识（"大模型有幻觉""上下文窗口有限"），用AI给你的是具体边界（"Planner在第3轮调用时tool_feedback超过800token就开始丢前文信息"）。前者谁都能Google到，后者只有你亲手撞出来。

**在Planner场景的操作版本：**

每周选3-5个"直觉上觉得模型应该能搞定但搞砸了"的badcase，不是直接归因为"模型笨"，而是逐层拆解：
- 把SP中相关工具描述和示例单独拎出来喂给模型，看它是否理解 → 工具描述问题
- 把完整上下文（含记忆/车况/情景）喂进去，看结果是否改善 → 上下文问题
- 信息全了还是错 → 模型能力问题

这个拆解过程就是"撞墙"——每次撞到不同的墙，你对Planner的边界地图就更完整。

### 原则二：Free the Claw——但要分区执行

Pedro说"少harness多上下文多token"，这在通用Agent场景是好建议，但在安全关键场景需要分区：

| 区域 | 策略 | 例子 |
|------|------|------|
| 安全核心区 | 严格围栏，不放手 | 车控指令（开天窗/方向盘/刹车）|
| 体验优化区 | 适度放手，给更多上下文 | 话术生成/推荐内容/情感回应 |
| 探索实验区 | 完全放手，看模型极限 | 内部测试环境/非客户facing场景 |

**分区依据：** 错误代价。错误代价不可逆（安全）→ 严格管控；错误代价可修复（体验）→ 放宽约束；错误代价为零（实验）→ 完全free。

**当前Planner的疑问：** 69个参考示例占SP约50%，有多少落在"安全核心区"（必须保留），多少落在"体验区"（可以减少靠模型自己判断）？这是一个值得做的audit。

### 原则三：一人公司思想实验——找到你的"不可替代三件事"

Pedro的实验不是鼓吹一人公司，而是用极端假设暴露真正稀缺的人类工作。

**PM版本的思想实验：**

假设你是Planner模块唯一的PM，模型可以帮你：
- 写SP草稿（基于badcase和工具定义）
- 跑回归测试（对比修改前后的输出质量）
- 分类badcase（按六大类A-F自动归因）
- 生成评审材料（数据看板/分类统计）

那你做什么？

**你的不可替代三件事（推断）：**

1. **分工边界判断**：Planner和Advisor谁负责什么？新增的工具应该给Planner直接调用还是走Advisor？这个决策需要同时理解模型能力、产品需求和用户心理。
2. **安全vs体验的取舍**：用户说"我想快点到"，Planner该不该推荐超速路线？该不该调用auto_drive以更激进的方式驾驶？这个取舍不能交给模型。
3. **Bug归因的终审**：一个badcase到底是模型能力问题（需要等模型升级）、SP设计问题（需要改提示）、还是链路问题（需要改架构）？错误的归因导致错误的投入方向，这需要人的判断。

### 原则四：Token记账——SP Section 贡献度审计

Pedro的Magpie系统追踪每一美元token去向。映射到Planner SP维护：

**SP的"token预算表"（概念模型）：**

| SP组成部分 | 估算占比 | 贡献类型 | 审计问题 |
|---|---|---|---|
| ①角色定义 | ~3% | 行为锚定 | 删掉后模型行为变化多大？ |
| ②输入类型定义 | ~3% | 格式规范 | 模型是否已经从示例中学会了？ |
| ③23个工具定义 | ~15% | 能力边界 | 哪些工具从未被调用？可否按场景动态加载？ |
| ④输出格式 | ~3% | 结构约束 | 必须保留 |
| ⑤10条注意事项 | ~5% | 安全围栏 | 哪些是"安全核心区"必须保留？ |
| ⑥69个参考示例 | ~50% | 行为教学 | 贡献最大也最可疑：每个示例的边际贡献是多少？ |
| ⑦车辆设备知识 | ~8% | 领域知识 | 模型是否已经从训练数据中知道D6X规格？ |
| ⑧工具使用tips | ~5% | 调用优化 | 与③重复多少？ |
| ⑨其他知识 | ~3% | 补充 | 使用频率如何？ |
| ⑩聊天风格 | ~3% | 人格设定 | 必须保留 |
| ⑪动态模板变量 | ~2% | 实时状态 | 必须保留（但是否所有变量每轮都需要？） |

**审计方法（SOP见下文）：**
核心问题不是"删哪个"，而是"每个section/每条示例换回了什么质量提升"。

### 原则五：上下文=稀缺资源——"灌什么"比"灌多少"重要

Pedro说"谁能持续补进现场数据，谁就能把通用智能调成自己的系统"。映射到Planner：

通用大模型不知道：
- D6X五座SUV的具体规格（确认来源：Director SP地面真相）
- 当前用户的驾驶偏好和记忆
- 此刻的车辆状态（车速/温度/窗户开关）
- 这段对话的Goal List进展

这些都是"稀缺上下文"——只有你的系统能灌进去。远区精简方案的本质就是在"上下文有限"的约束下做最优分配：保留高信号（用户query、talk_content），删除低信号（远区tool_feedback、action_list），近区完整保留。

**进一步的优化方向（推断）：** 动态判断"这轮对话中，哪些上下文是高信号"——比如用户正在执行导航任务时，导航相关的tool_feedback是高信号应该保留，音乐相关的可以精简。这需要一个比"远区/近区"更细的attention预算分配。

## 操作 SOP：SP Section 贡献度审计

### 输入
- 当前SP全文（~30K tokens）
- 20-30个已有badcase（覆盖各类型）
- 评测指标（正确率/安全率/话术自然度）

### 步骤

1. **基线测试**：用完整SP跑全部badcase，记录各指标基线值。
2. **逐section消融**：依次移除一个SP section（或将69个示例分组移除），重新跑badcase，记录指标变化。
3. **贡献度打分**：
   - 移除后指标下降 >5% → 高贡献，必须保留
   - 移除后指标下降 1-5% → 中贡献，考虑精简但不删除
   - 移除后指标下降 <1% → 低贡献，候选删除
   - 移除后指标上升 → 可能有害，优先删除
4. **交叉验证**：对"低贡献"section，换一批badcase重新跑，确认不是样本偏差。
5. **安全红线检查**：即使某section贡献度低，如果它覆盖安全关键场景（车控围栏），不删除。
6. **产出修改方案**：列出建议删除/精简的sections，预估节省token数和缓存命中率改善。

### 输出
- SP Section贡献度排名表
- 建议修改方案（分"安全保留"/"优化精简"/"建议删除"）
- 预估优化效果（token节省比例、缓存命中率改善）

### 失败模式
- 样本量太小（<20条）：结论不可靠
- 只用简单case测试：低估示例在复杂场景中的作用
- 忽视长尾安全case：删掉的section恰好是低频但高风险场景的围栏
- 一次删太多：无法区分是哪个section的缺失导致退化

### 判断标准
- 审计完成后，能回答"69个示例中，哪30个贡献了80%的质量"
- 修改后的SP在30条badcase上指标不低于基线的95%
- token节省 ≥15% 且无安全退化

## Prompt 模板

### 模板1：每日撞墙训练提问

```text
我是 [Planner/XX模块] 的PM。今天遇到一个badcase：

【场景】：[用户说了什么/什么车况/什么时间]
【期望行为】：[模型应该做什么]
【实际行为】：[模型做了什么]
【已排查】：
- 链路问题？[走对了模块/走错了]
- 工具描述？[相关工具定义清楚/不清楚]
- 上下文？[记忆/车况/情景是否完整注入]

请帮我分析：
1. 如果上下文全部补齐，模型是否能正确处理？（上下文问题 vs 模型能力问题）
2. 如果是模型能力问题，这个能力gap需要什么级别的解决方案？（SP调整/示例增加/模型升级）
3. 这个case暴露的边界，是否意味着还有一类相似badcase？
```

### 模板2：一人公司思想实验（周频自检）

```text
回顾我过去一周的工作清单：
[粘贴过去一周的todo/done list]

帮我标记每项工作：
- 🤖 模型可替代（写文档/跑测试/分类数据/生成报告）
- 🧠 需要人类判断（分工决策/安全取舍/归因终审/跨团队协调）
- 🤝 需要人际关系（沟通说服/stakeholder管理/共识建设）

然后回答：
1. 我花了多少时间在 🤖 上？如果这些交给模型，我能多做多少 🧠？
2. 有没有 🧠 被我当成 🤖 在做（比如手动分类badcase而非建立分诊框架）？
3. 有没有 🤖 我以为只有我能做但其实模型已经能做了？
```

## 检查清单

### "撞墙训练"执行检查
- [ ] 本周是否选了3-5个badcase做逐层拆解？
- [ ] 每个badcase是否区分了链路/工具描述/上下文/模型能力四层？
- [ ] 是否发现了至少一个之前不知道的边界？
- [ ] 发现的边界是否记录到Planner边界地图中？

### "Free the Claw分区"检查
- [ ] SP的每条围栏规则，是否标记了属于安全区/体验区/实验区？
- [ ] 体验区的规则，是否做过"去掉它模型是否依然做对"的测试？
- [ ] 有没有安全区的围栏规则其实是体验区的？（过度保守）

### "一人公司实验"检查
- [ ] 本周是否识别出至少一项"我在做但其实模型能做"的工作？
- [ ] 是否把省下的时间投入到"不可替代三件事"上？
- [ ] 是否更新了自己的"不可替代工作清单"？

### "Token记账"检查
- [ ] 是否知道当前SP的token分布（各section占比）？
- [ ] 是否做过至少一次section消融实验？
- [ ] 是否有一个"示例ROI排名"——哪些示例贡献最大？

## 判断标准

### 做对了的信号

- 你能画出一张"Planner边界地图"：哪些场景模型稳定搞定，哪些场景必须靠SP围栏，哪些场景模型能力目前不够。
- 你的SP修改有数据支撑：不是凭直觉加减示例，而是基于消融实验的贡献度排名。
- 你的周报里"🧠判断类"工作占比在增加，"🤖执行类"在减少。
- 你能清楚回答"如果Planner模块只有我一个人，我会把时间花在哪三件事上"。
- 发现新badcase时，你的第一反应不是"又出bug了"，而是"这是哪面墙？我之前撞过没有？"

### 仍然停留在表层的信号

- 你用AI的方式还是"问一个问题→拿一个答案"，没有把它塞进持续工作流。
- 你的SP修改方式是"遇到badcase就加一条规则"，没有审计整体结构。
- 你说不清Planner的69个示例中哪些贡献最大、哪些可以删。
- 你的"不可替代工作清单"和半年前没有变化——说明你没有持续重跑思想实验。
- 你的撞墙只停在"知道有这个问题"，没有系统记录墙的位置和类型。

## 误读警告

**最容易误读的地方：** 这篇文章最大的误读是把Pedro的"free the claw"当成通用建议。他是SaaS公司CEO，模型出错的代价是客服回答不准、销售提案不完美——这些都可修复。但在安全关键场景（车载AI），"放开模型"如果没有分区策略，一次错误的车控指令可能不可逆。正确的迁移不是"放开所有管控"，而是"分区放开——安全区严管，体验区放宽，实验区全放"。

**第二个误读：** 把"一人公司"当成经营建议而非思想实验工具。Pedro不是在说你应该成为一人公司，而是用这个极端假设来暴露"什么工作只有人能做"。工具的价值在于用完就放下，不在于照着做。
