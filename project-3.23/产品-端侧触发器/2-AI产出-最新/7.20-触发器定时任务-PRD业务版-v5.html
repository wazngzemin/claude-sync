1为什么必须做

评审第一分钟就要把「这事该不该做」焊死，后面才不会被人拿「有没有必要」反复拱。三个视角，从用户、系统、线上证据各来一刀。

1.1 用户视角：他要的是「车替我记着、到点替我做」

用户在车里说的是这样的话：「十分钟后关座椅加热」「明天9点提醒我出发」「每小时开一次按摩」「每分钟看看后排宝宝」。这些话有一个共同点 —— 发出的这一刻不需要马上做，用户是要系统「记着」，到未来某个时间点、或每隔一段时间再做。

往人性那一层挖（Y 模型的根）：人开车时注意力被路占死了，他没有余力时刻盯着温度、盯着时间、盯着宝宝。他要的不是「执行这一条指令」，是「车替我操心」—— 不用我记着待会要干嘛，车在合适的时机自己动。「时间」正是最高频的那个「合适的时机」。

1.2 系统视角：今天的执行模型是「说完就办」，缺一个时间维度的点火器

今天系统的执行模型是「说完就办」：用户说一句话，Planner（规划器）同步推理、同步执行，这一轮对话结束，事情就结束了。可现实是很多需求发出的这一刻就不该执行，而要等到未来某个时刻。

⚠️ 能力缺口讲清楚，研发才知道你在补什么
已上线的条件触发器管的是「车态信号变成什么样」—— 温度高于28度、车速大于30、档位到P。它不管「时间到了没有」。也就是说，系统里「时间」这个维度的点火能力目前是零。本 PRD 就是补这一格。

1.3 线上证据：这批 badcase 的根，就是没有「定时任务」这个载体

这不是设想出来的需求，是线上已经在爆的问题。下面几条都是上汽 DNA 的真实 P0/P1 badcase，它们的病根是同一个 —— 系统没有一个正经的「定时任务」载体，只能用会话逻辑硬凑，于是全线出错：

真实 Query / 现象暴露的问题定时任务怎么根治
P0 定时任务「说完立即执行」该「记着等到点」的事，被当场就办了定时任务先落「等待中」，到点才点火，天然不会立即执行
P0「10秒后关闭副驾车窗」回复会执行、实际任务不创建不执行承诺了、没兑现，用户以为设好了其实没有创建即在任务中心建卡片+登记定时器，创建成功才回复，做不到就明确拒绝
P0 删除了长时任务，但任务还在执行取消没真停，后台幽灵运行「取消＝报意图→后台真停→再消卡」，见铁律⑦
P0 重复回复两次，疑似重复创建了两次任务同一意图叠加出两个任务、两次响判重：同一意图只更新原任务，见铁律⑥
🔥 这页是你最硬的弹药
当有人问「这功能有没有必要、是不是过度设计」，你不用讲道理，直接把这张表推过去：「这四条都是线上 P0，根都是没有定时任务这个载体。不是要不要做的问题，是欠的债。」

1.4 竞争视角：这是 Driver Agent 2.0 甩开传统语音助手的代差

主动服务是 Driver Agent 2.0 相对传统语音助手的代差能力。而「主动」离不开「时间」—— 到点提醒、按周期照看，都是时间维度的主动。没有定时能力，产品在这块就退回「高级语音遥控器」：只能你说一句它答一句，做不到「你交代一次，它替你惦记」。

1.5 目标：补第三种触发类型「定时」，只做点火器

给云端触发器补上第三种触发类型 —— 「定时」。它是一个时间维度的点火器：到点只负责「点火 + 通知」，点火之后干什么（提醒、车控、看画面判断）复用触发器与 Task 服务已有的能力，不为定时单独建一套执行系统。

2定时任务是什么

2.1 一句话定义

定义
定时任务 = 用户发起后，不在当前对话里立刻执行，而是由系统按「时间条件」在未来激活、可能反复执行的任务。它是「任务（Task）」的一种，只是激活方式是「时间到了」，区别于条件任务（车态满足）和语义任务（画面命中）。

2.2 四种时间形态（用生活例子记，别用参数记）

时间形态一句话生活里的例子做几次
相对延时·一次性从现在起，过多久，做一次十分钟后关座椅加热一次
绝对时间·一次性定在某个钟点，做一次明天9点提醒我出发一次
固定间隔·循环每隔多久，反复做每小时开一次座椅按摩反复，直到被停
日历周期·循环按日历规则，反复做每个工作日8点提醒开会反复，直到被停

2.3 核心认知：定时是「点火器」，不是第三个执行系统

这句话你要在评审上第一时间讲出来，因为它一句话挡掉「是不是要建四套引擎/是不是又搞个独立系统」这类挑战。

✅ 一定要讲的这句
「每分钟看下宝宝在干嘛」拆开 = 定时（每60秒点一次火） + 语义（拿视觉描述判断宝宝状态）。
定时不是和「条件」「语义」并列的独立执行系统，它只是一个时间维度的点火器。到点之后干什么 —— 纯提醒、查车况、看画面判断 —— 全部复用触发器已有能力和 Task 已有执行链路。我们不为「定时」新建烟囱。

3和已上线条件触发器的关系

研发一定会问「这跟已经上线的条件触发器什么关系，是不是重复造轮子」。一张对照表钉死：同一套引擎，只是「点火的理由」不同。

维度条件触发器（已上线）定时触发器（本 PRD）
点火的理由车态信号变成某样（温度>28、车速>30、档位到P）时间到了（过了600秒 / 到了9点 / 又过了一小时）
盯的是空间维度：世界的状态时间维度：钟表
点火之后完全一样 —— 回调 Task 服务 → 建 Run → 执行前守卫校验 → 唤醒车端执行 → 按结束策略收尾
登记、判重、上限、生命周期共用一套，不各搞一套
✅ 口径
逻辑上它就是同一个触发器，只是多认一种「点火理由」：以前只认「信号变了」，现在也认「时间到了」。不是新系统，是给老引擎加一个时间时钟。

5八个业务场景

每个场景一条从「用户开口」到「执行完成」的主线，纯业务语言。重点看每个场景末尾的「产品为什么这么定」 —— 那是评审时会被问到的地方。

5.1 场景一 · 一次性（相对延时）：十分钟后关座椅加热

1用户说「十分钟后关座椅加热」，系统听懂：这是到点才做的事，不是现在做
2拆成两件事：什么时候做（从现在起600秒）+ 到点做什么（关座椅加热并语音提醒）
3在任务中心建一张卡片，状态＝待办，写明「十分钟后关座椅加热」
4把「600秒后叫我」交给云端定时器，开始计时，等待600秒
5到点，定时器点火，通知系统「该做那件事了」
6执行前再看一眼：人还在车上吗？座椅加热还开着吗？
情况已变人已下车 → 这次跳过，任务结束，不打扰用户
一切正常唤醒车端，关闭座椅加热并语音「已帮你关闭」→ 卡片进记录，任务结束（一次性不再重复）
产品为什么这么定
一次性任务＝点一次火就结束。中间「执行前再看一眼」是关键 —— 到点时情况可能已经变了（人都下车了还去关加热毫无意义），这种情况直接跳过、不打扰。跳过不算失败。

5.2 场景二 · 一次性（绝对时间）：明天9点提醒出发

1用户说「明天9点提醒我出发」，系统听懂：这是定在某个钟点的事
2拆解：什么时候做＝明天上午9点整；到点做什么＝提醒出发。建卡片（待办）
3交给云端定时器，盯住「明天9点」这个时刻
9点前反悔用户取消 → 定时器撤销，不再提醒
到9点整点火，唤醒车端语音「该出发了」→ 卡片进记录，任务结束
产品为什么这么定
和场景一唯一的区别是时间怎么给：延时是「从现在数秒」，绝对是「盯住某个钟点」。对用户和系统来说，后面的流程一模一样。到点之前用户随时可取消。

5.3 场景三 · 循环（固定间隔）：每小时开一次座椅按摩

1用户说「每小时开一次座椅按摩」，系统听懂：这是要反复做的事
2拆解：每3600秒一次；每次开按摩并告知；下车或取消才停。建卡片，带启用/关闭开关（因为是循环任务）
3交给云端定时器计时，每到一小时点火
4执行前看一眼：座椅上有人吗？
没人这次跳过，不算失败，等下一个小时
有人开座椅按摩并语音告知 → 本次完成，任务不结束，回去等下一个小时
◆随时可能发生：用户关掉开关＝暂停（可恢复）；下车 / 取消＝彻底结束
产品为什么这么定
循环任务靠三种方式结束 —— 下车、关开关、取消。两个细节要讲清：① 这次没人「跳过」但不算失败，下个小时照常；② 「关开关」是暂停（能恢复），「取消」是彻底结束，两者不是一回事。

5.4 场景四 · 定时 + 看画面（循环）：每分钟看后排宝宝有没有哭 二期

1用户说「每分钟看下后排宝宝有没有哭」，系统听懂：这是定时 + 看画面判断的组合任务
2安排两件事同时进行：① 车上摄像头持续描述后排画面（每十几秒报一次，如「宝宝睁眼」「脸上有泪」）；② 云端每60秒定时，取一次最新描述来判断
3每60秒到点，取最新画面描述，判断「宝宝哭了没」—— 结果三选一
看不清这次不算数，不更新记忆，等下一分钟
没哭记住「现在是没哭」，等下一分钟
哭了再看上一次 —— 上一次没哭吗？
  · 是（刚从没哭变成哭）→ 提醒一次，播报「宝宝哭了」并给安抚建议，记住「现在是哭了」
  · 不是（上一次就在哭）→ 不重复提醒，等下一分钟
🔥 这段最值钱，一定要讲透
为什么只在「从没哭变成哭」那一刻提醒？因为如果每分钟只要「哭了」就提醒，宝宝哭5分钟你会被烦5次。产品规则是抓「状态跳变」那一下，提醒一次就够；同一状态持续不重复打扰；「看不清」不更新记忆，防止一次误判打乱节奏。
这个规则不只用于看宝宝 —— 所有「持续满足的条件」都用它防止反复提醒。

5.5 场景五 · 用户取消（语音 / 卡片两个入口）

A用户想取消「每小时座椅按摩」，两个入口：语音说「取消」/ 在任务中心卡片上点取消按钮
B关键：卡片点取消，任务中心只是把「用户想取消」这个意图报上去，自己不擅自删卡片
C后台停掉计时器 → 结束当前这一轮 → 任务标记为「已取消」
D后台真停干净后，才通知任务中心把卡片移除 / 进记录
产品为什么这么定
点「取消」不是卡片自己消失 —— 是「报意图 → 后台真停 → 再消卡」。这正是线上 P0「删除了长时任务但任务还在执行」的解法：卡片先消失、后台还在跑，是最糟的体验。

5.6 场景六 · 上下电：下车、上车，任务怎么办

1车辆下电、用户下车熄火。逐个看还在等的定时任务：这是本趟车内的事，还是跨天的事？
本趟车内如「十分钟后关加热」「每小时按摩」→ 人都走了，做了没意义 → 任务结束，下次上车不自动恢复，用户要就重新说
跨天跨行程如「每个工作日8点提醒开会」→ 定时器在云端，不受下电影响，继续计时。若下车期间正好到点：车不在线这次做不了，跳过、不补做，等下一个到点时间，下次上车照常提醒
产品为什么这么定
定时器在云端，所以下车不会把跨天任务弄丢（这是放云端的核心好处）。区别只有一条 —— 本趟内任务下车即结束，跨天任务继续等，错过那次不补。

5.7 场景七 · 超上限：任务太多了

1用户又想加一个定时任务，先数一下：现在的定时任务到上限了吗？
没到正常创建
到了不新建，也绝不挤掉老的 → 语音告诉用户「任务已达上限，需先完成或删掉一个再加」→ 用户去任务中心删掉不要的，再来加
产品为什么这么定
满了就明确报错，绝不偷偷挤掉老任务 —— 否则用户之前设的「接孩子提醒」被无声顶掉，就是安全事故级别的体验问题。宁可让用户手动删，也不能系统擅自挤。

5.8 场景八 · 判重：用户重复设置同一个任务

1用户又说了一遍「十分钟后关座椅加热」，之前已经设过一个
2系统发现已经有一个一模一样的任务 → 不新建第二个，用新的时间/内容更新原来那一个
3还是一个任务、一张卡片，不会冒出两张卡、两个定时器同时响
产品为什么这么定
重复设置只更新原任务、不叠加。用户以为自己在「改」，不该冒出两张卡片。这正是线上 P0「重复回复两次、疑似重复创建两次」的解法。

6产品定死的八条铁律

把上面八个场景里的产品决策拎出来，一条一条钉死。每条都配「为什么 + 不这么做会怎样」 —— 这是评审现场最核心的弹药，别人挑战哪条，你就把这条的理由甩出去。

① 一次性点一次火就结束；循环靠「下车 / 关开关 / 取消」三种方式结束
循环任务没有自然终点，必须有明确的停止手段。不这么做：循环任务永远停不下来，成了甩不掉的幽灵。
② 执行前再看一眼：到点后、真执行前，还要看当下情况，情况变了就跳过（跳过不算失败）
「定时」定的是时间，但「该不该做」要看到点那一刻的现实。不这么做：人都下车了还去关加热、座椅没人还去开按摩，全是无意义打扰。
③ 只在「状态跳变」那一刻提醒：从不符合变成符合才提醒一次，持续符合不重复；看不清不算数
用户要的是「事情发生了告诉我」，不是「这事一直存在所以一直烦我」。不这么做：宝宝哭5分钟被提醒5次，用户直接关掉功能。
④ 定时器放云端，下电不丢跨天任务；本趟内任务下车即结束，跨天任务继续等、错过不补
「每工作日8点提醒」这种跨天任务，端上熄火就没了，只有云端能扛住。不这么做：用户设的通勤提醒，下班熄火就丢，第二天不响。
⑤ 满了报错不挤旧：到上限就明确告诉用户去删一个，绝不系统擅自挤掉老任务
老任务里可能有「接孩子提醒」这种要命的事。不这么做：无声顶掉关键提醒＝事故。
⑥ 重复设置只更新原任务、不叠加，不冒出两张卡两个定时器
用户重复说＝想改，不是想要两个。不这么做：线上 P0 —— 重复创建，两个定时器同时响。
⑦ 取消＝报意图 → 后台真停 → 再消卡，不是卡片自己擅自消失
卡片消失≠任务停止，两者必须严格对齐。不这么做：线上 P0 —— 删了任务后台还在跑。
⑧ 用户只管说人话：「十分钟后」变600秒、「明天9点」变具体时刻，这种翻译由系统做
用户不该被要求说结构化指令。注意分工：这个「翻译」由 Task 侧的规格构建器做，触发器永不理解人话（见第7章）。

7产品说了算 vs 研发说了算

这一章是你在评审上不被带偏的护栏。研发拿技术细节挑战你时，先分清这件事到底是「产品决策」还是「技术实现」—— 是前者，你说了算，讲理由；是后者，你把口径给出来，实现方式交给他们，别硬答自己答不了的。

议题归属产品口径（你必须能张口就说的）
要不要做定时、做哪几种形态产品做四种，一期做前三种。理由见第1、4章
到点后「执行前再看一眼」、状态跳变才提醒、满了报错不挤旧……产品八条铁律，逐条有理由，见第6章 —— 这些不接受技术方便性理由被改
数量上限是几产品产品给数字（暂定定时6 / 定时+语义1），研发只反馈「扛不扛得住」，见 Q6
定时器放云端还是端侧产品 定诉求产品要的是「下电不丢跨天任务」；据此放云端。技术怎么实现云端定时器，是研发的事
「十分钟后」怎么变600秒研发 实现产品只要求「用户说人话就行」；在哪个模块解析、怎么解析，是技术实现，你不用答
时间规则用什么字段/协议传给触发器研发 实现产品只定义「四种形态 + 各自要哪些参数」；塞进哪个协议字段，是刘杨、庆磊的技术评审内容，见 Q7
定时引擎内部是不是四套调度研发 实现产品口径是「对用户是四种形态，对系统是一个点火器」；内部怎么归一化调度，研发定，见 Q1
回调失败重试几次、连续取不到画面多少轮报错研发 建议产品给原则（周期任务不因单轮失败注销）；具体次数听研发的稳定性建议
🛡️ 一句话护身符
被问到技术实现细节、你答不上来时，不要硬编、不要脸红，就说：
「这是实现方式，我不越界替你们定。产品口径是 ___，只要满足这个口径，具体怎么做你们最专业。」
这不是心虚，这是边界清楚 —— 产品经理答不出「用哪个字段」很正常，答不出「为什么要做这个规则」才是出丑。

8研发会怎么挑战你，你怎么答

这些是从过往几场真实评审里，研发追问的高频模式。每条给你：他在担心什么 → 你张口能说的答案 → 底层道理。评审前把这章过一遍，同样的问题砸过来你就能接住。

Q1 · 是不是要做四套定时引擎？四种形态各一套？刘杨 / 庆磊
他在担心：四种形态＝四套调度逻辑＝四倍工作量和维护成本，架构会烂。
你答：不是四套。对用户是四种说法，对系统只有一个动作 —— 算出「下一次该点火的时刻」。四种形态的差别，只是「下一次点火时刻怎么算」不同：延时＝现在+600秒，绝对＝给定时刻，间隔＝上次+一小时，日历＝按规则算下一个。算完之后，「到点、点火、通知」完全同一套。所以是一个引擎、四种算法，不是四套引擎。
底层道理：把四种形态归一化成一个「下一次点火时刻」，是最省事也最不容易出错的做法。你不用替研发设计这个归一化，但你要知道「对系统只有一个点火动作」这个口径，才不会被「四套」这个词吓住。
Q2 · 定时器为什么放云端？放端上不是更准、断网也能跑吗？研发常问
他在担心：云端掐表有网络延迟，车在地库没信号怎么办，端上本地定时器不是更靠谱？
你答：要分清是「本趟车内的短定时」还是「跨天的长定时」。跨天任务（每工作日8点提醒）如果放端上，用户一熄火下电，计时就断了，第二天不响 —— 这是硬伤。放云端，下电期间照常走时，无需端侧持久化和上电重建。而且定时到点走的是触发器已有的回调路，不新建通道。所以本期先把云端定时做扎实，端侧本地定时器（断网倒计时那类）作为后续能力，不在本期。
底层道理：「放云端」不是拍脑袋，是被「下电不能丢跨天任务」这个产品诉求倒逼出来的。你守住诉求，实现细节让研发接。
Q3 · 每分钟看一次宝宝，1万辆车持续过模型，云端顶得住吗？庆磊（成本质疑）
他在担心：1万辆车 × 每分钟一次判断 = 天文数字的模型推理，烧钱烧到项目做不下去。
你答：不是每次都过大模型。判断做成三层漏斗：
· L1 死规矩能判的（情绪==哭、档位==P、温度>30）—— 直接比较，零模型，挡掉约80%；
· L2 抠关键词能判的 —— 正则/小模型从描述里抠「哭」字，成本极低，挡掉约15%；
· L3 真正开放长尾的 —— 才过一次云端大模型，而且限流：一分钟最多判一次，约5%。
算下来 1万辆 × 每分钟1次 ≈ 167次/秒，这个量扛得住。
会有人接着怼（庆磊/刘杨）：「判断挪到触发器，不还是云端推理？一样顶不住。」
你接：对，挪模块不省钱，省钱的是分层+限流，不是把模型挪给谁。所以我不主张「全过模型」，我主张「能不过模型的坚决不过」。这个账我认。
Q4 · 这跟已经上线的条件触发器什么关系？是不是重复造轮子？研发常问
他在担心：又搞一套新东西，跟现有条件触发器功能重叠、维护两套。
你答：不是新轮子，是给老引擎加一种「点火理由」。条件触发器认「信号变了」，定时触发器认「时间到了」。点火之后的路——回调、建Run、守卫、执行、收尾——完全共用，登记/判重/上限/生命周期也共用一套。见第3章对照表。
底层道理：逻辑上它就是同一个触发器多认一种理由。你把「同一套引擎」这个口径立住，就挡掉了「重复建设」的指控。
Q5 · 定时器到底归谁？之前不是说 Advisor 自己注册定时器吗？历史遗留
他在担心：归属反复横跳，最后又要返工。
你答：这事已经定论、不再讨论了。演进是这样：过去 Planner 一人包揽（「十分钟后」类需求直接丢失）→ 5.29 曾设想 Advisor 自建定时器（结果是条件走触发器、定时走Advisor，两套点火系统，判重/上限各搞一套，割裂）→ 7.16 刘杨口头认领到触发器 → 7.20 庆磊的 Task 方案白纸黑字：Trigger 负责「延时、周期、确定性条件和VLM语义匹配」，且后续不再有动态Advisor这个概念。所以定时器归触发器已是既成事实。本次评的不是「放哪」，是「触发器侧定时能力做成什么样」。
底层道理：统一归触发器＝收口，避免两套点火系统。归属这题别再让人翻出来重议，直接指向庆磊方案的分工表。
Q6 · 上限凭什么是6？这个数字哪来的？庆磊 / 刘杨
他在担心：产品拍脑袋定数字；而且庆磊技术方案里是「单一全局上限」，跟产品分类型的口径冲突。
你答：数字是产品的决定，当前暂定定时任务6个、其中定时+语义1个（跟条件任务对齐）。这个具体值和「分类型限制 还是 单一全局限制」的口径，跟庆磊有冲突，是本次要当场对齐的开放问题（见Q）。原则不变：满了报错、绝不挤旧。数字可以调，规则不让步。
底层道理：上限是产品对「用户认知负担 + 端侧并发能力」的判断，产品给值、研发反馈可行性，不是研发单方面定一个全局数就完事。
Q7 · 时间规则用现在的协议怎么表达？现有的运算符里没有「延时」「间隔」啊刘杨 / 韩杰（格式质疑）
他在担心：现有条件协议只有「大于/等于/包含」这类运算符，没一个能表达「延时600秒」「每隔一小时」，格式怎么定、定完别再改。
你答（诚实且守边界）：这确实是个待定义的空白，而且正是本次评审要解决的。产品这边负责给口径：四种形态分别要哪些参数（延时给秒数、绝对给时刻、间隔给周期、日历给规则）。至于把这些参数塞进哪个协议字段——放执行策略字段、还是新增一类时间条件——那是你和庆磊技术评审要定的。我不替你们选实现，但我保证产品口径先给全，不让技术方案替产品做决定。
底层道理：这题一半产品一半技术。你答「四种形态要哪些参数」（产品），不答「用哪个字段」（技术）。这也正是本 PRD 要赶在庆磊第二次技术评审之前过的原因——协议表达那场会，得先有产品口径垫底。
Q8 · 定时+看画面，一个自然语言描述（caption），触发器规则怎么用？韩杰 / 蒋文
他在担心：触发器要判「条件成立没」，得有确定字段（门==开）。给一坨自然语言「门看起来开着」，规则没法写；今天定的格式明天VLM一改全得跟着改。
你答：不硬拿自然语言描述做判断，分两种走：① 高频、必要的（位置、性别、睡没睡、门开没开）—— 精调成枚举字段，直接「字段==值」判，规则稳、不怕改；② 长尾、临时的（宝宝毯子掉没掉）—— 才落自然语言描述，走抠关键词/小模型的慢通道，不塞进你的规则引擎。你要的结构化字段，高频那批我保证给。
底层道理：caption 和枚举不是二选一，是分流。要写死规则的给枚举，偶尔看一眼的才用 caption。硬拿 caption 写规则确实不合理——这点研发是对的，我的方案就是不那么干。
Q9 · 定时引擎的活，庆磊排期表里根本没有，谁来做、几个人日？评审现场必提
这是你要主动提的，不是等着被问：庆磊排期表合计 Task Service 24人日 + GUI 5人日，里面没有触发器侧定时引擎这个工作项。
你说：定时引擎（四形态调度、到点回调、跟适配器对接）是刘杨侧的新增工作量，现有排期表没覆盖。「定时跟任务中心同版本上线」是我在7.16会上三次承诺的，要兑现，这块人日必须当场补进排期，否则承诺会落空。
为什么要主动提：排期缺口如果评审时没人提，上线时就是产品背锅。你先把它摆到台面，是把风险显性化，不是给自己找麻烦。
