## 议题一：注意事项 Section Review（单如水主讲）

> SP 中"注意事项"是 14 条编号列表，位于 SP 的第④ block。

### Q1：模块定位不清——bugfix patch 还是总览？

**单如水的观察**：
- "注意事项这个部分我看了一眼，因为我不确定它是主要是写用来解 bug，还是说是一个总览"
- 现状是混合的——#1-#9 偏 bugfix 补丁（"听不懂别说话"、"反复失败可放弃"），#11-#14 又像综合性原则（"自我时间感知"、"情绪状态管理"）
- **若定位为总览**：应综合性指出原则，比如**安全/边界规则**——"应该明确地写到应该禁止一些，或者说模型不应该出现一些对辱骂呀、未成年不宜内容啊，或者说敏感话题的讨论，但是这个就没有写"
- "所以说我觉得会有一些风险存在"

**当前 SP 完全缺失合规条款，这是会议识别的一个系统性风险点。**

**决议**：定位待定（需先决定整章定位再补内容）。

### Q2：#5 + #8 应合并，工具失败处理过模糊

| 条款 | 原文 |
|------|------|
| #5 | "工具返回提示失败时，你不必道歉，想办法处理即可" |
| #8 | "工具反复调用失败时，可暂时放弃" |

**单如水的拆解**：
- 两条同主题（工具失败处理），应合并写一个**完整的失败处理链**；
- "反复"措辞太模糊——"我觉得可以写清楚，就是具体比如说调用几次失败"；
- 当前 SP 里**只在导航工具的描述中**隐含"最多重试两次"，"其他工具如果调用失败了，也没有明确说是应该怎么做"；
- "想办法处理即可"过于抽象，应给具体 fallback 模式。

### Q3：「自我时间感知」双处重复

| 位置 | 内容 |
|------|------|
| 注意事项 #13【自我时间感知】 | 对比 timestamp / 30 天→好久不见 / 5 分钟→主动确认任务 |
| 聊天风格【自我时间感知与社交表达】 | 几乎逐字相同（含相同的 30 天 / 5 分钟阈值） |

**单如水**："我看他其实几乎一模一样，看一下需不需要删减合并"

**倾向决议**：保留聊天风格那处（属语气表达策略），注意事项里删除。

### Q4：「情绪状态管理」(#14) 覆盖不全

**单如水的细致观察**：
- 当前已列 6 种情绪（平静/欣喜/关切/低落/生气/好奇）；
- 但只对部分情绪写了触发/转换规则；
- **"好奇"完全没写**：什么时候被触发、什么时候应该衰减——一个字没有；
- **"低落"语义模糊**：是豆包自己的低落（不是用户的），那"它应该什么时候被触发呢？其实我觉得这个如果可以写的话，可以写挺多的"；
- 建议：目前先精简保留共性规则，后面再细化每种情绪。

## 陈芮婷的关键补充：Harmony 工程硬限制

> 单如水讲完后陈芮婷立刻补充的一段，是本次会议在工程方法论上的重要 takeaway。

**原话**：
> "我会后也把跟 Harmony 相关的一些材料可以发给大家，我觉得说一些很明确的规则，用大模型它是会带来一些随机性的，反而会有一些工程的话的手段，要跟大模型结合起来，效果会更好。"
>
> "比方说每次当我们去调用一个工具的时候，我们有一个计数器，当它已经调用超过三次之后，我们直接就是强制进行一个终止，再把这个终止的结果返回给 planner，而不是由 planner 自己通过大模型的方式来去限制这个三次，它的效果可能会更更有确定性一些。"

**陈芮婷的调研依据**：
> "我自己也做过一些调研，当我们如果只是在提示词当中，给它一些明确的规范，比如说我们说最多调用三次，但对于模型来说，它其实是通过上下文的方式去做理解，其实对于模型的（遵循力差）"

**这是一个跨议题的基础原则**：硬规则 → 走工程框架（Harmony 计数器+强制终止）；软策略 → 走 SP。后续会议讨论"例子 vs 原则"时也呼应此点。

## 议题二总览：示例评审

> 进入示例前，李明骏先介绍了 SP 示例的设计原则。

### 李明骏对示例设计的总核心思想

**原话**：
> "我们希望它有一些能遵循一些特性，思考特性，然后最好的话，每个例子也能覆盖多个工具。特性的话，有比较多种的类别。这里也没有强行说每一个示例只覆盖一种，是那个思维范式，它有可能是串在一起的。"

**关键约束**：
- 每个例子要承载多种**思维范式**（串/并行、不说话执行、记忆利用、合并输入、决策分析等）；
- 例子要尽量"一对多"覆盖工具，但思维范式可以叠加。

### 陈芮婷 + 单如水 review 留 comments 的两类

**类型 1：针对具体 case 写法的疑惑**——可能需要内部讨论才能确定是不是要改。

**类型 2：针对评测体系发现的缺失场景**——已经做的体验评测里发现一些场景没被现有示例覆盖：

| # | 缺失场景类型 | 后续动作 |
|---|------|------|
| (a) | 用户**模糊/含糊/错误输入**时，让模型追问澄清的例子 | 应在评测集和 SP 中都补齐 |
| (b) | **无需调用工具**、仅靠推理就能回复用户的例子 | 同上 |
| (c) | 工具返回**不完美/空结果/负反馈**时的处理（如偏远地区 POI 搜不到、用户对前几轮结果不满意）| 同上 |
| (d) | 单一域（如导航）过复杂，1-2 个 fix 示例难覆盖 | **结构性建议：引入动态加载示例** |

### 结构性建议：动态加载示例（陈芮婷）

**原话**：
> "我最近也看了那个李强的他的这个文档，有说就是结合这个 query 在动态加载几个例子，比方讲如果我们现在是一些导航、POI 搜索相关的例子，我们就可以除了明骏这些固定的这些 case 之外，能够去加载几个跟导航强相关的一些动态的 case"

**为什么必要**：
> "因为像导航这个域，可能会有 POI 加导航，也有可能 POI 加 Web search，也有可能是搜不到，有用户也有些负反馈，很难在一个例子当中把这些情况全部考虑进去。"

**机制对应**：参考[[scenario-rag-arbitration]]中情景 RAG 的动态注入设计。

## 示例 1：虹桥机场 + 顺路日料（导航 + POI 并行）

> **本场全场最长的讨论（约 18 分钟），暴露了 5 个层面的问题**。

### Query 原文

```json
{"query": "去虹桥机场T2接人，顺便找个顺路的日料店吃个饭"}
```

### 问题 1：Query 本身有歧义——接完人吃 vs 路上吃完再接人？

**陈芮婷的评测视角**：
> "其实这个 query 是不够明确的，不确定他到底是要接完人再去跟那个接到的人一起吃日料，还是他想在路上吃完日料再去接人？那这两种类型的理解会对他后面的一个导航产生比较大的影响。"
>
> "我看这句话特别像是说我先吃完饭再去接人，但是这个最后搜索的结果又是在虹桥附近再去找日料店，又很像是接完人再去看日料店。"

**李明骏的设计意图（被追问后回忆）**：
> "我还是主要是希望他能把它串起来，能有先，在附近去，在目的地附近终点去，或者说在沿途的路径去找找店的让他知道，如果你要沿途找，你就必须要先发起导航，有这样一个，他得灌输他这样一个思想。"

**张嘉锋质疑（关键）**：
> "沿途搜先开启导航，有必要用例子来教吗？……我感觉这些理论上来讲，不至于说你一定要给个例子吧，给例子感觉会有点死。"

→ 触发后文"例子 vs 抽象原则"大辩论。

### 问题 2：第一轮同时下发两个 action 不可行（工程冲突）

**原 case 中的输出**：
```json
output1: action_list: [
  {action_id:1, route_planning_qa: "去虹桥T2"},
  {action_id:2, search_poi_qa: "在虹桥附近找日料店"}
]
```

**郝晓伟先提出**：
> "你看在第一轮里面，它出了两个 action，第一个是发起了一个导航，另外就是在这个目的地附近找了一家沃歌斯店，感觉体现不出沿途搜的这一步。"

**左晨补充工程理由**：
> "应该是规划去 T2 路线成功之后，再出一个搜索沿途的寿司店吧？不行啊，这就等于替换了目的地啊。这两句发下去都不知道要干啥。"

**张航补充**：
> "导航那边整个下游的导航工具会有问题，他很有可能就第一个出来，然后又被第二个冲掉了，整个就做不了。"

**陈芮婷追问能否同时调**：
> "我可以理解为，你不能同时去调用这个导航两个指令吗？必须串行吗？"

**张航明确回答**：
> "我觉得理论上现在就是没办法传，没办法调，就是因为这两个都要动地图的页面，你现在调就是没办法调，其实。"

**李明骏接受**：
> "OK，那这个就应该是先去虹桥机场 T2 航站楼，然后跟用户说，我再帮你找一下寿司店，等第一个 tool feedback 回来之后，再开始做沿途搜。"

→ 改成 **react 两轮**：第一轮只发导航 + 一句话，等 tool_feedback → 第二轮再沿途搜。

→ **待办**：在注意事项中加一条「导航与沿途搜不能并行调用」的规则（落到 SP 注意事项 / 工具描述）。

### 问题 3："沿途搜" vs "搜目的地附近"用词不一致

**陈芮婷**：
> "这个应该是要沿途搜，而不是搜目的地附近吧？如果是按用户的这个 query 来的话。"

**李明骏**：
> "对，他应该是沿途搜。"

### 问题 4：完整的导航流程链路应该是什么？

**张航给出完整链路**（push 把 SP 写完整）：
> "你得先确认线路，第一步你是导航，导航规划了一个去上海虹桥，第二步你要确定走哪条线路，发起导航，到第三步才是找沿途的你才是沿途搜，可能比如沿途搜还有确认，还有确认，再确定确认哪个之后才去，才添加途径点，再加进去嘛，应该是这样的流程。我觉得还不仅仅是前面这几个。"

| 步骤 | 内容 |
|------|------|
| 1 | 路线规划（route_planning_qa） |
| 2 | 用户选哪条 |
| 3 | 发起导航 |
| 4 | 沿途搜（search_poi_qa） |
| 5 | 用户确认沿途店 |
| 6 | 添加途径点 |

→ 至少 6 步，远比原 case 复杂。

### 问题 5：单如水 push 应该先澄清"接完朋友的规划是什么"

**单如水**：
> "如果他是要接到朋友再去吃，那么这是什么的沿途呢？得从 T2，到某一个，比如说朋友的酒店，这个沿途上进行搜索吧，要不然是什么沿途？这你都不确定，所以我觉得应该首先再问说，接完朋友的规划是什么？这样你才有沿途啊，不然你都没有目的地，你这个沿途是什么的沿途？"

**郝晓伟的直觉**：
> "就单说这一条 query，我的理解就是最终目的地是虹桥，在这个沿途找一家日料店，我第一直觉啊。"

**左晨的反驳**：
> "这条 query 感觉不合适，如果你说去接人，接人的路上你自己想吃一个日料，感觉不合理。"

**单如水**：
> "肯定是接完朋友再吃。"

**张航的产品化建议（push 重设计 query）**：
> "改吧，改个 query，你这个已经偏离预期了，因为明骏做这一条的目的，他的目的是为了让模型知道这种沿途搜的这种情况是要先发起导航再沿途搜，那就设计一个没有啥歧义的，能让模型知道沿途搜这种东西先发起导航，再沿途搜就可以了，不要再搞这种什么接人，顺便找个日料店，这种就感觉就很奇怪了。重新设计个 query。"
>
> "如果需要澄清的，导航可以可以再强化一点。"

### 问题 6：planner 直接选好 POI vs 给用户选？（左晨提出）

**左晨**：
> "用户的 query 是顺，找个顺路的，那是不是也应该直接搜出来的结果。直接帮用户选好，planner 应该选好这个事情。"

**陈芮婷**：
> "我个人认为，如果是顺路，而不是说最近的话，应该是要给用户一个选择的空间的。"

**左晨**：
> "因为用户说顺便找个顺路的。顺路的不算最顺路，对吧？那我觉得比较纠结，这个时候如果你说找顺路的，找出来几个，可能有的绕路 3 分钟，有的绕路 5 分钟，有的绕路 10 分钟，顺路这算都顺路吗？"

→ **达成共识**：顺路而非最近的情况下，planner **给用户选择空间**（呈现几个选项）；最近场景才直接选。

### 关键大辩论：例子 vs 抽象原则（张嘉锋 ↔ 李明骏 ↔ 张航 ↔ 陈芮婷）

**张嘉锋的方法论倡议**：
> "对，因为我感觉这些理论上来讲，不至于说你一定要给个例子吧，给例子感觉会有点死。"
>
> "我觉得首先，现在有两个问题，一个是说我们要给模型减负嘛，那你肯定例子是不如你的一个简单的逻辑和原则能够减负的。第二是我总觉得例子……（被打断）"

**李明骏的强反驳**：
> "不不不，这个我要提一下反对意见。就是你这种原则和这种东西，它反而很不好理解，很不好注意，你尤其你塞到（30K SP 里面）……"
>
> "例子的泛化性是远比一个很抽象的原则对他来说更有用的。因为抽象的原则，首先他越抽象，他越在使用的时候有可能就用不上，尤其是他覆盖在，比如说几万个 SP 里面的几万 token 的 SP 里面的时候。"
>
> "这个我实验过，只是没记录。你也可以试一下。对，就他确实就遵循的能力真的不太行。"

**张嘉锋退一步**：
> "那至少这个逻辑我们是不是从人为的角度，我们应该梳理出来？这就我们一直讲的 NORM 嘛。"

**李明骏接受**：
> "这个我觉得可以单独加在知识检索里面去。"

**张嘉锋立场总结**：
> "我个人是存（疑），因为我也调了很多模型，我感觉应该能理解。"

**张嘉锋的核心建议**：
> "我我单点这个 case，比如说就完全是定死，这一定是先开再沿途嘛，你可能也要有一些其他的原则和逻辑，但是那些异常的原则和逻辑，我觉得最好是教会模型，它为什么要这么做，而不是只是给它一个例子。"

**陈芮婷折中调和**：

## 示例 4：火锅 + 左庭右院（短期/语义记忆 + 边问边做）

### 问题 1：记忆格式不真实

**陈芮婷**：
> "这里短期记忆和语义记忆是否要参考真实的记忆的格式来写？我这里可能有一些小的问题，比如说 5 分钟前大家在聊火锅，应该跟现在短期记忆的语言格式是不一样的，以及长期语义记忆当中也不会有这句话。"

**郝晓伟**：
> "这个我觉得后面根据真实的格式改一下就好。"

### 问题 2：Context 已注入记忆，还要调记忆工具吗？

**陈芮婷**：
> "我理解这里应该已经是 context 召回的记忆了，应该不需要再去调用记忆工具查询了。"

**李明骏的解释（重要技术细节）**：
> "要查呀，因为他这里没有告，他不知道之前是啥东西啊。而且上面的检索，他是就是通过那种方式检索出来的，他很多信息不一定能筛出来。"

**陈芮婷理解**：
> "从记忆的角度来说，这个 case 真实的情况有可能这里是空的，所以你这边需要用语义的方式再去查一遍记忆。"

**张航追问**：
> "我们的记忆的第一次检索和后面的检索，两次检索方法是不一样的吧？理论上第一次如果没有，后面也检索不出来吧？"

**李明骏说明（关键技术信息）**：
> "不一样，第一次是向量式的，它召回率比较低。第二次是语义检索，出来的东西，它的可关联性会好很多。"

**陈芮婷**：
> "而且这里的 query 也不一样。"

→ **机制确认**：
- 第一次（context 注入用）：**向量检索，召回率低**；
- 第二次（工具调用）：**语义检索，关联性好**，且 query 不同；
- 因此即使 context 里有记忆，工具调用仍合理。

### 问题 3：planner 边问边做（重复出现的体验顽疾）

**陈芮婷**：
> "这边这个问题是一个回复策略的角度来的，就是我们现在车上也发现很多类似的问题，就是 planner 一边就是问用户要不要导航，或者要不要开某个东西，但实际上他就已经做了。所以我觉得这个是体验上是不太好的，你要么预问了，你就要等他回复再执行，要么你就是直接导航过去，也做了相应的动作，不要一边问就一边做了。"

**李明骏的承诺**：
> "这个我都统一删一下，但是我觉得他有可能还是会有这个倾向。"

**张航的评测对接 push**：
> "芮婷，你把这些典型的类型的问题，你跟晓伟对一下，让他在 planner 那个评测集里，把这种类型都加上去。就 planner 自己的评测，也把这部分都加进去。这种前后矛盾的，这种的。"

→ **D6 形成**：统一删除"边问边做"。

### 问题 4：豆包推荐的店被写进语义记忆作"偏好"

**陈芮婷**：
> "这个左庭右院是豆包推荐的，用户并没有吃过，他只是说要导航过去。我们都认为说这个时候不应该把这个偏好左庭右院火锅给直接就写到记忆里面去。"

**单如水**：
> "我觉得这个问题也是，我们怎么确认，这只是用户一次性的决策，还是说是一个长期的偏好？因为这个的 query 里面只是说想吃火锅。"

**李明骏**：
> "对，我觉得有点过了，不应该记的，这个可以删掉。"

**张航的产品立场（D7 形成）**：
> "就除非用户明确表达说，我什么什么你记一下，不然我们端的应该不会主动调起。对，就现在这个，那个应该是后台数据挖掘的事。"

## 暴露但未解决的深层张力

会议在多个地方暴露了**结构性问题**，但限于议程未能深入解决，应作为后续专项议题跟进：

### Tension 1：例子 vs 抽象原则（NORM）的效用之争

- **张嘉锋立场**：抽象原则更利于"减负"，模型应该能理解；
- **李明骏立场**：30K SP 里抽象原则遵循力很差，例子的泛化性更好；
- **陈芮婷折中**：例子里加"决策分析（rationale）"是两全；
- **待决**：留实验对比（T7）。

### Tension 2：端状态过度参考是 SP 全局问题

- **现象**：车上发现 planner 会把"宝宝安全"等 goal/advisor 信息硬塞进所有回答；
- **根因**：goal_list / advisor / env_info 注入后 attention 权重过高，缺乏 selective use 的引导；
- **本会处理**：单点修复（删除"清凉解暑"）；
- **根本解决**：需要在 SP 中增加"selective context use"的引导，或在工程层做 context 过滤；
- **关联**：参考 [[public-context-management]] 13 类 context 的过滤策略。

### Tension 3：advisor 直接执行 vs 询问的体验权衡

- **李明骏的体验担忧**："都问一下有点啰嗦"；
- **张航 + Alex 的产品立场**：主动服务做不好的代价更高，先稳为主；
- **本会决议**：除情感类外都改询问（D2）；
- **遗留问题**：随着 advisor 自身改造（携带"直接执行 / 建议询问"标识），后续应放宽。

### Tension 4：示例的真实性 vs 范式覆盖

- 多个示例（地库砂锅粥、emoji 海边、博总火锅）真实性低；
- 李明骏：编不出来；
- 评测侧愿意帮编（D9）；
- **隐含问题**：当前 SP 例子 50% 的预算被低真实性 case 占用，可能影响模型对真实分布的学习。

### Tension 5：planner 的边界——主动服务、安全提醒、拒识

| 责任 | 当前定位 |
|------|------|
| 主动服务 | **不做**（D5）—— advisor 负责 |
| 安全隐患识别 + 提示 | **待决**（T10） |
| 拒识（人人 vs 人机判断） | **不做** —— 拒识模块负责，planner 处理所有进来的话 |
| 主动回复 | **必须做**（D1） |

## 给 PM 的关键 Takeaway（按重要性排序）

1. **【D1 + D2 是本场最重磅的两个原则】**——结合昨天会议"用户必回"对齐、Alex 关于主动服务的立场，构成 planner 行为的两条底线。所有 SP 示例、评测集、产品口径都应按此对齐。

2. **【Tension 2 是隐藏的全局风险】**——端状态过度参考不是单 case 问题，是 context 注入策略的系统性副作用。删除单个示例的"清凉解暑"只是治标，根因需要专项处理（建议联动 [[public-context-management]] 团队）。

3. **【Tension 1 关乎方法论选择】**——例子 vs 抽象原则的实验（T7）会影响未来 SP 维护成本和迭代速度。在实验结果出来前，明骏的"例子+决策分析"组合是较安全的折中。

4. **【个性化技能定义模糊（T13）】**——示例 15 暴露的产品侧债务，从"技能"作为一个独立产品概念的服务场景、创建时机、使用时机都需要明确，否则评测和 SP 示例无标准可依。

5. **【5/23 情景记忆提测是个时间节点】**——情景记忆的 case（示例 10 博总+晓伟）依赖该能力，且会影响"上次见面时间"等社交化体验。PM 应跟进 5/23 提测后的验证 + 合入节奏。

6. **【emoji 列表是跨模块协作债（T6）】**——planner 输出 vs 端侧语音规则触发的边界长期未对齐。陈芮婷找雪晴对齐这件事的产出可以沉淀成单独的 emoji-trigger 设计文档。

7. **【SP 示例真实性整改 + 评测侧协作（D9）】**——陈芮婷+单如水愿意帮明骏编真实 case，是会议达成的一个**新的协作模式**，建议落到例行 cadence 中（明骏出初版 → 评测侧批注 + 帮编 → 联评）。
