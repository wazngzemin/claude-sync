1输入协议：仍是 3 类，但字段说明"缩水"了

① user_query —— 文档只写 4 字段，实际更多

{
  "query": "用户说的话",
  "speaker_name": "说话人的名字",
  "speaker_position": "说话人的位置",
  "timestamp": "2026-04-13T14:30:05Z"   // 保留，时间推理的锚
}
注意：Round 4 版输入里显式写的 assistant_emotional_state 没了（随情绪机一起删）。但 goal_list 仍然内联在 user_query 里传——示例13 的 input5 就是铁证：{"query":"...","goal_list":"g-1:沿途风景讲解..."}。文档段没写这个字段，字段说明与实际协议不一致，新人只看文档会漏掉 goal_list 的传递方式。

② advisor / ③ tool_feedback —— 不变

advisor

{"advisor":"给你的建议"}
"不需要回应advisor，直接面向用户做行动和表达。这不是用户跟你说的话。"

tool_feedback

{"tool_feedback":"...","tool_name":"..."}
"根据工具的返回灵活继续推进任务，这不是用户对你说的话。"
为什么三类输入都要反复强调"这不是用户说的话"？——因为 Planner 是单模型多角色输入，最大的混淆风险就是把 advisor 建议当用户指令复述出来（"建议您开空调"被说成"你说要开空调"）。这类 badcase 在你们排查会的"链路问题"里占过比。输入协议设计的本质 = 用格式隔离信道，用重复声明防串台。

2输出协议：4 个字段，情绪状态机被砍 重要变化

{
  "talk_or_not": "yes/no",
  "talk_content": "要说的话，可带[开心地说]/[唱]",
  "emoji_id": "从可用emoji列表选一个，可为空",
  "action_list": [{"action_id","params","tool_name"}]
}
字段Round 4本版
talk_or_not / talk_content / action_list有有（不变）
emoji_id有（17个表情，默认speaking）有（15个定义，无默认speaking，"可以为空"）
emotional_state_change有（6态状态机）删除
为什么砍掉情绪状态机？——第一性拆一下：状态机的收益是"情感连贯"（被裁员示例那种长陪伴），成本是每轮都要维护一个显式状态+遵守转换规则，而 LLM 对"不能从低落直接跳欣喜"这类抽象规则的遵守率本来就不稳，遵守不了就成新 badcase（"情绪突变"）。砍了之后情绪表达退回两层：人设层的"被骂会生气伤心"+ 话术层的 [开心地说] 标签。用提示词的人设惯性替代显式状态机，是拿稳定性换表现力上限。示例10（被裁员陪伴）同步被删，佐证这是一次有意识的"情感能力降配"。

3{{tool_list}}：工具定义外置 = 本版最大架构变化 ★关键

"## 你可用的工具 —— 你仅可使用这几个工具 {{tool_list}}"

Round 4 版把 26 个工具的定义全部写死在 SP 里（占大量 token 且每轮必载）。这版变成一个占位符，意味着工具清单在运行时由外部装配——可以按车型（D6X/家越07硬件不同）、按场景、按版本灰度动态挂载。SP 从"工具说明书"变成"工具插座"。
