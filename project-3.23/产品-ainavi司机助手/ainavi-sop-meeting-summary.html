一、Ainavi 版本路线图

Navi 1.0

基础工具能力，如多点算路。面向基础导航复杂推理场景。

Navi 1.0+

基于高德 SDK Agent 能力 + 主线主动服务需求 + 重点客户诉求（赛力斯、DNA），扩展工具范畴。

例：多点顺路算路、预测目的地清单

Navi 2.0

结合舱驾融合业务，实现更深度的智驾场景。

例：「到前面的罗森停下」「找附近停车位帮我停进去」

📄 参考文档： Navi 交付方案（研发侧）— 涵盖架构、接入项目、端云数据流转等详细技术链路说明。

二、产品交付 SOP（六步法）

  1. 摸清 Ainavi 现有能力与工具清单

    阅读 PRD 了解体验目标、产品架构、合作沟通。核心能力：复杂推理 + 导航工具 + 第三方工具组合。
    参考工具 List 文档，标注版本归属、研发进度、基于客户需求的优先级。

  2. 了解方生号（FC）导航能力与工作流程

    简单控制（放大/缩小地图、开启组队等）走 FC 即可，不经过复杂推理模块。
    📹 参考视频：朱琳关于 FC 工作机制的讲解。

  3. 确认客户交互形态（双地图 vs 单地图）

    双地图方案：AI car 项目 — 复杂导航走 AI 小地图嵌入对话 UI，我方有主导权。
    单地图方案：比亚迪、奔驰 — 客户已有基础能力，由车企导航团队自行开发，我方只承担部分模块。

  4. 分析客户需求范围，初步评估实现方式

    以比亚迪项目为例，逐项标注能力归属：

    需求类型承接模块示例
    简单控制类FC打开/关闭地图、取消导航、退出地图
    复杂导航类Navi最近的公交站、导航到某地不走高速
    需推理+搜索Planner + Navi「我要洗车」— 推理出搜索洗车 POI 类型

    ⚠️ 客户强诉求但暂不支持 → 尽量拒绝；若必须交付 → 填写 Navi 产品需求收集表（需求池），标注项目、提出人、期望时间，由思晴评估可行性并给出预计完成时间。

  5. 输出产品方案（结合复杂推理 SOP）

    参考政伦整理的「复杂推理交付 SOP」文档 + Navi 工具定义 & UE 集合文档。
    Navi 工具三大模块：① 导航业务 ② 沿途问答 ③ 行程规划。
    每个工具有详细定义：能力说明、搜索策略（云端/端侧）、入参、示例、数据协议（Planner ↔ Navi 双向）。

  6. 构造产品必过集（评测用例）

    基于客户输入场景拆分：评测点、业务分类、对话轮次、query、考察点说明。
    覆盖：单工具调用、多工具组合（复杂推理 + POI 搜索 + 途经点等）。

三、交付依赖关系

3.1 对客户端的依赖

核心依赖：高德 SDK 数据接口。

FC 下发后 → 需通过高德 SDK 获取数据（如 POI 搜索接口）→ 端侧处理后上传 Navi 云端 → Navi 云端返回数据给复杂推理模块。

⚠️ 客户需将 SDK 接口封装后提供给字节，无法直接交付高德 SDK 全套。

3.2 对字节的依赖

项目类型依赖情况示例
语音主链路由字节提供 无链路依赖 比亚迪 — 开发在我方，语音+复杂推理均为内部模块
语音主链路由第三方提供 存在链路依赖 如美行等供应商 — 需在语音主链路上对接
📌 补充：复杂推理模块（Planner）目前在所有交付项目中均由我方提供，字节自研 Planner 仅在 AI car 项目使用。主动服务模块目前仅 AI car 项目在用，尚未上 AI car Lite。

四、关键讨论 & 技术风险点

4.1 TTS 播报冲突 高风险

当复杂推理链路发起导航后，车企原有导航会产生自己的 TTS 播报（如"全程20公里…"）， 与 Mind 期望由豆包统一播报产生冲突。

现状：AI car 项目已将导航 TTS 全部接入豆包语音链路。
挑战：导航 TTS 对时延要求极高（100–200ms），奔驰等车企导航走离线 ASR 播报，在线方案延迟通常 400–500ms，无法满足。
结论：这是共性难题，需按项目评估链路复杂度，已列为待跟进 Task。

4.2 可见可说 / 用户点击事件上云 待方案

用户点击 UI 结果（如「去第一家」）后，Mind 如何获知点击事件？

当前方案：端侧点击指令需上报到复杂推理模块。
问题：不同语音供应商（讯飞/思必驰）对可见可说的命中结果处理不同 — 有些 POI list 不往云端送。
思晴确认：方案框架去年已搭建，但主流项目中未纳入考虑（不仅是 Navi，音乐搜索选第几个也同理）。
结论：列为共性 Task，需研发+产品共同定方案。

4.3 SDK 版本兼容性与平台化适配

（智斌提出的关键问题）

场景可行性说明
同一平台多车型不同 SDK 版本（810 / 910 / 910+Agent） ✅ 可行 一套云端代码 + 不同工具集声明 + 端侧适配层
老版本 SDK 缺少某些能力 体验降级 能力缺失部分不上传云端，通过 Skill 边界限制告知用户「不支持」
无 NaviAgent 的车型通过第三方信源增强 ✅ 可行 可接入美团等第三方信源增强导航体验（如 POI 丰富度）
不同客户品牌的 Skill 定制（如奔驰搜五星酒店） ✅ 可行 同一套 1.0 代码 + 品牌专属 Skill 配置项
💡 卖点提炼：平台化一套代码 + 配置化工具声明 = 适配不同 OEM、不同车型、不同 SDK 版本，且可通过第三方信源提升无 Agent 版本的导航体验。
