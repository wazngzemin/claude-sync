给王泽民细化触发器架构图时,经过多轮纠偏后确认的真实诉求(一开始我反复跑偏:换成26/44节点别的结构、照抄复刻、问太多)。

**Why**:用户有一张自己设计的 30 节点全模块大图(产品-端侧触发器,原图 1-30 + L1-L33),要的是**在这张图基础上"细化"**,不是重画结构、不是复刻、不是另起炉灶。

**How to apply**(下次做触发器/类似图直接照这个来):
1. **骨架不要换**——他的 30 节点是对的(《业务流程讲解稿》《全链路详解》就是按它讲的)。只在其上加细,别改结构、别改编号。
2. **触发器模块要拆到原子级**——15云端拆 11 原子(注册校验/场景库/信号扫描/条件判定/抖动过滤/冷却频控/优先级仲裁/端云分类/标准事件/CallbackRouter/场景下发);24端侧拆 12 原子(规则接收/信号扫描/两级判定/本地定时器/优先级仲裁/冷却Always-on/运行指令/VLM调度/抢占恢复/结果回写/动作执行/弱网兜底)。
3. **每条线标"为什么"**——不是"L12",而是"信号变了→取快照求值""过频控→排优先级"这种因果。
4. **按流程分色**——注册流/信号流/判定流/下发流(端云粗线)/命中输出流/执行流,纵向流水线,别缠成一团。
5. **细化要有依据**——接口/DSL 字段必须自己逐字核对原文(接口文档/DSL定义【对外】),"待确认项"(range/持续时长/group/VLM并发)要标⚠不能当既定。

产出在 `产品-端侧触发器/diagrams-v2/`(gen*.py 可改重导)。最终图见 `触发器全模块大图-终版`(原图布局+4路径分色+15/24拆原子)。

**PRD 内流程图标准（7.5 深夜定版,与上面架构大图是两类图）**:用户看到简单 CSS flex 方框流程图直接否("太简单了没含量,我不知道该怎么讲")。他指着丁彬 VLM 会议里的图要求同款。经 3 轮迭代(充电桩 SVG v1→v3)拍板的规范:①**判断逻辑必须用标准菱形**(原话"你为什么判断逻辑用的不是菱形呢");②**每条判断线上必须写字、写为什么**("判断线条上要写字呀,要写为什么呀")——✅Yes 绿实线+理由 / ❌No 红虚线+为什么这么设计;③5 泳道按角色分层(配置平台/云端触发器/信号库Context/端侧AIBox/Director用户);④异常终态红色圆角节点;⑤底部时间轴(t=0→端到端秒数);⑥monospace 协议字段标注。模板=`需求详解PRD/SVG大图-充电桩链路-v3.html`;成品=`触发器PRD-v6-全流程SVG.html`(6 张:§3四入口三出口总览/§5A充电桩/§6B迎宾/§7C低电量三走向/§8D照看宝宝/§10异常四层决策树+三不变式)。以后给他画任何案例走查/链路流程图默认此规范,先画 SVG 别用 CSS 方框。
