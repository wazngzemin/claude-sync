§2 目标 · 1.5分钟翻到PRD §2.1 + §2.2
好，背景讲完了，大家知道为什么要做v4了。现在说一下这次到底要做哪几件事。

第一件事，我们的触发器引擎——刘杨那边的——现在只认识车速、车门、温度这些信号。它不认识"靠近某个POI"这种事。所以这次要给引擎加一种新信号，叫fence_event，就是围栏事件。引擎收到这个信号能匹配、能触发。

第二件事，端侧要能产出这个围栏事件。说白了就是端侧得有人管"我到没到"这件事。以前端侧没有这个能力，所以这次要新建一个模块——端侧围栏模块。这是本期最大的新建件，等下§3重点讲。

第三件事，解决one_shot难点。什么叫one_shot呢？就是——车开到500米的时候提醒你一次就够了，不能499米再提一次、498米又提一次，一路提到你停车提了几十次。只触发一次，然后闭嘴。这个等下§4我会详细讲怎么做到的。

第四件事，打通完整闭环——从用户开口说话，到选点、注册规则、下发围栏任务、端侧判定、事件上报、Advisor生成话术、TTS播报，整条链路端到端跑通。

就这四件事。不多，但每件事都有具体的Owner和交付物，等下§11分工的时候逐个对齐。

然后文字上云。②号，到了云侧之后干什么呢？先过仲裁——仲裁模型一看，这句话不是普通的聊天，它是一句"条件话"，有条件有动作。然后分给意图模块，意图模块把它拆成三块：条件是"快到外婆家"，动作是"打电话"，策略是"只触发一次"。接着调AINAVI选点工具——思晴6.25确认过这个工具可以复用——传入"外婆家"，拿回来经纬度。

③号，现在云端手上信息全了——POI名、坐标、半径500米、监听fence_enter事件。把这些写成一条规则，注册到配置平台。

④号，这是关键的一步——大家看这条黄色的跨界箭头。云端把围栏任务下发给端侧。说白了就是告诉端侧："你盯着这个坐标，500米范围，进了告诉我一声"。

到端侧了。大家看图中间这个红色的大框——端侧围栏模块，本期新建的。5个子模块都在这个框里。指令接收器收到JSON，解析完存到围栏存储里。然后⑤号，距离轮询器启动了——这个轮询器就是一个定时器，每秒问一次高德"我离外婆家还有多远"。如果外婆家是导航目的地，就直接读导航引擎的沿路线距离；如果不是，就调高德SDK距离查询接口。

⑥号，拿到距离之后，交给围栏状态机判。这个状态机很简单，就看一件事——距离d是不是≤480米，而且当前状态是不是"远"。如果两个条件都满足，那就是首次穿越进入围栏，状态从"远"翻到"已接近"，产出一个fence_enter事件。翻完就锁住了，后面477、476……再也不触发了。

⑦号，看这条红色的粗箭头——事件上报器把fence_enter发给云端。这里有一个非常重要的点——上报的内容只有4个字段：task_id、event_type、poi_id、timestamp。不传坐标、不传距离、不传任何用户隐私信息。坐标始终留在端侧，不出车。合规完全没问题。

⑧号，事件到了云端之后，后面的链路大家就熟悉了——触发器引擎匹配规则，push给Advisor，Advisor生成advice，Director拿着advice推理出话术，TTS播报。"快到外婆家啦~别忘了打电话哦！"这段从触发器引擎到TTS播报，和我们现有的链路完全一样，不需要改任何东西。

最后看底部这条绿色箭头——播报完了之后，端侧把围栏删了、轮询器停了，云端把规则清了。双侧清理，闭环结束。

所以整个方案就是这个图上画的——云端负责理解意图、选点、注册、下发；端侧负责围栏判定；云端负责触发和播报。新建件只有一个——中间这个红色大框，端侧围栏模块。

翻到PRD §3.2的端侧围栏模块组件表。
好，刚才大家看到架构图里那个红色大框——端侧围栏模块——里面5个子模块。我逐个说一下它们各自干什么，大家看一下PRD里这个组件表。

第1个，指令接收器。云端把围栏任务下发过来，端侧收到的就是一个JSON——里面有task_id、POI中心点经纬度、围栏半径、要监听什么事件。指令接收器就负责解析这个JSON，看看是"创建围栏"还是"删除围栏"，然后转给下游。说白了就是一个JSON解析器，很轻。

第2个，围栏存储。解析完了得存下来啊——因为这个围栏可能要活几十分钟甚至几小时。用户说完"快到外婆家提醒我"，可能一个小时以后才到。这中间如果app崩溃了或者重启了，围栏不能丢。所以要做本地持久化，SharedPreferences或者SQLite都行，帅哥你们看哪个方便用哪个。

第3个，距离轮询器。这个是干活的核心——一个定时器，每秒跑一次。每次做什么呢？遍历所有活跃围栏，逐个查"我现在离这个POI多远"。如果这个POI是导航目的地，就直接读导航引擎的沿路线距离，不用调外部接口；如果不是导航目的地，就调高德SDK距离查询接口。查完把距离d传给状态机。

第4个，围栏状态机。收到距离d之后，做判定。逻辑很简单——当前状态是"远"，而且d≤480米，就认为首次进入围栏，状态翻到"已接近"，产出一个fence_enter事件。翻完就锁了，后面d=477、476……不会再触发第二次。这个就是one_shot的实现，等下§4我展开讲。

第5个，事件上报器。状态机翻转的时候，把事件发给云端。就是发一个请求，payload只有4个字段：task_id、event_type、poi_id、timestamp。注意，不传坐标。如果发送失败了——比如弱网——本地缓存，恢复后重试。

5个子模块就是这些。仁杰你上次说过一句话我印象很深——"围栏逻辑本身比较简单，给个坐标，建个围栏，以一定频率查在不在"。现在高德SDK帮你包装的那层下架了，我们自己做这5个子模块。但你看每个拆开来都不复杂——JSON解析、本地存储、定时器、一个简单状态机、一个HTTP请求。复杂的不是单个模块，是把5个串起来跑通。
翻到PRD §3.3的距离获取方式对比表。
好，距离轮询器每秒要查一次"我离这个POI多远"——那这个距离数字到底从哪来？这个问题很关键，因为数据源的选择直接影响触发精度。

我们有两种数据源，端侧根据场景自动选。

第一种，导航沿路线距离。什么时候用呢？你正在导航，而且这个POI就是你的目的地或者途经点。这种情况下，导航引擎本身就在实时计算"还有多远到目的地"——你看导航界面上不是一直有个"剩余距离"吗？我们直接读这个值就行。好处是精度非常高。大家想一个场景——你在高架上面开，高架下面有一家火锅店，直线距离可能只有200米，但你下高架再绕过去其实还有2公里。如果用直线距离，200米就触发了，用户会觉得"明明还没到啊"。但导航沿路线距离是2公里，不会误触发。

第二种，高德SDK距离查询。什么时候用呢？POI不在你的导航路线上，或者你压根没开导航。比如"路过徐浦大桥提醒我"——你没有导航去徐浦大桥，导航引擎不会算到它的距离。这时候调高德SDK的POI详情接口，问它"我离徐浦大桥多远"，它告诉你一个数字。俊龙6.25说过这个接口能返回当前距离。返回的可能是直线距离，精度比第一种低一点——高架旁的场景可能有偏差。但对于"路过提醒"这种轻量场景，够用了。

还有第三种，GPS自算——这是兜底。如果高德SDK也调不通，比如隧道里信号差、弱网，端侧自己拿GPS坐标用Haversine公式算两点直线距离。精度最低，但至少围栏不会因为网络问题挂掉。

优先级：导航距离 > SDK距离查询 > GPS自算。三级降级，确保围栏在任何网络条件下都能工作。

§4 核心难点 · 2分钟翻到对战稿架构图2 + PRD §4.1 数字走查表格
好，这里是整个方案最核心的难点——one_shot。

大家想一下，如果我们简单粗暴地配一条规则"距离小于500米就触发"，会怎样？我给大家走一遍——用户说"快到公司提醒我买咖啡"，导航去公司。车从15公里外一路开过来，到500米的时候，触发了，没问题。但下一秒变成499米，还是小于500米——又触发了。498米——又触发。497、496、495……一路刷屏刷到你停车。"该买咖啡了""该买咖啡了""该买咖啡了"——触发了几十上百次。用户想骂人了。

所以不能用"小于500米"这种条件。必须是——从远到近、第一次跨过500米的那个瞬间、触发一次、然后闭嘴。这就叫one_shot。这也是刘杨你6.16号最先提出的问题——"怎么确保只触发一次"。
翻到对战稿架构图2（围栏判定详细流程）。手指着图从左边"定时器"开始讲。
怎么做到one_shot呢？答案就在这张图里——围栏状态机。

大家看图的左边开始——定时器每秒触发一次。先问"有没有活跃围栏？"有的话，判断A类还是B类——A类读导航距离，B类调SDK——拿到一个距离d。然后d传给状态机。

状态机就看两件事：第一，当前状态是不是"远"？第二，d是不是≤480米？

如果两个都满足——恭喜，首次穿越！大家看图右边那个红色大框——状态从"远"翻到"已接近"，产出fence_enter事件，然后one_shot锁锁住。锁住了意味着什么？意味着下一秒d=478、477、476……状态已经不是"远"了，第一个条件不满足，不会再触发。永远只触发这一次。

如果d还是481、490、500……没穿过480——看图下方，"等下秒，不做事"。继续等。

底部大家看状态机全景——4个状态，3次转换：

"远"——初始状态。距离很远，什么都不做。
"已接近"——d≤480时翻过来的。产出fence_enter事件。这个翻转不可逆——翻了就不回来了。
"已到达"——d≤50米而且挂了P档。说明车停在POI旁边了。
"已离开"——d>520米。说明车又开走了。围栏销毁，生命周期结束。

中间还有一条虚线——从"已接近"可以直接跳到"已离开"。什么场景呢？你在高速上路过一个服务区，靠近了但没停，直接又开远了——距离先到480以内再到520以外，跳过了"已到达"。
翻到PRD §4.3的三个坑。
这个状态机设计里面还有三个坑需要处理，大家注意一下。

第一个坑，GPS防抖。大家知道GPS是有误差的，特别是在城市里建筑物密集的地方。如果阈值设500米，GPS可能这秒报498、下秒报502、再下秒又报499——在500米附近来回跳。如果每次穿过500都触发一次，那又变成刷屏了。所以我们不用500米做阈值——进入用480米，离开用520米。中间480到520这40米是死区，在这个范围内不做任何判定。GPS怎么跳都无所谓，只有彻底穿过480才算进，彻底越过520才算出。

第二个坑，阈值可配。500米是默认值，但不同场景可能需要不同的阈值。比如"快到高速出口提醒我"可能需要1公里——因为高速上你来不及反应。这个阈值后续做成产品配置，现在先用默认值。

第三个坑，立体空间。刚才说了高架场景——直线距离300米但沿路线2公里。A类用导航沿路线距离，这个问题自动解决；B类用的是直线距离，高架旁可能提前触发。但"路过XX提醒我"这种轻量场景，提前触发不是灾难。后续可以加Z轴高度过滤，本期先不做。
§5 A/B两类Case · 1.5分钟翻到PRD §5 对比卡片
好，这个A/B分类前面提过了，这里展开讲一下。大家看PRD里这张对比卡片。

所有"快到XX提醒我"的场景，我们分成A和B两类。区分标准就一条——这个POI是不是在你的导航路线上。

A类，POI在导航路线上——要么是你的目的地，要么是途经点。这种情况好办，因为导航引擎本来就在实时算"剩余距离"，我们直接读这个值就行了。精度非常高——它算的是沿着路线走的真实距离，不是两点之间的直线距离。举个具体例子，你正在导航去外婆家，导航引擎告诉你"还有479米"——这个479米是沿着路开的真实距离。你即使在高架旁边，直线只有200米，导航距离照样显示2公里。不会误触发。

B类，POI不在导航路线上。比如"路过徐浦大桥提醒我右转"。你没有导航去徐浦大桥，它不是你的目的地也不是途经点，导航引擎压根不会算到它的距离。这种怎么办？调高德SDK距离查询接口，拿到的是直线距离。精度比A类低一些——高架旁的场景可能提前触发——但对于"路过提醒"这种轻量需求来说，可以接受。

这里有一个非常重要的变化，大家注意——B类在v3里是一个卡点。v3的方案是什么呢？B类需要"动态把这个POI插入成导航途经点"，然后才能用导航距离来判。仁杰上次说了，这个"能不能做不确定"。说白了，v3的B类是悬着的。

现在v4——端侧围栏方案天然支持任意POI坐标。你给我一个经纬度，我就能在端侧建一个围栏，开始查距离。跟你开没开导航没关系，跟这个POI在不在路线上也没关系。给坐标就能建。所以B类从"做不了/不确定"变成了"直接能做"——这是v4相比v3最大的价值之一。
B类"不再是卡点"是v4的核心卖点。讲到这里看一眼帅哥和仁杰，如果他们点头或者有表情反应，停下来让他们补充。如果有人问"B类精度够吗"，说"轻量场景可以接受，后续可以加Z轴高度过滤"。

Phase A，选点+注册。用户说完话到围栏建好，10秒内完成。系统听懂意图、选好POI、拿到坐标、注册规则、下发围栏任务给端侧。

Phase B，端侧距离监听和围栏判定。这个阶段是整个方案最有意思的部分。

现在车在路上开着，距离轮询器每秒查一次距离。我给大家走一下数字，感受一下这个过程——

d=15000米。太远了。状态机一看——"远"，不做事。
d=10000米、8000米、5000米……还是远，不做事。
d=2000米……用户可能在聊天、听歌，完全忘了半小时前说过"提醒我打电话"这件事。

d=520、510、500、490……快了快了，但还没穿过480。

这一秒就是关键的一秒。479≤480，而且当前状态是"远"——两个条件同时满足——首次穿越！围栏状态机做三件事：

第一，把状态从"远"翻到"已接近"——这个翻转不可逆。
第二，产出一个fence_enter事件。

陈俊龙"每秒调一次距离查询接口，QPS顶得住吗？有频率限制的"
嗯，这个确实是我最想跟你确认的一点。我目前设计的是1Hz——每秒查一次。但我知道SDK那边可能有限频。

我给你算一下我们能接受的范围——如果限到5秒一次，60km/h的车5秒走85米，85米远在死区范围（40米）之外，完全可以接受。如果限到10秒一次，120km/h的车10秒走333米，也还能命中480米阈值。

但是如果限到30秒一次——那就有点悬了。120km/h的车30秒走1000米，整个围栏范围500米可能直接跳过去了——进去了、出来了、状态机都没察觉。那方案就得改了。

所以关键问题是——你那边实际的QPS限制是多少？给我一个数字就行，哪怕是大概范围也行——是1级别的、5级别的、还是30级别的？
速算表备查：
60km/h ≈ 17m/s → 5秒=85m → 可接受
120km/h ≈ 33m/s → 5秒=167m → 可接受但精度降
120km/h → 10秒=333m → 勉强可用
120km/h → 30秒=1000m → 方案不可用，需改设计
如果他追问"为什么一定要每秒查"：不一定要每秒——5秒10秒都能接受。我只是设计了一个理想值。给我一个实际限制数字，我来调整轮询频率就行。我们的方案对QPS的容忍度其实挺高的，只要别到30秒就行。

这是6.25全员评审大家一起拍的结论——端侧判定、云端触发。今天不重新讨论方向。

任何人"高架旁误触发怎么办？"
分两种情况看——

A类，POI是导航目的地。距离数据用的是导航沿路线距离——你在高架上面开，直线距离可能只有200米，但沿路线还有2公里。导航距离不会误触发。这种完全没问题。

B类，POI不在导航路线上。距离用的是直线距离——高架旁确实可能触发。但大家想一下具体场景——"路过迪士尼提醒我拍照"。你在高架上，直线距离500米内，说明迪士尼确实就在你旁边了。提前触发一次"快到迪士尼了，别忘了拍照"——这不是灾难。用户不会因为早提醒了几分钟就觉得系统有问题。

如果真的要精确——比如某些场景对位置非常敏感——后续可以加Z轴高度过滤：高架上面海拔20米、下面海拔5米，差值大于阈值就不触发。但本期先不做，优先保证核心流程跑通。
任何人"如果高德SDK距离查询接口也不可用怎么办？"
有兜底方案。端侧自己拿GPS坐标，用Haversine公式算两点之间的直线距离。

仁杰说过——"围栏逻辑本身比较简单"。核心是什么？就是拿到两个坐标、算距离、和阈值比大小。有SDK的时候，SDK帮你算；SDK不可用的时候，自己算也一样。Haversine公式就是一个数学公式，几行代码的事，不依赖任何外部服务。

精度会降低——GPS本身有误差、直线距离不如路线距离准——但对于"快到某地提醒"这种场景，500米范围内差个几十米完全可以接受。

所以三级降级：导航距离 > SDK距离查询 > GPS自算。就算高德SDK真的不可用，围栏功能也不会挂——只是精度降级。
任何人"v3到v4变化太大了，方案稳不稳？会不会v5又推翻？"
我理解你的担心，但其实核心架构没有变。

v3和v4的架构是一样的——"云端理解意图+选点+注册 → 下发围栏任务给端侧 → 端侧判定 → 事件上报云端 → 触发+播报"。这个四步模型从v3到v4完全没变。

变的是什么呢？只有端侧判定这一步的数据来源。v3是"调高德围栏SDK，SDK帮你判"；v4是"调高德距离查询接口拿距离，自己用状态机判"。说白了就是——以前有人帮你做，现在自己做。但做的事情是一样的。

这次改方案的原因是——高德围栏SDK下架了，这是一个外部变化，不是我们推翻自己的设计。而且你换一个角度想——v4反而更稳。为什么？v3依赖高德围栏SDK这个第三方服务——它说下架就下架，我们控制不了。v4呢？距离查询接口+自建状态机，核心判定逻辑在我们自己手上，不依赖第三方的开关。可控性更强。

所以我不觉得会有v5推翻。除非——导航引擎和距离查询接口也同时下架——那是另一个话题了。
任何人"这个需求630上不了怎么办？"
我有降级方案准备好了。

630最小集——只做A类，就是"导航目的地靠近触发"。距离数据直接读导航引擎现有的"剩余距离"字段——不需要调SDK、不需要建完整的围栏模块。端侧改动最小——加一个距离监听和简单的阈值判定就行。覆盖的场景是"快到外婆家提醒我打电话"这种最典型的case。

B类（任意POI、不在导航路线上）后面迭代——因为B类依赖高德SDK距离查询接口的确认，如果来不及确认就来不及开发。
