📡 车状态信号数据源·车厂
是什么车上所有传感器/CAN总线产生的原始状态:座位占座、档位、车速、车门、温度、空调……都是结构化枚举值,天生适合运算符。
数据长啥样
{ "信号":"座位占座信息", "position":"主驾", "states":"已占用",
  "prev":"未占用", // prev→current 这个"沿"就是"变更为"判定的依据
  "capture_ts":"14:10:03" }
为什么对你重要:这类信号是你的"A档"——直接上 等于/变更为,零模型成本。你条件库里大部分条件都建在这上面。
📷 VLM 视觉(AIBox端模)丁彬/兰若
是什么车里的摄像头 + 视觉大模型,每隔几秒"看一眼"车内外,输出自然语言描述+少量标签。跑三个循环+按需单次:
四种任务 • 舱内高频(5秒):看变化快的——动作、神态
• 舱内低频(5分钟):看变化慢的——位置、性别、年龄、物品
• 舱外(3秒):天气、路面
• oneshot(按需一次):迎宾、上下车、充电桩、寻车
数据长啥样
// 低频任务每5分钟全量输出一次:
{ "乘客列表":[
   {"位置":"主驾区域","性别":"男","年龄":"青中年","动态":"端正坐姿…"},
   {"位置":"副驾后方区域","性别":"女","年龄":"儿童"} ],
  "物品或动物类型":"粉色书包" }  // ← 注意:物品是caption不是枚举!
你的点火作用:这三个循环不是自己转的——是你的端侧触发器㉔/SLS开关把它"打开"的(绿箭头▲)。开了才5秒一转。坑:低频任务5分钟才出一次结果,"上车即推荐"最坏等5分钟(见亲子案例K1)→ 要用占座事件触发oneshot插队。
🔥 端侧触发器㉔ / SLS开关你的
是什么你负责的"点火开关"。SLS舱内AlwaysOn开关拨下去,由端侧触发器执行"启动周期推理"这个动作(6.30定:点火权统一收口端侧触发器)。
为什么收口给你:如果推理入口有的从SLS进、有的从Advisor进,任务优先级就管不全了。入口唯一,排队才有意义。过渡期(0804):SLS直接点火,818你接管。
📮 端状态客户端李鑫·纯透传
是什么一个"快递员"——收到信号就转发上云,不处理、不存储、不缓存(7.4确认)。给它啥它传啥。
坑(关键):它不存 = 全链路没人记得"上一帧"。而你的"变更为"判断恰恰需要前帧对比 → 这就引出卡点C2(见下)。
🗄 端侧信号库 / 云端信号库仁杰
是什么把杂乱的原始信号标准化成统一格式(key/value/source/timestamp/ttl)。端侧的负责车上信号,云端的是"全集"(端信号同步上来 + 外部数据)。
数据长啥样
{ "key":"vlm_rear_left_object", "value":"毯子在地上",
  "source":"VLM", "capture_ts":"14:05:10", "ttl":300 }  // ttl=有效期,过期作废
⚠ AI service 数据中心卡点C2 · 赵衍·还没立项
该做什么端上信号上云的中转站。赵衍7.4说"AIBox传给AI service,AI service做端数据中心",但还没立项。
为什么卡你你判"从没狗变成有狗"需要上一帧,但端不存、透传不存 → 前帧得有人存。最合适的地方就是这个数据中心:每类信号保留"最近一次有效值+时间戳"。
🔴 讲之前先确认:Context是不是只存最新值?如果Context本身带历史,前帧就在Context里,这个卡点是伪命题。先问仁杰Context存储粒度,再决定C2怎么解。
☁ 云端 Context仁杰·全局事实源
是什么全系统的"当前状态总账本"。所有信号最后都汇进来。你的触发器、Director、Advisor 都从这读。
关键性质① 只存储+主动查询,没有被动通知(庆磊7.4)——你得自己去扫,它不会主动告诉你变了。② 带 ttl 防过期数据误判。③ 给Advisor/Director是"推理时固定拼接",不是订阅。
⚠ 这段的两个卡点: C1 回传怎么走VLM结果绕Context一大圈,还是原路直接返回?(下一段详说) C2 谁存前帧推AI service存"最近有效值",但先确认Context存储粒度。
🎯 这段记住一句:信号从"传感器/摄像头/外部数据"三方汇进Context这本记事本,你只读记事本不碰传感器(解耦)。但记事本"只有当前值、不主动通知你"——所以你要每5秒主动去扫,而且"变更判断"缺前帧,这是你要解决的第一个数据难题。

⑨ 拒识分流(四路)意图·斯华
是什么用户的话上云第一站,判断属于四类的哪一类,分流到不同的路。用"词表命中+意图模型+超时兜底"。
四路是啥 ①车控:"开车窗"这种明确指令 → 回端侧直接执行,不上大模型
②条件话:"超25度开空调""毯子掉了告诉我" → 走你的链路(重点)
③非车控query:"今天天气咋样" → 直达Director推理
④拒识:噪音、误唤醒 → 记日志丢弃
⑩ 意图解析意图
是什么把"条件话"从自然语言拆成结构化的"条件+动作"。
数据长啥样
"超过25度开空调" 解析成:
{ "condition":"temperature > 25", "action":"开空调" }
"下车提醒带伞" 解析成(注意补全了主语):
{ "condition":"主驾 下车", "action":"TTS:别忘带伞" }
🔀 条件话的两条分支你的核心入口
分支A·系统预设/静态产品提前配好的固定场景("温度>45自动开空调")→ 进你的⑬配置平台注册 → 存进⑮云端触发器场景库。这条走"注册"路径。
分支B·用户动态用户临时说的("毯子掉了告诉我")→ Director拆解 → 动态Advisor创建任务 → 给你的⑮拼SP(如果要看视觉)。这条走"拼SP"路径。
为什么分两条:A是"编译期约定"(提前配好,能做频控、能挂多个场景),B是"运行期约定"(用户一句话现造,用完即销毁)。A像公交线路,B像打网约车。
🎯 这段记住一句:用户说话先分四路,只有"条件话"进你这。条件话又分两条:产品预设的走"配置平台注册",用户临时说的走"Director→动态Advisor→你拼SP"。你在这一段是"接单口"——把用户的意图变成你能判断的规则。
第③段触发判定 —— 你的核心战场

🔍 云端触发器⑮ 内部:命中一条规则的6步
⑮ 云端触发器 · 判定流水线你的·P0
15-3 扫描每5秒心跳读一次Context快照,看有没有信号变化(主动轮询,不是等通知——因为Context不主动通知)。
15-4 判定拿注册的条件比对当前信号。支持 静态条件 AND 动态条件。
例:"主驾占座=已占用" AND "乘客列表里有儿童" → 两个都满足才命中。← 卡C8:一个是事件、一个是状态,怎么AND?
15-6 防抖信号在临界值反复跳(29.9→30.1→29.9)不该反复触发,要连续K帧一致才认。对付VLM的glitch(张兰若警告:狗突然消失又出现)。
15-7 频控同一条规则触发过就进冷却:总100次/天100/上电10/间隔60s。← 卡C12:但它挡不住"用户拒绝了"这种语义。
15-8 仲裁多条规则同时命中,按优先级P0~P4排谁先执行(安全类P0 > 娱乐类P4)。
15-13 组装事件命中后打包成标准事件,push给Advisor。
事件长啥样
{ "event_id":"evt_0001", "scene_id":"亲子出行上车", "priority":0,
  "hit_conditions":[
    {"signal":"座位占座.主驾","op":"变更为","value":"已占用"},
    {"signal":"乘客列表.年龄","op":"等于","value":"儿童"} ],
  "下游输入":{"scene":"亲子出行","push_text":"检测到亲子出行…"} }
四个出口(15-13后往哪送):①静态Advisor(黄线,你负责)②动态Advisor(红线,你负责,卡C1回传)③通知Director ④简单动作直接执行。走哪个由场景配置的callback决定。
🔍 动态视觉分支:拼SP怎么拼(卡C4/C5)
🧩 15-16/17/18 拼SP引擎卡C4归属·C5成本
流程Advisor给"人话需求"("毯子掉没掉")→ 15-16判枚举还是描述 → 15-17提槽判舱内/舱外 → 15-18拼SP → 经㉔下发VLM。
拼SP长啥样
原生SP: "prompt: aaaa, bbbb"          (AlwaysOn基础任务)
 + 动态SP: "cccc"                     (观察毯子状态)
 = 合并SP: "prompt: aaaa, bbbb, cccc"
KV槽位: answer1←aaaa / answer2←bbbb / answer3←cccc(毯子结果)
为什么归你不归Advisor:同时可能有3个Advisor各要看不同东西,VLM 5秒一个窗口只吃一个SP,必须有人合并、定槽位、再拆结果分发——能看到全部任务的只有管注册的触发器。刘杨反对这归属(C4),1万辆车用模型判断的成本也被质疑(C5)。

⑳ Director 推理大脑核心
是什么基于豆包大模型的推理中枢。输入=Advisor建议+Context,输出三个字段。
输出长啥样
{ "talk_or_not": true,
  "talk_content": "小朋友也在车上,帮你准备了儿童节目,要放吗?",
  "action_list": [{"tool":"push_card","params":{"template":"模版一","ttl_min":3}}] }
↩ 拒绝结果回流卡C12
问题益红配了"拒绝后本次上电不再推荐",但你的频控(execute_policy)只会数次数,不懂"用户拒绝了"。
怎么解Director识别拒绝 → 调你的"场景屏蔽接口"(作用域=本上电周期)→ 15-7频控下次直接拦。但这个接口现在不存在,要提给刘杨。
🎯 这段记住一句:你push事件之后,后面的话术/动作是Advisor和Director的活,你不管。但有一根线要连回来:用户拒绝的结果得回流到你的频控,否则"拒绝后不再推荐"实现不了。
🎯 一句话把四段串起来(以"到公司提醒拿外卖"为例)
用户说"到公司提醒拿外卖" → ②条件话 → 你在配置平台⑬注册成规则 → ①感知上行持续把"距公司距离"送进Context → 你的⑮云端触发器每5秒判"距离<500m" → 命中 → 组装事件push出行Advisor → Advisor出建议 → Director出话术 → ④输出:TTS"快到公司了,别忘了拿外卖"。

你在这条链路出现三次:注册(配置平台⑬)→ 判定(云端触发器⑮)→ 执行(端侧触发器㉔)。所有卡点都在"信号上来"和"你判定"的交界处——本质是同一个问题:VLM给的是描述,你要的是能判断的信号,中间的转换成本谁扛、判断放哪、结果怎么回、前帧谁存。
