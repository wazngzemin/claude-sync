## 2. 机器人侧事实梳理（吕壮口述还原）

### 2.1 AIOT SDK 是什么
- 机器人侧在**车机端**做了一个软件 **AIOT SDK**。
- 原始职责：**管理和存储所有的表情形象**，整套打包放在 APP 里。
- 它接收上游的调用指令来**展示对应表情**。

### 2.2 老模式（Banner 直接下发）
- 早期：通过 **Banner 直接下发"表情/动作"指令**。
- 机器人侧**纯被动接收**——"用户开始播放音乐了 → 展示'播放音乐'表情"，收到什么显示什么，自己**不做逻辑判断**。

### 2.3 新模式（Director 接管后，关键变化）
- 现在天气询问、节日询问等场景，走 **Director**。
- **Director / 意图侧不再直接告诉机器人"显示哪个表情"**，而是给三个元素 ↓

> **🔑 机器人展示接口 = 三元素：**
> 1. **形象/表情标签**（要调哪个形象）
> 2. **内容标签**（如"晴天"）
> 3. **TTS 播报的开始 + 结束时间**
>
> 机器人拿到这三样，**自己内部跑一套（较复杂的）逻辑**决定展示什么表情、什么时候展示。

- ⚠️ 注意：这三元素**目前是在「意图」侧产出的，不是 Director 产出**（吕壮原话）。→ 这条边界要回去和意图/Director 团队对齐。
- **语音播报本身** = Planner 自己定义，**机器人不管**。

- 交付：**吕壮会把这张表发给泽民**（"你把那个表先发我一下" → 已确认要发）。

> ⚠️ **一条要复核的边界**：触发器本身就是"信号/条件驱动"，而吕壮说"信号触发的部分与你无关"——这两者听起来矛盾。合理解释是：机器人**自己内部**的信号触发展示逻辑（在 AIOT SDK 里闭环），不经过我们的触发器。**但这点必须等表到手、看清字段后再确认**，别现在就下结论。

## 4. 暴露的关键开放问题：TTS ↔ 表情不同步

这是这场聊出来的、机器人侧真正的**痛点**，也是吕壮反过来问泽民的事：

- **现象**：TTS 和表情**串行**执行 → TTS 已经开始播（说话标签出来了），**表情形象还没加载出来**；有时 TTS 都播完了表情才出 → **用户根本没看到天气表情**。
- **期望**：TTS 播报**过程中并行**展示表情。
- **现状**：技术上目前是串行；泽民反馈"听他们聊过，后面想做成正常并行 / 做定型产品"。
- **吕壮诉求**：想知道**能不能解、是不是纯技术原因**导致的延迟。
- **归属**：这是 **Director / Planner 执行链路**的问题，不是触发器条件层的问题。

> 📌 行动：泽民需回去找 Director / 意图 / Planner 相关同学，确认"串行→并行"的**技术可行性和排期**，给吕壮一个回话。
