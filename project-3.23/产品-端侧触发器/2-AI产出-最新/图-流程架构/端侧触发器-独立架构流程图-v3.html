v3 核心变更（7.6下午 VLM链路专项会定版）
1. AlwaysOn 的 VLM 数据不再经过触发器透传，改为 VLM SDK 直送数据中心（数据通道）
2. 触发器只接收 VLM SDK 的状态回调（控制通道），管调度状态不管数据内容
3. ㉔-9 更名为「VLM调度+状态管理」，㉔-U 变更为「触发器自身事件上报」
4. 新增数据中心（李鑫）节点：订阅VLM SDK topic，整包缓存，按需透传云端
5. OneShot 回传路径待后续评审（丁彬原话: "one shot 等需求导入再聊"）

v2→v3 对比
• v2（7.6上午）：VLM → ②"谁调返给谁"回触发器 → ㉔-U整包透传 → AI Service（串行，VLM数据全部经过触发器）
• v3（7.6下午定版）：VLM SDK → ③数据直送数据中心 + ②状态回调给触发器（并行双通道，触发器不碰VLM数据）
• 原因：触发器是信号处理模块，不该承担数据透传职责；VLM数据量大、频率高，经触发器中转增加延迟且触发器无法解析利用

双通道运作要点：
• 数据通道（绿色）：VLM SDK 执行完成后，将原始输出发布到 topic → 数据中心订阅接收 → 整包缓存 → 按需透传云端。触发器全程不经手
• 控制通道（蓝色）：VLM SDK 同时回调触发器 ㉔-9，只传 status（success/empty/timeout/failed）→ ㉔-9 更新任务状态、释放互斥、安排下一周期
• 职责切割：触发器保证"调用请求发送成功"，VLM SDK 保证"收到后内部执行成功"
• 异常处理：AlwaysOn 由 VLM SDK 内部轮询保证（闲时常驻）；状态回调丢失时 ㉔-9 超时守卫兜底
• 与 OneShot 对比：AlwaysOn 双通道已定版（本图），OneShot 回传路径待评审（见图4）
