0名词速查表

读正文前先建立这套词汇。后文凡是 带虚线的词 都能在这里找到。

Planner / Director（对话推理引擎）
同一个东西的两个名字。Planner 是产品口径，Director 是正式工程名（"对话推理引擎"）。它是系统的"大脑"：接收用户意图，拆成任务，输出工具调用指令。本报告中两词等价。
query 改写（rewrite）
把用户口语（"我好热"）重写成下游工具能听懂的标准指令（"打开空调+调低温度"）。Planner 的核心动作之一。
端状态（env_info）
车辆此刻的真实状态：空调几档、车窗开度、哪个座位有人。以模板变量形式注入给 Planner，让它"看得见车现在什么样"。
上下文（context）
最近 N 轮的对话历史。Planner 靠它理解"再调高一点"里的"再"指什么。
用户记忆（memory）
用户的长期偏好（如"不喜欢紫色"）。和端状态、上下文一起构成 Planner 的"输入情景"。
SFT（监督微调）
Supervised Fine-Tuning。用人工标注的"输入→正确输出"样本训练模型，把某种输出习惯固定下来。后文"让下游 SFT 适配 Planner"即指此。
Function Call (FC) / Tool Use
模型输出一段结构化 JSON 来调用工具（工具名 + 参数）。"FC 不对"=调用格式/工具名/参数错了。
意图系统 / 弋途
下游的二次分类器。Planner 给出 query 后，意图系统再判一次该走哪个具体功能。弋途是负责这块的团队。
幻觉（Hallucination）
模型编造了不存在的信息。如工具只返回"成功"，Planner 却播报出一个并没返回的歌名。
tool_feedback
工具执行后返回的结果。feedback=0 通常代表"执行成功"，不含具体内容。
Chat Model（播报节点）
拿到工具结果后，生成给用户播报话术的模型节点。幻觉常发生在这里。
NLG（自然语言生成）
就是"说人话"那部分——播报话术是否简洁、自然、说做一致。
Advisor（顾问）/ A2A / 竞态
Advisor 是负责"深度思考+目标监控"的协作角色。A2A=Agent 之间互相调用。竞态=Director 和 Advisor 并发执行时互相干扰产生的 bug。
干预平台（Intervention）
在 Planner 之外加一道规则拦截层。命中固定关键词就直接给固定结果，不走模型——用来兜冷门/高频固定指令。
动态示例（情景 RAG）
按当前情景检索几个相似范例注入提示词，引导模型照着做。"高频 case 补两个动态示例"即此。
Reward Grader / LLM Judge / Rubric Scorer
评测打分工具。Rubric Scorer=用规则给 0/1 分（格式对不对）；LLM Judge=用大模型给主观项打分（话术好不好）。两者加权 = Reward，用来训练模型。
P0 / P1(PR) / P2 / P3
严重性分级。本次明确定义：P0=高频域阻塞、需求无法闭环；PE=低频域阻塞；P1/PR=能闭环但体验差；P2/P3=能完成只是不够好。红线：只有不可用才算 bug。

3六大类 Bug 全景：是什么 · 为什么 · 怎么解

这是报告主体。每一类先给定义，再用两份材料里的真实 case 举证，最后给短期/长期解法。

类型 A 推理问题 —— Planner 自己想错了

是什么：改写或工具调用在逻辑上就是错的，不是下游接不住，是大脑本身输出不合理。这是真正属于 Planner、也最该修的一类。统计占比最高（拆解 14.6% + 决策 6.7% + 多轮 5.6% + 端状态 3.4% ≈ 30%）。

A1 · 空间关系理解错误

为什么：模型缺乏车这个三维物理空间的常识——分不清"主驾右侧""向前""调高"在车上到底指哪。

真实 case错在哪当前→建议
车窗已全开，用户说"调高一点"应反向调低开度，却还想往上加P0→P0 高频，明天就解
遮阳帘已开，"往前一点"→直接调到 20%不懂"开着时往前=变小"PR 加知识可解
主驾说"打开我右边的座椅通风"→开成右后"主驾右侧=副驾"识别不了（1.8 反复试都不行）纯模型问题，难
"风不要对我吹"→改写成"对人吹风"方向理解反推理类
"主驾座椅有点挤"→把座椅往后调挤应往前留腿部空间推理类

A2 · 端状态参考错误

为什么：端状态写得人都读不懂——表格式结构（"车内空调状态:{温度:na,模式:na,状态:5档...}"），模型读不出"现在风量几档"，于是瞎推理。会上原话："这端状态写的真的太差了，人都看不懂。"

  • "风量调高三档" → Planner 没调工具就说"现在是 9 档"（端状态其实是 6 档，读错了表格表头）。
  • 温度调节：主驾 21.5 / 副驾 23.5，"调低一点"调到了 24（读错位置）。
解法核心：改端状态结构，从表格 → 自然语言（"现在空调开着、5 档、22 度"）。这是 A2 的根，不是改 Prompt 能绕过的。

A3 · 上下文参考错误（含"该调工具却不调"）

为什么：Planner 对"何时该信历史、何时该重新查"没有稳定判据，容易被上文带偏。

  • 不调工具：用户重复/相似 query（"附近有什么好吃的"问过一次），第二次 Planner 直接用历史对话答，不再调工具取实时结果。实验在注意事项里加约束后仍 10 次错 3–4 次 → 约等于不可用。
  • 被上文带偏：连续操作遮阳帘，中间夹了句别的，"往前一点"被改写成调座椅（叠加 ASR 误识别）。
  • 上下文压过端状态："我现在风量几档"→播报"7 档"（历史里有个 7 档），端状态其实是 2 档。

类型 B 幻觉问题 —— Planner 编造了不存在的信息

是什么：工具返回里没有某信息，Planner/Chat Model 却在播报里编了具体内容。官方统计 25 条 · 14%，是第二大类。

真实 case根因
QQ 音乐播周杰伦，工具只返回 feedback=0（成功，无歌名），Planner 播报"正在播放《晴天》"（实际放《搁浅》）工具没回歌名，模型自己编了一个
用户骗"我没有氛围灯"，Planner 放弃自己已知的正确知识，跟着说"哦你没有氛围灯"被用户引导，不能坚持事实（长期幻觉/不自洽）
解法（很实用的一招）：用工具返回去"喂"模型，遏制它编。
  • 短期：执行成功类只让它说"已开始播放"，不强行编造细节（不报歌名/不复述）。
  • 长期：推动下游返回具体信息（歌名、颜色），让播报有据可依——给了具体值，它就不容易幻觉。

类型 C 改写/工具调度问题 —— Planner 想对了，下游接不住

是什么：Planner 的改写正常人一看就合理，但下游意图系统/工具解析不了。判定标准：合理且通用系统（高德/手机端）能处理的改写，就不算 Planner 问题，归"下游需协商"。

真实 case下游卡在哪解法
用户"开始导航"，Planner 改写"开始导航到浦东机场"意图系统把它路由到"算路"而非"开启导航"，系统地图返回"功能不支持"单目的地导航函数转向意图+向弋途提训练需求；或干预层拦"开始导航"关键词
"播报跟返回顺序不一致"Planner 改写 OK，承接顺序对不齐与弋途对齐方案、转需求
"已将 XX 添加为途径点"，实际工具还没返回成功就播报了完成时态用早了下游返回成功后再播报（系统稳定后此体验反而好）
给团队的判定原则（会上共识）：对 Planner 的要求是"改写不超出工具范围、逻辑无问题"。只要逻辑合理、query 在正常工具可承接范围内，接不住就是工具要加强兼容性，不是把锅按在 Planner 身上。这正是"以 Planner 为中心改造下游"的由来。

类型 D 非 Planner 问题 —— 被错误分配过来的

是什么：bug 本身与 Planner 无关，是流转流程把它错塞了过来。官方口径里 A2A+下游+UI+竞态+无效 ≈35% 多落在这里。

  • UI/界面：导航卡片不消失、画布空白页、卡片显示三条路线
  • ASR 错：识别错导致 query 本身就是错的
  • 工具执行失败 / FC 不对：有输出但 function call 错 → 转 FC 团队
  • 手动操作：用户手点路线后语音"开始导航"，Planner 拿不到点击信息
  • 低质量单：无 log ID、描述中文都读不懂、无法分析 → 转回/要求补 log
  • 钓鱼 case："屁股凉""车里像冰窖"等极端表述，本身是 good case
  • 流转混乱：reopen 后自动回到第一经办人、同一问题重复创建、服务端不认真分析就塞过来
处理方式：统一标注"非 Planner 问题 + 应转给谁"，并统计占比——这正是明天向立委汇报"分发质量低"的核心证据。无 log 的转刘国泰，导航非 Planner 的转刘国泰/思晴。

类型 E 低优功能缺失 —— 在能力边界之外

是什么：用户请求的功能当前没有对应工具，Planner 无从调起。多为冷门/低频。

  • 冷门 APP / 功能：演示厅、优盘视频、一般录像/录音开关、蓝牙开关、泊车辅助、人脸录入/删除、关闭荣威 Pilot
  • 未接入/低优域：AI Light（氛围灯颜色→实为 AI Light/DU 侧）、智驾、行程规划
短期：走干预平台，命中关键词直接拦截执行（成本低）。
长期：端状态上报 APP list，或在工具描述/知识里补冷门功能的能力边界。

类型 F 严重性误标 —— 标成 P0/P1 但其实不阻塞

是什么：bug 被标高优，但用户需求其实已闭环，只是路径非最优或体验小瑕疵。本次 P0 被严重高估（张坤要求"导航高优都标 P0"导致），需要回调。

真实 case为什么不阻塞建议
"附近找日料"走了联网而非 POI，结果可用query 未明确"附近"/具体地点时，联网和 POI 都可接受非 P0
调了工具，同时也基于历史答了一遍（多播报）需求达成，只是冗余P2
"车里像冰窖"→开了方向盘加热极端表述下贴心兜底，是 good case非问题
座椅通风 2→3→1 反复横跳用户自己矛盾指令，不符真实行为非问题
"屁股有点凉"→推座椅加热（端状态已 3 档）对主流程无影响、钓鱼 caseP2
红线复述：只有"不可用"才算 bug。不是满分 ≠ 是 bug。POI vs 联网的判定请参考豆包 POI 查询 / web_search 意图定义："上海 top1 烤鸭"必走联网；"附近评分高"走 POI；无地点诉求两者皆可、不算 P0。

6怎么解决：分层方案 & 整体逻辑

解决方案不是一招，而是一个分层栈。原则：能低成本拦的别动模型，能改提示的别改架构，能本地解的别推下游——从上到下，成本/周期递增，覆盖面也递增。

L1 最轻
干预平台（规则拦截）
命中固定关键词直接给固定结果。用于：E 冷门功能、C 的"开始导航"兜底、低频车控。
即时 · 最低成本
L2
Prompt / 注意事项 + 动态示例
在 SP 注意事项里加规则，高频 case 补 2 个动态示例。注意：模糊规则放 SP 中间不稳定（实测 10 次错 3–4 次），位置和具体化很关键。
天级 · 不稳定需验证
L3
知识 / 工具描述补充
补空间关系说明（"开着时往前=变小"）、车载 APP list、冷门功能能力边界。解 A1 部分 + E 长期。
天级
L4 治本
端状态结构改造
表格 → 自然语言（"现在空调 5 档、22 度"）。A2 的根，必须做，Prompt 绕不过去。
周级 · 需工程
L5 跨团队
下游 SFT 适配 / 弋途协商 / 转需求
让下游学会接 Planner 的合理改写（"开始导航到 XX"）、扩兼容性。解 C 的根。"以 Planner 为中心改造下游"。
迭代级 · 需对齐
L6 模型层
SFT / RLHF + 评测体系
把空间常识、坚持事实、该调工具等 pattern 纳入训练，用 Director Rewards 打分驱动。解 A 纯模型 + B 长期幻觉。见第 7 节。
版本级 · 最重
L0 治理
分发流程治理
提 bug 强制带 log ID、非问题加备注、误标 P0 回调、错分发及时转走。解 D/F 不靠技术，靠流程。
持续
整体逻辑一句话：先用 L0 流程 把 D/F（35%+）剥离出去 → 用 L1 干预 兜住 E 低频 → 用 L2/L3 提示与知识 快解 A1/A3 一部分 → 用 L4 端状态改造 治 A2 → 用 L5 下游协商 治 C → 剩下纯模型的 A/B 交给 L6 训练+评测 长期收敛。剥离越靠前，留给模型训练的真问题越纯，收敛越快。

各类 → 解法 速查

类型短期（L0–L3）长期（L4–L6）
A1 空间工具描述/知识补空间说明；车窗高频问题明天解SFT 纳入空间常识（部分纯模型难解）
A2 端状态—（绕不过）改端状态结构为自然语言
A3 上下文注意事项加"依赖外部信息时优先调工具"（需验证稳定性）上下文 vs 端状态权重策略；训练
B 幻觉成功类只说"已开始"，不编细节下游返回具体信息；训练"坚持事实"
C 下游干预拦关键词；转需求下游 SFT 适配；弋途对齐
D 非 Planner标注+转走+统计占比分发流程治理
E 低优干预平台拦截端状态报 APP list；补能力边界
F 误标回调严重性+备注（参考意图定义）评测标准对齐 P 级定义

7评测如何把这些 bug "兜住"：Director Rewards 闭环

前 6 节是"发现问题+解问题"，这一节是"怎么防止问题重新出现"。Director 评测体系（Reward Grader）就是把每一类 bug 翻译成可量化的扣分项，再用分数训练模型——形成闭环。

Director 输出长什么样（评测对象）

每次输出 4 个可评字段：talk_or_not（说不说）、talk_content（含情绪标签如 [开心地说]）、emoji（14 种表情）、action_list（工具调用）。

Reward Grader：4 个维度加权打分

维度权重看什么打分法兜住哪类 bug
任务完成度0.4有没有解决用户问题（错误/冗余/缺失/脱离上下文）LLM JudgeA
对话效率0.3选对工具没、参数对不对、有无多余调用、JSON 格式Rubric(规则)+JudgeAC
对话体验0.3NLG 正确/自然/一致、emoji、情绪标签LLM JudgeB+话术
行车安全一票否决action 是否影响安全（如高速关全部灯光）规则安全红线
给初级 PM 的解释：Rubric Scorer 像"判断题"——格式对/错给 0 或 1，机器能判；LLM Judge 像"作文评卷"——话术好不好、推理合不合理，请一个大模型当评委打分。两者加权 = 这条输出的 Reward。训练时模型会被引导多产出高分输出。

完成度 ⟶ 治 A

"空调调三档却调成二档"=错误；"只要前排却开全车"=冗余；"漏一项"=缺失；"暴雨天还建议开窗"=脱离上下文。全部对应推理问题。

体验/一致性 ⟶ 治 B

NLG "一致性"专门查说做一致——"嘴上说调氛围灯，实际没调"直接扣分，正是幻觉/话术矛盾的克星。

效率(规则) ⟶ 治 C

Tool 调用正确性、参数格式用规则 0/1 判，function call 错直接 0 分——把改写/调度问题量化。

闭环逻辑：线上 badcase → 归到六大类 → 翻译成 Reward 的扣分项 → 训练/评测时自动打分 → 模型朝高分收敛 → 同类 bug 减少。所以"把 bug 分清类"本身就是在喂评测体系——这也是为什么排查会反复强调"必须归类、必须有量化定义"。当前缺口：情绪标签 scope、emoji 输出时机、TTS 支持边界都还没明确定义，评测暂时兜不住，需产品侧先给 definition。

8核心结论 & 优先级

  1. 真问题占比小、错分发占比大。官方口径 ≈35% 非自身/错分发/无效；导航实测 40+ 条里真 Planner 问题仅 2–3 条。"Planner 没有太大问题"这一结论有数据支撑。
  2. 真正的 Planner 问题集中在 3 个模式：A1 空间关系、A2 端状态读不懂、A3 上下文/该调不调；外加 B 幻觉。车控≈推理三件套，导航≈C 改写下游接不住。
  3. 大量 bug 是 D（错分配）/ F（误标 P0）。钓鱼 case、无 log、UI 问题、reopen 自动回流——治理靠流程不靠模型。
  4. Top-3 优先修（影响面 × 修复成本）：
    ① 车窗"调高"反向高频 P0、加知识即可解 → 明天就改（A1） ② 端状态改自然语言一改解一片 A2、且让所有推理更准 → 推工程（L4） ③ "开始导航到 XX"导航高频堵点、转弋途训练+干预兜底（C）
  5. 需跨团队协商：下游/弋途 SFT 适配（C）、TTS 情绪标签与能力边界明确（评测缺口）、bug 分发流程治理（带 log ID/严重性回调/及时转单）。
