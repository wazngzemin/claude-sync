一、DT为什么需要事件

DT中新增“事件"类信息的作用：本质还是做更全面信息同步，更及时的信息同步。也属于context工程的一部份

近期很多难点问题都可以通过事件这种消息类型进行解决

  • 一部分上下文注意力问题，尤其是用户的query跟最近发生的事件相关。
    • 手动操作类
      • 开始导航（这句话前点选了屏幕上某个路线）
      • 沿途介绍下风景（说完这句时已经更改了目的地，从去合生汇改到了去滴水湖）
      • 换个风格（用户说这句话前点开了生成的图片）
    • 系统自主事件类
      • 现在这首歌还挺好听的，谁唱的（自动切换到了一首新歌，包括谁做的等额外信息）
      • “联系四S店”（弹出了维养卡片，有两个选项，【联系4s店养护】【本月不再提醒】）
    • 异步消息
      • GUI操作
      • 工具（人脸注册等工具）
  • 补充更多context，做到更好的体验
    • 换首歌（大家聊到，精准提炼总结了人人对话事件里大家对于老歌的怀念，帮用户发现做更精准的搜索）

二、事件列表

处理类型说明：

  • ① Query关联事件：存入事件上下文队列，在用户提问时检索注入，确保回答与实时状态对齐，不主动说话或行动。
  • ② 重要主动服务事件：作为独立模型请求，立即触发AI互动，包括问候、提醒、预警、建议等。
  • ③ 目标关联事件：AI创建持续目标时向事件系统订阅，事件发生时携带目标上下文触发，实现精准闭环。部分事件同时具有②和③属性。

- “这条路”、“那条路”、“这个路线” → 检索导航/路线类事件

## 6. 不相关事件过滤
以下情况的事件不应被检索：
- 与Query主题完全不相关的事件（如问导航时不返回空调事件）
- 事件发生时间明显早于Query中暗示的时间范围
- 用户已在对话中明确知晓的旧事件

# 匹配判断准则

## 1. 条件匹配
将事件描述与每个目标的内容进行语义匹配，判断事件是否满足目标中设定的条件：
- **时间条件**：“10分钟后” → 匹配定时器到期事件
- **地点条件**：“到服务区时”、“快到超市时” → 匹配GPS到达/接近事件
- **人员状态条件**：“副驾睡着时” → 匹配副驾入睡/醒来事件
- **车辆状态条件**：“车速上80时” → 匹配车速阈值事件
- **下车条件**：“下车的时候” → 匹配电源OFF、安全带解开、车门打开等事件
- **持续性条件**：“一直到我笑为止” → 匹配用户发笑的表情/语音事件

## 2. 模糊匹配
当事件和目标措辞不完全一致但含义相同时，应判定为匹配：
- “提醒我拿后备箱的礼物” 匹配 “车辆熄火”或“到达目的地”
- “帮我照顾后排的宝宝” 匹配 “后排儿童哭闹”或“后排儿童入睡”
- “副驾睡着的时候提醒我” 匹配 “副驾入睡”事件

## 3. 一事件多目标
一个事件可能同时触发多个目标。此时应分别输出每个关联目标的触发结果。

## 4. 事件不重复触发
如果同一个事件在前一轮对话中已经触发过该目标，且目标未被更新或重置，则不应再次触发。

## 5. 优先级判断
当事件同时触发“提醒”和“终止”类目标时：
- 优先处理安全类、紧急类目标
- 终止类优先于提醒类
- 多个提醒按目标创建时间排序，先创建的先提醒

# 注意事项
1. 输出自然语言描述，供AI模型直接理解并生成对用户的自然交互
2. 触发提醒时，考虑当前车端状态（如后排睡觉则建议轻声提醒）
3. 关联用户记忆时，可补充偏好信息帮助AI做个性化响应
4. 如果目标已完成且不需要再跟踪，明确标注“后续建议”中是否需要调用goal_list_update删除目标
5. 若事件发生时车内正在进行重要通话或处于紧急状态，应在输出中标注“建议暂缓提醒”
