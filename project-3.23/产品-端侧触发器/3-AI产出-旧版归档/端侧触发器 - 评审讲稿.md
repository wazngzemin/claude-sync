# 端侧触发器 — 需求评审会讲稿

> 对照完整 PRD 逐节讲解。每节包含：讲什么、研发会怼什么、你怎么回。

---

## 开场（2 分钟）

今天评审端侧触发器 + 端侧信号库。两个模块一起评，因为信号库是触发器的输入底座，触发器是信号库的主要消费者，拆开评没有意义。

先说三句话把边界框死：

1. **我们不做 VLM 算法**。VLM 是黑盒，触发器只管"什么时候调、调什么、调完结果怎么处理"。
2. **我们不做 Adviser 的思考逻辑**。Adviser 怎么决策、说什么话术、什么时候推送——不归我管。触发器只管"条件命中了，通知 Adviser"。
3. **我们不做信号采集**。CAN 信号怎么读、摄像头怎么接——不归我管。信号库只管"别人写进来，我存好，别人来读，我给出去"。

这三个"不做"今天评审过程中任何人如果问到，我会直接指回这页。

## 第一节：需求背景（3 分钟）

**讲什么**：对照 PRD 1.0-1.2，讲清楚为什么现在必须做。

**核心论点**：当前方案是硬编码的 if-else，三个致命问题——

- 不可扩展：加一个任务 = 改代码 + OTA
- 无统一信号源：18 个任务各读各的 CAN 信号，结果不共享
- 无调度：P0 和 P3 撞车，谁跑得快谁赢——这是赌命

**预先堵住的三个问题**：

Q："短期方案也能跑，为什么非要升级？"
A：现在 15 个任务，短期方案能跑。下个季度变成 30 个任务、跨 3 个车型、每个车型信号定义不一样——你的 if-else 还能维护吗？这不是技术债，是技术高利贷。今天不做，三个月后你会在同一个会议室骂自己。

Q："信号库不就是个 KV 存储吗？有什么好设计的？"
A：KV 存储是技术实现，信号库是产品概念。信号库的核心价值不是存，是**标准化写入规范 + 统一订阅接口 + 结果可复用**。VLM 结果写进去，任何下游模块都能读。现在的问题是下车安全跑完的结果，停车位识别想用拿不到——信号库解决的就是这个。

Q："跟云端 context 模块的关系是什么？"
A：信号库是端侧的数据底座，context 是云端的数据消费方。信号库按策略上报，context 负责存储和分发。评审 context 的时候你们再找我，今天不展开。

## 第二节：整体架构（5 分钟）

**讲什么**：对照 PRD 第二章，翻到架构图。

**逐层讲解**：

**车端**：
- 左侧四类信号来源 → 写入信号库
- 中间信号库 ← 触发器读
- 触发器 → VLM 下发运行指令
- VLM → 信号库回写结果
- VLM → 触发器反馈任务状态

**云端**：
- 规则管理后台 → 触发器下发规则
- Adviser/Planner → 触发器动态注册任务
- 触发器 → Adviser/Planner 条件命中通知
- 信号库 → Context 上报

**关键设计点**：架构图里 Adviser 是**两条通路**——

1. Adviser 注册条件任务（Adviser → 触发器）
2. 触发器条件命中通知 Adviser（触发器 → Adviser）

这两条通路是独立闭环。Adviser 可以只注册不消费，也可以只消费不注册。

**预先堵住的问题**：

Q："为什么触发器不直接调 Adviser，而是通过信号库上报再通知？"
A：因为 VLM 结果不是只给 Adviser 用的。Planner 也要用、context 也要存、下游其他模块也可能消费。信号库是中转站，不是 Adviser 的专属通道。

Q："架构图里触发器和信号库是分离的，实际部署也是两个进程吗？"
A：部署形态我不定，研发定。产品层面它们是两个独立模块，各自有清晰的接口边界。你放一个进程还是两个进程，我不关心，我只关心接口对。

Q："触发器 → Adviser 通知和信号库 → Context 上报有什么区别？"
A：通知是事件驱动（"条件命中了，你快看"），上报是数据同步（"这是最新的信号快照，你收着"）。通知带 rule_id + 信号快照，上报带 task_id + 完整结果。两个通道，两个用途。

## 第三节：端侧信号库（8 分钟）

**讲什么**：对照 PRD 第三章。

### 3.2 信号写入规范

四个字段：key / value / source / timestamp。不复杂，但必须强制。

**预先堵住**：

Q："为什么只有四个字段？不用加类型/优先级/过期时间？"
A：类型从 value 推断，优先级是触发器的事，过期时间信号库自动管理。四个字段是最小必要集。多加一个字段就是多一个维护成本，我宁愿先不加，不够再加。

Q："同一 key 多个 source 写入，后写覆盖——不怕覆盖掉关键数据吗？"
A：信号库是状态存储，不是事件日志。你存的是"当前车门是什么状态"，不是"车门状态的变化历史"。如果要追溯历史，去 context 查上报记录。

### 3.3 信号分类

五类信号：车辆状态 / 用户识别 / OEM 信号 / 用户输入 / 场景信号 / 模型结果 / 任务状态。

**重点强调**：模型结果类信号（VLM 标签、VLM 描述）是信号库的核心增量价值。现在这些结果跑完就丢了，信号库让它们可复用。

### 3.6 VLM 结果状态定义

**这一页必须重点讲，因为这是最容易出 bug 的地方。**

五态：success / empty / failed / timeout / interrupted。

**核心原则**：
- timeout **绝对不能写成"无异常"**。VLM 没返回 ≠ 没问题。写"无异常"就是埋雷，下游以为安全，用户开门撞到障碍物——这是安全事故。
- failed 不触发下游主动服务。模型挂了就是挂了，不要强行给用户推一个"可能是 X"。
- P3/P4 不得覆盖 P0/P1 刚写入的关键结果。信号库里同一 key 被低优覆盖，下游拿到脏数据，P0 任务白跑。

**预先堵住**：

Q："empty 和 failed 对下游有什么区别？"
A：empty 表示"看过了，没有"——这是有效信息。failed 表示"没看成"——这是无效信息。Adviser 拿到 empty 可以说"一切正常"，拿到 failed 应该说"我没看到结果"。

Q："谁负责保证 P3 不覆盖 P0？信号库还是触发器？"
A：触发器。下发 P3 任务时检查目标 key 最近一次写入的 priority 字段，如果 ≤ 5s 内被 P0/P1 写过，拒绝写入或写入带 override_protection 标记。

## 第四节：端侧触发器（15 分钟）

**这是今天评审的核心。翻到 PRD 第四章。**

### 4.1 业务流程 + 两级判断

**讲什么**：触发器是"规则判断 + 任务调度 + 生命周期管理"。

**两级判断的核心**：

第一级：run_phase_conditions——"车现在在哪个人车阶段？"
第二级：trigger_conditions——"具体发生什么事了？"

**为什么分两层**：不是为了好看。是因为性能。run_phase_conditions 不满足时，本条规则**根本不订阅 trigger_conditions 中的信号**。一个行车中的任务，不会在下电后还在那监听信号变化。这是省电、省算力、省内存的工程决策。

**信号快照对齐**：同一规则的所有条件信号在 500ms 窗口内对齐后求值。为什么 500ms？因为 CAN 信号到达有先后——档位变化和车门解锁可能差 200ms。不等这 200ms，触发器会在"档位变了但车门还没解锁"的那一刻判定条件不满足，漏掉一次触发。

**预先堵住**：

Q："500ms 怎么定的？不同车型 CAN 总线延迟不一样。"
A：这是默认值，规则里可配。今天你需要给我一个合理默认值，我选了 500ms。你有数据就改，没有就用这个。

Q："两级条件其实就是两组 AND，为什么不合并成一个 AND 省事？"
A：一个是性能优化（不满足第一级就不订阅第二级的信号），一个是条件管理（PM 改触发条件不用关心运行时段逻辑）。你合并成一个 AND，18 个任务 × 每个任务 N 个条件信号全部订阅，信号库压力你自己算。

### 4.3 优先级

五级：P0（安全/即时）> P1（安全增强）> P2（持续服务）> P3（Always-on）> P4（动态）。

**这里有个必须现场拍板的问题**：P4——Adviser 动态任务的优先级——是否合理？

我的观点：P4 是合理的。因为：
1. Adviser 任务不是用户主动发起的，是系统"觉得应该做的事"
2. Adviser 任务可以被丢弃——用户不会因为"副驾疲劳关怀没触发"而投诉
3. 如果 P4 提到 P2，那动态任务会跟儿童照看、增强哨兵抢资源——这不合理

但这只是我的建议。如果 Adviser 团队认为某些场景（如安全 Adviser 注册的任务）应该是 P1，那我们在规则里支持配置。今天需要确认的是**默认值**。

**必须确认**：P4 默认优先级是否需要调整？

### 4.4 运行时段 + 时段切换

四个时段：离车后、上车前、停车时、行车中。

**重点讲时段切换**：行车中 → 停车时的过渡态。档位 D→P 是瞬变，但车门从 locked → unlocked 是用户操作，可能隔好几秒。过渡态期间，两个时段的规则共存：
- 行车中任务的 run_phase_conditions 失效后自动终止
- 停车时任务的 run_phase_conditions 全部满足后开始扫描

**不会出现"两个时段任务同时跑"**——互斥组保证 VLM 同一时刻只跑一个。

### 4.9 冷却策略

**两个关键设计**：

1. 冷却粒度是**按规则**，不是按 task_id。同一个下车安全任务，如果以后有不同的触发条件（比如"D→P 触发"和"车门解锁触发"两个规则），它们独立冷却。
2. notify_adviser 类型默认冷却 120s——比 VLM 任务的 60s 长一倍。为什么？因为 Adviser 通知是推送给用户的，60s 内连推两次"副驾疲劳"就是骚扰。

### 4.11 DSL 示例

**翻到三个 DSL 示例，逐条过一遍。**

强调 DSL 条件里的 `required` 字段：
- `required: true` → 信号缺失 ≥ 1s 跳过本拍
- `required: false` → 缺失不阻塞

**问研发**：这个 DSL 表达式引擎的工作量多大？条件操作符里时间窗和 since 本期不实现，剩余的比较/与或非/持续时间——能不能支持？

## 第五节：动态任务（8 分钟）

**翻到 PRD 第五章，这是另一个核心评审点。**

### Adviser 与触发器的交互路径

三条路径：
1. `run_vlm`：Adviser 要 VLM 跑一个识别任务 → 走正常调度
2. `notify_adviser`：Adviser 注册条件 → 触发器命中 → 通知 Adviser → Adviser 自己决定干什么
3. `notify_planner`：同上，目标是 Planner

### 通知策略——必须提前堵住的问题

**我主动把三个容易扯皮的点说了**：

1. **通知内容**：rule_id + task_name + 触发时刻信号快照（key/value 列表）。够不够？Adviser 还需要什么？现场确认。
2. **通知失败**：丢弃，不重试。如果 Adviser 需要可靠性——用 run_vlm，结果写信号库，Adviser 自己来查。通知通道是"尽力而为"，不是"保证送达"。
3. **冷却去重**：同一 rule_id 冷却期内不重复通知。120s 默认值。

**预先堵住**：

Q："通知失败丢弃——那用户下车时 Adviser 没收到通知怎么办？"
A：
- 下车安全这类 P0 任务走的是 run_vlm，不是 notify_adviser。结果写信号库，上报 context，Adviser 可以通过 context 拿到。
- notify_adviser 适用于非关键场景（如"副驾疲劳关怀"），丢了就丢了。
- 如果需要可靠性，请用 run_vlm + 信号库回查，不要依赖通知通道。

Q："Adviser 收到通知后又注册了新任务，新任务又触发通知——振荡怎么办？"
A：冷却 120s 是第一道防线。第二道防线是——Adviser 的设计者自己应该知道不要在通知回调里注册相同的规则。这不是触发器的问题，这是 Adviser 的逻辑问题。但我在通知内容里带了 rule_id，Adviser 可以据此去重。

Q："动态任务 16 条上限，满了怎么办？"
A：返回 rejected。注册方收到 rejected 后自行决定：是淘汰旧任务重新注册，还是放弃。这个数字 16 是我建议的，研发评估后给最终值。

### 5.6 三个场景

快速过一遍 Planner 定时任务、Adviser notify 任务、Adviser run_vlm 任务。这三个场景覆盖了动态任务的全部三种 action_type。

## 第六节：Fail 定义与验收（5 分钟）

**翻到 PRD 第六章，这一节不是讲故事，是立规矩。**

### 6.1 Fail 定义

11 种 Fail 类型，P0-致命的 6 种：

- 漏触发、误触发、优先级错误、信号库污染、队列阻塞、运行时段误判

**为什么这 6 个是 P0**：因为它们直接导致安全事故或功能不可用。

**特别强调**：信号库污染——"超时写成无异常"——这是最容易犯的错，也是最危险。测试团队请把这一条作为核心测试用例。

### 6.2 性能指标

7 个指标。最关键的三个：
- 条件命中→下发 VLM ≤ 1s
- P0 整体返回 ≤ 5s
- 超时释放 ≤ 1s

**问研发**：这三个指标在当前硬件上能否达到？如果不能，瓶颈在哪？

### 6.3 验收用例

12 条。我不逐条念，提三个最关键的：
- 下车安全 + Always-on（验证抢占恢复）
- 超时 → 不写伪结果（验证信号库污染防护）
- 动态任务弱网 + 重启（验证持久化）

## 结语（1 分钟）

总结三句话：

1. **我们在建的不是一个任务调度器，是端侧 AI 能力的基础设施**。信号库是数据底座，触发器是调度中枢，15 个 VLM 任务只是第一批消费者。

2. **本期不追求完美，追求闭环**。信号库 + 触发器 + VLM + 云端 context 四条链路全通，就是成功。规则管理后台、自然语言解析、用户自定义场景——下期再说。

## 附录：预判研发攻击清单

以下是评审会上研发大概率会攻击的点，以及你的反击策略。不是让你在会上念——是让你准备好。

### 攻击 1："两级判断就是伪概念，就是 AND 条件"

**反驳**：不是。第一级不满足时不订阅第二级信号——这是实打实的性能优化。你合并成一个 AND，18 个任务每个都要订阅全部条件信号，信号库每秒被读多少次你算过吗？

### 攻击 2："信号库就是 KV 存储，用 Redis 就行"

**反驳**：信号库的价值不是存储引擎，是标准化 schema + 写入规范 + 订阅机制 + 上报策略。你用 Redis 存，那 VLM 结果的 status 枚举谁来校验？信号版本谁来管理？过期谁来清理？这些都是信号库的职责，不是 KV 存储的。

### 攻击 3："冷却粒度按规则太细了，按任务就行"

**反驳**：现在是 15 个任务，大部分一个任务只有一个规则。但以后呢？下车安全可能要区分"左侧下车"和"右侧下车"，两个规则触发的条件不同。如果按任务冷却，第一个触发完第二个被冷却挡住——漏触发。P0 漏触发你负责吗？

### 攻击 4："通知失败丢弃是不是太粗暴了"

**反驳**：通知通道的设计目标是"轻量、事件驱动"，不是"可靠消息队列"。如果通知要保证送达，需要 ACK + 重试 + 持久化 + 超时队列——这一套做下来，触发器的复杂度翻倍。Adviser 需要可靠性？用 run_vlm，结果写信号库，Adviser 自己轮询。不要用通知通道做可靠传输。

### 攻击 5："16 条动态任务上限怎么定的，是不是拍脑袋"

**反驳**：是拍脑袋。我有两个依据：第一，同时运行的 P4 级别后台任务超过 16 个本身就不合理——你哪来那么多"顺便做一下"的事？第二，每条动态任务都要持久化 + 订阅信号 + 占用扫描周期，16 条是一个安全的默认值。研发评估后给我最终数字，我接受调整。

### 攻击 6："500ms 快照窗口太短/太长"

**反驳**：给数据。CAN 总线上档位变化到车门解锁的典型延迟是多少？你有数据我就改，没数据就用 500ms。重点是这个窗口**可配置**，不是写死的。

### 攻击 7："触发器要做的太多了，能不能砍掉动态任务和 Adviser 通知，第一期只做预设规则"

**反驳**：可以。如果动态任务是阻塞项，第一期可以只做预设规则的 run_vlm 调度，Adviser 通知和动态注册留接口。但你要明确告诉我：砍了动态任务，开发周期能缩短多少？如果砍了只省两周，不如不砍——因为第二期补这个功能时，触发器已经上线了，加功能就是改线上系统。
