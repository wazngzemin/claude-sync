§2 🟢 预设条件任务（20个场景详细trace）

预设条件任务特征：在⑬配置平台预先配置好条件→动作链路。
A类：车况预设 - 条件来自CAN车控信号（档位/温度/车速），触发后白名单车控直执
B类：视觉标签预设 - 条件来自VLM已精调枚举标签，触发后push Director执行分发
C类：组合条件 - 多种信号源组合，触发后push Director或白名单车控

2.1 A类：车况预设条件任务（3个场景）

A类特征：条件信号来自CAN车控总表（端状态已有），触发后执行白名单车控动作。
代表场景：充电桩开盖（链式触发）/ 雾灯（双向触发）/ 雨雪模式（双向触发）

Case A1：充电-自动充电口开启【P0 · 链式触发代表】

场景描述

★ 评审可能被问：
韩杰"为什么档位变更要拆成'穿越事件'而不是直接 == P？"
回答：避免重复触发——挂P档后车辆保持P状态，如果直接 == P 每个5s扫描周期都会触发。穿越事件只在R→P那一刻产生一次，符合"用户挂P这个动作"的语义。

刘杨"oneshot VLM调用怎么避免延迟？"
回答：oneshot VLM是按需调用（非周期），延迟~1.5s（1080P推理时长，丁彬7.2数据）。这个延迟用户能接受——挂P档后1.5s开盖，体感是"立即"。

Case A2：天气路况保护-雾天后雾灯【P0 · 双向触发代表】

场景描述

识别到雾天，自动开启后雾灯保护行车安全；天气恢复后执行恢复策略（关闭后雾灯）。难点 = 避免重复提醒（持续雾天不要反复触发）→ 频控解决。这是双向触发场景的代表：同一信号两个方向（变更开启 / 变更关闭）。

频控策略（双向各自独立）

嘉锋7.2原话："难点是避免重复提醒" + 泽民"我有配置——周期内几小时/几天/自然日一次"。
落地：execute_policy可配置维度 = 单次上电/单日/单周/单月/自然日/节假日。雾天场景默认配置：开启方向 per_day=2（避免反复触发）+ 恢复方向 per_day=2。这是触发器frequency能力的落地案例。

双向同时触发（边界）设计上不可能（weather_type不能同时含雾和不含雾）；防抖1帧保证
★ 评审可能被问：
董均"VLM上报和外部天气数据冲突谁为准？"
回答：本场景外部天气数据为主、VLM为备。VLM是兜底数据源（外部数据失效时启用）。多源冲突不在触发器层解决——触发器只看一个信号源（配置时指定），多源融合是Context层的事。

张荣"用户手动关了又自动开怎么办？"
回答：触发器检测到用户手动操作（fog_light_switch=off信号）→ 标记"用户意图" → 本场景本周期frozen → 不再自动开。这是产品规则，写在场景配置的"暂停推荐"字段里。

§4 🟣 动态 Advisor 视觉事件（5个代表场景详细trace）

动态 Advisor 特征：用户对话即时拆解 → 动态Advisor⑰注册触发器 → 触发器下发端侧VLM SP → AI Box周期观测 → 结果回传 → 判定 → push动态Advisor。
5种事件类型：出现/开始/结束/存在/移除
本节选取：5种类型各1个代表case

事件类型总览

事件类型语义运算符代表case
{位置}出现{非持续视觉行为}瞬时动作触发，单次appear / eq比OK手势 / 打哈欠
{位置}开始{持续视觉行为}状态切换为持续change_to + duration开始睡觉
{位置}结束{持续视觉行为}持续状态结束change_to（反向）结束睡觉
{位置}存在{物品/动物/人物}持续状态判定eq True存在小狗
{位置}移除{物品/动物/人物}从有到无change_to（反向）移除小狗
★ 重要区分："开始/结束持续性行为" vs "出现/移除物品"：
- 开始/结束：关注的是行为状态变化（睡觉：睁眼→闭眼），需要监听多帧
- 出现/移除：关注的是物体存在变化（小狗：从无到有），单帧判断即可
这影响VLM实现方式和触发器频控策略。

Case D2：{位置}开始睡觉【持续性行为开始代表】

场景描述

用户说"留意下后排宝宝"。动态Advisor拆解为"开始睡觉"事件——要求从没有睡着（睁眼）→睡着（眼睛一直闭合），而非对单帧图片的判断。

条件组合逻辑

#信号逻辑说明
①eye_state从睁眼→闭眼（change_to）需要监听变化，非单帧判断
②闭眼持续时间 ≥ 3秒duration_gte 3s避免瞬眼误判
重要：本场景的关键是"监听变化"而非"状态判断"。
- 错误做法：单帧判断eye_state==闭眼 → 每个周期都触发
- 正确做法：从睁眼→闭眼（change_to）+ 持续时间判定 → 仅触发一次
