## 定义

**AI First 重设计——别把你的人生耗在"永恒治理"上**。黄钊这篇文章表面讲"数据治理决定 AI 准确率"，但题眼是一个**反转**："关键恰恰不是数据治理"——治理是无底洞，跟着走会错过 AI 最大红利。真出路：**不要先产数据再治理，而是一开头就重新设计，让数据/Skill/触发条件从源头按规则产生**。本教程把它钉到你的工作：你「上线前问题汇报表」里那些 Skill / 触发条件清单的坑，**短期止血靠治理，根治靠重设计**，让它们以后根本不需要你天天救火。

> ⚠️ 最容易读错（我第一遍就错了）：把它读成"要好好做 Skill 治理"。作者亲口否定了这个结论。

## 一、先收下表层结论：模型是最后一层接口（21/95/65）

Anthropic 同一模型版本下：无 Skills **21%** → 有 Skills **95%**（+74pt 全来自治理）→ 不维护一个月 **65%**（−30pt，模型没变）。横跳证明：变量在治理层，模型只是接口。

**精确映射到你：**

| 文章 | 你的 Planner / 场景知识 |
|------|------------------------|
| Skills | 场景知识 / Selector SP / 23 工具描述 / 触发条件清单 |
| 无 Skills → 21% | 「上线前问题汇报表」的【Skill 质量差】：初版粗写、没实车验证、写了车厂没有的能力 |
| 不维护 → 65% | 信号库不锁版、清单从老表迁移后口径漂移 |
| 模型是最后一层接口 | Seed 1.8 已够强；决定 Planner 准不准的，是你喂的 Skill/数据 |

**这节只是踏脚石**：让你别再"一遇 badcase 就怪模型/调 prompt"（那 74 分模型给不了）。但到此为止还是表层。

## 二、关键反转：把"持续治理"当终极答案 = 掉进无底洞

文章第 7 节"透视"是题眼。黄钊先替你写出大多数人的结论（"那就好好做数据治理呗"），然后直接否定：

> "我觉得关键恰恰不是这个……会错过 AI 带来的最大红利。数据治理这事难度没变化，你整 3 年的数据治理，世界早不知道什么样了。"

**为什么是无底洞——三条规律在你身上的体现：**

| 文章论点 | 在你工作里长什么样 |
|---------|-------------------|
| 数据难度 ≠ 技术难度："知道怎么做 ≠ 能长期做到" | 你知道该锁版、该实车验证、该统一 key——但让你+刘洋+韩天亮+上汽数年都做到，几乎不可能 |
| 治理像治河污染，难在无限期持续，否则反弹 | 这次清干净，下版本信号库一变、上汽 DNA 一调，又乱回去 |
| **人是污染源**：越自主、越有创造力的人越不愿规整贡献数据 | 触发条件来自一堆有自主性的产品/研发/车厂，治理永远追不上 |

**结论**：把自己定位成"那个永远在治理清单的人"，就是在和"人是污染源"这条规律死磕——越治越累，还显不出价值。这正是作者警告的坑。下面第三节给你换打法。

## 三、真出路：AI First 重设计你的 Planner / 触发器（主体）

> 判据一句话：**这个环节，能不能让 AI 来干、人只做审定和兜底？** 前 3 个重设计把脏活从源头砍掉，第 4 个是不能绕、必须自己占的护城河。

### 重设计 1：触发条件清单——从"老表迁移 + 永久清洗"到"DSL 先行 + 源头合规"

**为什么**：你台账"行1"那 5 条（运算符配不出、key 不统一、类型标错、与上汽 DNA 对不上）全是"先产后治"的产物——人工旧表抄进来，事后逐条救火。根因是：**进库时没有结构约束，什么脏数据都能进。** 重设计的本质是：先立结构规范，让脏数据进不来。

**Step 1（半天）：定一版触发器 DSL 字段规范。** 每个触发条件必须由这些受约束字段构成，缺一不可：

```
触发条件 DSL 规范 v1（草案）
├─ signal_key      必填 · 唯一英文 key（全局不重复，禁止 states/status 混用）
├─ signal_name_cn  必填 · 中文显示名
├─ value_type      必填 · 枚举：bool | number | enum | str（禁止把枚举标成 str）
├─ operator        必填 · 枚举：==, !=, >, <, >=, <=, changed_to（number 类型必须支持后四个）
├─ value_domain    必填 · 取值域（number 给范围如 [0,100]；enum 给全集）
├─ unit            number 必填 · 单位（℃ / % / km / km·h⁻¹）
├─ source          必填 · 枚举：一手实车 | 二手老表迁移
├─ verified_date   必填 · 最后实车验证日期（二手未验证填 null + 标红）
└─ lib_version     必填 · 信号库版本（如 v2.5），锁版后不随意变
```

**Step 2（1 小时）：写一条校验规则 / 脚本。** 任何新条目入库前自动检查：9 个字段是否齐全？number 类型有没有配齐 4 个比较运算符？source=二手且 verified_date=null 的，自动标红进"待验证"。

**Step 3（持续）：新条目只能两种方式进。**（a）人填结构化表单（字段同 DSL，填不全不让提交）；（b）AI 从需求文本按 DSL 自动生成草案（用下方 Prompt），人审定。

**Step 4（一次性）：存量止血回灌。** 把现有清单逐条套进 DSL，缺字段的补、二手的标红、和上汽对不上的（9 条）单列"待确认"。**这是一次性动作，不是长期工作。**

**真实例子（你台账里的"温度>26℃ 配不出"）：**

```
旧（先产后治）：信号"座舱温度"，运算符只有"等于/变更为" → 配不出"温度>26℃" → 出 badcase 再补
新（DSL 先行）：
  signal_key: cabin_temp
  signal_name_cn: 座舱温度
  value_type: number
  operator: [==, !=, >, <, >=, <=, changed_to]   ← number 类型强制配齐，"温度>26℃"天生可配
  value_domain: [16, 32]
  unit: ℃
  source: 二手老表迁移
  verified_date: null   ← 标红，待韩天亮确认
  lib_version: v2.5
```

**Prompt 模板——从需求文本生成合规 DSL 草案（可复制）：**

```
你是触发器 DSL 生成助手。我给你一句需求，请严格按下面字段输出一条触发条件，字段缺一不可，number 类型必须配齐 6 个运算符，拿不准的字段标"待确认"，不要编造取值域：

字段：signal_key（唯一英文,小写下划线）/ signal_name_cn / value_type(bool|number|enum|str) / operator / value_domain / unit / source / verified_date / lib_version

【需求】：（例：电量低于 10% 时提醒充电）

输出一个 DSL 代码块 + 一行"我不确定、需你确认的字段"。
```

**常见错误**：① 为兼容历史接口保留多个 key——这是实体歧义温床，宁可一次性和刘洋统一命名改干净；② 把"DSL 先行"做成又一张要人工维护的大表——它的价值是**校验 + 生成自动化**，不是多一张表。

**检查点**：随便拿一个新触发需求，你能不能不碰旧表、直接用上面 Prompt 生成一条合规 DSL？能 → 重设计 1 成立。

### 重设计 2：Skill / 示例生产——从"人工粗写"到"能力白名单 + 模板生成"

**为什么**：Skill 第一版人工粗写，所以"写了车厂不具备的能力（如观影模式一级指令）""描述不清误导调用"。根因是：**写 Skill 时没有'车厂到底能干什么'的硬约束。** 重设计的本质：先立能力白名单，写 Skill 时强制对照。

**Step 1（2 小时）：建车厂能力白名单。** 把 D6X 真实支持的能力逐条登记，这是 Skill 写作的"硬边界"：

```
车厂能力白名单（节选示例）
| 能力 | 是否支持 | 调用路径 | 可控参数 | 备注 |
|------|---------|---------|---------|------|
| 座舱温度调节 | ✅ | vehicle_basic_control | temp[16-32] | — |
| 车窗开合 | ✅ | vehicle_basic_control | window, position[0-100] | — |
| 观影模式（一级指令） | ❌ | — | — | 车厂无此一级指令，禁止 Skill 写成可直接下发 |
| 宠物模式 | ⚠️走预置应用 | 预置应用，非编排 | — | Skill 必须写明走预置应用 |
```

**Step 2（1 小时）：定 Skill 写作模板。** 每条 Skill 必须含这些字段，且"调用路径"必须在白名单里能查到：

```
Skill 写作模板
├─ skill_name        Skill 名
├─ trigger_intent    触发它的用户意图（举 2-3 个真实 query）
├─ capability_ref    对应白名单里的哪条能力（必须可查到，否则不让上线）
├─ call_path         调用路径（工具名 / 预置应用），必须与白名单一致
├─ params            参数及取值域
├─ negative_examples 反例：哪些情况不该走这条 Skill
└─ verified          是否实车验证过（日期）
```

**Step 3（持续）：新 Skill 由 AI 按模板生成初稿 + 自动对照白名单校验**（用下方 Prompt），人只审定。`capability_ref` 在白名单查不到 / `call_path` 不一致 → 直接打回。

**真实例子（观影模式 / 宠物模式）：**

```
旧：Skill 写"用户说看电影 → 下发'观影模式'一级指令" → 车厂没有 → 调用失败
新：AI 按模板生成时，capability_ref 去白名单查"观影模式" → 命中 ❌不支持 → 自动打回，
    提示"车厂无观影模式一级指令，需拆成 调暗氛围灯+降座椅+开屏 的编排，或走预置应用"
```

**Prompt 模板——按白名单生成/校验 Skill（可复制）：**

```
你是 Skill 写作助手。下面是【车厂能力白名单】和一条要写的 Skill 需求。
请按模板生成 Skill，并强制校验：capability_ref 必须能在白名单里查到；call_path 必须和白名单一致；
若需求涉及白名单中"❌不支持"或"⚠️走预置应用"的能力，不要硬写成可直接下发，明确指出并给替代方案。

【车厂能力白名单】：（粘贴）
【Skill 需求】：（例：用户说"我要看电影"）

输出：Skill 模板代码块 + 一段"白名单校验结论（通过/打回，原因）"。
```

**常见错误**：白名单不全 = 校验形同虚设。先覆盖高频能力，再逐步补全；白名单本身要锁版、标来源（同重设计 1 的纪律）。

**检查点**：拿一条你怀疑"写了车厂没有的能力"的旧 Skill，丢进上面 Prompt，AI 能不能把它打回并给替代方案？能 → 重设计 2 成立。

### 重设计 3：badcase 闭环——从"人工逐条救火"到"AI 主导归因 + 你裁决"

**为什么**：你现在人工看 badcase、人工判断是链路/工具/上下文/模型问题，慢且累。badcase 修复这件事本身"能完全由 AI 干、人辅助"。

**Step 1：每个 badcase 走 AI 自动归因。** 用下方 Prompt，让 AI 按你 CLAUDE.md 的四层路径（实体歧义 → 数据陈旧 → 检索失败 → 模型能力）归因 + 给修改草案，你只裁决。

**Step 2：把归因结果回灌成重设计 1/2 的输入。** 关键升级——AI 归因时多输出一列"这能不能重设计根治"：
- 若根因是"实体歧义/key 不统一" → 沉淀进重设计 1 的 DSL 规范
- 若根因是"写了车厂没有的能力" → 沉淀进重设计 2 的白名单
- **这样你修的 badcase 越多，未来要修的越少**（救火 → 防火）。

**AI 四层归因 Prompt 模板（可复制）：**

```
你是我的 Planner badcase 归因助手。按四层顺序归因，命中一层就停，不要直接归到"模型不行"，最后给修改草案，由我裁决。

【粘贴 badcase】
- 用户输入：
- 当时车况/记忆/目标（env_info / memory / goal_list）：
- Planner 实际输出（action_list / talk_content）：
- 预期行为：

输出表格：
| 归因层 | 证据(具体到哪句 Skill / 哪个字段) | 修改草案 | 这是"治理止血"还是可"重设计根治"(若可，归到 DSL 规范 or 能力白名单) |

四层顺序：1 实体歧义（描述/ key 乱）→ 2 数据陈旧（信号/枚举过期）→ 3 检索失败（示例在库没命中）→ 4 模型推理（前三层都排除才认定）。
像审计一样冷酷，不要替模型开脱。
```

**常见错误**：只用它救当下的火，不回灌。一定要填最后一列，否则又退回"永恒治理"。

**检查点**：下一个 badcase，你能不能只当"裁判"——AI 出归因+草案，你 5 分钟内裁决并决定是否回灌规范？能 → 重设计 3 成立。

### 重设计 4：一手 context 治理——不能绕，是你唯一值得深耕的护城河

**为什么**：文章判据"规则在人之上、举动被充分数据化 → 数据精度极高"（骑手/司机/主播）。你的车端正是这种场景：**车况 {{env_info}}、用户记忆 {{memory}}、视觉感知 {{status}}** 是被传感器/系统充分数据化、规则在人之上的一手数据——别人拿不到、AI 也替代不了。前 3 个重设计是"把脏活砍掉"，这一个是"把唯一干净、唯一值钱的数据治理好"。

**具体怎么治理一手 context（这才是你该花时间的地方）：**

| 一手 context | 治理动作 | 衡量指标 |
|-------------|---------|---------|
| 车况 {{env_info}} | 确认 Planner 真的用上了关键字段（电量/温度/位置/车速），而不是忽略 | "本该用 env_info 却没用"的 badcase 数 |
| 用户记忆 {{memory}} | 记忆的写入/召回质量：该记的记了吗？召回准吗？ | 记忆相关意图的命中率 |
| 视觉感知 {{status}} | 视觉信号的时效与准确，喂给 Planner 的格式是否好消费 | 视觉相关决策的正确率 |

**做法**：把你"维护二手人工 Skill"的时间，逐步挪到"让 Planner 更好地消费这三类一手数据"上。这既是抗 AI 折旧的能力，也直接接你的"认知大脑 → 具身智能"主线（认知大脑的护城河 = 它能消费别人没有的一手 context）。

**检查点**：你能说出这三类一手 context，各自当前最大的一个质量问题、和一个改进动作吗？

## 四、判断：你是在"治理"还是在"重设计"

**还在"永恒治理"（坑里）：**
- 核心工作是"持续维护/清洗 Skill 和触发清单"；每次版本变更都要重对口径、补运算符；最忙时在救 badcase 的火；汇报的是"我在优化"。

**已在"AI First 重设计"（出坑）：**
- 你在定**结构规范/模板/白名单**，让新数据进来即合规；把 badcase 归因/Skill 生产/清单维护改成"AI 主导、你裁决"；时间越来越多花在**一手 context** 上；汇报的是"我重设计了 X，让这类问题从源头少发生 N%"。

## 五、升到转型主线：缝隙理论与具身智能

文章小结引《缝隙理论》："用好 AI 第一关键不在理解 AI 这门技术，而在身处缝隙中理解人与 AI 应该如何综合。"

| 诱惑（无厘头） | 真正的护城河 |
|--------------|-------------|
| 更懂 Seed 模型 / "模型即产品" | 重新设计"人—Planner—下游 Agent—车"的协作关系 |
| 让 Planner 话术更温柔 | 让认知大脑在缝隙里组合一手 context + 判断力 |
| 和脏的二手数据死磕治理 | 去占"规则在人之上"的干净一手 context |
