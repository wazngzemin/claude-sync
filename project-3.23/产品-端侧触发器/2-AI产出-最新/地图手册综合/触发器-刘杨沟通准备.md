## 2. 必问清单（按本次会议优先级排序）

### 🔴 A. 信号 → 参数 → 运算符的真实能力边界　【最重要，对应 item1/4 + 清单审计】

**问什么**
- 引擎对每个参数**实际支持哪些判定方式 / 运算符**？
  - **数值型**（温度 / 电量 / 距离 / 车速 / 里程）：支持 `>`、`<`、`≥`、`≤`、区间 吗？
    （我核对清单发现：31 个数值信号原本运算符只写了"等于 / 变更为"，我已批量补全，需你确认引擎真支持）
  - **状态型**：`变更为`（A→B，如"门：关→开"）能用吗？——这是分上车/下车的命根（需求5）
  - **抖动过滤**：怎么配？有没有统一规范（区间 + 周期）？
  - **双条件**：AND / OR 支持到什么程度？（机制表里写"430 版本先 follow 触发器现有机制"）
- 枚举值类型（关闭/1档/2档/3档、路况）后台是按 enum 校验还是 str？（我已把 13 条错标 str 的改成 enum）

**问到什么程度**
- 拿到一句明确结论：**哪些条件现在就能配、哪些要开发**。
- 最好确认："**就按我补全后的清单为准**"，或拿到一份"参数 × 支持运算符 × 取值类型"的真实能力对照。

**为什么**
- 这直接决定**需求1 / 主动服务能不能落地**。配场景的人（我/运营）按这个配，配错就是 badcase。
- 这是我本周主线，必须钉死，否则给意图的《条件清单》也定不了稿。

---

### 🟠 B. 弋途接口 & 协议一致性　【item2】

**问什么**
- 6 个 API（signals / actions / scene / update / query / status）+ DSL 五结构，对内（后台配置）和对外（弋途）是不是**同一份**？鉴权（HMAC-SHA256）怎么接？
- `query` 接口能不能列出"**已注册了哪些事件可订阅**"（需求12）？支不支持自动化验证？
- `conditions` 与 `dynamic_conditions` 能不能**合成一套**（需求6）？车门能不能拆**独立条件 / 任意位置**？

**问到什么程度**
- 确认"同源"这件事 + 拿到**对外文档维护方式**（我基于后台 DSL 同源维护）+ **弋途联调排期**。

**为什么**
- 我要对外交付弋途，意图模型也要对接同一协议；不一致 = 两边漂移、返工。

---

### 🟠 C. 导航 / POI / 地理围栏可行性　【item3，领导点名 P0】

**问什么**
- 引擎现在认不认 `距离 distance` / `目的地 address` / `收藏地址`（清单序号 121 / 124 有这些字段，但**实际能不能当触发条件**）？
- "**接近 / 离开 POI**"事件要不要**新开发**？工期多久？
- 地理围栏（圆心 + 半径 + 进出判定 + 防抖）**端侧算还是云端算**？**谁提供实时定位 + POI 坐标**？

**问到什么程度**
- 拿到判定："**能做 / 要开发 / 做不了**" + **工期** + **卡在谁**（云端定位由谁提供）。

**为什么**
- 领导点我名，我得能回话"能不能做、多久、卡在哪"，不能含糊。
