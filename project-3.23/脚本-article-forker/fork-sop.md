# Fork 文章 SOP

> 我的操作手册 — 把每篇好文章变成运行中的工具，而不是收藏夹里的尸体

---

## 一、为什么要这样做

**旧循环（大多数人）：**
读文章 → 觉得有道理 → 收藏 → 遗忘 → 再读…
> 知识在「消费」循环里转，永远不产生复利

**新循环（fork）：**
读文章 → 找到「代码」→ AI 做成工具 → 使用 → 迭代 → 持续运行
> 知识进入「运行」态，持续产生复利

---

### 核心原则

> 知识只在运行中产生价值。凡是不可运行的积累，几乎都是无效积累。「读」让你知道，「fork」让你拥有。

---

### 收藏 vs Fork 对比

|  | 收藏（消费） | Fork（运维） |
|---|---|---|
| **动作** | ~~存起来以后看~~ | 现在就让它跑起来 |
| **你的身份** | ~~读者~~ | 机械师 |
| **文章的身份** | ~~阅读材料~~ | 零件 / 源代码 |
| **3个月后** | ~~忘了~~ | 工具还在运行，已经迭代了好几版 |

---

## 二、操作 SOP（每次读到好文章时执行）

### 第 0 步：加载 Skill 标准

如果你在 Claude Code / Codex 里执行 article-forker，先让 agent 读取本目录的 `SKILL.md`。

这个 skill 的结构是：

- `SKILL.md`：入口，只放触发条件、主流程、gotchas 和验证要求
- `references/quality-standard.md`：高标准 fork 细则，做 `source.md / concept.md / tool.html` 前读取
- `references/feishu-workflow.md`：只有需要上传飞书时读取
- `assets/`：`source.md`、`concept.md`、`metadata.json` 模板
- `scripts/validate_fork.py`：产物完成后的本地检查

这套结构遵循一个原则：入口短，细节按需加载。不要把所有上下文塞进同一个超长提示词。

### 第 1 步：打住（3 秒）

你正要点收藏/转发/存笔记——**停下来**。问自己一个问题：

> **「这篇文章里的代码是什么？」**
> 即：它里面有没有一个可以被工具化的方法论、框架、流程、步骤？

---

### 第 2 步：判断文章类型（30 秒）

不是所有文章都值得 fork。快速判断：

| 文章类型 | 能 fork 出什么 | 值不值得 |
|---|---|---|
| **方法论型**（如何做用户访谈、写作5步框架） | 清单工具、流程引导器、提示词模板 | ✅ 最值得 fork |
| **框架型**（分析竞品的3维度、决策矩阵） | 分析模板、决策辅助工具 | ✅ 非常值得 |
| **案例型**（XX公司如何做增长） | 提取可复用模式，做成检查清单 | ⚠️ 看情况 |
| **纯观点型**（为什么我看好XX） | 思维透镜 | ➖ 一般不值 |
| **新闻/资讯** | — | ⬜ 跳过 |

---

### 第 3 步：提取「代码」（2 分钟）

用自己的话回答三个问题：

- [ ] 这篇文章的**方法/步骤**是什么？（不是观点，是动作）
- [ ] 这个方法的**输入**是什么？（用户要提供什么）
- [ ] 这个方法的**输出**是什么？（用户会得到什么）

**示例：** fork《别再读文章，要fork文章》这篇文章本身
- **方法：** 读到文章 → 识别可工具化的方法论 → 让AI做成工具 → 使用 → 迭代
- **输入：** 一篇好文章
- **输出：** 一个可运行的工具 + 持续迭代的习惯

> ⚠️ 如果你 2 分钟内提取不出明确的「方法/步骤」，这篇文章大概率是观点型，不值得 fork。正常收藏即可，不必强求。

---

### 第 4 步：扔给 AI（5 分钟）

把文章全文 + 下面的提示词一起发给 Claude 或 ChatGPT：

---

**提示词模板（直接复制使用）：**

```
我刚读了一篇文章，我想把它 fork 成一个我能实际用的工具。

先帮我做三件事：

1. 【提取代码】这篇文章里可以被工具化的方法论/框架/步骤是什么？
   像描述一个函数一样写清楚：方法名、解决什么问题、输入、输出、具体步骤。

2. 【工具提案】基于提取的代码，建议做一个什么工具？要求：
   - 必须是单个 HTML 文件，我双击就能打开
   - 不需要任何安装和配置
   - 界面干净好看
   - 非技术用户能直接用

3. 【直接生成】不要只给建议，直接把那个 HTML 文件的完整代码写出来。
   代码里不要有 TODO 和占位符，我要能直接保存运行。

文章全文如下：

[ 在这里粘贴文章全文 ]
```

---

> ⚠️ **关键：** 发送前回顾一下你在第 3 步提取的「代码」。如果你觉得 AI 可能理解偏了，在提示词里加一句：「这篇文章的核心方法是 XXX，请围绕这个做工具」。你的判断比 AI 的自动提取更准。

---

### 第 5 步：保存 & 跑起来（2 分钟）

AI 给你 HTML 代码后：

- [ ] 全选复制 AI 输出的代码
- [ ] 打开电脑上的「文本编辑」（Mac）或「记事本」（Windows）
- [ ] 粘贴进去
- [ ] 保存为 `工具名.html`（注意后缀是 `.html` 不是 `.txt`）
- [ ] 双击这个文件 → 浏览器自动打开 → 工具跑起来了

> Mac 用户注意：「文本编辑」默认保存为 .rtf 格式。保存时选「格式 → 制作纯文本」，然后文件名写 xxx.html。或者更简单：让 AI 直接帮你保存文件（Claude Code 里说「帮我保存到桌面」就行）。

---

### 第 6 步：用它（持续）

第一版不会完美。**先用 3 天。**

用的过程中你一定会发现：「这个地方不对」「缺了个功能」「这个设计不方便」。

**这些不满才是最有价值的东西** —— 它们是你下一次迭代的输入。

---

### 第 7 步：迭代（每次 5 分钟）

用了 3 天后，把你的不满告诉 AI：

**迭代提示词模板：**

```
这是我之前 fork 出来的工具（附上当前代码或描述）。

用了几天，我发现这些问题：
[ 写你的具体不满，越具体越好 ]

帮我改一版。改完直接给我完整代码。
```

每次迭代只改 1-2 个点，不要一次改太多。小步快跑。

---

## 三、怎么判断 fork 成功了

- [ ] 工具在你手机/电脑上**真的在用**（不是保存了没打开过）
- [ ] 你至少**迭代过一次**（第一版几乎不可能完美）
- [ ] 你已经**忘了原始文章的标题**，但工具还在帮你
- [ ] 你能用一句话说清楚**「这个工具帮我解决了什么问题」**
- [ ] 对高价值文章，已经生成 `source.md / concept.md / tool.html / metadata.json`
- [ ] 已运行 `python3 scripts/validate_fork.py forks/<fork-dir>` 并通过
- [ ] 如果上传飞书，`metadata.json` 里已经记录 Source / Concept 的 doc_id 和 wiki URL

> ⚠️ **反面信号：** 如果 fork 完一个工具，一周内你一次都没打开过——说明你提取的「代码」不对，或者工具的形态不适合你的场景。这不是失败，这是诊断信息。下次换个角度提取。
