§1 背景与产品定位

1.1 背景（源文档 7.5 版四点）

  1. 主动服务需求：用户不只需要"我说你做"的被动响应，还需要系统在合适时机主动服务——"车里热了自动开空调""10分钟后关座椅加热""副驾打瞌睡提醒降温"。需要一个"条件监控 + 自动触发"引擎。
    没有触发器会怎样：每个主动服务场景都要 Planner/Advisor 自己起轮询、自己记状态。LLM 没有常驻监控能力，做不到。做到也贵到离谱（每5秒调一次模型=无效推理成本爆炸）。
  2. 条件任务增长：赛力斯条件任务清单已有 21+ 个场景且持续增加。缺乏统一注册、判定、执行引擎，每场景各写一套逻辑，不可维护。
    没有触发器会怎样：21+ 场景 × 每场景一套 if-else = 排列组合式维护灾难。改一个信号格式要动 21 处。一个条件改了，谁通知其他条件？没有全局视角。
  3. 端云协同需求：定时器必须在本地（弱网/断网不能停摆），VLM 视觉监控必须在端侧（隐私+带宽+延迟），但复杂语义判定和冲突仲裁需要云端。纯云端或纯端侧都不够。
    没有触发器会怎样：纯云端——隧道里"10分钟后关加热"直接失约；纯端侧——车机算力扛不住语义判定，规则全集也没法统一管理。
  4. Advisor 感知订阅：动态 Advisor 需要"耳目"持续感知外界变化。感知订阅 = 注册触发器的 tool，把 Advisor 的自然语言需求翻译成触发器的结构化注册参数。触发器是 Advisor 视觉感知的底座。
    没有触发器会怎样：动态 Advisor 二期（DNA-616）的条件/定时/POI/VLM 四类任务全部无处落地。Advisor 变成瞎子——说了"帮我照看宝宝"然后什么都看不到。

1.2 一句话定位

触发器 = 采集信号 → 判断条件 → 驱动执行。根据采集到的各类信号，当满足一定条件时，驱动下游执行特定动作。它不是 Planner（大脑），不是 Advisor（参谋），不是执行层——它是介于信号采集和智能决策之间的自动化中间层，只管"何时唤醒"。

1.3 端侧触发器定位回应 —— "就是个 VLM 工具"（汪帅 7.2 质疑） 7.2

汪帅 7.2 评审原话："就是个 VLM 工具，不能叫端侧触发器。" 以下是正面回应——四条理由证明它不是 VLM 工具：

#端侧触发器做的事跟 VLM 的关系证据
1本地定时器：弱网/断网不停摆的倒计时 + 到期持久化 + 重启恢复跟 VLM 毫无关系——纯时间逻辑这是 ㉔ 存在的唯一不可替代理由（v2 §5.2 24-5）
2任务中心端云同步：持久化任务库 + 任务级/感知级字段 + enable/disable 反向写入接口跟 VLM 无关——存的是所有动态任务不只视觉任务佳杰只从端侧取数（6.22 定）
3AlwaysOn 统一点火/开关管理：响应上下电 & Button，把舱内/舱外三类周期任务开关统一收口管理的是 VLM 的开关，不是 VLM 本身丁彬 6.30："希望很多功能都由你这来触发，把逻辑统一。"
4围栏/时间/车况等非视觉条件的端侧执行VLM 只是它调度的信号源之一给 VLM 下发 SP、收五态回执，是"触发器管理感知任务"，不是"触发器 = 感知工具"
丁彬官方文档（7.1）明确写了端侧触发器职责：开关三种周期任务 + 响应系统上下电 & Button + 触发 one_shot + 透传 AI Service。没有一项叫"VLM 工具"。

3.5 【15-9】端云分类（cloud / edge 路由）

(1) 职责定义

对入库的每条场景规则打 target 标记：cloud（留云端 ⑮ 判定）/ edge（下发 ㉔ 端侧执行）。分错就出事：该端侧的留云端 = 断网就废（定时器失效）；该留云端的下发端侧 = 端上算力不够（跨信号组合判定）。

(2) 分类规则表

条件特征分类原因
含 timer（倒计时/cron/日期）edge断网必须本地运行（铁律：定时器在端侧）
仅含车况类信号（温度/档位/车速等 key-value）cloud（上汽）/ edge（赛力斯）上汽全云端判；赛力斯可端侧判（24-12运算符级）
跨信号组合（A 且 B 且 C）cloud组合判定逻辑复杂，端上算力受限
含 VLM 视觉信号edge（赛力斯有端模时）/ cloud（上汽无端模走VQA时）视觉判定需端模实时推理（赛力斯）；上汽走云端VQA
含 POI/位置类cloud需查云端POI库
含 web_search/web内容cloud必须联网
动作含 vehicle_control（白名单内）跟随条件分类动作不单独决定路由

(3) 处理逻辑

  1. 遍历 conditions，按上表规则投票
  2. 多类型混合时按"必须端侧"优先（timer + 跨信号 → edge，因为timer必须端侧）
  3. 写入 SceneRule.target = cloud|edge
  4. edge 类型 → 后续走 15-10 SP拼接 + 15-12 下发
  5. cloud 类型 → 留在 ⑮ 等待 15-3 信号扫描触发判定

(4) Case

Case A："10分钟后关座椅加热" → 含 timer → edge。

Case B："温度>25且行驶>30min" → 跨信号组合 → cloud（且含持续时长，端上算力不够）。

Case C："看到副驾睡着" → VLM 视觉信号 + 赛力斯有端模 → edge。

Case D："帮我照看后排宝宝30分钟" → VLM 视觉 + expire_at（30min）→ VLM优先 → edge。

3.6 【15-3】信号扫描（变更驱动 + 兜底心跳）

(1) 职责定义

持续感知 ㉑ Context 中的信号变更，找出哪些"激活场景"可能因这次变更命中条件。不判定，只过滤——把"全部场景集"缩减到"本次可能命中的小集合"，给 15-4 做精确判定。

(2) 输入

输入来源说明
信号变更事件 SignalChange㉑ Context 推送 / ⑮ 拉取{signal_key, old_value, new_value, vin, timestamp}
激活场景集 ActiveScenes15-2 场景库该 vin+user 当前 status=1 且 target=cloud 的场景

(3) 处理逻辑

  1. 变更驱动（主路径，低延迟）：㉑ 推送 SignalChange → 15-3 查 ActiveScenes 中哪些场景的 conditions 引用了该 signal_key → 命中场景加入候选集 → 送 15-4 判定。
  2. 兜底心跳（容灾路径）：每 5s 全量扫描该车所有激活场景的所有引用信号，对比上次快照。若变更驱动正常则心跳空跑；若变更驱动漏推则心跳兜底。
  3. 过滤优化：维护反向索引 signal_key → [scene_id, ...]，避免每次扫描遍历全部场景。成本 = 该车激活场景数，非全量。

(4) 输出（候选场景集 CandidateScenes）

{candidates: [{scene_id, triggered_by_signal, current_value}], batch_id}

(5) 异常分支

异常处理
变更驱动漏推（信号变了但 ⑮ 没收到）5s 心跳兜底发现 → 补送候选集
信号长时间无更新（TTL过期）由 24-10/24-11 上报 timeout；⑮ 标记该信号 stale → 依赖场景 frozen
Context 服务不可用心跳失败 → 告警 → 已激活场景继续判（last_value 缓存）→ 新场景无法注册

(6) Case

(7) ★泽民交付物

① 订阅信号清单（哪些 signal_key 走变更驱动）② 扫描频率分级（车况 5s/AlwaysOn 3-5s/低频 5min）③ 反向索引维护规则。
找谁：仁杰确认 Context 推送还是 ⑮ 拉取；李鑫确认端状态客户端推送频率。

3.9 【15-5】阈值穿越检测（F→T 才放行）

(1) 职责定义

对 15-4 判定为 True 的场景，再判一道"是不是从 False 变 True"。只有穿越事件（F→T）才放行进入 15-6 防抖。没有穿越检测 = 温度持续28度的每个5s都触发一次 = 用户被轰炸。

(2) 处理逻辑

  1. 读 SceneRule.last_result（per-scene 维护，云端）
  2. 本次 raw_result（来自 15-4）
  3. raw_result=True AND last_result=False → 产生穿越事件 CrossingEvent
  4. raw_result=True AND last_result=True → 不产生事件（持续满足，不重复触发）
  5. raw_result=False → 不论 last 都不产生事件；更新 last_result=False
  6. 更新 SceneRule.last_result = raw_result

(3) 前帧问题（GAP，重要）

端上不留存前帧（丁彬7.4"即用即弃"）。穿越需要的"上一帧"在哪？
解法：last_result 维护在云端 ⑮（每个场景1个 bool，存储成本极小）。端侧只无脑上报每帧结果，云端做穿越判定。
对上汽：完全适用（全云端判）。
对赛力斯端侧判定场景：端侧 24-12 需保留 last_result 字段（1 bool/场景），不是完整前帧。
方案B回传链路下：动态任务原路返回触发判定时，也要同步把每帧结果写入 per-scene state store，否则下次穿越判定缺前帧。

3.15 【15-13】状态→事件组装 · push 6.22 · 触发器存在的根本价值

4.9 【24-10】五态回执处理 （含安全红线）

(1) 职责

AI Box 执行后返回 5 种状态，24-10 按策略响应。不归一则下游消费无标准语义——"沉默失败"最危险。

(2) 五态响应表

状态含义响应策略Context写入
success推理成功，有结果回写信号库+上报正常值
empty帧内无目标对象记录"无目标"（不告警）{status:"empty"}
failed推理失败（模型错误）重试1次→仍失败→告警{status:"failed"}
timeout推理超时（>5s未返回）写 timeout 进 Context（安全红线）{status:"timeout"}（绝不能写"无异常"）
interrupted被 one_shot 插队打断容忍（设计内行为，不打断防抖计数）不写

(3) 安全红线（不可妥协）

6.4 前帧问题 GAP

问题：阈值穿越需要"上一帧 False + 本帧 True"——但端侧官方设计 = 即用即弃不留存。前帧在哪？

根因：丁彬7.4："本地不留存，即用即弃。未上云的就销毁。" —— 端上没有历史状态。那穿越检测怎么做？

解法（推荐）：穿越状态维护在云端触发器 15-5。per-scene 维护 last_result（1个bool值/场景）。端上只需无脑上报每帧结果，穿越判定在云端做。AlwaysOn 数据走蓝线上云后云端有全量历史可比对。

端上不存前帧、不做穿越判定。这对上汽（全云端判定）完全适用。赛力斯端上判定场景需保留 last_result 字段（1个bool/场景，极小存储），不是完整前帧。

残留风险：如果回传链路走方案B（原路返回不绕Context），云端触发器可能在某些场景下拿不到"上一帧值"（因为没经过Context存储）。→ 方案B需保证：即使原路返回触发判定，也要同步把每帧结果写入per-scene state store。

12.1 内部矛盾6组

#矛盾分析处置
C1铁律②"端侧结果统一走信号库→Context" vs 7.4"绕Context订阅被推翻"不矛盾——推翻的是"Advisor订阅Context触发推理"，不是"数据不进Context"。数据照进Context（存储审计），触发通路走原路返回。已修正铁律②（§2.2 第2条 7.4修正说明）
C2铁律②端侧"即用即弃不留存" vs 阈值穿越需要"上一帧"真矛盾——端上不留前帧，穿越判定怎么做？解法=穿越状态维护在云端⑮（per-scene 1个bool），端上只无脑上报每帧结果。详见§6.4。
C3"最多拼3个动态SP" vs "多个Advisor同时订阅视觉"潜在矛盾——4个Advisor同时订阅就超3上限。处置=⑮ 15-7仲裁层加SP资源调度（按Advisor优先级抢占slot）。需补设计。
C4"SP拼接归云端触发器" vs 刘杨7.2"不能放到触发器"术语矛盾可能性大——刘杨反对"语义理解"，不反对"字符串拼接"。周六前对齐确认。最坏备选=拼接挂Advisor侧。
C5"事件判断三层 L1覆盖80%" vs 实际覆盖率未确认未证实——L1覆盖率取决于丁彬信号表精调字段数。标注为"待确认估算值"。找丁彬拿准确数。
C6"AlwaysOn是即用即弃" vs "前帧问题需历史对比"跟C2同源——但更尖锐：AlwaysOn设计本意就是即用即弃，穿越判定天然需要前帧，这是设计层面的张力。解法同C2——穿越判定挪到云端。AlwaysOn数据走上云通路（紫→蓝）后云端有历史。
