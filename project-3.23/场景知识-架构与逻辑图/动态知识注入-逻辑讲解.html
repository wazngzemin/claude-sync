动态知识注入系统 — 逻辑全解

司机 Agent 2.0 / PRD 逻辑骨架 + 数据流完整讲解 / 2026-05-18

一句话理解整个系统

用户说一句话 → Selector 判断「这句话需要什么领域知识」→ 从知识库里挑出对应的行为指南 → 塞进 Director 的 prompt 里 → Director 带着这份"专业手册"来回答用户、调用工具。

核心思想很简单：不是把所有知识塞给模型，而是每一轮对话动态挑选最相关的知识注入。就像一个全科医生，接诊时根据症状先翻开对应科室的诊疗手册，而不是抱着全部 64 本手册看病。

二、第一层：Knowledge Selector（知识筛选 Agent）

它的职责只有一件事：根据"用户说了什么 + 当前什么状态"，从知识库里挑出最相关的知识 ID。它不回答用户，不调工具。

它看到什么（5 个输入）

1
user_query — 用户说的话
不只是文字，还包含谁在说（speaker_name）和坐在哪（speaker_position: 主驾/副驾/二排左…）。
同一句"好热啊"，主驾说和后排小孩说，需要的知识可能完全不同。
2
conversation_history — 最近 10 轮对话
让 Selector 理解上下文。比如用户前面说了"去机场"，现在问"还要多久"，Selector 需要知道这是在问去机场的行程，而不是随便问时间。
3
goal_list — 当前目标队列
正在进行中的持续目标。比如"导航去周末游目的地"还没结束，相关知识就不能删。
关键原则：目标在，知识就留。
4
env_info — 实时车辆状态
空调/座椅/速度/电量/位置/导航/温度/音量… 全车实时状态。
关键原则：趋势比绝对值更重要。温度从 22℃ 涨到 29℃ 比"当前 29℃"更有决策价值。
5
loaded_knowledge_ids — 当前已加载的知识
上一轮已经塞进 Director 的知识列表。Selector 输出的是增量——哪些要新加，哪些要删掉。
避免每次全量替换，减少不必要的上下文波动。

它有什么（知识库索引）

知识库按 6 个"专家角色" 组织，每个角色覆盖不同领域。一共约 64 条知识，用四段式编码：

代码 专家角色 数量 覆盖范围 举例
EXP 销售/体验官 15 条 展厅看车、试驾伴行、离车送别 EXP-01-03-01 主驾落座首次体验
DRV 司机/导游 20 条 通勤、接送人、办事、充电、兜风、泊车指引 DRV-01-01-03 接机/火车
COM 随侍/管家 8 条 调温/调光/调坐姿/晕车/小憩/冥想 COM-01-01-06 晕车干预
ENT DJ/主播 7 条 听歌/播客/新闻/听书/游戏/节目编排 ENT-01-01-01 听歌
LIFE 管家/助理 7 条 停车缴费/天气/航班/比赛/金融/快递/百科 LIFE-01-02-02 查航班
EMO 伙伴 7 条 迎宾仪式/生日彩蛋/闲聊共情 EMO-01-02-01 生日惊喜

它怎么判断（5 条筛选原则）

#原则具体含义例子
1 场景绑定推理 先判断用户处于什么具体场景，再匹配对应知识。不是关键词匹配，是场景理解。 "好热啊" + 暴晒刚上车 → 急速降温
"好热啊" + 开了一小时 → 调温度
2 关注环境变化 趋势比绝对值更有价值。温度在涨、人数在变、车在进地库——这些变化触发知识切换。 温度 5 分钟涨 7℃ → 需要 COM-01-01-01
车辆进入地库 → 需要 DRV-01-02-02
3 目标持续关联 只要目标队列里有进行中的目标，支撑它的知识就不能删。 目标"导航去餐厅"还在 → DRV 知识保留
用户说"算了不去了" → 才可以删
4 增量更新 只输出变化部分（新增 + 删除），不重复输出已加载的知识。 上轮已加载 ENT-01-01-01 → 这轮只输出新增的 COM-01-01-06
5 不虚构 信息不足以判断场景时，不猜。宁可输出空列表，让 Director 自己问用户。 用户只说"帮我弄一下" → 不瞎匹配，add 为空

它输出什么

{
  "add_knowledge_ids": ["DRV-01-01-03"],      // 本轮新增的知识
  "remove_knowledge_ids": ["ENT-01-01-01"],    // 本轮删掉的知识
  "reasoning": {
    "add_reason": "用户要去机场接人且提供航班号，匹配接机场景",
    "remove_reason": "之前的听歌知识在接机任务中不再需要"
  }
}

三、中间层：知识拼接

这一步没有 AI，纯机械操作：拿到 Selector 输出的 ID 列表 → 去知识数据库（Excel）里查出每个 ID 对应的全文 → 拼接成一个字符串 → 塞进 Director 的 system prompt 的 {{knowledge}} 占位符里。

每条知识全文约 1500-2500 字，不是一句话摘要，而是一份完整的行为指南，包含：

组成部分内容作用
场景名称 + 描述 这个知识覆盖什么情况 让 Director 知道"我现在在处理什么场景"
互斥锚定 容易混淆的相似场景怎么区分 防止 Director 用错知识
行为规则 具体怎么做，什么能做什么不能做 指导 Director 的回复和工具调用
执行步骤 1→2→3 的操作流程 告诉 Director 先做什么后做什么
示例 具体输入→输出的对照 让 Director 理解输出格式和语气
打个比方：Selector 是"导诊台"，告诉你去哪个科室；知识全文是那个科室的"诊疗手册"；Director 是拿着手册看诊的医生。

四、第二层：Director / Thinker（对话推理 Agent）

它是真正面对用户的角色。它拿到动态注入的知识后，负责：说什么话、调什么工具、什么时候沉默。

它的 System Prompt 长什么样

Director 的 SP 是固定模板 + 动态知识的组合：

// ===== 固定部分 =====
你是豆包，一台AI汽车的个人助手。你的工作是与用户交流并执行任务。

// 输入类型定义（3种）
// 工具列表（18个）
// 输出格式要求
// 注意事项（14条）
// 参考示例（8个多轮对话）

// ===== 动态部分（每轮都可能变） =====
## 动态专业知识：
<动态专业知识>
{{knowledge}}  ← Selector 选出的知识全文在这里
</动态专业知识>

它处理 3 种输入

user_query — 用户说的话
advisor — 其他 Agent 的建议
tool_feedback — 工具的返回结果

advisor 是特殊输入：比如"车内温度过高建议降温"——这是系统主动触发的，不是用户说的。Director 需要把 advisor 的建议融合进回复里，但不能让用户感觉是系统在替他决定，要解释"为什么这样做"。

它能调用 18 个工具（分 5 组）

类别工具什么时候用
车控类 车控 / 车状态查询 / 导航控制 开空调、查电量、开始导航
搜索类 天气 / POI / 路线规划 / 联网搜索 / 车书问答 找餐厅、查路线、查天气
媒体类 音乐 / 短视频 / AI 播客 / 图片生成 放歌、看视频、生成图片
记忆类 记忆检索 / 人脸注册 / 长期记忆更新 记住偏好、认识新乘客
管理类 目标管理 / GUI 操作 / 视觉问答 点外卖、持续追踪目标

它输出什么

{
  "talk_or_not": "yes",                          // 要不要说话
  "talk_content": "好的，我先规划去T2的路线",      // 说什么
  "action_list": [                                // 做什么
    { "action_id": "1",
      "params": {"query": "去虹桥机场T2航站楼"},
      "tool_name": "route_planning_qa" }
  ]
}

它的关键判断逻辑（14 条注意事项精华）

情况怎么做为什么
听不懂 / 不是对自己说的 沉默（不说话、不调工具） 车里人聊天不需要 AI 插嘴
工具成功返回（如"车窗已关"） 大多数情况沉默 用户看得见结果，不用复述
需求模糊 问一句就好 不要猜，但也不要连问三个
工具之间有先后依赖 串行调用，不并发 先查电量再决定充电站
工具之间互不依赖 并行调用 导航 + 搜餐厅可以同时发起
GUI 任务等待中，用户说了短话 优先当作 GUI 回复 "确认""好"大概率是回 GUI 的
工具反复失败 暂时放弃，不死循环 别让用户一直等

五、完整走一遍：用户说"帮我接我爸妈，航班 CA1234"

1
用户开口
{"query":"去机场接我爸妈，航班CA1234", "speaker_name":"明骏", "speaker_position":"主驾"}
同时系统采集到：车刚启动、车内 1 人、舱内视觉正常、目标队列为空、当前无已加载知识。
2
Selector 开始工作
读取 5 个输入 → 扫描知识索引 → 判断：
场景匹配：用户明确说"去机场接人" + 提供航班号 → 命中 DRV-01-01-03（接机/火车）
环境判断：车刚启动、1 人、无特殊状态 → 无需额外知识
已加载知识：空 → 纯新增，无需删除
3
Selector 输出
{ "add_knowledge_ids": ["DRV-01-01-03"], "remove_knowledge_ids": [] }
4
知识拼接
系统拿 DRV-01-01-03 去知识数据库查出全文（~2000字），内容包括：
• 实时跟踪航班动态、自动调整出发时间
• 航站楼精准导航至指定到达口
• 特殊人群接送适配（老人→走无障碍通道）
• 接到人后座椅/温度自动调整

全文塞进 Director 的 SP 的 {{knowledge}} 占位符。
5
Director 推理
Director 读到了接机知识，知道应该：
① 先查航班状态 → 但当前没有航班查询工具，用联网搜索代替
② 规划去机场的路线
③ 利用记忆（"用户父母常住外地"）做个性化回复 
{
  "talk_or_not": "yes",
  "talk_content": "好的，我帮你查一下CA1234的到达时间，同时规划去机场的路线。",
  "action_list": [
    {"action_id":"1", "params":{"query":"CA1234航班实时状态"}, "tool_name":"web_search"},
    {"action_id":"2", "params":{"query":"去机场T2航站楼"}, "tool_name":"route_planning_qa"}
  ]
}
6
工具执行 → 返回 → Director 继续
两个工具并行执行（互不依赖），结果返回给 Director：
• web_search："CA1234 预计 14:30 到达 T2"
• route_planning_qa："路线规划成功，预计 45 分钟"

Director 综合结果继续回复，这轮不需要重新触发 Selector（tool_feedback 不触发重新检索）。
7
下一轮：用户说新的话 → 重新走 Selector
当用户说了新的话（比如到达机场后说"爸妈到了，后座温度调高一点"），
Selector 重新判断 → 可能保留 DRV-01-01-03（接机还在进行）+ 新增 COM-01-01-01（温度调节）。
Director 拿到两份知识，既懂接机流程又懂温度调节。

六、当前实现 vs PRD 设计目标

目前系统中存在两套方案，对应不同阶段：

当前实现（场景匹配） PRD 目标（动态知识检索）
Selector 做什么 纯分类：query → 场景 ID（ID_001~013） 情境推理：5 个输入 → 知识 ID 增删
知识编码 三位数编码 ID_001 四段式编码 DRV-01-01-03
场景数量 13 个 64 个（6 专家角色）
输入信号 主要看 query + context query + 历史 + 目标 + 车辆状态 + 已加载知识
更新方式 每轮全量替换 增量更新（add + remove）
知识全文 已产出 13 条（~2500 字/条） 目标 64 条（产出中）
延迟 ~1.8s（需优化） 需控制在 500ms 以内
核心矛盾：当前实现的 13 个场景已经导致 1.8s 延迟。PRD 目标是 64 个场景 + 更复杂的 5 输入推理。如果不优化 prompt 结构和检索方式，延迟只会更高。

建议路径：先用压缩锚词表把当前 13 场景的延迟压到 500ms 以内 → 验证准确率 → 再逐步扩展到 64 场景 → 如果 64 场景锚词表仍然可控（~1500 tokens），继续用 LLM 分类；否则引入 embedding 预筛。
