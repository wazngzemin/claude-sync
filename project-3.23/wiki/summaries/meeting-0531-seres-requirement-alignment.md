# 5.31 赛力斯需求梳理对齐会

> **时间**：2026-05-31（周日）14:37 – 20:54（GMT+8），时长约 6h17min
> **被评审对象**：赛力斯（Seres）整车 AI Car 需求全集 + 赛力斯侧 PRD / CRD 文档
> **主持/产品决策**：张航（"航哥"）
> **会议性质**：豆包上车团队内部需求拉通——把赛力斯整车需求拆成评审组逐条过一遍，确定哪些按当前方向做、哪些调整、并标注赛力斯 PRD/CRD 中写得不清/有问题/遗漏的地方，反推给对方修改。

## 会议背景与目标

- **压力来源**：赛力斯（CRD 负责落地的产品 + 研发 + 产品）人数多、节奏快，长期施压要求豆包侧"快点锁定需求"。豆包此前一直以"赛力斯 CRD 前几天才开放、且早期 PRD 根本不算 PRD"为由争取到延后窗口，但本次到了必须给出反馈的节点。
- **CRD ≈ 赛力斯面向研发的需求拆解**；张航口径：**赛力斯的 PRD + CRD ≈ 豆包的 PRD**。实际审下来赛力斯很多 CRD 写得不清楚或有问题。
- **本次只过与场景强相关的差异项**：豆包自家 AI Car 主框架功能逻辑团队已清晰，不再过；重点过偏场景的、与上汽差异大的部分。
- **两大会议目标**：
  1. 基于 feature 逐条确认方向，需调整的当场调整；
  2. 找出赛力斯 PRD/CRD 写错、有问题、漏的、不清楚的地方，标注后让对方改。

## 逐场景 / 评审组纪要

### 舒适空间（00:12–01:25）

- 基于 Advisor + Planner 框架做调舒适项服务（冷热、冥想、休憩），含特殊人群模式（老人/抱婴/宠物留车）。主动服务目前多为**事件触发**（看到老人→后调座椅；宠物留车→保持空调通风；看到婴儿→照看）。
- **专家任务集**：每个主动服务/语音任务的具体效果在专家任务集里逐条展开，目标 **6 月底**完成（益红 + 仁杰）。
- **舒适空间 APP 首页移除 COT 推荐黑框**（→ **D1**）：用户进首页是为手动调节而非回溯历史；只保留静态文字形式的历史思考摘要（时间戳+简述），证明"AI 在思考、活着"。突出可操作按钮（调座椅/温度）。
- **沉浸态主题解耦为独立工具**（→ **D2**）：不预置定制页让用户先配参数；建独立"沉浸态主题工具"（预置约 20 种主题，后续可扩生成式），Planner 可在任意场景调用；长任务详情页（目标/状态/历史思考）与沉浸态分离、不强绑定；用户可语音重新唤起页面。

### 感知可视化 / PHUD（01:25–02:05）

- **设计原则（→ D3）**：必须传递"豆包持续在感知、活着"——连续/流式，而非离散事件闪现；与 always-on 感知开关联动（关掉则消失）。需表达三件事：车内有谁及其状态、豆包注意力在哪、豆包观察到了什么。
- 现方案（一个月亮）信息密度太低、过于抽象。倾向用"四个小人"对应四座位+状态（开车/睡觉/玩手机/宠物）。难点：PHUD 极窄（扁形），需高设计功力。参考小米 PHUD 密度。
- 默认态展示"四小人"还是"氛围/情绪"未定，可能叠加；豆包"闪念"心情文字在屏顶补充解释。**待雪晴重出设计方案**。

### 出行 / 导航（02:05–02:20）

- **充电**：充电站 POI 已挂高德，支持条件筛选；赛力斯需补字段（"支持预约""自营"标签）。独立**预约工具**以即时交互卡呈现，可语音发起。**充电狗**（人工送充电宝）仅语音触发 + 出二维码供手机扫，作 SFT 工具进即交卡。
- **行程/导航**："兜风"与导航规划同属行程规划子能力，未保存暂存常时任务页（依赖异步架构）。回家/公司地址**以赛力斯端为主真源**（手动拖地图改也要同步给豆包）。行程查看简化为"语义精确搜索"+ Planner 摘要。
- **明日（6/1）上午**拉思晴在车上体验 1h，过地图最近问题。

### 娱乐（02:05–02:48）

- **AI 电台**：类比 AI Navi，把歌单/节目当"行程"容器，支持前插/编排/混排（歌单+播客+新闻）。**放弃 QQ 音乐**（不允许混源、成本高），改用网易云/酷我/酷狗（曲库共享，足够覆盖）；汽水用于氛围。需补单曲多版本列表。P 号屏左右滑切换节目——**手势细微差异识别难，技术待验证**。
- **娱乐搭子（读屏陪玩）**：本质 VQA 原子能力，解耦场景限制，吸顶屏默认开/可关，中控屏场景待定。"附近有啥楼"类问题**不走 VQA，直接调地图更优**。上汽现采用"预处理历史帧"方案（每几秒抓一帧、保留 3 张历史图、查询时动态选屏/选摄像头拉回对应时刻图）。

### 停车支付（02:48–03:30）

- 分工见责任表。优惠流程（省钱大师）类型多（积分/拍照/文字支付），不同商场差异大，**需赛力斯提供全类型样本**，目标支付宝自动化覆盖 ~80%、豆包补剩余 20%。
- 二维码进 director 走"模拟 query"拉起最新二维码工具，而非直接给工具。二维码有效性判断（视觉 vs 接口）待确认。**不支持**凭空语音"帮我登录支付宝"。

### 触发器 / 条件任务 / 场景引擎（底层底座 · 王泽民 owns）

贯穿全场的底层能力，由王泽民负责的触发器（Trigger）产品承载：

- **触发器核心条件梳理（张航 → 泽民，近期）**：02:19:46 张航明确"触发器要做……触发器我跟泽民讲了……比如音乐的播放状态、导航状态的切换，这些都要放进去……近期要把触发器一些核心的条件全部梳理出来，要补一波"。
- **AI 电台节目链依赖触发器事件**：节目单"播完一个播下一个"需要事件驱动（播放状态结束→触发下一节目），编排成任务 list 写进节目单工具，靠触发器/定时器顺序推进。
- **条件"二次判断"能力缺口**：04:00:38 提出"当电量不足以到达目的地时提醒充电"这类需要对条件再算一次的条件任务——**现有框架不支持**；益红建议作为通用事件在上下文里反馈。
- **条件任务 vs 端侧闭环引擎归属（已厘清）**：端侧闭环（VLM 识别→直接执行、不走 Planner/query，如遗留物品/天气保护）与云端条件任务是**同一个场景引擎**，只是配置在端上、配置来源是一套逻辑；统一显示在条件任务中心可关。
- **天气路况保护并入条件任务**：赛力斯原想做成单独设置开关，决定与条件任务统一，用户在条件任务中心开关。
- **熟路 PLUS 条件归属**：赛力斯识别熟路条件下发；skill/knowhow 是否存为条件任务/知识/记忆即 A 组模块边界问题（下周重点）。

### 生活服务（03:21–04:30）

- **美团**延续上汽方案但问题较大：到店取餐**不接受美团 APP 一步步操作**（美团不给原子数据），必须打通数据或找替代。博洋此前与赛力斯对齐的方案仍是"封闭域+智能体"思路、未考虑 director 角色，**需重新 review**。
- "赶不上 820 节点"的说法不可接受——可经 director 分发实现；让瑞哥/加忠找**弋途**商务抓紧谈。
- **小程序**：仅"打开/关闭"无意义不做；若能真正控制小程序（且与支付宝对齐保证不被封）才做，需对齐支持范围清单。

### 安全 / 迎宾 / 情感（04:00–05:01）

- **迎宾打招呼**：打招呼前必须先识别到 FaceID（已一致）。耗时预估 FaceID 1s + Advisor 3s + Planner 2s ≈ 6s，时机要求高（"开门正好打招呼"）。简单招呼（固定文案）直接给 Planner 知识、不过 Advisor；高级情感招呼走**情感 Advisor（第一优先级）**，要"鲜活"、每次不同（场景/画像/生日彩蛋）。建议虹软替商汤（FaceID 更成熟，≤1s）。
- **遗留物品提醒**：改端侧闭环场景引擎直接执行（VLM 触发 + 写死 TTS），任务中心可见可关（强制类不可改）。配置分三类：用户不可见强制 / 用户可见云端推理 / 用户可见端侧闭环。
- **哨兵 / 天气路况保护**：见责任表，均端侧闭环、开关进任务中心。

### Always-on / 账号 / 隐私（04:14–05:50）

- **Always-on ≠ 主动服务（→ D4）**：always-on 仅表示"持续观察+持续聆听"（多音区免唤醒 + 人人对话进上下文），与主动服务推荐是独立开关。关掉 always-on 不一定关主动服务；关视觉子开关不影响系统层车外安全告警。
- **To C 文案**：大标题"持续观察与聆听" + 子开关"视觉持续观察"/"听觉持续聆听" + 小 i 说明车外基础安全视觉始终在（不含 VLM 识别）。不单列"多音区免唤醒"开关（太复杂）。**持续聆听上汽做不了**（无端侧 ASR，成本扛不住），赛力斯优先。
- **账号 = 豆包一方应用（→ D5）**：法务+安全确认运营主体是豆包，所有用户数据强制挂豆包账号、权限等同豆包手机 App。**6/2（周一）整理邮件抄送 路由 终审，不再重开讨论**。
- **声纹归属（→ D6）**：声纹带权限 → 放**账号**内；乘客管理里可同步展示，FaceID/声纹支持授权用户跨账号在同车流转。声纹主用于车外安全认证（车内有人时车外语音禁用、灯语告知）+ 车内摄像头被挡时作 FaceID 备份。
- **头像生成**：语音认识后**直接自动生成卡通头像**（基于性别/年龄区间预置/随机，不做人脸还原，规避合规），用户后续可在乘客管理改。
- **记忆删除**：支持一键全量清空（短期+对应长期）、按时间范围删除（如"删上周"）；不支持任意分段删（短期删了长期摘要还在=矛盾）。先按日期，慎做按人。开关（不产不召、数据留）vs 删除（物理删）要区分。

### 工具 / 手势 / 感知规范（05:01–05:25）

- **工具规范化**：三个 MCP 工具（含充电狗）输入/输出 description、流程、字段按豆包工具维护表模板填清楚，工具化接入不走重链路。
- **手势**：手指指向转**语义描述**（"指着车窗"而非"右侧"）；歧义（指向上=阅读灯/天窗）要给 Planner 可选项；手势与 query **时间戳对齐**；滑动交互用"所见即可说"页面注册制；OK/确认一律转 query。
- **埋点/感知**：清单对齐——赛力斯给埋点清单，豆包 check 并映射到标准清单，逐条对接。
- **VLM 输出**：先给原始自然语言+完整信息，由赛力斯自己设计标签转换/可视化（PHUD 等）。
