触发器：云端和端侧到底怎么分工，研发问懵丁彬的那些问题你怎么答

用大白话把业务讲清楚 · 不堆字段 · 每一步讲"为什么流到下一个人、他拿到手干什么" · 基于 7.2 / 7.4 两场真实评审
这份稿子不讲字段格式（那是给研发的），只讲一件事：这套东西的业务逻辑是怎么转的，哪些活在云端、哪些活在端侧，为什么这么分。后半段把 7.2、7.4 会上研发追着丁彬问、他答不上来的问题，一条条给你人话答案——目标是同样的问题砸到你脸上，你能张口就答。

第一部分
先把大局讲清楚：这东西是干嘛的，为什么劈成云端 + 端侧

Q1 · 一句话，触发器到底解决什么问题？

系统里所有会"动脑子"的东西——Director、Advisor、VLM——都是模型。模型有个躲不掉的毛病：它只在你叫它的那一秒活着。你不问，Director 不想；你不触发，Advisor 不出主意；你不给它画面，VLM 连看都不看。

可现实是一直在变的：电量在往下掉、车在往目的地开、宝宝随时会醒。这就拧上了——模型是"一问一答"的，世界是"一直变"的。那谁来盯着世界，在该出手的那一刻去捅模型一下？

这个"盯着 + 到点捅一下"的活，就是触发器。它自己不动脑子（不判断语义、不出主意、不动手），它只干一件事：把"一直在变的信号"，变成"少数几个值得叫醒模型的事件"。

打个比方：触发器 = 门口的保安 + 床头的闹钟。便宜的、不用模型的判断（电量低没低、到 P 档没、超 30 度没）它 7×24 盯着做；贵的、要模型想的事，只在条件命中那一秒才叫醒。用"死规矩"挡掉 99% 的没用信号，让模型只处理那 1% 值得处理的时刻。
Q2 · 为什么非得切成"云端触发器 + 端侧触发器"两个，做一个不行吗？
6.22 西钺当时就警告：不要开发两个触发器

逻辑上它就是一个触发器，一个条件只判一次。物理上却必须劈成两半，因为有几个物理约束，谁也绕不过去：

三个"这活只能在端侧"的硬理由
约束典型场景为什么云端做不了
掐表"10 分钟后关座椅加热"本地倒计时才准。云端掐表有网络延迟，而且那会儿车可能在地库、没信号
断网也得干进隧道前说"10 分钟后开窗"真到 10 分钟，车在隧道里，云端够不着。只能端上提前存好规则，本地自己触发
离摄像头近看舱内画面舱内的图合规上不了云。眼睛（VLM）只能长在车上
三个"这活只能在云端"的硬理由
约束典型场景为什么端侧做不了
解冲突"10 分钟后关窗" 撞上 "5 分钟后开窗"谁先谁后要有个看得见全局的大脑来排。端上各管各的，看不到别人在干嘛
懂人话"宝宝毯子掉了提醒我""毯子掉没掉"要理解画面描述的意思，得过模型。端上只认死规矩，理解不了这种话
登记要统一所有人要盯什么，先来登记得有一个统一的前台校验能力、查重、拼任务。散在各端上会乱套、没法审计
记住这句就够了：逻辑上一个触发器只判一次；物理上劈两半，纯粹是因为"断网 / 掐表 / 摄像头"这三件事云端天生做不到，"解冲突 / 懂人话 / 统一登记"这三件事端上天生做不到。

第二部分 · 核心
掰开揉碎：哪些在云端、哪些在端侧、各自用什么

这是你反复问的那块。先给一张"谁在哪、拿什么、干什么"的对照，再讲两边怎么交接。

☁️ 云端触发器

角色 = 前台登记处 + 会算账的大脑
  • 登记校验：谁要盯什么，全先来云端登记。查一下"这个能不能看/能不能做"（iCanDo 能力清单），查重，然后才放行
  • 排班判断：信号来了，跑条件判断、判断是不是"头一次成立"（穿越）、要不要防抖、够没够冷却时间
  • 解冲突（仲裁）：几件事撞一起，按优先级排谁先办
  • 懂人话（语义判断）："这个描述算不算哭了"这种要理解意思的，在云端过模型判
  • 拼任务（拼 SP）：把动态 Advisor 临时想看的事，拼到端侧本来就在跑的观察任务后面，管好槽位
  • 叫醒下游：条件成了，push 给对应的 Advisor / Director，或直接走车控白名单
它手里的料：Context（用户记忆 / 车况 / 日历都在这）、端状态、各方的注册请求

🚗 端侧触发器 / AI Box

角色 = 车上的手脚和眼睛 + 一个掐着表的闹钟
  • 掐表：本地定时器。"10 分钟后关加热"这种倒计时，断网也准，还管合并、最小间隔、最多循环 30 次
  • 点火（开关 AlwaysOn）：把 AI Box 上的周期观察任务（舱外 3 秒 / 舱内动态 5 秒 / 舱内静态更慢）打开或关掉
  • 让 VLM 看：调度摄像头，定时拍，跑哪个任务、什么优先级、超时就释放
  • 判死规矩：只认硬条件（档位 R 变 P 没、超 30 度没）。不动脑子、不判语义
  • 断网闭环 + 直接执行：弱网下本地直接触发简单车控（上车开窗），不绕云
  • 透传上报：把 VLM 结果按端状态协议往上送
它手里的料：摄像头 + 端侧 VLM、本地定时器、CAN 车控信号、端侧配置平台
同一件事，云端做前半段、端侧做后半段——三个例子
用户/场景云端触发器做什么端侧触发器做什么
照看后排宝宝哭了提醒我登记这个盯梢任务、拼"看宝宝情绪"进 SP、结果回来后判"算不算哭了"、叫醒 Advisor每 5 秒让 VLM 看一眼、如实描述"宝宝安静 / 在哭"、把描述透传上云
倒车停好自动开充电口盖（这条主要在端上闭环，云端不深度参与判断）盯档位 R→P、调 VLM 看"后面有没有充电桩"、两个条件都中才提示
电量低了推荐沿途充电7×24 盯电量 < X%、命中就叫醒出行 Advisor、本次上电只叫一次（电量是车况信号，端上直接上报，端侧不额外观察）

规律：要动脑子、要查资料、要管全局的归云端；要贴着摄像头、贴着车、要掐表、断网也得干的归端侧。

两边怎么交接？一条铁律 + 上下两条路
铁律：不许上游直接捅端侧

不管是 Advisor 还是 Planner，想让端侧看点什么，都必须先经云端触发器。不能绕过云端直接给端侧下命令。为什么？因为能力校验、查重、拼 SP、审计，全靠云端这道关口收口。放开了，端侧的规则就乱了、出了问题也查不清。

下行（怎么把任务下发到端侧）
Planner / Advisor　→　云端触发器（校验 + 拼 SP）　→　端侧触发器（注入到 AI Box 的周期任务里）
6.30 会上丁彬问"能不能直接给端侧触发器"，西钺答："不行，先到云端的。"
上行（视觉结果怎么回到云端）——这条有两种走法，是 7.4 吵得最凶的点
走法 A「绕 Context」：端侧结果 → 写信号库 → 落 Context → 云端定时去翻 Context 看有没有变化
走法 B「原路返回」：谁下发的 SP，端侧就把结果原路送回给谁（快一跳）
这两条怎么取舍，见后面 Q6。

第三部分
数据怎么一步步流转：以"照看宝宝"走一遍，每步讲清"为什么、流给谁、他干什么"

1用户张嘴 → Director。"照看后排宝宝，哭了告诉我"，ASR 转成文字先进 Director。
为什么先到 Director：它是总管，头一件事是判断这句话是"马上办的"还是"要盯一阵子的"。这句是后者。
2Director 发现是"长时任务" → 交给动态 Advisor。
为什么不自己盯：Director 是执行层，盯梢是个持续活，得交给专门的参谋去想"怎么盯、盯到了怎么办"。
3动态 Advisor 先看家底。宝宝的状态，舱内 AlwaysOn 其实一直在拍，Context 里可能已经有。有就直接订阅 Context，不麻烦端侧；没有（比如"毯子掉没掉"这种从没约定过的）才需要新开观察。
为什么先看家底：能不动端侧就不动，省算力。
4要新观察 → Advisor 把"盯宝宝情绪"注册给云端触发器。
为什么不直接甩端侧：铁律，所有登记先过云端。云端负责校验能不能看、拼 SP、排班。
5云端触发器拼 SP → 下发端侧。端侧 VLM 本来每 5 秒就拍一次舱内，云端把"顺便看看宝宝哭没哭"这句，拼到它本来的活儿后面，一起发下去。
为什么触发器拼、不是 Advisor 拼：见 Q8。一句话——只有触发器知道端侧现在有几个槽、都被谁占了。
6端侧 VLM 每 5 秒看一眼，如实描述。"宝宝闭眼安静" / "宝宝脸上有泪在哭"。注意：VLM 只描述，不判断"这算不算哭了、要不要提醒"。
为什么只描述：VLM 是眼睛，眼睛只管看见。判断是大脑的事。
7结果回云端，云端来判。前 19 分钟都是"没哭"，某一秒变"哭了"。这时云端才判：这个描述 match 上"哭了"这个目标了 → 达成。
为什么判断在云端：要理解"脸上有泪"约等于"哭了"，这是语义，端上认死规矩认不出来。
8达成了 → 出主意 → 执行 → 收摊。Advisor 出建议（车速 60 在高速，别停车，先语音安抚）→ Director TTS"宝宝好像在哭，要不要回头看看"，顺手把音乐调小。任务结束，把"看宝宝"这句从端侧 SP 里撤掉，槽位还原。

第四部分 · 弹药
7.2 / 7.4 研发追着丁彬问、他答不上来的，你这么答

每一条都是会上的真实原话。给你"他到底在担心什么 + 你张口能说的答案 + 底层道理"，还有几条会有人当场反驳，我把接招也写上。

Q6 · "为什么要绕 Context 一圈？你怎么下发就怎么回来啊"
7.2 超然 / 7.4 仁杰
超然："为啥要走端状态呢？这条链路跟端状态没一点关系。你怎么下发就怎么回来吧，没必要绕一圈。"
仁杰："谁下发 SP，就原路把信息返回，不应该再绕 context 去订阅。"
他担心什么

多绕一圈 = 多一跳延迟 + 多一层依赖。而且这个结果只有触发器用，凭什么塞进公共的 Context 让大家一起存。

你怎么答
他说得对，我认这个账。我的方案是混合，不是二选一：像"照看宝宝"这种一次性的盯梢任务，走原路返回（谁下发谁收，快）；像舱内 AlwaysOn 那种本来就要长期存、好几个模块都要用的规律信号，还是落 Context 做存档。
关键是不管走哪条，判断权始终在云端触发器——最后都是它来判"成没成、要不要叫醒 Advisor"。这一点没变。
底层道理
Context 是个"只读的档案柜"——你们自己在 7.4 确认过：它没有主动通知能力，只能被查。既然它不会主动喊人，靠它来"等宝宝哭"就得派人定时去翻柜子，这就是"绕"。原路返回省掉翻柜子这步。但档案还得存（别的模块要用、事后要追溯），所以落 Context 这条不砍，只是不拿它当"触发"的路。
Q7 · "持续俩小时每 5 秒过一次模型，1 万辆车顶得住吗？"
7.4 庆磊（全场最尖锐的成本质疑）
庆磊："持续两小时，每 5 秒上报一次宝宝没哭，直到最后一秒哭了，这里面两小时云端无效的模型计算。1 万辆车每人搞个任务，云的模型就顶不住了。端是完全没这个能力判断的吗？"
他担心什么

如果每次判"哭没哭"都要云端过一遍大模型，1 万辆车 × 每 5 秒一次 = 天文数字的推理，烧钱烧到项目做不下去。

你怎么答
不是每次都过模型。我把判断做成三层漏斗：
层判什么怎么判成本挡掉多少
L1能用死规矩判的直接比较。情绪==哭泣、档位==P、温度>30零模型~80%
L2抠个关键词能判的正则 / 小模型，从描述里抠"哭"字极低~15%
L3真的开放、长尾的才过一次云端大模型，而且限流：1 分钟最多判一次，不是每 5 秒有，但限住了~5%
1 万辆 × 1 次/分钟 ≈ 167 次/秒，这个量云端扛得住。
底层道理
庆磊的账算错在"默认每次都过模型"。真相是绝大多数判断根本不配用模型——用模型判"档位是不是 P"，就像请个博士来算 1+1。三层漏斗的意义就是：99% 的判断用最便宜的方式挡掉，模型只碰最后那 5%，还限流。
现场会有人接着怼（庆磊 + 刘杨）："事件判断挪到触发器，不还是云端推理吗？1 万辆车触发器一样顶不住。"
你接：对，挪模块不省钱。省钱的是分层 + 限流，不是把模型挪给谁。所以我不主张"全挪触发器"，我主张"能不过模型的坚决不过"。这个账我认。
Q8 · "SP 为什么触发器拼？Advisor 自己拼不行吗？"
7.2 刘杨（连着追问了三轮）
刘杨："我不理解为什么云端触发器要拼一个 SP。你原始 SP 是动态 Advisor 给的，新任务也是 Advisor 下发的，为啥不由 Advisor 直接拼，非得过触发器？"
他担心什么

触发器本该是个"判断条件成立没"的规则引擎，现在让它去拼 prompt，职责越界了，架构不干净。

你怎么答
先把"拼"这个词说清楚，它有歧义。我说的拼不是理解语义去改 prompt，是纯字符串拼接 + 管槽位。
端侧 VLM 同一时刻只有几个固定的观察槽（一个舱内 AlwaysOn、几个动态槽）。谁的任务拼进第几个槽、结果回来第几个槽对应谁，这个"记账"必须有一个人统一管。这个人只能是触发器，因为它是唯一同时知道"端侧现在有哪几个槽、都被谁占了"的模块。Advisor 只知道自己要看啥，不知道别人也在看啥，让它拼会撞车。
底层道理
拼 SP 的本质不是"写 prompt"（那确实是 Advisor 的活），是"分配和回收端侧的观察槽位"——这是资源调度，天然归管资源的触发器。像航空公司排座位，不是乘客自己挑（会撞），是系统统一排。
刘杨会说："那走注册场景的链路不就行了？"
你接：预置好的枚举信号（睡没睡、有没有狗）确实走"注册 → 判断"就够，那种我同意不用拼 SP。但"宝宝毯子掉没掉"这种用户临时说的、VLM 从没约定过的开放问题，没有现成信号可注册，只能临时拼一句话让 VLM 去看。枚举类走注册，开放类才拼 SP，两类分开。
Q9 · "端侧触发器不就成了个 VLM 小工具、规则转发吗？"
7.2 汪帅（不认端侧触发器的定位）
汪帅："端侧触发器仿佛就是接受云端 SP 指令再转给 box，它就变成规则转发了。这跟我理解的不一样，感觉它就是个 VLM 的服务工具，不能叫端侧触发器，就是个小工具。"
他担心什么

如果端侧触发器只是个传话筒，那它没有独立存在的价值，不配叫"触发器"。

你怎么答
如果它只会转发，汪帅说得对。但它不止转发，它有三样云端替代不了的活：
① 掐表——"10 分钟后关加热"这种本地倒计时，断网也得准，云端做不了；
② 断网闭环——进隧道前设的"10 分钟后开窗"，到点车在隧道里云端够不着，只能端上本地触发；
③ 离线判死规矩——档位 R 变 P 这种端上信号，直接在端上判掉，不用等云端来回。
转发 SP 只是它顺带做的一件事，不是全部。
底层道理
判断"某模块有没有独立价值"的标准是：抽掉它，有没有事情彻底做不了。抽掉端侧触发器，所有"断网 / 掐表 / 离摄像头近"的场景全塌——这些云端天生做不到（延迟、网络、合规三个物理墙）。所以它不是可有可无的转发层。
Q10 · "caption 一个字符串，触发器规则怎么用？格式定完别再改"
7.2 韩杰 / 蒋文
韩杰："触发器是个偏规则的东西，如果要做判断，一个字符串到底能不能用得上？我希望格式定完就不要改。"
蒋文："端侧触发器需要结构化字段，caption 再用规则去匹配，不合理。"
他担心什么

触发器要判"条件成立没"，得有个确定的字段（门==开）。你给我一坨自然语言（"门看起来开着"），规则怎么写？而且今天定的格式明天 VLM 一改，我全得跟着改。

你怎么答
韩杰的担心对，所以我不硬拿 caption 做判断，分两种走：
① 高频、必要的（位置、性别、年龄、睡没睡、门开没开）——我推方舟精调成枚举字段，直接 字段==值 判，规则稳、不怕改。
② 长尾、临时的（宝宝毯子、狗项圈掉没掉）——这种没法也不值得精调，才落 caption，走第二/三层（抠关键词或小模型），不塞进你的规则引擎。
你要的"结构化字段"，高频那批我保证给你；caption 只走它该走的慢通道。
底层道理
caption 和枚举不是二选一，是分流。判断确定性要求高的（要写死规则的），必须给枚举；确定性要求低的（偶尔看一眼的），caption 够用。硬拿 caption 写规则确实不合理——韩杰是对的，我的方案就是不那么干。
Q11 · "充电桩这任务哪来的？存哪？谁感知的？"（丁彬当场答成"我对这块了解靠后"）
7.4 庆磊（把丁彬问懵的那条）
庆磊："这任务怎么产生的？存哪？谁感知的？是车出厂就预置在客户端的吗？我都不知道这个任务怎么产生的。"
丁彬："系统预设任务……你可以理解为我对这块了解的靠后面一点，大概就是预设到里头的。"
你替他答清楚
充电桩这个是系统预设任务，不是用户说的，是赛力斯提的需求、车出厂就配好的。
· 配在哪：端侧的配置平台（不是云端）。
· 谁触发：端侧触发器盯两个条件——第一，档位从 R 变 P（倒车入库停好了）；第二，这时候调一次舱外 VLM 看一眼"屁股后面有没有充电桩"。
· 两个都满足，才提示要不要开充电口盖。为什么端上不直接开？因为开不开还要综合别的信息、还要跟用户语音确认一句，这个综合决策不在端上。所以端上只出"有没有充电桩"这个结果，交上去。
这个 case 值钱在哪
它一次踩了三个最容易混的点：① 配置在端侧不在云端；② 触发条件是"车况（档位）+ 视觉（充电桩）"两个一起；③ 端侧只出感知结果，决策还在上面。丁彬答懵是因为把这三点搅一起了，拆开就清楚。
庆磊接着问："那这不就是静态 watcher 吗？"
你接：原来静态 watcher 管这类预设条件。现在把"带视觉的预设条件"从 watcher 拆出来给端侧触发器管，所以它既不是纯静态 watcher、也不是动态 Advisor，是端侧触发器的系统预设任务。这个拆分逻辑你得能一口气讲出来。
Q12 · "VQA 没有 key，怎么存？两小时发 200 个，Context 全毁了"
7.4 刘正伟（收尾前引爆的那条）
刘正伟："VQA 输出的 key 是什么？没有 key 这怎么存？如果识别两小时，往 context 里发了 200 个，那 context 全毁了。"
西钺："VQA 是单任务，它就没有 key。"
他担心什么

往 Context 里塞东西得有 key 才能存、才能查。一堆没 key 的自然语言塞进去就是垃圾，还把 Context 撑爆。

你怎么答
两件事分开：
① AlwaysOn 那种多任务持续观察的，有 key——用任务名当主 key（西钺 7.4 也认这个），"宝宝情绪""狗在不在"各是一个 key，存 Context 没问题。
② VQA 是"拍一张问一次答一次"的单次问答，它压根不该进 Context——它是 Director / Planner 临时问一句"我车边有啥"，答完就用完了，谁问谁拿走，不落库。
刘正伟担心的"200 个冲爆 Context"根本不会发生，因为 VQA 就不进 Context。
底层道理
VQA 和 AlwaysOn 是两种东西，别混。AlwaysOn 是"持续订阅"（要存、要 key、要落档），VQA 是"一次性查询"（用完即走、不存）。刘正伟的问题成立的前提是"VQA 也往 Context 塞"，而正确的设计是 VQA 根本不塞。边界一划清，问题自己就没了。
Q13 · "端状态能不能主动通知下游、让人订阅事件变更？"
7.4 汪帅（问架构），庆磊/超然/刘正伟一起把它钉死
汪帅："端状态有变更能不能告诉下游？下游能不能订阅事件变更？还是只能主动读？"
庆磊："端状态、context 就是个相对存储，没有主动通知下游的能力，只能被查。"
他担心什么

Advisor 想知道"宝宝哭没哭"，要么一直去读（轮询、浪费），要么等人通知。如果 Context 能主动通知，很多问题就简单了。可它做不到。

你怎么答
对，Context 是个哑的档案柜，只能被翻，不会主动喊人。正因为它哑，才需要触发器。触发器就是那个"替所有人盯着、成了就主动喊人"的角色。
所以这个问题的答案不是"给 Context 加通知能力"（那是把触发器的活塞给存储），而是本来就该触发器干这事——它是系统里唯一的事件中枢。谁要盯什么，注册给触发器，触发器盯到了 push 给你，你别去轮询 Context。
底层道理
这其实又绕回 Q1：世界一直变、模型只能被叫醒、得有个不睡觉的哨兵。Context 是记事本，触发器是哨兵，两个角色不能合并。汪帅想让记事本兼职哨兵，行不通——记事本没有"判断 + 主动喊"的能力。

一句话收尾
整套东西记住这几句就不会乱

1. 触发器 = 不睡觉的哨兵 + 闹钟：便宜的死规矩它 7×24 判，贵的模型只在命中那秒叫醒。
2. 劈成云端 + 端侧，是因为断网 / 掐表 / 摄像头云端做不到，解冲突 / 懂人话 / 统一登记端上做不到。
3. 云端 = 前台 + 大脑（登记、判断、拼 SP、叫醒下游，手里有 Context）；端侧 = 手脚 + 眼睛 + 闹钟（掐表、点火、让 VLM 看、判死规矩，手里有摄像头和本地定时器）。
4. 交接一条铁律：所有登记先过云端，不许上游直捅端侧。
5. 成本不靠挪模块省，靠三层漏斗 + 限流：能不过模型的坚决不过。
6. caption 和枚举分流：要写规则的给枚举，偶尔看一眼的才用 caption。
