一、VLM 在系统里是什么——两条路径 + 三个角色

VLM（视觉语言模型）= "让摄像头看一个东西，告诉系统看到了什么"。它不是信号，是一次主动的视觉推理请求（占摄像头、跑模型、有超时、有结果状态）。系统里 VLM 有两条底层路径：

路径是什么车型谁发起
云端 VQA一次性问答——一张图 + 一个 prompt → 一个答案（true/false + 描述）上汽 ✅ + 赛力斯 ✅Director 工具（看一眼）/ 动态 Advisor 定时器（定时看）
端侧 VLM（㉕ 感知器）持续感知——Always-on 每 3-5 秒一帧，端侧推理出标签/状态仅赛力斯（需 AIBox）触发器 24-8 调度（一定经触发器）

VLM 和三个角色的关系：

角色和 VLM 的关系
触发器㉕ 感知器是触发器的"感知半"。触发器不碰 VLM 推理本身（推理在端模/AIBox），只做 SP 下发 + 事件组装 + 调度。
Advisor（⑯静态/⑰动态）Advisor 够不着 VLM（云端 LLM 没有手）。动态 Advisor 通过感知订阅让触发器去调 VLM；静态 Advisor 被动消费触发器组装好的事件。
Planner/Director（⑲⑳）Director 想"看一眼"可直接调云端 VQA（旁路，不经触发器）；持续视觉则创建动态 Advisor 走触发器。
6.22 一句话定调：端模目前只做"眼睛"——观测，不融入业务逻辑（丁彬 40:10："它现在就是作为眼睛去观测，还没融入逻辑，你告诉它看啥可以，但给它下个复杂任务它做不了"）。所以"看到什么 → 怎么用"被拆成两段：端模负责"看到/状态"，上层（触发器组装事件 + Advisor/Director 推理）负责"怎么用"。

二、VLM 全链路（动态视觉感知事件）· 端到端总图

这是 6.22 评审捋出的核心链路——以"照看二排宝宝/盯副驾睡没睡"这类持续视觉为例：

① ⑳ Director用户"照看宝宝"→建动态Advisor
▶
② ⑰ 动态Advisor拆解query→改写注意力SP
▶
③ ⑮ 云端触发器注册校验→编译→下发SP
▶
④ 端模 ㉕按SP观测特定目标
▶
⑤ ④ 信号库写"状态"(list,带时间)
▶
⑥ ⑮ 组装"事件"监听变化·阈值穿越
▶
⑦ ⑰/⑯ Advisor事件push(15→17)·消费
▶
⑧ ⑳ Director推理→执行(TTS/车控)
#环节谁/在哪做什么定了没
1Director 建监工⑳ 云端用户"照看二排宝宝" → Director 识别为持续任务 → 创建动态 Advisor定
2动态 Advisor 改写 SP⑰ 云端拆解 query"宝宝哭/踢被子就告诉我" → 改写成一条注意力 SP（"当宝宝哭/踢被子时告诉我"）定
3触发器注册 + 下发 SP⑮ 云端→端15-1 校验 → 编译 → 把动态 SP 下发给端模（铁律①：动态注册必经 ⑮ 中转）⚠ SP 走哪条没定
4端模按 SP 观测㉕ 端侧(AIBox)端模按 SP 观测特定目标（宝宝状态）。能不能做"看到X就告诉我"的条件式判断，是最大不确定性（见 §5）⚠ 端模能力没试过
5结果写信号库④ 端侧端模输出"状态"写入 ④（如"宝宝在哭"/"小狗在副驾"）。always-on 输出是纯客观状态、不带是/否判断；Context 里存成 list（5s前/3s前/当前）部分定
6触发器组装"事件"⑮ 云端触发器监听信号变化（阈值穿越/首次出现）→ 把"状态变化"组装成"事件"（如"宝宝刚开始哭"=高度简化语义）⚠ 组装放云/端没定
7事件 push 给 Advisor⑮→⑰/⑯ 云端触发器把事件 push 给订阅的 Advisor（15→17），不是 Advisor 轮询 Context（佳杰当场纠正，见 §4）定
8Advisor 推理 + 执行⑰→⑳ 云端Advisor 结合 Context（condition）出 advice → Director 拍板 → 执行（降音量/降温/TTS）定
这张图的三段分工（评审记这个）：
· 端模（兰若舱内/丁彬舱外）：只管"看到 + 输出状态"（眼睛）。
· 触发器（泽民）：SP 下发 + 监听状态变化组装"事件" + push Advisor + 调度。
· Advisor/Director（佳杰/晓伟）：消费事件 → 推理 → 执行。

三、两条视觉路径详解：云端 VQA vs 端侧 VLM

路径 A · 云端 VQA（一次性问答）—— 上汽/赛力斯均可用

⑰ 动态Advisor / ⑳ Director自带定时器/当场问
▶
端侧摄像头抓一帧上传
▶
☁ 云端 VQA图+prompt→result/desc
▶
回⑰ / ⑳推理→建议/TTS

特点：一次性问答、依赖网络、本期仅 1 个观察任务（1 个 prompt），多 Advisor 同订阅靠感知工具做 prompt 拼接。Advisor PRD 原话："vlm 只接 vqa，不接端模，云对云对接，一个文本/prompt 过去，返回 result 和 description"。

上汽当期就靠这条（佳杰 37:59："上汽这边直接用动态提示词接的云端 VQA"）。但有抢占问题（佳杰 39:07："端侧取图时会和其他业务抢占，比如和 VQA 抢，导致 VQA 用不了"）。

路径 B · 端侧 VLM（持续感知，经触发器）—— 仅赛力斯

⑬配置/⑰订阅VLM需求
▶
⑮ 15-5 编译→VLM task 7字段
▶
15-12 下发规则包→㉔
▶
㉔ 24-8 调度→AIBox运行指令
▶
㉕ 持续执行周期出结果
▶

VLM task 7 字段：camera_group / seat / query / interval / timeout / priority / mutex_group。特点：持续感知、断网也能跑、仅赛力斯有端侧算力。

同一需求两条路："盯副驾睡没睡"——上汽走路径 A（云端 VQA 定时轮询），赛力斯走路径 B（端侧 VLM Always-on）。产品做的事一样，技术链路不一样。

四、事件 ≠ 状态：触发器组装"事件"push，不轮询 Context（6.22 核心纠正）

这是评审里佳杰当场纠正泽民的核心链路问题，必须讲清：

概念是什么谁产出/存哪谁消费
状态 / condition客观、持续的当前事实（"车里有小孩"、"当前温度26°"、"小狗在副驾"）。always-on 输出纯状态、不带是/否端模写 ④信号库 → ㉑Context（list，带时间）推理时作为条件用
事件 event状态的变化，高度简化的语义（"宝宝刚开始哭"、"用户睡醒了"、"用户在倒车"）⑮云端触发器监听状态变化、阈值穿越/首次出现，组装出来Advisor 订阅，由触发器 push 给它（15→17）
✗ 原来的错画法㉑Context → ⑰Advisor 当事件消费（轮询）
≠
✓ 正确④状态 → ⑮组装事件 → push ⑰Advisor(15→17)
佳杰 26:35："我需要一个用户睡醒了的事件，我不会去轮询你的 context，我是注册了这个事件之后，就等待你的云端触发器给我发这个事件。"
佳杰 29:34："context 解决的是 condition。倒车是事件，车里有小朋友是 condition，做什么是 advisor 给的。我不可能 24 小时轮询，这就产生了触发器的需求。"
为什么事件放触发器、不放 Advisor：① 事件比状态更有价值（佳杰："如果触发器直接给状态，那我直接接信号库了"）；② 从推理 ROI 角度，不能让 Advisor 一直推理不相关的事（西钺 30:27）。所以触发器中间做了"从状态里挑出有意义的变化、组装成事件"这层活。

五、端模能力——整条链路的前提（最大不确定性）⚠

评审最该盯的一条：端模能不能做"看到 X 就告诉我"的条件式动态注意力？这个没试过，能不能成直接决定整条链路简单还是复杂。
情况链路会变成什么样谁说的
能（端模 prompt 限制有限集，指令遵循好）动态 Advisor 改 SP → 触发器下发给端模 → 端模"看到就上报" → 触发器收事件 → 给 Advisor。链路简单仁杰 41:12、丁彬 40:58
不能（端模只能固定周期输出观测内容）端模每帧输出状态 → 云端要再加一个模块从每次输出里判断"哪一次发现了 X" → 再触发动态 Advisor。链路变重仁杰 34:59 / 35:38
丁彬 40:58："通过 prompt 我理解是能限制住的，得试一下，目前还没试过。如果指令遵循比较好的话问题不大……如果你看的这几种事情比较固定（只看毯子、毛巾）基本能保证；但如果还要看手绢、小方巾，发散起来就是个问号。"

5.1 端模能力的两个约束

  • 有限集（SP 必须在可识别集合内）：丁彬 42:38："SP 有个要求是有限集，没见过的就靠基模能力了。"西钺那边端模现在能看的：睡觉、打电话、某些动作（丁彬 43:21）。
  • 只输出状态、不做判断：丁彬 45:49/46:24："还是输出状态，结果判断交给业务逻辑。输出'小狗在主驾/副驾/二排'，上层语义再去推理。"西钺 46:41："纯客观描述此时此刻的状态，不描述是/否——不说'坛子是否掉地上'，只说'坛子当前在地上'。"

六、通用视觉事件 / 端侧泛化感知通道（兜底长尾）

预置的视觉事件是枚举的、有限的；但用户的动态 query 是发散的。所以评审定了要留一条"通用视觉事件"通道兜底（西钺 + 佳杰提出）。

⑰ 动态Advisor泛化query"看XX"
▶
⑮ 通用视觉事件通道透传system prompt
▶
AIBox对prompt做增量更新
▶
端模㉕按新prompt观测
要点内容
定位backup 方案。先看现有视觉事件够不够，不够才走通用通道（佳杰 01:02:35）
怎么传把动态 Advisor 的泛化 query 透传给 AIBox 的 system prompt（佳杰 01:04:33）
分几类舱内静态 / 舱内动态 / 舱外 三类（丁彬 01:03:12 / 西钺 01:03:45）。舱外只用前摄（丁彬 01:03:51）
prompt 增量必须是"增量更新"，不能覆盖——否则原 prompt 没了、原感知内容也丢了（丁彬 01:03:12）。还要区分增到哪个任务的 prompt 里
体验兜底豆包可以先说一句"这个任务我现在可能做得不太好，你确定要让我试试吗"（佳杰 01:04:33）

七、调度 + 舱内/舱外分工（三个任务）

7.1 调度归属：在 AIBox，不在触发器（6.22 确认）

always-on / VLM 任务调度在佳杰的 AIBox 模块，对所有请求做优先级排序，不在触发器（西钺 57:25 提，佳杰 57:47 确认）。

佳杰 57:47："AIBOX 上有一个模块专门去对所有请求做优先级排序。always-on 优先级多高、VQA 优先级如何、其他视觉需求优先级如何，优先级低的会排队等。"
两层排队（解耦）谁做 / 做什么
请求侧排队触发器 ㉔ 24-7：两个触发器任务同时要摄像头时，按 P0-P4 决定先发哪个请求给 AIBox
执行侧排队AIBox 任务调度（斯华）：收到所有来源请求后，做最终资源分配和执行排序。可能因为在跑更高优的舱外 AV 而让你的请求继续等
