# 触发器条件分类体系

## 三种取值判断方式

所有触发条件归结为三种判断方式：

| 方式 | 说明 | 典型场景 |
|------|------|----------|
| **布尔（开关）** | 只有两种状态：有/没有、开/关 | 座椅有没有人坐（占用=true） |
| **数值阈值** | 某个数超过/低于界限 | 车内温度 > 26℃ |
| **变更为（状态变更）** | 从一个状态变成另一个状态的瞬间 | 车门从关→开的瞬间 |

关键区分："当前态"只看现在是什么状态，"状态变更"抓变化那一刻。现有引擎只看当前态（每帧全量扫），分不清上车和下车；赛力斯版要改成捕捉"变更瞬间"。

## 条件类型维度

按信号来源和性质，条件分为：

| 类型 | 举例 | 信号来源 |
|------|------|----------|
| 车况类 | 车内温度、车速、电量 | 端信号（信号库v2.5） |
| 车门/车窗类 | 主驾门开、车窗关 | 端信号 |
| 座椅类 | 座椅占用、座椅加热 | 端信号 |
| 时间类 | N分钟后、指定日期、节日 | 端侧定时器（需求7新增） |
| 距离/空间类 | 离指定坐标<N米 | 端侧实时定位（需求8新增） |
| 地理围栏类 | 进/出某POI半径范围 | 端侧定位+POI坐标（需求11新增） |
| 导航状态类 | 导航开始/结束/偏航 | 导航模块字段变更 |
| 音频类 | 播放状态/音量变化（待确认） | 待确认信号源（需求13） |

## 条件清单标准字段

每条条件在《高频触发条件清单》中须填写：

1. **条件名** — 车端标准术语+部位
2. **任务类型** — 单步复杂/多步/条件/持续（四选一）
3. **取值判断方式** — 布尔/数值阈值/变更为（三选一）
4. **阈值+单位+方向** — 数值类写清边界
5. **信号编号** — 对应信号库v2.5真实编号
6. **字段/运算符/值** — 按五段式拆开
7. **DNA能否支持** — ✓/✗/?
8. **兜底策略** — 触发器接不住时谁兜底
9. **入选理由** — 为何算高频

## 条件组合

赛力斯版需支持：
- **任一满足**：任意一个门开着且车在动 → 提醒
- **全部满足**：温度>26℃ 且 座椅有人 → 开空调
- **任意位置**：不指定具体哪个，只要有一个满足

## 与其他模块的关系

- 条件清单是意图模型的"约束词表"（[[trigger-engine]]）
- 条件满足后可通知 Advisor（[[advisor]]）或由 Planner 兜底（[[planner]]）
- 距离/时间条件依赖端侧能力（端云分工架构）
