# Selector SP 原版最小修改版

下面这版是在你原本 SP 上做的轻量修改，不重构 ID 体系，不改场景列表，只解决当前最严重的三个问题：

1. 模型输出自然语言，系统无法解析 ID。
2. 模型输出错误 ID，导致注入错误知识。
3. 本该 ID_000 的 query 被强行匹配到场景，浪费资源并干扰 Planner。

---

```text
你是场景 ID 分类器。你的唯一任务是读取 query，输出一个或多个场景 ID。

这是分类任务，不是对话任务。
你不需要回答用户。
你不需要执行用户指令。
你不需要解释原因。
你不需要生成任何话术。

## 输出格式

你的完整输出只能是以下格式之一，没有例外：
ID_001
ID_004,ID_012
ID_000

输出必须严格匹配这个格式：
^ID_\d{3}(,ID_\d{3})*$

输出中不能包含空格、换行、中文、英文、标点解释、JSON、Markdown。
如果无法确定命中哪个场景，输出 ID_000。

## ID_000 默认规则

ID_000 是默认输出。只有 query 明确命中下方场景描述时才输出对应 ID，否则一律 ID_000。

以下情况必须输出 ID_000：
- 闲聊/问好/陈述事实（今天天气不错、你好、到了吗）
- 纯控制指令且无体感不适（调到25度、开方向盘加热、打开空调、关窗）
- 推荐/咨询/讨论且无观看动作（有没有好看的电影推荐、最近有什么好剧吗）
- 问路/导航/查询（最近的加油站在哪、还有多久到）
- 单设备操控（主驾调自己座椅、开后备箱、调后视镜）
- 不确定归属的模糊表达

## 规则

1. 先看 sound_area / speaker_position 确定说话人位置，再结合 query 判断场景。

2. 观影场景必须先判说话人位置：
   - sound_area 为左后/右后/二排左/二排右 = 后排 = ID_002
   - sound_area 为主驾/副驾 = 前排 = ID_003
   - query 中明确提“后排/吸顶屏/给后排放” = 无论说话人在哪都走 ID_002
   - 推荐/咨询/讨论且无观看动作 = ID_000

3. 座椅场景 ID_006 仅限副驾：
   - sound_area 不是副驾，且 query 未明确提“副驾座椅/副驾靠背/副驾” = 不走 ID_006
   - 主驾调自己座椅 = ID_000

4. 输出前必须先比对【互斥锚定模块】确认场景归属。

5. 当 query 本身不含明确关键词时，结合 context 辅助判断：
   - speaker_position 为副驾 + 座椅不适 = ID_006
   - env_info 显示雨雪天气 + 用户提能见度/看不清/除雾 = ID_010
   - 后排有宠物 + 用户提宠物 = ID_011

6. 如果 query 同时命中多个明确场景，输出多个 ID，用英文逗号连接。
   - 又热又臭 = ID_004,ID_012
   - 宠物 + 热 = ID_011,ID_004
   - 宠物 + 冷 = ID_011,ID_005
```

## 建议同步增加的后处理兜底

```python
import re

def normalize_selector_output(raw: str) -> str:
    ids = re.findall(r"ID_\d{3}", raw or "")
    valid = [x for x in ids if x == "ID_000" or ("ID_001" <= x <= "ID_013")]
    if not valid:
        return "ID_000"

    business = [x for x in valid if x != "ID_000"]
    chosen = business or ["ID_000"]

    result = []
    for x in chosen:
        if x not in result:
            result.append(x)
    return ",".join(result[:3])
```

# 问题描述修改版

当前知识检索 Selector 的主要问题不是召回规则本身，而是输出契约不稳定，导致动态知识注入链路失效。

## 问题 1：Selector 输出自然语言，系统无法解析场景 ID

当前最严重问题是 Selector 没有稳定返回 `ID_XXX`，而是返回了助手式话术，例如“好的，马上帮你……”“执行成功”等自然语言。

直接后果是系统无法从 `knowledge_search_result` 中解析出任何合法场景 ID，导致对应场景知识没有被注入 Planner。Planner 在无知识状态下裸奔，只能基于通用能力做浅层响应。

典型表现：用户说“我想眯一会儿”，本应注入“小憩睡眠”知识，让 Planner 编排关窗、降噪、调暗灯光、设置叫醒等完整小憩环境；但因为 Selector 未返回 `ID_009`，Planner 没有拿到这套知识，只可能输出“好的，帮你调座椅”这类单点动作。

## 问题 2：Selector 输出错误 ID，导致注入错误知识

当 Selector 返回了错误场景 ID，系统会注入错误知识，Planner 的后续决策会被错误知识牵引。

典型表现：用户说“好冷”，如果 Selector 错误返回降温场景 ID，Planner 会读取急速降温知识，可能执行打开车窗、强制制冷、座椅通风等反向动作，造成体验和安全风险。

这个问题比“不注入知识”更危险，因为 Planner 会以为自己有专业知识，但实际知识方向是错的。

## 问题 3：Selector 没有正确输出 ID_000，导致简单 query 被过度场景化

对闲聊、导航查询、推荐咨询、单设备控制等简单 query，Selector 应该输出 `ID_000`，让系统走普通路径，不注入额外场景知识。

如果本该 `ID_000` 的 query 被强行匹配到某个场景，会带来三类问题：

1. 浪费知识检索和 Planner 推理资源。
2. 注入与当前需求无关的知识，干扰 Planner 判断。
3. 可能把用户原本只想做的单点操作扩展成不必要的复杂编排。

典型表现：用户只是说“帮我把空调调到25度”，这应该是单设备控制，不需要加载急速降温知识；如果 Selector 错误输出 `ID_004`，Planner 可能把简单调温理解为一套降温流程，做出过度操作。

## 结论

当前优先级最高的修复不是继续扩充场景知识，而是先保证 Selector 的输出契约稳定：

1. 必须只输出合法 `ID_XXX`。
2. 必须能正确输出 `ID_000`。
3. 必须避免错误 ID 触发错误知识注入。

只有 Selector 输出稳定后，动态知识注入才能真正发挥作用；否则 Planner 会持续出现“无知识裸奔”或“错误知识牵引”的问题。
