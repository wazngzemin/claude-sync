6.22会后 · 端侧触发器执行方案

产品经理：王泽民 | 基于2026-06-22端侧触发器评审会议 | 参会：丁彬、陈西钺、施佳杰、吴仁杰、赵益红、王泽民
文档定位：PM行动方案——6.22改了什么、现在该怎么做、谁给谁什么、什么时候交付

一、6.22改了什么——一句话

端侧触发器的scope大幅收窄：固定/标配的端侧闭环场景全部回赛力斯CDC底软，触发器只做动态/可变/用户临时设的场景。事件链路从"轮询Context"改成"云端触发器push给Advisor"。新增"动态视觉感知事件"和"任务中心打通"两块需求。

二、Scope变更：什么回CDC，什么留触发器

判断标准（仁杰13:14原话）

"比如像那种车辆安全、上车安全、下车安全，它很有可能都会在车辆设置里，是一个常驻的设置项，是所有车标配的功能，不会变，不涉及条件变化或用户改配置——这种直接底软做掉，不需要走触发器去绕。那种有生命周期的，用户只在这个时候创建了一下，消费完就没了，或者说条件是可能变化的、动态生成的——这类才走触发器。"
场景 归属 谁说的 理由
哨兵模式 回CDC 西钺09:24 + 丁彬16:17 固定功能，赛力斯自己做的逻辑，不耦合触发器
上下车安全 回CDC 仁杰13:14 "起步安全是所有车标配，固定功能，不需要动态配置"
起步安全 回CDC 仁杰14:07 "整个生命周期里这东西不会变化，就是一个固定功能"
无感迎宾 / 无感开车门 回CDC 佳杰10:29 "更适合做成系统feature，不涉及语音交互，多端信号打通，赛力斯自己就做就行了"
充电桩开盖 回CDC 丁彬08:07 + 西钺09:58 赛力斯CDC已有"充电大师"业务逻辑，两套会打架
睡眠检测/提醒（固定型） 回CDC 仁杰16:01 "它就是一个常驻的固定功能，没必要走触发器"
用户口头设的条件任务
"10分钟后关座椅加热"		 留触发器 共识 有生命周期，用户临时创建，消费完就没了
运营可配的动态场景
"圣诞播放圣诞音乐"		 留触发器 仁杰15:32 "可运营的，临时的，动态的，会变化的"
Advisor注册的感知订阅
"照看后排宝宝"		 留触发器 丁彬14:07 "很开放式的规则，没法写死，需要触发器来产生"
动态视觉感知事件
"用户睡醒了""驶出地库"		 新增 佳杰28:07 + 西钺01:01:29 Advisor需要消费的视觉类事件，需新建一类能力
对讲解稿的影响：原文的"端侧任务②简单条件判定""端侧任务③白名单车控直执"以及DMS安全监控、儿童遗留、乘客状态监控、手势控制的详细case表——这些描述的都是6.22之后回CDC的固定场景，不再是触发器的scope。

三、事件链路改了：push，不是轮询

佳杰的原话（26:35）

"我不会去轮询你的context，我是注册了这个事件之后，我就等待你的云端触发器给我发这个事件。"

佳杰对Context vs 事件的区分（29:34）

"context拿的是另外一部分信息，它是作为condition存在的。比如'用户在倒完车之后，如果车里有小朋友，就做什么'——倒车是事件，车里有小朋友是condition。context解决condition，触发器解决事件。第一趴（事件），我不可能24小时轮询，这就产生了触发器的需求。"

新链路 vs 旧链路

环节旧链路（讲解稿写的）新链路（6.22确认的）
Advisor怎么拿到事件 端侧检测 → 写④ → ⑫同步 → ㉑Context → Advisor轮询Context消费 端侧检测 → 写④ → ⑫同步 → ⑮云端触发器做阈值穿越+事件组装 → push给Advisor
Context的角色 Context是事件的载体，Advisor从中消费事件 Context只存condition/状态（比如"车内有小朋友=true"），不是事件通道
谁判断"发生了变化" 端侧㉔的24-10做阈值穿越 ⑮云端触发器做阈值穿越(15-10) + 事件组装(15-11)后push
对你的意义：这个改动说明触发器的"事件组装"能力是核心价值——不是简单转发状态，而是把原始信号变成Advisor能消费的语义事件。这是佳杰明确需要你提供的东西。

四、新增需求①：动态视觉感知事件

需求来源

佳杰+益红有一批和视觉强相关的事件需求：

佳杰（21:06）："它属于一个视觉类的状态变更事件，不是静态的。比如用户什么时候驶出了地库，什么时候在打电话、结束打电话，它更多是这种持续类的视觉，要检测到一个变化过程。"
佳杰（22:29）："大概就是睡着睡醒、驶出地库、打手机结束打手机、某个位置出现了某一个物品或被移除了。"

链路怎么走（会上讨论收敛到的方案）

#谁做什么
1动态Advisor⑰拆解出"需要一个视觉类条件" → 调触发器注册
2⑮ 云端触发器把语义目标编译成端模能理解的SP/注意力 → 下发端侧
3端模（AIBox）按SP观测，只做客观状态输出（"毯子在地上"），不做业务判断（"毯子掉了"）
4信号库④存储端模输出的观测结果
5⑮ 云端触发器从信号做阈值穿越/事件组装 → push给Advisor

五、新增需求②：任务中心打通

佳杰的需求（51:07-55:08）

能力佳杰原话
端云任务同步 "你必须第一个需要建设的能力就是端云的任务同步机制。比如定时任务、预设条件任务，它能够在端上有一个模块做持久化。"
区分任务级 vs 感知级 "你可能额外会有一些字段来区分它是任务级数据还是感知级数据。我在端侧做显示，不可能把所有触发器里很原始的场景都拿来显示——只有对用户有价值的'任务'才在任务中心显示。"
enable/disable接口 "我对你的操作：启用/不启用、删除/添加回来。你的数据库要支持我去写入，或者提供接口给我。"

六、澄清：AlwaysOn调度不归触发器

西钺（57:25）："调度always on是触发器来做吗？这个我记得佳杰他们有一个专门的AIBox上有一个模型调度的模块在做。"
佳杰（57:47）："AIBox的模块里面，有一个模块专门对所有请求做优先级排序。"

触发器可以请求开启/关闭AlwaysOn（比如"出地库→开always-on"），但调度中枢是AIBox任务调度，不是触发器。
