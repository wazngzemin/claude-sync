00.6 用一句“二排宝宝”把全骨架跑通

  1. 用户说“帮我照顾后排宝宝”进入 user_query，携带 speaker、timestamp、goal_list、env_info。
  2. 端侧/拒识/仲裁判断这不是普通闲聊，也不是简单车控，而是需要持续跟踪的目标类请求。
  3. Director SP 读取当前 goal_list，判断是否已有相似目标；若没有，调用 goal_list_update add_goal。
  4. 目标进入 goal list，后续由 advisor/goal reviewer 关注，不由 Director 直接更新细节进展。
  5. advisor 根据目标需要订阅车内视觉/安全带/睡眠/哭闹等感知信号（具体信号字段需后续文档确认）。
  6. 订阅模块未来被动态感知、条件或定时器触发，生成 advisor 输入回到 Director。
  7. Director 融合 advisor 建议、env_info、goal_list，决定是否说话、是否调车控、是否提醒用户。
  8. HMI/TTS/emoji 负责表达；如果用户取消“别盯了”，Director 调 goal_list_update delete_goal。
这就是完整骨架：端云分流解决“快不快”，RAG/Context 解决“知道什么”，Director SP 解决“怎么判断和编排”，工具解决“怎么执行”，goal/advisor/订阅解决“怎么持续”，HMI/交互治理解决“用户怎么感知”。后面每个章节都是在展开这六件事。

N逐节点骨架与流程详解：每个节点到底怎么运转

这一章按总图的一阶节点逐个拆。每个节点都包含：节点骨架图、内部流程图、输入、处理原理、输出、存在理由、优化/排查点。读这一章时，不要急着记名词，先看“这个节点把什么输入变成什么输出”。

节点 1：用户输入 / ASR

这两张图怎么读：左图是在讲“用户一句话如何变成 Director 能处理的结构化事件”。用户真正给系统的是语音，但 Director 不能直接吃语音，它需要的是一个带字段的 user_query：里面不仅有文字，还要有谁说的、坐在哪、什么时候说、当时车况如何、当前有哪些目标。

右图是在讲这个节点的内部加工顺序：先收音，再等 VAD 判定用户说完，再补齐元数据，最后进入分流。这里最容易被忽略的是“补元数据”这一步。产品经理看 query 往往只看文字，但车载场景里，同一句“打开窗户”，主驾说和后排说，安全权限、执行对象、体验都不一样。

例子：用户说“后面有点热”。如果 speaker_position 是二排，系统应该优先理解为后排体感；如果是主驾说“后面有点热”，可能是在替后排乘客表达。这个判断不是 Director 突然聪明，而是输入节点把“说话人位置”给了它。

维度详细讲解
输入用户语音、说话人位置、时间戳、上一轮情绪、车端环境。SP 最终接收的是结构化 user_query，不是裸文本。
处理先把语音转文字，再确定是否判停，再补说话人、座位、时间、goal_list、env_info 等元数据。
输出进入后续分流的 user_query 对象。
原理车载交互不是聊天框输入，必须先解决“谁在说、从哪个座位说、是否说完、是不是对助手说”的问题。
为什么存在：没有这个节点，后面的 Director 无法区分主驾/副驾/后排，也无法做时间感知、情绪连续、目标恢复。
优化/排查：首 token 慢先看 ASR/VAD 是否拖长；误触发先看是否把人人对话当成 user_query；多乘客错误先看 speaker_position。

节点 2：端侧快链路

这两张图怎么读：左图表达的是“能在端侧快解的，就不要进入云端大模型”。端侧快链路像高速收费站的 ETC：高置信、低风险、高频的请求直接放行执行；不确定的才交给后面的 RAG/Planner。

右图拆的是快链路内部判断：先看能力覆盖不覆盖，再看置信度够不够，再看权限和安全是否允许。三个条件都满足，才直接执行。只满足“看起来像车控”还不够，比如“打开车门”虽然是车控意图，但车辆设备知识里写了车门不支持语音控制，就不能端侧执行。

例子：“空调调到 24 度”可以走快链路，因为能力明确、参数合法、风险低；“我有点冷，帮我弄舒服点”就不一定适合端侧直接做，因为它可能要结合温度、座椅、风向、用户偏好，需要更复杂推理。

维度详细讲解
输入明确、高频、可端侧闭环的 query，例如简单车控、部分设置、导航基础控制。
处理判断端侧能力是否覆盖、置信度是否足够、安全/权限是否允许；满足则不进云端复杂 Planner。
输出端侧执行结果，或未命中后继续上云。
原理“能用简单方法搞定，不要漏到后面复杂模型”。这是漏斗架构的第一层降本降时延。
为什么存在：保障 150ms 级高频体验，不让打开空调这类任务被 500ms+ Planner 拖慢。
优化/排查：快链路命中过少会慢；命中过宽会误执行。坏 case 要看 query 是否本该端侧、是否被错误上云。

节点 3：RAG / 仲裁漏斗

这两张图怎么读：RAG/仲裁漏斗不是一个单点模型，而是一组“前置减负机制”。左图里同一个 query 会被分到句法 RAG、情景 RAG 或仲裁模型：句法 RAG 处理明确规则，情景 RAG 负责给模型补知识，仲裁模型负责决定最后是直接执行还是交给 Director。

右图强调“知识注入不是越多越好”。系统先用规则命中简单场景，再按需注入 few-shot/knowhow，最后仲裁路由。这样做的底层原因是：Director 的 SP 已经很厚，如果所有工具、所有示例、所有知识都塞进去，首 token 会慢，推理也会被噪音干扰。

例子：“导航去虹桥机场 T2”是明确路线规划，可以少注入甚至不注入复杂知识；“去机场接人，顺便找个顺路的日料店”就需要情景 RAG/Planner，因为它涉及先导航、再沿途 POI、再可能添加途经点。

维度详细讲解
输入未被端侧直接解决的 query，以及可召回的规则、知识、情景示例。
处理句法 RAG 负责明确表达，情景 RAG 负责模糊场景知识注入，仲裁模型决定直接执行还是进复杂 Planner。
输出directExecute、modelExecute、或进入 Director 的增强输入。
原理这是“层层过滤”和“不同阶段注入知识”的具体实现。
为什么存在：把重复、明确、可规则化的问题挡在 Director 外面，减少 token、时延和大模型不稳定。
优化/排查：召回错会污染模型；仲裁错会让简单任务变慢或复杂任务被草率处理。

节点 4：Director / Planner SP 决策器

这两张图怎么读：左图把 Director 的真实形态讲出来了：它不是一个薄薄的“意图分类器”，而是一个厚 SP 驱动的决策器。它同时读三类输入、26 个工具描述、大量规则、few-shot、车辆知识、聊天风格和目标状态，然后输出一个 JSON。

右图讲的是 ReAct 闭环。Director 第一次推理时不一定能完成任务，它可能先调用工具；工具返回后，tool_feedback 会重新喂给 Director，触发二次推理。二次推理再决定要不要说话、要不要继续调工具、要不要总结。

所以 Director 的关键不是“想一次”，而是“每次根据当前输入做下一步最合理的控制决策”。如果输入是 user_query，它要判断用户需求；如果输入是 advisor，它要融合主动建议；如果输入是 tool_feedback，它要根据执行结果继续推进。

例子：用户说“后排都系安全带了吗？”Director 第一次输出 search_visual_info；视觉工具返回“左后未系，右后已系”；Director 二次推理输出提醒话术。这里如果没有 tool_feedback 回灌，系统只能说“我看看”，无法完成闭环。

维度详细讲解
输入user_query、advisor、tool_feedback，以及公共 context、goal_list、env_info、记忆、视觉、车辆知识。
处理基于厚 SP 进行任务识别、工具选择、目标增删、话术/情绪/emoji 决策，并遵守串行/并行、静默、失败处理等规则。
输出talk_or_not、talk_content、emoji_id、emotional_state_change、action_list。
原理Director 是控制面，不是执行面。它决定“说不说、说什么、调什么、是否建目标、是否继续推理”。
为什么存在：把自然语言、上下文、工具能力和用户体验策略统一编排成一轮可执行决策。
优化/排查：action 错查工具描述和 few-shot；话术错查 talk_or_not/聊天风格；目标错查 goal_list_update 规则；慢查 SP 过胖和 context 注入。

节点 5：下游工具 / Agent

这两张图怎么读：左图是在讲 action_list 进入工具层之后，会被拆成两种情况：能并行的工具一起执行，有依赖的工具必须串行。Director 输出多个 action 不等于一定并行，能不能并行取决于任务之间有没有信息依赖。

右图讲的是工具调用的生命周期：先把 Director 给出的自然语言 params 转成工具能理解的请求，再由下游 Agent/API 执行，最后返回成功、失败或候选结果。返回结果不是终点，而是下一轮 Director 决策的输入。

例子：“播放音乐，同时把空调开到 20 度”可以并行，因为音乐和空调互不依赖；“先搜一家日料店，再把第二家设为途经点”不能并行，因为路线规划要等 POI 返回候选，用户或模型确认具体哪家之后才能继续。

维度详细讲解
输入Director 输出的 action_list，包括 tool_name 和 params。
处理执行车控、导航、POI、媒体、记忆、视觉、行程、目标等具体能力；区分可并行和串行依赖。
输出tool_feedback，可能是成功、失败、候选列表、执行结果、查询内容。
原理大模型不直接操作车辆，而是通过工具把自然语言决策落到业务系统。
为什么存在：把“想做什么”变成真实动作，同时通过 feedback 支持 ReAct 二次推理。
优化/排查：工具调错看 tool 描述；串并行错看依赖规则；失败处理差看 feedback 是否结构化、Director 是否 replan。

节点 6：goal list 目标账本

这两张图怎么读：goal list 不是普通上下文文本，而是“系统正在替用户管理的目标账本”。左图里 Director 只负责 add/delete，advisor 负责更新进展，HMI 负责把目标展示出来。这个写权限划分非常重要。

右图讲的是目标创建的内部逻辑：先识别这是不是持续目标，再和已有 goal 去重匹配，再决定新增或删除。不能用户一说“继续看着点”就新增一个目标，否则 goal list 会越来越脏，后续 advisor 会重复触发。

它最关键的价值是“恢复”。如果用户中途打断、过了几分钟再问“刚才那个还在吗”，Director 可以从 goal_list 里知道当前有哪些目标还在进行，而不是靠对话历史硬猜。

例子：用户说“沿途给我介绍风景”，Director 调 goal_list_update add_goal；后来用户说“后面不用讲了”，Director 删除“沿途风景讲解”这个目标，但不应该删除“后排乘客休息中，保持安静”这个另一个目标。

维度详细讲解
输入Director 的 goal_list_update add/delete、advisor 的进展更新、用户的取消/修改表达。
处理保存目标内容、目标状态、目标细节和进展；作为 Director、advisor、HMI 的共享状态。
输出当前目标队列，进入 SP 的 goal_list 字段，也用于 HMI 展示。
原理goal list 是“长期任务的账本”。Director 管增删，advisor 管细节和进展，避免写冲突。
为什么存在：没有 goal list，系统只能单轮响应，无法持续跟踪、恢复、取消和展示目标。
优化/排查：重复创建看去重；目标忘记看是否写入；取消失败看 delete_goal；进展错看 advisor 写回。

节点 7：advisor / 订阅

这两张图怎么读：advisor 是持续目标的“后台负责人”。左图里 goal 会派生出 dynamic advisor，也可能被静态 advisor 关注；advisor 再通过订阅模块连接未来事件。也就是说，Director 创建目标之后，不是自己一直盯，而是把后续跟进交给 advisor。

右图是 advisor 的运行循环：读 goal list，决定需要订阅哪些信号，等待感知/条件/定时触发，触发后生成 advisor input 给 Director，必要时更新目标进展。advisor input 进入 Director 时，SP 明确提醒“这不是用户说的话”，所以 Director 不应该像回应用户一样回应 advisor。

例子：“照顾后排宝宝”创建目标后，动态 advisor 可能订阅后排视觉状态。未来检测到宝宝睡着，advisor 给 Director 建议“降低后排风量、轻声提醒”。Director 再决定是否说话、是否调车控、是否更新用户可见状态。

维度详细讲解
输入goal list、公共 context、感知信号、条件触发、定时器、用户取消/修改。
处理静态 advisor 关注固定方向，动态 advisor 承接新目标；订阅模块负责未来唤醒。
输出advisor 建议输入给 Director，或更新 goal 的细节和进展。
原理把持续任务从 Director 单线程中拆出去，形成多线程目标跟踪。
为什么存在：持续目标不能靠用户每次开口，也不能让 Director 一直盯；advisor 是目标的后台负责人。
优化/排查：不触发看订阅；乱触发看条件/频控；目标进展错看 goal reviewer；主动服务打扰看排队策略。

节点 8：公共 Context 服务

这两张图怎么读：公共 Context 是全系统共享的“现场记忆”。左图里的端状态、视觉、用户记忆不是直接全塞进 Director，而是先经过预处理，再形成可注入、可查询的 context。这样做是为了减少 token 和噪音。

右图讲的是 Context 的工程流程：先从多源收集，再结构化、打时间戳、分类，再按场景/人物/任务过滤，最后注入给 Director 或 advisor。它不是一个静态大文本，而是一个会随环境变化不断更新的状态服务。

理解 Context 的关键是：它决定模型“看见什么”。很多坏 case 表面上是模型推理错，其实是 Context 没给对信息，或者给了过期/无关的信息。

例子：用户问“现在车里热吗？”如果 env_info 已经有“车内 35 度，空调关闭”，Director 可以直接参考，不必调用车辆状态查询；如果 context 没有温度，才需要调用 search_vehicle_status_info。

维度详细讲解
输入车辆状态、舱内外视觉、用户记忆、事件日志、屏幕信息、任务历史、工具结果。
处理统一存储、压缩总结、打标分类、提前过滤，按场景/人物/任务给模型注入。
输出给 Director 和 advisor 的上下文片段，或 SP 输入里的 env_info / 记忆 / 视觉信息。
原理模型不应每轮全量读世界，只应读和当前任务相关的现场。
为什么存在：解决情景割裂，让系统知道车内外、用户、目标和历史发生了什么。
优化/排查：信息缺失看过滤；幻觉看 context 来源/时效；时延高看是否全量注入和现查现拼。

节点 9：HMI / TTS / emoji / 交互治理

这两张图怎么读：HMI/TTS/emoji 不是“最后播一下”的简单尾巴，而是用户真正感知系统是否聪明的地方。左图里 Director JSON 会拆成 TTS、HMI 卡片和交互治理：说话、展示、打断、排队都在这里发生。

右图讲的是交互治理的判断顺序：先看本轮需不需要说话，再看有没有打断当前播报，再决定合并还是排队。车里只有一条语音通道，用户主动说话优先级最高，advisor 建议和条件任务不能随便抢话。

SP 里的 talk_or_not 很关键。很多工具成功后用户能直接感知，比如音乐已经响了、空调已经开了，就不需要再说“已为你打开”。这不是偷懒，而是让语音助手更像人。

例子：用户说“暂停音乐”，工具成功后音乐停了，最佳输出可能是 talk_or_not=no；但如果用户问“为什么没声音了”，就需要解释。是否说话取决于用户是否需要反馈，而不是工具有没有返回成功。

维度详细讲解
输入DirectorDecision、goal list、工具结果、用户新 query、advisor 主动建议。
处理决定是否播报、是否展示卡片、emoji 如何变化、是否打断/合并/排队/静默。
输出用户听到的话、看到的卡片、感知到的动作反馈和目标状态。
原理车内是单通路语音场景，用户主动输入优先，主动服务必须低打扰。
为什么存在：后台做对不等于体验好，必须把动作和状态转换成用户能理解、能控制的反馈。
优化/排查：废话多看静默规则；打断乱看 P0/P1；目标不可见看 HMI 卡片；首 token 慢看铺垫话术。

节点 10：tool_feedback / advisor feedback 反馈回路

这两张图怎么读：反馈回路是 Director / Planner 从“一次性判断”升级成“闭环执行”的关键。左图表示两个来源会把系统重新推回 Director：一个是工具返回的 tool_feedback，另一个是 advisor 在目标推进中给出的建议。

Director 收到反馈后，不是机械地把结果念给用户，而是重新判断：这个结果用户是否已经能感知？是否还缺一步？是否需要问用户？是否说明任务失败？所以工具返回只是“新证据”，真正决定下一步的仍然是 Director。

右图把反馈分成三类：成功、失败、候选列表。成功不一定要说话，比如“音乐已播放”用户自己听得到；失败不一定要道歉，SP 明确要求积极想办法；候选列表通常要把选择权还给用户，比如 POI 返回多个合生汇时，不能直接导航到某一个。

例子：用户说“去合生汇”，POI 工具返回“五角场店、虹桥店”。这时 feedback 不是终点，而是二次推理的起点。Director 应输出“有两个合生汇，你要去哪个？”而不是直接调 route_planning_qa。这就是反馈回路避免误执行的价值。

维度详细讲解
输入工具返回、advisor 建议、订阅触发事件。
处理Director 判断要静默、总结、继续调用、询问用户、replan 或暂时放弃。
输出下一轮 DirectorDecision，或者目标进展更新。
原理反馈回路让系统从“一次性决策”变成“执行后再判断”的闭环。
为什么存在：没有 feedback，ReAct 就断了，复杂任务无法根据真实结果继续推进。
优化/排查：工具返回成功但还废话看静默规则；失败不重试看失败处理；候选不问用户看澄清策略。

节点 11：拒识 / 过滤

这两张图怎么读：拒识 / 过滤不是一个单点模块，而是多层保护。左图表示 query 可能在端侧、云侧、Planner 内部被拦住；右图表示 Planner 内部至少要判断三件事：是不是对我说、我听懂了吗、这个请求是否安全或符合品牌/隐私/能力边界。

车载场景比手机助手更需要拒识，因为车里有多人对话、背景噪声、驾驶安全约束。不是每一句被 ASR 转成文字的话都应该进入工具链路，否则就会出现“乘客聊天被助手接话”或“玩笑话触发真实车控”的问题。

SP 里已经有一条很重要的规则：如果用户说的话听不懂，或感觉不是对你说的，不需要说话也不用调用工具。产品上这叫“不打扰”，技术上就是把误触发损失挡在最前面。

例子：主驾和副驾聊天说“等会儿把窗户关了吧”，如果没有唤醒或没有明确对助手下指令，系统应倾向静默；但如果用户明确说“豆包，关闭所有车窗”，才应该进入车控链路。差别不在字面动作，而在“是否是对助手发出的可执行指令”。

维度详细讲解
输入所有用户语音，尤其是人人对话、噪声、不完整表达、越权/不可执行请求。
处理多级判断：端侧、云侧、Planner 内部；SP 也规定听不懂或不是对你说可不说话不调工具。
输出静默、拒识回应、或继续进入任务链路。
原理车内场景必须避免“助手乱接话”和“误执行”。
为什么存在：不是每句话都应该被 AI 接管，尤其车里多人说话时。
优化/排查：误拒看拒识层级；误唤醒看 ASR/说话人；不该执行却执行看 Planner 内拒识和安全规则。

节点 12：耗时与交互调度控制面

这两张图怎么读：耗时控制面不是某一个独立工具，而是一组贯穿全链路的工程策略。左图按时间线看：首 token、Planner 推理、工具执行、交互调度，每一段都会影响用户体感；右图按架构策略看：能走快链路就别进慢思考，必须进慢思考时也要先做前置筛选。

这里最容易误解的是：耗时优化不是单纯“把模型换快”。Director 当前 SP 很厚，里面有工具注册表、车辆知识、聊天风格、情绪状态机、emoji、few-shot 示例。模型每轮都要在这些材料里找相关规则，所以前置筛选、工具裁剪、Context 裁剪，本质上是在减少模型每次要读的“书”。

从产品体验看，复杂任务可以慢一点，但不能让用户不知道系统在干嘛。所以耗时控制分两类：一类是缩短真实计算路径，比如端侧快链路、并行工具、缓存、裁 prompt；另一类是管理等待感，比如先说“我先查一下”，再让长工具异步执行。

例子：“空调调到 24 度”应该尽量走端侧快链路，用户几百毫秒内就感知动作；“帮我规划桐庐两日游”必然慢，因为要搜索、整理、保存行程，这时 Director 先给一句“我先搜搜山水古村和农家乐推荐”，再进入工具链路，用户就不会觉得系统卡住。

维度详细讲解
输入所有链路节点耗时、是否首 token 前、任务优先级、是否用户主动输入。
处理快慢分流、前置筛选、Context 过滤、并行工具、响应话术按需、打断/合并/排队。
输出更短首 token、更少无效等待、更清晰的用户反馈。
原理时延优化不是单点模型加速，而是把耗时从关键路径移走，把复杂任务异步化。
为什么存在：车载体验对“快”和“不打扰”都敏感，复杂任务可以慢，但不能让用户空等。
优化/排查：慢先区分首 token 慢、动作完成慢、总结慢；再查 ASR、分流、prompt、context、工具串行。

X逐节点深讲义：像讲文章一样把每个节点掰开

上一章是“图 + 节点卡”。这一章是讲义：每个节点按“它在解决什么问题 → 它吃什么输入 → 它内部怎么判断 → 它吐什么输出 → 坏 case 怎么查 → 架构怎么优化”展开。你可以把这一章当成汇报时的讲稿。

深讲 1：用户输入 / ASR 不是“把语音转文字”，而是把车内事件变成可推理对象

如果把 Director 比作大脑，ASR 输入层不是耳朵那么简单，它还要告诉大脑“谁说的、在哪里说的、什么时候说的、说话时车里是什么状态”。没有这些元信息，后面的推理会从第一步就偏。

输入到底包含什么
  • 语音内容，经过 ASR 形成 query。
  • 说话人名字和位置：speaker_name、speaker_position。
  • 时间：timestamp，用于时间间隔、久别重逢、任务是否继续。
  • 上一轮情绪：assistant_emotional_state。
  • 当前目标和车况：goal_list、env_info。
内部真正做的判断
  • 用户是否说完：VAD 判停影响首 token。
  • 是不是对助手说：不是所有车内语音都应触发。
  • 这句话有没有明确执行对象：主驾、副驾、后排含义不同。
  • 元信息是否足够支撑后续决策。
带着例子走一遍：用户说“后面有点热”。如果输入只给 Director 一个 query，它只能猜；如果输入同时给出 speaker_position=二排右、env_info=车内29度、goal_list=后排宝宝休息中，那么 Director 才能判断这是舒适诉求，可能需要调空调或询问是否降低后排风感，而不是泛泛回复“我帮你看看”。
常见坏 case：用户明明不是对豆包说话，系统却接话；后排乘客说“有点冷”，系统调了主驾温度；用户上一轮刚说“等会继续”，5 分钟后系统完全忘了。这些问题不一定是 Planner 不聪明，很多时候是输入事件缺字段或字段不准。
优化抓手：输入层要把“说话人、座位、时间、车况、目标队列”当成一等字段，而不是塞进自然语言里。坏 case 排查先看 input JSON 是否完整，再看 Director 输出。

深讲 2：端侧快链路是在保护体验，不是为了绕过大模型

车载助手里有一类请求不值得上云，也不应该等大模型慢慢推理，比如“空调 24 度”“暂停播放”“地图放大”。端侧快链路的核心价值是把高频、确定、低风险任务从慢链路里摘出来。

它吃什么输入
  • 结构化 query 和端状态。
  • 本地白名单、简单意图规则或轻量模型结果。
  • 权限和安全约束，例如智驾仅主驾可控、车门不支持语音。
它怎么决定是否放行
  • 能力覆盖：这个功能端侧是否能执行。
  • 参数明确：有没有可执行参数，例如温度、档位、开关。
  • 风险可控：不会造成安全/误执行风险。
  • 置信度足够：不要把“我有点冷”硬当成固定车控。
带着例子走一遍：“空调调到 24 度”满足能力、参数、低风险，所以快链路直接调车控；“我有点冷”虽然也可能调空调，但它是体感表达，可能涉及座椅加热、风向、温度、乘客位置，不应草率快解。
常见坏 case：快链路过宽会误执行，比如用户聊天说“窗户别开太大”被当成指令；快链路过窄会拖慢高频动作，比如每次暂停音乐都上云。两边都影响体验。
优化抓手：把快链路白名单分成“强确定可执行”和“需要 Planner 解释”的两层。不要只按意图名分流，要按“意图 + 参数完整度 + 安全约束 + 说话场景”一起分流。

深讲 3：RAG / 仲裁漏斗是在给 Director 减负，也是在防止知识乱入

Director 很强，但不能让它处理所有问题。漏斗层的职责是先把能规则化处理的、能知识补全的、能直接执行的任务筛掉，只把真正需要推理和编排的交给 Director。

它内部有三种力量
  • 句法 RAG：处理表达固定、规则明确的单轮任务。
  • 情景 RAG：补充场景相关知识、工具 tips、车辆知识。
  • 仲裁模型：决定 directExecute、modelExecute 或进入 Planner。
它输出的不是答案，而是路由
  • 简单明确：直接执行或走轻链路。
  • 知识型：带着召回材料进入模型。
  • 复杂多步：进入 Director 做工具编排。
  • 无关或不该答：进入拒识/过滤。
带着例子走一遍：“胎压正常吗”需要车辆状态查询；“胎压报警怎么处理”更像车况维养问答；“去医院顺路找个有充电桩的停车场”是多工具、多约束任务，应交给 Director 编排路线、POI 和可能的询问。
常见坏 case：RAG 召回太多，Director 被噪音带偏；仲裁太激进，复杂任务被当简单工具执行；仲裁太保守，简单任务都进 Director，耗时上升。
优化抓手：调 RAG 不只看召回率，还要看“召回材料是否真的进入了正确决策”。最好把分流结果、召回片段、最终工具选择放在同一条 trace 里看。

深讲 4：Director / Planner 是厚 SP 策略机，不只是意图识别

当前 Director 的真实形态，是一个由系统提示词驱动的大模型策略机。它同时负责理解用户、选择工具、决定说不说话、选择 emoji、维护情绪、增删目标、处理工具反馈。也就是说，它不是一个“分类器”，而是车载任务操作系统的调度内核。

它读什么
  • 三类触发：user_query、advisor、tool_feedback。
  • 上下文：goal_list、env_info、记忆、视觉、车辆知识。
  • 工具注册表：当前 SP 暴露 26 个工具。
  • 规则：串并行、静默、失败处理、情绪状态机、隐私、提示词保护。
它吐什么
  • talk_or_not：这一轮是否开口。
  • talk_content：要说什么。
  • emoji_id：形象动画。
  • emotional_state_change：情绪是否变化。
  • action_list：调用哪些工具，参数是什么。
带着例子走一遍：用户说“后排乘客都系好安全带了吗”。Director 不能直接回答，因为它不知道视觉结果；它应先说“我看看”，调用 search_visual_info。工具返回“后排左未系，右已系”后，Director 二次推理，输出提醒话术，不再调用工具。这就是 ReAct：想一步、做一步、看结果、再想。
常见坏 case：工具选错、该并行的串行、该串行的并行、该静默时废话、advisor 输入被当用户输入、tool_feedback 后不继续。这些都属于 Director 策略问题。
优化抓手：当前 SP 太厚，优化方向是动态裁剪工具、示例、车辆知识和 Context；把高频硬规则结构化外置，减少模型每轮在长 prompt 里“找规则”的成本。

深讲 5：工具 / Agent 层是执行边界，工具描述就是 Director 的“法律条文”

Director 自己不打开空调、不导航、不查天气，它只能产出 action_list。真正做事的是工具层。工具描述写得清不清楚，直接决定 Director 会不会选错工具、传错参数、并行错任务。

工具层包含什么
  • 车控、系统设置、通讯、车辆状态、天气、POI、路线、导航。
  • 音乐、视频、播客、图片生成。
  • 记忆、人脸注册、个性化技能、目标管理。
  • 视觉问答、智驾、氛围灯、车况、用车报告、行程记录。
工具调用的核心原则
  • 无依赖可并行，有依赖必须串行。
  • 查询类工具返回后通常需要二次推理。
  • 控制类工具成功且用户可感知时可静默。
  • 工具能力边界不能脑补，比如车门不支持语音控制。
带着例子走一遍：“放点音乐，车里太热了”可以并行调用音乐和车控，因为两者没有依赖；“去虹桥机场 T2，顺路找寿司店”不能简单并行所有动作，因为“顺路”依赖路线，通常要先规划终点路线，再沿途搜 POI。
常见坏 case：把媒体点播误交给系统设置；把目的地搜索交给导航基础控制；把模糊 POI 直接路线规划；工具失败后重复死循环。
优化抓手：工具 schema 要写清楚“能做什么、不能做什么、什么时候用、什么时候别用、返回多个候选怎么办”。很多 Planner 坏 case，本质是工具边界写得不够像判例法。

深讲 6：goal list 是长期任务账本，不是普通上下文

即时任务做完就结束；持续任务需要被记住、被展示、被未来事件唤醒。goal list 的价值就是把“用户托付给车的事”放进一个共享账本里，避免 Director 每轮靠对话历史硬记。

它记录什么
  • 目标内容：用户想持续完成什么。
  • 目标状态：进行中、完成、取消等。
  • 细节和进展：由对应 advisor 更新。
  • HMI 展示信息：用户能看到系统正在帮他盯什么。
读写权责
  • Director：根据 user query 增删目标。
  • advisor：更新目标细节和进展。
  • HMI：展示目标状态。
  • SP 明确：advisor 输入不使用 goal_list_update 创建目标。
带着例子走一遍：用户说“帮我一路介绍沿途风景”。这是持续任务，Director 创建 goal；后续 advisor 根据路线和位置触发介绍。用户说“后面不用讲了”，Director 删除这个目标，但如果同时还有“后排乘客休息中”的目标，不能一起删。
常见坏 case：目标重复创建；用户补充目标被当成新目标；删除一个目标时误删其他目标；Director 直接更新进展造成和 advisor 冲突。
优化抓手：goal list 最好结构化，至少要有目标 id、类型、owner、状态、创建来源、最近更新时间。当前文档里细节进展偏自由文本，这会是后续一致性风险。

深讲 7：advisor 是持续任务的后台负责人，Director 不是一直自己盯

如果让 Director 自己盯所有目标，它会越来越重。advisor 的设计，是把持续目标拆给专门的后台角色：它读 goal list 和 Context，必要时给 Director 发建议，或者更新目标进展。

advisor 分两类
  • 静态 advisor：舒适、出行、情感、内容，长期存在。
  • 动态 advisor：为动态目标创建，有生命周期。
  • goal reviewer：判断目标是否达成或需要推进。
它怎么工作
  • 读取对应目标。
  • 订阅需要的感知、条件或定时信号。
  • 在合适时机生成 advisor 输入。
  • Director 融合建议后决定是否行动和表达。
带着例子走一遍：“照顾后排宝宝”不应该让 Director 每轮都主动问宝宝怎么样，而是由 advisor 订阅视觉/哭闹/睡眠等信号，只有出现需要提醒的情况才生成建议。Director 再判断要不要说话、调空调或提醒用户。
常见坏 case：advisor 太积极导致打扰；advisor 太保守导致持续目标不推进；多个 advisor 同时建议，抢占语音通道；advisor 建议被 Director 当成用户指令。
优化抓手：advisor 需要频控、优先级、去重和冲突仲裁。产品上要定义什么叫“值得打扰用户”，技术上要记录 advisor 建议是否被采纳，以及为什么没采纳。

深讲 8：公共 Context 是共享现场，不是把所有信息都塞进 prompt

Context 的核心不是“更多信息”，而是“恰好相关的信息”。车载系统有大量状态：车辆、屏幕、视觉、记忆、事件日志、目标队列。如果全部塞给 Director，会慢、贵、吵，还可能泄露隐私。

Context 装什么
  • 端状态：空调、导航、车速、媒体等最新状态。
  • 情景和视觉：舱内外场景、屏幕信息。
  • 用户记忆：偏好、关系、历史事件。
  • 任务状态：goal list 和历史任务。
  • 工具知识、车辆知识、通用经验。
Context 服务做什么
  • 统一维护和存储。
  • 压缩总结，减少长上下文。
  • 打标分类，方便按任务取用。
  • 提前过滤，减少噪音和隐私风险。
带着例子走一遍：用户说“有点闷”。如果 env_info 已经有车内温度、空调状态、空气质量，就不必先查一堆工具；如果没有关键状态，Director 才调用车辆状态查询。Context 做得好，能减少工具调用和模型猜测。
常见坏 case：状态过旧导致错误动作；无关记忆注入导致模型过度个性化；隐私记忆给错人；视觉信息没有更新却被当实时事实。
优化抓手：Context 要按场景、人物、任务过滤。优化 Planner 时不要只看模型输出，要同时看“当轮注入了什么 Context、哪些应该注入但没注入、哪些不该注入却注入了”。

深讲 9：HMI / TTS / emoji 是体验出口，不是装饰层

用户不会感知到内部工具链路，只会感知到系统说了什么、做了什么、有没有打扰、屏幕有没有变化、表情是否合适。所以 HMI/TTS/emoji 是架构闭环的一部分，不是最后的 UI 包装。

它接收什么
  • Director 输出的 talk、emoji、emotion。
  • 目标状态和任务进展。
  • 工具成功/失败状态。
  • 当前语音通道是否被占用。
它决定什么
  • 是否播报，还是静默。
  • 是否显示卡片。
  • 是否打断当前播报。
  • 多个任务是否合并、排队、丢弃。
带着例子走一遍：用户说“暂停音乐”，工具成功后音乐已经停了，最佳体验可能是静默；用户问“为什么没声音了”，就必须解释。是否说话不是由工具是否返回成功决定，而是由用户是否需要反馈决定。
常见坏 case：每个成功都播报导致烦；advisor 主动服务抢话；目标进行中但 HMI 不展示，用户不知道系统还在盯；emoji 和情绪不匹配，显得割裂。
优化抓手：建立“可感知成功可静默、不可感知成功要反馈、失败要 replan、候选要询问”的表达策略。HMI 卡片要能承接 goal list，不能只靠语音。

深讲 10：tool_feedback / advisor feedback 是 ReAct 闭环的生命线

没有反馈回路，Director 只能做一次性计划；有了反馈，它才能根据真实执行结果继续推进、修正、询问或结束。复杂任务的“智能感”很大一部分来自反馈后的二次判断。

反馈有哪些
  • 工具成功：动作完成或查询有结果。
  • 工具失败：参数不合法、能力不支持、执行失败。
  • 候选列表：POI、路线、搜索结果。
  • advisor 建议：目标推进或主动服务。
Director 怎么处理
  • 成功：静默、总结或进入下一步。
  • 失败：换方案、询问、重试或暂时放弃。
  • 候选：让用户选择，不随便代选。
  • advisor：融合用户上下文，不当成用户说话。
带着例子走一遍：POI 返回两个“合生汇”，Director 应问用户去哪个；路线规划成功后如果用户能在屏幕看到路线，可以不再长篇解释；音乐播放成功后通常可以静默，因为声音就是反馈。
常见坏 case：工具失败后直接道歉结束；工具返回候选却直接执行第一项；工具成功仍重复播报；advisor 建议与用户当前意图冲突但未仲裁。
优化抓手：工具返回要结构化标注 success/fail/candidates/need_user_choice。Director 的二次推理 prompt 要强调“反馈只是证据，不是必须播报的话术”。

深讲 11：拒识 / 过滤是在保护用户，也是在保护系统边界

车内不是安静的输入框，用户可能和家人聊天、打电话、唱歌、开玩笑。拒识层的目标不是少服务，而是避免系统在不该介入的时候介入，在不能执行的时候假装能执行。

拒识判断什么
  • 是不是对助手说。
  • 是否听懂、是否有足够信息。
  • 是否在工具能力范围内。
  • 是否涉及安全、隐私、品牌或提示词保护。
可能输出什么
  • 静默：不是对我说或连续重复输入。
  • 澄清：信息不足。
  • 拒绝或引导：能力不支持、隐私不允许。
  • 进入正常链路：确认是有效请求。
带着例子走一遍：副驾对主驾说“等会儿窗户关一下”，这可能只是人人对话，不应触发；用户明确说“豆包，关一下副驾窗户”，才是可执行请求。拒识的难点是同样几个字，在不同对话关系里含义不同。
常见坏 case：助手乱接人人对话；用户问隐私行程，系统直接查；用户问提示词/工具描述，系统泄露；不支持语音的车门/儿童锁被误执行。
优化抓手：拒识不能只靠关键词，要结合唤醒、说话对象、座位、上下文、能力边界。坏 case 排查时先看“是否应该进入 Planner”，再看 Planner 内部输出。

深讲 12：耗时控制面要把“计算慢”和“用户等得烦”分开优化

车载体验里，慢有两种：一种是真慢，模型和工具耗时长；另一种是体感慢，用户不知道系统是否听懂、是否在做。架构优化要同时处理真实耗时和等待感。

真实耗时来自哪里
  • ASR/VAD 判停。
  • RAG 召回和仲裁。
  • Director 厚 SP 推理。
  • 工具串行执行。
  • tool_feedback 后二次推理。
等待感来自哪里
  • 用户说完后没有任何反馈。
  • 系统先说一大段再执行。
  • 复杂任务没有阶段性提示。
  • 多个主动服务抢语音通道。
带着例子走一遍：“帮我规划桐庐两日游”不可能像“暂停音乐”一样快。正确体验是先快速确认“我先查查”，然后搜索、整理、保存行程；而不是沉默十几秒后突然给结果。复杂任务可以慢，但要让用户知道系统在推进。
常见坏 case：首 token 慢；工具串行导致总耗时长；prompt 太胖导致每轮推理慢；静默规则不当导致用户以为没反应；主动服务排队策略不清导致打断乱。
优化抓手：先把耗时拆成 ASR、分流、RAG、Director、工具、二次推理、TTS 七段，再判断是计算问题还是交互问题。技术上做端侧快链路、prompt 裁剪、Context 裁剪、工具并行；产品上做阶段性话术、HMI 状态和静默策略。
读完这一章你要形成的底层模型：Director 架构不是“一个大模型接一堆工具”，而是“输入事件结构化 → 快慢分流 → 知识/Context 裁剪 → Director 决策 → 工具执行 → 反馈二次推理 → 目标长期托管 → HMI/TTS 交互治理”。优化也不能只盯模型，要沿着这条链路逐段查。

AAdvisor 业务逻辑与架构：后台专家系统如何接入 Director

这一章整合 advisor 文件夹里的 8 份文档：舒适、出行、情感、内容四个静态 advisor，动态 advisor 一期/二期，感知订阅 OnePage、技术设计和流程要点。核心结论：advisor 不是“另一个聊天机器人”，而是后台专家控制面；它不直接面向用户说话，而是把 advice、目标进展、任务卡片、触发事件交给 Director/HMI 承接。

一句话定义 advisor：advisor 是围绕某类长期目标运行的“后台专家 Agent”。它读 Context、目标、触发事件和历史反馈，判断当前是否值得推进，生成 advice 给 Director；Director 再决定是否说话、是否调用工具、是否展示卡片。advisor 负责“观察和建议”，Director 负责“最终表达和执行”。

舒适保障车内每个人的舒适：温湿度、空气、空间、光线、声音、乘坐、振动等。事件触发；1 分钟内无新事件且有舒适子目标时自动 review。视觉描述、端信号、情景记忆、语义记忆、世界信息、人员信息、用户反馈。即时舒适建议；舒适子目标，如“极速制冷中”“升温中”。不是简单空调控制，而是多乘员、多区域、多因素舒适性推理。
出行在安全和效率前提下，让出行省心、省时、省钱、个性化。行前/行中事件触发；目的地不变时定时 review；兜风/充电等场景定时触发。导航、目的地/途经点、路线、时长、距离、拥堵、世界信息、历史出行偏好。目的地/目标预测；路线规划建议；沿途 POI/充电/休息/吃饭建议。不是导航工具本身，而是“为什么去、怎么安排一路”的出行专家。
情感让用户感知到被关心、被理解、被陪伴。事件触发；Waiting 状态且有子目标未完成时 review。事件、当前人车环境、感知输入、goals、记忆、疲劳/安全/情绪状态。即时情感建议；跨时间追踪的情感子目标；goal_operations。子目标必须是情感表达方向，不是具体执行动作；信息不足时不能强行猜。
内容成为车内“首席内容建议官”，先判断要不要介入，再决定推荐什么。默认不工作；Trigger 唤醒；长时间静默 review；其他 advisor 协同信号。时间、天气、剩余行程、道路复杂度、驾驶状态、用户状态、最近内容反馈、偏好记忆。介入等级 L0-L3；内容形态、风格、信息密度、标签、AIGC/URL/文本等。不是传统推荐列表，而是“场景-用户-内容”的体验提案；高压/争吵/拒绝时 L0 静默。

A.4 静态 Advisor 的共同工作流：Trigger 不等于一定要打扰

共同输入

事件触发、定时 review、目标状态、Context 快照、用户显性/隐性反馈、其他 advisor 协同信号。

共同输出

即时 advice、子目标/目标操作、介入等级、策略参数、任务卡片/状态更新建议，最终给 Director 承接。
  1. 默认静默：尤其内容 advisor 明确“没有 Trigger，就没有内容”。这说明 advisor 不是常驻刷屏，而是低打扰后台服务。
  2. Trigger 唤醒：事件、定时 review、目标未完成、其他 advisor 协同信号都可能唤醒。
  3. Context 汇聚：把视觉、端信号、记忆、世界信息、目标状态拉成当前快照。
  4. 是否值得介入：先判断当前是不是适合建议；复杂驾驶、争吵、用户刚拒绝内容时要降级或静默。
  5. 生成 advice/目标操作：给出即时建议，或新增/修改/删除子目标，或输出内容介入等级。
  6. 分发给 Director：Director 决定是否开口、怎么说、是否调用工具、是否展示卡片。
  7. 反馈回收：用户接受、拒绝、打断、完播、工具结果都会回到 Context，影响下一次 review。
PM 复述版：静态 advisor 像四个后台专家：舒适专家盯体感，出行专家盯路线和目的，情感专家盯情绪和关系，内容专家盯车上时间是否值得被内容填充。它们都不是“看到机会就说”，而是先判断“现在打扰用户是否合适”。

A.5 动态 Advisor：承接非即时任务的任务操作系统

动态 advisor 文档把任务分成四类，技术设计里明确编号：1 长时任务、2 持续任务、3 条件任务、4 定时任务。流程要点文档还补充：定时循环任务属于 type=4 + cycle，条件任务可单次/循环。

类型定义典型例子触发机制上限/约束执行特点
长时任务 type=1非结构化、过程型、陪伴型，持续一段时间，不依赖固定触发点。介绍沿途风景、照顾车上老人/小孩、冥想、陪玩游戏。定时 review + Context 查询；可结合 VLM 状态。技术设计：长时任务从 1 调整到 2；流程要点提到长时+持续要考虑推送节奏。AI 根据状态和历史 QA 推进，二期希望升级为模型生成任务编排脚本。
持续任务 type=2主要为 GUI Agent 服务，偏离后拉回，目标是持续推进一个结构化外部任务。点咖啡/订座/停车等生态闭环任务。GUI 状态、用户响应超时、生态应用 SOP/状态。持续任务 1；生态应用类 advisor 存在唯一实例。用户长时间不回应时提醒继续，任务卡片实时显示生态状态。
条件任务 type=3满足某个条件时触发执行，可单次或循环。副驾上车时开玩笑、车速高于 80 提醒、车内温度高于 25 调低空调。向触发器模块注册事件；callback 后触发 advisor。条件任务上限 6；条件/定时/循环强制推送保证 Planner 收到。二期从“触发时推理生成策略”改为“创建后提前生成策略”。
定时任务 type=4明确时间点或延迟时间后执行一次；固定间隔则是循环定时。5 分钟后关空调；每 10 秒播报比分；明天上午首次上车提醒。云端/端侧定时器；VLM 定时会先捞图请求 VLM。定时任务 6；VLM 定时 1；超高频循环拒绝；循环最大执行 30 次。二期新增端侧定时器和倒计时卡片，弱网/断网仍能 AI PUSH。
重要差异：动态 advisor 一期更像“触发后再推理”；二期目标是“创建任务时预生成策略，触发时直接按策略生成 advice 给 Director”，这样能减少触发链路上的一次模型推理耗时。PRD原文

A.6 动态 Advisor 创建链路：从用户一句话到后台任务

1. 用户表达目标例如“5分钟后帮我关闭空调”“如果我低头看手机就提醒我”“帮我照顾一下二排宝宝”。Director 先识别这是非即时任务，不应只当普通聊天。
2. 任务分类任务分类器输出 type：长时、持续、条件、定时；同时判断是否需要视觉信息、是否循环、执行次数。
3. 订阅/定时/VLM 改写条件任务注册触发器；定时任务注册定时器；视觉任务把目标改写为 VLM 可判断的 event/state query。
4. 目标/卡片创建创建 goal、任务卡片、任务岛/状态栏；二期要求卡片展示具体条件和执行内容，而不是只有简略 widget。
5. 策略预生成二期中条件任务 advisor 提前生成策略，触发后不再重新从零推理，降低事件触发到 advice 的耗时。
6. Director 承接任务创建成功/失败要回到 Director 说清楚；触发执行时，advice 发给 Director，由它最终决定说/做/静默。
例子：如果我低头看手机就提醒我。分类为条件任务 type=3；由于依赖视觉，VLM 改写为 event：用户低头看手机；触发器或定时 VLM 监测到事件后，动态 advisor 根据预生成策略生成 advice；Director 再输出提醒话术，必要时 HMI 卡片显示任务状态。

A.7 感知订阅：三种触发器怎么选

订阅类型适用场景文档规则例子风险
触发器事件偏瞬时、需要精准时间点 review 的条件。“偏瞬时需要精准时间点 review 的走触发器，此时不配置定时”。副驾上车、车速高于 80、接近某 POI、离开某 POI。触发器能力不足时无法注册，需要失败提醒或 fallback。
定时器不需要精准瞬间，周期性 review 即可。其他都走定时；可选 10s/30s/60s 等，特殊情况支持自定义。每 5 分钟播报比分、持续检查冥想状态、循环提醒。高频循环会打扰且耗资源，二期要求拒绝秒级/超高频。
VLM 定时需要视觉判断，但没有端侧事件可直接触发。定时触发前捞 VLM；演示 5s，量产 30-60s 或更长；VLM 定时上限 1。后排宝宝是否睡着、主驾疲劳、乘客把手伸出窗外。VLM 不是端模事件，云对云文本请求，实时性和成本都要控制。

A.9 四个静态 Advisor 的“业务逻辑深讲”

舒适 Advisor：不是“热了开空调”，而是全员舒适状态管理

舒适 advisor 的底层问题是：车内舒适不是单变量。温度、风感、空气、光线、噪声、座椅、空间、人员偏好都会互相影响。它的输出不应该只是工具调用，而应该是“当前舒适问题是什么、建议怎么调、是否形成子目标”。

例子：夏季主驾上车，车内 38℃，空调和座椅通风关闭。舒适 advisor 分析为温度舒适不满足，给 Director 建议“快速制冷 + 座椅通风”；如果车内有老人儿童且偏好不同，则要考虑分区、风向避人吹、座椅局部调节，而不是统一调到最低温。
坏 case：只看车内温度，不看乘员位置和偏好；只调空调，不考虑座椅/车窗/除雾/光线；重复建议用户已经拒绝的舒适动作。

出行 Advisor：不是导航，而是出行意图和全程安排预测

出行 advisor 解决的是“用户为什么出门、可能去哪、一路怎么更省心”。它覆盖行前目的地/目标预测，行中路线规划推荐，沿途充电、吃饭、休息、POI 的统筹建议。

例子：用户周一早上上车，历史上常去公司，但今天日历有医院预约。出行 advisor 不应只按高频目的地预测公司，而要结合日历/时间/历史路径，低置信时主动建议确认。
坏 case：把出行 advisor 当 POI 搜索；只看导航目的地，不推断用户出行目标；长途中只推荐充电站，不考虑吃饭、休息、排队和天气。

情感 Advisor：即时共情 + 跨时间情感子目标

情感 advisor 的输出分两类：一类是本轮可以立即执行的情感建议，比如安抚、夸赞、调侃、警告；另一类是需要跨时间追踪的情感子目标。文档明确要求子目标不是执行动作，而是情感表达方向，并且要有明确完成/删除条件。

例子：主驾疲劳驾驶时，情感 advisor 应输出安全相关即时建议，并可创建“持续关切主驾疲劳状态”的子目标；如果用户采取有效手段暂时抑制疲劳，则可以结束当前子目标，并说明原因。信息不足时不能强行猜用户情绪。
坏 case：把“问用户要不要开座椅按摩”当情感子目标；没有座椅占座信息却创建服务对象不明的目标；目标完成后仍输出无意义建议。

内容 Advisor：先判断能不能打扰，再判断推荐什么

内容 advisor 和传统推荐系统的根本区别：它不是“用户喜欢什么就推什么”，而是“当前场景、用户状态、认知负荷适合什么内容体验”。文档里有 L0-L3 介入等级：L0 静默，L1 氛围，L2 碎片，L3 沉浸。

例子：周一早高峰拥堵，用户略疲惫，有商业播客习惯。内容 advisor 不该推 60 分钟深度内容，而应推 3-5 分钟轻量商业故事；如果车内夫妻正在吵架，即便满足触发条件，也应 L0 静默。
坏 case：一有空档就推荐内容；用户刚关闭内容仍继续推荐；复杂路况下推强叙事内容；忽略上一条内容完播/跳过反馈。

A.11 动态 Advisor 任务创建：每一步到底做什么

动态 advisor 的创建链路不是“Director 判断一下然后建任务”这么简单。文档里实际拆成了多个模型/工程节点：任务分类、VLM 改写、触发器注册、定时器注册、策略生成、任务卡创建、创建结果回传。下面按真实执行顺序拆。

步骤输入内部逻辑输出失败/分支
1. Director 初判用户 query、goal_list、env_info、Context。判断是不是非即时任务：是否跨时间、跨条件、需要持续观察、需要 GUI/生态继续推进。进入动态 advisor 创建流程，或留在 Director 当场闭环。如果是普通即时任务却误进动态 advisor，会造成卡片污染和过度后台化。
2. 任务分类器用户目标文本。按规则输出 type、vlm_use、cycle、max_execution。长程/GUI 优先级最高；定时优先于条件；条件是否循环由“每次”等表达决定。{"type":"1|2|3|4","vlm_use":true/false,"cycle":true/false,"max_execution":-1/正整数}分类错会导致后面订阅方式全错，比如条件任务被当长时任务就不会精准触发。
3. 上限仲裁任务类型、当前任务数量。检查上限：长时 2、持续 1、定时 6、VLM 定时 1、条件 6。允许创建，或返回 error 给 Planner。文档明确从“新的挤掉旧的”改成“达到上限直接 error，等待完成或删除后再创建”。
4. VLM 改写用户目标 + 是否依赖视觉。把自然语言目标改成 VLM 能判断的 event 或 state 查询。{"type":"event|state","query":"..."}；无法视觉判断则 query="none"。如果把 state 误写成 event，会凭空制造触发点；如果把 event 写成 state，会导致触发不及时。
5. 注册触发/定时分类结果、VLM query、触发器能力、已有事件库。条件任务先查事件库，能 select 就选；不能选再根据触发器能力 create 脚本；定时任务转 APScheduler trigger。触发器订阅 id / 定时器 id / VLM 定时配置。如果常识判断无法生成合理脚本，要告知不能输出，不能硬造。
6. 策略生成目标、触发条件、执行内容、Context 过滤器。一期在触发时推理；二期要求创建后提前生成策略，触发时直接按策略生成 advice。可执行策略、advice 模板、任务卡字段。策略不提前生成，触发时会多一次模型推理，链路变慢。
7. 任务卡/HMI目标、条件字段、执行字段、倒计时/状态。任务中心显示任务岛、状态栏、小卡；二期要求显示具体条件和执行内容。用户可见任务卡。只显示简略 widget，用户不知道“什么时候触发、会做什么”。
8. 创建结果回传创建成功/失败。成功/失败都要给 Director 承接；二期要求成功也提醒。成功话术：“将在{具体条件}满足时，做{具体执行项}”。失败话术：“这个条件暂时不支持”。成功静默会让用户以为系统没创建；失败不说会导致任务丢失感。

A.12 任务分类器规则：不是按关键词，而是按“执行机制”分类

判断顺序规则例子为什么这样分
1. 是否长程/GUI只要涉及点单、缴费、订座、外部生态系统能力，归 type=2。帮我点杯咖啡到公司；帮我订座。这类任务有外部应用状态和用户多轮参与，需要生态闭环 advisor。
2. 是否明确时间“X 分钟后”是定时单次；“每隔 X 时间”是定时循环；定时优先级高于条件。5 分钟后关空调；每 10 分钟提醒我喝水。时间触发有明确调度器，应该注册定时器而不是靠 review 猜。
3. 是否包含条件表达出现“如果/当/一旦/检测到…就…”归条件任务；是否循环看“每次”等表达。如果我低头看手机就提醒我；每次副驾上车都打招呼。条件任务需要等未来事件发生，必须注册触发器或 VLM 检测。
4. 其他持续过程非结构化、过程型、陪伴型、持续观察型任务归长时 type=1。照顾二排宝宝；介绍沿途风景；比赛开始后播报赛况。这类任务不是单次触发，而是要根据状态和上下文持续推进。
特殊规则：“比赛开始后每 5 分钟播报”文档归长时任务 type=1，因为本质是过程型陪伴，而不是纯调度任务。这个点很容易和定时循环混淆，实际分类要看业务本质，不只看有没有“每 5 分钟”。

A.13 VLM 改写逻辑：event 和 state 绝不能混

VLM 不是让模型“理解用户完整任务”，而是把目标改写成视觉能判断的最小命题。文档明确分为 event 和 state：event 要求 VLM 输出 true/false，state 只描述当前状态，不编造触发事件。

类型什么时候用改写原则示例
event有明确触发动作，或用户表达“如果/当/一旦/检测到…就…”。保留触发语义，输出“语义状态 + 视觉依据”，要求可 true/false 判断。“如果我低头看手机就提醒我” → {"type":"event","query":"用户低头看手机"}
event识别到某个动作/姿态后做事。可以用 1-2 个可观测特征有限泛化。“识别到我比较累的时候给我放首歌” → 用户处于疲劳状态（如打哈欠或眼睛闭合等明显疲劳表现）。
state没有明确触发瞬间，是长期/持续关注目标。不要编造触发事件，不扩展应该做什么，只保留对象 + 状态存在性/表现。“帮我照顾一下二排的宝宝” → 二排存在宝宝（画面中出现婴儿或儿童）。
none无法通过视觉判断。输出 query=none，不能硬造视觉依据。“提醒我股票涨了就告诉我”不是视觉任务。
典型错误：把“照顾二排宝宝”改成“宝宝哭闹时触发”，这就是编造触发事件。文档要求 state 型任务只描述“二排存在宝宝/宝宝当前状态”，具体怎么照顾要由后续 advisor review 和 Director 决策，不在 VLM 改写阶段脑补。

A.14 条件任务注册：先选已有事件，再创建脚本，不能硬造

  1. 输入事件库和触发器能力描述：模型会拿到当前可选择的事件库，以及触发器能支持哪些信号/状态。
  2. 优先 select：如果事件库里已有相关事件，优先选择已有事件，避免重复创建脚本。
  3. 再 create：如果没有现成事件，再根据触发器能力描述生成动态脚本。
  4. 不能生成就失败：当常识判断无法生成合理脚本时，必须告知不能输出，不能为了创建成功而伪造条件。
  5. callback 触发：条件满足后，触发器 callback 给动态 advisor，advisor 生成 advice 给 Director。
例子：“车速高于 80 时提醒我慢点开”如果触发器已有车速阈值事件，就 select；如果没有但支持端信号脚本，就 create 脚本；如果用户说“世界毁灭时提醒我”，常识上无法注册合理事件，应失败并让 Director 说明不支持。

A.15 定时任务逻辑：云端、端侧、循环、合并、上限

逻辑点文档规则产品含义
定时器类型支持 interval、cron、once；无法解析输出 none。“5 分钟后”“每天早上”“每隔 10 分钟”底层不是同一种 trigger。
VLM 定时定时获取图片请求 VLM，VLM 判断事件是否触发，触发后再推理产生 advice。视觉状态不能一直算，靠周期检查控制成本。
端侧定时器二期新增端侧定时器，定时业务逻辑端侧闭环，不完全依赖云端。弱网/断网也能 AI PUSH，适合“5 分钟后提醒”。
倒计时展示任务小卡显示相对时间任务倒计时。用户能看到任务在计时，降低“是不是没记住”的不确定感。
超高频拒绝每隔 5 秒提醒这类超高频循环拒绝创建；最小循环间隔 1 分钟。保护用户注意力和系统资源。
分钟级合并多个提醒任务自动合并到分钟级；不支持 9:01:05 这种秒级播报。Director 要明确预期，不承诺秒级精度。
最大执行次数循环任务最大执行 30 次，超过后提醒用户需要重新创建。防止无限循环任务长期污染任务中心。

A.16 执行阶段：触发后不是直接说话，而是 advice → Director

任务类型触发入口执行阶段内部逻辑输出到哪里
长时任务review/事件/Context 变化。advisor 根据目标状态、历史 QA、当前 Context 判断是否推进；二期希望由任务编排脚本推进。advice 给 Director；任务卡更新。
持续/GUI 任务GUI 状态变化、用户长时间无响应、生态 SOP 变化。不主动推进业务本身，而是在合适时机轻量提醒用户继续参与 GUI 任务。advice 给 Director；生态状态进任务中心/Docker。
条件任务触发器 callback。一期：查询目标 + 推理生成 advice；二期：根据提前生成的策略直接生成 advice。advice 给 Director。
普通定时定时器触发。直接请求模型给出 advice，可能产生动态 advisor 内容。advice 给 Director / AI PUSH。
VLM 定时定时器触发后取图。先让 VLM 输出 true|reason / false|reason；true 才继续推理 advice。advice 给 Director。
重点：触发不是直接播报，触发只是把“该检查了/条件发生了”送进 advisor。advisor 产出 advice，Director 再做最终用户交互。这是避免后台任务越权说话的关键。

A.17 关闭和清理：任务不能只会创建，不会退出

关闭场景清理动作为什么重要
条件任务达到 max_execution 上限注销触发器；主动关闭 advisor，reason=执行结束。不清理会导致同一条件继续触发。
定时任务达到 max_execution 上限注销定时器；主动关闭 advisor，reason=执行结束。不清理会导致定时器长期空转。
用户取消任务删除 goal/任务卡，注销对应触发器或定时器。只删 HMI 卡不删订阅，会出现“幽灵触发”。
目标达成advisor review 后更新/删除目标；必要时静默删除。情感 advisor 文档明确：目标已完成时仅删除 goal，不输出无执行意义建议。
对象离车/情境结束删除关联对象的子目标。比如乘员离车后继续关心该乘员，会显得系统失真。

A.18 条件任务创建交互：二期要讲清楚“条件”和“动作”

旧流程问题

用户表达条件任务 → 豆包给肯定回复 → 后台异步创建 5-10s → 成功只显示简略 widget 且静默 → 用户不知道任务到底创建没、什么条件触发、会执行什么。

新流程要求

用户表达条件任务 → 豆包中性回应“好的，让我来帮你创建这个任务吧” → 后台创建 → 卡片展示条件字段和执行内容 → 成功/失败都提醒。
节点话术/展示要求产品意义
创建中中性回应，不提前承诺一定成功。避免“收到，下次一定执行”但实际创建失败。
创建成功“刚才的任务创建成功啦，将会在{具体条件}满足时，做{具体执行项}”。让用户知道触发条件和动作，建立可控感。
创建失败说明当前条件暂不支持。失败可感知，避免用户以为系统记住了。
卡片字段展示条件字段、执行内容字段、倒计时/状态。任务中心不是装饰，是任务可解释性界面。
多轮澄清任务不符合预期时支持用户继续修改。把任务创建从一次性黑盒变成可编辑对象。
把 advisor 加进来后的完整系统观：Director 是同步对话和工具执行的中央调度器；advisor 是异步目标和主动服务的后台专家群；Context 是共同现场；触发器/定时器/VLM 是未来事件入口；goal/task card 是长期状态载体；HMI/TTS 是用户感知出口。优化时要沿着“目标创建 → 分类 → 订阅/定时/VLM → advisor review → advice → Director → HMI/工具 → 反馈回流”整条链路查。

1当前架构范式是什么：为什么是"漏斗 + ReAct + 目标驱动"

架构 = 一种"怎么把一句话变成行动"的组织方式。这套系统同时叠了三种范式，各管一段。

范式一：混合漏斗（系统的骨架）PRD原文

"提出'句法RAG+情景RAG+简单/复杂Planner+agents'的混合方案。句法RAG作为前置模块，可配置，可平台化；引入 planner 模型作为系统'大脑'，配合情景RAG的知识注入。"

三条设计原则（这是理解整套系统"为什么这么搭"的总纲）：

"(1) 层层过滤，能通过简单方法搞定的不要漏到后面复杂模型去处理；(2) 在不同阶段选择合适的节点注入知识，注入方式从简单（固定）到复杂（泛化）；(3) 通过用户数据回流逐步把后面复杂任务的处理推到前面知识注入环节。"
产品视角翻译：漏斗 = "便宜的活儿别让贵的模型干"。越往后越聪明也越慢越贵，所以拼命把流量在前面拦掉。Planner 在漏斗最底，既是兜底，又是云端 teacher——它处理的难 case 会被回流成数据，反过来训练前面的层，让漏斗"自己变聪明"。

范式二：ReAct + Function Call（Planner 本体的工作方式）PRD原文LIVE

"基于模型 react 特性，部分 case 呈现多步任务特征"；"对话推理引擎收到工具返回后，结合 context 进行二次推理，决策是否需要生成交互内容或任务序列"。

ReAct = Reason + Act 的循环：推理→调工具→看工具结果→再推理。Planner 用 Function Call 能力把"要调哪些 agent"输出成一个工具调用列表。为什么选 FC 模式？因为纯 SP（只给提示词不训）连强模型都压不住：

"在仅采用 SP 的情况下，强如 deepseek agent 模型存在：不使用 agent 模式调用（严重）、指令跟随欠缺、上文捕捉薄弱、任务拆解稳定性不足（每次拆解结果均不相同）、回复总结幻觉。"

范式三：目标驱动 + advisor 协作（正在叠加的新范式）PRD原文LANDING

旧范式的根本病："我讲一句，它应一句，需求被处理成一锤子买卖，导致需求不闭环"。新范式把"整体目标"显式持久化成一份 goal list，由目标驱动系统持续运转，而不是每轮靠 user query 临时触发。

"针对以上问题，提出目标和动态 advisor 的驱动方式"；做法是"对话推理引擎和 advisors 共同维护一套 goal list"。
一句话原理：所有事情都有目标，只是有些 Planner 当场就闭环了（立刻能做完、不用持续观察的）；做不完、要持续盯的，才升级成 goal + 动态 advisor。这就是从"Planner 一人包揽单线程"到"团队协作多线程"的转折点。

3Director 本体深拆：它收什么、想几次、吐什么

Director = Planner = 对话推理引擎，三个名字一个东西。定位："全链路中的对话中枢和推理中枢"。

3.1 输入：三种形态，同时以数组传入PRD原文

Director 的触发输入有且仅有三种形态（外加 context 作为"参考背景"，不算触发）：

形态含义示例
user query用户主动说的话{"query":"我好热啊","speaker_name":"晓伟","speaker_position":"主驾"}
advisor inputadvisor 给的建议（主动服务/目标推进）{"advisor":"晓伟上车了，车内温度高，可以调空调快速降温"}
tool feedback工具/agent 执行返回，触发二次推理{"tool_feedback":"浙江共有22家5A景区…","tool_name":"web_search"}

参考背景（context，非触发）：人设背景 / 车辆基础信息 / 工具描述 / Context（情景、端状态、用户记忆、人人对话、事件日志、屏幕信息）/ 功能范围 / 场景 knowhow。

3.2 输出：三个字段PRD原文

{
  "talk_or_not": "yes",                  // 这一轮要不要开口（核心：可以不说话）
  "talk_content": "好啊，我先帮你找个招商银行",  // 话术，可带 [开心地说] 等情绪/角色标签
  "action_list": [                       // 动作序列（可并行多个）
    {"action_id":"1", "action_content":"附近的招商银行", "tool_name":"search_poi_qa"}
  ]
}

3.3 怎么工作：一次调用其实是两次 LLM 推理（ReAct 闭环）

① 第一次推理

输入 = user query + advisor input（数组同传）+ context
输出 = {talk_or_not, talk_content, action_list}
→ talk_content 去 TTS，action_list 去调工具

② 二次推理

工具执行返回 tool_feedback →"结合 context 进行二次推理，决策是否需要生成交互内容或任务序列"→ 总结话术 / 继续下一步 / 结束
这就是 ReAct 闭环的产品含义：Planner 不是"一次想完"，而是"想一点→做一点→看结果→再想"。所以同一个 query 可能触发多轮 LLM 调用。Schema 你 CLAUDE.md 说的"同一请求内多轮调用复用前序 KV、只算增量"，对应的就是这个二次/多次推理——这是缓存优化最值钱的地方。

3.4 职责清单（带优先级）PRD原文

  • 【P0】目标增删：基于 user query 推理是否需要增删目标，需要则发起并完成。
  • 【P0】动态 advisor 增删：基于 user query 推理是否需要增删动态 advisor。
  • 【P1】合理性仲裁（增删时执行），三类校验：
    • 感知/触发条件是否合理：能否被感知、触发可能性、频率合理性（"每200ms播报NBA新闻不合理"）
    • 动作是否合理：价值观、可执行性、动作冲突（"同条件下又开又关空调"）、条件与动作是否矛盾
反直觉但极其关键：Director 明确"不更新"目标的"细节和进展"。
"为避免更新冲突，任务细节和进展统一由对应的 advisor（goal reviewer）更新，planner 不会更新任务的细节和进展。planner 的推理工作由 advisor 来触发。"
产品含义：写权限被故意收窄——Director 只管"建/删目标"和"快速响应"，"盯进度/改进展"交给 advisor。这是新架构区别于旧架构的核心约束，也是"减轻 Planner 过载"的具体落地。

3.5.6 从耗时角度看：这份 SP 为什么会慢

工具描述长26 个工具，每个工具都有 describe / param / notice，模型每轮都要在大工具空间里检索匹配。
示例多且长few-shot 覆盖丰富，但会增加上下文长度；好处是能力稳定，坏处是首 token 和推理成本上升。
领域知识重车辆设备知识、工具 tips、聊天风格、基础问答、世界信息、视觉、记忆都在 prompt 里，容易造成噪音。
输出约束复杂不仅要选工具，还要决定说不说、说什么、什么 emoji、情绪是否变化。
状态变量多timestamp、emotion、goal_list、env_info、memory、vision 都可能影响结果，模型需要做多维判断。
因此前置筛选很关键这份 SP 进一步证明：按 query 裁工具、裁示例、裁 knowhow、裁 context，不是锦上添花，而是性能和稳定性的核心。

4.2 静态 advisor vs 动态 advisor（永久监工 vs 临时监工）PRD原文

静态 advisor（永久合同监工）

固定 4 个：舒适 / 出行 / 情感 / 内容，一对一绑定 static 目标。三职责：①目标更新者（更新细节进展）②主动服务（给 Director 建议）③goal reviewer（盯达成度）。生命周期：永久。

动态 advisor（临时合同监工）

由 Director 据 query/建议创建，一目标一 advisor，基于 prompt 模板工程自动注册。职责同静态。生命周期：动态目标创建→达成/取消/变更结束。
"静态 advisor 是永久合同监工；动态 advisor 由对话推理引擎根据 user query 或 advisor 建议创建，是临时合同监工，有明确的生命周期。"
静态 advisor负责的长期目标典型触发输出重点
舒适车内每个人的舒适：温湿度、空气、光线、声音、空间、乘坐等。事件触发 + 目标 review；舒适子目标存在时可周期复查。即时舒适建议、舒适子目标、调节方向。
出行出行目的、目的地、路线、沿途充电/吃饭/休息安排。行前/行中事件；目的地不变时定时 review；兜风/充电场景复查。目的预测、路线推荐、沿途 POI/补能/休息建议。
情感用户被关心、理解、陪伴；安全/疲劳/情绪类情感子目标。事件触发；Waiting 状态且有未完成子目标时 review。即时情感建议、情感子目标、goal_operations。
内容车上时光是否需要内容介入，以及适合什么内容负荷。默认静默；Trigger 唤醒；长时间静默 review；其他 advisor 协同信号。L0-L3 介入等级、内容形态/标签/风格/文本。
关键判断逻辑（新版）：静态 advisor 管四个永久目标域；动态 advisor 不只管“长时/持续”，还承接 长时任务、持续/GUI 任务、条件任务、定时任务 四类非即时任务。Director 仍是最终前台仲裁者，advisor 负责后台观察、review、策略和 advice。

  • 三维过滤：按场景（驻车不加载行车记忆）/ 按人物（过滤非共同记忆避免隐私泄露）/ 按任务（点外卖只检索外卖相关知识偏好）。
对应你 Schema 里的"前置筛选模块"：PRD原文 Director PRD 也独立提到——"对话推理引擎中有较多内容（工具描述、50+ 示例注入、knowhow）在处理具体问题时是冗余的，冗余注入会导致性能和推理质量下降，解决方案是增加前置筛选模块，根据 query 筛选相关内容再加载"。Schema 你 CLAUDE.md 说的 KV Cache / 远近区分区 是更细的 SP 侧优化，在另一份 Sp 优化专项.html，不在这五份里。

6任务与交互管理：为什么车上是"单通路"，怎么治理冲突PRD原文

这层决定"用户体感顺不顺"。底层物理约束只有一句：车里的语音交互是单线程、单通路管道，同一时间只能回应一次。

"豆包在车里和用户之间的语音交互是单线程的，是一条单通路管道，同一时间只能对一次请求做出回应。对于这种交互特性，只能在输入阶段进行治理。"

输入端治理：四种输入 + 调度策略

输入优先级策略
用户主动说的话（即时）P0 最高打断式
定时/条件任务触发（非用户）P1 统一排队式
Tool feedback—分流（车控/aigc/长程各异）
advisor 建议—排队，同一 advisor 只保留最新一条

核心：打断 vs 合并 vs 排队（按"是否已出话术"决定）

新 query 来时·首 token 还没出

→ 打断当前推理，把新旧 query 合并后重新推理
话术体现："车里是有点热呢，我来调一下，顺便帮你找今天的热点新闻"

新 query 来时·已经在播报

P0用户：直接打断，按新 query 处理，被打断的不恢复
P1非用户：排队，等上一条 tts 播完再播（可预推理+缓存结果）
合并的三条特殊规则（很容易踩坑）： ① 长程任务（订座/停车/点咖啡/点外卖）不参与合并——它们有自己的交互界面（工程逻辑）； ② aigc（生图/ailight）不走 Planner、无语音播报，完成后走通知卡系统通知； ③ 长程/aigc 的状态靠工具自带 exe_sts 字段随 tool_feedback 露出，任务中心解析后更新显示。

失效兜底（排查超时类 badcase 必看）

  • 端侧上报超时：端侧白名单召走→采纳端侧不等云；端侧有结果非白名单→超时采纳端侧；端侧无结果→超时播报兜底。
  • 云端工具超时（ReAct）：①工程判断超时，tool_feedback 反馈超时 → ②告知用户后重试 → ③重试失败告知用户后结束任务。
  • 被端侧召走的 query：云端并行在跑但丢弃云端结果，被合并的任务恢复排队；云端历史保留端侧处理信息、不保留被打断的云端推理。

7任务类型 → 架构怎么承接（即时任务 vs 非即时任务）

加入 advisor 文档后，这里要更新：不是只有“长时/持续”才进动态 advisor。动态 advisor 技术设计明确承接 长时任务 type=1、持续/GUI 任务 type=2、条件任务 type=3、定时任务 type=4。即时任务仍由 Director 当场闭环，非即时任务进入动态 advisor/任务中心/触发订阅体系。

任务类型是否非即时是否进动态 advisor怎么承接（原理）
简单即时 / 单步复杂否❌Director 当场闭环，不进任务中心；工具成功且可感知时可静默。
多步即时通常否❌Director 在 ReAct 里串联 tool_feedback 完成依赖；如果变成 GUI/生态持续流程，则转 type=2。
长时任务 type=1是✅过程型/陪伴型任务，靠 review、Context、可能的 VLM 状态持续推进。
持续/GUI 任务 type=2是✅面向生态/GUi Agent 的持续推进，用户长时间无响应时提醒继续，任务卡片显示状态。
条件任务 type=3是✅注册触发器；条件 callback 后动态 advisor 根据策略生成 advice 给 Director。
定时任务 type=4是✅注册云端/端侧定时器；循环定时带 cycle；VLM 定时先捞图判断再触发。
一句话记住整套架构的"分配逻辑"： 能当场做完的 → Director 闭环（不留痕）； 要跨时间、跨条件、跨 GUI 状态继续推进的 → 进入动态 advisor/任务中心； 要等条件的 → 注册触发器； 要等时间的 → 注册定时器； 要靠视觉判断的 → 走 VLM 改写 + 定时/事件 review。 "恢复"不靠 Director 记忆，靠 advisor 读 goal/任务状态重新触发任务——这就是"目标驱动"和"对话管理"的耦合点，也是高闭环率的来源。

9耗时模型与时延优化：一轮请求到底慢在哪里

这一章专门回答“耗时”。不要只看总耗时，要把它拆成：输入链路、分流链路、Planner 思考、工具执行、二次推理、TTS/HMI、交互治理。真正优化时延，先要知道哪一段在关键路径上。

9.1 PRD 给出的硬指标：先建立时延预算表 PRD原文

链路段目标耗时是否通常在首 token 前产品含义优化方向
ASR → NLU 完成200 + 300ms是用户说完到系统拿到可理解输入。流式 ASR、提前判停、端侧轻量 NLU。
FC.SFT 解析（简单）300ms是简单指令快速转成函数调用，不进深度慢思考。简单任务识别、端侧/小模型解析、工具 schema 收窄。
Planner 思考500ms是Director/Planner 做意图、任务拆解、工具选择。前置筛选、减少示例注入、Context 过滤、模型蒸馏。
Planner 回答（慢思考）500ms可能是复杂任务需要更深推理、组织话术或总结。COT 中间结论截取、边想边搜、响应话术按需生成。
端上执行（车控）150ms通常否动作已确定后，端侧执行控制。把明确车控留在端侧，减少云端往返。
TTS 生成200ms否，但影响用户感知用户听到声音的时间和流畅度。首句短话术、TTS 流式、减少不必要铺垫。
总目标P50 首 token ≤500ms；P90 ≤800ms核心指标用户是否觉得“快”。所有首 token 前的串行段都要压缩。
关键理解：首 token 不是“整个任务完成时间”，而是用户第一次感知到系统响应的时间。复杂任务可以总耗时长，但必须让用户尽早知道“我听懂了，正在处理”。所以架构里要区分 首 token 时延、动作下发时延、任务完成时延、总结播报时延。

9.2 三种典型链路的耗时账本

链路类型串行关键路径哪些可并行耗时风险体验策略
简单车控快链路
例：打开空调、调座椅		 ASR/NLU → 简单 FC 解析 → 端上执行 → TTS/HMI 端侧执行和简短确认话术可以接近并行；部分明确指令可跳过复杂 Planner。 如果误进云端 Planner，会把 150ms 级动作拖成 500ms+。 能端侧解决就端侧；不必强制生成铺垫话术。
复杂多步 ReAct 链路
例：先查天气再规划路线		 ASR/NLU → Planner 第一次思考 → tool call → tool_feedback → Planner 二次推理 → 下游执行/总结 无依赖的 action_list 可并行；有依赖的步骤必须串行等待 tool_feedback。 每多一次 ReAct 循环，就多一次“工具耗时 + Planner 二次推理”。 首 token 先给轻话术；复杂结果出来后再总结，不要让用户空等。
长时/持续目标链路
例：帮我盯一下宝宝		 入口识别 → Director 建目标/建 advisor → 注册订阅 → 未来事件触发 → advisor 推进 → 写回 HMI 建目标后，advisor 可以后台持续；多个 advisor 可并行 review goal list。 总时长可能跨分钟/小时，不能用一次请求的首 token 指标衡量全部。 先确认“已开始跟踪”，后续用目标卡片/提醒表达进展。

9.3 首 token 为什么会慢：真正卡住的是这些串行段

① 输入未完成ASR/VAD 没判停，Planner 不能稳定决策。优化点：流式识别、提前判停，但误判会导致打断体验差。
② 前置分流犹豫端侧白名单、拒识、仲裁模型没快速给出“快链路/慢链路”结论。优化点：把高频明确指令前移。
③ Prompt 太胖工具描述、50+ 示例、knowhow、Context 全塞进去，模型首 token 变慢，推理也更容易被噪音干扰。
④ Context 现取现拼如果每轮都临时查记忆、查端状态、查情景，再拼进 prompt，会把检索耗时放到关键路径。
⑤ 强制铺垫话术任务探索文档里也提到铺垫话术影响时延；简单查询/端状态场景，强制“我来帮你...”反而慢且重复。
⑥ 多轮 ReAct 依赖第二次推理必须等 tool_feedback，串行依赖越多，总完成时间越长。

9.4 ReAct 多轮耗时公式：为什么复杂任务天然慢

总完成耗时 ≈ 输入理解
        + Planner 第一次思考
        + max(可并行工具耗时)
        + Σ(串行依赖工具耗时 + 二次推理耗时)
        + HMI/TTS 展示耗时

首 token 耗时 ≈ 输入理解
        + 前置分流/Context 准备
        + 生成第一段可播报内容的模型耗时

所以复杂任务有两种优化思路：一种是压缩每段耗时，另一种是改变“用户等待方式”。如果必须慢，就先给用户一个可信的轻量响应，比如“我先看一下路线和沿途天气”，然后后台继续跑工具和二次推理。

9.5 打断/合并/排队对耗时的影响 PRD原文

时刻策略耗时影响为什么这样设计
新 query 来时，上一轮 Planner 还没出话术打断当前推理，把新旧 query 合并后重推理上一轮已消耗的推理时间沉没，但避免播出过时回答。用户还没听到结果，此时重算比播错更划算。
新 query 来时，上一轮正在播报P0 用户 query 直接打断；P1 非用户事件排队P0 降低用户等待；P1 避免主动服务打断用户。车内语音是单通路，用户主动输入优先级最高。
advisor 建议/定时条件任务触发排队，同一 advisor 保留最新一条牺牲实时性，换取低打扰和低噪音。主动建议不能频繁抢话，否则体验崩。
长程任务/aigc不参与合并避免重算成本和状态错乱。这类任务有自己的界面/长流程，强行合并风险高。

9.6 并行不是万能：哪些能并行，哪些不能并行

对象能否并行原因时延优化含义
多个无依赖 action/tool可以action_list 中无前后依赖时，下游 agent 可并发执行。总耗时取 max(tool)，不是 sum(tool)。
有依赖的多步任务不可以下一步需要上一工具的 tool_feedback。要优化拆解，减少不必要串行。
多个 advisor review goal list可以目标驱动体系就是为了多线程协作。后台可并行，但前台汇报要合并，不能逐条打扰。
Context 预处理部分可以压缩、打标、提前过滤可在请求前或后台做。把耗时从在线关键路径挪到离线/准实时。
TTS 与执行部分可以短确认话术可以和端侧动作接近并行。用户先听到响应，动作随后完成。

9.7 时延优化抓手：按收益和风险排序

优先级抓手主要降低哪类耗时风险适用范围
P0快慢分流：明确车控/简单查询不进复杂 Planner首 token、动作下发分错会漏掉复杂需求高频明确指令
P0前置筛选：按 query 裁工具描述、示例、knowhowPlanner 首 token、推理质量裁掉必要知识会调错工具所有云端 Planner 请求
P0Context 三维过滤：场景/人物/任务按需注入token、推理噪音过滤过猛会信息缺失记忆、端状态、情景复杂场景
P1响应话术按需生成首 token、TTS 总时长太少会让用户觉得没反馈端状态查询、简单控制、短链路任务
P1工具并行与依赖识别任务完成耗时依赖判断错会导致执行顺序错多 agent、多工具任务
P1KV/Prompt Cache多轮 ReAct 推理这批 PRD 未细讲，需看 SP 优化专项固定 SP 前缀、多次推理复用
P2COT 流式截取中间结论动作执行等待实验中，格式不稳会误截取慢思考实验链路
P2边想边搜首 token、RAG 初始注入量实验中，检索质量和时序复杂需要状态/知识补充的复杂推理

9.8 你排查“慢”的时候，应该问这 8 个问题

  1. 这条请求本该走快链路，还是本来就该走慢思考？
  2. 首 token 慢，还是动作完成慢，还是总结播报慢？不要混为一谈。
  3. 慢之前有没有等 ASR/VAD 判停？有没有被新 query 打断重算？
  4. 前置筛选有没有把大量无关工具、示例、knowhow 塞进 prompt？
  5. Context 是否在线现查现拼？有没有可提前预处理的部分？
  6. Planner 是一次推理还是 ReAct 多次推理？每次 tool_feedback 等了多久？
  7. action_list 里哪些能并行，哪些被错误串行了？
  8. 是否强制生成了不必要的铺垫/总结话术，导致用户感知变慢？
最终结论：耗时优化不是单点“模型变快”，而是把请求分成三类处理：快链路直接闭环、复杂链路尽早出首 token 并后台 ReAct、长时目标用 advisor/订阅异步推进。这才是当前架构里“耗时”和“体验”同时优化的底层逻辑。
