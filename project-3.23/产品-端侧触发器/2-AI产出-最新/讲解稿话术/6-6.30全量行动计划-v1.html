一、关键术语表（中英对照 + 含义解释）

术语（英文）中文翻译含义解释（在本项目中的具体意思）
SP
(System Prompt)		系统提示词给VLM（视觉语言模型）的指令文本。告诉VLM"你这次推理要看什么"。
例如：AlwaysOn SP里写着"描述画面中人物的性别、位置、动作"，如果动态追加一个SP1，就变成"…并观察宝宝身上的毯子状态"。
在本项目中：SP = VLM每5秒推理时读取的全部指令文本，触发器负责拼接、管理、删除。
AlwaysOn SP常驻提示词VLM端模始终运行的基础SP。即使没有任何动态任务，VLM也会按照AlwaysOn SP的指令每5秒做一次推理。
类比：像手机的"息屏显示"——始终在运行的基础功能。动态任务是在它上面临时加东西。
KV槽位
(Key-Value Slot)		键值返回位每个被拼入SP的动态任务，都在VLM的输出中预留一个"返回位"。
例如：SP1的answer放在key="blanket_status"，SP2的answer放在key="dog_collar"。这样触发器可以精确取到每个任务的结果，不会混在一起。
丁彬原话(33:33)："要加个对应槽位，专门从KV值里头去拿，SP1的KV值，SP2的KV值"
VLM
(Vision Language Model)		视觉语言模型
（端侧）		部署在车机AIBox上的端侧视觉AI模型。每5秒拍一帧舱内画面→按SP指令推理→输出自然语言描述。
核心约束：①只输出客观描述（"毯子在地上"），不做条件判断（不会说"毯子掉了"）②频率固定5秒 ③舱内图片不上云（合规）④可能不带标签（丁彬：动作太多标签输出太长）
VQA
(Visual Question Answering)		视觉问答
（云端）		部署在云端的视觉AI。舱外图片可以上云→VQA模型推理→返回结果。
内部有多个路由：问车型→懂车帝数据库 / 问建筑→LBS地理服务 / 泛化问题→方舟VLM云端模型。
与VLM的区别：VQA在云端、处理舱外；VLM在端侧AIBox、处理舱内。
Caption描述性文本VLM输出的自然语言描述，是一段话而不是固定标签。
例如"后排有一名女性，手持手机，穿白色衬衫"。
与标签的区别：标签是有限集（男/女），Caption是开放集（任意描述）。标签可以直接做条件匹配（==判断），Caption不能。
标签 (Label/Tag)枚举标签VLM输出中的固定字段，值是有限集合（枚举）。
例如：gender=男/女，position=主驾/副驾/后排左/后排右。
关键性质：因为值是有限的、确定的，触发器可以直接用运算符（==、!=、变更为）做条件判断，不需要语义理解。
提槽工具
(Slot Extraction Tool)		方向/位置提取工具原来在VQA工具内部的一个子模型，负责判断用户需求的方向（direction）：是舱内还是舱外，具体什么位置。
6.30会议决定：西钺把它从VQA里拆出来，做成独立公共服务，触发器可以直接调用。
西钺原话(55:02)："人人尽可用，触发器直接调，不用绕VQA"
Direction方向（舱内/舱外）提槽工具的核心输出。判定用户需求涉及的视觉观察方向：
· 舱内(in-cabin)：需要看车内画面→走端侧VLM（图片不上云）
· 舱外(out-cabin)：需要看车外画面→走云端VQA（图片可上云）
为什么重要：舱内图片因合规不能上云，所以必须分流到不同链路处理。
穿越检测
(Threshold Crossing)		阈值穿越检测触发器核心原子15-13的功能。持续监听信号值变化，当某个信号从"不满足条件"变为"满足条件"（False→True穿越）时，组装事件并push给Advisor。
例如：持续监听"毯子在地上"信号，当VLM从"毯子盖着"变为"毯子在地上"→穿越→触发。
动态Advisor
(Dynamic Advisor)		动态顾问/监工用户通过语音发出需求后，Director临时创建的一个"小程序"，负责监控这个特定任务是否完成。
例如用户说"毯子掉了告诉我"→Director创建一个动态Advisor→它订阅触发器的事件→拿到"毯子确实掉了"后告知用户→任务完成→自动销毁。
与静态Advisor区别：静态是预设的4个永久监工（舒适/出行/情感/内容），动态是临时的、用完即毁。
静态Advisor
(Static Advisor)		静态顾问/监工系统预设的4个永久运行的监工模块，不需要用户发起，系统启动就存在：
① 舒适Advisor ② 出行Advisor ③ 情感Advisor ④ 内容Advisor
它们持续订阅触发器事件（如"有人擤鼻涕"→舒适Advisor→建议调温度）。
Context上下文/情景信息存储车辆当前状态、乘客信息、环境信息的数据池。
关键定义（6.30西钺确认）：Context给Advisor和Director的方式是"固定拼接"而不是"订阅"——推理时自动带上，不是Advisor主动去查。
只存condition/状态，不存event/事件。
SFT
(Supervised Fine-Tuning)		监督微调用标注数据对AI模型进行专项训练的技术。
在本项目中：如果需要VLM直出某个新标签（如"情绪=开心"），需要做SFT训练。好处是输出准确、稳定；坏处是训练周期长、且会限制模型对其他内容的泛化能力。
丁彬原话(23:21)："如果很重要的，可能就是要去加标签，去做SFT，导给方舟"
后处理模型
(Post-processing Model)		后置标签转换模型在VLM输出Caption之后，额外挂一个轻量模型，把自然语言描述转换成结构化标签。
例如：VLM输出"这个人看起来很高兴"→后处理模型→emotion=happy。
嘉锋提议(20:32)："外挂一个模型去解析，用light model几百毫秒出标签"
丁彬确认：放在云端做比较好。
感知可视化
(P-HUD / Perception Visualization)		感知可视化屏幕赛力斯车上的一块屏幕，实时展示AI"看到了什么"的动画效果。
例如：识别到有小狗→屏幕上出现小狗动画；识别到乘客开心→氛围灯变暖。
问题：P-HUD需要的标签数量远超VLM当前输出能力（徐洋会上列的表有很多gap）。
DT / Director导演/决策大脑系统的核心推理模块，理解用户意图→制定计划→调度工具执行→跟踪反馈。
底层模型是豆包Seed 1.8，部署在云端。SP约30K tokens。
SLS
(Serverless)		无服务器计算丁彬流程图中的"SLS闭环业务"：端侧应用触发VLM推理→oneshot（单次）或周期推理→结果返回给SLS应用。
这是VLM的另一个调用方，与触发器的SP下发是并行的。

二、两场会议的核心结论（为什么要做这些事）

会议①：动态Advisor视觉识别下发链路（14:00-15:01）

参会：陈西钺 / 丁彬 / 张兰若 / 吴仁杰

本次会议解决的核心问题：动态Advisor需要VLM去"看"东西（比如毯子掉没掉），这个链路怎么走？谁拼SP？谁判断？谁清理？
#结论详细内容原话依据
1SP拼接全归触发器Advisor只给"人话需求"（如"毯子掉没掉"），不拼SP。触发器负责把人话转成工程SP、拼入AlwaysOn、管理KV槽位、任务结束后删除还原。
触发器新增能力："更新prompt"能力——原来只做条件判断+事件push，现在要加SP管理。		丁彬(13:42)："这就是要加的"
西钺(30:18)："触发的时候带上SP是自然动作"
2枚举/描述两条路径枚举类（性别/位置/有无小狗）= 标签 = 有限集 → 触发器直接条件匹配（像空调开关一样用等号判断）
描述类（着装/毯子/项圈）= 非标签 = 开放集 → 需要走动态SP链路（拼SP→VLM推理→KV返回→穿越检测）		丁彬(20:22)："性别/位置是标签，动作/着装没有固定标签"
西钺(49:38)："但凡不是枚举类型的，都需要去声明SP"
3舱内/舱外分流舱内图片不能上云（合规）→ 走端侧VLM
舱外图片可以上云 → 走云端VQA
提槽工具（从VQA拆出来的direction判断模型）来做这个分流。		西钺(03:03)："舱内的VQA以及动态Advisor需要看舱内的信息，都会走端侧VLM"
西钺(56:23)："你就只有判断是否需要额外观察，额外观察就调提槽"
4VLM只报客观·不做判断VLM输出"毯子在地上"（客观描述），不输出"毯子掉了"（条件判断）。
条件判断由触发器做（穿越检测：从"毯子盖着"→"毯子在地上" = 状态变更 = 触发）。		西钺(18:43)："我们只说毯子的状态在哪，掉不掉应该是应用方来判断"
西钺(01:00:07)："所有条件判断类任务都交给触发器"
5Advisor创建前先查Context动态Advisor创建任务前，先检查Context里有没有已有信息。有的话不重复下发VLM任务。
例如：用户说"车上有没有女人"→ Context里已经有性别标签=女 → 不需要下发VLM任务，直接用。		西钺(44:32)："必须过滤，你要先去看当前Context里有没有这个信息，有的话就不应该再下发"
6多任务合并同一SP同时观察宝宝+小狗+红绿灯，必须拼在一起放到同一个5秒推理周期。不能分别下发三个独立推理任务（否则周期变长）。西钺(28:04)："如果同时要观察后排宝宝和后排小狗和窗外红绿灯，肯定得拼在一起"
丁彬(28:32)："周期5秒，每次下发窗口5秒，一次得下发完"
7VLM不做时间判断VLM不知道"小狗出现了多久"、"闭眼了多长时间"。所有时间相关的判断（出现/消失/持续时长）都是触发器做的。张兰若(35:36)："所有跟时间相关的，都不在VLM返回信息当中，都需要触发器自己搞定"
8VLM输出会抖动(glitch)VLM可能在连续推理中出现不稳定输出（第1次=有小狗，第2次=没有，第3次=又有了），触发器需要做防抖处理。张兰若(39:39)："但凡出现一次错误，比如狗突然没了，第二次可能有个glitch又出现了，触发器就会有问题"

会议②：VLM视觉模型的标签转化（20:45-21:12）

参会：徐洋 / 张航 / 丁彬 / 张嘉锋 / 张兰若 / 陈西钺 / 王泽民

本次会议解决的核心问题：VLM输出自然语言描述（Caption）+ 少量标签，但下游模块（感知可视化P-HUD / 触发器 / Advisor）需要更多标签。这个gap怎么填？
#结论详细内容原话依据
1Caption ≠ 标签Caption（描述）= 开放集，"这个人很高兴"→ 不能直接做条件匹配
标签 = 有限集，emotion=happy → 可以直接 == 判断
VLM当前主要输出Caption，标签很少（只有天气、路况等几个）		丁彬(17:18)："标签就是一个字——高兴、难过；Caption就是一段描述——这个人很难过"
2不建议VLM直出过多标签视觉产品反馈：让VLM输出过多非高频标签会限制模型泛化能力（能出固定标签了，但其他自然语言描述能力下降）徐洋(01:05)："视觉产品反馈不建议让VLM直接输出过多的非高频标签，会限制模型泛化能力"
3两种解法① SFT训练：非常重要的标签→做监督微调→VLM直出标签。准确但耗时
② 后处理模型：在VLM输出Caption后，云端挂一个轻量模型做语义→标签转换。快但可能不够精准
先梳理需求量，再决定用哪种		丁彬(22:51)："很重要的→加标签做SFT；对准确性要求不高的→加个后模型"
嘉锋(20:32)："外挂一个模型，用light model几百毫秒出标签"
4泽民的任务：梳理触发器标签需求丁彬直接点名：触发器PM需要把动静态Advisor对VLM的标签依赖列出来，导给VLM团队。
包括：需要哪些标签、用在什么场景、当前有没有、缺的怎么补。		丁彬(25:00)："你那边需要监听的一些标签，就是这个事"
丁彬(25:24)："把这块梳理出来，给我们导入"
5P-HUD需求要先砍一轮感知可视化屏幕的标签需求远超VLM能力，但很多是赛力斯内部没对齐的。
徐洋负责找P-HUD产品先砍掉不常用的，剩下的再看走SFT还是后处理。		嘉锋(03:50)："看起来已经超过视觉给兰若的清单了"
徐洋(22:02)："先让P-HUD产品内部确认一轮，砍掉不常用的"
6动态任务不需要标签用户动态创建的任务（"毯子掉了告诉我"）通过SP拼接→VLM推理→Caption返回→Advisor语义判断。不需要VLM出标签，整个链路靠丁彬画的那个流程图闭环。丁彬(14:13)："像睡觉这种就不用再生成新的了，已有枚举值直接去让云端触发器订阅信号"
徐洋(14:55)："完全可以靠拼prompt或直接读KV的方式去判断"
