# 豆包车载 AI Planner Prompt 优化方案

## 概述

本文档针对豆包车载 AI Planner（Director）的六类 badcase 问题，逐一分析根本原因，并提供可直接写入 `prompt.md` 的具体优化内容和验证标准。

---

## 问题 1：GUI Agent 澄清指令被当成闲聊处理

### 根本原因

**状态建模缺失 + 上下文感知不足。**

Planner 缺乏对"当前是否有 GUI Agent 处于等待用户输入状态"的显式追踪。当 GUI Agent 通过 `add_info` 机制返回一个澄清问题（如"请问您用哪种方式登录？"），Planner 没有将此状态记录为"等待补充信息"，导致用户的下一条回复被路由到通用对话处理逻辑，而非回传给 GUI Agent。

根本上是缺少一个**任务状态机**的概念：GUI Agent 的任务存在 `running / waiting_for_user / completed / failed` 四种状态，Planner 的 Prompt 中没有对 `waiting_for_user` 状态的处理规则。

### Prompt 优化动作

**插入位置：** `prompt.md` 中第 10 条工具使用 tips（gui_agent_operation 段落）末尾，新增独立规则块"GUI Agent 任务状态管理"。

**插入内容：**

```
### GUI Agent 任务状态管理

GUI Agent 在执行过程中可能进入"等待用户补充信息"的中间状态，你必须识别这种状态并正确处理。

**状态识别规则：**
当 GUI Agent 的 tool_feedback 中包含以下特征时，判定为"等待用户补充信息"状态：
- 返回内容包含疑问句或选项列表（如"请问您希望用哪种方式登录？"）
- 返回内容明确提示需要用户输入（如"需要您输入验证码"、"请确认订单信息"）
- 返回内容描述任务被暂停或中止，等待指示

**处理规则：**
1. 当 GUI Agent 进入等待状态后，你需要将 GUI Agent 的问题原文转述给用户，不得自行替换或改写问题内容。
2. 将用户的下一条回复，通过 {"type":"add_info","query":"<用户原话>"} 回传给 GUI Agent，不得将其识别为新任务或闲聊。
3. 在用户回复之前，不得主动发起新的 GUI Agent 任务调用。
4. 如果用户的回复与当前等待的问题无关（如突然切换话题），需先通过 {"type":"delete_task","query":"取消当前任务"} 明确终止当前任务，再处理新意图。
```

**配套 Few-shot 示例（补充至 prompt.md 示例库）：**

```
示例：GUI Agent 要求输入验证码
input:
{"tool_feedback":"当前页面需要输入短信验证码，请告知验证码内容。","tool_name":"gui_agent_operation"}
output:
{
  "talk_or_not": "yes",
  "talk_content": "需要你输入短信验证码，你看一下短信",
  "action_list": []
}

input:
{"query":"3872","speaker_name":"婉婷","speaker_position":"主驾驶位"}
output:
{
  "talk_or_not": "no",
  "talk_content": "",
  "action_list": [
    {
      "action_id": "1",
      "params": {"type":"add_info","query":"验证码是3872"},
      "tool_name": "gui_agent_operation"
    }
  ]
}
```

### 验证标准

| 测试场景 | 期望行为 | 通过标准 |
|---|---|---|
| GUI Agent 返回验证码请求，用户输入验证码 | Planner 调用 `add_info` 回传验证码 | 不出现闲聊回复 |
| GUI Agent 返回登录方式选择，用户回复"手机号" | Planner 调用 `add_info`，不新建任务 | type 字段必须为 `add_info` |
| GUI Agent 返回订单确认，用户回复"确认" | Planner 调用 `add_info` 传递确认指令 | 不得触发新任务创建 |
| 用户在等待期间切换话题（如说"播放音乐"） | Planner 先 `delete_task` 再处理新意图 | 两步操作均出现 |

## 问题 2：Planner 澄清内容与小程序页面不符

### 根本原因

此问题包含两个子问题：

**子问题 2a：** Planner 对特定 App（如美团）的实际支持的登录方式不了解，导致主动询问用户时给出了该 App 不支持的选项（如微信登录、扫码登录）。根本原因是**工具知识库（domain knowledge）不准确**。

**子问题 2b：** Planner 主动预判澄清内容（"你想用哪种方式登录？"），而非等待 GUI Agent 执行��基于实际页面状态反馈再决定是否需要用户介入。根本原因是**缺乏"延迟澄清"原则**，Planner 与 GUI Agent 之间的信息流向倒置。

**子问题 2c（原标注"无法解决"的重新评估）：**
播报内容与页面不一致，约 60% 可通过约束 Planner 的播报信息来源来缓解（已在问题 4 处理）；剩余 40% 属于 GUI Agent 自身能力边界问题，Planner Prompt 无法修复，标记为持续优化项。

### Prompt 优化动作

**插入位置：** `prompt.md` 中"工具知识"章节 gui_agent_operation 段落末尾，新增应用知识约束 + 澄清策略原则。

**插入内容：**

```
### 主流小程序登录方式约束（车载场景）

以下为车载场景下各小程序实际支持的登录方式，澄清时只能在支持范围内提供选项：

- 美团：支持手机号+短信验证码、手机号+密码。不支持微信登录、扫码登录。
- 捷停车：支持手机号+短信验证码。
- 美味不用等：支持手机号+短信验证码。
- 星巴克：支持手机号+短信验证码、星巴克账号。

注意：如 GUI Agent 返回的实际页面选项与上述约束不符，以 GUI Agent 返回的实际页面为准。

### 澄清策略：延迟���清原则

优先调用 GUI Agent 执行任务，让 GUI Agent 根据实际页面状态决定是否需要用户介入。
仅在以下情况才在调用前主动澄清：
1. 用户意图模糊，GUI Agent 无法确定执行哪个任务（如未说收货地址）
2. 任务涉及不可逆操作（如支付、下单），需确认关键参数

禁止行为：在 GUI Agent 执行前，主动询问"你想用哪种方式登录"等用户自行无法决策的问题。
```

### 验证标准

| 测试场景 | 期望行为 | 通过标准 |
|---|---|---|
| 用户要求用美团点外卖（未登录状态） | Planner 不主动给出微信/扫码登录选项 | 澄清选项严格来自约束列表或 GUI 返回 |
| GUI Agent 返回美团登录页，只有手机号选项 | Planner 告知用户"需要手机号登录" | 不出现约束以外的登录方式 |

## 问题 3：Planner 受对话历史影响（原方案遗漏）

### 根本原因

**历史依赖污染 + 任务上下文边界不清晰。**

此问题包含三个子问题：

**子问题 3a（历史失败污染）：** 多次失败后，Planner 将历史失败结果作为当前任务的预判依据，直接报失败而不重新尝试。

**子问题 3b（历史地址污染）：** 历史对话中出现的外卖地址被 Planner 沿用到��任务，即使用户未指定。

**子问题 3c（打断后 type 参数传错）：** 任务被打断后，Planner 恢复执行时使用了 `add_info` 替代 `new_task`，将"重新发起"误判为"补充信息"。

### Prompt 优化动作

**插入位置：** `prompt.md` 注意事项章节末尾，新增"历史隔离与任务重启规则"。

**插入内容：**

```
### 历史隔离与任务重启规则

**规则 3a：任务独立执行，不受历史失败影响**

每一次用户发起的任务请求，必须作为全新任务独立执行，不得因为历史对话中出现过类似失败而预判当前任务会失败。
禁止行为：对话历史中有"美团登录失败"记录，用户再次要求点外卖时，直接回复"之前登录失败了"而不尝试执行。
正确行为：无论历史是否有失败记录，都重新调用 {"type":"new_task",...} 发起新任务。

**规则 3b：新任务参数不得默认沿用历史信息**

用户发起新任务时，地址、收货人等关键参数必须来源于：
1. 用户在当前对话轮次中明确说明的信息
2. 用户账号的默认设置（由 GUI Agent 在实际页面中读取）
3. 用户明确表示"和上次一样"

禁止行为：因上一轮外卖地址是"公司"，就自动将新任务地址填写���"公司"。
正确行为：调用 GUI Agent 时不在 query 中预填充历史地址，由 GUI Agent 根据实际页面处理默认地址。

**规则 3c：打断后恢复任务使用 new_task，而非 add_info**

判断规则：
- 原任务已被 delete_task 或超时终止后用户要求重新执行 → 使用 new_task
- GUI Agent 处于等待状态，用户提供所需信息 → 使用 add_info
- 不确定任务是否仍在运行时 → 默认使用 new_task，比错误使用 add_info 更安全
```

**配套 Few-shot 示例：**

```
示例：打断后重新发起任务（应用 new_task）
[上下文：用户之前说"帮我叫滴滴去机场"，后说"等等先不叫了"，任务已 delete_task 终止]

input:
{"query":"好了，帮我叫滴滴去机场","speaker_name":"婉婷","speaker_position":"主驾驶位"}
output:
{
  "talk_or_not": "yes",
  "talk_content": "好，重新帮你叫车",
  "action_list": [
    {
      "action_id": "1",
      "params": {"type":"new_task","query":"叫滴滴去首都机场"},
      "tool_name": "gui_agent_operation"
    }
  ]
}
```

### 验证标准

| 测试场景 | 期望行为 | 通过标准 |
|---|---|---|
| 历史有失败记录，用户重新发起同类任务 | Planner 重新调用 `new_task`，不预判失败 | 必须出现 `new_task` 调用 |
| 历史有外卖地址，用户新任务未指定地址 | Planner 不在 query 中预填历史地址 | query 中不出现历史地址字符串 |
| 用户打断任务后重新要求执行 | Planner 使用 `new_task` 而非 `add_info` | type 字段必须为 `new_task` |

## 问题 4：GUI Agent 传参 Type 类型传错 + 播报过度承诺

### 根本原因

**工具使用规范不清晰 + 语义总结边界缺失。**

**子问题 4a（type 传错）：** Planner 对三种 type 的语义边界理解不够精确，在特定场景下选错 type。

**子问题 4b（播报过度承诺）：** Planner 将 GUI Agent 返回的"任务开始执行"等中间状态总结为"已完成"（如"已帮您点好奶茶"），缺乏对 GUI Agent 返回状态的严格分级。

### Prompt 优化动作

**插入位置：** `prompt.md` ��� gui_agent_operation 工具参数说明，替换现有的 param 说明，并新增"GUI Agent 返回状态播报规则"。

**插入内容（type 完整规范 + 决策规则）：**

```
### GUI Agent 工具参数 type 完整规范

`new_task`
使用条件：发起一个全新的、独立的任务。
触发时机：用户提出新操作意图 / 原任务已终止需重新执行 / 用户打断后重新发起。
不能用于：向正在等待的任务传递用户补充信息。

`add_info`
使用条件：向当前处于"等待用户补充信息"状态的任务传递用户回复。
触发时机：GUI Agent 返回了需要用户输入的��题 / 用户在同一意图下补充遗漏参数。
不能用于：发起新任务 / 向已终止的任务传递信息。

`delete_task`
使用条件：终止一个正在进行中的任务。
触发时机：用户明确取消 / 用户切换到冲突新任务 / 任务遇到不可恢复的错误。

type 选择决策：
- 用户有新意图 + 当前无等待任务 → new_task
- 用户回复 GUI Agent 的问题 → add_info
- 原任务被终止后重新发起 → new_task（而非 add_info）
- 不确定时 → 优先 new_task

### GUI Agent 返回状态播报规则

播报时必须严格与 GUI 返回状态一致，禁止"超前播报"已��成的结果：

| 返回状态关键词 | 播报措辞规范 |
|---|---|
| "任务开始"、"正在操作"、"操作中" | "已为您开始操作，请稍候" |
| "操作完成"、"已提交"、"已下单" | "已为您完成[具体操作]" |
| "需要确认"、"等待输入" | 原文转述 GUI Agent 的问题 |
| "操作失败"、"出现错误" | "操作遇到问题：[具体原因]，是否重试？" |

禁止行为：GUI Agent 返回"奶茶点单任务已开始操作"，Planner 播报"已帮您点好奶茶"。
```

**配套 Few-shot 示例：**

```
示例：正确处理 GUI 中间状态播报
input:
{"tool_feedback":"收到补充信息，任务已继续操作，正在选择白雾红尘口味中","tool_name":"gui_agent_operation"}
output:
{
  "talk_or_not": "yes",
  "talk_content": "好，正在帮你加上白雾红尘，稍等一下",
  "action_list": []
}
```

### 验证标准

| 测试场景 | 期望行为 | 通过标准 |
|---|---|---|
| GUI 返回"任务已开始操作中" | Planner 播报"开始操作"类措辞 | 不出现"已完成"、"已帮您"等完成态表述 |
| GUI 返回"订单提交成功" | Planner 播报完成态 | 措辞与 GUI 返回状态一致 |
| 需�� `new_task` 的场景 | type 为 `new_task` | 无 `add_info` 误用 |
| 需要 `add_info` 的场景 | type 为 `add_info` | 无 `new_task` 误用 |

## 问题 5：Goal 创建/拆分问题（混合指令处理）

### 根本原因

**任务分类规则缺失 + 混合意图拆分逻辑不清晰。**

用户的混合指令（如"帮我点外卖，同时营造浪漫氛围"）包含两类性质不同的任务：
- **B 类 GUI 长时任务**：需要 GUI Agent 执行，时间较长（点外卖、叫车）
- **A 类即时动作**：可立即执行，无需 GUI Agent（调灯光、播音乐）

Planner 未对这两类任务做区分，导致即时动作被错误排队等待，或把非 GUI 指令传给 GUI Agent。

### Prompt 优化动作

**插入位置：** `prompt.md` 中"工具使用 tips"章节 goal_list_update 段落之后，新增"混合指令处理规则"。

**插入内容：**

```
### 混合指令处理规则

当用户指令同时包含多种性质不同的任务时，按以下规则拆分：

A 类即时动作：调节音量、灯光、空调、播放音乐、开关车窗等——立即执行，调用车载系统控制接口。
B 类 GUI 长时任务：点外卖、叫车、交停车费、订座等——通过 gui_agent_operation 发起，并配套创建 goal。

拆分规则：
1. 识别���令中所有子任务，分别归类为 A 类或 B 类。
2. A 类任务优先在同一轮回复中立即触发，不等待 B 类任务。
3. B 类任务独立发起，通过 GUI Agent 异步执行。
4. 向用户的播报需分别说明两类任务的执行状态。
5. A 类任务不得混入 gui_agent_operation 的 query 中，GUI Agent 无法处理车载系统控制指令。
```

      "tool_name": "vehicle_basic_control"
    },
    {
      "action_id": "2",
      "params": {"query":"播放轻柔浪漫的背景音乐"},
      "tool_name": "search_and_control_music"
    },
    {
      "action_id": "3",
      "params": {"type":"new_task","query":"到吉野家点招牌牛肉饭套餐外卖"},

### 验证标准

| 测试场景 | 期望行为 | 通过标准 |
|---|---|---|
| 用户指令包含点外卖 + 调灯光 | 灯光立即执行，外卖通过 GUI 发起 | A 类任务不进入 GUI 队列 |
| 用户指令包含叫车 + 播放音乐 | 音乐立即播放，叫车通过 GUI 发起 | 两类任务独立执行 |

| "点外卖+换音乐+交停车费" | 换音乐立即执行，点外卖和交停车费各自发起 GUI 任务 | 三种任务正确分类 |

## 问题 6：Planner 多任务互相影响

**结论：符合当前产品定义，无需处理。**

当前版本设计只允许一个 GUI 长时任务同时存在，"点咖啡"后下发"讲笑话"指令被打断，是符合预期的行为。无需修改 Prompt。

---

## 问题 2c 补充分析：播报内容与页面不一致

**可解部分（约 60%）：** Planner 的播报内容只能来源于 GUI Agent 的返回文本，不得自行推断或补充。通过问题 4 中的"播报状态规则"约束信息源，可消除 Planner 主动"脑补"与页面不符内容的情况。

**暂无法解部分（约 40%）：**
- GUI Agent 自身理解页面错误，返回内容与实际页面不符——属于 GUI Agent 能力边界，Planner Prompt 无法修复。
- 网络延迟、页面加载中等动态状态——Planner 无法实时感知页面变化。

**处置建议：** 标记为持续优化项，由 GUI Agent 侧逐步提升返回内容的准确性。
