# 触发器引擎（Trigger Engine）

## 定位

触发器是 Driver Agent 2.0 的"条件任务执行底座"——不直接面对用户，而是在后台持续监听车端信号，当预设条件满足时自动触发动作（车控/TTS/通知 Advisor）。产品负责人：王泽民（你）。研发负责人：刘洋。

## 在系统中的位置

```
用户语音 → ASR → 意图模型（解析条件话术）→ 触发器协议
                                                ↓
                              触发器引擎（云端注册 + 端侧判断触发）
                                                ↓
                              条件满足 → 执行动作 / 通知 Advisor / Planner 兜底
```

触发器处理的是"条件任务"——即"满足某个条件才执行"的任务，区别于用户说完立刻做的普通指令。

## 核心能力

| 能力 | 现状 | 目标（赛力斯版） |
|------|------|------------------|
| 条件判断 | 每帧全量扫描当前态 | 捕捉状态变更瞬间 + 条件组合 |
| 协议结构 | 套娃嵌套（门→位置→开关） | 拍平，各门独立一级 + 任意位置 |
| 时间条件 | 不支持 | 端侧定时器（定时/日期/节日） |
| 距离条件 | 不支持 | 端侧实时定位 < N米触发 |
| 地理围栏 | 不支持 | 圆心+半径+进出判定+防抖 |
| 端云分工 | 全绕云端 Planner | 云端注册仲裁 + 端侧判断执行 |
| 事件透出 | 不透出 | 接口列出可订阅事件 + 自动化验证 |

## 核心协议：五段式

条件在后台的标准表达格式：

```
条件类型 → 信号 → 字段 → 运算符 → 值
例：温度   → SIG-T-012 → temp → > → 26
```

## 端云分工铁律

- 云端 = 注册 + 冲突仲裁 + 承载模型生成事件
- 端侧 = 判断触发 + 定时器 + 执行策略
- **不能两边都判**——云端注册、端侧触发
