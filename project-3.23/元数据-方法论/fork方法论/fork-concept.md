## 定义

**Fork 文章方法论**：一种将传统"阅读-收藏"的知识消费模式，转变为"阅读-提取-工具化-运维"的知识运行模式的方法。核心思路来自程序员的 fork 操作——把别人的代码复制到自己名下并在自己的环境中运行。

在 AI 时代，这种操作不再局限于代码：任何一篇有方法论的文章，都可以被当作"源代码"来 fork——让 AI 帮你把其中的方法变成可运行的工具，然后在使用中持续迭代。

---

## 详细说明

### 一、核心原理：为什么"读"不够

传统的高质量学习动作——认真阅读、画重点、用 AI 生成总结、写感想、存笔记——在这套方法论中被统一归类为**知识消费**。

消费的特征是：信息流过你的大脑，激起一阵愉悦的涟漪，然后消散。三天后你可能还记得一些，三个月后几乎全忘。

这不是因为你不够认真。这是"读"这个动作的**结构性缺陷**：读完之后，知识没有进入运行态——它只是被存储了，不会自动产生价值。

> **核心判断标准**：凡是能让你真正起步的积累，几乎都是**可运行的**积累。不可运行的积累，无论看起来多努力，几乎都是无效积累。

这解释了"为什么大多数人一直在学，但却始终无法起步"：他们的学停留在消费/存储层面，从未进入运行态。

---

### 二、新旧循环对比

理解这两个循环的区别，是改变行为的起点。

**旧循环（知识消费）：**

读文章 → 觉得有道理 → 收藏/写感想 → 遗忘 → 偶然再次遇到 → 再读 → 再感动 → 再收藏 → …

特征：看起来一直在学，实际上同一批知识在消费环里反复流转。每次循环消耗时间但不产生增量。

**新循环（知识运行）：**

读文章 → 识别其中的"代码"（可工具化的方法论） → 让 AI 做成工具 → 在真实场景中使用 → 发现不满 → 迭代改进 → 工具越来越贴合自己 → 知识在运行中持续产生价值

特征：每次迭代都是增量。一个月后工具跟原文已经长得不一样了，它被你调教成了完全贴合自己需求的东西。原文的标题可能已经忘了，但它的"代码"一直在运行。

**关键区别不是 fork 本身，而是 fork 之后的运维。** 没有运维的 fork 就是穿着行动外衣的收藏。

---

### 三、收藏 vs Fork 对比

| 维度 | 收藏（消费） | Fork（运维） |
|------|------------|-------------|
| **你做了什么** | 存起来以后看 | 现在就让它跑起来 |
| **你的身份** | 读者 | 机械师 |
| **文章的身份** | 阅读材料 | 零件/源代码 |
| **知识的状态** | 静态存储 | 运行中、有反馈回路 |
| **3个月后** | 忘了在哪里存的 | 工具还在用，已经迭代了好几版 |
| **价值曲线** | 递减（遗忘曲线） | 递增（复利曲线） |

### 四、操作 SOP：7 步完整流程

以下是每次读到好文章时可以执行的标准流程。每步都附有详细说明和注意事项。

---

#### 第 1 步：打住（3 秒）

你正要点收藏、转发、或者存进笔记——**停下来**。

这是整个流程中最难的一步，因为收藏是条件反射。你需要用一个新的条件反射去替代它：

**问自己：「这篇文章里的代码是什么？」**

这里的"代码"不是指编程代码，而是指：这篇文章里有没有一个**可以被工具化的方法论、框架、流程或步骤**。

如果第一反应是"好像有"——继续下一步。
如果第一反应是"这篇文章主要是在表达观点/讲故事"——正常收藏即可，不是所有文章都值得 fork。

> **类比**：就像当年从掏现金切换到扫码支付，中间需要十几次刻意练习，之后才变成肌肉记忆。fork 文章也是同样性质的习惯切换。

---

#### 第 2 步：判断文章类型（30 秒）

不是所有文章都值得 fork。花 30 秒做一个快速判断：

**最值得 fork 的文章类型：**

| 类型 | 特征 | 能 fork 出什么 | 值得程度 |
|------|------|--------------|---------|
| **方法论型** | 文章给了具体的步骤/流程，如"如何做用户访谈的5步法"、"写作框架" | 清单工具、流程引导器、检查表 | ★★★★★ |
| **框架型** | 文章提出了一个分析框架，如"评估竞品的3个维度"、"决策矩阵" | 分析模板、决策辅助工具 | ★★★★ |
| **案例型** | 讲某个公司/团队怎么做成了某件事 | 提取其中可复用的模式，做成检查清单 | ★★★ |
| **观点型** | 主要在表达作者的看法和判断 | 思维透镜（可以用这个视角审视别的东西） | ★★ |
| **新闻/资讯** | 告诉你发生了什么 | 不适合 fork | 跳过 |

**判断技巧**：如果你能用"先做X，再做Y，然后做Z"来复述文章的核心内容，那它大概率是方法论型或框架型——最值得 fork。

---

#### 第 3 步：提取「代码」（2 分钟）

这一步是关键。你需要用自己的话回答三个问题：

1. **这篇文章的方法/步骤是什么？** （不是观点，是动作。如果提取不出动作，可能不值得 fork）
2. **输入是什么？** （用这个方法时，你需要提供什么）
3. **输出是什么？** （用完之后你会得到什么）

**示例 — 以本文为例：**

- **方法**：读文章 → 识别可工具化的方法论 → 让 AI 做成工具 → 使用 → 迭代
- **输入**：一篇有方法论的好文章
- **输出**：一个可运行的工具 + 持续迭代的习惯

**示例 — 假设你读了一篇"费曼学习法"的文章：**

- **方法**：选一个概念 → 用最简单的话教给一个不懂的人 → 发现卡壳的地方 → 回去补 → 再教一遍
- **输入**：一个你想学会的概念
- **输出**：一段你能流畅讲出来的简单解释

> **注意**：如果你在 2 分钟内提取不出明确的"方法/步骤"，这篇文章大概率是观点型。正常收藏即可，不必强求——fork 不是万能的，强行 fork 一篇没有方法论的文章只会浪费时间。

---

#### 第 4 步：扔给 AI（5 分钟）

把文章全文 + 下面的提示词一起发给 Claude 或 ChatGPT。

**提示词模板（直接复制使用）：**

```
我刚读了一篇文章，我想把它 fork 成一个我能实际用的工具。

先帮我做三件事：

1.【提取代码】这篇文章里可以被工具化的方法论/框架/步骤是什么？
   像描述一个函数一样写清楚：方法名、解决什么问题、输入、输出、具体步骤。

2.【工具提案】基于提取的代码，建议做一个什么工具？要求：
   - 必须是单个 HTML 文件，我双击就能打开
   - 不需要任何安装和配置
   - 界面干净好看
   - 非技术用户能直接用

3.【直接生成】不要只给建议，直接把那个 HTML 文件的完整代码写出来。
   代码里不要有 TODO 和占位符，我要能直接保存运行。

文章全文如下：

[在这里粘贴文章全文]
```

**为什么要求生成 HTML 文件？**

- HTML 是浏览器原生支持的格式，双击就能打开，零安装零配置
- 单文件可以包含界面 + 逻辑 + 样式，不需要依赖任何外部服务
- 非技术用户能直接用，不需要懂编程

**关键注意事项：**

发送前回顾一下你在第 3 步提取的"代码"。如果你觉得 AI 可能理解偏了，在提示词里加一句明确的引导：

```
这篇文章的核心方法是 [你提取的方法]，请围绕这个做工具，不要做其他东西。
```

**你的判断比 AI 的自动提取更准。** AI 容易被文章里的修辞和比喻带偏，做出一些泛泛的东西。你明确告诉它该做什么，出来的工具才会精准。

---

#### 第 5 步：保存并跑起来（2 分钟）

AI 给你 HTML 代码后：

1. **全选复制** AI 输出的代码（从 `<!DOCTYPE html>` 开始到最后一个 `</html>`）
2. **Mac 用户**：打开"文本编辑" → 菜单栏选"格式" → "制作纯文本" → 粘贴 → 保存为 `工具名.html`
3. **Windows 用户**：打开"记事本" → 粘贴 → "另存为" → 文件名填 `工具名.html` → 编码选 UTF-8
4. **双击这个文件** → 浏览器自动打开 → 工具跑起来了

> **更简单的方式**：如果你用 Claude Code，直接说"帮我保存到桌面"，它会帮你把文件保存好，省去手动操作。

---

#### 第 6 步：用它（至少 3 天）

第一版工具不会完美——这是正常的，不是 AI 的问题，也不是你的问题。

**你需要做的是：真实使用至少 3 天。**

在使用过程中，你一定会产生这些感受：
- "这个地方设计得不好用"
- "我其实更需要的是 XXX 功能"
- "这个流程太复杂了，能不能简化"
- "结果不够精准，需要调整"

**这些不满就是最有价值的东西。** 它们是真实使用产生的反馈信号，是你下一次迭代的输入。没有使用就没有反馈，没有反馈就没有迭代，没有迭代就没有复利。

> **常见陷阱**：做完第一版后就放在那里不用了。这等于 fork 之后不运维——跟收藏没有本质区别。

---

#### 第 7 步：迭代（每次 5 分钟）

用了 3 天后，把你的具体不满告诉 AI：

**迭代提示词模板：**

```
这是我之前 fork 出来的工具（附上当前代码或描述功能）。

用了几天，我发现这些问题：
1. [写你的第一个具体不满]
2. [写你的第二个具体不满]

帮我改一版。改完直接给我完整代码。
```

**迭代原则：**

- **每次只改 1-2 个点**。不要一次提 10 个需求，改出来的东西会乱。小步快跑。
- **不满要具体**。"不好用"是没用的反馈，"第三步的提示语看不懂"是有用的反馈。
- **不需要等到完美再用**。80% 好用就继续用，发现问题再改。

> 一个月后，这个工具大概率跟原文的方法论已经长得很不一样了。它被你的真实需求塑造了。**原始文章可能已经忘了标题，但它的价值一直在运行。**

---

### 五、怎么判断 fork 成功了

四个判断标准：

1. **工具真的在用** — 不是保存了没打开过，而是在日常工作/学习中实际使用
2. **至少迭代过一次** — 第一版几乎不可能完美，如果你没有迭代过，说明要么没用，要么还没用够
3. **忘了原始文章的标题** — 但工具还在帮你。这说明知识已经从"存储态"进入了"运行态"
4. **能一句话说清楚工具解决了什么问题** — 如果说不清，工具可能太泛了

**反面信号：**

如果 fork 完一个工具，一周内你一次都没打开过——说明：
- 提取的"代码"可能不对（方法论没有找准）
- 工具的形态可能不适合你的场景（比如做了个网页但你更需要清单）
- 这篇文章可能本来就不值得 fork

这不是失败，是**诊断信息**。下次遇到类似文章时，换个角度提取。

### 六、Fork 记录表

每次 fork 后填一行。这个表是你的**知识复利账本**——记录你把哪些文章变成了运行中的工具。

| 日期 | 文章标题 | 提取的方法 | 做了什么工具 | 迭代次数 | 还在用吗 |
|------|---------|-----------|------------|---------|---------|
| 2026-05-19 | 别再读文章，要fork文章 | 文章→工具→迭代循环 | 本教程 + SOP + 提示词模板 | 0 (初版) | 刚开始 |
| | | | | | |

定期（比如每月一次）回顾这个表：
- 哪些工具还在用？→ 它们是真正的复利资产
- 哪些一次都没打开过？→ 反思提取方法是否正确
- 迭代次数最多的工具是什么？→ 这可能是你最核心的需求领域

---

### 七、深层原理：编程和写作是同一件事

纳瓦尔说编程和写作是普通人非常值得尝试的两项工作。这篇文章的作者认为更深一层：**编程和写作本质是同一件事——把想法变成可运行的东西。**

代码跑在机器上，文字跑在读者头脑中。

AI 出现后，这个"可运行"的边界大幅扩展。以前只有程序员能把方法论变成可运行的工具，现在任何人都可以。**Fork 文章方法论本质上是在说：每个人都可以成为自己知识系统的程序员。**

## 相关概念

- **间隔重复**（Spaced Repetition）：一种通过定期复习来对抗遗忘曲线的记忆方法。Fork 方法论比间隔重复更激进——它认为记住不够，要让知识运行起来。
- **Zettelkasten 卡片笔记法**：一种通过原子化笔记和关联来构建知识网络的方法。Fork 方法论可以看作 Zettelkasten 的升级版——不止于笔记，而是做成工具。
- **Karpathy LLM Wiki**：一种用 LLM 维护三层知识库的方法。本知识库本身就采用这种架构。Fork 方法论可以作为向这个知识库贡献内容的一种工作流。
