## 定义

**Planner感知差距诊断——你的SP是60分还是95分？**：原文揭示了Agent质量差距的"不可见性"——你用60分的Agent会觉得"还不错"，直到对比才知道差距。本教程将这个诊断框架映射到你的Planner SP维护工作，给你一套可量化、可执行的SP质量评估体系：知道差距→测量差距→缩小差距。

## 详细说明

### 一、核心问题：你的SP是几分？

文章的核心等式：**你感知到的质量 ≪ 实际质量差距**。

映射到你的工作：

| 文章中的角色 | 你的场景中的对应 |
|------------|--------------|
| "我的Agent"（精心配置） | 你的当前SP（30K+ tokens，精心维护） |
| "家人的Agent"（未配置） | 一个只有基础prompt、没有示例/注意事项的SP |
| 底座模型一样 | 都是Seed 1.8 |
| "家人觉得不错" | 车主觉得"豆包还行" |
| "拉到同一个群对比" | 两版SP同时跑同一批query对比 |
| "我才发现新Agent笨得不行" | 对比后发现基础SP的工具选择、参数传递、多步规划全面落后 |

**关键判断**：你确实知道"精心维护的SP > 基础SP"。但你不知道的是：**"精心维护的SP"和"理论最优SP"之间的差距有多大？** 你有对比基准吗？

### 二、"笨"的五个维度——映射到Planner的具体评估指标

文章列出了Agent"笨"的五个维度。每个维度在Planner中都有精确的评估方法：

#### 维度1：思维深度 → Planner的多步规划质量

| 表现层级 | 60分SP的表现 | 95分SP的表现 | 怎么测 |
|---------|------------|------------|-------|
| 简单请求 | "打开车窗"→正确执行 | 一样（差距不体现） | 不用测，都对 |
| 隐含需求 | "我好热"→开空调 | "我好热"→查车况+开空调+降温+问是否开窗 | 设计10个隐含需求case |
| 多步任务 | "去昨天游泳馆"→搜索"游泳馆" | "去昨天游泳馆"→查记忆→取地址→导航 | 设计5个多步case |
| 矛盾需求 | "开暖气但我想通风"→只开暖气 | →开暖气+告知通风会降温+建议折中方案 | 设计3个矛盾case |

**关键**：60分和95分在简单请求上表现一样。差距只在复杂场景暴露——这就是为什么"不对比你感知不到"。

#### 维度2：指令遵循能力 → SP规则的执行一致性

| 场景 | 60分 | 95分 | 怎么测 |
|------|------|------|-------|
| 输出格式 | 大部分时候符合JSON规范 | 100%符合JSON规范 | 跑100条，统计格式合规率 |
| 注意事项遵循 | 偶尔违反第3条注意事项 | 10条注意事项全部遵循 | 每条注意事项设计2个测试case |
| 边界条件 | 极端case时忽略规则 | 极端case也保持规则 | 专门测试极端case |

**诊断方法**：你的SP有10条注意事项——设计20个case（每条2个），跑一遍看几条被严格遵循。遵循率 < 80% → 你的SP可能只有70分。

#### 维度3：元认知 → Planner"知道自己不确定"的能力

| 场景 | 60分 | 95分 | 怎么测 |
|------|------|------|-------|
| 信息不足 | 瞎猜+执行 | 主动确认"你是想…还是…？" | 设计5个模糊query |
| 超出能力 | 给一个"听起来合理的回复" | 明确说"这个我做不了" | 设计3个超出能力边界的case |
| 多义指令 | 选一个意思执行 | 列出可能的理解，让用户选 | 设计3个歧义query |

**这是文章说的"合理解释通病"在Planner中的直接体现**：Planner处理不了复杂请求时，不会说"我不知道"——它会给一个"流畅但没做对事"的回复。你的badcase排查应该专门关注这类case。

#### 维度4：工具选择和使用 → 23个工具的调用精确度

| 场景 | 60分 | 95分 | 怎么测 |
|------|------|------|-------|
| 单工具 | 选对工具 | 选对工具+参数完整 | 每个工具1个标准case |
| 多工具协作 | 选了相关工具但漏了一个 | 所有相关工具都调了+顺序正确 | 设计5个多工具case |
| 工具不需要 | 调了不必要的工具 | 不需要工具时纯对话 | 设计3个不需要工具的case |

**量化方法**：工具选择准确率 = 正确调用次数 / 应调用次数。低于90% → SP的工具描述需要优化。

#### 维度5：关键任务技巧 → 特殊场景处理能力

| 场景 | 60分 | 95分 | 怎么测 |
|------|------|------|-------|
| 安全相关 | "帮我开120" → 照做 | → 提醒安全限速 | 设计3个安全case |
| 情感场景 | 用户说"我心情不好" → 搜歌 | → 共情+问想聊聊还是听歌 | 设计3个情感case |
| 长对话记忆 | 第10轮忘了第2轮说的 | 第10轮还记得第2轮的偏好 | 设计2个长对话case |

### 三、"合理解释"识别方法——Planner的优雅失败诊断

文章说Agent"做不到时给你一个合理解释"。这在Planner中的表现：

| 表象 | 实际情况 | 诊断方法 |
|------|---------|---------|
| talk_content很流畅友好 | action_list为空或错误 | 对比talk和action是否一致 |
| "好的，我来帮你" | 调了工具但参数错了 | 检查工具参数是否精确 |
| "抱歉我无法执行" | 其实可以执行，SP描述不够清晰 | 手动验证是否真的不可以 |
| "已经为您执行了" | 只执行了部分，漏了关键步骤 | 检查action_list完整性 |

**Prompt模板——排查"优雅失败"：**

```
以下是5条Planner输出日志，每条包含user_query、talk_content、action_list。

{粘贴日志}

请逐条检查是否存在"优雅失败"——即talk_content听起来合理但action_list不正确：

判断标准：
1. talk_content承诺做的事，action_list是否真的做了？
2. 用户隐含的需求，action_list是否覆盖了？
3. 如果action_list为空，是真的不需要执行操作，还是Planner没能力理解？
4. 工具参数是否精确（特别是位置、时间、数量等关键参数）？

对每条标记：✅正确执行 / ⚠️部分执行（漏了什么）/ ❌优雅失败（说了但没做对）
```

### 四、建立你的"对比基准"——从感知不到到量化诊断

文章说"除非你能对比，否则你很难知道"。你需要建立三种对比基准：

#### 基准1：内部纵向对比（和自己的历史版本比）

| 做法 | 怎么做 | 意义 |
|------|--------|------|
| SP版本对比 | 保存每个里程碑版本的SP，定期用同一批case对比新旧版本 | 知道自己是否在进步 |
| Badcase趋势 | 按月统计badcase数量和类型分布 | 同类badcase减少=SP在这个维度变好了 |
| 覆盖率跟踪 | 列出所有场景类型，标记SP的示例覆盖率 | 覆盖率提升=控制颗粒度在提升 |

#### 基准2：外部横向对比（和竞品/同行比）

| 做法 | 怎么做 | 意义 |
|------|--------|------|
| 竞品体验 | 定期体验竞品车载AI（小鹏/蔚来/理想），用同样的query测试 | 知道行业水准在哪里 |
| 同行交流 | 和字节内部其他AI产品的PM交流Agent配置方法 | 取长补短 |
| 公开评测 | 关注学术/行业的车载AI评测（如果有） | 外部标准化基准 |

#### 基准3：理论上限对比（和"理想Planner"比）

| 做法 | 怎么做 | 意义 |
|------|--------|------|
| 定义"完美回答" | 对20个core case写出你认为的"完美Planner输出" | 知道95分长什么样 |
| 差距分析 | 比较Planner实际输出和"完美输出"的差距 | 精确定位SP需要改进的维度 |
| 理想SP推演 | 基于"完美输出"反推SP应该怎么写 | 从结果倒推SP优化方向 |

**最重要的是基准3**——因为你在做基准3的过程中，会强迫自己定义"好"是什么样子。这个定义过程本身就是提升。

### 五、从"感知不到差距"到"主动诊断差距"——SP质量评估SOP

综合上面的维度和基准，形成一套可执行的评估SOP：

#### 每月一次"SP健康度体检"（共4小时）

| 步骤 | 做什么 | 工具 | 时间 |
|------|--------|------|------|
| 1. 准备评测集 | 从五个维度各选4个case = 20个case | 参考维度表 | 30分钟 |
| 2. 跑当前SP | 20个case全部跑一遍，收集Planner输出 | API调用 | 1小时 |
| 3. 人工打分 | 每个case按5分制打分（1=完全错，5=完美） | 打分表 | 1小时 |
| 4. 诊断"优雅失败" | 专门检查talk和action是否一致 | Prompt模板 | 30分钟 |
| 5. 差距分析 | 和上月得分对比，找到下降/停滞的维度 | 趋势表 | 30分钟 |
| 6. 定优化方向 | 选最低分的1-2个维度，作为下月SP修改重点 | — | 30分钟 |

**评分维度模板：**

| 维度 | Case编号 | 得分(1-5) | 备注 |
|------|---------|-----------|------|
| 思维深度 | C1-C4 | | |
| 指令遵循 | C5-C8 | | |
| 元认知 | C9-C12 | | |
| 工具选择 | C13-C16 | | |
| 关键技巧 | C17-C20 | | |
| **总分** | | **/100** | |

60分以下 = 需要大幅改进 / 60-75分 = 基础可用但有明显盲区 / 75-85分 = 良好但特定维度有短板 / 85-95分 = 优秀 / 95分以上 = 行业顶尖

### 七、判断标准

**你已经有"差距感知能力"的信号：**
- 你能给自己的SP打分（不是"还行"，是"78分，元认知维度最弱"）
- 你有一个评测case集，定期跑
- 你知道竞品车载AI在你关注的场景上表现如何
- 你专门排查"优雅失败"——不只看"做错了"，也看"说了但没做对"
- 你的SP优化有方向——不是"哪里有badcase改哪里"，而是"这个月重点提升元认知维度"

**你还在"感知不到差距"的信号：**
- 你评价自己SP时用的是"还行""挺好的"而不是量化分数
- 你不知道竞品的表现（没对比基准）
- 你的badcase排查只关注"明显做错了"，忽略"说了但没做对"
- 你的SP优化方向是被动的——哪里出badcase改哪里，没有主动的维度规划
- 你从来没写过"理想Planner输出"应该是什么样

**终极判断**：文章说"你的超常发挥，也许不过是别人的吃饭饮水"。反过来——如果你建立了评估体系、持续量化追踪、主动诊断差距——你就是那个"吃饭饮水就是别人超常发挥"的人。
