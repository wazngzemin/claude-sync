6.22 会议结论：端侧触发器架构

2026-06-22 下午会议 · 参会：王泽民、赵益红、丁彬、陈西钺、施佳杰、吴仁杰

这份文档只讲一件事：6月22号下午这场会，大家对端侧触发器的架构定了什么、没定什么、各模块该怎么配合。

一、最大的结论：端侧闭环场景全部回CDC

会上仁杰提了一个判断标准，大家都认了：

仁杰 13:14：像那种车辆安全、上车安全、下车安全，它是所有车一个标配的功能，不会变，不涉及条件变化或用户改配置，那这种直接底软做掉，不需要走触发器。像那种有生命周期的、用户临时创建、消费完就没了、条件可能变化的，那走触发器。

丁彬总结得很干脆：

丁彬 16:17：那看来原来端侧闭环的，现在还继续是闭环的，就几乎不需要触发器去做什么。

所以哨兵模式、上下车安全、起步安全、迎宾、充电桩开盖、温控、睡眠检测——这些固定场景全部留在赛力斯CDC底软里闭环，不走触发器。触发器只接动态的、可变的、用户自己设的、可运营的场景。

判定标准：
固定 / 不变 / 标配 / 整个车辆生命周期都不会变 → CDC底软闭环
动态 / 有生命周期 / 用户创建 / 可运营 / 条件会变 → 走触发器

二、VLM链路：动态视觉感知事件怎么走

这条链路解决的问题是：用户说"宝宝毯子掉了就提醒我"，这种需要持续视觉观测的动态任务，端到端怎么走。

会上仁杰把这条链路捋清了（42:01），佳杰补充了事件组装的逻辑（24:05-24:31），整条链路是这样的：

1
Director 接到用户query，识别出这是一条规则型任务（"当X就Y"），创建 动态Advisor 云端
↓
2
动态Advisor 拆解query，发现需要视觉类条件，改写出一条注意力SP（比如"观测毯子是否掉落"），调 云端触发器 注册这个场景 云端
↓
3
云端触发器 把动态SP下发给 端模(VLM)，端模的注意力prompt增加一项观测目标 云→端
↓
4
端模(VLM) 按周期观测，输出结果写入 端侧信号库（纯客观状态描述，不做判断，比如"毯子当前在地上"） 端侧
↓
5
云端触发器 监听信号库变化，执行业务逻辑（阈值穿越、首次出现等），把原始状态组装成语义事件（"用户睡醒了"） 云端
↓
6
云端触发器 把事件 push 给 动态Advisor，Advisor消费事件后执行对应动作 云端

这里有个关键的设计选择——事件走push，不走轮询：

佳杰 26:35：我需要一个用户睡醒了的这样一个事件，我不会去轮询你的context，我是注册了这个事件之后，我就等待你的云端触发器给我发这个事件。

事件和条件（Context）的区别

佳杰用了一个例子把这事讲得很清楚（29:34）：

"用户在倒车"——这是事件，不能24小时轮询，需要触发器监听到了push过来。
"车里有小朋友"——这是条件（condition），放在Context里，Advisor查的时候去读。

所以Context解决的是"当前状态是什么"，触发器解决的是"什么时候发生了变化"。两件事，两条路，不混。

端模能力边界

丁彬讲了端模现在的情况（40:10-40:58）：端模目前是解耦的，纯当眼睛用——你告诉它看什么，它就看什么，输出观测结果。但你给它下一个复杂任务（"看到毯子掉了就怎么样"），它做不了后半段，只能做到"看到毯子"这一步。

如果通过prompt限制观测目标，在有限集内（比如只看毯子、毛巾这几类固定的东西），效果基本能保证。发散了就靠基模能力，效果不确定。

动态视觉感知事件清单

佳杰和益红在梳理具体要哪些事件（22:29），目前大致包括：

  • 睡着 / 睡醒
  • 驶出地库
  • 打手机 / 结束打手机
  • 某个位置出现了某个物品 / 物品被移除

待定：佳杰和益红周内出清单给泽民，7月中前这块要全部定下来（仁杰 42:34）。

两类视觉事件

西钺和丁彬在后半段讨论里（01:00:14-01:02:35）分出了两类：

类型说明走法
有限集事件 能穷举的、预训练好的（睡醒、驶出地库、打电话等） 预置在端模SP里，周期观测上报
通用视觉事件 用户临时提的、泛化的（"帮我看宝宝毯子掉没掉"） 动态Advisor改写SP → 触发器下发 → 端模增量观测

通用视觉事件是兜底通道，效果不一定好。佳杰说到时候可以让豆包先问用户"这个我可能做得还不是那么好，你确定要让我试试吗？"（01:04:33）。

三、VQA链路：一次性视觉问答怎么走

VQA和VLM不是一回事。VLM是持续观测（"一直帮我看着毯子"），VQA是一次性问答（"现在窗外是什么天气"）。

上汽这边没有端模，所以视觉全走云端VQA。佳杰在会上提到了上汽的做法（37:59）：

1
用户提问（"窗外什么天气"），Director 判断需要视觉信息，调 search_visual_info 工具 云端
↓
2
端侧摄像头抓帧，图片上传 端→云
↓
3
云端VQA服务 拿到图片+用户query，推理出答案 云端
↓
4
结果返回给 Director，Director组织话术回复用户 云端

VQA和VLM的核心区别：

VQAVLM
触发方式用户主动问系统持续观测
调用频率一次性按周期（3-5秒）
推理位置云端端侧
跟触发器的关系无关，走Director→search_visual_info触发器注册/下发SP/组装事件

佳杰提到一个现实问题（39:07）：VQA和VLM会抢占摄像头资源。上汽这边已经遇到了，端侧取图的时候跟其他业务有抢占，导致VQA用不了。所以佳杰倾向于把所有信号类的东西并到一个模块去提供，避免重复造轮子和资源竞争。

四、端云模块分工

云端 要做的事

  • 云端触发器：接收Advisor注册的场景 → 监听端侧信号变化 → 执行规则/阈值判断 → 把状态组装成事件 → push给Advisor
  • 动态Advisor：拆解用户query → 改写注意力SP → 注册触发场景 → 消费触发器push的事件 → 决定执行什么动作
  • Director：识别任务类型 → 分发给对应Advisor → VQA场景直接调search_visual_info
  • VQA服务：接收抓帧图片+query → 推理 → 返回结果
  • 任务中心（云端部分）：接静态Advisor和动态Advisor的任务，和端上的任务中心同步

端侧 要做的事

  • 端模(VLM)：按SP观测 → 纯客观输出状态 → 写入端侧信号库。不做判断逻辑，只当眼睛
  • 端侧信号库：存所有信号（车态+视觉观测），供云端触发器监听
  • 端侧触发器：执行云端下发的动态任务（有定时器能力），不做CDC闭环场景
  • 任务中心（端侧部分）：本地持久化 → 弱网可用 → 支持enable/disable → 向佳杰的AIBOX提供数据接口
  • CDC底软：所有固定闭环场景（哨兵、上下车安全、起步安全、迎宾、温控、睡眠检测等）

五、任务中心的对接需求

这块是佳杰在51:07-55:08提出来的，之前没聊过，会上第一次摆到台面。

佳杰需要触发器端侧有一个模块，能让AIBOX的任务中心来读取和操作触发器的任务数据。具体需求：

操作说明
读取任务列表AIBOX从端侧触发器拉取当前所有任务（只拉对用户有显示价值的"任务级"数据，不拉底层"感知级"的原始场景）
enable / disable用户在任务中心启用或禁用某个任务，触发器要真的停掉
删除 / 找回从待办列表移除，或从场景中心里找回
端云同步云端配置的任务要能同步到端侧持久化，弱网也能用

关键区分：触发器里的数据分两种。一种是"任务级"的（用户能感知的，比如"充电桩靠近自动开盖"），要在任务中心显示。一种是"感知级"的（底层信号订阅，用户看了也没意义），不显示。触发器需要一个字段来区分这两类。

六、AlwaysOn调度不归触发器

西钺问了这个问题（57:25），佳杰回答得很明确（57:47）：AIBOX自己有一个任务调度模块，专门做所有请求的优先级排序——AlwaysOn优先级多高、VQA优先级多高、其他视觉需求优先级多高，低优先级的排队等。这个不在触发器里面。
