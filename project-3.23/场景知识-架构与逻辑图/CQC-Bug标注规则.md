# CQC Bug 分拣标注规则（标注规范）

> **文档性质**：这是 bug 分拣的**唯一判定标准**。CQC 标注时以本文为准，带教人以本文为评分依据。
> **维护规则**：
> 1. 本文是**活文档**。后续遇到新的、规则没覆盖的 case，**追加到第 9 节「案例库」对应分类下**，并在第 10 节「变更记录」登记一行。
> 2. 出现「规则之间冲突」或「新边界」时，由带教人（王泽民）+ 李强/单如水 确认后，**修订对应规则条目**，并在变更记录注明。
> 3. 不要删旧案例。判错的、被下游退回纠正的，也保留并标注"❗已修正"，作为反面教材。

---

## 0. 一句话总则

> **看「模型输出评分」列，0/1/2 分的都要分发；3 分合格不动，NA 跳过。**
> **分发 = ①判问题分类 → ②定修复出口 → ③（动态示例类）写决策分析 → ④标风险 → ⑤填处理人。**
> **核心判据：成批同因 → 静态示例；孤立单点 → 动态示例；拿不准下游/要写多个「如果」→ 停手标记上交。**

---

## 1. 标注总流程（每个 bug 走一遍）

```
Step1  看评分列：0/1/2 分 → 进入分发；3 分/NA → 跳过
Step2  看「问题分类」列（若已填，作参考）+ 读 query 和模型输出
Step3  判【问题分类】（第3节，8 类）
Step4  判【修复出口】（第4节，6 个出口）
Step5  动态示例类 → 写【决策分析】（第7节模板）
Step6  标【风险】：疑似下游 / 边界存疑 / 复杂建议静态
Step7  填【处理人】+【去重标记】（第8节）
```

## 2. 评分门槛（哪些要分发）

| 评分 | 含义 | 是否分发 |
|------|------|---------|
| 0 分 完全失败（人工智障/存在风险） | 必分发，最高优先级 | ✅ |
| 1 分 极差体验（帮了倒忙） | 必分发 | ✅ |
| 2 分 勉强接受（能办事但笨/磨叽） | 必分发 | ✅ |
| 3 分 符合预期 | 合格，不动 | ❌ |
| NA | 无效/未评 | 跳过 | ❌ |

| # | 问题分类 | 定义 | 判定信号 | 典型 case |
|---|---------|------|---------|----------|
| A | **工具调用类问题** | 该调的工具没调 / 调错了工具 / 没传必填参数 | 输出里工具名错、漏调、参数缺失 | 驾驶风格激进→没调车控；删任务→没传 goal_id |

| G | **内容幻觉问题** | 模型编造了与事实不符的内容 | 输出与实际状态/数据不符 | 行程说"换了顺序"实际没换 |
| H | **Case 设计问题** | query 本身构造有误 / 系统无此功能 | query 看不懂、测了个不存在的功能 | 打开 MV 跳转短视频工具 |

## 4. 修复出口定义（6 个出口）

> 这一层回答**「该谁修、怎么修」**。这是分发的最终目的。

| 出口 | 定义 | 判定标准（关键） | 处理人 |
|------|------|----------------|--------|
| **① 动态示例** | 低频、明确、**孤立单点**的工具路由错，一句话能说清调哪个工具 | 「一条 query 配一个执行答案」就能解，**无法抽象成一类** | 李强 |
| **② 静态示例 / SP** | **能用一条规则覆盖一批** query 的共性问题 | 同因 case ≥ 2 个；或要写多个「如果…就…」的复杂逻辑 | 单如水 |
| **③ 动态知识** | 模型缺的是车辆功能/规格的**事实知识** | 属硬件认知盲区，补知识即可，不是路由错 | 算法 |
| **④ 下游 / 链路** | Planner 决策对，是下游工具或链路执行问题 | 决策分析写出来是对的，但下游不支持/没执行 | 左晨（下游）/ 工程（链路）|
| **⑤ 非问题** | query 构造有误 / 系统无此功能 | query 看不懂、测了不存在的功能 | CQC 自己评论关闭 |
| **⑥ 模型能力（难解）** | 信息全、工具对，模型仍推理/幻觉错 | 排除以上全部后剩下的 | 算法深查 |

## 6. 边界与易混规则（最容易判错的，逐条钉死）

### 规则 6.1　单点 vs 共性（动态 vs 静态）—— 最重要
- **判据**：能不能用「一条规则」覆盖一批 query？
  - 能 → **静态示例**（如"日期一律查 timestamp"覆盖所有日期 query）
  - 只能就事论事 → **动态示例**
- **硬信号**：同一根因的 case **出现 ≥ 2 次**，一律按共性走静态，**不许逐个写动态示例**。
- 例：8 个"认识副驾的 X"都是 json 格式错 → 改 1 处 SP，**不是写 8 个动态示例**。

### 规则 6.2　复杂逻辑必须上交静态
- 写决策分析时若需要 **多个「如果…就…」** 才能说清 → 停手，标"复杂建议静态"，转单如水。
- 例："有点闷透透气"（要按车速/温差/天气/车窗状态判断）→ 静态示例，不进动态。

### 规则 6.3　带条件/带时间 → goal_list_update，不是车控
- 「…后 / …时 / 持续 / 每次 / 高于…就」= 条件触发或持续任务 → **goal_list_update**。
- 例：30 秒后开车窗、温度低于 22 度关空调、持续介绍风景。
- 对照：「直接打开车窗」无条件 → vehicle_basic_control。

### 规则 6.4　music vs media
- **要搜/放新内容** → search_and_control_music（放周杰伦、音乐歌曲播放）
- **控制正在播放的** → media_basic_control（暂停、关 MV、视频暂停）

### 规则 6.5　查状态 vs 控制
- 「…开了吗 / 现在是什么 / 多少」= 查询 → **search_vehicle_status_info**，不是控制工具。
- 例：空调开了吗、氛围灯现在啥颜色 → 查状态，不要去调控制。

### 规则 6.6　声场/音效/音量边界模糊 → 标"边界存疑"
- 这块连历史标注都不一致（"切换声场"→系统设置，"场景音效"→车控）。
- **拿不准就标"边界存疑"上交，不硬判。**

### 规则 6.7　记忆/技能工具混用
- 客观事实记录 → operate_user_memory；个性化交互习惯 → personal_skill_record；查记忆 → search_user_memory。
- 已在 memory 里的，**直接回答，不必再调记忆检索工具**。
- 三者混用是**共性问题 → 静态示例**。

### 规则 6.8　下游问题照写决策分析再标记
- 判为"疑似下游"时，**仍要写出你认为正确的决策分析**（方便下游对照），再标"疑似下游"进左晨群。
- **写了 ≠ 解决**。下游不支持的，光提动态示例修不好。

### 规则 6.9　非问题的判定与关闭
- query 本身乱、系统无此功能 → 评论"无此功能/query 构造有误"关闭。
- 关闭前确认：是评测集构造的 query 问题，还是真实用户需求被误判。看提单人/来源。

## 7. 决策分析书写规范（仅动态示例出口需要）

**模板：**
```
query：<原文，不改写>
决策分析：<是否改写 + 调哪个具体 tool_name + 干什么>
```

**三条死规矩：**
1. **必须点名具体 tool_name**（从 23 工具里选），不能口语化（"开空调"❌ → "调用 vehicle_basic_control"✅）。
2. **一句话能说清就别展开**；要写多个「如果」→ 改判静态示例。
3. **称呼/联系人不改写**（"给妈妈打电话"不改成号码）。

**自查（写完必做）：决策分析里的动作，和 query 对得上吗？**
（反例：query"打开录音开关"，决策分析却写"结束录音纪要"=串行错。）

## 8. 填表 / 字段规范

| 列 | 填什么 | 必填 |
|----|--------|------|
| 问题分类 | 第 3 节 8 类之一 | 是 |
| 修复出口 | 动态示例/静态示例/动态知识/下游/非问题/模型能力 | 是 |

**分流规则**：动态示例→李强；静态→单如水；动态知识/难解→算法；下游→左晨群；链路→工程；非问题→自己关闭。

## 附：30 秒快速判定 Checklist（贴工位）

```
□ 评分是 0/1/2 吗?           否→不处理
□ query 乱/无此功能?         是→非问题，关闭
□ 没到 Planner?              是→链路，转工程
□ 同根因 ≥2 个 / 要写多个如果? 是→静态示例，转单如水
□ 缺的是车辆功能事实知识?     是→动态知识，转算法
□ 孤立单点、一句话能说清?     是→动态示例，写草稿给李强
□ 决策对但下游可能不支持?     是→照写+标"疑似下游"，进左晨群
□ 声场/音效/拿不准?          是→标"边界存疑"上交
□ 写完自查: 动作和 query 一致吗?
```
