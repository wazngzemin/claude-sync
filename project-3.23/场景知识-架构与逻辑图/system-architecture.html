一、系统全景 —— 豆包是谁，它收到什么，它能做什么
系统的核心是豆包——一个车载 AI 助手。它直接跟用户对话、调用工具、执行任务。
豆包周围有一圈"后勤系统"：筛选 Agent 给它挑知识，advisor 给它推建议，18 个工具给它干活的能力。

输入类型 1：user_query

用户说的话

  • query：用户的原话
  • speaker_name：谁说的
  • speaker_position：在哪个座位
{"query":"好热啊",
"speaker_name":"明骏",
"speaker_position":"主驾驶位"}

输入类型 2：advisor

其他专家智能体的建议

  • 豆包不回应 advisor
  • 直接面向用户做行动和表达
  • advisor 不是用户说的话
{"advisor":"车内温度35度，
建议开启空调强力制冷
和座椅通风"}

输入类型 3：tool_feedback

工具执行后的返回结果

  • tool_feedback：工具返回信息
  • tool_name：是哪个工具返回的
  • 不是用户说的话
{"tool_feedback":"执行成功",
"tool_name":"vehicle_basic_control"}
▼ ▼ ▼ 三种输入都发给豆包 ▼ ▼ ▼

豆包 — 车载 AI 助手本体

说话风格像温柔聪明的高知女生，以短回复为主

5 个上下文槽位（prompt 里的模板变量）

{{status}} 舱内视觉感知 — 摄像头看到的人、物、动作
{{goal_list}} 目标队列 — 正在追踪的持续任务
{{env_info}} 车辆状态 — 空调/座椅/车速/导航等实时信息
{{memory}} 用户记忆 — 偏好、习惯、关系
{{knowledge}} 动态专业知识 — 筛选 Agent 选中的知识注入到这里

输出格式（严格 JSON）

{ "talk_or_not": "yes/no", "talk_content": "要对用户说的话", "action_list": [ { "action_id": "1", "params": {"query": "..."}, "tool_name": "工具名" } ] }

关键决策点：
· 要不要说话？（有时沉默更好）
· 说什么？（语气、长度、是否带情绪标签）
· 调哪些工具？（能并行就并行，有依赖就串行）

二、{{knowledge}} 从哪来？—— 筛选 Agent 的工作
豆包 prompt 底部有一个 <动态专业知识> {{knowledge}} </动态专业知识> 槽位。
这里面的内容不是固定的，而是由另一个 Agent——知识筛选 Agent——实时决定放什么进去。
筛选 Agent 定义在 prompt.md 中

它同时看到的 8 个信号

1. 用户当前 Query

{"user_query":"好热啊", "speaker_name":"明骏", "speaker_position":"主驾"}

2. 对话历史（最近10轮）

3. 舱内视觉感知

摄像头看到的：表情、动作、着装、物品

4. 目标队列

正在追踪的持续任务

5. 最新车辆状态

优先级高于对话历史中的车辆状态

6. 车辆状态变化点 ⚡ 最关键

"趋势比绝对值更具决策价值"
温度22→29比"当前29度"信息量大得多

7. 用户长期记忆

8. 当前已加载知识 ID 列表

上一次筛选后，哪些知识已经在 {{knowledge}} 里

它的 5 条决策原则

1

场景绑定推理

综合全部输入，先判断场景，再匹配知识 ID。不是关键词匹配。
2

环境变化优先

变化点信号 > 绝对值。温度急升比"当前温度高"更重要。
3

目标在就不卸载

目标队列有进行中任务 → 支撑知识保留，直到用户终止。
4

增量更新

只输出差异：要新增的和要删除的，不重复已有的。
5

不确定就不加

信息不足 → add 为空，在 reasoning 中说明。

它的输出

{ "add_knowledge_ids": ["DRV-01-01-04"], "remove_knowledge_ids": ["DRV-01-01-12"], "reasoning": { "add_reason": "温度5分钟急升+乘员出汗，需急速降温知识", "remove_reason": "通风知识已不需要，降温知识覆盖范围更大" } }

豆包会参考它来决定怎么说话、调什么工具

关键理解

知识的消费者是豆包本身，不是一个独立的"思考者 Agent"。
豆包读到你写的示例后，要能判断：该不该说话、说什么、调哪个工具、参数填什么。
所以你的多轮示例格式必须跟豆包的输出格式一致：input → output JSON，而不是自然对话体。

四、advisor 和 knowledge 是两条并行的路径
豆包同时从两条路径获得"场景指导"，但机制完全不同。

路径 A：advisor（实时建议）

其他专家智能体主动推送的一句话建议。

跟 user_query 可以同时到达（见豆包 prompt 示例2）：

input1：[ {"query":"放点音乐", ...}, {"advisor":"车内温度35度，建议开启空调强力制冷和座椅通风"} ]

特点：
· 一句话建议，不是长知识文档
· 豆包不回应 advisor，直接面向用户行动
· 需要豆包自己解释"为什么这么做"

路径 B：knowledge（场景知识注入）

筛选 Agent 选中的完整场景知识，注入到 prompt 的 {{knowledge}} 槽位。

内容来自 xlsx 表格，包含：

使用时机 场景要点 执行逻辑 示例

特点：
· 结构化的长文档（数百到上千字）
· 持续存在于上下文中，直到被卸载
· 教豆包在这个场景里"怎么思考和行动"
· 你要写的多轮示例就在这里

两条路径的关系

advisor 是"临时提醒"——"嘿，温度高了你注意一下"。
knowledge 是"操作手册"——"遇到高温场景，按这个流程来处理"。

它们可以同时生效。比如 advisor 提醒"温度高了"的同时，{{knowledge}} 里已经有急速降温知识告诉豆包具体该怎么操作。

· 为什么并行两个工具？ 急速制冷和座椅通风互不依赖，并行最快。
· 知识怎么起作用的？ {{knowledge}} 里的急速降温知识告诉豆包：要急速制冷+配合座椅通风双管齐下。
· advisor 怎么融合的？ advisor 建议了同样的操作，豆包直接采纳但不提"advisor告诉我"，用自己的判断（视觉看到出汗）解释原因。

豆包收到 tool_feedback

output2：{ "talk_or_not": "no", "talk_content": "", "action_list": [] }

决策：沉默。 车控成功是预期结果，用户能感受到（风开了、通风开了），不需要多说。
这符合豆包 prompt 注意事项第4条："车控返回成功时不必再说话"。

这就是多轮的价值所在：
第一轮豆包开了急速制冷（大风量），用户不满意。
如果你的 {{knowledge}} 示例里覆盖了"风大不适"的分叉情况，豆包就知道该怎么调整。
如果没覆盖，豆包只能靠通用能力猜，大概率不够好。

14:06 · 豆包输出

output3：{ "talk_or_not": "yes", "talk_content": "风量降下来了，出风口帮你调成不直吹头部。座椅通风在给你散热，温度保持低一点慢慢降就好。", "action_list": [ {"action_id":"1","params":{"query":"前排风量调到2档，出风模式调为吹脚"},"tool_name":"vehicle_basic_control"} ] }

决策分析：

· 新增认知： 用户对强风敏感 → 不能只追求降温速度
· 策略调整： 降风量+改出风方向，保留座椅通风作为替代散热手段
· 知识支撑： 车辆设备知识里说"前排风量2档及以下较为安静"，所以调到2档
· 没有过度反应： 用户说"头疼"但没有持续追问，不需要升级到健康关怀场景

六、豆包 prompt 的 10 条行为规则（写示例必须遵守）
豆包 prompt 里有明确的注意事项和聊天风格规则。你写的多轮示例必须符合这些规则，否则示例和 prompt 会打架。

输出纪律

1 仅输出 JSON，不含任何额外文本

2 听不懂 / 不是对自己说的 → 不说话不调工具

3 闲聊类 → 只说话，不一定需要工具

4 tool_feedback 成功 → 不一定要说话（车控/音乐成功可沉默）

工具使用纪律

5 有串行依赖 → 不能一次性输出多个工具

6 工具反复失败 → 可暂时放弃

7 需要未知的车辆信息 → 先查 search_vehicle_status_info

对话风格纪律

8 不重复说相同的话

9 问了用户意见 → 没回之前不自作主张执行

10 语气：温柔高知女生，以短为主，偶尔长；不说"哇塞""超级"等幼态词

七、你的多轮示例最终长什么样？—— 格式模板
基于以上所有理解，xlsx I 列里的多轮示例应该长这样（以急速降温 DRV-01-01-04 为例）。

多轮示例模板（注入到 {{knowledge}} 后，豆包读到的就是这段内容）

多轮示例：用户反馈风太大，需要调整降温策略 【背景】车内温度急升至29度，空调刚切急速制冷，主驾通风已开。 input1： {"query":"风太大了吹得我头疼","speaker_name":"明骏","speaker_position":"主驾驶位"} output1： { "talk_or_not":"yes", "talk_content":"风量降下来了，出风口调成不直吹头部。座椅通风在散热，温度慢慢降就好。", "action_list":[{"action_id":"1","params":{"query":"前排风量调到2档，出风模式调为吹脚"},"tool_name":"vehicle_basic_control"}] } 决策分析：用户对大风量敏感，降风量+改方向，保留座椅通风替代。前排2档以下较安静。 input2： {"tool_feedback":"执行成功","tool_name":"vehicle_basic_control"} output2： { "talk_or_not":"no", "talk_content":"", "action_list":[] } 决策分析：车控成功可沉默，用户能感受到风量变化。 input3： {"query":"这样舒服多了","speaker_name":"明骏","speaker_position":"主驾驶位"} output3： { "talk_or_not":"yes", "talk_content":"嗯，等温度降到舒服了我帮你切回自动模式。", "action_list":[] } 决策分析：用户确认满意，告知后续计划（自动回切），体现闭环意识。不需要额外操作。

这个模板的关键特征

1. 格式跟豆包主 prompt 的示例完全一致：input/output JSON + 决策分析
2. 每一轮都有新的情境推进：风大不适 → 调整成功 → 用户确认 → 闭环
3. 决策分析解释"为什么"：不只是教豆包说什么，还教它为什么选这个策略
4. 遵守豆包的行为规则：工具成功时沉默、不重复说话、语气匹配

八、待确认的问题
以下几点我不确定，需要你确认。

问题 1：advisor 是谁发的？

豆包 prompt 示例2里 advisor 说"车内温度35度，建议开启空调强力制冷和座椅通风"。
这是筛选 Agent 发的，还是另一个独立的监控系统？还是就是你要写的 xlsx 知识内容的一部分？

问题 2：知识注入时机和 advisor 的先后关系

在实际系统中，是先注入 {{knowledge}} 再给豆包 input？还是 knowledge 和 input 同时到达？
这影响示例的设计——如果知识先于用户说话就已经注入，示例可以假设豆包已经"知道"场景要点。

问题 3：xlsx 表格里哪些列的内容最终进入 {{knowledge}}？

是所有列都注入（使用时机+场景要点+执行逻辑+执行逻辑简版）？还是只注入其中部分列？
两列"执行逻辑"和"执行逻辑.1"是什么关系——一个是完整版一个是简版？还是不同用途？
