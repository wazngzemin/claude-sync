## 1. 需求背景（这东西到底要干嘛，三个真实场景）

触发器现在压根不认"导航目的地""距离"这些概念，所以下面这类主动服务**一个都做不了**。本期就是把它补上。

| 场景 | 用户体验 | 背后要的能力 |
|---|---|---|
| 快到公司提醒拿咖啡 | 早上导航去公司，距公司 500m 时车主动说"快到啦，要不要提前点杯咖啡" | 监控"距目的地剩余距离"，跨过阈值触发一次 |
| 快到外婆家找充电桩 | 导航回外婆家，快到时主动找附近充电桩 | 同上 + 识别目的地是不是某个收藏地址 |
| 接续导航/到站提醒 | 快到目的地了，提醒收拾随身物品别落车上 | 距离/时间快到目的地时触发 |

**本期定位**：这些导航信号在清单里**本来就有定义（序号 119~133）**，只是之前被删了/没开放。本期作为**新增**补回开放——不是从零开发。

# 4. 🔴 接近 / 离开 / 到达 判定逻辑设计（刘杨点名要你想的核心，请重点看）

> 6/16 刘杨明确：「**这个根据现有的信号协议是做不了的，泽民你还得想想怎么做**」。本节就是这套设计。

## 4.0 先用一个完整场景走一遍（带数字，看完就懂为什么要专门设计）

**场景**：周一早 8:00，用户在家发起导航去公司（出发时距离 15km），产品想做"距公司 500m 提醒拿咖啡"。

| 时刻 | 导航状态 | 剩余距离 | 触发器应该做什么 |
|---|---|---|---|
| 8:00:00 | 未发起 → 导航中 | 15km | 触发【发起导航】 |
| 8:00–8:25 | 导航中 | 15km → 501m | 一路递减，**都 >500m，不触发** |
| 8:25:30 | 导航中 | **500m（第一次 ≤500m）** | 触发【接近公司】→ 提醒拿咖啡（**只这一次**） |
| 8:25:31 起 | 导航中 | 499m、498m… 50m | **都不再触发**（已经触发过了） |
| 8:30:00 | 导航中 | ≈0m **且** 档位 → P | 触发【到达公司】 |
| 18:00 离开 | — | 0m → >500m | 触发【离开公司】 |

**这张表暴露了核心难点**：「接近」**不等于**「距离 < 500m」。要是直接配"距离<500m 就触发"，那从 500m 一路开到公司（499m、498m…50m）**每次信号刷新都满足"<500m"，会触发几十上百次**，用户被"该拿咖啡了"刷屏。所以必须是「**从远到近、第一次跨过 500m 线 → 触发一次 → 然后闭嘴**」——这就是 one shot，也正是现有协议给不了的东西。

## 4.1 为什么现有信号协议直接做不了（说给刘杨听）

- 现有协议判断的是「**当前值满不满足**」（距离<500m？满足/不满足），它是**无记忆**的——每次刷新独立判断，不知道"上一次是不是已经触发过了"。
- 而「接近」要的是「**距离从 >500m 变成 ≤500m 的那一个瞬间**」，这需要三样现有协议没有的东西：① **记住**上一次的距离/状态（有状态）；② 判断**方向**（在变近还是变远）；③ 触发后**标记"已触发、别再发"**。
- 所以这事现有协议做不了，要么给协议**加一种"过程型/状态型判定"**，要么**端上报时就把"接近事件"算好了**再丢给触发器（两条路见 §4.4）。

## 4.2 判定规则 + 伪代码（给每个 POI 维护一个小状态机）

给每个被监控的目的地/途经点维护一个状态：`远 → 已接近 → 已到达 → 已离开`。每次"剩余距离"刷新时跑一遍：

```text
state = 该POI当前状态           # 初始：远
d     = 当前剩余距离

if state == 远  and  d <= 接近阈值(500m):
    触发【接近】;  state = 已接近      # ← one shot 关键：状态一变，接近就不会再触发
if state == 已接近  and  d <= 到达阈值(50m)  and  档位 变更为 P:
    触发【到达】;  state = 已到达
if state in (已接近, 已到达)  and  d > 离开阈值(500m):
    触发【离开】;  state = 已离开
```

| 事件 | 判定规则 | 触发类型 | 说明 |
|---|---|---|---|
| **接近** | `远` 态下，剩余距离**第一次 ≤ 接近阈值**（如 500m） | one shot | 施佳杰：从远到近到某阈值"**只触发一次**" |
| **到达** | `已接近` 态下，剩余距离 ≤ 到达阈值（≈0/50m）**且** 档位变更为 P | one shot | 组合条件 |
| **离开** | `已接近/已到达` 态下，剩余距离 **> 离开阈值** | one shot / 循环 | 反向判定 |

## 4.3 必须处理的三个坑（每个给具体例子）

**坑 1 · 防抖（最容易踩）**
- 具体：用户在距公司 500m 的路口**等红灯**，GPS 抖动让剩余距离在 498m ↔ 502m 反复跳。没防抖的话，每跳进 ≤500m 就触发一次"接近"，一个红灯 1 分钟能触发好几次。
- 处理：① 状态机天然防一部分（触发后进入"已接近"就不再触发接近）；② 再加**缓冲死区**——`<480m 才算接近、>520m 才算离开`，中间 480~520m 这段不动作，避免边界抖动。

**坑 2 · 阈值谁定 + 要不要展示**
- 具体：500m 这个数谁拍？不同场景还不一样（加油站可能 2km 就该提醒、咖啡 500m 才合适）。要不要给用户看"距公司 500m"？
- 处理建议：阈值**按场景由产品配**，给个默认值（待定）；是否展示给用户也由产品定。

**坑 3 · 立体空间 corner case（施佳杰提）**
- 具体：你家在高架旁，导航得绕到下一个出口再绕回来——**直线距离只有 300m，但导航沿路线剩余距离 2km**。如果用直线距离算，会提前误触发"接近"。
- 处理：一律用「**导航剩余距离（沿路线）**」，不用直线距离。

## 4.4 给刘杨的核心问题（请你拍方案 + 工期）

「剩余距离单调递减到阈值的 one shot 事件」到底怎么落地，两条路，请你评估：

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **A · 触发器侧加过程型判定** | 状态机放触发器，触发器存每个 POI 的"远/已接近/已到达"状态 | 统一、其它距离类需求可复用；端上不用各写一遍 | 触发器要存状态、改动相对大 |
| **B · 端上报算好事件** | 端上报时就算出"接近事件"，触发器只收事件、不算 | 触发器不用改 | 逻辑散在端上，多端要各写一遍、难统一 |

> 我（产品）倾向 **方案 A**（统一、可复用），但**最终听你的**——请你评估哪条可行、工期多少。这是本需求**最关键的卡点**。
