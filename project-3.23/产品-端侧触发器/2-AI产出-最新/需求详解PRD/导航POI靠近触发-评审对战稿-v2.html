§3 技术方案 · 5分钟翻到PRD §3.1 核心原理——这是今天最核心的部分
§3.1 方案一句话：端侧定时问高德"我离这个POI多远"，拿到距离，自己判进出，状态变化时告诉云端。
翻到§3.1的四步架构流程图。用手指着四个框从左到右讲。
你们看这个四步架构图：

步骤①，云端理解意图+选点。用户说"快到外婆家提醒我打电话"，意图模型拆出条件和动作，AINAVI选点工具拿到外婆家的经纬度。思晴6.25确认这个工具可以复用。

步骤②，云端注册规则，下发一个围栏任务给端侧。告诉端侧：盯着这个坐标，500米范围，进了告诉我。

步骤③，端侧定时查距离。A类（POI是导航目的地）用导航沿路线距离；其他用高德SDK距离查询接口——就是俊龙6.25说的"POI详情信息去请求，有当前距离"。距离小于阈值，端侧自己判"进了"，上报一个事件给云端。坐标不出车，合规没问题。

步骤④，云端收到事件，引擎匹配，push给Advisor，Director说人话，TTS播报。这段和现有链路完全一样，不用改。
步骤③讲的时候，看一眼俊龙和仁杰。如果他们有反应（点头/皱眉），暂停让他们说。如果没反应，继续。
翻到§3.2的端侧围栏模块组件表。
§3.2 端侧围栏模块拆了5个组件——距离数据源（高德SDK）、围栏存储（本地持久化）、距离轮询器（定时查距离）、围栏状态机（判进出）、事件上报器（端→云）。

每个组件都不复杂。仁杰上次说过"围栏逻辑本身比较简单"——给坐标，建围栏，以一定频率查在不在。现在高德SDK帮你包装的那层下架了，我们自己做，但逻辑是一样的。
翻到§3.3的距离获取方式对比表。
§3.3 距离获取有两种方式，按场景自动选：

第一种，导航沿路线距离——导航中且POI是目的地/途经点，精度高，高架不误触发。
第二种，高德SDK距离查询——通用兜底，任何场景都能用，精度中等（直线距离）。

优先用第一种，不行再用第二种。如果第二种也不行——比如弱网——端侧自己用GPS坐标算直线距离。三级降级。
快速翻一眼§3.4的方案选型对比表。
§3.4 方案选型对比，我就不一行行读了，大家会后看。结论是三列里我们选第一列——端侧自建围栏。坐标不出车，覆盖全场景，开发量可控。云端围栏已排除，仅用导航距离覆盖不全。这个结论是6.25会上大家一起拍的。
翻到§3.5的高德SDK接口详细能力待确认表。
§3.5 这个表是今天需要和俊龙对齐的。高德SDK距离查询接口的几个关键问题——调用频率上限、参数格式、返回内容是直线距离还是路线距离、非导航状态能不能用。

最关键的是QPS。我设计的是每秒查一次。如果SDK有限频，比如5秒一次也能接受——60km/h的车5秒走85米，不会错过480m的阈值。但如果限到30秒一次那就很紧了。俊龙，这个你帮确认一下？
这里就是要点名让俊龙说话。他如果当场能给结论最好。如果说"需要回去确认"，接着说"那这周内能给我吗？这是端侧围栏设计的前置"。
§4 核心难点 · 2分钟翻到PRD §4.1 数字走查表格
§4.1 为什么不能直接配"距离小于500m"？看这个表——用户导航去公司，从15km一路开过来，到500m的时候第一次触发没问题。但是499、498、497……一路刷屏到停车，触发几十上百次。用户被"该拿咖啡了"刷死了。

所以必须是：从远到近、第一次跨过500m → 触发一次 → 然后闭嘴。这叫one_shot。
翻到§4.2的围栏状态机图。
§4.2 怎么做到one_shot？给每个POI维护一个状态机：远 → 已接近 → 已到达 → 已离开。

距离≤480m → 状态从"远"翻到"已接近"，产出fence_enter事件。翻了就不回来了——478、477……都不再触发，这就是one_shot。

距离≤50m且挂P档 → "已到达"。距离>520m → "已离开"，围栏销毁。
翻到§4.3的三个坑。
§4.3 三个必须处理的坑：

坑1防抖——GPS在500m附近抖动498↔502m，所以不用500m做阈值，用480/520死区。只有≤480才算进，>520才算出，中间40m不做事。
坑2阈值——不同场景阈值不同，产品配，给默认值。
坑3立体空间——高架旁直线300m但沿路线2km。A类用导航路线距离不受影响；B类用直线距离，可能提前触发，但可以接受。
§5 A/B两类Case · 1分钟翻到PRD §5 对比卡片
§5 所有"快到XX提醒我"的case分两类：

A类——POI就是导航目的地/途经点，导航引擎已经在算到它的沿路线距离，直接用。精度高，高架不误触发。
B类——POI不在导航路线上，或者没开导航。用高德SDK距离查询，直线距离。

v4一个关键变化——B类以前是卡点，需要"动态插途经点"（仁杰之前说需要确认能不能做）。现在端侧围栏天然支持任意POI坐标，B类不再是卡点了。

陈俊龙 / 宫丽萍 · 地图团队
高德SDK距离查询接口 · 整个方案最核心的依赖方 · 你在讲§3.5的时候就应该直接cue他
陈俊龙"每秒调一次距离查询接口，QPS顶得住吗？有频率限制的"
这个确实需要你帮我确认一下QPS上限。我目前设计的是1Hz——每秒查一次。但如果SDK限频，比如5秒一次也能接受。60km/h的车5秒走85米，不会错过480米的阈值。如果限到30秒一次那确实就很紧了。你那边实际的限制是多少？
60km/h ≈ 17m/s → 5秒=85m → 在死区范围(40m)外，可以接受
120km/h ≈ 33m/s → 5秒=167m → 仍在合理范围，但精度下降
如果限频30秒 → 120km/h走1000m → 可能跳过整个围栏 → 方案不可用
陈俊龙"距离查询返回的是直线距离还是路线距离？"
你6.25会上说过"POI详情信息去请求，有当前距离和预计到达时间"。如果返回的有路线距离那最好，直线距离也够用——对于"快到某地"这种场景，直线500米已经是"确实近了"。高架旁的误触发是已知限制，但"提醒打电话"这种轻量场景下可以接受。
陈俊龙"非导航状态下这个接口能用吗？"
你6.25也确认过——"POI详情请求"不依赖导航引擎，非导航状态也能调。我再跟你确认一下：非导航时返回的距离，是直线距离对吧？
宫丽萍"围栏SDK都下架了，距离查询接口不会也下架吧？"
围栏是一个独立的服务模块，标注"已下架"。距离查询是POI详情接口的一部分，属于基础能力——和围栏不是一回事。不过这个风险确实要考虑，俊龙你帮看一下距离查询接口在高德auto SDK roadmap上的状态？
陈俊龙/汪帅"能订阅吗？还是只能被动查？"（6.25汪帅问过）
如果能支持订阅那更好——端侧不用自己轮询，省资源。但目前方案按"被动查询"设计的，也能跑。所以这个是P1优化项，不是阻塞项。能订阅，后续切过去。

这个需要你定。两种路径：复用现有端云长连接（如果有的话），或者走推送通道。核心需求就是云端能把一个JSON包下发到端侧指定模块。你看哪种现有通道最合适？
汪帅"端侧→云端的事件上报走什么链路？"
和上一个问题成对。上报量很小——整个过程就1-2次（进入一次、离开一次），不是每秒上报。看能不能复用现有的端→云信号上报链路？
汪帅"有必要绕到云端吗？端侧直接播TTS不行吗？"（6.25他问过）
你6.25提过这个——"端上其实也能播出来的，可以不经过Director"。纯播报确实可以。但用户说"快到公司帮我找充电桩"——这种需要Director调工具的，端侧做不了。而且走Director能保证统一的话术风格、能结合Context做个性化。所以端侧管判定，播报还是走云端。
汪帅6.25原话（00:48:18）："端上其实也能做一些简单的触发器...我们端上其实也能播出来的，可以不经过Director"
仁杰"坐标不出车你怎么做到的？"
端侧自己调高德SDK查距离，拿到一个数字（比如479米），自己判"进了"——然后上报给云端的只有4个字段：task_id、event_type、poi_id、timestamp。不传坐标、不传距离。云端拿task_id匹配注册规则就够了。
仁杰"多围栏并发怎么办？用户同时设了3个POI"
3个围栏就是每秒3次SDK距离查询。如果SDK没有QPS限制，这个量级很小。如果有限制——比如只允许1Hz——那3个围栏就是每秒轮流查一个，3秒一轮。60km/h走3秒=50米，在死区范围内还是能接受。具体数字等俊龙确认QPS后再定。
汪帅"630来不来得及？"
630是不是能上取决于两件事：一是俊龙本周确认SDK接口能力，二是你们评估围栏模块工期。今天先对齐方案可行性，工期评估我们单独拉——你看SDK确认后两个工作日内能给我工期评估吗？

如果工期紧，我有降级方案：630先上A类最小集（导航目的地距离），用导航引擎现有距离字段，不建围栏模块。B类后面迭代。
刘杨 · 触发器引擎
引擎适配 · v4对他是利好——不用开发云端状态机了
刘杨"我这边需要做什么改动？"
v4对你是好消息——不需要开发云端状态机了。判定下沉到端侧，你这边只需要做引擎适配：支持fence_event类信号的场景匹配。收到端侧上报的fence_enter事件后做15-11 Callback路由，push给Advisor。和现有事件匹配逻辑一样，只是多一种事件类型。
刘杨"fence_event和现有信号上报有什么区别？"
现有信号是端侧每秒上报原始数据，你这边要读数据做判断。fence_event是端侧已经判完了，直接告诉你"进了"——你只需要匹配task_id、检查event_type是不是fence_enter，命中了就走Callback。工作量比云端状态机方案小很多。
施佳杰 · 弋途Advisor
Advisor订阅fence事件 + 生成advice
施佳杰"Advisor需要订阅一种新的事件类型？改动大吗？"
对，新增订阅fence_enter事件。payload里有poi_name和action（比如"提醒打电话"），你拿这两个字段组装advice就行。和现在订阅其他事件的逻辑一样，不是新模式。
施佳杰"B类Case用户没开导航，Context里导航信息缺失怎么办？"
没影响。Advisor拿到的是fence_enter事件的payload——里面有poi_name和action，这是生成advice的核心信息。导航状态是Context的一部分，有就用、没有Director也能推理。"快到外婆家了，提醒打电话"这种场景，有没有导航信息不影响话术。
思晴(Jojo) / 李明骏 · AINAVI + 记忆模块
选点工具返回经纬度 · 阻塞项 · 这个必须在会上拿到结论
思晴"AINAVI选点工具加经纬度需要改什么？"
你6.25确认过AINAVI选点工具可以复用。现在需要在返回字段里加上lat和lng两个字段。具体看经纬度存在哪——如果记忆模块里有，直接取；如果没有，用poiid调地图接口换坐标。这块需要和明骏确认：记忆模块里存经纬度会不会影响现有功能。
李明骏"记忆模块加经纬度字段会影响现有功能吗？"
这个需要你评估。两种路径：①记忆模块的POI记录直接带经纬度（改存储结构），②不改记忆模块，选点时用poiid实时查地图接口换坐标。第二种对你零改动，但多一次网络调用。你看哪种更合适？
韩天亮 / 任军 · 信号库 + 运算符
前置依赖 · 需要先完成才能开工
韩天亮"fence_event信号编号怎么定？"
注册一个fence_event主信号，下面3个子类型：fence_enter、fence_arrive、fence_leave。另外payload里要支持task_id字段——匹配围栏任务的唯一标识。和现有nav_event信号格式类似。你看能不能这周先注册？这是引擎适配的前置。

需要确认现有eq运算符是否已经支持对event_type和task_id做字符串匹配。如果已经支持字符串eq，不用改；如果只支持数值，需要扩展。你帮确认一下？

全场高频挑战问题（任何人都可能问的）

任何人"为什么不直接在云端做围栏判定？不用端侧改那么多"
会上讨论过。仁杰原话："走端上比较合规，代价也比较小。如果把围栏放在云端，需要把坐标出车，合规可能会麻烦"。帅哥原话："难度指数级"。

云端围栏意味着端侧每秒上报GPS坐标到云端——合规风险、网络依赖、上报量都是问题。端侧自建围栏，坐标不出车，判完了只上报一个事件。这是6.25全员评审的结论。
仁杰6.25（00:43:42）："这个功能走端上比较合规，代价也比较小。如果把围栏放在云端...需要把坐标出车，合规可能会麻烦"
汪帅6.25（00:44:39）："难度就指数级的了"
任何人"高架旁误触发怎么办？"
两层处理：第一，A类（POI是导航目的地）用导航沿路线距离，高架不会误触发。第二，B类用直线距离，高架旁确实可能触发。但想一下场景——"路过迪士尼提醒我拍照"，高架旁直线500米内确实离得很近了，误触发一次也不是灾难。真要精确，后续加Z轴高度过滤。本期先不做。
任何人"如果高德SDK距离查询接口也不可用怎么办？"
有兜底方案：端侧自己用GPS坐标+Haversine公式算直线距离。仁杰说过"围栏逻辑本身比较简单"——核心就是拿两个坐标算距离，和阈值比大小。没有SDK包装，自己算也行。精度降低但满足围栏判定需求。
任何人"v3到v4变化太大了，方案稳不稳？会不会v5又推翻？"
核心架构没变——还是"云端下发任务、端侧判定、事件上报"这个四步模型。变的只是端侧判定的数据来源：从"围栏SDK"换成"距离查询接口+自建状态机"。这次改方案是因为高德围栏SDK下架这个外部变化，不是我们推翻自己。新方案反而更稳——不依赖第三方服务的开关，自己控距离查询+自建状态机，可控性更强。
任何人"这个需求630上不了怎么办？"
有降级方案。630最小集 = A类（导航目的地距离）+ B类收藏地址。只需要端侧围栏模块 + AINAVI选点 + 信号编号。完整版（包含任意POI的B类）还需要高德SDK距离查询确认，如果来不及，后面迭代。

但今天先不聊降级，先确认技术可行性和各方工期。如果方案没问题，工期单独对。
