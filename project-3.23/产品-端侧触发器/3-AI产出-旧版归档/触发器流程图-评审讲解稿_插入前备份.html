开场 + 怎么读图

各位好。今天讲的是这张「端云协同触发器」的全模块流程图。我先把范围说死啊：我讲的就是"一句话、或者一个信号进了系统，云端和车端怎么分工、一步步把它办了的"。VLM 算法、大脑怎么推理、信号怎么采，这三样今天不讲，也不归触发器。

骨架你先记三个词：两个入口（用户说话①、车自己冒信号③）、一个出口（用户听到看到㉙）、四条路径。所有的路最后都汇到右边那个大脑 ⑲⑳。线我按业务分了色，加粗的那种线就是跨端云、要过网络的，弱网会卡。

① 简单车控② 对话③ 条件任务④ 主动服务信号底座

★ 核心

云端 vs 车端：这张图最重要的一条分工线

先说结论：云端负责"登记 + 裁决"，车端负责"判断 + 执行"，绝不能两边都判。为什么这么分？就三条理由：定时器得放本地（不然车一进隧道断网就废了）、弱网底下也得本地能闭环、复杂的冲突仲裁得在云端用语义判，车端没那个脑子。落到具体环节上看：

环节云端做（登记+裁决）车端做（判断+执行）
注册云 规则注册、校验、编译成协议（13/15-1~3）。车端不吃自然语言。端 只接收已校验规则、本地落地（24-1）。
判定云 复杂条件、语义、跨任务仲裁（15-4~9）。端 简单条件、定时、弱网本地条件（24-3~7）。同条件不两边判。
执行云 需大脑二次推理的复杂动作回 Director（20）。端 简单车控/TTS/弱网闭环本地直接做（24-12/7）。
结果云 从 Context 统一消费（21）。端 写信号库→上报 Context，不直推 Advisor（24-11）。
速览

四条路径（每条一段，标清在云还是在端）

云+端① 简单车控（蓝）· "打开车窗"

① 用户说→② ASR（车端）→上云 ⑨ 拒识判意图（云端）→甩回端 ⑤ 白名单→⑥命中→⑦ 车端本地执行→⑧ 生效。意图在云判，其余全压车端，毫秒级、弱网也能用。

云② 对话/复杂（绿）· "我好热"

⑨ 判复杂→L5 直达 ⑲ 汇入→⑳ 推理（全在云端，要结合车内温度+记忆）→扇出：L29 talk、L30 动作 下到车端落地，L27/L32 工具调用回炉。

云③ 条件/定时任务（橙）· "超25度开空调"

⑨判条件话→⑩意图翻成协议（云端）→⑬后台注册场景→⑮引擎判（云端）；信号线 ③→④→L9→⑫→⑮；命中 L21 通知大脑执行。注册一定在云，判定看分类（15-9）走云还是下发端。

云+端④ 主动服务（紫）· "上车打招呼"

信号命中→⑯静态/⑰动态 Advisor（云端）出建议→大脑决定主动说/做；⑰还会 L22 下发视觉订阅给 ㉕ 感知器（车端执行 VLM）。判定与消费在云，感知执行在端。

★ 你点名要细的

VLM 链路 · 掰到原子级（订阅什么→谁转化→谁调度谁执行）

VLM 这块我得单独拎出来讲。为啥？它跟温度、车速那种普通信号不一样，也不是叫一下就完事的普通工具。它是个视觉感知任务，得调摄像头、调模型、管超时、管互斥、还得管结果状态。我就拿"帮我盯着副驾有没有睡着"这个例子，一步一步走，每一步我都点名是哪个原子、它拿到什么、又转出去什么。

17云注册"视觉订阅"

订阅的是什么？不是图、不是工具调用，是一个视觉目标：订阅对象=副驾、识别目标=是否睡着、触发语义=从"没睡"变"睡着"、执行方式=周期观察、优先级、超时、能否被抢占。17 自己不调 VLM，只把"要盯什么"交给触发器。

15-1云注册校验

查 iCanDo：这车有副驾摄像头吗？VLM 能识别"睡着"吗？当前场景允许注册吗？能→放行（select/create）。

★15-5云转化就在这个原子：把"人话目标"编译成 VLM task

转化的是什么？把"副驾是否睡着"这句自然语言，翻译成机器可执行的视觉任务字段：

camera_group=CABIN · seat=passenger · query=是否睡着 · mode=state · interval · timeout · priority · mutex_group

订阅进来的是"人话目标"，15-5 转出去的是"VLM 运行规格"。这就是你说没讲清的那个"转化"。

15-9 → 15-12云分类为 VLM → 打包下发

15-9 判定不能留云端（云端没实时摄像头）、也不能只写 Context（还没结果）→ 分类成 edge_vlm_task；15-12 打包成端侧 24 能收的规则包（task_id/camera_group/query/interval/timeout/priority/mutex_group/expire_at/callback_policy）下发（L14）。

24-1 → 24-7端接收 → 定摄像头资源

24-1 端侧落地、校验版本/有效期/车型；24-2 持久化（短时断网不丢）；24-7 资源仲裁：CABIN 摄像头被更高优占了吗？被占就抢占让路。

★24-8端这就是"24 调度 25"：VLM 调度

24-8 把规则包转成 25 的运行指令，告诉 25：看哪个摄像头（CABIN）、看什么（副驾是否睡着）、跑多久、超时多久、能否被抢占。

25 → 24-9/10/11端执行 → 五态 → 阈值穿越 → 回写

25 感知器拉图→调 VLM/VQA→出结果，先交 24-9 不直接给 17；24-9 归五态（success/empty/failed/timeout/interrupted，timeout 不写无异常）；24-10 只在 false→true 那一下生成事件；24-11 写信号库+上报 Context，17/19/20 从 Context 消费。

◆ VLM 总链（评审照念）
17 视觉订阅 → 15-1 校验 → 15-5 编译成 VLM task → 15-9 分类 edge_vlm_task → 15-12 下发 → 24-1 接收 → 24-7 定资源 → 24-8 调度 25 → 25 执行 → 24-9 五态 → 24-10 阈值穿越 → 24-11 回写 Context
★ 评审必问

这两个引擎是不是全由研发代码写？产品做什么？

对，15、24 最后都是研发写成引擎代码的（15 是云端的服务，24 是车端的能力）。但你别以为产品就只画个框。你看前面那两张表就明白了：引擎自己那套活，接口、存储、扫描、调度、状态机、日志、重试、性能，是研发写代码；可喂给引擎的那套规则、参数、清单、分级、边界、协议，得产品定。产品一行代码不写，可这些产品不定清楚，引擎就是个空壳，跑不起来。

所以产品在这两个引擎上要交的东西其实挺集中的，就 7 类。这 7 类一交，研发就能照着把代码写出来、排上期：

#产品交付物对应哪些原子 · 对应需求
1条件清单（信号/取值/编号/DNA支持）15-1/15-2 · 需求1（总源头，四方共用）
2运算符范围 + 每条件的触发时机15-4 · 需求5（变更为/状态变更重构）
3频控/冷却/优先级的具体数值15-6/15-7/15-8 · 抖动区间、cooldown、P0-P4 分级
4端云分类原则（哪些走云/端/VLM/兜底）15-9 · 需求9（架构级，你主导）
5互斥组划分 + 定时能力边界24-5/24-7 · 需求7（定时器）、CABIN/FRONT 互斥
6五态下游处理 + 端侧白名单24-9/24-12 · timeout不写无异常、哪些动作可端侧直执
7接口/DSL 协议 + 各规则包字段15-12→24-1、24-8→25、24-11→Context · 需求12
跟老板就一句话：这两个引擎研发能把架子搭起来，但真正让它跑起来的，是产品交的这 7 类东西。缺一类，引擎就有一块是死的，也排不上研发的期。
收口

异常 · 降级 · 待拍板

问题结论 + 落到哪个原子
弱网怎么办已下发的定时/白名单任务，24 本地履约（24-5/24-12）；没下发或需复杂语义的，不能假装执行成功。
VLM 异常25 失败→failed、超时→timeout；24-9 归一；24-11 把 empty/failed/timeout 都写回。timeout 不写无异常、failed 不触发主动服务、empty 是有效"未发现"。
冲突云端 15-8 做业务优先级+语义互斥；端侧 24-7 只做本地资源抢占（摄像头/推理），不做复杂语义仲裁。
端侧能执行啥需白名单拍板：TTS/卡片/AI Push/低风险车控可端侧直执（24-12）；涉安全/确认/复杂语义必回 Director。
接口/DSL15-4 的 range/持续时长/group/任意位置车门/POI、15-5 的 VLM task 字段、15-12→24-1 规则包、24-8→25 运行指令、24-11→Context 事件，这些都得明确是一期支持还是二期支持⚠。

结语

这张图评审通过的关键，不是大家认可"要做触发器"，而是认可：① 端云边界（谁判谁执行）② 每个原子干什么 ③ 研发写代码 vs 产品交 7 类 ④ 异常降级 ⑤ 跨团队接口。否则图画得再完整，也落不到研发排期。

三铁律收尾：① 触发器只读信号库；② 云端注册裁决、车端判断执行，不两边都判；③ 触发器只喊一嗓子，复杂决策回大脑。
