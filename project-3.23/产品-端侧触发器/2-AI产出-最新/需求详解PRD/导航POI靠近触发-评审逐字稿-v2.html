开场：今天评什么

用户说"快到外婆家提醒我打电话"——系统能做吗？现在不能。触发器不认"目的地""剩余距离"这些概念，注册不了、监测不了、触发不了。今天评审的就是怎么把这个能力补上。

我用这张流程图把整条链路讲一遍。大家不用看 PRD，看这张图就行。讲完之后我说一下 B 方案的难点、和传统地理围栏的区别，然后我们过分工和待确认项。

整条链路分两段。第一段叫注册——用户说完那句话，系统在 10 秒之内理解意图、注册条件、下发端侧。第二段叫触发——可能半小时后，车真的快到外婆家了，端侧算出"接近"事件，回传云端，Advisor 决策，播报给用户。

我先走第一段。

第一段 · 注册链路（从用户开口到系统记住）

对着图，从左下角开始，跟着箭头走

① 用户语音 query（图左下角）

用户坐在车里说了一句"快到外婆家提醒我打电话"。

端侧拿到的原始数据：语音 + speaker_name（谁说的）+ position（哪个座位说的）+ timestamp。

① →② ASR 转文字

语音到了端侧 ASR，转成文字："快到外婆家提醒我打电话"。然后这段文字上传云端。

这一步没什么特别的，所有语音交互都走这条路，不细说。

文字上到云端 → ⑳ Director 推理（图右侧中间）

Director 拿到这句话，理解意图：条件是"接近外婆家"，动作是"提醒打电话"。

Director 判断——这是一个"设定条件、等条件满足了再执行"的需求，不是现在马上要干的事。所以它不会直接播报，而是去调工具注册一个触发器。

⑳ Director → 调用 ㉒ 工具库（图最右侧）里的触发器注册工具

Director 从工具库里挑出"条件任务注册工具"，发起注册请求。

这里有个岔路——A 类和 B 类 Case 走法不一样：

A 类（"快到目的地提醒"）：目的地就是导航的目标，地址现成的，直接往下走。

B 类（"快到外婆家提醒"）：外婆家不在导航路线上。这时候系统得先做一件事——把"外婆家"解析成一个真实地址。怎么解析？图右侧画了三个来源：收藏地址（用户存的家/公司/外婆家）、记忆（历史对话里提到过的地址）、POI 库（地图搜索）。匹配上了，拿到经纬度或地址字符串，再动态插入导航的途经点，这样地图才会开始算到它的剩余距离。

@导航产品 @宫丽萍 ——地图能不能动态插途经点？这个能力直接决定 B 类 Case 做不做得了。做不了的话，本期 B 类就得砍。
注册请求到达 ⑬ 规则配置平台（图左上角）

注册的东西长这样：

event_type eq "approach" AND poi_id eq "外婆家"
Action: callback → 动态Advisor
Policy: one_shot

翻译成人话：当"接近外婆家"这个事件发生时，通知动态 Advisor，只通知一次。

配置平台把这条规则生成条件 DSL，传给云端触发器。

⑬ 配置平台 → 注册场景 + 条件 DSL → ⑮ 云端触发器

云端触发器收到条件 DSL，做两件事：

  1. 把这条规则存下来，等着后续事件来匹配。
  2. 把需要端侧盯的信息（POI 地址、阈值 500m、状态机初始状态"远"）下发到端侧，端侧持久化。

@刘杨 ——这一步你这边需要支持 event_type 类信号的场景匹配。B 方案下引擎不做状态机，你这边改动相对小，主要是收事件、匹配、推 Advisor 这条链路要通。

条件下发端侧，注册完成

到这里，第一段结束。用户说完话大概 10 秒，系统已经记住了："快到外婆家就提醒打电话"。端侧持久化了规则，状态机初始化为"远"，开始盯导航剩余距离。

大家看流程图上这条注册链路有没有问题？有的话我们先对齐，没有我接着讲触发链路。

第二段 · 触发链路（真的快到了，到播报出来）

半小时后，用户正在开车接近外婆家。回到图的左下角。

硬件/导航 → 信号入库 → ④ 端侧信号库（图下方中间）

导航一直在跑，地图 SDK 持续输出"到目的地的剩余距离"。这个数字写进端侧信号库，标准化存成快照。

图最底下那行标注写了：distance: 15000 → 479，整数米。意思是出发时 15000 米（15 公里），一路递减到 479 米。

@导航产品 @宫丽萍 ——这里有一个确认项：地图 SDK 给的剩余距离，格式到底是整数米（479），还是带单位的字符串（"479m"）？如果是字符串，得有人剥单位。

④ 端侧信号库 → 端侧 POI 状态机（B 方案核心）

这是整条链路最关键的地方，我展开讲。

状态机给每个被监控的 POI 记一个状态，只有四种：远 → 已接近 → 已到达 → 已离开，单向翻，翻过去就不回来。

我用数字走一遍。

距离刷新状态机里发生了什么
d = 15000m, 10000m, …, 501mstate=远，d 都大于 480m，条件不满足，不动。
d 从 501 变到 479mstate=远 且 d≤480 → 条件满足！触发 approach 事件。
state 立刻翻成 已接近。
d = 478m, 477m, …, 50mstate=已接近（不是"远"了），接近判定进不去，不会再触发。
d ≤ 50m 且 档位 → P触发 arrive（到达）事件，state 翻成 已到达。
d > 520m（开走了）触发 leave（离开）事件，state 翻成 已离开。
为什么判定线是 480 不是 500？——防抖。

用户在距外婆家 500 米的路口等红灯，GPS 在 498 和 502 之间来回抖。如果阈值卡在 500，每抖进来一次就触发一次"接近"。

所以设了死区：小于 480 才算接近，大于 520 才算离开，480 到 520 之间不动作。产品配的阈值是 500，但实际判定用死区值。
为什么用"沿路线距离"不用直线距离？

你家在高架旁边，直线距离 300 米，但导航得绕到下一个出口再绕回来，沿路线 2 公里。如果用直线距离，300 米就触发"接近"了——可你实际还要开 2 公里。误触发。

一律用导航给的沿路线剩余距离，不用直线。

这就是 one_shot 的全部原理。状态翻一格，接近事件只响一次，用户不会被刷屏。

@金龙 @汪帅 ——这套状态机逻辑跑在端侧，由你们实现。四个状态、三条判定规则、加上死区防抖和状态重置（导航取消/离车时清空），这就是 B 方案你们要做的核心。

端侧状态机 → 事件回传 → ⑫ 云端信号库（图左侧中间）

端侧产出 approach 事件后，同步到云端信号库。

上报的内容大概是：{ event_type: "approach", poi_id: "外婆家", timestamp: ... }

⑫ 云端信号库 → 匹配 → ⑭ 条件理解 & 解析 → ⑮ 云端触发器

云端触发器拿到 approach 事件，去匹配之前注册的条件——event_type eq "approach" AND poi_id eq "外婆家"。命中了。

@任军 ——这里需要 eq 运算符能匹配 event_type 这种字符串类型的信号。你确认一下引擎支不支持。

⑮ 云端触发器 → 注册成功，事件 push 给 Advisor → ⑰ 动态 Advisor

条件命中，触发器按注册时的 Action（callback → 动态 Advisor）把事件推给动态 Advisor。

推过去的 payload 里带着 poi_id、event_type、注册时绑定的用户动作（"提醒打电话"）。

⑰ 动态 Advisor（图右侧中上）

动态 Advisor 收到事件，干四件事（图上画了四行）：

  1. API 调状态检测——确认用户现在在不在车上、导航还在不在跑
  2. 分发/定时智能工具调用——看需不需要调额外工具（比如搜充电桩）
  3. 知道 4 级目标状态——看 Goal List 里有没有和这个 POI 相关的目标
  4. 使用 AI 记忆工具——调记忆看看"外婆家"有没有相关的上下文

最终组装出一条 advice（建议），推给 Director。

@施佳杰 ——这一步是你的。你要订阅 approach / pass_by 事件，编写 advice 脚本，验证 payload 格式。"接近外婆家"和"接近公司"触发的 advice 不一样，按 poi_id 分。

⑰ 动态 Advisor → 建议 → ⑳ Director 推理

Director 收到 Advisor 的建议，做最终决策：

  • talk_or_not：要不要说话？——要。
  • talk_content：说什么？——"快到外婆家了，要不要给外婆打个电话？"（带情绪/emoji）
  • action_list：要不要同时干别的？——看情况，可能还调个拨号工具。

Director 做决策的时候还会看 ㉑ Context（偏好/情境/车状态/记忆/目标）和 ⑱ Goal List，这两个不展开了，跟之前讲的触发器系统一样。

⑳ Director → 播报

Director 决策完，TTS 播报："快到外婆家了，要不要给外婆打个电话？"

整条链路走完了。

回顾一下整条链路经过的节点：

注册段： ① 用户语音 → ② ASR → ⑳ Director → ㉒ 工具库（注册工具） → ⑬ 配置平台 → ⑮ 云端触发器 → 下发端侧持久化

触发段： 硬件/导航 → ④ 端侧信号库 → 端侧状态机（approach 事件） → ⑫ 云端信号库 → ⑮ 云端触发器（匹配） → ⑰ 动态 Advisor → ⑳ Director → 播报

大家对着图看一下，整条链路还有没有漏的节点、不通的地方。有问题我们现在说。

B 方案的难点和我的顾虑

链路走完了，大家看到 B 方案的核心就是——状态机放在端侧跑。这个方案 630 来得及、引擎改动小，但有几个问题我得摆出来。

难点一：多端各写一遍

上汽一套端、赛力斯一套端。状态机跑在端侧，每个端都得实现一遍。逻辑一样，代码各写各的。后面产品要改死区从 480/520 改成 450/550，我得找每个端的人说一遍"你改一下"，漏了就出 bug。

如果是 A 方案（状态机放引擎），改一个地方全生效。但 A 方案改引擎，630 工期顶不上。

我的态度：本期先这么做，能跑起来。长期看端多了维护成本上来了，再往引擎迁。接口不变，改的只是"事件谁来算"。

难点二：B 类 Case 的生死线——动态插途经点

A 类（"快到目的地提醒"）没问题，目的地在导航里，距离现成的。

B 类（"快到外婆家提醒"）——外婆家不在导航路线上，地图没在算到它的距离。得先把它动态插入途经点，地图才开始算距离。

地图能不能动态插途经点？做不了，B 类 Case 本期直接砍。A 类不受影响。

@导航产品 @宫丽萍 ——这是我今天最需要你们确认的一件事。

难点三：地址匹配

用户说"快到外婆家"，系统里存的收藏地址是"上海市浦东新区XX路XX号"。导航目的地的 address 也是一串字。这两串字格式不一定一样。

匹配用字符串等值比对还是经纬度距离比对？定清楚了端侧才知道怎么写。

@导航产品 @金龙

难点四：状态重置

状态机翻到"已接近"就不回"远"了。那如果用户取消导航再重新导航去同一个地方呢？状态得清掉重来。什么时候重置——离车熄火？取消导航？重新发起导航？

@金龙 @汪帅 ——这个你们评估一下，定个策略。

说完四个难点，停一下。"大家觉得哪个最棘手？有方案的现在说，没方案的给我个什么时候能回复的时间。"

和传统地理围栏的区别

可能有人会说——这不就是地理围栏吗？手机上早就有，进入一个区域触发通知。

不一样。我说一下关键区别。

维度我们的方案（沿路线距离 + 状态机）传统地理围栏
距离怎么算 用导航给的沿路线剩余距离。高架上要绕出口再回来，导航会告诉你"还有 2 公里"。 画一个圆，用 GPS 直线距离判断你在不在圈里。
高架旁的情况 沿路线 2km，不触发。准。 直线 300m，进圈了，触发了。但你还得绕 2 公里。误触发。
前提条件 必须在导航状态下。没在导航，没有"沿路线距离"这个数。 不需要导航。GPS 在就行。
高德支持吗 导航剩余距离——支持，现成的。 地理围栏 API——宫丽萍已确认：高德不支持。

所以我们选沿路线距离这条路，两个原因：准（高架/隧道/绕行不会误触发）、高德有这个数据（围栏 API 没有，要做得自己搭）。

代价是必须在导航状态下才能用。纯兜风、没发起导航，这方案做不了。

如果 B 方案做不了，走围栏需要什么

退路先说清楚。如果大家评估下来觉得端侧状态机做不了，或者 630 来不及，唯一的退路是自己实现地理围栏。

围栏方案得做这些事，大家感受一下工作量：

要做的事谁做有多重
自己写围栏引擎 金龙/汪帅 高德不给围栏 API，得自己实现"GPS 点在多边形内"的判定，还要处理 GPS 漂移、采样频率低导致"飞过围栏"漏判、省电模式下的定位唤醒。比状态机重得多。
获取 POI 经纬度 导航产品 + 地图 "外婆家"得变成经纬度坐标才能画圈。收藏地址有经纬度吗？
定围栏大小 产品 加油站画 2 公里？咖啡画 500 米？每个场景都要定。圆形还是带状？
进出状态管理 端侧 围栏也得做"进去了不再触发"，本质还是一个状态机。省不掉。
高架误触发 端侧 + 产品 围栏用直线距离，高架旁 300 米就进圈了。要么忍，要么叠加海拔/路线修正，更复杂。

围栏不是更简单，是更重。状态机省不掉、高架问题还多了一个、围栏引擎还得自己写。

所以我的建议是：B 方案能做就做 B。围栏是最后的退路。

应答策略

金龙/汪帅："端侧状态机工期太紧了，630 来不及"

"那我们拆一下——A 类 Case（目的地/途经点）的状态机不复杂，四个状态三条规则。能不能先做 A 类，B 类排后面？A 类做出来，'快到公司提醒拿咖啡'这个主场景就跑了。"

有人："为什么不做地理围栏？"

"两个原因。高德不支持围栏 API，得自己从头写引擎，工期更大。而且围栏用直线距离，高架旁 300 米就误触发了。我们用沿路线距离不会有这个问题。"

有人："直接配'距离小于 500'不就行了？"

"不行。499、498、一直到 50，每帧都满足'小于 500'，触发器会连着响几十次。必须只响一次就闭嘴，这就是状态机要解决的。"

刘杨："引擎不改可以，但 event_type 匹配你给我定义清楚"

"PRD 事件定义表里写了——event_type 的值是 approach / arrive / leave / pass_by，字符串，eq 匹配。你看文档，有问题随时对。"

导航产品："动态插途经点做不了"

"那 B 类 Case 本期砍。A 类不受影响，目的地本来就在导航里，照做。B 类记个技术债，看地图能力什么时候到位。"

施佳杰："Advisor 怎么区分接近公司和接近外婆家？"

"事件 payload 里带 poi_id 和 address。按 poi_id 分，不同 POI 触发不同 advice。"

有人："这方案以后能不能迁到引擎统一做？"

"能。接口不变——端上报事件、引擎匹配。变的只是'事件谁来算'。本期端侧算，以后迁引擎算，下游都不用改。"

有人："没在导航怎么办？"

"做不了。没在导航就没有沿路线距离。纯兜风、没发起导航的场景，本方案就是做不了。这个是能力边界，不是缺陷。要做无导航的触发，得靠围栏或 VLM，那是另一个需求。"

收口：散会要拿到的 5 件事
