1. 背景

1.1 现状与痛点

触发器系统目前不认识"导航目的地""剩余距离"这些概念。用户说"快到外婆家提醒我打电话"，系统无法注册、无法监测、无法触发。整类"快到某地就主动服务"的场景一个都做不了。

这些导航信号在《触发器条件清单》里本来就有定义（序号 119~133），只是之前被删了/没开放。本期是把它们作为新增重新开放，并补上核心缺失的"接近/到达/离开"判定逻辑。

1.2 地图侧确认结论（2026-06-24 宫丽萍）

高德地图SDK没有现成的地理围栏API。但可以根据导航距离判断。
注意：围栏可以在端侧自建（用GPS坐标自行计算进出），只是高德不提供现成接口。
产品决策：优先走导航剩余距离方案（方式A：动态插途经点 → 复用距离判定），工作量最小、复用度最高。端侧自建围栏作为备选。

3. 两类Case的本质区别

看起来都是"快到XX提醒我"，但触发条件的性质完全不同，必须分开处理：

A类 · 目的地/途经点距离（能做）

  • POI就是当前导航的目的地或途经点
  • 导航引擎已经在实时算到它的剩余距离
  • 信号现成（124/128/130），只缺one_shot状态机
  • 例："离目的地还有1km提醒我拿咖啡"
  • 例："快到途经点帮我找停车位"

B类 · 靠近任意POI（有前置依赖）

  • POI不是当前导航目标（"外婆家"可能不在路线上）
  • 导航引擎没有在算到它的距离
  • 需要先把它变成导航系统认识的途经点
  • 然后问题退化为A类
  • 例："快到外婆家提醒我打电话"
  • 例："路过徐浦大桥提醒我右转"
核心洞察：既然高德不支持围栏，B类的唯一路径是把任意POI转成途经点 → 退化为A类。所以"地图SDK能否动态插入途经点"（仁杰确认中）是B类case的生死问题。

4. 核心难点：为什么不能直接配"距离<500m"

4.1 数字走查：一个真实场景

周一早8:00，用户导航去公司（出发距离15km），产品想做"距公司500m提醒拿咖啡"。

时刻剩余距离如果配"距离<500m"正确行为
8:00~8:2515km → 501m不触发 ✅不触发 ✅
8:25:30首次到500m触发 ✅触发一次 ✅
8:25:31499m触发（第2次）❌不触发 ✅
8:25:32498m触发（第3次）❌不触发 ✅
......一路刷屏到50m ❌安静 ✅
8:30:00≈0m，挂P档还在触发 ❌触发"到达" ✅
问题：从500m开到公司（499、498...50m），每帧都满足"<500m"，会触发几十上百次——用户被"该拿咖啡了"刷屏。所以必须是"从远到近、第一次跨过500m → 触发一次 → 然后闭嘴"——这就是 one_shot，也正是刘杨说"现有信号协议做不了"的原因。

4.2 现有协议为什么做不了

现有协议判断的是"当前值满不满足"（距离<500m？），它是无记忆的——每次刷新独立判断。而"接近"需要三样东西：

  1. 记住上一次的状态（有记忆）
  2. 判断方向（在变近还是变远）
  3. 触发后标记"已触发、别再发"（one_shot自锁）

4.3 解决方案：POI状态机

给每个被监控的目的地/途经点维护一个状态：远 → 已接近 → 已到达 → 已离开。

远
d ≤ 480m
→
触发【接近】
已接近
d ≤ 50m 且 挂P
→
触发【到达】
已到达
d > 520m
→
触发【离开】
已离开
state = 该POI当前状态           # 初始：远
d     = 当前剩余距离(沿路线·米)

if state == 远  and  d <= 接近阈值(480m):        # 死区下沿
    触发【接近】;  state = 已接近                  # one_shot：状态翻了就不再触发

if state == 已接近  and  d <= 到达阈值(50m)  and  档位变更为P:
    触发【到达】;  state = 已到达                  # 组合条件

if state in (已接近, 已到达)  and  d > 离开阈值(520m):
    触发【离开】;  state = 已离开                  # 反向判定

4.4 三个必须处理的坑

坑具体场景处理方案
坑1 · 防抖在距公司500m路口等红灯，GPS抖动498m↔502m反复跳缓冲死区：<480m才算接近、>520m才算离开，中间不动作
坑2 · 阈值不同场景阈值不同（加油站2km、咖啡500m）阈值按场景由产品配，给默认值；配置平台支持
坑3 · 立体空间高架旁直线300m，沿路线要绕2km一律用导航沿路线距离，禁用直线距离

6.3 Case 3：路过徐浦大桥 → 准备右转（B类·pass_by）

这个case完全不同——是"路过"不是"靠近目的地"，判定方式换了

"路过"和"靠近"是两回事，别搞混：

  • 靠近（approach） = 外婆家/公司是你的目的地，你一路开过去距离越来越近 → 用导航剩余距离判 → 状态机"远→已接近"
