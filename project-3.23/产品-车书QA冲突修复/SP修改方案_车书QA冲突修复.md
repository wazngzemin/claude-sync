# Director SP 修改方案 — 车书QA与车控冲突修复（基于实际FORNAX SP）

> 目标：让Director在收到 `{{car_function_simple_qa}}` 车书QA检索知识时，仍能正确判断"用户要执行操作"还是"用户在问怎么操作"，避免车控指令被QA知识的文本回答替代。
> 
> 所有修改均在FORNAX的Director SP中完成，不需要工程配合。

---

## 修改一：改 `{{car_function_simple_qa}}` 的section引导语（最关键）

**当前内容**：
```
## 基础车辆功能问答:
可能会用用上的一些基础车辆问答信息
<基础车辆功能问答>
{{car_function_simple_qa}}
<基础车辆功能问答>
```

**改为**（直接替换）：
```
## 基础车辆功能问答:
以下是从车辆手册中检索到的参考知识，描述了相关功能的手动操作方法和使用说明。使用原则：
- 当用户意图是直接执行操作（如"打开XX""关闭XX""调高XX"）或查询实时状态（如"现在时速多少""电量还剩多少"），必须优先调用对应工具（vehicle_basic_control或search_vehicle_status_info），不要用以下知识做文字回答
- 仅当用户明确在询问操作方法（如"怎么用""如何设置""步骤是什么"）时，才参考以下知识回答
- 以下知识可作为工具调用失败时的兜底说明
<基础车辆功能问答>
{{car_function_simple_qa}}
</基础车辆功能问答>
```

**为什么这是最关键的改动**：
- 原来的"可能会用上的一些基础车辆问答信息"太模糊，模型把它当成了"可以直接用来回答的答案"
- 新的引导语明确告诉模型：这是"参考知识"而非"答案"，并给出了三条清晰的使用规则
- 这个改动只有~100字，占用极少token，但直接改变了模型对这块知识的"理解框架"

**额外注意**：原SP中 `<基础车辆功能问答>` 的闭合标签写的是 `<基础车辆功能问答>` 而不是 `</基础车辆功能问答>`（缺少斜杠），建议顺手修复为 `</基础车辆功能问答>`。

**工作量**：5分钟

## 修改二：注意事项中追加第15条

**位置**：`### 注意事项` 的第14条之后

**追加内容**（直接复制粘贴到第14条后面）：

```
15. 【车控工具优先】当用户说出明确的执行指令（如"打开定速巡航""关闭车窗""把温度调到24度"）或查询实时状态（如"现在时速多少""剩余电量多少"），即使「基础车辆功能问答」中包含了该功能的手动操作说明，你也必须优先调用vehicle_basic_control或search_vehicle_status_info来执行。「基础车辆功能问答」中的知识仅在用户询问"怎么操作""如何使用"等方法性问题，或工具调用失败需要兜底时使用。
```

**为什么需要这条**：与修改一形成双保险。修改一是在知识注入处的"就近提醒"，修改二是在全局规则层的"强制约束"。模型会同时看到两处一致的指令，遵循概率大幅提升。

**工作量**：5分钟

## 修改三：追加专项示例（示例22）

**位置**：在现有示例的最后一个之后（示例21之后），追加示例22

**追加内容**（直接复制粘贴）：

```
---
### 示例22：车书知识存在时的工具优先决策（执行 vs 咨询 vs 状态查询）
覆盖工具：vehicle_basic_control、search_vehicle_status_info
思维范式：车控工具优先于知识回答、知识仅兜底或方法咨询时使用

<舱内视觉感知>
主驾驶：张先生，开车中
</舱内视觉感知>
<基础世界信息>星期二，上午，晴，26度</基础世界信息>
<用户记忆>无</用户记忆>

--- 场景A：执行型指令（应调工具，不用知识回答）---

input1：
{"query":"打开定速巡航","speaker_name":"张先生","speaker_position":"主驾驶位","timestamp":"2026-05-19T10:30:05Z","assistant_emotional_state":"平静","goal_list":"","env_info":"车速80km/h，定速巡航未开启"}
（此时「基础车辆功能问答」中包含：定速巡航开启方法：进入【车辆设置】-【辅助驾驶】，打开【自适应巡航辅助】，然后按方向盘PILOT按键激活。）
output1：
{
  "talk_or_not": "yes",
  "talk_content": "好的，正在为你开启定速巡航。",
  "emoji_id": "speaking",
  "action_list": [{"action_id": "1", "params": {"query": "打开定速巡航"}, "tool_name": "vehicle_basic_control"}]
}
**决策分析**：
- **为什么不用「基础车辆功能问答」的知识回答？** 用户说"打开定速巡航"是执行指令（带"打开"动作词），不是在问操作方法。虽然检索到了手动操作步骤，但用户期望的是语音直接控制。
- **判断标准**："打开""关闭""调高""开启"等动作词 = 执行 → 调vehicle_basic_control。

input2（工具失败时才用知识兜底）：
{"tool_feedback":"执行失败，当前车速不满足开启条件","tool_name":"vehicle_basic_control"}
output2：
{
  "talk_or_not": "yes",
  "talk_content": "现在没法直接开，车速可能还不够稳定。你也可以手动操作：进入车辆设置的辅助驾驶，打开自适应巡航，再按方向盘PILOT键。",
  "emoji_id": "speaking",
  "action_list": []
}
**决策分析**：工具失败后，才是「基础车辆功能问答」知识发挥作用的时候——作为兜底告诉用户手动操作方法。

--- 场景B：实时状态查询（应调工具查实时数据）---

input3：
{"query":"现在时速多少","speaker_name":"张先生","speaker_position":"主驾驶位","timestamp":"2026-05-19T10:32:05Z","assistant_emotional_state":"平静","goal_list":"","env_info":"车速80km/h"}
（此时「基础车辆功能问答」中包含：当前车速是仪表屏常显信息，非P挡时你可以直接在仪表屏查看。）
output3：
{
  "talk_or_not": "no",
  "talk_content": "",
  "emoji_id": "speaking",
  "action_list": [{"action_id": "1", "params": {"query": "查询当前车速"}, "tool_name": "search_vehicle_status_info"}]
}
**决策分析**：
- **为什么不用知识回答"看仪表盘"？** 用户问"现在时速多少"是查实时数据，不是问"在哪里看时速"。直接调工具返回具体数字才是正确响应。
- **判断标准**："XX多少""当前XX""剩余XX" = 查实时状态 → 调search_vehicle_status_info。

--- 场景C：操作方法咨询（这时才用知识回答）---

input4：
{"query":"定速巡航怎么用啊","speaker_name":"张先生","speaker_position":"主驾驶位","timestamp":"2026-05-19T10:35:05Z","assistant_emotional_state":"平静","goal_list":"","env_info":"车速80km/h"}
（此时「基础车辆功能问答」中包含：定速巡航开启方法：进入【车辆设置】-【辅助驾驶】，打开【自适应巡航辅助】，然后按方向盘PILOT按键激活。）
output4：
{
  "talk_or_not": "yes",
  "talk_content": "定速巡航可以这样开：进入车辆设置，找到辅助驾驶，打开自适应巡航辅助，然后按方向盘上PILOT键就行了。你也可以直接跟我说打开定速巡航，我帮你操作。",
  "emoji_id": "speaking",
  "action_list": []
}
**决策分析**：
- **为什么这次用知识回答？** 用户说"怎么用"是在咨询操作方法，不是要求执行。「基础车辆功能问答」中的手动操作步骤正好是答案。
- **附加价值**：回答末尾主动告知"也可以直接让我操作"，引导用户下次使用语音控制。
- **判断标准**："怎么""如何""步骤""方法" = 咨询 → 参考「基础车辆功能问答」回答。
```

**为什么这个示例设计有效**：
1. **三种场景对比**：执行型（调工具）、查询型（调工具）、咨询型（用知识回答），覆盖了所有冲突场景
2. **包含工具失败兜底**：展示了知识的正确使用场景（兜底），避免模型完全忽略知识
3. **明确判断标准**：每个场景都给出了"判断标准"，模型可以直接套用
4. **格式完全一致**：遵循现有SP的示例格式（包含timestamp、emoji_id、emotional_state等新字段）

**工作量**：20分钟

## 关于动态专业知识 `{{knowledge}}` 部分

泽民负责的 `{{knowledge}}` 模块当前不需要改动。原因：
1. 问题出在 `{{car_function_simple_qa}}`（车书QA知识），不在 `{{knowledge}}`（场景知识）
2. 场景知识的64条知识按角色/场景组织（EXP/DRV/COM/ENT/LIFE/EMO），结构不适合加这种通用的"知识使用元规则"
3. Knowledge Selector的触发条件难以精确匹配"所有可能涉及车控的query"
4. 直接在SP层面修改效果更好、更快、更可控

但如果后续需要，可以在 `{{knowledge}}` 中增加一条类似 `COM-XX-XX-XX` 的知识条目，描述"当车书知识和车控工具冲突时的处理逻辑"，作为额外的一层保障。
