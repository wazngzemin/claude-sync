# 场景支持走查会议纪要

## 一、背景与当前状态

### 场景 Planner 已暂时下架

**问题：** 场景 Planner 当前端到端耗时约 800ms，拖慢整个豆包响应链路，已被令平决定暂时下架。

**为什么：** Planner 每次调用需要向云端 LLM 发送 ~40K-50K tokens 的请求，场景编排类任务涉及多步推理和多工具调用，耗时无法满足车载实时交互要求。

**当前影响：** 所有需要 Planner 编排的场景功能暂不可用；行程规划功能尚未上线，后续会下放到下游工具。

### 618 前计划重新上线

计划在 618 之前重新上线场景支持。近期重点：
- 大量实车测试，收集 badcase
- 修复 skill 知识中的错误（包含车厂不具备的能力描述）
- 优化检索 SP 的准确性（当前场景命中率约 60-70%）

---

## 二、核心问题清单

### 问题 1：预置场景 vs Planner 编排的路径混乱

**问题是什么：** 部分场景（如离车保电、洗车模式、宠物模式等）本应直接调用预置场景应用（阈值/规则化），不需要 Planner 做编排。但当前 Planner 会"自己去 plan"，导致：
- 本该直接调用的功能，Planner 尝试自行拆解后失败
- 或者 Planner 下发了不存在的 function call 给方向靠（如"使用宠物模式"），方向靠没有该指令，调用失败

**为什么会这样：**
1. 方向靠（车控协议）中缺少这些预置场景的入口指令（如离车保电、宠物模式等车控指令未注册）
2. Skill 知识中写了车厂实际不具备的能力，误导 Planner 以为可以直接调用
3. Planner 分不清哪些走预置场景、哪些走自主编排

**正确架构应该是：**

```
用户语音
  ↓
Planner 判断
  ↓
┌─────────────────────────┐
│  function call 下发      │
├────────────┬────────────┤
│ SOA 车控调用 │ 预置场景调用 │
│（原子化功能） │（阈值/规则） │
└────────────┴────────────┘
```

Planner 分析后决定走哪条路：
- **原子化功能**：通过方向靠下发具体车控指令（如调空调温度、开关车窗）
- **预置场景**：直接调用已有的预置应用（如洗车模式、离车保电），不做编排

### 问题 2：Skill 知识质量差

**问题是什么：**
- Skill 中包含车厂实际不具备的功能描述，Planner 据此下发指令后必然失败
- Skill 描述不清晰，导致 Planner 对场景理解偏差（如观影模式的 skill 引导 Planner 去给车控下发"观影模式"指令，但车控没有这个指令）
- 部分场景（如通风模式）的触发条件过于复杂，Skill 未做好条件分支

**为什么：** Skill 是第一版粗写的，没有经过实车验证，也没有和车控协议做逐条对齐。

### 问题 3：Planner 回答前后矛盾

**问题是什么：** 多个 badcase 中 Planner 出现"一会支持一会不支持"的行为：
- 先说"帮你打开XXX模式"，随后又说"遇到小问题打不开"
- 先说可以做，然后说不支持，接着又说"我帮你手动模拟一个"，最后实际没执行
- 罗列了一堆要执行的操作（如车控调节列表），但实际没有调用工具/没有返回结果

**为什么：** Planner 在推理过程中发现 function call 失败或工具不存在，但没有良好的 fallback 策略，导致回复逻辑混乱。

### 问题 4：端状态信息利用不足

**问题是什么：**
- Planner 反问用户"是否处于P档"——但 P 档信息是端状态可以直接获取的，不应该反问用户
- 通风模式无法识别下雨天、车内 PM2.5 等环境信息来做智能调整
- 部分场景触发条件（如车辆长时间暴晒、湿气重、空气浑浊）缺乏端状态信号支持

**为什么：** 端状态（通过 OMS/DMS 等传感器提供）的接入范围有限，部分信号（如抽烟检测、PM2.5）尚未接入或不稳定。

### 问题 5：场景回滚能力缺失

**问题是什么：** 所有预置场景都设计了"回滚"功能——进入前什么状态，退出后恢复原状。但 Planner 编排的场景没有回滚能力，"只有去无回"。

**举例：** 用户进入小憩模式 → 座椅放平、遮阳帘关闭、白噪音播放 → 用户说"退出小憩" → 无法恢复座椅/遮阳帘/音乐到进入前的状态。

**为什么重要：** 联动控制多（座椅、空调、遮阳帘、音乐、氛围灯等），不可能让用户逐个手动恢复。这是所有编排场景的共性问题。

### 问题 6：记忆/FaceID 功能不可用

**问题是什么：**
- 用户让 Planner 记住偏好（如"下次说打开通风模式时同步开窗"），Planner 信誓旦旦说记住了，实际没有保存
- 自定义场景无法持久化——用户通过语音创建的场景无法保存，每次都要重新让 Planner 生成，且每次结果可能不同
- 用户无法查看自己创建了多少个自定义场景

**为什么：** FaceID 尚未就绪（预计近日修复），导致用户身份识别和记忆绑定均不可用。记忆工具本身也存在问题。

### 问题 7：UI 展示问题（非 Planner 侧）

**问题是什么：**
- 调一个工具出现一次卡片，调多个工具罗列多次，占满整屏需要翻页
- 工具调用卡片过大，一页看不完所有操作
- 部分 UI 只展示了工具调用但没有实际执行结果

**处理方式：** 这些是 UI 侧的 bug，需要提交给测试团队转 UI 团队，不由 Planner 侧处理。后续相同工具将不再重复展示。

## 三、场景分类梳理结果

会议逐条过了场景列表，明确了每个场景的处理方式：

### A. 不需要编排，直接调预置场景（需提需求给方向靠/左晨）

| 场景 | 说明 |
|------|------|
| 离车保电 | 车上有该模式，但方向靠车控协议中未注册，需提需求 |
| 洗车模式 | 已有预置，直接调用 |
| 宠物模式 | 已有预置（含大屏"主人马上回来"图），直接调用 |
| 露营模式 | 已有预置，直接调用 |
| 大床模式 / 一键躺平 | 已有预置，直接调用 |

**Action：** 整理这 5 个场景的标准 query 示例（含变体表达），提交给左晨团队在方向靠中注册支持。

**Query 示例思路（以宠物模式为例）：**
- 标准 query："打开宠物模式"
- 变体 query："我现在要临时下车，车里有只狗，你帮我看一下"
- 需要明确哪些变体表达也能触发预置场景

### B. 需要 Planner 编排的场景

| 场景 | 编排要点 | 备注 |
|------|----------|------|
| 后排观影 + 驻车前排观影 | **合并为一条**。区分：驻车时前后排均可看；行驶中仅后排可看 | 前排需 P 档前置条件 |
| 5D 观影 | 视情况与前排/后排观影合并。Planner 只负责调腾讯视频播放指定影片；3分05秒等时间点的座椅/氛围灯联动由腾讯视频协议触发，非 Planner 职责 | 需要 5D 片源才能触发 |
| 极速降温 + 极速升温 | **合并为一条**。降温=空调最大+风量最大+前1分钟开窗+温度达标后降挡；升温=空调+座椅加热+方向盘加热（视配置） | 当前规则化是"暴力调到最大"，后续可智能化 |
| 车内小憩 | 需编排：BGM（轻音乐/白噪音选择）+ 座位选择 + 定时闹钟 + 结束回滚。好的点：Planner 已会主动问"休息多久"；还需：问哪个位置小憩 | 调 skill 优化 |
| 冥想空间 | 与小憩类似但不睡觉，保持独立不合并 | 收尾问题：结束后 BGM 未停、座椅未回滚 |
| 快速通风 | Planner 可做得比规则化更好：识别下雨天→减小开窗幅度；根据实际情况调整通风时长。但需端状态支持（温度、PM2.5 等） | 当前规则化无法判断天气 |
| 儿童陪伴 + 后排勿扰 | **合并为"用车关怀"**。覆盖儿童睡觉、大人睡觉等场景 | 扩大适用范围 |
| 副驾调节（女王模式） | 当前先做副驾，后续再扩展到后排右座等多座位判断（需 VRA/触发器支持） | 多座位联动暂不做 |
| 保驾护航（雨天关怀） | 恶劣天气下打开雾灯、除霜、辅助驾驶等 | 暂不支持语音触发 |
| 新手模式 | 用户自述"我是新手帮我开车"→ 生成辅助模式 | 当前 Planner 罗列了操作但实际未执行 |

### C. 合并汇总

| 合并前 | 合并后 |
|--------|--------|
| 后排观影 + 驻车前排观影 (+视情况 5D 观影) | 观影模式（含前排/后排/5D 分支） |
| 极速降温 + 极速升温 | 极速调温 |
| 儿童陪伴 + 后排勿扰 | 用车关怀 |

**原则：** 尽量合并同类项，减少 skill 数量，二级分类即可，不用到三级。

## 五、关键决策记录

1. **场景 Planner 暂时下架**，618 前重新上线
2. **5 个预置场景不走 Planner 编排**，直接调用预置应用，需方向靠注册支持
3. **合并同类项**：观影合并、调温合并、关怀合并，原则上到二级分类即可
4. **5D 观影中的时间点联动（如 3 分 05 秒座椅震动）由腾讯视频协议负责**，Planner 只管调起/关闭腾讯视频
5. **回滚是所有编排场景的共性需求**，需 Planner 侧设计方案
6. **记忆问题等 FaceID 修复后再验证**
7. **UI 问题走测试 bug 流程**，不由 Planner 侧处理
