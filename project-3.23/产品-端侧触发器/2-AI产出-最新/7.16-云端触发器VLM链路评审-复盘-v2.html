03技术短板补课：会上别人说的内容里，你没吃透的 6 个概念

这一节回答你问的「会上其他人说的内容我哪里是有短板的」。每条 = 会上原话 → 正确理解 → 你要做的动作。

① 上报驱动 vs 轮询（刘杨 37:34 纠正你两次）

正确理解：触发器没有自己的时钟，上游转发一次它判断一次。说「我11秒轮询一次」在架构上不成立。动作：PRD v2 里把「判断频率」一节改写成「事件驱动：以上游转发事件为触发源」，并把端到端时延公式写出来（时延 = 上报间隔10–20s + 网络 + 模型推理）。

② 任务编号 = 配对主键（丁彬/刘杨/韩杰 12:46–15:22 三人接力讲）

正确理解：并发3个任务时，VLM 上报的每条结果必须带 task_id，触发器靠它把「结果」和「注册时的感知目标」配对，才能喂给模型判断。不是优先级、不是枚举值，是主键。动作：PRD 协议章节补「配对图」（见回合2的补课动作）。

③ 整包无标签（丁彬 26:20：「就这三个包整包给到端状态，并没有进行解析」）

正确理解：端状态里的 VLM 数据是未结构化的描述文本大包裹，没有「宝宝哭=1」这种原子信号。所以 A 路线（枚举标签+字符串比对）地基不存在，全模型判断是唯一现实路线——但它是成本路线。动作：成本重算 + 标签化写成二期降本方向，反喂端侧提需求（端侧提取标签的排期问丁彬）。

④ context 链路 vs 高频UP链路是两条互不相交的路（刘杨 19:50 起反复确认）

正确理解：绿框（乘客列表，默认 alwayson 任务）→ 走端状态 → 进云端 context → 触发器订阅；红框（动态任务结果）→ 走任务中心长时任务字段模块 → 转发触发器。两条路的报文、topic、路径完全不同，触发器对两条路的取数方式也不同（订阅 vs 接收转发）。动作：PRD 链路图把两条路画成不同颜色+标注「订阅/转发」字样，下一场评审直接投这张。

⑤ 订阅 vs 工具调用（陈西钺 50:51–51:25 给你的现场教学）

正确理解：舱外 VQA 不是「你订阅他上报」，是「你需要时调用、他当场返回」——「我是工具，你订阅不了」。这两种集成模式的触发逻辑完全不同：订阅=数据推给你你再判断；工具=你的模型判断时主动发起调用拿数据。动作：舱外场景（即将驶出地库、风景介绍）找晓伟对齐需求归属后，单独约西钺把「触发器→VQA工具调用」链路也评了——这是下一场会的内容，别拖。
