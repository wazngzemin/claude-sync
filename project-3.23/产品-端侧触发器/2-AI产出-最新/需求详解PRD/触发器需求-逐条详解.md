# 需求1 ｜ 整理《高频触发条件清单》  〔P0 · 上汽610 · 本周第一优先级〕

### ① 这是什么 · 为什么做
触发器要能判断"什么情况下做什么事"，前提是先有一份**标准化的"情况"清单**——也就是系统认得的"触发条件"有哪些。现在的问题是：每个任务各自读 CAN 信号、各写各的，**没有一份统一的条件字典**（PRD 1.2："端侧没有统一信号库，任务判断缺少稳定输入"）。结果就是条件没法复用、模型没法解析、下游也对不齐。
你这条需求，就是把散落的条件收成**一张干净、通用、能被模型解析的标准条件清单**。它是后面需求2、3、5 全部的"总源头"——它定稿，全线能动；它拖，全线塌。

### ② 整体链路一图流
```
你（PM）精选条件清单
   └─▶ 分发四方：意图(婉婷) / 动态Advisor(佳杰) / 主动服务(益红) / 研发(刘洋·叶芳)
          └─▶ 研发录入后台配置器、注册成系统可用条件
                 └─▶ 测试(萝莉)找车实测，标记哪些真能触发
                        └─▶ 后台有一版标准条件库，成上汽/赛力斯通用基线
```

### ③ 第1棒：你（PM）产出什么
**交付物：《高频触发条件清单》一张表，一行一条件。** 列固定为 6 列，别自创：

| 条件名 | 取值判断方式 | 信号库v2.5编号 | DNA能否支持 | 归属任务类型 | 备注 |
|--------|--------------|----------------|:----------:|--------------|------|
| 车内温度 | 数值阈值（>25℃） | （填编号） | ✓ | 舒适 | |
| 车门 | 布尔 / 变更为（关→开） | | ? | 安全 | 套娃结构，见需求6 |
| 座椅占用 | 布尔（有人/无人，按座位） | | ✓ | 安全 | |
| 档位变化 | 变更为（D/R/N→P） | | ✓ | 安全 | 下车安全用 |
| 车速 | 数值阈值（<10 / <20km/h） | | ✓ | 出行 | |

**"取值判断方式"全系统只有三种，背下来，跟谁都这么说：**
- **布尔开关** —— 开/关、有/无（座椅占用、安全带）
- **数值阈值** —— 大于/小于某个数（温度>25、车速<10）
- **变更为 / 从无变有** —— 状态切换（车门 关→开、档位 D→P）

**做到什么算"产出完成"**：
- 范围 **20~30 条**，对标理想"任务大师"，**精选高频、不贪全**（初级 PM 最容易犯的错就是想做全，图里专门写了"不贪全"）。
- 每条 6 列都填全；DNA 不确定的填 `?`，留给需求3 一起判。
- 形式：飞书表 / Excel，可评审。

### ④ 第2棒：交给谁 · 怎么交 · 交时说什么
**这一份清单同时分发四方，各取所需**（不是分别做四份）：

| 交给谁 | 角色 | 你交的时候要说清楚 |
|--------|------|---------------------|
| 廖婉婷 | 意图 | "这是触发器支持的条件全集，请把它当约束去调模型"（→ 这就是需求2） |
| 施佳杰 | 动态Advisor | "用户实时订阅事件请按这张清单解析" |
| 赵益红 | 主动服务 | "后台配预设事件请照这张清单的条件来" |
| 刘洋 / 叶芳 | 研发 | "请把这些条件录进后台、注册成系统可用条件" |

交付动作建议：**拉一个 30 分钟的小会**把清单过一遍，而不是甩个文档。开场一句话定调：
> "这是触发器的标准条件清单，20 多条高频的。今天想跟四边对一下：① 你那边用这版有没有解析不了/配不出来的；② 有没有该加的高频条件我漏了。对齐完我锁版本，大家照这版往下做。"

### ⑤ 第3棒：他拿到后做什么 · 回你什么
- **意图（婉婷）**：把清单当词表/约束去调模型 → 回你一个"模型调试要几天"（这是需求2 的核心）。
- **主动服务（益红）**：照清单在后台配"预设事件" → 回你"哪些她要、哪些配不出"。
- **研发（刘洋/叶芳）**：把条件录进后台配置器、注册上线 → 回你"录入完成 / 哪几条信号后台还没接"。

### ⑥ 第4棒：研发 → 测试 → 上线
1. **叶芳/刘洋**：在后台配置器把条件逐条录入、上线（PRD 9.1 本期"用简单配置方式，后台后续补"）。
2. **测试·萝莉**：找车实测，**标记两类**——① 哪些条件真能触发、② 哪些其实端上没部署（信号库里没有对应信号）。
3. **回炉**：没部署的条件，要么砍掉、要么去催端信号接入。

### ⑦ 他们开发需要什么（前置——缺了开不了工）
| 缺什么 | 找谁 | 为什么 |
|--------|------|--------|
| 《信号库 v2.5》最新版 + **锁定版本号** | 韩天亮 | 这是判每条"有没有端信号、填哪个编号"的唯一字典；信号库一直在变，不锁版本做到一半全废 |
| 后台配置器登录权限 | 文涛 / 芭比 | 条件录不进后台 = 不算"系统可用" |

### ⑧ 怎么算做完（验收）
- 后台有一版**干净、通用、能被模型解析**的标准条件库；
- 20~30 条定稿、过评审；
- 成为上汽 + 赛力斯**通用基线**（别的项目直接复用，不重做）。

### ⑨ 节点 · 依赖 · 风险
- **节点**：6/15 前评审、向 610 交需求。
- **下游强依赖**：需求2（意图）、需求3（益红）、需求5（触发机制）全等这张清单。
- **风险**：信号表一直在变 → **先锁一个版本号再开工**；别等齐所有端信号才动手，你先精选定稿。
