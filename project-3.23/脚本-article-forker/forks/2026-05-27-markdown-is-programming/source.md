## 摘要

王建硕（百姓网创始人）提出一个颠覆性框架：Markdown 是新时代的编程语言，LLM 是新一代的编译器。AI 写出的"屎山代码"不需要被治理——它是编译产物，你该在上一层（自然语言/.md文件）做修改。他展示了一个从用户反馈到 APP 自动更新的完整闭环（5分钟），以及由 PM Agent + 工程 Agent + 市场 Agent 组成的全自动团队。核心判断："你写得越多 .md，控制得越细；你越偷懒，AI给你的就是它的缺省值。" 程序员是这一轮最危险的群体——不是因为笨，是因为太熟练地做一份正在消失的工作。

## 关键要点

- **层的革命**：AI代码 = 新一代汇编，.md = 新一代源代码，LLM = 新一代编译器——不要在编译产物层做治理
- **5分钟闭环**：用户反馈→GitHub Issue→Claude Code自动写代码→测试→合并→build→TestFlight→上线
- **PM Agent**：AI自主审反馈、排优先级、判断一致性，甚至自主产出需求——"它自己更新自己"
- **Agent团队**：市场（每天6点）+ 工程（每4小时）+ PM + 客服 = 一个人就是一家公司
- **聊天 ≠ 写程序**：聊天是debug方法，程序是可被定期执行的确定产出；90%的人还停在聊天层
- **手写自然语言的颗粒度**："你亲手一个字一个字敲的自然语言的数量 ≈ 你对项目的控制颗粒度"
- **Token是GDP**：公司的总Token使用量是唯一硬指标；省Token会遮蔽你的双眼
- **程序员最危险**：离新技术越近的产业工人越惨——工业革命淘汰熟练工，不淘汰农民

## 逐段拆解

### 第一段：层的革命——屎山代码不需要被治理（"屎山只要不被人看到，它就不是屎山"）

**功能**：核心框架建立——重新定义"应该在哪一层工作"

**深度分析**：这是全文最有思想重量的一段。王建硕用汇编→C语言的类比，把AI代码重新定位为"编译产物"。关键逻辑链：①没有人去改汇编修bug，他们改C语言然后重新编译 → ②AI写的代码就是新一代"汇编" → ③你跑去代码那一层修改，等于在改汇编 → ④你站错了层。这不是说徐文浩的治理方法错了（skill、最佳实践、harness都对），而是说这些工具应该用在"自然语言层"，不是"代码层"。"代码不存在错误，只存在跟你的intention不一致"——这个重新定义极其关键：所谓bug = 你没把意图说清楚。LLM给你"白色"不是bug，是你没说要"蓝色"。修复方式不是改Python代码，是改你最上面那条自然语言。

**隐含假设**：LLM的"编译"足够可靠——即你改了.md，重新"编译"（让AI重写代码）后，输出一定比直接改代码更好。但当前LLM的"编译"并不确定性——同样的输入可能产出不同输出。真正的编译器是确定性的（同一C代码永远编译出同一汇编），LLM不是。

**与你工作的直接关联**：这一段直接重新定义了你的工作本质。你维护的SP（System Prompt）就是王建硕说的".md源代码"。Planner输出的JSON决策（talk_or_not/talk_content/action_list）就是"编译产物"。当Planner出badcase时，你不应该直接改Planner的输出逻辑（那是在改"汇编"），你应该改SP（改"源代码"）。你已经在做对的事——但你可能没意识到，你正在做的是"编程"。

### 第二段：5分钟闭环——用户反馈到APP更新（"过几个小时就会更新"）

**功能**：具体实现——展示"在正确的层工作"带来的效率跃迁

**深度分析**：技术链路朴素但极其完整：用户反馈→GitHub Issue→Claude Code监听hook→读Issue→写代码→跑测试→合主分支→build→TestFlight→审核上线。这不是PPT里的架构图，是实际跑通的生产流程。5分钟从反馈到可测试版本，1-2天审核后上线到所有人手机。这个速度意味着：传统产品开发的"需求→排期→开发→测试→上线"5步被压缩成1步——用户写反馈。更深层的意义：**产品经理的角色被重新定义了**。过去PM是"翻译器"（把用户需求翻译成开发可执行的PRD），现在用户直接对AI说话，AI直接执行——PM作为"翻译器"的角色消失了。

**隐含假设**：用户反馈足够清晰，AI能正确理解并实现。但实际上大多数用户反馈是模糊的（"感觉不太好用"），AI很难从模糊反馈中提取明确的代码修改。这也是为什么王建硕后来加了PM Agent。

**与你工作的直接关联**：想象一下这个闭环应用到Planner——用户在车里说"豆包刚才理解错了"→自动生成一个badcase Issue→AI分析Issue→自动修改SP中的相关规则→重新部署→下次同类问题就不会再错。你现在的"手动看日志→标badcase→改SP→上线"循环，可以部分自动化。

### 第三段：PM Agent——"它自己更新自己"（"我就很好奇——今天它又变成了什么样"）

**功能**：升级——从"人驱动"到"AI自驱动"

**深度分析**：这是5分钟闭环的进化形态。PM Agent做的事：审反馈、排优先级、做一致性判断、过滤奇怪需求——这就是一个标准PM的日常工作。更惊人的是王建硕说"连用户反馈也不需要了，它自己想想要更新什么"。他每1-2小时收到TestFlight推送，"很好奇今天它又变成什么样了"——**开发者变成了用户**。这打破了30年来的软件开发基本假设：开发者和用户是两种身份。现在合二为一了。注意王建硕用的词——"很好奇"。这不是焦虑，不是监控，是好奇。这个情感状态说明他信任这个系统，同时对它的输出保持开放。

**隐含假设**：PM Agent的判断质量足够高——能正确排优先级、正确判断一致性。但如果PM Agent做了错误的优先级判断（比如把一个critical bug排到低优先级），自动化闭环反而会加速问题扩大。

**与你工作的直接关联**：你的工作中有哪些部分可以交给"PM Agent"？候选：①badcase自动分类和优先级排序 ②SP中示例的自动健康度检查（标记老化示例）③新用户行为模式的自动发现。这些不需要一步到位——可以先做最简单的（badcase自动分类），积累经验后再做更复杂的。

### 第四段：Agent团队——"一个文件夹+一个schedule"（"一个人就是一家公司"）

**功能**：组织模型——用Agent替代传统团队结构

**深度分析**：四个Agent角色：市场（每天6点执行）、工程（每4小时+随时响应）、PM（审核过滤）、客服（处理用户问题）。王建硕对"团队"的定义极其朴素：一个子文件夹里的一堆代码 + 一个定期执行 = 一个团队。但他反复强调不是甩手掌柜——"光你自己一个字一个字敲的自然语言，可能至少要敲个几百行"。**你手写的自然语言数量 ≈ 你对项目的控制颗粒度**。这句话是全文最重要的判断之一——它说的是：自动化不等于放手，自动化等于把控制从"代码层"迁移到"自然语言层"。你不是不控制了，你是在更高的层控制。

**隐含假设**：一个人能写出足够精确的自然语言来覆盖市场+工程+PM+客服四个维度。实际上这需要极强的全栈产品能力——大多数人在自己不擅长的领域写出的.md会很模糊。

**与你工作的直接关联**：你的Planner SP就是一个"Agent团队"的指令集——①角色定义 ②工具定义 ③输出格式 ④注意事项 ⑤示例 ⑥知识 ⑦风格。这些就是王建硕说的"文件夹里的一堆.md"。SP的长度（30K-35K tokens）≈ 你对Planner的控制颗粒度。如果SP太短太模糊→Planner给你的是"缺省值"（通用但不精确）。SP越详细→Planner的行为越像你想要的。

### 第五段：几万行.md就是源代码（"千万别再去碰代码"）

**功能**：框架完整化——.md的工程化方法论

**深度分析**：王建硕具体展示了他的.md目录结构：角度.MD（推广角度）、VI.MD（视觉规范）、voice.MD（口吻和用词）、platform/（各平台规格）、generation/（图像生成规则）。这不是随意的笔记——这是一个精心组织的"代码项目"，有目录结构、有分工、有层级。"跟我过去20年写C和Python的工作流本质一模一样"——区别只是语言从C/Python变成了.md。每天有一个schedule定期扫描所有.md，从入口往下执行。这就是一个"编译-执行"的循环。

**隐含假设**：.md的组织方式可以达到和传统代码一样的精确度。但自然语言本质上是模糊的——同一句话不同人理解不同，同一个.md不同版本的LLM执行结果也不同。代码的确定性 vs .md的模糊性是这个框架的根本张力。

**与你工作的直接关联**：你的SP已经是一个组织好的.md项目——①角色定义 ②输入类型 ③工具定义 ④输出格式 ⑤注意事项 ⑥示例 ⑦设备知识 ⑧工具tips ⑨聊天风格 ⑩动态变量。这就是王建硕说的"目录结构"。你的SP=你的源代码。SP的质量=你的"编程能力"。SP的每次修改=一次commit。你需要像对待代码一样对待SP——有版本控制（SP changelog）、有测试（badcase回测）、有code review（SP评审会）。

### 第六段：聊天和写程序是两件事（"少跟AI聊天，多写程序"）

**功能**：行为区分——消耗型使用 vs 积累型使用

**深度分析**：王建硕定义了"程序"：确定的、可以被定期或被trigger执行的、给你出结果的东西。它的反面：每次都从零开始、打开窗口从头聊。"90%的人还停在打开Python命令行学语言那一层——不积累、不沉淀、明天再来还是一样。" 这和前一篇fork的"L0-L3分层"高度对应——L1（只聊天）就是"不积累"的状态。两个金句：**普通人，少聊天多写程序。程序员，少写代码多写程序。** 第一句给普通人：别停在聊天，要把和AI的交互固化成可复用的流程。第二句给程序员：别停在写代码，要在更高层（.md）写程序。

**隐含假设**："可积累"是好的。但有些交互天然是一次性的（探索性提问、brainstorm），不需要也不应该被固化成"程序"。

**与你工作的直接关联**：你和AI的交互有两种模式：①聊天模式（在对话框里问"这个badcase怎么解决"）②编程模式（修改SP中的规则/示例，这些改动会被Planner每次调用时"执行"）。王建硕说的"少聊天多写程序"意味着：与其每次遇到badcase在脑子里想"怎么办"，不如把解决方案写进SP——让它变成一条永久的规则，以后每次调用自动生效。这就是"编程"。

### 第七段：抬轿子、车、和驾驶员（"离新技术越近的产业工人，越惨"）

**功能**：身份判断——谁最危险

**深度分析**：三层隐喻——抬轿子的（旧技能）、教父（教旧技能的人）、驾驶员（驾驭新工具的人）。程序员被困在"教父"位置——他们的代码能力可被AI替代，但因为情感和惯性，学"开车"的意愿反而低于普通人。"离新技术越近的产业工人越惨"——工业革命淘汰熟练工不淘汰农民，互联网淘汰金牌编辑不淘汰文盲。历史模式：**在技术范式转换期，旧范式中最熟练的人损失最大，因为他们的沉没成本最高、转型意愿最低。**

**隐含假设**：编程能力会被完全替代。但目前的AI编程能力在系统架构、性能优化、安全审计等维度仍有明显短板——高级工程师的一部分工作暂时还不可替代。

**与你工作的直接关联**：作为PM，你反而不在"危险区"——因为你的核心技能是"理解用户需求+判断产品方向"，这些是"开车"的能力，不是"抬轿子"的能力。但如果你只把自己定位为"写SP的人"而不是"用.md编程的人"，你也可能沦为新时代的"熟练工"。关键区分：你是在"写文档"（旧定位）还是在"写程序"（新定位）？

### 第八段：Token使用量是GDP（"省Token会遮蔽你的双眼"）

**功能**：衡量标准——一个硬指标判断公司AI使用深度

**深度分析**：王建硕一天烧2000美金Token——这不是炫耀，是证明。"公司总Token使用量×单价=每天在Token上烧多少钱"——像GDP一样的硬指标。如果这个指标上不去，其他认知都是白谈。"省Token遮蔽双眼"的洞察很深：我们这代人在"智能稀缺"的世界长大（脑子贵、时间贵），下意识用"节约"解决问题。但AI时代智能变便宜了，瓶颈不再是"能不能做"而是"敢不敢做"和"想不想到"。省Token让你以为"AI不够好"，实际上是你给的太少。

**隐含假设**：烧Token = 有效使用。但很多公司可能在无效地烧Token（重复调用、低质量prompt、没有积累的对话）。量≠质。

**与你工作的直接关联**：Planner的Token消耗直接关系到性能和成本——但王建硕的判断让你重新思考：你在"省Token"（上下文精简、远区压缩）的同时，有没有因此丢失了关键上下文？"省了Token但Planner变笨了"就是"省Token遮蔽双眼"的一个实例。也许有些场景应该"拥抱浪费"——宁可多给上下文、多花Token，也要让Planner的输出更准确。

## 论证链

"AI代码=新汇编，不该在这层修改"（核心框架）→ "你该在.md层修改"（正确的层）→ "用户反馈→5分钟更新"（闭环证明）→ "PM Agent自主更新"（进化形态）→ "一个文件夹+一个schedule=一个团队"（组织模型）→ "几万行.md=源代码"（方法论）→ "聊天≠编程，要积累"（行为标准）→ "手写自然语言量=控制颗粒度"（核心等式）→ "Token使用量是GDP"（衡量标准）→ "程序员最危险，因为太熟练"（警告）

## 隐含假设

1. **LLM的"编译"足够可靠** — 但LLM不是确定性编译器，同一输入可能产出不同结果
2. **自然语言可以达到代码级精确度** — 但自然语言本质上是模糊的，这是这个框架的根本张力
3. **一个人能写出覆盖全栈的.md** — 需要极强的全栈产品能力，大多数人做不到
4. **烧Token = 有效使用** — 量≠质，无效的Token消耗只是浪费
5. **编程能力会被完全替代** — 高级工程师在架构/性能/安全维度的工作暂时不可替代

## 观点边界

1. **适用于个人项目和小团队**：王建硕一个人做APP可以这么玩；大团队的协作、权限、合规让"自动合主分支"变得不现实
2. **适用于"输出可预期"的场景**：调UI、加功能可以自动化；涉及安全、隐私、金融交易的修改不应该自动执行
3. **"屎山不用管"只在代码成本为零时成立**：如果每次重新"编译"成本很高（比如大型系统），有时候修改编译产物反而更经济
