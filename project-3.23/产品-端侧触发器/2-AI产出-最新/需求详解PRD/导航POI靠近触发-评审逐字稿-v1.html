① 开场：今天评什么、要拿到什么

~1 分钟

今天评审导航 POI 靠近触发这个需求。一句话说就是——让触发器能做"快到某个地方就提醒你"这件事。

触发器目前压根不认"目的地""剩余距离"这些概念。用户说"快到外婆家提醒我打电话"，系统注册不了、监测不了、触发不了，一个字都做不了。本期就是把这个能力补上。

今天我要跟大家对齐三件事：

  1. B 方案（端侧做状态机）的完整链路，大家看看能不能落地。
  2. B 方案有几个难点，我说出来，请各方评估。
  3. 每个人要做什么，工期什么时候能给我。

PRD 文档已经发到群里了，今天我不念 PRD，我只讲关键的几个点，细节你们对着文档看。

② 背景：这事到底要干嘛

~2 分钟

先说三个真实场景，让大家有个画面。

场景一：早上导航去公司，距公司 500 米的时候车主动说"快到啦，要不要提前点杯咖啡"。
场景二：导航回外婆家，快到的时候主动帮你搜附近充电桩。
场景三：路过徐浦大桥，提醒你"准备右转下匝道"。

这三个场景，630 量产要上。但现在触发器全都做不了——因为它不认"距离"这个信号。本期就是把导航状态、目的地地址、剩余距离、收藏地址这些信号补进触发器，让它能用起来。

两类 Case 要分清

刚才三个场景，其实分两类，性质不一样。

A 类：目的地/途经点距离

POI 就是你当前导航的目的地或途经点。
导航已经在实时算到它的剩余距离。
端状态信号已支持，只缺 one_shot 状态机。

例："离目的地还有 1km 提醒我拿咖啡"
B 类：靠近任意 POI

POI 不是当前导航目标。
导航没有在算到它的距离。
需要先把它动态添加为途经点，地图才能给你距离。

例："快到外婆家提醒我打电话"

A 类相对简单，导航在算距离，我们加个状态机就行。
B 类难很多——"外婆家"不在导航路线上，系统不知道到它多远。要先把"外婆家"从用户记忆/收藏地址里解析出一个真实地址，再动态插入导航的途经点，地图才开始算到它的距离。地图能不能动态插途经点，是 B 类 Case 的生死问题。

停一下。这个区别很重要，后面讲链路的时候还会用到。大家心里先有这个数。

③ 核心难点演示：为什么"距离 < 500m"行不通

~3 分钟

有人可能会想——这事不是很简单吗？配一条规则"剩余距离小于 500 就触发"不就行了？不行。我用数字走一遍你就知道了。

周一早 8 点，用户从家出发导航去公司，出发时 15 公里。产品想做"距公司 500 米时提醒拿咖啡"。

时刻剩余距离如果配"<500m 就触发"
8:00 - 8:2515km → 501m一路递减，都 >500m，不触发。没问题。
8:25:30500m第一次 ≤ 500m，触发！提醒拿咖啡。
8:25:31499m又满足 <500m → 又触发
8:25:32498m还满足 → 又触发
8:25:33 …497m … 50m每一帧都满足 → 一路触发几十上百次

看到了吧。从 500 米一路开到停车场，距离 499、498、一直到 50，每次刷新都满足"小于 500"，触发器没有记忆——它不知道刚才已经提醒过了——就会从 499 开始"该拿咖啡了、该拿咖啡了"连着刷屏几十次。用户直接崩溃。

我要的是什么？第一次跨过 500 米这条线的时候提醒一次，然后就闭嘴。像门铃——按一下响一声，不是防空警报一直响。这就是 one_shot。

现有的信号协议做不了这事。它每次刷新独立判断"当前值满不满足"，没有三样东西：

  1. 记不住上一次的距离和状态——不知道之前是不是已经提醒过了。
  2. 判不了方向——不知道你是在靠近还是在远离。
  3. 标不了"已触发"——触发了不能自动闭嘴。

所以必须加一套 POI 状态机——给每个被监控的目的地记一个状态，走过就不回来。

状态机长这样（四个格子，单向翻）：

远 ──d≤480──→ 已接近 ──d≤50+挡P──→ 已到达 ──d>520──→ 已离开

每条箭头只走一次。"远"的时候距离跌破 480 米，翻成"已接近"，触发一次接近事件，以后再也不触发了。这就是 one_shot。

为什么是 480 不是 500？防抖。用户在 500 米路口等红灯，GPS 会在 498 和 502 之间来回抖。如果阈值卡在 500，每抖一下进 500 就触发一次。所以真正的判定线往里收 20 米——小于 480 才算接近，大于 520 才算离开，480 到 520 之间是死区，不动作。

④ B 方案怎么做 + 链路走一遍

~2 分钟

之前周会上讲过 A/B 两条路。A 是引擎自己做状态机，B 是端上把事件算好了报给引擎。后来综合评估，630 本期走 B 方案——端侧做状态机、端上算好"接近"事件再同步云端。

整条链路分两段：

第一段 · 注册（用户说出话的那一刻，~10 秒内完成）

用户说"快到外婆家提醒我打电话"。系统听懂了——条件是"接近外婆家"，动作是"提醒打电话"。

  1. Director 理解意图 → 调工具注册到云端触发器
  2. 云端触发器生成条件 DSL，包括 type:"approach"、poi:"外婆家"、threshold:500
  3. 条件下发到端侧，端侧持久化，状态机初始化为"远"

注册完成。系统记住了"快到外婆家就提醒打电话"，端上盯着这个 POI。

第二段 · 触发（可能半小时后，真的快到了）

  1. 端侧信号库持续收地图给的剩余距离，写入标准化快照
  2. 端侧状态机每次距离刷新都跑一遍判定：state==远 且 d≤480？
  3. 条件满足 → 产出 approach 事件 → state 翻成"已接近" → 以后不再触发
  4. 事件同步到云端信号库
  5. 云端触发器拿到事件 → 匹配到注册的条件 → push 给 Advisor
  6. Advisor 组装 advice → Director 决策 → 播报"快到外婆家了，要不要打电话？"

这条链路大家对着 PRD 里的流程图看一下。重点在第 2、3 步，状态机逻辑全在端侧跑。

⑤ B 方案的难点和我的顾虑

~2 分钟

B 方案能做，630 来得及，但有几个问题我得说清楚。

难点一：状态机逻辑散在端上，多端要各写一遍

上汽一套端、赛力斯一套端，以后可能还有更多车型。B 方案的状态机跑在端侧，意味着每个端都得自己实现一遍"远→已接近→已到达→已离开"这套判定。逻辑一样，但代码各写各的。

我的顾虑：写的时候可能不一致，改的时候更容易漏。比如后面产品想把死区从 480/520 改成 450/550，我得找每个端的人说一遍"你改一下"，漏了就出 bug。如果是 A 方案（引擎统一做），我改一个地方就全生效了。

但这不阻塞 630。本期先这么做，能跑起来。长期看，如果端越来越多，再考虑迁到引擎也来得及。

难点二：B 类 Case——地图能不能动态插途经点

A 类 Case（"快到目的地提醒"）好办，目的地本来就在导航里，距离现成的。

B 类 Case（"快到外婆家提醒"）问题大——外婆家不在导航路线上。你得先把"外婆家"解析成一个真实地址，再动态插到导航的途经点里去，地图才会开始算到它的距离。

地图能不能动态插途经点？这个能力直接决定 B 类 Case 做不做得了。@宫丽萍 @导航产品 需要你们确认。做不了的话，B 类 Case 本期就只能砍掉，只做 A 类。

难点三：收藏地址匹配——"外婆家"怎么变成经纬度

用户说"快到外婆家"，系统里存的收藏地址可能是"上海市浦东新区XX路XX号"这样一串字。导航目的地的 address 也是一串字。这两串字格式不一定一样——一个写"上海浦东新区"，一个写"浦东新区"，字符串直接比对就对不上。

得定清楚：匹配是用字符串等值比对还是经纬度距离比对？@导航产品 @金龙

难点四：状态重置

状态机翻到"已接近"之后不会再翻回"远"。那如果用户中途取消导航、再重新导航去同一个地方呢？状态得清掉重来。什么条件下重置？离车熄火？重新发起导航？要定清楚，不然出怪 bug。@金龙 @汪帅

说完这四个难点，停一下。我想听一下大家觉得哪个最棘手、哪个有方案了，我们一个一个过。

⑥ 和传统地理围栏的区别

~2 分钟

可能有人会问——这东西不就是地理围栏吗？手机上早就有了，你进入一个区域就触发通知。

不一样。我说清楚区别，这个很关键。

维度我们的方案（导航沿路线距离）传统地理围栏
距离怎么算 用导航给的沿路线剩余距离。你在高架上要绕到下一个出口再绕回来，导航会告诉你"还有 2 公里"。 画一个圆或多边形，用 GPS 直线距离判断你在不在圈里。
高架旁场景 沿路线 2km → 不触发"接近"。准。 直线 300m → 触发了，但你其实还得绕 2 公里才到。误触发。
前提条件 必须发起导航，有目的地/途经点。没在导航就没有"沿路线距离"。 不需要导航。只要 GPS 在，进围栏就触发。
能做"路过"吗 A 类能做"接近目的地/途经点"。B 类"路过任意 POI"需要动态插途经点，依赖地图能力。 天然支持。画个围栏在那里，路过就进去了。
地图 SDK 支持 高德已有导航剩余距离。可用。 宫丽萍已确认：高德不支持地理围栏 API。
状态机 需要我们自己做（端侧状态机） 围栏本身就有进出状态，但高德没提供这个能力

所以我们选"导航沿路线距离"这条路，核心原因有两个：

  1. 准。高架、隧道、绕行这些场景，沿路线距离比直线距离靠谱得多。
  2. 高德没给围栏 API。想做围栏得自己从头搭，重得多。

代价是——必须在导航状态下才能用。没发起导航、纯兜风，这套方案做不了。

⑦ 如果 B 方案大家觉得做不了，走围栏需要什么

~1 分钟

我先把退路说清楚。如果大家评估下来觉得 B 方案的端侧状态机做不了，或者工期撑不住，那唯一的退路就是自己实现地理围栏。

围栏方案要做什么？我列一下，大家感受一下工作量。

要做的事谁来做难度
实现围栏引擎 端侧研发（金龙/汪帅） 高德不提供围栏 API，得自己写。核心是持续的 GPS 点和围栏多边形做"点在区域内"判定，还要处理 GPS 漂移、低频采样漏判、省电模式下的唤醒。这不是写个 if-else 的事。
POI 经纬度获取 导航产品 + 地图 用户说"外婆家"，你得先把它变成经纬度坐标才能画围栏。收藏地址有经纬度吗？地图搜索结果能拿到吗？得确认。
围栏形状和半径 产品定义 每个场景画多大的围栏？加油站画 2 公里、咖啡画 500 米？圆形还是沿路线的带状？产品得一个一个定。
进出状态管理 端侧 围栏也得有"进入过了就不再触发"的逻辑，本质还是要一个状态机。
高架/隧道误判 端侧 + 产品 直线距离 300 米但沿路线 2 公里，围栏方案天然会误触发。要么忍、要么叠加海拔/路线修正，更复杂。

大家看到了——围栏不是更简单，是更重。没有高德的围栏 API 兜底，所有东西都得自己写。而且围栏用直线距离，高架旁场景它天然不准。

所以我的建议是：B 方案能做就做 B。除非端侧状态机的工期确实顶不上 630，再谈围栏。
