## 开场：一句话框死范围

各位好，今天讲的是 6.22 评审后的修正版全模块流程图。

**一句话：一个信号或一句话进了系统，云端和车端怎么分工、一步步把它办了。** 图中间那条虚线是端云分界线，跨越它的每一条粗线都要过网络，弱网会卡——这是全篇命门。

## 第一段：端侧 — 信号从哪来

### ①用户语音 → ②ASR

用户在车里说了一句话，比如 **"帮我照看一下后排的宝宝"**。①用户语音模块拿到原始音频，传给②ASR做语音转文字。

### ③端信号源（6类） → ④端侧信号库

除了语音，车上还有6类传感器信号在持续产生数据：
- **CDC信号**：车速、档位、转向角（比如"P档"、"车速60km/h"）
- **导航信号**：当前位置、目的地、剩余里程
- **蓝牙信号**：手机连接状态、来电状态
- **温度/环境**：车内温度、车外温度、雨量
- **埋点信号**：用户操作行为
- **VLM输出**：端侧视觉模型的感知结果（比如"舱内有一名儿童在后排"）

这些信号**全部写入④端侧信号库**，统一存储为标准格式的端状态快照。

**6.22 修正：端模只报客观状态。** 比如端侧视觉模型看到毯子，上报的是 **"坛子当前在地上"**，而不是"坛子掉了"——掉没掉是判断，判断交给上层业务逻辑做。这是丁彬和西钺在会上确认的。

> **举例：** VLM 每3秒输出一次观测："后排左侧座位有一名儿童，处于睡眠状态"、"后排地板有一条蓝色毛毯"。这些就是客观状态，写入信号库。

### ④端侧信号库的三条出路

信号库拿到数据后，有三条路走：

1. **→ ⑫云端信号库（端信号同步上云）**：所有端信号的全集同步到云端，云端信号库是端侧的镜像。
2. **→ ㉑Context（按策略上报）**：端侧信号库按策略（比如变化时才上报、或定时上报）把关键状态推到云端 Context，作为 Advisor 和 Director 决策的事实依据。
3. **→ ㉔端侧触发器引擎（读/订阅）**：端侧触发器读取信号库，执行本地判定逻辑。

## 第二段：端侧 — 语音走车控快通道

### ②ASR → ⑨拒识/仲裁/意图 → ⑤离线fc+白名单 → ⑥命中?

ASR把语音转成文字后，上传到云端⑨做意图判断。⑨是所有语音的第一关，它做分流：

- **车控指令**（比如"打开车窗"、"把空调调到24度"）→ 走蓝色线送到端侧⑤离线fc+白名单
- **复杂需求**（比如"帮我照看后排宝宝"）→ 走灰色线直送⑳Director大脑
- **条件任务**（比如"10分钟后关座椅加热"）→ 走橙色线到⑩意图模型解析
- **拒识成功** → 写入㉑Context事件日志

**⑤离线fc+白名单** 是端侧的函数映射表。"打开车窗"这种指令，直接在端侧就能映射到车控API，不需要云端推理。

**⑥命中?** 就是判断白名单里有没有这个指令：
- **命中** → 直接送㉘车控执行，端侧闭环，延迟极低
- **未命中** → 回到云端⑲Director汇入，走完整推理链路

> **举例：** 用户说"把车窗打开"→ ASR转文字 → ⑨判断为车控 → ⑤白名单命中"车窗" → ⑥命中 → ㉘直接执行开窗。全程不需要大模型推理。

## 第三段：云端 — 条件任务注册链路

### ⑨ → ⑩意图模型解析 → ⑬规则配置平台 → ⑮云端触发器引擎

用户说了一句条件型的话，比如 **"10分钟后关座椅加热"** 或 **"到公司了提醒我拿外卖"**。

⑨判断这是条件任务，送到⑩意图模型解析，把自然语言拆成 condition + action 的结构化格式：
- condition: `时间=10分钟后` 或 `位置=到达公司`
- action: `关闭座椅加热` 或 `语音提醒"拿外卖"`

解析结果送到⑬规则配置平台查信号可用性，确认能注册后，**注册成⑮云端触发器引擎的一个场景**。

### ⑮云端触发器引擎 — 系统的调度中心

⑮是整个触发器系统的核心枢纽，它负责：
- **注册校验**：来自⑬的静态场景、来自⑰动态Advisor的动态注册、来自⑳Director的对话产生的条件任务，全部在这里注册
- **仲裁**：多个触发条件同时满足时做优先级仲裁
- **端云分类**：判断任务应该在云端执行还是下发到端侧
- **中转下发**：向端侧㉔下发任务（弱网也能跑）

**6.22 新增职责：事件组装 + push。** ⑮持续监听端侧信号库（通过⑫云端信号库的镜像），当检测到阈值穿越（比如"用户状态从睡眠变为清醒"）或首次出现（比如"首次检测到充电桩"），就组装成事件，push给Advisor。

> **举例：** 端侧VLM每3秒上报"用户处于睡眠状态"，某次上报"用户处于清醒状态"。⑮检测到"睡眠→清醒"这个阈值穿越，组装成"用户睡醒了"事件，push给订阅了这个事件的⑰动态Advisor。

## 第四段：云端 — Context、Advisor、Director 三角关系

这是系统最核心的决策链路，顺序是：**㉑Context → ⑯⑰Advisor → ⑲⑳Director**。

### ㉑ Context — 统一事实源

Context 是全系统的"事实黑板"，所有模块都从这里读取当前状态。它存的是：
- 端侧信号库按策略上报的车况/环境状态
- VQA模型注入的视觉观测结果
- 拒识模块写入的事件日志

**6.22 关键修正：Context 只存 condition/状态，不存事件！**

佳杰在会上当场纠正了泽民：

> 佳杰原话："我不会去轮询你的Context。我注册了事件之后，就等待你的云端触发器给我发这个事件。"

什么是 condition？ 是一个持续存在的状态描述：
- "车里有一名儿童"
- "当前车速80km/h"
- "坛子当前在地上"
- "用户最近一次停车位置：公司B2层"

什么是事件？ 是一个状态变化的瞬间通知：
- "用户刚刚睡醒了"
- "车辆刚刚驶出地库"
- "首次检测到后方有充电桩"

**condition 走 Context → Advisor；事件走 ⑮触发器 push → Advisor。** 这是6.22最核心的修正。

### ⑯静态Advisor ×4 — 常驻守望者

四个静态Advisor（舒适/出行/内容/情感）是长期运行的，不需要用户触发。它们：
- 从 Context 读取 condition（比如"车内温度32度"）
- 从 ⑮触发器 接收 push 事件（比如"用户上车了"）
- 综合判断后，给⑲Director汇入发送 advice

> **举例 — 舒适Advisor：** 读到 Context 中"车内温度32度" + 接到⑮ push 的"用户上车"事件 → 生成 advice："建议开启空调制冷至24度并打开座椅通风"→ 发给⑲Director。

### ⑰动态Advisor — 临时监工

动态Advisor是按需创建的，每个目标对应一个实例。比如用户说"帮我照看后排宝宝"，Director就会创建一个专门的动态Advisor实例来盯这件事。

它的工作方式：
1. 从⑱GoalList收到监工任务
2. 从 Context 读取 condition（"后排有儿童"）
3. 从 ⑮触发器 接收 push 事件（"检测到毯子位置变化"）
4. 必要时调用云端VQA确认情况
5. 综合推理后给⑲Director发 advice

**6.22 修正：** 动态Advisor 订阅事件等 push，不轮询 Context。这大大降低了系统负载。

> **举例 — "照看宝宝"：** Director创建动态Advisor → Advisor拆解出需要观测"毯子位置" → 向⑮注册"毯子位置变化"事件 → ⑮下发给端侧VLM → 端模持续上报"坛子在座椅上/坛子在地上" → 某次⑮检测到"座椅上→地上"穿越 → push事件给⑰ → ⑰调VQA确认 → 确认毯子掉了 → 给Director发advice："毯子掉了，建议提醒用户"。

### ⑱GoalList 共享看板

Director 和 Advisor 之间的共享任务板：
- ⑳Director 通过 goal_list_update 建/删目标
- ⑱把新目标分配给⑰动态Advisor建监工
- ⑰完成后回写进展到⑱

### ⑲Director汇入 → ⑳Director推理（大脑）

⑲是 Director 的输入口，它把三种信号同时送进⑳做推理：
- 用户的 query
- Advisor 们的 advice
- Context 中的 condition

⑳是大脑（Seed 1.8 大模型），输出三件事：
- `talk_or_not` — 要不要回话
- `talk_content` — 说什么
- `action_list` — 做什么（调哪些工具）

### ⑳Director 的输出去向

Director做完决策后，动作分发到5个地方：
1. **→ ㉒工具库** — 调23个下游工具执行（action→工具），工具结果回到⑲做二次推理
2. **→ ㉓任务中心** — 条件/定时任务写入 Widget 卡片
3. **→ ⑮云端触发器** — 对话中产生的条件任务去注册（比如"到了就提醒我"）
4. **→ ㉖TTS播报** — 话术发给端侧 TTS 播报
5. **→ ㉘车控执行** — 直接车控指令
6. **→ ㉙→用户** — 卡片/HMI展示

> **举例 — "帮我照看后排宝宝"的完整链路：**
> 1. 用户说"帮我照看后排宝宝" → ①②ASR → ⑨判断为复杂需求 → ⑳Director
> 2. ⑳建目标到⑱GoalList："监控后排宝宝状态"
> 3. ⑱建监工 → ⑰创建动态Advisor实例
> 4. ⑰拆解：需要持续视觉观测 → 向⑮注册"后排宝宝异常"事件
> 5. ⑮下发观测任务到㉔端侧触发器 → ㉔发运行指令给㉕感知器
> 6. ㉕调度端侧摄像头，端模持续上报舱内状态到④信号库
> 7. ④同步到⑫ → ⑮监听到"宝宝从睡着变哭了" → push事件给⑰
> 8. ⑰调VQA确认 → 确认宝宝在哭 → advice给⑲
> 9. ⑲+Context(后排儿童3岁) → ⑳推理："宝宝在哭，可能需要家长关注"
> 10. ⑳ → ㉖TTS播报："检测到后排宝宝在哭哦，您要不要回头看看？"

## 第五段：VQA 和 VLM 链路（重点讲清楚）

这是大家最容易混淆的部分。先理清两个概念：

### VLM vs VQA 是什么关系？

| | VLM（端侧视觉大模型） | VQA（云端视觉问答） |
|---|---|---|
| **在哪** | 端侧 AI Box 上 | 云端服务器 |
| **谁** | ㉕感知器调度的端模 | 云端VQA模型节点 |
| **做什么** | always-on 持续观测、输出客观状态描述 | 按需被调用、回答具体视觉问题 |
| **输入** | 摄像头实时画面 + 观测SP | 抓帧图片 + prompt问题 |
| **输出** | 客观状态文本（"后排有儿童在睡觉"） | 针对性回答（"是的，毯子掉在地上了"） |
| **写入** | ④端侧信号库 → ⑫ → ㉑Context | 直接注入㉑Context + 返回⑰ |
| **调度** | ㉔端侧触发器指挥㉕感知器 | ⑰动态Advisor通过视频通话接口调用 |

**简单说：VLM是车上的"眼睛"，always-on在看；VQA是云端的"放大镜"，有问题才问。**

### 链路一：VLM 端侧持续感知链路

```
㉔端侧触发器 → ⑦运行指令 → ㉕感知器(端侧)
                                    ↓
                              端侧摄像头(抓帧)
                                    ↓
                              端侧VLM模型(AI Box上运行)
                                    ↓
                              输出客观状态文本
                                    ↓
                    ㉕ → ④端侧信号库(⑤识别结果写信号库)
                                    ↓
                         ④ → ⑫云端信号库(端信号同步上云)
                         ④ → ㉑Context(按策略上报)
                                    ↓
                         ⑮云端触发器监听⑫的信号变化
                         检测到阈值穿越 → 组装事件 → push给⑯⑰
```

> **举例 — always-on 舱内监控：**
> - ㉕感知器每3秒调端侧摄像头拍一帧舱内画面
> - 端侧VLM处理后输出："主驾男性，清醒状态，双手握方向盘；副驾女性，睡眠状态；后排无乘客"
> - 写入④信号库 → 同步到⑫云端信号库
> - ⑮检测到"副驾从清醒变为睡眠" → 组装事件"副驾入睡" → push给⑯舒适Advisor
> - 舒适Advisor结合Context（车内温度22度，时间21:30）→ advice："建议调低音量、关闭氛围灯"

> **举例 — 动态视觉观测（看宝宝毯子）：**
> - ⑰动态Advisor拆解"照看宝宝"→ 生成注意力SP："当观测到毯子/被子位置变化时上报"
> - 向⑮注册 → ⑮下发给㉔ → ㉔指令㉕感知器新增此观测项
> - ㉕在always-on的舱内SP上**增量追加**此prompt（不覆盖原有SP）
> - 端模后续每次输出多一项："后排蓝色毛毯当前在座椅上" / "后排蓝色毛毯当前在地板上"
> - 写入④→⑫→ ⑮检测到"座椅→地板"穿越 → push"毯子位置变化"事件给⑰

### 链路二：VQA 云端按需问答链路

```
⑰动态Advisor → prompt → 视频通话接口
                              ↓           ↓
                         图+prompt    抓帧指令↓端
                              ↓           ↓
                       云端VQA模型   ← 端侧摄像头(帧上传↑云)
                              ↓
                    result回⑰ + 注入㉑Context
```

VQA 和 VLM 最大的区别：VQA 是**一问一答**，不是持续跑的。

> **举例 — 确认毯子掉落：**
> - ⑰动态Advisor收到⑮ push的"毯子位置变化"事件
> - ⑰需要确认细节，生成prompt："后排的毯子是否掉到了地上？宝宝身上还有没有盖着东西？"
> - 发给视频通话接口 → 接口向端侧摄像头发抓帧指令
> - 端侧摄像头拍一张高清帧 → 上传到云端
> - 图+prompt 一起送入云端VQA模型
> - VQA返回："是的，蓝色毛毯已从座椅滑落到地板上，宝宝身上没有覆盖物。"
> - result返回⑰ → ⑰确认毯子确实掉了 → 同时VQA结果注入㉑Context
> - ⑰给⑲发advice → ⑳推理 → ㉖TTS："检测到宝宝的毯子掉到地上了哦，要不要在下一个服务区停一下帮宝宝盖好？"

> **举例 — Director直接调VQA（search_visual_info工具）：**
> - 用户说"我车后面有没有充电桩？"
> - ⑳Director直接通过㉒工具库调 search_visual_info
> - 走同一条VQA链路：抓帧 → 上传 → VQA → 回答
> - VQA："后方5米处有一个特斯拉超充桩。"
> - ⑳ → TTS："您后方大约5米处有一个充电桩哦。"

### 链路三：VLM 和 VQA 的资源竞争问题

佳杰在会上特别提到：上汽那边虽然用的是云端VQA，但在端侧取图时**会和其他业务抢占**。比如VQA要抓帧的时候，always-on也在用摄像头，导致VQA用不了。

所以佳杰主张：**把所有信号类的东西都并到一个模块（信号库）去提供，事件类的也放到一个模块（触发器）去提供**，避免各自重复造轮子和资源竞争。

## 第六段：端侧触发器和弱网执行

### ㉔端侧触发器引擎

㉔是端侧的执行中枢。它从⑮接收中转下发的任务，从④读取端侧信号，做两级判定后执行。

**6.22 关键修正：scope 大幅收窄。**

仁杰在会上给出了清晰的判定标准：

| 归属 | 判定标准 | 举例 |
|------|---------|------|
| **CDC底软** | 常驻/固定/标配/key-value/整个车生命周期不变 | 起步安全、上下车安全、睡眠提醒、无感迎宾、充电大师、哨兵 |
| **端侧触发器** | 有生命周期/动态/可变/可运营/临时 | 云端下发的定时任务、动态视觉感知事件、通用视觉通道、可运营临时场景 |

> 仁杰原话："像起步安全，你放在触发器里面专门去配一下，反而还要让触发器去接底层信号、考虑稳定性和延时，不如直接在底软就把事做了。"
> 
> 西钺原话："不要开发两个触发器，不管从开发工作还是部署资源上都是浪费。端侧触发器只在CDC，VLM在AI Box。"

### ㉔的弱网执行能力

当网络断了（弱网），㉔可以独立执行：
- **→ ㉘车控执行**（红粗线：弱网本地执行）
- **→ ㉖TTS播报**（红虚线：弱网TTS）

> **举例：** 车进了隧道没信号，但之前云端已下发了"到达目的地500米时提醒"的定时任务。㉔端侧触发器在本地定时器到时后，直接调㉖TTS播报："快到目的地啦"——不需要云端参与。

### ㉕感知器（端侧）

㉕是端侧VLM的调度器，不是VLM本身。它负责：
- 接收㉔的运行指令（要观测什么）
- 调度端侧摄像头抓帧
- 管理always-on + 动态观测任务
- 用**阈值穿越**（last=False & this=True）做上升沿检测，防止重复触发

感知器的结果写回④端侧信号库，形成闭环。

## 第七段：6.22 新增区域

### 赛力斯 CDC 底软 · 固定闭环区

这是 6.22 最大的 scope 变更。以下功能**全部归还赛力斯 CDC 底软**，不走触发器：

| 功能 | 为什么归CDC |
|------|-----------|
| 起步安全 | 所有车标配，不变化，纯key-value判定 |
| 上下车安全 | 固定规则，不需要动态配置 |
| 睡眠提醒 | 常驻功能，设置菜单里的button |
| 无感迎宾 | 系统feature，不是场景引擎 |
| 充电大师开充电盖 | 赛力斯自己的业务逻辑，直接读信号执行 |
| 哨兵模式 | 赛力斯端侧闭环 |

CDC底软直接读④端侧信号库 → 直接执行㉘车控，不绕触发器链路。

> 佳杰举了个很好的例子区分：**"无感迎宾"是系统功能**——它出现在设置列表里有个专门的button，用户enable/disable。而**"靠近公司时推荐去哪里拿咖啡"是场景引擎**——它有生命周期，是动态的。前者归CDC，后者归触发器。

### 任务中心（端侧）

佳杰的新依赖。端侧需要一个持久化任务库，支持：
- **任务级 vs 感知级**字段区分：用户能看到的"帮我照看宝宝"是任务级，底层注册的"观测毯子位置"是感知级
- **enable/disable/删除/添加回来**接口：用户在任务中心UI上操作，端侧触发器要真的关停对应任务
- **端云任务同步**：端侧任务库 ↔ ㉓云端任务中心双向同步

> **举例：** 用户在任务中心看到一条"照看后排宝宝"任务，点了"暂停"。这个disable信号写入端侧任务中心 → 同步给㉔端侧触发器停止相关观测 → 同步给㉓云端任务中心更新状态。

### AIBox 任务调度（佳杰）

任务优先级排序在佳杰的 AIBox 模块里做，不在触发器。比如 always-on 的优先级、VQA 的优先级、其他视觉需求的优先级——优先级低的排队等。

### 通用视觉事件通道

这是兜底长尾的"万能通道"。当预设的视觉事件集都不能满足需求时，通过这个通道透传动态query给AIBox。

分三路：
- **舱内静态**：性别、年龄等长时间不变的
- **舱内动态**：动作、表情等快速变化的
- **舱外**：只用前摄

**关键：prompt 是增量追加，不是覆盖。** 否则加了新的观测项，原来的感知就没了。

> **举例：** 动态Advisor需要观测一个全新的东西——"后排有没有人在吃东西"，现有事件集里没有这个。走通用视觉通道 → 生成prompt增量"观测后排乘客是否在进食" → 追加到舱内动态的SP里 → 端模后续输出多一项："后排乘客正在吃三明治" → 写入信号库 → 云端触发器组装事件 → push给Advisor。

## 第八段：⑮→Advisor push 事件 — 最关键的修正

这是 6.22 会议最核心的修正，用一个完整 case 把它串起来：

**场景：用户说"倒完车之后，如果车里有小朋友，就把儿童锁打开"。**

**佳杰在会上的经典拆解：**
- "用户倒完车" = **事件**（等⑮ push）
- "车里有小朋友" = **condition**（读㉑ Context）
- "把儿童锁打开" = **action**（Advisor给出）

**完整链路（修正后）：**

1. 用户语音 → ASR → ⑨判断为条件任务 → ⑩解析 → 得到 condition="倒车完成 & 车内有儿童", action="开儿童锁"
2. → ⑬注册 → ⑮云端触发器建立场景
3. ⑮监听⑫云端信号库中"档位"字段
4. 某时刻，端侧信号上报"档位从R变为P" → ⑮检测到阈值穿越 → 组装事件"倒车完成"
5. ⑮ **push** "倒车完成"事件给订阅了此事件的⑯静态Advisor（舒适Advisor）
6. 舒适Advisor收到事件后，去㉑Context读condition → 发现"后排有一名3岁儿童"
7. 事件+condition都满足 → 给⑲Director发advice："建议开启儿童锁"
8. ⑲汇入 → ⑳推理 → action_list: [vehicle_basic_control: 开启儿童锁]
9. ⑳ → ㉘车控执行 + ㉖TTS："检测到您倒好车了，后排有小朋友，已帮您开启儿童锁哦。"

**如果不修正（旧链路错误做法）：**
- Advisor 轮询 Context → "档位=P" 什么时候变的？不知道！Context 只存状态不存变化时间点
- Advisor 要自己判断"从R变P"这个变化 → 每次都要和上一次对比 → 轮询开销大、逻辑复杂、容易漏

**修正后（正确做法）：**
- ⑮触发器专门做"状态变化检测+事件组装" → push给Advisor → Advisor只需要等推送 → 简洁高效

2. **端模条件式注意力能力**：端模能不能做到"看到X才上报"这种prompt限制？丁彬说"通过prompt能限制住，但得试一下，目前还没试过"。如果做不到，就只能全量上报再由上层过滤。

## 收尾：一句话总结

修正后的链路最关键的变化是两个字：**解耦**。

- **状态和事件解耦**：Context 存状态（condition），触发器 push 事件（event）
- **感知和判断解耦**：端模只报客观状态，判断交给上层
- **固定和动态解耦**：固定闭环归 CDC 底软，动态任务走触发器
- **感知和调度解耦**：感知在VLM/触发器，调度在佳杰AIBox

这四个解耦做到了，整个系统就清楚了。
