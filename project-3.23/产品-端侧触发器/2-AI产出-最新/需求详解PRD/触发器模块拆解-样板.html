📌 这个模块是什么

让触发器认识"导航 + 距离"，产出 接近 / 到达 / 离开 三种事件，给动态advisor（弋途）订阅，从而做"快到公司提醒拿咖啡"这类主动服务。

🎬 一个完整 case（带编号，比图16更细）
  1. 用户 8:00 导航去公司，端上（金龙）报 导航状态=导航中、目的地=公司、剩余距离=15km → 写入 端侧信号库
  2. 端侧信号库 增量同步到云端信号库（延迟<1s）
  3. 云端触发器引擎 每 5s 扫描云端信号库 → 剩余距离一路递减（15km→…→501m，都不触发）
  4. 8:25 剩余距离 首次 ≤ 500m → 命中"接近公司"事件（one shot，状态置"已接近"，之后 499m…不再触发）→ 频控通过 → 无冲突
  5. Callback 路由：把"接近公司"事件 透出给动态advisor（弋途） + 写入 Context
  6. 动态advisor 消费事件 → 产出 advice"提醒拿咖啡" → Director 推理 → 播报"快到啦，要不要点杯咖啡"
⚙️ 配置平台（火山后台）配哪些、怎么配
  • signals（注册信号）：注册 3 个导航信号 —— 导航状态 status、目的地位置 address + distance。现在被删/没开放，要刘杨重新开放
  • scene（注册场景/事件）：配"接近"事件 → condition = 剩余距离 ≤ 阈值，trigger_type = one_shot
  • 阈值：500（单位 米/千米 待金龙确认）；到达事件再加 档位变更为 P 组合条件
  • execute_policy（频控）：one shot 类设"单次上电触发 1 次"，避免重复
✅ 现状（配置平台现在有什么）

配置平台现有：上汽车控信号（车窗/车内灯/上车场景等）、部分静态条件。导航的"目的地位置/剩余距离"信号被删了/没开放；VLM/视觉条件还没配过（后面支持）。

🕳 还缺什么
  • 导航 3 信号未开放（要刘杨开）
  • "接近"=距离单调递减到阈值的 one shot，现有协议不支持（要刘杨定方案 A 改协议 / B 端上算好事件）
  • 距离单位、时间格式未定（金龙/产品定）
  • distance 的大于小于运算符引擎是否支持（任军确认）
