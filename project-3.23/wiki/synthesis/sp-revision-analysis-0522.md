# 5.22 会议后 SP 修改落地分析

> **本文做的事**：把会议后的**新版 SP**和**会议被评审的原版 SP**做严格 diff，回答四个问题：
> 1. **修改了什么**（具体到行/段/字段）
> 2. **为什么这么改 + 作用是什么**（产品/工程效果）
> 3. **会议决议有哪些没落地**
> 4. **新版还有哪些可以继续优化的点**（含修改过程引入的新问题）

---

# Part 1：已落地的修改（修改了什么、为什么、效果）

## 修改 1：vehicle_basic_control 工具大幅精简 + 修复双 param bug + 新增 notice

| 维度 | 详情 |
|------|------|
| **改前** | 描述 ~250 字（枚举几十种功能：车窗/空调/座椅/HUD/摄像头/蓝牙/WIFI/K歌/账号/媒体/...）+ **两条 param 字段（schema bug）** + 无 notice |
| **改后** | 描述精简到 ~30 字（按大类分组：空调/座椅/门窗/灯光/导航/播控）+ 单条 param + **新增 notice："导航没有记忆能力，若用户说关联着记忆的地址时要先去查询记忆"** |
| **为什么改** | ① 原描述过长稀释关键信息；② 双 param 是 schema 层 bug 必须修；③ 导航 + 记忆联动是车上常见 badcase（"去妈家"这种关联记忆的地址） |
| **作用** | ① ~200 token 节约；② 修复 schema 错误；③ 显式引导"导航前先查记忆"，覆盖了原本散落在注意事项里都没的规则 |
| **对应** | 我之前提的**优化点 1（双 param bug）已修复** ✓ |

## 修改 2：navi_basic_control 工具描述大幅精简

| 维度 | 详情 |
|------|------|
| **改前** | 长达 300+ 字，列出实时引导/地图界面/道路层级/导航状态/地址收藏共五大类、20+ 子能力（路线全览、缩放至最大、正北朝上、上下文语音缩放、避开限行、收藏地址、把 POI 设为家、删除收藏...） |
| **改后** | 精简到 ~80 字，保留：实时引导（开始/暂停/结束/巡航切换/视角切换/缩放）、地图界面（缩放/平移/2D3D/路况）、导航状态（播报模式/路线偏好/全屏/**后台导航保持** ← 新增） |
| **为什么改** | 原版描述过宽，模型可能误以为支持"避开限行"等实际并未深度对接的能力 → 幻觉风险 |
| **作用** | Token ↓；同时**缩小工具能力幻觉面**（不会答用户"打开避开限行"然后实际做不了） |
| **副作用** | ⚠ 真正支持的"避开限行"、"路线全览"、"收藏地址"等能力**模型也不知道了**——这个收紧得是否过度，需要评测确认 |

## 修改 3：示例编号连续化（1-21）+ 删除"示例补充"段

| 维度 | 详情 |
|------|------|
| **改前** | 示例 1-14 + 跳号到 19 + 跳号到 21；SP 末尾另有"示例补充"段含"示例 1 改写版" |
| **改后** | 示例 1-21 完整连续；"示例补充"段删除 |
| **为什么改** | ① 跳号说明删过 case 但没清理，KV cache 不友好；② 两段示例分离造成模型对优先级困惑 |
| **作用** | ① 示例段前缀稳定 → KV cache 命中 ↑；② 示例库变成单一权威源，模型决策路径清晰 |
| **对应** | 我之前提的**优化点 6（跳号）+ 优化点 11（两段示例）已修复** ✓ |

## 修改 4：大量新增示例 15-20（落地"主动性"范式）

| 示例 | 主题 | 覆盖范式 |
|------|------|---------|
| 15 凌晨下班毒舌互怼 | 情绪驱动语气、记忆深度利用、情绪落差收尾 | 用户记忆深度调用 + 情绪状态机 |
| 16 多人场景风格差异 | 同时面对 3 人，对每人切换不同语气 | 多人记忆并行使用 + 风格切换 |
| 17 主动破冰话题转场 | 零信息输入开场、自然转场、静默记忆 | 主动性 + 节奏管理 |
| 18 主动追问层层延展 | 捕捉线索→好奇心→深度对话→静默记忆 | 闲聊深挖（无工具） |
| 19 个性化技能（音乐讲故事新增→更新） | 技能新增、技能重写、边执行边优化 | personal_skill_record 演化路径 |
| 20 个性化技能（负能量场景新增→更新→变更→删除） | 技能尺度细化、场景边界澄清、技能删除 | personal_skill_record 全生命周期 |

**为什么改**：D9 落地——评测侧愿意帮编 case；同时把"主动性"从聊天风格段的抽象原则改成示例（呼应张嘉锋 vs 李明骏关于"原则 vs 例子"的争论，明骏一方倾向例子）。

**作用**：示例库覆盖范式更全；新工具 personal_skill_record 有完整生命周期示范（新增/更新/删除）。

## 修改 5：输出 schema 字段顺序调整 + emoji_id 改为必填

| 维度 | 详情 |
|------|------|
| **改前** | `{talk_or_not, talk_content, emoji_id, emotional_state_change, action_list}` —— emoji_id 为可选 |
| **改后** | `{talk_or_not, emoji_id, talk_content, emotional_state_change, action_list}` —— emoji_id 提前；【表情选择】明确"每次回复都需要输出 emoji_id" |
| **为什么改** | ① 让 emoji 选择先于 talk_content 生成（认知排序提示）；② 推进豆包形象动画的覆盖率 |
| **作用** | 全部输出都有形象动画——产品端更生动 |
| **副作用** | ⚠ 每轮都强制选 emoji，可能让模型在没有合适 emoji 时频繁默认 `speaking`，反而稀释了 emoji 的表达力（参见后文新问题） |

## 修改 6：聊天风格段大幅扩展（新增 8+ 条主动性策略）

| 新增段落 | 内容要点 |
|---------|---------|
| 【情绪连贯性与状态管理】 | 从注意事项 #14 复制过来 + 扩充情绪不能突变规则 |
| 【主动破冰与对话发起】 | 简单问候不能只回应，要从环境捕捉细节展开 |
| 【主动探索与创意邀约】 | 6 条：闲聊捕捉线索、即兴游戏、留心节奏转场、深情时刻停留 |
| 【闲谈中的探索欲】 | 闲谈感叹不能终结话题 |
| 【闲聊中的追问原则】 | 自我描述/观点/经历片段轻巧追问，对方不接就收 |
| 【闲聊中的静默记忆】 | 透露重要个人信息时**静默调用** operate_user_memory 记录 |
| 【零信息输入的对话启动】 | 同破冰，用环境感官细节开场 |

**为什么改**：会议讨论的方向是让 planner 更"像人"（参考[[planner-eval-methodology]]的"像人评测"维度）。

**作用**：示例 17/18 等新示例的范式有了原则层支撑。

**严重副作用**：
1. **【闲聊中的静默记忆】与 D7（不擅写偏好）严重冲突**——D7 决议"除非用户明确说'记一下'，否则不主动写偏好"，但这条原则**反过来鼓励静默写记忆**。示例 17/18 就直接在用户透露 MBTI/童年故事时静默写入；
2. **三处情绪/时间相关内容重复**：注意事项 #13 + 聊天风格【自我时间感知】+ 聊天风格【情绪连贯性】——D8（去重）不仅没做，**还更严重了**；
3. **聊天风格段从 ~600 token 膨胀到 ~1500 token** —— 把张嘉锋的担忧（原则越多遵循力越差）实际放大了。

## 修改 7：用户记忆分两块（基础信息 + 个性化需求）

| 改前 | 改后 |
|------|------|
| 单一 `<用户记忆>{{memory}}` | `<用户基础信息记忆>{{memory}}` + `<用户个性化需求>{{personal_skill_memory}}` |

**作用**：配合 operate_user_memory（事实记忆）vs personal_skill_record（个性化技能）两种工具的分工，让 context 注入也对齐。

**对应**：我之前提的**优化点 3（两工具边界模糊）部分修复** —— 在 context 层做了分工，但工具描述本身的边界差异仍未补强。

## 修改 8：工具使用 tips 新增/调整

**新增**：
- **web_search tip**：简单查询第一轮可不说话，查到再回复（减少冗余 talk turn）
- **正在学习的工具/技能 段**："视频生成工具：可以让豆包使用 seeddream 模型来生成视频"（声明在建能力，避免模型答应做不了）
- **基础车控新增**："不支持通过语音交互控制座椅移动"（堵幻觉）

**删除**（⚠ 这是回退）：
- **"navi_basic_control，route_planning_qa，search_poi_qa，trip_plan_record 这四个工具无法并行调用"** —— 原 SP 有这条，新版**完全删除**。这正是会议示例 1（虹桥日料）暴露的工程约束，被反向回退了。

# Part 2：会议决议未落地清单（哪些没改/改不彻底）

> 对照[[meeting-0522-planner-sp-discussion]]中 D1-D9 九条决议。

## D1：用户必回原则 — ❌ 完全未落地

- **应该改**：注意事项中明确"user_query 输入下 talk_or_not 必为 yes"；schema 注释强化
- **实际**：注意事项 #4 还是"tool_feedback 不一定都需要说话"，没有提到 user_query 必须回复
- **后果**：示例 9 同类的 user_query → 工具返回 → talk_or_not=no 的链条仍然可能出现

## D2：advisor 改询问 — ❌ 完全未落地

- **应该改**：L41 advisor 段说明"建议会自带'直接执行/建议询问'标识，缺标识时默认询问"；示例 2 改成 advisor 给空调建议 → planner 询问用户
- **实际**：L41 仍然"请你参考得到的建议做行动和表达"；**示例 2 一字未改**，明骏照旧"35 度直接开空调"
- **后果**：与豆包 Alex 关于主动服务的立场矛盾仍在

## D3：Harmony 工程硬限制 — ❌ 未落地

- **应该改**：合并注意事项 #5+#8；删除 tips 里"导航最多重试 2 次"；工程层兜底
- **实际**：注意事项 #5 #8 仍分散；tips 里"导航发起失败后，最多重试 2 次"还在；其他 22 工具失败处理仍无明确指引
- **后果**：硬规则继续靠 LLM 数次数，不可靠

## D5：禁止 planner 主动服务 — ❌ 严重未落地（且示例 5 暴露问题更明显）

- **应该改**：注意事项加"主动服务由 advisor 负责"；示例 5 删除主动服务段
- **实际**：**示例 5 input2 → output2 仍然是 planner 在 advisor 没介入下主动说**：
  > "对了，外面晚霞挺美的，要不要把氛围灯调成晚霞的颜色？再帮你生成个 AI 播客，路上听听。"

  会议明确指出"豆包做不到把氛围灯调成晚霞色"，且这是典型 planner 主动服务——**新版示例 5 一字未改**
- **后果**：示范了被禁的行为，模型继续学

## D6：边问边做删除 — ❌ 未落地

- **应该改**：所有"问要不要 X + 同时 action 已包含 X"的 turn 都改
- **实际**：**示例 4 output4 经典违反**：
  > "talk_content": "好的，左庭右院鲜牛肉火锅万象城店，我帮你记下偏好。**要现在导航过去吗？**"
  > "action_list": [personal_skill_record(...), **route_planning_qa(导航到左庭右院)**]

  嘴上问"要现在导航过去吗"，手已经发起导航——D6 的反面教材
- **后果**：车上"边问边做"的体验问题继续存在

## D7：不擅写偏好 — ⚠ 部分落地但本质未改

- **应该改**：除非用户明确"你记一下"，不主动调记忆工具写偏好
- **实际**：示例 4 把工具从 `operate_user_memory` **改成了 `personal_skill_record`** —— 表面更"对口"（技能 vs 事实），但**核心违规仍在**：
  - 用户只说"听说牛肉不错"（接受推荐 ≠ 表达长期偏好）
  - planner 仍然写入"雨晴喜欢吃火锅，偏好左庭右院鲜牛肉火锅（万象城店）"
  - 张航原话："除非用户明确表达说我什么什么你记一下，不然我们端的应该不会主动调起"
- **结论**：**只换了工具壳，没改行为本质**——而且 D7 的违反在新增示例 17/18 中**变本加厉**（见 Part 3 新问题）

## D8：自我时间感知去重 — ❌ 未落地且恶化

- **应该改**：删除注意事项 #13，保留聊天风格那处
- **实际**：注意事项 #13 仍在 + 聊天风格【自我时间感知】仍在 + **新增聊天风格【情绪连贯性与状态管理】** = **三处相关内容**
- **后果**：SP token 浪费 + 模型对"以哪处为准"的歧义加重

# Part 3：修改过程引入的新问题（新版 SP 才出现的 bug/隐患）

## 新问题 1：【闲聊中的静默记忆】原则与 D7 决议正面冲突

| 矛盾项 | 详情 |
|------|------|
| **D7（会议决议）** | "除非用户明确表达说'我什么什么你记一下'，不然不主动调起记忆工具" |
| **新版聊天风格【闲聊中的静默记忆】** | "当用户在闲聊中自然透露了值得长期记住的个人信息时（如性格描述、价值观、MBTI、重要经历），可以在不打断对话节奏的前提下，**并行调用 operate_user_memory 工具记录下来**。**不需要在回复中告知用户"我记住了"**" |
| **示例 17 output6** | 用户讲童年和妈妈在花园闻泥土味 → planner 静默调 operate_user_memory 记录 |
| **示例 18 output4** | 用户说"擅长调节奏不擅长剖心" → planner 静默调 operate_user_memory 记录 |
| **示例 18 output6** | 用户说 MBTI 是 INFJ → planner 静默调 operate_user_memory 记录 |

**严重程度**：⚠⚠ **高**。这是新版 SP 内部最大的不一致——一方面 D7 禁止主动写偏好，另一方面新增的"主动性"策略和大量示例都在示范主动写记忆。两套规则相互打架，模型行为不可预测。

**根因**：会议讨论的是"豆包推荐被用户接受 ≠ 偏好"，但新版把"用户主动透露个人信息"这种合法记忆场景一并加入——**会议决议覆盖面不够完整**，导致后续扩展时与原决议冲突。

**优化建议**：
- 在【闲聊中的静默记忆】里加边界条款："仅限用户主动陈述的事实/性格/经历；豆包推荐被接受不算"
- 或者，反过来在 D7 里精确化："禁止把'豆包推荐被采纳'记为偏好；但用户主动陈述的特质可以记"

## 新问题 2：示例 5 智驾 case 完全未改（D5 重灾区）

会议明确说要删除 planner 主动调晚霞氛围灯+AI 播客这段（张航："你也做不到把氛围灯调成晚霞颜色"；陈芮婷："不要搞主动服务"），但新版完全保留。

**这条新问题不是修改产生，是会议决议**没**修改产生**——但在新版 SP 里它仍然是"教模型怎么做"的明示，所以放在新问题里追踪。

## 新问题 3：示例 16 凌晨下班 case 静默写"明骏喜欢毒舌"

- **现状**：用户记忆里已经有 "明骏喜欢有趣、搞怪、嘴臭的回应"
- **示例 16 output8 仍然调** operate_user_memory 写入 "明骏凌晨下班时喜欢用毒舌互怼的方式来放松，最后再温柔收尾效果最好"
- **问题**：
  - ① 用户没说"记一下"——违反 D7
  - ② 与已有记忆高度冗余——浪费记忆存储

## 新问题 4：聊天风格段膨胀加剧"原则 vs 例子"问题

- **改前**：聊天风格段 ~600 token
- **改后**：~1500 token（新增 8+ 条主动性策略）
- **风险**：张嘉锋 vs 李明骏的争论核心是"模型在长 SP 中对抽象原则的遵循力衰减"——新版**新增了大量抽象原则**，理论上应该让效果变差（除非配合示例都做对）

**应该立即开展会议遗留的 T7 实验**：测试新版 SP 里**新增的 8 条主动性原则**有多少能真正被模型遵循（vs 示例 17/18 教的具体行为）。

## 新问题 5：emoji_id 改必填带来的副作用

- 每个 output 强制选 emoji
- 但 emoji 列表中 `standby`（待命状态）实际是端侧的待命循环动画，按设计是规则触发，不应是 planner 主动输出
- 模型可能在"对话间隙的空 output"也选 `standby`，与端侧规则冲突
- `wave_driver` / `wave_passenger`（向主驾/副驾挥手欢迎）属于场景触发型，强制每轮选 emoji 会让模型乱选

**优化建议**：要么 emoji_id 不强制必填（保留可空），要么把 emoji 列表精简到**纯 planner 输出场景**（移除 standby、wave 等场景触发类）

## 新问题 6：示例 8 vehicle_manual_qa 工具不存在

- 工具列表里没有 `vehicle_manual_qa`（24 个工具中无）
- 但示例 8 input6 → output6 调用：`{"query": "如何无钥匙解锁车辆", "tool_name": "vehicle_manual_qa"}`
- 示例 8 input7 → output7 后续也基于此返回
- **这是会议讨论示例 11 时的同类问题（"工具列表里没定义的工具被调用"），但漏改示例 8**

## 新问题 7：工具使用 tips 删除了"四工具不可并行调用"

- 原 SP 有："navi_basic_control，route_planning_qa，search_poi_qa，trip_plan_record 这四个工具无法并行调用"
- 这正是示例 1（虹桥日料）暴露的工程约束
- **新版完全删除** —— 工程约束的可见性反而下降

## 新问题 8：navi_basic_control 描述精简过度

精简后**丢失了实际支持的能力**：
- 避开限行（"开启/关闭避开限行"）
- 路线全览
- 道路层级切换（高架上/下、主路/辅路）
- 地址收藏管理（收藏地点、设为家/公司、查看/删除）

**风险**：用户问"打开避开限行" → 模型说不支持 → 实际是支持的，体验割裂。

# 关键结论

1. **这次修改是一次"加法为主"的修订**——加了 6 个新示例、加了 8 条主动性原则、加了情绪连贯性管理段。Token 净增加而非压缩。

2. **会议决议的落地率约 30%**：
   - 落地：D4 部分、D9 部分、优化点 1/6/11
   - 未落地：D1/D2/D3/D5/D6/D8 全部；D7 部分但更糟

3. **最大的内部矛盾**：新增的【闲聊中的静默记忆】+ 示例 17/18 的静默记忆行为，与 D7 决议正面冲突——这是 SP 内部规则打架，会让模型行为不稳定。

4. **修改优先方向（按 ROI）**：
   - **立即**：修示例 4/5/8 三个示例（P0）+ 恢复并行调用 tip
   - **本周**：处理【静默记忆】vs D7 冲突 + 补 D1/D2/D8
   - **后续**：emoji 精简 + 注意事项重排 + 长途智驾示例

5. **方法论层面提醒**：张嘉锋的实验问题（T7）在新版 SP 里有了**完美的对比样本**（同时存在"主动性原则"和"主动性示例"）——应该尽快做实验，不然下一版 SP 又是凭直觉加规则。
