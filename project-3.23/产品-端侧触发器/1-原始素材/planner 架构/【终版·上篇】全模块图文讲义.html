1.2 讲解：整套系统在干什么

豆包上车是搭载在上汽荣威 D6X（五座 SUV）上的车载 AI 助手。系统被设计成三层漏斗：第一层句法 RAG（规则匹配，毫秒级）→ 第二层情景 RAG + 仲裁（轻量模型判简单/复杂）→ 第三层 Director/Planner（大模型深度推理）。设计三原则（原文）：① 层层过滤——能用简单方法搞定的不要漏到后面；② 分阶段注入知识——注入方式从固定到动态；③ 数据回流——Planner 处理的难 case 训练前面的层。

在漏斗之上叠了一层目标驱动协作：Director 只管"快速响应+建/删目标"，做不完的任务交给 advisor（永久监工×4 + 临时监工）持续盯进度，通过 goal list 共享看板协作。感知订阅让 VLM/触发器/定时器自动唤醒 advisor 推进任务。

全系统的交互走单通路语音管道（同时只能回一次），多个输入同时到达时按优先级打断/合并/排队。

一句话概括：用户说话 → 三级拒识分流 → 简单走端侧 150ms / 复杂进 Director → Director 用 ReAct + 26 工具把需求拆解执行 → 做不完的升级成 goal + advisor 持续盯 → 感知订阅自动触发推进 → 单通路管道把结果说给用户。

2.1 流程图：一句话怎么被分流

2.2 讲解：为什么是三级拒识

约 30% 的车控指令是模糊/复杂表达（吉利量产数据），单看 query 分不准。所以系统用三级冗余换稳定：

第一级（端侧白名单+LLM）：车上本地的小模型配白名单。"打开车窗"这种明确高频指令直接命中，端侧就地执行约150ms。越多流量在这里拦住，系统越快。

第二级（云侧前置拒识）：白名单没命中的上云。拒识模型判定"妈妈帮我拿水杯"不是车机能力→reject。但reject不丢弃——写进context事件日志被advisor感知（可能触发"车里有小孩需要关怀"的主动服务）。

第三级（Director内部兜底）：Director在plan_describe前再拒一次，防漏网的消耗推理资源。

仲裁模型（Seed1.6 flash SFT）：拒识放行后判"简单or复杂"。输入是query+情景RAG注入的few-shot（{"type":"modelExecute","knowledge":"...","agentLimit":[...]}，不直接给答案只给参考）。简单→FC快车道(300ms+150ms)；复杂→Director慢思考(500ms+500ms)。

同一句"我好热"：空调没开时→简单车控→端侧调温；空调已开到最冷→需要结合车况判断→复杂→进Director→回复"已经开最低了"。仲裁要结合情景RAG的车况知识才能判对。

= Planner = 对话推理引擎。底层Doubao-Seed-1.8，ReAct+Function Call范式。

3.1 流程图：一次调用的ReAct闭环

3.2 讲解：三种输入各是什么

user_query（用户说的话）——真身带 7 个字段：query（话本身）、speaker_name（说话人）、speaker_position（座位）、timestamp（ISO8601，用于推算时间间隔——超 30 天未见→"好久不见"，超 5 分钟→确认上个任务）、assistant_emotional_state（上一轮情绪，保持连贯）、goal_list（当前目标队列）、env_info（最新车辆状态）。后两个字段随输入直传——车况和目标不用再调工具查，降时延。

advisor——{"advisor":"车内温度高，建议开空调降温"}。这不是用户说的话，是 advisor 的主动服务建议。Director 不回应 advisor，直接面向用户做行动和表达。

tool_feedback——{"tool_feedback":"导航成功","tool_name":"route_planning_qa"}。工具执行完的返回，触发 Director 二次推理。简单成功（车窗关了）Director 通常选择不说话。

3.3 讲解：五个输出字段

talk_or_not：可以选择不说话——车控成功、音乐开始播、导航画面出来了，不必多嘴。emoji_id（★新增）：17 个预定义动画之一驱动屏幕上豆包的形象。emotional_state_change（★新增）：仅情绪变化时输出。6 种状态：平静（默认）/ 欣喜 / 关切 / 低落 / 生气 / 好奇。规则：不突变（低落→不能直接欣喜）；关切惯性（用户没好转不能秒变欣喜）；久别可直接欣喜。

"去昨天游泳馆"两轮ReAct：第一轮→Director调search_user_memory查记忆→返回"昨天去了虹桥游泳馆"→第二轮→调route_planning_qa导航→返回"导航成功"→Director不说话（用户能直接看到导航）。一次调用、两轮推理、两次工具调用。

3.4 讲解：Director的职责边界

P0 目标增删 · P0 动态advisor增删 · P1 合理性仲裁（感知条件合理？动作可执行？矛盾？）。

关键约束：Director 明确"不更新"目标的"细节和进展"——写权限收归 advisor，避免双写冲突。Director 只管"建/删+快速响应"，"盯进度/改进展"交给 advisor。

四、SP 结构与优化：Director 脑子里装了什么、怎么瘦身的

SP=System Prompt，是Director每次推理时读的"大脑"。13块骨架，从32806 tokens优化到16000 tokens。

4.1 骨架图：SP 13块的真实排列

4.2 讲解：为什么"固定在前动态在后"

①~⑫ 固定前缀每次调用完全匹配 → KV Cache 复用已算好的注意力向量，不重算这几万 token。只有 ⑬ 的动态部分需要算新 token。把"变"的放最后，缓存命中率最大化。

4.3 讲解：SP 优化六项措施

  1. 输出格式压缩——talk_or_not→speak省2token，talk_content→content省1token，首token降~90ms
  2. 示例拆分静态+动态——64个全量→静态10个常驻+动态5个按query检索加载（最大的一项压缩）
  3. 检索加载——示例放入VikingDB向量引擎，语义检索top-5（需解决多轮query改写）
  4. 示例压缩（抽象化）——删掉tool_feedback中冗余字段（完整POI信息→只留预估时间/里程/红绿灯）
  5. 基于人工反馈——用error case持续迭代示例（如天气查询缺位置→加示例教模型推断位置）
  6. 渐进式架构——静态示例+skills分类+动态检索，三级加载

4.4 D6X设备知识：防车控幻觉的红线

系统红线（不支持/受限）
空调后排不可调温/风量（跟随主驾）；制热=AC+≥27°C
座椅按摩仅前排，后排不支持；靠背不支持度数
氛围灯全车统一，不分区
导航无记忆；选点用序号不用具体名；途经点≤15
智驾/雨刮/车门/儿童锁/天窗/HUD不支持语音控制/无HUD

4.5 26工具三个易混点

  1. 媒体播控（上一首/暂停/循环）归vehicle_system_settings，不归音乐工具（后者只管搜歌）
  2. 记忆分两类：客观事实→operate_user_memory；交互习惯→personal_skill_record
  3. 4导航工具不可并行：navi_basic_control/route_planning_qa/search_poi_qa/trip_plan_record

五、公共Context + 前置筛选（知识检索）

所有模块共享的"现场与记忆"，加上token优化的核心落点。

5.1 流程图：Context的写入→预处理→注入

5.2 讲解：Context 13类内容

P0：情景描述(感知Always on)/历史对话(ASR-TTS)/事件日志/端状态(仅最新)/屏幕截图(近5min)/用户记忆(按人物id)/任务状态队列/基础联网。P1：豆包探索/通用任务经验。P2：工具知识/专有知识/性格。

优化不是"远近区"，而是：①不全量加载按需抽取 ②情景描述压缩总结 ③三维过滤（按场景：驻车不加载行车记忆；按人物：过滤非共同记忆防隐私泄露；按任务：点外卖只检索外卖知识）。

5.3 讲解：前置筛选（知识检索专家）

为什么做：SP原来32806 tokens太大→注意力稀释→推理质量降→时延高。用一个独立小模型先跑一遍，只挑当前场景相关的几条知识注入。

怎么做：输入5字段（query/近10轮对话/goal_list/env_info/已加载知识ID）→ 在6类专家×66条知识库里匹配 → 输出 {"add_knowledge_ids":["COM-01-01-01"],"remove_knowledge_ids":[],"reasoning":{...}}（增量更新只输出变化）→ 注入SP末尾{{动态专业知识}}块。

6类专家：EXP(销售17条)/DRV(司机20条)/COM(管家8条)/ENT(娱乐7条)/LIFE(助理7条)/EMO(伙伴7条)。5筛选原则：场景绑定/关注变化(趋势>绝对值)/目标持续关联/增量更新/不足不虚构。

"好热啊"+env_info显示温度5分钟从22°C升到29°C → 原则②触发 → 命中COM → add:COM-01-01-01(调温度) → 该知识注入SP → Director推理时就能参考"分区温控/用户偏好联动"来决策。

六、goal list + 交互冲突 + 执行层

目标看板、单通路治理、26工具执行与HMI呈现。

6.1 goal list 流程图：一个目标的生命周期

读写权责：创建/删除=Director；更新进展+review达成=advisor；Director 明确不更新进展（避免双写冲突）。"细节和进展"是自由文本，advisor写得含糊会导致误判→badcase源。

6.2 交互冲突：单通路治理

核心判据：Director的首token出了没？未出→打断+合并重推理（话术用"顺便""对了"串联）；已播报→P0用户直接打断不恢复，P1非用户排队等播完（可预推理缓存）。

策略(优先级低→高)行为例子
①扔弃新任务新的丢正回话，advisor下发→丢弃
②排队等旧的播完正回话，主动服务→排队
③静默执行新的悄悄做正找景点，开车窗→静默开
④整合(核心)暂停旧的重规划规划行程，"再加个点"
⑤插队新的插播完旧的继续播新闻，"小心行人"→插完→继续
⑥停止旧任务旧的直接丢正开窗，"下个路口左转"→停窗做导航

三特殊规则：长程任务不参与合并；aigc不走Planner走通知卡；同一advisor只保留最新建议。

恢复机制：打断→存checkpoint；新query处理完→advisor读goal list的"细节进展"决定是否恢复（不靠Director记忆，靠看板）。

6.3 执行层：26工具+HMI

一agent一任务，串行依赖不一次多调。成功→二次推理（多数静默）；失败→Replan（回传原始任务+失败原因→Director重新规划）。

HMI三类卡：小卡(AI语音卡≤1我方)/大卡(AI画布≤1上汽容器,与小卡互斥)/通知卡(push可多)。播报归属：车控音乐短视频GUI→Director统一回；本地生活vqa车书s2s→agent自播；websearch→Director总结播；aigc→通知卡。
