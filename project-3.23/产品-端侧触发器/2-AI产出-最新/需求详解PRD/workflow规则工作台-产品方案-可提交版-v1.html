一背景与目标

1.1 现状问题

目前车上的业务规则（"什么情况下车做什么"）以两种方式存在：一是硬编码在各模块代码里，二是散落在配置平台的表单式场景里。带来三个问题：

  • 迭代慢：加一条规则或改一处逻辑，要走研发开发—测试—发版，周期以周计，产品无法自助。
  • 表达弱：现配置平台是表单式，只能表达"多条件 AND 后执行"，无法表达"先判断 A、满足走一条、不满足走另一条"这类带流程和分支的复杂逻辑（7.15 韩杰明确指出"中间一定有复杂条件逻辑，不是触发了就直接执行"）。
  • 重复建设：触发器、事件日志、Director 条件/定时任务各自开发一套条件判断逻辑，成本高、口径不一。

1.2 目标

把全系统的业务规则，从"埋在代码里、每条要研发写"，升级为"一个可视化画布，产品/运营自己拖积木即可配置、发布、运营"的规则引擎。一次搭好，触发器、Director、主动服务、各 Advisor 共用，不再各建一套。

二产品定位与边界

2.1 定位

本工作台是端云统一的规则编排引擎——系统的"规则中枢"。它消费信号中枢提供的信号原料，产出可执行的规则（剧本），下发端/云执行。对应 7.9 白板会给触发器的定位："规则引擎，走 workflow 画布"。

2.2 边界（关键，避免重复建设）

归本工作台（编排层）归信号中枢（原料层）归研发/下游（执行层）
可视化画布、原子能力库、事件+条件编排、K2 时间窗、频控、优先级仲裁、callback 出口路由、运营（版本/灰度/统计）信号接入、原子状态监控、事件生成（穿越检测）、缓存端云同步下发、端侧离线执行、动作真正落地（TTS/车控/Advisor）
原则（引 7.9 白板会共识）："原料归信号中枢，编排归工作台，不开发两个条件引擎。"信号原料只建一处（信号中枢），规则编排只建一处（本工作台），两边通过标准接口对接。

三核心概念

一条规则 = 触发 + 条件 + 动作，三段式：

段含义类比（闹钟）
触发规则的起点。某信号事件发生时唤醒规则到 7 点
条件唤醒后的过滤。查此刻状态是否满足，可多条件 且/或，可分支如果是工作日
动作条件通过后执行。播报/车控/触发 Advisor/记日志就响铃

执行模型为漏斗：事件（起点）→ 条件（过滤）→ 动作（出口）。对应 7.9 韩杰的"事件 + 条件"漏斗，替代旧的"多条件并行 AND"。

四画布功能规格

4.1 节点类型

节点作用参数
Start规则入口无
Trigger 触发规则起点，绑定触发信号显示名、信号源、运算符（变更为/有变更）、目标值（可多选）、去抖动(ms)、仅点火触发(开关)、记录事件日志(开关)
Condition 条件状态过滤判断信号、位置（至少1个/全部，多选 key）、运算符（等于/不等于/大于/大于等于/小于/小于等于/包含/不包含）、目标值、多条件 且/或
Action 动作执行动作动作类型 + 类型对应参数（TTS：音色+文本；车控：指令；Advisor：场景）
Branch 分支条件的 Yes/No 出边Yes→下一节点；No→结束 或 另一分支
End规则结束无
运算符约束（避坑）：事件类仅"变更为、有变更"两个；条件类仅上述 8 个。画布上档位可显示"切入/切出"作友好包装，底层映射为"变更为/有变更"。与研发对接一律用底层运算符，不用"切入/切出/穿越"等口头说法（7.10 曾因此产生分歧）。数值信号的"越阈值"由条件运算符"大于/小于"+ 触发引擎抓边沿实现，非独立运算符。

4.2 连线与分支

节点自上而下连线；Condition 节点必有 Yes/No 两条出边；支持多条件串联、多级分支；支持一条规则挂多个动作出口（callback 路由到不同下游）。

4.3 自然语言预览

画布自动将规则翻译为一句自然语言（如"当档位切入 D 档，且至少 1 个车窗打开，TTS 播报…"），用于评审、汇报、审核。

4.4 草稿 / 试运行 / 发布 / 版本

  • 草稿：编辑中，不生效。
  • 试运行：在 mock/联调环境验证规则逻辑。
  • 发布上线：规则生效，按发布范围下发。
  • 版本：每次发布产生版本号（v0.x），可回滚；展示触发次数、命中率等统计。

七端云同步机制

7.1 离线要求驱动端云归属

每条规则由产品标注一个离线要求（断网是否必须执行），作为研发判定端云归属的输入：

规则特征端云归属
断网必须执行 + 信号均为端侧本地（档位/车窗/车速/舱内 VLM）下发端侧执行
依赖云端数据/API（POI/天气/节日/舱外 VLM）云端执行
信号端侧有但逻辑复杂/需问用户研发权衡
分工声明：产品提供"离线要求 + 实时性要求"，研发据此结合端侧数据可用性与性能，判定端云归属并回填。产品不预先指定端云实现。（依据 7.15 韩杰"产品定义场景，研发定端云"。）

7.2 下发与执行

云端配好规则（剧本）→ 按归属下发端侧或留云端 → 端侧规则支持断网离线执行 → 执行结果按需回写信号库/上报 Context。

八运营能力

  • 发布范围（灰度）：渠道 / 指定车辆 VIN（多选）/ 车型 / 软件版本，支持灰度发布。
  • 执行策略（频控）：执行总次数 / 单次上电周期内次数 / 自然日内次数 / 最小间隔（默认 30s）。
  • 统计：触发次数、命中率、版本对比。
  • 优先级仲裁：多规则冲突时按优先级仲裁。
