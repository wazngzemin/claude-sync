1.2 原方案 vs 现实

原方案：高德SDK围栏（已不可用）

  • 给SDK一个坐标 + 半径 → 创建围栏
  • SDK主动回调：进入/离开围栏
  • 开发量小，SDK内部处理频率/精度
  • 现状：auto SDK围栏服务已下架

替代方案：端侧自建围栏判定（本方案）

  • 利用高德SDK"POI距离查询"接口获取实时距离
  • 端侧自建围栏状态机：距离 vs 阈值判定
  • 状态变化时上报云端触发器引擎
  • 高德SDK距离查询能力已确认可用

3. 推荐方案详解：端侧自建围栏

3.1 核心原理

一句话：端侧定时问高德"我离这个POI多远" → 拿到距离 → 自己判断进出 → 状态变化时告诉云端。

组件谁干什么频率
距离数据源 高德SDK 提供当前自车位到指定POI的距离（直线 or 粗略路线） 被动接口，端侧按需调用
围栏管理器 端侧·新建模块 管理围栏生命周期：创建/删除/查询 按事件
距离轮询器 端侧·新建模块 定时调高德SDK查距离，喂给状态机 ~1Hz（每秒一次）
围栏状态机 端侧·新建模块 distance vs threshold 判定，产出进入/离开事件 随轮询，~1Hz
事件上报 端侧 → 云端 围栏事件（进入/离开）上报云端触发器引擎 仅状态变化时

3.2 距离获取方式（两种，按场景选）

方式适用场景数据来源精度限制
① 高德SDK POI距离查询 任何场景（导航中 + 非导航） 调高德SDK接口：传入POI ID/经纬度 → 返回当前距离 + 预计到达时间 中 — 直线距离或粗略路线距离 需端侧定时调用（非push）；调用频率需和高德确认上限
② 导航引擎剩余距离 仅导航中 + POI为目的地/途经点 导航引擎每秒native输出的沿路线剩余距离 高 — 沿实际路线计算 仅导航状态可用；POI必须在路线上
优先级策略：如果正在导航且POI是导航目的地/途经点 → 用②导航剩余距离（更精准）；否则 → 用①高德SDK距离查询（通用兜底）。

参数值说明

出栏阈值（leave）520m距离 > 520m 判定"已离开" — 死区上沿
死区480m ~ 520m在此区间内不做状态切换，防GPS抖动
到达判定d ≤ 50m + 档位=P双条件防误判（高架经过不算到达）
关于"离开"判定（会上刘杨/汪帅的疑虑）：
刘杨说"距离都是正数，怎么判离开"，汪帅说"离开很难"。实际上用距离+死区可以判：用户到达后继续行驶 → 距离从小变大 → 当 d > 520m → 判定"已离开"。与接近判定对称。关键在于死区（480~520m）防止在边界反复跳。

但对于 one_shot 场景（如"到外婆家提醒我打电话"），只需"已接近"一个事件，触发一次即完，不需要判离开。

7. 与原 PRD 方案的关系

维度原PRD方案（导航剩余距离）本方案（端侧自建围栏）关系
距离数据来源 导航引擎native输出的沿路线距离 高德SDK POI距离查询 / GPS haversine 不同来源，互补
状态机位置 云端（信号上报后云端判定） 端侧（本地判定，只上报事件） 核心差异：判定从云端下沉到端侧
上报内容 每秒上报distance原始值到云端 只在状态变化时上报围栏事件 本方案上报量显著减少
A类case（导航目的地） 直接可用 — 导航引擎有距离 也可用 — 距离来源更通用 A类可用任一方案，导航距离更精准
B类case（非导航POI） 需先插途经点，退化为A类 直接可用 — 不依赖导航 本方案天然支持B类
非导航状态 不可用 可用 — GPS直线距离兜底 本方案覆盖面更广
后续trigger链路 相同：事件 → 引擎匹配 → Advisor → Director → TTS 后半段完全复用
结论：本方案（端侧自建围栏）替代了原方案中"距离信号每秒上报云端 + 云端状态机判定"这一段。后半段（触发器引擎匹配 → Advisor → Director → TTS）完全复用，不受影响。对于A类case（导航中+目的地），可以优先用导航剩余距离（更精准），围栏方案作为通用兜底。
