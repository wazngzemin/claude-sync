# 端侧触发器 - PRD

## 一、需求背景与目标

### 1.0 为什么现在要做？

当前端侧 VLM 任务触发采用短期方案：在 AI BOX SDK 中硬编码订阅状态，满足条件直接调用识别任务。存在三个瓶颈：

1. **不可扩展**：每增加任务或适配新车型，都需改代码发版；
2. **无统一信号源**：每个任务各自读 CAN 信号，VLM 结果不能复用，云端 Adviser 拿不到标准化端侧状态；
3. **无调度能力**：VLM 任务已增至 15+，缺乏优先级仲裁、打断恢复和超时释放。

现需升级至长期方案，建设端侧信号库 + 端侧触发器。

### 1.0.1 本期目标

- 已有 15 个 VLM 任务全部转为规则驱动（脱离硬编码）
- 支撑任务优先级仲裁、抢占恢复、超时降级
- 支持 Adviser/Planner 动态注册条件任务
- VLM 结果统一写入信号库，供云端 context 订阅

### 1.0.2 模块职责边界

| 模块 | 职责 | 不负责 |
|---|---|---|
| 端侧信号库 | 统一存储所有状态（端状态/OEM/场景/VLM结果/任务状态）；对外提供写入接口和订阅接口；按策略上报云端 context | 不负责信号的原始采集 |
| 端侧触发器 | 加载规则 → 读取信号 → 两级条件匹配 → 任务调度 → 下发运行指令；接收动态任务注册 | 不负责 VLM 内部推理；不负责 Adviser 思考逻辑 |

### 1.1 端侧触发器模块是什么？

端侧触发器是 AI 汽车端侧感知任务的调度中枢，负责把"端侧信号库里的实时状态"进行匹配，在条件满足时触发对应的 VLM任务，并管理任务的优先级、执行状态、超时、打断与恢复。

是端侧任务执行前的"规则判断 + 任务调度 + 生命周期管理"模块。

> 信号库负责存"车知道了什么"，触发器负责判断"现在该跑哪个任务"，VLM/小模型负责执行"具体看什么"，结果再回到信号库并上报云端 context。

目前要同时建设两个基础能力：

1.  **端侧信号库**：统一存储端状态、OEM/用户信号、导航/场景信号、VLM 结果和任务状态；
2.  **端侧触发器**：基于规则和信号判断任务是否触发，并负责任务排队、优先级、打断、超时和结果回写。

### 1.2 目前问题点？

#### 端侧任务触发是写死逻辑，无法进行扩展

当前状态是车上的端侧识别逻辑是固定写死：在 AI BOX SDK 中订阅几了个状态，满足条件后直接调用识别任务，识别完把结果给云端。但是当不同车型规则不一致时，无法统一管理。

#### 端侧没有统一信号库，任务判断缺少稳定输入

-   **触发器没有稳定输入**：每个任务自己去读 CAN 信号，读的方式都不一样
-   **VLM 结果无法复用**：下车安全跑完的结果，停车位识别任务想用，拿不到
-   **云端 Advisor 订阅不到标准化端侧状态**：Advisor 想知道"刚才 VLM 检测到什么"，没有统一接口
-   **任务失败没有记录**：VLM 超时了、报错了，没有任何地方记录，后面排查问题找不到日志

#### VLM 任务数量已经明显增加，需要管理任务

没有统一调度会出现：

| 问题 | 场景 |
| --- | --- |
| Always-on 占用算力，用户 VQA 响应变慢 | P3 阻塞 P0 |
| 安全任务被低优任务挡住 | 下车安全 P0 被 Always-on P3 占位 |
| 低优任务覆盖高优任务结果 | Always-on 旧 caption 覆盖 VQA 结果 |
| 任务超时后队列不释放 | VLM 推理卡住，后续任务全部阻塞 |

## 三、端侧信号库

### 3.1 业务流程

![](images/WnfCcuTJB4QhCsAhevA8U1aqWHZzcHpN.png)

**流程说明**（对应信号库流程图）：

信号产生 → 判断信号类型（任务状态 / 端状态信号 / OEM用户信号 / 导航场景信号 / 模型结果）→ 标准化为统一字段（key/value/source/timestamp）→ 写入端侧信号库 → 更新信号版本/有效期 → 分两路输出：**通知触发器或等待扫描** ｜ **按策略上报云端 Context**

### 3.2 信号写入规范

所有信号统一为以下 4 个标准化字段写入信号库（对应流程图）：

| 字段 | 说明 | 是否必填 |
|---|---|---|
| key | 信号唯一标识，如 `gear_state`、`vlm_result_3_1` | 是 |
| value | 信号值，支持字符串/数值/结构化 | 是 |
| source | 写入方标识，如 `car_service`、`vlm_agent` | 是 |
| timestamp | 写入时间戳 | 是 |

写入后信号库自动更新信号版本和有效期。

### 3.4 VLM 结果写入规范

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| task\_id | 任务编号，如 3-1 | 是 |
| task\_name | 任务名称，如下车安全 | 是 |
| task\_type | single/periodic/long\_running/always\_on | 是 |
| priority | P0/P1/P2/P3/P4 | 是 |
| result\_type | label/description/mixed/empty/failed | 是 |
| label\_result | 结构化标签结果 | 按任务 |
| desc\_result | 自然语言描述结果 | 按任务 |
| status | success/empty/timeout/failed/interrupted | 是 |
| timestamp | 结果时间 | 是 |

### 3.5 VLM 两条回路说明

对应架构图，VLM 执行后有两条独立的数据回路：

| 回路 | 方向 | 内容 | 用途 |
|---|---|---|---|
| 任务状态反馈 | VLM → 触发器 | queued / running / timeout / interrupted | 触发器据此做抢占、超时释放、恢复决策 |
| 识别结果写入 | VLM → 信号库 | 标签 / 描述 / empty / failed | 信号库存储，供下游消费和云端上报 |

### VLM 结果要怎么写

| 状态 | 含义 | 能不能当有效结果用 |
| --- | --- | --- |
| success | 识别成功，有效标签/描述 | 可以 |
| empty | 模型正常执行，但没有发现目标 | 可以，表示无目标 |
| failed | 模型执行失败 | 不可以 |
| timeout | 模型超时未返回 | 不可以 |
| interrupted | 被高优任务打断 | 不可以，等待恢复或终止 |

| 结果 | 写入 | 注意事项 |
| --- | --- | --- |
| 成功识别 | 写标签和/或描述 |     |
| 未发现目标 | 写 empty | 表示模型正常执行但无目标 |
| 超时 | 写 timeout | **不允许写成"无异常"** |
| 失败 | 写 failed | 不触发下游主动服务 |
| 被打断 | 写 interrupted | 等待恢复或终止 |
| 低优覆盖限制 | — | P3 Always-on 不得覆盖 P0/P1 刚写入的关键结果 |

## 四、端侧触发器

端侧触发器就是端侧任务的"调度员"。

它做这几件事：

`看规则 看信号 判断任务是否命中 多个任务同时命中时排优先级 告诉 VLM/小模型去看什么` `处理任务成功、失败、超时和打断`

### 4.1 业务流程

![](images/Ym5PlJUAi5N1h7isN32kEyGCXelua4en.png)

**流程说明**（对应触发器流程图）：

加载任务规则 → 读取/监听信号库 → **判断运行时段** → ◇运行时段满足？→ 否：回到监听 → 是：**判断触发条件** → ◇触发条件满足？→ 否：回到监听 → 是：生成候选任务 → 检查任务状态（是否已运行/冷却/完成）→ ◇允许执行？→ 生成运行指令 → 调用 VLM/小模型 → 结果回写信号库

> 关键设计：条件匹配为**两级判断**——先匹配运行时段（离车后/上车前/停车时/行车中），再匹配具体触发条件。运行时段不满足时不检查触发条件，避免无效扫描。

### 4.2 功能列表

| 一级功能 | 二级功能 | 详细描述 |
| --- | --- | --- |
| 规则管理 | 规则加载 | 启动全量拉取 + 运行中增量订阅 + 本地缓存 |
| 规则管理 | 热更新 | 双 buffer 切换，切换前 dryrun 校验 |
| 规则管理 | 灰度 | 按 VIN / 车型 / 比例灰度，规则带 gray\_tag |
| 规则管理 | 回滚 | 异常 ≤ 5min 回滚到上一版本 |
| 规则管理 | 动态注册 | 接收 Planner/Adviser 运行时注册的条件任务（见第五节） |
| 条件扫描 | 两级判断 | 第一级：运行时段匹配；第二级：触发条件匹配 |
| 条件扫描 | 信号订阅 | 按规则反推信号 key，向信号库订阅 |
| 条件扫描 | 去抖与合流 | 同 key 在 debounce\_ms 内只保留最后一次 |
| 条件扫描 | 条件 DSL | 比较 / 与或非 / 持续时间 / 时间窗 / since |
| 条件扫描 | 缺失策略 | 关键信号 ≥ 1s 缺失 → 跳过本拍 |
| 任务调度 | 优先级队列 | P0 > P1 > P2 > P3 > P4，同优 FIFO |
| 任务调度 | 互斥仲裁 | 同 mutex\_group 任一时刻仅 1 个 Running |
| 任务调度 | 冷却去重 | 同任务默认 60s 冷却 |
| 任务调度 | 抢占恢复 | 高优完成后被挂起任务自动恢复 |
| 指令下发 | 运行指令 | 向感知模块下发 task\_id / task\_goal / priority / timeout（见 4.10） |
| 指令下发 | 超时处理 | 按规则降级（小模型兜底 / 静默丢弃） |
| 指令下发 | 上报 | 脱敏后写入云端 context |
| 结果处理 | 结果回写 | 识别结果写入信号库（步骤 16） |
| 结果处理 | Adviser/Planner 通知 | 结果上报后通知云端 Adviser/Planner 消费（步骤 18） |
| 结果处理 | 任务续接/恢复 | 周期任务续接下一轮；被打断任务恢复；单次任务标记结束（步骤 19） |

### 4.3 优先级与打断恢复

| 优先级 | 任务类型 | 抢占规则 | 示例 |
| --- | --- | --- | --- |
| P0 | 用户即时/安全强相关/离车即时提醒 | 立即抢占任意低优 | 下车安全、遗留物品、VQA |
| P1 | 安全/场景增强类 | 抢占 P2-P4，不抢 P0 | 停车寻车、地下停车场识别 |
| P2 | 持续服务类 | 仅 FIFO + 排队 | 增强哨兵、舱外迎宾、儿童照看 |
| P3 | Always-on 描述类 | 仅 FIFO，可被丢弃 | 舱内/舱外 Always-on |
| P4 | 动态 Adviser/Planner 任务 | 弱触发，可静默 | Adviser 条件任务、Planner 定时任务 |

**关于 4-5（安全提醒）与 4-6（儿童照看）并发说明**：

两任务同属 VLM_CABIN 组，同一时刻仅能运行一个。本期采用**合并任务目标**方案：

- **仅当 4-5 和 4-6 同时处于触发状态时合并**：触发器向 VLM 下发合并指令 `task_goal = "全舱安全观察 + 儿童状态观察"`，VLM 一次运行返回两组结果，触发器分别写入两个任务的状态。
- **仅触发其中一个时**：按单任务正常下发，不合并。
- **合并运行中其中一个结束时**（如用户关闭儿童照看）：触发器下发更新后的 task\_goal，退回单任务模式。

### 4.8 抢占恢复规则

**业务原则**：高优先级任务到达时，低优先级任务应被暂停；高优先级任务完成后，低优先级任务应在合理条件下自动恢复。

**恢复条件**：
- 高优任务已完成
- 被抢占任务的触发条件仍满足
- 用户阶段未发生变化（如停车时任务，恢复时用户已挂 D 档准备离开，则不恢复）
- 被抢占任务未超过自身超时时间
- 无其他更高优先级任务在运行

**恢复失败处理**：
- 若触发条件不再满足或用户阶段已切换，任务终止，不写信号库
- 若超过自身超时时间，标记为 Timeout
- 若仍有更高优先级任务在运行，继续等待恢复

### 4.9 冷却策略

- 冷却粒度：**按任务**
- P0 任务默认**不冷却**（安全任务每次满足条件都必须触发）
- P1-P4 默认冷却 60s，可在规则中单独配置
- 队列需设置容量上限，超出后低优先级任务丢弃，P0 任务排队数需有上限保护

### 4.10 运行指令协议

对应架构图，触发器向端侧感知模块下发的运行指令包含以下字段：

| 字段 | 说明 | 是否必填 |

| task\_goal | 识别目标描述（静态任务为预设值，动态任务由注册方提供） | 是 |

## 五、动态任务（Adviser/Planner 注册）

### 5.1 概述

对应架构图云侧"动态任务来源（Adviser/Planner/用户Query）→ 动态任务/即时任务 → 端侧触发器"链路。

Planner/Adviser 可在运行时向触发器注册条件任务，触发器将其作为普通规则纳入两级条件扫描，命中后执行对应动作。

### 5.2 注册接口

| 字段 | 说明 | 是否必填 |
|---|---|---|
| task\_name | 任务名称 | 是 |
| registered\_by | 注册方标识（如 `planner`、`comfort_adviser`） | 是 |
| run\_phase | 运行时段 | 是 |
| run\_phase\_conditions | 运行时段条件，格式同静态规则 | 是 |
| trigger\_conditions | 触发条件 DSL，格式同静态规则 | 是 |
| action\_type | `run_vlm` / `notify_adviser` / `notify_planner` | 是 |
| action\_target | 目标标识 | action\_type 非 run\_vlm 时必填 |
| task\_goal | VLM 识别目标描述 | action\_type = run\_vlm 时必填 |
| priority | 默认 P4 | 否 |
| mutex\_group | 需要占用的 VLM 资源组 | action\_type = run\_vlm 时必填 |
| expire\_at | 过期策略：`current_trip_end` / 具体时间戳 / `never` | 是 |

**注册结果回调**：

| 结果 | 说明 |
|---|---|
| registered | 注册成功，返回 rule\_id |
| rejected | 注册失败（条件格式错误 / 信号 key 不存在 / 队列已满） |

### 5.3 动态任务状态流转

```
registered → active → triggered → executing → done / expired / cancelled
                ↑                      ↓
                └──── cooldown ←───────┘（周期任务回到 active）
```

### 5.4 核心规则

- 接收并保存动态任务（本地持久化，弱网可执行）
- 按注册条件纳入正常两级条件扫描
- 执行前检查上下文有效性（如用户是否仍在车内）
- 支持注册方主动撤回（传入 rule\_id）
- 车机重启后自动恢复未过期的动态任务
- 动态任务总数上限：建议 16 条

### 5.5 两种 action_type 的区别

| action_type | 触发后行为 | 是否占用 VLM | 对应架构图链路 |
|---|---|---|---|
| run\_vlm | 触发器下发运行指令给感知模块，走正常优先级仲裁 | 是 | 触发器 → 运行指令 → 端侧 VLM |
| notify\_adviser | 触发器通知云端 Adviser 条件命中，Adviser 自行决定后续 | 否 | 触发 → adviser 主动服务 |
| notify\_planner | 触发器通知 Planner 条件任务已触发 | 否 | 触发 → Planner |

### 5.6 示例场景

**Planner 条件任务**：用户说"十分钟后帮我关掉座椅加热"，Planner 解析后注册条件任务（trigger\_conditions: time == 10min later, action\_type: notify\_planner），10 分钟后触发器通知 Planner 执行。

**Adviser 条件任务**：舒适 Adviser 注册"副驾有人 + VLM 识别到连续打哈欠 ≥ 30s → notify\_adviser"，条件命中后触发器通知舒适 Adviser，Adviser 自行决定是否建议调整环境。

**Adviser 需要 VLM**：Planner 注册"看到 XXX 上车就告诉我"（action\_type: run\_vlm, task\_goal: "持续关注是否有特定人上车", mutex\_group: VLM\_FRONT），触发器将其作为 P4 VLM 任务调度。

## 六、Fail 定义与验收标准

### 6.1 Fail 定义

| Fail 类型 | 定义 | 典型案例 | 严重程度 |
|---|---|---|---|
| 漏触发 | 条件满足但任务未触发 | D→P + 解锁了，下车安全没跑 | P0-致命 |
| 误触发 | 条件未满足但任务触发 | 还在 D 档开着车，下车安全触发了 | P0-致命 |
| 重复触发 | 同一事件触发多次 | 一次锁车，遗留物品检测跑了 3 次 | P1-严重 |
| 优先级错误 | 高优被低优挡住 | 用户问了 VQA，Always-on 没让路 | P0-致命 |
| 信号库污染 | 超时/失败写成伪结果 | 下车安全超时，信号库写了"无障碍" | P0-致命 |
| 抢占不恢复 | 高优完成低优不恢复 | VQA 跑完了，Always-on 没恢复 | P1-严重 |
| 队列阻塞 | 超时不释放 | VLM 卡住，后续全排队 | P0-致命 |
| 规则未生效 | 热更新后规则没起效 | PM 改规则 5 分钟后端侧还在跑旧规则 | P1-严重 |
| 动态任务丢失 | 重启后动态任务未恢复 | 注册"10分钟后关加热"，重启后丢了 | P1-严重 |
| 动态注册无反馈 | 注册失败但注册方不知道 | Adviser 注册条件任务，格式错误但没返回失败 | P1-严重 |
| 运行时段误判 | 时段判断错误导致触发/漏触发 | 行车中误判为停车时 | P0-致命 |

### 6.2 性能验收指标

| 指标 | 目标值 | 测量方式 |
|---|---|---|
| 条件命中 → 下发 VLM | ≤ 1s | 信号库写入满足条件 → 触发器下发指令 |
| P0 任务整体返回 | ≤ 5s | 触发条件满足 → 结果写入信号库 |
| P1 任务整体返回 | ≤ 8s | 同上 |
| P2 任务整体返回 | ≤ 10s | 同上 |
| Always-on 执行周期 | 10s ± 2s | 两次 Running 之间间隔 |
| 超时释放 | ≤ 1s | timeout 发生 → 队列释放 |
| 规则热更新生效 | ≤ 5s | 云端推送 → 端侧 dryrun 通过 → 切换 |
| 崩溃重启恢复 | ≤ 3s | 进程重启 → 加载规则 → 恢复队列 |
| 动态任务注册响应 | ≤ 500ms | 注册请求 → 返回结果 |

### 6.3 线上验收用例

| 用例 | 输入 | 预期 |
|---|---|---|
| 下车安全 + Always-on | D→P + 解锁 + Always-on 在跑 | Always-on interrupted，3-1 执行，完成后恢复 |
| VQA 抢占 | Always-on 在跑 + 用户 VQA | Always-on interrupted，VQA 执行，完成后恢复 |
| 超时 | 触发 1-5 + VLM 超时 | 写 timeout，不写伪结果，队列释放 |
| 风暴防护 | 雾气信号 60s 内抖动 30 次 | 只触发 1 次（冷却期内去重） |
| 动态任务弱网 | 注册"10 分钟后关加热" + 断网 + 重启 | 10 分钟到点仍执行 |
| Adviser 注册触发 | 舒适Adviser注册"副驾打哈欠≥30s" + 条件命中 | 通知 Adviser，不占用 VLM |
| Adviser VLM 任务 | Planner注册 run\_vlm 任务 + P0 任务抢占 | 动态任务被挂起，P0 完成后恢复 |
| 动态任务过期 | 注册 expire\_at=current\_trip\_end + 行程结束 | 任务自动清理，不再触发 |
| 动态任务撤回 | Planner 注册后主动撤回 | 状态变为 cancelled，从扫描中移除 |
| 4-5+4-6 合并 | 安全提醒在跑 + 儿童照看触发 | 合并 task\_goal 下发，两组结果分别写入 |
| 4-6 单独结束 | 合并运行中 + 用户关闭儿童照看 | task\_goal 退回单任务模式，4-5 继续 |
| 运行时段切换 | 行车中任务在跑 + 用户停车（D→P） | 行车中任务终止，停车时任务规则开始扫描 |

## 七、Context 上报策略

### 7.1 上报内容

| 字段 | 是否上报 | 说明 |
|---|---|---|
| task_id, task_name, status | 是 | 基础信息 |
| label_result / desc_result | 视任务 | P0/P1 全量上报；P3 Always-on 只上报变化 |
| timestamp | 是 | 端侧生成时间 |
| 车辆状态快照 | 是 | 上报时刻车速/档位/车门（用于云端追溯） |

### 7.2 上报时机

对应主流程图步骤 17-18：

- 任务状态变为 Success/Empty/Failed/Timeout 时立即上报
- Always-on 周期任务可合并上报（减少流量）
- 动态任务状态变更（triggered/done/expired/cancelled）上报
- 支持网络恢复后批量补报
- 上报后通知云端 Adviser/Planner 消费（步骤 18）

### 7.3 脱敏规则

- 人脸/人体图像不上传原始图，只传标签和描述
- 车牌号不上传或脱敏处理
- VQA Query 文本可上报，关联语音不上传

## 八、下电后行为与离车后任务

### 8.1 下电后触发器与信号库状态

- 车辆下电后，触发器和信号库进入**低功耗模式**，仅保留离车后任务（1-1/1-2/1-3/1-4/1-5）的运行能力。
- 电源由车辆常电提供，需与整车厂确认功耗预算。
- 下电后信号库不再接收 CAN 信号（CAN 休眠），仅保留已缓存的信号和 VLM 结果。
- 下电后动态任务中 expire\_at = current\_trip\_end 的自动清理。

### 8.2 离车后任务最长运行时间

| 任务 | 建议最长运行时间 | 超时处理 |
|---|---|---|
| 1-1 增强哨兵 | 8h 或电量 < 20% 时停止 | 记录日志，通知用户 |
| 1-4 豆包抢车位 | 30min 或用户取消 | 标记 Cancelled |
| 1-5 遗留物品 | 下电后 5min 内完成 | 超时标记 Timeout |
| 1-2 / 1-3 停车寻车 | 下电后 2min 内完成 | 超时标记 Timeout |

## 九、本期范围与长期方向

### 9.1 本期不做

1. 不建完整云端规则管理后台 → 用简单配置方式，后台后续补
2. 不定义 VLM 内部算法 → VLM 是黑盒，触发器只管调接口
3. 不做用户自定义场景 → 用户不能说"帮我盯一下这个"
4. 不做复杂自然语言规则解析 → 规则用结构化 JSON，Planner 侧负责解析后再注册
5. 不定义车控安全策略 → 触发器只管"触发感知任务"

### 9.2 长期方向

- 云端按 VIN/车型/版本灰度下发规则
- Adviser/Planner 大规模动态注册
- 端侧算力预算管理（多模型并发调度）
- 自学习触发阈值（按用户行为微调）
- 触发可解释 UI（向用户展示"为什么触发"）
