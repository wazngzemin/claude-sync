§1 项目背景：为什么需要触发器

场景故事：用户开着荣威 D6X 带宝宝出门。出发前说了句"帮我照看后排宝宝，哭了告诉我"。然后车开了 20 分钟，期间宝宝一直安静。突然，后排宝宝开始哭——这时候系统应该立刻主动告诉驾驶员"宝宝好像在哭呢"。

问题来了：谁在这 20 分钟里一直盯着宝宝？Director 不行——它只在用户说话那一秒活着，说完就走。Advisor 不行——它是出主意的，不是盯梢的。VLM 不行——它只在被调用那一刻看一帧图，没人叫它就不看。

答案就是触发器。触发器 = 7×24 不下班的保安 + 到点叫醒你的闹钟。它自己不动脑子，只做一件事：把"一直在变的信号"变成"少数几个值得叫醒模型的事件"。

§2 触发器是什么 · 一句话定位

触发器 = 门口的保安 + 床头的闹钟。
便宜的、不用模型的判断（电量低没低、到 P 档没、超 30 度没）它 7×24 盯着做；
贵的、要模型想的事，只在条件命中那一秒才叫醒。
用"运算符"挡住 99% 的噪音，让模型只处理 1% 值得处理的时刻。

§3 云端 vs 端侧：各是什么、各干什么、为什么分两半

3.1 为什么必须分两半

6.22 西钺警告过"不要开发两个触发器"。逻辑上它就是一个触发器，一个条件只判一次。但有 三件事只能在端侧做、三件事只能在云端做——物理约束谁也绕不过：

🚗 只能在端侧做的三件事☁️ 只能在云端做的三件事
① 掐表："10 分钟后关加热"，本地倒计时才准。云端掐表有网络延迟，车可能在地库没网

② 断网也得干：进隧道前说"10 分钟后开窗"，到点车在隧道里，云端够不着

③ 离摄像头近：舱内画面合规上不了云，VLM 眼睛只能长在车上 		① 解冲突："10 分钟后关窗"撞"5 分钟后开窗"——要有全局视角排优先级。端上各管各的，看不到别人在干嘛

② 懂人话："毯子掉没掉"要理解画面描述的语义，得过模型。端上认死规矩，理解不了这种话

③ 统一登记：谁要盯什么先来登记、校验能力、查重、拼任务。散在端上会乱、没法审计

§4 核心引擎：闸门链状态机

为什么不能"信号一变就触发"？假设直接配"battery_level ≤ 20%"就触发，看会怎样：

8:20 电量首次降到 20% → 触发 ✅
8:21 电量仍然 20% → 又触发 ❌ 重复弹窗
8:25 电量 19% → 又触发 ❌
8:35 充了点又降回 20% → 又触发 ❌ 二次骚扰

所以不能只判"≤20%"，要有完整闸门链：① 条件判定 → ② 穿越（只认F→T边沿） → ③ 防抖 → ④ 冷却。四道闸门缺一不可。

11.1 异常处理总表

异常处理设计原则来源
网络波动·端到云积压周期类压包丢弃只传最新 / one_shot暂存续发一次周期=时效优先 / one_shot=可靠优先7.4 通过
VLM 推理超时写 timeout 进 Context（绝不写"正常"）→ 不重试，下轮自然覆盖宁可不触发不误触发—
VLM 模型崩溃重试1次 → 仍失败 → 告警 + 标 degraded有限重试不死循环—
信号源离线（TTL过期）信号作废 → 依赖场景 frozen防过期数据误触发7.4 通过
云端触发器不可用端侧定时器照跑（断网容错）端侧独立性—
L3 模型不可用场景 frozen + 告警（不降级默认True）宁可不触发不误触发—
时序脏数据（人换座位）切换时先上报空数据清前值防旧 face_id 误判7.4 超然
用户拒绝推荐记 user_rejected → 本行程 frozen不打扰用户配置表
3min 失效（弹窗类）触发器启 3min 计时器 → 到期发 event_expired避免弹窗长期挂起—
HMI 删任务走模拟query/notification 回传 Director不能只在 trigger 删7.3 晓伟

11.2 安全红线（永不执行）

三条不变式：① 宁可不触发不误触发 ② 异常必须可见不静默吞 ③ 端侧断网也要活（已下发承诺端侧独立兑现）

永不执行的动作：行驶中开门窗（speed>0 拦截）/ 行驶中调座椅 / 智驾中接管 / 儿童在座儿童锁需二次确认
