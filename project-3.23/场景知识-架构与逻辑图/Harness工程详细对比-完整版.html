组件 1：Agentic Loop（执行循环）

① 解决什么问题

裸 LLM 只能回答一次——你问一句它答一句就结束了。但真实任务需要多步：先查天气 → 根据天气决定要不要开窗 → 再查车内温度 → 最终决定空调设几度。Agentic Loop 让模型可以循环执行：思考 → 行动 → 观察结果 → 再思考 → 再行动，直到任务完成。

② 业界标准怎么实现

TAO 循环（Thought-Action-Observation）

也叫 ReAct Loop，是 2026 年所有主流 Agent 框架（LangChain/LangGraph/OpenAI Agents SDK/Anthropic Claude Code）的标准模式：

业界标准 Agentic Loop 伪代码：

while not done:
    # 1. 组装 Context（每轮都重新组装）
    context = assemble_context(
        system_prompt,
        retrieved_knowledge,   # 动态知识
        retrieved_examples,    # 动态示例
        tool_definitions,      # 可用工具
        memory,                # 记忆
        conversation_history,  # 对话历史
        current_observation    # 上一轮工具返回的结果
    )

    # 2. 调 LLM 推理
    response = llm.generate(context)

    # 3. 解析输出
    parsed = parse_output(response)  # JSON 解析

    # 4. 输出验证（Guardrails）
    if not validate_output(parsed):
        retry_count += 1
        if retry_count > max_retries:
                return error("验证失败，超过重试次数")
        current_observation = f"你的输出不合法：{validation_error}"
        continue  # 回到循环开头，让模型重新生成

    # 5. 如果模型决定调工具
    if parsed.has_tool_calls:
        for tool_call in parsed.tool_calls:
            # 5a. 工具执行前校验（Guardrail Layer 1）
            if not validate_tool_call(tool_call, current_state):
                current_observation = f"工具调用被拒绝：{reason}"
                continue

            # 5b. 执行工具
            result = execute_tool(tool_call)

            # 5c. 工具执行后校验（Guardrail Layer 2）
            validated_result = validate_tool_output(result)

            # 5d. 结果回传作为下一轮的 observation
            current_observation = validated_result

    # 6. 如果模型决定结束
    if parsed.is_final_answer:
        # 最终输出校验（Guardrail Layer 3）
        if validate_final_output(parsed.answer):
            done = True
            return parsed.answer

关键特征：
  - 循环次数不固定，模型自己决定什么时候结束
  - 每轮都可以调多个工具
  - 工具返回结果自动成为下一轮的输入
  - 三层 Guardrail 嵌在循环内部
  - 输出验证失败会触发重试（retry loop）
  - 有超时/预算控制（token budget / max iterations）

来源：Oracle: What Is the AI Agent Loop · OpenAI Agents SDK: Guardrails

LangGraph 的实现：有向图 + 状态机

LangGraph 把 Agentic Loop 建模为一个有向图：每个节点是一个计算步骤（LLM 推理 / 工具执行 / 条件判断），边是状态转移。状态是一个共享对象，在图中流动，存储消息、变量、中间结果、决策历史。支持 checkpointing（断点续传）、线程级记忆、跨会话持久化。

来源：SpaceO: Agentic AI Frameworks 2026

③ 我们实际怎么实现的

豆包 Planner 的实际执行循环：

# 1. Selector SP 检索动态知识（独立 LLM 调用，500ms-1.8s）
knowledge_ids = selector_sp.classify(query, env, history, goals)

# 2. vikingDB 向量检索动态示例（~30ms）
examples = vikingdb.search(query_embedding, top_k=5)

# 3. 组装 SP
sp = assemble_sp(
    static_sp,              # 固定部分 ~16K tokens
    knowledge[knowledge_ids], # Skill 全文
    examples,               # few-shot 示例
    env_info,               # 端状态
    memory,                 # 用户记忆
    conversation_history    # 最近 10 轮
)

# 4. Planner LLM 推理（~800ms）
response = planner_llm.generate(sp + user_prompt)

# 5. 解析输出
parsed = parse_json(response)  # talk_or_not / talk_content / action_list

# 6. 执行工具
for action in parsed.action_list:
    result = execute_tool(action.tool_name, action.param)
    # tool_feedback 回传

# 7. 如果有 tool_feedback 需要二次推理
if needs_second_pass(result):
    # 把 tool_feedback 加入对话历史
    # 重新调 Planner 推理一次
    response2 = planner_llm.generate(sp + tool_feedback)
    # 通常到这里就结束了，最多 2 轮

缺失的部分（对比业界标准）：
  ✗ 没有 validate_output（输出直接执行，不校验）
  ✗ 没有 validate_tool_call（工具调用前不校验）
  ✗ 没有 validate_tool_output（工具返回后不校验）
  ✗ 没有 retry loop（输出不合法时不重试）
  ✗ 循环深度有限（通常 1-2 轮，不能做 3-4 步编排）
  ✗ 没有 token budget 控制
  ✗ 没有超时退出机制

④ 差距精确定位

业界 Loop 里有的步骤我们有没有影响
Context 每轮重新组装✅ 有（Selector + vikingDB + env + memory 每轮都跑）—
LLM 推理✅ 有—
JSON 解析✅ 有—
输出结构校验（JSON schema + tool_name 合法性 + param 格式）❌ 没有tool_name 写错也直接下发→方向靠报错→badcase
验证失败 → 重试（retry loop，让模型重新生成）❌ 没有一次输出错了就没机会纠正
工具执行前校验（非P档+调座椅→拒绝）❌ 没有安全隐患！小明说"可能车毁人亡"
工具执行后校验（检查返回值是否合理）❌ 没有工具返回异常也原样回传给 Planner
多轮深度循环（3-4轮 → 模型自己决定结束）⚠️ 部分（最多2轮）复杂编排做不了（先查温度→判断开窗→查天气→定幅度）
Token budget / 超时退出❌ 没有极端情况可能死循环或超时无响应

⑤ 怎么补

最小化改动方案（在现有架构上加 3 层校验）：

# 在 Planner 输出之后、工具执行之前，加一个校验函数：

def validate_planner_output(parsed, env_info):
    # 校验1：tool_name 是否在 23 个合法工具里
    valid_tools = ["vehicle_basic_control", "search_poi_qa", ...]
    for action in parsed.action_list:
        if action.tool_name not in valid_tools:
            return False, f"工具名 {action.tool_name} 不存在"

    # 校验2：安全红线（Guardrail 硬校验）
    if env_info.gear != "P":
        for action in parsed.action_list:
            if "座椅" in action.param or "靠背" in action.param:
                return False, "非P档禁止调座椅（安全红线）"

    # 校验3：param 格式基本合法性
    # ...

    return True, "通过"

# 校验不通过时，让 Planner 重新生成（retry）：
for retry in range(max_retries=2):
    response = planner_llm.generate(context)
    parsed = parse_json(response)
    ok, reason = validate_planner_output(parsed, env_info)
    if ok:
        break
    else:
        # 把校验失败原因加入 context，让模型修正
        context += f"你的上一次输出不合法：{reason}，请重新生成"

这是纯代码逻辑，不需要 LLM，几 ms 就能执行。
加了这一层，就补上了业界标准 Loop 里最关键的 3 个缺失步骤。

工作量：中等（几天）。找谁做：明骏 / 超然。优先级：P0。

⑥ 评分

★★★☆☆ 3/5 有基础 Loop 但缺少校验层和重试机制。Context Assembly 做得好但 Loop 内部的防护缺失。

组件 2：Context Assembly（上下文组装）

① 解决什么问题

模型的上下文窗口是有限的（Seed 1.8 约 128K tokens 但实际 SP ~35K）。不能把所有信息都塞进去——需要一个系统在每一步决定"这一轮该给模型看什么"。Anthropic 的原话："找到最小的高信号 token 集合，最大化期望输出的准确率。"

② 业界标准怎么实现

Anthropic 的 Context Engineering 指南

把上下文窗口当作有限的注意力预算来管理。每多塞一条信息，就占用一部分预算，可能稀释其他信息的注意力权重。目标不是塞得多，而是塞得精。

MindStudio 的"新鲜上下文"模式

每一轮循环的 Context Assembly 流程：

1. 任务到达 → Orchestrator 接收
2. Context Builder 查询 Memory
   → 从记忆库提取与当前任务相关的条目
   → 不是全量加载，是按相关性筛选
3. 组装新鲜 Context
   → task + memory + tools + env
   → 每轮组装一次，不复用上一轮的
4. Orchestrator 分解任务 → 分发给 Specialist Agents
5. 每个 Specialist 用自己的新鲜 Context 执行
6. 结果聚合 → 输出
7. 反馈写入 Memory
   → 有价值的结果存入长期记忆
   → 供后续任务的 Context Assembly 使用

关键设计原则：
  - 每轮 Context 都是"新鲜组装"的，不是累积的
  - Memory 是分层的：短期（本轮）/ 长期（跨轮）/ 工作（当前任务）
  - 工具定义按需加载（不全量塞）
  - 有裁剪机制：Context 超限时自动压缩

来源：MindStudio: Agentic OS Architecture · Atlan: Context Layer Harness Engineering

Atlan 的"治理 Context 层"

最先进的做法是在 Context 组装之前做权限和治理检查——即使 prompt 没有排除 PII 列，元数据层也不会把 PII 暴露给模型。"PII 执行是结构性的，不依赖 prompt"——安全约束在 Context Assembly 层面硬编码，而不是在 prompt 里写"请不要使用 PII"。

③ 我们实际怎么实现的

豆包 Planner 的 Context Assembly 架构：

━━━ 静态部分（固定在 SP 里，不变）━━━
  ① 角色定义        ~500 tokens
  ② 工具定义 x23     ~4000 tokens  ← 全量加载，不按需筛选
  ③ 输出格式+注意事项 ~800 tokens
  ④ 静态示例 10条     ~4000 tokens  ← 优化后从64条减到10条
  ⑤ 车辆知识+人设     ~2300 tokens

━━━ 动态部分（每轮检索+组装）━━━
  ⑦ 动态知识 {{knowledge}}
     检索方式：Selector SP（LLM 推理），500ms-1.8s
     内容：1-3 条 Skill 全文，~2000 字/条

  ⑧ 动态示例 {{examples}}
     检索方式：vikingDB（向量检索），~30ms
     内容：5+5 条 few-shot，~800 字/条
     问题：多轮时 query 改写没做好

  ⑨ env_info {{env_info}}
     端状态：P档/温度/SOC/车窗/座椅/速度 等

  ⑩ memory {{memory}}
     用户记忆：偏好温度/FaceID/历史
     问题：FaceID 未就绪→记忆写不进去

  ⑪ 对话历史
     最近 10 轮 user/assistant 交替
     问题：10 轮后信息丢失，无会话摘要压缩

我们做得好的地方：
  ✓ 双路检索（LLM + 向量）是比较先进的
  ✓ 示例拆分静态+动态，SP 从 32K 压到 ~16K
  ✓ 首 token 延迟从 2309ms → 998ms（降 57%）
  ✓ 准确率从 79.38% → 88.38%（升 9%）

④ 差距精确定位

业界标准做法我们有没有影响
动态知识检索（按场景注入相关知识）✅ 有（Selector SP）—
动态示例检索（向量检索 few-shot）✅ 有（vikingDB）—
端状态 / 记忆注入✅ 有—
示例拆分静态+动态✅ 有（10静态+5+5动态）—
工具定义按需加载（只加载当前场景需要的工具）❌ 全量加载 23 个4K tokens 的工具定义稀释注意力
Context 超限自动裁剪/压缩❌ 没有长对话时可能超限或注意力严重分散
对话历史摘要压缩（老对话压缩成摘要）❌ 没有10 轮后信息丢失
多轮 query 改写（"好的那就这家"→ 改写为完整表达再检索）❌ 没有多轮对话时示例检索不准
Context 层安全治理（PII 在组装层拦截）❌ 没有安全约束只在 prompt 层
Selector 延迟 ≤500ms⚠️ 当前 1.8s性能瓶颈

⑤ 怎么补

  1. 工具动态加载：Selector 输出场景 ID 后，只加载该场景需要的工具定义（如降温只需要 vehicle_basic_control，不需要 image_generate）。省 ~2K tokens。→ 明骏做
  2. 对话摘要压缩：超过 10 轮的对话，用小模型把前 8 轮压缩成 ~200 字摘要，只保留近 2 轮原文。→ 晓伟 + 超然本周定方案
  3. 多轮 query 改写：SP 优化专项里已经提了但没做。方案：在 vikingDB 检索前，用规则或小模型把短 query（"好的那就这家"）改写为完整表达（"去鮨一日本料理用餐"）。→ 研发排期
  4. Selector 延迟优化：压缩索引表（每条从 ~100 字→~30 字锚词），减少 Selector 的输入 token。同场景连续对话跳过 Selector 用缓存。→ 我压缩索引表 + 明骏做缓存逻辑

⑥ 评分

★★★★☆ 4/5 这是我们最强的组件。双路检索+示例拆分是先进的。但工具全量加载、无摘要压缩、query 改写未做、Selector 延迟高是扣分项。

组件 3：Tools & Execution（工具与执行）

② 业界标准

业界标准：工具用 JSON Schema 定义参数（name / description / parameters），框架自动序列化/反序列化。支持 MCP（Model Context Protocol）标准接口让工具跨 Agent 复用。工具数量可动态加载——只暴露当前任务需要的工具。Oracle 总结："跨所有厂商，工具集成遵循相同的结构。"

来源：Oracle: AI Agent Loop

③ 我们的实现

23 个工具全量定义在 SP 里，占 ~4K tokens。每个工具有 tool_name / tool_describe / param / notice 四个字段。通过 SOA 车控指令下发到方向靠。工具列表固定，不按场景动态加载。

④ 差距

业界我们差距
工具按需加载23 个全量加载中。多了不需要的工具 → 分散注意力 + 浪费 token
标准 JSON Schema 参数自定义 param 格式中。param 格式不统一，模型容易写错
MCP 标准接口自定义 SOA 接口低。车载场景不需要跨 Agent 复用

⑥ 评分

★★★☆☆ 3/5 工具本身够用但加载方式粗放。

组件 4：Memory（记忆系统）

② 业界标准

多层记忆架构（2026 共识）

业界标准的 4 层记忆（来自 Red Hat + IBM + Mem0 2026）：

Layer 1: 工作记忆 (Working Memory)
  = 上下文窗口里的内容
  = 当前对话 + 当前任务状态
  类比：人的"此刻在想的事情"
  实现：Context Window 里的对话历史

Layer 2: 短期记忆 (Short-term / Episodic Memory)
  = 单次会话内的信息
  = 包括对话摘要、中间结果、工具返回的数据
  类比：人"今天经历的事情"
  实现：会话级缓存 + 对话摘要压缩

Layer 3: 长期记忆 (Long-term / Semantic Memory)
  = 跨会话持久化的知识
  = 用户偏好、历史行为、身份信息
  类比：人"长期积累的知识和经验"
  实现：向量数据库 + 用户画像存储

Layer 4: 程序记忆 (Procedural Memory)
  = "怎么做事情"的经验
  = 从历史任务中学到的技能/模式
  类比：人"骑自行车不需要重新学"
  实现：技能库 / 经验模板

Mem0 的多维度记忆标记

2026年4月 Mem0 发布的新算法：每条记忆写入时标记身份维度（user_id, agent_id, run_id, app_id），检索时按维度组合自动合并和排序。实现了"同一个用户在不同 Agent 之间共享记忆"。

来源：Red Hat: Architecting Memory for AI Agents · Mem0: State of AI Agent Memory 2026 · IBM: What Is AI Agent Memory

③ 我们的实现

豆包 Planner 的记忆系统：

Layer 1: 工作记忆 → ✅ 有
  对话历史（最近 10 轮 user/assistant 交替）
  问题：10 轮后信息丢失，无摘要压缩
  6/3 晚例会超然确认："这就是我们的一个缺陷，一个空洞"

Layer 2: 短期记忆 → ⚠️ 部分有
  Goal List（goal_list_update 管理目标队列）
  工具返回结果（tool_feedback 回传）
  问题：没有对话摘要压缩机制

Layer 3: 长期记忆 → ⚠️ 功能有但不可用
  search_user_memory / operate_user_memory 工具
  FaceID 身份识别
  问题：FaceID 未就绪 → 记忆写不进去
  问题：Planner 说"记住了"但实际没保存
  6/3 小明走查："他说信誓旦旦的说给我记了，但我也没办法确认"

Layer 4: 程序记忆 → ❌ 没有
  没有从历史任务中学习的机制
  场景编排结果不持久化（每次重新编排，可能结果不同）

④ 差距精确定位

记忆层业界标准我们差距严重度
工作记忆对话历史 + 超限时自动压缩对话历史 10 轮，超了就丢严重——10 轮后用户说的事情全忘了
短期记忆会话级缓存 + 对话摘要有 Goal List，无摘要中

程序记忆技能库 / 从历史学习没有中（可以后期补）

★★☆☆☆ 2/5 架构有但关键功能不可用（FaceID + 10 轮丢失）。

组件 5：Guardrails（安全护栏）★☆☆☆☆ 1/5

② 业界标准

三阶段校验 + Tripwire 机制

业界标准的三阶段 Guardrail（来自 OpenAI Agents SDK + Vectara）：

阶段 1：工具执行前（Input Tool Guardrail）
  触发时机：模型输出了一个 tool_call，即将执行之前
  校验内容：
    - 参数合法性（必填参数是否齐全？值是否在合法范围内？）
    - 安全条件（当前状态下是否允许这个操作？）
    - 权限检查（当前用户是否有权限做这个操作？）
  校验结果：
    allow  → 放行，正常执行
    reject → 拒绝，返回一条消息给模型说明原因
    tripwire → 紧急停止，整个 Agent 执行中止

  举例：
  模型要调 "vehicle_basic_control" 参数 "座椅放倒"
  → 校验函数检查 env_info.gear
  → gear == "D"（行驶中）
  → reject: "非P档禁止调座椅（安全红线）"
  → 这条消息回传给模型，模型知道被拒绝了

阶段 2：工具执行后（Output Tool Guardrail）
  触发时机：工具执行完毕，返回结果之后
  校验内容：
    - 返回值是否合理？（如温度查询返回了 -999°C → 异常）
    - 返回值是否包含敏感信息？（如 PII → 脱敏）
  校验结果：
    pass → 原样回传给模型
    modify → 修改后再回传（脱敏/修正）
    block → 不回传，给模型一条错误消息

阶段 3：最终输出前（Final Output Guardrail）
  触发时机：模型生成了最终回答，即将返回给用户之前
  校验内容：
    - 是否包含有害/不当内容？
    - 是否泄露了系统内部信息？
    - 格式是否正确？
  校验结果：
    pass → 返回给用户
    rewrite → 修改后返回
    block → 不返回，给用户一条安全消息

Vectara 的自动重试机制

如果工具调用被 Guardrail 拒绝，系统自动触发重试：发出 GuardrailCheckEvent（包含具体的失败原因），模型收到后修正调用，最多重试 max_retries 次。"大幅减少了工具幻觉和格式错误，不需要开发者写复杂的验证代码。"

ToolHalla 的纵深防御架构（5 层）

纵深防御（Defense-in-Depth）：
  Layer 1: 输入筛查（prompt injection / PII / 毒性检测）
  Layer 2: 对话控制（话题限制）
  Layer 3: LLM 生成控制（结构化输出 / JSON mode）
  Layer 4: 输出验证（Pydantic schema + Guardrails AI）
  Layer 5: 业务规则后校验

"Pydantic schema 验证（Layer 4）是最低标准——免费、快速、
 捕获最常见的失败模式。"

③ 我们的实现

豆包 Planner 的安全护栏：

阶段 1（工具执行前）：❌ 没有代码级校验
  只有 SP 注意事项里的文字约束："非P档禁止调座椅"
  这是 LLM 推理层面的概率性约束，不是代码硬拦截
  → 99.9% 会遵守，0.1% 可能不遵守
  → 小明原话："哪怕 99.9% 都规避了，有那万分之一，可能就是车毁人亡"

阶段 2（工具执行后）：❌ 没有
  工具返回什么就原样回传给 Planner
  不检查返回值是否合理

阶段 3（最终输出前）：❌ 没有
  Planner 的 talk_content 直接 TTS 播报
  不检查是否包含不当内容

唯一的"护栏"：
  层1: SP 注意事项（文字约束，概率性）→ 已做
  层2: Adviser 前置判断（部分场景）→ 部分做了
  层3: 车控底层代码兜底 → 待定方案！周五沟通！

④ 差距

这是我们和业界标准差距最大的组件。业界的 Guardrail 是代码级硬校验（三阶段 validate + tripwire + 自动重试），我们几乎完全依赖 SP 里的文字约束（LLM 推理层面，概率性的）。在车载安全场景下，这个差距是致命的。

⑤ 怎么补

见组件 1 的"怎么补"部分——在 Planner 输出后加一层 validate_planner_output 函数，做 tool_name 合法性 + 安全红线 + param 格式校验。这是最小化改动，几天就能做。
