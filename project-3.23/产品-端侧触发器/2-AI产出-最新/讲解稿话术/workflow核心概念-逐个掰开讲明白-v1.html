workflow 核心概念 · 逐个掰开讲明白

每个词都：大白话是什么 + 举个例子 + 为什么要它 · v1
这份东西的规矩：不放一个"术语一句话带过"。每个概念都像口头讲那样——先用大白话说是什么，再举个车上的例子，再说为什么要它。你从头看一遍，这摊事里所有的词你就都能自己讲给别人听了，上桌不怕被问。
A · 一条规则的三段
一条规则 = 触发 + 条件 + 动作
大白话

规则就是"车在什么情况下做什么"。永远拆成三段：什么时候开始看（触发）、还得满足啥（条件）、那就做点啥（动作）。

例子
你手机闹钟：到 7 点（触发）→ 如果是工作日（条件）→ 就响铃（动作）。车上一模一样。
为什么

这三段是所有规则的骨架，配任何规则都是往这三个格子里填东西。记死它，后面全好懂。

触发 Trigger
大白话

规则的起点。像个哨兵，平时不动，盯着某个信号，信号一变它就"醒"，规则才开始跑。

例子
"档位切换"这个哨兵盯着挡把，你挂进 D 档这一下，它就醒了。不挂 D 档，它一直睡，不费一点电。
为什么

有了起点，规则平时就睡着、不占资源，只在该看的时候才醒。这是"漏斗"省电的根子（见 B）。

条件 Condition
大白话

哨兵醒了之后，先别急着动手，要检查的几道判断题。查的是"此刻是什么状态"。

例子
挂了 D 档（触发醒了），先查一道题："车窗有没有开着？"开着才往下，全关就到此为止。
为什么

光有触发不够——挂 D 档但车窗本来就关着，就不用提醒。条件是用来"筛掉不该做的情况"。

动作 Action
大白话

判断题都过了，那就真动手做事：播报、控制车、触发 Advisor、记一条日志。

例子
挂 D 档 + 车窗开着，都满足了 → 动作：语音播报"关窗省电"。
为什么

动作是规则的出口，是用户真正感受到的那一下。一条规则可以挂多个动作（既播报又记日志）。

B · 这三段怎么执行（最重要）
漏斗模型 vs 旧的"多条件并行 AND"
大白话

指这三段是"从宽到窄、层层过滤"地跑：一个事件当起点（漏斗口），后面几个条件当过滤（漏斗身），层层筛，最后真执行的很少（漏斗出口）。

例子

想实现"挂 P + 充电口盖关 + 电量低 → 提醒充电"。

旧做法（多条件并行 AND）：三个条件平等并排，"档位=P 并且 充电口盖=关 并且 电量<30%"，三个同时成立才提醒。毛病：三个一直同时盯着，没起点，你在高速上没挂 P 它也在那儿空算。

新做法（漏斗）：挑"挂 P"当起点，其余当过滤——

你开车的所有时刻
　　↓ 事件：挂 P 了吗？　←漏斗口，只有挂P才进来
　挂 P 的时刻
　　↓ 过滤：充电口盖关？
　关着的
　　↓ 过滤：电量<30%？
　电量低的
　　↓
　提醒充电　　　　　　←漏斗出口，真执行的很少
为什么（漏斗好在哪，3 点）

① 省：平时没挂 P，规则睡着不算；旧的从头到尾一直算。
② 清楚：有明确起点（挂 P），逻辑一目了然；旧的一团条件分不清主次。
③ 能拐弯：漏斗每层能分支（电量不低？走另一条路）；旧的 AND 不满足就是不执行，没有别的路。
这就是 7.9 韩杰定的方向，也是你画布比旧配置平台先进的地方——不是换界面，是执行模型升级了。

C · 跟"触发"有关的词
事件 Event（= 触发的本质）
大白话

就是"一个状态变化的时刻"。"触发"是画布上节点的名字，"事件"是它的本质，同一个东西两个叫法，别被绕。

例子
档位从别的挡"变成"D——这个"变化"就叫一个事件。韩杰 7.9 叫它"事件"，你画布上叫"触发"。
为什么

知道它俩是一回事，你看文档时"事件"和"触发"来回换就不会晕。

变更为 / 有变更 事件的两个运算符
大白话

"变更为 X" = 值变成了指定的那个 X 才算；"有变更" = 值变了就算，不管变成啥。

例子
档位"变更为 P" = 只有挂到 P 才触发；档位"有变更" = 挂到任何挡（P/R/N/D）都触发。差一个"为"字，"为"就是"成为某个值"。
为什么

你截图确认过，平台事件运算符只有这两个，没有"切入/切出/穿越"。跟研发对接只用这两个词，别用口头说法惹分歧。

穿越 / 抓边沿 数值越过阈值
大白话

数值信号（车速、温度）"越过一条线"的那一下。注意：它不是一个独立运算符，是用条件"大于/小于"配，触发引擎在"从不满足变成满足"的那一刻抓一下。

例子
车速"穿越 120" = 从 ≤120 升到 >120 的那一下抓一次。不是一直大于 120 都触发（那会疯狂触发），是"刚越过"那一下。
为什么

数值一直在变，如果"大于120就触发"会触发无数次。只抓"刚越过"那一下（边沿），才有意义、才不轰炸。

去抖动 Debounce
大白话

信号抖一下又弹回来，不算数。设个时间窗（如 300ms），窗内的重复触发合并成一次。

例子
挂挡时挡把晃了一下、瞬间划过 D 又回来，去抖 300ms 就把这种"手滑"过滤掉，不误触发提醒。
为什么

物理信号常有毛刺、抖动。不去抖，一个动作可能误触发好几次，体验很吵。

one_shot 一次性 / 防重复
大白话

一个状态持续满足时，只在"刚满足"那一下触发一次，别反复触发。

例子
导航"接近目的地（<500m）"，你在 500m 内会待好几分钟，one_shot 保证只在刚进 500m 那下提醒一次，不是每秒念一遍。
为什么

没有它，"持续满足"的条件会把用户念崩溃。它和"穿越/抓边沿"是一个道理——只在"变化那一下"动。

D · 跟"条件"有关的词
多选 key 至少 1 个 / 全部
大白话

有些信号是一组的（4 个车窗、5 个座位）。配条件时一次勾选这一组里的几个，再配"任一满足"或"全部满足"。

例子
"车窗至少 1 个打开" = 勾选 4 个车窗 + "任一为开"。要是没有多选，你得写"左前开 或 右前开 或 左后开 或 右后开"四条，很啰嗦。
为什么

车门、座位、车窗这些天生成组，多选让"任一门开""后排有人"这种一条搞定，不用堆条件。

条件运算符 等于/大于/包含… 8 个
大白话

判断"此刻的值"用的：等于、不等于、大于、大于等于、小于、小于等于、包含、不包含。就 8 个。

例子
"车内温度大于 26"、"驾驶模式不等于泥泞"、"当前道路包含'高速'"。
为什么

它跟事件运算符（变更为/有变更）分开：事件看"变没变"，条件看"此刻是啥"。别搞混。

K2 时间窗 事件后 N 分钟内状态补齐算命中
大白话

事件发生后，给条件留一个时间窗——窗内条件补齐了，也算这条规则命中。不要求"事件和条件严丝合缝同一秒"。

例子
"上车（事件）后 2 分钟内，如果系好安全带（条件补齐）"就播报。不是要求上车那一秒就系好，给 2 分钟窗口。
为什么

现实里事件和条件常有时间差（人上车总要几秒才系带）。有时间窗，规则才符合真实节奏，不会因为差几秒就漏掉。（这是你 6.30 亲子案例里就发现的，是你的现成方案。）

E · 跟"执行/运营"有关的词
callback 出口 满足后往哪派
大白话

规则满足后，把结果"派"到哪个下游去执行。有几个出口：TTS 播报、触发 Advisor、通知 Director、直接车控。

例子
同一个"进停车场"事件，可以同时派两个出口：一个通知 VLM 关舱外、一个记事件日志给 Director。
为什么

一个事件常要通知多个下游。callback 出口就是这个"分发路由"，是编排层（你）的活。

频控 执行策略
大白话

限制一条规则多久能触发一次、一共触发几次。防止同一个提醒反复骚扰用户。

例子
"关窗提醒"设：最小间隔 30 秒、单次上电周期内最多 3 次、一天最多 5 次。不然你一路开车它一路念。
为什么

没有频控，高频事件会把用户逼疯。频控是"动态规则"的刚需，旧的静态开关根本没有"频率"一说。

优先级仲裁 多规则冲突谁先
大白话

同一时刻好几条规则都想动手，或者想抢同一个资源（都想播报），按优先级决定谁先、谁被压住。

例子
"超速提醒"和"关窗提醒"同一秒都想播报，仲裁决定：安全类的超速提醒优先，关窗提醒往后排或这次跳过。
为什么

规则一多必然打架。得有一个统一裁判，不能各播各的、用户听一团乱。这个裁判在编排层（你）。

F · 跟"架构/分工"有关的词
原料 vs 编排 信号中枢 vs 工作台的边界
大白话

"原料" = 信号从哪来、怎么监控、怎么缓存（信号中枢·韩杰）；"编排" = 用这些信号搭规则、频控、仲裁、派发（工作台·你）。菜市场供料，厨房炒菜。

例子
"档位现在是 P"这个信号谁提供、怎么缓存 = 原料（韩杰）；"挂 P 就查充电口盖再提醒"这条规则怎么配 = 编排（你）。
为什么

7.9 白板会定的"原料归你、编排归我，不开发两个条件引擎"——就是这条边界。它是你 owner 的地盘划分，上桌必须守。

端侧 vs 云端
大白话

端侧 = 车里那个盒子（断网也能干活）；云端 = 远处服务器（算力大，但断网够不着）。

例子
档位、车窗这些信号在车上（端侧）；地图、天气、日历在云端。钻隧道断网时，端侧照跑，云端歇菜。
为什么

一条规则放端还是放云，就看它"断网要不要能用"。这是你跟研发分工的关键（详见"离线要求"）。

离线要求 断网要不要能用
大白话

一条规则断网时是否必须还能执行。产品标这一个，研发就能推出它该落端侧还是云端。

例子
"关窗提醒"断网也得提醒 → 落端侧；"推荐充电桩"要查地图、断网本来也没用 → 云端。
为什么

端侧存在的唯一硬理由就是"断网也要跑"。所以离线要求 = 端云归属的判断依据。你只答这个（业务问题），端云实现研发定（技术问题）。

原子能力库 三箱积木
大白话

把触发、条件、动作抽成一个个可复用的"积木块"，放三个箱子里，配规则时从箱子里拖。

例子
"车窗"这块条件积木，D档关窗提醒用、驻车通风提醒也用，不用每条规则重做一遍。
为什么

规则一多，如果每条各配各的，重复劳动。积木化后加规则几乎零成本——这是韩杰"不用基于不同业务开发多套"的意思。

事件日志 → Context 规则触发记一笔给 Director
大白话

规则触发时，记一条结构化日志（几点几分、发生了啥），上报到云端 Context，给 Director/大模型看。

例子
"车速超 120、当前国道"触发时，记一条事件日志进 Context，Director 后面推理时就知道"这人刚才在国道超速了"。
为什么

规则不光是当场做个动作，它产生的"事件"还是喂给大模型的记忆。图里"记录事件日志"那个开关就是干这个。
