② 7.9 信号中枢三表评审
2026-07-09 · 韩杰主持，嘉锋/明骏等 · 泽民缺席 · 端状态升级"信号中枢"，拆三张表（整理自你的《7.6→7.9 边界详解》，非原始逐字稿）
议题1 · 信号中枢是什么
韩杰："端状态承担的是一个信号中枢——把车端多元信号（车企总线/导航媒体智驾 APP/AI Box 的 VLM）转化成业务需要的信息。Context 要信息文本；触发器/Advisor 要信息的变更——我们叫它事件。同时做数据缓存、实时处理（如从档位+充电状态推出'停车充电/长时停车/临时停车'）。"
议题2 · 三张表 泽民的条件库被拆进这里
表装什么给谁用
状态查询表原端状态高频精选，key-value，只描述"现在是什么值"。信号 ID 改研发代码风格；新增舱内外视觉识别Context → Director；条件计算的数据源
事件表状态的变化。必须"穿越"才算（"从低于 120 到 120 之上才算"）。三个来源：状态变化 / 纯埋点 / 组合事件触发器（起点）、事件日志 → Context → DT
条件表判断逻辑的可视化配置。把工程代码（大于/小于/等于）变成可视化操作。1:1 复用给触发器 + 事件日志 + Director触发器、事件过滤漏斗
议题3 · 执行逻辑改成"漏斗"

旧：五个条件并行 AND。新：事件（起点/主触发）→ 条件（串并联过滤）→ 执行。例如"档位变更为 P"当起点，再查"所有座位有人？车门关？"当过滤，都过才执行。

翻译成泽民的运算符：韩杰的"事件"=你的"变更为/无变有/有变无"；韩杰的"条件"=你的"等于/大于/包含"。漏斗=一个变更沿+若干状态判断。这不是新发明，是你运算符体系的重排。
议题4 · 没吵完的（白板会真正议题）
未决点嘉锋韩杰
事件和条件是不是两层"都是条件，事件是条件的一个选项，我就不用触发器再开发一遍条件""事件是起点，条件是过滤，不是一回事"
监控模块归谁—"可能是触发器，或者我们要监控的模块"（含糊，谁不吭声就归对方）
组合事件怎么拼能组合就行"还要再研究"（没人设计，你带方案去就是定义者）
议题5 · 韩杰给触发器的新定位 王座
韩杰（原话）："触发器就代表规则引擎，将来做运营，大概率通过 workflow 画布。"
议题6 · 白板会立场（边界）
原料归信号中枢（接入/原子监控/缓存），编排归触发器（事件条件编排/K2 窗口/频控/仲裁/callback 四出口/workflow 画布）。韩杰原话："不要开发两个条件引擎。"

⚠ 泽民缺席的代价：条件库被当"种子"拆（"把泽铭触发器里赛力斯需要的先加进来"），监控模块归属没人替你争。

③ 7.15 workflow 触发器
2026-07-15 14:11-14:46 · 35 分钟 · 王泽民 / 韩杰 / 丁彬 三人 · 主题：端侧触发器 + VLM 感知能力技术讨论（实为韩杰推 workflow 大方向）
14:11议题1 · VLM 能力边界

韩杰问：VLM/AI Service 能不能直接判断"是否雨雪天气"给 yes/no？丁彬：LLM 没这任务；VLM 有，但只针对舱内（问宝宝睡没睡），舱外还没有。

丁彬："这是个动态插件、query 类任务，返回结果千奇百怪（睡着了/睡一会又醒了），不是枚举值，是描述。" 韩杰：输入 caption、输出 caption，无限集。
结论：端侧拿字符串没法解析。要结构化文本才能解，但要工程开发。现阶段只有能枚举的值能做条件/触发，caption 类做不了。
14:15议题2 · 用 workflow 把端侧串起来
韩杰："通过剧本/流程图/workflow 把端侧串起来，一定有起点，起点是时间/上电。" 丁彬："你这个起点就是触发器，任务内置，上电就开始监测 if 条件，不停向数据中心轮询。"

韩杰的规则骨架：上电（起点）→ 加判断条件（当前天气）→ 去信号中心请求 + 判断逻辑 → 满足才动作。

14:16议题3 · 中间的条件逻辑很复杂

韩杰：雨雪切驾驶模式，不是识别到雨雪就立马切。中间要判断：当前驾驶模式要不要改（已是泥泞模式就终止）。泽民 14:17：还要判断预设模式是否开着，开着就不动作。

韩杰 14:17："端云都有复杂逻辑。比如识别在高速要开运动模式，但当前电量很低，开运动模式能耗更高，就矛盾了。所以一定不是触发了就执行。"
14:21议题4 · 端云"子集"之争 泽民追得不错

泽民质疑："为什么云端成了端侧的子集？不应该是端侧是云端的子集吗？"

韩杰："云端来源于端侧，不可能比端侧多。车端状态信息一定是端侧 AI Service 的子集。但整个云端 Context 会比 AI Service 多，因为有些信息不来源于车端信号（比如后台日志）。"
澄清：车端状态信息 ⊂ AI Service ⊂（车端部分）；但 Context 整体 ⊃ AI Service（含非车端信息）。
14:26议题5 · 动作映射 / 二次整形
丁彬："stream 描述行为不可控、没训过。要把这个 string 再过一次模型，问它有没有某个动作（比如笔译），有就输出标签，这已经在云端二次整形了。"

泽民 14:27：二次整形谁做？丁彬：云端做，就是昨晚 0818 那个需求；依赖你给我一个动作映射表，我再找西钺问舱内能不能都输出。

14:28议题6 · 原子能力剥离 + 平台化 这场会的核心
韩杰 14:29："我想做一个原子能力的剥离，将来配置的就是一个个脚本。雨雪切模式=触发+条件判断。我不告诉研发这条件端侧还是云端有，我提需求，研发去看端侧有没有、云端有没有、怎么获取最好最快最准。"
韩杰 14:33："director 的条件任务、定时任务都用这一套架构，所有条件、动作、触发都一样，我不用基于不同业务开发多套。"
三件事（14:33）：① 一个平台（就是你现在用的那个，升级一下）② 端云同步逻辑（平台配好剧本，下发端侧执行）③ 有负责方把信号监控、状态获取实现掉。之后各业务自己创建脚本、订阅结果。主动服务、Advisor 都能直接用。
14:30议题7 · 断网场景（端侧存在的理由）
丁彬："断网的时候，这个功能是不是要保证执行？" 韩杰："对，这有影响。如果这些场景断网都要能执行，那这一套能力都要放端侧。"

这是全场对你最有用的一句：离线要求 = 端侧存在的硬理由 = 你跟研发分端云的判断依据。

14:35议题8 · 韩杰反对"端侧触发器搞不搞" 替泽民翻案
韩杰："我比较反对研发一直说'端侧触发器搞不搞'。我的认知是我们提需求——我们有 VLM 能力，就拉着丁彬、研发聊，怎么把这个能力作为条件接进来。到底直接请求 VLM 还是通过 AI Service 透传，让研发去聊。"

这句话直接把 7.8 汪帅打死泽民的那个问法（"端侧触发器搞不搞"）作废了。

14:42议题10 · 动作类型 + 产品研发边界反思
