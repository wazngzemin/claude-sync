豆包上车 Director/Planner 架构讲义 · 上篇

系统骨架与 Director 全链路——从一句话进来到结果出去，中间经过了什么、每个模块为什么存在、怎么工作、输入输出是什么、做到了什么程度。基于 planner 架构/ 和 advisor 架构/ 下全部 21 份文档精读。

一、整套系统在干什么——一句话概括

豆包上车是搭载在上汽荣威 D6X（五座 SUV）上的车载 AI 助手。用户在车里说一句话，系统要做的就是理解他要什么→决定怎么做→把事做了→告诉他结果。如果这件事不是一次能做完的（比如"帮我照顾后排宝宝""一路介绍沿途风景"），系统还要持续盯着、到时候主动推进。

为了做到这件事，系统被设计成一个三层漏斗：

  1. 句法 RAG（第一层）——明确的单轮指令（"打开车窗"），规则匹配，毫秒级直接下发，不需要大模型。
  2. 情景 RAG + 仲裁模型（第二层）——模糊的指令（"我好热"），注入场景知识做 few-shot，由轻量模型判断该走简单快链路还是复杂慢思考。
  3. Director/Planner（第三层）——复杂任务（"帮我规划个周末自驾游"），云端大模型深度推理，多步拆解+调工具+看结果+再推理。

设计三原则（原文）：① 层层过滤——能用简单方法搞定的不要漏到后面复杂模型去处理；② 分阶段注入知识——注入方式从简单（固定规则）到复杂（动态检索）；③ 数据回流——Planner 处理的难 case 回流成训练数据，反过来增强前面的层。

为什么是漏斗而不是一个大模型包揽一切？因为车载场景对时延极敏感——"打开车窗"用户期望 150ms 响应，如果每句话都进大模型就要等 500ms+。越简单的指令越应该在前面被拦住用便宜快的方式处理，只有真正需要"思考"的才进大模型。Planner 同时充当兜底（前面都处理不了的它接）和云端 teacher（它处理完的数据可以反过来训练前面的层，让漏斗自己变聪明）。

二、入口分流：一句话进来，先过三道关

2.1 为什么需要分流

用户在车里说的话千差万别——有的是"打开车窗"这种明确到可以直接执行的指令，有的是"我好热啊"这种需要结合车况才能判断的模糊表达，还有的是"帮我规划去莫干山的两天行程中间要充电和休息"这种需要多步推理的复杂需求。根据吉利的量产数据，约 30% 的车控指令是模糊/复杂表达，单看 query 文本分不清该走哪条路。

如果不分流，所有话都进大模型，简单指令时延爆炸（500ms+ vs 应有的 150ms）；如果只用规则匹配，复杂需求处理不了。所以系统用多级拒识 + 仲裁分流来解决。

2.2 三级拒识：端侧白名单 → 云侧前置 → Director 内部

第一级：端侧白名单 + 离线 fc 模型——车上本地跑的小模型，配一张指令白名单。如果用户说的是"打开车窗""空调调到25度"这种明确高频车控，白名单直接命中，端侧就地执行，约 150ms，根本不上云。这是性能的关键——越多流量在这里拦住，系统整体越快。

第二级：云侧前置拒识——白名单没命中的 query 上到云端，先过拒识模型。如果判定这句话不该由车机处理（比如"妈妈帮我拿一下水杯"），就 reject。但 reject 不是丢弃——拒识的内容会写进 context 事件日志，被 advisor 感知。为什么？因为"妈妈帮我拿水杯"虽然车做不了，但 advisor 可能据此推断"车里有小孩，可能需要关怀"，从而触发主动服务。

第三级：Director 内部兜底——即使过了前两道，Director 在 plan_describe 阶段之前还会再做一次拒识判断，防止漏网的非车机 query 消耗推理资源。

2.3 仲裁模型：判"简单"还是"复杂"

拒识放行的 query，接下来要判断走快链路还是慢思考。这由仲裁模型（Seed1.6 flash SFT）完成——一个轻量级判别模型，只做"简单 or 复杂"二分类。

输入：召回的 query + 情景 RAG 注入的知识（以 few-shot 形式，格式 {"type":"modelExecute","knowledge":"...","agentLimit":[...]}，不是直接给答案，只是给仲裁几个参考案例帮它判断）。

输出：简单 → 云侧 LLM 快速生成指令下发（FC.SFT 300ms → 端控 150ms）；复杂 → 交给 Director 慢思考（思考 500ms + 回答 500ms）。

同一句"我好热"，空调没开时该控温（简单车控），空调已开到最冷时该回话"已经开最低了"（对话回复）。仲裁模型要结合情景 RAG 注入的车况信息才能判对。这就是为什么仲裁不是纯规则——它需要"带着情景知识做判断"。

三、Director/Planner 本体：系统的"大脑"

3.1 它是什么

Director = Planner = 对话推理引擎，三个名字一个东西。它是三层漏斗的最深一层，底层模型 Doubao-Seed-1.8（S08 从 kimi 切换），采用 ReAct + Function Call 范式。它不直接控车，只负责"理解意图 → 拆任务 → 调工具 → 读反馈 → 再决策"。

3.2 一次调用是怎么工作的（ReAct 闭环）

Director 不是"想一次就完事"的模型调用，它是一个循环——想一点、做一点、看结果、再想。内部四步：

  1. 流式对话响应——收到输入后先快速回一句话（首 token 在 500ms 内出来），让用户感到"它听到了"。
  2. 任务拆解——把需求拆成原子动作（"帮我导航去公司"拆成：先查记忆找"公司在哪"→ 再调导航）。
  3. 结构化任务序列——拆好的动作变成 action_list，发给 26 个下游工具之一执行。
  4. 总结与反思——工具返回 tool_feedback（如"导航成功"或"地址未找到"），Director 读反馈决定：成功 → 总结告诉用户（或静默）；失败 → 重试或换方案（Replan）；需要更多信息 → 继续调另一个工具。然后回到第 1 步——这就是ReAct 闭环。
"去昨天游泳馆"——第一轮推理：Director 发现需要查记忆才知道"昨天游泳馆"是哪，先调 search_user_memory。工具返回"昨天去了虹桥游泳馆"。第二轮推理：有地址了，调 route_planning_qa 导航。工具返回"导航成功"。Director 判断任务完成，不说话（静默——因为用户能直接看到导航画面）。这就是一次调用里的两轮 ReAct。

3.3 三种输入

Director 的触发输入有且仅有三种形态（以数组形式同时传入）：

形态格式处理原则
user_query{query, speaker_name, speaker_position, timestamp, assistant_emotional_state, goal_list, env_info}回应用户。注意 goal_list 和 env_info 直接随输入传——车况和目标信息不用再调工具查，这是降时延的设计。timestamp 用于自我时间感知（超30天未见→"好久不见"）。
advisor{advisor:"建议文本"}不回应 advisor，直接面向用户做行动和表达。这不是用户跟你说的话。
tool_feedback{tool_feedback, tool_name}工具返回，触发二次推理。这也不是用户说的话。

3.4 五个输出字段

{
  "talk_or_not": "yes/no",                    // 是否说话——可以选择不说
  "talk_content": "要说的话",                  // 可带 [开心地说]/[唱] 等情绪标签
  "emoji_id": "speaking",                      // 17选1的形象动画（★真身新增）
  "emotional_state_change": "关切",            // 仅情绪变化时输出（★真身新增）
  "action_list": [{action_id, params, tool_name}]  // 调工具的指令列表
}

情绪状态机是真身 SP 的重要新增。6 种状态：平静（默认）/ 欣喜 / 关切 / 低落 / 生气 / 好奇。转换规则：不能突变（低落→不能直接欣喜，需经平静过渡）；关切有惯性（用户情绪没好转就不能秒变欣喜）；久别重逢可直接进欣喜。

3.5 Director 的职责边界（非常关键）

  • P0 目标增删——据 query 推理是否需要新建/删除目标
  • P0 动态 advisor 增删——据 query 推理是否需要创建/删除动态 advisor
  • P1 合理性仲裁——增删目标/advisor 时校验：感知条件是否合理、动作是否可执行、是否矛盾
关键约束：Director 明确"不更新"目标的"细节和进展"。写权限收归 advisor，避免 Director 与 advisor 双写冲突。Director 只管"建/删目标"和"快速响应"，"盯进度/改进展"交给 advisor。这是新架构区别于旧架构的核心设计。

四、SP 结构与优化：Director 脑子里装了什么

4.1 SP 的 13 块骨架

Director 每次推理 = 读一份 System Prompt（SP）+ 当前轮的用户输入（UP）。SP 当前真身按以下顺序组织：

#区块内容
①角色定义"你是豆包，一台AI汽车的个人助手"
②输入格式3 类输入的 JSON 格式定义
③26 个工具逐个定义 tool_name / describe / param / notice
④输出格式5 字段 JSON
⑤14 条注意事项行为宪法（串行依赖不一次多调 / 品牌立场 / 时间感知 / 情绪状态机）
⑥参考示例示例 1~21（SP 体积主力，约占 87%）
⑦emoji 表情17 个形象动画定义
⑧D6X 设备知识荣威 D6X 五座 SUV 全规格 knowhow（防车控幻觉的护栏）
⑨工具 tips10 条使用技巧（含"4 导航工具不可并行"）
⑩其他知识导航/冥想/游戏/隐私/提示词保护
⑪聊天风格温柔高知女生人设 / [情绪]标签 / 静默规则
⑫基础车辆功能问答语音唤醒/小憩模式等固定 QA
⑬末尾动态 context基础世界信息 / 行程 / {{舱内视觉感知}} / {{用户记忆}} / {{动态专业知识}} = 前置筛选注入点
①~⑫ 固定在前，⑬ 动态在后——这是 KV Cache 友好的关键设计。固定前缀每次调用完全匹配，可复用已算好的注意力向量，不需要重算这几万个 token；只有 ⑬ 的动态部分（车况/记忆/知识每轮变化）需要计算新 token。把"变"的放最后，缓存命中率最大化。

4.2 SP 优化：从 32806 tokens 压到 16000 tokens

这是 Sp 优化专项 文档的核心内容——之前一直没讲过。

优化前的问题

SP 共 32806 tokens。其中参考示例占 26164 tokens（80%），其次是工具描述 2769 tokens，车辆知识 914 tokens。示例太大导致：推理慢（首包时延 P50 = 2309ms）、总耗时长（P50 = 12231ms）、KV Cache 容易被挤出。

六项优化措施

1. 输出格式压缩——字段名缩短：talk_or_not → speak（省 2 token）、talk_content → content（省 1 token）。看起来微小，但首 token 延时降了约 90ms。

2. 示例拆分为"静态 + 动态"——原来 64 个示例全塞在 SP 里。优化后拆成：静态 10 个示例（常驻 SP，教模型通用能力，如多轮 react、氛围灯调色）+ 动态 5 个示例（根据当前 query 检索出最相关的 5 个，按需加载）。这是 token 压缩最大的一项。

3. 检索加载示例——把所有示例放入向量检索引擎（VikingDB），根据 query 做语义检索取 top-5。需要解决多轮检索和 query 改写的问题（如第二轮的 query 可能只是"好啊"，需要结合历史改写成完整意图再检索）。

4. 示例压缩（抽象化）——原来示例是"裸示例"（完整的 tool_feedback 原文，如 POI 返回的详细地址/评分/营业时间全贴进去）。优化后只保留关键决策信息（如只留预估时间/总里程/红绿灯数），削掉冗余字段。

5. 基于人工反馈的优化——用 error case 持续迭代示例。例：查天气时用户未提地理位置 → 模型不知道查哪里的天气 → 加一条示例教模型"结合对话历史推断位置"。这是示例的 bugfix 循环。

6. 渐进式加载架构——静态示例 + skills（分类后的示例组，如导航类/车控类/GUI类）+ 动态示例，三级加载。

优化效果（实测数据）

指标优化前（Baseline）优化后（PE3-0415）
首包时延 P502309 ms998 ms（降 57%）
首包时延 P903439 ms1317 ms（降 62%）
总耗时 P5012231 ms6179 ms（降 49%）

静态示例 vs 动态示例的本质区别（来自「动态示例解 bug 指南」）：
· 静态示例：常态化注入 SP，授模型以"渔"——教会它处理某一类问题的通用能力（如彩虹氛围灯、座椅模式体验等场景的多轮 react 能力）
· 动态示例：按需检索注入 SP，授模型以"鱼"——解决低频且无法归类抽象的单点 bug（如让模型学会使用通讯录）。动态示例只含 query + 执行答案（调什么工具），不需要完整的多轮 input/output。

4.3 26 个工具（关键易混点）

当前真身 SP 定义了 26 个工具。完整清单：vehicle_basic_control / vehicle_system_settings / vehicle_communication_control / search_vehicle_status_info / search_weather / search_poi_qa / route_planning_qa / navi_basic_control / search_and_control_short_video / search_and_control_music / search_user_memory / face_id_register / operate_user_memory / personal_skill_record / web_search / goal_list_update / ai_broadcast_generate / search_and_control_broadcast / auto_drive / ambient_light_control / car_log / car_care_qa / recording_minutes / trip_plan_record / image_generate / search_visual_info。

三个最容易调错的点（评测重点）：

  1. 媒体播控（上一首/暂停/单曲循环）归 vehicle_system_settings，不归 search_and_control_music（后者只管搜歌/推荐）。另外有独立的 media_basic_control 工具在动态示例版本里出现（见动态示例解bug指南），负责收藏/点赞/评论/清晰度等精细播控。
  2. 记忆分两类：客观事实（家在哪/年龄/宠物）→ operate_user_memory；交互习惯偏好（"放歌时我要听创作背景"）→ personal_skill_record（支持 add/update/delete）。
  3. 4 个导航类工具不可并行：navi_basic_control / route_planning_qa / search_poi_qa / trip_plan_record 必须串行调用。

4.4 D6X 设备知识（防车控幻觉的护栏）

SP 第 ⑧ 块是荣威 D6X 的全规格 knowhow。它的作用不是"教模型怎么控车"，而是划清"能做/不能做"的边界——车控幻觉 badcase 基本都是这块没遵守。

系统红线（不支持 / 受限）
空调后排不可调温/风量（跟随主驾）；无后排独立开关；制热 = AC + ≥27°C

氛围灯全车统一，不支持分区
导航无记忆（不知道"家"在哪）；选地点用序号不用具体名

五、公共 Context 服务：所有模块共享的"现场与记忆"

5.1 为什么需要独立的 Context 服务

Director、4 个静态 advisor、动态 advisor、各下游 agent 都需要知道"车里现在什么情况""用户是谁""之前聊了什么"。如果每个模块各自去拉这些信息，就会重复拉取、重复处理。所以系统设了一个独立的公共 Context 服务——统一存储 + 预处理（压缩/打标/过滤），各模块按需取各自需要的子集。

5.2 装了 13 类内容

P0（必须有）：情景描述（感知模块 Always on）/ 历史对话（ASR-TTS）/ 事件日志 / 端状态（仅最新）/ 屏幕截图（近 5 分钟）/ 用户记忆（按人物 id 加载）/ 任务状态队列 / 基础联网信息（天气/时间/节气）。P1：豆包探索 / 通用任务经验。P2：工具知识 / 专有知识 / 性格自我认知。

5.3 Context 的优化机制（真实抓手）

不是"远区精简近区完整"（那个在这批 PRD 里没有），而是三条：

  1. 不全量加载，按需抽取——不一次性加载全部长期记忆，按场景变化提前取一部分。短期用过的记忆缓存住不重复检索。
  2. 情景描述压缩总结——把 VLM 持续输出的流式描述压成要点。
  3. 三维过滤——按场景（驻车不加载行车记忆）/ 按人物（过滤非共同记忆，防隐私泄露）/ 按任务（点外卖只检索外卖知识）。

六、前置筛选（知识筛选专家）：token 优化的核心落点

6.1 为什么需要前置筛选

Director 的 SP 里有示例、工具描述、knowhow 等大量知识内容（优化前 32806 tokens）。全塞给 Director 有两个害处：① token 多 → 注意力稀释 → 推理质量下降；② 体积大 → 首包时延高 → 用户等太久。所以需要一个独立的小模型，在 Director 推理之前先跑一遍，只把当前场景相关的知识挑出来注入。

6.2 怎么做的

输入 5 个字段（来自「动态知识检索&使用 需求说明」）：

  1. {{query}}：用户当前 query（含 speaker_name/position）
  2. {{conversation_history}}：近 10 轮对话（仅说话内容，不含工具调用）
  3. {{goal_list}}：当前进行中的目标
  4. {{env_info}}：最新车辆状态（空调/座椅/位置/导航/媒体/温度/电量）
  5. {{loaded_knowledge_ids}}：当前已加载的知识 ID 列表

知识库：6 类专家 × 四段式 ID（专家代码-一级-二级-三级）：EXP（销售/体验官，17 条）/ DRV（司机/导游，20 条）/ COM（管家，8 条）/ ENT（娱乐，7 条）/ LIFE（助理，7 条）/ EMO（伙伴，7 条），共 66 条场景知识。

输出：{"add_knowledge_ids":["COM-01-01-01"], "remove_knowledge_ids":[], "reasoning":{add_reason, remove_reason}}——增量更新，只输出变化部分。

5 条筛选原则：① 场景绑定推理 ② 关注环境变化（趋势比绝对值更重要——温度骤升 → COM-01-01-01）③ 目标持续关联（目标在则知识保留）④ 增量更新 ⑤ 信息不足不虚构。

"好热啊" + env_info 显示车内温度 5 分钟从 22°C 升到 29°C → 原则②触发 → 命中 COM（管家）→ 输出 add:COM-01-01-01（调温度知识）→ 该知识的具体内容注入 SP 末尾的 {{动态专业知识}} 块 → Director 推理时就能参考"分区温控、用户记忆偏好联动"等知识来做决策。

七、goal list（共享目标看板）

7.1 为什么需要 goal list

旧架构的病：用户说一句，Director 应一句，做完就忘——"需求被处理成一锤子买卖，导致不闭环"。goal list 就是治这个病的：把"正在为用户做的事"显式写成一张看板，Director 和 advisor 共同盯着它。

7.2 数据结构

[
  {"目标":"照顾二排宝宝", "目标状态":"进行中", "目标细节和进展":"宝宝睡着了，持续监测中"},
  {"目标":"每5分钟播报NBA", "目标状态":"进行中", "目标细节和进展":"已完成第三次播报"}
]

分 static 区（固定 4 目标：舒适/出行/情感/内容）和 dynamic 区（临时目标：功能介绍/照顾宝宝/点咖啡等）。在 SP 里以 goal_list 字段随 user_query 每轮直传。

7.3 读写权责（最关键的设计）

操作Directoradvisor
创建/删除目标✅—
更新"细节和进展"❌ 明确不更新✅（各管自己的目标）
review 目标是否达成❌✅（goal reviewer）
"细节和进展"是自由文本——不是结构化状态枚举，而是 advisor 用自然语言写的一句话。这灵活但有隐患：如果 advisor 写得含糊（"差不多了"），Director 下一轮读到时可能误判目标是否达成 → 该删不删（目标残留）或不该删却删（任务被中断）。

八、任务与交互管理：单通路下怎么治理冲突

8.1 物理约束

车里的语音交互是单通路管道——同一时间只能回应一次请求。所以多个输入同时到达时，必须在入口做"交通管制"。

8.2 打断 vs 合并 vs 排队

核心判据：Director 的推理结果（第一个 token）开始输出了没有？

  • 首 token 未出 → 打断当前推理，新旧 query 合并后重推理。话术串联用"顺便""对了"。
  • 已在播报 → P0 用户主动说话：直接打断，不恢复；P1 非用户（定时/advisor）：排队，等播完再处理。

8.3 六种冲突策略

策略行为例子
① 扔弃新任务新的到了就丢正在回话，advisor 下发 → 丢弃
② 排队等旧的播完再播正在回话，主动服务推送 → 排队
③ 静默执行新的悄悄做不打断正在找景点，打开车窗 → 静默开窗
④ 整合（核心）暂停旧的，整合上下文重规划规划行程中，"再加个点" → 重规划
⑤ 插队暂停旧的，新的播完，旧的继续播新闻中，"小心右侧行人" → 插播后继续
⑥ 停止旧任务旧的直接丢弃正开窗，"下个路口左转" → 停止开窗，处理导航

优先级矩阵：安全通知 > 导航通知 > 普通系统消息 > 配置类主动服务 > 条件任务 > 非配置类主动服务 > 简单 query > 复杂 query。

8.4 三条特殊规则

  1. 长程任务（订座/停车/点咖啡/外卖）不参与合并——它们有自己的 GUI 界面。
  2. aigc（生图/ailight）不走 Planner、无语音，完成后走通知卡。
  3. 同一 advisor 排队期间的多条建议只保留最新——防旧建议堆积。

8.5 任务工程管理（打断/恢复/挂起）

打断：端上请求中断 → 标记 plan 为 interrupted + 存 checkpoint → 新 query 处理。

恢复：新 query 处理完后，由 advisor（goal reviewer）决定是否恢复，不是 Director 自己记住。恢复 = 带上下文（含目标"细节和进展"）重新触发任务。续播：已缓存 → 从断点上一句续播；未缓存 → 基于上下文重新生成。

挂起：定时/条件任务注册后进入"挂起"，Always on 模块持续监听，条件满足通知 Planner 恢复执行。

为什么恢复靠 advisor 而不是 Director？Director 是无状态的 LLM——每次调用只读 SP+UP，它没有"持续记忆"。但 goal list 上的"目标细节和进展"是持久化的，advisor 读它就知道"上次说到哪了"。这就是目标驱动架构的核心价值——恢复不靠模型记忆，靠看板上的结构化进展。

九、执行层：26 工具 + HMI 卡片

9.1 工具执行原则

一个 Agent 只执行一个任务。串行依赖的工具（先查记忆拿地址→再导航）不能一次性全输出——注意事项第 6 条。但无依赖的可以并行（调空调 + 放音乐）。

9.2 成功 vs 失败

成功：tool_feedback 回灌 → Director 二次推理 → 很多情况选择不说话（车窗关了用户能感受到，音乐响了用户能听到）。

失败 → Replan：端上回传"未成功的原始任务 + 失败原因" → Director 综合当前上下文重新生成计划。注意事项说"不必道歉，想办法处理"。

9.3 HMI 三类卡片

卡片数量谁开发互斥
小卡（AI 语音卡）≤1我方互斥
大卡（AI 画布）≤1上汽容器，我方集成
通知卡（AI push）可多个上汽容器不互斥

播报归属：车控/音乐/短视频/GUI → Director 统一回复；本地生活/vqa/车书/s2s → agent 自己播；websearch → Director 总结后播；aigc → 不走 Planner，完成后走通知卡。
