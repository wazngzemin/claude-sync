VLM 条件任务 ·「模型解析」能力说明

以「后排宝宝哭了告诉我」为例，讲清楚这件事是什么、链路怎么走、需要各方准备什么
v1 · 2026-07-16 · 供各方对齐讨论
这份材料给谁看、讲什么：
面向 VLM（视觉大模型）算法、服务端研发、产品 三方。围绕一类新需求——"VLM 条件任务"（如"后排宝宝哭了告诉我""有人抽烟提醒我"），讲清楚其中一个关键环节：怎么把 VLM 看到的画面，判断成"条件达成了没有"（本文暂称"模型解析 / 语义匹配"）。全文用一个场景从头走到尾，读完你会清楚：这个环节是什么、放在链路哪里、输入输出是什么、有几种实现路线、需要各方准备什么。

一、先看一个场景：用户说"后排宝宝哭了告诉我"

这是一句很自然的话，但系统要完成它，其实要做三件事：

步骤要做什么谁来做
① 盯着持续观察后排宝宝车上的 VLM（摄像头 AI，每隔几秒看一次）
② 判断判断"宝宝到底哭了没有"本文要讲的环节 —— 模型解析 / 语义匹配
③ 播报哭了就语音告诉用户Planner（规划器，负责最终执行）
一句话：整件事的难点不在"看"（VLM 能看），也不在"说"（Planner 能说），而在中间那步——把"看到的画面"变成"哭了 / 没哭"的判断。这一步就是本文的主角。

二、核心难点：VLM 只会"描述画面"，不会"下结论"

为什么"判断宝宝哭没哭"需要单独一个环节？因为 VLM 有一个特点——它只输出对画面的客观描述（caption，画面描述文本），不直接给"是 / 否"的结论。

举例，同样是后排宝宝，VLM 每 5 秒可能这样上报：

第 1 次："后排儿童安静，闭眼休息"
第 2 次："后排儿童睁眼，面部平静"
第 3 次："后排儿童睁眼，脸上有泪珠，嘴角向下"

用户要的是"哭了就告诉我"——但 VLM 从头到尾没说过"哭了"两个字，它只说了"有泪珠、嘴角向下"。谁来把"有泪珠、嘴角向下"判断成"这就是哭了"？这就是"模型解析"要做的事。

为什么 VLM 不直接给"哭了=是/否"？因为 VLM 是一个通用的感知能力，它不预先知道每个用户会提什么需求。今天用户问"哭没哭"，明天问"睡没睡"，后天问"有没有玩手机"——VLM 没法为每一种可能的问题都内置一个"是/否"判断。它能做的是如实描述画面，把"要不要判定为哭"这个和具体需求相关的判断，交给后面的环节。

三、这个"判断"环节放在整条链路的哪里

把整个场景端到端画出来（每个盒子标了是哪个模块负责）：

① 用户"后排宝宝哭了告诉我"
↓
② Task Service（任务服务）把这句话变成一个持续任务，生成一个 高频任务（编号如 044，目标="宝宝哭了"），下发给 VLM
↓ 下发观察任务
③ 端侧 VLM（视觉大模型）每 5 秒看一次后排，上报画面描述（如"有泪珠、嘴角向下"）
↓ 每 5 秒一条画面描述
④ 模型解析 / 语义匹配　【本文主角】判断这条画面描述，是否达成了目标"宝宝哭了"
↺ 每来一条判一次，没达成就继续等下一条
↓ 判定"达成了"
⑤ Task Service收到"达成"信号，触发播报；任务完成后通知 VLM 停止观察
↓
⑥ Planner（规划器）→ 用户语音播报："后排宝宝哭了"
看这张图记住一件事：第 ④ 步"模型解析 / 语义匹配"是这条链路里唯一负责"判断条件达成没达成"的环节。前面 VLM 只管看、只管描述；后面 Task/Planner 只管触发和播报；中间"到底算不算达成"这个判断，就在第 ④ 步。

四、VLM 到底给什么：两类结果分开走（红框 / 绿框）

这里有个重要设计：VLM 把舱内高频观察的结果，拆成两部分、走两条订阅线，别混。用真实的输出样例看：

// VLM 舱内高频每 5 秒的输出（真实结构） { "乘客列表": [ // 🟩 绿框：车内整体情况 {"位置": "副驾后方", "动态": "头依靠在中间乘客肩膀上睡觉"} ], "动态任务": [ // 🟥 红框：用户 query 对应的观察结果 {"任务编号": "044", "识别结果": "后排儿童睁眼，脸上有泪珠"} ] }
哪部分内容走哪条线用来干嘛
🟥 红框
动态任务结果		 {"任务编号","识别结果"}
识别结果 = 用户 query（用户指令）对应的观察		 给"模型解析"环节 判断条件达成没达成
🟩 绿框
乘客列表		 车内每个位置有谁、在干嘛（整体情况） 走 Context（上下文）线 给对话/推理提供背景，不参与本文的判断
为什么要拆开："宝宝哭了告诉我"这个具体任务，只关心红框里编号 044 的识别结果，不需要整车情况。拆成两框、走两条线，让"判断某个具体任务"这件事拿到的输入干净、聚焦。

五、两个不同的周期，别混：看得多勤 ≠ 报得多勤

这里容易搞混，用场景讲清楚。假设用户说的是"每 60 秒告诉我宝宝在干嘛"：

周期这个例子里含义
VLM 上报周期5 秒VLM 每 5 秒看一次、给一条画面描述（供判断用）
用户反馈周期60 秒用户希望每 60 秒被告知一次结果
一句话：VLM 5 秒给一次"素材"，不代表要 5 秒打扰用户一次。"多久看一次 / 判一次"和"多久告诉用户一次"是两件事，需要分开设定。

六、"判断"具体怎么实现：两条路线

把"画面描述 → 达成没达成"这一步做出来，业界通常有两条路。它们没有绝对好坏，适用场景不同：

对比路线 A：标签提取路线 B：语义匹配
做法预先定义好一批标签（如"哭泣/平静/睡眠"），让 VLM 从画面里提取出标签，后面直接比对标签（等于"哭泣"就算达成）不预定义标签，用一个模型直接判断画面描述是否符合目标语义
举例适合"有人抽烟提醒我"——"抽烟"是高频、稳定、边界清晰的标签，VLM 直接提取"抽烟=是/否"适合"宝宝哭了告诉我"——"哭"的表现多样（泪珠/咧嘴/皱眉/哭声），难穷举成固定标签，靠模型理解
要准备标签清单 + VLM 的描述要包含相关信息 + 种子数据（训练样例）一个语义判断模型 + 少量样例
能覆盖只有预定义过的标签场景用户任意说的新需求，都能尝试判断
短板来个新需求就要加标签、要 VLM 支持、要重新准备数据，不够灵活模型判断会有误判样例（badcase），准确率需要打磨
◎ 建议：不二选一，分层组合
  • 高频、稳定的场景（抽烟、没系安全带、儿童站起来等）→ 走路线 A 标签提取：预置标签，VLM 提取，比对即可，又快又准。
  • 长尾、任意的场景（用户随口说的各种需求）→ 走路线 B 语义匹配：靠模型理解，保证"用户想到什么都能试着支持"。
这样既能把高频场景做得又快又稳，又能覆盖用户五花八门的长尾需求。两条路线共存，按场景选用。

七、输入 / 输出协议（草案，字段待各方确认）

7.1 输入：VLM（红框）→ 模型解析环节

字段类型说明 / 待确认
任务编号 task_nostring（字符串）Task Service 生成任务时分配，VLM 原样带回
识别结果 resultstring用户 query 对应的观察，如"后排儿童睁眼、有泪珠" 粒度待定
上报时间 tsint（时间戳）这一帧画面的时间
空值约定—没识别到 / 无异常时，输出什么？待定（建议：空串=未达成）
置信度 confidencefloat（小数）建议增加若 VLM 能给，判断可用它控制误判

7.2 输出：模型解析环节 → Task Service

字段类型说明
任务编号 task_nostring达成的是哪个任务
回调编号 callback_idstring本次触发的唯一标识
达成时的结果 resultstring触发时 VLM 的原始描述（"有泪珠"），供播报参考
触发时间 triggered_atint判定达成的时刻

八、需要各方共同确认 & 准备的事项

下面是把这件事落地，需要各方一起对齐的问题。标了"建议"的是初步倾向，欢迎讨论。

#待确认 / 待准备主要涉及初步建议
1路线选择：分层（高频标签+长尾语义）/ 纯标签 / 纯语义产品 + VLM算法 + 研发建议分层
2需要提取哪些标签（高频场景的标签体系）丁彬提的依赖①产品牵头整理先从高频场景整初版
3VLM 的描述能否支撑标签提取（没相关描述就提不出）依赖②VLM算法核对 caption对照《端状态格式表》逐项核；缺的改 prompt 或走语义
4构建种子数据（给研发训练/调试用）依赖③产品给 case + 算法构建高频标签场景需要；语义路线用少量样例
5标签/结果给谁用、存哪依赖④研发 + 产品供"模型解析"环节消费；存端状态/数据中心供订阅
6识别结果的粒度：VLM 给近结论（"在哭"）还是纯描述（"有泪珠"）VLM算法 + 产品越接近结论越好判，最终判定仍由解析环节做
7空值约定：没识别到时输出什么VLM算法约定空串=未达成，写进协议
8两个周期：上报周期 vs 反馈周期怎么解耦研发 + 产品判断与反馈分开设定
9判断准确率：误判/漏判怎么衡量、谁打磨算法 + 产品产品定策略（宁漏报不误报），算法打磨模型
10模型/成本：判断模型谁提供、调用量级算法 + 研发轻量模型、非常驻，成本另行测算
下一步：建议就上面 10 条，小范围先对一轮，形成初步分工与结论后，再进入正式技术方案评审（对齐接口、字段、排期）。
