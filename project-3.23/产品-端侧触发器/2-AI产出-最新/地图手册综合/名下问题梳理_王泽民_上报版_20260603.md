# 一、场景知识模块

> 场景知识 = "什么场景下该做什么"的知识，喂给 Selector（分类）、Skill（编排逻辑）、Planner（注入参考）。当前问题集中在**可用性、知识质量、分类准确率**。

## P0-1 ｜ 场景 Planner 已下架，618 前需重新上线
- **现状**：场景 Planner 端到端 ~800ms，拖慢整链路，已被令平决定暂时下架。需 Planner 编排的场景（小憩/观影/冥想/通风等 8 个）全废；另 5 个预置场景因未注册也无法语音触发 → **13 个场景全部不可用**。
- **要做**：① 618 前大量实车测试 + badcase 收集，驱动 Skill/SP 迭代；② 配合明骏优化 Planner 耗时（SP 压缩、动态加载）；③ **由我定"重新上线"的产品验收标准**（耗时降到多少、命中率达到多少）。
- **卡点**：耗时优化是研发侧，但上线验收标准要我来定。

## P0-2 ｜ Skill 知识质量差，写了车厂不具备的能力
- **现状**：13 个场景的第一版 Skill 存在——① 写了方向靠车控协议里**实际没有**的指令（如"观影模式"一级指令），Planner 据此下发必失败；② 误导调用路径（宠物模式写成编排，实际应走预置场景）；③ Skill 与车控协议从未逐条核对；④ 复杂触发条件没做分支（通风没写"是否下雨决定开窗幅度"）。
- **影响**：Planner 生成的 action_list 调不通 → 反馈前后矛盾（先说能做、失败后又说不行），体验极差。这是 P1-2 的根因。
- **要做**：① 拿小明的 bug 走查表逐条核对，删掉所有"Skill 写了但车上没有"的能力；② 找左晨确认方向靠完整车控清单后逐条对齐；③ 优先重写 badcase 高发的宠物/观影/通风/保驾护航。
- **卡点**：缺一份左晨提供的干净车控能力对照表。

## P0-3 ｜ 场景分类器（Selector SP v11）命中率仅 60–70%
- **现状**：约 1/3 场景 query 被错分/漏分。已知问题：① 边界判不准（"我好累"走小憩还是冥想）；② 多场景命中处理不一致（"好热好臭"该同时命中降温+通风）；③ 纯控制指令过滤不彻底（"调到25度"被误判极速降温）；④ 上下文信号（speaker_position/env_info）利用不足。
- **要做**：① 建 Selector 评测集（每场景 ≥20 条，含正/反/边界）；② 量化各场景 precision/recall；③ 针对低分场景改判定逻辑和互斥锚定规则。
- **卡点**：评测集未建；改 SP 需明骏配合上线验证。

## P0-4 ｜ 5 个预置场景未在方向靠注册，语音无法触发
- **现状**：离车保电/洗车/宠物/露营/大床 这 5 个场景车上有预置应用，但方向靠没有注册入口指令，Planner 无法 function call 调用，自己编排又不对 → 失败。
- **6/3 上午会议已决策**：这 5 个不走 Planner 编排，直接调预置场景。
- **要做**：① 整理 5 个场景的标准 query + 变体提给左晨注册；② Selector 命中后不注入 Skill、改走预置应用调用；③ 删除/改写这 5 个原编排 Skill。
- **卡点**：左晨注册排期未回。

## P0-5 ｜ 场景合并方案需落地到 Selector SP 和 Skill
- **6/3 上午会议已决策合并**：
  | 合并前 | 合并后 |
  |--------|--------|
  | 后排观影 + 驻车前排观影 + 5D 观影 | 观影模式（一份 Skill 内部分支）|
  | 极速降温 + 极速升温 | 极速调温（内部判升/降）|
  | 儿童陪伴 + 后排勿扰 | 用车关怀 |
- **要做**：① 3 份新 Skill 替代 7 份旧 Skill；② 更新 Selector 互斥锚定规则；③ 更新场景数据总览表。
- **卡点**：5D 观影是否合入还在犹豫（依赖腾讯视频协议确认）。

## P1-1 ｜ 场景退出/回滚 + AIBOT 表情结束信号缺失
- **现状**：Planner 编排的场景**没有任何回滚机制**——退出小憩后座椅/遮阳帘/音乐/空调不恢复原状。今天下午会议新增子问题：**AIBOT 表情 loop 没有结束信号**，场景关闭后表情一直 loop 到下个场景覆盖。
- **要做**：① 与明骏对齐"退出场景"时 action_list 能否加回滚动作；② 设计统一退出协议（车控回滚+表情结束+音频停止）；③ 确认 Planner 是否需新增 `robot_expression_control` 工具控制机器人表情。
- **卡点**：明骏今天下线需约时间；机器人表情控制接口技术方案不清。

## P1-2 ｜ Planner 回答前后矛盾、调工具后无结果
- **现状**：实车走查大量 badcase——先承诺"帮你打开 XX"调用失败后又反悔；自称"手动模拟"但实际没调工具；每调一个工具罗列一次操作，信息冗余。
- **根因**：① Skill 写了不存在能力（P0-2）；② Planner SP 缺"调用失败时的行为规则"（12 条注意事项没覆盖 tool_feedback 失败应对）；③ UI 卡片重复展示（非 Planner，已提测试）。
- **要做**：① 源头治理 Skill（P0-2）；② Planner SP 增加一条"tool_feedback 失败时不要先承诺再反悔，直接说暂不支持"；③ 等 bug 走查表逐条定 root cause。

## P1-3 ｜ 端状态信息利用不足，Planner 反问已知信息
- **现状**：Planner 反问"是否在 P 档"——而 P 档信息 env_info 里就有；通风不知是否下雨、降温不知当前温度。
- **根因**：① `{{env_info}}` 注入字段不全；② SP 没规则"env_info 有答案就别反问"；③ 部分信号（PM2.5、车外下雨）没接入。
- **要做**：① 找明骏要 env_info schema 确认缺哪些字段；② 在触发器需求1 里确认哪些端状态可作场景编排输入；③ SP 增规则"env_info 已有则禁止反问"。

## P2-1 ｜ 记忆 / FaceID 不可用，用户偏好无法保存
- **现状**：FaceID 未就绪 → "记住偏好"说记了实际没存；自定义场景无法持久化；无法查看已建场景列表。
- **要做**：FaceID 修复后重验记忆链路（写入 / 检索 / 列出自定义场景）。
- **卡点**：FaceID 修复时间不确定。

## P2-2 ｜ 自定义场景持久化 + 管理 + 分享（长期）
- 场景生成非确定、无"我的场景"管理界面、无分享功能。属长期需求，不在 618 目标内。

## 2.2 引擎能力

### P0-6T ｜ 需求5：触发机制只看"当前态"，分不清状态变更
- **现状**：引擎只扫当前信号值，看到"门是开的"但不知是"刚从关变开"，分不清上车/下车 → "上车打招呼""下车提醒拿东西"判不准。
- **要做**：重构为"状态变更 + 条件组合"（上车 = 门:关→开 且 座椅:无→有）。**这是需求7（端侧定时器）、需求8（距离事件）的前置。**

### 需求6：协议两套、配不出"任意位置"
- conditions（持续满足）与 dynamic_conditions（状态变更）两套协议；车门套娃结构配不出"任意一个门开"。→ 协议拍平合一 + 车门拆独立条件。

### 其他引擎问题
- 数值抖动过滤规则缺统一规范（区间/周期，如车速130 区间125-135、30s不重复）→ 不配会扰民、配错会漏触发。
- 频次类条件（计数）不支持（"1分钟切5首歌"），需求13 音频做不了。（长期）

## 2.3 下游链路 / 对外接口

### P0-7T ｜ 上汽无静态 Advisor，触发器下游链路断裂
- **现状**：原设计 `触发器→L17→静态Advisor→L19→Director`，但上汽实际没部署静态 Advisor，L17 断了。**触发器命中事件后不知道往哪发。**
- **影响**：① 主动服务链路跑不通（触发了没人消费）；② 多事件合并能力丢失。
- **要做**：① 临时方案——命中后走 L21 直连 Planner，让 Planner 替代 Advisor 决策；② 找吴仁杰重配下游链路；③ 评估 Planner 替代 Advisor 的性能影响（Planner 已过载）。
- **卡点**：6/3 下午会（婉婷拉），需 +张航、弋途的人一起定。

### P0-8T ｜ 无静态 Advisor 导致多事件合并丢失
- **现状**：主驾上车触发一次、副驾 3 秒后再触发一次 → 用户听到两次打招呼。
- **要做**：短期降级（分开打招呼）；中期在触发器冲突仲裁里加"同类事件 N 秒内连续 → 合并"。

### P0-9T ｜ 需求2：意图模型无法解析条件话术
- **现状**：用户说"超25度就开空调""到家附近找充电桩"，意图模型解析不出/不准，协议没跟后台 DSL 对齐。
- **要做**：基于需求1 给意图（婉婷）《条件词表》当约束 + 《协议范围说明》对齐五段式；确认其调试工期倒推我的交付 deadline。
- **卡点**：依赖需求1 定稿。

### P0-10T ｜ 需求3：益红 14 条主动服务未落地（Sheet3 9 处差异）
- **现状**：益红 14 条诉求做三 Sheet 核对（能做8/回掉/视觉排除），Sheet3 还有 9 处差异未拍板。
- **判定规则**：信号库查得到=能做、查不到=回掉、强依赖视觉=划掉。
- **卡点**：依赖需求1；9 处差异待与益红开会定。

### 需求10/11：导航 POI / 地理围栏能力缺失（领导点名 P0）
- 触发器不认导航目的地/距离，做不了"接近/离开某POI"。**前提**：需先评估云端能否拿到实时定位+POI 坐标（还没确认）。底层用地理围栏（圆心+半径+进出判定+防抖）。

### 需求12：事件接口未透出 + 缺自动化验证
- 下游（动态 Advisor）不知道触发器注册了哪些事件可订阅，query 接口不透出、没法自动化验证。

## 2.4 端侧能力
