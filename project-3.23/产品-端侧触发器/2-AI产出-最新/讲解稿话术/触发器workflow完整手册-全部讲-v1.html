第一篇 · 把这事彻底搞懂
拿去干嘛：给你自己。通读这一篇，workflow 这摊事你就再没有盲点。

1.1 workflow 到底是什么

以前你想在车上做个小功能——"挂 D 档、车窗还开着，就提醒关窗"——得写需求交研发，研发写代码、编进系统、发版本，好几周，你自己动不了，改一次再等一次。

workflow 就是来解决这个的：给你一个可视化界面，你不写代码、拖拖拽拽就能配出一条规则，配完点发布就生效。研发不用为每条规则写代码，你也不用等发版，所有规则清清楚楚摆在一个平台上。
一条规则，就是你手机上的闹钟：到 7 点（触发——什么时候开始）→ 如果是工作日（条件——还得满足啥）→ 就响铃（动作——那就做啥）。车上的规则一模一样，永远这三段。

拿你那张图，一秒一秒走一遍

平时
规则躺着，"档位切换"这个哨兵盯着挡把，不挂 D 档它啥也不干，不费电。
这一刻
你挂进 D 档，哨兵"叮"地醒了。（去抖 300ms：挡把抖一下又弹回不算，防误触发。）
下一步
往下走到条件：查"四个车窗有没有至少 1 个开着？"
岔路口
有窗开着 → 走 Yes 到动作；全关着 → 走 No，直接结束。
动手
走到动作：默认女声播报"行车中关闭车窗可以降低能耗哦"。
收工
播完到 End，哨兵回去睡，等下次挂 D 档。

脑子里有画面了吧：一条规则 = 一个哨兵盯着某信号，信号来了就醒，醒了先做判断题，题过了就动手。底下那行人话，是系统自动把你拼的积木翻译成的——你以后汇报，念那一句就行。

1.2 它跟你的配置平台差在哪

你现在的配置平台是"填表单"，workflow 是"画流程图"。一个把条件平铺着填进一张表，一个把逻辑画成有起点、能拐弯的流程。
比什么配置平台（现在）workflow（升级后）
形态填表单画流程图
能不能拐弯不能，条件满足就执行能，有 Yes/No 分叉
触发逻辑几个条件一起 AND事件当起点 + 条件过滤（漏斗）
给谁用触发器自己整个系统（Director/Advisor/主动服务）
积木各场景各配原子能力库，复用
运营配好就完草稿/试运行/发布/版本/命中率
端云偏云端配置端云同步下发 + 离线标记
最核心就一条：workflow 能表达"起点+流程+分叉"，配置平台不能。韩杰 7.15 反复说"中间一定有复杂逻辑，不是触发了就执行"——那种拐弯，表单装不下，得靠画布。它俩是同一个东西的两代，你的活就是主持这次"从表单到画布"的升级。

1.3 两个维度别搞混：谁做 / 放哪

这是你最容易乱的地方，记死：有两个完全独立的维度。

维度问的是两个选项
编排 vs 底层谁来做这个逻辑编排（你拖画布）/ 底层（研发写代码造积木）
端侧 vs 云端这东西放哪台机器跑端侧（车上盒子）/ 云端（远处服务器）

两个维度正交，四种组合都真实存在：

例子谁做放哪
D 档车窗提醒编排·你拖端侧（断网要用）
挂 P 推充电桩编排·你拖云端（查地图）
舱内识别宝宝底层·研发造端侧（舱内 VLM 在车上）
舱外识别雨雪底层·研发造云端（端侧还没这能力）
这条对你救命：画布上没有的积木，你配不出来。舱外识别雨雪这个积木端侧还没造出来，那这条规则现在就配不了——不是你设计得不好，是底层积木没有。跟研发别承诺"画布什么都能配"。

1.4 Signal Hub 是什么 + 背后六层能力

那张图叫"Signal Hub 规则工作台"。Signal Hub = 信号中枢（韩杰的原料层），管信号从哪来、怎么监控缓存；规则工作台 = 画布（编排层，该是你的）。合起来 = 建在信号中枢之上的规则编排画布。（名字挂着 Signal Hub，是个归属信号，第二、五篇细说。）

要让这厨房真出菜，从"信号进来"到"动作落地"，背后是六层能力：

能力层干什么归谁
1 信号接入各种信号收进来、标准化信号中枢·韩杰
2 信号加工监控变化生成事件、缓存、实时处理信号中枢·韩杰
3 规则编排引擎画布+分支、原子库、条件判断、频控仲裁、callback 派发、人话预览工作台·你
4 端云同步云端配好下发端侧、离线执行研发
5 运营管理草稿/试运行/发布/版本/命中率/灰度你 + 研发
6 动作执行TTS 播、车控调接口、Advisor 触发下游模块
第 3 层是你的护城河，拆开是五样硬能力：①可视化画布 ②原子能力库 ③条件判断引擎 ④频控+仲裁 ⑤callback 四出口。第 1、2 层是韩杰的原料，第 4、6 层是研发和下游的执行——只有第 3 层是你，守住它。

2.2 你现在的定位

原料归信号中枢（韩杰）——信号接入、监控、缓存。编排归触发器（你）——事件条件编排、K2 时间窗、频控、仲裁、callback 四出口、workflow 画布。7.9 白板会原话："不要开发两个条件引擎。"这句是全组共识，是你的免死金牌——谁想把编排挪走，你用这句挡。
第三篇 · 端云分工 · 第一性原理
拿去干嘛：跟研发对场景时用。不背台词，从根上推，谁问都不慌。

3.1 从一条事实推出全部

端侧和云端中间隔着一条网，这条网会断。

三个躲不掉的硬道理，全从这条推出来：

  1. 断网时端侧够不着云端 → 断网也要能用的功能，必须整个待在端侧（硬约束）。
  2. 端侧算力小、云端算力大 → 又重又复杂的判断（跑大模型识别雨雪），倾向云端。
  3. 数据天生在哪边 → 车上信号（档位/车速）在端侧、地图天气在云端，判断倾向在数据那边做。

判断任何功能放哪，就问这三个问题。而谁来答这三个问题，是关键：

问题要懂什么谁答
断网要不要能用业务/体验你（你的专业）
算力够不够技术研发
数据端侧有没有技术研发
7.8 你就栽在"数据端侧有没有"这道技术题上——你写"端侧判断雨雪"，端侧根本没这数据能力，你答了不该你答的题，当场被戳穿。以后你只答"断网要不要用"，别碰另外两道。

3.2 跟研发就说三句话

① 编排归我"这规则怎么拐弯，我在画布配。"
② 积木你造"里面用到的积木，哪些你已经有、哪些要你造？没有的我配不出来。"
③ 端云你定"这条断网要不要能用我告诉你，落端落云你定。"

三句话三件事各归各。你 7.8 漏洞百出，就是把三件搅一件、还替研发答了技术题。拆开、各答各的，就再没人能当场戳穿你。

4.1 workflow 画布产品方案（目录 + 节点规格）

方案该包含这几章：①定位（整个系统的规则统一在这配）②节点类型 ③连线逻辑 ④自然语言预览 ⑤草稿/试运行/发布/版本 ⑥端云同步（每条规则打离线标记）⑦原子库管理。

每种节点配什么参数

节点参数
触发显示名、信号源、运算符（变更为/有变更）、目标值（可多选）、去抖 ms、仅点火触发、记事件日志
条件信号、位置（至少1个/全部，多选）、运算符（等于/大于…）、值、多条件且/或
动作动作类型 + 对应参数（TTS 配音色文本、车控配指令、Advisor 配场景）
分支/起止Yes→下一步 / No→结束或另一动作；Start / End 无参数

5.3 给老板汇报 · 一页要点

一句话讲清这摊事：我们要把整个系统的规则，从"埋在代码里、每条都要研发写"，升级成"一个可视化画布、产品运营自己搭积木配"，这就是 workflow 规则引擎。

为什么要做：规则会越来越多，代码写不过来、改一次等发版太慢；画布化之后加规则几乎零成本，还能看数据做运营。

我负责什么：这个规则编排画布的产品设计——原子能力库、场景剧本、编排逻辑、频控仲裁。信号原料归信号中枢，端云实现归研发，规则怎么配这一层归我。
