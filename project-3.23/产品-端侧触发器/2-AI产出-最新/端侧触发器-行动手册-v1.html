01

两个概念:触发事件 vs 触发条件

这两个词你这几天一直没绕明白,先花五分钟彻底讲清。讲清了,后面所有事你才判断得了好坏。

先说结论:这两个名字是韩杰新起的,东西你早就配过。触发事件就是你配置平台里的动态条件,触发条件就是静态条件。会上刘杨当面点破,韩杰认了。

= 动态条件
触发事件
关注:状态变没变
是什么
车上某个状态,从一个值变成另一个值的那一下。你盯的是"变化"发生的瞬间。
举例
  • 档位从 P 切到 D
  • 主驾座位从没人到有人
  • 车门从开变成关
一句话记住
从 A 变成 B,重点在"变"。
= 静态条件
触发条件
关注:此刻是不是
是什么
当前这个状态,是不是某个值。你盯的是"此刻的状况",不管它怎么变来的。
举例
  • 车窗此刻是不是有一扇没关
  • 车速此刻是不是 0
  • 温度此刻是不是高于 28℃
一句话记住
现在是 B,重点在"是"。
同一个信号,看你怎么用就是不同的东西。车速从 0 变成非 0 是事件(起步了);车速此刻是不是 0 是条件。
于庆磊 · 会上确认
"动态是不是就理解成状态从 A 到 B,静态就是当前的状况是 B?" 韩杰:"对,是的。"

这两样,你分别要做什么

要产出的东西怎么做
触发事件 ① 哪些状态变化算一个事件
② 每个事件的描述拼接规则
③ 运算符怎么定		 把旧条件表里"变化型"的挑出来,每条写清:监控哪个信号、什么变化算数、拼成的描述长什么样(比如"档位"+"切入"+"D 档"="档位切入 D 档",这句话事件日志直接拿去用)
触发条件 ① 哪些是过滤用的判断
② 串行还是并行
③ 和事件怎么组合		 把"是不是型"的挑出来,标清判断的值(车窗=有未关);再标它是事件响了之后第几道过滤、几道之间是且还是或
这里有个坑,刘杨已经当场挑出来了。 韩杰把你表里"变更/变更为"这类运算符,改名叫"切入/切出",但底层根本没实现,语义也对不上。刘杨问:"我们现在不支持切入切出啊,你是把旧运算符改个名,还是要真实现一个?" 这个洞是你表里的,得你来定口径,不能让它错着往下走。
02

你的触发器架构,从哪变到哪

7.10 韩杰把触发器从"一张并列条件表"重构成了"三段式漏斗"。先看整体变化,再看逐层清单。

以前的触发器
配置平台一张表:
动态条件＋ 静态条件
所有条件一起轮询,全满足就触发
动作只有一种:调 Advisor
定位:开关 VLM、调 Advisor 的条件判断器
→
7.10 之后
拆成三段,串行漏斗:
触发事件→ 条件→ 动作
事件先响,再逐道过滤,最后执行
动作变多种:Advisor / 事件日志 / TTS / 规则引擎
配置平台升级成可拖拽的 workflow 画布

逐层看,每一层要改成什么、谁在动

层以前现在谁在动
条件数据 动态/静态条件,一张表 拆成触发事件表 + 条件表,并进韩杰的信号中枢 韩杰已动手
你要复核
运算符 变更为 / 变更 等 改名成切入/切出,底层没实现,语义待定 你定口径
刘杨实现
执行逻辑 并列轮询,全满足触发 串行漏斗:事件→条件→动作 韩杰架构
动作类型 只调 Advisor Advisor + 事件日志 + TTS + 规则引擎 你定扩展点
配置界面 表单式填条件 拖拽节点的 workflow 画布 归属要抢
产品定位 条件判断器 编排层 owner?还是中枢的数据源?没定 你必须抢
一句话看懂这张表: 上面四层是数据和逻辑,韩杰已经在改,你的活是复核校准,别让它错着走。下面两层(画布、定位)是 7.10 悬着没定的,你的活是上桌去抢。数据改错了能补,定位让掉了就没了。
03

为什么要这么改

先讲韩杰为什么拆(他的四个理由),再讲为什么这事你必须掺和,不能任由它发生。

韩杰拆条件的四个理由

1事件日志需要一句人话
Director 要记"14:03 档位切入 D 档"这种一句话的事件。旧的并列条件里,档位、车窗、车速地位一样,系统不知道哪个是"发生的事",拼不出这句话。拆开之后,触发事件天生就是那句话。
2旧结构只能做最简单的规则
并列条件只能表达"A 且 B 且 C 就触发"。车上真实的场景经常要"先等档位变,延时三秒,再看车窗,不同结果走不同分支"。有先后、有时延、有分支的流程,并列做不了,串行漏斗才行。
3要做可拖拽的运营画布
韩杰的终点是把配置平台变成 workflow 画布,运营拖节点就能配规则。画布得有不同类型的节点才拖得起来。三类节点(事件、条件、动作)是画布的前提,一堆一样的并列条件画不成流程图。
4多个下游共用一套底座
现在不止 Advisor 用这套"状态变化→判断→执行",事件日志也用,以后规则引擎、任务大师都用。拆成通用的事件+条件+动作,大家挂不同的动作就行,不用每个下游各搭一套。

为什么这事你必须掺和

上面是韩杰的理由。但对你来说,真正的问题是:这套重构会顺手把你的东西拿走,你在场不发声,就等于默认。

你不掺和
会发生什么
  • 条件表被别人拆,拆错了也没人替你把关(这次是刘杨替你抓的)
  • "端侧触发器"这个名字被消解(韩杰已经在说"不提这概念了")
  • 编排画布归了韩杰,触发器只剩个数据源
你掺和
能拿回什么
  • 重新握住条件的定义权(映射规则你写)
  • 拿下编排画布,这是触发器的新产品
  • 把端侧触发器的场景立起来,产品有实体
04

做到什么程度算够

"做到什么程度"最怕没标准,越做越多也不知道停。给你三层,每层一个"够了"的判断线。

第一层 · 数据
把条件表映射干净,交给韩杰能直接用

每条旧条件都归好类(事件还是条件)、每个事件写好描述规则、运算符标清楚(切入切出到底是新实现还是改名)。

够了的标准: 韩杰拿去不用回来问你,刘杨照着能直接实现。
第二层 · 定位
把"端侧触发器"和"编排画布"定在你名下

出一份端侧触发器场景清单(断网要保哪些能力、预置哪些脚本),再和韩杰当面把"编排归触发器"对齐。

够了的标准: 下一场会再有人问"要不要端侧触发器",答案是你给的,不是韩杰替你给的。
第三层 · 你自己
对每个改动能当场判断好坏

别人动触发器的任何一处,你当场能反应过来这对你是有利还是掏空,不用等会后再琢磨。

够了的标准: 不再需要开完会追着问"他刚才那样改是什么意思"。
三层的先后: 第一层是本职,不做交代不过去;第二层是关键,决定触发器还是不是一个产品;第三层是底子,前两层都靠它。时间紧就先保第二层的定位,再补第一层的数据。定位让掉了,数据做得再干净也是给别人的画布打工。

06

你去找谁,说什么

前面都是分析,这一节是可以直接照着做的。四个人,四件事,连开场白都给你写好了。

1找赵衍本周 · 最急
做什么
出一份"端侧触发器场景清单":哪些能力断网也要能用、预置哪些场景脚本、哪些走端侧运营。这正是他在会上点名"必须产品定"的东西。
怎么开口
"衍哥,你 7.10 说端侧触发器要闭环哪些场景得产品定,这个我认。我出一版场景清单,离线要保的、预置的、运营的,列出来咱们对一遍。"
为什么先做这个
他当众把定义权交给你,韩杰却替你让了。趁他的话还在纪要里、韩杰方案还搁置着,窗口开着。过了这村不一定有这店。
2找韩杰本周
做什么
单独约半小时,把两条边界当面钉死:一,端侧触发器要闭环哪些场景由你定;二,触发事件→条件→动作怎么编排成规则,那块画布是触发器的产品。信号原料归他的中枢,这个你支持。
怎么开口
"韩杰,信号中枢我支持,原料归你没问题。但两点得先跟你对齐:端侧场景清单我来定,编排画布归触发器。你对外说'和泽民对齐过'之前,咱俩先真的对齐一次。"
为什么
"和泽民对齐过了"这句话很危险,它把你的沉默变成了同意。你都不知道自己被"对齐"掉了什么。当面说清,下次才不会又被代言。
3找刘杨7.30 前
做什么
把"切入/切出"这个运算符定清楚:是要底层新实现一个,还是就复用旧的"变更为"改个名。顺便一起复核你那张条件表转成两表之后,运算符有没有丢。
怎么开口
"杨哥,你 7.10 提的切入切出那个点我来定口径。咱俩过一遍我的条件表转成事件/条件两表,哪些运算符要真实现、哪些是改名,我给你个明确清单。"
为什么
这是你的表,这次是他替你抓的错。你对自己数据的把关不能比外人慢。补上这个洞,顺手就把条件的定义权重新握回来。
4找明骏7.30 前
做什么
对一下他的事件日志要的事件描述,长什么样、要哪些字段(谁、几点、做了什么)。你的触发事件拼接规则,得拼出他能直接用的那句话。
怎么开口
"明骏,你 Director 要的事件日志,一条长什么样?要不要带人、带时间?我把触发事件的拼接规则按你的格式来定,省得你拿到还得再加工。"
为什么
事件描述是触发事件的产出物,也是触发器给 Director 的价值证明。对齐了格式,这条链路才真的通,你的事件表也才有明确的产出标准。

这份手册说到底就一件事:7.10 你没上桌,你的表被拆、名字被让、画布往韩杰那边滑。这些都还没定死,窗口还开着。

前面五节是让你看懂局面,第六节是让你动起来。先做第一件,找赵衍,把他递给你的场景清单接住。这一步做完,"端侧触发器"这四个字才重新是你的。
