图①系统主链路 —— bug 在哪几个节点注入

怎么看：中间一行是一句话从语音到执行的流水线；上方是喂给 Planner 大脑的 4 种输入；每个节点下方的彩色小牌=这里会注入哪类 bug。蓝色 Planner 是唯一真该背锅(A/B)的大脑，其余节点的红牌都是"别处的锅"。
A 推理 B 幻觉 C 下游接不住 D 非Planner 蓝绿=喂给大脑的输入

图②分诊决策树 —— 拿到一个 bug，6 步判出它属哪类

怎么看：从顶上一个 bug 进来，沿中间黄色判断框一路往下；每个框只问一个是非题，命中就向右拐到对应类型（彩色出口），不命中就继续往下。走到底没拐出去的，才是真正归 Planner 大脑的 A 推理。这张图就是把"怎么甩锅/怎么定责"标准化。

图③解法分层漏斗 —— bug 怎么一层层被剥掉，只剩真问题给模型

怎么看：100% 的 bug 从左进，每经过一层处理就剥掉一批（蓝条变矮）；条形下方是这层用的手段，灰色掉落是这层剥走的 bug。核心思想：越靠前用越轻的手段把"不该大脑背的"剥干净，最后只剩 3–4 个顽固问题留给最重的模型训练。
产品/工程可快速做(L0–L3) 端状态改造(治本) 跨团队·下游 算法训练 顽固残留

图④评测闭环 —— 为什么"把 bug 分清类"本身就在防复发

怎么看：顺时针一圈。线上抓到的 badcase，先按图②归到 A–F，再翻译成 Director Rewards 的可量化扣分项，评测/训练时自动打分，模型朝高分收敛，同类 bug 减少 —— 下一轮 badcase 就更少。所以分类不是为了写报告，是在喂训练系统。
