一、scope（范围）变更：哪些场景归触发器做，哪些场景不归触发器做 6.22结论

先说结论

6.22之前的理解：端侧触发器要接所有端侧场景，包括起步安全、上下车安全、睡眠检测这些。
6.22纠正：这些场景全部回赛力斯CDC（底层控制软件），触发器不做。

判定逻辑——怎么判断一个场景归谁做

判定逻辑（仁杰 13:14 提出，所有人认了）

拿到一个场景，问三个问题：

这个场景的触发条件会不会变？
"起步安全"的条件永远是"档位从P切到D + 安全带未系"，不会变。
"帮我看宝宝毯子掉没掉"的条件是用户临时说的，下次可能不看毯子看水杯。
这个场景需不需要用户创建/配置？
"起步安全"不需要用户做任何事，出厂就有。
"到公司了提醒我拿外卖"需要用户主动说一句话来创建。
判断条件是纯数值比较（key-value 键值对），还是需要AI推理？
"起步安全"就是 档位==D && 安全带==未系，纯数值比较。
"毯子掉了"需要VLM（视觉语言模型）持续看，看完还要云端模型判断语义。

规则：三个问题只要有一个答案是"不变/不需要/纯数值"——>大概率归CDC（底软）。三个答案都是"会变/需要/要AI"——>归触发器。

仁杰 13:14：像那种车辆安全、上车安全、下车安全，它是所有车一个标配的功能，不会变，不涉及条件变化或用户改配置。那这种直接底软做掉。那种有生命周期的、用户临时创建、消费完就没了、条件可能变化的，走触发器。
为什么不让触发器也做固定场景？

仁杰 14:32 解释了：如果让触发器做起步安全，触发器要去接底层所有信号（档位、安全带、车门状态...），还要考虑触发器本身的稳定性和延时能不能满足安全要求。而CDC（底软）本来就在读这些信号，本来就在做车辆控制，直接在里面加一个判断就完了，链路短、延时低、稳定。

西钺 16:26 的原话："这样很清晰很干净，不然两边触发器会打架。"

这个结论带来的开发影响

  • 开发做 端侧触发器㉔的scope（范围）收窄——不再接CDC（底软）闭环场景的信号，只处理云端⑮下发的动态任务。之前如果有对接起步安全/上下车安全信号的代码，可以去掉
  • 产品(我)做 PRD（产品需求文档）里删掉"端侧任务②简单条件判定"和"端侧任务③白名单车控直执"
  • 产品(我)做 跟赛力斯确认他们各业务场景之间的仲裁机制——西钺 17:14 发现赛力斯内部多个业务同一时机同时提醒，没做仲裁
边界情况：CDC（底软）和触发器职责交叉

同一个时机两边都想响应：比如用户起步时，CDC做起步安全检查，触发器也注册了一个"起步时播放音乐"的任务。这两个不冲突——CDC管安全，触发器管体验。但如果CDC的安全检查阻止了起步（比如安全带没系），触发器的"播放音乐"也不该执行。→ 需要一个仲裁机制，西钺 16:26 说了"到时候跟赛力斯做一个系统的仲裁"。

赛力斯CDC能力不够怎么办：有些场景虽然"固定"，但CDC做得粗糙。这种情况触发器不越界，CDC做到什么程度是赛力斯自己的事。

二、四条铁律：系统的四个不可违反的设计约束 已定 + 铁律④是6.22新定的

先过一遍这四条，后面讲逻辑时会反复引用。

四条铁律
铁律①：动态注册必须经⑮云端触发器中转
意思是：不管是用户说的条件任务（"10分钟后关加热"），还是Dynamic Advisor（动态顾问）注册的视觉观测任务（"帮我看毯子"），都不能直接发给端侧㉔，必须先到⑮云端触发器注册，由⑮决定是留在云端执行还是下发给端侧执行。
为什么：如果允许各个模块直接绕开⑮给端侧发任务，端侧㉔不知道云端还有没有冲突的任务，冲突仲裁就做不了。⑮是唯一知道"全局有哪些任务在跑"的地方。
铁律②：端侧结果必须走④信号库→㉑Context (上下文)，不能直推不跳级
意思是：比如VLM (视觉语言模型) 观测到"后排有个小孩在睡觉"，这个结果必须先写入④端侧信号库，再通过L9同步到⑫云端信号库，再按策略上报到㉑Context (上下文)。不能VLM直接推给Advisor (顾问)。
为什么：如果允许VLM直推Advisor，那信号库里就没有这条数据，触发器读信号库时就读不到，Context里也没有，Director (导演) 推理时也拿不到这个信息。所有模块都从信号库读数据，保证数据源唯一。
铁律③：触发器只读④信号库，不碰③原始信号源
意思是：端侧触发器㉔要读车速，只能从④信号库读，不能直接去问③CDC信号源"现在车速多少"。
为什么：③原始信号源有6类（CDC信号、VLM输出、埋点、围栏、音频、时间），格式各异。④信号库已经把它们统一成标准格式（key键/value值/source来源/timestamp时间戳/ttl有效期）。触发器只面对一种数据格式，不用关心底层信号是从哪来的。
铁律④（6.22核心新增）：condition（条件/状态）→ ㉑Context供读取；event（事件/变化）→ ⑮组装后push（主动推送）给Advisor
这条是6.22整场会最核心的架构决策，下面第三章专门展开讲逻辑。

三、event（事件）vs condition（条件）：核心设计选择的完整逻辑 6.22结论

先说问题——为什么需要区分这两个概念

在6.22之前，我的理解是：所有数据都存到㉑Context (上下文) 里，Advisor (顾问) 需要什么就去读什么。佳杰在会上纠正了这个理解。

佳杰 26:35：我需要一个"用户睡醒了"的事件。我不会去轮询你的Context (上下文)。我注册了这个事件之后，就等待你的云端触发器给我发这个事件。

佳杰的逻辑——用一个具体case拆解

case："用户倒完车之后，如果车里有小朋友，就把儿童锁打开"（佳杰 29:34）

这句话拆成三个部分：

  1. "用户倒完车"——这是一个event（事件）。它描述的是"刚刚发生了一个变化"：档位从R (倒档) 变成了P (驻车档)。
  2. "车里有小朋友"——这是一个condition（条件/状态）。它描述的是"当前持续存在的状态"：车里现在有一个小孩。
  3. "把儿童锁打开"——这是action（执行动作）。Advisor (顾问) 判断"event满足了 + condition也满足了"之后，给Director (导演) 发advice (建议)，Director决定执行这个动作。

为什么event不能存到Context里让Advisor轮询？

逻辑推演：如果event也存Context，会出什么问题
假设我们把"用户倒完车"也存到Context里。 Context里就会有一条数据：{档位: P, 上次档位: R, 变化时间: 14:32:05}。
Advisor要知道"用户刚刚倒完车"，就要每隔几秒去轮询Context，看看档位有没有变化。 轮询频率设多少？1秒？3秒？5秒？
问题来了：
  • Advisor不止盯"倒车完成"一个事件。舒适Advisor要盯"用户上车"、"温度变化"、"下雨"...。出行Advisor要盯"到达目的地"、"路线偏离"...。如果每个Advisor都在轮询Context，轮询量很大。
  • 而且大部分时候轮询的结果是"没变化"。用户可能一天只倒一次车，但Advisor每3秒问一次"倒车了没？倒车了没？"——浪费。
  • 佳杰 29:34 原话："第一趴'车在倒车'这个event，我不可能无限制的24小时在轮询。"
正确做法：Advisor不轮询，触发器push（主动推送）。
  • Advisor在注册时就告诉⑮触发器："我关注'倒车完成'这个事件"。
  • ⑮持续监听⑫云端信号库 (④端侧信号库的云端镜像) 的信号变化。
  • 当⑮检测到"档位从R变P"（阈值穿越），⑮自己组装一个语义化的event："倒车完成"。
  • ⑮主动push这个event给订阅了它的Advisor。Advisor平时什么都不用干，等推送就行。
Advisor收到event之后，再去读Context拿condition。
  • 收到"倒车完成"event → 去Context读"车里有没有小朋友"这个condition → 有 → 给Director发advice："建议打开儿童锁"
  • 如果Context里没有"车里有小朋友"的记录 → 不触发action，什么都不做
总结这个设计选择：

condition（条件/状态）= 持续存在的、变化慢的、需要时去读的。
→ 存在㉑Context (上下文) 里。
→ 举例："车里有小朋友"、"车内温度32度"、"当前车速80"、"毯子在座椅上"。

event（事件/变化）= 瞬间发生的、需要第一时间知道的、不轮询等推送的。
→ ⑮检测阈值穿越后组装，push给Advisor。
→ 举例："用户刚刚醒了"、"车刚刚驶出地库"、"毯子从座椅上掉到地上了"、"档位从R变P了"。

两条路不混。Advisor不轮询Context拿event，触发器不存condition到Context。
边界情况：同一个信号既是event又是condition

"车速"：如果Director (导演) 推理时需要知道"当前车速是多少"——这是condition，存Context。如果Advisor注册了"车速超120时提醒我"——"超过120"这个穿越瞬间是event，⑮检测到后push。同一个信号可以同时产生event和condition，两条路各走各的。

"毯子的位置"：VLM每3秒输出"毯子在座椅上"——这是condition，存到④信号库→㉑Context。某次输出变成"毯子在地板上"——⑮检测到穿越（"座椅上→地板上"），组装event"覆盖物位置变化"push给Advisor。两条路不冲突。

边界情况：event push失败了怎么办

Advisor实例正好在重启：⑮保留最近N条未确认的event，Advisor恢复后补发。

Advisor已经被销毁了（任务结束了）但event还在push：⑮检测到订阅者不存在，自动取消该event的订阅，清理资源。

多个Advisor订阅了同一个event：⑮同时push给所有订阅者，各自独立消费。可能导致多个Advisor同时给Director发advice → Director自己做去重合并。

四、VLM（端侧视觉模型）链路：一步一步怎么走 6.22结论

VLM链路是6.22讨论最久的（19:19-42:34），因为它涉及端云之间的多次交互。下面用"照看后排宝宝，毯子掉了提醒我"这个case，一步一步拆完整逻辑。

第一段：用户说话 → Director建任务 → Dynamic Advisor拆解

步骤1-5：请求从用户到Dynamic Advisor（云端）
用户说"帮我照看后排宝宝，毯子掉了就提醒我"。
①拿到音频 → ②ASR (语音识别) 转成文字 → L2上传云端。
⑨拒识仲裁 (意图判断) 做四路分流。
这句话不是简单车控（不是"打开车窗"），不是纯条件任务（不是"10分钟后关加热"），是一个需要持续视觉观测的复杂需求。
→ 判定结果：走L5送到⑲Director汇入 (导演输入口)。
⑲Director汇入 + ㉑Context (上下文) 注入 → ⑳Director (导演) 推理。
Director拿到用户的话，同时拿到Context里的当前状态（"后排座椅有一名3岁儿童"）。
Director推理出来的判断：
- 这是一个持续监控类任务，不是一次性的。
- 需要持续视觉观测后排区域。
- → 在⑱GoalList (目标看板) 建一个目标："监控后排宝宝毯子状态"。
- → 同时回复用户一句话（TTS播报 语音合成）："好的，我帮你看着宝宝的毯子。"
⑱GoalList建目标 → 创建⑰Dynamic Advisor (动态顾问) 实例。
⑰是一个临时"监工"，专门盯"后排宝宝毯子"这一个任务。任务完成/用户取消后⑰就销毁。
⑰Dynamic Advisor拆解用户的需求。
⑰拿到的输入是："帮我看后排宝宝，毯子掉了就提醒我"。
⑰要做的判断：
A. 需要什么数据来完成这个任务？
→ 需要后排的视觉信息，具体是"毯子/覆盖物的位置"。这是一个持续观测需求，不是问一次就行的。
B. 触发条件是什么？
→ "毯子从座椅上到了地上"——这是一个阈值穿越事件（位置从A变到B）。
C. 拆解结果：
→ 改写出一条注意力SP (系统提示词)："持续观测后排区域，上报覆盖物（毯子/被子/毛巾等）的位置（座椅上/地板上/人身上）"
→ 向⑮云端触发器注册一个视觉类event：当"覆盖物位置"发生穿越时push给我
边界情况：用户的需求太模糊怎么办

如果用户说的是"帮我注意车里的情况"——太泛了，⑰Dynamic Advisor (动态顾问) 没法改写成具体的SP (提示词)。应该让豆包追问："您希望我帮您看什么？比如看宝宝、看宠物、还是看物品？"

第二段：触发器注册 → SP下发端侧 → VLM开始观测

步骤6-9：任务从云端下发到端侧VLM开始跑
⑮云端触发器收到⑰的注册请求。
⑮要做的事：
- 校验：信号源可用吗？（端侧有摄像头吗？端侧AIBox AI运算盒 在线吗？）
- 校验通过 → 注册成功，存入场景库
校验不通过怎么办
→ ⑮返回注册失败+原因（"端侧摄像头不可用"）
→ ⑰收到失败，给Director发advice："当前设备不支持视觉监控"
→ Director告诉用户："抱歉，现在没法帮您看后排"
⑮做步骤h：端云分类——这个任务在端侧执行还是留云端？
判断逻辑：这个任务需要VLM (端侧视觉模型) 持续观测，VLM跑在端侧AIBox上，所以任务要下发到端侧。
→ 走步骤i：通过L14下发场景定义 + 动态SP (提示词) 给㉔端侧触发器。
㉔端侧触发器收到云端下发的场景。
㉔做的事：
- 存到本地场景库（持久化，重启不丢）
- 准备运行指令，包含7个字段：task_id (任务编号) / task_goal (任务目标) / priority (优先级) / timeout (超时时间) / mutex_group (互斥组) / trigger_source (触发来源) / expire_at (过期时间)
- 发运行指令给㉕感知器 (VLM调度器，注意：感知器≠VLM本身，感知器负责调度VLM)

步骤b：信号扫描

①信号更新时立即扫（⑫云端信号库收到新数据时触发）
②每5秒定时扫一遍（兜底，防止信号更新事件丢了）
扫描范围：只扫当前这辆车的VIN (车辆识别号) 下、这个场景关注的信号字段。不是全量扫所有信号。
如果端侧断网，⑫长时间没有新数据
→ 5秒定时扫兜底，但扫到的是旧数据。旧数据跟上一次快照一样，不会触发穿越。所以不会误触发。
步骤c：规则判定
分两类规则：
死规则（纯数值比较）
温度 > 45？距离 < 100m？播放次数 == 10？
这种直接比较，没有歧义。
如果是非结构化数据（VLM的文本输出）
VLM输出的是自然语言文本，比如"后排座椅上有蓝色毛毯"。不能直接做数值比较。
→ 进入步骤d：模型推理。
步骤d：模型推理（只有非结构化数据才需要）
用一个轻量级模型把VLM的文本输出转换成结构化数据：
输入："后排地板上有蓝色毛毯"
输出：{rear_cover_position: "地板上"}
转完之后就可以拿去做规则判定了。
模型推理耗时长或失败
→ timeout (超时) 机制，超时按"未满足"处理，下次扫描周期重试
步骤e：条件全达成？
一个场景可能有多个condition（条件）。只有全部满足才往下走。
举例：场景"主驾上车 且 舱内<10℃ 且 舱外<5℃"
当前信号：舱内9℃ / 舱外3℃ / 主驾上车 → 三个condition全中 → 往下
如果舱内12℃ → 不满足第二个 → 不触发，等下次扫描

DSL (领域语言) 支持AND（全满足）和OR（满足其一即可）的逻辑组合。
步骤f：数值抖动过滤
防止信号在阈值附近来回跳导致反复触发。
规则：30秒内同一个穿越方向不重复触发。
举例：车速在128→132→129→131之间跳。"车速>130"这个穿越在30秒内只触发一次，不会触发4次。
VLM场景也一样：毯子在座椅边缘，VLM一会说"在座椅上"一会说"在地板上"→ 30秒内只算一次穿越。
步骤g：冲突仲裁
多个场景同时触发时，谁先执行？
- 每个场景有一个优先级（1-10，数字越小越高）
- Planner (规划器) 创建的场景默认最高优先级
- 同级场景随机选一个

6.22之后安全类场景归CDC了，所以⑮里不再存在"安全类和体验类冲突"的问题。剩下的都是体验类之间的仲裁。
步骤h：端云分类
判断这个场景在端侧执行还是留在云端。
留云端：需要大模型推理的、需要云端API (接口) 的（天气、POI 兴趣点）
发端侧：需要VLM (视觉模型) 的、需要定时器的、弱网也要能跑的
步骤i：场景下发
通过L14下发到㉔端侧触发器。
三种下发方式：全量（第一次同步所有场景）/ 增量（只发新增/修改的场景）/ 灰度（试运行，只下发到部分车辆）
弱网下发失败
→ 重试3次，间隔递增 → 超过重试 → 标记"待下发"，网络恢复后补发
★ 步骤j：event（事件）组装 + push（推送）—— 6.22核心新增
这是6.22加的最重要的一步，也是铁律④的工程实现。
⑮做的事，一步一步：

j-1：持续监听⑫云端信号库的信号快照。每次有新数据来，或每5秒定时检查一次。

j-2：拿这一次的快照和上一次的快照做对比。逐字段看有没有变化：
- rear_cover_position: "座椅上" → "地板上" ← 变了，阈值穿越
- gear_position: "P" → "P" ← 没变，跳过
- cabin_temp: "32" → "32" ← 没变，跳过

j-3：检测到穿越的字段，过抖动过滤（同步骤f，30秒去重）。

j-4：过了过滤的穿越，组装成语义化event (事件)：
- 原始穿越：{field: "rear_cover_position", from: "座椅上", to: "地板上"}
- 组装后event：{name: "后排覆盖物位置变化", detail: "从座椅到地板", signal: "rear_cover_position", time: "14:32:09"}

j-5：查订阅表——哪些Advisor (顾问) 订阅了这个event？
- ⑰Dynamic Advisor（"照看宝宝"那个实例）订阅了 → push给它
- 如果⑯Static Advisor（舒适顾问）也订阅了"覆盖物变化" → 也push给它
- 各自独立消费，不互相影响

一个信号变化同时触发多个event
比如"档位从R变P"同时触发"倒车完成" + "车辆停稳"两个event → 各自独立组装，各自push给各自的订阅者
网络恢复后信号批量同步，瞬间产生大量event
→ 去重+合并：同一个穿越只算最新的一次。不会因为补发了5分钟的信号就触发50个event

七、㉔端侧触发器引擎：内部流水线的逻辑 已定 + 6.22 scope收窄

㉔的核心职责：接收云端⑮下发的场景 → 在端侧本地做规则判定 → 调度VLM执行 → 弱网也能跑。
6.22之后scope收窄了：不再接CDC闭环场景，只处理云端下发的动态任务。

内部流水线逐步逻辑

步骤a：场景更新
接收L14云端⑮下发的场景。三种方式：全量（第一次同步）/ 增量（新增/修改）/ 灰度（试运行）。
收到后本地持久化存储——重启不丢。
端侧重启后场景丢失
→ 从本地持久化恢复 + 同时向云端请求一次全量同步做校验
云端下发的和本地缓存的冲突
→ 以云端为准，覆盖本地
步骤b-c：信号扫描 + 规则判定
从④端侧信号库读取信号快照。只扫这个场景关注的字段。
端侧只做死规则（纯数值比较）。温度>阈值？定时器到了？围栏进入了？
不做模型推理——端侧没有轻量级语义解析模型，非结构化数据（VLM文本输出）的判断留在云端⑮做。
步骤d：优先级排序 P0-P4
这是㉔内部的优先级，管的是"触发器自己内部多个场景之间谁先执行"。
注意区分：这个优先级和AIBOX任务调度模块的优先级是两套。㉔管"场景触发顺序"，AIBOX管"VLM任务执行顺序"。

P0 (最高)：VQA (视觉问答) / 下车安全 / 遗留检测
P1：停车寻车
P2：儿童照看
P3：AlwaysOn (常驻观测)
P4 (最低)：通用视觉通道 / 低优任务

互斥组 (mutex)：舱内摄像头同一时间只能给一个VLM任务用。
- VLM_CABIN (舱内摄像头互斥组)
- FRONT (前摄像头互斥组)
高优抢占低优，低优排队等。P0不冷却（立即执行），高频任务3秒冷却，低频5秒冷却。
步骤e-f：冷却 + 下发运行指令
冷却 = 两次执行之间的最小间隔时间。防止同一个任务1秒内触发100次。
给㉕感知器 (VLM调度器) 下发运行指令，7个字段：
task_id (任务编号)、task_goal (任务目标)、priority (优先级)、timeout (超时时间)、mutex_group (互斥组)、trigger_source (触发来源)、expire_at (过期时间)
步骤g-h-i：VLM执行 → 终态 → 回写信号库
㉕感知器调度VLM执行观测任务。
任务执行完后，四种终态：
- success (成功)：得到了有效观测结果
- empty (空)：观测了但没发现目标
- timeout (超时)：规定时间内没完成 → 释放摄像头资源，下个扫描周期可以重新触发
- failed (失败)：摄像头异常/模型崩溃 → 上报错误日志，不自动重试
VLM的观测结果写回④信号库 (L16)。
注意：回写后可能触发㉔自己的二次规则判定（循环）。比如VLM输出"温度38度"→ 写入信号库 → ㉔的另一个场景正好是"温度>35就开空调" → 被触发。
会不会无限循环？
→ 步骤f的抖动过滤兜底：30秒内同值不重复触发。加上cycle_count计数，超过阈值停止循环。
步骤j：非VLM的动作直接执行
如果不是VLM类任务（比如是定时器到了/围栏触发了），㉔直接在端侧执行：
→ ㉘车控执行（弱网也能跑）
→ ㉖TTS (语音合成) 播报（弱网有本地TTS兜底）
→ ㉙HMI (人机界面) 屏幕显示

弱网场景：车进隧道断网了，㉔本地定时器到了 → 不需要云端 → 直接㉖TTS播报"快到目的地啦"。这是端侧触发器存在的价值——弱网闭环。

八、任务中心对接：佳杰需要触发器提供什么 6.22结论

这块是佳杰在51:07-55:08提出来的，之前没聊过，会上第一次摆到台面。

佳杰的需求逻辑

佳杰的问题和推导
任务中心需要在端侧显示当前有哪些任务。
用户在车机屏幕上能看到："充电桩靠近自动开盖"、"帮我照看宝宝"、"10分钟后关加热"...可以开关、删除、找回。
这些任务的数据从哪来？
佳杰 51:59 说了："我是只从你端上取数据的。你需要第一个建设的能力就是端云的任务同步机制。"
→ 云端配置的任务（运营预设的+用户临时创建的）要同步到端侧。端侧要有一个持久化数据库存这些任务。弱网时也能看到任务列表。
问题：触发器里的数据不全都是"任务"。
佳杰 52:33 说了关键区分："你可能需要一些字段来区分它是任务级的数据，还是感知级的数据。因为我不可能把所有你触发器里面很原始的订阅的信号都拿过来做显示。"

任务级（用户看得懂，在任务中心显示）：
- "充电桩靠近自动开盖" — 用户能理解的功能
- "帮我照看后排宝宝" — 用户自己说的
- "10分钟后关座椅加热" — 定时任务

感知级（底层实现细节，用户看了没意义，不显示）：
- "订阅信号：rear_cover_position变化" — 这是"照看宝宝"任务的底层实现
- "监听信号：charger_distance < 50m" — 这是"充电桩自动开盖"的底层实现

→ 触发器需要一个字段来区分，比如 display_level: "task" / "signal"。任务中心只拉 display_level = "task" 的数据。
佳杰需要的接口 (API)：
接口做什么举例
read (读取)拉取所有 display_level="task" 的任务列表返回 ["充电桩自动开盖", "照看宝宝", "10分钟后关加热"]
enable/disable (启用/停用)用户关掉某个任务，触发器要真的停掉这个任务用户关掉"充电桩" → ㉔不再监听充电桩信号
delete (删除)从待办列表移除（软删除，标记不显示）用户删了"充电桩" → 任务中心不显示，但场景定义还在
restore (找回)从场景中心找回已删除的任务用户在"推荐场景"里又点了"充电桩" → 恢复显示
边界情况
  • disable (停用) 后再 enable (启用)：任务状态要恢复到disable之前。中间漏掉的event不补发。
  • 弱网时云端有更新但端侧拿不到：网络恢复后做一次全量对账 + 增量补发。
  • 怎么判断哪些是任务级：佳杰 53:54 说"这些都是预设的，一定是可穷举的"。产品先标一轮，大家看一下。

九、AlwaysOn（常驻观测）调度归属 6.22结论

西钺问了（57:25）：AlwaysOn (常驻观测) 的调度是触发器㉔做吗？

佳杰 57:47：AIBOX上有一个模块是专门对所有请求做优先级排序的。AlwaysOn优先级大概是多少、VQA优先级如何、其他视觉类需求优先级如何——优先级低的排队、等。
结论：AlwaysOn (常驻观测) 的调度在佳杰的AIBOX任务调度模块里，不在触发器㉔里。

触发器管"什么时候触发"（什么condition满足了就触发什么action）。
AIBOX任务调度管"谁先执行"（VLM同时有多个任务时优先级排序）。

㉔内部也有P0-P4优先级，那是管触发器自己场景之间的触发顺序，和AIBOX的任务执行优先级是两层不同的东西。
