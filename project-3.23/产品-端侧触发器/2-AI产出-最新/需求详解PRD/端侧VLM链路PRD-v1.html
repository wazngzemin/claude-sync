一、需求背景与问题

1.1 为什么需要这份PRD

端侧VLM（视觉语言模型）是Driver Agent 2.0在赛力斯车型上的核心能力：通过AIBox搭载的端侧VLM引擎，持续感知车辆内外环境（天气、路面、乘客、充电桩等），为云端Planner提供视觉信息。

问题是：VLM从"被触发"到"数据到达云端"这条链路，涉及4个团队（触发器/VLM SDK/数据中心/云端端状态），之前没有一份文档把整条链路的产品需求写清楚。7.6下午的专项会上，链路设计发生了重大变更（从串行变并行双通道），必须有一份PRD把定版结论落地成可执行的需求。

1.2 7.6之前的链路问题

旧链路（7.6上午之前的理解）：触发器调VLM → VLM执行 → 结果返回给触发器 → 触发器整包透传给AI Service → AI Service上报云端。

问题：
① 触发器承担了"VLM数据透传"的职责，但触发器本质是信号处理模块，不该碰VLM原始数据
② VLM数据量大、频率高（AlwaysOn每3-5秒一帧），经触发器中转增加延迟
③ 触发器无法解析VLM原始输出（一个大JSON），数据过触发器没有任何业务价值
④ 职责不清导致异常处理模糊——VLM执行失败了，到底谁负责重试？谁负责上报？

1.3 7.6定版的核心变更

#变更点旧方案新方案（7.6定版）
1VLM数据流向VLM → 触发器 → AI Service → 云端（串行）VLM SDK → 数据中心（数据通道·直送）
VLM SDK → 触发器（控制通道·状态回调）
（并行双通道）
2触发器角色触发 + 接收结果 + 透传数据只触发 + 只接收状态回调，不碰VLM数据
3数据中心角色被动接收端状态上报主动订阅VLM SDK topic + 整包缓存 + 透传云端
4异常职责不清晰触发器保证"发送成功"，VLM SDK保证"执行成功"
5㉔-U模块定义VLM数据透传上报触发器自身事件上报（开关变更/异常/心跳）

二、整体架构：双通道分离

2.1 一句话说清楚

触发器负责调度（告诉VLM SDK"去看什么"）→ VLM SDK负责执行（调摄像头、跑模型）→ 执行完分两条路回传：
• 控制通道：状态回调给触发器（"我执行完了，成功/失败"）→ 触发器据此管调度状态
• 数据通道：VLM原始输出直送数据中心（"这是我看到的东西"）→ 数据中心整包透传云端

核心原则：触发器管调度不管数据，数据中心管数据不管调度。两条通道并行，互不依赖。

4.3 待评审：数据回传路径

7.6会上未定版。丁彬原话："one shot 等需求导入再聊。"
AlwaysOn的双通道已确认（数据直送数据中心），但OneShot的VLM数据如何到达云端还有两个方案：
方案路径优点缺点倾向
A走数据中心直通（同AlwaysOn）
VLM SDK → topic → 数据中心 → 云端		• 架构一致，端侧只有一条数据通道
• 触发器完全不碰VLM数据
• 维护成本低		• 触发器不知道推理的具体内容
• 无法在端侧做基于结果的本地决策		—
B经触发器中转
VLM SDK → 触发器 → 数据中心 → 云端		• 触发器能知道具体结果（如"看到充电桩"）
• 可在端侧做本地决策（如直接下发车控指令）		• 触发器要解析VLM数据
• 增加耦合和延迟
• 与双通道原则矛盾		—
产品建议：优先方案A（与AlwaysOn统一），除非后续有明确的"端侧基于VLM结果做本地决策"的需求（如充电桩识别后端侧直接开充电口盖，不走云端）。预计7月中旬需求导入后评审。

五、数据中心需求

5.1 核心职责

数据中心在VLM链路中承担"数据汇聚+透传"角色，是VLM原始输出从端侧到云端的唯一通道。

#需求详细描述会议依据
1订阅VLM SDK topic向VLM SDK订阅指定topic（按具体topic，不全量订阅）
目前需订阅3个AlwaysOn任务的topic
新增任务时按需添加		斯华："有topic的，具体的topic"
丁彬："数据中心就订阅这三个就行"
2接收VLM原始输出订阅后，每次VLM推理完成，数据中心自动收到完整JSON
无需触发器中转		汪帅："AI Box产生的事件，结果返回直接到数据中心"
3整包缓存收到后在内存中缓存最新一包
key = task_id（如outdoor_perception）
value = VLM原始JSON		按需存储，不全量缓存所有历史
4整包透传云端按端状态上报格式，将VLM原始输出整包透传到云端
不解析、不拆字段		丁彬："直接整包透传到云端去解析"

5.2 订阅机制

项目说明
订阅时机数据中心启动时即注册订阅，不依赖触发器是否已开启AlwaysOn
（盲订——订阅后有数据就收，没数据就等）
订阅粒度按具体topic订阅（如 outdoor_perception / cabin_dynamic / cabin_static）
不订阅不相关的topic（如赛力斯增强哨兵、迎宾等）
新增topic后续新增VLM任务时，由产品提需求告知数据中心需要订阅哪个topic
（如OneShot的 charging_port）

5.3 存储策略

7.6会议暂定按需存储
汪帅建议全量缓存（四五百个事件约10M内存），李鑫倾向按需。暂定：
• VLM原始输出：收到即透传，最新一包缓存在内存
• 车控信号K-V：后续触发器有需求时再提（如需要从数据中心订阅某个信号的变更事件）
• 全量缓存方案保留，待评估内存开销后再定

5.4 透传规范

项目说明
上报格式按现有端状态上报通道的格式（已有链路，赛力斯四里已打通）
不解析不拆解VLM原始JSON中的字段（如不单独提取 weather 或 passenger_count）
整包作为一个value上报
后续拆解如果触发器未来需要依赖VLM输出中的某个字段做条件判断，该字段必须被数据中心拆为独立key → 需产品提需求

六、接口定义

6.1 触发器 → VLM SDK：运行指令

{
  "task_id":         "outdoor_perception",     // 任务唯一标识
  "task_goal":       "识别前方天气路况",         // 任务目标描述
  "camera":          "front",                  // 摄像头: front / cabin / rear
  "priority":        "P3",                     // 优先级: P0-P4
  "timeout":         3000,                     // 超时时间(ms)
  "mutex_group":     "VLM_FRONT",              // 互斥组: VLM_CABIN / VLM_FRONT
  "trigger_source":  "alwayson",               // 触发来源: alwayson / condition_match / timer
  "expire_at":       null,                     // 过期时间(null=常驻)
  "execution_type":  "periodic"                // 执行类型: periodic / oneshot
}
调用方式：同步接口。调用后立刻返回发送状态（成功/失败），推理结果通过异步回调返回。

6.2 VLM SDK → 触发器：状态回调（控制通道）

{
  "task_id":    "outdoor_perception",
  "status":     "success",           // success / empty / timeout / failed
  "timestamp":  1720252800000
}
status值含义触发器处理
success推理成功，有识别结果更新任务状态 → AlwaysOn等周期后下一轮 / OneShot标记完成释放互斥
empty推理成功但没看到目标（有效结论，非异常）同success处理
timeout推理超时记录异常 → ㉔-U上报 → 不写伪结果（红线）
failed推理失败（模型崩溃/摄像头异常等）记录异常 → ㉔-U上报 → OneShot一次重试，AlwaysOn由SDK恢复

6.3 VLM SDK → 数据中心：topic发布（数据通道）

topic: "vlm.outdoor_perception"

payload: {
  "task_id":    "outdoor_perception",
  "result":     { ... },             // VLM原始输出(完整JSON)
  "timestamp":  1720252800000
}
数据中心通过订阅 vlm.outdoor_perception / vlm.cabin_dynamic / vlm.cabin_static 等topic接收数据。具体topic命名需斯华/李鑫对齐。

6.4 数据中心 → 云端：端状态上报

复用现有端状态上报通道，将VLM原始输出整包作为一个字段上报。具体格式沿用赛力斯四里已打通的链路，由李鑫/韩杰对齐。

七、异常处理

7.1 职责切割原则

触发器保证"调用请求发送成功"——发了就完事，不管VLM内部怎么跑
VLM SDK保证"收到后内部执行成功"——自己做鲁棒，不依赖触发器补发
数据中心保证"收到的VLM数据完整透传云端"——订阅后就透传，不加工

每一方只保证自己这一段，不替别人兜底

7.2 异常场景全量

#异常场景谁发现处理方式会议依据

八、产品红线

红线1：超时/失败时不写伪结果。VLM没看到就是没看到（empty），超时/崩溃不能编造一个"没看到"的结果写入信号库或上报云端。
红线2：触发器不碰VLM原始数据。VLM的原始JSON只走数据通道到数据中心，触发器只收状态回调（4个status值），不接收、不解析、不透传VLM数据。
红线3：端云必须基于同一份规则。端侧触发器和云端触发器对同一条规则的判定结果必须一致，否则就是bug。规则由云端下发，端侧执行。
红线4：数据中心整包透传，不解析。在产品明确提出拆字段需求之前，数据中心对VLM原始输出不做任何解析/拆解/过滤。
