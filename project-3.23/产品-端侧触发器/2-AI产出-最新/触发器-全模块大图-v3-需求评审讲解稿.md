## 〇、全局视角：这张图在讲什么

**一句话**：一个用户请求或一个传感器信号，从进入系统到最终执行或回复用户，经过了哪些模块、做了什么判断、数据怎么流转。

**三方架构**（6.22定版）：
- **云端（字节服务器）**：⑨拒识仲裁 → ⑬规则配置 → ⑮触发器引擎 → ⑯⑰Advisor → ⑲⑳Director → ㉒工具库
- **端侧AIBox（字节）**：①②用户输入 → ③④信号库 → ⑤⑥白名单 → ㉔端侧触发器 → ㉕感知器
- **CDC底软（赛力斯）**：固定闭环功能（起步安全/迎宾/充电大师等）→ 直读信号 → 直接执行

**端云分界线**就是图中间那条虚线——每条穿越它的橙色粗线都要走网络，弱网会卡。

**四条铁律**：
1. 动态注册经⑮云端触发器中转
2. 端结果→④信号库→㉑Context（不直推不跳级）
3. 触发器只读④信号库（不碰原始信号源）
4. **condition→㉑Context供读取 / event→⑮组装push给Advisor**（6.22最核心）

## 一、端侧输入层：信号从哪来

### ① 用户语音query → ② ASR语音转文字

用户在车里说了一句话。①拿到原始音频，②在端侧做ASR转成文字。

**连线 L1**：①→② 纯端侧传递。
**连线 L2**（橙色粗线）：②→⑨ 跨端云上传——文字上到云端⑨做意图判断。

> 例：用户说"帮我照看一下后排的宝宝" → ASR转文字 → L2上云

### ③ 端信号源（6类）→ ④ 端侧信号库

除了语音，车上有6类传感器持续产生数据：

| 类别 | 举例 |
|------|------|
| 1. 端状态 | 四门两盖/车速/档位/座椅/安全带/上下电/充电 |
| 2. 视觉VLM | 端模输出：人物动作/物品位置（客观状态） |
| 3. 埋点 | App操作/点击/滑动 |
| 4. 电子围栏 | 进出区域（[需10/11]） |
| 5. 音频 | 音量/切歌/循环（[需13]） |
| 6. 时间 | 定时器/日期（[需7]） |
| + 赛力斯CDC事件 | 接续导航/展车点击等 |

**连线 L8**：③→④ 所有信号写入端侧信号库，统一存储为标准格式快照（key/value/source/timestamp/ttl/编号SIG-T-012）。

**6.22新增原则——端模只报客观状态**：VLM输出"坛子当前在地上"，不判断"坛子掉了"。掉没掉是上层逻辑的判断。

### ④ 端侧信号库的三条出路

| 连线 | 去向 | 说明 |
|------|------|------|
| L9（橙色粗线） | ⑫云端信号库 | 端信号全集同步上云 |
| L15 | ㉔端侧触发器 | 端侧触发器读信号做本地判定 |
| （隐含） | ㉑Context | 按策略上报（变化时/定时），作为Advisor和Director的决策依据 |

**铁律③**：触发器只读④信号库，绝不直接监听③原始信号源。

## 二、云端意图分流：⑨ 拒识仲裁意图模型

用户的文字上到云端后，第一关是⑨。它做四路分流：

| 连线 | 分流结果 | 去向 |
|------|---------|------|
| L3（橙色粗线） | ①车控指令（"打开车窗"） | → 端侧⑤离线fc+白名单 |
| L4 | ②条件话（"10分钟后关加热"） | → ⑩意图模型解析 |
| L5 | ③非车控复杂（"照看宝宝"） | → ⑲Director汇入 |
| L6 | ④拒识成功 | → ㉚Context事件日志 |

### ⑤ 离线fc+白名单 → ⑥ 命中？

⑤是端侧的函数映射表。"打开车窗"这种简单车控，直接映射API。

⑥判断白名单里有没有：
- **命中**（L7）→ ⑦原子车控执行 → ⑧卡片→用户。端侧闭环，延迟极低
- **未命中** → 回到云端⑲Director汇入，走完整推理链路

> 例："把车窗打开" → ASR → ⑨判车控 → L3回端 → ⑤白名单命中 → ⑦开窗 → 全程不走大模型

## 三、条件任务注册：⑩ → ⑬ → ⑮

用户说条件型的话（"10分钟后关座椅加热"/"到公司了提醒我拿外卖"），走这条路：

### ⑩ 意图模型解析 [需2]

把自然语言拆成 condition + action：
- "超25度开空调" → {temp>25, 开空调}
- "到公司了提醒我拿外卖" → {位置=到达公司, 语音提醒"拿外卖"}

**连线 L11**：⑩→⑬ 注册到规则配置平台。

### ⑬ 火山后台/规则配置平台

这是触发器的"管理后台"，内部展开：

| 子项 | 说明 |
|------|------|
| 6 API | signals / actions / scene / update / query / status |
| DSL五结构 | Condition / Option / Action / Param / ExecuteInfo |
| 场景定义 | conditions + dynamic_conditions + actions + priority(1-10) + effective_scope + execute_policy |
| 频控 | 总100次/天100/上电10/间隔60s |
| 注册来源 | 用户App / 产品预设 / 运营 / 静态Advisor / Planner |

**连线 L13**：⑬→⑮ 场景注册进触发器引擎。

## 四、⑮ 云端触发器引擎——系统心脏 + 事件组装中心

这是整张图最核心的模块。内部流水线展开 a→j：

| 步骤 | 说明 |
|------|------|
| a. 场景注册/生成 | 收⑬配的场景，排队扫描 |
| b. 信号扫描 | 信号更新或每5s扫一遍（只扫该车） |
| c. 规则判定 | 死规则：温度>45 / 距离<100m / 循环10 |
| d. 模型推理 | 非结构化：VLM"明骏上车了" |
| e. 条件全达成？ | 例：配"主驾上车 且 舱内<10℃ 且 舱外<5℃" → 信号"舱内9/舱外3/主驾上车" → 全中 → 开空调26度风量3 |
| f. 数值抖动过滤 | 例：车速>130，区间125-135，30s内不重复 |
| g. 冲突仲裁 | 多场景→优先级最高(1-10)，同级随机，Planner默认最高 |
| h. 端云分类 [需9] | 决定场景在端执行还是留云 |
| i. 场景下发 | 全量/增量/灰度 → L14到㉔端侧触发器 |
| **j. 事件组装+push（6.22核心新增）** | 监听⑫信号变化→阈值穿越检测→组装事件→push给Advisor |

### j. 详解——6.22最关键的变更

**之前的理解**：Advisor轮询Context拿状态变化 → 自己判断是不是"事件"
**6.22纠正**：⑮负责做"状态变化检测 + 事件组装"，然后push给Advisor。Advisor只管等push。

**具体机制**：
1. ⑮持续监听⑫云端信号库的信号快照
2. 检测到阈值穿越（last=F & this=T），如"档位从R变为P"
3. 组装成语义化事件："倒车完成"
4. push给订阅了此事件的Advisor

**输出四个方向**：

| 连线 | 方向 | 说明 |
|------|------|------|
| L17（粗线） | → ⑯静态Advisor | push事件 |
| L18（粗线） | → ⑰动态Advisor | push事件 |
| L21（虚线） | → ⑲Director | 条件已触发通知 |
| L14（橙色粗线） | → ㉔端侧触发器 | 场景下发到端侧执行 |

> 佳杰原话："我不会去轮询你的Context。我注册了事件之后，就等待你的云端触发器给我发这个事件。"

### ⑮的铁律④诠释

用佳杰在会上的经典拆解：

> "倒完车之后，如果车里有小朋友，就把儿童锁打开"

- "用户倒完车" = **事件**（等⑮ push）
- "车里有小朋友" = **condition**（读㉑ Context）
- "把儿童锁打开" = **action**（Advisor给出）

## 五、Context → Advisor → Director：决策三角

### ㉑ Context——统一事实源

**6.22定版：只存condition/状态，不存事件。**

什么是condition：持续存在的状态描述
- "车里有一名儿童"
- "当前车速80km/h"
- "坛子当前在地上"

什么是event：状态变化的瞬间通知
- "用户刚刚睡醒了"
- "车辆刚刚驶出地库"
- → 这些不存Context，由⑮push

**连线 L26**：㉑→⑲ Context注入Director（情景/端状态/记忆）

### ⑯ 静态Advisor ×4——常驻守望者

4个永久监工：舒适 / 出行 / 情感 / 内容。

**6.22工作模式**：
1. 从⑮接收push事件（如"用户上车了"）
2. 从Context读取condition（如"车内温度32度"）
3. 综合判断 → 给⑲Director发advice

**连线 L19**：⑯→⑲ advice

> 例：⑮ push"用户上车"事件 + Context"车内32度" → 舒适Advisor → advice："建议开空调24度+座椅通风"

### ⑰ 动态Advisor——临时监工

一目标一实例，用完删。

**内部展开**：
- 4类任务（含上限）：长时×2 / 持续GUI×1 / 条件订阅×6 / 定时(含vlm)×6
- 创建脚本：{iCanDo, action: select/create}
- ↔触发器接口集（P0）：订阅/取消 · callback回调 · 事件触发 · 能力描述 · 文本→事件类型 · 优先级 · 兜底

**6.22**：订阅事件等push，不轮询Context。

**连线 L20**：⑰→⑲ advice
**连线 L22**：⑰→㉕ 感知订阅下发
**连线 L23**：㉕→⑰ 结果回传

> 例 — "照看宝宝"：Director建目标 → ⑱GoalList → ⑰建监工 → 拆解"观测毯子位置" → 向⑮注册事件 → ⑮下发㉔ → ㉔指令㉕ → 端模持续上报 → ⑮检测"座椅→地板"穿越 → push事件给⑰ → ⑰调VQA确认 → advice给Director

### ⑱ GoalList 共享看板

Director和Advisor之间的共享任务板。

**连线 L24**：⑱→⑲ 目标注入
**连线**：⑳→⑱ goal_list_update 建/删目标
**连线**：⑱→⑰ 建监工

### ⑲ Director汇入 → ⑳ Director推理（大脑核心）

⑲是6路输入汇入口：
1. query（用户的话）
2. 静态建议（⑯ advice）
3. 动态建议（⑰ advice）
4. tool_feedback（㉒工具返回）
5. 触发通知（⑮ 条件已触发）
6. GoalList（⑱ 目标队列）

**前置筛选**：按query挑相关工具/示例/knowhow + L26注入Context

**连线 L25**：⑲→⑳ 汇入推理

⑳是大脑（Seed 1.8），SP约30K tokens（角色+23工具定义+69示例+车辆知识D6X+风格"温柔高知女生"+动态变量）。

**输出三件事**：talk_or_not / talk_content / action_list

### ⑳的输出去向

| 连线 | 方向 | 说明 |
|------|------|------|
| L27 | → ㉒工具库 | action→23个下游工具执行 |
| L28 | → ㉓任务中心 | 条件/定时→Widget卡片 |
| L29（橙色粗线） | → ㉖TTS播报 | talk_content发到端侧TTS |
| L30（橙色粗线） | → ㉗㉘机器人/车控 | action→车控执行 |
| L33（虚线） | → ⑮触发器 | 对话产生的条件任务回注册 |

## 六、VQA 和 VLM 链路

### 链路一：VLM端侧持续感知

```
㉔端侧触发器 → ⑦运行指令 → ㉕感知器
                                 ↓
                           端侧摄像头(抓帧)
                                 ↓
                           端侧VLM(AIBox上运行)
                                 ↓
                           输出客观状态文本
                                 ↓
                  ㉕ → ④信号库(L16回写) → ⑫云端(L9同步)
                                 ↓
                  ⑮监听信号变化 → 阈值穿越 → 组装事件 → push Advisor
```

> 例：端模每3秒输出"副驾女性，睡眠状态"。某次变为"清醒状态" → ⑮检测穿越 → push"副驾醒了" → 舒适Advisor结合Context → advice

### 链路二：VQA云端按需问答

```
⑰动态Advisor → prompt → 视频通话接口
                              ↓           ↓
                         图+prompt    抓帧指令(L_V3 橙色↓端)
                              ↓           ↓
                       云端VQA模型   ← 端侧摄像头(L_V4 帧上传↑云)
                              ↓
                    result回⑰(L_V5) + 注入㉑Context(L_V6)
```

VQA是一问一答，不是持续跑的。⑰需要确认细节时才调。

> 例：⑰收到"毯子位置变化"事件 → 生成prompt"毯子是否掉地上？宝宝还盖着吗？" → 抓帧→VQA → "是，毯子在地板上" → ⑰确认 → advice给Director

## 七、㉔ 端侧触发器引擎（6.22scope收窄）

内部流水线 a→j：

| 步骤 | 说明 |
|------|------|
| a. 场景更新 | 接收L14云端下发 |
| b. 信号扫描 | 读④端侧信号库快照 |
| c. 规则判定/命中 | 本地逻辑判定 |
| d. 优先级P0-P4 | P0:下车安全/VQA/遗留 |
| 互斥组 | VLM_CABIN(舱内)/FRONT(舱外) |
| e. 冷却/Always-on | P0不冷却 ｜ 高频3s/低频5s |
| f. 下发运行指令 | 7字段:task_id/task_goal/priority/timeout/mutex_group/trigger_source/expire_at |
| g. VLM业务(6.22后) | VQA P0 / 儿童照看P2 / 停车寻车P1 / Always-on P3 / 通用视觉通道 / 动态任务 |
| h. 终态 | success/empty/timeout/failed |
| i. 回写信号库 | 结果L16回写→二次触发（循环） |
| j. 动作执行 | 端本地→车控/TTS 弱网闭环 |

### 6.22 scope收窄

**仁杰判定标准**：

| 归CDC底软 | 归端侧触发器 |
|-----------|-------------|
| 常驻/固定/标配/key-value | 有生命周期/动态/可运营/临时 |
| 起步安全、上下车安全 | 云端下发的定时/条件任务 |
| 睡眠提醒、无感迎宾 | 动态视觉感知事件 |
| 充电大师、哨兵 | 通用视觉通道（兜底长尾） |

> 仁杰原话："像起步安全，你放在触发器里面专门去配一下，反而还要让触发器去接底层信号、考虑稳定性和延时，不如直接在底软就把事做了。"

### 弱网执行

网络断了，㉔可独立执行：
- → ㉘车控执行（弱网本地执行）
- → ㉖TTS播报（弱网TTS）

> 例：车进隧道，云端已下发"到目的地500米提醒"。㉔本地定时器到时 → ㉖TTS "快到目的地啦" — 不需要云端。

## 八、CDC底软 · 固定闭环区（6.22大幅扩展）

以下功能**全部归赛力斯CDC**，不走触发器：

| 功能 | 为什么归CDC |
|------|-----------|
| 起步安全 | 所有车标配，纯key-value判定 |
| 上下车安全 | 固定规则，不需要动态配置 |
| 睡眠提醒 | 系统设置button，常驻功能 |
| 无感迎宾 | 系统feature，不是场景引擎 |
| 充电大师 | 赛力斯自己的业务逻辑 |
| 哨兵模式 | 赛力斯端侧闭环 |

**数据流**：直读④端侧信号库 → 直接执行㉘车控/㉖TTS，不绕触发器链路。

> 西钺原话："不要开发两个触发器，不管从开发工作还是部署资源上都是浪费。端侧触发器只在CDC，VLM在AI Box。"

## 九、6.22新增支撑模块

### 任务中心（端侧）—— 佳杰依赖

端侧需要一个持久化任务库：
- **任务级 vs 感知级**字段区分：用户能看到的"帮我照看宝宝"=任务级，底层注册的"观测毯子位置"=感知级
- **enable/disable/删除/添加回来**接口：用户在任务中心UI上操作，端侧要真的关停对应任务
- **端云任务同步**：端侧任务库 ↔ ㉓云端任务中心双向同步

### AIBox任务调度

优先级排序在佳杰的AIBox模块里做，不在触发器。always-on、VQA、其他视觉需求——优先级低的排队等。

### 通用视觉事件通道

兜底长尾的"万能通道"。当预设事件集不够用时，透传动态query给AIBox：
- **舱内静态**：性别、年龄等长时间不变的
- **舱内动态**：动作、表情等快速变化的
- **舱外**：只用前摄

**关键**：prompt是增量追加，不是覆盖。否则加了新的观测项，原来的感知就没了。

## 十、完整case串讲——"照看后排宝宝"

1. 用户说"帮我照看后排宝宝" → ①②ASR → L2上云
2. ⑨判断为复杂需求 → L5送⑲Director汇入
3. ⑲+Context → ⑳推理 → 建目标到⑱GoalList："监控后排宝宝状态"
4. ⑱建监工 → ⑰创建动态Advisor实例
5. ⑰拆解：需要持续视觉观测 → 向⑮注册"后排宝宝异常"事件
6. ⑮做端云分类 → L14下发给㉔端侧触发器
7. ㉔发运行指令给㉕感知器 → ㉕调度端侧摄像头
8. 端模持续上报"后排蓝色毛毯在座椅上" → ④信号库 → L9同步⑫
9. ⑮监听到"座椅上→地上"穿越 → 组装事件"毯子位置变化"
10. **⑮ push事件给⑰**（L18）← 6.22核心
11. ⑰调云端VQA确认："是的，毯子在地板上，宝宝身上没有覆盖物"
