把 Planner 当成"开车的副驾"：Layer 1 是驾校的必修课本 · Layer 2 是上高速前打开的高速驾驶手册 · Layer 3 是遇到不认识的路标时掏出手机查一下相似案例。一个人不可能把所有路况都背下来，但有了这套机制，他能在每一刻都"刚好懂当前要做什么"。

03

动态示例 vs 动态知识 — 必须分清

#kvk

会议里反复出现"动态示例"和"动态知识"两个词，听起来都带"动态"，但 它们是两个完全不同的对象，工作方式也完全不同。 这是评审会最容易混乱的地方，本节单独讲透。

3.1核心区别一表

维度 动态示例 (Example) 动态知识 (Knowledge)
本质Few-shot 演示（input → output 完整 JSON）陈述性内容（规则 / 事实 / 经验 / Q&A）
教什么"怎么做" — 通过模仿学习"是什么 / 边界 / 规则"
形态一个完整 case，含 user input + Planner 输出一段文本，可以是规则、参数表、Q&A、tips
粒度200–500tokens / 条50–200tokens / 条
SP 来源块第 ⑤ 块「参考示例」26,164 t69 个第 ③ 车辆知识 + 第 ④ 工具 tips 等 ~5,000 t
检索 key当前 query（改写后）/ 当前 tool_feedback 类型任务类型 / 涉及的工具 / 涉及的车型 / 涉及的域
更新触发Badcase 驱动 — 新场景就补示例车型升级 / 工具迭代 / 新经验沉淀
主要受众模型（让它模仿）模型（让它知道边界）

3.2看实际内容 · 同一 query 的两种注入

用户在主驾说"我好热"（车内 26℃，高速行驶 80 km/h）。Planner 需要看到两种东西才能正确决策：

→ EXAMPLE · 动态示例（示例 #25）
// VikingDB 召回的相似 case
{
  "query": "我好热",
  "speaker": "超然",
  "zone": "主驾驶位"
}

// Planner 期望输出
{
  "speak": "yes",
  "content": "好嘞，给你降温",
  "actions": [
    { "tool": "search_vehicle_status_info",
      "param": "查询空调状态" },
    { "tool": "vehicle_basic_control",
      "param": "空调温度调到 22℃" },
    { "tool": "vehicle_basic_control",
      "param": "风量调到 4 档" }
  ]
}
→ KNOWLEDGE · 动态知识（车辆设备 + 任务经验）
# 车辆设备知识 / D6X 空调
- 温度范围: 16 – 32℃
- 调节步长: 1℃
- 风量档位: 0 – 7
- 自动模式: AUTO 键启动

# 工具使用 tips · vehicle_basic_control
- 温度参数必须在范围内
- 风量参数为整数
- 不要重复发同一指令

# 通用任务经验
- 用户喊"热 / 冷"等模糊词时:
  · 先查当前车况
  · 再判断调到合适温度
  · 默认目标温度 22℃
两者是配合关系，不是替代关系

没有示例，模型不知道把"我好热"拆成几个 action；
没有知识，模型可能把空调温度调到 50℃（超出范围）。
只有两个都注入，Planner 才能做出"既懂方法又懂边界"的决策。

PM Talking Point · 评审会问答

问："示例和知识是不是重复了？合并维护不是更简单？"

答：① 形态不同 — 示例是 JSON case，知识是规则文本，模型学习方式不一样。② 检索逻辑不同 — 示例按 query 召回相似 case，知识按"涉及哪个工具/车型"圈出规则。③ 更新节奏不同 — 示例随 badcase 高频迭代，知识随车型/工具升级低频迭代。
合并维护反而成本更高，因为索引逻辑要做并集，召回精度反而下降。

04

Opt 1 · 输出格式精简

#opt1

最容易做、最快出收益的一项。字段名越短，首包延时越低。

原字段 优化后 省 tokens 影响
user_queryquery1每轮都出现
speaker_namespeaker1每轮都出现
speaker_positionzone2每轮都出现
talk_or_notspeak2输出首字段
talk_contentcontent1输出第二字段 · 影响首包
tool_nametool1每个 action 都出现
action_idid1每个 action
action_contentparam1每个 action
核心收益

首 token 延时减少 ~90ms。这 90ms 不是被浪费 — 它正好够做一次 VikingDB 检索，所以"延时省下来"立刻被"动态检索"用掉，总延时持平但能力大幅增强。

PM Talking Point

这一项的最大价值不是省 token，是挪出 90ms 的延时预算去做更有价值的事。
汇报时把这两件事一起讲：90ms 怎么省的、怎么花的，逻辑闭环。

05

Opt 2 · 静态 / 动态示例拆分

#opt2

这是整个专项最核心、也最难的一项。在看示例之前，必须先把"一轮"的定义搞清楚。

5.1前置概念：什么叫"一轮"

最容易误解的地方

"一轮" 不是"用户说一次话"。
"一轮" 是 "Planner LLM 被调用一次"。
用户说一句话能引发 N 轮，因为每次工具返回都要再调一次 Planner。

Planner 是个"循环大脑" — 每次用户说话或工具返回都会触发一次 Planner LLM 调用 = 一轮。

5.2为什么一句话能变成 6 轮

因为 Planner 不能"一次性想完所有步骤"。原因有三个：

原因具体表现类比
工具是真实执行的 Planner 没法"预知"工具返回什么数据。必须真调一次 POI 搜索拿到真实结果，才能基于结果决策下一步。 厨师不知道冰箱里到底有什么，必须打开看一眼
工具结果可能引发新问题 "虹桥机场"搜出 3 个 POI（T2 / T3 / P7），歧义只有看到真实数据才暴露。必须停下来问用户。 地图搜出 3 个同名地点，必须让你选
用户意图会演进 用户原本说"机场附近找日料"，听到 P7 停车场后改主意"那就去停车场接人"。意图变了，整条规划要重来。 本来想吃日料，路上看到火锅店改主意

5.4逐轮拆解 · 每轮的完整数据流

下面把 6 轮全部展开。每一轮按 5 个阶段拆：输入数据 → Planner 内部推理 → 输出 JSON → 系统侧动作 → 触发下一轮的边界。看完你会清楚"轮"到底是怎么一回事。

P1 user.in 第一轮 · 用户首次说话，Planner 拿到原始 query
①输入

用户语音 → ASR → user prompt（UP）打包送给 Planner：

// UP (user prompt)
{
  "query": "我现在从这里出发，去虹桥机场，
           然后在机场附近找个好吃的日料店去吃个饭",
  "speaker": "超然",
  "zone": "主驾驶位"
}
SP 这轮注入：静态 10 个示例 + query 检索动态 5 个
②Planner
内部推理
  1. 任务识别：一句话里有 2 件事 — 去虹桥机场 + 找日料店
  2. 关键瓶颈："虹桥机场"是模糊地点（可能 T1 / T2 / T3 / 停车场），不能直接规划路线
  3. 决策：先搜 POI 把"虹桥机场"具体化，日料的事先放着，下一步再处理
  4. 话术：要先吱一声让用户知道"听到了"，再去调工具，否则会有沉默
③输出

Planner 给系统返回一个 JSON：

{
  "speak": "yes",        // 要不要播报
  "content": "好",        // 铺垫话术
  "actions": [
    {
      "id": "1",
      "tool": "search_poi_qa",
      "param": "虹桥机场"
    }
  ]
}
④系统侧
动作
  1. TTS 立刻播报 "好"（用户听到响应，~1s 内）
  2. 异步调用 search_poi_qa("虹桥机场")
  3. Planner 挂起，等待工具结果
  4. 工具异步执行 ~1.5s 后返回
⑤为什么
必须停

核心：Planner 没法预知工具会返回什么数据。 它不知道 search_poi_qa 会返回 1 个 POI、3 个 POI、还是空，所以不能"提前规划"。只能等真实数据回来 → 触发 P2。

tool.return ↓
P2 tool.return POI 工具返回 3 个候选，Planner 要决定下一步
①输入

工具结果包装成新的 user-side 输入送回（注意：tool_feedback 也走 user 角色，因为对 Planner 来说它就是"外部信息进来了"）：

{
  "tool_feedback": "<poi1>名称:T2航站楼 距离:0km ...</poi1>
                     <poi2>名称:P7停车场 距离:0.2km ...</poi2>
                     <poi3>名称:T3航站楼 距离:0.3km ...",
  "tool": "search_poi_qa"
}

同时历史保留 — Planner 看到的完整 UP 是：
[user: P1 query] → [assistant: P1 output] → [user: tool_feedback]

SP 这轮注入：静态 10 + tool 检索 5（POI 类示例）+ 上轮 query 检索 5
②Planner
内部推理
  1. 解读返回：3 个候选，性质各不同 — 2 个航站楼 + 1 个停车场
  2. 能不能自己选？ 用户原话只说"虹桥机场"，没指定 T 几或是停车场 → 不能默认选最近
  3. 决策：必须反问用户消除歧义，先不调任何工具
③输出
{
  "speak": "yes",
  "content": "你是想去虹桥机场的T2或是T3的
              出发层，还是去P7停车场？",
  "actions": []      // 空数组 = 不调工具，纯反问
}
④系统侧
动作
  1. TTS 播报反问
  2. 没有 action → 不调任何工具，进入等待用户状态
  3. 麦克风开启 → 监听用户回复 → 触发 P3
⑤为什么
必须停

歧义不能由 Planner 自己消除。 即使 Planner 想替用户选"最近的那个"，也会引发"用户没要求过你这么做"的体验灾难。必须把决策权交回用户。

user.in ↓
P3 user.in 用户回话，但意图变了 — Planner 要重新组织规划
①输入
{

  1. 意图演进识别：原本是"去虹桥机场航站楼吃饭"，现在变成了"去 P7 停车场接人 + 在 P7 附近吃日料"
  2. 影响判断：之前那次 POI 搜索（搜的是"虹桥机场"）的结果已经作废 — 焦点变了
  3. 决策：用新的 query 串重新搜 POI，把"P7 附近"和"日料"两个约束都带上

  "actions": [

      "id": "1",

}

注意 param 已经组合了"P7 停车场附近 + 日料店"两个约束 — 这是 Planner 把多轮信息融合的能力

④系统侧
动作
  1. TTS 播报 "好"
  2. 异步调 search_poi_qa("虹桥机场P7停车场附近好吃的日料店")
  3. 等待返回 → 触发 P4
⑤为什么
必须停

同 P1 — 工具结果未知，不能盲规划路线。

tool.return ↓
P4 tool.return 3 家日料店返回 — Planner 这次能自己做选择题
①输入
{
  "tool_feedback": "<poi1>名称:鮨一日本料理(虹桥店) 
                     评分:4.8 POI榜单:虹桥日料热门榜第1名</poi1>
                     <poi2>名称:和亭日式料理 评分:4.5</poi2>

SP 这轮注入：静态 10 + tool 检索 5 + 原 query 检索 5
②Planner
内部推理
  1. 评估可选项：3 家都是日料，评分 4.2 - 4.8
  2. 能不能直接调路线？ 不能 — 用户没指定哪家，自己选会强加意见
  3. 能不能反问？ 可以，但 3 个候选有明显优劣（鮨一是热门榜第一），主动推荐比把 3 个原样念给用户体验更好
  4. 决策：推荐评分最高的鮨一，等用户确认（如果用户不爱可以再换）
③输出
{
  "speak": "yes",
  "content": "我找到3家，你可以选选，

  "actions": []
}
④系统侧
动作
  1. TTS 播报推荐
  2. 等待用户回复 → 触发 P5
⑤为什么
必须停

用户还没确认最终选择。即使 Planner 推荐了一家，也不能直接调路线 — 万一用户不喜欢推荐的呢？"推荐"和"决定"是两件事，必须等用户答 "好啊"。

②Planner
内部推理
  1. 评估：路线 1 距离短稍快（25 分钟、10 红绿灯）；路线 2 略远但红绿灯少（27 分钟、6 红绿灯）
  2. 能自己选吗？ 不能 — 这是用户偏好题（赶时间 vs 怕红绿灯多走停停），没有客观最优解
  3. 决策：把 2 条特点说清楚，让用户选
③输出
{
  "speak": "yes",
  "content": "有两条路线，第一条距离短也稍快，
              第二条红绿灯少，你看看选哪个",
  "actions": []
}
④系统侧
动作
  1. TTS 播报对比
  2. HMI 同步展示 2 条路线（地图上高亮）
  3. 等待用户选择（"第一条" / "第二条"）— 这会触发 P7（不在本例范围）
⑤为什么
必须停

2 条路线各有优劣，是偏好问题不是事实问题。Planner 不能假设用户更在乎时间还是红绿灯数量，必须让用户自己选。

两种触发源的通用模式

把 6 轮抽象一下，其实只有 2 种处理模式：

触发源固定处理路径典型出口
user.in
P1, P3, P5		 识别意图 → 评估"信息够不够下决定" → 不够则调工具
话术固定输出"好/嗯/在听"作为铺垫		 actions ≠ [] — 调工具，挂起等返回
tool.return
P2, P4, P6		 解读返回数据 → 评估"能不能自己拍板" → 不能则反问用户/给选项
能拍板就继续调下一个工具		 多数情况 actions = [] — 等用户回复
关键认知 · 这就是"Agent 循环"的本质

Planner = 一个永远在等"下一个事件"的状态机。每一次 user.in 或 tool.return 都是一次事件。它的工作就是"看到这次事件 → 输出该做什么 → 挂起等下一次事件"。
"6 轮"不是固定的，是这个对话的复杂度（用户改主意 + 工具歧义）凑出来的结果。简单 query 可能只要 2 轮，复杂多任务可能 10 轮 +。

SP 优化为什么收益被放大

正因为每个复杂请求都要 6+ 轮，每轮都要把 SP 发一遍 — SP 越长，总成本就被乘以 N 倍放大。
压缩 1 次 SP = 实际省下 N 次的计算量。这是"长度优化"收益非线性放大的根本原因。

PM Talking Point · 运维 SOP

静态名单不是一锤子买卖。上线后跑一段时间，建立动态调整机制：

· 某静态示例从未被模型实际"模仿"到 → 降级到动态库
· 某动态示例被高频召回（top-3 命中率 > 70%）→ 升级到静态层
这条 SOP 要写进运维文档，否则静态层会越来越僵化。

Opt 3 · 通过检索加载

#opt3

动态库放哪、怎么取？这一项解决"工程链路"问题。两条并行检索路径：

路径输入处理代价对策
Query 路径 用户口语化 query 小模型改写 → embedding → VikingDB 召回 top-5 多一次模型调用 ~50ms 与其他 context 模块并行
Tool 路径 上一轮 tool_feedback 直接按 tool name 取该类示例 top-5 几乎零延时 —
为什么需要 query 改写

口语化召回率低。"我好热" 在向量库里的相似示例可能是 "降温 / 开空调 / 风量调大"。直接 embed "我好热"，召回率可能 40%；改写成 "降温车控请求" 后召回率提升到 85%+。

#opt4

当前是"裸示例" — 直接把真实 tool_feedback 粘进去，每条 500+ tokens。模型只需要看到核心字段。

原则

PM Talking Point · 验证机制

这一项的关键决策：压成什么样不影响模型学习。建议立一个评测集：对比"裸示例 vs 抽象示例"两版在同一批 query 上的准确率。差距 <1pp 就放心压。

Opt 5 · 人工反馈闭环

#opt5

        ↓

        ↓

        ↓

8.1真实 case · 洗车天气

文档里给的一个例子非常说明问题：

// 用户："帮我看看下午天气怎么样，适合洗车不"

❌ 错误执行
search_weather("查询今天下午的天气情况以及洗车指数")
        → FAIL · API 不知道是哪里的下午

✅ 正确执行
search_weather("查询今天上海下午的天气情况以及洗车指数")
        → OK

根因：用户没说地点，模型也没结合 context（车辆当前位置）去推断。

修复：在动态库新增示例 #63 · 天气场景，明确演示"若用户未提地理位置，结合 context 推断"。下次类似 query 来，VikingDB 召回这个示例，模型就会模仿。

为什么动态注入比"改 SP"高级

改 SP 是全局污染 — 加一条规则全员看到。
动态注入是精准投放 — 只有触发该场景的 query 才会看到。
Badcase 越多，SP 不会变长，但召回精度持续上升。

PM Talking Point · 运营节奏

这套闭环需要建一条"Badcase → 示例"的人工流水线 — 谁收集、谁分析、谁写示例、谁验证。
建议每周固定 2 小时和算法同学一起处理新 badcase，形成节奏。否则动态库就死了。
